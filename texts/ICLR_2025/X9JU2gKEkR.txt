009                                                             A BSTRACT
010
011              Recent studies have been increasingly demonstrating that high-quality data is cru-
012              cial for effective pretraining of language models. However, the precise defini-
013              tion of “high-quality” remains underexplored. Focusing on the code domain, we
014              introduce Arctic-SnowCoder-1.3B, a data-efficient base code model pretrained
015              on 555B tokens through three phases of progressively refined data: (1) gen-
016              eral pretraining with 500B standard-quality code tokens, preprocessed through
017
                 basic filtering, deduplication, and decontamination, (2) continued pretraining
                 with 50B high-quality tokens, selected from phase one by a BERT-style qual-
018
                 ity annotator trained to distinguish good code from random data, using posi-
019
                 tive examples drawn from high-quality code files, along with instruction data
020              from Magicoder and StarCoder2-Instruct, and (3) enhanced pretraining with 5B
021              synthetic data created by Llama-3.1-70B using phase two data as seeds, adapt-
022              ing the Magicoder approach for pretraining. Despite being trained on a lim-
023              ited dataset, Arctic-SnowCoder achieves state-of-the-art performance on Big-
024              CodeBench, a coding benchmark focusing on practical and challenging program-
025              ming tasks, compared to similarly sized models trained on no more than 1T to-
026              kens, outperforming Phi-1.5-1.3B by 36%. Across all evaluated benchmarks,
027              Arctic-SnowCoder-1.3B beats StarCoderBase-3B pretrained on 1T tokens. Addi-
028
                 tionally, it matches the performance of leading small base code models trained on
                 trillions of tokens. For example, Arctic-SnowCoder-1.3B surpasses StarCoder2-
029
                 3B, pretrained on over 3.3T tokens, on HumanEval+, a benchmark that evaluates
030
                 function-level code generation, and remains competitive on BigCodeBench. Our
031              evaluation presents a comprehensive analysis justifying various design choices for
032              Arctic-SnowCoder. Most importantly, we find that the key to high-quality data is
033              its consistency with the distribution of downstream applications.

048   1    I NTRODUCTION
049
050   Pretraining large language models (LLMs) has generally relied on vast quantities of data. This
051   emphasis on data volume is especially true in specialized domains like code, where researchers
052   obtain massive code pretraining datasets by crawling platforms like GitHub (Li et al., 2023a; Rozière
053   et al., 2024; Guo et al., 2024; Lozhkov et al., 2024; Mishra et al., 2024; DeepSeek-AI et al., 2024b).
      Recent studies, however, have increasingly showed that high-quality data is crucial for effective


                                                                             1
      Under review as a conference paper at ICLR 2025


054
      pretraining (DeepSeek-AI et al., 2024a; Penedo et al., 2024; Li et al., 2024; Abdin et al., 2024),
055   including the code domain (Gunasekar et al., 2023; Li et al., 2023b; DeepSeek-AI et al., 2024b).
056
057   In the general domain, researchers have explored various techniques to curate high-quality pre-
058
      training data for language models. FineWeb-Edu (Penedo et al., 2024) uses a linear regressor built
      on Snowflake-arctic-embed-m (Merrick et al., 2024) embeddings to assess the educational
059
      value of web pages and select high-quality content, while the DCLM (Li et al., 2024) approach
060
      employs a fastText-based (Bojanowski et al., 2017) filter trained on positive examples from
061   high-quality online sources (Wei, 2024) and instruction data (Wei et al., 2024b), and random neg-
062   ative web pages to identify high-quality text. These model-based quality filters have been shown
063   to significantly enhance language model performance on downstream tasks, compared to using un-
064   filtered, large-scale datasets. Similarly, researchers have recognized the importance of high-quality
065   code data for pretraining, with Phi-1 (Gunasekar et al., 2023) using a random forest classifier on
066   CodeGen (Nijkamp et al., 2023) embeddings to select educational code samples, and DeepSeek-
067   Coder-V2 (DeepSeek-AI et al., 2024a) employing a multi-stage fastText-based (Bojanowski
068   et al., 2017) pipeline to recall web-related code data and high-quality code from GitHub, achieving
069
      state-of-the-art coding performance.
070   In this paper, we introduce Arctic-SnowCoder-1.3B, a high-performing small code model created
071   by a novel three-step training methodology focused on progressive improvements in data quality.
072   As a result of this methodology, Arctic-SnowCoder-1.3B outperforms StarCoderBase-3B (Li et al.,
073   2023a) across all evaluated benchmarks and exceeds Phi-1.5-1.3B (Li et al., 2023b) by 36% on the
074   complex and practical BigCodeBench benchmark (Zhuo et al., 2024), a benchmark that truly matters
075
      for real-world programming. As shown in Figure 1, Arctic-SnowCoder is developed through a three-
      stage, data-efficient pretraining process that progressively refines the quality of the data used. The
076
      first stage involves general pretraining for a 500B token horizon using 400B unique raw code data,
077
      which have been preprocessed through basic filtering, deduplication, and decontamination. The
078   400B raw corpus is primarily derived from the coding data used to train Snowflake Arctic (Snowflake
079   AI Research, 2024), combining cleaned The Stack v1 (Li et al., 2023a) and GitHub crawls. This is
080   followed by continued pretraining on 50B tokens, utilizing a smaller, high-quality subset of 12.5B
081   code files, repeated four times. The high-quality tokens are selected from phase one by a BERT-
082   based (Devlin et al., 2019) quality annotator trained to distinguish good code from random data,
083   using positive examples drawn from publicly available high-quality code files (Wei, 2024), along
084   with instruction data from Magicoder (Wei et al., 2024b) and StarCoder2-Instruct (Wei et al., 2024a).
085   Finally, the model undergoes an enhanced pretraining phase for 5B tokens, leveraging roughly 2B
086
      synthetic data generated by Llama-3.1-70B (Dubey et al., 2024). This process uses the phase two
      data as seeds and adapts the OSS-Instruct methodology from Magicoder (Wei et al., 2024b) by
087
      transforming lower-quality seed code into high-quality code documents. Notably, all training phases
088
      of Arctic-SnowCoder derive data from the same raw pretraining corpus, ensuring that minimal new
089   knowledge is introduced.
090
091   Arctic-SnowCoder-1.3B achieves state-of-the-art results on BigCodeBench (Zhuo et al., 2024), a
092
      coding benchmark focusing on practical and challenging programming tasks, among models of sim-
      ilar size trained with ≤ 1T tokens. Particularly, it outperforming Phi-1.5-1.3B (Li et al., 2023b) by
093
      36%. Despite being trained on 555B tokens, compared to other state-of-the-art small code models
094
      trained on trillions of tokens, Arctic-SnowCoder matches or surpasses the performance of these mod-
095   els on several benchmarks. For instance, Arctic-SnowCoder-1.3B beats StarCoderBase-3B (Li et al.,
096   2023a), trained on over 1T tokens, across all evaluated benchmarks. Arctic-SnowCoder-1.3B out-
097   performs StarCoder2-3B (Lozhkov et al., 2024), trained on over 3T tokens, on HumanEval+ (Chen
098   et al., 2021; Liu et al., 2023) (28.0 vs. 27.4), a benchmark evaluating function-level code genera-
099   tion, while remaining competitive on BigCodeBench (19.4 vs. 21.4). We conduct comprehensive
100   ablation studies to validate the design decisions behind training Arctic-SnowCoder:
101
102          • First, our findings indicate that, in general pretraining, organizing file-level data into repos-
103            itories after partitioning by programming language significantly outperforms the approach
104            of grouping data solely by repository names.
105
106          • Additionally, we determine the optimal learning rate schedule, which involves a re-warmup
107            phase followed by linear decay, as well as the ideal repetition of high-quality data during
               continued pretraining, which we find to be four times.

108
             • More importantly, our comparisons of model-based quality annotators, trained on various
109            data combinations, highlight that the consistency of pretraining data distribution and down-
110            stream tasks is crucial for achieving superior performance.
111
112   In summary, we make the following contributions:
113
114          • We introduce Arctic-SnowCoder-1.3B, a high-performing small code model trained on
115            555B tokens that benefits from progressive improvements in data quality.
116          • We demonstrate that high-quality data and synthetic data can significantly improve the
117            model performance despite being seeded from the same raw corpus.
118          • For the first time, we demystify the notion of data quality in code pretraining by systemat-
119            ically comparing model-based quality annotators trained on different data combinations.
120
             • We provide practical insights into optimal design choices for repo-level grouping in general
121
               pretraining, and optimal learning rate schedules and repetitions of high-quality data during
122            continued pretraining, providing practical guidelines for future model development.

147   2.1   R AW DATA
148
149   The raw pretraining data used to train Arctic-SnowCoder-1.3B consists exclusively of code, pri-
150
      marily derived from the coding data used to train Snowflake Arctic (Snowflake AI Research, 2024).
      This data combines cleaned versions of The Stack v1 (Li et al., 2023a) and GitHub crawls. From this
151
      data, we select 18 popular programming languages for training, similar to StarCoder2-3B (Lozhkov
152
      et al., 2024). These languages include Python, Java, C++, C, JavaScript, PHP, C#, Go, TypeScript,
153   SQL, Ruby, Rust, Jupyter Notebook, Scala, Kotlin, Shell, Dart, Swift, amounting to a total of 400B
154   unique tokens.
155
156
      2.2   G ENERAL PRETRAINING
157
158   In general pretraining, the model is trained for 500B tokens with a sequence length of 8,192 and a
159   batch size of 512 using Adam (Kingma & Ba, 2017). The learning rate follows a cosine decay after a
160   linear warmup of 600 iterations. We set the maximum learning rate to 5.3 × 10−4 and the minimum
161   to 5.3 × 10−5 , following DeepSeek-Coder (Guo et al., 2024). In this phase, we use the entire
      400B raw data without applying additional quality filtering. We start by partitioning code files by


                                                       3
      Under review as a conference paper at ICLR 2025


162
      programming language, grouping them by repository, and then concatenating them in random order,
163   similar to the StarCoder2 (Lozhkov et al., 2024) approach. In §3.3, we show the advantage of first
164   partitioning code files by programming language. We name the model produced by this phase as
165   Arctic-SnowCoder-alpha.
166
167   2.3   C ONTINUED PRETRAINING WITH HIGH - QUALITY DATA
168
169   After general pretraining, we continue pretraining Arctic-SnowCoder-alpha with 50B high-quality
170   tokens sourced from the same raw pretraining corpus. The 50B high-quality tokens are formed by
171   repeating 12.5B top-percentile code file tokens for 4 times scored by our code quality annotator.
172   Inspired by FineWeb-Edu (Penedo et al., 2024) and DCLM (Li et al., 2024), we train a linear classi-
173
      fication head on top of Snowflake-arctic-embed-m (Merrick et al., 2024), a state-of-the-art
      embedding model based on BERT (Devlin et al., 2019). The training data comprises 300k posi-
174
      tive examples, sampled from a blend of 220k high-quality open-source code files (Wei, 2024), 80k
175
      high-quality instruction data from Magicoder (Wei et al., 2024b) and StarCoder2-Instruct (Wei et al.,
176   2024a), and 300 randomly selected code documents from the pretraining corpus. Prior research on
177   code quality, such as Phi-1 (Gunasekar et al., 2023), often overemphasizes the “educational value”
178   of code, skewing models towards simpler benchmarks like HumanEval+ (Chen et al., 2021; Liu
179   et al., 2023). In §3.2, we show that our annotation leads to a more balanced enhancement of model
180   capabilities. Furthermore, given that these code documents typically exceed 1000 tokens, surpass-
181   ing the BERT context window size of 512, we improve over FineWeb-Edu’s pipeline to calculate the
182   score for each file by averaging the scores from the top, middle, and bottom sections as produced
183   by the quality annotator. In this phase, we rewarmup the learning rate for 1000 iterations from 0 to
184
      5.3 × 10−4 , the maximum pretraining learning rate, followed by a linear decay to 0. The model pro-
      duced in this phase is referred to as Arctic-SnowCoder-beta. In §3.4, we perform a comprehensive
185
      analysis that validates all of our design choices.
186
187
      2.4   E NHANCED PRETRAINING WITH SYNTHETIC DATA
188
189   In the enhanced pretraining stage, we generate even higher-quality data than in continued pretraining
190   leveraging Llama-3.1-70B-Instruct (Dubey et al., 2024) and increase the Python mix ratio to approx-
191   imately 50% while keeping the proportions of the other languages unchanged. Phi-1 (Gunasekar
192   et al., 2023) demonstrates that synthetic, textbook-like pretraining data can significantly enhance
193   model performance. However, overemphasis on such data risks skewing the model’s distribution,
194   potentially impairing its effectiveness in real-world coding tasks. For example, we show in §3.2 that
195   Phi-1.5 excels in HumanEval+ (Chen et al., 2021; Liu et al., 2023) and MBPP+ (Austin et al., 2021;
196
      Liu et al., 2023), which resemble textbook exercises, but performs less effectively on the more com-
      plex and practical coding tasks in BigCodeBench (Zhuo et al., 2024). To address this, we adapt the
197
      OSS-Instruct method from Magicoder (Wei et al., 2024b) for pretraining purposes. Originally, OSS-
198
      Instruct was originally designed to generate realistic instruction-tuning data by prompting a model
199   to create question-answer pairs inspired by open-source code snippets. In contrast, we produce
200   high-quality synthetic pretraining data by using Llama-3.1-70B-Instruct to generate high-quality
201   and problem-solving oriented code files, seeded with code documents scored in the top percentile
202   during the continued pretraining phase. In §3.2, we conduct an extensive evaluation to demonstrate
203   that each pretraining phase significantly outperforms the previous one, highlighting the effectiveness
204   of progressively enhancing data quality.

279   3.3     R EPO - LEVEL DATA IN GENERAL PRETRAINING
280
281   In the general pretraining phase, we adopt StarCoder2’s approach to group file-level data randomly
282
      into repositories through a random concatenation of file contents (Lozhkov et al., 2024). In Table 3,
      we study two methods: (1) grouping files just by repository names, meaning that each training doc-
283
      ument can be a mix of multi-lingual code files if the repository is written in different languages, and
284
      (2) partitioning files into different programming languages before grouping them into repositories,
285   meaning that each training document only focuses on one single language.
286

288   repo” treats each repository as a single training unit with possibly mixed languages, and (2) “Group
289   by language and repo” partitions data by programming language before grouping by repository.

296
      We can observe that the second approach, which we finally adopt in general pretraining, performs
      significantly better than the first one. The primary reason for enhanced performance when grouping
297
      by language before the repository is that grouping by repositories can result in training instances
298
      containing mixed file types, such as configuration files and programming files. During training,
299   we align the compute, meaning that the “grouping by repositories” approach processes fewer to-
300   kens specifically from programming files. Additionally, since files are randomly ordered, code files
301   from different languages are often unrelated. Consequently, each training example may include two
302   entirely unrelated files, which can negatively affect learning.
303
      A promising hybrid approach could involve grouping files by language within each repository. This
304
      method ensures that training examples can include multiple programming language files while main-
305   taining the cohesion of files in the same language within each group.

      4.1     C ODE PRETRAINING CORPUS FOR LANGUAGE MODELS
427
428   Code data is essential to improving the reasoning capabilities of large language models
429   (LLMs) (Aryabumi et al., 2024; Madaan et al., 2022; MA et al., 2024; Yang et al., 2024b; DeepSeek-
430   AI et al., 2024b). Typically, researchers obtain massive code pretraining data by crawling from pub-
431   lic platforms hosting code repositories such as GitHub (Li et al., 2023a; Rozière et al., 2024; Guo
      et al., 2024; Lozhkov et al., 2024; Mishra et al., 2024; DeepSeek-AI et al., 2024b). For example


                                                       8
      Under review as a conference paper at ICLR 2025


432
      The Stack v1 (Kocetkov et al., 2022) is a 3.1 TB dataset consisting of permissively licensed source
433   code mined from GitHub in 30 programming languages. Its successor The Stack v2 (Lozhkov et al.,
434   2024), built on the Software Heritage archive (Cosmo & Zacchiroli, 2017), is an order of magni-
435   tude larger, with a raw dataset of 67.5 TB spanning 619 programming languages. However, directly
436   using these massive unfiltered code for pretraining is suboptimal, because the code documents may
437   contain undesired contents or duplicates. Therefore, further preprocessing steps are needed to down-
438   scale the raw corpus, which can include deduplication (Li et al., 2023a; Rozière et al., 2024; Guo
439   et al., 2024; Lozhkov et al., 2024; Mishra et al., 2024; DeepSeek-AI et al., 2024b; Team et al., 2024),
440   PII (Personally Identifiable Information) redaction (Li et al., 2023a; Lozhkov et al., 2024; Mishra
441
      et al., 2024), benchmark decontamination (Li et al., 2023a; Lozhkov et al., 2024; Guo et al., 2024;
      DeepSeek-AI et al., 2024b), and model-based filtering (DeepSeek-AI et al., 2024b). As an example,
442
      StarCoder2 (Lozhkov et al., 2024) selects only 3 TB of data for pretraining from the 67.5 TB total
443
      data available in The Stack v2. The code pretraining corpus of Arctic-SnowCoder follows a similar
444   preprocessing pipeline, comprising approximately 400B unique tokens from a mix of filtered The
445   Stack v1 and GitHub crawls.
446
447
      4.2   M ODEL - BASED QUALITY FILTERING
448
449   In addition to common preprocessing steps like deduplication and heuristic filtering, a recent
450   trend is using model-based quality filters to select high-quality pretraining data. Phi-1 (Gu-
451   nasekar et al., 2023) employs a random forest classifier trained on top of the CodeGen (Nijkamp
452   et al., 2023) embedding layer on GPT-4 annotations, to assess the educational value of files.
453   This filter selects high-quality The Stack v1 and StackOverflow content, significantly enhanc-
454
      ing coding performance. FineWeb-Edu (Penedo et al., 2024) employs a linear regressor built on
      Snowflake-arctic-embed-m (Merrick et al., 2024), an advanced embedding model based
455
      on BERT (Devlin et al., 2019). This regressor, trained on 400k Llama-3 (Dubey et al., 2024)
456
      annotations rating the educational value (0-5) of FineWeb dataset documents, significantly en-
457   hances STEM performance. DCLM-Baseline (Li et al., 2024) uses a fastText (Bojanowski
458   et al., 2017) filter trained on positives from OpenHermes 2.5 (Teknium, 2023), high-scoring posts
459   from r/ExplainLikeImFive, and random negatives. It outperforms FineWeb-Edu in top-
460   10% selection. DeepSeek-Coder-V2 (DeepSeek-AI et al., 2024b) follows DeepSeek-Math (Shao
461   et al., 2024) by leveraging a multi-stage fastText-based pipeline to recall high-quality code and
462   math contents. Llama-3 (Dubey et al., 2024) uses fastText for recognizing text referenced by
463   Wikipedia (Wikipedia contributors, 2004) and Roberta-based (Liu et al., 2019) classifiers trained
464   on Llama-2 (Touvron et al., 2023) predictions. While prior work focuses on initial pretraining,
465
      Arctic-SnowCoder demonstrates that high-quality data from the pretraining corpus can significantly
      enhance model performance during continued pretraining. We are also the first to uncover the secret
466
      of data quality, revealing the importance of matching data distribution with downstream tasks.
467
468
469
      4.3   H IGH - QUALITY CODE DATA FOR PRETRAINING
470   Phi-1 (Gunasekar et al., 2023) is one of the first to study the impact of high-quality code data.
471   It first uses a random forest classifier to filter out high-quality code data from The Stack v1 and
472   StackOverflow, and then creates synthetic textbook-like data and exercises using GPT-3.5 (OpenAI,
473   2022), showing significant coding performance with only 50B+ training tokens. DeepSeek-Coder-
474   V2 (DeepSeek-AI et al., 2024b), pretrained for around 14T tokens in total, achieves state-of-the-art
475   coding performance, with a multi-stage fastText-based (Bojanowski et al., 2017) pipeline to
476   recall web-related code data as well as high-quality GitHub code. Arctic-SnowCoder utilizes a
477   high-quality code annotator to extract high-quality code from pretraining datasets. It then generates
478
      synthetic files seeded from this high-quality data, adapting Magicoder OSS-Instruct (Wei et al.,
      2024b) into pretraining.

481   5     C ONCLUSION
482
483   We introduce Arctic-SnowCoder-1.3B, a high-performing code model that underscores the critical
484   importance of data quality in the pretraining process. Trained on 555B tokens, Arctic-SnowCoder-
485   1.3B achieves competitive results with state-of-the-art small code models while using significantly
      fewer tokens. Our three-stage pretraining process begins with 500B tokens of general pretraining on


                                                        9
      Under review as a conference paper at ICLR 2025


486
      a raw code corpus, followed by 50B high-quality tokens scored by a quality annotator, and concludes
487   with 5B tokens of synthetic data for further enhancement. This work demystifies the notion of high-
488   quality data in code pretraining by demonstrating the key to high-quality data is its alignment with
489   the distribution of downstream applications. Additionally, the paper offers practical guidelines for
490   repo-level data grouping, learning rate scheduling, and the repetition of high-quality data, paving
491   the way for more efficient and effective code model development.
