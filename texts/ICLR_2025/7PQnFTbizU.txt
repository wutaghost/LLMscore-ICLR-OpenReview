011                                                A BSTRACT
012
013              Web agents that can automate complex and monotonous tasks are becoming es-
014
                 sential in streamlining workflows. Due to the difficulty of long-horizon planning,
                 abundant state spaces in websites, and their cryptic observation space (i.e. DOMs),
015
                 current web agents are still far from human-level performance. In this paper, we
016
                 present a novel web agent, Agent-E † . This agentic system introduces several ar-
017              chitectural improvements over prior state-of-the-art web agents, such as hierarchi-
018              cal architecture, self-refinement, flexible DOM distillation, and change observa-
019              tion to guide the agent towards more accurate performance. Our Agent-E system
020              without self-refinement achieves SOTA results on the WebVoyager benchmark,
021              beating prior text-only benchmarks by over 20.5% and multimodal agents by over
022              16%. Our results indicate that adding a self-refinement mechanism can provide an
023              additional 5.9% improvement on the Agent-E system without self-refinement. We
024              then synthesize our learnings into general design principles for developing agentic
025
                 systems. These include the use of domain-specific primitive skills, the importance
                 of state-sensing and distillation of complex environmental observations, and the
026
                 advantages of a hierarchical architecture.

029   1       I NTRODUCTION
030
031   Recent studies indicate that generative AI and automation tools could handle 60-70% of an em-
032   ployee’s tasks (Chui et al., 2023). By reducing cognitive load, saving time, and optimizing
033   workflows, these tools can potentially contribute between $2.6 trillion and $4.4 trillion to global
034   GDP (Chui et al., 2023). With the rise of digital jobs and advancements in the reasoning abilities
035   of large language models (LLMs), these models are increasingly being integrated into autonomous
036   systems for a variety of tasks. LLM-agents can be seen in applications like software engineering
037
      tasks (Jimenez et al., 2023; Huang et al., 2023a; Zhang et al., 2024b; Schick et al., 2023), personal
      device control (Yan et al., 2023; Wu et al., 2024; Zhang et al., 2024a), and web navigation (He
038
      et al., 2024; Zhou et al., 2023; Putta et al., 2024b; Lutz et al., 2024b). However, while these agents
039
      have demonstrated promising results in some areas, their performance in web automation remains
040   limited.
041
042   Several unique challenges make planning difficult in a web navigation context. First, websites are
043
      represented in HyperText Markup Language (HTML) Document Object Models (DOMs), which
      organize elements in a nested format. These lengthy, dynamic text-based representations complicate
044
      the identification of key elements in the observation space (Lutz et al., 2024b). Furthermore, DOMs
045
      often exceed the context windows of current state-of-the-art LLMs. Second, while humans can
046   naturally execute complex web tasks, agents require careful, multi-step planning. Even a simple
047   task, like a Google search (e.g. clicking the search bar, typing each key, and pressing enter),
048   involves multiple fine-grained actions. Lastly, current state-of-the-art web agents remain error-prone
049   and unreliable for deployment, underscoring the need for further advancements in this area to create
050   a more reliable system (Wornow et al., 2024; He et al., 2024; Zhou et al., 2023).
051
      In this paper, we introduce Agent-E, a state-of-the-art web agent capable of performing complex
052   web-based tasks. Our system presents several design elements that elevate challenges faced by prior
053

054
      web navigation systems. Central to Agent-E are three LLM-powered components: the planner agent
055   browser navigation agent, and verification agent. The planner agent is responsible for high-level
056   planning and task management. It breaks down the user task into a sequence of high-level tasks
057   and delegates them to the browser navigation agent. The browser agent then plans and executes
058   the lower-level steps necessary to complete the delegated task. This tiered system breaks down the
059   planning into fine-grained actions that are more manageable tasks; this insulates the planner agent
060   from the low-level details of the observation space. To further improve the interpretability of DOMs,
061   our system utilizes different DOM distillation techniques. These techniques emphasize features in
062   the DOM relevant to completing an action to prevent an LLM agent from becoming overwhelmed
063
      with the difficult observation space. In addition, our system employs a validation agent at the end of
      each task. This validation agent provides feedback on incomplete tasks, leading to a self-correcting
064
      system.
065
066   Using our proposed system, we demonstrate that web agents can achieve state-of-the-art perfor-
067   mance on realistic web navigation tasks without additional supervision. By combining our hierar-
068   chical system with DOM distillation techniques, we attain a new state-of-the-art 73.1% result on
069
      the WebVoyager benchmark (He et al., 2024), which is 20.5% higher than previous text-only web
      agents (Lutz et al., 2024b) and 16% higher than previous multi-modal web agents (He et al., 2024).
070
      Additionally, we achieve a 5.9% boost in performance using a self-refinement mechanism.

      2     AGENT-E: S YSTEM D ESCRIPTION
098
      Agent-E is built using Autogen, the open-source programming framework for building multi-agent
099
      collaborative systems (Wu et al., 2023b) and Playwright* for browser control. Our system simplifies
100   complex, long-horizon planning for web navigation workflows. Agent-E hierarchical system is
101   composed of three LLM-powered agents: Planner, Browser Navigation Planner, Validation Agents,
102   and one execution component. Each component plays an integral role in the system’s successful and
103   reliable workflow execution.
104
      To manage the different granularity of sub-tasks necessary to complete a full workflow, our system
105
      is split into a hierarchy: 1) high-level planning, which is performed by the planner agents, and 2)
106   low-level planning and execution, which is handled by the browser navigation planner and executor.
107
            * https://playwright.dev/

108
      Given a new user task, the planner agent decomposes the task into a sequence of high-level steps.
109   Then throughout the workflow, the planner agent delegates the execution of each high-level step to
110   the browser navigation subsystem and adapts to the plan based on the observation from the browser
111   navigation subsystem. Finally, once the planner indicates the workflow is completed, the validation
112   agent verifies the workflow. During workflow failures, the validation agent returns feedback to the
113   planner agents and prompts them to correct its workflow. The self-refinement mechanism is further
114   explained in Section 3.
115   To tackle the challenges of large observation spaces and fine-grained action space in browser in-
116   teractions, we introduce the notion of skills, a set of predefined actions the agents can execute.
117   These predefined can be associated with the execution of actions, or related to sensing the current
118   observation space.

150   2.1   S KILLS D ESIGN & DOM D ISTILLATION FOR B ROWSER NAVIGATION AGENT
151
152
      There are two key novel components in skills design used in Agent-E.
153
            • Sensing Skills & DOM Distillation: Agent-E supports three different DOM distillation
154
              techniques (text only, input fields, all fields) that allow the browser navigation agent to
155
              choose the approach best suited for the task (see Figure 2). If the task is to summarize
156           information on a page, it can simply use Get DOM with text only content type. If the task
157           is to identify and execute a search on a page, it can use the content type input fields. If the
158           task is to list all the interactive elements on a page, it can use all fields. This optimizes the
159           information available to the agent and prevents the problems associated with noisy DOM.
160           Another key aspect is that our DOM distillation techniques for all fields and input fields at-
161           tempt to preserve the parent-child relationship of elements wherever possible and relevant.
              This is unlike some previous implementations which use a flat DOM encoding (e.g. Lutz


                                                         3
      Under review as a conference paper at ICLR 2025


162
               et al. (2024b)). Further, to make identifying and interacting with HTML elements easier,
163            Agent-E injects a custom identifier attribute (mmid) in each element as part of sensing,
164            similar to Zhou et al. (2023) and He et al. (2024).
165
166          • Action Skills: All the action skills are designed to not only execute an action but also report
167
               on any change in state as an outcome of the action, a concept we call change observation.
               This is conceptually similar to the Reflexion paradigm (Shinn et al., 2024) which uses
168
               verbal reinforcement to help agents learn from prior failings. However, a key difference
169
               is that change observation is not directly associated with or limited to a prior failure. The
170            observation returned can be any type of outcome of the action. For example, a click action
171            may return a response Clicked the element with mmid 25. As a consequence, a popup has
172            appeared with the following elements. Such detailed skill responses nudge the agent toward
173            the correct next step.

197   to self-correct incorrect workflows. We complement our planner and browser navigation agents with

199   the agent leverages the validator’s feedback to revise its strategy and reattempt the task. The high-
200   level mechanism illustrated in Figure 3, will allow the agent to self-correct in detected failures. Note

      the validation agent is only invoked once the planner agent finishes its workflow.

      3      E VALUATION
217
218
      In this section, we test and demonstrate that our Agent-E system outperforms other web agent sys-
219   tems with the use of no additional supervision. Our results indicate that our hierarchical architecture,
220   sensing and action space, and use of self-refinement are able to make better use of LLM context win-
221   dows for planning.
222
223   WebVoyager Benchmark WebVoyager (He et al., 2024) is a recent web agent benchmark that
224   consists of web navigation tasks across 15 real websites (e.g. Amazon, Google Flights, Github,
225   Booking.com). Each website has about 40-46 tasks resulting in a benchmark dataset of 643 tasks.
226   We chose WebVoyager since it covers a diverse range of tasks across dynamic, live websites that are
227   representative of real-life use cases for web navigation. In contrast, alternative benchmarks either
228   focus narrowly on a single task domain (Yao et al., 2022), lack dynamic website observations (Deng
229
      et al., 2023), or rely on custom websites with significantly less complex DOM structures than those
      found in real-world environments (Liu et al., 2018; Zhou et al., 2023).
230
231
      Experimental Details The entire benchmark was divided among 5 human annotators. For each
232
      task, an annotator was instructed to classify the task as pass or fail along with a textual reason in
233
      case of failures. A task is considered complete only if the agent successfully finishes all parts of the
234   instructed task and remains on the designated website. Overall accuracy measures the percentage
235   of times the validator’s label and human annotator’s labels match. To remain consistent with prior
236   work benchmarks on WebVoyager (He et al., 2024), we utilize GPT-4-Turbo (gpt-4-turbo-preview)
237   as a planner and browser navigator in our Agent-E implementation. And for the validation agent,
238   we employ GPT4-Omni(gpt-4o).

      T HE NEED FOR MULTIPLE DOM OBSERVATION METHODS
335
336
      Typical HTML DOMs can be extremely large (e.g., the YouTube homepage with all DOM elements
337
      and attributes is about 800,000 tokens). Thus, it is important to denoise and encode the DOM such
338
      that only task-relevant information is presented to the LLM. However, information relevant to a
339   given task is very dependent on the task at hand. Some tasks may only need a complete textual
340   representation (e.g., summarise the current page), and some tasks may only need input fields and
341   buttons (e.g., search on google). On the other hand, more exploratory tasks may need a complete
342   representation of the page (e.g., what elements are on this page).
343
      Most previous web agents have used a single DOM representation, e.g. (Zhou et al., 2023) used
344
      an accessibility tree,(He et al., 2024) used screenshots and (Lutz et al., 2024b) used direct encod-
345   ing and denoising of the HTML DOM. However, in our view, there is no single DOM observation
346   method that suits all the tasks. Thus, Agent-E supports three different DOM representation methods
347   text only, input fields, all fields. This allows Agent-E to flexibly select the DOM representation that
348   it feels is best suited for the task. Also, this allows Agent-E to fall back to different representations,
349   when one representation unexpectedly does not work well. There were numerous examples in our
350   benchmark where these multiple DOM representations were useful. Appendix A: Figure 6 illus-
351   trates an example where Agent-E adaptively uses all fields DOM representation for interaction and
352   text only for summarization. Refer to Appendix D for quantitative evaluation comparing the flexible
353
      DOM distillation and directly using the accessibility tree.
354
355
356   C HANGE OBSERVATION HELPS GROUNDING
357
358
      Change observation is a technique where each action execution is accompanied by observation of
359
      state changes, and this is returned via linguistic feedback to the LLM. A typical scenario where this
360   is useful is when the browser navigation agent tries to click on a navigation item (e.g., click on the
361   soccer link on ESPN.com), and instead of navigating to the relevant section, the page instead opens
362   a popup menu that requires further selection. In this example, the interaction is not yet complete
363   (since completion requires clicking a popup link or selecting a drop-down entry, respectively), but
364   the browser navigation agent may assume it is complete. With change observation, the click skill
365   will return feedback to the LLM that as a consequence of the click, a menu has appeared where you
366   may need to make further selection. See Figure 4 in Appendix F for an example.
367   The purpose of change observation is to provide linguistic feedback to the LLM on whether the
368   action led to any tangible change in the environment, to inform subsequent actions. We also envision
369   efficiency improvements if the change observation can return a list of elements so that LLM can
370   make subsequent selections without again using the Get DOM skill to observe the state of the DOM.
371
      Change observation is adjacent to the concept of Reflexion (Shinn et al., 2024). However, there
372   are nuanced differences between the two. The Reflexion technique provides feedback on a prior
373   failure, by using an LLM to analyze the scalar ‘success/failure’ signal based on an action and current
374   trajectory. In contrast, change observation is not a binary signal and instead observes the change in
375   the environment as a consequence of an action (e.g. new elements added to DOM, pop-up expanded,
376   etc). Change observation is implemented using mutation-observer API to observe the consequence
377   of an action and provide linguistic feedback of actions to help the system be better aware of the new
      state of the environment, and nudge the system towards the correct next action.

      4   AGENT D ESIGN P RINCIPLES
379
380   In this section, we synthesize our learnings from the development and evaluation of Agent-E into a
381   series of agent design principles. We believe these principles can be generalized beyond the domain
382   of web automation.
383
384        1. Design with a Core Set of Primitive Skills to Enable Versatile Use Cases: An ensemble
385           of well-crafted foundational skills can serve as a building block to support more complex
386           functionalities. LLMs can effectively combine these skills to unlock a broad range of use
387
              cases. These skills should be domain-dependent; in the case of Agent-E, these primitive
              skills were click, enter text, get DOM, Open URL and Press Keys. These are only a subset of
388
              potential user actions on a page (e.g. we do not support drag, double click, right-click, tab
389
              management, etc). We consider the skills enabled in Agent-E enough for the vast majority
390           of general web automation tasks. However, websites with specialized interaction patterns
391           (e.g. right-click to select functionality) may benefit from additional skills. Examples of
392           prior related work leveraging domain-specific primitive skills include (Irpan et al., 2022;
393           Nakano et al., 2022; Lutz et al., 2024b) among several others, highlighting the generality
394           of the design principle.
395        2. Adopt Hierarchical Architectures for Managing Complex Task Execution: The idea
396           of using hierarchical AI planning for complex tasks has existed for decades (Tate, 1977;
397           Nau et al., 1991; Marthi et al., 2007); see Russell & Norvig (2009) for details. In agents
398           with multiple LLM-based components, a hierarchical architecture excels in scenarios where
399           tasks can be decomposed into sub-tasks that need to be handled at different levels of granu-
400           larity. In the case of Agent-E, this allows the high-level planner to be agnostic of browser-
401
              level details. Additionally, it aids in the identification of tasks that can be executed in
              parallel, leading to performance enhancements. It also supports the development and im-
402
              provement of various components in isolation. Note that hierarchical architectures may not
403
              always be necessary. In the case of Agent-E, if all we had to support were simple tasks like
404           navigating to specific URLs or performing a web search, a hierarchical architecture might
405           be over-complicated, and a simpler architecture may have sufficed.
406
           3. Domain-Specific State Processing Improves Efficiency and Accuracy: Depending on
407
              the domain, there may exist a large amount of environmental information, much of which
408           is irrelevant. An example is HTML DOMs for websites which may have hundreds of thou-
409           sands of tokens. This may lead to suboptimal LLM performance, especially for sequen-
410           tial decision-making tasks. Agent-E employs a variety of domain-specific processing and
411           sensing techniques to distill only task-relevant data. These include multiple DOM filtering
412           approaches that the agent adaptively uses given the task requirements. Removing as much
413           noise as possible from the environment before the system begins processing, is a crucial
414           requirement while building such agentic systems.
415        4. Integrate Linguistic Feedback to Summarize State Changes: Agent-E’s actions change
416           the state of the page, often in complex ways. We have found that, rather than relying on the
417           filtered DOM alone, explicitly detecting and summarizing state changes through linguistic
418           feedback enables the agent to more effectively understand the consequences of an action
419
              (e.g., a dialog box appeared as a consequence of the click action). Change observation
              helps refine the agent’s subsequent actions by providing a clear narrative of cause and
420
              effect, and also improved awareness of the environment. This idea is also applicable in
421
              other contexts, for example, in use cases such as desktop automation or automation in the
422           physical space (e.g robot control). Examples of systems that use related ideas in other
423           domains include (Wang et al., 2023c; Song et al., 2023; Wang et al., 2023a) among others.
424           Descriptive logging and tracking are highly beneficial in agentic systems.
425
           5. Leverage Past Experience: For agentic systems to be adopted widely, they need to achieve
426           close to human-level performance. One approach is for agents to routinely reflect and learn
427           from their past experiences. Our Agent-E system introduces the use of self-refinement
428           for web automation. The 5.9% boost in performance achieved by this mechanism shows
429           that agents are capable of identifying and self-correcting their mistakes throughout a single
430           task execution. A more efficient approach to leveraging past workflows is to establish
431           offline workflows that routinely analyze, reflect on, and aggregate past tasks and human
              demonstrations to convert them to more classical automation workflows. These automated


                                                       8
      Under review as a conference paper at ICLR 2025


432
               workflows could then be re-triggered upon a new task if it matches a workflow that has been
433            encountered in the past, with the exploratory agentic approach used only as a fallback. This
434            would enable faster and cheaper task completion, which should be a primary requirement
435            of agentic systems. Other examples of leveraging past experience can be found in prior
436            work on self-improving systems e.g. (Zelikman et al., 2022; Hosseini et al., 2024; Wang
437            et al., 2023c) and others.
438
            6. Balance Between Generic and Task-Specific Agent Design: Generic agentic systems by
439            definition can perform a wide range of tasks. However, in many practical implementations,
440            a more focused set of capabilities may be desirable. For example, Agent-E is a generic
441            web agent that can perform a wide range of tasks on the internet but is not necessarily
442            optimized for any specific task. It would be possible to optimize Agent-E for specific types
443            of tasks (e.g., form filling) or specific websites (e.g., Atlassian Confluence pages) to achieve
444            significantly higher performance. Depending on the use case, an optimized agent may suit
445            better for certain workflows than a generic version.

448
449
      LLM-based Planning and Reasoning Over the last few years, Large language models (LLMs)
450   have excelled in text generation, code generation, and the generation of multistep reasoning. This has
451   spurred the use of LLMs to solve multi-step reasoning and planning problems. The many variants of
452   ‘chain-of-thought’ techniques (Wei et al., 2022; Chu et al., 2023) encourage the LLM to produce a
453   series of tokens with causal decoding that drive toward the solution of problems in math, common-
454   sense reasoning and other similar tasks (Chowdhery et al., 2022; Fu et al., 2023; Li et al., 2023; Mitra
455   et al., 2024). With tool-usage for sensing and acting, LLMs have also been used to drive planning
456   in software environments and embodied agents e.g., (Baker et al., 2022; Wang et al., 2023a;c; Irpan
457   et al., 2022; Bousmalis et al., 2023; Wu et al., 2023a; Bhateja et al., 2023). Finally, there has
458
      been related work investigating the limits of LLMs when it comes to planning and validation. For
      examples of negative results, see (Valmeekam et al., 2023b; Momennejad et al., 2023; Valmeekam
459
      et al., 2023a; Huang et al., 2023b; Kambhampati et al., 2024) among others. In this paper, we
460
      investigate multi-step planning for specialized web agents. We find that domain-specific techniques
461   including sensing (through DOM distillation and change-observation), hierarchical planning (with a
462   low-level browser agent), and multimodal self-refinement, are crucial for state-of-art performance.
463
464
      Specialized Agents for Repetitive Tasks Beyond the examples above, and as described in Sec-
465   tion 1, there has been much recent interest in building specialized agents for the web (Zheng et al.,
466   2024a; He et al., 2024; Lutz et al., 2024b) and on device (Bai et al., 2024; Wen et al., 2024). Also
467   related is recent work on building agentic workflows to replace robotic process automation (Wornow
468   et al., 2024). Further, the work on building agents and training language models for API usage is
469   also related, given that many software tasks and workflows involve the use of APIs; examples in-
470   clude (Hosseini et al., 2021; Patil et al., 2023; Qin et al., 2024) and many more. As described in
471   Section 1, our proposed web agent employs multiple novel ideas that enable it to achieve state-of-art
472   performance on realistic web navigation tasks, significantly outperforming previous specialized web
473
      agents.
474
475   Hierarchical Planning The notion of hierarchical AI planning has been around for five decades or
476   more. Instead of planning directly in the space of low-level primitive actions, planning in a space of
477
      ‘high-level actions’ constrains the size of the plan length (and hence the size of the planning space),
      which can result in a more effective and efficient search. Examples from prior work include (Tate,
478
      1977; Nau et al., 1991; Marthi et al., 2007) and many more; see Russell & Norvig (2009) for more
479
      details. Also related is the use of temporal abstractions in planning and reinforcement learning, for
480   example, the use of options in (Sutton et al., 1999; Bacon et al., 2017). In recent years, multiple
481   papers have proposed the use of hierarchical planning for solving tasks in complex environments
482   or with embodied agents; examples include (Wang et al., 2022; Irpan et al., 2022) and others. In
483   this paper, we introduce a hierarchical architecture for web agents where responsibility for planning
484   and execution of complex web tasks is separated between a planner agent and a browser navigation
485   agent. We show hierarchical planning is a promising solution for long-horizon planning in web
      navigation.

      Self-Improving Agents Recent research has focused on enhancing the capabilities of LLMs dur-
487   ing training or inference without additional human supervision (Wei et al., 2022; Chen et al., 2023;
488   Wang et al., 2023b; Kojima et al., 2023). Techniques like chain-of-thought prompting and self-
489   consistency, as used in Huang et al. (2023b), aim to generate higher-quality outputs. Other meth-
490   ods, such as Self-refine (Madaan et al., 2023), Reflexion (Shinn et al., 2023), and REFINER (Paul
491   et al., 2024), focus on iterative refinement of outputs using actor-critic pipelines or task decomposi-
492   tion. These approaches have been successfully applied to web agents, improving the performance of
493   LLMs in web automation tasks (Putta et al., 2024a; Pan et al., 2024; Lutz et al., 2024a). In this paper,
494   we design and evaluate three different auto-validators, and use these to create self-refinement mech-
495
      anisms for our web agent. Our results indicate that self-refinement, using our text- and vision-based
      auto-validators, shows notable additional gains in web navigation tasks.

498   6   C ONCLUSION
499
500
      This paper introduced Agent-E, a web agent that significantly advances the ability to handle complex
501
      web navigation tasks. Web-based automation faces key challenges such as the complexity of DOM
502   interpretation and long-horizon task planning. Agent-E addresses these with flexible DOM distil-
503   lation techniques to focus on relevant content, hierarchical task management to reduce error-prone
504   low-level decisions and a self-refinement mechanism that allows the agent to correct its workflow
505   without human intervention. Our evaluation of the WebVoyager benchmark demonstrates Agent-E’s
506   ability to overcome these web navigation challenges. With a 73.1% success rate, Agent-E without
507   self-refinement sets a new state-of-the-art for web agents, surpassing prior text-based and multi-
508   modal systems by 20.5% and 16%, respectively. We observe another 5.9% improvement when self-
509   refinement is added to this system. We presented our learnings in the form of eight general design
510
      principles for developing agentic systems that can be applied beyond the realm of web automation.
511   Although Agent-E presents state-of-the-art results on web navigation tasks, there are several key
512   observations and space for improvement. First, unlike prior state-of-the-art agents from He et al.
513   (2024), our planning and browser navigation agent is not multimodal. Transitioning the browser
514   navigation agent to handle multi-modal observations may improve its sensing capabilities. Second,
515   although self-refinement shows the best performance, this outcome-based refinement system comes
516
      at a cost (i.e. requiring tasks to take 1.2-2x longer to complete). Lastly, we observed that differ-
      ent modalities of validation agents perform best for different tasks. This highlights the need for
517
      task-specific validation systems. In conclusion, Agent-E’s novel approach effectively tackles key
518
      challenges in web navigation, offering a robust, adaptable framework that advances agentic sys-
519   tems in web automation and beyond. While task completion times can still be optimized, Agent-E
520   provides a significant leap forward in agent performance and reliability.

      E THICS S TATEMENT
523
524
      As web agents like Agent-E move beyond research prototypes, they can raise important ethical con-
525   cerns. First, web agents that operate on a personal device may introduce privacy issues for the
526   user. These agents may have access to user sensitive information including passwords and finan-
527   cial data. Second, such agents, if used by a malicious user, could potentially be used for harmful
528   purposes like sending spam and unauthorized web scraping. Thirdly, the widespread deployment
529   of web agents could violate websites’ terms of service. While our research advances the technical
530   capabilities of web agents, we recognize the critical importance of understanding failure modes and
531   potential risks before real-world deployment. We acknowledge that benchmark performance alone
532   is insufficient for ensuring safe deployment. Future work must establish robust security frameworks,
533
      access controls, and oversight mechanisms before web agents can be safely entrusted with user data
      and credentials. We emphasize that human oversight remains essential for deploying these systems
534
      responsibly.

       D    F LEXIBLE DOM D ISTILLATION A BLATION
1027
1028
       A key distinction between Agent-E and other web agents (e.g. He et al. (2024) and Lutz et al.
1029   (2024b)) is that Agent-E supports multiple DOM observation techniques that the LLM can choose
1030   from, given the task at hand. Our DOM distillation method consists of three DOM observation
1031   techniques which can be selected by the Browser Navigation Agent depending on the task:
1032
1033        1. all fields: This is the most comprehensive DOM representation, provided in JSON format.
1034           It starts with the Accessibility Tree (AXTree) of the webpage—a simplified version of
1035           the DOM that omits non-semantic elements like <div> tags used purely for styling. We
1036           the enrich this view with additional details, such as the names of HTML tags and inner
1037
               text content where necessary. This representation is useful for tasks requiring detailed
               interaction with page elements.
1038
1039        2. input fields: This is a subset of all fields where only input fields and interactive el-
1040           ements from the DOM are returned. This strips away all the non-interactive text elements
1041
               and allows the agents to use a much more succinct version of the DOM for purely interac-
               tion purposes.
1042
1043        3. text only: This is a plain text view of the current page (gathered by using
1044           body.innerText in JavaScript of the current page). This will not have DOM iden-
1045
               tifiers to interact with screen elements but will have full text visible on the page. This is
               best suited for summarizing page content or answering specific questions from the page
1046
               (e.g., What is the price of iPhone 16? or Is this product waterproof?). Answering such
1047
               questions with all fields is a lot more challenging since the information can be frag-
1048           mented across multiple DOM fields and thereby multiple JSON nodes.
1049
1050   Prior work typically uses a single DOM observation method such as a simplified version of the
1051   HTML DOM based on heuristics (Lutz et al. (2024b)) or directly uses the accessibility tree (AxTree)
1052   of the current page (e.g. Zhou et al. (2023), He et al. (2024)).
1053
       To better understand the value provided by the flexible DOM distillation, we conducted an evaluation
1054   comparing Agent-E with flexible DOM distillation with a variant that directly uses AxTree. We
1055   performed the evaluation using a subset of WebVoyager (75 tasks = 5 tasks randomly sampled tasks
1056   from each website * 15 websites), the same as described in Appendix C.2. The results are presented
1057   below in Table 10. Note the experiment below is Agent-E without self-refinement.
1058
       Flexible DOM distillation significantly improves success rates (+16%) by tailoring observations to
1059
       task-specific needs. Using AXTree directly is marginally faster since the AXTree enrichment steps
1060   we perform for ’all fields’ and ’input fields’ take some processing time (typically an additional 1-
1061   2 seconds per call depending on the complexity of the webpage). These findings emphasize the
1062   importance of adaptive DOM distillation in enhancing Agent-E’s effectiveness across diverse web
1063   navigation tasks.

1083   because websites are implemented using diverse approaches. For example, some websites dynam-
1084   ically add new elements to the Document Object Model (DOM) after an action. Other websites
1085   achieve similar effects by modifying properties like visibility, opacity, position or display styles of
1086   existing elements, without adding new ones. In Agent-E, we implement Change Observation us-
1087   ing two complementary approaches: tracking changes in aria-expanded attribute and tracking new
1088   elements added using Mutation Observer.
1089   Tracking changes in aria-expanded attribute: The ariaexpanded attribute is a standard accessi-
1090   bility feature that indicates whether a particular element (e.g., a menu or dropdown) is expanded or
1091   collapsed. By observing if aria-expanded changes from False to True, we can infer if the element
1092   has changed state (e.g. “Click action on the element [mmid=25] was performed successfully). As
1093   a consequence a menu has appeared where you may need to make further selections. Get all fields
1094   DOM to complete the action.” a relatively straightforward approach that tells the LLM that a menu is
1095   now open and likely further actions are needed. This method works effectively on websites that ad-
