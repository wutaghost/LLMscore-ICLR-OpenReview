009                                               A BSTRACT
010
011            Building precise simulations of the real world and invoking numerical solvers to
012            answer quantitative problems is an essential requirement in engineering and sci-
013            ence. We present FEABench, a benchmark to evaluate the ability of large lan-
014            guage models (LLMs) and LLM agents to simulate and solve physics, mathemat-
015            ics and engineering problems using finite element analysis (FEA). We introduce a
016
               multipronged evaluation scheme to investigate the ability of LLMs to solve these
               problems end-to-end by reasoning over natural language problem descriptions and
017
               operating COMSOL Multiphysics® , an FEA software, to compute the answers.
018
               In addition to testing state of the art LLMs, we design a language model agent
019            equipped with the ability to interact with the software through its Application
020            Programming Interface (API), examine its outputs and use tools to improve its so-
021            lutions over multiple iterations. Our best performing strategy generates executable
022            API calls 88% of the time. However, this benchmark still proves to be challenging
023            enough that the LLMs and agents we tested were not able to completely and cor-
024            rectly solve any problem. LLMs that can successfully interact with and operate
025            FEA software to solve problems such as those in our benchmark would push the
026            frontiers of automation in engineering. Acquiring this capability would augment
027
               LLMs’ reasoning skills with the precision of numerical solvers and advance the
               development of autonomous systems that can tackle complex problems in the real
028
               world.

031   1   I NTRODUCTION
032
033   While there has been a series of works demonstrating the significant potential of large language
034   models (LLMs) on analytical mathematical and scientific reasoning (Lewkowycz et al., 2022; Yang
035   et al., 2024b; Hendrycks et al., 2021; Rein et al., 2023; Trinh et al., 2024), addressing the degree of
036   complexity required in numerical simulation-intensive science and engineering workflows remains
037   an outstanding challenge. Many quantitative tasks that form the cornerstone of these workflows
038   require numerical analysis performed with sophisticated computational modeling software. For ex-
039
      ample, the development of a modern smartphone requires detailed modeling of the mechanical,
      thermal, and electrical behaviors of its many subcomponents. Finite element analysis (FEA) (eg:
040
      Courant et al. (1994)) software develops approximate solutions to the underlying partial differential
041
      equations for a physical system, by building discretizations (or meshes) over geometries. The result-
042   ing equations are then solved numerically. The vast relevance of FEA to domains like mechanical,
043   biomedical and aerospace engineering, consumer electronics, manufacturing, and scientific research
044   has given rise to software such as Ansys® (Ansys, Inc.), Abaqus® FEA (Dassault Systèmes), and
045   COMSOL Multiphysics® (COMSOL Multiphysics® , b; Multiphysics, 1998), that are indispensable
046   to modeling complex systems with the interplay of non-trivial geometries, and multiple physical
047   phenomena.
048   Despite the potential impact, the application of LLMs to numerical analysis tasks like FEA remains
049   largely unexplored. In this paper, we begin to bridge this gap by asking whether LLMs and LLM-
050   agents can be used to solve problems using finite element analysis (FEA). This task requires the
051   ability to reason over a natural language problem description, plan actions needed to solve the prob-
052   lem and successfully operate FEA software. We selected COMSOL Multiphysics® as the framework
053   for our benchmark because it supports a wide range of physics models and is extensively used for
      commercial engineering analysis as well as for scientific research. However, because the FEA work-

      Our contributions are the following:
075
             • We create a benchmark intended for LLM and agentic research. The benchmark consists of
076
               (1) FEABench Gold: 15 manually verified problems, in addition to (2) FEABench Large:
077            a larger set of 200 algorithmically parsed problems. The problems in FEABench Gold
078            are (a) quantitatively verifiable, that is, if solved completely and correctly, a desired target
079            value will be computed and exported to a table, (b) manually confirmed to have input
080            problem descriptions that are self sufficient and do not omit information necessary to solve
081            the problem (c) manually verified to be solvable, i.e. we confirmed that if the steps to
082            model the problem are followed faithfully in COMSOL Multiphysics® the desired target
083            value is computed. The target values are expected to be largely independent of the modeling
084            software. The objective of the LLM is to read the problem specification and interact with
085
               the software by reasoning over the problem and operating COMSOL Multiphysics® . The
               skills this requires of an LLM include the ability to correctly (a) infer spatial dimensions,
086
               representations of objects as compositions of geometrical primitives, and required physics
087
               features like boundary conditions and their properties, (b) follow instructions that describe
088            the association of these features with geometrical entities and analysis steps and finally, (c)
089            generate the sequence of Java calls to the API that encode these decisions.
090
             • We further define two versions of the tasks in FEABench Gold– ModelSpecs and Plan , to
091            probe different versions of task complexity.
092
             • We introduce a holistic evaluation strategy with intermediate metrics that seek to measure
093            different facets of the ‘distance to the correct solution’. We benchmark different SOTA
094            LLMs on their baseline (single-turn) performance with these metrics.
095
             • Finally, we design an interface in which an LLM can interact with the COMSOL
096            Multiphysics® API and with specialized auxiliary functions and can use execution feed-
097            back to improve its solution over multiple turns. To mitigate the lack of familiarity of
098            LLMs with COMSOL Multiphysics® as a domain-specific language, one of the tools in
099            our agentic environment is a retriever that looks up a corpus of LLM-generated annotations
100            of individual code blocks.

      3   E VALUATION M ETRICS

      Reasoning correctly about the problem and issuing the right calls to operate the API poses a chal-
165   lenging task for even SOTA LLMs, since a model will only be able to compute the correct target
166   value if it was able to generate all the code necessary to set up and solve the model successfully. This
167   makes conventional code evaluation metrics such as the ‘pass@k’ metric (Chen et al., 2021; Kulal
168   et al., 2019) harder to apply to this setting, since most solutions are unable to completely solve the
169   problem. We introduce a multipronged evaluation strategy with metrics that measure the correctness
170   of the solution, even when a target value could not be computed (Table 1). These additional metrics
171   offer the advantage of being continuous, unlike the relative error, which can only be computed if the
172   LLM’s solution computed a ‘valid’ target value. Metrics denoted by † require execution of the API
173
      calls. We delineate the metrics, and the facets they probe here:.
174
             • Executability† : Executable lines as a fraction of parsed API calls in an LLM solution. The
175            COMSOL sandbox returns a ‘reply’ to each line of code. A given line may be invalid if it
176            is syntactically incorrect or if it refers to an invalid action (like modifying a property under
177            a non-existent node).
178          • Model Tree Score† : Similarity score between the LLM solution’s model tree and a GT
179            tree. This is normalized so that a solution with no parsed lines of code is scored 0. If it
180            was equivalent to the GT tree, the score would be 1. This measures the alignment of the
181            model’s solution path with a successful path.
182          • Code Similarity Score: Simple similarity score between the solution and the GT code.
183            We mainly report this metric as a baseline measure of code similarity, and to motivate our
184            introduction of domain-specific metrics. The preponderance of boilerplate syntax, along
185
               with the fact that two different code blocks could generate equivalent model subtrees, are
               factors that contribute to the lack of meaningful variation of this metric across experiments.
186
             • Physics Metrics: The metrics above analyzed the entire solution or its derived artifacts.
187
               The code is a basis to represent the actions the LLM takes to model the problem. Since
188
               the physics block is both the most diverse across problems and the most challenging (Fig-
189            ure 4), we additionally evaluate specifically the LLM’s physics actions. The most basic
190            physics action sequence involves: Create Interface (eg: HeatTransfer) → Create Feature
191            under Interface (eg: TemperatureBoundary) → Modify Feature Properties (eg: T0, to set
192            a temperature). Our Physics Metrics include (a) Interface Factuality: What fraction of
193            interfaces created by the LLM are real COMSOL Multiphysics® interfaces and not halluci-
194            nated? (b) Interface / Feature / Feature Property Recall: How many interfaces / features
195            / feature properties created / modified by the GT solution were also in the LLM solution?
196            (c) Feature Dimension: For features created by both, does the feature’s spatial dimension
197
               match? As an example, if an LLM chose to set a temperature boundary condition on a 1D
               geometry, this metric would check whether it correctly deduced that the boundary condi-
198
               tion should be 0 dimensional (i.e. a point), by comparing the dimension with that of the
199
               boundary condition in the GT solution. While these metrics offer a granular look into the
200            LLM’s physics reasoning path, some nested physics metrics, such as ‘Feature Dimension’
201            will not be valid for a problem when there is no overlap between the GT and the LLM code:
202            we mask out these problems while computing the means for that metric.
203          • Target Relative Error† : At evaluation, we entask an LLM (Gemini-1.5-Pro) to check
204            that the computed value in the exported table matches the target description and that the
205            exported quantity is not a default value, and to parse the response, if so. Valid Target is
206            the number of problems in the benchmark for which the LLM judges the exported table to
207            be valid. We then compute the relative error between the last value in the exported table
208            and the GT answer. Relative Error | Strict computes the mean relative error only over
209
               problems for which Valid Target is True, AND the relative error is less than 10%. Relative
               Error | Strict is the principal metric we use to assess whether the problem was truly solved.

238   Recent work has sought to explore the space of designing optimal Agent-Computer Interfaces (Yang
239
      et al., 2024a; Wang et al., 2024b) primarily for software engineering. However, these frameworks

240
      software development, but of limited relevance to FEA / numerical analysis workflows. Our single-
241
      turn results, particularly on executability and hallucinated interfaces, highlight the need to ground
242   the LLM’s responses with feedback from and documentation about the API. Given a single try, the

244   nature and location of the errors will allow it to correct those errors in a directed fashion. We thus
245   build an environment in which the LLM can interact with the API, receive feedback and attempt to
246   correct its solution. Our design includes the features below, aimed at addressing specific challenges
247   posed by FEA problems:

372   intermediate states. During evaluation, the best solutions are read in and evaluated. If there are
373   multiple best solutions (in cases where multiple solutions were able to compute a target value), the
374   top best solution is the one that maximizes the following formula: Executability + bool(Computed
375   Value) + [(1.0 - Target Relative Error) if (Target Relative Error<1) AND (Valid Target) else 0].
376   The three conditions together prioritize solutions that (1) had high executability, (2) were complete
377   enough to export any value, albeit incorrect or the wrong quantity and, (3) exported a ‘Valid Target’
      within 100% of the desired value.

409   6   R ESULTS
410
411   Comparison across LLMs: Although JAVA API commands to COMSOL Multiphysics® are
412   somewhat ‘out of distribution’ since they are unlikely to account for a significant fraction of code
413   in the LLM training data, we find that all models are able to generate code with moderately high
414   executability in the range 0.60-0.79, implying that LLMs appear to know the higher-level grammar
415   and syntax of COMSOL Multiphysics® API calls or are able to infer it from the one-shot example.
416   Getting more granular choices correct proves to be more challenging: LLMs are prone to halluci-
417   nating the interface choice (factuality between [0.54-0.85]). This is likely a significant contributor
418   to the non-executable lines because an invalid interface declaration will render all physics lines of
419
      code acting under this interface invalid. We also compare the performance of the three LLMs on 200
      problems in FEABench Large. Note, unlike the problems in the human-verified FEABench Gold,
420
      these problems do not have a single final target artifact, so we only evaluate these against metrics that
421
      don’t require execution. Claude 3.5-Sonnet consistently has the best performance on most metrics
422   on both benchmarks. The open-weights LLMs generally perform worse than closed-source LLMs,
423   especially on the alignment-probing metrics such as the Model Tree Score and Physics Recall met-
424   rics. The feature recall is so low for these problems, that the feature dimension metric can only be
425   evaluated for fewer than 5 problems in each of the experiments involving models in the Gemma
426   family.

466   The interactive Multi-Turn Agent has the highest performance of all experiments on the ModelSpecs
467   task across several metrics including executability (0.62 →0.88). Figure 4 analyzes the executability

472

507   7     D ISCUSSION
508
509
      Our benchmark seeks to inspire rigorous evaluations of the capabilities of LLMs on solving prob-
510
      lems that require simulating physical phenomena in the real world and performing numerical anal-
511   ysis. Such problems are ubiquitous in science and engineering, and solving them requires synthe-
512   sizing reasoning over the physics domain with the ability to leverage numerical analysis software
513   such as FEA. Although the problems in our benchmark are already challenging for SOTA LLMs,
514   an extension could be to use imported Computer-Aided Design (CAD)-built geometries to be more
515   aligned with industrial workflows. While datasets such as FEABench Large provide a useful sta-
516   tistical signal on the quality of code solutions generated across a large number of problems, adding
517   more human verified problems would be valuable.
518   Our multiphysics agentic interface devised the basic elements to facilitate LLMs to interact with the
519   API in a targeted fashion and we further designed one realization of an agentic framework that incor-
520   porates these elements. It would be valuable to port blocks such as the Evaluator and the specialized
521   functions into generalist agentic frameworks like AutoGPT and LangChain (Significant Gravitas;
522   Chase, 2022) to explore possible performance gains and understand the optimum way to distil vi-
523   sual information from the GUI. Using an LLM-annotated corpus to boost code executability might
524   facilitate code generation in other low-resource domain-specific language contexts. Conversely,
525
      code generation approaches for other low-resource languages (Cassano et al., 2024) might reduce
      the bottleneck of translating predefined decisions into code (the Plan task). We examined the perfor-
526
      mance of a fine-tuned model relative to a baseline in Appendix G: while the fine-tuned checkpoint
527
      can outperform the untuned checkpoint in a Zero-Shot setting, the untuned checkpoint prompted
528   with a One-Shot example outperforms both. Research (Ding et al., 2024) on increasing the effective
529   context lengths in fine-tuning will likely benefit our setting. Other work (Dziri et al., 2024) has
530   identified the challenge of getting transformers to reason over complex compositional tasks and it
531   would be interesting to explore whether alternative approaches could mitigate this.
532
      Our dataset serves as a novel testbed to evaluate the ability of LLMs and agentic approaches to
533
      interact with feedback from an execution environment, error-correct and learn how to master a rel-
534   atively unfamiliar software well enough to solve problems. The ability to quantitatively analyze a
535   problem and operate scientific software would augment LLMs’ reasoning skills with the numerical
536   precision and inbuilt checks offered by FEA software, and significantly push the ceiling on problems
537   that LLMs can currently accurately solve. Unlocking this ability would bring LLMs a step closer to
538   being able to serve as grounded ‘engineering assistants’ that can autonomously run precise simula-
539   tions to innovate and optimize designs and answer quantitative questions about physical phenomena
      in the real world.

721   LLMs and Agents for Code Several studies have focused on benchmarking coding in general-
722   purpose programming languages, with a particular focus on software engineering tasks (Austin et al.,
723
      2021; Chen et al., 2021; Jimenez et al., 2023; Li et al., 2022), and less commonly, science problems
      (Tian et al., 2024). FEA software emerged because simulating and numerically solving real-world
724
      problems from scratch in mainstream languages would require significantly more effort without

765   B.1.2    G ENERATION P ROCEDURE :
766
767
      Without any modification, the tutorials might export a single value, a table, or not export any target
      quantity at all, with the final output being qualitative in nature, such as in the form of plots or figures.
768
      For our benchmark, however, we specifically wanted every problem to have a numerically verifiable
769
      target value, in order for there to be an absolute notion of correctness (i.e. if the code was fully
770   correct, and aligned with the intent of the problem, it should be able to export this value). This also
771   enables easier evaluation of the problems. The following procedure and guidelines were adopted to
772   curate the benchmark:
773
774           • For an initial set of 2-3 problems, model specifications and plans were annotated by hand,
775             by an expert user of COMSOL Multiphysics® .
776           • For subsequent problems, we speed up the benchmark generation procedure by following
777             an initial LLM-assisted data generation process, with the final verification steps involving
778             humans. An LLM is provided with a tutorial, as well as a two-shot prompt with the expert
779
                annotated model specifications.
780
              • The LLM is entasked with returning a model specification for the tutorial that has the same
                format. This requires the LLM to identify an appropriate target value from the tutorial
781
                which it does from either the text or the figures, and returning a model specification for
782
                computing this target value.
783
              • The LLM is then asked to create a plan corresponding to the model specifications, using a
784
                two-shot prompt with two plans. The utility of the tutorials are that the plan is closest to
785             the GUI instructions listed in the tutorial, while model specifications is more concise.
786
              • A ground truth code that can compute the correct value is then generated for the problem.
787             We manually verify that the ground code when run, exports the desired target value. This
788             step also involves simultaneously ensuring that all information required to build the model
789             is contained in the plan, and in the model specifications by editing the LLM-generated
790             drafts and ensuring that no Translation Errors are encountered when parsing and executing
791             the ground truth code in COMSOL Multiphysics® using the bridge described in Appendix
792             D.1 or that any errors if encountered are in non-crucial lines and do not prevent the solu-
793             tion from being computed. Any missing or incorrect information is fixed, and the selec-
794             tion information field, that contains numerical identities of boundaries and points is also
795
                created.
796           • We add an instruction to export the output to OUTPUT PATH/output.txt in the model
797
                specifications and plan.

1178   D.1     T HE P YTHON -COMSOL M ULTIPHYSICS® B RIDGE
1179
       The raw output of the LLM is a string containing COMSOL Multiphysics® API commands in Java.
1180
       An interface between Python and COMSOL Multiphysics® is needed to execute this code and inter-
1181
       act in other ways with the API. We use the Python package MPh (mph) and Rpyc for this. MPh
1182   is a scripting interface built on JPype (jp) that enables a Python program to communicate with and
1183   build a model in COMSOL Multiphysics® . Each Java API command in the LM’s output can be
1184   ‘pythonized’ algorithmically. In most cases, the pythonized line is near identical to the Java line.
1185   However, due to differences in Java and Python syntax there exist some corner cases that need to
1186   be handled separately. Eg: ‘new String[]’ is exclusively a Java construction, while the notation
1187   for booleans in Python is True / False as opposed to true / false in Java. Thus a ‘pythonizer’ is
       constructed that parses and translates Java API calls to their Python counterparts.


                                                        22
       Under review as a conference paper at ICLR 2025


1188
       The setup involves the following assumptions: an MPh client object is created. This behaves like a
1189   stateful ‘sandbox’, where models can be built by LLMs, code can be evaluated, or information such
1190   as the current state of the model tree, properties under a node and the exported table can be queried
1191   and retrieved. Although multiple models can be created and set under the client, for simplicity we
1192   work with settings that involve a single model. Before running a new solution, the existing model is
1193   deleted and a new blank model is created. The LLM actions will modify this blank model. Thus, by
1194   design, all lines of code the LLM outputs, should start with ‘model.’ and end with ‘;’.

1197
1198         1. Geometry, if any: This involves identifying the dimensionality of the problem, and con-
1199            structing a representation of the object being modelled, say a cup, by creating and com-
1200            posing primitive shapes such as ellipses or rectangles to build the object. While already
1201            constructed geometries can also be imported from other software such as CAD, in our
1202            benchmark, we currently restrict ourselves to models for which we construct the geometry
1203            from scratch in COMSOL. This typically starts with a ‘model.component(“comp1”).geom’
1204            pattern.
1205         2. Physics: This will include specifying all the physical conditions for the problem, including
1206            initial or boundary conditions, forces, properties or in the case of mathematics problems,
1207            the differential equation. This typically starts with a ‘model.component(“comp1”).physics’
1208            pattern. Some problems may additionally have lines that begin with , and set up the cou-
1209            pling between different kinds of physical phenomena. We categorize these lines, if any as
1210            ‘physics’ in Figure 4 and 6.
1211
             3. Material: Creating materials and assigning them to domains. One can either assign known
1212            materials such as ‘Copper’ and the object will inherit the default properties of that mate-
1213            rial, or define a blank material and its properties such as conductivity from scratch. This
1214            typically starts with a ‘model.component(“comp1”).material’ pattern.
1215
1216         4. Mesh: Usually a shorter step that involves meshing the surfaces of the geometry to set up
1217
                elements. This typically starts with a ‘model.component(“comp1”).mesh’ pattern.
1218         5. Study / Solver: This involves specifying the conditions of the analysis and solver, such
1219            as the number of timesteps. While the solver code can be modified to override defaults,
1220            COMSOL also has the ability to automatically populate the model with the default solvers
1221            most apt for a given problem. This typically starts with a ‘model.study’ or ‘model.sol’
1222            pattern respectively. In Figure 4 and 6, we categorize both patterns as ‘solver’.
1223
             6. Results: Once the numerical solver has completed the analysis, one will likely postpro-
1224            cess the problem, in order to generate desired plots or tables. This typically starts with a
1225            ‘model.result’ pattern.

       The unfamiliarity of LLMs with permissible options and arguments to the COMSOL
1397
       Multiphysics® calls is a significant factor contributing to the difficulty of the benchmark. This raises
1398   the prospect of exploring whether fine-tuning can boost the performance of LLMs on generating
1399   code. We used the Google AI Studio platform (Google LLC) to tune the ‘gemini-1.5-flash-001-
1400   tuning’ checkpoint on 180 problems in FEABench Large for 5 epochs. This platform imposed a
1401   limit of 4 × 104 characters on the inputs and 5000 characters on the outputs. All but two of the
1402   180 FEABench Large code outputs exceed this limit. We used a shorter, Zero-Shot prompt (with-
1403   out the One-Shot example) and truncated the dataset’s inputs and outputs to adhere the limit during
       fine-tuning.

       Under review as a conference paper at ICLR 2025


1404
       At inference time, we examine three LLMs x Prompting scenarios in terms of their performance on
1405   the FEABench Gold problems and on the task ModelSpecs , namely (1) Baseline | Zero-Shot: the
1406   untuned checkpoint (‘gemini-1.5-flash-001-tuning’) paired with a Zero-Shot prompt similar to that
1407   used during training (2) FT | Zero-Shot: the Fine-Tuned model paired with the same prompt and,
1408   (3) Baseline | One-Shot: the untuned checkpoint paired with the One-Shot prompt used in other
1409   experiments in this paper.
1410
       With the Zero-Shot prompt, the untuned LLM performs abysmally on several metrics including
1411   Executability. This is unsurprising, since the LLM sees no template for how its code should be
1412   structured. In this setting, the Fine-Tuned LLM seems to offer advantages, in terms of enabling
1413   the LLM to generate more executable code (Executability: 0.08 → 0.50). However, the untuned
1414   LLM prompted with the One-Shot example outperforms the fine-tuned LLM across most metrics,
1415   especially evident in the stark difference in the Physics Recall Metrics and the Model Tree Score.
1416
       The failure of fine-tuning in yielding significant gains can be attributed to several factors in this
1417   experiment. First, the fine-tuned checkpoint overfits to the training distribution. Even when the code
1418   is reasonably ‘executable’ (0.50), it is likely misaligned with what the prompt actually requires the
1419   LLM to do – observe the Model Tree Score is 0.24 (FT) vs 0.49 (Baseline | One-Shot). This was
1420   also qualitatively noticeable since the outputs during inference were also truncated midway, similar
1421   to the truncated outputs in the training distribution.
1422
       Using the same checkpoint’s tokenizer, the median number of tokens in the input zero-shot prompt
1423   and the output code, (before truncation) is 4036 and 7122 tokens respectively, across the 180 prob-
1424   lems. The limits imposed during fine-tuning exacerbate the performance of the fine-tuned LLM.
1425   Since the linewise mapping of the inputs (natural language modeling instructions) to code is not
1426   one-to-one, the truncation only allows the LLM to see the first chunk of the correct answer. Lastly,
1427   the training distribution is not identical to the test-time distribution: the FEABench Large inputs
1428   use API-specific explicit instructions from the tutorials. The problem descriptions corresponding to
