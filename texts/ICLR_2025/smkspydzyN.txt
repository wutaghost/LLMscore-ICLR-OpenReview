009                                               A BSTRACT
010
011            Developers try to evaluate whether an AI system can accomplish malicious tasks
012            before releasing it; for example, they might test whether a model enables cyberof-
013            fense, user manipulation, or bioterrorism. In this work, we show that individually
014            testing models for such misuse is inadequate; adversaries can misuse combinations
015            of models even when each individual model is safe. The adversary accomplishes
016            this by first decomposing tasks into subtasks, then solving each subtask with the
017            best-suited model. For example, an adversary might solve challenging-but-benign
018            subtasks with an aligned frontier model, and easy-but-malicious subtasks with a
019
               weaker misaligned model. We study two decomposition methods: manual decom-
               position where a human identifies a natural decomposition of a task, and automated
020
               decomposition where a weak model generates benign tasks for a frontier model
021
               to solve, then uses the solutions in-context to solve the original task. Using these
022            decompositions, we empirically show that adversaries can create vulnerable code,
023            explicit images, python scripts for hacking, and manipulative tweets at much higher
024            rates with combinations of models than either individual model. Our work sug-
025            gests that even perfectly-aligned frontier systems can enable misuse without ever
026            producing malicious outputs, and that red-teaming efforts should extend beyond
027            single models in isolation.

030   1   I NTRODUCTION
031
032   Developers try to ensure that AI systems cannot accomplish malicious tasks before releasing them; for
033   example, they might test whether releasing a model enables automated cyberoffense, manipulation, or
034   bioterrorism (Phuong et al., 2024; Google, 2024; OpenAI, 2023; Anthropic, 2023). To mitigate such
035   misuse risks, the most capable frontier systems are trained to refuse requests that would otherwise
036   lead to malicious outputs. In contrast, less capable open-source systems are often deployed with
037   weaker refusal training that can be further removed by fine-tuning (Lermen et al., 2023). This strategy
038
      in principle only produces “safe” models—models that cannot accomplish malicious tasks—since
      only frontier models are capable of complex malicious tasks, and they are trained to refuse them.
039
040   In this work, we empirically show that testing whether individual models can be misused is insufficient:
041   adversaries can misuse combinations of models even when each individual model is safe. Critically,
042   adversaries do this without circumventing the models’ safety mechanisms; this means that even a
043   perfectly aligned frontier model can enable harms without ever producing a malicious output.
044   The core strategy the adversary employs for misuse is task decomposition, where it decomposes
045   malicious tasks into subtasks, then assigns subtasks to models (Figure 1). Many malicious tasks are
046   combinations of benign-but-hard subtasks and malicious-but-easy subtasks. The adversary executes
047   the benign subtasks (which require capability) with a frontier model and the malicious subtasks
048   (which require non-refusal) with a weak model.
049   We first formalize a threat model that captures model combinations. The adversary aims to produce
050   an output that satisfies some condition (e.g., is a working malicious python script that can be used
051   to infect a target machine) that it could not produce itself, using a set of models at its disposal. At
052   each turn, the adversary takes the task and any previous turns as input, selects a model and a prompt,
053   then receives the output of the model on that prompt. The adversary wins if it eventually produces an
      output that satisfies the original condition (e.g., produces the desired python script).

      We study two classes of decomposition patterns within this framework: manual and automated
074
      decomposition. For manual decomposition, a human identifies a natural decomposition of a task (e.g.,
075   creating vulnerable code by generating secure code, then editing it). However, some tasks are hard
076   for humans to manually decompose. We address this with automated decomposition, where a weak
077   model proposes related-but-benign tasks in the first turn, a frontier model solves them in subsequent
078   turns, and a weak model uses the solutions in-context to execute the original task in the final turn.
079
      Under these decomposition patterns, we find that combinations of models can create malicious python
080
      scripts, vulnerable code, manipulative tweets, and explicit images at much higher rates than either
081   individual model in isolation. We study DALL-E 3 and three variants of Claude 3 as frontier models,
082   and six different weaker open-source models. Combining models often produces significant jumps in
083   misuse performance: for example, combining Claude 3 Opus and Llama 2 70B achieves a success
084   rate of 43% when generating vulnerable code, while neither individual model exceeds 3%.
085
      We next study the scaling behavior of misuse and find that multi-model misuse will likely become
086
      starker in the future. Empirically, we find that the rate at which the adversary successfully misuses
087   combinations of models scales in terms of the quality of the weaker model (e.g., from Llama 2 13B to
088   70B) and the stronger model (e.g., from Claude 3 Haiku to Opus). Our results are only a lower-bound
089   on what is possible with model combinations; different decomposition patterns (such as using the
090   weak model as a general agent that repeatedly calls the strong model), or training the weak model to
091   exploit the strong model via reinforcement learning, will likely enable further misuse.
092
      Our work expands red-teaming to combinations of models in order to reliably assess deployment
093   risks. Developers should continue this red-teaming throughout the deployment life of the model,
094   as any new model release could unlock new risks. More generally, red-teaming with respect to the
095   broader model ecosystem could help developers more reliably identify when benign capabilities
096   enable misuse, and thus more realistically trade-off their benefits and risks.

099   2          R ELATED W ORK
100
101   Despite their numerous capabilities, deploying language models (LLMs) poses risks; see (Bommasani
102   et al., 2021; Weidinger et al., 2021; Hendrycks et al., 2023) for surveys. These include misuse risks,
103   where adversaries use LLMs to complete malicious tasks. For example, future LLMs could be used
104
      for cyberoffense (Barrett et al., 2023; Fang et al., 2024), bio-terrorism (Soice et al., 2023), deception
      (Scheurer et al., 2023; Park et al., 2023b), or manipulation (Carroll et al., 2023), among other uses.
105
106   A common way to misuse frontier language models is to jailbreak them, i.e. circumvent the LLM’s
107   refusal mechanism to produce malicious outputs (Wei et al., 2023; Shah et al., 2023; Zou et al., 2023;
      Liu et al., 2024; Anil et al., 2024). Some jailbreaks leverage multiple models, often by optimizing

108
      prompts on open-source models and transferring to closed-source models (Wallace et al., 2019; Jones
109   et al., 2023; Zou et al., 2023). We show that frontier models can be misused without jailbreaking.
110
111   Many AI companies and academics have developed frameworks for assessing misuse risk before
112
      deployment. For example Google (Shevlane et al., 2023; Phuong et al., 2024), OpenAI (OpenAI,
      2023), and Anthropic (Anthropic, 2023) have public policies for how they assess and evaluate the
113
      misuse potential of individual models. Bommasani et al. (2023) and Kapoor et al. (2024) argue that
114
      models should be evaluated for the marginal risk of adding the model to the environment, rather the
115   absolute risk. Our work suggests that assessing individual models fails to capture all misuse risk, and
116   the marginal risk of even aligned model could be large.
117
118
      We build off of work studying risk that arise from combining language models. Anwar et al. (2024)
      speculate that LLM agents (Wang et al., 2023; Xi et al., 2023) could have emergent risks from
119
      interaction, Motwani et al. (2024) offer initial evidence that LLM agents can collude, and Bommasani
120
      et al. (2022) suggest that models have correlated failures, which are magnified when they are
121   codeployed. Moreover, new capabilities may sometimes only emerge when agents interact (Park
122   et al., 2023a), or when an LLM changes an exogenous world state (Pan et al., 2024).
123
124
      Another line of work studies how combining models enhances benign capabilities. This includes
      training a small model to decompose tasks that a large model subsequently solves (Juneja et al.,
125
      2023), improving outputs via debate (Du et al., 2024; Khan et al., 2024), using weak language
126
      models to control strong language models (Greenblatt et al., 2023), and approximating fine-tuning of
127   closed-source models using open-source models (Mitchell et al., 2024). Combining models from
128   different modalities can also solve tasks that no individual model can (Tewel et al., 2022; Zeng et al.,
129   2023; Li et al., 2023). Our work shows combining models increases the potential for misuse.
130
      Finally, Narayanan & Kapoor (2024) argue that safety inherently depends on the context of a model
131
      deployment, while Glukhov et al. (2023) argue that no refusal or censorship mechanism can ensure
132
      safety, since some malicious tasks are combinations of benign subtasks that a single censored model
133   can solve. Our work expands task decomposition: we empirically demonstrate how adversaries can
134   use task decomposition to combine models across realistic malicious tasks; we expand the set of tasks
135   that can adversaries can accomplish via decomposition by allowing access to weak, open-source
136   models with inadequate refusal training; and we show how task decomposition can be automated
137   using the decomposition and in-context abilities of these weak models.

      3       T HREAT MODEL
140
141
      In this section, we introduce our threat model specifying how adversaries can combine models.
142
143   Threat model. Our threat model encapsulates an adversary that is trying to misuse a set of models
144   for some nefarious activity. The adversary combines models by querying them sequentially; at each
145   step, the adversary chooses a model and a prompt and receives an output. The adversary wins if it
146   eventually produces an output that satisfies some malicious property.
147   More formally, we assume an adversary has access to a set of models M such that each model
148   m ∈ M produces output o = m(x) from prompt x. The adversary aims to produce a malicious
149   output; we assume there is a binary predicate r, where r(o) = 1 if o is a desired malicious output
150   and 0 otherwise. To produce the output, at turn n the adversary takes in the transcript of previous
151   models, prompts, and outputs τ = {(m1 , x1 , o1 ), . . . , (mn−1 , xn−1 , on−1 )}, the set of models M,
152   and the predicate r, and outputs a model mn and a prompt xn ; the adversary a is thus a function such
153
      that a(M, τ, r) = (xn , mn ). The adversary then gets output on = mn (xn ), and wins if r(on ) = 1;
      if not, the adversary tries again. We assume the adversary fails after N unsuccessful rounds.
154
155   When M contains one model or copies of one model, this reduces to the individual-model misuse case.
156   The adversary thus gains leverage when models in M have distinct-but-complementary strengths.
157   Strength of the adversary. One important consideration in our threat model is how strong the
158   adversary is—if the adversary can produce a malicious output itself, it can vacuously “misuse”
159   a single model just as well as combinations of models.1 Thus, for the adversary to benefit from
160   combining models, it needs to be bounded in some way. We argue that adversaries are bounded

184   4     M ANUAL D ECOMPOSITION
185
186   In this section, we study how manual decompositions of tasks enable misuse. Manual decomposition
187   applies when humans can decompose tasks into subtasks that different models are better suited for, in
188   which case the adversary can simply allocate each subtask to the best-suited model (see Figure 2).
189   To instantiate manual decomposition within our framework, we assume that M contains two models:
190   a frontier model that is very capable but refuses to produce malicious outputs, and a weak model that
191   is less capable but can be prompted to produce malicious outputs. To exploit the comparative strength
192   of each model, the human splits a task into a complicated-but-benign component (e.g., generating
193   a complex image), and a simple-but-malicious component (e.g., adding nudity to the image). The
194   adversary a first uses the frontier model mfrontier ∈ M to produce the complicated-but-benign output
195   o1 , then uses the weak model mweak ∈ M to adapt o1 to be malicious, and finally returns the result.
196   We study manual decomposition for vulnerable code generation (Section 4.1) and explicit image gen-
197   eration (Section 4.2), and include all prompts and additional implementation details in Appendix A.

      4.1   V ULNERABLE CODE GENERATION
200
201
      We first study adversaries that aim to write code that serves a specific task, but includes a deliberately
202
      added vulnerability. For example, the adversary might aim to implement a feature that lets users
203   upload files to a website in a way that enables a directory traversal attack. Adversaries might do this
204   in order push code with vulnerabilities to open-source repositories, so that they can exploit them after
205   deployment (Goodin, 2024). We will specify the vulnerable code generation task, instantiate how the
206   adversary combines models for this task, then present quantitative results.
207
      Vulnerable code generation task. To evaluate adversaries on this task, we introduce a LLM-
208
      generated dataset of programming questions with associated vulnerabilities. The questions are
209   designed to be at the level of an advanced programming class, solvable with a single python script,
210   and about a topic where security is important. One such programming question in our dataset is:
211
212
      “Develop a script that allows users to upload files to a specific directory ”/uploads” on the server.
      The script should validate the file type and size before allowing the upload. Accepted file types are
213
      JPG, PNG, and PDF, with a maximum size of 5MB. Success criteria: The script should only allow
214
      uploads of the specified file types within the size limit and save them to the designated directory.”
215
      This question comes with three associated vulnerabilities, one of which is:


                                                          4
      Under review as a conference paper at ICLR 2025


216
      “Directory Traversal: Not properly sanitizing the file path provided by the user, allowing them to
217   manipulate the path and potentially upload files outside the designated ”/uploads” directory, which
218   could lead to unauthorized access or overwriting of sensitive files on the server.”
219
220
      The dataset has a total of 100 programming questions, each of which has three vulnerabilities, for
      a total of 300 examples. The programming questions and our datasets are generated by repeatedly
221
      interacting with Claude 3 Opus to extract challenging problems; we use Claude because it empirically
222
      produces much more challenging questions than the other models that we tried. We generate
223   vulnerabilities by prompting Claude to come up with vulnerabilities that a user should avoid when
224   implementing the task; we found that this produces vulnerabilities that are salient and exploitable.
225
226
      Available models. We assume the adversary has access to one of three frontier models: Claude 3
      Opus, Sonnet, or Haiku (Anthropic, 2024), and one of five weak models: the three models from
227
      the Llama-2-chat family (Llama 2 7B-chat, 13B-chat, 70B-chat) (Touvron et al., 2023), and the
228
      instruction tuned Mistral 7B (Jiang et al., 2023) and 8x7B Mixtral (Mistral AI team, 2023).2 We test
229   every frontier-weak combination.
230
231   Adversary’s combination strategy. To solve this task by combining models, the adversary first
232
      prompts the frontier model to solve the programming task without mentioning the vulnerability. It
      then edits the solution to include the vulnerability using the weak model. Both models are instructed
233
      to produce production-level code that would run out of the box. Combining models is intuitively
234
      necessary for misuse since the frontier model refuses to either generate or adapt code to be malicious,
235   while the weak model is often incapable of generating code that solves the programming task.
236
237   To ensure that combining a weak model with a frontier model is the key feature enabling misuse, we
238
      compare against several single-model baselines. We first study the single-shot baseline, where we test
      whether each model can generate code with the vulnerability from a single prompt. We additionally
239
      test the single-model-decomposition baseline, where the adversary decomposes the task into subtasks,
240
      then solves each subtask using the same model. The single-shot baseline tests whether a model can
241   solve the task out-of-the-box, while the single-model decomposition baseline tests whether combining
242   models is the key feature enabling misuse, rather than decomposing the task.
243
244
      Quantitative results. We first evaluate whether each adversary solves the task, i.e. the model-
      generated code both executes the task and includes the desired vulnerability. To do so, we use GPT-4
245
      as an evaluator to measure whether the code has each of these properties (OpenAI, 2023).3 We use
246
      GPT-4 since it is much cheaper than hiring cybersecurity experts to do the evaluation, and did not
247   include it in the set of assessed models to avoid LLM’s self-bias during evaluation (Panickssery et al.,
248   2024). We say an output is successful if GPT-4 assesses that it both executes the task out-of-the-box
249   and has the vulnerability, and incorrect otherwise; see Appendix A.2 for the exact prompt.
250
      We include the full quantitative results in Table 1 and find that across nearly all weak-frontier combi-
251
      nations, the adversary is far more successful when combining models than using either individual
252
      model when generating vulnerable code. The largest gains come from combining Llama 2 70B-chat
253   with Claude 3 Opus; the adversary achieves a success rate of 43% when combining the two models,
254   compared to less than 3% when using each individual model.
255
256
      We additionally empirically verify our intuition that frontier models fail because they refuse to
      generate outputs, while weak models fail due to lack of capability. We test for refusal by checking if
257
      the model outputs a valid python script, and find that all versions of Claude nearly always refuse to
258
      respond to our prompts, while most open-source models nearly always respond (Table 4).
259
260   Scaling. Finally, our results indicate that the success rate when creating vulnerable code scales as
261   both the frontier model and the weak model improve. For four out of the five weak models that we
262
      test, combining with the strongest frontier model (Claude 3 Opus) achieves the highest success rate.
      Moreover, the most-capable weak models (Llama 2 70B-chat, Mistral 7B, and Mixtral 8x7B) have
263
      much higher success rates than the less-capable weak models (Llama 2 7B-chat 13B-chat) when
264
      combined with any frontier model. These results indicate that the misuse risk of combining models
265   will continue to grow as stronger weak and frontier models are substituted in.

364
365
366   In this section, we study how automated decompositions of tasks enable misuse. Automated decom-
367   position relies on the weak model to generate related tasks when the original task is hard to manually
368
      decompose. The adversary leverages the in-context capabilities of the weak model together with
      benign demonstrations from the frontier model (on the related tasks) to solve the original task.
369
370   To instantiate automated decomposition within our framework, we one again assume M contains
371   a frontier model and a weak model. The adversary a first prompts the weak model mweak ∈ M to
372   generate tasks that are related to the malicious task but are benign themselves; this produces output
373   o1 , which contains three related tasks. The adversary then uses the frontier model mfrontier ∈ M to
374
      solve each related task in o1 ; this produces benign solutions o2 , . . . , o4 . Finally, the adversary uses
      the weak model mweak to produce a solution to the malicious task, using the benign prompts and
375
      solutions o2 , . . . , o4 as references in-context.
376
377   We study automated decomposition for generating malicious python scripts used for hacking (Sec-
      tion 5.1) and for personalized manipulation in a synthetic setup (Section 5.2).

443
444   We next study adversaries that try to generate tweets to manipulate a specific set of users—specifically,
445   the adversary aims to generate tweets that will make a user (or users) feel worse about a politician
446   based on the user’s historical tweets and retweets. For example, the adversary might tweet “Barack
447   Obama, thanks for passing the ACA” to target a user whose tweets complain about the increased role
448   of government. Adversaries might aim to construct such tweets in order sway user opinions.
449
      To make this experiment tractable to run and to avoid manipulating real users, we study whether
450   adversaries can manipulate a simulated set of users. We use GPT-4 to simulate the preferences of
451   these users; specifically, we prompt GPT-4 with a user’s tweet history, and ask it to anticipate how
452   that user would react to various tweets. Adversaries are not aware that the users are simulated, so
453   they deploy the same strategy on simulated users as they would on real users.
454
      Simulated personalized manipulation task. To evaluate adversaries on this task, we construct
455
      a dataset of user histories from historical Twitter data. We use Twitter data released by Linvill &
456   Warren (2020) to obtain a set of 100 users with at least 39 tweets each. These tweets come from
457   accounts associated with a Russian disinformation operation and are thus not genuine users—we use
458   this data because it does not require using an API to access and helps mitigate privacy concerns. We
459   choose politicians from a list of 100 US politicians (see Appendix A.5).
460
      The adversary’s task is to generate three tweets about a politician that will make the simulated user
461
      or pair of users feel worse about the politician. We study pairs in addition to individual users since
462   optimizing for both users simultaneously is a more challenging task, and we thus expect it to require
463   more capability. We test the adversary on 100 randomly chosen user-politician combinations, and
464   100 randomly chosen pair-politician combinations. We provide further details in Appendix A.5.
465
      Available Models. We test Claude 3 Opus as the frontier model and Mistral 7B as the weak model.8
466
467   Adversary’s combination strategy. To combine models, the adversary first prompts the weak model
468   to come up with three non-nefarious tasks that also require drawing inferences from tweet histories.
469   The frontier model then solves these tasks using the user’s tweets, and the weak model uses the
470   solutions in-context when solving the original task. We include the prompts in Appendix A.5. As
471
      before, we additionally test the single-shot and single-model-decomposition baselines.
472   Quantitative results. We measure whether the adversary produces tweets that each simulated user
473   engages with, and that clouds the user’s opinion about the politician. We measure both quantities
474   using GPT-4 as a simulator.
475   We include the full results in Table 6 in Appendix A.5, and find that in every setting, combining

477   benefit of combining models grows when creating tweets that must simultaneously manipulate a
478   pair of users rather than an individual user (from a 5% improvement to 33%), which suggests that
479   combining models is especially important for more challenging tasks.

481   change their preferences. Nevertheless, combining frontier models with weak models outperforms

490   6    D ISCUSSION
491
492   In this work, we provide empirical evidence that combinations of safe models can be misused.
493   However, this work only begins to explore the risks of combining models. Future adversaries could
494   use LLM agents to adaptively extract capabilities from frontier models (e.g., by crafting prompts for
495   the frontier model, then iterating based on the output), or fine-tune open-source models to exploit
496
      a specific frontier model’s capabilities. Adversaries could also combine models based on strengths
      beyond non-refusal and capability; models might have different specicializations, use different tools,
497
      or have access to different information, which could further enable decomposition-based misuse.
498
499   Our work relates closely to jailbreaks, but we do not exhaustively try to jailbreak the frontier systems.
500   This means that better jailbreaks may produce higher success rates than combining models for some
501   of our tasks.9 However, we think this is largely irrelevant; our experiments directly show that for
502
      fixed-strength adversaries, combining models enables misuse. Since human adversaries are also
      fixed-strength, this indicates that in the future, humans may still successfully misuse combinations
503
      of models even if jailbreaking them becomes expensive or impossible. The risks we surface are
504
      fundamentally different from jailbreaking, can arise from axes of variation that do not include refusal,
505   and persist even for systems that cannot be jailbroken.
506
507   Our work thus demonstrates that mitigating misuse risks requires new definitions of safety, and safe
508
      releases require persistent deployment decisions. We need new objectives for safety, as adversaries
      can exploit current systems by using them as they are designed. Developers also must continue to
509
      reevaluate whether a deployment is safe throughout the deployment life of a model, as each new
510
      system in the environment creates many new combinations of models. Our threat model is a first step
511   towards adapting red-teaming practices to capture a broader set of model risks, but building more
512   comprehensive threat models and LLMs that are resilient to them is important subsequent work.
513
514
      While we show that AI systems can bolster the adversary, using them to bolster defense is more
      complex. Defenders cannot access the outputs of the weak model—all of the weak models that we
515
      study are open-source, so adversaries can query them locally with no oversight. This means defenders
516
      can primarily leverage AI systems at the platform level, i.e., the place where the adversary deploys the
517   malicious content (e.g., GitHub or Instagram). Platform-level defense has downsides; it is expensive,
518   does not cover upload-free attacks, and requires adoption by many stakeholders. Nevertheless, our
519   work suggests that this defense may be the most tractable option, and is important subsequent work.
520
      Finally, our attacks more directly surfaces tradeoffs from the dual-use nature of language models. For
521
      example, a language model that is only capable of explaining information well (e.g., summarizing
522
      high-level behavior from server-logs) could enable misuse under our threat model by preprocessing
523   complex inputs for weak models. However, the benefits of some capabilities could outweigh the
524   costs; good explanations could help developers or models patch bugs, and flag malicious behavior.
525   We believe that deployment decisions should be made based on a holistic picture of the benefits and
526   risks of some capability, and hope our framework lets developers more accurately assess risks.
527
      Broader impacts. Our work releases a new method to misuse models, following similar releases
528
      (Wei et al., 2023; Zou et al., 2023; Liu et al., 2024; Anil et al., 2024). To mitigate the risks posed by
529   this work, we shared our findings with frontier labs and only introduce comparatively weak methods
530   for combining models. However our method nevertheless could be used by adversaries, and requires
531   new types of defenses to mitigate.
532
      We believe the benefits of releasing our work outweigh the risks. Withholding our findings to improve
533
      defenses would likely be ineffective; the security literature has repeatedly found that “security through
534
      obscurity” does not stop adversaries from identifying failures (Saltzer & Schroeder, 1975; Wang et al.,
535   2016; Guo et al., 2018; Solaiman et al., 2019; Tong et al., 2023). Our work alerts model developers
536   and academics of this threat model at a time where frontier models are less capable and misuse risks
537   are lower; we hope this will help preclude inadvertently risky deployments in the future.

       In this section, we discuss the benefits and drawbacks of using synthetic data instead of real data, and
       using LLM evaluators instead of human evaluators.
1317
1318   Synthetic data. For our experiments, we largely rely on LLM-generated data to construct our
1319   datasets. We do so in part because we could not find existing datasets for the exact misuse risks we
1320   were worried about; synthetic datasets allow us to generate data for the exact task that we have in
1321   mind, and allow us to easily modulate difficulty. In general, the quality of the synthetic datasets we
1322
       generate is also very high—the examples in isolation qualitatively seem like they are well-written
       and salient to the desired task. Synthetic data is also cheap—we generate these datasets with only a
1323
       few API queries—while generating analogous datasets with humans would be costly.
1324
1325   We find that the primary downside of using synthetic data is question diversity; in particular, the
1326   sets of questions we generate qualitatively have slightly less variation than sets of questions humans
1327   would construct. However, we empirically see that there is enough variation to capture differences in
1328
       model performance. If the dataset were relatively homogeneous, models or combinations of models
       would likely tend towards either 0% or 100% accuracy. However, we find that models frequently
1329
       achieve success rates that are comfortably in between these.
1330
1331   We think that using synthetic data did not change our high-level takeaways; the takeaways are valid
1332   for the datasets we use, and we expect that the specific dataset is not responsible for gains from
1333   combining models. We think further assessing the benefits and drawbacks of using synthetic data that
1334
       is tailored for a specific task, rather than real data generated for a more general task, is an interesting
       direction for subsequent work.
1335
1336   LLM evaluation. Our experiments largely rely evaluation that uses an LLM. LLM evaluation enables
1337   us to automatically measure how well language models perform on tasks that do not have single
1338   correct answers, or require long-form outputs. It is also significantly cheaper than human evaluation
1339   on the domains we study, and we think it is high-quality; for a different task, Pan et al. (2023) find
1340
       that LLMs match human labels better than a majority-human ground truth.
1341   Nevertheless, the primary risk of language model evaluation is that it is not accurate. In our settings,
1342   lack of accuracy due to capability would likely affect both combinations and individual models
1343   equally, so it is unlikely to affect our results. Thus, the primary risk is that LLM evaluation is biased
1344   towards combinations over individual models. We think this is unlikely to be the case; for example,
1345   when generating malicious python scripts in Section 5.1 and Section 5.2, the same language model
1346
       ends up producing outputs in the single-model baseline and multi-model cases, yet the LLM evaluator
       favors the combination. To reduce the bias of the LLM evaluator, we additionally use a held-out
1347
       language model for evaluation from those used in the experiments.
1348
1349   We also see similar qualitative results—combining models outperforms individual models—in the
       explicit image experiments in Section 4.2 which relies on human evaluation. LLM evaluation enabled
