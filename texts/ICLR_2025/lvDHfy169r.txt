009                                                A BSTRACT
010
011            Large Language Models (LLMs) have the potential to automate reward engineering
012            for Reinforcement Learning (RL) by leveraging their broad domain knowledge
013            across various tasks. However, they often need many iterations of trial-and-error to
014            generate effective reward functions. This process is costly because evaluating every
015            sampled reward function requires completing the full policy optimization process
016
               for each function. In this paper, we introduce an LLM-driven reward generation
               framework that is able to produce state-of-the-art policies on the challenging Bi-
017
               DexHands benchmark with 20× fewer reward function samples than the prior
018
               state-of-the-art work. Our key insight is that we reduce the problem of generating
019            task-specific rewards to the problem of coarsely estimating task progress. Our
020            two-step solution leverages the task domain knowledge and the code synthesis
021            abilities of LLMs to author progress functions that estimate task progress from a
022            given state. Then, we use this notion of progress to discretize states, and generate
023            count-based intrinsic rewards using the low-dimensional state space. We show that
024            the combination of LLM-generated progress functions and count-based intrinsic
025            rewards is essential for our performance gains, while alternatives such as generic
026            hash-based counts or using progress directly as a reward function fall short.
027
028   1    I NTRODUCTION
029
030   Automated reward engineering aims to reduce the human effort required when using Reinforcement
031   Learning (RL) for sparse-reward tasks. Multiple recent efforts towards automated reward engineering
032   have focused on leveraging Large Language Models (LLMs) to provide reward signals—either
033   employing the LLM output directly as the reward (Kwon et al., 2023), or using the LLM to generate
034   code for a dense reward function (Yu et al., 2023; Ma et al., 2023).
035
      Traditionally, constructing an effective dense reward function is an intricate process of identifying
036   key task elements and carefully weighing different reward terms (Sutton & Barto, 2018; Booth
037   et al., 2023). Prior works on reward function generation attempt to use LLMs both for their domain
038   knowledge and to optimize quantitative aspects of reward engineering (weighting, rescaling) (Yu
039   et al., 2023; Ma et al., 2023)–they can require many training runs as they search through many reward
040   functions in order to find a candidate that is effective for training (Ma et al., 2023).
041
      Our core insight is that we can reduce the problem of reward generation to the task of generating
042   rough measures of task progress. Our framework uses LLMs to generate code for progress functions:
043   task-specific functions that map environment states to scalar measures of progress. For a given genre
044   of tasks, we follow the example from prior work on reward code generation (Yu et al., 2023) by
045   asking practitioners to provide a small helper function library (ex. dist(x,y)) for the particular genre’s
046   observation spaces. Then, given both the helper function library and a single-sentence description of
047   a task within the domain (ex. “This environment require a closed door to be opened and the door
048   can only be pushed outward or initially open inward.”), we leverage LLMs to generate the progress
049   functions.
050   We find that progress functions are most empirically effective when used within a count-based
051   intrinsic reward framework: we treat the outputs of the progress functions as a simplified state
052   space that groups together similar states from the environment, we discretize the states, and we
053   compute state visitation counts across the discretized state space. Then, we treat the inverse square
      root of the visitation count as a count-based intrinsic reward, as in prior work (Kolter & Ng, 2009;

054
      Tang et al., 2017), and learn policies using these count-based intrinsic rewards as task rewards. It
055   may seem straightforward to instead directly sum the outputs of the progress function and use the
056   sum as a reward – providing larger rewards for reaching states corresponding with greater progress
057   through the task. However, this approach neglects the typical reward scaling and weighting issues
058   common to dense reward shaping approaches (Sutton & Barto, 2018; Booth et al., 2023). Our
059   progress-and-counts algorithm, ProgressCounts, achieves SOTA performance on the challenging
060   Bi-DexHands benchmark, outperforming Eureka (Ma et al., 2023) by 4%. By limiting the role of
061   LLMs to generating progress functions and applying count-based intrinsic rewards to simplified
062   progress-based states, ProgressCounts achieves significantly greater sample efficiency compared to
063
      approaches that use LLMs directly for estimating reward weighting and scaling (Yu et al., 2023; Ma
      et al., 2023). Specifically, ProgressCounts requires 20 times fewer training runs than Eureka.
064
065   Specifically, we make the following contributions:
066
067         1. We re-frame the problem of reward generation in terms of generating coarse measures of
068            task progress. Given a single-sentence task description and a small domain-specific library
069            of helper functions, we leverage LLMs to generate progress functions.
070
            2. We generate rewards from progress functions by treating the output of the progress function
071
               as a reduced state representation, discretizing progress in order to measure state visitation
072            counts, and generating count-based intrinsic rewards.
073
074         3. We demonstrate that our algorithm ProgressCounts is effective by obtaining state-of-the-art
075            performance on the Bi-DexHands benchmark. We outperform the prior state-of-the-art
076            Eureka (Ma et al., 2023) by 4% while requiring 20× fewer samples.

079   2    R ELATED WORK
080
081   Automated reward engineering. Learning directly from sparse rewards can be challenging (Ng
082   et al., 1999; Hare, 2019; Vecerik et al., 2017). It is common for practitioners to carefully engineer
083   dense reward functions to shape the learning process (Ng et al., 1999)—a labor-intensive (Sutton &
084   Barto, 2018) and brittle (Booth et al., 2023) process. Advancements in automation such as Population
085
      Based Training (Jaderberg et al., 2017) have shown promise in refining this process by automating
      searches over fixed design spaces.
086
087   Foundation models offer the opportunity to automate reward engineering with powerful priors. The
088   output of foundation models can be used to propose tasks for open-ended learning curricula (Du
089   et al., 2023; Zhang et al., 2023) and add high-level structure to learning (Mirchandani et al., 2021).
090   Foundation models can also be used directly as rewards (Fan et al., 2022; Kwon et al., 2023; Sontakke
091
      et al., 2024), and have the potential to generate reward function code, either scaffolded by reward
      function templates (Yu et al., 2023) or in more freeform fashion (Ma et al., 2023; Venuto et al., 2024;
092
      Li et al., 2024). We also leverage LLMs to inject task knowledge via code, but rather than attempting
093
      to generate complex reward functions, we simply ask them to identify a few key features associated
094   with task progress.
095
096
097   Count-based intrinsic rewards. Our algorithm leverages the idea of count-based intrinsic rewards
098   in order to convert coarse progress-based state representations into rewards for policy learning.
099
      Count-based intrinsic rewards are one of the main approaches to intrinsic motivation: estimating
      the “novelty” of a given state via state visitation counts (Tang et al., 2017). Approaches that
100
      hash continuous spaces to discrete representations have shown considerable promise when the
101
      discretization function is domain-specific (Tang et al., 2017; Ecoffet et al., 2021). In particular, the
102   Go-Explore algorithm (Ecoffet et al., 2021) pairs count-based intrinsic rewards along with simulator
103   state resets in order to achieve state-of-the-art results on several challenging Atari (Bellemare et al.,
104   2013) tasks. The main downside to these approaches is that domain-aware discretization typically
105   requires significant human engineering (Tang et al., 2017; Ecoffet et al., 2021). In our algorithm, by
106   discretizing task progress, we already have access to automated domain-specific state discretizations.
107   Absent the need for extensive human-engineered discretization functions, count-based intrinsic
      rewards are both practical and effective for learning policies from progress functions.

168   4     M ETHODS
169
170   In this section, we first introduce our algorithm for leveraging LLM domain knowledge to generate
171   progress functions, which distill key features from a high-dimensional environment state space to a
172   coarse low-dimensional notion of progress in the task. Then, we outline how we use the generated
173   progress functions: we view progress as measure of state, and leverage count-based intrinsic rewards
174
      for learning.
175
176   4.1     P ROGRESS F UNCTIONS
177
178   4.1.1     P ROGRESS F UNCTION D EFINITION
179
      Given a new task description, the first step of our process is to generate a progress function P : S →
180
      Rk , which takes environment features s ∈ S as input, and outputs information about the current
181   progress of the agent on the task. Especially for more complex tasks, it may be difficult to distill a
182   task to a single feature that tracks overall progress. Therefore, a progress function, given a state, is
183   asked to emit a positive scalar measure of progress for one or more subtasks. For instance, for the
184   SwingCup task (Figure 1-A), which involves 1) gripping the handles of the cup, 2) rotating the cup to
185   the correct orientation, a good progress function would break the task into two sub-tasks and return
186   scalars measuring progress for both sub-tasks.
187   Specifically, the progress function outputs [x1 , x2 , ...xk ] where xi ∈ R tracks task progress for
188   sub-task i. It also outputs additional variables [y1 , y2 , ...yk ] that inform our framework whether the
189   progress variables xi are increasing or decreasing.
190
191   4.1.2     P ROGRESS F UNCTION G ENERATION
192
193   For any given task, domain knowledge is required in order to determine what features from the
194   environment are useful for assessing progress, and how to compute progress from those features. We
195   derive that domain knowledge from an LLM, which is used to generate code for the progress function
196   P . In order to translate domain knowledge into an effective progress function, we provide the LLM
197   with three inputs:
198
              1. Function inputs: we specify the features available as inputs to the progress function via a
199
                 description of the features in the environment state (Figure 1-A, grey). This information is
200
                 available via the simulator.
201
202           2. Function outputs: we specify the desired output of the progress function via a short task
203              description (Figure 1-A, green). Humans specify this information on a per-task basis.
204           3. Function logic: we structure the process of translating feature inputs to progress output by
205              providing the LLM with access to a environment feature engineering library (Figure 1-A,
206              green). This library offers a palette of additional, higher-level features to optionally use to
207              compute progress, and also indirectly suggests that certain types of feature transformations
208              are beneficial to compute progress (ex. l2 dist(x, tgt)). Humans create this library once per
209              genre or benchmark of tasks.
210
211   Note that on a per-task basis, ProgressCounts only requires a user to provide the short task description.
212
      Please see Appendix A.3.2 for the libraries used for our Section 5 benchmarks.
213   Given the high-level task description, the small feature engineering library, and code describing the
214   environment state space, we follow standard LLM prompting strategies to generate the progress
215   function code (Figure 1-A), blue). Please see Appendix A.4 for several examples of generated
      progress functions.

      4.2     F ROM P ROGRESS TO R EWARD
217
218   Given a perfect estimate of task progress, it may seem natural to directly useP progress as a dense
219   reward: for a given state s, compute the sum of the progress outputs Rsum = i xi , and use progress
220   sum Rsum as the reward for reaching s. However, learning from dense rewards can be brittle–small
221   mistakes in reward design can often lead to a failure to learn effective policies (Booth et al., 2023).
222   Progress functions offer highly simplified state representations—and given the coarse nature of these
223   representations, we look for a more forgiving mechanism to generate rewards from these simplified
224
      representations. Therefore, we use a count-based intrinsic reward approach inspired by prior work
      that achieves state-of-the-art performance with domain-specific discretizations (Ecoffet et al., 2021).
225
226

302   against three baselines. 1) Sparse extrinsic rewards upon task success, 2) Dense: expert-written dense
303   extrinsic rewards from the original benchmark, 3) rewards generated using the Eureka LLM-based
304   reward generation algorithm (Ma et al., 2023), the current state-of-the-art reward generation method
305   on Bi-DexHands.
306

471   6    D ISCUSSION
472
473   The state-of-the-art results achieved by ProgressCounts demonstrate two key takeaways:
474   First, LLM-generated progress functions offer a compelling mechanism to generate coarse task-
475   specific state representations, and alongside count-based intrinsic rewards offer an empirically-
476   superior alternative to using LLMs to engineer reward functions. ProgressCounts outperforms Eureka,
477   which uses LLMs to engineer reward functions, both in terms of performance and sample efficiency.
478   One reason for this success is the structure (task progress, count-based rewards, etc.) we build into
479   the ProgressCounts framework, which increases the quality and reliability of LLM responses, and
480   reduces the need for trial and error across reward weights and scaling. Perhaps more interestingly,
481   we hypothesize that ProgressCounts also benefits from count-based intrinsic rewards being robust to
482
      non-optimal binning functions, unlike reward functions where even minor errors can easily lead to a
      failure to solve tasks successfully.
483
484   Second, despite being relatively under-utilized in recent research, count-based intrinsic rewards can
485   be surprisingly effective at training policies that operate in complex high-dimensional state spaces
      when given an adequate binning function. Interestingly, the results achieved by ProgressCounts


                                                         9
      Under review as a conference paper at ICLR 2025


486
      suggest that these binning functions do not need to be hugely complex; ProgressCounts outperforms
487   state-of-the-art, human-engineered dense reward functions using count-based exploration driven by
488   binning functions that contain less than 20 lines of code.
489
490
      Overall, we believe ProgressCounts represents a novel and promising strategy for injecting domain
      knowledge from large language models into an RL training loop. We hope that these results will
491
      encourage further research into (and more general usage of) count-based intrinsic methods, as well as
492
      exploration of other novel methods for leveraging LLMs to assist in solving reinforcement learning
493   tasks.

       A.10    E XPERIMENTAL DETAILS AND HYPERPARAMETERS
1351
1352   Environments We train our policies on two environments: Bi-DexHands (Chen et al., 2022), and
1353   MiniGrid (Chevalier-Boisvert et al., 2024).
1354
1355   Code For the Bi-DexHands benchmark, we build upon the codebase from Eureka (Ma
1356   et al., 2023): https://github.com/eureka-research/Eureka. The repo uses
1357   the RLGames implementation of PPO for training (Makoviichuk & Makoviychuk, 2021).
1358   For the MiniGrid benchmark, we build upon the codebase from NovelD (Zhang et al.,
1359
       2021): https://github.com/tianjunz/NovelD. Our experimental code is is avail-
       able at the following anonymous link: https://drive.google.com/drive/folders/
1360
       1G88Je0K4BuexWhE8ZLgoM0WM6vHcBrvt?usp=sharing.
1361
1362
       Hyperparameters For all Bi-DexHands tasks, we scale extrinsic rewards by 0.05, and normalize
1363
       intrinsic rewards to a mean of 0.001. Elsewhere, we use the default hyperparameters associated
1364   with Eureka, which are the default parameters from the original Bi-DexHands benchmark. Progress
1365   functions are discretized into 1020 bins for count-based exploration (for tasks with two subtasks,
1366   the first subtask is discretized to 20 bins, and the second subtask to 1000 bins). For all MiniGrid
1367   tasks, we set an intrinsic reward coefficint of 0.5, and leave all other hyperparameters at default
1368   values. Progress functions are discretized into 50 bins for count-based exploration (for tasks with two
1369   subtasks, each subtask is discretized to 25 bins).
1370
1371   Compute All experiments were run on a machine with 8 NVidia Tesla V100 GPUs across 3 weeks.
