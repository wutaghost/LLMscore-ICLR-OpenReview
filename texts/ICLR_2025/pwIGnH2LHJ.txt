008                                                               A BSTRACT
009
010                Large Language Models (LLMs) in Software Engineering (SE) can offer assistance for coding. To
011                facilitate a rigorous evaluation of LLMs in practical coding contexts, Carlos et al. introduced the
012                SWE-bench dataset, which comprises 2,294 real-world GitHub issues and their corresponding pull
013                requests, collected from 12 widely used Python repositories. Several impressive LLM-based toolkits
014                recently are developed and evaluated on this dataset. However, a systematic evaluation of the quality
015                of SWE-bench remains missing. In this paper, we addressed this gap by presenting an empirical
016
                   analysis of the SWE-bench dataset. We conducted a manual screening of instances where SWE-
                   Agent + GPT-4 successfully resolved issues by comparing the model-generated patches with the
017
                   actual pull requests. SWE-Agent+GPT-4 was at the top of SWE-bench leaderboard during the time
018
                   of our study. Our analysis reveals some critical issues with the SWE-bench dataset: 1) 32.67% of
019                the successful patches involve “cheating” as the solutions were directly provided in the issue report
020                or the comments. We refer to as ‘solution leakage’ problem. 2) 31.08% of the passed patches are
021                suspicious patches due to weak test cases, i.e., the tests were not adequate to verify the correctness
022                of a patch. When we filtered out these problematic issues, the resolution rate of SWE-Agent+GPT-4
023                drops from 12.47% to 3.97%. We also observed that the same data qualify issues also exist in the
024                two variants of SWE-bench, i.e., SWE-bench Lite and SWE-Bench Verified. In addition, over 94% of
025                the issues were created before LLM’s knowledge cutoff dates, posing potential data leakage issues.
026                The critical problem in the current versions of SWE-bench dataset motivated us to refine it to build
027                a more rigorous evaluation dataset SWE-Bench+. We created SWE-bench+ by collecting GitHub
028                issues that were created after the training cutoff dates of the LLMs to prevent the potential data
029
                   leakage problem. We also ensure that the issues collected do not contain solutions in their reports
                   or comments. After carefully analyzing the passed instances from the SWE-Agent + GPT-4 model
030
                   with the new dataset, SWE-Bench+, we observed a decline in the pass rate, dropping from 3.97%
031
                   (as seen on the refined SWE-Bench) to a resolution rate of 0.55%. We further evaluated SWE-RAG
032                + GPT-4, SWE-RAG + GPT-3.5, and AutoCodeRover + GPT-4o models on the new dataset to verify
033                our findings, where the resolution rates of the models drop significantly, which are 0.73%, 0.55%,
034                and 3.83%, respectively.

037   1       I NTRODUCTION
038
039   SWE-bench aka Software Engineering Benchmark dataset is created to systematically evaluate the capabilities of an
040   LLM in resolving software issues. The dataset contains 2,294 complex issues from GitHub Jimenez et al. (2024).
041   Given as input the issue information to an LLM, the task for the LLM is to modify the code base to address the
042
      issue (i.e., resolution). Each input for an issue consists of a description and a pull request with a reference to the
      corresponding buggy code repository. Each issue can be either a bug report or a new feature request. The pull
043
      request contains the code changes made by developers to address the issue, along with test cases designed to check
044
      if the feature is properly implemented or if the bug is successfully fixed. Two variants of the SWE-bench datasets are
045   recently developed: SWE-bench Lite1 and SWE-bench Verified2 . SWE-bench Lite focuses on 300 issues related to bug
046   fixing. SWE-bench Verified contains 500 verified issues with clear issue descriptions and strong test cases.
047
048
      A significant body of work from both academia and industry has so far utilized SWE-bench and its variants to develop
      and to test LLM coding capabilities Chen et al. (2024); Zhang et al. (2024a); Xia et al. (2024); Yang et al. (2024b);
049
      Zhang et al. (2024c); Rosa et al. (2024); Zan et al. (2024). Given an issue and its associated buggy code repository,
050
      these LLM-based approaches can perform a series of complex tasks, such as reasoning about the target bug’s location,
051   analyzing the root cause of the issue, proposing strategies for fixing the bug, and ultimately writing a patch to fix the

      In this paper, we answer the above question by offering two contributions. First, we present an empirical study of
      state-of-the-art (SOTA) LLMs on SWE-bench Full that explores 1) the quality of SWE-bench issues with a focus on
076
      the testing adequacy of the test cases used for validating patches and 2) the quality of patches generated by the LLMs
077
      to fix the issues. Second, we present an enhancement of SWE-bench Full, which we call SWE-bench+.
078
079   During the time of our empirical study, SWE-Agent+GPT-4 was at the top of the SWE-bench online leaderboard. Other
080   top approaches (e.g., Honeycomb, Amazon Q Developer Agent, and Factory Code Droid) were either closed-sourced
081
      commercial tools or not verified by the SWE-bench team regarding reproducibility. SWE-agent Yang et al. (2024b)
      allows LLM agents to execute basic file operations via shell commands to achieve interaction between the LLM engine
082
      and a software repository. First, we picked issues that were claimed as resolved by SWE-Agent+GPT-4. We did this by
083
      filtering only the instances with evaluation logs showing that all tests passed. Second, we performed a patch validation
084   study by comparing the gold patches (i.e., original) to the model patches (i.e., generated). We did this by comparing
085   the files changed, the lines changed, and the code changes made in the fixes (both original and generated). Third,
086   we determined eight patterns in the generated fixes by reviewing the issue reports, the corresponding tests, and the
087   available discussions of issues (which are treated as hints to LLMs).
088
      Our identified six patterns in the 251 SWE-Agent+GPT-4 patches can be broadly divided into two types: suspicious
089
      fixes and correct fixes. Suspicious fixes corresponded to 63.75% (i.e., 160) of the patches. Two patterns were prevalent
090   in those fixes: 1) Answer Leak. In 32.67% of the resolved instances, the solutions were outlined directly in the issue
091   reports or comments. 2) Weak Tests. In 31.08% of the resolved instances, the changes made by the model are either
092   incorrect, incomplete, or applied to different files or functions compared to the gold patch. Despite these discrepancies,
093   the changes pass the tests, indicating that the tests are too weak to catch such errors. In addition, the dataset can also
094   suffer from potential data leak issues. This is because 94% of the instances in SWE-bench and their pull requests were
095   created prior to the training cut-off dates of the LLMs, meaning that all issue reports in the full SWE-bench dataset
096   may have been exposed to the LLMs during their training phases, raising concerns about potential data leakage.
097   Based on the above observation, we considered fixes corresponding to issues with answer leakage and weak tests as
098   ‘suspicious fixes’. In Figure 1a, we show the distribution of such fixes in the three literature datasets (i.e, SWE-bench
099   full, lite, verified). In Figure 1b, we show that after filtering these suspicious fixes, the correct resolution rate of
100   SWE-Agent+GPT-4 dropped to 3.97% from 12.47%. This drastic drop in resolution rate raises concerns about the
101   robustness of the model-generated patches and the reliability of the SWE-bench dataset itself.
102   To address the problems in SWE-bnech datasets, we created SWE-bench+ dataset, which ensures that: 1) the data
103   were created after the models’ training cut-off dates, and 2) the issues do not include solutions in the issue description
104   or comments. SWE-bench+ dataset is created by following the same data collection methodology described in the
105   SWE-Bench dataset; except we filtered out issues with answer leakage problems. Considering the training cut-off
106   dates of the LLMs used in our study—GPT-3.5 (turbo-16k-0613) with a cut-off in September 2021, GPT-4 (1106) up
107   to April 2023, and GPT-4o (2024-05-13) up to October 2023, we opted to collect data starting a month after the most
      recent model’s cut-off date. To the end, we gathered issues from the period of 2023-11-01 to 2024-08-22. As we show

137   2       ROBUSTNESS A NALYSIS OF SWE-B ENCH
138
139   We conducted an empirical study of SWE-Agent+GPT-4 generated patches for issues in the SWE-bench Full dataset.
140   The goal of our study was to identify whether the patches exhibit any potential problems. Figure 2 outlines the major
141
      steps we followed in our study. The input is the set of all issues in SWE-bench. Each issue contains a description and
      the patch to address the issue. Each patch is a diff of code changes. We call this a “gold patch”. We picked SWE-Agent
142
      + GPT-4 and applied it on each issue to create a fix. We refer to the model output for an issue as a “generated patch”.
143
      We then compared the gold and generated patches to an issue by analyzing the corresponding files changed in the pull
144   requests on GitHub with the same instance id. As we studied the model generated patches, we also examined the logs
145   and trajectories generated by the model. Logs provide the step-by-step execution of the models. The trajectory data
146   provide a detailed record of the models’ decision-making processes while making a resolution as a patch.
147
      To reduce potential biases during the comparison between gold and generated patches, three authors independently
148
      performed the patch validation study. Each author carefully examined the files and lines changed, reviewed the issue
149
      descriptions, and evaluated the implementation styles and intentions behind both the model-generated and developer-
150   generated patches. The disagreements were resolved through a broader discussion involving all the authors.
151
152   For the model generated patches, we focused on instances where the generated patches resolved the issue and passed
153
      all associated tests. As a result, we identified 251 instances from the SWE-Bench Full dataset. Note that, to ensure the
      patches passed all tests, we reviewed the evaluation logs of 286 instances initially marked as resolved by SWE-Agent
154
      + GPT-4 in the results.json file from the SWE-Bench Full evaluation repository and selected only those with logs
155
      confirming that all tests passed following the application of the generated patches.

179   2.1.1     PATTERNS EXTRACTED FROM THE SUSPICIOUS FIXES

181   We observed four patterns in the suspicious fixes, one is attributed to solution leakage and the other three are attributed

184   comments on GitHub. Since both the issue descriptions and comments (referred to as hints text in the SWE-Bench
185   study) are provided as input to the models, these LLM models can extract the solutions directly from this information
186   instead of generating it independently. 32.67% of the successfully resolved issues followed this pattern, making it the

188   validity of the SWE-Bench instances as benchmarks. If a model is simply copying the solution it already has access

      assessment of its ability to generate new solutions. The example shown in Figure 3 illustrates issue report 166694

      2.1.2    PATTERNS EXTRACTED FROM THE CORRECT FIXES
293   1. Different fixes from gold patches: This pattern refers to cases where the model-generated patches present an
294   entirely different solution to the issue compared to the gold patch. Although the coding style and implementation
295   differ, the model-generated patches correctly resolve the issue. Instances in this pattern are considered a correct
296   resolution of the issues.
297
      2. More comprehensive fixes: In contrast to the previous pattern, this refers to instances where the model-generated
298
      patches are more comprehensive than the gold patches. For example, the model-generated patches may include ad-
299   ditional if-else cases or other logic that the gold patch does not address. This showcases a strength of LLMs, as they
300   can generate more thorough solutions that cover scenarios developers might overlook, resulting in potentially safer
301   and more robust solutions (e.g., by incorporating try-catch statements or handling edge cases more effectively). We
302   consider this pattern as a correct resolution of the issue.

399   limits compared to the Claude models. Initial trials with SWE-Agent+Claude 3 Opus and RAG-based Claude models
400   (SWE-RAG+Claude 3 Opus and SWE-RAG+Claude 2) revealed that the token limits were exceeded before completing
401
      experiments on all 548 SWE-Bench+ instances. As the leaderboard frequently updates, with new models surpassing
      older ones, AutoCodeRover+GPT-4o recently ranked among the top three models, and since it is open-source, we also
402
      included it in the study. We following their instructions to run the models on our SWE-Bench+ dataset.
403
404   Our evaluation followed a four-step technique, described in Figure 2, to ensure that the resolution rates of the models
405   on SWE-Bench+ accurately reflected their correctness. Specifically, in Step 1, we stored all the model-generated diff
406   files as patches in separate json files, we then utilized the scripts provided by the SWE-Bench to evaluate the generated
407
      patches. In Step 2, once the evaluation results were generated, we manually reviewed the instances marked as resolved
      and selected only those where all tests in PASS TO PASS and FAIL TO PASS had passed. In other words, we filtered
408
      out instances where the patches were successfully applied, and all associated tests passed. In Step 3, after filtering
409
      the resolved instances where all tests passed, we analyzed the resolution rates of the models. Finally, in Step 4, we
410   performed a similar patch validation study as described in Section 2, comparing the model-generated patches to the
411   gold patches. We found that solution-leak related issues are now resolved. However, a prominent issue with weak
412   test cases persists. As shown in Table 3, on average, around 67.72% of the resolved instances did not truly resolve
413   the issue, despite passing all the tests. The prominent pattern identified was the models’ inability to accurately locate
414   the buggy files or lines. This raises concerns about the models’ ability to locate the bug. In other cases, the model
415   generated incomplete or incorrect fixes, not resolving the bug. These findings show that the issue with weak tests still
416   persists and needs future investigation.
417   As a result of the patch validation analysis, the resolution rates were 0.73% for SWE-RAG+GPT-4, 0.55% for SWE-
418   RAG+GPT-3.5, 0.55% for SWE-Agent+GPT-4, and 3.83% for AutoCodeRover+GPT-4o. These rates are significantly
419   lower than the reported resolution rates on the SWE-Bench leaderboard were 1.31% for SWE-RAG + GPT-4, 0.17%
420   for SWE-RAG + GPT-3.5, and 18.83% for AutoCodeRover + GPT-4o. This drop in performance for all models when
421   moving from SWE-Bench Full to SWE-Bench+, highlights the impact of excluding suspicious fixes on the overall
422   resolution rates.

      LLM for Software Engineering. Large Language Models (LLMs) have emerged as powerful tools and demon-
464
      strated impressive capabilities in various software engineering tasks, including code generation Jiang et al. (2024);
465   Li & Döhmen (2024); Chen et al. (2021); Luo et al. (2024); Du et al. (2024), program repair Zhang et al. (2024b);
466   Yang et al. (2024a); de Fitero-Dominguez et al. (2024) and bug detection Alrashedy & Binjahlan (2024); Hossain
467   et al. (2024). The development of code generation benchmarks has been crucial for evaluating LLM performance.
468   Notably, HumanEval Chen et al. (2021) was introduced to assess the functional correctness of code generated by
469   LLMs. Building on this foundation, AlphaCode Li et al. (2022) demonstrated competitive performance in solving
470   complex programming problems. To address limitations in existing benchmarks, EvalPlus Liu et al. (2024) enhanced
471   HumanEval with more comprehensive test cases and revealed a significant overestimation of LLM performance in
472   previous evaluations. LLMs also have shown promising results in program repair and bug detection. For example,
473
      AlphaRepair Xia & Zhang (2022) employed a zero-shot learning approach that outperformed traditional automated
      program repair (APR) tools. Further research demonstrated that LLMs could surpass existing APR techniques, par-
474
      ticularly when fine-tuned on domain-specific data Xia et al. (2023). The application of LLMs in bug detection with
475
      innovative approaches like FUzzGPT Deng et al. (2023b) and TitanFuzz Deng et al. (2023a) leveraging these models
476   to generate edge-case test inputs and perform mutation-based fuzzing for deep learning libraries. There are several
477   comprehensive studies have explored LLM applications across various software engineering domains Fan et al. (2023);
478   Hou et al. (2024), delved into the natural language to code generation Zan et al. (2023), and analyzed the evolution
479   and performance of Code LLMs across different tasks Zheng et al. (2024).
480
481   Benchmark Dataset Quality for Code Generation. To achieve accurate and reliable outcomes in code generation
482   tasks with LLMs, it is essential to use high-quality evaluation benchmark datasets during the training and evalua-
483   tion steps Jimenez et al. (2024). With the growing interest and significance of code generation and program repair
484   in software engineering, this highlights the critical need for trustworthy and reliable evaluation benchmark datasets
485   Jimenez et al. (2024); Zan et al. (2024). To tackle this challenge, SWE-bench was developed to assist developers in
      evaluating LLMs using real-world GitHub issues Jimenez et al. (2024). A critical part of making a reliable dataset


                                                       9
      Under review as a conference paper at ICLR 2025


486
      is to make it representative of real-world and complex software challenges and issues Chen et al. (2024); Tao et al.
487   (2024). The “Diversity Empowers Intelligence” (DEI) framework Zhang et al. (2024a) further shows how diversity
488   and comprehensiveness of data can enhance the performance of Large Language Models (LLMs) in code generation
489   tasks. Given the critical need for high-quality benchmark datasets in evaluating code generation models, Liu et al. Liu
490   et al. (2024) introduce EvalPlus, a framework specifically designed to improve the evaluation of LLM-generated code.
491   This framework addresses challenges like insufficient tests and noisy benchmarks by augmenting existing datasets
492   with automated test input generation using LLM and mutation-based strategies. Their findings show the need for more
493   rigorous testing in LLM code generation to improve benchmark quality in the field.
494
495   7   C ONCLUSION
496
497   In this paper, we presented the first empirical study on the robustness of the SWE-Bench dataset. Our study identified
498   significant limitations in the original SWE-Bench dataset, particularly issues with solution leakage and weak test cases,
499   which undermined the reliability of previous model assessments. To address these challenges, we introduced SWE-
500   Bench+, a dataset free from solution leakage and built with issues created after LLM training cut-off dates to ensure
501   more rigorous and accurate evaluations. Through extensive testing, we demonstrated that while SWE-Bench+ resolves
502   the data leakage concerns, weak test cases continue to pose challenges, with model resolution rates dropping further
503   in this refined environment. Despite the reduced pass rates, SWE-Bench+ establishes a more reliable framework for
504
      assessing the true capabilities of LLMs in software development, offering insights into how these models can be
      better developed and evaluated. Future work should focus on improving the test case robustness in SWE-Bench+ and
505
      exploring more effective strategies for filtering data to further minimize biases and inaccuracies.
506
507   For future work, further investigation into the issue of weak test cases is needed, along with suggestions for improving
508   test quality to create more accurate and relevant test suites. Another potential avenue of research could explore the
509   underlying causes of the high failure rates and propose strategies to mitigate them. Additionally, similar studies could
510
      be conducted on other leading evaluation benchmarks, such as Human-Eval, to compare results and identify broader
      patterns.
