    1. Enhanced Normalization Rules: MalTrans introduces new normalization rules for assembly

       due to its unsuitable normalization rules and less effective model architecture. In contrast,
       MalTrans achieves superior malware detection across ISAs, thanks to its improved model
       architecture and tailored normalization schemes.
    3. Broader ISA Coverage: While UnsuperBinTrans is limited to two ISAs (x86-64 and ARM
       32), MalTrans extends the evaluation to a wider range of ISAs, demonstrating both broader
       applicability and improved performance.

We hope this clarifies the distinct advantages of MalTrans compared to UnsuperBinTrans.

translation quality based on structural similarity. However, downstream malware detection depends
more on preserving code semantics rather than exact n-gram matches. This may explain why
ARM32→X86 achieves higher detection accuracy despite a lower BLEU score, as the translated
ARM32 code may better retain semantic features relevant to detection.

8. Reviewer's Comment:
“Why did the paper utilize 4-gram precision? Why not higher than 4-gram precision?”

Authors’ Response:
We have clarified in the first paragraph in Section 5.3 (Line 355) that the BLEU scores are computed
as an average of unigram, bigram, trigram, and 4-gram precision.

We did not consider n-grams higher than 4-grams for the following reasons. (1) Higher n-grams (5+
grams) become increasingly sparse. This issue is exacerbated in code translation, where instruction
sequences are typically shorter than natural language sentences, making higher n-grams less reliable as
evaluation metrics. (2) The original BLEU metric proposed by [16] demonstrated that 4-gram BLEU
provides an optimal balance between accuracy and computational efficiency. Beyond 4-grams, the
benefits diminish due to increased computational costs and reduced reliability.

----------------------------------------------------------------------------------------------------------------------------

9. Reviewer's Comment:
“How does the translation quality compare with other works such as [11] and [12]?”

Authors’ Response:
The work in [11] utilizes neural machine translation techniques for binary code similarity comparison
but does not perform binary code translation across ISAs. Specifically, it employs only the encoder of a
neural machine translation model to generate embeddings for two pieces of binary code, and measures
similarity based on embedding distance. In contrast, our work focuses on translating binary code across
different ISAs.
The work in [12] focuses on source code translation (e.g., C to Java), which differs significantly from
binary code translation (e.g., assembly code from x86-64 to ARM32).

As the objectives and languages differ, a direct comparison of translation quality is not applicable.

092   Signature-Based Detection. Traditionally, malware detection relied heavily on signature-based
093   methods, where known patterns of malicious code are identified and stored in databases Sebastio

099   of programs at runtime Liu et al. (2011); Burguera et al. (2011); Aslan et al. (2021); Saracino et al.
100   (2016). It looks for suspicious activities, such as unusual system calls or network behavior Burguera
101   et al. (2011); Aslan et al. (2021). However, these approaches are unscalable and suffer from false-
102   negative rates if the malicious behavior is not triggered during monitoring.
103   Machine/Deep Learning-Based Detection. In recent years, machine/deep learning has emerged as a
104   powerful tool for malware detection. Machine/deep learning techniques can analyze vast amounts of

106   et al. (2018), Support Vector Machines (SVMs) Hasan & Rahman (2017), and neural networks Jeon
107   et al. (2020), including models like CNNs, RNNs, and LSTM, have been widely applied in malware

123   2.2   S OURCE C ODE T RANSLATION
124
      Source code translation refers to the process of converting code written in one programming language
125
      into another. Early work in source code translation uses transpilers or transcompilers Andrés &
126   Pérez (2017); Tripuramallu et al. (2024), which rely on handcrafted rules. However, they produce
127   unidiomatic translations that prove hard for human programmers to read. Another issue is incomplete
128   feature support. For instance, a feature available in the source language might not have a direct
129   equivalent in the target language, leading to functional gaps or the need for workarounds.
130
      Recent advancements in deep learning have introduced new approaches to source code transla-
131
      tion Roziere et al. (2020); Weisz et al. (2021); Lachaux et al. (2020). But as malware is closed-source
132   where the source code is usually unavailable, source code translation does not work for malware.
133
134
      2.3   P ROGRAM A NALYSIS -BASED B INARY C ODE T RANSLATION
135
136   Several approaches leverage program analysis techniques to convert binary code from one ISA
137   to another. They can be broadly categorized into static analysis-based translation and dynamic
138   analysis-based translation. Static analysis-based translation analyzes and translates the binary code
139   before execution Shen et al. (2012); Cifuentes & Van Emmerik (2000). However, ensuring that all
140   paths are accurately translated can be challenging. Dynamic analysis-based translation performs
141
      translation during execution Chernoff et al. (1998); Ebcioglu et al. (2001), but it introduces runtime
      overhead and requires sophisticated runtime environments, which can increase complexity and
142
      resource consumption. Additionally, both approaches face challenges related to architecture-specific
143
      features and encounter difficulties in achieving accuracy, performance, and compatibility.
144
145   In this work, we propose unsupervised binary code translation inspired by recent advances in neural
146   machine translation. This represents a new and emerging direction. To the best of our knowledge,
147
      UnsuperBinTrans Ahmad & Luo (2023) is the first and only existing work that applies deep learning
      techniques to binary code translation. Our evaluation demonstrates that our model outperforms
148
      UnsuperBinTrans in binary code translation and achieves superior malware detection across ISAs.

      their development, driving a parallel increase in malware targeting multiple ISAs. This highlights the
155
      critical need to extend malware detection capabilities across ISAs. However, existing deep learning-
156   based methods typically rely on large datasets of malware samples for training. The diversity of ISAs
157   exacerbates this challenge, as sufficient malware data is often unavailable for many ISAs, hindering
158   effective detection. Furthermore, collecting and labeling malware to build comprehensive datasets for
159   each ISA is both labor-intensive and time-consuming. To reduce the burden of data collection, our
160   idea is to translate a binary from one ISA to another ISA with sufficient malware samples. We refer
161   to the former ISA as the source ISA, and the latter one as the target ISA. Subsequently, we can use a
      malware detection model trained on the target ISA to test the translated binary.

      Overview. Figure 1 shows the workflow of applying our binary translation model, called M ALT RANS,
163   to detect malware across ISAs. Note that our goal is to reduce the effort required to collect task-
164
      specific datasets—which contain labeled malware samples (as related to the step 2 in Figure 1)—for
165
      training the malware detection model. This contrasts with mono-architecture datasets that can be
166
      conveniently created using open-source programs (as related to the step 1 ) for training M ALT RANS.
167
168   In step 1 , we use widely available open-source programs to train M ALT RANS for translating
169   binaries from the source ISA to the target ISA (such as X86-64). As the training is unsupervised,
170   we cross-compile the open-source programs on different ISAs using cross-compilers to build mono-
171   architecture datasets. Notably, the training of M ALT RANS does not require any malware samples.
172   Moreover, it should be highlighted that malware is typically a closed-source program, where the
173   source code is usually unavailable. Thus, cross-compilation that generates binaries across ISAs from
174   source code does not apply to malware.
175
      In step 2 , we train a deep learning-based malware detection model using a task-specific dataset
176
      containing malware and benign samples in the target ISA.
177
178   Finally, in step 3 , when dealing with a binary in the source ISA, we use M ALT RANS to translate
179   the binary to the target ISA and reuse the malware detection model trained on the target ISA to test
180   the translated code for detecting malware.
181   In summary, our approach leverages the abundance of malware samples available for the target ISA
182   to enhance detection capabilities for other ISAs, thereby reducing the burden of data collection for
183   malware detection. More importantly, collecting sufficient malware samples for certain ISAs can be
184   particularly challenging. Our approach overcomes this difficulty, making robust malware detection
185   feasible even for ISAs where malware collection is difficult.

      4.1   I NSTRUCTION N ORMALIZATION
193
      A binary, after being disassembled, is expressed in an assembly language. Given this insight, a surge
194   of NLP-inspired binary analysis approaches have been proposed Li et al. (2022; 2023); Zan et al.
195   (2022); Ye et al. (2023). A binary is represented as a sequence of instructions. An instruction includes
196   an opcode (specifying the operation to be performed) and zero or more operands (specifying registers,
197   memory locations, or literals). For example, mov eax, ebx is an instruction where mov is an opcode
198   and eax and ebx are operands.
199
      In NLP, the out-of-vocabulary (OOV) issue is a well-known problem, and it exacerbates significantly
200   in our case, as constants, strings, and address offsets are frequently used in instructions. To mitigate
201   the OOV problem, we employ the normalization strategy. Furthermore, according to a study by Jean
202   et al. (2015), learning a translation model with a large vocabulary can significantly increase the
203   computation complexity and hamper translation performance. Thus, normalizing instructions can
204   also reduce the vocabulary size of both source ISA and target ISA, as well as minimize the vocabulary
205   size discrepancy, thereby enhancing translation performance (see the evaluation).
206   Below are the normalization rules. Appendix A shows examples by the application of these rules.
207
208          • (R1): We use IDA Pro IDA (2023) to disassemble binaries, which generates dummy
209            names Dummy name (2023). We replace dummy names with their respective prefixes. E.g.,
210            i) word_, dword_, and xmmword_ represent data of different lengths. They are replaced
211
               with <WORD>, <DWORD> and <XMMWORD>. ii) off_ and seg_ represent offset pointer value
               and segment address value. They are replaced with <OFF> and <SEG>. Other symbols are
212
               replaced with <TAG>.
213
214
             • (R2): Function names are replaced with <FUNC>.
215          • (R3): Number constants are replaced with <VALUE>. Hexadecimal numbers are replaced
               with <HEX>. Minus signs are preserved.

228   4.2       M ODEL A RCHITECTURE
229
      M ALT RANS contains a shared encoder for both source and target ISAs and a separate decoder for
230
      each ISA. We use a multi-layer bidirectional Transformer to design the encoder and decoders. The
231   model architecture is shown in Figure 2. We regard opcodes/operands as words and basic blocks as
232   sentences. A basic block is a straight-line sequence of instructions with no branches inside it.
233
234
      We train M ALT RANS in an unsupervised manner. Pretraining is a key ingredient of unsupervised
      neural machine translation Conneau et al. (2020); Conneau & Lample (2019). Studies have shown
235
      that the pretrained cross-lingual word embeddings has a significant impact on the performance of an
236
      unsupervised machine translation model Yang et al. (2018). We adopt this and pretrain the encoder
237   and decoders of M ALT RANS to bootstrap the iterative process of our binary translation model.
238
239
      4.3       M ODEL P RETRAINING
240
241   We employ the causal language modeling (CLM) and masked language modeling (MLM) objectives
242   to train the encoder and decoder. (1) The CLM objective involves training the model to predict a
243   token et , given the previous (t − 1) tokens in a basic block P (et |e1 , ..., et−1 , θ). (2) For MLM, we
244   randomly sample 15% of the tokens within the input basic block and replace them with [MASK] 80%
245   of the time, with a random token 10% of the time, or leave them unchanged 10% of the time.
246   The first and last token of an input basic block is a special token [/s], which marks the start and
247   end of a basic block. We add position embedding and architecture embedding to token embedding,
248   and use this combined vector as the input to the bi-directional Transformer network. Position
249   embeddings represent different positions in a basic block, while architecture embeddings specify the
250   architecture of a basic block. Both position and architecture embeddings are trained along with the
251   token embeddings and help dynamically adjust the token embeddings based on their locations.
252
253   4.4       M ODEL T RAINING FOR C ODE T RANSLATION
254
      We train M ALT RANS in an unsupervised manner using the following learning objectives: denoising
255
      auto-encoding (DAE) and back translation (BT).
256
257   Denoising Auto-Encoding (DAE). The DAE reconstructs a basic block from its noised version, as
258   depicted in Figure 2(a). Given the input source block, Bsrc , we introduce random noise into it (e.g.,
                                                                                                        ′
259   altering the token order by making random swaps of tokens), resulting in the noised version, Bsrc .
               ′
260   Then, Bsrc is fed into the shared encoder, whose output is analyzed by the source decoder. The
261   training aims to optimize both the shared encoder and source decoder to effectively recover Bsrc .
262   Through this, the model can better accommodate the inherent token order divergences. Similarly, the
263   shared encoder and source decoder are optimized when the input is a target basic block, Btgt .
264   Back Translation (BT). We adapt the back-translation approach Feldman & Coto-Solano (2020) to
265   train our model in a translation setting, as shown in Figure 2(b). As an example, given an input source
266   block Bsrc , we use the model to translate it to the target ISA (i.e., applying the shared encoder and the
267   target decoder), as shown in the process 3 . We then feed the translated block B     etgt to the model and
268   train it to predict the original block Bsrc (i.e., applying the shared encoder and the source decoder),
269   as shown in the process 4 . As training progresses, the model produces better synthetic basic block
      pairs through back translation, which serve to further improve the model in the subsequent iterations.

      5.1   E XPERIMENTAL S ETTINGS
277   We implemented M ALT RANS using Transformer. Specifically, for each shared encoder and separate
278   decoder, we used a 4-layer Transformer with 64 hidden units, 4 heads, ReLU activations, a dropout
279   rate of 0.1, and learned positional embeddings. Appendix C presents the details of the model. All
280   the experiments were conducted on a computer with a 64-bit 3.6 GHz Intel Core i9-CPU, a Nvidia
281   GeForce RTX 4090, 64GB RAM, and 4TB HD.
282
      Model Comparison. We consider two types of models for comparison as follows.
283
284          • Baseline Model 1: UnsuperBinTrans Ahmad & Luo (2023). To the best of our knowl-
285            edge, UnsuperBinTrans is the first and only existing work that focuses on binary code
286            translation using deep learning techniques. UnsuperBinTrans employs an encoder-decoder
287            architecture, using RNNs for the encoder and decoder, and is based on unsupervised training.
288          • Baseline Model 2: IR-based malware detection model. Intermediate representation (IR) can
289            abstract away architecture differences inherent in different ISAs, and represent instruction
290            sets of different ISAs in a uniform style angr (2024). We consider a malware detection
291            model trained on IR code as a baseline.
292          • Optimal Model: Same-ISA model. We consider a malware detection model trained and tested
293            on the same ISA, without employing any translation, as the optimal model. As expected,
294            this model, if trained with sufficient data, is likely to outperform a model trained on one ISA
295            and tested on another, representing the best-case scenario. Moreover, if the performance
296            difference between the best model and our model is small, it indicates effective translation.

299
      We consider six ISAs: X86-64, i386, ARM64, ARM32, MIPS32, and s390x. Our goal is to reuse the
300
      malware detection model trained on X86-64 (where sufficient malware samples are available) for the
301   other four ISAs. To achieve this, we translate binaries from the other five ISAs to X86-64.
302
303   Mono-Arch Datasets for Training M ALT RANS. We first collect various open-source programs,
304
      including openssl-1.1.1p, binutils-2.34, findutils-4.8.0, and libgpg-error-1.45. They are widely used
      in prior NLP-based binary analysis works Ding et al. (2019); Li et al. (2021). We compile these
305
      programs on each ISA using GCC-11.4.0 with different optimization levels (O0-O3). We then
306
      disassemble each binary using IDA Pro IDA (2023) and collect basic blocks, which are normalized
307   and deduplicated. Finally, we create a mono-architecture dataset for each ISA: 2, 789, 119 blocks for
308   X86-64, 2, 803, 557 blocks for i386, 7, 413, 083 blocks for ARM64, 5, 812, 795 blocks for ARM32,
309   4, 813, 685 blocks for MIPS32, and 5, 365, 474 blocks for s390x. We evaluate the adequacy of our

      testing the translation capability of M ALT RANS and (2) the testing dataset used in the malware
313   detection task. The details of these datasets are introduced in the following sections.

315   Byte Pair Encoding (BPE). After creating the mono-arch dataset for each ISA, we use byte pair
316
      encoding (BPE) (Sennrich et al., 2016) to process the datasets. The BPE merge times can change the
      vocabulary size. Based on our investigations, we find that the vocabulary size discrepancy between
317
      the source and target ISA plays a critical role in the model’s translation performance. Therefore, to
318
      enhance M ALT RANS’s translation capability, we set the BPE merge times based on the following
319   principles derived from empirical experiments and theoretical insights presented in prior research:
320
321          • The vocabulary size discrepancy between the source and target ISA should not exceed 15%.
322            A large vocabulary discrepancy can lead to an imbalanced learning problem, where the
323            model disproportionately focuses on the larger vocabulary, resulting in inefficiencies or
               overfitting to the less-represented vocabulary Gowda & May (2020).

      model, due to the insufficient malware samples used to train the optimal model. For ARM64, only
      80% of 58 ARM64 malware samples are used for training, while for s390x, only 1 out of 2 malware
475
      samples is used. This highlights the importance of a sufficiently large training dataset to achieve
476
      desirable performance. While increasing the dataset size could enhance the model’s performance,
477   collecting malware samples for certain ISAs can be challenging. Our approach—translating binaries
478   to x86-64—addresses this data collection challenge effectively. In an extreme case, if only one binary
479   in a given ISA is available, we can still detect whether it is malware using the X86-64-trained model.

530
531
      Applying deep learning to malware detection has drawn great attention. The limited availability of
      malware in certain ISAs, however, hinders deep learning-based malware detection. In this work, we
532
      proposed M ALT RANS, which translates binaries across ISAs. The training of M ALT RANS does not
533
      require malware and relies only on mono-architecture datasets created from open-source programs.
534   We apply M ALT RANS to malware detection across six ISAs. Considering that X86-64 is the most
535   data-rich ISA, we train M ALT RANS to translate binaries from the other five ISAs to X86-64, and reuse
536   a malware detection model trained on X86-64 to test the translated code. Our approach effectively
537   reduces the burden of data collection. Compared to UnsuperBinTrans, M ALT RANSachieves better
538   malware detection across ISAs, thanks to its improved model architecture and tailored normalization
539   schemes, contributing to its superior translation capability. Furthermore, we expanded the evaluation
      to cover more ISAs, showcasing M ALT RANS’s broader applicability and improved performance.

782   In NLP, it is widely recognized that a comprehensive dataset, which ensures that the vocabulary
783
      covers a wide range of words, is crucial for training effective code translation models. We assess
      the adequacy of our mono-architecture datasets. Specifically, we study the vocabulary growth as
784
      we incrementally include programs. Our findings indicate that while the vocabulary size initially
785
      increases with the inclusion of more programs, it eventually stabilizes. Take x86-64 as an example,
786   including openssl-1.1.1p results in a vocabulary size of 23, 029. The size increases to 36, 770 (a 60%
787   growth) when binutils-2.34 is added, and then increases to 39, 499 (a 7.4% growth) and 39, 892 (a
788   0.9% growth) when findutils-4.8.0, libpgp-error-1.45 are included, respectively. The growth trend is
789   similar for other ISAs. It shows that the vocabulary barely grows in the end when more programs
790   are added. According to the vocabulary growth trend as well as the high performance achieved, our
791   mono-architecture datasets are adequate to cover instructions and enable effective code translation.

      By comparing the translated X86-64 block with the target X86-64 block, we observe that M ALT RANS

928   predicts a different register and memory cell for the operands, while preserving the functionality of

1006   Uniqueness of Our Work. We propose an entirely different approach compared to the works
1007   in Wang et al. (2023b; 2024), which also aims to reuse models across ISAs. While their approach
1008   achieves this by learning cross-architecture instruction embeddings, our method focuses on translating
1009   binary code across ISAs. By translating code to a high-resource ISA, our approach offers several
1010
       advantages. First, it allows the direct application of existing downstream models—trained on the
       high-resource ISA—to other ISAs through testing the translated code. In contrast, the works in Wang
1011
       et al. (2023b; 2024) require retraining the model using cross-architecture instruction embeddings.
1012
       Furthermore, translating code from one ISA to another assists human analysts in understanding code
1013   from unfamiliar ISAs, supporting broader applications in code comprehension.
1014
1015   InnerEye Zuo et al. (2018) applies neural machine translation techniques for binary code similarity
1016
       comparison but does not perform binary code translation across ISAs. It uses two encoders from
       neural machine translation models, where each encoder generates an embedding for a piece of binary
1017
       code of a given pairs, and measures similarity based on embedding distance. In contrast, our approach
1018
       focuses on translating binary code across ISAs.

1020   Packed or Obfuscated Malware. To address packed malware, we can incorporate advanced
1021   unpacking tools, such as PEiD PEiD (2008) and OllyDbg OllyDbg (2000), to first unpack the
1022
       malware and then analyze the unpacked content.
1023   The malware samples used in our study were from VirusShare.com VirusShare (2020), a repository
1024   that collects malware observed in the wild. It is widely recognized that such malware often employs
1025   obfuscation techniques to evade detection by antivirus systems. However, we lack ground truth
       regarding the specific obfuscation techniques applied to each malware sample, making it difficult to


                                                                19
       Under review as a conference paper at ICLR 2025


1026
       assess resilience to specific obfuscation techniques. Identifying the obfuscation techniques used in a
1027   given malware sample is a challenging and open research problem in its own right.
1028
1029   Our study primarily addresses the challenge of data scarcity in low-resource ISAs by translating
1030
       binaries from these ISAs to a high-resource ISA using M ALT RANS. Future work could systematically
       explore the impact of obfuscation techniques on detection performance. A key challenge is the
1031
       absence of a high-fidelity dataset mapping malware samples to their specific obfuscation techniques.
1032
       Filling this gap will be a focus of our future research.
