009                                                A BSTRACT
010
011            Large language models (LLMs) have shown significant advancements in code
012            generation, but still face challenges on tasks beyond their basic capabilities. Re-
013            cently, the notion of self-debugging has been proposed to boost the performance of
014            code generation by leveraging execution feedback from tests. Despite its promise,
015            the availability of high-quality tests in real-world scenarios is limited. In this con-
016            text, self-debugging with self-generated tests is a promising solution but lacks a
017            full exploration of its limitations and practical potential. Therefore, we investigate
018
               its efficacy on diverse programming problems. To deepen our understanding, we
               propose two distinct paradigms for the process: post-execution and in-execution
019
               self-debugging. Within the scope of self-contained Python programming tasks,
020
               we find that post-execution self-debugging struggles on basic problems but shows
021            potential for improvement on competitive ones, due to the bias introduced by self-
022            generated tests. On the other hand, in-execution self-debugging enables LLMs to
023            mitigate the bias by solely leveraging intermediate states during execution, thereby
024            enhancing code generation.

027   1   I NTRODUCTION
028
029   Large language models (LLMs) have demonstrated considerable progress in code generation, but
030   still face challenges to perform complex programming tasks beyond their basic capabilities. The
031   tasks require LLMs to understand the given natural language specifications and generate programs
032   that could pass all the private tests. Recently, self-debugging has emerged as a promising approach
033
      to boost the performance of LLMs in code generation (Chen et al., 2024b; Jiang et al., 2023; Zhong
      et al., 2024). This approach enables models to refine their own output through an iteration of genera-
034
      tion and execution for the programs utilizing pre-built oracle tests. However, in real-world scenarios
035
      of software development, oracle tests are not available for each code snippet.
036
037   To address this challenge, recent studies have introduced self-generated tests into self-debugging
038   process (Shinn et al., 2024; Huang et al., 2023; Ridnik et al., 2024). As illustrated in Figure 1,
039
      in this framework, the model first generates an initial program and a suite of tests based on the
      natural language specifications of the problem. The program is then executed on the self-generated
040
      tests with an executor (e.g. code interpreter). If it raises any error, the signal or message will be
041
      collected as execution feedback, which the model uses to generate a revised version of the program.
042   It helps reduce the reliance on external feedback from humans or stronger models and thus holds the
043   potential to be generally applied in various code generation tasks.
044
045
      Nonetheless, the efficacy of self-debugging with self-generated tests remains underexplored. Reflex-
      ion (Shinn et al., 2024) leverages feedback from self-generated tests to debug but evaluates the code
046
      before repair with hidden oracle tests. AlphaCodium (Ridnik et al., 2024) first iterates on public
047
      oracle tests and then on model-generated tests with a technique of test anchors. The improvements
048   observed using oracle tests do not accurately demonstrate the true self-debugging capabilities of
049   LLMs. This highlights the need for more transparent evaluation to better understand the inherent
050   debugging potential with self-generated tests. To study this, we first clarify the concept of self-
051   debugging in practice, a scenario wherein the model attempts to debug and repair its own programs
052   without reliance on human supervision or guidance from stronger models. Beyond leveraging the
053   model’s intrinsic capabilities, execution feedback from self-generated tests also serves as additional
      signals to help LLMs identify bugs in its programs according to specifications. Depending on the ex-

082   We summarize our observations as follows:
083
084   • In the context of self-contained Python programming tasks, post-execution self-debugging strug-
085     gles with relatively basic problems, such as those in HumanEval and MBPP. However, it shows
086     potential for improvement on more challenging programming problems in LiveCodeBench.
087   • This discrepancy is attributed to the bias introduced by self-generated tests, which refers to the
088     misalignment between self-testing labels and true labels for the programs. In addition to the

091

093     results demonstrate promising improvements for both basic and competitive tasks.
094
095   Through our study, we aim to shed light on the practicality of self-debugging with self-generated
096   tests, contributing valuable insights into the future development of LLMs in code generation tasks.

119   Self-Debug with LLMs. As LLMs have evolved, the idea of using models to refine their own
120   output has become more popular. In code generation, several techniques have explored how LLMs
121   can refine the code they generate. Most of these methods rely on prompting LLMs with execution
122   results to improve the code. These methods often rely on pre-existing or generated tests to execute
123
      the code, capturing execution information that is then used to refine the output code (Olausson
      et al. (2024); Wang et al. (2024); Dong et al. (2023); Madaan et al. (2023); Zhang et al. (2023a)).
124
      Self-Debugging (Chen et al., 2024b) introduces a framework in which LLMs iteratively debug their
125
      own generated code by utilizing execution results and self-generated explanations. Self-Edit (Zhang
126   et al., 2023a) builds on the example tests provided in programming problems for execution to help
127   the model correct its own output. LDB (Zhong et al., 2024) utilizes runtime execution information
128   to help debug generated programs. Jiang et al. (2024) enhance LLM self-debugging by training
129   on an automatically collected dataset for code refinement and explanation. Madaan et al. (2023)
130   conduct a broad evaluation of self-debugging in code models, highlighting that performance can be
131   improved with higher-quality feedback or human intervention. In this work, we aim to explore the
132   potential as well as limitations of execution-based self-debugging methods, particularly with self-
133   generated tests. We provide a detailed analysis of these methods and propose a unified framework
134
      in the following Section 3.

136   3    S ELF -D EBUGGING WITH S ELF -G ENERATED TESTS
137
138
      We focus on evaluating the self-debugging capabilities of large language models (LLMs) through
139
      execution on self-generated tests. Figure 1 provides a comprehensive overview of this process.
140   Given a problem with a natural language specification, the LLM (denoted as M) first generates an
141   initial program C along with a suite of test cases, denoted as {(Xi , Yi )}N
                                                                                 i=1 , where Xi represents
142   the input and Yi represents the expected output for the i-th test. To enhance the model’s debug-
143   ging performance beyond its intrinsic reasoning capabilities, we utilize execution feedback as an
144   additional signal to help the model identify bugs in its generated program according to the problem
145   specification. Specifically, we employ an executor (denoted as E) to run the generated program on
146   the test suite and collect execution information as feedback.
147   There are various implementations for utilizing execution feedback, which we categorize into two
148   distinct paradigms: Post-Execution and In-Execution self-debugging. These paradigms reflect
149   the type of information employed in the self-debugging process. Post-execution information refers
150   to content obtained after the program’s execution, such as execution outputs or error messages. In
151   contrast, in-execution information refers to intermediate states observed during execution, providing
152   finer-grained insights into the program’s behavior. We now formally define these paradigms.
153
154   Post-Execution Self-Debugging. The paradigm leverages information obtained after the actual
155   execution of the program. A widely adopted implementation involves comparing the execution
156   output with the expected output (Olausson et al., 2024; Wang et al., 2024; Dong et al., 2023; Madaan
157   et al., 2023; Zhang et al., 2023a; Chen et al., 2024b; Jiang et al., 2024), as shown in Figure 1.
158   Consider an initial program C and a generated test set {(Xi , Yi )}N       i=1 . An executor, denoted as E,
159   processes each input Xi , yielding the corresponding execution output Ỹi = E(C, Xi ), i ∈ [1, N ].
160   The executor then assesses whether the execution output Ỹi aligns with the expected output Yi to
161   determine if the test is passed. If a discrepancy occurs, the test is marked as failed. The system
      then utilizes the failed test case (Xi , Yi ), the execution output Ỹi , and any related error messages to


                                                          3
      Under review as a conference paper at ICLR 2025


162
      refine the program. This process encourages the model to generate a revised version of the program,
163
      denoted as C̃ = M(C, Xi , Yi , Ỹi ).
164
165
      In-Execution Self-Debugging. Post-execution self-debugging typically overlooks the intermedi-
166
      ate states of the program, which can provide valuable insights for program refinement. To address
167   this limitation, in-execution self-debugging leverages feedback from the intermediate states during
168   program execution (Zhong et al., 2024; Ni et al., 2024; Bouzenia et al., 2023). Formally, a program
169   C can be divided into multiple basic blocks, denoted as C = [B 1 , B 2 , ..., B K ], where B k represents
170   the k-th basic block and K is the total number of blocks in the execution trace. Each basic block is
171   defined as a linear sequence of program statements with a single entry and a single exit point.
172
      Given a test input Xi , i ∈ [1, N ], the executor E initializes the input as the initial variable set
173   Vi1 and executes it through the first block B 1 . The execution updates the variable set to Vi2 =
174   E(B 1 , Vi1 ), where Vi2 denotes the set of variables after executing block B 1 . This process is repeated
175   iteratively, with the executor processing Vik+1 = E(B k , Vik ) for each subsequent block B k until the
176   program execution is complete. The sequence of intermediate states represented as the execution
177   trace T = [B 1 , Vi1 , ..., B K , ViK ], provides a detailed view of how the program behaves over time.
178   By analyzing this trace, the LLM M identifies potential issues within specific blocks and refines the
179   program accordingly, resulting in the updated version C̃ = M(C, Xi , T ).

255   tests on basic programming problems and evaluate how it compares to self-debugging with oracle
256   tests. Consistent with implementations in most existing literature, we perform self-debugging by uti-
257   lizing post-execution information. In this process, program correctness is determined by comparing
258   the actual output with the expected output for a given test case. If the generated program successfully
259   passes all tests, the iterative process terminates, and no further self-debugging is conducted.
260

261   that can be utilized from post-execution results. The first type is the correct label, which indicates
262   whether the model’s previous program was correct or not. If the program is incorrect, an instruction
263   for repair will be provided to the model. The second type is the detail of the failure, including the
264   test input, expected output, and execution output. In cases where the program raises an exception
265   during execution, the error message is incorporated into the detail in place of the execution output.

494   5   D ISCUSSION
495
496   Directions for future work. In this work, we demonstrate that post-execution self-debugging with
497   self-generated tests struggles on basic problems due to biased evaluations, despite the significant
498   potential shown by LLMs in automated test generation. This highlights the necessity for the re-
499
      search community to focus on the quality of LLM-generated tests before utilizing execution feed-
      back derived from them. Developing techniques that enhance high-quality test synthesis is crucial
500
      to mitigate bias for post-execution self-debugging. It could be beneficial to implement an iterative
501
      refinement process wherein execution information is leveraged to improve the tests. This could in-
502   volve using techniques like test-driven development where tests are continuously updated based on
503   code changes and debugging outcomes.
504
505
      As demonstrated in Section 4.4, leveraging enriched runtime information from execution is a promis-
      ing avenue for self-debugging. In particular, in-execution self-debugging has shown superior per-
506
      formance compared to post-execution in certain tasks, suggesting that more nuanced and reliable
507
      feedback leads to better performance. Designing more sophisticated methods for collecting and
508   analyzing runtime information is a promising direction for further enhancing self-debugging capa-
509   bilities. For instance, improving the intelligibility of execution trace representations for LLMs may
510   prove beneficial (Ni et al., 2024). Additionally, beyond variables, other types of runtime information,
511   such as code coverage and execution paths, may also be utilized effectively (Chen et al., 2024a).
512
      Effective self-debugging with self-generated tests hinges on several core capabilities of LLMs. In
513
      terms of refinement, the model should be capable of accurately recognizing and localizing faults
514   within the program. Additionally, more advanced reasoning capabilities are needed to analyze exe-
515   cution feedback thoroughly. The model should comprehend the relationship between the code logic
516   and the feedback, thereby deducing the runtime structure of program statements and variables.
517
518
      Applications. Self-debugging opens up possibilities for developing more advanced LLMs without
519   reliance on human supervision or guidance from stronger models (Burns et al., 2023). Traditionally,
520   human-generated test cases serve as a strong supervisory signal for aligning code generation, but
521   the collection of such tests is labor-intensive, leading to a sparsity of labeled data for effective
522   code refinement. Self-generated tests, by contrast, offer a viable path for self-improvement (Tao
523   et al., 2024). They alleviate the burden of manual test generation and pave the way toward truly
524   autonomous self-correcting code generation systems (Chen et al., 2024b).
525

      6   C ONCLUSION
528
529   This paper investigates the concept of self-debugging in code generation for LLMs, with a focus
530   on leveraging self-generated tests. We establish a structured framework for self-debugging which
531
      is essential for real-world applications where high-quality annotations and human supervision are
      often unavailable. We introduce and formalize two distinct paradigms within the execution-then-
532
      feedback process: post-execution and in-execution self-debugging. Through comprehensive experi-
533
      ments on both basic and competitive programming tasks, our findings highlight the unique strengths
534   and weaknesses. Specifically, we observe that: 1) post-execution self-debugging encounters dif-
535   ficulties in basic tasks; 2) bias from self-generated tests can lead to inconsistency across different
536   levels of problems; and 3) in-execution self-debugging, which leverages intermediate runtime in-
537   formation, consistently outperforms post-execution approach on both basic and competitive tasks,
538   indicating significant potential for future development. Overall, our work provides valuable insights
539   into the mechanics of self-debugging using self-generated tests, paving the way toward more au-
      tonomous and self-evolving code generation systems.

752   the problem. By contrast, in-execution self-debugging leverages test inputs and their corresponding
753   runtime information to assess program correctness. As depicted in Figure 3, this approach enables
