009                                               A BSTRACT
010
011            Mixture of Expert (MoE) models improve the efficiency and scalability of dense
012            language models by routing each token to a small number of experts in each layer
013            of the model. In this paper, we show how an adversary that can arrange for their
014            queries to appear in the same batch of examples as a victim’s queries can exploit
015
               expert-choice routing to the full disclosure of a victim’s prompt. We successfully
               demonstrate the effectiveness of this attack on a two-layered Mixtral model. Our
016
               results show that we can extract the entire prompt using O(Vocabulary size ×
017
               prompt length2 ) queries or a maximum of 100 queries per token in the setting we
018
               consider. Our work is the first of its kind data reconstruction attack that originates
019
               from in a flaw in the model architecture, as opposed to the model parameterization.

022   1   I NTRODUCTION
023
024   In recent years, the Mixture of Experts (MoE) architecture has emerged as a powerful and efficient
025   approach for inference of large-scale machine learning models (Jiang et al., 2024; Shazeer et al.,
026   2017; Aljundi et al., 2017; Eigen et al., 2013; Jordan and Jacobs, 1994; Jacobs et al., 1991), particu-
027
      larly in natural language processing (Shazeer et al., 2017; Fedus et al., 2022; Riquelme et al., 2021;
      Du et al., 2022). This architecture distributes computational load across multiple expert modules,
028
      each specializing in different aspects of the underlying task. In the context of language, this means
029
      dividing the input text into smaller units called ‘tokens’ and assigning them to the expert(s) best
030   suited to handle them.
031
032   However, this specialization can also introduce new vulnerabilities. Prior work from Hayes et al.
033
      (2024) identified a critical architectural flaw in MoE models prone to token dropping. Token drop-
      ping refers to a phenomena that occurs when an expert, already at full capacity, gets assigned addi-
034
      tional tokens; these additional tokens cannot be processed and, as a result, are either routed to other
035
      experts or dropped. Hayes et al. demonstrated that if an adversary can ensure their data is processed
036   in the same batch as another user’s data, they can exploit token dropping to launch denial-of-service
037   attacks, by filling the buffers of experts that the user’s data relies on. The attack of Hayes et al.
038   reduced quality of the model responses to victim queries, causing denial of service.
039
      Building on this foundation, we show that the same vulnerability has far more significant conse-
040
      quences and can be exploited to compromise user privacy. We demonstrate that by carefully crafting
041
      a batch of inputs, an attacker can manipulate the expert buffers within the MoE model, leading to
042   the full disclosure of a victim’s prompt that is included in the same batch. This represents the first
043   data reconstruction attack of its kind that originates in a flaw within the design of the model’s ar-
044   chitecture itself, rather than in its learned parameters. The core principle underpinning this attack
045   is that token dropping introduces a shared information channel; if one user’s data can affect the
046   routing pattern of another user’s data, this introduces an information channel (albeit a low entropy
047   one) that can be used to infer which tokens were in another user’s message. In other words, we
048   show that the MoE architecture that uses Expert Choice Routing (Zhou et al., 2024) is vulnerable
049   to Conditional Adversarial Token-Dropping attack, where an attacker can infer what data a vic-
050
      tim submitted to the model. A high level description of Conditional Adversarial Token-Dropping
      attack is given in Figure 1. Although our findings are limited to a specific choice of token to expert
051
      routing algorithm (Zhou et al., 2024), we hypothesise that other MoE routing strategies that break
052
      the fundamental assumption of independence across a batch of inputs may also be vulnerable.
053

      2.1   M IXTURE OF E XPERTS
100
      The concept of Mixture of Experts (MoE) was first introduced by Jacobs et al. (1991) and Jordan and
101
      Jacobs (1994), but has more recently become a popular tool for efficient inference on transformer
102   based LLMs (Shazeer et al., 2017; Zoph et al., 2022; Renggli et al., 2022; Jiang et al., 2024), primar-
103   ily because it allows the model to selectively activate only a small fraction of the total parameters
104   for any given input. An MoE layer in a large language model (LLM) consists of n expert modules
105   and a gating function, which routes a token to an expert (or subset of the n experts). Each expert
106   module can specialize in different aspects of language tasks (e.g. coding) and the gating function
107   decides which experts are relevant for a particular input. Since only a subset of experts is activated
      for an input, the number of parameters activated is significantly smaller than the overall number of


                                                                         2
      Under review as a conference paper at ICLR 2025


108
      parameters in the LLM, which translates to fewer floating-point operations and faster inference. This
109   in turn allows one to build extremely large networks without a corresponding increase in inference
110   costs. Some of the best performing modern LLMs utilize MoE architectures e.g. Gemini-1.5 (Team,
111   2024) , Mixtral (Jiang et al., 2024), and Grok-1 (xAI, 2023).
112
113
      2.2   V IOLATIONS OF USER PRIVACY
114
115   Although previous work has investigated how user privacy can be compromised in
116   LLMs (Debenedetti et al., 2023; Shen et al., 2024), none have thus far investigated vulnerability
117   of user data privacy due to the underlying model architecture, specifically how input representations
118   can be influenced by other data within the same processing batch. Hayes et al. (2024) demonstrated
119   previously that batch composition can be used by adversaries to exploit MoE routing and to launch
120   denial-of-service attacks; we instead exploit it to leak private user supplied prompts. Note that Ex-
121
      pert Choice Routing considered in this work is one of many different routing strategies that breaks
      the implicit batch independence; we hypothesise that other routing strategies may be similarly vul-
122
      nerable.

127   3.1   T HREAT MODEL
128
129   In this paper, we make the following simplifying assumptions. First, we assume that the adversary
130   has white-box access to the model that uses an MoE with cross-batch Expert Choice Routing strat-
131
      egy (Zhou et al., 2024). This can apply in a setting where a third party is using the base model that
      is available publicly e.g. implementation available through t5x (Roberts et al., 2022). Second, the
132
      adversary can control the placement of its and the user inputs in the batch. Third, the adversary can
133
      query the model repeatedly ensuring that the the user supplied input is consistently in the same batch
134   as its own inputs; the adversary and user inputs are always batched together and sent to the model
135   for processing. At face value, this threat model is unrealistic as the second and the third assumptions
136   require access to the serving infrastructure to control in-batch placement. However, the ability to
137   mount man-in-the-middle attacks over networks is a commonplace assumption in security research.
138   We defer a more detailed discussion of the practicalities of the attack and potential methodological
139   improvements to Section 6.
140
141   3.2   P RIMER ON L ANGUAGE M ODELS AND M IXTURE OF E XPERTS
142
143   A transformer based large language model is a function fθ : V l → P(V ) that takes as input a
144   sequence of tokens from a vocabulary V and outputs a probability distribution over the vocabulary,
145   P(V ). In particular, we are interested in functions of the form fθ (z) = softmax(W · hθ (z)), where
146
      W is an unembedding matrix and W · hθ (z) gives a set of logits over V.
147   We assume that the model hθ consists of multiple MoE layers. A MoE layer consists of n expert
148   functions {e1 , e2 , . . . , en } where ei : Rd → Rd is a feed forward layer that takes in d-dimensional
149   token representations and outputs new features of the same dimensionality. The MoE layer also
150   consists of a gating function g : Rd → Rn which is used to assign token representations to experts
151   by outputting a probability distribution over the n experts.
152   Large language models are commonly ran on batches of inputs to improve hardware utilization and
153   efficiency. This means fθ in reality operates on the domain V l×b , where l is the sequence length
154   of an input, and b is the batch size. For models that do not use MoE layers, the computation is
155   entirely parallel over a batch of inputs; the computations of one input in the batch cannot affect the
156   computations of another input in the batch. For models that do use MoE layers, this is no longer
157   true, as the gating function g can only assign a limited number of token representations from a batch
158
      to a specific expert. There are many different choices for how to assign tokens to experts given the
      output of the gating function; this is also commonly known as the routing strategy (Cai et al., 2024).
159
160   In this work, we focus on Expert Choice Routing which allows each expert to independently select
161   its topk assigned tokens from a batch of tokens (Zhou et al., 2024). The value k represents the
      fixed capacity of each expert, signifying the number of tokens it can process; we refer to this as

192   4    C ONDITIONAL A DVERSARIAL T OKEN -D ROPPING ATTACK
193
194   This section details our two exploits against models that use MoE architectures with the Expert
195   Choice Routing strategy. Both exploits leverage the primitives explained in Section 4.1 to compro-
196   mise user message privacy. The attacks are:
197
            1. Oracle attack: The attacker guesses the victim’s prompt and verifies it with only two
198
               queries to the model. (Described in detail below)
199
200         2. Leakage attack: The attacker sequentially executes the oracle attack to extract the victim’s
201
               prompt without prior knowledge.
202
      We successfully demonstrate the leakage attack on a two-layered Mixtral model (Jiang et al., 2024)
203
      with Expert Choice Routing strategy (results are in Section 5, by two-layer we mean we only use
204   the first two layers of a larger model).
205
206   At a high level, the attacks are based on the ability to create dedicated adversarial inputs (Figure
207
      2) that when included in a batch, intentionally shape what is included in an expert’s buffer in the
      first MoE layer (Figure 3) in such a way that information about user prompts within the batch leaks.
208
      In particular, the adversary finds a set of tokens, which we refer to as blocking tokens, that have
209
      high priority of being routed to an expert. The adversary then guesses the target token that they
210   are trying to leak from the victim input, and uses the blocking tokens to ensure that this (guessed)
211   token is at the boundary of the expert buffer (i.e. has the k th priority). The reader will recall from
212   Section 3.1 that the adversary can control the position of the victim’s message within the batch,
213   and so creates an adversarial batch that consists of blocking tokens, the guessed message and the
214   victim’s message (the message they are trying to infer). The adversary then sends two queries: one
215   where the adversarial batch contains the victim’s input in a batch position before the guessed input
      token, and one where the the guessed input token is before the victim’s input. If the victim and


                                                           4
      Under review as a conference paper at ICLR 2025


216
      guessed inputs are identical, then the model output on these two queries will be different, as either
217   the victim or guessed message target token will be dropped depending on their order within the
218   batch. Importantly, the guessed message tokens that are processed will be different in each query (in
219   one query the guessed token is dropped and in another it is not), which causes a difference in model
220   outputs. By equating difference in outputs to token dropping, the adversary can infer if their guess
221   was correct.
222
      We concentrate on the description of the oracle attack, as the token-by-token leakage attack is simply
223   a sequential execution of the oracle attack. Next, we give a more detailed overview of the attack
224   along with the primitives we make use of for execution.
225
226   4.1     ATTACK OVERVIEW
227
228   The goal of the attack is to force a scenario in which an attacker’s token is conditionally dropped
229   depending on the relative order in the same batch of an identical token belonging to the victim. We
230   discuss the attack in more detail in Appendix E, but at a high level the exploit is triggered by the
231   following steps:
232
              1. Step 1: Guess the next Token and Position: The attacker guesses the target token and its
233
                 position in a chosen expert’s buffer, assuming the prefix is known (initially empty).
234
235           2. Step 2: Construct the Adversarial Batch: Using the primitives described in Section 4.2,
236
                 the attacker crafts an adversarial batch that:
237               (a) Places blocking tokens to fill the expert buffer, leaving one spot for the guessed token.
238               (b) Includes the probe sequence with the known prefix and guessed token, with the goal
239                   of triggering tie-handling.
240               (c) Adds a padding sequence to extend the buffer size and ensure stable sorting.
241           3. Step 3: Send Two Queries: The attacker sends the adversarial batch twice, changing the
242              order of the victim’s message and the probe sequence.
243
              4. Step 4: Map Observed Logits to Routing Paths: The attacker uses a local model to find
244
                 a mapping between observed logits to routing paths of the probe sequence.
245
246
              5. Step 5: Verify the Guess: A correct guess is indicated if the guessed target token is routed
                 to the chosen expert in the first query (where the probe sequence comes first) but not in the
247
                 second query (where the victim’s message comes first). This difference in routing paths
248
                 arises because the input order of identical tokens influences the tie-breaking behavior. If
249              the guess is incorrect, the routing paths should be consistent regardless of the input order.
250
251
      4.2     ATTACK PRIMITIVES
252
253   We now describe the primitives behind each of the attack steps.
254
255   4.2.1     H ANDLING TOKEN PRIORITY TIES IN EXPERT ROUTING
256
257   In Expert Choice Routing, each expert buffer serves as a priority queue. As discussed in Section 3.2,
258   priorities are given to each token by way of the matrix G of shape (t, n), where t is the total number
259
      of tokens in the batch and n is the total number of experts. With token dropping the topk most
      prioritized tokens in each column of G will be processed by the corresponding expert and the rest
260
      will be dropped. If two tokens have identical priority (they have the same probability of assignment
261
      to an expert) and they have priority k, then which of the two tokens is selected and routed to the
262   expert is entirely down to the tie-handling behavior of the topk algorithm implementation. This is
263   a feature (further discussed in Appendix B) in the CUDA implementation of topk which is based
264   on a stable sort, with a “first come first serve” tie breaking policy. This distinguishable behavior
265   which is a function of the relative order to tokens within a batch is what we exploit in Conditional
266   Adversarial Token-Dropping.
267
268   4.3     E XTENDING AN EXPERT ’ S BUFFER SIZE
269
      Extending the expert’s buffer size, k, is crucial to the attack for two reasons:


                                                          5

             • To make sure the victim’s token are not dropped by default.
271
272          • The expert buffer capacity needs to be > 32 for predictable behavior in CUDA topk tie
273            breaking (see Appendix B).
274
275   The expert buffer capacity is given in Equation (1).
276   In essence, it is the expected number of tokens per expert factored by the capacity factor c, which
277   usually sits in the range of [0.6, 2] (Gale et al., 2023). By sending a long query the total number
278   of tokens will increase, the other inputs will be padded to match the sequence length l and thus the
279   effective size (or space) in each expert buffer will increase, as padding tokens are masked to achieve
280   the lowest priority possible.
281
282   4.4   C ONTROLLING TARGET- TOKEN PLACEMENT WITH BLOCKER SEQUENCES
283
284   Each token is assigned a priority (distribution over experts) for each expert; tokens with lower pri-
285   ority are pushed towards the end of the expert buffer, or dropped altogether if their priority is larger
286   than k. We leverage local access to the model weights to pre-compute a sequence per expert with to-
287   kens that are assigned high-priority in the first MoE layer. We refer to this as a blocking sequence, as
288   these tokens block other tokens from being assigned to an expert because of a higher priority. Since
289   our goal is to place the target token at the edge of the expert buffer we need to guess its position in
290
      an expert buffer. We will then use our pre-computed blocker sequences to precisely set the correct
      number of blocker tokens needed within the adversarial batch. We further discuss the details of the
291
      approach we took for finding blockers in Appendix C.
292
293   Notice that although target token can be positioned in any position in an expert’s buffer, the attacker
294   controls all of the tokens in the batch except for the victim’s. Therefore the position search-space is
295   bounded to the victim’s message length.

346   4.5.1   D O WE REALLY NEED TOKEN ROUTING PATH MAPS ?
347
348   Initially, one may question if computing token routing paths is necessary. We now discuss why this
349   is crucial to the attack for models with more than one MoE layer.
350   In a single layer transformer model that utilizes MoE, the layers of the model are: (1) embedding,
351   (2) attention, (3) MoE, (4) unembedding. Importantly, the attention layer does not affect the repre-
352   sentation of the target token after token collisions due to priority ties in the MoE layer. A collision
353   would have an immediate impact on the model output (logits). The attack is performed sequentially,
354   token-by-token, the attacker needs to correctly guess the target token, and its position in a chosen
355   expert. Any change to the logit output will then indicate this token was affected by token dropping.
356   For a two layer model, the layers are: (1) embedding, (2) attention, (3) MoE, (4) attention, (5) MoE,
357   (6) unembedding. The second attention layer does affect the representation of the target token after
358   the token collisions due to priority ties in the MoE layer. We therefore present an attack that handles
359   the effect of attention that may interfere with our ability to understand how and when tokens are
360   dropped. The second layer of attention is impacted by MoE token dropping, and that means that the
361   prefix could also be dropped due to the impact of the adversarial batch and the unknown suffix of
362   the victim message. Prefix tokens will also change the representation of the target token and thus
363   its routing path through the second MoE layer, and therefore a method to recover the target token
364
      routing path given the model output is necessary.

459   6   D ISCUSSION
460
461   Methodological improvements In this paper we show that it is possible to exploit Expert Choice
462   Routing to extract private victim data placed in the same batch as an adversary’s data. Conditional
463   Adversarial Token-Dropping currently requires 2 queries per guess for verification, and 2n×L per
464   token queries for general extraction. This makes it infeasible at present to use our attack against
465
      real world systems. Yet, we believe that performance of the attack could be significantly improved.
      Firstly, we hypothesize that refining the buffer shaping process could enable the selection of blockers
466
      that prevent inter-batch token interference. We discuss this further in Appendix C. Secondly, we
467
      suspect that an alternative approach exists to determine the processed token without exhaustively
468   constructing all possible 2n×L expert combinations, potentially by learning a mapping between
469   outputs and routing paths. We discuss this further in Appendix D. Thirdly, targeting the final MoE
470   layer instead of the first may eliminate the need for routing path tracking altogether. Fourth, the
471   current attack requires precise matching in its exploitation for tie-handling. We hypothesise that
472   relative placement of tokens can similarly be used to signal what victim token is used. Finally, we
473   believe that a black-box variant of this attack is feasible – a local clone of the model at present is
474   only used to find blocking sequences and for inverting token routing paths. We hypothesise that both
475   of the tasks could inefficiently be deduced from black-box access.
476   Optimisation–Security trade-off Within the domain of computer security, it is well-established that
477   prioritizing performance optimization often inadvertently introduces vulnerabilities to side-channel
478   attacks (Anderson, 2010). In the case of MoE models, the Expert Choice Routing strategy, which
479   optimizes for efficiency, inadvertently creates a side channel that allows the attacker to exploit the
480   model. Our work highlights the importance of rigorous adversarial testing of any optimization
481   introduced into machine learning pipelines to safeguard user privacy. While we focus on a specific
482
      routing strategy, we anticipate that similar vulnerabilities may exist in other strategies that violate
      implicit batch independence.
483
484   Defences Having established general vulnerability of MoE-based models with Expert Choice Rout-
485   ing, we now shift our focus to potential defense strategies. A crucial first step in mitigating these
      vulnerabilities is to preserve in-batch data independence, particularly across different users. This en-


                                                                  9
      Under review as a conference paper at ICLR 2025


486
      sures that adversaries cannot exploit the routing strategy. Second, the current attack design requires
487   precise shaping of the expert buffer, therefore, introducing any form of non-determinism into the
488   system can effectively disrupt the attacker’s ability to exploit this vulnerability. This could involve
489   incorporating randomness into various aspects of the model, such as the expert capacity factor, the
490   batch order, the input itself, or the routing strategy.

      7   C ONCLUSION
494
495
      In classical dense LLMs, it is essentially impossible for one user’s data to impact another user’s
      output. But MoE models introduce a side-channel: one user’s queries can impact a different user’s
496
      outputs. The magntidue of this leak is minuscule challenging to detect. But by carefully crafting
497
      adversarial input batches, we show how to manipulate the expert buffers within the MoE model
498   with Expert Choice Routing, leading to the full disclosure of a victim’s prompt included in the same
499   batch. At present, Conditional Adversarial Token-Dropping is only possible when Expert Choice
500   Routing is used, yet we hypothesise that other routing strategies can be similarly vulnerable. While
501   the current threat model assumes unrealistic attacker capabilities, we believe that future research can
502   extend the practicality of these attacks.
503
      More broadly, attacks such as this highlight the importance of system-level security analysis at all
504   stages of model deployment, starting as the design of the architecture, and extending towards as late
505   as the actual deployment of the model and how different user queries are batched together. Studying
506   any one component in isolation may give the appearance of safety, but only when the system as a
507   whole is analyzed is it possible to understand vulnerabilities such as this. We hope that future work
508   will perform other analysis of this type on future advances.

      model we expect to observe a deviation in the model outputs of our probe sequence, depending on
748

749

           It is therefore important to devise a method that can trace the deviations in model outputs to the
757        expected token-dropping the attacker is aiming to induce. Our approach is robust but conservative;
758        it is computationally slow and is the main bottleneck of our attack and the reason we didn’t scale
759        the attack to more than two layer models. We first estimate the routing path of the prefix tokens
760        and store their attention activations using KV-caching by sending our adversarial batch without the
761        guessed token.
762

763        with our target token and the previously computed KV-cache. Our local model supports disabling
764        experts based on a given routing path B, a binary matrix of shape n × L. We then store an expert
765        allocation routing path table mapping each model output to its corresponding routing path bitmap,
766        where each bij ∈ B is equal to one if token j is routed to expert ei , and zero otherwise.
767

768
           to our stored table and recover the token routing path. We expect to see a special distinguishable
769        behavior in which the routing path of the probe sequence of the first adversarial batch query is not

771        keys up to some Lp distance to account for some of the floating point errors.

826   could locally compute the priority pj of the guess appearing in this expert’s buffer. With that we

830   we could precisely place the guess token on the edge of the buffer, and if the guess is correct it

832   function of the order of the inputs in the batch. We can then map the different logits to a routing path
833   of expert assignment, and with that infer whether a dropout from the correct expert has occurred.

838   and a stable sort for long (32+) sequences. Hence, the attacker batch should be comprised of three
839   different inputs: (1) probe sequence: prefix + guess; (2) blockers that are used to block K positions
840   on some target expert; finally, (3) padding sequence – some long sequence that forces the expert
841   capacity to be ≥ 32 to ensure sorting stability.
842
      Second, careful management is required given that token placement needs to be precise. In paratic-
843
      ular, the adversary needs to take into account the expert capacity, the blockage from the prefix,
844   blockage from the the blockers we get from the probe sequence. We launch the attack by the adver-
845   sary searching for the blockers using a local copy of the model, recursively assuming some random
846   user input.
847
      Third, when dealing with more than one layer, we should take into account the possibility that the
848
      prefix tokens were dropped in previous layers, and thus even if we know the routing path of the
849
      guessed token, its’ logits can change as a function of the prefix droppings in prior layers. Since we
850   know the prefix, we can use our local model known upfront the prefix path. We need to send our
851   adversarial batch to a local model, assume it is not drastically affected by the user input, and store
852   the key-values cache for this execution. Then we send our guess tokens assuming different possible
853   further expert token dropping combinations. With L >> 3-layers it seems completely infeasible
854   – each layer out of L layers has n experts, and with Expert Choice Routing all 2n assignments are
855   possible. Thus each token ∈ V would have at most 2Ln routing paths. By mapping our guessed
856   tokens logits to the experts-path it went through, we can later observe the logits and infer if this
857   logits comes from a routing path that include the target expert or not. We look for a change in inputs
858
      order, that would affect the guessed token routing path. We use a hooked local model, that allows
      us to disable experts given some token dropping routing path, we then bruteforce for all possible
859
      routing paths and store aside the logits to expert assignment mapping.
