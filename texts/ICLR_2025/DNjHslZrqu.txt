009                                                A BSTRACT
010
011            Many real-world applications of tabular data involve using historic events to predict
012            properties of new ones, for example whether a credit card transaction is fraudulent
013            or what rating a customer will assign a product on a retail platform. Existing
014            approaches to event prediction include costly, brittle, and application-dependent
015            techniques such as time-aware positional embeddings, learned row and field en-
016            codings, and oversampling methods for addressing class imbalance. Moreover,
017            these approaches often assume specific use-cases, for example that we know the
018
               labels of all historic events or that we only predict a pre-specified label and not
               the data’s features themselves. In this work, we propose a simple but flexible
019
               baseline using standard autoregressive LLM-style transformers with elementary
020
               positional embeddings and a causal language modeling objective. Our baseline
021            outperforms existing approaches across popular datasets and can be employed for
022            various use-cases. We demonstrate that the same model can predict labels, impute
023            missing values, or model event sequences.

      1    I NTRODUCTION
027
028
      What does the past tell us about the present? What will happen in the future? These are fundamental
029
      questions pervasive in many scientific domains. While generative models have shown remarkable
030   performance in language, vision, audio and even graphs, the study of predicting events remains
031   underexplored. Event data consists of sequences of discrete samples that arrive at sporadic and
032   uneven intervals. These elements can be univariate, but can also contain multiple features describing
033   the event, or collected concurrently with the event.
034
      Event data shares properties with other modalities, but it is unique in that the ordering and timing of
035
      events may contain important information. Unlike time-series data, which is commonly represented
036   as uniformly spaced measurements, the time between events is important, and these intervals can
037   vary drastically within a single event sequence. Additionally, event data shares many of the same
038   challenges as tabular data, including heterogeneous fields (columns) that often span diverse data types
039   (Luetto et al., 2023). However, unlike traditional tabular data, the ordering and timing of samples
040   in event data is a primary consideration in prediction tasks. Event data can be viewed as a special
041   case of sequential data (e.g. language) or as a special case of tabular data. However, event prediction
042   models are often used to forecast not only properties of future events, but also when they will occur.
043   Therefore it is not sufficient to use existing tabular or time-series specific models for event prediction;
044
      event stream data should be studied separately.
045   Recent methods have attempted to model event data by adapting tabular learning methods, like
046   boosted trees, to handle additional features containing temporal information (Kang & McAuley,
047   2018). However, recent advances in deep learning models for tabular data (Padhi et al., 2021; Luetto
048   et al., 2023; Gorishniy et al., 2022; Somepalli et al., 2021; Zhang et al., 2023) have enabled researchers
049   to add contextual information to tables, including temporal information.
050   While these approaches have demonstrated that sequential tabular data can be modeled using deep
051   learning and that neural networks can outperform non-parametric models, the models in this sparsely
052   explored field are often overly complex, in both architecture and training routine. Further, existing
053   deep learning approaches are trained specifically for a particular prediction task and must be finetuned
      for new tasks on the same dataset, for example predicting a new event attribute.

086   In this work, we bring recent ideas in casual language modeling to bear for processing sequential
087   tabular data with a simplified pipeline. Our Simple Transformer for Event Prediction (STEP)
088   is a decoder-only auto-regressive model that surpasses state-of-the-art methods on popular event
089
      prediction datasets. We depart from prior methods in the following key ways:
090
             • Our approach converts event data into a sequence of tokens to be trained with a causal
091
               language modeling loss. This makes our approach relatively simple compared to hierarchical
092
               Masked Language Modeling (MLM) pipelines specially tailored to event data. We observe
093            that causal models outperform MLM models on a range of event prediction tasks.
094
095
             • We find that causal models perform best when using standard positional encoding to represent
               event ordering. We depart from existing methods (Zhang et al., 2023) that use specialized
096
               positional encoding for temporal information; instead we represent time as a feature.
097
098          • We introduce two data augmentations during training that enable STEP to perform useful
099
               event prediction tasks without complicating the training pipeline and architecture: (1) We
               shuffle the columns within a event, allowing the model to infer unknown features of an event
100
               conditioned on any subset of observed features, a task usually unique to MLM models. (2)
101
               We mask labels from previous events, reflecting the fact that we may not know the labels of
102            previous events while predicting the label of a new event.

105   2   R ELATED W ORK
106
107   Structured data is commonly found in tabular form, with each row representing a sample and each
      column representing a feature. This type of data is pervasive in industrial settings but only recently


                                                        2
108
      has become of interest to the deep learning community. Traditional and still predominant methods for
109   supervised and semi-supervised learning on tabular datasets involve non-parametric tree-based models
110   like XGBoost (Chen & Guestrin, 2016), CatBoost (Prokhorenkova et al., 2019), and LightGBM (Ke
111   et al., 2017). These models are highly competitive in performance metrics (e.g., accuracy, AUC) and
112   offer additional benefits such as interpretability and effectiveness in both high and low data regimes.
113
      One disadvantage of tree-based models is that they do not explicitly model inter-row dependencies,
114
      which are common in event data where rows represent sequential events, such as customer transactions
115   over time. Attention-based “tabular transformers” have been proposed recently to address this
116   disadvantage. These transformers differ in (1) the way that they employ the attention mechanism,
117   (2) the type of positional encoding used, (3) the encoding of numerical features, and (4) the training
118   regime (masked language modeling vs. causal language modeling). For example, two early works,
119   TabNet (Arik & Pfister, 2019) and TabTransformer (Huang et al., 2020), use field-level (intra-row)
120   attention whereas SAINT (Somepalli et al., 2021), TabBERT (Padhi et al., 2021), FATA-Trans (Zhang
121   et al., 2023) and UniTTab (Luetto et al., 2023) employ both intra-row and inter-row attention which
122   enables them to deal with sequential data where rows can be dependent on one another. Most of the
123
      tabular transformers use a masked language modeling approach whereas TabGPT (Padhi et al., 2021)
      uses a causal language modeling approach for generating synthetic data by predicting future events
124
      from an ordered sequence of 10 historical events.
125
126   Prior work has explored ways to handle specialized features present in event data, such as temporal
127   and numeric features. In language modeling, positional encoding is used to model the order of the
128   tokens. While most tabular models (e.g., TabBERT (Padhi et al., 2021), UniTTab (Luetto et al., 2023))
129
      use a similar technique to capture the order of the events, FATA-Trans uses a time-aware positional
      embedding mechanism to also capture the non-uniform time interval between events. To address
130
      numeric features, researchers have used both a quantization approach where continuous values are
131
      discretized into bins (e.g., TabBERT) as well as learning a dense representation using a neural net
132   (e.g., FATA-Trans, SAINT). UniTTab (Luetto et al., 2023) represents numerical values as feature
133   vectors derived from various frequency functions. Azabou et al. (2023) model events (specifically
134   neural brain activity) using a novel tokenization scheme combined with a pretrained model that has
135   been finetuned to predict future neural activity.
136
      Transformer-based approaches have been used to improve performance on time series tasks.
137
      PatchTST (Nie et al., 2023) subdivides time series into overlapping patches, treating each patch as a
138   token for the transformer, with multivariate series handled as independent univariate series sharing
139   the same embeddings. iTransformer (Liu et al., 2024) embeds entire time series of each variate as
140   tokens, applying self-attention and feed-forward networks for series representation. TimesFM (Das
141   et al., 2024) uses patches and MLP blocks with residual connections for tokenization, while Gruver
142   et al. (2023) argue that foundation models like GPT-3 can serve as zero-shot forecasters by tokenizing
143   data as numerical strings.
144   Temporal Point Processes (TPPs) have become a standard approach for modeling event sequences
145   in continuous time. EasyTPP (Xue et al., 2024) offers an open benchmark with various datasets
146   and neural TPP algorithms (e.g., (Shchur et al., 2020; Yang et al., 2022; Zhang et al., 2020)) for
147   comparative evaluation. MOTOR (Steinberg et al., 2023) is a foundational time-to-event model
148   designed for structured medical records. Additionally, EventStreamGPT (ESGPT) (McDermott et al.,
149   2023) is an open-source toolkit specifically for modeling event stream data. Importantly, none of
150   these methods are decoder-only, as each encodes the events before passing the learned embeddings to
151   a model (generally a decoder) for prediction.

156   3.1   P ROBLEM D EFINITION
157
158   We consider event data, represented as tabular data with a temporal feature and arranged sequentially
159   such that the rows are first grouped by a meta-feature (such as user) and then organized by time.
160   Because our goal is to predict a user’s future events based on their previous events, it is imperative
161   that we separate the data by user so that events in a sequence correspond to precisely this task. We
      define the data as follows:

166   a meta-feature (e.g. a user ID denoted f0meta ) and at least one is a temporal feature (i.e. a timestamp
167   denoted f1time ). Meta-features are columns that are not used as model inputs during training, but
168   instead identify which rows should be grouped together (e.g. rows belonging to the same user
169

170

173   where each xj in the sequence has the same value for f0meta .
174
      While it is natural for the target label to be oriented as the last feature of each event (i.e. fk ), this is
175
      not a strict requirement. Similarly, for a given dataset, the target label location may vary from task to
176   task. Therefore, the model should be capable of accepting each record with the features enumerated
177   in any order. This functionality provides two primary benefits. Firstly, the model can be evaluated
178   with any of the columns as the target label. Specifically, any feature of xj ([f1time , f2 , ..., fk ]), can
179   be designated as the target feature during evaluation. Secondly, when predicting the label for the
180   target event, the model can be evaluated with any number of the input features (and in any order).
181   For example, if the target label is fk , any number of [f1time , f2 , ..., fk − 1] can be passed in before
182   querying the model to predict fk .

184   3.2   STEP: A S IMPLE T RANSFORMER FOR E VENT P REDICTION
185
186
      We introduce STEP: a Simple Transformer for Event Prediction. Like the above methods, we use
      a transformer-based architecture to learn a rich semantic representation of the data, however our
187
      construction is decoder-only. Decoder-only transformers have demonstrated a remarkable ability to
188
      model sequences of tokens – we rely on that ability to model the sequential relationship between
189   different records within a sequence of events.
190
191   In order utilize the causal masking present in these transformers, we first convert event data into the
192
      language space, by unrolling records of event data into sequences and tokenizing the resulting rows
      before passing the sequences into the model. As with prior methods, we quantize numeric data and
193
      arrange the data into sequences by user. Because we treat our tabular data like language, we arrange
194
      our data table in a manner similar to common language modeling setups:
195
196         1. We tokenize each feature separately by column to ensure that entries with identical string
197            values have differing semantic meanings depending on the source column. For example,
198            if there are multiple binary columns in the data table, we want to differentiate between
199            true/false in each column. Numeric features are tokenized by quantizing the values into
200            32 bins (separated by quantile). The entries are not sub-word tokenized into word pieces
201
               as with common language tokenizers but instead are treated as atomic tokens similar to the
               word level tokenization algorithm (i.e. split by column). The result is that each processed
202
               event (xj ) with k + 1 features is tokenized into exactly k + 1 tokens such that
203
204                                 xj = [f0meta , f1time , f2 , ..., fk ] ⇒ [tmeta time
                                                                               0 , t1 , t2 , ..., tk ]
205            where each ti is the tokenized version of fi .
206

207

208                                 Xn = [x1 , ...xl ] = [f1,1  , f1,2 , ..., f1,k , ...f2,1  ...fl,k ]
209            For sequences with less than l events, samples are right-padded with 0s so that each sequence
210            Xn is a single sample with exactly k ∗ l tokens that are ordered temporally and share a meta
211            feature.
212         3. We use the same absolute positional encoding that was used for GPT, another decoder
213            transformer (Radford et al., 2019). Therefore, since we do not designate the temporal
214            feature time with special embedding but rather as a normal feature, there can be more than
215            one temporal feature in the table, but there should be a time-ordered way of arranging the
               records.

238   3.3   DATA P REPROCESSING
239
      A key step in our training pipeline is converting event data into text tokens to train our decoder-only
240
      transformer. We transform the columns of the table by respective data type: categorical values are kept
241
      as is, but numeric values are bucketed into 32 distinct buckets per column. Then we pretokenize the
242   data by splitting splitting on column (rather than whitespace or digits as is common with traditional
243   LLM pretokenization schemes). We train a word-level tokenizer on the data with a maximum
244   vocabulary size of 60, 000 words. In contrast to Byte-Pair Encoding, word-level tokenization is a
245   top down approach to building a vocabulary which simply takes the words in the corpus (in this case
246   each individual cell is a “word”) and uses the most common of these words as the vocabulary. While
247   it would be possible to further split the cells into tokens for the model to learn information about the
248   actual string values within the cell, we choose word-level tokenization as it ensure that each event
249   (and therefore sequence of events) has the exact same number of tokens. Additionally, we tokenize
250
      the data such that each column in the table has a non-overlapping subset of the vocabulary to ensure
      that words that have the same values are perceived differently by the model (i.e. True has a different
251
      meaning in column IsFraud vs. column IsOnline). Each sequence is composed of l consecutive
252
      records by the same user where l is a dataset specific hyperparameter. Each record is separated by a
253   token that delineates new events within the sequence.
254
255   In order to capture the temporal nature of each event, we replace the absolute timestamps with “time
256
      since last” to capture the relative time between each record. This column is treated as a normal feature
      (unlike Zhang et al. (2023) that uses this information as a special positional encoding).

268
269   The data preprocessing required for STEP is relatively simple compared to traditional tabular pro-
      cessing pipelines – we employ an out-of-the-box word-level tokenizer and a standard causal language


                                                                   5
270
      model training setup. STEP uses the same architecture in each setting except for the context length,
271   which is a function of the number of events per sequence and the number of features per event.
272   Additionally, because we use a word-level tokenizer the maximum vocab size is 60000, but the actual
273   vocab size can be much less and that directly affects the size of the embedding layer. For all datasets,
274   STEP is trained using a next-token-prediction objective but is not finetuned on some downstream task.
275   Rather, STEP has the flexibility to be used to for downstream tasks without the need for finetuning
276   depending on which information is masked when passed to the model. Despite its lack of bells and
277   whistles, we demonstrate the performance and flexibility of STEP compares favorably to prior work.
278   See table 2 in A.4 for a full list of hyperparameters for our experiments.

292

      4.2     BASELINE COMPARISONS
328
329   We consider the work of TabFormer (Padhi et al., 2021), UniTTab (Luetto et al., 2023), and FATA-
330   Trans (Zhang et al., 2023). Padhi et al. (2021) present TabFormer, which introduces the concept of
331   using field transformers to learn row embeddings, which can be used in finetuning a model to predict
332   the final label in a sequence. Additionally, Padhi et al. (2021) produces the widely used synthetic
333   credit card transaction dataset by using a rule-based generator.
334   Luetto et al. (2023) introduce UniTTab, includes an additional linear layer between the field trans-
335   former and the sequence transformer which handles datasets with multiple types of events. Addition-
336   ally, they employ a combinatorial timestamp and label smoothing to improve performance.
337
      FATA-Trans (Zhang et al., 2023) uses a similar field transformer to encode each row, but includes
338
      time-aware positional embeddings, and field-aware positional embeddings to distinguish between
339   static and dynamic features. They do extensive pretraining and demonstrate multiple ablations.
340
341   Each of these methods introduces performance improvements at the expense of increased complexity
342
      and rigidity. These sophisticated embeddings and training techniques marginally improve perfor-
      mance on this class of data but lack the flexibility to adapt to new prediction or modeling tasks on the
343
      fly. On the other hand, decoder-only foundation models are known to be adaptable to a diverse set of
344
      tabular data tasks (Gruver et al., 2023) in a zero shot setting despite tabular data containing a unique
345   structure that differs from natural language. However, despite their flexibility, autogregressive LLMs
346   underperform task specific models, but their ability to adapt to this setting serves as motivation that
347   applying causal models to event data is a natural area for exploration. We will see that this simple
348   approach can actually outperform highly engineered existing methods as well as current state-of-the
349   art foundation models in a zero shot setting.
350
      Additionally, we compare STEP against Llama-3-8B-Instruct (AI@Meta, 2024), which we
351   access via the Huggingface API. We convert the table into text of the form <header1>:<value1>
352   <header2>:<value2>...[ROW] and prompt the model for the last label in each sequence,
353   restricting its output to only valid tokens. Results are included in table 1 (see A.2 for more details).

355   4.3     E XPERIMENT VARIATIONS
356
357   We now define our experimental setup. For each experiment, we train models with 5 different random
358   seeds. For all datasets, we reserve 2% of each dataset as the test set, leaving 98% of the data to be
359   used for training. We do this for each training run, meaning that the results are averaged over the 5
360
      different evaluation splits. During training, we optionally mask some information about labels related
      to our desired task. For the credit card, Czech loan, and churn datasets, we mask the labels for all
361
      events as we likely do not know the labels of previous transactions ahead of time. For example, we
362
      do not know if previous transactions in the sequence are fraudulent when the model predicts if the
363   final transaction is fraudulent. For amazon reviews, scores are likely immediately accessible after
364   giving a review so we do not mask previous scores in the user sequence.

366   4.3.1    P REDICTING THE LAST LABEL IN A SEQUENCE
367
368   First, we show that in the setting considered by prior work (predicting the last label in the sequence),
369   STEP outperforms the benchmarks. Events are considered in non-overlapping sequences. In this
370
      setting, the model is tasked with predicting the label for the final event in each sequence.
371
      4.3.2    R ANDOMIZING FEATURES WITHIN EACH EVENT
372
373   Because the features of an event all “occur” at the same time, we want to remove the causal bias
374   inherent in decoder-only models. That is, for a given event, we want to break the intra-row dependency
375   of the features from column 1 preceding column 2, column 2 preceding column 3, etc. The affect of
376   this causal bias has been explored in the literature (Uria et al., 2014; Germain et al., 2015). In order
377   to break this dependency, we train step with the ordering of features within a given event randomized
      so that the model learns not to expect a specific ordering of columns.


                                                         7
378
      Prior work has primarily utilized MLM objectives, which are able to mask features within the input
379   rather than just at the end. We achieve similar behavior by randomizing the order of the features
380   within an event during training, which allows the model to predict any subset of a row’s features
381   using any of other features in addition to any number of prior transactions in the sequence.
382
383   4.3.3     H ANDLING MISSING OR PARTIAL INFORMATION
384
385   A natural consequence of the model’s ability to handle events with the features occurring in any order
386   is that there is no set “label” during training – at evaluation time any feature can be predicted. Further,
387
      predictions can be made with any subset of features by simply passing in the known features for an
      event and predicting the missing ones. Similarly, while the model is trained with a fixed number
388
      of events in its context, it can make predictions with fewer events seen. We explore the trade-off
389
      between model performance and the amount of information seen by STEP.

        6               D ISCUSSION AND O UTSTANDING C HALLENGES FOR S EQUENTIAL
466                     TABULAR DATA
467
468    The baseline we propose in this work is flexible and can accommodate a variety of use-cases, but our
469    work leaves several open problems:
470    Multiple event types. The datasets we use have the property that all events possess the same features.
471    In many real-world applications, events may be of various types, each with their own features.
472    For instance, a single user’s history may include credit card payments, bill payments, and ATM
473    transactions. Building models that can ingest and make predictions on data with multiple event types
474    may require appropriate architectures or positional embeddings as well as training procedures. One
475    benefit of our decoder only method is that it should be able to easily handle heterogeneous event data
476
       by simply adding special tokens indicating data source.
477    Long context lengths. In our experiments, we limit the number of tokens in a model’s context by
478    limiting the number of events. Other tasks may require loading many more events or features into
479    the context. A disadvantage of the transformer architecture we employ is that the cost of attention
480    increases quadratically in the context length. Future works may consider modifying the architecture
481    to accelerate inference over long context lengths, for example straight-forwardly adopting existing
482
       tools from the language modeling literature or engineering new tools to select a subset of features per
       event or to select a subset of historic events that are useful for making predictions.
483
484    Formatting model inputs. One way to reduce the number of tokens in the context is by converting
485    each event into a single token embedding, for example using a small embedding network, thus
       dividing the context length by the number of features per event. However, such techniques may


                                                          9
486
      entail either different formats for input and output or may require predicting all attributes of an event
487   simultaneously. If we naively predict an entire event simultaneously, we may not be able to compute
488   joint probabilities of event attributes, which are required for some applications, such as modeling
489   future events. For example, one approach might involve first embedding each event into a single
490   token embedding using an embedding network, predicting the embedding corresponding to the next
491   event, and then decoding that embedding autoregressively into the associated column entries. This
492   approach would simultaneously reduce the context length while still enabling us to compute joint
493   probabilities, but on the other hand would be more complexity with additional moving parts.
494   Diverse benchmark datasets. We focus on several popular datasets in our experiments, but appli-
495   cations of sequential tabular data are wide ranging, spanning scientific applications, finance, retail,
496   recommendation, and more. An impactful area for future work is to collect diverse datasets and
497   compile them into a systematic benchmark to improve the rigor of method comparisons. A future
498   benchmark might also include datasets with multiple event types as described above.
499   Limitations. A major limitation of our experimental setup is the small number of publicly available
500   datasets in this field, compared to the broader tabular data literature where vast numbers of datasets
501   are in popular use. We endeavored to add multiple datasets and baselines but currently a primary
502   challenge is the lack of publicly available datasets and easy to use baselines. We hope to expand and
503   diversify datasets in future work and caution practitioners to carefully benchmark methods on their
504   own data. We also only support categorical and numerical features, but in principle there is no reason
505   to only consider these kinds of features. Real-world events might have text or graph descriptors as
506   well.
507   Additionally, while our implementations are relatively straightforward, we view the simplicity of our
508   methods as a feature of STEP. Prior work attempted to implement architectural and data ablations but
509   we believe that demonstrating that simplicity is sufficient for superior results is a contribution of our
510   method.
511   Ethics. Sequential tabular event data is common in financial settings where user information can be
512   particularly sensitive. While this should be a consideration for anyone deploying a model like STEP,
513   for this paper we used only publicly available datasets on synthetic or anonymized users.
514
      Reproducibility. We have taken steps to make this work as reproducable as possible. All datasets
515
      described in section 4.1 are publicly available and we have provided cited each data source. A large
516
      part of our modeling pipeline is the event preprocessing that is described in section 3.3. Lastly, the
517   code used to train STEP is publicly available on our github6 .
