009                                                A BSTRACT
010
011            We introduce AutoCoder, a code Large Language Model that surpasses GPT-
012            4 Turbo 2024-04-09 and GPT-4o 2024-08-06 in pass@1 on the Human
013            Eval benchmark test (90.9% vs. 90.2%). In addition, AutoCoder offers a more
014            versatile code interpreter compared to GPT-4 Turbo and GPT-4o. Its code in-
015            terpreter can install external packages instead of limiting to built-in packages.
016            AutoCoder’s training data is a multi-turn dialogue dataset created by a system
017            combining agent interaction and external code execution verification, a method we
018
               term AIEV-I NSTRUCT (Instruction Tuning with Agent-Interaction and Execution-
               Verified). Compared to previous large-scale code dataset generation methods,
019
               AIEV-I NSTRUCT reduces dependence on proprietary large models and provides
020
               execution-validated code dataset. The code and the demo video is available in
021            supplementary materials.

024   1    I NTRODUCTION
025
026   Code generation is a critical aspect of modern software development. It significantly enhances devel-
027   opment efficiency and quality by increasing productivity, reducing errors, standardizing code, acceler-
028   ating prototyping, and supporting complex systems Li et al. (2024; 2023a); Buscemi (2023).Recently,
029   Large Language Models (LLMs), such as GPT-4 OpenAI (2024) and Claude 3.5 Sonnet Anthropic
030   (2024), have achieved significant advancements on code generation. These models have shown high
031   accuracy in producing code that meets user requirements and have been widely adopted in real-world
032   software development.
033   Training large language models requires extensive high-quality data Hoffmann et al. (2022). This is
034   particularly crucial for code generation tasks that demand high accuracy Chen et al. (2021). OpenAI
035   once hired people to help annotate the Code Instruct dataset for training their InstructGPT Ouyang et al.
036   (2022). However, manually annotating large-scale code instruction datasets is both economically
037   and time-consuming Xu et al. (2022). To address this challenge, previous work has employed
038   various automated code annotation methods, such as S ELF -I NSTRUCT Wang et al. (2022), E VOL -
039
      I NSTRUCT Luo et al. (2023), and OSS-I NSTRUCT Wei et al. (2023). S ELF -I NSTRUCT enhances
      LLMs’ instruction-following capabilities by using strong teacher models to generate synthetic coding
040
      instructions for fine-tuning weaker student models. E VOL -I NSTRUCT improves LLMs’ coding
041
      abilities by iteratively increasing the complexity of seed code instructions through various heuristics.
042   OSS-I NSTRUCT generates diverse and realistic coding problems by drawing inspiration from open-
043   source code snippets. The essence of these methods lies in distilling the knowledge of a powerful
044   teacher model (such as GPT-4 Turbo) to guide a smaller model. This leads to a problem: While the
045   small model can achieve significant performance improvements, the final accuracy of the small
046   model is unlikely to surpass that of the teacher model. Because both the correct and incorrect
047   knowledge from the teacher model are transferred to the small model. Moreover, although using
048   closed-source models reduces costs compared to manual annotation, the cost of using closed-source
049   models remains high. According to our tests, even with the relatively cheaper GPT-4 Turbo model,
050
      generating an average of 6,500 high-quality entries for the code instruction dataset costs $1,000.
051   This raises two questions:
052
053         1. Can we correct the incorrect knowledge generated by the teacher model to provide more
               accurate code for the student model?


                                                         1
      Under review as a conference paper at ICLR 2025


054
                           2. Instead of relying on expensive closed-source teacher models, can we enable our student
055                           model to learn autonomously?
056
057   To address the first issue, we designed a new large-scale code instruction dataset annotation method
058   called AIEV-I NSTRUCT. It is an interaction system comprising two agents: a questioner and a
059   programmer. These agents interact to simulate the process of programmers constructing code
060   according to project requirements and conducting unit tests. In each dialogue round, we extract
061
      the code generated by the programmers and execute it. The execution results are returned to the
      questioner to inform the next round of questions. This process continues until the programmers’s
062
      code passes the unit tests, ensuring the accuracy of the generated code dataset.
063
064   To address the second issue, we sperate AIEV-I NSTRUCT into two stages: the Teaching Stage and
065   the Self-learning Stage. In the Teaching Stage, we rely on proprietary large models as agents for
066   code annotation, similar to previous methods. Once our model surpasses the proprietary models in
067
      accuracy on the test set, we transition to the Self-learning Stage. In this stage, we use our own model
      as the agent for code annotation. For detailed methodology, refer to Section 3.

085   Under the support of AIEV-I NSTRUCT, we obtained 169K high-quality code instruction data sam-
086   ples. Using this dataset, we trained the AutoCoder series models, including AutoCoder (33B)
087   and AutoCoder-S (6.7B). As shown in Figure 1, AutoCoder demonstrates higher accuracy. In
088   the HumanEval Base Test, we compared our results with several models featured on the EvalPlus
089   Leaderboard as of September 2024 evalplusleaderboard (2024). The performance of Claude 3.5
090   Sonnet Anthropic (2024) was obtained from its official website, while the GPT-4o 2024-08-06
091   results were self-implemented. Remarkably, AutoCoder’s Pass@1 even outperforms some of the
092
      top-ranked models, including GPT-4 Turbo 2024-04-09 and GPT-4o 2024-08-06.
093   Moreover, as illustrated in Figure 2, AutoCoder boasts a more versatile Code Interpreter function
094   compared to GPT-4o. The Code Interpreter is an external program execution environment that large
095   models utilize to execute the code they deem necessary. While GPT-4o and GPT-4 Turbo can identify
096   the code that needs to be executed, they fail to provide the Code Interpreter with the necessary
097   instructions to install external packages required by the programs. This limitation significantly
098
      restricts the capabilities of the Code Interpreter. In contrast, AutoCoder can correctly supply the
      Code Interpreter with the appropriate external package installation instructions, thereby enabling it to
099
      execute a wide variety of code. As far as we know, as of September 2024, AutoCoder is the only
100
      model that supports automatically installing external packages in the Code Interpreter.
101
102   To comprehensively evaluate the capabilities of AutoCoder, we tested it on several datasets: Hu-
103   manEval Chen et al. (2021), HumanEval+ Liu et al. (2024), MBPP Austin et al. (2021), MBPP+ Liu
104
      et al. (2024), MultiPL-E Cassano et al. (2022), DS-1000 Lai et al. (2023) and LiveCodeBench Jain
      et al. (2024) . To analyze the contribution of different components to AutoCoder’s performance, we
105
      compared it to its base model, Deepseek-Coder Guo et al. (2024). The performance and detailed
106
      experimental procedures can be found in Section 5.

      Large Language Models for Code. Recently, LLMs have shown remarkable abilities in understand-
122   ing and generating code Kazemitabaar et al. (2023). Trained on extensive datasets covering various
123   programming languages and tasks, these models excel in code completion, bug fixing, and code
124   synthesis Jin et al. (2023). Closed-source models like OpenAI’s GPT-4 OpenAI (2024), Claude.ai’s
125   Claude AnthropicAIteam (2024), and Google’s Gemini deepmindteam (2024) series have demon-
126   strated superior performance on code tasks. Meanwhile, open-source models specialized for code,
127   such as DeepSeek-Coder deepseekteam (2024), CodeQwen Qwen (2024), Magicoder Wei et al.
128   (2023), OpenCodeInterpreter Zheng et al. (2024), and WizardCoder Luo et al. (2023), are also emerg-
129   ing. Generally, closed-source models outperform open-source ones due to their larger parameter sizes
130
      and broader knowledge base.
131   Code LLMs Instruction Tuning. After pre-training large models, we use instruction tuning to
132   optimize them Gao et al. (2020), enhancing their ability to understand and execute specific instruc-
133   tions Chang et al. (2024). A major challenge in Instruction Tuning for Code LLMs is the lack of
134   high-quality instruction datasets for code Rao (2024). Code tasks, such as Text-Code and Code-Code
135   translation, are difficult and time-consuming to annotate manually. OpenAI used human annotators
136
      to label various tasks and train InstructGPT Ouyang et al. (2022), but they noted that annotating code
      tasks is prohibitively expensive for large-scale datasets. Since the advent of GPT-4, an increasing
137
      number of researchers have leveraged GPT-4 for code annotation to create high-quality instruction
138
      tuning datasets. Currently, there are three primary methods: S ELF -I NSTRUCTWang et al. (2022),
139   E VOL -I NSTRUCTLuo et al. (2023), and OSS-I NSTRUCT Wei et al. (2023). S ELF -I NSTRUCT boosts
140   LLMs’ instruction-following skills by using strong teacher models to generate synthetic coding
141   instructions for fine-tuning weaker student models. E VOL -I NSTRUCT iteratively enhances LLMs’
142   coding abilities by increasing the complexity of seed code instructions. OSS-I NSTRUCT creates
143   diverse coding problems inspired by open-source code snippets. These methods distill the expertise
144   of powerful teacher models like GPT-4 to guide and improve smaller models.

149
150   Figure 3 illustrates the overall architecture of AIEV-I NSTRUCT, divided into two stages: the Teaching
151   Stage and the Self-Learning Stage. In the Teaching Stage, the model learns primarily by distilling
152   knowledge from a teacher model. In the Self-Learning Stage, it learns autonomously.
153
      In the Teaching Stage, we obtain open-source code snippets and use GPT-4 Turbo as the teacher model
154
      to supplement and correct them. The process consists of four main steps. In I : Initialization, we
155
      initialize the necessary components. GPT-4 Turbo is assigned two roles: questioner and programmer.
156   It can ensure the generated data is diverse, resulting in a more uniform probability distribution
157   rather than converging to a specific dialogue template. The dialogue messages are initialized as an
158   empty list, which will be used throughout the process to store data. Eventually, this list will contain
159   multiple rounds of dialogue, and the entire conversation will be added as a single data entry to our
160   final dataset. Additionally, we need to initialize a Docker container as our Code Interpreter. This
161   container is responsible for installing the required external packages and executing the code that
      needs verification throughout the process. In II : Propose the question, we first utilize GPT-4 Turbo

197   require GPT-4 Turbo to provide some Unit Tests. These Unit Tests further ensure the accuracy of the

200   we use multiple rounds of execution feedback to check the generated code, thereby improving the
201
      quality of the dataset. First, we input the code snippet generated in the second step into the Code

202

203
      language description based on the Stderr. This natural language description is also appended to the
204   dialogue messages (➃). Next, both the natural language description and the Stderr are provided as
205   new questions to the programmer, who will continue to modify the code. The dialogue messages will
206   append the new code it generates (➄) and repeat this process. In IV: Termination, we also use the
207   Code Interpreter to run the code generated by the programmer. If the program executes successfully,
208   the Stdout is appended to the dialogue messages (➅). This completes the analysis of one data entry.
209
      After analyzing every 2000 data entries, we split the new data into a test set and a training set in a 1:9
210   ratio. The training set is used to train the student model (AutoCoder). After training, we use the test
211   set to evaluate both the teacher model and the student model. Upon completion of the evaluation, we
212   compare the Pass@1 of the two models. If the teacher model performs better, we continue executing
213   the Teaching Stage. If the student model performs better, we move to the Self-Learning Stage. The
214   difference between the Self-Learning Stage and the Teaching Stage is that in the Self-Learning Stage,
215   we replace the original teacher model with the student model. The student model itself is assigned as
      the questioner and programmer , and it completes the entire execution feedback process.

      3.2   DATASET A NALYSIS
217
218   Dataset Generation. To prevent data contamination in test sets from resulting in overly high
219   performance on certain benchmark datasets (such as HumanEval), we used code from two datasets
220   that had already undergone contamination detection: Magicoder-Evol-Instruct and Magicoder-OSS-
221   Instruct Wei et al. (2023). We collected a total of 186K original code entries from these two datasets.
222   After de-duplication, we input these data into our AIEV-Instruct pipeline to generate the dataset. We
223   set the maximum number of execution feedback iterations in AIEV-Instruct to 7. If the generated
224   code fails to execute successfully and pass all unit tests after 7 attempts, that data point is discarded.
225   The gpt-4-turbo-2024-04-09 is used as the teacher model. Sample demonstrations of some
226
      data points are provided in the Appendix D.
227   Dataset Comparision. We compared our dataset AutoCoder-AIEV-Instruct with several current large
228   code instruction datasets. The comparison results are shown in Figure 4. The dataset AutoCoder-AIEV-
229   Instruct contains 169K data samples, totaling 241K rounds of dialogue. Among these, 150K rounds
230   are contributed by multi-round dialogue data samples. Besides including the main function, it also
231   encompasses subsequent package installations, code execution errors, or results, as well as various
232
      error analyses. Compared to the original Magicoder-Evol-Instruct and Magicoder-OSS-Instruct, it
      adds unit tests, which further enhances the accuracy of code-related tasks. Additionally, compared
233
      to Code-Feedback Zheng et al. (2024), it includes more execution feedback results, reducing the
234
      multi-round dialogues for code block concatenation and enhancing the coherence of the context.
235
236   Dataset Decontamination. Similar to the data processing method used by StarCoder Li et al.
237   (2023b), we also performed decontamination for AutoCoder-AIEV-Instruct. Specifically, we tested
238
      each code snippet from HumanEval, MBPP, DS-1000, and MultiPL-E against every code snippet in
      AutoCoder-AIEV-Instruct using Levenshtein distance. If the similarity exceeded 90%, the data entry
239
      was removed. Through this process, we excluded a total of 113 data entries.
240

242   experiments, we provide some theoretical analysis in the Appendix B to explain why datasets
243   generated using the AIEV-INSTRUCT method achieve higher accuracy compared to previous
244
      OSS-I NSTRUCT and E VOL -I NSTRUCT methods. Specifically, AEvol < AOSS < AAIEV .

      Under review as a conference paper at ICLR 2025


324
      In the post-processing stage, we mix the natural language of the Code execution request into the
325   User’s natural language, enabling the model to correctly learn when to execute the code. Then, we
326   mix the code execution request response into the Assistant’s response, so it can generate coherent
327   answers. Finally, we add special tokens before and after the bash commands and code blocks in the
328   Assistant’s original response, allowing the model to learn to correctly identify the bash commands
329   and code blocks that need to be executed. Sample demonstrations of data points after post-processing
330   are provided in the Appendix D.

      4.2   T RAINING D ETAILS
333
334
      We fine-tuned two base models, Deepseek-Coder 6.7B and 33B, using the AutoCoder-AIEV-Instruct
335
      dataset to obtain our AutoCoder 33B and AutoCoder-S 6.7B. We utilized the AutoTokenizer
336   package from the transformer library to add four special tokens to these models to enable the
337   Code Interpreter feature for AutoCoder. For hardware, we used 10 nodes with a total of 40 80GB
338   A100 GPUs on a Simple Linux Utility for Resource Management (SLURM) cluster. The NVIDIA
339   Collective Communications Library (NCCL) handled communication between GPUs. In terms of
340   training parameters, we used the ZeRO-Stage 3 feature from the deepspeed library to partition
341   model parameters, with a batch size of 8 per GPU, a gradient accumulation step of 4, a learning
342   rate of 5e-5, and bf16 as the parameter type. The max sequence length was set to 5120 and the total
343   epochs was set to 2. We adopted a full-parameter tuning approach to train the model.

529   6                 C ONCLUSION
530
531
      We propose AIEV-I NSTRUCT, a novel method for creating high-quality code instruction datasets. It
532
      simulates programmers writing code and conducting unit tests through agent interactions, ensuring
533
      accuracy with execution validation. It includes both a teaching stage and a self-learning stage,
534   reducing reliance on expensive closed-source models during the annotation process. Using the
535   dataset generated with AIEV-I NSTRUCT, we trained the AutoCoder code LLM. It exhibits excellent
536   performance and surpass the current top models, GPT-4 Turbo and GPT-4o on the HumanEval
537   benchmark. Furthermore, AutoCoder extends the functionality of previous code interpreters by
538   allowing them to automatically install external packages, thus extending the applicability of the
539   code interpreter. Overall, our work provides the community with excellent open-source code large
      language models and offers new insights for generating high-quality large code instruction dataset.

719   B    DATASET ACCURACY T HEORETICAL A NALYSIS
720
721   In this analysis, we compare the theoretical maximum accuracies of different datasets (E VOL -
722
      I NSTRUCT, OSS-I NSTRUCT, and AIEV-I NSTRUCT) to understand how various factors, such as
      problem generation, code alignment, and iterative validation, affect the overall accuracy of code
723
      generation models. The following assumptions is introduced.
724
725
      Assumption 1. (Teacher Model Accuracy for Code Generation) The theoretical maximum accu-
      racy of the E VOL -I NSTRUCT dataset should closely match the teacher model’s accuracy in generating
726
      correct code c for given problems p. Therefore, AEvol ≈ P(c | p).
727
728
      Assumption 2. (Teacher Model Accuracy for Problem Generation) The theoretical maximum
      accuracy of the OSS-I NSTRUCT dataset should closely match the teacher model’s accuracy in
729
      analyzing and interpreting open-source code c. Therefore, AOSS ≈ P(p | c).
730
731
      Assumption 3. (Rarity of Code) The number of valid code solutions c for a problem p is smaller
      than the number of possible problem descriptions. Thus, P(c) < P(p).
732
733   Assumption 4. (Improved Alignment through Unit Tests) Adding unit tests to the original code
734   improves the alignment of problem descriptions and code. Therefore, we assume P(p | c) < P(p∗ |
735
      c∗ ), where p∗ and c∗ represent the new problem description and code after adding unit tests.
736   Assumption 5. (Iterative Validation Increases Correctness) The probability of correctness im-
737   proves with each iteration during the iterative validation and correction process. If the probability
738
      of error in each iteration is 1 − P(p∗ | c∗ ), then the probability of correctness after n iterations is
      AAIEV ≈ 1 − (1 − P(p∗ | c∗ ))n .
739
740   Remark 1. All assumptions above are mild. Assumption 1 is necessary to establish that the theoretical
741
      accuracy of the E VOL -I NSTRUCT dataset is fundamentally linked to the teacher model’s capacity to
      generate correct code for a given problem. Assumption 2 reflects that the OSS-I NSTRUCT dataset’s
742
      accuracy depends on the teacher model’s ability to interpret and understand the open-source code, a
743
      natural requirement for problem generation. Assumption 3 acknowledges the common observation
744   that there are fewer valid code solutions than possible problem descriptions, making code inherently
745   rarer. This assumption is essential for applying Bayes’ theorem in the analysis. Assumption 4
746   states that adding unit tests to the original code enhances the alignment between the code and its
747   corresponding problem description, which is a well-accepted practice in software engineering to
748   ensure code correctness. Finally, Assumption 5 posits that iterative validation and correction increase
749   the probability of achieving a correct solution. This is a common concept in optimization processes,
750   where each iteration helps refine and improve the overall accuracy.
751
      Based on these assumptions, the following theorem is derived.
752
      Theorem 1. (Relative Accuracy of Datasets) Given the assumptions 1 - 5, the accuracy of
753
      AUTO C ODER -AIEV-I NSTRUCT is higher than that of M AGICODER -OSS-I NSTRUCT, and the
754
      accuracy of M AGICODER -OSS-I NSTRUCT is higher than M AGICODER -E VOL -I NSTRUCT, that is,
755
                                             AEvol < AOSS < AAIEV .


                                                         14
      Under review as a conference paper at ICLR 2025


756
      Proof. From Assumption 1, we have AEvol ≈ P(c | p). By applying Bayes’ theorem, one can get
757
      P(c | p) = P(p|c)·P(c)
                    P(p)     . Based on the Assumption 3, it follows that AEvol ≈ P(p|c)·P(c)
                                                                                     P(p)     < P(p | c).
758
759   From Assumption 2, the accuracy of the OSS-I NSTRUCT dataset is given by AOSS ≈ P(p | c).
760
      Next, by Assumption 4, adding unit tests improves the alignment, so P(p | c) < P(p∗ | c∗ ).
761
762   By Assumption 5, the probability of correctness after n iterations in the iterative validation process
763   is AAIEV ≈ 1 − (1 − P(p∗ | c∗ ))n . Since P(p∗ | c∗ ) > P(p | c) and n > 1, one can get
764   1 − (1 − P(p∗ | c∗ ))n > 1 − (1 − P(p | c))n .
765   Thus, AAIEV > AOSS . Combining these results, one can get
766
                           P(p | c) · P(c)
767              AEvol ≈                   < P(p | c) ≈ AOSS < 1 − (1 − P(p | c))n < AAIEV ,
768                            P(p)
769   that is, AEvol < AOSS < AAIEV . This completes the proof.
