                                                        Abstract
012
013            Preference learning provides a promising solution to address the limitations of supervised
014            fine-tuning (SFT) for code language models, where the model is not explicitly trained
015            to differentiate between correct and incorrect code. Recent findings demonstrate that
016            on-policy data is the key to successful preference learning, where the preference data is
               collected using the same policy LM being trained. Inspired by this, we propose PLUM,
017
               an on-policy Preference Learning framework Augmented with test cases for code LMs.
018            The framework operates in three key stages: (1) automatic generation of test cases from
019            natural language instructions, (2) creation of a preference data by evaluating candidate code
020            solutions sampled from the policy, which can then be used to (3) train the policy LM. PLUM
021            levitates the need to train reward models, allowing for large scale on-policy and online
               preference data collation. PLUM is evaluated on both standard benchmarks (HumanEval,
022
               MBPP) and more challenging ones (LiveCodeBench), delivering substantial improvements
023            over original SFT’ed models and other execution-feedback-driven approaches. We show
024            PLUM’s benefits are consistent across various widely-used code LMs even they have been
025            well-trained with SFT. For example, PLUM increases pass rates by up to 4.8% on average
026            on standard benchmarks and 11.8% on LiveCodeBench, demonstrating its effectiveness
               and generalizability. We also demonstrate the benefits of on-policy and online preference
027
               learning by comprehensive experimentation.

030   1    I NTRODUCTION
031
032   Language models pre-trained on code corpora have excelled at code generation (Rozière et al.,
033   2024; Li et al., 2023). Supervised Fine-Tuning (SFT) enhances their ability to follow natural
034   language prompts but focuses on reproducing patterns from training data rather than ensuring code
035   correctness (Wei et al., 2023; Zheng et al., 2024). This leads to models that generate syntactically
036   correct but functionally flawed code, unable to meet real-world requirements like edge cases or
037   algorithmic accuracy (Chen et al., 2024). Works like AlphaCode (Li et al., 2022) and LeTI (Wang
038   et al., 2024b) have introduced test outcomes as a means to define functional correctness in code
039
      generation. Building on the insights from these efforts, we propose leveraging preference learning for
      refining model behavior. Preference learning trains models to prefer certain solutions (e.g., factual,
040
      helpful, or harmless) over undesirable ones (e.g., inaccurate, unhelpful, or harmful). Despite its
041
      success in aligning models with human values and improving reasoning in other domains (Dong et al.,
042   2023; Guo et al., 2024a; Yuan et al., 2024; Wang et al., 2024a; Pang et al., 2024), the application of
043   preference learning as a principled and efficient approach in code generation remains under-explored,
044   largely due to the lack of high-quality training data.
045
      Recent research shows that reducing the likelihood of incorrect outputs is more effective for improving
046
      model performance than simply maximizing correct responses (Setlur et al., 2024; Tajwar et al.,
047
      2024). Mode-seeking objectives, which prioritize minimizing errors, have been found to outperform
048   maximum likelihood methods by more efficiently redistributing probability mass across potential
049   outputs. This underscores the importance of applying on-policy and online approaches to enhance
050   preference learning algorithms (Tajwar et al., 2024; Guo et al., 2024b; Setlur et al., 2024; Liu et al.,
051   2024). Unlike offline preference data, on-policy data remains in-distribution with the model, reducing
052   the risk of misalignment (Guo et al., 2024b; Zhang et al., 2024; Tang et al., 2024; Fisch et al., 2024).
053   The main challenge now is how to efficiently obtain preference labels for on-policy data at scale (Yang
      et al., 2024b). In programming tasks, test cases present as a native and powerful candidate solution to

      address this issue. Being able to automatically produce high-quality test cases unlocks the possibility
055   of collecting preference data over programming questions at any scale.

077   To this end, we propose preference learning framework augmented with test cases for training code
078   language models (PLUM), which integrates the process of deriving test cases from natural language
079   specifications into the training process to obtain preference labels for model’s on-policy candidate
080   solutions. PLUM utilizes natural language instructions from well-established datasets such as OSS-
081   Instruct (Wei et al., 2023), Evol-Instruct-Code (Luo et al., 2023), and ShareGPT (Cleston, 2023). For
082   each instruction, high-quality test cases are constructed, and multiple solutions are sampled from the
083   model. These solutions are evaluated using the generated test cases, with preference labels assigned
084
      based on the results: solutions passing the tests are preferred, while failures are dis-preferred. This
      dataset then trains the policy using established preference learning algorithms (Rafailov et al., 2023;
085
      Ethayarajh et al., 2024; Azar et al., 2023). By relying solely on policy model’s self-generated solutions,
086
      PLUM eliminates the need for external, synthetic off-policy data, reducing the risk of distributional
087   shifts and poor generalization commonly observed for synthetic off-policy data, enhancing the
088   model’s robustness and improving its ability to differentiate between correct and incorrect solutions.
089   In addition, by showcasing the effectiveness of our framework, we demonstrated the feasibility of
090   bypassing tedious (and potentially unstable) reward model training (Liu et al., 2023a; Le et al., 2022)
091   and manual labeling, by automating the process of test case synthesis. These simplicity advantages
092   of PLUM makes online preference training of language models possible.
093   We evaluate PLUM on a diverse set of state-of-the-art code language models under different set-ups,
094   on commonly used benchmarks: HumanEval(+) and MBPP(+) (Chen et al., 2021; Austin et al.,
095   2021; Liu et al., 2023b) as well as more challenging code generation datasets like LiveCodeBench
096   and LeetCode (Jain et al., 2024; Guo et al., 2024a). We demonstrate that our approach seamlessly
097   integrates with various models in a plug-and-play manner, relying solely on coding instructions to
098   enhance models’ code generation capabilities. Furthermore, we show that online training, facilitated
099   by automated test case generation, further boosts model performance particularly on difficult coding
100   benchmarks, echoing findings from other domains (Xiong et al., 2024b).

145   2.1      T HE PLUM
146
147   As illustrated in Algorithm 1 and Figure 1, our approach takes in a base policy model, a set of natural
148   language programming instructions, and a test case generator. For each iteration, it produces multiple
149   test cases for the batch of instructions. Then we sample solutions from policy πθ , and execute them
150   against the generated test suite to obtain preference labels. We then update the policy πθ := πθ′ .

152   2.2      G ENERATING T EST C ASES
153
154   A crucial factor in making PLUM successful is the ability to synthesize high-quality test cases for
155   programming questions. In the following subsections, we provide a detailed explanation of the test
156
      case generation process, outlining how it contributes to the overall effectiveness of PLUM.
157   The test cases in PLUM are generated with a test-case generator model over natural instructions
158   from established code generation datasets.1 In automated testing, ensuring the correctness and
159   completeness of test cases is a persistent challenge due to the lack of reliable oracles to validate
160   test outputs. We adopt two strategic principles: 1) employing self-consistency as an approximate
161

             We use GPT-4-1106 as the generator model.

      oracle, and 2) generating diverse test suites to minimize overfitting to any particular test instance and
163   mitigate under-specification.
164
165
      Collecting instructions from established datasets We collect natural language instructions from
166
      established datasets including OSS-Instruct (Wei et al., 2023), Evol-Instruct-Code (Luo et al., 2023),
167
      and ShareGPT (Cleston, 2023). 2 These datasets provide a diverse range of programming tasks and
168   instructions. Although they come with gold/silver solutions in the training splits, these solutions are
169   never used in PLUM. Instead, they allow us to directly compare PLUM’s performance against SFT,
170   which relies on gold solutions, as we investigate in our experiments. Not requiring gold solutions for
171   training broadens the applicability of PLUM to a wide variety of real-world coding tasks and user
172   requirements.
173
174   Generating high-quality test cases Given a training instruction in natural language, we prompt
175   a generator model to produce a reference solution, a starter code snippet specifying the function
176   signature, and a suite of test cases using the prompt in Figure 1. The generated test cases are critical
177   for ensuring that the solutions meet the functional requirements specified in the instructions. The
178   correctness of the test cases is central to the success of preference learning. We adopt a consistency-
179   based approach inspired by Chen et al. (2023) and Rozière et al. (2024) for quality control.
180   We check for consistency between the generated reference solution and the test cases. Pairs where the
181   test cases do not accurately reflect the solution, or the solution does not pass the test cases, are filtered
182   out. This process helps minimize potential noise and enhances the quality of the test cases used in the
183   following stages. The generated reference solutions serve only to control the quality of the test cases
184   and are never used in training. Similarly, the solutions provided with the instruction data are never
185   used in PLUM. On average, each instruction is paired with 3–5 test cases depending on the dataset.

189   Many preference learning algorithms assume that the preference data is in-distribution for the policy,

192
      sampled from different models than the policy, leaving the data out of distribution Bai et al. (2022);

193
      same instructions before applying preference learning (Rafailov et al., 2023; Yuan et al., 2024). This
194
      ensures that the policy has a similar distribution to that from which the preference data are sampled.

455   On-Policy vs Off-Policy We now address whether "on-policy" training makes a significant differ-
456   ence. To test this, we apply the same method but use preference data sampled from other models (i.e.,
457   off-policy). We selected models of varying sizes, including DeepSeek-Coder-1.3B-Instruct, Qwen-
458
      2.5-Coder-1.5B-Instruct, CodeStral-22B (MistralAI, 2024), and DeepSeek-Coder-33B-Instruct. We
      followed the exact same data collection and training processes. Our results show that while off-policy
459
      training still improves model performance, it generally underperforms compared to on-policy training.
460
      These findings are consistent with prior research (Xiong et al., 2024a; Dong et al., 2024; Tajwar
461   et al., 2024; Xu et al., 2024).
462
463   Further, we investigate the effect of synthetic negatives. To this effect, we use a mutation-based
464
      approach for synthetically introducing errors into Python code while maintaining its syntactic
      correctness, as detailed in Appendix A.8. This method uses Abstract Syntax Tree (AST) manipulation
465
      to apply mutations like argument swapping, operator replacement, control flow changes, off-by-one
466
      errors, and return value modification. By injecting these errors, it generates valid but behaviorally
467   altered code. We apply this to on-policy positive examples, creating off-policy negatives for model
468   training under the same setup. Observing that the synthetic negatives could even potentially harm
469   the performance, we confirmed the importance of preference training with more natural, and ideally
470   on-policy negative samples.
471
      This highlights our contribution in demonstrating a method empowered by automated test cases and
472
      efficient on-policy preference learning. This approach can be easily adopted to scale the collection of
473   test cases, providing a robust supervision signal for model training.
474
475
      Importance of Test Case-Based Preference Learning We experiment with including only non-
476   executable samples as rejected solutions, while using the same set of chosen solutions. As shown
477   in Figure 2, we observe that this is worse than PLUM in most cases. More importantly, it does not
478   always improve the model’s performance and may even hurt. This has also been noted in previous
479   studies (Wang et al., 2024b). Although the positive examples used are the same as our oracle-based
480   preference learning, lower-quality negative examples do not necessarily help the model improve due
481   to the additional noise in the preference signal.
482
483   Test Cases Allow By-passing Reward Model Training For Iterative Alignment We demonstrate
484   in Table 7 that PLUM enables iterative on-policy alignment while providing accurate preference
485   signals without requiring reward model training. Notably, iterative preference learning, facilitated by
      the online feedback loops generated through the test case collection procedure, outperforms offline

492   4   R ELATED W ORKS
493
494   Reinforcement Learning and Preference Learning For Reasoning-Related Tasks Preference
495   learning algorithms like Direct Preference Optimization (DPO) (Rafailov et al., 2023) and Kahneman
496   & Tversky’s Optimization(KTO) (Ethayarajh et al., 2024) are popular for their cost efficiency and
497   training stability. Beyond controlling model-user interactions, these methods are now applied to more
498   complex reasoning tasks. Ocra-Math (Mitra et al., 2024) uses iterative preference learning to improve
499   math reasoning in SFT-ed language models, while Eurus (Yuan et al., 2024) leverages preference
500
      trees for solving complex problems through multi-step interactions with external feedback.
501
502   Code Generation with Large Language Models Code generation has become a key application
503   of generative language models. Pre-training on code corpora has led to strong performance in models
504   like StarCoder (Li et al., 2023), StarCoder2 (Lozhkov et al., 2024), and DeepSeek-Coder (Guo et al.,
505   2024a), while others like CodeQwen (Bai et al., 2023) and CodeLlama (Rozière et al., 2024) benefit
506   from continued pre-training on additional code data. To enhance these models, supervised fine-tuning
507
      on instruction-response pairs has been employed (Luo et al., 2023; Wei et al., 2023; Zheng et al.,
      2024). Reinforcement learning techniques, like those in CodeRL (Le et al., 2022; Shojaee et al.,
508
      2023; ?), and reward models used in DeepSeek-Coder-V2 (DeepSeek-AI et al., 2024) also improve
509
      performance using test feedback.
510
511
512   Test Case Generation with Language Models Automated test case generation (Pacheco et al.,
513
      2007; Fraser & Arcuri, 2011; Panichella et al., 2015) is crucial for ensuring software quality and
      safety and has long been a key topic in software engineering. The advent of LLMs has inspired
514
      works using transformer models for test generation, either by training models (Tufano et al., 2021; Li
515
      et al., 2022) or prompting them (Chen et al., 2023). Test cases also help clarify user intent, aligning
516   model-generated programs with user requirements (Fakhoury et al., 2024; Endres et al., 2024).
517
518
519
      The synergy between test cases and code generation Programming-by-examples (Gulwani, 2016)
      and test-driven programming (Perelman et al., 2014) focus on using test cases to automatically refine
520
      programs to meet specifications. This concept has been adapted to enhance deep learning approaches
521
      to code synthesis (Kulal et al., 2019; Chen et al., 2023; Zelikman et al., 2023). Recent methods, like
522   CodeT (Chen et al., 2023) and Parsel (Zelikman et al., 2023), use test cases to reduce the search
523   space during inference, while ALGO (Zhang et al., 2023) employs brute-force solutions as oracles to
524   generate test outputs for competitive programming. Our approach, similar to Haluptzok et al. (2023),
525   leverages test cases during training to improve models’ inherent programming capabilities.
526
527
528   5   C ONCLUSION
529
530   In this paper, we introduced PLUM, a novel preference learning framework designed to improve
531   the ability of code language models (LMs) to distinguish between correct and incorrect code by
532   leveraging test cases. Our framework tackles the limitations of traditional supervised fine-tuning
533   approaches by embedding on-policy learning directly into the training process. Through the automatic
534   generation and evaluation of test cases, PLUM enables models to learn from their own outputs without
535   requiring separate reward models or manual labeling, offering a scalable and flexible solution.
536   The results from our experiments demonstrate the effectiveness and generalizability of PLUM.
537   Furthermore, we performed careful experiments and showed that on-policy preference learning
538   outperforms various off-policy methods, highlighting the crucial role played by on-policy training.
539   Further, we demonstrated PLUM allows for effective online preference learning that further pushes
      the performance on challenging coding benchmarks.

937   A.3   A DDITIONAL R ESULTS : W HAT IF DIRECTLY USING COMPETITIVE CODING DATASETS ?
938
939
940   There are existing competitive coding datasets like C ODE C ONTESTS Li et al. (2022) which are
941   equipped with test cases. Using more general datasets like OSS-I NSTRUCT (Wei et al., 2023),
942
      paired with synthetic test cases, is more advantageous for preference learning in code language
      models than using competitive coding datasets like Code Contests. Competitive coding datasets
943
      present complex problems with intricate edge cases, which can overwhelm the model and obscure
944
      fundamental instruction-following and preference-learning goals. In contrast, OSS-Instruct provides
945   more accessible, more uniform instructions that allow for cleaner and more straightforward alignment.
946   This helps models learn functional correctness more effectively without being distracted by the
947   nuances of competitive coding. Additionally, OSS-Instruct, sourced from real-world open-source
948   projects, avoids domain-specific biases that can arise from competitive coding, making it more
949   generalizable and applicable across diverse programming environments.
950
      We also conducted experiments to repeat the same process using C ODE C ONTESTS dataset. Due
951   to the challenging nature of this dataset, we sampled more to get the same number of positive and
952   negative cases as in OSS-I NSTRUCT and S HARE GPT etc. As shown in Table 8, training on this
953   dataset seems ineffective.
