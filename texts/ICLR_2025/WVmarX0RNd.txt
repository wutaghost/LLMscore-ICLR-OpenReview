009                                              A BSTRACT
010
011            As large language models (LLMs) grow in popularity for their diverse capabili-
012            ties, improving the efficiency of their inference systems has become increasingly
013            critical. Batching requests during LLM inference increases throughput by allow-
014            ing multiple requests to be processed in parallel, making better use of hardware
015            resources such as GPUs. However, the autoregressive nature of LLMs presents
016            a challenge: requests often have varying execution times, causing resource un-
017            derutilization, as hardware must wait for the longest-running request in the batch
018
               to complete before moving to the next batch. We propose Multi-Bin Batching, a
               simple yet effective method that can provably improve LLM inference throughput
019
               by grouping requests with similar execution times into predetermined bins. We
020
               evaluate multi-bin batching on various settings, showing consistent throughput
021            improvements compared to standard batching approaches.

024   1   I NTRODUCTION
025
026   Large Language Model (LLM) inference systems are becoming increasingly popular due to their
027   various abilities, such as text generation (Li et al., 2024), coding assistance (Chen et al., 2021),
028   and question answering (Jiang et al., 2021). As the demand for LLM inference systems grows, so
029   does the need to optimize their efficiency. Several techniques have been proposed to improve the
030   efficiency of LLM inference systems, and batched inference (Sheng et al., 2023; Kwon et al., 2023;
031   Jin et al., 2023) is one of the most promising techniques among them.
032   With batched inference, multiple requests are processed simultaneously, using the underlying hard-
033   ware’s parallelism to improve throughput. It can be seen in Figure 1a that generating 100 tokens for
034   each request in an increasing batch size improves throughput. We measure throughput for the Phi-
035   3.5 Mini Instruct model by prompting it with“once upon a time” in various batch sizes, generating
036   100 tokens per batch index on an NVIDIA A100 80G GPU. Throughput is calculated as total tokens
037   generated across all indices divided by total generation time, using greedy sampling.
038   However, batched inference comes with some critical drawbacks. The execution time of each re-
039   quest depends linearly on the number of tokens generated. In standard batched inference systems, a
040   computing unit remains locked until the entire batch is processed, meaning all requests in the batch
041   must be completed before the system is released. This can result in underutilization of resources,
042   offsetting some of the throughput gains achieved through parallelism in batched inference. Re-
043   cent studies have proposed dispatching additional requests to the computing node before the current
044
      batch is fully processed. This approach, known as continuous batching (Yu et al., 2022), requires
      fine-grained control of hardware, which is not always feasible. In distributed or cloud-based environ-
045
      ments, hardware control is typically abstracted or inaccessible, making it impossible to implement
046
      continuous batching.
047
048   Inspired by this, a natural question arises: can we achieve near-optimal throughput from batched
049   inference without depending on fine-grained, hardware-level controls? Addressing this challenge
050
      is crucial for achieving high LLM inference throughput, particularly in such environments where
      continuously dispatching additional requests is not feasible.
051
052   We propose a novel approach for optimizing batched inference by binning requests based on their
053   output lengths. Instead of placing all requests into a single queue, we create multiple “bins”, each
      serving as a waiting area for requests with similar output lengths. Incoming requests are assigned

      Why is this approach beneficial? Consider the example illustrated in Figure 1b. Suppose four
073   requests arrive at nearly the same time, with execution times of 1, 5, 2, and 6 seconds, respectively.
074   In a standard batching system with a batch size of B = 2, the requests would be grouped based on
075   their arrival time, forming two batches: (Request 1: 1s, Request 2: 5s) and (Request 3: 2s, Request
076   4: 6s). The total execution time would be 11 seconds (5 seconds for the first batch and 6 seconds for
077   the second).
078
      Now, consider our binning approach. Assume we have two bins: one for requests with output lengths
079   between 1 to 3 seconds and another for those between 4 to 6 seconds. In this case, Request 1 and
080   Request 3 would be placed in the first bin, while Requests 2 and 4 would go to the second bin. The
081   resulting batches—(Request 1: 1s, Request 3: 2s) and (Request 2: 5s, Request 4: 6s)—would reduce
082   the total execution time to 8 seconds (2 seconds for the first batch and 6 seconds for the second).
083   This simple binning strategy demonstrates how aligning requests by output length can significantly
084   improve LLM inference throughput.
085   Inspired by the toy example above, we propose multi-bin batching, a simple yet effective method
086   that can provably improve LLM inference throughput by grouping requests with similar execution
087   times into predetermined bins. We evaluate multi-bin batching on various settings using Microsoft’s
088   Phi-3.5-mini-instruct model on an Nvidia A100-80G GPU, demonstrating consistent throughput im-
089   provements compared to standard batching approaches. For instance, with the GSM8K dataset and
090   an oracle output length estimator, multi-bin batching enhances throughput by up to 75% compared to
091   standard batching systems. Our experiments span simulated results, and end-to-end LLM inference
092   with oracle lengths, all showing significant performance gains as the number of bins increases.
093   To summarize, our contributions are as follows:
094
095                                                           • We propose a novel binning-based batching system that can improve the throughput of
096                                                             LLM inference systems. Our batching system groups requests with similar execution times
097                                                             together based on predetermined bins.
098
099                                                           • We use queueing-theoretical analysis to show that our multi-bin batching strategy can im-
100                                                             prove the throughput of LLM inference systems. We also show that how many bins are
101                                                             needed to achieve any desired throughput improvement.
102
103                                                           • Our comprehensive experiments on real-world LLM models demonstrate that our proposed
104
                                                                multi-bin batching system can enhance throughput by up to 75% compared to standard
                                                                batching approaches.

      response length prediction using a transformer-based model and implemented QoS-friendly length

      control, resulting in improved throughput and latency of LLM inference systems.

152   We begin by introducing the system model and key assumptions that form the foundation of our
153   analysis. This model represents a typical LLM inference system as a queueing system with specific
154   characteristics. Following this, we will propose our novel batching algorithm, which leverages a
155   multi-binning approach to optimize request processing.
156   Assumption 3.1. The LLM inference system is a single-server queueing system with an infinite
157   queue length capacity. The system receives requests from a Poisson process with rate λ.
158
159   Assumption 3.1 is a standard assumption in queueing theory, and it is well-suited for LLM inference
160   systems since requests are typically generated by users in a random manner. The single server
161   assumption is reasonable; it can further be easily extended to multi-server systems by assuming that
      the servers are identical and requests are served in a first-come-first-serve manner. In that case, the


                                                        3
      Under review as a conference paper at ICLR 2025


162
      effective arrival rate λ can be divided by the number of servers, and one can derive similar results
163   following the same analysis.
164
165   The system forms batches of size B and serves them in a“first completed batch, first served” manner.
166
      The serving time of a batch of requests is the maximum of the serving times of the requests in the
      batch. This batching approach is suitable for LLM inference systems, as it enhances efficiency by
167
      optimizing the utilization of computing resources. The “first completed batch, first served” approach
168
      means that as soon as a batch is fully formed with B requests, it becomes eligible for service,
169   regardless of when its individual requests arrived. This allows for more efficient processing of
170   requests, especially when combined with our batching strategy.
171
172   Assumption 3.2. The service time of each request is independent and identically distributed (i.i.d.)
173   with a uniform distribution in the range [lmin , lmax ], i.e., l ∼ U(lmin , lmax ).
174
175   We make this assumption to simplify the analysis, and it is justified since LLM answer lengths
176   typically fall within a specific range due to maximum token length limitations. We also extend our
177   analysis to the case where the service time is exponentially distributed in the Appendix A.4.
178   In our theoretical analysis, we assume that the system always forms batches of size B and then start
179   processing them. However, in real systems, there could be a parameter that specifies the maximum
180   time a batch can wait before it is processed. This way the system can ensure that the latency of a
181   request does not exceed a certain threshold and the quality of service is maintained for all requests.

      We define the latency of a request as the time taken to complete a request from the time it is submitted
318   to the system. The latency of a request consists of two components: the queuing time and the service
319   time. In the previous section, we discussed the expected service time of a request for our multi-bin
320   batching system. In this section, we analyze the queuing time of a request. The queuing time
321   of a request is the time it spends waiting in the queue before it is processed. This time can also
322   be decomposed into two components: the time spent waiting to complete the current batch and
323   the time spent waiting for the current batch to start processing. In this analysis, we focus on the
      time spent waiting to complete the current batch, as it is the dominant component of the queuing


                                                         6
      Under review as a conference paper at ICLR 2025


324
      time in an underloaded system, which are common in cloud computing environments. In such
325   systems, it is reasonable to assume that the time spent waiting for the current batch to complete
326   remains the dominant factor in queuing time. This is because underloaded systems typically have
327   shorter queues and less contention for resources, making the wait time between batches relatively
328   insignificant compared to the time required to complete the batch itself. Therefore, we make the
329   following simplifying assumption for the purpose of latency approximation:
330   Assumption 5.1. The number of servers in the system is infinite. Therefore, whenever a batch is

      This paper introduced multi-bin batching, a novel approach to optimize Large Language Model
460
      (LLM) inference systems without relying on fine-grained hardware controls. By grouping requests
461
      with similar output lengths, our method provides a provable throughput increase, mitigating resource
462   underutilization in standard batched inference systems. Experiments demonstrated significant per-
463   formance gains compared to standard batching. Our scalable solution contributes to LLM inference
464   optimization and can be readily integrated into existing systems. As LLMs grow in importance,
465   multi-bin batching enables more efficient deployments across various computing environments, es-
466   pecially where fine-grained hardware control is unfeasible. Future work could refine bin prediction
467   models and explore adaptive binning strategies.
