                                                     A BSTRACT
008
009
                Scaling up inference-time compute has proven to be a valuable strategy in improv-
010
                ing the performance of Large Language Models (LLMs) without fine-tuning. A
011
                task that can benefit from such additional inference-time compute is self-repair:
012             given an initial flawed response, the LLM has to correct its own mistake and pro-
013             duce an improved response. We propose leveraging the in-context learning ability
014             of LLMs to perform self-repair. The key contribution of this paper is an approach
015             to synthesise and select a golden set of pairs, each of which contains a problem
016             with an initial guess, and a consequent fix, both generated by the LLM. Each
017             golden example pair, or AuPair1 , is then provided as an in-context example at in-
018             ference time to generate a candidate repaired solution with 1-shot prompting; in
019             line with best-of-N the highest-scoring response is selected. Given an inference-
020
                time compute budget of N LLM calls, our algorithm selects N AuPairs in a man-
                ner that maximises complementarity and usefulness. We demonstrate the results
021
                of our algorithm on the coding domain for code repair on 4 LLMs across 7 com-
022
                petitive programming datasets. The AuPairs produced by our approach provide a
023             significant boost in performance compared to best-of-N , and also exhibit strong
024             generalisation across datasets and models. Moreover, our approach shows strong
025             scaling with the inference-time compute budget.

028   1    I NTRODUCTION
029
030   Recent progress in the field of Large Language Models (LLMs) has resulted in models that keep get-
031   ting better at generating responses to user queries. When providing these already powerful models
032   with more inference-time compute—increasing number of LLM calls—methods that sample dif-
033   ferent responses and then select the best among them, such as best-of-N (Stiennon et al., 2020)
034   or self-consistency (Wang et al., 2023b), have shown clear benefits. While these approaches are
035   more breadth-focused, another way to leverage inference time compute is to improve or repair the
036
      LLM’s initial guesses by generating better fixes. We propose combining the benefits of both these
      approaches to generate a wide set of repaired solutions for poor initial LLM responses, and then
037
      select the best as final answer.
038
039   To generate a wide range of repaired solutions for each initial LLM response, we exploit the in-
040   context learning capability exhibited by LLMs. The main contribution of this paper is an algorithm
041   that produces a golden sequence of pairs of guesses and fixes, which can each be provided as in-
042
      context example for generating repaired solutions. Each such AuPair consists of the problem de-
      scription, the initial guess, and the consequent fix, along with their respective scores. An example
043
      AuPair is illustrated in Fig. 2. Given an inference-time compute budget of N LLM calls, our al-
044
      gorithm provides an ordered set of N golden example pairs or AuPairs. These AuPairs are used
045   to generate N fixes at inference time, out of which the highest scoring one is selected as the final
046   output response.
047
048
      A core ingredient of our proposed algorithm is the selection of these AuPairs. We propose a submod-
      ular approach based on the ability of each pair to solve different problems in a held-out validation
049
      set. Since the list of AuPairs is constructed by taking the greedy pair at each step, only those pairs
050
      that increase the score of the fix on a subset of problems are selected, resulting in useful AuPairs.

084
      In this paper, we focus on the code repair task: given a coding problem, an initial guess which is
      LLM-generated code, and a set of test cases that are used only to evaluate the correctness of the
085
      generated code, can the LLM generate an improved fix for the problem? We show that the fixes
086
      generated with AuPairs provided as in-context examples are significantly more useful and diverse
087   than those generated using best-of-N (§3) for the same inference-time compute budget.
088
089   The key contributions of this paper are the following:
090
091          • An inference-time algorithm, AuPair, which constructs a golden set of code repair exam-
092
               ples that boost performance significantly when used as in-context examples (§2).
093          • Reliably outperforming best-of-N across 4 different model sizes: Gemma-9B, Gemma-
094            27B, Gemini-1.5-Flash, Gemini-1.5-Pro, and 7 competitive programming datasets (§3.1).
095
             • Strong scaling performance with inference time compute, with far less diminishing returns
096
               than best-of-N (§3.3).
097
098          • Robust out-of-distribution generalisation, w.r.t. both model size and dataset (§3.4).
099          • Demonstrably higher diversity of solutions, without performance trade-off (§3.6).

      This process of iteratively constructing the set of AuPairs ensures that they improve performance
223   on disjoint parts of the problem space. The AuPairs that we obtain from this phase are then used in
224   the same manner at inference time, as 1-shot examples, to improve code repair performance. The
225   compute budget N at inference time determines the number of AuPairs that we can use at inference
226   time. Since the AuPairs form an ordered set, the first N AuPairs are used at inference time for
227   budget N . The final solution for each problem is the best among all generated solutions, i.e., the one
228   that passes the most test cases.

      3   E XPERIMENTS
233
234   Datasets: We use 7 datasets that contain problems and test cases from competitive program-
235   ming contests: 1) CodeForces (8.8k problems), 2) AtCoder (1.3k problems), 3) HackerEarth (1.2k
236   problems), 4) CodeChef (768 problems), 5) LiveCodeBench (400 problems), 6) CodeJam (180
237   problems), and 7) Aizu (2.2k problems) (Li et al., 2022a; Jain et al., 2024). We choose Code-
238   Forces and AtCoder, separately, for in-distribution testing, and use the rest exclusively for out-of-
239   distribution testing. Our train / val / test split proportions for the CodeForces and AtCoder datasets
240   are 37.5/12.5/50%. Some datasets have difficulty levels as part of the problem; for those datasets
241   we maintain the same stratified distribution of questions in the training, validation, and test datasets.
242   Models: We use 4 models of different sizes: Gemma-9B, Gemma-27B, Gemini-1.5-Flash and
243   Gemini-1.5-Pro. In addition to using these models for dataset curation and pair generation, we
244   look at the transfer capabilities of our method with respect to different models in Section 3.5.
245
      Evaluation: We use each AuPair as 1-shot example, in context with the problem text and initial
246   guess in the repair prompt. The structure of the prompt is the same as the one used earlier (A.3).
247   We perform two types of evaluation: in-distribution and out-of-distribution. For in-distribution
248   evaluation, we use the test split from the same dataset as the one used for pair generation and AuPair
249   extraction. This ensures that the format of questions and test cases in the test questions matches
250   that of the AuPairs. Out-of-distribution evaluation uses a different coding dataset; this means that
251   the test samples have different format of questions, difficulty, types of problems and test cases than
252   the AuPairs. Another axis of out-of-distribution evaluation that we look at is the model axis: we
253   report the performance obtained using AuPairs produced by a different model than the one used at
254
      inference time.
255   Metrics: Our primary metric is the best-of-N accuracy, which we calculate as the average of the
256   best response across N LLM calls for all points in the test dataset. In our case, we have grounded
257   feedback available in the form of the number of unit tests passed by each LLM response, when
258   executed using the Python interpreter. We use this feedback to compute the maximum score out
259   of the N generated outputs for our approach and baselines, all of which involve N LLM calls at
260
      inference time for fair comparison. For our approach we pick the first N AuPairs from A.
261   Baselines: We compare the effectiveness of our proposed approach with best-of-N (Stiennon et al.,
262   2020) and self-repair (Olausson et al., 2024). Best-of-N is currently the strongest baseline to im-
263   prove model performance by allowing multiple (N ) LLM calls at inference time. To have a strong
264   best-of-N baseline, we set the temperature to 1.0, to ensure sampling of diverse responses while
265   preserving good quality responses (Renze & Guven, 2024) Self-repair uses the N LLM calls to ei-
266
      ther generate verbal feedback or repaired code. In our experiments, for a budget of N = 32 LLM
      calls, we use 4 LLM calls to generate verbal feedback and 7 LLM calls to generate repaired code for
267
      each verbal feedback.
268

      performance (§3.1 and 3.2), scalability and generalisation (§3.3 to 3.5), and diversity (§3.6 to 3.8).

288   3.1   S IGNIFICANTLY B OOSTED C ODE R EPAIR P ERFORMANCE
289
      The first step to assess code repair performance is to measure in-distribution performance; namely
290
      generate and selecting AuPairs on the training and validation sets that match the test dataset, and
291   using the same model at evaluation as for construction. We do this for 2 datasets (CodeForces
292   and AtCoder) and all 4 models. Fig. 4 shows the resulting comparison between the best-of-N
293   baseline and our AuPair approach, for a budget of N = 32 LLM calls at inference time.4 AuPair is
294   clearly superior to best-of-N on all models and datasets, sometimes by wide margins. This clearly
295   establishes that our proposal of providing a different in-context example of code repair in each LLM
296   call can significantly boost performance.
297   An interesting side-result is visible in initial performance, i.e., the performance of the initial re-
298   sponses of the LLMs to the problems, which have to then be repaired. Gemini-1.5-Pro, despite
299   being a superior model to Gemini-1.5-Flash, shows worse initial performance. Since the code gen-
300   erated has certain conditions that allow successful execution, we observe that many initial guesses of
301   generated code fail because they do not obey these conditions (see Appendix §A.4). In such cases,
302   code repair with best-of-N is unlikely to give us high boost in performance since the initial solution
303   is badly formatted. This is one clear case where having an AuPair in context significantly improves
304   performance. As a result, using AuPairs in conjunction with high performing models leads to large
305
      performance improvements despite poor initial performance, as we can see for both CodeForces and
      AtCoder with the Gemini-1.5-Pro model in Fig. 4.

308   3.2   S ELECTION M ATTERS : AU PAIRS ARE M ORE E FFECTIVE THAN R ANDOM PAIRS
309
      We design an ablation to disentangle the two possible sources of improvement that our approach
310
      demonstrates, namely 1) in-context learning and 2) the choice of AuPairs. It is not implausible for
311   the boost in performance to result from the LLMs’ in-context learning ability, and that the same
312   result could be achieved by including any set of pairs. On the other hand, our approach specifically
313   targets complementarity during construction of AuPairs in that subsequent AuPairs are selected
314   based on their ability to solve problems that previous AuPairs were unable to solve. To resolve
315   this, we compare the full method to a random-pair baseline that randomly selects pairs from the full
316   candidate set (the result of Phase 1), deduplicating the problems that the random pairs solve (which
317   makes it a stronger baseline). Fig. 5 shows that AuPair significantly outperforms the random-pair
318   baseline for N = 1, ..., 32. Note that for any fixed candidate set, as N grows toward the size of the
319
      full set of pairs, the performance of the random-pair baseline will equal that of AuPair.
320       4

321   the total number of AuPairs can be less than 32. To have a fair comparison in that case, we set the same compute

      The aim of this set of experiments is to determine whether our approach exhibits out-of-distribution
372   generalisation, i.e., given AuPairs collected on a different dataset, see if we can retain the perfor-

399
400   Now that we have seen that our approach can exhibit very good out-of-distribution generalisation
401   along the data axis, we evaluate it on its ability to generalise on the model axis, i.e., we look at the
402   performance of AuPairs collected using a different model. We evaluate this cross-model transfer
403   capability for all model combinations on CodeForces. The resulting 16 ablations are shown in
404   Fig. 7(a), and help disentangle the impact of the AuPairs versus the code repair capabilities of the
405
      inference model. A key takeaway is that the Gemma models exhibit worse performance, regardless
      of the quality of AuPairs used at inference time, indicating that they are inferior at the capability of
406
      code repair. Gemini-1.5-Flash performs much better at code repair, and its sensitivity to the source
407
      of AuPairs is negligible: it is equally performant for each source. Gemini-1.5-Pro, on the other hand,
408   is sensitive to the source of AuPairs; in particular, when Gemini-1.5-Pro uses AuPairs collected by
409   the same model, it achieves the best performance by a large margin. With AuPairs selected using
410   other models, Gemini-1.5-Pro achieves comparable performance to Gemini-1.5-Flash. One reason

418   3.6   H IGH C ODE - SPECIFIC D IVERSITY
419
420   We dive a bit deeper into the nature of fixes generated using different AuPairs. There are several
421   ways to analyse code; we choose Abstract Syntax Trees (ASTs) since they mostly capture the struc-
422   ture of changes. More concretely, since we have N fixes for each problem (here N = 32), we
423
      measure the diversity per problem as the number of unique changes made to the guess over all N
      fixes for that problem. The diversity score is calculated as the average number of unique abstract
424
      syntactic subtrees generated per problem. More concretely, we perform the set difference of all
425
      subtrees in the fix AST that are not in the guess AST and normalize with the maximum number of
426   subtrees. We plot this diversity metric against the score in Fig. 7(b) to get a sense of how diverse
427   and useful the AuPairs are. We also include diversity results of the best-of-N baseline, see A.2 for
428   further details on the diversity score computation. The results show that while AuPairs always in-
429   crease performance, they result in higher diversity of fixes when given to the more competent models
430   (Gemini-1.5-Pro and -Flash), and lower diversity for Gemma models. It is worth highlighting that
431   the exceptional performance of AuPairs produced and used by Gemini-Pro (Fig. 7(a), bottom right)
      correspond to highly diverse fixes (Fig. 7(a), top right).

      4     R ELATED W ORK
466
467
      Automatic Program Repair (APR) has been a longstanding research area in the field of machine
468   learning (Devlin et al., 2017; Bhatia & Singh, 2016; Chen et al., 2019; Feng et al., 2020; Berabi
469   et al., 2021; Chakraborty et al., 2022; Yuan et al., 2022). Most methodologies rely on supervised
470   finetuning to adapt LLMs to the task of code generation using labeled pairs of broken / fixed code
471   pairs, which is costly to obtain and often task- and problem-specific (Hu et al., 2022; Jiang et al.,
472   2021; Xia & Zhang, 2022; Dinella et al., 2020). On the other hand, unsupervised APR is challenging
473   since it requires syntactic and semantic understanding of code, and most automatic code breaking
474   approaches tend to be out-of distribution with real samples. Yasunaga & Liang (2021) train both a
475   breaker and a fixer in order to learn to propose new code fixes that are realistic, and uses a compiler
476
      to verify its correctness. Close to our approach we use partial fixes generated by the model as the
      initial broken code to be fixed iteratively.
477
478   More recently, a few unsupervised approaches have been proposed based on the capability of LLMs
479   to generate code (Chen et al., 2021; Nijkamp et al., 2023; Chowdhery et al., 2024; Li et al., 2022b;
480   Fried et al., 2023; Li et al., 2023). The APR task still remains challenging, even though models are
481   better at generating code (Olausson et al., 2024; Chen et al., 2023). Zhao et al. (2024) use a step-by-
482
      step method to repair code using a reward model as a critic, providing feedback to finetune an LLM.
      Shypula et al. (2024) propose a retrieval based few-shot prompting approach with Chain-of-Thought
483
      (CoT) reasoning traces, and use supervised fine-tuning (SFT) to finetune a model using self-play.
484
485   The main disadvantage of using SFT approaches comes from the need to finetune the model to
      the task, which becomes much more costly with ever-growing model sizes. In recent years the in-

      context learning (ICL) paradigm (Brown et al., 2020) has been shown to be a flexible and compute

      efficient adaptation approach to new tasks (Von Oswald et al., 2023; Akyürek et al., 2023). Le et al.

      (2022) use an LLM to generate code and a critic network to predict functional correctness of the

507   correctness is specified by the number of test cases the generated code passes. Gou et al. (2024)
508   combine the use of LLMs with tools to provide feedback for the LLM to self-correct via additional
509   calls to evaluate its own output in a validation setting. Wang et al. (2023a) also make use of external
510   tools and use an LLM in a learner / teacher role to provide a chain of repairs to fix the code.
511
      Yin et al. (2024) propose an automated self-repair approach with few-shot prompting but using CoT
512   and execution feedback information. Agarwal et al. (2024) also use CoT rationales but remove them
513   from context when few-shot-prompting the model. Olausson et al. (2024) show that using an LLM
514   as a feedback source for self repair has its limitations when compared with the same number of
515   independent model calls for the same problem since the ability to generated better code may be
516   interconnected with the ability to identify its faulty behaviour. Welleck et al. (2023) decouple the
517   generation and the correction phase, by independently training a corrector with scalar and natural
518   language feedback to correct intermediate imperfect generations. We use self-corrections, since
519   we use the same model for generating the fixes and the broken code pairs, but the improvement is
520
      grounded on the number of passing tests, avoiding degenerate behaviours.
521   Yuan & Banzhaf (2017) propose a multi-objective evolutionary algorithm to search over possible
522   correct code patches; Romera-Paredes et al. (2023) use an island-based evolutionary method to
523   encourage exploration of diverse programs, and perform iterative best-shot-prompting to improve
524   the quality of the generated code. In this paper, we use a generative approach; closer to the work
525   of Shirafuji et al. (2023), we make use of ICL abilities of LLMs to generate improved code repairs,
526
      but we provide an extra submodular process to select the samples, that encourages diversity.

528
      5   C ONCLUSIONS AND F UTURE W ORK
529
530
531
      We propose an algorithm, AuPair, which produces a set of golden example pairs that can be provided
      as in-context examples using 1-shot prompting to improve code repair performance at inference
532
      time. Our approach is highly scalable, showing significantly better outcomes than best-of-N , which
533
      is the current state-of-the-art method that improves performance as inference compute is scaled
534   up. In addition to this, the AuPairs generated using our algorithm show strong out-of-distribution
535   generalisation and thus can be reused at inference time to solve a wide range of problems. While
536   in this paper we have explored repair in the coding domain, our algorithm is general and can be
537   used in any setting in which an initial solution generated by an LLM can be improved via repair.
538   Additionally, the choice of coding implies that all our feedback is grounded, but our algorithm is
539   general enough to accommodate ungrounded feedback from reward models, so future work merging
      this line of research into the full LLM pipeline might be worth investigating.

      A.3    P ROMPTING
961
962   There are 2 types of prompts that we use: 1) guess generation prompt, and 2) repair prompt. The
963   guess generation prompt is used during dataset creation, for obtaining the initial guesses for all prob-
964   lems in the dataset. The repair prompt is used throughout the rest of the paper: in the Pair Generation
965   (Phase 1, §2.1 with k = 32 random examples) and in the AuPair Extraction (Phase 2, §2.2). The
966   function signature indicates that the function expects a string as an input. The instruction specifies
967   that the final answer is meant to be printed inside the function, and that the main function is not
968
      meant to be written.
969   The structure of our repair prompt is as follows: there is a generic instruction at the top, followed by
970   the few-shot examples in the format of question, followed by the guess or bad solution, followed by
971   the fix or good solution. We also add the score achieved by the guess and the fix for the in-context
      example pairs. Following this, we add the text and initial guess solution for the problem and the


                                                          18
       Under review as a conference paper at ICLR 2025


972
       LLM then has to generate a better fix. Note that we do not provide any extra execution feedback in
973    the form of execution traces; this could potentially be explored by future work. Our aim is clear:
974    the pairs indicate a certain type of change and we provide these pairs in context to aid the LLM in
975    generating an improved solution for the given problem. Some different prompting strategies that we
976    tried out were the following:

1038
1039   When the LLM generates a fix for any problem, we
1040   call the solve() function for each test case associ-
1041   ated with that problem. We then compare the output
1042   with the ground truth and give a partial score corresponding to the proportion of test cases passed
1043
       by this fix.
1044   An important point to note is that the solve() function has to take as input a string, which is then
1045   parsed into the correct variables. This formatting requirement is a key reason for the poor initial and
1046   best-of-N performance of Gemini-1.5-Pro in Fig. 4. Since the instruction for generating the initial
1047   guess is not correctly followed by the model, a lot of guesses end up invariably having incorrect
1048   parsing of the input, leading to low scores. A lot of AuPairs extracted using these models, as a
1049
       result, contain this formatting fix, as we will see in Section A.5.
