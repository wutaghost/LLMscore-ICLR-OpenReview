010                                              A BSTRACT
011
012            Large language models (LLMs) have the potential to revolutionize how we design
013            and implement compilers and code translation tools. However, existing LLMs
014            struggle to handle long and complex programs. We introduce LEGO-Compiler,
015
               a novel neural compilation system that leverages LLMs to translate high-level
               languages into assembly code. Our approach centers on three key innovations:
016
               LEGO translation, which decomposes the input program into manageable blocks;
017
               annotation-based Chain-of-Thoughts, guiding LLMs through the compilation pro-
018            cess with LLM-annotated context; and a feedback mechanism for self-correction.
019            Supported by formal proofs of code composability, LEGO-Compiler demonstrates
020            high accuracy on multiple datasets, including over 99% on ExeBench and 100%
021            on industrial-grade CoreMark, and successfully handles programs far exceeding
022            the length limitations of native LLM translation. This work opens new avenues for
023            applying LLMs to system-level tasks, complementing traditional compiler tech-
024            nologies.

      1   I NTRODUCTION
027
028
      The rapid development of Large Language Models (LLMs) has led to an expansion of their applica-
029
      tions and effectiveness across various domains (Rombach et al., 2022; OpenAI, 2023; 2024; Ziegler
030   et al., 2024). One important area where LLMs have shown impressive results is code translation,
031   including tasks such as code generation from natural languages (Zan et al., 2023) and transformation
032   between programming languages (Yang et al., 2024). In code translation, LLMs have demonstrated
033   remarkable accuracy and readability, often surpassing manually crafted translators.
034
      While LLMs have shown promising results in translating between high-level programming lan-
035
      guages (Rozière et al., 2020; Roziere et al., 2021; Szafraniec et al., 2023) and in decompilation
036   tasks (Fu et al., 2019; Cao et al., 2022; Armengol-Estapé et al., 2023), their application to trans-
037   lating from high-level languages to low-level assembly languages remains a relatively unexplored
038   area. This can be attributed to two main factors. Firstly, the dominance of traditional compilers
039   in this domain has left little incentive for exploring alternatives in such a mature field. Secondly,
040   the complexity of compilers and the bitwise precision required in compilation tasks for semantic
041   accuracy have made it challenging for LLMs based on statistical learning.
042   Despite these challenges, LLMs have shown promising capabilities in compilation-related tasks.
043   Cummins et al. (2023; 2024) has demonstrated their proficiency in optimizing compiler options and
044   their excellent ability to mimic compiler code behavior, producing high-quality IR code. Further-
045   more, preliminary explorations in translating from high-level languages to assembly languages, such
046   as C-x86 (Armengol-Estapé & O’Boyle, 2021) and C-LLVM IR (Guo & Moses, 2022), have indi-
047   cated the potential feasibility of using LLMs in compilation tasks. However, these existing works
048   have not fully addressed the boundaries of LLM capabilities in compilation tasks – specifically, what
049   LLMs can and cannot do in this domain. Compilation is typically divided into two main aspects:
050
      functionality and optimization. This work focuses on exploring and answering questions about LLM
      capabilities in the functionality aspect of compilation.
051
052   LLMs are pre-trained on vastly large code corpora. some are monolingual, and some may be bilin-
053   gual (where LLMs can learn the translation rule between two languages). However, most of these
      LLMs do not disclose their training datasets, so their capabilities can only be assessed through

054
      empirical testing. We primarily find that LLMs learn the neural compilation process from directly
055   compiler-generated bilingual corpora, which is a relatively easy way to construct pretraining dataset.
056   However, we found that assembly code directly generated by traditional compilers is hard to learn
057   for LLM-based generation due to several challenges. These include the presence of semantically
058   opaque labels, symbols or numeric values that LLMs struggle to translate accurately, and the need
059   to handle symbol renaming for identifiers with the same name in different scopes, etc. Although
060   style migration or modifications to existing compilers can be made, these approaches still rely on an
061   existing compiler to perform the neural compilation job, which doesn’t outperform existing designs.
062   Our work takes a different approach where we do not require bilingual corpora. As a result, we don’t
063   rely on an existing compiler. Regarding LLMs’ strong in-context learning abilities (Min et al., 2022;
064   Song et al., 2024), we propose the following methods: through high-quality examples and compiler
065   knowledge guided Chain-of-Thoughts, LLMs can perform step-by-step neural compilation. This
066   approach involves generating annotations highly corresponding to source code statements and data
067   structure layout annotations, leading to substantial improvements in the compilation generation task.
068   More importantly, the scalability of current code translation is also a big problem. Although ad-
069   vanced LLMs already have hundreds of thousands tokens context limit, they can not merely com-
070   pile a code with 2.6k tokens in CoreMark (Gal-On & Levy, 2012), which is just a 200-LOC func-
071   tion. To address this limitation, we have an intuitive thought: can we split the program into finer-
072   grained components, compile each component, then assemble them together? We can surely do it
073   in function-level, since functions are trivial semantic units (Ibrahimzada, 2024). However, we still
074   think function-level is still too coarse-grained, and we seek to further break it.
075   Based on these insights, we propose a novel approach called LEGO translation, which draws inspi-
076   ration from the modular and composable nature of LEGO blocks. This method breaks down large
077   programs into manageable, semantically-composable control blocks, analogous to LEGO pieces.
078   These blocks are then independently translated and rebuilt to form a much larger scale transla-
079   tion. We apply the LEGO translation method to the compilation domain and, guided by a series of
080   compilation-specific expert knowledge, design the LEGO-Compiler, a scalable, LLM-driven system
081
      that leverages the power of LLMs to perform neural compilation tasks.
082   LEGO-Compiler can correctly compile over 99% of the code in ExeBench (Armengol-Estapé et al.,
083   2022), a large scale dataset through careful unit-testing. We can also correctly compile 100% Core-
084   Mark (Gal-On & Levy, 2012), an industrial-grade code that encompasses most common program-
085   ming language features in C. Regarding scalability, we have verified that LEGO translation method
086   can significantly scale up the capability of code translation/compilation performed by LLMs, where
087
      we propose the LongFunction dataset for very long code translation and compilation evaluation,
      and the LEGO translation method can sufficiently translate it for both neural compilation or code
088
      translation purposes.
089
090   The main contributions of this work are as follows:
091
092
             • We propose a set of novel methods for neural compilation, including LEGO translation for
093            breaking down large programs, annotation-based Chain-of-Thoughts (CoTs) that explicitly
094            generate intermediate results to aid translation, and a feedback-driven self-correction mech-
095            anism. These methods collectively address the challenges of applying LLMs to complex
096            compilation tasks.
097
098          • We introduce LEGO-Compiler, a comprehensive neural compilation system that integrates
099            our proposed methods. LEGO-Compiler incorporates control flow annotation, struct an-
100            notation, and variable mapping to ensure accurate and scalable compilation across various
101            architectures and programming languages.
102
103          • We provide both theoretical and empirical support for our approach. We present a formal
104            proof of code composability that underpins the LEGO translation method. Empirically,
105            we demonstrate LEGO-Compiler’s effectiveness through extensive evaluations, achieving
106            over 99% accuracy on ExeBench and 100% accuracy on the industrial-grade CoreMark
107            benchmark. Our system successfully handles programs far exceeding the length limitations
               of direct LLM translation, showcasing its scalability.

132   2.1        P ROBLEM D EFINITION
133
134   Before introducing our method, we first define the neural compilation problem. Neural compilation
135   can be viewed as a specialized version of code translation problem, as defined in Definition 1, with
136
      the goal of translating high-level programming language as the src language (such as C) into low-
      level assembly language as the dst language (such as x86, ARM, or RISC-V). Unlike general code
137
      translation, compilation needs to handle more low-level details, such as memory layout and calling
138
      convention, while ensuring the functional correctness of the translated result.
139
140   Definition 1. There are two programming languages: Lsrc and Ldst , each is an infinite set of valid
141
      program strings. There exists a unary relation ⇀ from Lsrc to Ldst . The problem is to perform a
      translator function T : ∀x ∈ Lsrc , (∃u ∈ Ldst , x ⇀ u) → (x ⇀ T (x)), T (x) ≡ x semantically.

144   2.2        LEGO T RANSLATION : C ORE M ETHOD
145
146
      As depicted in (a) in Figure 1, previous neural code translation methods typically convert entire
      programs at the function or file level. While this approach may be effective for smaller programs,
147
      it struggles with larger programs due to significant accuracy degradation. These methods translate
148
      code at a coarse granularity, making it challenging to translate very long functions using LLMs.
149   Taking neural compilation as an example, all current LLMs fail to compile a C function with larger
150   than 2.6k tokens using direct translation, although some advanced LLMs already have 128k-200k
151   context limit. They could also perform code-snippet level translation, but they lack guidelines and
152   necessary information to compose the code-snippet level results together, and there is also no clear
153   formal proof to the composability of code. Despite these limitations, we observe an inherently
154   composable nature in code. In the context of neural compilation, we propose the following insights
155   to enhance translation scalability:
156
157
                   • Fine-grained translation: Instead of translating an entire program at once, focus on trans-
                     lating smaller code snippets accurately. By ensuring each part is correctly translated, they
158
                     can be combined to form a semantically equivalent complete translation.
159
160                • Contextual Awareness: Effective translation of smaller code snippets requires understand-
161                  ing their contextual positioning within the code. This includes recognizing the relationship
                     with preceding and succeeding snippets to maintain semantic coherence.


                                                                                                3

162
              • Symbol Handling: Accurate translation involves reasoning about necessary symbols and
163             constructs within each snippet to enhance alignment with the target language’s syntax and
164             semantics, thereby aiming to preserve the intended functionality.
165
166   Inspired by Wang et al. (2024), where this process is similar to the destruct and rebuild process of a
167   LEGO toy, we named the fine-grained translation technique as LEGO translation and our system
168   built upon it as LEGO-Compiler. As depicted in (b) in Figure 1, LEGO translation first breaks
169   down large programs into manageable, self-contained blocks, analogous to LEGO pieces (Part
170   split). Then these blocks are independently translated (Part translation) and finally recombined,
171
      enabling scalable and accurate translation of complex programs (Part rebuild). All these methods
      rely on an inherently nature in programming languages, the composability in control block level,
172
      which reflects the linearization process in compiler design (Wirth et al., 1996), where tree-structured
173
      control flow can be linearized, and therefore, composable. We have formally proved the widely
174   applicable composability of programming languages using a constructive approach in Appendix B.

      2.3.1    C ONTROL F LOW A NNOTATION
204
205   The first annotation process addresses the positioning issues. Control flow in high-level languages
206   consists of structures like if, while, for, and switch statements, which are linearized into branch-label
207   constructs in assembly. In modern programming languages like C, most control statements (except
208   goto) are encapsulated, meaning their generated labels remain within their scope. This property
209   makes C programs composable at the control block level, which is formally proved in Theorem 5.
210   Control flow annotation is where Part Split is performed. Algorithm 1, which is also inspired by
211   the composability proof algorithm in Figure 2, describes how to use LLM to split program into
212   reasonably sized blocks. For small programs or control statements with low nesting levels, splitting
213   may be unnecessary. For deeply nested control statements, further splitting may be required. This
214   process maintains composability and encapsulation, ensuring correctness when rebuilding the full
215   translation. In extreme cases, the program can be divided into basic blocks (Definition 3) or even
      sequential statements (Definition 2).

      2.3.2    S TRUCT A NNOTATION
240
      The C language type system includes numerous basic types based on integer or float, their corre-
241
      sponding pointers, and compound types such as struct, union, and array, composed of basic types.
242   Translating basic types and their instructions is relatively simple, as LLMs have learned this knowl-
243   edge through extensive pre-training. However, for compound types, like structs (and similarly
244   unions), the challenge arises from the infinite possible combinations of basic types.
245
      To address this, we adopted a Chain-of-Thought approach. Instead of directly memorizing the map-
246
      pings of variables from source-assembly language pairs in pretraining stage, we prompt the LLM
247
      to perform a separate thought process to reason about the memory layout of compound types based
248   on structure, which includes size, offset, and alignment. Since compound types are ultimately com-
249   posed of basic types, and LLMs understand basic type memory layouts, this pass can effectively
250   infer the memory layout of compound types, like struct and array.
251
      The Struct Annotation result is also verifiable using front-end tools like IntelliSense (Microsoft Cor-
252
      poration, 2024) or Clangd (LLVM Project, 2024b). After Struct Annotation, we obtain the symbol
253
      attributes for each type in the symbol table (LLM’s context).
254
255
256   2.3.3    VARIABLE A NNOTATION
257
258   After type attribute inference, we need to determine where each type appears in the program and
259   which variable identifier represents it. Our designed prompt guides the LLM to infer all declarations
260
      and arrange variable stack allocations according to their declaration order
261   Using x86 assembly as an example, global variables are stored in the data segment and indexed by
262   same-name labels, providing clear binding relationships. For local variables, our method involves
263   treating them as stored on the stack, assigning each a specific offset relative to a base address.
264   By iterating through variable definitions, we can update these offsets relative to the base address,
265   achieving effective stack allocation in most cases.
266   For compound type variables like structs, following the System V ABI (2018) for x86 assembly,
267   access to sub-elements is achieved by adding offsets. This process is generally accurate with LLMs,
268   relying primarily on precise binary integer arithmetic operations. After the Variable Annotation
269   pass, we obtain a correspondence between variables in C and assembly languages, allowing for
      simple substitution during compilation, as illustrated in Figure 2.

271   this through a renaming pass at the source program level, eliminating name conflicts and ensuring

275   2.3.4    S ELF - CORRECTION THROUGH ERROR FEEDBACK
276
277   To address potential errors in the LLM-generated code, we implement a comprehensive self-
278
      correction mechanism. This system classifies errors into three categories: assembly semantic errors
      (detected by the assembler), runtime errors (identified through execution and caught by debuggers
279
      like gdb), and behavioral errors (discovered through result comparison). The error information
280
      is collected and fed back to the LLM for self-correction. Assembly semantic errors are typically
281   straightforward to fix, while runtime errors, often caused by null pointer dereferences, are addressed
282   by tracing instructions step-by-step to pinpoint the problematic area. Behavioral errors, being the
283   most complex, may require multiple iterations to resolve. This iterative feedback and correction
284   process significantly enhances the robustness and accuracy of the LLM-based compilation system,
285   which is evaluated in the following section.
286
      In general, LEGO-Compiler uses Chain-of-Thoughts in neural compilation task by either explicitly
287   annotating the source code or generating intermediate results. These annotations and results are
288   stored in the LLM’s context, allowing the model to integrate them effectively. Through in-context
289   learning, LEGO-Compiler is able to perform neural compilation tasks step by step, with each step
290   being a simpler subtask that the LLM can handle. For more details, see subsection D.1.

352
              viding error feedback, LLMs will reflect on their generation and focus on solving the er-
353           rors. In comparison, assembler feedback is the most efficient feedback message, because
354           it directly points out the errors; runtime feedback is helpful as well, though LLMs need

356           since it lacks clear information about where is wrong in the hypothesis, although it’s some-
357           how helpful, LLMs’ guesses on which part is problematic are usually wrong. In general,
358           by enabling the feedback-driven LLM self-correction method, all models get significant
359           improvement on their Pass@5 accuracy. Typically, the advanced LLMs majorly solve the
360           simple and medium code, while the mini LLMs will still face some problems in simple
361
              cases.

496   3.4   L ONG F UNCTION E VALUATION
497
498   Since the LEGO translation method significantly scales up the capability of long code translation,
499   we design LongFunction, a synthesized dataset for testing very long code translations, particularly
500   for evaluating the effectiveness of our LEGO translation method. The dataset is made up of 50 syn-
501   thesized programs, ranging from 317 to 238737 tokens in length, each program is self-contained and
502   can be compiled and run independently. We evaluate the neural compilation task on LongFunction
503
      dataset for all x86 64, arm-v8a and riscv64 architectures, and the neural code translation task by
      migrating C to Python/C++/Rust.
504
505   Examples of LongFunction and the evaluation details can be found in subsection D.3 due to page
506   limits. In conclusion, our proposed LEGO translation method breaks the complexity of long code,
507   boosting their capability of handling long code, where all LLMs we tested passed the whole Long-
508   Function dataset. In comparison, the current best model, o1-preview, can only maximally translate
509
      a 5772 token sized case using direct translation method.
510   The results of LongFunction dataset evaluation give us a strong insight: Code, or programming
511   languages, unlike the natural languages, no matter how long they are, their complexity can be divide-
512   and-conquered into two levels: The first is on the control flow level, which combines each block of
513   code logic together to form a long and complicated code, where both current compilers and our
514   LEGO-Compiler methods can iteratively split the code into smaller and smaller, managable code
515
      snippets to overcome this complexity. The second is on the handling of each code snippet, where
      the size is not large, and the major difficulties are on how to correctly translate each statement of
516
      it correctly, where we also find out that for these advanced LLMs, they perform strongly on this
517
      level. Since the basic operations are also limited in programming languages, within proper size,
518   the code snippet level translation or compilation can be gradually improved to near 100% with the
519   advancement of more powerful pretrained LLMs.

525
      We have presented LEGO-Compiler, a novel approach to neural compilation that leverages Large
      Language Models (LLMs) to translate high-level programming languages into assembly code. Our
526
      LEGO translation method breaks down large programs into manageable, self-contained blocks
527
      through the composable nature of code, significantly extending the scalability of neural code trans-
528   lation. By incorporating a series of Chain-of-Thought stages guided by classical compiler design
529   and self-correction mechanisms, LEGO-Compiler effectively addresses key challenges in compi-
530   lation tasks, achieving significant improvements in accuracy and scalability. Our experimental re-
531   sults demonstrate the effectiveness of LEGO-Compiler, as it achieves over 99% accuracy on the
532   ExeBench dataset and fully compiles the industrial-grade CoreMark benchmark correctly. We
533   also introduce LongFunction, a new dataset designed to evaluate the translation and compilation
534   of lengthy code, demonstrating the effectiveness of the proposed LEGO translation method.
535   These findings provide important insights into the capabilities and limitations of LLMs in neural
536   compilation tasks. While our current implementation incurs higher computational costs compared
537   to traditional compilers, it offers unique advantages in adaptability and potential for rapid integration
538   of new instruction sets or language features. As LLM capabilities continue to improve, approaches
539   like LEGO-Compiler are poised to play an increasingly important role in the future of software
      development and compilation, complementing and enhancing traditional compiler technologies.

      A.1   C ODE T RANSLATION
867
868   Code Translation has evolved from traditional statistical methods (Nguyen et al., 2015b) to neural-
869   based approaches that capture programming language structures (Chen et al., 2018). Current
870   neural code translation researches can be majorly categorized to two types: learning-based tran-
871   spilers (Rozière et al., 2020; Roziere et al., 2021; Wen et al., 2022) and pre-trained language mod-
872   els (Feng et al., 2020; Wang et al., 2021; Lu et al., 2021; Rozière et al., 2022; OpenAI et al., 2023;
873   Anthropic, 2023). The former majorly studies the scarcity of parallel corpora (Xie et al., 2023) and
874   develops unsupervised learning methods to overcome it. The latter using Large Language Models’
875
      vast pretrained knowledge, can also perform code translations well without training (Yang et al.,
      2024; Liu et al., 2024a).
876
877   Analysis of neural code translation is equally crucial. Studies have examined common pitfalls in
878   language models for code intelligence (She et al., 2023; Jiao et al., 2023), investigated the reliability
879   and explainability of these models in automated program generation (Liu et al., 2024b), and the
880   attention paid by LLM during code generation that differs from human (Kou et al., 2024).
881   As for compilation related translations, Armengol-Estapé & O’Boyle (2021) first gives a try on
882   neural compilation. Guo & Moses (2022) further studies on C-to-LLVM IR translation. However,
883   they only perform limited investigations on the methods, and their results are still preliminary. There
884   are also works on the reverse decompilation process (Fu et al., 2019; Cao et al., 2022; Armengol-
885   Estapé et al., 2023) and works on code optimizations (Cummins et al., 2023; 2024). The most related
886   work is Zhang et al. (2024), which achieves state-of-the-art 91% Pass@1 accuracy compared to non-
887
      LLM methods on the C-x86 task using a finetuned CodeLlama model, which our work surpasses.
      Besides, their approach relies on compiler-generated bilingual corpora, while our methods can ef-
888
      fectively eliminate such dependency by reasoning the steps of how a compiler works.
889
890   Finally, the breakdown of neural code translation is also less studied, Nguyen et al. (2015a)
891   first breaks the translation of Java-C# into syntaxemes level to lower the translation difficulty in

1252   The proof of composability presented in this paper is based on a simplified model of C-like languages
1253   and unoptimized translation. It’s important to note several key points about the scope and limitations
1254   of this proof:
1255

1256
                translation process do not compromise the validity of the proof. The core of our argument
1257            relies on the decomposition of programs into control blocks and the composability of these
1258            blocks. The internal structure of basic blocks, while important for actual compilation, does
1259            not affect the composability principle we’ve established.
1260
             2. Unoptimized Translation: Our proof assumes a straightforward, unoptimized translation
1261            process. This assumption is crucial for maintaining the direct correspondence between
1262            source code structures and their translations.
1263
             3. Limitations for Complex Language Features: The composability principle as proved
1264            here can be applied to C-like languages, but may not hold for more complex language
1265            features. For example:
1266
                   • Exception Handling: Languages with sophisticated exception handling mechanisms,
1267
                      such as Python, introduce complexities that can break composability. These mecha-
1268                  nisms often require:
1269
                        – Guarded execution of code blocks.
1270                    – Runtime type information (RTTI) for determining appropriate exception handlers.
1271                    – Non-local control flow that can’t be easily decomposed into our model of control
1272                      blocks.
1273               • Coroutines and Generators: Features that allow for suspending and resuming execu-
1274                  tion mid-function can introduce state that is not easily captured in our model of control
1275                  flow.
1276               • Reflection and Metaprogramming: Languages that allow for runtime modification of
1277                  program structure or behavior can invalidate static composability assumptions.
1278            Although not applicable to some specific language features, it doesn’t mean the compos-
1279            ablity and its derived LEGO translation method is not applicable to the whole programming
1280            language, as long as these features are not used in the code, the composability will still stand
1281            and the LEGO translation will still work.
1282         4. Optimizations Across Basic Blocks: Our proof assumes that the boundaries of control
1283            blocks are respected in the translation process. However, many real-world compiler opti-
1284            mizations operate across these boundaries. Examples include:
1285               • Loop unrolling
1286
                   • Function inlining
1287
                   • Global value numbering
1288
                   • Code motion optimizations
1289
1290
                Such optimizations can reorder, eliminate, or combine operations from different control

1291
1292         5. Applicability: Despite these limitations, the composability principle proved here is valu-
1293            able for:
1294               • The foundation of LEGO translation method, the proof reveals the composable na-
1295                  ture of code in at least control block level, which is a major difference than natural
                      languages.

1324   C.2   M ANAGING H IGHLY C OMPLEX E XPRESSIONS
1325
       One of the primary challenges in neural compilation arises when dealing with expressions or state-
1326
       ments of high complexity. In such cases, LLMs struggle to accurately evaluate these expressions
1327   through next token prediction. To address this, we propose two solutions:
1328
1329          • External Tool Integration: We can utilize external parsing tools to generate tree structure
1330            information for complex expressions evaluation. This tree structure is then provided to the
1331            LLM, offering an explicit traversal order and guiding the evaluation process.
1332
              • Expression Decomposition: Without relying on external tools, we can design a new pass
1333            where the LLM identifies high-complexity expressions and rewrites them as a combination
1334            of lower-complexity expressions. This approach ensures that the entire program consists
1335            only of expressions within a proper LLM’s evaluation capabilities.

       E    L IMITATIONS
1783
1784
       Optimization Capabilities: The current focus of LEGO-Compiler is on functional correctness
1785   rather than code optimization. Traditional compilers excel at producing highly optimized code, a
1786   capability not yet matched by our neural approach. Future work could explore integrating optimiza-
1787   tion techniques into the neural compilation process.
1788
       Performance Overhead: As noted in the discussion, the computational cost of neural compilation
1789
       is significantly higher than traditional methods. This limitation may restrict its practical application
1790
       in scenarios where compilation speed is critical.
1791
1792   Complex Expression Handling: The paper acknowledges challenges in managing highly complex
1793   expressions, proposing external tool integration or expression decomposition as potential solutions.
1794
       This indicates a current limitation in LLMs’ ability to handle intricate code structures independently.
1795   Architecture-Specific Knowledge: While the paper demonstrates success with x86, ARM, and
1796   RISC-V architectures, expanding to a broader range of architectures, especially more specialized
1797   ones, may require significant additional training or fine-tuning of the LLMs, or by providing large
1798   RAG database to provide such knowledge in the context.
1799   Security and Reliability: The stochastic nature of LLM outputs raises concerns about the consis-
1800   tency and security of the generated assembly code. Ensuring deterministic outputs and preventing
1801   potential vulnerabilities introduced by the neural compilation process remains a challenge.
1802
       Handling of Language-Specific Features: The paper primarily focuses on C-like language com-
1803
       pilation, and proves the availability of functionality in neural compilation through both theoretical
1804   and empirical results. However, extending the approach to other programming languages can result
1805   in more tailored problems, for example:
1806
1807          • RAII idiom: Languages with class properties, like C++, have an important programming
1808            idiom called Resource Acquisition Is Initialization(RAII), which pose significant chal-
1809            lenges for LLMs. For instance, constructor and destructor functions in these languages are
1810            implicitly called based on scope. This implicit behavior is difficult for LLMs to accurately
1811
                model and implement in assembly code, but this could be solved using external mangling
                tools like c++filt (Free Software Foundation, 2023).
1812
1813          • Name Mangling:Languages like C++ and Rust use name mangling mechanisms for func-
1814            tion overloading and template instantiation. This requires special handling of global sym-
1815
                bols such as function names during compilation, which may be challenging for LLMs to
                consistently implement without explicit training on these concepts.
1816
1817          • Dynamic Language Features: Some language features violate the composability principle
1818            that LEGO translation relies on. For example, Python’s exception handling mechanism,
1819
                which can cross scope boundaries, would make the LEGO translation method ineffective
                for such features.
1820
1821   It’s important to note that many of these challenges are not unique to neural compilation. Traditional
1822   compilers also struggle with highly dynamic features like exception handling and Run-Time Type
1823   Information (RTTI). Languages like Python achieve their flexibility by sacrificing native code gen-
1824   eration in favor of interpretation or JIT compilation. Therefore, these limitations are not specific to
1825   our work but rather inherent to any approach based on static compilation analysis.
1826
       The ability to handle these diverse language features represents an area for future research in neural
1827   compilation. It may require developing specialized techniques or combining neural methods with
1828   traditional compiler approaches to address these complex language-specific challenges.
1829
1830
       Scalability to Very Large Codebases: While the LEGO translation method significantly improves
       scalability, handling entire large-scale software projects or operating systems may still be beyond
1831
       the current capabilities of this approach. However, It is noteworthy that repository complexity is nat-
1832
       urally reduced into files or functions, therefore, LLM-based compilers and translators are potential
1833   to translate them with more advanced models and more carefully designed methods.
