009                                                   A BSTRACT
010
               Recently, Large Language Models (LLMs) attained impressive performance in math and
011
               reasoning benchmarks. However, they still often struggle with multi-step reasoning which
012
               is relatively easy for humans. To further investigate this, we introduce a new benchmark,
013            SearchBench, containing 11 unique combinatorial problems that avoid training contamina-
014            tion (each equipped with automated pipelines to generate an arbitrary number of instances)
015            and analyze the feasibility, correctness, and optimality of LLM-generated solutions. We
016            show that even the most advanced LLMs fail to solve these problems end-to-end in text,
017            e.g., GPT4 and o1-preview respectively solve only 1.4% and 18.6% correctly. SearchBench
018            problems require considering multiple pathways to the solution and backtracking, posing a
019            significant challenge to auto-regressive models. Instructing LLMs to generate code that
020            solves the problem helps only slightly. We next introduce an in-context learning approach
021
               that prompts the model to implement A*, an informed search algorithm, to comprehensively
               traverse the problem state space, improving the performance of models. We further extend
022
               this approach and propose the Multi-Stage-Multi-Try inference method which breaks down
023
               the A* algorithm implementation into two stages and auto-verifies the first stage against
024            unit tests, raising GPT-4’s performance above 57%.

028   1   I NTRODUCTION
029
030   The advent of Large Language Models (LLMs) has revolutionized the field of natural language processing,
031   with models like Llama3.1 (Meta, 2024), GPT4 (OpenAI, 2023), and o1-preview (OpenAI, 2024) demon-
032   strating unprecedented performance on math and science QA benchmarks, such as GSM8k (Cobbe et al.,
033
      2021) and GPQA (Rein et al., 2023). However, LLMs still exhibit surprising failures on some intuitive
      tasks (Bian et al., 2023; Qin et al., 2023; Marcus, 2020) and struggle with multi-step compositional reasoning,
034
      combinatorial problems, and planning (Dziri et al., 2024; Valmeekam et al., 2022; Wu et al., 2023). Inspired
035
      by these observations and to further investigate LLMs’ reasoning abilities, we offer a new benchmark of
036   search problems, SearchBench. SearchBench is designed to evaluate the general reasoning capabilities of
037   language models in performing search and backtracking to previous search states as part of it.
038
039
      SearchBench is challenging to LLMs due to several factors. Current LLMs have an autoregressive architecture
      that forces them to solve problems sequentially, posing challenges for tasks that require backtracking (Dziri
040
      et al., 2024). Moreover, natural language is less suited for describing or updating accurate representations
041
      of complex intermediate states. Lastly, the number of feasible states in combinatorial problems grows
042   exponentially with its size. Our empirical results show that even the most capable models can solve less
043   than 20% of SearchBench problems. In order to successfully solve SearchBench, a model must backtrack to
044   correct errors, consider multiple chains of reasoning, and determine the most optimal outcome among the
045   many feasible options. These capabilities are required for robust reasoning, making SearchBench a valuable
046   benchmark for evaluating LLM reasoning capabilities as they continue to evolve.

073   SearchBench has five problem categories: (i) pathfinding, (ii) puzzles, (iii) subset sum, (iv) sorting, and (v)
074
      under-determined systems; further divided into 11 unique problem types. Each problem type is inspired
      by known puzzles and combinatorial problems but augmented with modified rules to ensure substantial
075
      differences from similar problems LLMs encountered during their training. We generate ∼100 instances of
076
      varying difficulty per problem type using an automatic pipeline, resulting in 1107 fixed problem instances
077   in total. Each problem type in SearchBench is equipped with an automatic pipeline that evaluates LLM-
078   generated solutions on three dimensions: feasibility, correctness, and optimality. Feasibility checks whether
079   the actions taken follow the problem’s rules; correctness verifies if a feasible solution reaches the goal state;
080   and optimality checks if the least cost solution was found.
081
      To alleviate backtracking bottleneck of LLMs, we introduce A* prompting that uses code execution to
082
      find the solution. This method offloads some of the non-linear computations involved in searching the
083   state-space from the model. Using this method, the task of the model is changed to devising a A* search
084   strategy (Wikipedia, a), which has advantages over other search algorithms that are either computationally
085   inefficient (BFS) (Wikipedia, b) or do not guarantee an optimal solution (DFS) (Wikipedia, c).
086

087
      coding it without any errors. Our experiments show that the model often makes coding mistakes, such
088

091   two steps. First, we prompt the model to write a general A* algorithm for the problem type. Here, we verify
092   the the A* implementation against a set of unit tests: (i) the code is executable; (ii) it returns a list as output;

      2     S EARCH B ENCH B ENCHMARK
129
130   SearchBench includes five categories of problems: puzzles, subset sum, sorting, pathfinding, and under-
131   determined systems. In theoretical computer science, combinatorial problems are classified into four types:
132   existence, construction, enumeration, and optimization problems (Wilson, 2016). To ensure broad representa-
133   tion, we selected one problem category from each of these types for SearchBench. Particularly, subset sum
134   problems represent the existence category, where the task is to determine if a subset of a given set sums to a
135   specified value (refer to Tab. 1 for an example problem in this category). The 8-puzzle and 8-puzzle words
136   fall under construction problems, which involve solving puzzles. Sorting problems, such as color sort and
137   restricted sorting, are enumeration problems. Pathfinding problems are categorized as optimization problems.
138   Additionally, we introduce a new category of NP-hard combinatorial problmes in SearchBench, under-
139   determined system problems. These problems consist of constraint satisfaction problems which are typically
140   solved by defining a system of linear equations, and do not require search over states. We modified them

                                                                                       3
      Under review as a conference paper at ICLR 2025


141
      to include fewer constraints than unknown variables, allowing for multiple correct solutions, and defined
142
      a unique cost function to enable search for a single optimal solution. This category was added in order to
143   evaluate models’ ability to generalize to novel combinatorial problems.
144
145   We selected 2-3 problem types for each category, resulting in 11 total problem types. Each type has a unique
146
      state space. For example, in 8-puzzle words, each state is an n × m table of characters, while in coin exchange,
      each state is an ordered subset of given coins (See Appendix sec. G for more examples). Generally, our
147
      problems involve an initial state, a goal state, and a set of possible actions, and the task is to find a sequence
148
      of actions from the initial to the goal state with minimum cost. We modified the rules to ensure that solved
149   instances of SearchBench were not encountered during the LLMs’ massive internet-scale training. The
150   SearchBench taxonomy and rule modifications are illustrated in Fig. 1.
151
152
      To construct SearchBench, we implemented an automatic generation pipeline for each problem type, ensuring
      each generated instance is solvable. We generated approximately 100 instances per type, resulting in a total
153
      of 1107 problem instances. The benchmark is then fixed. The generation pipelines can create instances
154
      with adjustable difficulty levels. Difficulty is defined by the state space size of the instance, with minimum
155   difficulty requiring a few actions and maximum difficulty set such that problems could be solved correctly but
156   not optimally by humans (See Appendix Sec. F for an analysis of the search space size). Hence, maximum
157   human performance on SearchBench could be considered approximately 100%. Moreover, studies like
158   Pizlo & Li (2005); Chronicle et al. (2006) show that humans can solve the classic versions of SearchBench
159   problems, but their performance declines as the state space size increases.
160
      In contrast to other reasoning benchmarks (Saparov & He, 2022; Cobbe et al., 2021; Hendrycks et al.,
161   2021; Patel et al., 2021; Clark et al., 2020; Tafjord et al., 2020; Sap et al., 2019; Le et al., 2019) that only
162   measure correctness, to gain a more comprehensive understanding of LLM performance on SearchBench, our
163   evaluation pipeline assesses LLM solutions across 3 dimensions: Feasibility, Correctness, and Optimality.
164   Feasibility determines if any of the actions chosen violate the problem rules (e.g. passing through labyrinth
165   walls). Correctness requires that the solution is both feasible and reaches the goal state from the given
166   start state. Optimality indicates that the solution is both correct and has the minimum cost w.r.t. known
167   optimum. For each SearchBench problem, we implemented a fast A* algorithm with a provably admissible
168   and consistent heuristic, to produce the optimal solution. We ran this implementation for each instance in the
169
      benchmark to obtain its unique optimal solution.
170   We note that even though correctness implies feasibility, and optimally implies correctness, feasibility and
171   correctness are valuable intermediate metrics in determining how close the models are to generating the fully
172   correct solution. For example, in traffic problems, GPT-4 often fails to record the first city visited, resulting in
173   a feasible but incorrect solution. Defining feasibility helps distinguish this mostly correct implementation
174   from more erroneous solutions. Correctness is stricter than feasibility and indicates that search-related tasks
175
      were implemented correctly, but the heuristic or recorded cost is incorrect, leading to non-optimal solutions.

179   We use the following 3 baseline prompting methods to evaluate LLMs on SearchBench: 0-shot text, 4-shot
180   CoT text, and 0-shot code. Additionally, we introduce two new code-based methods: 4-shot A* prompting
181   and MSMT A*. The full prompts for each of the 5 approaches and GPT-4’s responses for an example problem
182   in SearchBench are provided in Appendix Sec. H.
183   To ensure the generality of our prompting methods, we selected one in-context example from each of the four
184   SearchBench categories that are different from the category of the evaluated problem. This minimizes the
185   similarity between the rules and context of the solved examples and the evaluated problem, and tests whether
186   the model can solve unrelated problems. Thus, if a model finds an optimal solution using these methods, it
187   demonstrates true generalization rather than prompt-specific improvements. In Sec. 6, we further analyze the

208   0-shot text and 4-shot CoT text prompting methods. In the text-based prompting methods, we instruct the
209   model to solve the problem in an end-to-end manner, using text only. In 4-shot CoT prompts, the in-context
210   examples include a representation of the intermediate states drawn using ASCII characters after each action
211   to prevent hallucinations and illogical leaps in reasoning.
212   0-shot code prompting method. This method instructs the LLM to produce a Python code that solves the
213   given problem. The generated code is then executed to derive the final answer.
214
      A* Prompting. In this approach, we prompt the LLM to implement an A* algorithm that solves PiC - a
215
      problem instance number i of problem category C, providing four in-context examples of A* codes for four
216
      unrelated problems PjĈ from different categories Ĉ ̸= C. To implement A* for the target SearchBench
217
      problem, the LLM must perform abstract reasoning to devise a search strategy applicable to any state within
218   the search space. This contrasts with solving problems end-to-end in text, where the model has access to the
219   variables of each state, eliminating the need for abstract reasoning or a general strategy. However, end-to-end
220   approaches requires the model to perform every step of the non-linear computations involved in the search.
221
      The in-context examples include detailed comments before each code segment, explaining the reasoning used
222
      to develop the strategy implemented within the code segment. These comments serves as CoT reasoning for
223
      devising the search strategy implemented in the code.
224
225   Multi-Stage-Multi-Try (MSMT) A* Prompting. In this method, the model receives the same in-context
226   examples as the ‘A* prompting’, with different instructions. Here, the inference is done in two stages as
227
      demonstrated in Fig. 2. In the first stage, the model is instructed to implement the code as two functions: the
      ’a_star’ function includes an instance-agnostic A* algorithm for the target problem type, and the ’initialize’
228
      function encodes the variables given in the problem statement. We then verify if the generated code satisfies
229
      the following set of unit tests: (i) code is executable; (ii) code returns a list; (iii) and the list elements match
230   the data type specified by the problem statement. If the code fails any unit test, MSMT re-generate the code.
231   Next, in the second stage, the LLM is instructed to implement an ‘initialize’ function, conditioned on the
232   verified ‘a_star’ function from stage 1 for each instance of the problem type. The inclusion of simple unit tests,
233   which can be expanded to more detailed tests if needed, offers a robust method for filtering out erroneous
234   samples from the model’s generations.

      In our MSMT A* prompting approach, the model generates the full A* algorithm end-to-end without any
236
      external feedback, similar to how text-based prompting methods operate. Importantly, our MSMT A* does
237   not rely on the majority vote of multiple solutions. Instead, the solution returned by the first model-generated
238   code that passes the unit tests is taken as the final answer. This results in increased efficiency of MSMT A*,
239   requiring only up to 1.5x number of inferences per problem on average compared to 5x-100x in majority vote
240   approaches (Wang et al., 2022).

245
      Mathematical and Reasoning Benchmarks: The evaluation of LLMs (Brown et al., 2020; OpenAI, 2023;
246
      2022; Chung et al., 2024; Chowdhery et al., 2022; Rae et al., 2021; Taylor et al., 2022; Thoppilan et al., 2022)
247
      on mathematical and reasoning tasks has been a focus of recent research in natural language processing,
248   leading to the development of benchmarks such as BIG-BENCH (Srivastava et al., 2022), GSM8K (Cobbe
249   et al., 2021), AQUA (Ling et al., 2017), SVAMP (Patel et al., 2021), CommonsenseQA (Talmor et al., 2018),
250   StrategyQA (Geva et al., 2021), and MATH (Hendrycks et al., 2021). However, these benchmarks have
251   limitations. For instance, GSM8K problems are relatively simple and often require a repetitive reasoning
252   pattern to solve. The MATH dataset, while more challenging, may not accurately reflect a model’s reasoning
253   or problem-solving capabilities due to the advanced mathematical skills required. Tasks in BIG-BENCH are
254   mostly single-step reasoning tasks that don’t challenge models to combine multiple steps for solving complex
255   compositional problems. When prompted to solve problems end-to-end using CoT prompting, LLMs perform
256
      well on these tasks; however, they fail on our benchmark’s problems, indicating that these benchmarks offer
      limited insight into LLMs’ ability to systematically explore a state space.
257
258   Application of LLMs to Combinatorial Problems: Recent work (Yang et al., 2023; Liu et al., 2024; Masoud
259   et al., 2024; Mittal et al., 2024; Iklassov et al., 2024) has explored solving combinatorial problems using
260   LLMs. Yang et al. (2023); Liu et al. (2024); Masoud et al. (2024) investigated prompting LLMs to solve the
261   Traveling Salesman Problem through multiple inferences, while Mittal et al. (2024) introduced a dataset of
262
      combinatorial problems, "PuzzleBench". However, they only selected problems that can be represented in
      a symbolic solver (SMT2.0) and assumed there exists fixed pre-defined symbolic representations for input
263
      problems and outputs, limiting their datasets’ generalizability. Moreover, problems selected by Mittal et al.
264
      (2024) and Iklassov et al. (2024) are instances of the classical combinatorial problems, raising issues of
265   memorization as algorithm implementations for instances of such problems are often available online.
266
267   SearchBench stands out in several ways (i) Generalizability: Unlike PuzzleBench, SearchBench problems are
268
      described only in natural language, with no restrictions on rules or actions, ensuring that a model capable of
      solving SearchBench can generalize to other combinatorial problems. (ii) Uniquely Modified Rules: This
269
      prevents memorization, as algorithms for classic versions of the problem are available online. (iii)Optimal
270
      Solutions: Each problem type has a uniquely defined cost, ensuring a single optimal solution and avoiding
271   multiple valid answers. (iv)Multi-Dimensional Evaluation: This provides deeper insights into how close
272   models are to deriving the unique optimal solution. (v)Automated Instance Generation: This avoids data
273   leakage or contamination, as new instances can be generated on demand.
274
      Prompting and Inference Strategies: Sophisticated prompting strategies have been developed to enhance
275
      models’ reasoning abilities. One notable approach is Chain-of-Thought (CoT) prompting (Wei et al., 2022),
276
      which prompts LLMs to generate intermediate steps leading to the final output. This technique has led to
277   advanced variations, including Tree-of-Thoughts (Yao et al., 2023a; Long, 2023), and Graph-of-Thought (Yao
278   et al., 2023b; Lei et al., 2023; Besta et al., 2023) methods that maintain a tree of intermediate generations
279   a to enable systematic exploration of "thoughts". However, these methods rely on evaluating and rejecting
280   intermediate steps, which does not integrate well with our problems. In search problems, intermediate states
281   can’t be easily classified as correct or incorrect, and all possible actions must be considered to find the optimal

      consolidate evidence and verify segments of the LLM’s output.
313   In this work, we propose the A* prompting strategy, where we prompt the model to solve problems by imple-
314   menting a unique A* algorithm. Similarly, our A* MSMT approach decomposes the task of implementing
315   the search algorithm into two stages and checks the model’s generations against external validators; we use
316   simple unit tests instead of external data sources or solved solution instances in our work.

431   What are the most common reasoning errors made in GPT-4’s A* implementations? We manually
432   analyzed 50 A* codes generated by GPT-4 that returned non-optimal solutions across five problems: three
433   pathfinding problems and two puzzle problems. These problems were chosen because GPT-4 showed the
434   least and greatest performance improvement, respectively, using A* prompting compared to 0-shot code (see
435   Appendix Sec. 3). We identified seven distinct failure modes in the GPT-4-generated A* implementations.
436   Each failure mode corresponds to a critical subtask within the overall search strategy. Failing any one of these
437   subtasks results in a suboptimal solution. The results are summarized in Tab. 2 where the percent of ‘correct
438   reasoning’ (disregarding coding errors) is reported for each subtask. As shown, in pathfinding problems,
439
      the most common reasoning mistake was failing to record the list of coordinates visited (13% success rate).
      Specifically, the model often did not record the start coordinate in the list of visited states. This in turn led
440
      to feasible yet incorrect solutions. For the puzzle problems, the most frequent reasoning mistake was in
441
      encoding the goal state. This is likely because our puzzle problems featured unique expected goal states, e.g.,
442   different from the conventional 8-puzzle problem.

      7    C ONCLUSIONS , L IMITATIONS , AND B ROADER I MPACT
459
460
461   In this work, we introduced SearchBench, a pioneering benchmark designed to assess the reasoning capabili-
462
      ties of large language models (LLMs) in solving challenging and ubiquitous search problems using various
      text-based and code-based prompting methods. We demonstrated that advanced LLMs can, to some extent,
463
      successfully solve search problems by implementing structured algorithms, especially when the models’
464
      implementations are verified against unit tests. This suggests a potential future path for automating the
465   addition of new problem types to SearchBench. Specifically, we could leverage LLMs to generate evaluation
466   and instance generation pipelines for new problem types based on a natural language description of the
467   problem. Our MSMT approach could then verify the accuracy of these pipelines using comprehensive unit
468   tests, significantly streamlining the process of scaling the dataset to include new problem types. Limitations
469   and broader impact are discussed in the Appendix Sec. D.

      B    D ETAILED ANALYSIS OF GPT-4’ S PERFORMANCE ON S EARCH B ENCH
732
733
      Tab. 3 details GPT-4 code-based method performance for each of SearchBench’s 11 problems. Consistently
734
      4-shot A* prompting outperforms 0-shot code for most problems. Interestingly for problems in the pathfinding
735   category, prompting GPT4 with 0-shot code outperforms A* prompting.
736
737   Examining closer, GPT-4 mainly uses DFS for pathfinding in 0-shot code. While simpler than A*, DFS
738
      doesn’t guarantee optimal solutions, as reflected in GPT-4’s high feasible and correct rates but lower optimal
      rates. Implementing A* with an admissible and consistent heuristic requires the model to implement a more
739
      complex strategy in the code involving additional constraints and more sophisticated data structures. This
740
      increases the likelihood of reasoning or coding errors, which could explain the dip in GPT-4’s performance
741   using A* prompting compared to 0-shot code when solving these problems.
742
743   Figure 7 further analyzes the relationship between problem difficulty (quantified by state space size of the
744
      problem) and the performance of GPT-4. As observed, the model’s performance is generally higher on easier
      problems, particularly in terms of the rate of correct solutions. This is expected, as easier problems have a
745
      smaller state space to explore. However, the performance of the model does not change drastically across
746
      different difficulty levels. This indicates that the combinatorial problems in SearchBench are intrinsically hard
747   for LLMs to solve in text due to the requirement for backtracking. Moreover, the difference in implementing
748   an A* search algorithm for a difficult or easy instance of SearchBench is limited to encoding the initial and
749   goal states. The rest of the algorithm implementation task remains the same. This is the reason why the
750   model’s performance is comparable across different difficulty levels, both using text-based and code-based
751   methods.

      results underscore the intrinsic difficulty of SearchBench problems, even when addressed through code

1057         while queue:
1058             _, g, actions, state = heapq.heappop(queue)
                 # If the current state is the goal state, return the actions taken
1059             if state == goal_state:
1060                 return actions

1061           empty spot with any of the 4 neighboring tiles
1062             empty_spot_coord = [(i, j) for i, row in enumerate(state) for j, element in enumerate(
               row) if element == ’_’][0]

1064                 new_row, new_col = empty_spot_coord[0] + d_row, empty_spot_coord[1] + d_col

1065           the puzzle
1066                 if 0 <= new_row < num_rows and 0 <= new_col < num_cols:
                          # Generate the new state
1067                      new_state = [list(row[:]) for row in state]
1068                      # Swap the empty spot with the new tile

           def heuristic(state, goal):
1083          # An admissible and consistent heuristic for this problem is the sum of the Manhattan
1084            distances from each tile in its current position to its position in the goal state
              # This heuristic relaxes the constraint that only the empty spot can be moved, and assumes
1085            that each tile can be moved to its goal position independently of the others
1086          # Thus the heuristic reports a lower estimate on the cost of reaching the goal state and is
                 admissible
1087          # The heuristic is consistent because the estimated cost from the current state to the goal
1088             can never be greater than the sum of cost from the current node to a successor node plus
                the estimated cost from the successor node to the goal because the cost of moving the
1089            empty spot to an adjacent tile is 1, which is equal to the decrease in the Manhattan
1090            distance of the tile that was swapped with the empty spot
              # Also the cost of the goal state is 0, as all tiles are in their goal positions

1092          for i in range(len(state)):

                row) if element == state[i][j]][0]
1095                       h += abs(i - goal_coord[0]) + abs(j - goal_coord[1])
1096          return h
