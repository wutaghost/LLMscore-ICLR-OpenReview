009                                                                 A BSTRACT
010
011            Understanding the behavior of an end-to-end 3D visual grounder is challenging,
012            especially when the grounder makes an unexpected prediction. Despite the llm-
013            agent-based grounders performing step-by-step interpretable reasoning, the cost
014            for evaluation at scale is prohibitive. To address the challenges, in this work, we
015            propose a novel fully interpretable symbolic framework for 3D visual grounding,
016            namely Evolvable Symbolic Visual Grounder (E A S E), with much less inference
017            cost and superior performance. Given a symbolic expression of a grounding de-
018            scription translated by an LLM, E A S E calculates the feature of each concept uti-
019
               lizing a set of explicit programs in Python learned from a tiny subset of the training
               data. To learn this program library, we introduce a learning paradigm that continu-
020
               ously optimizes the programs on the training dataset by an LLM-based optimizer.
021
               We demonstrate that our paradigm is scalable when more data is involved. Ex-
022            periments on ReferIt3D show E A S E achieves 50.7% accuracy on Nr3D, which
023            surpasses most training-free methods and has considerable advantages in infer-
024            ence time and cost. On Sr3D, E A S E also has comparable overall performance
025            with these approaches. Moreover, we perform extensive experiments to analyze
026            the interpretability and feature quality and reveal the potential for reasoning and
027            condition level grounding.

118   2.1   P ROBLEM STATEMENT
119
      3D visual grounding tasks involve a scene, denoted as S, represented by an RGB-colored point
120
      cloud containing C points, where S ∈ RC×6 . Accompanying this is an utterance U that describes
121
      an object within the scene S. The objective is to identify the location of the target object T in the
122   form of a 3D bounding box. In the ReferIt3D dataset (Achlioptas et al., 2020), bounding boxes for
123   all objects are provided, making the visual grounding process a task of matching these bounding
124   boxes to the scene S. In contrast, the ScanRefer dataset (Chen et al., 2020) provides only the scene
125   point cloud, requiring additional detection or segmentation modules to accomplish the grounding
126   task.
127   2.2   G ROUNDING PIPELINE
128
129   We adhere to the previous SOTA neuro-symbolic framework for 3DVG (Hsu et al., 2023; Feng
130   et al., 2024). The grounding pipeline is composed of three main components: the semantic parser
131   that converts U into a structured expression E; encoders to compute the features for descriptive terms
132   such as near and small within E. Subsequently, an executor performs logical reasoning over E
133   and computes matching scores between S and each object.
134
135   Semantic parser. We employ GPT-4o (OpenAI, 2024) as the semantic parser and structure the
136
      expressions in JSON format, which consists of the following components:
137
             • category: A string representing the category of the target object referenced in U.
138
139          • relations: A list defining the spatial constraints relative to the target object. Each entry in
140
               this list includes:
141            relation name, a string specifying the spatial relation mentioned in U, such as “near” or
142
               “above.”; objects, a list of objects that share the specified relation with the target object.
               Each element is represented as a separate JSON object; negative, A boolean indicating that
143
               if set to true, the target object should not exhibit this relation.
144
145   For example, the phrase “chair near the table and under a shelf” can be represented as:
146
147         {"category": "chair", "relations": [{"relation_name": "near",
148         "objects": [{"category": "table"}]}, "relation_name": "below",
149         "objects": [{"category": "shelf"}]}]}
150
      Human-annotated natural language expressions exhibit diverse descriptions of relations, leading to
151
      a long-tail distribution of relation name in parsed expressions. To address this issue, we pre-define
152
      a set of common relation names and prompt LLM to select from them for E instead of using the
153   original word from U. Based on the number of associated objects, the relations are categorized into
154   unary, binary, and ternary (Feng et al., 2024). Table 1 presents our predefined set of relations
155   along with their classifications. For simplicity, attributes that describe properties of a single object,
156   such as “large” or “at the corner,” are treated as special types of relations.
157
158   Relation encoder. Neuro-symbolic approaches always use neural network-based encoders to com-
159   pute features corresponding to category names and relation names in E for grounding. Unlike previ-
160   ous neuro-symbolic approaches, our relation encoder is a Python class generated by LLM. It directly
161   utilizes the scene’s point cloud data, performing computations in a transparent and interpretable
      manner. Additionally, object classification is conducted using a pre-trained point cloud classifier.

171   The feature of categories and unary relations, denoted as funary ∈ RN , can be seen as the matching
172   score between objects and their respective categories or relations, where N is the number of objects
173   in the scene. Features of binary relations fbinary ∈ RN ×N represent the likelihood of binary rela-
174   tions existing between all possible pairs of objects. For instance, the element fneari,j quantifies the
175   probability that the i-th object is “near” the j-th object. The structure of ternary features follows a
176   similar pattern.
177
178   Executor. The executor use E and associated features to identify T . Since elements in the features
179   represent the probabilities or corresponding relation or category, the logical conjunction in E can be
180   represented through the product operation. For a symbolic expression E, the classification score ccls
181   of its category is calculated initially by a classifier. Subsequently, the executor processes each
182   relation individually by referring to the relations field. For every relation, the relation feature
183
      frelation is computed, and the grounding scores scoresub ∈ RN for the corresponding objects can
      be recursively calculated. Since the number of associated objects is always less than the feature
184
      dimension by one, the grounding result of the individual relation can be determined through a dot
185
      product between feature and grounding scores. Once all grounding results have been computed, the
186   final score score ∈ RN is obtained by performing an element-wise (Hadamard) product between ccls
187   and all scorerelation . The target object’s index is then determined by applying the argmax operation
188   to the final score.

      2.3     R ELATION E NCODER
191
192   The sizes and positions of objects in 3D scenes are mathematically related to specific relations.
193   For example, “near” is related to the distance between objects, and “large” is related to the volume
194   of an object. In E A S E, we represent each relationship in a modular format using Python classes.
195   Each class contains a highly interpretable computing process. These classes are generated by LLM
196   through generation and self-refinement (Madaan et al., 2024) on a little scale of data from the train-
197   ing set. Previous works Yuan et al. (2024); Fang et al. (2024) also employ functions for relation
198   computation. However, a key distinction to E A S E is that it eliminates the need for human anno-
199   tations or specific prompts—either at the code or text level—thereby reducing reliance on human
200
      knowledge. 2.3.1 outlines the overarching structure of our classes, while the data collection process
      is detailed in 2.3.2. In 2.3.4, we describe the procedures for generating and refining the classes. The
201
      overall framework is depicted in Figure 3.
202
203
204   2.3.1    C LASS STRUCTURE
205
      For each relation, we define a corresponding Python class. The class is initialized with the point
206   cloud data of the scene, represented as a PyTorch tensor, and has two primary methods. One is
207    init param, which is used to compute all parameters for derive features such as the distances
208   between pairs of objects in the case of the “near” class. The other is forward executes numerical
209   computations, including operations such as inversion and exponentiation, and ultimately returns the
210   computed feature.
211
212   2.3.2    U NIT TEST
213
214   LLMs may not always generate a perfect code within one attempt(Olausson et al., 2023). Therefore,
215   we incorporate relation-specific data from the training set to enhance both accuracy and quality,
      enabling the LLM to refine its code based on these data samples.


                                                         4
      Under review as a conference paper at ICLR 2025


216
      The symbolic framework facilitates the collection of this data. For each relation, we use the heuris-
217   tics method to filter the training set data based on the parsed expressions, excluding samples with
218   simple structures. These data typically include only object categories and relationship labels, lacking
219   more complex logical constructs. The data scale for each relation is small (less than 100).
220
      This approach allows the use of straightforward numerical relationships for testing. We write a test
221
      suite to compute the relation feature using generated code and compare the corresponding value of
222
      the target object with that of distractors from the same class. The code passes the test case if the
223   target object’s corresponding value is the highest. The test suite also provides detailed feedback on
224   failed cases. For example, in the case of the relation “small”, the target object, being smaller, should
225   have a larger corresponding feature value compared to distractors. If a distractor exhibits a larger
226   value, this test case fails, and bounding box information of both the distractor and target object will
227   be given for the feedback message in 2.3.4.
228
229   2.3.3   P ROMPT
230
231   Our code generation prompt is primarily composed of two sections: the general instruction and the
232   in-context example.
233
234   General Instruction The general instruction provides fundamental details, such as the task de-
235   scription, execution environment, and class schema. To ensure more precise code generation for
236   view-dependent relationships, we include additional specific guidelines. Please refer to the appendix
237   for the prompt.
238
239   In Context Example In-context examples can assist LLM generate more accurate re-
240   sponses (Brown et al., 2020). However, fixed annotated in-context examples may not work for
241   all relations. On the other hand, annotating specific examples for every relation needs a lot of hu-
242   man effort and could lead to the model relying more on the explicit human-provided knowledge
243   within the examples than on the knowledge encoded in the model itself. Some relations have exhib-
244   ited computational similarities. For example, code for computing relation “near” and “far” may be
245
      largely identical in computing pair-wise distances but differ only in the numerical processing. Once
      a well-refined code for “near” is developed, it can serve as an in-context example for generating
246
      responses related to “far”. By employing dynamic in-context examples, this method offers two key
247
      benefits: 1) reduced reliance on extensive human annotation compared to manually annotated in-
248   context examples (ICE), and (2) enhanced efficiency and precision in generating similar relational
249   data when compared to settings where no ICE is available.
250
251
      We prompt LLM with instruction select the relations that may be relative to
      ... from following to construct a directed acyclic graph G to implement that. In G, each
252
      node represents a relation, and an edge from node A to node B means that when generating code for
253
      B, code of A is used as ICE. The entire code generation process can then follow a topological sort
254   order. We show the graph in Figure 7.
255
256
      2.3.4   C ODE R EFINEMENT
257
258   The code generation and refinement are done in many iterations. In the initial iteration, we give
259   the prompt in 2.3.3 and relation name to LLM to generate Nsample responses. where Nsample is a
260   hyperparameter. Following this unit tests in 2.3.2 are executed on all generated codes. If a code can
261   pass all the test cases, we use it as the final choice and stop the generation. Otherwise, we randomly
262   select 3 failure cases from the test suite 2.3.2 and formulate a feedback message for refinement. This
263   feedback specifies the expected execution outcomes and includes bounding box information from
264   the test suites, instructing the LLM to modify the code accordingly.
265   In subsequent iterations, codes having top k highest pass rate on test cases in the last iteration are
266   selected for further refinement. For each code, its feedback message is appended and the LLM
267   generates another Nsample modified version based on the old version and the feedback message.
268   The same testing and refinement process is applied to these new samples. After Niter iterations, the
269   code with the highest pass rate is chosen as the final version. If multiple samples achieve the same
      pass rate, we select from which is refined more times.

408   Relation constraints Feng et al. (2024) propose that some spatial relations are symmetric, like
409   “near” or “far”, which means if object A is “near” B, B should be also “near” A. So the features
410   of these relations should be symmetric. Some other relations should be asymmetric like “left” or
411   “right”. In these cases, if a feature element is positive (indicating the presence of the relationship), its
412
      corresponding symmetric element should be zero (indicating the absence of the reverse relationship).
      Feng et al. (2024) add loss for these constraints during training for regularization. Even if there is not
413
      any training or special instruction from humans on them, we observe that in our generated data, the
414
      constraints can be ensured on relation “near”, “far”, “left” and “right” because of the deterministic
415   execution of code. This means the feature of ‘near” and “far” is guaranteed to be symmetric. For
416   features f for “left” and “right”, if fi,j > 0, fj,i is guaranteed to be 0.
417
418   Error Breakdown We randomly selected 100 error cases from E A S E on the Nr3D to conduct
419   a detailed error analysis. Results are shown in section 3.3. The primary failure source is related
420   to errors in feature computation. These errors come from flaws in codes or inaccuracies in object
421   classification. Another significant source of failure is the limitations of our system design. Specif-
422   ically, we simplify the scene representation into a list of 3D bounding boxes with predicted labels,
423   while omitting critical details such as object orientation, shape, color, and other visual attributes.
424
      Furthermore, the system does not incorporate region or room-level segmentation of the scenes. Ad-
      ditionally, some failure cases are linked to issues in the semantic parsing process. On the one hand,
425
      we only have a limited set of common relations, which is insufficient for grounding utterances in
426
      real-world scenarios. On the other hand, LLM cannot ensure the generation of accurate symbolic
427   expressions that align with the grounding utterance, which can result in incorrect grounding results.

493
      3D Visual Grounding 3D visual grounding focuses on finding an object in 3D scene based on
      descriptions to appearance or location. ScanRefer (Chen et al., 2020) and ReferIt3D (Achlioptas
494
      et al., 2020) are 2 popular benchmarks on this task, providing rich and diverse object-utterance
495
      pairs on ScanNet (Dai et al., 2017). Traditional approaches train a end-to-end model to work on
496   this task. By encoding more features on visual information (Huang et al., 2022), designing fine-
497   grained encoder on spatial relationship or training on large scale well-annotated data (Ziyu et al.,
498   2023). Recent approaches employ large multi-modal models for reducing training data, enhancing
499   perception and reasoning. Neural symbolic approaches (Hsu et al., 2023; Feng et al., 2024; Yuan
500   et al., 2024) use LLM to parse natural language grounding utterances into executable structured
501   expressions, improving data efficiency. R2G (Li et al., 2024) employ scene graph to modeling object
502   attributes and relations, It surpasses previous methods on view dependent utterances. Agent-based
503   approaches (Yang et al., 2024; Fang et al., 2024) create virtual environments where LLM can take
504
      actions, get observations and do iterative reasoning. Xu et al. use VLM and images from the scene
      to figure out the target object without the need for detection or segmentation modules.
505
506
      LLM Programming LLM can generate executable code (Roziere et al., 2023; Zhu et al., 2024).
507
      Many works use LLM’s programming ability on reasoning (Li et al., 2023a), robotic control-
508
      ling (Liang et al., 2023), reward designing (Xie et al.; Ma et al., 2023). For high-quality and stable
509   code output, Le et al. (2022); Chen et al. (2023) use the feedback from the outside environment to
510   fine-tuning LLM, Ma et al. (2023) use RL training trajectories to select and improve functions.
511
512   Neural Symbolic Reasoning Neural symbolic reasoning methods parse natural language to sym-
513   bolic expression and get reasoning results by executing expressions. Li et al. (2023b) provides a
514   neuro-symbolic interface that supports many logical reasoning and seamless integration with other
515   pretrained model seamlessly. Cheng et al. (2023) use LLM query to assist SQL execution. Mao et al.
516   (2019); Hsu et al. (2023) define domain-specific language and use neural networks to encode labels
517   and relations. Zero-shot visual programming (Gupta & Kembhavi, 2022; Yuan et al., 2024) use
518
      Python code as the symbolic language and execute by interpreter equipped with predefined APIs.
519
520   5   C ONCLUSION & L IMITATION
521
522   In this work, we propose a way to encode relations in Python programs for symbolic 3D visual
523   grounding and a weakly supervised framework for filtering and refining LLM synthesized codes.
524   We directly use knowledge distilled from LLM for relation encoding instead of human annotation
525   or supervised learning. The whole system is highly explainable. We demonstrate its advantages in
526   performance, data efficiency, and inference cost.
527   However, some limitations still remain. In natural language referring sentences, there are diverse
528   descriptions of the relation. But we constraints the domain of keywords in a prompt and pass a
529   minor part of spatial relation names during semantic parsing. Besides, we treat the scene as a list of
530   3D bounding boxes and ignore the object’s appearance, orientation, and room-level information of
531   the scene.

892
893   To demonstrate the effect of filtering generated code by its accuracy on training cases, we choose 6
894   relations and plot the pass rate on training cases on the x-axis and the number of passed examples in
895
      all relative test examples. For some easy relations like “near” or “far”, GPT-4o can pass all the tests
      at once, so we only show the cases having multiple refinement steps.
896
897   The result is shown in Figure 9. For 5 of 6 relations (except relation behind), the code having
898   the highest performance on the training cases can have the top-tier performance on the test set. As
899   for relation behind, using the best code on training cases causes about 15 cases loss on the test
900   case compared to using about 70 percent accurate code. But it’s still better than using most of the
901
      codes whose accuracy is less than 0.5. This might be caused by the bias of training data collection.
      But in general, choosing codes according to performance on the training set is useful for overall
902
      performance on the test set.
