009                                               A BSTRACT
010
011            Transformer-based Large Language Models (LLMs) have become increasingly
012            important. However, due to the quadratic time complexity of attention computa-
013            tion, scaling LLMs to longer contexts incurs extremely slow inference speed and
014            high GPU memory consumption for caching key-value (KV) vectors. This paper
015            proposes RetrievalAttention, a training-free approach to both accelerate attention
016            computation and reduce GPU memory consumption. By leveraging the dynamic
017            sparsity of attention mechanism, RetrievalAttention proposes to build approximate
018
               nearest neighbor search (ANNS) indexes for KV vectors in CPU memory and
               retrieve the most relevant ones through vector search during generation. Unfortu-
019
               nately, we observe that the off-the-shelf ANNS indexes are often ineffective for
020
               such retrieval tasks due to the out-of-distribution (OOD) between query vectors
021            and key vectors in the attention mechanism. RetrievalAttention addresses the OOD
022            challenge by designing an attention-aware vector search algorithm that can adapt to
023            the distribution of query vectors. Our evaluation demonstrates that RetrievalAtten-
024            tion achieves near full attention accuracy while only requiring access to 1–3% of
025            the data. This leads to a significant reduction in the inference cost of long-context
026            LLMs, with a much lower GPU memory footprint. In particular, RetrievalAttention
027            only needs a single NVIDIA RTX4090 (24GB) to serve 128K tokens for LLMs
028            with 8B parameters, which is capable of generating one token in 0.188 seconds.

      1    I NTRODUCTION
032
033
      Recent transformer-based Large Language Models (Vaswani et al., 2017) have shown remarkable
      capabilities in processing long contexts. For instance, Gemini 1.5 Pro (Team, 2024) has supported
034
      the context window of up to 10 million tokens. While this is promising for analyzing extensive data,
035
      supporting longer context windows also introduces challenges for inference efficiency due to the
036   quadratic complexity of attention computation. To enhance efficiency, KV caching, a technique that
037   retains past key and value vectors, has been widely adopted to prevent redundant computations. How-
038   ever, KV caching-based systems face two primary issues: (a) substantial GPU memory requirements,
039   particularly for long contexts, e.g., the Llama-3-8B model requires approximately 125GB per million
040   tokens; and (b) inference latency increases linearly to the context size, primarily due to the time
041   needed to access cached tokens — a common issue across various computing devices, including
042   GPUs. Therefore, reducing storage costs and token access is vital for enhancing inference efficiency.
043   The solution lies in leveraging the dynamic sparsity inherent in the attention mechanism (Deng
044   et al., 2024). This refers to the phenomenon where each query vector significantly interacts with
045   only a limited subset of key and value vectors, with the selection of these critical vectors varying
046   dynamically based on individual queries. Prior work (Tang et al., 2024; Xiao et al., 2024a; Ribar
047   et al., 2024; Lee et al., 2024; Singhania et al., 2024) has proposed various techniques to capitalize on
048   this observation to improve the efficiency of attention computation. However, most of these methods
049   identify important tokens either statically (Xiao et al., 2024b; Li et al., 2024) or heuristically (Xiao
050
      et al., 2024a; Ribar et al., 2024; Tang et al., 2024), leading to imprecise approximations that often
      result in a significant performance drop.
051
052   We observe that the Approximate Nearest Neighbor Search (ANNS) index, such as proximity
053   graph (Malkov & Yashunin, 2018), is particularly effective in this context. ANNS index is used
      to efficiently find the most similar vectors to the query and is widely adopted in various domains

077   Leveraging ANNS for attention mechanism presents a unique challenge: the out-of-distribution
078   (OOD) problem between query and key vectors. Most ANNS engines operate under the assumption
079   that both query and key vectors are drawn from the same data distribution. However, this assumption
080   does not hold in this context due to the different projection weights for query and key vectors in
081   attention mechanism. The Mahalanobis distance (Mahalanobis, 2018) shows that query vectors
082   deviate more than 10× farther from key vectors compared to that between in-distribution query and
083   key vectors. Unfortunately, the effectiveness of ANNS degrades significantly under OOD problem. In
084
      particular, our empirical analysis indicates that maintaining an acceptable level of inference accuracy
      requires conventional ANNS scanning 30–50% of all key vectors to identify the critical ones, which
085
      fails to fully leverage the inherent sparsity of the attention mechanism and impairs the inference
086
      latency. To the best of our knowledge, we are the first to identify the challenge of OOD in using
087   ANNS index for attention computation, a factor that is crucial for inference efficiency and accuracy.
088
089   In this work, we present RetrievalAttention, an efficient method for accelerating long-context LLM
090
      generation. RetrievalAttention employs dynamic sparse attention during token generation, allowing
      the most critical tokens to emerge from the extensive context data. To address the challenge of
091
      OOD, RetrievalAttention proposes a vector index tailored for the attention mechanism, focusing
092
      on the distribution of queries rather than keys. This approach allows for the traversal of only a
093   small subset of key vectors (1–3%) to identify the most relevant tokens, yielding accurate attention
094   scores and inference accuracy. In addition, RetrievalAttention reduces GPU memory consumption by
095   retaining a small number of KV vectors in GPU memory following static patterns (e.g., similar to
096   StreamingLLM (Xiao et al., 2024b)) and offloading the majority of KV vectors to CPU memory for
097   index construction. During token generation, RetrievalAttention efficiently retrieves critical tokens
098   using ANNS index on the CPU and merges the partial attention results from both the CPU and GPU.
099   This strategy enables RetrievalAttention to perform attention computation with reduced latency and
100   minimal GPU memory footprint.
101   We evaluate the accuracy and efficiency of RetrievalAttention on both commodity GPUs (RTX4090)
102   and high-end GPUs (A100) on three long-context LLMs across various long-context benchmarks
103   like ∞-Bench (Zhang et al., 2024b) and RULER (Hsieh et al., 2024). For the 128K context on the
104   RTX4090 GPU, RetrievalAttention achieves 4.9× and 1.98× decoding-latency reduction compared to
105   the retrieval method based on exact KNN and traditional ANNS index, respectively, while maintaining

145   2.2   E XPENSIVE L ONG -C ONTEXT S ERVING
146
147   Due to the quadratic time complexity of attention operation, serving long-sequence input incurs
148   extremely high costs. Table 1 shows the inference latency of Llama-3-8B without KV cache. When
149   the prompt length reaches 1 million tokens, generating every token requires 1,765 seconds with
150   over 96% of latency spent on attention operations. Although KV cache can reduce the decoding
151
      latency, it demands a huge amount of GPU memory for long contexts. As shown in Table 1, 125 GB
      memory is necessary for storing the KV cache when the context length reaches 1 million tokens,
152
      which is far beyond the GPU memory capacity of commodity GPUs such as the RTX4090 (24GB)
153
      or even high-end GPUs like A100 (40GB or 80GB). This necessitates either scaling to more GPUs
154   to accommodate the large KV cache (Liu et al., 2024a) or repetitively offloading and reloading
155   the entire KV cache between CPU and GPU memory over PCIe (Sheng et al., 2023), resulting in
156   excessive communication overhead. Neither approach provides an efficient and cost-effective solution
157   for long-context inference on commodity GPUs.

179   We quantify the attention sparsity by calculating the cumulative sum of attention scores of top-k

199   2.4                C HALLENGES OF O FF - THE - SHELF V ECTOR S EARCH
200
      To reduce the latency of long contexts inference while maintaining performance, we require a
201
      method to accurately identify the critical tokens to the current query in sub-linear time complexity.
202
      Additionally, given the constrained GPU memory, it would be beneficial if such a method could
203   efficiently utilize CPU memory to manage the KV vectors. Based on Equation 1, one key vector is
204   critical for a query vector if they have a large inner product. With the inner product as a similarity
205   function, performing searches on ANNS indexes aligns well with the goal of the attention mechanism
206   to efficiently find critical key vectors for a query vector.
207
      Traditional ANNS indexes generally cluster similar (close) vectors and select the representative vector
208
      for each cluster (Sivic & Zisserman, 2003) or directly build connections between similar vectors to
209   form a proximity graph (Wang et al.).2 For cluster-based indexes, the query first compares with all
210   representative vectors and then only accesses the most similar clusters, whereas, in the proximity
211   graph, the query performs a greedy search, moving closer to the most similar vectors at each hop.
212   Both methods typically require scanning a limited subset of all vectors (e.g., 1%) to identify the most
213   similar vectors to the query, achieving high search efficiency and accuracy. However, we find that

237
      queries and keys, which are projected by different weights as 2.1. Figure 3a (focus on Q to K for
238   now) demonstrates the performance of widely-used vector indexes supported by Faiss (Douze et al.,
239   2024) using a query vector to retrieve the most similar key vectors. It compares the percentage of
240   keys scanned and the corresponding recall achieved (i.e., the overlapping ratio between the retrieved
241   top-100 results and the ground truth). Cluster-based IVF (Sivic & Zisserman, 2003) requires scanning
242   ∼30–50% data for a recall rate higher than 0.95, and graph-based HNSW (Malkov & Yashunin,
243   2018) falls into a local optimum. The results show that traditional vector indexes require scanning a
244   large number of vectors to achieve a high recall, highlighting the challenge of performing efficient
245   vector searches for attention.
246   Fundamentally, the difficulty is due to the OOD between query and key vectors. We quantify this
247   using Mahanobis distance (Mahalanobis, 2018), which measures the distance from a vector to a
248   distribution. We sample 5,000 vectors from Q and K respectively as the query set and compute the
249   the Mahanobis distance from the query set to the remaining vectors in K. Figure 3b shows that the
250   queries from Q are significantly distant from the K vectors (OOD) while K themselves are very
251   close. Therefore, traditional index building based solely on the closeness between key vectors does
252
      not align with the attention mechanism, which requires to retrieve critical tokens as nearest neighbors
      from the query vectors’ viewpoint. In contrast, Figure 3a shows that using sampled K as the queries
253
      (K to K) can easily achieve a high recall by only scanning 1–5% vectors because they are in the
254
      same distribution. Similarly, query vectors in each attention head also follow the same distribution as
255   they are projected by the same model weight. For efficient vector search, the index must consider the
256   OOD characteristic of the attention computation by design.

259   3    R ETRIEVAL ATTENTION D ESIGN
260
261   In this work, we focus on the acceleration of token generation and assume the prefill of the long-
262   context prompt is done in advance, which is widely supported by existing LLM service providers
263   (e.g., context caching (Google Cloud, 2024) or separation of prefill and decoding (Patel et al., 2024;
264   Qin et al., 2024)).
265   We propose RetrievalAttention that leverages attention-aware vector search to approximate atten-
266   tion computation by CPU-GPU co-execution accurately. Figure 4a shows the overall design of
267   RetrievalAttention. Based on our observation in §2.3, We derive an approximated attention by
268   selectively retrieving relevant key-value vectors while discarding those that are negligible(§3.1).
269   To efficiently support long context, we offload most KV vectors to the CPU memory, build vector
      indexes, and use attention-aware vector search to find critical tokens. (§3.2). To better exploit the

288   3.2   ATTENTION - AWARE V ECTOR S EARCH
289
      For each pair of key and value vectors, we first decide whether to hold them in CPU or GPU memory
290
      (the decision method is elaborated in §3.3). The KV vectors offloaded to CPU memory will be
291
      indexed by ki ∈ Rd and queried by qt to find the most relevant ones.
292
293   To accelerate the vector search during token generation, RetrievalAttention diverges from traditional
294   indexes that only consider the closeness between key vectors for index building. Instead, it leverages
295
      the existing query vectors in the prefill phase to guide the index building for key vectors, efficiently
      mitigating the distribution gap. As shown in Figure 4b, during the index construction, RetrievalAt-
296
      tention explicitly establishes connections from the query vector to its nearest key vectors (i.e., exact
297
      k-nearest neighbors, or KNN). The KNN results can be efficiently computed via GPU, forming a
298   mapping from query vector distribution to key vector distribution. Using this structure, the decoding
299   query vector can first search its nearest query vectors and then obtain the most relevant key vectors
300   through the distribution mapping.
301
      Therefore, the KNN connections from query vectors to key vectors serve as a bridge to reconcile their
302
      distribution differences. However, this structure still has imperfections in both memory overhead and
303
      search efficiency because we need to store and access query vectors besides key vectors. To address
304   this problem, we leverage the projection technique from the state-of-the-art cross-modal ANNS index
305   RoarGraph (Chen et al., 2024a) to eliminate all query vectors. Specifically, we project the KNN
306   connections into key vectors by linking key vectors that are connected to the same query vectors,
307   which efficiently streamlines the search. This process connects key vectors that are perceived as close
308   from the query vectors’ perspective, allowing efficient index traversal for future query vectors.
309
      Our evaluation shows that, by effectively modeling the proximity relationship between the query
310   and key vectors, the vector database only requires scanning 1–3% key vectors to reach a high
311   recall, significantly reducing the index search latency by 74% compared with IVF indexes (Sivic &
312   Zisserman, 2003).
313
314   3.3   CPU-GPU C O -E XECUTION
315
316   To exploit GPU parallelism and accelerate attention computation, RetrievalAttention decomposes the
317   attention computation into two disjoint sets of KV cache vectors, the predictable ones on GPU and
318   the dynamic ones on CPU, and then combines the partial attention outputs together.
319   We leverage the patterns observed in the prefill phase to predict KV vectors that are consistently
320   activated during token generation. Similar to StreamingLLM (Xiao et al., 2024b), our current
321   implementation uses fixed initial tokens and the last sliding window of the context as the static pattern
322   and persists them in the GPU cache. RetrievalAttention can be adapted to utilize more complex static
323   patterns (Li et al., 2024; Jiang et al., 2024), achieving the best trade-off between low inference cost
      and high accuracy. During the prefill phase, we physically separate the static tokens in the GPU

      memory with the remaining tokens, which are offloaded to the CPU memory indexed by the ANNS.
325   To minimize data transfer over the slow PCIe interface, RetrievalAttention independently computes
326   the attention results for the CPU and GPU components and then combines them, inspired by the
327   FastAttention (Dao et al., 2022).

      In this section, we compare the performance of RetrievalAttention in long-context LLM inference
332
      against full attention and other state-of-the-art methods. Through experiments, we mainly explore
333
      the following questions: (1) How does RetrievalAttention affect the model’s inference accu-
334   racy? Specifically, we assess the generation accuracy of RetrievalAttention and other methods
335   across various downstream tasks (§4.2) (2) Can RetrievalAttention efficiently reduce the token
336   generation latency of long-context LLM inference? We compare the end-to-end decoding latency
337   of RetrievalAttention with that of other baselines (§4.3).

339   4.1   E XPERIMENTAL S ETUP
340
341   Testbed, Models, and Configurations. We conduct experiments on a server equipped with one
342   NVIDIA RTX4090 GPU (24GB memory) and an Intel i9-10900X CPU with 10 physical cores (20
343
      logical cores) and 128GB DRAM. The experiment results using NVIDIA A100 GPU can be found
      in §A.4. We implement RetrievalAttention on three state-of-the-art long-context LLMs, including
344
      Llama-3-8B-Instruct-262k (Gradient AI, 2024), Yi-6B-200K (01-ai, 2024a), and Yi-9B-200K (01-ai,
345
      2024b). To show a practical speedup of RetrievalAttention and ensure the CPU memory consumption
346   in long contexts does not exceed the DRAM capacity, we follow previous work (Tang et al., 2024) to
347   run the benchmark in real-world single-batch scenarios.
348
349
      Baselines. We compare RetrievalAttention with the following training-free baselines. (1) Full
      attention without KV cache as well as the version with KV cache using vLLM (Kwon et al., 2023).
350
      (2) StreamingLLM (Xiao et al., 2024b): it retains initial tokens along with fixed-length recent tokens
351
      in the GPU memory and discards remaining tokens. (3) SnapKV (Li et al., 2024): it only caches
352   the critical tokens observed from the last window of the prompt. (4) InfLLM (Xiao et al., 2024a):
353   it separates the KV cache of continuous token sequences into blocks and selects representative
354   vectors for each block. In the decoding phase, the current query scans all representative vectors and
355   retrieves top-k blocks with the highest similarity. (5) Quest (Tang et al., 2024): it keeps track of the
356   minimal and maximal key values in KV cache pages and estimates the criticality of a page using the
357   query vector. (6) InfiniGen (Lee et al., 2024): it prefetches only the essential KV cache entries by
358   speculating important tokens required for subsequent attention layers.
359   To better assess the effectiveness of our method, we introduce two additional baselines using tradi-
360   tional vector search methods from Faiss (Douze et al., 2024). Specifically, Flat is an exact KNN
361   method that performs a linear scan of all key-value vectors, whereas IVF indexes key vectors through
362   clustering. By default, all indexing-based methods retrieve the top-100 nearest key vectors.
363
      Benchmarks. We adopt three representative long-context benchmarks for evaluation.

365   • ∞-Bench (Zhang et al., 2024b): this benchmark consists of 7 tasks, including three retrieval tasks

508   5   R ELATED W ORKS
509
510
      To accelerate the long-context LLM inference, some works (Zhang et al., 2023; Liu et al., 2024b;
511
      Xiao et al., 2024b; Han et al., 2024; Ge et al., 2024; Li et al., 2024) attempt to compress the size
512   of the KV cache by leveraging the sparsity of attention. However, these methods often suffer from
513   significant model accuracy drops due to the dynamic nature of attention sparsity.
514
515
      FlexGen (Sheng et al., 2023) and Lamina (Chen et al., 2024b) offload the KV cache to CPU memory,
      but they struggle with slow and costly full-attention computation. By identifying the dynamic nature
516
      of important KV vectors for different queries, recent work chooses to retain all of the KV cache and
517
      dynamically attend to different parts of KV vectors based on the current query. Quest (Tang et al.,
518   2024) partitions the KV cache into blocks and selects a representative key vector for each block.
519   For a given query, it scans all representative key vectors and attends top-k blocks with the highest
520   attention scores. InfLLM (Xiao et al., 2024a) adopts a similar strategy as Quest but offloads most
521   KV cache blocks to the CPU memory to support longer contexts. Due to block-based organization
522   and retrieval, the accuracy of representative vectors significantly impacts the effectiveness of those
523   methods for obtaining important tokens. SparQ (Ribar et al., 2024), InfiniGen (Lee et al., 2024),
524   and LoKi (Singhania et al., 2024) approximate the most relevant top-k keys corresponding to a
525   given query by reducing the channel dimension. RetrievalAttention instead organizes the KV cache
526
      using ANNS indexes, allowing the retrieval of important tokens with high recalls and low cost. The
      concurrent work MagicPiG (Chen, 2024) and PQCache (Zhang et al., 2024a) employ LSH and PQ
527
      centroids to retrieve critical tokens, respectively. However, they fail to address the OOD issue in
528
      attention, necessitating retrieving a large portion of KV cache (e.g., 20%) for high model accuracy.
529

531   6   C ONCLUSION
532
533
      We propose RetrievalAttention, a method that offloads most KV vectors to CPU memory and lever-
534   ages vector search for dynamic sparse attention to minimize inference cost. RetrievalAttention
535   identifies the different distributions of the query and key vectors and employs an attention-aware
536   approach to efficiently find critical tokens for model generation. Experimental results demonstrate
537   that RetrievalAttention effectively achieves 4.9× and 1.98× decoding speedup than exact KNN and
538   traditional ANNS methods, on a single RTX4090 GPU for a context of 128K tokens. RetrievalAtten-
539   tion is the first system that supports running 8B-level LLMs with 128K tokens on a single RTX4090
      (24GB) GPU with an acceptable latency cost and without compromising model accuracy.

949    C     I MPLEMENTATION
950
951    RetrievalAttention builds one individual vector index for the KV cache in one attention head. Re-
952    trievalAttention has implemented several optimizations to optimize the prompt prefill, accelerate the
953    vector search, and reduce CPU memory usage.
954
955    Optimization for the Prefill Phase. During the prefill phase, full attention computation is required
956    to generate the output vector for the next layer of the LLM. Simultaneously, we move the KV vectors
957    to the CPU side for the ANNS index building. To accelerate the overall prefill process, we overlap
958    the cache movement to the CPU with the full attention computation on the GPU in a pipeline manner.
959    To minimize peak GPU memory usage during the prefill phase, attention computation is performed
960    sequentially across multiple attention heads. This approach only slightly impacts the attention
961    computation speed, as longer prompts can fully leverage GPU parallelism with FlashAttention.
962    For simplicity, our current implementation does not integrate the KNN computation from Q to K into
963    FlashAttention. However, the KNN computation can be fused into FlashAttention for acceleration
964    because (1) FlashAttention inherently includes the step of calculating the inner product of queries and
965    keys, and (2) the top-K algorithm only consumes registers that are redundant in the flash-attention
966    kernel for modern GPUs. We can initialize a min-heap for each query in registers and update it with
967    the matrix multiplication results as the thread block moves through key blocks. Heap operations are
968
       performed in CUDA cores and may take up to 2-3× the prefill latency, which is acceptable. The
       detailed algorithm is shown in Algorithm ??.
969
970    Handling Newly Generated Tokens. In our current implementation, we do not update the index
971    and, consequently, the nearest-keys set for prefilled queries during the decoding phase. Instead, we
       maintain the newly generated tokens in the GPU memory as a static pattern and include all of them

1007   without updating the nearest-keys of prefilled queries. Specifically, we can utilize the update strategy
1008   from RoarGraph (Chen et al., 2024a) to achieve this. For each new key vector (v), we find the
1009   closest prefilled query (q), get the previous nearest-key set (S), and use it to select neighbors for
1010   v without modifying the pre-existing set. This insert strategy is shown to be highly efficient while
1011   maintaining the indexing quality (Chen et al., 2024a). To exemplify this, we insert every key vector
1012   of the generated token into the index in a summary task and test recall rates among all heads in
1013   Llama-3-8B. Figure 11 presents the index of RetrievalAttention maintains its quality at high recall
1014   regimes when continuing to insert newly generated key vectors into the index.

1068      Sparse Transformers. Since the quadratic complexity of attention has become the bottleneck of
1069      LLM efficiency for long context applications, numerous works have studied to design sparse trans-
1070      formers to reduce the computational and memory complexity of the self-attention mechanism. Some
1071
          works restrict the attention computation to predefined patterns, including sliding windows (Child
          et al., 2019), dilated windows (Beltagy et al., 2020), or a mixture of different patterns (Zaheer et al.,
1072
          2020; Ainslie et al., 2020). Some approaches use cluster-based sparsity based on hash value (Kitaev
1073
          et al., 2020) or KNN algorithms (Bertsch et al., 2024; Mao et al., 2024). These solutions either
1074      require pre-training a model from scratch or target limited scenarios like CPU-only, which do not
1075      work for our target to out-of-box usage of LLMs on the GPU-CPU architecture. Although some
1076      approaches (Xiao et al., 2024a; Ribar et al., 2024) exploit the dynamic sparse nature of LLMs, they
1077      often use some estimation using low-rank hidden states or post-statistical approaches, which incurs
1078      high overhead but with low accuracy. Moreover, all these approaches have to maintain full KV
1079      vectors on GPU with only accelerated inference by reduced memory movement, which does not solve
          the challenge of commodity GPUs with limited GPU memory.
