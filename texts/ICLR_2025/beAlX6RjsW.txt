                                                   A BSTRACT
008
009
010            Many inference services based on large language models (LLMs) pose a privacy
011
               concern, either revealing user prompts to the service or the proprietary weights to
               the user. Secure inference offers a solution to this problem through secure multi-
012
               party computation (MPC), however, it is still impractical for modern LLM workload
013
               due to the large overhead imposed by MPC. To address this overhead, we propose
014            M ARILL, a framework that adapts LLM fine-tuning to minimize MPC usage during
015            secure inference. M ARILL introduces high-level architectural changes during fine-
016            tuning that significantly reduce the number of expensive operations needed within
017            MPC during inference, by removing some and relocating others outside MPC
018            without compromising security. As a result, M ARILL-generated models are more
019            efficient across all secure inference protocols and our approach complements MPC-
020            friendly approximations for such operations. Compared to standard fine-tuning,
021            M ARILL results in 2.2−11.3× better runtime and 2.4−6.9× better communication
022
               during secure inference across various MPC settings, while typically preserving
               over 90% performance across downstream tasks. Anonymous code is available at
023
               https://anonymous.4open.science/r/MPC-auto-B100.

027   1    I NTRODUCTION
028
029   Transformer-based large language models (LLMs) have revolutionized machine learning (ML).
030   Since the announcement of ChatGPT, we have seen the release of a plethora of proprietary LLMs
031   like GPT-4 (OpenAI, 2023), Claude 2 (Anthropic, 2024), and Gemini (Google, 2024), as well as
032   open-source LLMs like Llama (Touvron et al., 2023) and Mistral (Jiang et al., 2023) that are now
033
      competitive against their proprietary counterparts (Chiang et al., 2024; Wang et al., 2023; Yan et al.,
      2024; Liu et al., 2024). Recently, companies have started to finetune these models on domain-specific
034
      data to improve their performance on downstream tasks such as chatbots, virtual assistants, and
035
      copilots (OpenAI, 2023; Anyscale, 2023; Cohere, 2024).
036
037   Using these finetuned models to power such user-facing services, however, raises significant privacy
038   concerns. On one hand, the providers of these finetuned models do not want to expose their models’
039
      weights, as these models are often trained on proprietary data and represent competitive differentiation.
      On the other hand, users do not want to send their queries to these providers as these queries might
040
      contain sensitive or proprietary information (e.g. IP-protected code or user data). In fact, some
041
      enterprises prohibit their users from using LLM services, e.g., Samsung recently banned the use of
042   external LLM services after an employee accidentally leaked sensitive code to ChatGPT (Ray, 2023).
043
044   Secure inference is a promising solution to address this challenge as it can provide privacy for both
045
      parties through secure multi-party computation (MPC) (Goldreich et al., 1987; Yao, 1986). There is a
      long line of work on MPC-based secure inference (Mohassel & Zhang, 2017; Mishra et al., 2020;
046
      Rathee et al., 2020; 2021; Wagh et al., 2019; Tan et al., 2021; Jawalkar et al., 2024) offering different
047
      performance and security tradeoffs, with the recent work focusing on secure transformer inference (Li
048   et al., 2023b; Wang et al., 2022; Dong et al., 2023; Lu et al., 2025; Hou et al., 2023; Gupta et al.,
049   2024). In principle, the service provider can use any of these recent secure inference protocols to
050   support its privacy-preserving service. However, despite massive strides in efficiency, these protocols
051   are still impractical for today’s LLMs. For instance, the state-of-the-art solution (Gupta et al., 2024)
052   requires 23 s and 15.9 GB of communication for the first token generation on a small 137M parameter
053   model with 1024 input tokens. We expect the runtime and communication to degrade to around 6.5
      minutes and 240 GB for a more typical 7B parameter model, which is impractical.

077   To reduce this overhead, prior work has focused on expensive low-level operations within MPC and
078   proposed MPC-friendly approximations for those operations (§ 2). In this work, we consider an
079
      orthogonal approach targeting high-level architectural changes, that offer a complementary way to
      minimize the MPC overhead. Instead of simplifying operations, such architectural changes reduce
080
      the number of expensive low-level operations needed within MPC. Importantly, this strategy does not
081
      (necessarily) eliminate these operations from the inference process entirely; rather, it relocates them
082   outside of MPC without compromising security, where their cost is relatively negligible. Our work
083   is the first to explore this MPC-minimization strategy. Notably, our strategy targets the fine-tuning
084   phase, which is performed locally by the service provider before deploying the model through a
085   secure inference API. The key insight is that fine-tuning, when carefully tailored to secure inference,
086   can unlock significant opportunities for MPC-minimization during inference. Since our focus is on
087   MPC-minimization, these fine-tuning changes do not accelerate plaintext inference (see Appendix C)
088   and only serve to reduce secure inference costs.
089   Building on the above insight, we propose a secure inference fine-tuning framework M ARILL1 , which
090   relies on MPC-guided techniques that diverge from standard fine-tuning to improve the cost of secure
091   inference. Models produced by M ARILL are (i) MPC-minimized without compromising security (§ 3),
092   and (ii) achieve ML performance close to that of standard fine-tuned models through knowledge
093   distillation (§ 5). Crucially, since M ARILL essentially compresses the model within MPC during
094   inference, the resulting models are significantly more efficient across all secure inference protocols
095   (§ 6.1). Furthermore, M ARILL only introduces high-level architectural changes that complement
096
      MPC-friendly approximations. We show that integrating these approximations with M ARILL leads to
      further efficiency improvements (§ 6.3).
097
098   M ARILL builds on three techniques, all focused on the shared goal of minimizing a high-level model
099   component within secure inference MPC via fine-tuning. The first two techniques adapt well-known
100   fine-tuning methods to the MPC-based secure inference setting, representing a novel application of
101   these ideas in this context. While these methods have previously also been used to accelerate secure
102
      machine learning in trusted execution environments (TEEs) (Zhang et al., 2024; Huang et al., 2024),
      their adaptation to MPC involves key distinctions due to the differences in the cost profile of MPC and
103
      TEEs, which we summarize in Appendix D.2. Our third technique is novel and specifically designed
104
      to optimize for the unique performance characteristics of MPC-based secure inference. Now, we
105   present a brief overview of our techniques and the model component (underlined) they minimize:

      • Leveraging open-sourced models: As alluded to earlier, open-source LLMs have become more
109     powerful and are now competitive against proprietary models (Chiang et al., 2024; Wang et al.,
110     2023; Yan et al., 2024; Liu et al., 2024). Consequently, a trend has emerged where an increasing
111     number of service providers opt to fine-tune these open-source models with their private datasets
112     instead of pre-training their own proprietary models (Anyscale, 2023; Cohere, 2024). Since the
113     open-source model weights are publicly available, they can be utilized to improve secure inference
114     cost as they do not have to be kept private from the user. However, standard fine-tuning updates
115     all the model weights with the private data, thereby necessitating that all weights be private and
116     precluding any potential benefits of the open-source model weights. In light of this, we adapt two
117
        existing fine-tuning techniques to effectively leverage the public weights and minimize MPC:
118       – Layer Freezing (§ 5.1): We reduce the number of transformer layers that need to be evaluated
119         within MPC by restricting fine-tuning updates (and thus, private weights) to just the final layers
120         of the pre-trained model. We resort to such strict demarcation because alternating private and
121
            public layers still require the bottleneck operations in the public layers to run within MPC (§ 4),
            and simply pruning the public layers leads to poor task performance (§ 6.4).
122
123       – Low-rank Adaptation (LoRA) (§ 5.2): Recent parameter-efficient fine-tuning techniques like
124
            LoRA (Hu et al., 2022) have shown that it is possible to achieve comparable task performance by
            fine-tuning only a small fraction of the model’s weights. Although LoRA was originally designed
125
            to reduce memory requirements during fine-tuning, we show that it can be repurposed such that
126
            the typical MPC cost of linear layers is incurred only for a smaller weight matrix dimensionality
127         – a runtime bottleneck in the natural two-party setting as well as during decoding (Appendix B)-
128         stages in other MPC settings (§ 5.2).
129
      • Modifying self-attention architecture: We analyzed the cost profile of secure LLM inference
130
        under various MPC settings and identified self-attention as the bottleneck in the most efficient
131     settings (§ 5.3). Fine-tuning can mitigate this overhead by enabling the pruning of certain heads in
132     the (multi-head) self-attention architecture (Michel et al., 2019). However, to achieve significant
133     improvements, we have to prune up to 75% heads (and their corresponding parameters) and this
134     leads to a large accuracy drop despite fine-tuning (§ 6.4). To address this loss, we propose a novel
135     head reduction technique that preserves accuracy with fine-tuning even for a large head reduction:
136       – Head-merging (§ 5.3): We reduce the number of attention heads in self-attention by merging
137         m heads into one, but simultaneously, we also increase the head dimension proportionally to
138         preserve all the pre-trained parameters. While it seems that we did not gain anything because the
139         computational FLOPs remain the same, we show that head-merging actually matches the secure
140         inference cost of head-pruning (§ 6.4). This is based on the key observation that the self-attention
141         operations that are the bottleneck in MPC only scale with number of heads and not the head
142         dimension. Our experiments show that if the heads are merged carefully, head-merging achieves
143
            much better accuracy than head-pruning since it preserves all the pre-trained parameters (§ 6.4).
144
      The end-to-end workflow of M ARILL is summarized in Fig. 1. Compared to standard fine-tuning,
145
      M ARILL-generated models have 2.2 − 11.3× faster runtime and 2.4 − 6.9× lower communication
146   across state-of-the-art secure inference frameworks in various MPC settings (§ 6.1). We evaluate
147   the ML performance of M ARILL on three different kinds of tasks, namely, code generation (Chen
148   et al., 2021), chatbot (Zheng et al., 2023), and machine translation (Kocmi et al., 2022). Across
149   these benchmarks, we show that M ARILL typically preserves over 90% of the standard fine-tuned
150   performance (§ 6.2).

      3    T HREAT M ODEL AND S ETTING
183
184
      In the secure inference setting, there is a user with a private input and a logical server with a private
185   model. The model architecture is assumed to be public, the service provider only wants to hide the
186   private model weights, and the user wants to hide its private input.
187
188
      Traditional threat model. We focus on the threat model commonly considered by prior works on
      secure transformer inference: a semi-honest (or passive) adversary that compromises either the user
189
      or the logical server. In settings where the logical server is implemented via a set of MPC participants,
190
      the adversary compromises an unknown subset of these participants (see Appendix A). M ARILL is
191   not limited to a semi-honest adversary and we discuss malicious security in Appendix F.
192
193   Open-source pre-trained model setting. M ARILL applies to the emerging open-source pre-trained
194
      model setting where the service provider fine-tunes a (public) open-source model with its private data
      to produce an inference model. Unlike prior works, M ARILL produces models where not all weights
195
      must be kept private from the user. We now explain why M ARILL’s models uphold the principles of
196
      secure inference despite using a mix of public and private weights:
197
198   • Public-private architecture does not leak private data: M ARILL statically determines which
199     weights should be kept public or private through simple configurations (layers frozen, LoRA rank),
200     and this is done independently of the private dataset. The public weights in a M ARILL-generated
201     model are exactly the open-source model weights frozen during fine-tuning, and thus, making them
202     public does not reveal any extra information about the private weights or the private dataset.
203   • Public parts of the model evaluated outside MPC do not leak additional data: since M ARILL
204     is black-box in the underlying MPC protocol, all that remains to be proven is that the public parts
205     M ARILL evaluates outside MPC do not leak additional data about the private inputs. The proof is
206     straightforward (see Appendix E) and we show that evaluating just the private part of the model
207     within MPC is equivalent to evaluating the whole model within MPC, i.e., client only learns the
208
        output tokens and the server learns nothing. Given just the output tokens, what can be learned about
        the private weights is an orthogonal question (see Appendix D.3 for discussion on recent work).

211   4    P ERFORMANCE C HARACTERISTICS OF S ECURE I NFERENCE
212
213   Secure inference relies on secure multi-party computation (MPC) (Goldreich et al., 1987; Yao, 1986),
214   a cryptographic primitive that allows mutually distrusting parties to compute any function on their
215   private inputs without revealing anything beyond the function output. Prior secure inference works,
      specifically, have considered three MPC settings (Appendix A), each making different assumptions


                                                          4
      Under review as a conference paper at ICLR 2025


216
      about the participants. In this section, we highlight the unique cost profile of MPC in these settings
217   and discuss how it motivates the design of our techniques in § 5.
218
219   Interaction costs. Unlike plaintext computation, most operations within MPC require interaction
220
      among the MPC participants. This imposes two additional performance overheads in addition
      to computation size, namely, communication size and rounds of communication. For most MPC
221
      protocols, this cost of interaction ends up being the bottleneck and it is the primary reason why MPC
222
      is orders of magnitude slower than plaintext computation.
223
224   Multiplications with public weights come for free. Since MPC operates natively over integers,
225   recent secure inference works use fixed-point representation to emulate real-number arithmetic.
226
      Additionally, prior works maintain the invariant that the intermediate state after every network layer
      is arithmetically secret-shared (ASS) among MPC participants. This approach minimizes the cost of
227
      arithmetic operations, such as integer multiplications and additions, which dominate ML workloads.
228
      In an ASS scheme, a secret value x is split among n MPC participants such that (i) each party Pi
229   receives a share xi and any set of n − 1 shares reveals nothing about x, and (ii) the sum of all shares
230   reconstructs the secret x = x1 + . . . + xn . The linear nature of this reconstruction function allows
231   secret-shared values to be added locally (without interaction) by simply adding the corresponding
232   secret shares, making additions within MPC relatively so inexpensive that they are considered “free".
233   Similarly, any affine operation with public coefficients on secret-shared values, such as a matrix
234   multiplication with public weights, also becomes free. In § 5.2, we show how low-rank adaptations
235   can leverage this property to reduce the number of multiplications between secret-shared values.
236   Non-arithmetic operations are the bottleneck in the most efficient MPC settings. Non-arithmetic
237   operations are used to implement comparisons in maxpool, activation functions such as ReLU and
238   GeLU, exponentiation and division in softmax, as well as the truncation operations in fixed-point
239   multiplications. We analyzed state-of-the-art secure inference frameworks (§ 6.1) in the most efficient
240   MPC settings, namely, 3PC and 2PC-Dealer (Appendix A), and found that non-arithmetic operations
241   account for over 88% of the runtime and communication during secure inference with a sequence
242   length of 2048. This is in stark contrast to plaintext computation where non-arithmetic operations
243
      have a minimal contribution to the total FLOPs and the inference latency. Guided by this insight, we
      proposed head-merging in § 5.3, a technique that preserves the FLOPs and still yields significant
244
      performance improvements.
245
246   A mix of public and private weights typically does not speedup secure inference. Since multi-
247   plications with public weights come for free, one would expect significant improvements to secure
248   inference if most of the weights were public. However, to preserve the standard guarantees of the
249
      MPC, an intermediate state that depends on both the private input and any private weight must not
      be revealed to any party. Consequently, once the computation involves a single private weight, all
250
      subsequent non-arithmetic operations need to be performed within MPC, which as we just discussed
251
      are the bottleneck in the most efficient MPC settings for secure inference. This restriction motivated
252   the design of layer-freezing in § 5.1, which separates the public and private weights across layers
253   such that the non-arithmetic operations in public layers are performed outside MPC.

259   MPC. We start with layer-freezing (§ 5.1) that reduces the number of layers evaluated within MPC.

296   5.2   L O RA A DAPTATION
297
      In § 4, we discussed how multiplication with public weights is free during secure inference. Here, we
298   demonstrate how LoRA (Hu et al., 2022), a technique developed for parameter-efficient fine-tuning,
299   can be repurposed to minimize integer multiplications during inference. These operations account for
300   up to 95% of the runtime in the state-of-the-art 2PC work Bumblebee (Lu et al., 2025). Beyond the
301   2PC setting, we found that multiplications also dominate the decoding (see Appendix B) runtime in
302   3PC and 2PC-Dealer settings, which are otherwise bottlenecked by non-arithmetic operations (§ 4).
303   This occurs because the linear layers during decoding perform matrix-vector multiplications instead
304   of matrix multiplications, making key matrix-multiplication optimizations from Mohassel & Zhang
305   (2017) no longer applicable.
306   A LoRA adapter on a weight matrix W ∈ Rn×k is a product of two low-rank matrices A ∈ Rn×r and
307   B ∈ Rr×k , where r ≪ min(n, k). During fine-tuning, only the low-rank matrices are updated, and
308   at inference time, A × B is merged into the pre-trained weight W to minimize inference overhead.
309   This approach updates all the model weights and we do not get any benefit from the public pre-trained
310   weights. In our solution, we crucially do not merge the product A × B with the pre-trained model
311   weights and keep the matrices separate as shown in Fig. 2b. To see why this reduces multiplications,
312
      consider the evaluation of a LoRA-adapted linear layer: for input X ∈ Rm×n , the evaluation function
      can be written as X × (W + A × B). Naïvely, the complexity of this expression is O(mnk).
313
      However within MPC, the product X × W comes for free (§ 4). To evaluate the remaining expression
314
      X × A × B, instead of computing A × B first, we can first evaluate X × A and then multiply it with
315   B. This reduces the overall complexity to O(mr(n + k)); for n = k = 3200 and r = 64, this idea
316   reduces the number of multiplications by 25×.

513   6.4    A BLATION S TUDY
514
515   Layer-freezing vs layer-pruning. In layer-freezing, we froze the bottom layers of the transformer
516
      to move some layers outside of MPC. An alternative strategy to minimize layers within MPC is to
      simply prune some layers. We experimented with layer-pruning on the HumanEval benchmark and
517
      evaluated the best-performing strategy from Sajjad et al. (2020), namely, top-layer pruning. For half
518
      of the layers pruned, we found that the accuracy drops from 61% for the baseline to just 49.4% post
519   layer-pruning. In contrast, layer-freezing achieved an accuracy of 56.7%, a 12% increase in relative
520   performance, highlighting the importance of preserving the pre-trained weights of the pruned layers.
521
522
      Head-merging vs head-pruning. We compared head-pruning (Michel et al., 2019) and head-
      merging § 5.3 on HumanEval, configuring head-pruning to prune an equal number of heads from each
523
      layer to avoid additional leakage about the private dataset. Table 1b summarizes the results for both
524
      techniques when the heads are reduced by 2× and 4×. First, we note that head-merging achieves
525   similar efficiency improvements to head-pruning for both head reduction factors, with head-pruning
526   being at most 10% faster and 2% more communication efficient. ML performance of head-merging,
527   on the other hand, is much better since it preserves all the head parameters. In particular, head-merging
528   has up to 8% better accuracy than head-pruning, and HM= 4 even outperforms HP=2 in both ML
529   and secure inference performance. Note that these improvements only apply to similar head-merging,
530   not adjacent head-merging, which naïvely combines adjacent heads. These results demonstrate the
531   significance of preserving head parameters as well as merging heads based on similarity.
532
533   7     C ONCLUSION
534
535   In this work, we designed a framework M ARILL, that leverages open-sourced LLMs and introduces
536   high-level architectural changes through fine-tuning to minimize MPC usage during secure inference.
537   We demonstrated that M ARILL is effective in minimizing secure inference costs across MPC settings
538   in exchange for a reasonable accuracy tradeoff. In particular, M ARILL-generated models are 2.2 −
539   11.3× more efficient for secure inference compared to a standard fine-tuned model, and they typically
      preserve over 90% relative performance across multiple challenging LLM tasks.

      A    MPC S ETTINGS
865
866
867
      • 2-party computation (2PC): this setting assumes two MPC participants who do not trust each
        other, and thus, it is the most natural setting for secure inference.
868
869   • Honest-majority 3-party computation (3PC): this setting has an additional helper party that also
870     participates in MPC, and the adversary can corrupt at most any one of the three parties. Prior works
871     considered this setting because having this helper party improves the MPC performance by orders
872     of magnitude.
873
874   • 2PC with trusted dealer (2PC-Dealer): in this setting, there is an additional trusted dealer that is
875
        only responsible for distributing input-independent correlated randomness to the computing parties
        in a pre-processing phase. The parties can then use this randomness to accelerate 2PC on their
876
        private inputs.

879   B    LLM I NFERENCE S TAGES - P REFILLING AND D ECODING
880
881
882
      In this section, we briefly describe the two stages in LLM inference. Firstly, users provide a prompt
      in natural language to the system. The system then uses tokenizers to map the natural language into
883
      a vector x1 , ...xn through a process called tokenization (Sennrich et al., 2015). Then the system
884
      performs the main inference process using LLMs. The inference process consists of two phases -
885   the pre-filling phase and the decoding phase. Formally, the pre-filling phase computes probablity
886   of the first token conditioned on the previous n tokens P (xn+1 |x1 , ...xn ) (Sheng et al., 2024). It
887   then samples from the distribution and predicts the first token xn+1 . The decoding phase iteratively
888   computes the next token based on the same logic. For instance, the first step in the decoding computes
889   P (xn+2 |x1 , ...xn+1 ) and samples to obtain xn+2 . The decoding phase terminate when the new
890   token is an ending token, often referred to as the “end-of-sentence" token (EOS). Interestingly, the
891   left-to-right decoding nature has made the computation characteristics different (Kwon et al., 2023;
892   Yu et al., 2022; Sheng et al., 2024) in these two stages. Thus, we distinguish between the two phases
893
      when evaluating our techniques in this work.

      C    M ARILL D OES N OT ACCELERATE P LAINTEXT I NFERENCE
896
897
898   M ARILL introduces changes during fine-tuning that minimize MPC usage during secure inference.
899   In this section, we argue that these changes are specifically tailored to reduce secure inference costs
900
      and do not accelerate plaintext inference:
901
      • Layer freezing (§ 5.1): although layer freezing restricts the private weights to just the final layers,
902
        plaintext inference still has to evaluate all the layers, resulting in no performance improvement.
903
        Secure inference also evaluates all the layers, however, the expensive MPC overhead is only paid
904     for the layers with private weights.
905
906   • LoRA (§ 5.2): LoRA introduces low-rank matrices A and B to every weight matrix W of the
907     model such that the linear layer computation on input X becomes (W + A × B) × X, as opposed to
908     W × X without LoRA. Thus, running plaintext inference with LoRA actually increases overhead.
909     Again, secure inference also evaluates (W + A × B) × X for all linear layers, however, the
910
        expensive MPC overhead is only paid for computing A × B × X since multiplication with public
        weights comes for free in MPC (§ 4).
911
912   • Head merging (§ 5.3): head-merging reduces the number of heads h but also proportionally
913     increases the head-dimension d such that c = h ∗ d remains the same. Plaintext inference cost is
914     dictated by the FLOP count, which for self-attention is O((b2 + bc) · c) where b is the sequence
915     length. Note that head merging does not change the FLOP count, and thus the plaintext inference
916     cost remains the same. Secure inference benefits from head merging because it reduces non-
917     arithmetic operations that have a negligible contribution to FLOP count but are the bottleneck in
        MPC due to its unique cost profile (§ 4).

920   D.1    M ARILL VS MPC- FRIENDLY APPROXIMATIONS .
921
922   M ARILL is complementary to MPC-friendly approximations as it makes high-level changes to the
923   architecture, as opposed to the underlying operations. Additionally, M ARILL differs from these works
924   in two key aspects: (i) while these works output models where all weights are private, M ARILL
925   produces models that have a mix of public and private weights, and (ii) the model architecture in
926   NAS-based works depends on the private training data and leaks additional information, whereas
927
      M ARILL is statically configured independent of the training data. We show in § 6.3 that these
      approximations can be combined with M ARILL to yield further performance improvements. We
928
      do not include NAS-based approximations here because they leak information beyond the standard
929
      secure inference guarantees, which conflicts with the strict security requirements we aim to preserve.

      D.2    L AYER - FREEZING AND L O RA IN TEE- BASED SECURE ML
932
933   Both M ARILL and prior work on TEE-based secure ML (Zhang et al., 2024; Huang et al., 2024)
934   divide the model weights into public and private components to optimize computation (either for
935   secure training or inference), and the specific techniques of interest are LoRA and layer freezing. In
936   this section, we highlight the key differences in how these techniques are adapted in each setting.
937
      • LoRA: In TEE-based works, LoRA is used to shift computations involving public weights outside
938
        the TEE during inference, provided these computations are preprocessed securely within the TEE
939
        during an equally expensive inference-specific preprocessing phase (Tramèr & Boneh, 2019). In
940     constrast, M ARILL performs all LoRA-related computations (on both public and private weights)
941     within the MPC environment, leveraging the observation that arithmetic operations on public
942     weights have the same cost as the corresponding plaintext operation (§ 4). As a result, M ARILL
943     can accelerate secure inference with LoRA without requiring a preprocessing phase.
944   • Layer-Freezing: Prior work on TEE-based secure inference (Zhang et al., 2024; Mo et al., 2020;
945     Elgamal & Nahrstedt, 2020; Hou et al., 2022; Shen et al., 2022; Sun et al., 2023) has adopted
946     alternating public and private layers. While effective for TEE-based secure inference (where
947     arithmetic operations are the bottleneck), this approach offers limited benefits in MPC-based
948     settings (e.g., 2PC-Dealer and 3PC) since the bottleneck in these settings lies in non-arithmetic
949     operations that must still occur within MPC (§ 4). Instead, M ARILL introduces a strict separation
950     between public and private components to optimize secure inference. Finally, while the recent work
951
        on TEE-based secure federated learning (Huang et al., 2024) employs (split) layer-freezing like
        M ARILL, and this idea can be extended to TEE-based secure inference, we employ this technique
952
        in the MPC-based secure inference context for the first time and provide new and comprehensive
953
        results on its accuracy (in the traditional non-federated setting) and MPC performance (Table 2).

964   E     S ECURITY P ROOF
965
966   In this section, we prove that secure inference on M ARILL-generated models satisfies the standard
967   guarantee of secure inference, i.e., the user only learns the output tokens and the server learns
968   nothing from a secure inference execution. Since M ARILL is black-box in the underlying secure
969   inference protocol and the model architecture itself does not leak private data, all we need to prove
970   is that evaluating public parts of M ARILL-generated model outside MPC does not reveal additional
971   information. We prove the same by showing that evaluating just the private part of the model within
      MPC is equivalent in terms of security to evaluating the whole model within MPC.

           This functionality is parameterized by the model architecture A.
979
980        • Client Prompt: Receive prompt p for A from client C, and store p internally.
981        • Server Weights: Receive model weights W for A from server S, store W internally.

983        • Decoding: If yprev ̸= ⊥, receive token x from the client C. If x = yprev , update the state

985          y ← A.decode(st), update yprev ← y, and send y to the client C.

989
       Ideal functionality of secure inference. The secure functionality provided by a secure inference
990
       protocol can be described using a (simplified) ideal functionality FA (Fig. 6) that is parameterized
991
       by a model architecture A. Note that FA does not leak any information to the server, and the client
992    learns nothing beyond the output tokens. The ideal functionality allows the client to choose the latest
993    token x, but the ideal functionality makes sure that this token must match the previously generated
994    token yprev . It is important to note this ideal functionality is simplified for exposition and there are
995    some additional considerations when it is realized with a secure inference protocol in practice:
996
997    • Secure inference protocols typically emulate real-number arithmetic with fixed-point arithmetic
998      which incurs numerical errors. Thus, the functionality needs to be modified to faithfully perform
999
         each operation according to the fixed-point schema used by the specific secure inference protocol.
1000   • Many of the secure inference protocols (Demmler et al., 2015; Knott et al., 2020; Mohassel &
1001     Zhang, 2017; Wagh et al., 2019; Lu et al., 2025; Dong et al., 2023; Mishra et al., 2020; Tan et al.,
1002     2021; Wagh et al., 2021; Huang et al., 2022) we cite employ probabilistic truncation to boost
1003     efficiency, and it was shown in Li et al. (2023c) that these protocols can not be proved secure w.r.t.
1004
         any ideal functionality. Thus, our proof only applies to protocols that do not have this limitation.
1005
       Modelling the public-private architecture of M ARILL’s models. M ARILL introduces three
1006   techniques that make the following modifications to the model architecture:
1007
1008   • Layer freezing: it splits the model into public and private layers, where the bottom layers are all
1009     public and the top layers are all private.
1010   • LoRA: it makes the majority of weights in the private layers public.
1011
       • Head-merging: it changes the number of heads in the private layers.
1012
1013
       It is straightforward to argue that LoRA and head merging satisfy the standard secure inference
1014   guarantee because they only impact the private layers which are entirely run within MPC. Even
1015   for LoRA, the public weights just make the MPC much more efficient (§ 4) but the computation
1016   is still run entirely within MPC. The changes they make can simply be seen as running a different
1017   architecture within MPC, and MPC ensures that only the output of the private layers (i.e., the output
1018   tokens) is revealed. Thus, we only need to prove that splitting the model into public and private
1019   layers is secure because this actually moves operations outside MPC. To this end, we model the
1020   public-private architecture as follows: Mpb denotes the public layers run outside MPC, and Mpr
1021   denotes the private layers run within MPC. From our layer-freezing strategy, M = Mpb ∥Mpr denotes
1022
       the complete inference architecture, which is basically a concatenation of the public layers with the
       private layers. Now, we look at the security proof which proves that only evaluating Mpr within MPC
1023
       has the same security as running M entirely within MPC.

       F    M ALICIOUS S ECURITY
1081
1082
       Our work is not limited to a semi-honest adversary and can also support a malicious adversary
1083   that deviates from the protocol arbitrarily. Given a maliciously-secure protocol, our work inherits
1084   malicious security against the server directly as the server does not have any additional capabilities in
1085   our system. The simulator for a corrupted server also remains the same. Security against client needs
1086   careful assessment because the client in our system inputs a hidden state (output of a transformer
1087   layer), as opposed to a sequence of tokens in traditional secure LLM inference. This does not impact
1088   semi-honest security because the client will follow the protocol and input the right hidden state.
1089   However, a malicious client can input a state that doesn’t correspond to any sequence of input tokens4
1090   to potentially learn the model weights, or input a different token from what was generated to deviate
1091
       the generation process. This issue can be fixed by making the following changes to the protocol:
1092
       • In step 2, the client additionally provides a zero-knowledge proof-of-knowledge (ZKPoK) (Gold-
1093     wasser et al., 1985) proving that the hidden state it is secret-sharing corresponds to an actual
1094     sequence of tokens of the appropriate length.
1095
       • The secure inference protocol will output the token as well as a hiding commitment and its
1096
         randomness to the client. Now, when the client will secret-share the hidden state for the latest token
1097
         yi−1 in step 4a, it’ll additionally provide a ZKPoK proving that this state is consistent with the
1098     commitment received during the previous token generation.
1099
       • If either proof fails, the protocol will be aborted.
1100
1101
       To complete the argument for malicious security, the simulator will be updated as follows:
1102
1103   • Since the adversary is now malicious, the simulator does not have direct access to its input. Instead,
1104     the simulator will receive ZKPoK proofs in addition to the hidden states from the adversary A. It
1105     will extract the adversary’s input from these proofs. The rest of the simulation follows exactly the
1106     same way.

       The modifications we make to the model for MPC-minimization change its learned knowledge during
1111   pre-training, and simply fine-tuning it leads to a large accuracy loss. To bridge this accuracy gap, we
1112   turn to knowledge distillation (KD) (Hinton et al., 2015) in this work.
1113
1114
       Fig. 1 summarizes our distillation workflow. First, we take the pre-trained model and apply the
       transformations that lead to an MPC-minimized architecture; the model thus obtained is the student.
1115
       Then, we take the pre-trained model and fully fine-tune it to get the teacher model, representing the
1116
       performance baseline we want to match. Finally, we use KD to ease the fine-tuning of the student
1117   model by matching its intermediate states with the teacher model. The student model thus obtained
1118   can then be used for secure inference.
1119
1120
       For layer-freezing and LoRA, we have a one-shot distillation procedure because they preserve the
       pre-trained knowledge. Head-merging, on the other hand, requires a two-stage distillation process,
1121

1122

1160
       M ARILL’s techniques target various potential bottlenecks that occur in secure inference protocols.
       In this section, we discuss which combination of techniques is the most suitable for a given secure
1161
       inference protocol.
1162
1163   • If the protocol is bottlenecked on arithmetic operations, one should use LoRA because it provides
1164     an asymptotic reduction in these operations5 .
1165
1166   • If the protocol is bottlenecked by non-arithmetic operations, consider the sequence length of
1167
         the inference task. If the sequence length is large, prefilling will dominate the overall cost and
         self-attention will be the bottleneck. Head-merging will reduce all the non-arithmetic operations in
1168
         self-attention and provide significant runtime and communication improvements. If the sequence
1169
         length is small, decoding is likely to dominate the cost, and LoRA will present better runtime
1170     improvements.
1171
1172   • If there is no specific bottleneck, use layer-freezing and it will reduce overheads irrespective of the
1173
         cost profile of the underlying protocol. For half the layer frozen, layer-freezing alone offers 2×
         improvements across all inference scenarios and protocols. Otherwise, first apply one of the other
1174
         two techniques, and then add layer-freezing on top.

1266   M     L IMITATIONS
1267
1268   Availability of open-source pre-trained model. In this work, we introduce a novel paradigm that
1269
       shows how the publicly available weights of an open-source pre-trained model can be leveraged to
       accelerate secure inference. This makes sense in many settings because the provider doesn’t have to
1270
       go through a very expensive pre-training process, and the best open-source models are among the
1271
       best models out there Chiang et al. (2024); Liu et al. (2024); Yan et al. (2024); Wang et al. (2023).
1272   However, there could be domains that require specialized knowledge which does not benefit from
1273   the pre-trained knowledge of the available open-source models. In such cases, the provider has to
1274   pre-train their own model, and layer-freezing and LoRA improvements will no longer apply. We note
1275   that if there is significant relevant public data available for that domain, the provider also has the
1276   option to open-source its own pre-trained model to leverage our techniques.
1277
       Delegation setting. In this work, we focus on the secure inference threat models considered by prior
1278   work. These works assume that client is one of the MPC participants, and thus, having it evaluate
1279   a part of the network locally with layer-freezing actually reduces its overhead. This is because the
1280   MPC overhead on each participant is orders higher than plaintext inference Gupta et al. (2024); Li
1281   et al. (2023b). However, one could also imagine a weaker threat model for all of these settings where
1282   the client does not participate in the MPC at all. Rather, an additional server is introduced to the
1283   MPC with the additional trust assumption that it will not collude with the other servers involved in
1284   the MPC. In this case, our layer freezing technique is indeed adding additional overhead on the client,
1285   which might not be acceptable in some cases.

1287   N    S OCIAL I MPACT
1288
1289   This paper presents work that enables privacy-preserving inference, where both the user’s input as
1290   well as the service provider’s model weights stay private. While user privacy is needed in many
1291   applications and desirable in general, there is a potential concern of model misuse through malicious
1292   user prompts. This is not a fundamental issue though, as the checks that the services perform
1293   today on user prompts can also be performed within MPC without revealing them to the service
1294   provider. Alternatively, at the cost of additional client overhead, the client could be asked to create a
1295
       zero-knowledge proof (Goldwasser et al., 1985) proving that its input satisfies some criteria.
