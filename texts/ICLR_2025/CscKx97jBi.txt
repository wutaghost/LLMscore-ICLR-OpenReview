                                                  A BSTRACT
008
009
010            As advancements in Large Language Models (LLMs) continue to accelerate, an
011
               increasing number of researchers are exploring the potential of these models to
               assist in everyday tasks. Despite their remarkable achievements in various down-
012
               stream applications, several challenges must be addressed. This paper delves into
013
               applying LLMs in coding tasks, such as ChatGPT and LLama. Initial observa-
014            tions suggest that directly employing these LLMs does not yield optimal results.
015            However, we have identified that LLMs demonstrate enhanced performance when
016            given appropriate feedback. This includes providing information on the accuracy
017            of the code generated, supplying test cases relevant to the task, and indicating the
018            correct or incorrect outputs for these test cases. Furthermore, we have developed
019            an innovative architecture miming human debugging. This approach supplies lo-
020            cal variable information to the LLM while executing the generated code. Our
021            architecture facilitates providing feedback to the LLM and simulates the human
022
               debugging experience, thereby significantly improving the LLM’s code generation
               capabilities. Utilizing our proposed architecture, our model surpasses the current
023
               benchmarks of state-of-the-art models in the MBPP and Humaneval datasets. We
024
               also present comprehensive analyses and ablation studies to substantiate the effi-
025            cacy of our methods. These findings open new avenues for enhancing the utility
026            of LLMs in coding tasks, offering a more interactive and practical approach to
027            leveraging these advanced technologies.

      1   I NTRODUCTION
031
032
033
      As Large Language Models (LLMs) continue to advance, their integration into various aspects of
      daily work has become increasingly prevalent. For instance, LLMs have been explored in the health-
034
      care sector to offer insightful analyses of health-related data (Cascella et al. (2023)). Their applica-
035
      tion in education is also being researched to enhance learning and teaching processes. While these
036   efforts mark significant strides in applying LLMs to improve daily life, their potential applications
037   still need to be fully realized.
038
039
      One area of recent focus is the use of LLMs in programming, specifically in code writing, as in-
      dicated in LEVER (Ni et al. (2023)). However, employing LLMs like ChatGPT for direct coding
040
      tasks often yields suboptimal results. A notable challenge is the correction of inaccuracies in the
041
      code generated by these models. Several methods have been attempted to address this issue. For
042   example, LEVER (Ni et al. (2023)) generates multiple code variations and employs an executor to
043   select the most effective solution. Meanwhile, LATS (Zhou et al. (2023)) introduces a combination
044   of tree search and an evaluation module to ascertain the most suitable code output. Despite these
045   innovations, a critical gap still needs to be in rectifying errors in the code produced by LLMs.
046
      To understand why LLM perform poor in code generation, we used GPT-3.5-turbo to generate test
047
      cases for the MBPP dataset and identified several common issues contributing to poor performance.
048   First, incorrect function and variable names: the MBPP dataset typically provides a problem de-
049   scription like ”Write a function to find the longest chain which can be formed from the given set of
050   pairs,” but LLMs cannot accurately deduce the true function and variable names required for the test
051   cases. Second, incorrect data structures: for example, given a problem like ”Write a function to find
052   the similar elements from the given two tuple lists,” the correct solution may need a tuple, but LLMs
053   might return a set, which fails the test cases. Finally, logic errors in the generated code: these are
      particularly challenging to address and are a common issue with LLM-generated solutions.

095   2.1   A DVANCEMENTS IN L ARGE L ANGUAGE M ODELS (LLM)
096
097   Recent advancements in Large Language Models (LLM) have been remarkable, driven by the ex-

099

101
      ties, including companies and researchers, started developing their own LLMs. Examples include

105   InstructGPT (Ouyang et al. (2022)) have enhanced human-model interaction by aligning LLMs with
106   human intent, allowing them to address a broad spectrum of queries. The emergence of open-source
107   LLMs such as Vicuna (Peng et al. (2023)) and Alpaca (Taori et al. (2023)) based on LLaMA further
      demonstrates the potential of these models.

      2.2   E MPLOYING LLM AS AN AGENT
120
121   As LLMs have evolved, efforts have been made to utilize them as agents in everyday scenarios.
122   In this context, an ’agent’ refers to an entity where we input information such as objectives, envi-
123   ronment, and feedback, allowing the LLM to assist in decision-making. Starting with the concept
124   of Chain-of-Thought (Wei et al. (2022)) and Tree-of-Thought (Yao et al. (2023)), researchers have
125   been teaching LLMs to approach goals incrementally, similar to human processes. In ReAct (Yao
126   et al. (2022)), there is a shift towards guiding LLMs to generate reasons in addition to actions. Addi-
127   tionally, recent studies (Chang et al. (2023),Hu et al. (2023)) are exploring the use of reinforcement
128
      learning to provide feedback to LLMs, enhancing their performance. The application of LLMs as
      agents spans numerous fields. For instance, they have been used in gaming (Qiao et al. (2023),Ak-
129
      oury et al. (2023)), customer service (Cai et al. (2023), Soni (2023)), and code generation (Ni et al.
130
      (2023), Zhou et al. (2023)). The expansion of LLMs continues to open up new possibilities for their
131   application in various domains.

      2.3   C ODE G ENERATION
134
135   The task of generating code based on given tasks has a long history, with earlier methods being
136   predominantly rule-based, as seen in works done by Woods (Woods (1973)) in 1973. However,
137   with the advent of deep learning, there has been a shift towards leveraging these techniques for
138   code generation, exemplified by Xiao (Xiao et al. (2016)). Recently, LLMs have shown remarkable
139   efficacy in this field, leading to numerous innovations aimed at refining LLM-generated outputs.
140   Some recent works (Ni et al. (2023), Zhou et al. (2023)) have experimented with adding an executor
141   to run the generated code and select the best option from multiple outputs. Concurrently, approaches
142
      like ReAct (Yao et al. (2022)) focus on providing appropriate feedback based on the generated code
      and its test cases. Despite these advancements, LLMs face challenges correcting errors in their
143
      generated code. In our research, we simulate the human debugging process and have achieved
144
      significant progress compared to prior methodologies.

147   3     A RCHITECTURE
148
149   3.1   A RCHITECTURE OVERVIEW
150
151   In our study, we aim to replicate the human approach to identifying and rectifying coding errors.
152
      Typically, a programmer detects and corrects bugs by isolating a failing test case, running the code
      line by line to find the source of the error, and revising the problematic section until it passes all
153
      tests. Drawing from discussions with fellow programmers and personal experience, this multi-step
154
      debugging process informs our research.
155
156   We use a Large Language Model (LLM) to mimic this human debugging process. Recognizing the
157   various stages of human code writing, we divided the task into multiple phases, each simulated by
158
      the LLM. The whole architecture can be seen in Fig2 The first phase involves the code generator,
      which replicates the initial code writing. The second phase introduces a module that simulates
159
      human testing and evaluation of the code’s correctness through a code executor.
160
161   Given the extensive use of test cases in practical applications, we designed a pseudo test case gener-
      ator to provide numerous test scenarios for code assessment. Our architecture’s fourth component is


                                                                                               3
      Under review as a conference paper at ICLR 2025


162
      the debug module, which examines intermediate variables during execution to locate errors, similar
163   to how a human would. The fifth phase involves bug correction. Once a bug is identified through
164   a test case, our feedback module channels this information back to the code generator to facilitate
165   code revision, mirroring the iterative debugging process humans use.
166
      In summary, our architecture includes five key components: an LLM as the main code generator, a
167
      pseudo test case generator for diverse test scenarios, an executor to validate the generated code and
168
      monitor intermediate variables, a debug module for error analysis, and a feedback module that uses
169   execution results to guide code regeneration. This holistic approach closely mimics the iterative
170   process of human code debugging and refinement.

      3.2   C ODE G ENERATOR
173
174
      In the human coding process, when a programmer is presented with a specific task—such as ”Write
175   a function to calculate the volume of a cube given its side length”—they begin by understanding
176   the problem and planning how to implement a solution. They might start by defining the function
177   signature, choosing appropriate function and parameter names, and writing an initial version of the
178   code that implements the basic logic.
179
      Similarly, in our architecture, the code generation process starts with inputting the coding task into
180
      the LLM, which acts as the programmer. We provide the LLM with predefined function names and
181   input parameters to mimic how a programmer would set up their code structure, reducing simple
182   errors like incorrect function names or mismatched parameters. The LLM then generates an initial
183   code template that includes the function definition and preliminary implementation.
184
      This template is analogous to a human programmer’s first draft of code. At this stage, while we
185
      obtain a functional code output, its correctness is not yet assured, nor can we identify potential
186
      bugs within the code. Just as a human programmer would proceed to test and refine their code, we
187   rely on the subsequent components of our architecture to validate and improve the generated code.
188   This mirrors the initial step in human coding, where the programmer writes code based on their
189   understanding before entering the testing and debugging phases.
190
191
      3.3   C ODE E XECUTOR
192
193   In the human coding process, after writing the initial code, a programmer typically runs the code
194   to test its functionality. They execute the code with various test cases to see if it produces the
195   expected results. Importantly, when errors occur, programmers don’t just look at the final output;
196   they also examine the intermediate states of the program—such as variable values at different points
197   in execution—to identify where the code is deviating from expected behavior.
198   Our architecture simulates this aspect of human debugging through the code executor component.
199   While previous methodologies may only check for correctness of the output, our executor not
200   only validates whether the generated code produces the correct results but also meticulously tracks
201   changes in intermediate variables during execution. This is akin to how a programmer might use
202   debugging tools or insert print statements to monitor variable states.
203
      Specifically, we implement mechanisms within the executor to record the dynamics of intermediate
204   variables throughout the code execution process. Upon completion of the test cases, the executor first
205   evaluates if the generated code satisfies all tests. If the code passes, it is deemed correct. If not, the
206   executor compiles a comprehensive report—including the failed test cases, their expected outputs,
207   the actual outputs from the generated code, and the state of intermediate variables. This detailed
208   information mirrors what a human programmer would observe and analyze when debugging their
209   code, providing valuable insights for the next debugging steps.
210
211   3.4   P SEUDO T EST C ASE G ENERATOR
212
213   In human coding practices, programmers often create their own test cases beyond the examples
214   provided, especially when they anticipate edge cases or wish to thoroughly validate their code. This
215   involves thinking critically about different input scenarios that could potentially cause the code to
      fail, such as boundary values, unusual inputs, or invalid data.

      Our architecture emulates this aspect of human debugging through the pseudo test case generator.
217   Recognizing that standard datasets may lack sufficient test cases to fully challenge the code, we use
218   an LLM to generate additional, diverse test scenarios. Just as a programmer might devise various
219   inputs to test their code’s robustness, the LLM generates a range of test cases that may include edge
220   cases or uncommon inputs.

      While these LLM-generated test cases may not all be perfect, they serve a crucial role in simulating

      the human approach to uncover hidden bugs. By incorporating this pseudo-test case generator, we
223   significantly enhance the robustness of our testing process. We execute these test cases using the
224   code produced by our architecture, and any failures encountered during these tests are meticulously
225   recorded. This approach mirrors the comprehensive testing methodology a human programmer
226   would employ to ensure the reliability and correctness of their code.

      3.5   D EBUG M ODULE
229
230   In the human debugging process, when a programmer’s code does not perform as expected, they
231   analyze the failed test cases to understand why the code is not working correctly. This involves
232   checking whether the test cases themselves are valid and then examining the code to identify logical
233   errors or incorrect assumptions. Programmers often inspect variable values at different points in
234   execution, consider alternative code paths, and hypothesize about the root causes of the errors.
235   Our architecture’s debug module simulates this critical step of human debugging. When the code
236   executor reports failures, the debug module uses an LLM to analyze the detailed information col-
237   lected—including the failed test cases, expected and actual outputs, and intermediate variable states.
238   The first objective is to verify the validity of the test cases, ensuring that any potential errors in the
239   pseudo-test cases are not misleading the debugging process.
240   Once the test cases are confirmed to be correct, the LLM delves into the code itself, identifying
241   specific sections that may contain bugs. It offers explanations and possible solutions, much like
242   a human programmer would reason about why the code is failing under certain conditions. This
243   involves examining the logic flow, considering the values of variables at different execution points,
244   and pinpointing where the code diverges from expected behavior. By simulating this analytical
245   process, the debug module helps guide the LLM in understanding and correcting the underlying
246   issues in the code.
247
248   3.6   F EEDBACK M ODULE
249
250   In human programming, after identifying potential errors and their causes, a programmer consoli-
251   dates their findings and uses this insight to modify and improve their code. This iterative process
252   involves applying the knowledge gained from debugging to refine the code, testing the new version,
253   and repeating this cycle until the code functions correctly. Programmers also recognize when they
254
      are not making progress and may decide to approach the problem differently to avoid endless cycles
      of unproductive changes.
255
256   Our feedback module emulates this crucial step in the human debugging process. After the debug
257   module analyzes the errors and suggests possible fixes, the feedback module compiles this informa-
258   tion and presents it back to the code generator. This includes detailed explanations of the errors, the
259   reasoning behind them, and specific suggestions for code modifications.
260   Upon receiving this feedback, the code generator embarks on an iterative process of refining and
261   adjusting the code. It strives to produce a version of the code that addresses the identified issues and
262   aligns with the expected outcomes of the test cases. To ensure this iterative process remains efficient
263   and does not get trapped in perpetual cycles of regeneration, we establish a cap on the number
264   of attempts allowed for code correction. This limit is crucial for maintaining a balance between
265   thoroughness and efficiency in the debugging process.
266   By incorporating this cyclical process of feedback and refinement, our architecture mirrors the way
267   human programmers learn from their mistakes, apply critical thinking, and iteratively improve their
268   code until it meets the desired specifications. This balanced methodology ensures that our architec-
269   ture debugs efficiently and effectively, closely emulating human cognitive processes in coding and
      debugging.

276
      MBPP dataset: The MBPP dataset contains basic Python programming problems stated in natural
      language. The dataset contains 974 problems. For every problem, the dataset contains three different
277
      test cases. Following the previous methods(Ni et al. (2023)), we use the first test cases as part of the
278
      prompt to generate the template containing the function signatures. We use all three test cases during
279   the test, and only when the generated code passes all three test cases do we think the generated code
280   is correct. If the generated code fails in any test case, we will think the generated code is wrong and
281   have bugs.
282
      HumanEval dataset: The HumanEval dataset provides 164 comment descriptions of functions
283
      paired with a canonical implementation of each function and several input–output pairs that the
284
      function should pass. We follow the same evaluation method as the MBPP dataset.

      5    C ONCLUSION
465
466   This paper presents a novel architecture that enhances code generation and debugging using Large
467   Language Models (LLMs) by explicitly simulating the human debugging process. Our approach
468   addresses key limitations in current LLM applications, particularly in identifying and fixing cod-
469   ing errors. The architecture’s essential components—the code generator, advanced code executor,
470
      pseudo test case generator, and innovative feedback module—collaboratively mirror the step-by-step
      methods a human programmer employs during debugging. This emulation significantly improves
471
      the accuracy and efficiency of LLM-generated code.
472
473   The advanced code executor goes beyond merely checking code correctness; it meticulously tracks
474   changes in intermediate variables during execution, providing detailed feedback akin to a program-
475   mer’s observational insights. The pseudo test case generator augments standard datasets like MBPP
476
      and HumanEval by creating diverse and comprehensive test scenarios, enhancing the robustness and
      depth of the debugging process.
477
478   Crucially, the debug module identifies error sources in the generated code or test cases, offering
479   detailed insights and potential solutions, much like a human programmer diagnosing and resolving
480   issues. This comprehensive and iterative process ensures efficient and thorough code generation and
481   debugging.
482   Our architecture achieves state-of-the-art performance on benchmark datasets, demonstrating its
483   effectiveness. By simulating human debugging and providing detailed feedback, we have advanced
484   automated code generation and debugging. These insights will guide the development of more
485   sophisticated and user-friendly coding tools, broadening the applications of LLMs in real-world
      programming.
