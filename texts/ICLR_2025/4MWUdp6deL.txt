008

010
               Large Language Models (LLMs) have recently demonstrated remarkable coding
               capabilities. However, assessing code generation based on well-formed properties
011
               and aligning it with developer preferences remains challenging. In this paper, we
012
               explore two key questions under the new challenge of code preference learning: (i)
013            How do we train models to predict meaningful preferences for code? and (ii) How
014            do human and LLM preferences align with verifiable code properties and developer
015            code tastes? To this end, we propose C ODE FAVOR, a framework for training pairwise
016            code preference models from synthetic evolution data, including code commits and
017            code critiques. To evaluate code preferences, we introduce C ODE P REF B ENCH, a
018            benchmark comprising 1364 rigorously curated code preference tasks to cover three
019            verifiable properties—correctness, efficiency, and security—along with human pref-
020            erence. Our evaluation shows that C ODE FAVOR holistically improves the accuracy
021
               of model-based code preferences by up to 28.8%. Meanwhile, C ODE FAVOR models
               can match the performance of models with 6 ∼ 9× more parameters while being 34×
022
               more cost-effective. We also rigorously validate the design choices in C ODE FAVOR
023
               via a comprehensive set of controlled experiments. Furthermore, we discover the pro-
024            hibitive costs and limitations of human-based code preference: despite spending 23.4
025            person-minutes on each task, 15.1 ∼ 40.3% of tasks remain unsolved. Compared
026            to model-based preference, human preference tends to be more accurate under the
027            objective of code correctness, while being sub-optimal for non-functional objectives.

030   1   I NTRODUCTION
031
032   Large Language Models (LLMs) for code (Chen et al., 2021; GitHub, 2023; Amazon Web Services,
033   2023) have become instrumental in modern software development. Code LLMs assist developers
034   in various scenarios, from suggesting code completions and generating functional code based on user
035   instructions to proposing complex code changes to resolve bug reports and feature requests.
036   Instruction-tuned LLMs (Luo et al., 2024; Wei et al., 2024) are increasingly adept at generating
037   functional code based on natural language instructions. However, evaluating the quality of
038   LLM-generated code remains challenging, particularly regarding code correctness, efficiency, security,
039   adherence to best practices, and alignment with developer preferences. Effectively and efficiently
040   assessing LLM-generated code against these properties is crucial for both evaluation (Liu et al., 2023b)
041   and preference optimization for code LLMs (Weyssow et al., 2024). Nevertheless, the subject of
042   learning code preferences has been largely under-explored, motivating us to study code preferences
043   systematically and train code preference models with new data and modeling methods.
044   Following the established format in LLM-as-a-judge (Chiang et al., 2024), we define the code preference
045   task as follows: Given a user query, a pair of two candidate code responses, and optionally a preference
046   criterion, code preference is demonstrated by choosing one response over the other. Specifically, current
047   approaches estimate code preference based on three proxies, each with advantages and limitations:
048   • Code execution: Code preference in another way can be confidently determined by execution
049     statuses (Liu et al., 2023a). However, applying code execution to arbitrary programs poses challenges
050     due to (i) setup complexity, (ii) code incompleteness, and (iii) execution overhead. For instance,
051     code execution may necessitate specific hardware (e.g., GPUs) and precise software versions, which
052     are challenging to deduce from the code and, even if inferred, are too cumbersome to set up and run.
053   • Human annotation: Human-labeled preferences are often seen as the standard oracle in developing
        LLMs, such as in the RLHF for OpenAI’s GPT models (Ouyang et al., 2022) and LLM evaluation

054
        in Chatbot Arena (Chiang et al., 2024). However, applying human labeling to code is particularly
055     challenging and cost-intensive. Programs are inherently abstract and complex, labeling them requires
056     experienced developers to perform detailed analysis and testing. Meanwhile, human preference
057     is inherently subjective, influenced by the annotators’ code tastes and expertise, which can cause
058     noisy preferences for code, whose quality could otherwise be concretely defined and measured.
059   • LLM-as-a-judge: Prominent LLMs have also been employed to evaluate LLM responses (Chiang
060     et al., 2024; Zheng et al., 2023; McAleese et al., 2024). This method is more scalable than
061     human labeling and can be generalized to a wider range of programs compared to code execution.
062     However, its reliability often hinges on the reasoning capabilities of high-cost proprietary LLM
063
        judges (Weyssow et al., 2024), subject to inherent biases (Zheng et al., 2023).
064   While scaling human- and execution-based preference for code is human-resource- and engineering-
065   challenging1 , improving model-based code preference becomes emerging and crucial, beyond directly
066   prompting off-the-shelf models (Weyssow et al., 2024). Furthermore, how exactly human developers
067   and prominent LLMs determine code preference remains obscure, with little research on quantifying
068   or analyzing their performance across various code criteria. To this end, this work attempts to explore
069
      two critical questions in code preference learning:
070   1. Technical question: How can we build effective and efficient code preference models regarding
071      modeling approaches and data sources?
072   2. Empirical question: What are the preferences of human annotators and LLMs, and to what extent
073      do they align with verifiable code properties and human judgments?
074   C ODE FAVOR. We propose C ODE FAVOR, a novel framework for training code preference models.
075   Specifically, C ODE FAVOR employs pairwise modeling to predict preference within a code pair accord-
076   ing to a user-specified criterion. We propose two synthetic data generation methods to construct pref-
077   erence ranking samples from code evolution: (i) Commit-Instruct transforms the pre- and post-commit
078   code snippets to code preference pairs; and (ii) Critic-Evol samples faulty code from a draft LLM and
079   has another critic LLM to improve the broken code. These methods allow us to curate synthetic pref-
080   erence data efficiently, leveraging the natural evolution of code and the capabilities of existing LLMs.
081   C ODE P REF B ENCH. To evaluate code preferences labeled by various approaches, we introduce
082   C ODE P REF B ENCH, a collection of 1,364 carefully curated preference tasks. These tasks target
083   verifiable properties including correctness, efficiency, and security, while additionally considering
084   general developer preferences. Using C ODE P REF B ENCH, we extensively analyze the effectiveness
085   and cost of code preferences derived from developer agreement, general LLMs, and C ODE FAVOR
086   models. Our study demystifies key insights on the pitfalls of different approaches over different coding
087
      criteria. Our results also demonstrate that our models not only achieve top performance in effectiveness
      but also are significantly more cost-efficient compared to existing solutions.
088
089   We summarize our main contributions below:
090
      1. Dimension & Technique: We propose C ODE FAVOR, the first open recipe to train pairwise
091      code preference models. At the heart of C ODE FAVOR is a pairwise modeling design and two
092      complementary methods for generating synthetic preference pairs from code evolution.
093   2. Benchmark & Code: We present C ODE P REF B ENCH, the first comprehensive developer
094      preference benchmark with 1,364 labeled by three verifiable oracles (correctness, efficiency,
095      security) and general developer preferences from 18 annotators. We release the data and code at
096      [ANONYMOUS-URL-FOR-PEER-REVIEW].
097   3. Study & Results: Based on C ODE P REF B ENCH, we comprehensively quantify and conduct case
098      studies on code preferences derived from human developers and LLMs. We show that C ODE FAVOR
099
         can significantly improve the accuracy of model-based preference by up to 28.8%. C ODE FAVOR
         models can match the preference accuracy of models that are larger by 6 ∼ 9×, while being cheaper
100
         by 34×. We also conduct extensive controlled experiments to validate our design choices.

      2.2    S YNTHETIC C ODE P REFERENCE FROM C ODE E VOLUTION
156
157   Training a pairwise preference model requires a rich set of contrastive code pairs, along with the
158   corresponding instructions and evaluation criteria. Collecting complex (Luo et al., 2024) and
159   diverse (Wei et al., 2024) code pairs is crucial yet challenging, given such resources are neither readily
160   available nor curated by prior work. To this end, we propose to create code preference training data
161   using synthetic code evolution, based on code commits (§2.2.1) and code critiques (§2.2.2). We argue
      that code evolution is a practical source for synthesizing code preferences, not only because of its natural

      indication of preferences2 , but also thanks to their general availability and diversity. We focus on the
163   general methodology in this section and defer the detailed prompting implementation to Appendix A.1.

      2.2.1      Commit-Instruct: P REFERENCE L EARNING FROM C ODE C OMMITS
166
167   We propose Commit-Instruct, a synthetic data generation method transforming raw code commits
168   into code preference training samples. Specifically, the workflow (middle of Figure 1) employs a critic
169   LLM to analyze each raw code commit and produce a training sample in a desired format §2.1. Each
170   raw commit can be denoted by r = (m,ypre ,ypost ), where m is the commit message, and {ypre ,ypost }
171   are the pre- and post-commit code snippets. Commit-Instruct processes each commit in three steps:
172   1. Reasoning: The critic LLM is instructed to reason and explain code changes from ypre to ypost .
173   2. Filtering: Given the explanation, the critic LLM first determines whether or not the code change
174      is meaningful. If so, we proceed to the next step; otherwise, the commit is discarded. This step
175      aims to ensure the quality of synthetic data by excluding trivial or ambiguous code changes.
176   3. Rephrasing: Based on the commit r and its explanation, the critic LLM synthesizes a preference
177      sample in the desired format x = {i,yA ,yB ,c} (§2.1). Specifically, yA and yB are rephrased from
178      ypre and ypost to emphasize the actual change. i is the instruction generated to describe ypre and
179
         ypost and the criterion c is concluded by how ypost improves ypre . The rephrased version of ypost
         is regarded as the chosen response y + in model training.
180
181   Figure 3 in Appendix A.1 provides the detailed prompt implementation for Commit-Instruct.

183   2.2.2      Critic-Evol: P REFERENCE L EARNING FROM C ODE C RITIQUES
184
185   While synthetic evolution can be gathered from human data such as code commits, it can also be derived
186   from synthetic data. As such, we propose Critic-Evol which generates synthetic code preference data
187
      by asking a stronger critic model π + to revise the code generated by a weaker draft model π − .
188   Specifically, Critic-Evol takes a set of coding instructions I = {i1 ,i2 ,··· ,in } as inputs, for each of
189   which (ik ) we apply steps below to either transform it into a synthetic training sample or simply drop it:
190
      1. Fault sampling: This step starts with a suitably weak but functional model, denoted as π − , which
191      statistically allows us to sample improvable code solutions yk− ← π − (ik ).
192
      2. Critique & filtering: We instruct the critic LLM π + to code review yk− , by pointing out noticeable
193      code quality issues and defining the criterion (e.g., c) regarding the code defects. π + may also be
194      satisfied with yk− and thus we stop synthesizing code preference data for (ik ,yk− ).
195   3. Revision: If the critique from π + suggests yk− can be significantly improved, π + creates yk+ by
196      revising yk− to meet the desired criterion c. As such, a new synthetic code preference sample is
197      composed as {ik ,yk− ,yk+ ,c}, with yk+ being the chosen response.
198
199   Figure 5 in Appendix A.1 provides more details on implementation of Critic-Evol.

201   2.3      DATASETS
202
203   We apply our techniques to create two synthetic datasets for code preference learning:
204   Commit-Instruct-EditPack consists of 20,641 code preference samples synthesized from EditPackFT-
205   Multi (Cassano et al., 2023) and Llama3-70B-Instruct (Dubey et al., 2024). After filtering out
206   non-permissive code, we obtain 22,469 blessed Python commits from EditPackFT-Multi for use as
207   the raw commits and prompt Llama3-70B-Instruct (Dubey et al., 2024) to perform the Commit-Instruct
208   strategy. 91.9% of the commits are successfully transformed into code preference data (§2.1) and
209   8.1% of them are filtered out due to lack of clear significance.
210   Critic-Evol-SOSS has 41,595 synthetic code preference samples using the Critic-Evol technique.
211   Specifically, we run Llama3-8B-Instruct as the draft model (i.e., π − ) over 50,661 coding instructions
212   from the Self-OSS-Instruct dataset (BigCode, 2024) to produce initial code solutions. 82.1% of these
213   initial attempts are revised and extended by Llama3-70B-Instruct as the critic model, whereas the
214   rest 17.9% are deemed good enough such that a revision is unnecessary.

235   To systematically evaluate code preferences across different methods, we create the C ODE P REF B ENCH,
236   a set of 1,364 preference tasks. It covers four objectives in code preference evaluation: correctness,
237   efficiency, security, and human preference. Table 1 provides an overview of the four categories of tasks.
238   This section presents the curation process of C ODE P REF B ENCH (§3.1) and the results from human
239   (§3.2) and LLMs (§3.3), along with the controlled experiments in §3.4. Additional details, such as case
240   studies (Appendix A.4) and contamination analysis (Appendix A.5), are deferred to the Appendix.
241
242   3.1   B ENCHMARK S ETUP
243
244   In C ODE P REF B ENCH, we evaluate code preference approaches over four objectives, covering three
245   verifiable properties (i.e., correctness, efficiency, and security) and human preference. For verifiable
246   objectives, we generate oracle labels via code execution and static analysis. For human preference,
247   we engage three annotators to label each code pair to form the evaluation set and establish baselines
248
      To ensure benchmark quality, we only use clear-cut good-bad pairs and exclude tie pairs due to their
      inherent ambiguity. The creation of the dataset for each evaluation category is detailed below:
249
250   Objective #1: Correctness. We construct correct-wrong pairs from EvalPlus datasets (Liu et al.,
251   2023b), i.e., HumanEval+ (164 tasks) and MBPP+ (378 tasks), as they rigorously test LLM solutions
252   with extensive test cases that can detect subtle bugs. We derive at most two contrastive code pairs
253   for evaluation from each seed task. In each code pair, the wrong code comes from test-falsified LLM
254
      solutions while the correct is the human-written ground truth. Finally, we obtain 660 correct-wrong code
      pairs. The number is smaller than 2×(164+378) as “wrong” samples do not exist in some easy tasks.
255
256   Objective #2: Efficiency. We construct fast-slow pairs from EvalPerf datasets (Liu et al., 2024a).
257   EvalPerf exercises the performance of LLM-generated correct solutions using 121 performance-
258   exercising tasks equipped with performance-exercising test inputs. The EvalPerf dataset provides
259   fast-to-slow reference solutions with distinct performance for each task. Therefore, we sample
260
      fast-slow pairs over the reference samples at a step size of 3, and obtain 352 fast-slow pairs.
261   Objective #3: Security. We construct secure-vulnerable code pairs from CyberSecEval (Bhatt et al.,
262   2023), which includes 351 Python vulnerabilities detected by security analyzers. We prompt GPT-4o to
263   fix each vulnerability and rerun the security analyzers to guarantee the fix. Additionally, we equip each
264   code pair with a generalized instruction generated by GPT-4o, so the instruction is not biased towards
265   any candidate. Finally, we obtain 207 secure-vulnerable code pairs to evaluate code security preference.
266   Objective #4: Human preference. We established a team of 18 developers to annotate pairs of code
267   responses sampled from DeepSeek V2 over the latest open and close domain coding benchmarks, i.e.,
268   148 BigCodeBench-Hard (Zhuo et al., 2024) tasks and 161 LBPP (Matton et al., 2024) tasks. Specif-
269   ically, we sample 8 solutions per task at a temperature of 0.8 and select the code pair with the largest
      edit distance. We follow the same annotation criteria as Chatbot Arena (Chiang et al., 2024): given


                                                          5
      Under review as a conference paper at ICLR 2025


270
      two responses, users select the one they would use for the instruction (or skip it if both are tied). Lastly,
271   we obtained 145 preference pairs without conflicting preferences out of three annotations per pair.
272
273   Additionally, we evenly shuffle the order of code pairs within each category to prevent positional
274
      bias. By default, we remove code comments when evaluating tasks focused on verifiable objectives,
      as comments should not affect the outcome. At evaluation, LLMs predict each code preference task
275
      using greedy decoding, following criteria aligned with the benchmark objective.

465   Criteria. Table 6 studies the impact of criteria in the prompt given to C ODE FAVOR models. In our
466   evaluation, by default we let the criterion statement be objective-specific. Specifically, using an empty
467   criterion substantially decreases the preference accuracy, especially for code security (i.e., by 13.2 ∼
468   19.5%). Using a generalist criterion can also lightly degrade the overall performance by up to 2.3%.
469   These findings suggest using fine-grained, domain-specific criterion statements for code preference.
470   To comment or not to comment? Table 6 further studies how code comments impact the code prefer-
471   ences of C ODE FAVOR models in both training and inference. Our default setting as the baseline is both
472   trained and evaluated without code comments. Specifically, enabling code comments when evaluating
473   our default models (i.e., trained without comments), we observe a 6.2 ∼ 10.4% drop in the preference
474   accuracy for code correctness, while other dimensions are barely impacted. Meanwhile, if we both train

517   principles are more specialized (e.g., efficiency and security) and difficult to label than the general
518   human preference for natural language. Weyssow et al. (2024) score code snippets by employing a group

      of prominent LLMs as judges, following Cui et al. (2024), whereas our work trains LLM raters and

520

521
      be even suboptimal compared to LLM-based preference. Furthermore, we provide extensive insights
522   by expanding the studied criteria beyond correctness (e.g., efficiency and security), quantifying human
523   preference, and evaluating a broader set of models. Data-wise, CriticGPT applies bug injection (Just,
524   2014) with human assistance, whereas C ODE FAVOR collects contrastive code pairs from code evolution.

527   5                         C ONCLUSION
528
529   In this paper, we studied human and LLM preferences for code generation. We introduced
530   C ODE FAVOR, a novel framework for training pairwise code preference models using synthetic
531   code evolution data, derived from code commits and LLM critiques. For evaluation, we curated
532   C ODE P REF B ENCH, a benchmark comprising of 1,364 high-quality code preference tasks. This
533   enables us to investigate (i) the alignment of human and LLM-based preferences with correctness,
534   efficiency, and security, and (ii) the consistency of LLM-based preferences with human preferences.
535   Our evaluation demonstrates the effectiveness of C ODE FAVOR: C ODE FAVOR fine-tunes instruction-
536   following models, significantly improving their abilities to learn code preferences. C ODE FAVOR is also
537   cost-effective, achieving on-par performance models of 9× more parameters while being 34× cheaper.
538   Despite the high cost of human-based code preference evaluation, our results reveal that human pref-
539   erences can be sub-optimal for non-functional objectives. Finally, our controlled experiments provide
      a comprehensive validation of the advantages and limitations of design choices within C ODE FAVOR.

      We showcase our prompt implementation for synthetic data generation via concrete examples below:
832   Commit-Instruct. Figure 3 details Figure 1 by exemplifying the prompt we implemented for
833   Commit-Instruct. Given a raw commit that replaces string literals to systematic enum-like constants,
834   we first ask the critic model to explain the code changes. The model then responds with the observed
835   key changes and is asked by the user if the code change improves certain code properties. The critic
836   model answers “[YES]” by pointing out that the code change improves maintainability and readability.
837
      Next, we instruct the critic model to generate a well-formatted version of synthetic preference data
      including an instruction, the criteria, a code preference pair, and feedback to explain the improvements.
838
839   Meanwhile, Figure 4 illustrates a case when the critic model filters out a code commit whose
840   improvement is regarded as unclear. Specifically, the demonstrated code commit changes the original
841   code to import a package in a try-catch block, rather than doing it directly. This prevents execution
842   termination when the package is not needed. However, the critic model thinks such a change is minor
843
      and thus turns down the commit being processed to synthetic preference data.
844   Critic-Evol. Figure 5 demonstrates the prompting for Critic-Evol. Specifically, we implemented
845   the Critic-Evol prompt in single-turn using few-show prompting. The prompt includes five
846   few-shot examples (e.g., the square root example), followed by an instruction extracted from the
847   Self-OSS-Instruct dataset and an initial coding solution provided by Llama3-8B-Instruct. Next, the
848   critic model Llama3-70B-instruct is prompted to generate reflection, criteria, an improved version
849
      of code, and feedback to form synthetic code preference data. More specifically, in the reflection
      section, the critic model is expected to explain the initial implementation and reason for potential
850
      improvements. If the model thinks there is little to improve, the generation stops here and this sample is
851
      filtered out for synthetic preference data generation. In the case of Figure 5, the model suggests that the
852   get all words function can be implemented using dictionary comprehension which is more concise
853   and efficient. Following this, the critic model proposes a related criterion based on code conciseness
854   and efficiency, resulting in an improved version of code, i.e., [ATTEMPT 2].

1113    A.4.1             FAULTY P REFERENCES IN C ODE C ORRECTNESS
1114
1115   We examine and compare the generations of prominent LLMs, our model (Mistral-7B-v0.3-Instruct
1116   classification model trained with Critic-Evol), and human judgments using the code correctness dataset
1117   in C ODE P REF B ENCH. Specifically, in C ODE P REF B ENCH, the oracle for code correctness is via the
1118   execution of massive test-cases (Liu et al., 2023b).
1119    Erroneous reasoning due to LLM hallucination. Preference over code correctness is essentially
1120    a reasoning task. We observe that prominent LLMs frequently make faulty preferences for code
1121    correctness due to reasoning hallucination. For example, Figure 6 shows a task that requires extending
1122    the input string to form the shortest palindrome. There is only a subtle difference in Line 4: the correct
1123    implementation (Code A) searches for the largest suffix palindrome from left to right whereas Code
1124    B erroneously searches it reversely. Interestingly, while human developers consistently made the
1125
        right preference, prominent LLMs such as Claude 3.5 Sonnet and DeepSeek V2.5, as well as our
        models, prefer the wrong code. Taking a closer look, the faults originate from unsound findings in
1126
        their generation. For example, Claude 3.5 Sonnet’s generation includes a false claim, saying that
1127
       “CODE A iterates from the beginning of the string” will make the right code (Code A) “not always
1128    find the longest palindromic suffix.” Similarly, DeepSeek V2.5 also hallucinates that Code A would
1129    incur index errors when i is 0 which is also not true: when i is 0, the if condition in Line 5 is equivalent
1130    to that in Line 2 as string[0:] is the string itself, making the Line-5 condition never true. In other
1131    words, if Line 5 is true when i is 0, Line 2 would also be true and has already returned. In addition,
1132    Figure 8 also presents cases when LLMs collect irrelevant findings and use them as reasons to falsify
1133    the correct code. Our findings double-confirm the phenomena of “Counterfeit Conundrum” proposed
        by Gu et al. (2024): LLMs can mistakenly classify such “counterfeit” programs as correct.

1439
       Following Riddell et al. (2024) that quantifies the contamination in evaluating code generation, we
1440   employ surface-level matching to measure the contamination level between the training and evaluation
1441   data. The contamination quantification is based on the Levenshtein similarity score between the source
1442   and target strings. We measure the code similarity of all training-evaluation code pairs. Specifically,
1443   for each test-set code snippet, we present the contamination upper-bound using the top-1 similarity
1444   score from the most similar training code snippet.
1445
       Figure 16 illustrates the cumulative distribution of the top-1 similarity score on two training sets created
1446   by Commit-Instruct and Critic-Evol respectively, with code snippets from all 1,364 evaluation tasks
1447   (Table 1). Specifically, it shows that there are only 0.1 ∼ 1.7% positive samples in the test-set code
1448   pairs that can find training-set positive samples with a similarity score above 80. This demonstrates
1449   that our training set is almost contamination-free to our evaluation set. As a reference, Riddell et al.
1450   (2024) show that 50.8% and 63.4% of code samples in the widely used code corpus dataset, i.e.,
1451   the Stack (Li et al., 2023), can reach over 80 similarity scores with ground-truth code samples in
1452   MBPP (Austin et al., 2021) and HumanEval (Chen et al., 2021) respectively. The low contamination
1453   can be partially inherited from their seed datasets (Cassano et al., 2023; BigCode, 2024) which have
1454
       been decontaminated upon creation. Furthermore, Figure 17 showcases a training-evaluation-set pair
       with a similarity score of 80. While they share a similar dataflow structure, their semantic and detailed
1455
       branch condition present different meanings.
1456
1457   Interestingly, overall the similarity level of positive-to-positive training-evaluation code pairs is smaller
       than that of other categories, with the negative-to-negative code pairs most similar.

       A.7    L IMITATION AND F UTURE W ORK
1621
1622   While C ODE FAVOR has demonstrated effectiveness in learning code preferences, there are several po-
1623   tential areas of improvement to enhance the scale, applicability, and accuracy of code preference models:
1624
       1. Scaling up synthetic data: One limitation in our implementation is the scale of synthetic training
1625
          data, as our preliminary dataset only includes a total of 62,236 samples, which may be modest for
1626      model fine-tuning. Larger-scale datasets could further improve the generalizability and robustness
1627      of preference models for code generation. Since the idea of C ODE FAVOR is rather general, we
1628      plan to scale up the synthetic data generation by collecting more code commits for Commit-Instruct
1629      and more LLM samples for Critic-Evol. Orthogonally, we may consider using multiple and more
1630      powerful models in Commit-Instruct and Critic-Evol to further improve the quality and diversity
1631      of generated synthetic data.
1632   2. Contextualized code preferences: Code generation in real-world software development often
1633      involves broad context such as repository-level information (e.g., (Ding et al., 2024; Zhang et al.,
1634
          2023a)) and knowledge of external dependencies. Currently, C ODE FAVOR focuses on code
          preferences of self-contained code snippets, which could limit the applications of code preference
1635
          models in practically complex and context-dependent scenarios. Therefore, one future direction
1636
          is to extend our framework to curate more context-sensitive code pairs for contextualized code
1637      preference learning.
1638   3. Benchmark improvements: Our evaluation benchmark, C ODE P REF B ENCH, while carefully
1639      curated, also presents potential limitations related to the diversity and practicality of candidate
1640      code samples due to their synthetic nature. There may also be limitations due to the validity
1641      and consistency of human annotations, which are inherently subjective, particularly in assessing
