009                                                A BSTRACT
010
011            Efficiently mapping tasks to processors and data to memories is a cornerstone
012            of parallel programming to achieve high performance. Traditionally, this critical
013            task has been handled by expert-crafted mapper programs, tailored for specific
014            machine architectures and problem domains. However, creating customized map-
015            pers for each unique application is labor-intensive and time-consuming. Large
016            language models (LLMs) have recently demonstrated remarkable capabilities in
017            understanding and generating code, as well as in self-improvement for optimiz-
018
               ing specific performance metrics. Inspired by these advancements, we intro-
               duce the task of mapper generation (MAGE), which frames generating high-
019
               performance mappers as a discrete optimization problem aimed at maximizing
020
               compute throughput. To solve this optimization problem, we leverage agentic
021            LLMs in the mapper generation process. At the core of our approach lies a novel
022            domain-specific language (DSL), which provides a high-level interface for LLMs
023            to generate the mapper code without getting entangled with complicated, low-
024            level system programming. Moreover, our DSL defines a structured and con-
025            strained search space for RL to explore, guiding LLMs to discover the optimal
026            mapping policy. The evaluation shows that our LLM-generated mappers can sur-
027            pass expert-written mappers in performance, achieving up to 34% speedup across
028            9 benchmarks. Notably, our approach improves the throughput of parallel matrix
029
               multiplication algorithms by up to 31%, reducing development time from several
               days to just a few minutes.

      1   I NTRODUCTION
033
034
      Task-based programming (Slaughter et al., 2015; Bauer et al., 2012; Augonnet et al., 2009; Cham-
035
      berlain et al., 2007; Kaiser et al., 2014; Heller et al., 2017; Chandra et al., 2001; Duran et al., 2011;
036   Moritz et al., 2018; Barham et al., 2022) has recently emerged as a prominent paradigm in paral-
037   lel programming. The core idea is to decompose computations into self-contained functions called
038   tasks that do not communicate with other tasks except through their collection (typically tensor or
039   multi-dimensional array) arguments or data.
040
      One performance-critical aspect of executing task-based applications is mapping: assigning tasks to
041
      processors, data to physical memories, and managing other low-level physical resources. We refer
042   to a concrete mapping policy as a mapper. Different from directly modifying application-level (e.g.,
043   CUDA-level) code, mappers operate at a higher level (e.g., task and processor level) and do not
044   change the correctness of an application’s output; they only affect its performance. The difference
045   between a good mapper and a bad mapper is easily an order of magnitude or even more in perfor-
046   mance. Currently, writing mappers is a manual, labor-intensive, and time-consuming process that
047   requires deep knowledge of the application, the machine, its resource limits, and the C++ mapping
048   APIs. Even for experienced performance engineers, developing an effective mapper can take several
049   days due to the complexity and iterative refinement required.
050   In this paper, we focus on using LLMs to automatically generate mappers. There are two main
051   challenges in using LLMs for mapper code generation. First, when given a specific mapping policy
052   described in natural language, LLMs struggle to generate the corresponding mapper code, which
053   requires generating a few hundred lines of low-level C++ code. This code generation task contrasts
      significantly with method- or function-level code generation, which is more isolated and where

075

076   The second challenge is to identify the optimal mapping strategy that maximizes the performance of

078   category of problems that is well-suited to agentic LLMs that are inspired by classical reinforcement
079   learning (RL) techniques. Notably, RL has achieved tremendous success across a wide range of
080   applications, such as AlphaTensor (for finding better matrix multiplication) (Fawzi et al., 2022) and
081
      AlphaGeometry (for generating math proofs for geometry problems) (Trinh et al., 2024). For the
      mapper generation problem, our DSL constructs a more structured and constrained search space for
082
      LLMs to explore.
083
084   For the search process, we combine the LLM’s code generation capabilities with an agent-based
085   solution built on the Trace framework (Cheng et al., 2024). In this approach, LLMs act as optimiz-
086   ers, generating mappers in the DSL and receiving iterative feedback through an interactive feedback
087
      mechanism as shown in Figure 1. This feedback loop allows LLMs to refine their mapping strategies
      efficiently, reducing the time required to develop high-performance mappers to just a few minutes.
088

089
      tageous. Our experiments show that mappers generated by LLMs not only match but can surpass
090   expert-written mappers, achieving up to 34% speedup across 9 benchmarks. Notably, the search
091   process enhances the performance of parallel matrix multiplication algorithms by up to 31%. We
092   also demonstrate, through ablation studies, that the critical role of high-quality feedback in guiding
093   LLM optimizers to discover more efficient mapping strategies.

      Mapping in Parallel Programming Many parallel programming systems allow users to make their
      own mapping decisions, such as Legion (Bauer et al., 2012), StarPU (Augonnet et al., 2009; 2010),
112
      Chapel (Chamberlain et al., 2007), HPX (Kaiser et al., 2014; Heller et al., 2017), Sequoia (Fata-
113
      halian et al., 2006), Ray (Moritz et al., 2018), TaskFlow (Huang et al., 2021), and Pathways (Barham
114   et al., 2022). Several techniques have been proposed to automate mapping, including machine learn-
115   ing models (O’Boyle et al., 2013; Wang & O’Boyle, 2009), static analysis (Poesia et al., 2017;
116   Ren et al., 2008), traditional reinforcement learning techniques (Mirhoseini et al., 2017), and auto-
117   tuning (SFX Teixeira et al., 2023). Unlike previous work, we use an agent-based reinforcement
118   learning approach with LLMs, exploring a larger search space for mappers than traditional methods.
119
      LLM Code Generation With the rise of LLMs, various models (Chen et al., 2021; Nijkamp et al.,
120   2022; Li et al., 2023; Ouyang et al., 2022; Wei et al., 2023) have been trained on code, enabling them
121   to assist with programming tasks and automate software development. While LLMs have shown
122   success in generating isolated function-level Python code (Chen et al., 2021; Austin et al., 2021),
123   they face challenges when generating system-level C++ code in large repositories due to complex
124   APIs and broader contextual dependencies (Du et al., 2023; Wang et al., 2024). Thus, adopting
125   LLMs to generate code in real-world software remains difficult (Jimenez et al.). Our work addresses
126   this by developing a domain-specific language (DSL) that provides a high-level abstraction to help
127   LLMs generate mapper code more effectively.
128   LLM Optimization with Feedback More recently, LLMs have been used to solve optimization
129   problems. Usually, optimization problems exist in numerical domains (Boyd & Vandenberghe,
130   2004). AhmadiTeshnizi et al. (2023) used LLMs to solve structured optimization problems, such
131   as mixed-integer linear programming (LP). Yang et al. (2024) used LLM to solve unstructured prob-
132   lems as black-box optimization. Follow-up work by Nie et al. (2024) showed the effectiveness of
133   feedback in finding the global minima of black-box functions. Building on such ideas, Cheng et al.
134
      (2024) formally defined a class of optimization problems solvable by LLMs and proposed an op-
      timizer based on the execution graph of a program. Later, Yuksekgonul et al. (2024) proposed an
135
      LLM optimizer inspired by gradient descent. These recent developments provide an opportunity to
136
      use LLMs to optimize code not for task completion but for improving over a performance metric.

139   3   M APPER G ENERATION TASK
140
141
      Task Definition The problem we address is the automated generation of high-performance map-
142
      pers for the Legion parallel programming framework (Bauer et al., 2012). Mappers determine how
143   tasks are assigned to processors and how data is placed in memory to optimize performance. A well-
144   designed mapper can achieve up to 10× speedup compared to random mapping strategies. However,
145   mappers are highly context-specific and must be carefully tailored to an application’s input and the
146   machine’s architecture. Finding an optimal mapper is akin to solving a combinatorial optimization
147   problem—a process that is time-consuming, labor-intensive, and traditionally performed by experi-
148   enced performance engineers. The search space for discovering the best mapping strategy is vast,
149   growing to 214 even for the simplest scientific applications (as shown in prior work with a smaller
150   search space (SFX Teixeira et al., 2023) than ours), making it challenging to efficiently explore all
151
      possible solutions. Moreover, even with a clear mapping strategy, writing the corresponding mapper
      requires experts to produce hundreds of lines of low-level C++ code, a task that can take several
152
      hours. As a result, the entire performance tuning process can take several days to complete.
153
154
155   Mapping Decisions Now we elaborate on the decisions that a mapper has to make. The first
156   critical decision is the processor selection for each task, determining whether a task is better suited
157   for GPUs, CPUs, or OpenMP runtime. This choice depends on factors such as task size, GPU
158
      memory, and kernel launch latency. For example, tiny tasks that require very little computation may
      prefer to run on CPUs due to the GPU kernel launch overhead, whereas tasks with large memory
159
      footprints might prefer to be assigned to OpenMP or CPU when GPU memory is insufficient.
160
161   Next, the memory placement of data across different memory spaces is crucial to performance. A
      mapper must decide where to place data — in the GPU’s FrameBuffer for faster access, in ZeroCopy

256   4.2      L EARNING TO G ENERATE VIA I NTERACTIVE F EEDBACK
257
258   We conceptualize the mapper generation problem as a search for valid DSL programs. Since ef-
259   fective mappers must account for both specific application inputs and the underlying hardware, this
260
      task is well-suited to reinforcement learning (RL).
261   Our method combines the code-generation strengths of LLMs with an agent-based framework built
262   on Trace (Cheng et al., 2024). In this framework, LLMs act as optimizers, iteratively generating
263   mappers in the DSL and refining them based on real-time feedback. This feedback loop dramatically
264   reduces the time required to produce high-quality mappers, from days to minutes, making the RL-
265   driven agent particularly effective in this context-sensitive task.

269   erated code, ensuring the LLM starts with valid DSL syntax, a reasonable initial strategy, and an

343   Failure Case Analysis The compilation errors in C++ mapper generation arise because LLMs
344   are unable to manage the framework-specific contextual dependencies inherent to low-level system
345   software programming. For instance, LLMs generate variable names or references that do not exist
346   within the provided codebase. The reason is that the documentation and example code are so long
347   and complex that LLMs fail to retrieve the related information and hallucinate the variable names.
348   In the cases where the code compiles but fails the test cases, the failures are exclusively in C++.
349   The root cause lies in the complexity of implementing a C++ mapper, which requires a deep un-
350   derstanding of the API documentation and reasoning about the code examples. As outlined in the
351   documentation, multiple APIs must work in concert to accomplish specific tasks, yet LLMs are
352   unable to grasp this level of coordination. For instance, to implement the index mapping feature,
353   LLMs need to override several different functions together (e.g., one function deciding the target
354   node index and the other deciding the target processor index, and some other functions to inform the
355   runtime that the heuristics has been changed), which is too challenging for LLMs. In contrast, this
356
      has been simplified to just one simple function in the DSL (as shown in Figure 3).
357   While compiler feedback can guide LLMs in avoiding trivial errors (e.g., such as generating non-
358   existent variable names), it cannot bridge the gap in understanding the intricacies of low-level C++
359   mapping APIs. This limitation explains why all attempts at C++ code generation ultimately fail.
360   In contrast, only two test cases fail in the DSL context on the single trial, both due to compilation er-
361   rors stemming from incorrect syntax. This is understandable given that the DSL is a newly designed
362   language with limited training data available for the LLM. Thus, for performance optimization ex-
363   periments, we do not attempt to have LLMs optimize over C++ mappers, as results sufficiently
364   demonstrate that LLMs struggle to generate C++ code that meets the specifications of what a de-
365   sired mapper should do, let alone explore a large search space of different mapping strategies.

457
      We follow the same experimental setup as described in Section 5.2, with the key difference being
458   the focus on matrix multiplication algorithms in this subsection. Unlike the applications in Sec-
459   tion 5.2, where processor selection and memory placement are the critical performance factors, the
460   performance-critical mapping decision here is index mapping, which is the choice of how to launch
461   concurrent tasks and distribute the tiles of matrices across multiple GPUs.
462
      In the DSL mapper, achieving an effective index mapping requires specifying a function that maps
463
      the iteration space (defined by the parallel for loop) to the processor space of the machine (please
464   see the IndexMap statement discussed in Section 4.1). Based on our estimates, each algorithm has
465   approximately 109 possible choices for index mapping. We show some of the mapping functions in
466   Appendix A.8
467
      We target 6 different matrix multiplication algorithms: Cannon’s (Cannon, 1969), SUMMA (Van
468
      De Geijn & Watts, 1997), PUMMA (Choi et al., 1994), Johnson’s (Agarwal et al., 1995),
469
      Solomonik’s (Solomonik & Demmel, 2011), and COSMA (Kwasniewski et al., 2019). Each al-
470   gorithm exhibits different performance characteristics and may be the preferred implementation of
471   matrix multiply depending on the target machine, input size, and mapping decisions. Our goal is to
472   explore and identify better mappings for these algorithms. We categorize and elaborate further on
473   these algorithms in Appendix A.7.
474
475   Results As shown in Figure 6, the expert mappers’ compute throughput is normalized to 1.0,
476   reflecting the mapping decisions specified by the algorithms. Random mappers yield the lowest
477   throughput, underscoring the critical role of well-designed mappers. The best mappers found by
478   Trace-OptoPrime consistently outperform the expert mappers, achieving speedups ranging from 9%
479   to 31%. For both PUMMA and Solomonik’s, the throughput of the best mapper discovered by
480   Trace-OptoPrime is significantly higher than the average optimization trajectory across 5 runs. This
481   variability is due to the inherent randomness of LLMs in our experiments, where even subtle changes
482
      in mappers can lead to notable performance differences. When comparing optimization trajectories
      of Trace-OptoPrime and Trace-OPRO, Trace-OptoPrime shows significant performance gains in
483
      SUMMA, PUMMA, Johnson’s, and COSMA, while achieving similar results on other benchmarks.

          6              C ONCLUSION
529
530      In this paper, we addressed the challenges of automating mapper generation in task-based program-
531      ming through the use of LLMs and a Domain-Specific Language (DSL). By designing a high-level
532      DSL, we effectively simplified the complex task of generating low-level C++ mappers, enabling
533      LLMs to handle mapper generation without deep knowledge of system intricacies. Additionally, we
534      formulated the mapper generation task as a discrete optimization problem, leveraging reinforcement
535      learning (RL) techniques to explore a structured and constrained search space defined by the DSL.
536      Our experimental results demonstrate the efficacy of this approach, with LLM-generated mappers
537      achieving up to 34% speedup over expert-written mappers across 9 benchmarks. For matrix multi-
538      plication algorithms, we observed a performance boost of up to 31%. These findings show that our
539      LLM-enhanced DSL significantly reduces development time from days to minutes, benefiting both
         human developers and the performance of parallel applications.

      We outline how our implementation translates our DSL into Legion’s low-level C++ mapping in-
811   terface. The Legion runtime (and other modern tasking runtimes) employs a pipelined execution
812   model, where multiple tasks are going through different stages of analysis and execution concur-
813   rently. Each pipeline stage handles a different portion of a task’s analysis and execution, and some
814   pipeline stages may interact with the user’s mapper through the mapping interface. Since many
815   tasks progress through the pipeline simultaneously, callbacks are often interleaved with callbacks
816   for other tasks.
817   Legion is a distributed runtime—multiple ranks of the runtime run in different parts of the machine.
818   Each instance of the runtime exclusively manages some subset of the machine’s resources.
819
820
      Figure A2 shows a simplified version of Legion’s task pipeline. We are not concerned with the
      details of what the stages do; we focus on the parts relevant to the mapping callbacks. We focus
821
      on the translation of IndexTaskMap and Region statements, as these illustrate all of the interesting
822
      issues.
823
824   When the application launches an index task, the local rank first invokes the shard function of the
825   user’s mapper to choose the ranks where subsets of the index launch should be sent to complete the
826
      rest of the mapping process. Some tasks may remain in the pipeline on the local rank, while other
      tasks are sent to other ranks for mapping. Regardless of where the tasks are sent, the subsets of tasks
827
      sent to each rank are themselves still represented as an index launch.
828
829   After the sharded index task launches arrive at the ranks where they will finish mapping, the call-
830   back slice_task is invoked. The slice_task function maps the tasks in the index launch onto
831   processors, typically (but not necessarily) the processors of the rank that the (subset of the) index
832
      launch has been sharded to. We translate IndexTaskMap commands into the needed shard_task
      and slice_task callbacks, ensuring the two are properly coordinated and any concurrency issues
833
      are avoided. The key to this translation is that all of the machine transformations are invertible,
834
      which makes it possible to take a processor selected in a transformed machine model and map it
835   back (possibly through multiple layers of machine transformations) to the corresponding processor
836   in the physical machine.
837
838
      After slice_task finishes, tasks in the pipeline are all individual point tasks. The distribution stage
      transfers any tasks that have been mapped to a processor owned by a different rank to their final
839
      destination. Once on the rank with the task’s assigned processor, the mapper callback map_task
840
      decides how to map the regions, i.e., in which memory a physical instance of the region will be
841   placed and with what layout. The compiler automatically generates the logic for map_task from the
842   region and layout statements.

855

856
      space m′ but DSL can translate such uses back into the original processor space m to identify which
857   concrete processors to use. In this example, m′ [j0 , j1 , j2 ] = m[j0 + j1 × 2, j2 ].

949   A.7     PARALLEL M ATRIX M ULTIPLICATION A LGORITHMS
950
951   2D Algorithms Cannon’s (Cannon, 1969) introduced a systolic communication pattern with
952
      tiled data partitioning for distributed matrix multiplication. PUMMA (Choi et al., 1994) and
      SUMMA (Van De Geijn & Watts, 1997) extended this approach by supporting non-square ma-
953
      trices and improving communication efficiency through pipelining. They are called 2D algorithms
954
      because they partition the matrices into 2D tiles and then map them onto the processor space.
955
956
      Non-2D Algorithms Johnson’s (Agarwal et al., 1995) introduced a 3D algorithm that partitions
957
      the input matrices into 3D tiles and uses additional memory per processor to reduce communication
958
      compared to 2D algorithms. Solomonik’s (Solomonik & Demmel, 2011) balances between 2D and
959   3D approaches by using extra memory to further minimize communication. COSMA (Kwasniewski
960   et al., 2019) takes a different approach by optimizing the processor grid and parallelization strategy
961   based on the input size and the machine size.
