009                                                    A BSTRACT
010
011             Predicting if LLMs will succeed on individual task instances (i.e., prompts) is es-
012             sential to ensure their reliability in high-stakes applications. To do so, we can eval-
013             uate a LLM on a set of instances and train an assessor to predict its performance.
014             However, this requires evaluating each new LLM on sufficiently many instances.
015             In this work, we build a generic assessor predicting the performance of any LLM
016             on an instance by using the LLM’s performance on a small set of reference in-
017             stances and the features of the considered instance. In practice, we make use of
018             existing evaluation results to extract the representative instances and train the as-
019
                sessor. Thus, the performance of a new LLM can be predicted by only testing it on
                the reference instances, leveraging the information contained in other LLMs’ eval-
020
                uations. We conduct empirical studies on HELM-Lite and KindsOfReasoning, a
021
                new collection of existing reasoning datasets that we introduce, where we evaluate
022             all instruction-fine-tuned OpenAI models until gpt4-0125-preview. We find
023             that a few instances (around 100) are enough to achieve predictive power compa-
024             rable to the LLM-specific assessors trained on the complete set of several thousand
025             instances. Interestingly, randomly selecting the reference instances performs com-
026             parably to the advanced selection methods we tested. Finally, we identify a sharp
027             drop in predictive power of the generic and specific assessors in out-of-distribution
028             scenarios, suggesting that the inherent predictability of LLMs is low.

031   1    I NTRODUCTION
032
033   Large Language Models (LLMs) are being used as components of multiple services and products,
034   such as agents performing general computer tasks (Kim et al., 2024), performing ML experiments
035   (Huang et al., 2024), and even operating unmanned aerial vehicles (Javaid et al., 2024). These sys-
036   tems typically query an LLM on a specific instance (i.e., a specific prompt) of a task and use their
037   output to determine an action. For some of these uses, it is essential to determine if the output
038
      produced by the LLM on a specific task instance is likely to be correct (or, more generally, “valid”
      (Zhou et al., 2023)) before the subsequent steps are executed1 . A nascent line of research (Zhou
039
      et al., 2022; Hernández-Orallo et al., 2022; Drapal et al., 2024) is addressing this problem by devel-
040
      oping “assessors”, namely, independent modules that predict the correctness (or a continuous score)
041   of an AI system on an instance based on features intrinsic to the latter (such as linguistic features or
042   sentence vector embeddings). Assessors can be specific to an AI system, or “generic”, in which case
043   they also take as input features of the AI system at hand and are trained to predict the performance
044   of different LLMs on different instances.
045
      Meanwhile, the rate at which new LLMs are released has drastically increased. Some providers, such
046
      as OpenAI, are iteratively retiring old versions when new ones are released, forcing developers to
047   update the LLM version used in their applications (see OpenAI (2024a)). An even larger explosion is
048   occurring in the open-source world, fuelled by inexpensive fine-tuning techniques (Hu et al., 2022).
049   To build an assessor specific to a new LLM version, users must evaluate it on a sufficiently large set
050   of task instances, causing the costs to rise quickly when considering many LLM versions. On the
051   other hand, the system information one might use to build a generic assessor, such as the number

      As such, this paper investigates the following question: can we combine information across LLMs
075
      to predict the performance of a new LLM on a new instance by relying only on observational
076
      (or behavioural) features of the LLMs? In practice, we propose to characterise each LLM by
077   its performance on a small set of reference instances and to build a generic assessor using those as
078   system features. More precisely, we first select a small set of reference instances from the labelled
079   dataset on which past LLMs were evaluated. Then, we train the generic assessor on the concate-
080   nation of instance-specific features and the LLM-specific success vector on the reference instances.
081   Finally, to estimate the probability of success of a new LLM on a novel instance, it suffices to evalu-
082   ate the former on the reference instances, concatenate its performance to the features of the instance,
083   and apply the trained generic assessor. See Fig. 1 for a graphical representation of this procedure.
084   In our empirical studies, we rely on HELM-Lite (Liang et al.), which provides instance-level results
085   for 30 LLMs from different providers (at the time we conducted our experiments), and a collection
086   of previously existing datasets we introduce, named “KindsOfReasoning”, on which we evaluated
087   the full set of instruction-following models from OpenAI until gpt4-0125-preview. We only
088   consider tasks with binary correctness score (therefore discarding the datasets in HELM-Lite that
089   do not satisfy this) and thus build binary assessors.
090   We train specific assessors using different prompt features and find that OpenAI embeddings (Ope-
091   nAI, 2024b) lead to better or comparable in-distribution performance than simpler methods such
092   as Word2vec (Mikolov et al., 2013), FastText (Bojanowski et al., 2017), and n-grams. Subse-
093   quently, we build generic assessors using various methods to select the reference instances and com-
094   bine the performance on these with the instance-specific features. When predicting performance on
095   instances with the same distribution as those used to train the generic assessor, we find the latter to
096   perform comparably to the specific assessors, which require the LLM to be evaluated on many more
097
      instances. Additionally, we find that a random selection of reference instances performs as well as
      the advanced selection methods we tested. However, in out-of-distribution scenarios, the predictive
098
      power of all assessors declines significantly, indicating a lack of general predictability in LLMs.
099
100   In essence, the main contributions of our work are the following:
101
102
             • We propose a framework combining evaluation results across LLMs to predict the perfor-
               mance of a new LLM by only evaluating it on a a small set of reference instances.
103
104          • We study the performance of various methods for selecting the reference instances and
105            combining their performance with instance-specific features to build the generic assessor.
106          • Finally, we introduce KindsOfReasoning: A new compilation of existing datasets testing
107            various kinds of reasoning and release the raw outputs of a large number of models from
               OpenAI.

      2       R ELATED WORK
109
110
      2.1      I NSTANCE - LEVEL PREDICTION OF SUCCESS OF AI SYSTEMS
111
112   Zhou et al. (2023) advocates for the importance of instance-level success predictions for AI systems
113   and coins the term “predictable AI”; in particular, they highlight how ensuring predictability should
114   be prioritised over increases in average performance for high-stakes use cases. Following this
115   motivation, Hernández-Orallo et al. (2022) introduces the concept of an assessor model, which
116   accompanies an ML system and estimates the probability of success of the system on individual
117
      instances. In particular, an assessor can be trained on the evaluation results of the ML system on
      test data (i.e., which has not been used for training the ML system). In similar spirit, Drapal et al.
118
      (2024) combines novelty detection and meta-learning to reject instances where a ML system is
119
      likely to fail. Other similar approaches are described in Section 4 of Hendrickx et al. (2024). Zhou
120   et al. (2022) shows how a smaller LLM can be used to predict the performance of a bigger LLM on
121   individual instances without passing the latter through the model. They also find it possible to reject
122   almost half of the failure cases before running much larger LLMs, resulting in a significant saving
123   of compute. Finally, an application of success prediction is ”routing” between different LLMs, as
124   explored in Shnitzer et al. (2023) and Hu et al. (2024).
125
126   2.2      P REDICTABILITY OF AGGREGATED BENCHMARK SCORES FROM LLM FEATURES
127
128   Two works (Ye et al., 2023; Owen, 2024) studied the extent to which an LLM’s aggregate per-
129   formance on BIG-Bench tasks (Srivastava et al., 2022) can be predicted using information on the
130
      LLM such as number of parameters or the amount of used compute. In contrast, our work does
      not rely on these quantities, which are often unavailable, instead characterising LLMs according to
131
      their performance on reference samples. Moreover, while these works focus on predicting aggregate
132
      performance, our work and the ones mentioned in the previous subsection provide instance-level
133   predictions for new unlabelled instances.
134
135
      2.3      E XTRACTING LLM- SPECIFIC FEATURES FROM EXISTING EVALUATIONS
136
137   Recently, Ruan et al. (2024) built “observational scaling laws” that link performance on complex
138   downstream tasks to hypothesised latent capabilities, whose values can be inferred by decomposing
139   the performance of various LLMs on different benchmarks into components linked by a log-linear
140   relation with compute measures for LLM training. Once this relation is established, the performance
141   of a new model on downstream tasks can be predicted by knowing its performance on simple
142   benchmarks and its compute cost. Their work is similar to ours in determining LLM-specific
143
      features by using evaluation results of multiple LLMs and using them to predict the performance
      of a new LLM. However, we aim to predict the performance of the new LLM on a novel individual
144
      instance by evaluating on as few instances as possible, while Ruan et al. (2024) instead aims to
145
      avoid the cost of evaluating complex downstream tasks and predict the performance on the latter
146   from results on simple benchmarks and compute measures, which they assume to be available.
147   Moreover, our method can be applied to predict the performance on instances for which no ground
148   truth is available, while the simple benchmarks and the downstream tasks employed in Ruan et al.
149   (2024) must have a grading mechanism.
150
151   2.4      P REDICTING PERFORMANCE BY BENCHMARK SUBSAMPLING
152
153   Several works share our motivation of reducing the number of evaluations (and hence the cost)
154   needed to evaluate a LLM. For instance, a “Lite” version with a reduced number of tasks was intro-
155   duced alongside the BIG-Bench benchmark (Srivastava et al., 2022); similarly, HELM-Lite (Liang
156
      et al.) is a revised and reduced version of HELM (Liang et al., 2022). However, both of these
      perform the reduction at the level of tasks (i.e., datasets) of which the benchmark is constituted.
157
      Instead, Vivek et al. (2024) subsample a dataset by clustering models’ confidence to predict the
158
      overall accuracy on the whole dataset, while MixEval (Ni et al., 2024) extracts a subset of instances
159   from various benchmarks which is most predictive of the performance on Chatbot Arena2 , an online
160   platform performing pairwise comparison of LLM outputs.
161

162
      Closer to our work is TinyBenchmarks (Polo et al., 2024), which selects informative instances from
163   HELM-Lite and estimates the performance of a new LLM on the whole benchmark by evaluating
164   it only on those instances. In particular, TinyBenchmarks uses Item Response Theory (IRT) on the
165   successes of each LLM to extract a vector of item demands and LLM capabilities. Then, it uses
166   either the item demands or the raw LLM success on each instance to build a representative subset
167   of instances by clustering the items and taking the cluster centroids. Similarly to our approach, a
168   new LLM is then only evaluated on the representative subset; however, in contrast to our work, they
169   aim to predict the aggregate score on the benchmark, while we predict instance-level performance.
170   In practice, their IRT method provides instance-level predictions, but these predictions are limited
171
      to instances on which previous LLMs have been evaluated (as this is necessary to obtain the item
      demands), which requires access to the ground truth. In contrast, our approach relies on “intrinsic”
172
      (model-agnostic) features of the instances (alongside the performance on the reference samples,
173
      see Fig. 11), thus making it applicable to new instances with unknown ground truth, as the trained
174   assessor does not require any information beyond the intrinsic features of test instances.
175
176   A similar work to Polo et al. (2024) is metabench (Kipnis et al., 2024), which considered 6
177
      different datasets, and performed a two-step procedure (random sampling for each dataset, followed
      by item selection based on the Fisher information matrices of IRT item parameters) to extract a
178
      small set of instances, the performance on which accurately predicts aggregate performance on the 6
179
      datasets. As they fit the IRT model only the pre-selected instances, their method is unable to predict
180   instance-level performance. Finally, despite not tackling predictability directly, Siska et al. (2024)
181   finds that the vector of successes of different LLMs is correlated across instances belonging to 4
182   benchmarks, and, for one of those benchmarks, the similarity between the embeddings or a pair of
183   instances predicts the similarity between the success vectors; this suggests that patterns in success
184   across LLMs can be found and related to the embeddings.

      2.5    E VALUATIONS OF REASONING IN LLM S
187
188   Burnell et al. (2023a) found reasoning to be one of three factors in the capabilities of LLMs. Indeed,
189   reasoning in LLMs has been extensively studied: see Mondorf & Plank (2024) for a survey on LLM
190   reasoning evaluations and Huang & Chang (2023) for a broader survey also encompassing ways to
191   improve and elicit reasoning in LLMs.
192
      Recently, several collections of reasoning datasets have been introduced. GLoRE (Teng et al., 2023)
193   collects 12 logical reasoning datasets with three different types of tasks (multiple choice, natural lan-
194   guage inference, and binary answers). Similarly, LogiGLUE (Luo et al., 2023) collects 24 datasets
195   related to inductive, deductive and abductive reasoning, with four different types of tasks (the same
196   ones as GLoRe and free-form question answering); they only selected datasets that do not require
197   external domain knowledge, but some of these datasets are poorly formatted. Finally, CALM-Bench
198   (Dalal et al., 2023) is a collection of 6 diverse tasks requiring both causal reasoning and knowledge.
199   KindsOfReasoning, the collection we introduce combining previously existing datasets testing vari-
200   ous kinds of reasoning, partly overlaps with each of the aforementioned collections; however, Kind-
201
      sOfReasoning aims to include a broader range reasoning types (logical, common sense, inductive,
      deductive, abductive, counterfactual, causal, analogical, spatial and arithmetic reasoning) over 22
202
      different datasets; see Appendix B for more information on the dataset construction.

222   3.1    P REDICTING SUCCESS OF A LLM USING FEATURES INTRINSIC TO THE PROMPT
223
      Let us consider a single LLM, say m1 ; we aim to train a classifier (termed “assessor”) to predict the
224
      performance z1,i from the prompt pi . To do so, we split the test dataset D into different splits used
225

226
      and Dtest , such that D = Dtrain ∪ Dval ∪ Dtest and Dtrain ∩ Dval = Dval ∩ Dtest = Dtrain ∩ Dtest = ∅.
227   In a real-world scenario, Dtest will represent instances for which we did not evaluate the considered
228   LLM (and for which we may not have access to the ground truth); in contrast, available evaluation
229   results are split into Dtrain and Dval .
230

231

232
      of negations or the vector embeddings of the sentence). Formally, we consider a loss function ℓ and
233   a family of classifiers hφ , where φ denotes the parameters of the classifier (for instance, the weights

288
      of clusters dictates the number of selected instances.

294

295

300   these methods with a random reference subset; moreover, we also draw 20 random reference
301
      subsets, fit an assessor using the performance on the reference instances, and pick the random

305
306   As mentioned above, we have multiple ways to define the reference set as well as multiple choices for
307   the intrinsic features f . We can also choose multiple families of classifiers hφ and hyperparameters
308   of the optimisation algorithm to minimise equation 2. As such, we pick the combination of options

      4.3   W HAT PROMPT FEATURES LEAD TO BETTER PREDICTIVE PERFORMANCE ?
413
414
415   We first train an assessor specific to each considered LLM to identify the set of prompt features f
416   that maximizes predictive performance. In particular, we consider the prompt embeddings computed
417
      from the OpenAI API (with the text-embedding-3-large endpoint OpenAI (2024b)) and
      those obtained with Word2vec (Mikolov et al., 2013) and FastText (Bojanowski et al., 2017);
418
      the latter two generate a vector for each word in the prompt, which we average to form a vector
419
      representing the entire prompt. Lastly, we consider 1-gram vectors, calculated as the frequency of
420   words in a specific prompt, normalized by the frequencies across the entire set of training prompts.
421   For each choice of features and test LLM, we train various base classifiers (logistic regression with
422   l2 and l1 penalty and xgboost) on Dtrain , compute the AUC of each on Dval , pick the one with the
423   highest validation AUC, and report the AUC of that classifier on Dtest .
424
      Our results, available in Fig. 2, show that the OpenAI embeddings always perform better for the
425
      KindsOfReasoning dataset, while no clear winner emerges for HELM-Lite, where all features lead
426   to similar performance . Therefore, we will use the OpenAI embeddings in all experiments below.
427   Moreover, the OpenAI embeddings obtained from the endpoint text-embedding-3-large
428   were trained using Matryoshka Representation Learning (Kusupati et al., 2022), which allows them
429   to be truncated (by removing the final elements of the vector) without the embedding losing its
430   concept-representing properties. As such, we investigate the performance of the specific assessor
431   by truncating the OpenAI embeddings (Fig 3) and found that the performance saturates using 1024
      (out of a total of 3072) embeddings; hence, we’ll apply this truncation below.

      5     C ONCLUSION
524
525
      We proposed a novel framework for predicting the performance of a new LLM on individual task
526
      instances by leveraging the evaluation results of previously tested LLMs. Our approach minimises
527
      the number of evaluations required for a new LLM by introducing a generic assessor combining
528   instance-specific features with LLM-specific ones derived from performance on a small set of
529   reference instances. While we focus on LLMs, our methodology can be seamlessly applied to
530   predict the performance of other AI systems, by using suitable system-specific and instance-specific
531   features. Similarly, our approach can also be extended to non-binary correctness metrics, the
532   investigation of which we leave to future work.
533
      Our empirical studies on the HELM-Lite and KindsOfReasoning dataset collections showed how
534   the generic assessor performs only slightly worse than the specific one in distribution, while outper-
535   forming simpler baselines. Moreover, we found that the generic assessor is mostly unsensitive to the
536   specific set of reference instances used. Finally, out of distribution, the predictive performance de-
537   creases drastically for all methods, which raises awareness of the low inner predictability of LLMs.
538   To foster research in making LLMs more predictable (Zhou et al., 2023), we release the instance-
539   level results of all instruction-finetuned GPT3 and GPT4 models until gpt4-0125-preview on
      KindsOfReasoning.

844   We repeat all experiments discussed in the main text (Sec. 4) considering different choices for the
845   train, validation, and test splits Dtrain , Dval and Dtest for both dataset collections. The main text

847   instances of all datasets. In addition, we consider multiple out-of-distribution (OOD) splits, where
848   we keep one set of datasets as Dtest (according to some criteria), and Dtrain and Dval are obtained

851   the data where the selected assessor will be evaluated on. Details on the various splits are given in

      Table 4. Section 4.5 in the main text reported results using the second OOD splits for both dataset
      collections.
