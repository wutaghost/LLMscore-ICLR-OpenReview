008                                               A BSTRACT
009
               Although large Language Models (LLMs) and LLM agents have been widely
010            adopted, they are vulnerable to indirect prompt injection attacks, where malicious
011            external data is injected to manipulate model behaviors. Existing evaluations of
012            LLM robustness against such attacks are limited by handcrafted methods and
013            reliance on white-box or gray-box access—conditions unrealistic in practical de-
014            ployments. To bridge this gap, we propose AutoHijacker, an automatic indirect
015            black-box prompt injection attack. Built on the concept of LLM-as-optimizers,
016            AutoHijacker introduces a batch-based optimization framework to handle sparse
017            feedback and also leverages a trainable memory to enable effective generation of
018
               indirect prompt injections without continuous querying. Evaluations on two public
               benchmarks, AgentDojo and Open-Prompt-Injection, show that AutoHijacker out-
019
               performs 11 baseline attacks and achieves state-of-the-art performance without
020
               requiring external knowledge like user instructions or model configurations, and
021            also demonstrates higher average attack success rates against 8 various defenses.
022            Additionally, AutoHijacker successfully attacks a commercial LLM agent platform,
023            achieving a 71.9% attack success rate in both document interaction and website
024            browsing tasks.

027   1   I NTRODUCTION
028   Large Language Models (LLMs) (Brown et al., 2020; Touvron et al., 2023; OpenAI, 2023; Anthropic,
029   2024) have revolutionized various domains by enabling sophisticated natural language processing
030   tasks with unprecedented accuracy and flexibility. These models, empowered by vast amounts of
031   data and complex architectures, are now embedded into a wide array of applications and intelligent
032   agents (LangChain, 2023; Weber, 2024; Gravitas, 2023; Yao et al., 2022b; Wang et al., 2023b; Yao
033
      et al., 2022a), reshaping industries ranging from customer service to content generation. The profound
      impact of these models, however, comes with substantial challenges in security and trustworthiness.
034
035   Indirect Prompt Injection Attacks. A significant threat is indirect prompt injection attacks (Gre-
036   shake et al., 2023; Yi et al., 2023; Debenedetti et al., 2024). They occur within LLM-integrated
037   applications and agents when a query combines user instructions with external data. If this ex-
038   ternal data is manipulated to include hidden commands, LLMs, which process inputs in natural
039
      language, may inadvertently execute these hidden instructions. This occurs because LLMs often
      cannot distinguish between legitimate user commands and maliciously crafted external inputs, leading
040
      to altered user instructions and potentially harmful outcomes. With the growing need to understand
041
      and mitigate indirect prompt injection vulnerabilities in LLMs and LLM agents, it is essential to
042   explore the practical threat landscape these systems face in real-world scenarios. To this end, it is
043   important to develop a black-box attack method that fully reveals how resilient LLMs and LLM
044   agents are against indirect prompt injection inputs. This approach would imitate real-world scenarios
045   where attackers launch attacks without detailed knowledge of the models’ internal workings.
046
      Limitations of Existing Efforts. Existing works on prompt injection attacks have two primary
047
      limitations. Firstly, many of these methods are handcrafted rather than automated (Perez & Ribeiro,
048   2022; Toyer et al., 2024; Liu et al., 2024b; Debenedetti et al., 2024). While these approaches can
049   occasionally expose critical vulnerabilities, they are limited by the designer’s cognitive biases and do
050   not provide a comprehensive robustness exploration. In contrast, automated methods in adversarial
051   machine learning (Goodfellow et al., 2015; Croce & Hein, 2020; Zou et al., 2023), have demonstrated
052   the ability to systematically conduct robustness evaluation via generating adversarial examples. The
053   absence of automated methods leaves a gap in thoroughly assessing the robustness of LLMs and
      LLM agents against indirect prompt injection attacks.

073   Secondly, despite a few studies making efforts on automated method Liu et al. (2024a); Pasquini
074   et al. (2024), these methods rely on white-box or gray-box accessibility to the victim models. The
075   white-box attacks require full access to the model’s internal parameters and loss functions, while
076
      the gray-box approaches depend on internal knowledge, such as prompt information or specific
      configurations of the model. For instance, white-box gradient-based attacks, such as MGCG (Liu
077
      et al., 2024a) and NeuralExec (Pasquini et al., 2024) attacks, depend on having access to the internal
078
      gradients of the model. And gray-box attacks like the tool-knowledge attack Debenedetti et al. (2024)
079   and the combined attack (Liu et al., 2024b) need to know the internal design, such as users’ prompts
080   and tool information, of the victim LLMs and LLM agents. However, these accessibilities to the
081   victim models are often impractical in real-world scenarios where such access is restricted. Therefore,
082   there is a significant need for automated prompt injection robustness evaluation techniques that can
083   operate under black-box conditions, reflecting real-world challenges more accurately.
084
      To bridge this gap, we propose AutoHijacker, an automatic prompt injection vulnerability scanning
085   tool that is designed to evaluate the indirect prompt injection robustness of victim models (and agents)
086   under black-box conditions. Our method can automatically identify potential vulnerabilities in LLMs
087   and LLM agents without human intervention. Specifically, our approach is built upon the concept of
088   LLM-as-optimizers (Yang et al., 2024; Yuksekgonul et al., 2024; Chao et al., 2023; Mehrotra et al.,
089   2024) and utilizes LLMs to generate indirect prompt injection test cases. LLM-as-optimizers use
090   the responses provided by target LLMs as feedback to generate corresponding test cases. Despite
091   in scenarios where LLMs-as-optimizers are proven to be effective, such as in jailbreak attacks, one
092   significant challenge in applying LLM-as-optimizers for prompt injection attacks is the sparse
093
      nature of feedback from victim models. Typically, LLM-as-optimizers require very fine-grained
      feedback to simulate a “gradient,” allowing them to optimize and produce better solutions. Yang
094
      et al. (2024); Yuksekgonul et al. (2024). In contrast, prompt injection attacks typically receive sparse
095
      feedback. As shown in Fig. 1, when an attacker repeatedly attempts to execute attacks against a
096   victim LLM, in most cases, the model’s response does not reflect the injected content, leading to the
097   issue of “sparse” feedback and poor performance. To address this issue, we introduce a batch-based
098   optimization framework. By optimizing over a batch of data during the training stage, the model can
099   better handle sparse feedback states, smoothing the optimization process. Then a key challenge
100   posed by batch-based optimization is how to transfer and apply the attack knowledge discovered
101   by LLM-as-optimizers across different samples. Our method addresses this by implementing a
102   two-stage attack strategy and constructing an attack memory. During the training phase, we carefully
103   build an attack memory that selects and stores the history of previous attacks. In the test phase, we
104
      leverage this log to generate effective prompt injection test cases. Excitingly, This design allows for
      attacks without the need for continuous querying during testing, making it especially valuable in
105
      real-world scenarios where service providers may limit the number of allowed queries.
106
107   To evaluate the effectiveness of our method, we conducted comprehensive assessments using two
      public benchmarks: AgentDojo Debenedetti et al. (2024) and Open-Prompt-Injection Liu et al.

      2       R ELATED W ORKS
119
120   Prompt injection attacks have emerged as a significant threat to LLMs and their applications. Because
121
      LLMs are designed to process inputs in natural language, they often struggle to distinguish between
      user commands and external inputs, making them vulnerable to such attacks. This vulnerability has
122
      been extensively documented in recent studies (Greshake et al., 2023; Wang et al., 2023a; Pedro et al.,
123
      2023; Yan et al., 2023; Yu et al., 2023; Salem et al., 2023; Yi et al., 2023; Yip et al., 2024; Debenedetti
124   et al., 2024; Zhan et al., 2024b; Liu et al., 2024a; Pasquini et al., 2024; Shi et al., 2024). The
125   phenomenon was first identified in academic research by Perez & Ribeiro (2022), who demonstrated
126   that LLMs could be misdirected by simple, handcrafted inputs, leading to goal hijacking and prompt
127   leakage. Liu et al. (2023) developed a framework for prompt injection attacks, applying it to study
128   36 LLM-integrated applications and identifying 31 as vulnerable. Further research has evaluated
129   handcrafted prompt injection methods for both goal hijacking and prompt leaking (Toyer et al., 2023),
130   as well as scenarios where attackers aim to shift the LLM’s task to a different language (Liu et al.,
131   2024b). Prompt injection vulnerabilities in LLM agents have also been assessed in (Debenedetti et al.,
132
      2024; Zhan et al., 2024a). Beyond academic findings, online posts (Harang, 2023; Willison, 2022;
      2023) have highlighted the risk of prompt injection across various commercial LLM platforms, raising
133
      widespread concerns. In this paper, we focus primarily on indirect prompt injection attacks (Greshake
134
      et al., 2023; Yi et al., 2023; Zhan et al., 2024a; Liu et al., 2024a; Abdelnabi et al., 2024), where the
135   injection data originates from external resources. Existing prompt injection attacks have significant
136   limitations. They are mainly handcrafted rather than automated, limiting systematic exploration due
137   to human biases. Moreover, many strong attacks depend on white-box or gray-box access to models,
138   requiring internal parameters or configurations such as user instruction and tool knowledge. Such
139   access is impractical in real-world black-box scenarios. Our work addresses these limitations by
140   introducing a black-box automatic indirect prompt injection attack.

145
      Preliminaries. Our objective is to design an algorithm that can automatically convert original
      external data (e.g., documents, websites) into injected data that misleads LLMs and LLM agents
146
      into achieving an unintended attack goal when processing these external data. Formally, we aim to
147
      develop an algorithm Fθ that satisfies the following condition: I(LM (U, Fθ (D)), G) = 1, where
148   LM represents the victim LLMs or LLM agents, U represents the user instruction (e.g., “Summarize
149   this PDF.”), D represents the original external data (e.g., a PDF document), and I(·, ·) is an indicator
150   function that determines whether the former input satisfies the latter input.2 Specifically, in the above
151   formulation, it judges whether the output of LM satisfies the attack goal G.
152
      Threat Model. We assume that the attack algorithm cannot access internal information about the
153
      victim model’s response process. This includes internal outputs (e.g., the intermediate actions of
154   LLM agents), the user’s requests, knowledge of tool functionalities, or any user-specific information,
155   such as the user’s name. These types of information are often leveraged in existing attack methods
156   to construct stronger attacks, as discussed in Sec. 1 while, in practice, it is typically infeasible for
157   attackers to obtain such details. We assume the attacker can have a reasonable guess about the
158   foundation LLM used behind the victim system but does not have white-box access (e.g., knowing the
159   detailed parameters of the model) to it. The attacker can only observe the responses of the foundation

184   To achieve our goal, we introduce AutoHijacker, an automated black-box indirect prompt injection

      Framework Structure. We introduce three LLMs that cooperate in a multi-agent system, consisting
188
      of an attacker, a prompter, and a scorer. The prompter takes the original external data D, the attack
189
      goal G, and a trained attack memory A, and outputs a meta-prompt M containing design instructions
190   to guide the attacker in generating effective injection data. The attacker, using the meta-prompt
191
      M , original external data D, and attack goal G, generates the injection data D̂. The scorer takes
192   the response from the LM and the attack goal G, and returns a score S. This score will guide the
193   subsequent rounds of generation, as we will describe later. Note that in our framework, we introduce
194   an individual prompter to generate a meta-prompt that guides the attacker, rather than having the
195   attacker directly generate injection data based on the input or using approaches like Chain-of-Thought
196   (CoT) (Wei et al., 2022). This design ensures that clearer instructions are provided to the attacker,
197   mitigating potential performance drops that could occur due to long-context scenarios, especially
198   when the attack memory is provided entirely to the attacker.
199   Attack Pipeline. AutoHijacker operates in two main stages: the training stage and the test stage.
200   In the training stage (Sec. 3.2), it develops the attack memory mentioned earlier. In the test stage
201   (Sec. 3.3), it utilizes the trained attack memory to perform a one-step generation of injection data.

203   3.2   T RAINING S TAGE - BATCH - BASED OPTIMIZATION
204
205
      Handling the Sparse Feedbacks. In Sec. 1, we mentioned that a significant challenge in prompt
      injection attacks is the sparse feedback they typically receive, whereas LLMs-as-optimizers usually
206
207
      rely on fine-grained feedback. As shown in Fig. 1, for a single injection data D̂ generated by Fθ , the
      feedback states across multiple query times will likely remain the same, leading to sparse feedback
208
      states. Namely, both the previous and current rounds of optimization may likely yield similarly low
209
      scores, making the optimization process difficult to advance. To address this issue, we argue that
210   generating multiple diverse injection data instead of a single instance can mitigate the sparsity of
211   feedback. This is because, for different injection data, the feedback states are less likely to align (i.e.,
212   scenarios where the scorer returns all zeros for the entire batch are less likely to occur). In this case,
213   it is more likely that certain instances will result in more effective injection data compared to the
214   previous round during optimization rounds. Leveraging this information to optimize on a broader
215   set of data increases the chances of discovering further opportunities for improvement, allowing the
      scores to continue improving and guiding the optimization process in a productive direction.

      Thus, the training stage of AutoHijacker focuses on optimizing the generation of effective prompt
217   injection inputs by leveraging a batch-based optimization framework. As depicted in Alg. 1, the
218   training process operates over N training data points, each consisting of external data Dn , an attack
219   goal Gn , and a user instruction Un . The attack goal Gn specifies the desired malicious behavior we
220   aim to induce in the victim model. For each epoch i (up to a maximum of I epochs), and for each
221   data point n, the following steps are performed:
222     1. Meta Prompt Generation: We generate a meta prompt Mi,n using the prompter LLM that takes
223        the current attack memory A, the external data Dn , and the attack goal Gn as inputs. The meta
224        prompt encapsulates potential attack strategies and guides the attacker to generate injection data.
225
        2. Injection Data Generation: An attacker LLM uses the meta prompt Mi,n , along with Dn and
226
           Gn , to produce the injection data D̂i, n. This injection data is designed to manipulate the victim
227
           model into exhibiting the desired malicious behavior specified by Gn .
228
        3. Victim Model Interaction: The injection data D̂i, n is combined with the user instruction Un and
229
           input to the victim foundation LLM. The victim foundation LLM generates a response Ri,n .
230
        4. Scoring: A scorer LLM evaluates the victim model’s response Ri,n against the attack goal
231
           Gn , producing a score Si,n . The score reflects how successfully the injection data induced the
232
           desired behavior in the victim model.
233
        5. Attack Memory Update: The data point, along with its score, is added to the attack memory A
234        according to the procedure outlined in Alg. A. The attack memory retains the most effective and
235        least effective attacks, which are used to inform future generations of injection data.
236
      Attack Memory Construction. The above batch-based optimization requires the sharing of attack
237
      knowledge between different training samples. To address how the attack knowledge discovered by
238
      LLM-as-optimizers can be transferred and applied across different samples, we introduce the attack
239   memory A. This critical component of AutoHijacker acts as a repository for past attacks and their
240   effectiveness, guiding the generation of future injection data by offering examples of both successful
241   and unsuccessful attacks. As outlined in Alg. A in the appendix, the attack memory is updated
242   after each iteration during the training stage. When a new data point Dn , Gn , Mi,n , D̂i, n, Si, n is
243   obtained, the following steps are performed:
244
        1. Memory Augmentation: The new data point is added to the existing attack memory A, resulting
245        in an augmented memory A′ .
246     2. Scoring and Sorting: All entries in A′ are associated with their respective scores Sj . The entries
247        are sorted in descending order based on the scores to identify the most effective attacks and in
248        ascending order to identify the least effective ones.
249     3. Memory Pruning: To maintain a manageable size and focus on the most informative examples,
250        we retain the top ktop entries with the highest scores and the bottom kbottom entries with the
251        lowest scores. These entries constitute the updated attack memory A.
252   By retaining both the most and least successful attacks, the attack memory provides a balanced
253   perspective that helps the prompter and attacker LLM generate effective injection data. The inclusion
254   of unsuccessful attacks is important as it informs the model about strategies that do not work,
255   preventing it from repeating ineffective approaches. Moreover, it enables our method to perform
256   one-step generation during the test stage, eliminating the need for additional queries.
257

      3.3   T EST S TAGE - O NE S TEP G ENERATION
259   In the test stage, AutoHijacker leverages the attack memory A constructed during the training stage
260   to generate effective prompt injection inputs without the need for iterative optimization or continuous
261   querying on the victim model. Superficially, given new external data D, an attack goal G, the
262
      following steps are performed: (1). Meta Prompt Generation: The prompter LLM generates a meta
      prompt M by utilizing the attack memory A, along with the external data D and attack goal G; (2).
263
      Injection Data Generation: The attacker LLM uses the meta prompt M , along with D and G, to
264
      generate the injection data D̂. This step mirrors the injection data generation in the training stage but
265
      relies solely on the attack memory without additional interaction with the victim model.
266
267   Details can be found in Alg. B in the Appendix. By using the attack memory to inform the generation
268   process, AutoHijacker can produce potent prompt injection inputs in a single step, suitable for
269   black-box settings where querying the victim model may be limited or infeasible. After generating
      the injection data D̂, it can be further evaluated using indirect prompt injection evaluation protocols,

298   Setups. The AgentDojo benchmark (Debenedetti et al., 2024) consists of test suites across four
299   distinct environments: Workspace, Slack, Travel, and Banking. The benchmark features a total of
300   70 tools, 97 realistic user tasks, and 27 injection tasks. We utilize the attack success rate (ASR,
301   denoted as target attack success rate in the original paper) as the metric. For baselines, we use the
302   attacks that are already included in the AgentDojo benchmark as baselines, including Direct, Ignore
303
      Previous, Important Instructions, Tool Knowledge, and InjectAgent. In addition, we introduce three
      additional baselines. These baselines share a similar ideology to our method, which are also built
304
      on LLM-as-optimizer. The first is HOUYI (Liu et al., 2023), which is a query-based direct prompt
305
      injection attack. The second and third are PAIR (Chao et al., 2023) and TAP (Mehrotra et al., 2024),
306   which are query-based jailbreak attacks, and we extend them into prompt injection attacks. Unless
307   specified otherwise, we set the query times of these three query-based attacks as 20 in this and the
308   following evaluations. We choose this number of queries to achieve the best performance under a
309   similar computational cost compared with our method. For defenses, we evaluate the defenses that
310   are included in the benchmark, including three defenses while excluding those that significantly
311   influence the benign performance of the LLM agent. These three defenses are Spotlighting with
312   Delimiting, Repeat User Prompt, and Tool Filter. Details are in Appendix A.2.

460   Batch-based Optimization. In our method design, we argue that leveraging a batch of diverse
461   data can mitigate the issue of sparse feedback in indirect prompt injection attacks, making it more
462   feasible for LLMs-as-optimizers to work effectively. Here, we evaluate this claim and evaluate
463   the effect of different algorithm designs. Specifically, we compare two approaches: (1) Single-
464   instance optimization, which uses the same data throughout training like existing LLM-as-optimizers
465
      attacks (Liu et al., 2023; Chao et al., 2023; Mehrotra et al., 2024), and (2) Batch-based optimization,
      which uses a batch of different data to jointly training the injection data, following the setup of our
466
      method as outlined in Sec. 4.1. We present the average score curves across all training samples,
467
      with consistent training epochs maintained for both approaches. As shown in Fig. 2, the training
468   score curves demonstrate that our batch-based optimization addresses sparse feedback problem. The
469   batch-based approach provides richer feedback signals, enabling continuous improvement.

511   Construction of Attack Memory. In our method, the construction of the attack memory involves
512   two hyperparameters. The first is the selection of ktop and kbottom . By setting these values as non-zero,
513
      we can store both the most effective and least effective attacks in the attack memory, thereby using a
      “contrastive learning-like” approach. We evaluate this design by comparing it to another approach,
514
      top-k sampling, where only the most effective k attacks are saved in the attack memory. Specifically,
515
      we test the effectiveness of our method with contrastive sampling of the attack memory (ktop = 15
516   and kbottom = 15) and top-k sampling of the attack memory (k = 30). As presented in Tab. 6
517   (Top), the contrastive sampling method outperforms the top-k sampling across all models, with ASR
518   improvements ranging from approximately 10% to 14%. The inclusion of both the most and least
519   effective attacks allows the attacker to learn from a wider range of examples, akin to contrastive
520   learning. This approach helps the attacker discern not only what strategies lead to success but also
521   what leads to failure, enabling the avoidance of ineffective patterns. The enhanced learning through
522   contrast and the prevention of overfitting to specific attack patterns contribute to higher ASR.
523   Another hyperparameter is the length of the attack memory. We evaluate its influence by setting
524   the length of the attack memory to 10, 20, and 30, respectively, and testing the ASR of our method.
525   Our findings, shown in Tab. 6 (Bottom Right), reveal that increasing the attack memory length
526   generally enhances the ASR for Llama-3.1-70B and Command-R+, with ASR values rising as the
527   memory length increases. For example, Llama-3.1-70B’s ASR improves from 46.1% at length 10
528   to 62.4% at length 30. However, for GPT-4o, the highest ASR occurs at a memory length of 20
529   (71.0%), suggesting an optimal memory size beyond which performance may plateau or decline
530
      due to cognitive overload. GPT-4o-mini exhibits fluctuating performance. On average, our method
      achieves the best performance with a memory length of 30. These results suggest that an appropriately
531
      longer memory length can provide richer information for the attacker to exploit.

533   5      C ONCLUSIONS AND L IMITATIONS
534   We introduce AutoHijacker, an automatic black-box indirect prompt injection attack against LLMs
535   and LLM agents. By addressing the challenge of sparse feedback with batch-based optimization
536   and an attack memory, our method effectively generates test cases without continuous querying.
537   Experiments demonstrate state-of-the-art performance on public benchmarks and commercial LLM
538   agents. A limitation of our approach is that it requires query time during the training phase, despite
539   enabling one-step generation in the testing phase. Additionally, the proposed method achieves better
      performance when the attacker knows the foundation LLM used by the agent.

      Algorithm A Attack Memory Construction
865
866    1: Input: Data point {Dn , Gn , Mi,n , D̂i,n , Si,n }, previous attack memory A
       2: Parameter: ktop (number of top scores to retain), kbottom (number of bottom scores to retain)
867
       3: Initialize: A′ ← A ∪ {{Dn , Gn , Mi,n , D̂i,n , Si,n }}
868
       4: Extract scores: S = {Sj | {Dj , Gj , Mj , D̂j , Sj } ∈ A′ }
869
       5: Sort A′ in descending order of Sj to obtain Asorted_desc
870    6: Let Atop = Asorted_desc [0 : ktop ]
871    7: Sort A′ in ascending order of Si to obtain Asorted_asc
872    8: Let Abottom = Asorted_asc [0 : kbottom ]
873    9: Update attack memory: A ← Atop ∪ Abottom
      10: return Updated attack memory A

888   A.1    OVERVIEW
889
890   We evaluate our method using two public benchmarks and a real-world commercial LLM agents
891   platform. To assess the effectiveness of our method on LLMs, we utilize the Open-Prompt-Injection
892   benchmark (Liu et al., 2024b). To evaluate its effectiveness on LLM agents, we employ Agent-
893   Dojo (Debenedetti et al., 2024). In both the Open-Prompt-Injection and AgentDojo benchmarks, we
894   include the strongest baselines provided within these benchmarks and other query-based methods,
895   alongside the defense methods presented. Additionally, to test our method’s effectiveness in real-
896
      world LLM agents, we evaluate it on a commercial platform that enables LLMs to use tools and RAG.
      We defer the detailed experimental settings to the corresponding sections.
897
898
899   A.2    AGENT D OJO
900
901   Experiment Setups. The AgentDojo benchmark (Debenedetti et al., 2024) consists of test suites
902   across four distinct environments: Workspace, Slack, Travel, and Banking. The benchmark features a
903   total of 70 tools, 97 realistic user tasks, and 27 injection tasks. The Workspace environment includes
904   24 tools, 40 user tasks, and 6 injection tasks. The Slack environment features 11 tools, 21 user tasks,
905   and 5 injection tasks. The Travel environment includes 28 tools, 20 user tasks, and 7 injection tasks.
906   Lastly, the Banking environment incorporates 11 tools, 16 user tasks, and 9 injection tasks.
907   Metric. We utilize the ASR (denoted as target attack success rate in the original paper) as the metric,
908   which measures the fraction of security cases where the agent executes the malicious actions.
909

      similar computational cost compared with our method.

924   Experiment Setups. The Open-Prompt-Injection benchmark (Liu et al., 2024b) contains seven
925   natural language tasks: duplicate sentence detection, grammar correction, hate content detection,
926   natural language inference, sentiment analysis, spam detection, and text summarization. Specifically,
927   the benchmark use MRPC dataset for duplicate sentence detection (Dolan & Brockett, 2005), Jfleg
928   dataset for grammar correction (Napoles et al., 2017; Heilman et al., 2014), HSOL dataset for hate
929
      content detection (Davidson et al., 2017), RTE dataset for natural language inference (Warstadt et al.,
      2019; Wang et al., 2019), SST2 dataset for sentiment analysis (Socher et al., 2013), SMS Spam
930
      dataset for spam detection (Almeida et al., 2011), and Gigaword dataset for text summarization (Graff
931
      et al., 2003; Rush et al., 2015). The benchmark uses each of the seven tasks as a user (or injected)
932   task. Note that a task could be used as both the user task and the injected task simultaneously. As a
933   result, there are 49 combinations in total (7 user tasks × 7 injected tasks). A user task consists of
934   a user instruction and external data, whereas an injected task contains an injected instruction and
935   injected data. For each dataset of a task, the benchmark selects 100 examples uniformly at random
936   without replacement as the user (or injected) data.
