                                                  A BSTRACT
013
014            In the era of data-driven decision-making, efficiently acquiring and analyzing
015            diverse datasets is critical for accelerating research and innovation. Yet, tra-
016            ditional manual approaches to dataset discovery, preparation, and exploration
017            remain inefficient and cumbersome, especially as the scale and complexity of
018            datasets continue to expand. These challenges create major roadblocks, slow-
019            ing down the pace of progress and reducing the capacity for data-driven break-
020            throughs. To address these challenges, we introduce DataSEA (Search, Eval-
021            uate, Analyze), a fully automated system for comprehensive dataset process-
022
               ing, leveraging large language models (LLMs) to streamline the data handling
               pipeline. DataSEA autonomously searches for dataset sources, retrieves and or-
023
               ganizes evaluation metadata, and generates custom scripts to load and analyze
024
               data based on user input. Users can provide just a dataset name, and DataSEA
025            will handle the entire preparation process. While fully automated, minimal user
026            interaction can further enhance system accuracy and dataset handling specificity.
027            We evaluated DataSEA on datasets from distinct fields, demonstrating its robust-
028            ness and efficiency in reducing the time and effort required for data preparation
029            and exploration. By automating these foundational tasks, DataSEA empowers
030            researchers to allocate more time to in-depth analysis and hypothesis genera-
031            tion, ultimately accelerating the pace of innovation. The code is available at
032            https://github.com/SingleView11/DataSEA.

      1   I NTRODUCTION
035
036   In the age of artificial intelligence, the significance of data is undeniable. The volume of data gen-
037   erated and published online is rapidly increasing, yet searching for structured data on the internet
038   remains challenging [Kacprzak et al. (2018)]. On one hand, quickly and clearly understanding the
039   structure and content of large datasets—spanning social sciences, life sciences, high-energy physics,
040   climate science, and other fields—has become increasingly difficult. On the other hand, as the scale
041   and complexity of data grow, the efficiency of manually searching for and downloading datasets
042   diminishes, turning into a daunting task. Traditional manual methods for discovering, preparing,
043   and exploring data are becoming increasingly cumbersome and inefficient. Consequently, swiftly
044
      acquiring and analyzing diverse datasets has become a crucial factor in driving research and innova-
      tion.
045
046   With the advancement of prompt engineering, large language models (LLMs) have demonstrated
047   impressive performance across various fields [Wei et al. (2022); Kojima et al. (2022); Wang et al.
048   (2022); Zhou et al. (2022); Madaan et al. (2024); Bai et al. (2022); Chen et al. (2023)]. The strong
049   capability of LLMs to process vast amounts of textual information opens new possibilities for the
050
      automated discovery, evaluation, and analysis of datasets. By utilizing LLMs to analyze the web
      information related to specific datasets, it is possible to organize this data into structured formats,
051
      facilitating further computational processing and user interaction.
052
053   Thus, we propose DataSEA (Search, Evaluate, Analyze), a comprehensive automated dataset pro-
      cessing system based on large language models. Additionally, by treating the collected data as a

055   adaptability to specific data processing needs. This allows users to gradually understand the various

057   like environment.

093   of others, and facilitating easier access to information and its sources for data journalists [Brickley
094   et al. (2019)]. Experimental results indicate that DataSEA exhibits strong robustness and efficiency,
095   significantly reducing the time and effort required for data preparation and exploration. Addition-

097   lowering the cost of understanding the content of datasets.

103   side improvements in long text processing and logical reasoning capabilities, DataSEA is likely to
104

      Efficient Dataset Discovery and Exploration Efficient dataset discovery, preparation, and explo-
112
      ration are critical components of the data-driven research pipeline. However, traditional approaches
113
      often require significant manual effort, involving labor-intensive tasks such as searching for relevant
114   datasets, preparing them for analysis, and creating appropriate exploratory tools. Recent advances in
115   automated data management have sought to alleviate these challenges by streamlining data discov-
116   ery, cleaning, transformation, and exploration. Automated data integration tools such as advanced
117   dataset search systems Brickley et al. (2019) and AutoML frameworks Zöller & Huber (2021) have
118   demonstrated considerable promise in minimizing manual workload, though many solutions still
119   face limitations regarding the flexibility and comprehensiveness of the automated workflow. These
120   advancements reflect a growing interest in reducing human intervention and enhancing efficiency
121   through intelligent data management solutions.
122   Automated Dataset Processing Frameworks Several automated dataset processing tools have
123   emerged to address various parts of the data handling workflow. Tools such as Trifacta and Open-
124   Refine Petrova-Antonova & Tancheva (2020) focus on data wrangling, emphasizing interactive user
125   experiences for cleaning and transforming data. Although these tools significantly improve the ef-
126   ficiency of data preprocessing, they require extensive user involvement throughout the process and
127   lack fully automated workflows, particularly in terms of dataset discovery and evaluation.
128   The development of systems like AutoML He et al. (2021) has further paved the way for automation
129   by addressing tasks like feature engineering and model selection. However, while AutoML tools
130   effectively handle model training and hyperparameter tuning, they often depend on structured, pre-
131   prepared datasets. The processes of discovering datasets and assessing their suitability for analysis
132   largely remain manual, limiting the overall automation potential in the data science pipeline Biswas
133   et al. (2022).
134   Leveraging Language Models for Automation LLM models such as GPT-4 Achiam et al. (2023)
135   and LLAMA Touvron et al. (2023), have demonstrated significant capabilities in understanding nat-
136   ural language, generating code, and automating workflows in complex domains. Previous research
137   has leveraged LLMs to generate scripts for data processing Biswas & Talukdar (2024); Nejjar et al.
138   (2023); Patiny & Godin (2023), streamlining the creation of custom data handling scripts. These
139   efforts highlight the potential of LLMs in automating repetitive tasks, but they often focus narrowly
140
      on code generation without addressing the end-to-end dataset processing pipeline.
141   The recent work on LLM-based assistants (e.g., GPT-4, LLAMA) has further demonstrated the
142   applicability of these models for responding to natural language queries related to data analytics
143   Ram et al. (2024), offering on-demand support for exploratory data analysis (EDA) Ma et al. (2023)
144   and visualization Sah et al. (2024). However, these applications are reactive, requiring substantial
145   user intervention in specifying datasets, parameters, and contexts for each step.
146   Comprehensive Dataset Automation DataSEA builds on these advancements, aiming to deliver
147   a fully automated framework for dataset processing that encompasses not only code generation but
148   also dataset discovery and evaluation. Unlike existing semi-automated tools that require significant
149   human interaction, DataSEA autonomously manages dataset discovery, metadata extraction, and
150   script generation, reducing the need for user input to a minimum. Data-centric AI Zha et al. (2023)
151   suggests that focusing on automating data-handling processes can significantly accelerate research
152
      outcomes Mittal et al. (2023), and DataSEA aligns closely with this vision by implementing an
      automated pipeline that integrates search, evaluation, and analysis.
153
154   By leveraging LLMs to automate not only the preparation but also the discovery and evaluation
155   of datasets, DataSEA contrasts with existing solutions that focus predominantly on either prepara-
156   tion or analytics. This comprehensive approach empowers researchers to reduce the time spent on
157   foundational tasks, allowing for more in-depth analysis and exploration of the data.
158   Summary While previous work has made significant strides in automating parts of the data prepa-
159   ration and analysis workflow, DataSEA is among the first to provide a fully integrated solution for
160   dataset discovery, evaluation, and custom analysis using large language models. By autonomously
161   handling these key stages, DataSEA extends beyond the capabilities of current LLM-driven tools
      and represents a significant step toward automating the entire data lifecycle. This approach aligns

      3.1     S YSTEM OVERVIEW
169
      DataSEA is composed of three core modules: Search, Evaluate, and Analyze. The system leverages
170   large language models to intelligently locate dataset sources, extract metadata, and generate custom
171   scripts for loading and visualizing the data. Users can input a dataset name and optional descrip-
172   tion, and DataSEA autonomously handles the remainder of the process. The architecture allows for
173   minimal user interaction, but additional input can improve the system’s accuracy.
174
      To enhance the effectiveness of LLMs, DataSEA employs instruction-prompting [Brown (2020)]
175
      and a multi-chunk strategy [Liu et al. (2024)] to handle long inputs, ensuring that even large datasets
176   can be processed effectively by breaking the data into manageable sections while maintaining con-
177   text across chunks.
178
179   3.2     S EARCH M ODULE
180
181   The Search Module in DataSEA automates the process of discovering dataset websites by leveraging
182   search engines and LLMs to filter and rank relevant results. The user starts by inputting a dataset
183   name and, optionally, additional dataset details to refine the search. Drawing inspiration from tools
184   like Google Dataset Search [Brickley et al. (2019)], the system generates optimized search queries
185
      based on the input and send them to search engines such as Google to retrieve the top-ranking links.
186   Once the top links are retrieved, the system performs web content extraction on each page. The
187   contents are then analyzed by the LLM, which generates evaluation info. Similar to work on using
188   LLMs for content understanding and retrieval tasks [Brown (2020)], the LLM helps filter out irrele-
189   vant or low-quality pages. The links are ranked based on their evaluation info, with the top-ranking
190   results being those most likely to contain useful dataset information. More detail can be found in
191
      Appendix.

236   3.3.2    R EFERENCE PAPER R ETRIEVAL
237
238   The second stage focuses on retrieving and ranking research papers that reference the dataset. The
239   system uses the Google Scholar API to find papers that may have cited the dataset. These papers
240   are ranked by citation count, following the widely accepted practice of using citation metrics as
241
      indicators of a paper’s impact (Bornmann & Daniel, 2008). For each paper, the metadata extracted
      in the first step is used to validate whether the paper indeed references the correct dataset, as there
242
      may be cases of duplicate names or other inaccuracies. The system filters the most impactful papers,
243
      and the user can specify the number of papers to be collected.

      3.3.3    M ETADATA E XTENSION
247
      In the final step, the system extracts additional metadata from the validated reference papers. This
248   may include more detailed descriptions of the dataset’s features, specific application examples, and
249   additional context provided by the authors. The extracted metadata is then combined with the orig-
250   inal information from the dataset’s website, resulting in an enriched and comprehensive dataset
251   profile.

      3.4     A NALYSIS M ODULE
254
255   The Analysis Module in DataSEA is used to download the dataset and analyze it by generating code
256   and test them. The final generated code can load and visualize dataset samples, and user can input a
257   customized requirement and the code generation will be adapted to satisfy the requirement.
258
259   3.4.1    DATASET D OWNLOAD
260
261   In this phase, the module utilizes metadata from the Evaluation Module to generate hypotheses for
262   downloading the dataset from the identified websites. For each website containing dataset informa-
263   tion, the system generates hypotheses regarding possible download methods. It then creates code by
264   combining the hypothesis, the website content, and the dataset metadata, and executes this code to
265   attempt to download the dataset.
266   The generation of hypotheses has proven to be an effective method for handling uncertainty in data
267   retrieval processes, as it allows the system to explore multiple potential download strategies simul-
268   taneously. This approach is inspired by the inductive reasoning capabilities of language models,
269   as demonstrated in Hypothesis Search (Wang et al., 2023), where generating and testing multiple
      hypotheses leads to more robust and successful outcomes.

293
294   3.4.2    DATASET A NALYSIS AND V ISUALIZATION
295
      After successfully downloading the datasets, the system generates custom analysis code by calling
296
      the LLM with prompts that include the dataset metadata and initial portions of the downloaded data.
297
      This analysis code is designed to load the dataset, read the samples of the raw dataset, and visualize
298   key aspects of the dataset’s structure and contents.
299
300   The analysis code generation process incorporates a self-repair mechanism, drawing from ap-
301
      proaches like CodeT5 (Wang et al., 2021). If the generated code fails during execution, the system
      automatically collects the error log and combines it with the original code to form a more detailed
302
      context. This context is sent back to the LLM, which attempts to identify and fix the issues in
303
      the code. The system iterates through this feedback loop until a working version of the code is
304   produced, significantly improving the reliability and robustness of the analysis code. This feature
305   allows the system to autonomously handle failures and continuously improve the generated code
306   without requiring user intervention.
307
      In addition to the automated processes, users have the ability to write customized requirements. The
308
      system will generate and test code based on the user’s input, allowing for tailored analysis that fits
309
      specific research needs. This user interaction complements the fully automated pipeline, providing
310   flexibility for users to guide the analysis toward more specific goals if needed.

418   5   L IMITATIONS
419
420   DataSEA faces several challenges, including its inability to process databases and databanks, which
421   limits its application in biological fields like genomics [Sherry et al. (2001) ] and proteomics [Abola
422   et al. (1984)]. Additionally, its performance depends heavily on LLMs, and the system exhibits a
423   trade-off between speed and accuracy. While the self-repair mechanism can handle common errors,
424   complex dataset structures may still require manual intervention. The automatic dataset download
425   process also struggles with anti-crawling mechanisms and login/email requests, and visual informa-
426
      tion is often lost during HTML content extraction, suggesting the need for methods that integrate
      neural optical understanding [Blecher et al. (2023)].

      This section provides a detailed description of the code structure for the DataSEA system, which
649   is composed of three main parts: Search, Evaluate, and Analyze. Each part contains an integrated
650   pipeline to automate the dataset processing workflow.
651
652   A.1   S EARCH P IPELINE
653
654   The Search Pipeline is responsible for identifying and retrieving relevant datasets based on the user-
655   provided input. It utilizes large language models (LLMs) to search dataset repositories and official
656   websites. The key steps are as follows:
657
            1. Dataset Query: The system accepts a dataset name as input and sends it to the LLM for
658            generating search queries.
659
            2. Web Search: These queries are used to search for datasets across multiple sources, includ-
660
               ing Kaggle, UCI, Zenodo, and official dataset websites.
661
            3. Link Retrieval: The system collects potential dataset links, filtering and ranking them
662
               based on relevance and credibility.
663
664         4. Search Output: The final output is a set of ranked dataset links which are passed to the
665
               next pipeline for evaluation.
666
667
      A.2   E VALUATE P IPELINE
668   The Evaluate Pipeline processes the dataset links retrieved from the search phase to extract useful
669   metadata and identify the most reliable sources. This pipeline consists of three parts:
670
671         1. Metadata Extraction: Using LLMs, the system extracts relevant metadata from the dataset
672            links, such as dataset size, format, domain, and source information. This step utilizes a
673            combination of preset and user-specified property names.
674         2. Reference Paper Retrieval: The system retrieves research papers that reference the dataset
675            by querying academic databases. The papers are ranked by citation count, and their meta-
676            data is validated.
677         3. Metadata Extension: Metadata from the papers is integrated with the original dataset
678            metadata to provide a more comprehensive evaluation. The system uses cross-references
679            to ensure accuracy and consistency.
680   The output of this pipeline includes the final metadata and a list of reference papers, which are
681   passed to the Analyze Pipeline.
682
683   A.3   A NALYZE P IPELINE
684
685   The Analyze Pipeline is responsible for downloading, organizing, and analyzing the dataset. It
686   performs several critical tasks:
687
            1. Dataset Download: The system generates hypotheses for downloading the dataset based
688
               on metadata and content from the dataset website. For each hypothesis, it attempts to
689            download the dataset and stores the results in corresponding folders.
690
            2. Code Generation: After downloading, the system uses LLMs to generate Python code that
691
               can load, read, and visualize the dataset. The code is based on the metadata and sample
692            points extracted from the dataset.
693
            3. Self-Repair: If the generated code fails during execution, the system captures the error log
694
               and re-invokes the LLM to attempt self-repair. This step improves the robustness of the
695            generated code.
696
            4. User-Defined Requirements: Users can specify custom analysis requirements, and the
697
               system generates corresponding code to meet these needs. The code is automatically tested,
698            and any failures are corrected using the self-repair mechanism.
699
700   The final output of the Analyze Pipeline includes a fully downloaded dataset, analysis code, and vi-
701   sualized sample data points. Users can interact with the system to modify or delete failed downloads
      as needed.
