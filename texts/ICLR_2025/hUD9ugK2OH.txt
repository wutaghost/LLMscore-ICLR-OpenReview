009                                               A BSTRACT
010
011
               Long-context LLMs are increasingly in demand for applications such as retrieval-
012            augmented generation. To defray the cost of pretraining LLMs over long contexts,
013            recent work takes an approach of synthetic context extension: fine-tuning LLMs
014            with synthetically generated long-context data in a post-training stage. However,
015            it remains unclear how and why this synthetic context extension imparts abili-
016            ties for downstream long-context tasks. In this paper, we investigate fine-tuning
017            on synthetic data for three long-context tasks that require retrieval and reason-
018            ing. We vary the realism of “needle” concepts to be retrieved and diversity of the
019            surrounding “haystack” context, from using LLMs to construct synthetic docu-
020
               ments to using templated relations and creating symbolic datasets. We find that
               models trained on synthetic data fall short of the real data, but surprisingly, the
021
               mismatch can be interpreted and even predicted in terms of a special set of at-
022
               tention heads that are responsible for retrieval over long context, retrieval heads
023            (Wu et al., 2024). The retrieval heads learned on synthetic data have high overlap
024            with retrieval heads learned on real data, and there is a strong correlation between
025            the recall of heads learned and the downstream performance of a model. Further-
026            more, with attention knockout and activation patching, we mechanistically show
027            that retrieval heads are necessary and explain model performance, although they
028            are not totally sufficient. Our results shed light on how to interpret synthetic data
029            fine-tuning performance and how to approach creating better data for learning
030            real-world capabilities over long contexts.

033   1   I NTRODUCTION
034
035
      The quadratic memory requirement of Transformer attention imposes a strong computational con-
036   straint on our ability to train and do inference on long-context models. This disrupts the typical
037   pre-training pipeline: pre-training must be done at as large a scale as possible, but pre-training a
038   long context model would necessarily reduce the number of observed tokens able to fit on the GPU.
039   One solution for this is to rely on synthetic data, now common in post-training settings such as SFT
040   (Xu et al., 2023b; Yue et al., 2024; Xu et al., 2024; Che, 2024) and RLHF/DPO (Yang et al., 2023).
041   Recent prior work has proposed using synthetic data to extend the long-context abilities of LLMs
042   after pre-training (Xiong et al., 2024; Zhao et al., 2024).
043   This use of synthetic data is particularly necessary for long context tasks since they are so laborious
044   for humans to manually label. Synthetic data is also configurable: it can exhibit different reasoning
045   skills and “teach” models have to make certain types of inferences (Du et al., 2017; Yu et al., 2018;
046   Agarwal et al., 2021; Tang et al., 2024; Divekar & Durrett, 2024). One way to do this is using
047   templates to express pieces of information that must be reasoned over and to create symbolic tasks
048   that are thought to mirror the reasoning required in the real task (Hsieh et al., 2024; Prakash et al.,
049   2024; Saparov & He, 2023; Li et al., 2024). However, past work has shown varying results from
050
      training on data for this kind of context scaling (Fu et al., 2024); we lack general understanding of
      what is needed here.
051
052   In this paper, we explore several methods of creating synthetic long context data across three tasks.
053   Our goal is to examine what makes synthetic data effective for this kind of context scaling. While
      more realistic data is often better, it is unreliable–certain types of more synthetic data can exhibit

097   2.1       BACKGROUND : S YNTHETIC DATA FOR T RAINING L ANGUAGE M ODELS
098
099   Formally, consider a supervised learning setting for a pre-trained transformer language model M.
100
      Given a task T , we assume a distribution pT of real-world task instances. We assume that a small,
      limited set of input-label pairs DT = (xT , yT ) drawn from the distribution pT is available as seed
101
102
      data. A synthetic dataset D̃T is a set of input-label pairs sampled from the outputs of a data generator
103   G given the seed data or the known properties: D̃T ∼ p((x̃, ỹ) | DT ). Benchmarking or training
104
      M on a synthetic dataset that can be used to represent properties of the real dataset is expected to
      evaluate or teach M the capabilities that can be transferred to the real-world distribution pT .
105
106   A recent line of work has shown that training short-context LLMs on simple heuristic-based syn-
107   thetic datasets can achieve surprisingly transferability on context extension, a post-training scenario
      where LLMs that have been pre-trained on short-context corpora are further trained on long-context


                                                                                            2
      Under review as a conference paper at ICLR 2025


108
      tasks to extend the effective context window (Fu et al., 2024; Zhao et al., 2024; Xiong et al., 2024).
109   For example, Xiong et al. (2024) finds that fine-tuning on a synthetic simple dictionary key-value
110   retrieval task can even outperform models fine-tuned on realistic in-domain data.
111
112
      We call these approaches synthetic context extension: using synthetic data to extend the context
      window of LLMs. It remains unclear how and why synthetic data, especially when drawn from a
113
      very different distribution from the real data, can be effective despite results that support the contrary
114
      (Chen et al., 2024; Liu et al., 2024b). There is also a lack of general principles for creating synthetic
115   data for training beyond dataset-specific constructions in the literature. We start by constructing
116   synthetic datasets varying in systematic ways to unify these variants from the literature.

      2.2    E XPERIMENTAL SETUP
119
120   Following Xiong et al. (2024), we focus on fine-tuning LLMs for long-context retrieval and reason-
121   ing tasks where training on high-quality synthetic data has been shown to outperform real data. We
122   also extend to multi-hop settings. We experiment on three datasets where, given a long context C
123   and a context-based query q, a language model M needs to retrieve one or more “needles concepts”
124   f1 , . . . , fm from C (pieces of relevant information), reason over that information, and then generate
125   a response ỹ ∼ p(y | C, q) where p(y | C, q) is the conditional distribution that M places over the
126   vocabulary Σ∗ given the context and the query. We consider extending the context window from
127
      8K to 32K tokens to be representative of synthetic context extension following Chen et al. (2023).
      Specifically, we use the following three datasets.
128
129   MDQA (Liu et al., 2024a): MDQA is a multi-document question answering (QA) dataset where
130   only one paragraph in C contains the gold answer to a single-hop query; that is, there is a single f
131   which directly addresses q. We extend the original MDQA dataset in 4K context to 32K context by
132   retrieving additional distractor paragraphs from Natural Questions-Open (Kwiatkowski et al., 2019;
133
      Lee et al., 2019) with Contriever (Izacard et al., 2021).
134   MuSiQue (Trivedi et al., 2022): MuSiQue is a multi-hop QA dataset where the model must identify
135   a piece of relevant information from a different document for each hop of the question in order to
136   retrieve the final correct answer from the context. We use the linear three-hop subset of MuSiQue
137   and extend the dataset to 32K by adding padding paragraphs 1 to the original context. In this setting,
138   the facts f1 , f2 , f3 are natural language sentences containing knowledge graph relations.2
139   SummHay Citation (Laban et al., 2024): Summary of a Haystack (SummHay) is a long-context
140   retrieval dataset where the model is given a set of documents with controlled “insights,” and asked
141   to produce a list of key points. Additionally, the model must cite the correct documents in support
142   of each key point. We isolate the citation component and construct a task where, given a haystack
143   of 10 documents and a key point (“insight”), the model must correctly identify the two documents
144   that support the point and their associated document IDs. The two facts f1 , f2 may span multiple
145
      sentences and may be substantially paraphrased versions of the insight.
146
147   Training Configuration For each task, we fine-tune two short-context LLMs, Llama-3-8B-
148
      Instruct (Dubey et al., 2024) and Mistral-7B-Instruct-v0.1 (Jiang et al., 2023). Prior work indicates
      that attention heads are largely responsible for implementing algorithms (Olsson et al., 2022) and
149
      using information within the context (Stolfo et al., 2023; Lieberum et al., 2023) while MLP layers
150
      are responsible for parametric knowledge (Geva et al., 2021). In addition, when adapting to long
151   contexts, the attention heads in particular must handle new position embeddings and softmax over
152   more context tokens. Therefore, we fine-tune attention heads only. 3
153
154
      To extend models from their original 8K pretrained context length to 32K, we follow Gradient (2024)
      in calculating new RoPE (Su et al., 2024) theta values, using 6315088 for Llama-3-8B-Instruct
155
      and 59300 for Mistral-7B-Instruct-v0.1. We scale the sliding window accordingly for Mistral-7B-
156
      Instruct-v0.1 to 16k context. These are the only adjustments we make to the models, following Fu
157   et al. (2024). Our hyperparameters and hardware setup can be found in Appendix C.1.

183   3.1      P RINCIPLES U NDER C ONSIDERATION
184
      To create a representative range of synthetic data for each task, we partition the input text C into
185
      (A) text containing relevant information {f1 , . . . , fm } (“needle concepts”) and (B) the surrounding
186         ffi C\{f , . . . , f }. This allows us fi
      context        1          m                   to categorize any task as having a variant of concept
187   expression (how the target information fi is expressed) and context diversity (the naturalness and
188   relevance of the surrounding information). In the following paragraphs, we discuss common variants
189   found in synthetic data literature. We single out and emphasize a highly structured variant of concept
190   and context–symbolic tasks–for being devoid of natural language yet noted to transfer to realistic
191   tasks (Xiong et al., 2024).
192
193   Concept Expression A common procedure for creating synthetic data involves exploiting task
194   asymmetry (Josifoski et al., 2023; Xu et al., 2023a; Lu et al., 2024; Chandradevan et al., 2024;
195   Chaudhary et al., 2024; Tang et al., 2024), where asking an LLM to generate natural language data
196   based off of a label (e.g. a sentence based off of a knowledge triple) is easier than predicting the
197   answer from text of the same complexity and domain. In this scenario, the LLM is asked to create
198   diverse “needle” target concept expressions fi . In task specific cases, it is beneficial to make this
199   data less realistic while encouraging generalization. For example, prior synthetic datasets have made
200   use of fictional entities (Saparov & He, 2023) or nonsense phrases (Wei et al., 2023) in place of real
201
      entities and properties, or swapped out nouns to augment the dataset (Lu et al., 2024) and prevent
      overfitting to specific entities. In long context benchmarks (Hsieh et al., 2024; Li et al., 2024), it is
202
      common to express the needle concepts in short, templated sentences.
203
204
      Context Diversity We can also vary the expression of C\{f1 , . . . , fm }, the “haystack.” This
205
      ranges from distractor needles which may have the same form (template) as the target concept to
206
      padding with repeated sentences. We use the repeated set of sentences “The grass is green. The sky
207   is blue. The sun is yellow. Here we go. There and back again.” as our low-diversity padding to
208   compare with context that is synthetically generated by an LLM, following Hsieh et al. (2024) and
209   Mohtashami & Jaggi (2023).
210
211   Symbolic Tasks We also experiment with purely symbolic (involving dictionary key-value or list
212   retrieval) versions of our real tasks, since such tasks are believed to recruit similar model abilities as
213   their natural language counterparts. For example, prior work has indicated that pre-training on code
214   helps on Entity Tracking (Prakash et al., 2024) and that fine-tuning on a symbolic dictionary key-
215   value retrieval task can provide greater benefits than even real data (Xiong et al., 2024). Additionally,
      RULER (Hsieh et al., 2024) introduced a variable assignment task for long-context value tracking.

217   variables that are assigned the value 12345.” that do not contain meaningful natural language,

226   MDQA Given a training example of MDQA training data (C, q, y) ∈ DT , we combine the query
227   q and answer y into our needle f that will be put into the context and that needs to be retrieved by the

229   in the query and answer (high expression), and (2) replacing the real entities with 4-character sym-

      bolic entities (low expression). We create the context surrounding the needle claim with two levels

236   ate two simplification levels of concept expression by (1) keeping the real entities in the query and

238   expression), and constructing fi by prompting GPT-4o-mini to write sentences or via template. We
239   create two levels of context diversity by (1) prompting GPT-4 to write a paragraph containing the

      SummHay Citation We derive the fi from the insights in one of two ways. (1) We prompt GPT-

      4o-mini to rephrase the insights to create the query, and then prompt again to split rephrased insights
247

250   context diversity by (1) padding each document with distractor insights from the same topic, (high

      4      R ETRIEVAL H EADS ARE N ECESSARY FOR C ONTEXT E XTENSION
291
292
      One of the key features of our tasks is the need for retrieving needles fi embedded in a long context.
293   Work from the mechanistic interpretability literature has shown that some attention heads in pre-
294   trained (Olsson et al., 2022; Lieberum et al., 2023) or fine-tuned (Panigrahi et al., 2023; Yin et al.,
295   2024) transformers specialize in retrieving and synthesizing information from the context in princi-
296   pled ways. 5 Notably, recent work (Wu et al., 2024) indicates that there exists a special, intrinsic set
297   of attention heads in pre-trained transformers that attend to relevant information fi in long context
298   C given a query q and copy it to the output ỹ. Wu et al. (2024) dub them as retrieval heads.
299   Given the nature of our task, we analyze these attention heads as a proxy for the subnetworks being
300   recruited and learned during fine-tuning with synthetic data. Our core hypothesis is that we can
301   attribute the performance of synthetic context extension to how well the models learn to adapt the
302   attention heads relevant to retrieving and using information from long context, as indicated by the
303   retrieval scores of attention heads. Building on prior work, we extend identification of retrieval
304   heads to multi-hop settings in MuSiQue and SummHay Citation.
305
306   4.1      D ETECTING R ETRIEVAL H EADS
307
308
      Following Wu et al. (2024), we detect retrieval heads by computing retrieval scores. To compare
      across fine-tuned models, we consider any attention head with a positive retrieval score to be a
309
      retrieval head, and later compute cosine similarity to account for the strength of scores. Given a
310                         ′
      fine-tuned model M , we evaluate it on a dataset D∗ = {(C ∗ , q ∗ , y ∗ )} where the answer y ∗ needs
311                                                                                                 ′

312
      to be identified from some needles f ∗ in C ∗ and copied to the model output ỹ ∗ . When M generates
      an output token w ∈ ỹ ∗ , we examine whether or not an attention head places the most attention
313
      probability mass on the same token in the answer span y ∗ in the context. If so, we consider the token
314
      w to be retrieved by the attention head. Given an evaluation example (C ∗ , q ∗ , y ∗ ), let Gh = {wh }
315   be the set of all tokens w that are retrieved by a head h during decoding. We define the retrieval
316   score Sh for head h on a single example as:
317

349   4.2    C ONNECTION WITH DOWNSTREAM PERFORMANCE
350
351   The presence of retrieval heads does not necessarily offer a concrete connection to downstream per-
352   formance; we do not know that models are attending to long context using these heads, or whether
353   these heads are correlated with other model capabilities. We conduct two experiments to elaborate
354
      on this: an intervention experiment where we mask out retrieval heads to see the impact on per-
      formance, and an observational experiment where we correlate the presence of retrieval heads with
355
      downstream performance across our different synthetic data variants.
356
357
      Activation Masking We show that these attention heads are responsible for model performance
358
      on the real tasks by comparing activation masking on the top-k retrieval heads versus k random
359   heads for various k as in Wu et al. (2024). Specifically, we select the top-k retrieval heads based on
360   retrieval score, and zero out the outputs of those attention modules. As shown in Figure 3, masking
361   even the top-10 retrieval heads causes a sharp drop in performance whereas masking 10 random
362   heads over 3 repeated trials results in a marginal (< 0.05) or no drop in performance, with one
363   exception (Llama-3-8B-Instruct on MDQA).
364
365   Synthetic Data Performance and Retrieval Heads As noted previously, when the synthetic data
366   induces fewer retrieval heads, they tend to be subsets of those active on the real data. Following this
367   for each synthetic dataset, we calculate the recall of non-zero scoring attention heads against the
368   real dataset (first column of Tables 5-10 in Appendix D). As shown in Table 4, we find that this is
369   strongly correlated with F1 on the real task for MuSiQue and SummHay Citation. This holds more
370   strongly for Llama-3-8B-Instruct than for Mistral-7B-Instruct-v0.1.

468   5       R ETRIEVAL H EAD PATCHING
469
470   Given datasets of the same conceptual reasoning and retrieval task, it is peculiar that fine-tuning on
471   some datasets results in fewer retrieval heads. Do the attention heads common to all datasets better
472   capture the core capability required for the task? For the common attention heads, do models learn
473
      a better way of updating them from the real data than the synthetic data? To investigate these, we
      follow Prakash et al. (2024) to perform cross-model activation patching of retrieval heads in the
474
      intersection and complement between the real dataset and the synthetic datasets. Specifically, given
475
      the set of retrieval scoring attention heads on the real data, Hreal , and the set of retrieval scoring heads
476   on a synthetic dataset, Hsynth , we take the complement Hcompl = Hreal \ Hsynth and the intersection
477   Hinter = Hreal ∩ Hsynth . For a fair comparison, we sample nheads = min(|Hcompl |, |Hinter |) without
478   replacement from both sets. Additionally we compare with nheads randomly sampled attention heads.
479   For each set, we patch activations from the model trained on the real data to the model trained on
480   the synthetic data. Implementation details can be found in Appendix F.
481
482   Synthetic Data Affects Required Model Components Less Effectively Our results in Tables 3
483   and 11 show that patching intersection heads outperforms patching both random and complement
484
          7
485      SummHay Retrieval Heads attend to the final answer (document number), whereas SummHay Insight
      Heads attend to the insight text within the document.


                                                              9
      Under review as a conference paper at ICLR 2025


486
      heads. The improvement is the greatest for synthetic tasks with the lowest performance on the cor-
487
      responding real task, and negligible or negative for the best synthetic tasks. The efficacy of patching
488   Hinter indicates that while a synthetic dataset may target the necessary retrieval heads for the real
489   task, they are insufficient in learning how to best utilize the required model components. One expla-
490   nation is that fine-tuning induces upstream changes so that a different representation distribution is
491   passed to the retrieval heads when learning on synthetic data. This allows retrieval heads to learn to
492   be effective for the synthetic task while failing on out-of-distribution real data representations.
493
494
495   Intersection Heads are Core Attention Heads So what do the “extra” retrieval heads in the
496   complement do? Wu et al. (2024) finds that Llama-2-7B contains 12 core retrieval heads while the
497
      rest are dynamically activated. We confirm this by finding that the average retrieval scores of the
      intersection heads are much greater than those of the complement heads (see Table 12).
498
499
500   Implications We established in Section 4 that retrieval heads are necessary for synthetic context
501   extension. The fact that “better” heads in the intersection can be patched in to improve performance
502   indicates that learning these heads alone is not sufficient. We see our work as contributing a useful
503   analytical tool for understanding the behavior of synthetic context extension. At the same time, this
504
      presents a challenge for future work to tackle: can we come up with a more complete mechanistic
      explanation of synthetic context extension that accounts for these observations as well?

      6    R ELATED W ORK
508
509
510   Prior work has shown that benchmarking or training LLMs on synthetic data can reveal or obtain
511
      capabilities that can be transferred and generalized to real tasks, especially in settings where human-
      annotated data is hard to obtain such as long-context tasks. For this purpose, synthetic data are
512
      commonly used and believed to represent a simple reduction of the kinds of abilities employed in
513
      linguistically complex settings. The Needle-In-A-Haystack (NIAH) introduced by Kamradt (2023)
514   involves placing a needle statement at a random position within a haystack consisting of unrelated
515   essay text. Subsequent work (Hsieh et al., 2024; Li et al., 2024) has expanded this task to multi-value
516   retrieval and used simple templated needle sentences to include distractor needles in the context.
517   Hsieh et al. (2024) additionally parameterized its test suite by the diversity of the input context (essay
518   text, repeated text, or distractor needles) and the target value type (words, numbers, or UUIDs).
519
      Leveraging the potential generalizability of synthetic data, a line of work in interpretability literature
520   generates synthetic data to perform controlled experiments to probe the inner workings of LLMs.
521   For example, Kim & Schuster (2023) shows that a synthetic version of entity tracking can be used
522   to mechanistically understand how fine-tuning enhances existing capabilities of pre-trained LLMs
523   via mechanistic intervention techniques, and Kim et al. (2024) shows that the transformer circuit
524   responsible for syllogistic reasoning in LLMs can be identified by evaluating on synthetic logical
525   statements. However, there is a lack of understanding of when and how the mechanism discovered
526   from synthetic tasks generalizes to real-world capabilities.
527   Our work bridges these directions by providing mechanistic explanations for the transferability of
528   synthetic context extension while motivating the pursuit of better usage of synthetic data to evaluate,
529   enhance, and understand the capabilities of LLMs.
530
531
532   7    C ONCLUSION
533
534   In this paper, we investigated the relationship between the nature of synthetic data for synthetic
535   context extension and performance on downstream tasks. Different synthetic datasets give widely
536   varying performance, partially because of the different numbers of retrieval heads they induce in a
537   model. We showed that these heads are causally connected to the performance, and that these heads
538   are necessary (but not sufficient) for a strong downstream model. We believe this work paves the
539   way for further mechanistic understanding of long context behavior and the ways in which synthetic
      data induces new capabilities in language models.

1218   We implemented retrieval head patching with Baukit.8 Given an example from the test set and a set
1219   of attention heads to patch, we run a forward pass with the model fine-tuned on the real data and ex-
1220   tract the attention output from the selected attention heads before being projected and concatenated
1221   back to the residual stream. Then, we use the same example and run a forward pass with the model
1222   fine-tuned on a synthetic dataset. We replace the attention outputs of the aforementioned selected
1223   attention heads with the attention outputs extracted from the model fine-tuned on real data. Using
1224   the procedure described above, we patch the attention outputs of the selected attention heads into
1225   the model fine-tuned on a synthetic dataset for all input tokens. We then use the patched inputs to
1226   generate and decode output tokens without patching any activations for the output tokens.
