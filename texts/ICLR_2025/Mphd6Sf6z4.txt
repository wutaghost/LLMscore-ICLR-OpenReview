                                                  A BSTRACT
008
009
               This review aims to evaluate and compare various static analysis tools across
010
               multiple programming languages for memory management. The tools and tech-
011            niques under scrutiny include pattern matching, symbolic execution, CppCheck,
012            SharpChecker, FindBugs, CheckStyle, and Pylint. When examining the methods,
013            pattern-matching, and symbolic execution, we identified implementations using
014            pattern-matching and symbolic execution for each programming language. We fo-
015            cus on understanding the full scope of their capabilities and effectiveness in man-
016            aging internal and external memory components such as RAM, SRAM, PROM,
017            Cache, Optical Drive, etc. While static analysis tools do not directly analyze phys-
018            ical memory components, they are crucial in enhancing memory behavior. By de-
019
               tecting and addressing memory-related issues early in the development process,
               these tools contribute significantly to the overall quality of software systems. This
020
               review will thoroughly examine the strengths and weaknesses of each static anal-
021
               ysis tool, aiding in selecting the most suitable tool or combination of tools for
022            effective memory management across diverse programming environments.

025   1     I NTRODUCTION
026
027   Software systems have become more complex as they must ensure their software’s reliability,
028   security, and management. Not upkeeping software systems may result in software defects. Those
029   include (Emanuelsson & Nilsson, 2008) logical or functional errors, where the program sometimes
030   computes incorrect values; runtime errors, where the program typically crashes; resource leaks,
031
      where the performance of the program degrades until the program freezes or crashes; and mi-
      nuscule security vulnerabilities that malicious attackers can exploit to obtain control over computers.
032
033
034   Various methods are employed to identify software errors and defects, using tools compiled explic-
035   itly for certain programming languages or implementing a feature to assist. Tools specifically com-
036   piled for certain languages include CppCheck for C/C++ programs, FindBugs for Java programs,
037
      and many more. Features implemented to assist include pattern matching and specific function-
      alities within a given tool. Software developers use those tools and features to mitigate the risks
038
      mentioned earlier.
039
040
      1.1   D EFINITION AND O BJECTIVES
041
042   Static analysis Rival & Yi (2020) is a technique aimed at discovering expressive properties of
043   program code without executing it. It shouldn’t be confused with dynamic analysis, (Ernst, 2003)
044   which operates by running a program and observing its outputs or executions. Static analysis
045   tools are (Lenarduzzi & Fabio, 2023) instruments that examine a program’s source code without
046   executing it to discover potential quality issues such as resource management, syntax errors, and
047   runtime behavior. These tools benefit developers, as they help build high-quality software, reduce
048   the risk of security breaches, and minimize debugging time. However, some techniques lack
049   scalability, and analyzing large software programs may require more advanced computational tools
050
      to ensure accurate results, which current tools may not consistently deliver.
051
052   Our objective is to gather research on static memory analysis and identify the most efficient tools
053   for memory management. This work comprehensively reviews static memory analysis techniques
      and tools to detect source code defects across programming languages, as presented in Table 1.

      2.1   W HAT M AKES A G OOD S TATIC A NALYSIS T OOL ?
100
101
      As previously mentioned, the primary purpose of a static analysis tool is to discover runtime errors,
102   resource leaks, and security vulnerabilities without executing the code. However, finding a tool that
103   identifies all defects without limitations is challenging. One common issue is the prevalence of false
104   positives (i.e., reporting non-issues) and false negatives (i.e., missing actual problems). Managing
105   and detecting these is crucial, but many tools do not explicitly label them as functionalities. Instead,
106   they address these through configurable settings, algorithm improvements from user feedback, and
107   data-flow or path-sensitive analysis integration that allows a user to determine the feasibility of a
      path and optimize resource usage.

136   2.2     S TATIC A NALYSIS T OOLS

138   2.2.1    PATTERN M ATCHING
139
140   Pattern matching is a technique that (Ferrara, 2023) for checking if an expression conforms to a
141
      particular pattern by examining a series of case expressions. Static analysis tools (Wen, 2024) utilize
      pattern matching alongside techniques such as symbolic execution and separation logic to identify
142
      issues such as poor use of language constructs and violation of coding guidelines. Identifying
143
      those issues helps (par, 2024) prevent defects such as resource leaks, performance and security
144   issues, and API misuse. Tools that implement pattern matching, like CppCheck, have been observed
145   by developers in its earlier versions (Gulabovska & Porkolab, 2019) to report relatively low false
146   positives and identified that well-written regular expressions are more effortless to predict. An
147   implementation that we reviewed to use pattern-matching was Mach7.
148
149   2.2.2    C PP C HECK
150
151   CppCheck (Moerman, 2018) is an open-source tool similar to C++test, which is capable of analyzing
152
      C and C++ code with simple control flow analysis. While CppCheck cannot filter certain defects
      through command-line flags, it offers useful features like sorting defect reports by severity and
153
      analyzing entire code bases using a compilation database that can manually remove the need to
154
      provide paths to all source files.
155
156
      2.2.3    S YMBOLIC E XECUTION
157
158   Symbolic execution (Păsăreanu & Visser, 2009) is a notable static analysis technique that uses sym-
159   bolic values to represent program inputs rather than initialized data, allowing for the manipulation of
160   program expressions symbolically. It is considered highly powerful as it (Gulabovska & Porkolab,
161   2019) leverages program structure, type systems, and data-flow information while following
      function calls. Symbolic execution can handle complex constructs such as recursive data structures,


                                                        3
      Under review as a conference paper at ICLR 2025


162
      arrays, preconditions, and multithreading. However, when applied to programs with loops or
163   recursion, it can result in an unbounded number of execution paths, leading to resource-intensive
164   processes. Those results occur because loops and recursion can generate infinite possible states. As
165   a result, symbolic execution may not terminate due to the explosion of configurations. 1
166
167
      Additionally, the analysis state is tracked using an exploded graph, which can grow exponentially in
168
      size due to program control branches (i.e., conditions). To mitigate the resource cost, constraints or
169   heuristics are introduced to limit exploration depth or prioritize paths, ensuring efficient execution.
170
171
172
      Tools that we observed to implement symbolic execution were the Clang Static Analyzer (CSA),
      the Language-Independent Tool for Symbolic Execution in C#, Symbolic JPF, and the Python
173
      CHEF Engine using Symbolic Execution.
174
175
176   2.2.4     S HARP C HECKER
177
178   SharpChecker is a static analysis tool (Koshelev & Belevanstev, 2017) that can detect 30 differ-
179   ent types of bugs through parsing, data-flow analysis, and (the most significant) context and path-
180
      sensitive intraprocedural analysis. SharpChecker employs the Roslyn infrastructure to work with C#
      build system files and compile a program’s source code. SharpChecker only uses the part of Roslyn
181
      that is responsible for parsing a project and solution files to determine a set of files for building a
182
      given program or library and to set an adequate environment.
183
184
185   2.2.5     F IND B UGS
186
187   FindBugs (Lenarduzzi & Fabio, 2023) is a static analysis tool mainly used to evaluate Java byte
188
      code. However, the tool’s capability goes beyond that, as it can also highlight the exact position
      of an issue if the tool is provided the source code. The analysis is performed by detecting bug
189
      patterns that, according to FindBugs (Shen, 2011), can arise due to complex language features,
190
      misunderstood API features, misunderstood invariants when code undergoes modification during
191   maintenance, and garden variety mistakes. Then, these bug patterns are categorized into nine cate-
192   gories: bad practice, correctness, experimental, internalization, malicious code vulnerability,
193   multi-threaded, performance, security, and dodgy code. Per their observations, Shen et al.
194   categorize three main ranking options for FindBugs’ error reports as sorting reports in alphabetical
195   order (e.g., bug patterns, packages), sorting reports in severity (i.e., assigning each error report a
196   priority value of low, medium, or high), and sorting reports through user designation (i.e., allows
197   users to manually designate each bug report according to categories (e.g., needing further study, not
198   a bug, etc.).
199
200   Despite the number of ranking methods in FindBugs, users still face the risks of false positives and
201   high inspection costs when dealing with large-scale software systems. Thus, FindBugs’ developers
202   (Shen, 2011) have two main goals for FindBugs:
203
204           1. The developers want to achieve a low false positive rate, particularly in correctness cate-
205              gories, which are primarily used to identify genuine errors in software systems.
206
207           2. Error reports of the same bug pattern(s) or bug kind(s) should ideally have the same desig-
208              nation since it would facilitate the management of error reports in groups.
209
210   FindBugs has been succeeded by SpotBugs. SpotBugs continued the legacy of FindBugs by pro-
211   viding advanced bug detection capabilities, with improvement and support for newer versions of
212   Java.

339   4.1.1    C OMPARISONS : M ACH 7, C PP C HECK , AND THE C LANG S TATIC A NALYZER
340
341   CppCheck and CSA allow custom checks, extendable frameworks, or modules to enhance code
342   analysis and can detect issues such as dead code and uninitialized variables. Mach7 and CSA
343   are made to implement standard static analysis techniques (e.g., pattern matching and symbolic
344   execution).
345
346   Mach7, CppCheck, and CSA enhance code quality through different analysis methods: Mach7
347   relies on pattern matching, CppCheck uses data-flow analysis, and CSA uses symbolic execution.
348   CppCheck amplifies error detection by categorizing errors and integrating a flag to alert for more
349   checks in checks that would increase false positives. However, CppCheck doesn’t provide as many
350   checkers as CSA and executes them quickly.
351
352   We discussed two different observations/implementations of CSA: Kovacs et al. introduced a new
353   module to enhance an established checker in CSA, and Balogh and Szalay evaluated the established
354   checkers on different test cases. Balogh and Szalay observed that CSA has difficulty detecting false
355   positives, especially in low-level contexts. At the same time, Kovacs et al. attempted to reduce false
356   positives by enhancing their specialized checks. In evaluating CSA, Balogh and Szalay conclude
357   that CSA offers comprehensive reporting, but long bug paths may hinder user comprehension and
358
      require more resources to analyze the code. On the other hand, Kovacs et al.’s approach aims to
      simplify bug tracing by providing more context on specific memory issues by implementing new
359
      modules.

      5     C ONCLUSION
412
413
      The tools we observed, shown in Table 1, revealed that five of the ten static analysis tools (e.g., Cp-
414   pCheck, FindBugs, CheckStyle, Pylint, SharpChecker) we reviewed were independent tools. At the
415   same time, the other five (e.g., Mach7, Clang Static Analyzer, Symbolic JPF, Language-Independent
416   Tool, Python CHEF Engine) were implemented as solutions using features of the techniques: pattern
417   matching and symbolic execution to make up for functionalities that the current tools did not possess.
418
419
      However, there is a significant difference between the two identified techniques– pattern matching
420   and symbolic execution. Pattern matching is commonly used in static analysis tools that identify
421   specific bug patterns (e.g., CppCheck, FindBugs, SharpChecker) and uphold common coding
422   standards. Those are valuable applications of pattern matching, but those applications are also
423   limited. Pattern matching can detect common bug patterns only if predefined, making it easier
424   to dismiss undefined ones. On the other hand, symbolic execution, while also utilized by static
425   analysis tools, is more used as an implementation or branch of a tool rather than a function.
426
427   The other static analysis tools, while effective in identifying their designated issues, lack some
428   crucial functions for memory management and error detection, making them unsuitable for a fully
429   efficient static analysis tool. For instance, CheckStyle, despite preventing some common errors
430   by enforcing coding standards that maintain readability and consistency, does not significantly
431   contribute to memory management. FindBugs’ detection of memory issues is limited to recogniz-
      able patterns, making it ineffective in detecting unknown bug patterns and upholding code quality.


                                                          8
      Under review as a conference paper at ICLR 2025


432
      CppCheck, while effective in detecting a wide range of memory-related issues (e.g., memory leaks,
433   array-bound errors), is lacking due to its generation of false positives, a common problem for static
434   analysis tools. Pylint, although helpful in maintaining code quality, like CheckStyle, cannot perform
435   deep memory analysis and, therefore, cannot detect memory management issues like memory leaks.
436
437
      Overall, our objective was to gather research on static memory analysis and identify the most effi-
438
      cient tools to utilize in our automated process to perform a memory analysis on a virtual machine.
439   From our findings and observations, symbolic execution is the most effective technique because it
440   is helpful in memory management and can be implemented with various static analysis tools. How-
441   ever, in terms of developing our automated process, it seems that it would be best to utilize symbolic
442   execution with another static analysis so that a broader range of potential memory issues could be
443   addressed.

445   5.1   L ESSONS L EARNED
446
447   Quite a few lessons have been learned in researching and examining the static analysis tools:
448
             • We now understand the limits of static analysis more in-depth. Static analysis tools have
449
               proven worthwhile but are limited, as they can discover potential issues in source code
450            pre-runtime but not all problems, especially those involving dynamic behavior.
451
             • A few of the tools we observed, like CppCheck and CheckStyle, while differing in language
452
               support, are similar in functionality, like upholding coding standards. Style issues detected
453            in CppCheck for C/C++ source code could also be detected in CheckStyle but for Java
454            source code.
455
             • The capabilities of each static analysis tool vary, as one may be more helpful in enforcing
456
               coding standards, while another may be more useful in detecting memory leaks. There is a
457            more effective tool, but there is no right tool as some projects may require a combination
458            of tools and others with just one specific feature.

460   5.2   F UTURE W ORK
461
462   Our future work will extend the concept of static memory analysis to a VM (virtual machine). We
463   will develop an automated process to perform a complete memory analysis on a virtual machine. We
464   will examine various aspects of its memory, including Internal (e.g., RAM, DRAM, SRAM, ROM,
465   PROM, EPROM, Cache) and External (e.g., Optical Drive, Solid State Drives, virtual memory).
466
      We will also identify any memory-related issues the tools detect (e.g., uninitialized variables and
      memory leaks). We will compare and contrast the results and develop multiple data visualizations
467
      to improve understanding.
