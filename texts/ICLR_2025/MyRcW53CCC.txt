011                                                A BSTRACT
012
013            The widespread usage of online Large Language Models (LLMs) inference services
014            has raised significant privacy concerns about the potential exposure of private
015            information in user inputs to malicious eavesdroppers. Existing privacy protection
016
               methods for LLMs suffer from either insufficient privacy protection, performance
               degradation, or large inference time overhead. To address these limitations, we
017
               propose PrivacyRestore, a plug-and-play method to protect the privacy of user
018
               inputs during LLM inference. The server first trains restoration vectors for each
019            privacy span and then release to clients. Privacy span is defined as a contiguous
020            sequence of tokens within a text that contain private information. The client then
021            aggregate restoration vectors of all privacy spans in the input into a single meta
022            restoration vector which is later sent to the server side along with the input without
023            privacy spans. The private information is restored via activation steering during
024            inference. Furthermore, we prove that PrivacyRestore inherently prevents the
025            linear growth of the privacy budget. We create three datasets, covering medical
026            and legal domains, to evaluate the effectiveness of privacy preserving methods.
027
               The experimental results show that PrivacyRestore effectively protects private
               information and maintain acceptable levels of performance and inference overhead.

      1   I NTRODUCTION
032
033
      Large Language Models (LLMs) have emerged as powerful tools in various domains, including
      healthcare (Chen et al., 2023; Xu et al., 2023), law (Wu et al.; Deng et al., 2023a), and finance (Wu
034
      et al., 2023; Xie et al., 2023). With the exception of a very small portion of users who have the
035
      resources and expertise to deploy LLMs locally, the vast majority of users access and interact with
036   these powerful models through online inference services.
037
038   However, the widespread usage of online LLMs inference services has raised significant privacy
039
      concerns, especially regarding the potential risk of private information being leaked through user
      inputs when interacting with LLMs deployed on cloud platforms. User inputs often contain sensitive
040
      information such as details in medical records and legal cases. Potential threats may arise from
041
      eavesdropper attackers intercepting user queries during transmission to cloud platforms for malicious
042   purposes. For example, in sensitive domains like medical diagnosis, if a user’s input containing
043   personal health information, such as "I was previously diagnosed with HIV, and lately I’ve been
044   experiencing fever and diarrhea..." is disclosed, it may cause troubles to their life.
045
      In this paper, we focus on protecting the private information contained in user inputs during LLM
046
      inference stage. In this setting, the client submits inputs to the server (also known as the service
047
      provider) and there is a risk that inputs might be disclosed by attackers. Current methods for protecting
048   user inputs can be categorized into two categories: Secure Multi-Party Computation (SMPC) and
049   Differential Privacy (DP). SMPC based methods (Hao et al., 2022b; Li et al., 2023a; Liang et al.,
050   2024) utilize encryption protocols and algorithms to enable collaborative computation without
051   revealing original data to others. However, SMPC methods have large inference time overhead,
052   making them impractical for real-time applications. For example, running a single pass inference
053   on the RoBERTa-Base (Liu et al., 2019) requires 168.43 seconds (Hao et al., 2022a). DP based
      methods (Feyisetan et al., 2020; 2019; Xu et al., 2020; Bo et al., 2021) introduce the definition of


                                                         1
      Under review as a conference paper at ICLR 2025


054
      dχ -privacy (Chatzikokolakis et al., 2013; Alvim et al., 2018) and apply a word-level text-to-text
055   privatization on data locally before transmitting data to the server. Nevertheless, DP based methods
056   inevitably degrade the performance of downstream tasks due to noise injection, which is known as the
057   privacy-utility trade-off. Additionally, as text length increases, the performance degradation becomes
058   pronounced, This phenomenon is known as the linear growth of the privacy budget Mattern et al.
059   (2022b) in differential privacy. Hence, there is a need to develop privacy-preserving methods which
060   can effectively safeguard the privacy of user inputs while maintaining high-quality outputs, without
061   incurring prohibitive computational costs.
062   We propose PrivacyRestore which directly removes privacy spans in user inputs and restores private
063   information via activation steering (Li et al., 2023c; Turner et al., 2023; Hernandez et al., 2023)
064   during model inference. Our method is based on two key assumptions: (a) Private information is
065   confined within specific a contiguous sequence of tokens, termed “privacy span”, rather than
066   being dispersed throughout the entire input. Privacy span is defined as a contiguous sequence of
067   tokens within a text that contain private information. The removal or proper redaction of privacy
068   spans significantly impedes unauthorized parties from reconstructing or inferring the underlying
069
      private information. For instance, if privacy spans "HIV" , "fever" and "diarrhea" are removed from
      a medical record "I was previously diagnosed with HIV, and lately I’ve been experiencing fever
070
      and diarrhea...", attackers can not recover any private information. (b) In a particular domain,
071
      the number of potential privacy spans is limited and finite. For example, in the application of
072   medical diagnosis, privacy spans generally pertain to symptoms and disease names, and the number of
073   possible symptoms and disease names is inherently limited. Moreover, although medical knowledge
074   and terminology inevitably evolve, the core set of symptoms and diseases remains relatively stable
075   and finite.
076
      PrivacyRestore operates in two stages: the preparation stage and the inference stage. In the prepa-
077
      ration stage, we first identifies the attention heads where the activation steering occurs. Second,
078   each privacy span is encoded to a vector named restoration vector. This stage is conducted on the
079   server side. Our method is plug-and-play, requiring only the restoration vectors to be trainable, while
080   keeping the LLM frozen. Once training is complete, the users keep all restoration vectors on the
081   client side. In the inference stage, the user construct a meta vector by first estimating the importance
082   of each privacy span in the input and then calculating a weighted sum of the corresponding restoration
083   vectors. The user then removes the privacy spans from the input and submits the remaining input
084   along with the meta vector to the server. The server uses the meta restoration vector to restore the
085   removed privacy spans through activation steering.
086   To prevent the leakage of privacy spans via reverse-engineering on the meta vector, the dχ -privacy
087   mechanism (Feyisetan et al., 2020) is applied on the meta vector before transmission on the client side.
088   dχ -privacy mechanism is a variant of the differential privacy mechanism (Dwork et al., 2016). By
089   applying dχ -privacy to the meta restoration vector instead of words, our method inherently addresses
090   the issue of linear growth of privacy budget (Mattern et al., 2022a) commonly encountered in dχ -
091   privacy and other DP variants. Experimental results demonstrate that the proposed method effectively
092
      protects private information and maintains satisfactory performance and inference efficiency.
093   The contributions of our paper are summarized as follows,
094
095
          • We propose a plug-and-play privacy protection method that removes privacy spans in the input
            and restores private information via activation steering during inference.
096
097       • We propose Attention-aware Weighted Aggregation to construct the meta vector and apply the
098         dχ -privacy mechanism to the meta vector, inherently addressing the problem of the linear growth
099         of privacy budget.
100       • We construct three datasets, covering the medical and legal fields, to evaluate our method. Ex-
101         perimental results demonstrate its capabilities of privacy protection. It also maintains acceptable
102         performance and inference efficiency.

107   In this section, we introduce the related works on user input protection methods, which are currently
      divided into two categories: SMPC-based methods and DP-based methods.

      2.1   S ECURE M ULTI -PARTY C OMPUTATION (SMPC)
109
110
      Secure multi-party computation (SMPC) methods utilize multi-party encryption algorithms to enable
111
      collaborative computation among multiple parties while protecting the privacy of their data. However,
112   most nonlinear operations in LLMs cannot directly support secure multi-party computation. To
113   address this challenge, current SMPC methods focus on two optimization directions: model structure-
114   oriented optimization and protocol-oriented optimization.
115
      The model structure-oriented approach aims to replace SMPC-unfriendly nonlinear operations with
116
      SMPC-friendly alternatives. For instance, MPC-Former (Li et al., 2023a) approximates nonlinear
117
      operations in Transformer using polynomials and maintains performance through model distillation.
118   MERGER (Liang et al., 2024) integrates previous techniques to natural language generation (NLG)
119   tasks by bypassing embedded computation and reorganizing linear operations in Transformer modules,
120   further enhancing computational efficiency and model performance. In contrast, the protocol-oriented
121   approach focuses on designing efficient SMPC operators for nonlinear operations in LLMs while
122   preserving the original model structure. Recent works Hao et al. (2022b); Liu & Liu (2023); Zheng
123   et al. (2023b); Gupta et al. (2023) have improved the efficiency of nonlinear operations in privacy-
124   preserving LLMs inference by utilizing various SMPC protocols, such as confusion circuit and
125   function secret sharing.
126   Although SMPC-based methods can be applied to protect user inputs during model inference, they
127   still suffer from large inference time overhead. For example, inference on the RoBERTa-Base model
128   takes 168.43 seconds (Hao et al., 2022a), making current SMPC methods impractical for online LLM
129   inference services.
130
131
132   2.2   D IFFERENTIAL P RIVACY (DP)
133
134   Differential Privacy (DP), as introduced by Dwork et al. (2016), is designed to protect individual
135   privacy by preventing attackers from identifying specific participants in a dataset. Several variants
136   of DP have been developed to enhance privacy protection across various settings, adapting the core
137
      principles of DP to different types of data and threat models. Notable examples include Centralized
      Differential Privacy (CDP), Local Differential Privacy (LDP) and dχ -privacy.
138
139   CDP (Dwork et al., 2016) operates under the assumption that all data has been stored in a central
140   repository. It guarantees that attackers cannot distinguish between any two adjacent repositories based
141   on query results. In contrast, LDP (Duchi et al., 2013) provides a stronger guarantee, ensuring that
142   attackers cannot distinguish between any two adjacent inputs. Mattern et al. (2022b) and Utpala et al.
143
      (2023) propose using paraphrasing techniques to achieve LDP on user inputs. The formal definitions
      of CDP and LDP are provided in Appendix D.
144
145   LDP allocates the same privacy budget ϵ to all adjacent input pairs, regardless of their similarity.
146   Applying the same ϵ forces each user input to be indistinguishable from any other, which can
147   negatively impact data utility. However, it is sufficient for privacy protection to make each user input
148   indistinguishable only from its closer counterparts. To address this, dχ -privacy (Feyisetan et al.,
149
      2019), a relaxed version of LDP, incorporates metrics that measure the similarity between inputs,
      allowing for more flexible control over the privacy budget. dχ -privacy is defined as,
150
151   Definition 2.1. (dχ -privacy). A randomized mechanism M : I → O fulfills ϵ-dχ -privacy if for all
152   adjacent inputs I, I ′ ∈ I and all possible outputs O ⊂ O,
153
154
                               P (M(I) ∈ O) ≤ exp(ϵdχ (I, I ′ ))P (M(I ′ ) ∈ O) ,
155
156
157   where dχ is a distance function defined on I. Recent works (Feyisetan et al., 2020; Xu et al., 2020;
158   Li et al., 2023d; Qu et al., 2021) leverage dχ -privacy to safeguard user inputs during both inference
159   and fine-tuning phases. However, as noted by Mattern et al. (2022b), all of the aforementioned DP
160   variants suffer from the linear growth of the privacy budget. A larger privacy budget indicates weaker
161   privacy protection, meaning that as the input length increases, the effectiveness of privacy protection
      diminishes.

      3     T HREAT M ODEL
163
164
      We consider a threat model involving two parties: a server that holds the LLM weights and a client
165   holds user inputs containing privacy spans. Privacy span is defined as a contiguous sequence of
166   tokens within a text that contain private information. The server provides services through an
167   API, enabling the client to transmit inputs and receive responses while maintaining the confidentiality
168   of the LLM weights. The server may be vulnerable to attacks by adversaries seeking to steal privacy
169   information in user inputs. Our task is to protect these private spans in user inputs from being
170   intercepted to the adversaries, even when the adversaries can attack the server directly.
171
172   4     M ETHODOLOGY
173
174   To protect privacy, PrivacyRestore transmit the input with privacy spans removed instead of the entire
175   input text to the server. The information in the privacy spans is encrypted as a vector, which is then
176   injected with noise and is also sent to the server. We propose to use activation steering (Li et al.,
177   2023c; Turner et al., 2023; Hernandez et al., 2023) to restore privacy information. Activation steering
178   methods modify the activations of a language model at inference time to predictably alter its behavior.
179   Activation steering is widely used to truthfulness enhancement(Li et al., 2023c), LLMs detoxifying
180   (Li et al., 2024), and sentiment modification (Turner et al., 2023). To our best knowledge, it is first
181
      attempt to use activation steering for privacy information restoration. We include the preliminaries of
      our methodology in Appendix C.
182
183   PrivacyRestore operates in two stages, i.e., the preparation stage and the inference stage.
184
      (1) Preparation stage: This stage takes place on the server. We first identify the edited attention heads
185   and train the restoration vectors for each privacy spans. After training, these vectors are released
186   to the clients. The preparation stage is conducted offline, prior to the server beginning to offer its
187   services.
188
      (2) Inference stage: This stage involves collaboration between the client and server. The client
189
      constructs a meta vector which is later transmitted to the server along with the input where the privacy
190
      spans are removed. The server then performs inference on the input with privacy spans removed and
191   restores those privacy spans via modifying activations.
192
193   An overview of PrivacyRestore is shown in Figure 1. Detailed descriptions of the preparation stage
194
      and the inference stage are provided in §4.1 and §4.2 respectively. The definitions of all notations
      used in this paper can be found in Appendix A.

      5     A NALYSIS OF P RIVACY B UDGET
325
326   In this section, we analyze the privacy budget of DP variants and PrivacyRestore.
327
328
      Theorem 5.1. The DP variants including CDP, LDP and dχ -privacy are constrained by a privacy
      budget that grows linearly with the length of the protected text.
329
330   The detail proof of Theorem 5.1 is presented in Appendix E. As the length of the protected text
331   increases, the growing privacy budget makes these DP variants more vulnerable to adversarial attacks,
332   thus compromising their robustness. We also provide empirical evidence demonstrating the linear
333   growth problem of dχ -privacy in Section 6.3. We implement two types of attack, i.e., prompt injection
334
      attack (Perez & Ribeiro, 2022; Suo, 2024) and attribute inference attack (Li et al., 2022), across
      three privacy-preserving datasets. As shown in Figure 3(a) and 3(b), attack performance increases
335
      with the length of the protected text, highlighting the linear growth problem of the privacy budget in
336
      dχ -privacy.
337
338   Theorem 5.2. PrivacyRestore fulfills dχ -privacy and provides a privacy budget of ϵ||Z ′ − Z||, where
339
      ϵ denotes privacy hyperparameter, and Z ′ and Z represent any pair of normalized weighted sums of
      restoration vectors concatenated across all edited heads. The privacy budget of PrivacyRestore is
340
      independent of the length of protected text.
341
342   The detail proof of Theorem 5.2 is provided in Appendix F. The ϵ is privacy hyperparameter,
343   independent of the length of protected text. The ||Z ′ − Z|| represents the distance between two
344   vectors, also independent of the length of protected text. Since the privacy budget in PrivacyRestore
345
      is independent of the length of the protected text, our method effectively protects privacy even with
      longer protected text, inherently addressing the linear growth issue in DP variants. We also provide
346
      empirical evidence to support the theorem. We present the attack performance of PrivacyRestore
347
      across varying protected text lengths, as detailed in Section 6.3.

      Datasets. We evaluate our method in medical and legal domains. However, existing benchmarks,

      SLJA (Deng et al., 2023b) for legal judgement, do not specify privacy spans in the input. To address
356   this gap, we leveraged GPT-3.5 (Ouyang et al., 2022) to classify symptoms in DDXPlus/NLICE
357   and case details in SLJA into sensitive and non-sensitive categories, treating the sensitive data as
358   privacy spans. The classification prompt is shown in Appendix O.1. Based on classification results,
359   we curated three privacy-preserving datasets: Pri-DDXPlus, Pri-NLICE and Pri-SLJA, with 149,
360   64 and 142 types of privacy spans, respectively. The process of dataset construction and statistical
361   information can be found in Appendix B.
362   Metrics. The evaluation access both performance and inference efficiency. For performance
363   evaluation, we employ MC1 and MC2 (Lin et al., 2021) to measure the model’s accuracy in selecting
364   the correct answer among 4 options. Each sample in Pri-DDXPlus, Pri-NLICE, and Pri-SLJA is
365   assigned with 4 options, including one correct and three incorrect options. The detailed calculation
366
      of MC1 and MC2 is outlined in Appendix J. We also evaluate the model’s generation ability using
      ROUGE-L and LLM-Judge (LLM-J)(Zheng et al., 2023a). For ROUGE-L, the reference text is the
367
      initial output produced by the backbone LLM. As ROUGE-L primarily focuses on n-gram overlap
368
      between generated text and reference texts, which may not fully capture the semantic meaning or
369   overall quality of the generated content, we further use a LLM (i.e., GPT-3.5) to assess the quality of
370   outputs considering relevance, clarity, and accuracy. The assessment prompt is shown in Appendix
371   O.3. The LLM-J score ranges from 1 to 10, with higher scores indicating better quality. For inference
372   efficiency, we use Throughput (TP), defined as the number of tokens generated per second, to
373   evaluate the inference efficiency.

      6.2   M AIN R ESULTS
413
414   As shown in Table 1, we evaluate the performance and inference efficiency of PrivacyRestore and
415   other compared methods across three privacy-preserving datasets. Compared to dχ -privacy and
416   paraphrase, dχ -privacy on privacy spans solely apply dχ -privacy mechanism to those privacy spans
417   and achieves higher scores in MC1/2, ROUGE-L and LLM-J. The possible reason for this is that
418   both dχ -privacy and paraphrase operate on the entire user input, instead of specific privacy spans.
419   Injecting noise into the entire input creates larger disturbances during inference compared to only
420   corrupting a limited number of privacy spans.
421   PrivacyRestore achieves best scores in MC1/2 and LLM-J compared to other privacy-preserving
422   methods. In terms of the ROUGE-L evaluation metric, PrivacyRestore achieve the best result in
423   Pri-NLICE while ranking second in the other two datasets. This discrepancy likely stems from
424   ROUGE-L’s dependence on n-gram overlap between the reference text and the generated output,
425   which does not fully reflect the quality of generated outputs. As demonstrated by the examples in
426   Figure 6 and Appendix N, PrivacyRestore often generates outputs with different sentence structures
427   while still providing accurate answers. Consequently, our method achieves slightly lower ROUGE-L
428
      scores but significantly higher LLM-J scores compared to dχ -privacy on privacy spans. Furthermore,
      the ROUGE-L metric displays larger variance than the LLM-J metric, potentially due to its sensitivity
429
      to expression rather than the underlying meaning of the generated output.
430
431   As for inference efficiency, dχ -privacy achieves the highest throughput. In contrast, dχ -privacy on
      privacy spans requires prior identification of privacy spans, while paraphrase necessitates rephrasing

440   attack methods to empirically evaluate the privacy protection capability of our approach. PrivacyRe-

529   7     CONCLUSION
530
531   We propose PrivacyRestore which protects the privacy within user inputs during inference in online
532   LLM inference services. PrivacyRestore achieves privacy protection by directly removing privacy
533   spans in the user input and then restoring these privacy spans via activation steering. PrivacyRestore
534   provides a practical and efficient solution for protecting privacy while maintaining satisfactory
535   performance and inference efficiency. We demonstrate that PrivacyRestore inherently addresses
536   the linear growth problem of the privacy budget found in differential privacy variants. We curate
537   three privacy-preserving datasets covering medical and legal fields, and PrivacyRestore achieves
538
      strong performance and inference efficiency across all datasets. Additionally, we implemented two
      types of attacks, and the experimental results demonstrate PrivacyRestore’s robust privacy protection
539
      capabilities.

      B.1   C ONSTRUCTION P ROCESS
819
      We used GPT-3.5 (Ouyang et al., 2022) to classify symptoms in DDXPlus and NLICE, as well as
820
      case details in SLJA, into five levels ranging from non-sensitive to highly sensitive. The assessment
821   prompt template is shown in Appendix O.1. A higher level indicates that the symptom or case detail
822   is more sensitive. We define all symptoms and case details with a sensitivity level greater than 3 as
823   privacy spans.
824
      We assign each sample a correct answer along with three randomly selected incorrect options. For
825
      DDXPlus and NLICE, we randomly select three diagnosis results to combine with the correct
826
      diagnosis as the choices. In the SLJA dataset, we randomly select three legal judgments to pair with
827   the correct one as the options.
828
829   The initial dataset is extensive, and we observed that for most samples, removing all privacy spans
830
      often yields outputs similar to those obtained when privacy spans are provided. Privacy preserving
      for these samples is meaningless because users can directly hide those privacy spans and obtain
831
      approximate result outputs. In real-world scenarios, sensitive privacy spans often play a crucial role in
832
      medical diagnoses and legal judgments, making privacy preservation highly valuable. Our dataset is
833   designed to benchmark various privacy-preserving methods and must include samples where privacy
834   spans are crucial for generating outputs. We utilize the KL divergences to measure the importance
835   scores of samples. We calculate the KL divergence between the model output distributions with
836   and without the privacy symptoms included. A higher KL divergence indicates that the absence
837   of sensitive privacy spans may lead to different or incorrect outputs. We selected only samples
838   with high KL divergence to construct the privacy-preserving datasets. As a result, we curated three
839   privacy-preserving datasets: Pri-DDXPlus and Pri-NLICE for medical diagnosis, and Pri-SLJA for
840   legal judgment.

       Definition D.1. (CDP). A randomized mechanism M : Q → G fulfills (ϵ, δ)-differential privacy if
973    for all adjacent queries Q, Q′ ∈ Q and all possible query results G ⊂ G,
974
975                                P (M(Q) ∈ G) ≤ exp(ϵ)P (M(Q′ ) ∈ G) + δ.
976
977    CDP (Dwork et al., 2016) ensures that adversaries cannot distinguish between Q and Q′ based on G
978    due to the similar probabilities, meaning the query results are probabilistically indistinguishable. This
979
       prevents adversaries from inferring characteristics about the repository based on multiple queries and
       their results.
980

983    However, privacy risks can also emerge during the data collection process itself, as attackers may
984    intercept user inputs while they are being transmitted to the central repository. LDP (Duchi et al.,
985    2013) protect user inputs during transmission process by ensuring that attackers cannot distinguish
986    between any two adjacent inputs, which is defined as,
987
       Definition D.2. (LDP). A randomized mechanism M : I → O fulfills (ϵ, δ)-differential privacy if
988    for all adjacent inputs I, I ′ ∈ I and all possible outputs O ⊂ O,
989
990                                P (M(I) ∈ O) ≤ exp(ϵ)P (M(I ′ ) ∈ O) + δ.
991
992    The mechanism A processes the user input before transmitting it. LDP ensures that, even if attackers
993    intercept O, they cannot distinguish between the initial user input I and the adjacent one I ′ .

       M.2    ATTRIBUTE INFERENCE ATTACK
1300
1301   Attribute inference attack attempts to steal user inputs by performing classification on the garbled
1302   inputs, where the target labels correspond to the token IDs of the original inputs. Since each input
1303   contains multiple tokens, this classification task is naturally a multi-label classification problem.
1304   Following Li et al. (2022), we utilize a multi-layer perceptron (MLP) model as the classifier. The
1305   input dimension is 4096 and the output dimension is the size of the whole vocabulary size. To
1306   evaluate the attack’s performance, we calculate the F1 score of the classification, where a higher F1
1307
       score indicates a more successful attack. The attack targets can include garbled text from dχ -privacy,
       paraphrased text, or the meta vector from PrivacyRestore. The implementation details for text and
1308
       vectors may vary slightly.
1309
1310   Attribute inference attack on meta vector. To attack the meta vector from PrivacyRestore, we can
1311   directly use a fully-connected layer to transform the meta vector’s dimension from 128 × K to the
1312   classifier’s input dimension 4096. We then perform classification on the transformed meta vector.
1313   Attribute inference attack on text. For garbled text from dχ -privacy or paraphrased text, we first
1314   transform the text into a vector representation. We utilize Llama2-chat-7b to process the text input
1315   and obtain the last token’s hidden state as the vector representation. Classification is then performed
1316   on this hidden state.
1317
1318   M.3    E MBEDDING INVERSE ATTACK
1319
1320   Different from attribute inference attack, embedding inverse attack steal the user inputs through
1321
       the generative model to generate the original user inputs. We utilize the GPT-2 model (Radford
       et al., 2019) as the generative model and set the maximum generation length to 256. We finetune the
1322
       GPT-2 model on the training set for 20 epoch using the learning rate of 1e-5. To evaluate the attack’s
1323
       performance, we compute the ROUGE-L score between the generated output of the GPT-2 model
1324   and the original user input, where higher scores indicate better attack effectiveness. Similar to the
1325   attribute inference attack, the implementation of the embedding inverse attack differs between text
1326   and meta vectors.
1327
       Embedding inverse attack on meta vector. We use a fully-connected layer to transform the meta
1328
       vector’s dimension to the dimension of hidden state of GPT-2 model. Then we directly input the
1329
       transformed meta vector as the input embedding.
1330
1331   Embedding inverse attack on text. We use Llama-2-chat-7b to process the text input and extract
1332   the last token’s hidden state as the vector representation. This vector is then transformed by a fully
1333
       connected layer to match the hidden state dimension of the GPT-2 model. Finally, it is fed into the
       GPT-2 model as the input embedding for subsequent generation.
