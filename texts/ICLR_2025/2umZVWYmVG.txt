                                                   A BSTRACT
006
007
               Frontier large language models (LLMs) consider reasoning as first-class citizens:
               they learn to refine their reasoning process and try different strategies during train-
008
               ing. Thereby, when prompted, can think through problems and respond better with
009
               proper reasoning. For programming tasks, this makes code reasoning a must. In
010            this paper, we propose the task of Code Execution Simulation (CES) as a proxy
011            for evaluating the code reasoning capabilities of LLMs. CES defines the notions
012            of valid or invalid reasoning process, which enables it to promptly (1) determine
013            where the execution simulation diverges from ground truth for incorrect output
014            predictions (essential to understanding limitations of LLMs in code reasoning)
015            and (2) identify suspiciously correct output predictions (essential to understand-
016            ing reasoning shortcuts, hallucinations, or potential data leakage). In addition
017            to evaluating LLMs’ execution reasoning on a program with a single test, CES
018
               measures their reasoning consistency across tests with the same or different prime
               path coverage. This enables it to evaluate the code reasoning of LLMs in a spec-
019
               trum: strong, weak, and random. Our results show that LLMs, to a great ex-
020
               tent (83.32%), follow a valid reasoning process (results in 30.79% correct and
021            51.53% incorrect output predictions). However, their reasoning is mostly random
022            (55.59%) or weak (41.69%), which explains their weakness in programming tasks
023            that require flow- or path-sensitive program analysis to succeed.

025   1   I NTRODUCTION
026   Large Language Models (LLMs) have shown emerging abilities in code/test synthesis, bug/vulnera-
027   bility detection, code translation, and program repair (Roziere et al., 2023; Zhu et al., 2024; Achiam
028   et al., 2023; Lozhkov et al., 2024; Reid et al., 2024; Yang et al., 2024; Mishra et al., 2024). The
029   extent to which they can reason about code execution is still under investigation. C RUX E VAL (Gu
030   et al., 2024) and C ODE M IND (Liu et al., 2024a) proposed the output prediction task for given inputs
031   to evaluate LLMs for code execution reasoning. RE VAL (Chen et al., 2024) took one step further
032   and evaluated LLMs using four runtime behavior prediction tasks: for given inputs and a statement
033
      in the program, predict (1) if the statement is covered during execution, (2) variable values after the
      execution of it, (3) the next statement executed after it, and (4) final output.
034
035   These techniques lack the following essential features: Flow Sensitivity. Assessing LLMs for fol-
036   lowing the correct execution path from start to end is essential for code reasoning. C RUX E VAL and
037   C ODE M IND only concern output prediction without evaluating intermediate program states. RE VAL
038   evaluates runtime behavior prediction of a subset of statements in programs, not all, due to computa-
039
      tional overhead inherent in their design. It also prompts an LLM separately per individual statement
      without strategies to combine all predictions for the entire program. Diagnosis. None of these tech-
040
      niques can promptly and reliably determine (1) where the LLM’s execution reasoning diverges from
041
      the ground truth and results in incorrect output prediction, or (2) flags suspiciously correct output
042   predictions. The former is essential to understanding the limitations of LLMs in code reasoning and
043   improving the next generation of Code LLMs. The latter can reveal data contamination, halluci-
044   nations, and reasoning shortcut cases, which are important trustworthiness concerns (Zhang et al.,
045   2023; Shi et al.; Ding et al., 2024a). Reasoning Consistency. Prior techniques evaluate the reason-
046   ing of LLMs per single test, failing to study the consistency of reasoning across multiple tests with
047   potentially different coverage. Investigating such consistency can reveal the strength of inductive
048   reasoning in LLMs: a model that correctly reasons about execution across tests with different cov-
049   erage is unlikely to succeed by chance (Strong Reasoning), and a model that correctly reasons about
050
      the execution across tests with the same coverage, but not those with difference coverage has a Weak
      Reasoning abilities. Otherwise, the reasoning of the model can be considered random.
051
052   We propose Code Execution Simulation (CES) task for assessing LLMs in code reasoning. CES
053   unifies output prediction and intermediate program state predictions into one prompt. Asking LLM
      to predict all intermediate program states and output makes the task complex and can confuse it,

153   3    C ODE E XECUTION S IMULATION (CES)

      A program P = {si |si ∈ Sassign ∪ Sloop ∪ Scondition ∪ Sreturn } is a set of statements that can
155
      assign a value to a variable (Sassign ), introduce recursions in the logic (Sloop ), cause branches in the
156   control flow (Scondition ), or terminate the program execution (Sreturn ). Given input(s) I to P , CES
157   evaluates LLMs in predicting properties related to Sloop , Scondition , and Sreturn statements. These
158   statements identify the start or end of basic blocks in the program and can capture mispredictions in
159   the assignment statements inside the block. Furthermore, asking a model to predict all intermediate
160   variable values can make the task complex, preventing proper responses to evaluate them (Liu et al.,
161   2024b). In this section, we first define these properties (§3.1) and explain how the CES prompts
      models and evaluates their abilities in simulation execution (§3.2). We also define the notions of

165   3.1     P ROGRAM P ROPERTIES D EFINITIONS
166   Definition 1. Loop Properties. A program may contain m (m ≥ 0) loop statements. A loop
167   statement lj ∈ Sloop = {l1 , . . . , lm } consists of two main components: loop variable (Vlj ) and
168   loop iterable (Ilj )1 . The loop variable keeps track of the iterations, and the loop iterable defines
169   the values or orders of the loop variable. In the example of Figure 1-b, the statement in Line 6
170   is a loop statement. The loop variables are e and o, and the loop iterable is zip(evens,odds).
171   For each lj in a given program P and concerning input(s) I, CES will ask the LLM to predict the
172   values of all Vlj s. The loop iterable can be a compound, i.e., consists of multiple variables or API
173   calls. As a result, CES asks the LLM to predict values of all sub-components. The rationale here
174   is to correctly identify the root cause for simulation divergence. In the example of Figure 1-b, the
175
      LLM may diverge from ground truth by mispredicting the values of evens or odds, or it may fail
      to understand the logic of zip API in Python library, mispredicting the return value of it even with

285   4 T HE S PECTRUM OF C ODE E XECUTION R EASONING
286
      The code that LLMs generate should be comprehensively tested using tests that cover all the ex-
287
      ecution paths in the program. Otherwise, one can falsely claim an incorrect code to be correct.
288   Similarly, one cannot claim victory on code reasoning until the LLM can correctly simulate all the
289   execution paths. CES introduces the spectrum of code reasoning and evaluates the models based on
290   their reasoning consistency across different execution paths.
291
      Figure 5 shows a program with its control flow graph. Due to the existence of a loop, the number
292
      of execution paths for this program depends on the input. To account for this issue and having a
293
      bounded number of execution paths, CES considers prime paths in the control flow graph (Ammann
294   & Offutt, 2017). A prime path in a cyclic graph is a path between two arbitrary nodes that does not
295   visit any node more than once except for the starting and ending nodes. Figure 5-c shows three tests
296   for the program of Figure 5-a and their prime path coverage during test execution. The union of
297   these tests covers all the prime paths, hence, critical execution sequences in the program.
298

504   5.5    RQ4. AGREEMENT B ETWEEN CES AND OTHER P ROGRAMMING TASKS
505
506   LLMs should incorporate their knowledge of programming languages and examples they have seen
507   to solve programming tasks. Otherwise, one cannot expect them to generalize to different tasks,
508
      perform reasonably on real-world programs, or trust them. We consider three programming tasks
      that, outside of the LLM world, performing them require execution awareness: bug prediction, bug
509
      localization, and bug repair. Ideally, if an LLM can correctly simulate an execution path, it is more
510
      likely to detect, localize, and repair a bug in that specific execution path. Similarly, we expect that
511   if a model cannot correctly simulate an execution path, it fails to detect, localize, or fix the bug.
512
513   To investigate these two hypotheses, we used OctoPack (Muennighoff et al., 2023), a dataset of
514
      bugs generated by humans injected into HumanEval programs. We first executed OctoPack tests
      and identified the failed tests on the buggy version of HumanEval programs. We then checked
515
      if those were among our three sampled tests and kept the programs and tests passing the check.
516
      We prompted the seven best-performing models in CES with the prompts in §A.2 to perform bug
517   prediction, localization, and repair.
518
519   Table 2 illustrates the performance of models in these tasks, as well as the Cohen’s Kappa (McHugh,
520
      2012) coefficients representing the pairwise agreement between CES and these tasks. Cohen’s
      Kappa is a statistical test that checks the agreement between the two groups in terms of the va-
521
      lidity of the property (how well Group A’s validity predicts Group B’s validity). It accounts for
522
      agreement by chance, making it more reliable than a simple percentage of agreement. Its coefficient
523   (κ) takes a value between -1 and 1. The κ = 1 shows perfect agreement; κ = 0 indicates agreement
524   by chance; negative values demonstrate no agreement or systematic disagreement.
525
526
      From these results, we can see that there is, at best, a slight agreement between CES 4 and these
      tasks, rejecting our mentioned hypotheses. Figure 8 (§A.6 in Appendix) demonstrates the degree of
527
      overlap between CES and individual bug-related tasks across the programs. To better understand
528
      the reasons for agreements and disagreements, we manually investigated instances where models (1)

670   Figure 14 shows the detailed prompt template used in CES, consisting of five key components: the
671   system message, adaptive in-context learning examples, instruction, question, and response. In both
672   the question and the in-context learning examples, we begin by commenting on the program us-
673   ing ”sub-questions,” which include variables or expressions related to loop or conditional construct
674   properties, along with placeholders marked as ”??”. Within the in-context learning examples, we
675
      also provide sample answers where the placeholders are replaced with the correct values or states.
      The instruction specifies how the LLMs should respond to the questions: answers related to loop pre-
676
      dictions should be enclosed in ”[STATE]” and ”[/STATE]”, condition-related predictions in ”[CON-
677
      DITION]” and ”[/CONDITION]”, and branch predictions in ”[BRANCH]” and ”[/BRANCH]”. In
678   the question section, we annotate the program similarly to how it is done in the in-context learning
679   examples, and we expect the LLMs to generate responses in the format outlined in the instruction.

681
      With the growing capabilities of large language models (LLMs), in-context learning (Dong et al.,
      2022) has emerged as an important paradigm for natural language processing (NLP) tasks. More-
682
      over, research (Wu et al., 2022; Wan et al., 2023) suggests that selecting well-performing in-context
683
      learning examples tailored to different inputs can enhance LLMs’ ability to generate correct outputs.
684   Building on the concept of adaptively selecting in-context learning examples, we developed 11 sets
685   of examples based on program constructs and their locations: if, elif, nested if, for loop, while loop,
686   nested loop, if inside while loop, if outside while loop, if inside for loop, if outside for loop, and
687   if inside nested loop. For each problem in the benchmark, we utilize the tree-sitter 5 to extract code
688   constructs and their locations, and then select the corresponding examples from our predesigned
689   in-context learning candidates.
