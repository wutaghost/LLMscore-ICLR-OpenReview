009                                                                      A BSTRACT
010
011
012              Large language models (LLMs) are increasingly being used to synthesize and
013              reason about source code. The libraries and API functions they invoke are continu-
014              ously evolving, with functionality being added or changing. Yet, no prior work has
015
                 studied how an LLM’s knowledge about code API functions can be updated. To fill
                 this gap, we present CodeUpdateArena, a benchmark for knowledge editing in the
016
                 code domain. An instance in our benchmark consists of a synthetic API function
017
                 update paired with a program synthesis example that uses the updated functionality;
018              our goal is to update an LLM to be able to solve this program synthesis example
019              without providing documentation of the update at inference time. Compared to
020              knowledge editing for facts, success here is more challenging: a code LLM must
021              reason about the semantics of the modified function rather than just reproduce its
022              syntax. Our dataset is constructed by first prompting GPT-4 to generate atomic and
023              executable function updates. Then, for each update, we generate program synthesis
024              examples whose code solutions are prone to use the update. Our benchmark covers
025              updates of various types to 54 functions from seven diverse Python packages, with
026
                 a total of 670 program synthesis examples. Our experiments show that fine-tuning
                 open-source code LLMs (i.e., DeepSeek, CodeLlama) on documentation of a new
027
                 update does not allow them to incorporate changes for problem-solving. How-
028
                 ever, prepending the same information does help, establishing that the information
029              is present, and careful fine-tuning on examples demonstrating the update shows
030              improvement, paving the way for better knowledge editing techniques for code.

      1       I NTRODUCTION
055
056
      Large language models (LLMs) have demonstrated strong abilities to synthesize code to solve
057   problems (Chen et al., 2021; Li et al., 2023; DeepSeek-AI et al., 2024; Guo et al., 2024a). This
058   capability enables them to use external libraries: they can invoke standard libraries for data science-
059   related tasks (Lai et al., 2023), program SMT solvers (Ye et al., 2023), or use external modules
060   for tasks like computer vision (Gupta & Kembhavi, 2023). However, such APIs are not static and
061   adherence to older APIs can cause failures. For example, in a live demo,1 GPT-4 failed to correctly
062   implement a Discord bot due to outdated API knowledge. To be maximally useful, LLMs for code
063   generation need to stay in sync with API updates, even those that occur after pre-training.
064   A separate line of research studies knowledge editing for LLMs on simple facts. Typical use-cases
065   here are teaching LLMs about new entities (Onoe et al., 2023), updating roles of existing entities like
066   who the British prime minister is now (De Cao et al., 2021; Mitchell et al., 2022), and other such
067   temporally-sensitive knowledge (Zhang & Choi, 2021). A number of techniques have been presented
068   for these settings to efficiently update the parameters of LLMs, such as with a single gradient update
069   (Mitchell et al., 2022; Meng et al., 2023) or with a small number of updates (De Cao et al., 2021;
070   Meng et al., 2022; Padmanabhan et al., 2023; Akyürek et al., 2024; Chen et al., 2023).
071   These studies suggest a natural parallel in the code setting: can we efficiently update a pre-trained
072   model’s knowledge of an API? In this work, we construct a benchmark to evaluate this capability.
073   Our benchmark instances, shown in Figure 1, consist of a problem setting defined by a synthetic API
074   update, such as an additional boolean flag in a function like numpy.argsort. We choose synthetic
075   updates, as information about any real API function update will likely be used as a pre-training corpus
076   by the next generation of pre-trained models. Then, for each function update, we have a number of
077
      program synthesis problems requiring the use of that update. Although there are solutions that do not
      use the update, the most parsimonious solutions do use the API functionality in question, and models
078
      are prompted to do so.
079
080   Our evaluation assesses whether LLMs can, after being updated on the synthetic API function update
081   (docstring, example usage, etc.), solve these program synthesis examples using the given API function
082   without being provided the update at inference time.
083   Our final benchmark, CodeUpdateArena, contains 670 program synthesis tasks, covering 54 functions
084   from 7 Python packages. Our benchmark is synthetically constructed by a carefully designed data
085   generation pipeline driven by GPT-4, enabling it to be scaled or updated with new instances in the
086   future. We manually filter our generated API updates and conduct a number of additional intrinsic
087   evaluations of dataset quality to establish the correctness of dataset instances.
088   Our experimental evaluation focuses on how existing small-scale LLMs (e.g., CodeLlama (Rozière
089   et al., 2023)) perform at this update setting when combined with existing knowledge updating
090   techniques. GPT-4 and Claude-3.5 are able to solve program synthesis examples when prompted
091   with the API update in context, with Claude-3.5 outperforming GPT-4 on its own generated data. We
092   then present two intuitive baselines for how practitioners would utilize API update information. The
093   first method, fine-tuning models on a docstring explaining the update does not improve performance.
094   However, fine-tuning on examples of the update being used does lead to improvement, and even
095
      outperforms having the API update in context. Through our ablation study, we found that the mix of
      training examples and learning rate are important for successful fine-tuning, but there is a tradeoff
096
      between efficacy and specificity of the update (impact on unrelated settings). We believe our dataset
097
      can provide a testbed for developing better methods for code knowledge editing in the future. Our
098   code is released at ©.

102   2       BACKGROUND AND R ELATED W ORK
103
104   Knowledge editing Knowledge editing involves updating a pre-trained model’s parameters to
105   contain additional knowledge that was not present in its pre-training corpus. Suppose we have a
106   model M and let (c, u) denote the additional knowledge u that should be returned in context c.
107

108
      Past work has focused on finding a model M′ such that M′ ≈ M and M′ (c) returns u with high
109   probability. For instance, suppose c = “the prime minister of the UK is” and u = “Rishi Sunak”; we
110   want to update the model’s knowledge about the UK’s prime minister with as little change to other
111   facts (e.g. Eiffel tower is in Rome) as possible.
112
      Prior work quantifies model editing success by measuring whether M′ can return u when prompted
113
      with c. A second goal is to preserve the original M as closely as possible, measured by ensuring that
114
      the model’s predictions on irrelevant contexts are not changed. The techniques for knowledge editing
115   typically involve gradient updates (De Cao et al., 2021), including meta-learned updates (Mitchell
116   et al., 2022), localized updates leveraging interpretability methods (Meng et al., 2022), and updates
117   on a collection of related examples (Padmanabhan et al., 2023; Akyürek et al., 2024).
118
      A third goal involves knowledge propagation (Onoe et al., 2023; Padmanabhan et al., 2023; Cohen
119
      et al., 2024; Powell et al., 2024; Zhong et al., 2023), where an LLM must be able to reason about the
120
      injected knowledge in contexts that may seem unrelated on the surface. However, current literature
121   has many negative results for this setting (Cohen et al., 2024; Hua et al., 2024). Our benchmark will
122   allow us to evaluate the state of affairs in the code setting, and whether functional competence around
123   code updates is more easily obtained than functional competence around textual knowledge.
124
125   Updates in Source Code Despite a large body of work on knowledge editing (Wang et al., 2023),
126   past work in this space has not explored the ramifications for code language models. Rather than
127   just reproduce an update like in knowledge editing settings (e.g. be able to generate Python 3.12
128   has lifted restrictions on the usage of f-strings), a user would likely expect a code LLM to be able to
129   generate, debug, or otherwise reason about code containing these updates.
130
      To the best of our knowledge, existing benchmarks mainly focus on general coding capabilities
131   of LLMs rather than their capability in dealing with API updates or historical versions existent in
132   pretraining corpus. Although some recent research has also explored providing documentations of
133   functions (or tools) (Zhou et al., 2022; Su et al., 2024; Zhang et al., 2023b; Hsieh et al., 2023) and
134   code snippets (Su et al., 2024; Zhang et al., 2023a; Phan et al., 2024; Shrivastava et al., 2022) to
135   LLMs in a retrieval-augmented framework (Chen et al., 2017; Guu et al., 2020; Lewis et al., 2020),
136   our main focus is on enabling LLMs to internalize this knowledge in an update (in-weight) and
137   propagate it during program synthesis as opposed to using it in-context. Therefore, our work also
138   relates to more general program synthesis using LLMs (Austin et al., 2021), especially those on
139
      developing benchmarks (Chen et al., 2021; Liu et al., 2023; 2024; Gu et al., 2024; Jimenez et al.,
      2024; Ding et al., 2023; Du et al., 2023; Guo et al., 2024b; Xie et al., 2024; Lai et al., 2023).
140
141
142   Defining an update taxonomy The goal of this work is to assess models’ abilities to be updated
143
      with realistic changes to functions in APIs. Most of the time when new functionality is introduced,
      the update extends existing methods in an atomic way. For example, a new sorting algorithm is
144
      supported for argument kind in numpy.argsort. To systematically capture different types of updates,
145
      we create a taxonomy for function updates, capturing what operation (add/modify/delete) is used to
146   update what component (function/argument/output) in what way.
147

149   3   TASK : CodeUpdateArena
150
151   We define f ← u to be the update made to an existing function f when providing it with new
152   semantics u. Our task involves understanding whether a pretrained code language model M can be
153   updated with f ← u. We assume that some kind of parametric update is made to yield a new model
154   M(f ←u) ; this can be done via various fine-tuning methods that have been proposed for knowledge
155   editing. We will describe exactly how u is conveyed to the language model in Section 6.1; here, we
156   focus on what capabilities we want the updated model M(f ←u) to exhibit.
157   To evaluate M(f ←u) , we provide a set of program synthesis examples P (f ←u) . Each program
158   synthesis example consists of a problem scenario si , a problem specification pi , and a set of T unit
159                (f ←u)
      test cases Ti       = {(ti,1 , ai,1 ), (ti,2 , ai,2 ), · · · (ti,T , ai,T )}.

      Each example scenario and specification is related to the updated semantics u. Let c̃i ←
163
      M(f ←u) (si , pi ) denote the result of predicting a code solution to problem i for update u. We
164
      want to evaluate M(f ←u) for three broad capabilities: (1) edit success: ∀j, c̃i (ti,j ) = aj (the update
165
      passes all test cases); (2) use of f : c̃i contains a call to the updated function f ; (3) specificity: the
166   update minimally changes the language model. See examples in Figure A.1.
167
168   Measuring whether samples from a code LLM pass test cases is typically done with pass@k (Chen
169
      et al., 2021). Drawing k samples from an LLM, what is the probability that one of those samples
      passes the test cases? This can be computed analytically without bias by drawing n > k samples,
170
      observing what number c of those samples pass the test cases, and using the formula from Chen et al.
171
      (2021) (reproduced in Appendix D). In this work, we set n = 5 and k ∈ {1, 2, 5}.
172
173   UPass@k Our main evaluation metric captures both edit success and use of f . We define UPass@k
174   as the standard pass@k except that it only counts solutions that meaningfully use the updated function
175   as “correct”.
176
      We run a solution against test cases with different function implementations at runtime:
177
178       a) when executing with the updated function in the environment, the solution must pass all tests.
179       b) when executing with the old function in the environment, the solution must fail some tests.
180
181   Details of how to do this execution are described in Appendix D. The first check is the standard one
182   used in pass@k. This second check ensures that the new functionality of f is leveraged in a nontrivial
183   way. Detecting a call to f is insufficient; if, for example, the update provides a new argument, we
184
      want the model to use that new argument rather than use f in its pre-update form.
185   Our program synthesis examples are designed to be naturally suited to the updated function f ← u.
186   It is, of course, possible for a code LLM to produce a solution that passes the tests but sidesteps the
187   usage of f altogether; however, in Section 6, we will see that prompted GPT-4 frequently does use
188   the update in successful solutions.
189
190   SPass@k captures how well the update is specific in that model’s other capabilities are not affected
191   (specificity) before (M) and after (M(f ←u) ) injecting each update. We discuss details in Section 6.1.

196   CodeUpdateArena, following recent work on generating synthetic datasets for complex tasks with

200   (described in Section 4.1) and generate program synthesis examples (Section 4.2). The output from
201
      each generation step is validated through manual inspection and heuristics. Figure 2 outlines our

241   Step 2: Generate a suite of unit tests Once the description of the update is available, we create a
242   set of unit tests to verify the correctness of the updated function f ← u.
243   To make the tests comprehensive, we ask GPT-4 to generate 10 unit test functions, testing edge cases
244   (e.g., empty input) and interaction with existing arguments (e.g. reverse=True and axis=1).
245
      See Appendix B.5 for the details of the prompt. We first generate unit test “skeletons”, unit test
246
      function with initialization of the input. Fig. 2 shows an example. Each skeleton takes the format of a
247
      unit test function with two placeholders — @ANSWER@ for answer and @ASSERT@ for assertion. Given
248   a unit test skeleton, GPT-4 generates the answer and assertion statement(s). The details of answer
249   and assertion generation can be found in Appendix B.2.
250
251
      Step 3: Generate an updated function f ← u We now prompt GPT4 to generate the source code
252   for the updated function f ← u given the function f and update specification u. We prompt using
253   the original function implementation (e.g. original argsort) to implement the new version. This
254   typically involves an implementation that wraps the original version of the function; for instance,
255   if a new boolean flag is added, call the function normally in one case and otherwise call it with a
256   transformed input or output.
257
      We validate the generated function with unit tests from the previous step. Specifically, we accept
258   the updated function if (1) it passes 70% of unit tests and (2) it passes more unit tests than the
259   original implementation.2 To improve the coverage, we sample up to three implementations if earlier
260   implementation does not satisfy two criteria above. After this process, on average, around 41% of
261   update specifications are paired with an updated function implementation. The rest are discarded.
262
263   Step 4: Filtering and deduplication Lastly, to verify the quality of generated data, the authors
264   of this paper manually examine the update specifications and filter duplicates and trivial update
265   specifications (e.g., change the return type from list to tuple). This process removes roughly
266   53% of examples on average, and the filtering percentage differs per package. We also filter update
267   specifications for which we could not generate at least 3 valid program synthesis examples (37% of
268   update specifications), as described in the next section.

      4.2   Arena (P ROGRAM S YNTHESIS E XAMPLES ) G ENERATION
282
283   Having generated update semantics u and the updated function implementation f ← u, we now gen-
284   erate program synthesis (PS) examples; see bottom half of Figure 2 and more details in Appendix B.6.
285
286   Step 1: Problem specification Given the update rationale generated as a part of update specification
287   u, GPT4 generates: (1) a scenario si that a problem is situated in; (2) the problem specification pi
288   that a solution function is mean to fulfill; and (3) the solution’s function signature, according to the
289   problem specification. See an example at Appendix A.2.
290
291   Step 2: Unit tests We then generate a set of unit tests meant to test that the solution to the program
292   synthesis example is correct. Note that these do not necessarily depend on the update, but only on the
293   specification of the problem from Step 1; they do not test whether the function is used.
294   We allow GPT-4 to include updated function in its generation, in contrast with update generation,
295   where GPT-4 could only call the old function through old_[function name]. Other than the
296   difference above, the generation process is identical to Step 2 in update generation.
297
298   Step 3: Reference Solution The prompt instructs GPT-4 to solve the problem by using the new
299   function as part of its solution. This helps to ensure that there exists a solution that uses the updated
300   function. We define a threshold δ = 0.6 of a fraction of tests that the implementation must pass
301   in order to be included in the benchmark. We found this quality bar to be high enough given the
302
      presence of bad tests, which we discard next.
303
304
      Step 4: Filtering and Deduplication Finally, we implement several filters of low-quality examples.
      First, we discard unit tests that generated solution doesn’t pass, as well as unit tests checking for
305
      exceptions (try/catch behavior). Our inspection of these cases showed that failed unit tests are
306
      almost invariably incorrect while the generated reference solutions are correct. Second, for each
307   update, we remove program synthesis cases for which reference solutions are too similar, to avoid
308   GPT-4 generating essentially similar solutions. Example of duplicate reference function in Figure 9.
309   See more detailed description at Appendix B.3
310
311   Step 5: Literalize answers in unit tests During generation, many unit tests initially rely on calling
312   updated APIs themselves to produce the correct answer to the test programmatically. However, this
313   causes unintended failures in the unit tests when running the old API updates in the environment,
314   leading to false positives for UPass@k even when the synthesis code does not use the new function. We
315   “literalize” the unit tests to remove these usages of the API; we provide more details in Appendix B.4.

410
      Non-applicability of existing knowledge editing methods Although a number of methods for
411
      knowledge editing have been proposed, not all of them are applicable to our setting. A line of methods
412   including ROME, MEMIT, and REMEDI (Meng et al., 2022; 2023; Hernandez et al., 2023) assume
413   the injected data follows a strict knowledge triplet format of (subject, relation, object);
414   this triplet structure is required for localization. Applying those methods to CodeUpdateArena is
415   not straightforward, as code entities do not exhibit these knowledge graph-like relations. Other
416   methods designed for similar settings do not assume this structure. However, even these more flexible
417   approaches like MEND (Mitchell et al., 2022) and others (Hartvigsen et al., 2023; Huang et al.,
418   2023) are optimized for models regurgitating the right short phrase response, typically less than 10
419   tokens. Our reference solutions contain 175.3 tokens on average. Furthermore, these methods have
420
      not proven effective in more related natural language settings such as Onoe et al. (2023).

459   6.3      A BLATION S TUDY
460
461   Our method FT (PS) serves as a starting point for future editing methods on our dataset to build on. In
462   this section, we study its design choices and demonstrate the difficulties and the trade-offs to achieve
463   efficacy and specificity at the same time. For efficiency, we follow the same evaluation procedure as
464
      Table 4 and but only use 1 random program synthesis example per update (in total, 161) to calculate
      efficacy. We focus on the DS-Coder-v1, which achieves the best specificity and overall performance,
465
      and DS-Coder-v1.5, which achieves the highest efficacy in finetuning experiments.
466
467
      Impact of training data for FT (PS) In our main experiment, the training set consists of c = 2
468
      copies of Nu examples from target update and Nr = 2 examples from r = 1 random updates.3
469   In this section, we investigated how the construct of training data affects knowledge injection, by
470   changing the values of c and r while fixing c + r.
471
472
      Having the target update is important: when we train the model on only program synthesis examples
      from random updates, Tables 5 and 11 show little or negative gain from learning only the task format.
473
      However, the random synthesis examples also matter: when we exclude them, the performance
474
      decreases as well, although to a less extent (see Table 11).
475
476   In a more complete hyperparameter sweep, we found that repeating the examples from target update
477   twice (c = 2) is generally the optimal hyperparameter, beyond which we observe diminishing gains
478
      in efficacy and drops in specificity. Second, although different models have different optimal values,
      we found that larger number of random updates r will continue to decrease models’ performances for
479
      efficacy and specificity. This is a different observation from prior work (Gangadhar & Stratos, 2024).
480
      See more details in Appendix E.4.

519
520
521
      In this paper, we presented CodeUpdateArena, a benchmark of API updates and corresponding
522   program synthesis examples. We demonstrated that our approach to synthesizing these leads to
523   high-quality examples. Across three LLMs, we conduct experiments for two simple baselines. One of
524   the baselines greatly outperforms prepending update information in context, which is different from
525   observation from knowledge editing in entity-driven scenarios. We further conducted a comprehensive
526   ablation study to inform future exploration. We hope our initial exploration could spur future work to
527   develop new knowledge updating methods for code LLMs to benchmark on this setting.
528   Limitations: One limitation of CodeUpdateArena is that certain APIs are difficult to test with our
529   dataset synthesis framework. For instance, it is difficult to generate unit tests for machine learning
530   APIs, and can be very involved to generate tests if a significant setup is needed (e.g., a mock
531   web server backend). Furthermore, our focus on synthetic API updates is necessary to avoid data
532   contamination, but at the same time decreases the realism of our dataset. It would be ideal to have real
533   software engineers annotate these kinds of updates at scale, but in preliminary experiments, we found
534   it very difficult to come up with creative and realistic updates. Finally, our examples are restricted to
535   Python and English-language descriptions; we believe a multilingual version of the benchmark (both
536
      human languages and code languages) would be useful.
537   Reproducibility Statement: Our dataset and code are available at ©. The dataset construction
538   procedure is fully automated and documented, enabling future researchers to generate similar or
539   related datasets. For our experiments, we detailed our hyperparameters in Appendix E. Our ablation
      study (Section 6.3 and Appendix E.4) depicts the model’s behavior across different design choices.

      A.1    U PDATE TAXONOMY
813
814   To systematically capture different types of updates, we first create a taxonomy for update types,
815   rooted in updates to functions.
816
      Recall that we define f ← u to be the update made to an existing function f when providing it with
817
      new semantics u. We assume f always takes the form of Function([argument1, argument2, · · · ]) →
818
      Output. We view u as consisting of three independent components: (1) the Action that the update is
819   applying to the API function (e.g., deprecate); (2) the Locus that the action happens at (e.g. argument
820   or output); (3) and the Aspect that the action is applying at some place for (e.g., name and data type).
821   See Table 6 in the appendix for possible values of each component. We note that we do not focus on
822   Action=deprecate in this work, as techniques for knowledge unlearning are different than those for
823   knowledge editing.
824
      An update type is a tuple with values for each component listed in Table 6. For example, the
825   add-argument-NULL update type means the update is adding a completely new argument to the
826   existing arguments of a function, and modify-argument-name update type means the update is
827   modifying the name of an existing argument (i.e. renaming). We note that not all combination
828   makes sense, e.g. modify-output-name; or some update types might overlap with another e.g.,
829   add-function-semantics overlaps with modify-function-semantics. We remove those and
830   obtain 17 update types.
831
832                                    Table 6: Update Taxonomy components
833
834                            Component        Values
835
836                            Action           {add, modify, deprecate}
837                            Locus            {function, argument, output}
838                            Aspect           {NULL, name, data_type,
839                                             default_value, supported_value}

       E.3   E VALUATION PROCEDURE FOR FINETUNING EXPERIMENTS
1964
1965   Recall that our benchmark CodeUpdateArena is structured by pairing each executable API update
1966   with n program synthesis examples. We treat the n program synthesis examples as an ordered list.
1967   The training sets of FT (U) and FT (PS) slightly differ and we will describe them separately.
1968
1969   FT (U) The training set only consists of a single copy of the API update information following the
1970   template in Prompt E.4.
1971
1972   FT (PS) As mentioned in Section 6.1, the single-edit training set contains c copies of Nu = 2
1973   unique program synthesis examples and Nr examples from r updates from the rest of the benchmark.
1974   Since the number of program synthesis examples per update could be as few as 3, we adopt a
1975   cross-validation scheme to evaluate a model for each update. To give a concrete example: when
1976   testing the model on program synthesis example i ∈ [0, n), we take the “previous” Nu examples —
1977   example (i − 1) mod n and example (i − 2) mod n; we then repeat them c times to obtain the
1978
       final set of examples for target update. Then, we take two unique program synthesis examples from
       each of the r random updates; and combine them with examples from the previous step to obtain the
1979
       final training set. Each training instance is formatted with Prompt E.5.
