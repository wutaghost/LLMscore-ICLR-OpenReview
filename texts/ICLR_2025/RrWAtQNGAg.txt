012
013            Code large language models (LLMs) have shown remarkable advances in code
014            understanding and generation tasks. Programming corpora serve as the founda-
015
               tion for various code LLMs. In reality, repositories consist of multiple files with
               numerous cross-file dependencies. Leveraging the dependency information can
016
               effectively enhance the code understanding and generation capabilities. However,
017
               existing works fail to utilize dependencies effectively. Consequently, there is a
018            pressing need for an open dataset that specifically focuses on capturing and lever-
019            aging the cross-file dependencies. To fill in this gap, we release C ODE C HAIN, an
020            augmentation of the code dataset at the repository level, provides a rich context
021            for code LLMs to learn from. Specifically, to capture the cross-file dependen-
022            cies, we first parse the code project into a topological graph where nodes rep-
023            resent files and edges denote dependencies. Then, we employ a novel random
024            walk method to determine the code chain and concatenate the corresponding files.
025            To utilize such corpus for supervised fine-tuning, we design C HAIN -I NSTRUCT to
026
               enable the model to thoroughly learn the code contents and its dependencies. Ulti-
               mately, we produce 562,587 code chains and 1,021,550 instruction samples. With
027
               C ODE C HAIN, we train our model on multi-task learning objectives and evaluate
028
               on the public benchmarks. The experimental results demonstrate that model by
029            learning the interconnected nature of codes significantly outperforms the previ-
030            ous methods, showcasing the effectiveness of C ODE C HAIN in advancing the code
031            understanding and generation.

034   1   I NTRODUCTION
035
036   Code large language models (LLMs) have shown remarkable advances in code understanding, com-
037   pletion, and generation tasks. Code corpora, as the foundation of large code models Li et al. (2023);
038   Guo et al. (2024); Zheng et al. (2023); Rozière et al. (2023); Luo et al. (2023), are from diverse
039
      sources: open-source repositories, platforms, forums, and so on. Models pre-trained and finetuned
      with such data exhibit strong code understanding and analysis capabilities. For repository-level
040
      code corpora, previous work decide to directly concatenate code files, which ignores the depen-
041
      dency information between files within the project. To utilize dependency information, DeepSeek-
042   Coder Guo et al. (2024) adopts a topological sorting method, which orders files based on the number
043   of its dependencies of each file. Nonetheless, only considering the the number of dependencies can
044   not reflect the true dependency order between files. Moreover, it struggles to determine the correct
045   order when multiple files have identical dependency counts, leading to errors. More comparison
046   analysis are in Section 4.1.
047
      To fill in this gap, we design a simple yet effective random walk method based on file dependencies
048   to determine the code chain. Specifically, we adopt the approach used by DeepSeek-Coder Guo
049   et al. (2024) to convert code files into a topological graph. In this graph, nodes represent files, and
050   edges symbolize the dependencies between them. The random walk begins by selecting a random
051   starting node on the graph. From there, subsequent nodes are chosen based on their dependency
052   relationships, continuing until the cumulative in-degree of the selected nodes surpasses a predefined
053   threshold. Our method not only captures the correct order of files when multiple nodes have the
      same in-degree but also maps out indirect call relationships between files.

136   3.2   D EPENDENCY G RAPH G ENERATION
137
138
      After downloading repositories from GitHub, our next step is to get the dependency relationships for
      each repository. In order to better construct the code-chains from the dependency relationships, we
139
      save the dependency relationships in graphs which we call dependency graphs. Each dependency
140
      graph is a directed graph that can describe the calling relationships between files within a repository.
141   In this graph, each node represents a code file and it points to the code files that import it. To
142   generate the dependency graphs, we read every code file in the same repository and consider various
143   expressions in code for importing modules to extract the dependency relationships. The expressions
144   included are as follow:
145
146          • Basic import(and rename): Use the ”import” keyword followed by the module name or
147            ”as” keyword to give it an alias, like ”import xx”,”import xx as xxx”.
148          • Import specific content from a module(and rename): Use the ”from” keyword followed
149            by the module name and ”import” keyword (and ”as” keyword), such as ”from a import b
150            as c”.
151          • Import multiple functions: use a comma-separated list within the import statement for
152            importing multiple functions from a module or package,like ”from mymodule import func-
153            tion1, function2”.
154          • Absolute references: An absolute reference specifies the complete path to a resource from
155            the root directory. For instance, ”from mypackage.mymodule”,”import myfunction”.
156
             • Relative references: A relative reference specifies a path starting from the current location
157            in the directory structure. For example, ”from . import sibling module”.
158
159   Our dependency parser can analyze these expressions to identify the imported modules. It then
160   searches to ascertain whether the referenced modules correspond to files within the same repository.
161   This process starts from the root directory of the current program and adheres to code’s import
      mechanism. The process of our dependency parser is written as the HasDependency() function in

207   We next generate code-chains from the dependency graphs. If we select a path on a dependency
208   graph, the path could be seen as a sequential chain representing a subset of the file dependency
209   relationships within a repository, where each node is imported by the next node it points to. We term
210   a path like this a code-chain. The Algorithm 1 describes the process of code-chains generation on
211   the dependency graphs in one repository. The generation process consists of two steps: In-degree
212
      calculation and Random Walk.
213   To compute the in-degree for each node, the algorithm initializes with the creation of two data
214   structures: an empty adjacency list dictionary named ”graphs” to map dependencies between files,
215   and an empty dictionary named ”in degree” to keep track of the number of dependencies each file
      has. The algorithm proceeds by iterating over each file to establish dependencies between pairs of

      files. If ”fileA” depends on ”fileB”, ”fileA” is added to the adjacency list of ”fileB” in ”graphs”,
217   which means creating an directed edge from ”fileB” to ”fileA”. Concurrently, the in-degree of
218   ”fileA” is incremented in the ”in degree” dictionary. Then, we set up a threshold to limit the number
219   of chains generated by restricting the cumulative sum of in-degrees of all chain nodes across the
220   entire repository. We build the code-chains from the dependency graphs based on their degrees and
221   through a way of random selections. Specifically, The process begins with the random selection
222   of a node within the graph, which serves as the starting point for initializing a chain. Then, this
223   process continues by randomly selecting the next node from among those that are dependent on the
224   current one, until no further nodes are available. If the cumulative degree stays below a predefined
225
      threshold, the newly formed chain is added to a list of selected chains. This process repeats until the
      total degree of all chains exceeds the threshold. When we gain all the code chains, we remove the
226
      duplicate ones.

230
231   After generating the code chains, each code file within the same chain is concatenated to form a
232   training sample. A comment indicating the chain between files is added at the beginning of each

234   Section 3.1, we exclude any repository that does not encompass all its files in the chains or lacks
235   dependency relationships to maintain file integrity. Furthermore, to prevent contamination from
236   public benchmarks like HumanEval, we adopt the same n-gram filtering process used by DeepSeek-
237   Coder Guo et al. (2024). Specifically, we remove any code segment from C ODE C HAIN that contains
238   a 10-gram sequence matching one found in the test data. For test data strings shorter than 10-grams
239
      but at least 3-grams long, we use precise matching to ensure exclusion. These quality screening
      measures reduce the dataset to 61.5% of its original size.

      4.1   C ODE C HAIN VS. D EEP S EEK -C ODER
295
296
297
      In DeepSeek-Coder Guo et al. (2024), the dependencies among files are analyzed on topological
      graphs, with the in-degree of a node indicating the number of dependencies a file has. The files are
298
      then sorted based on in-degrees as input for the model. However, this method does not accurately re-
299
      flect the actual dependencies in a repository. Figure 2 demonstrates specific errors that occur during
300   the sorting. DeepSeek-Coder’s inaccurate file sorting stems from two main issues: 1) The algo-
301   rithm produces incorrect ordering when multiple nodes share the same in-degree. In Case 1, within
302   a branched structure, it is obvious that two files at a fork do not have any dependency. However,
303   through topological sorting, each file is forcibly assigned an order. Moreover, when encountering
304   multiple nodes with identical minimum in-degrees, the algorithm assigns their order randomly. This
305   leads to an inaccurate file sequencing based on the order of invocation, overlooking the indirect
306   invocation relationships among the files, as evidenced by nodes d and f in case 2.
307   2) The in-degrees might not accurately reflect the actual file dependencies in the invocation chain.
308   Placing the node with the highest in-degree at the end would lead to incorrect ordering. For exam-
309   ple, node c in case 2 has the highest in-degree yet is located in the middle of the chain. To address
310   this, we adopted a random walk strategy to establish an invocation chain based on file dependencies.
311   The method starts by randomly choosing an initial node and then continues by randomly selecting
312   subsequent nodes that are dependent on the current node. This continues until no further nodes are
313   available. The whole iteration ends until the cumulative in-degree of all nodes exceeds a predeter-
314
      mined threshold. This approach ensures that the sequence accurately mirrors the true invocation
      order and maintains the time complexity at O(n). The proof of the complexity can be seen in
315
      Appendix E

716   code pre-training dataset at the repository level. This richer context can lead to improved model per-

      formance and understanding of code semantics. We release a comprehensive data curation pipeline
      for anyone to use to get more different code language repositories. Our method ensures the genera-

      tion of dependencies in the true file order, which is crucial for handling code repositories. This could

723

      authorization or consent. Like any large-scale dataset sourced from online repositories, CodeChain
724
      may inadvertently capture and perpetuate biases present in the original data sources. This could

      lead to biased model predictions and reinforce existing societal inequalities in code development

              • n: The total number of files in the repository.

809           • degree(x): The degree or in-degree of item x (which represents its connections or depen-
                dencies).

      E.1    O UTER L OOP
811
812   The outer loop runs while the condition total degree < threshold is satisfied. Initially, the degree
813   is set to 0. A random item is chosen from the file list, denoted by:
814
                                       Initial item = RandomChoice(files)
815
816   This selected item is added to a new list:
817                                     newlist ← newlist ∪ {Initial item}
818
      The degree is updated by adding the in-degree of the selected item:
819
820                                 degree ← degree + inDegree(Initial item)
821   Each of these operations—random selection, list update, and degree update—has a constant time
822   complexity of O(1).
823
824   E.2    I NNER L OOP
825
826   When a random item exists (i.e., random item ̸= ∅), the inner loop is executed. A new random
827   item is selected from the graph:
828                          random item = RandomChoice(graphs[random item])
829
      This item is then added to the list:
830
831                                    newlist ← newlist ∪ {random item}
832   The degree is updated once again by adding the in-degree of the newly selected item:
833
                                   degree ← degree + inDegree(random item)
834
835   Before appending the updated list to the set of selected chains, the algorithm ensures that this list is
836   not already present:
837                   if newlist ∈
                                 / selected chains, then append newlist to selected chains
838
      Finally, the total degree is incremented by the newly computed degree:
839
840                                   total degree ← total degree + degree
841
842   E.3    T IME C OMPLEXITY A NALYSIS
843
844
      The operations in the inner loop, like random selection, list append, and degree updates, all have
      constant time complexity O(1). The only exception is the check to see whether newlist ∈          /
845
      selected chains, which, in the worst case, has a time complexity of O(n), where n is the num-
846
      ber of files, since it involves searching the list.
847
848   The outer while loop runs while total degree < threshold. Since the total degree accumulates
849   based on the file selections, the number of iterations of the outer loop is proportional to n. Therefore,
850
      the overall time complexity can be expressed as:
851                                                           k
                                                              X
852                                                T (n) =        c
853                                                           1

854   where k represents the number of iterations of the outer loop, and c is the constant time per iteration.
855   Given that k is proportional to n, the total time complexity simplifies to:
856                                          T (n) = O(n · c) = O(n)
857
      Thus, the algorithm has a time complexity of O(n).
