009                                                                 A BSTRACT
010
011                 One of the most reliable ways to create deployable models for specialized tasks is
012                 to obtain an adequate amount of high-quality task-specific data. However, for
013                 specialized tasks, often such datasets do not exist. Existing methods address
014                 this by creating such data from large language models (LLMs) and then distill-
015                 ing such knowledge into smaller models. However, these methods are limited
016                 by the quality of the LLMs output, and tend to generate repetitive or incorrect
017
                    data. In this work, we present Retrieval Based Distillation (ReBase), a method
                    that first retrieves data from rich online sources and then transforms them into
018
                    domain-specific data. This method greatly enhances data diversity. Moreover, Re-
019
                    Base generates Chain-of-Thought reasoning and distills the reasoning capacity of
020                 LLMs. We test our method on 4 benchmarks and results show that our method
021                 significantly improves performance by up to 7.8% on SQuAD, 1.37% on MNLI,
022                 and 1.94% on BigBench-Hard. 1

027   How can we effectively obtain high-quality models for specific tasks? Large Language Models

029

      rameters, making them computationally expensive to use in practice, and in many cases these models
031
      underperform smaller models fine-tuned on task-specific data (Mosbach et al., 2023; Viswanathan
032   et al., 2023b; Bertsch et al., 2024).

072   these challenges, we propose Retrieval Based Distillation (ReBase). As shown in Figure 1, ReBase
073   is a framework that first retrieves data from an abundant and reliable labeled data source, then trans-
074   forms them into the content and format necessary for the user’s task. This data is then used to train
075   a domain-specific model. Initially, ReBase scrapes online data and encodes them into a large datas-
076   tore; Then, ReBase uses the user’s instruction and the user’s provided examples to retrieve the most
077   relevant items from the large datastore. Finally, using an LLM, ReBase transforms the retrieved
078   data point into a data that contains a query and an answer field for the specific task, this includes
079   transforming the content and transforming the format. Different from previous methods, ReBase
080

081

082
      2022) where the LLM transforms the output into a step-by-step reasoning. This enables the small
083   model to be trained on the reasoning generation by the large model, which is especially useful for
084   reasoning tasks (Suzgun et al., 2022).

088
      for 1.94%, on SQuAD for 7.8%, and on MNLI for 1.37% over previous methods. Our method
089   suggests the benefit of using data retrieved from multiple sources to train a specific model.

      2     P ROBLEM F ORMULATION
093
094
095   We formulate the problem as follows: Input: The input contains an instruction of a task and few-
096   shot examples. Output: The output contains a new dataset with the field (query, answer) that could
097   be used to directly finetune a model. It also contains a task-expert model trained for this task.
098
      Objective: Our high-level objective is to generate a high-quality dataset that effectively boosts a
      model’s performance on this task. Specifically, we assume that we have access to the abundant
099
      existing datasets online and access to LLMs. Our goal is to effectively harness the ability of LLMs
100
      and use the rich content of the existing datasets to create a high-quality dataset for the new task.
101   Then use this dataset to train a task-expert model.

      3.1   DATASTORE C ONSTRUCTION
109
110   Our datastore construction process begins with collecting datasets from Hugging Face Datasets
111   (Lhoest et al., 2021), which consists of over 75,000 datasets. A Hugging Face dataset contains a
112   dataset description that describes the purpose of the dataset. It also contains multiple rows entries
113   and columns. Each row represents a data entry, and each column represents a specific attribute of
114   that data entry. (eg. row id, content, source url, label)
115   For each row in these datasets, we do not directly encode the entire row entry because some attributes
116   are redundant and may introduce noise (eg. attributes such as row id or url are often not useful.)
117   Instead, we encode each column separately. Specifically, for the jth row entry in dataset i, we iterate
118   through each column c in the row entry and encode it into a vector:
119
                                      vi,j,c = Encode (column value) .
120
121   This vector has a unique identifier in the format:
122
                                     {dataset id, row num, col name}
123
124   We then add the key-value pair ((i, j, c), vi,j,c ) to the datastore. Additionally, for each dataset i, we
125   encode its corresponding dataset description:
126                               vi = Encode (dataset description) .
127
128   This value is identified by the dataset id i. We put the key-value pair ((i), vi ) into the datastore.

187   3.3           DATA T RANSFORMATION
188
189   After retrieving the relevant data, we employ a large language model (LLM) to transform the data
190   into a format and content suitable for the specific task. This transformation process includes the
191   following steps: 1. Salient Field Classification: The LLM identifies the relevant fields in each
192   retrieved row based on the domain-specific requirements. 2. Content Adaptation: The LLM
193   transforms the content to align with the target domain, ensuring it meets the specific needs of the
194   task. 3. Chain-of-Thought (CoT) Generation: For reasoning-intensive tasks, the LLM generates
195   outputs using CoT, providing detailed step-by-step reasoning to enhance the quality and accuracy of
196
      the transformed data.
197   In our experiments, we use Claude 3 Haiku (Anthropic., 2024) as the LLM underlying the dataset
198   transformer due to its competitive performance / cost tradeoff. The detailed prompt used to instruct
199   the LLM is provided in the Appendix B. For tasks that require complex reasoning, such as the
200   BIG-Bench Hard tasks, previous works have shown that Chain-of-Thought (CoT) (Wei et al., 2022)
201   reasoning can greatly improve the model’s performance on reasoning tasks (Suzgun et al., 2022) and
202
      finetuning on CoT data can further boost the reasoning ability (Chung et al., 2024) and can distill
      the reasoning capacity in LLMs to smaller models (Ho et al., 2022). Therefore, we leverage Chain-
203
      of-Thought generation. For these tasks, we prompt the LLM to generate a CoT reasoning followed
204
      by the final for the answer part instead of directly generating the final answer. The generated CoT
205   data is then used for further training to improve the downstream model’s performance as well. We
206   demonstrate the transformation process in Figure 3.
207
208
      Our transformation approach ensures that the transformed data is tailored to the new task in terms
      of both content and format and can be directly used for further finetuning. This process also in-
209
      corporates the reasoning process of LLMs and distills such reasoning capacities to the task expert
210
      model.

      Datasets The datasets we used in this work include: (i) MultiNLI (MNLI) (Williams et al.,
      2018) tests the model’s ability to recognize textual entailment between two sentences. It is one
232
      of the largest corpora for natural language inference, containing 433k samples across 10 distinct
233
      domains. We chose this task to test the method’s performance on traditional language understanding.
234   (ii) SQuAD (Rajpurkar et al., 2016) is a reading comprehension dataset that contains questions and
235   context based on Wikipedia articles. We choose this task as another standard language understanding
236   task. (iii) MCoNaLa (Wang et al., 2023b) is a multilingual benchmark to test models’ ability to
237   generate code from multi-lingual natural language intents. We focus on the Japanese-to-Python
238   subtask, as it is a challenging task with no task-specific annotated data available. (IV) BIG-Bench
239   Hard (BBH) (Suzgun et al., 2022) is a challenging reasoning benchmark. It is a subset of BIG-
240   Bench (BIG-bench Authors, 2023) containing challenging tasks where LLMs underperform humans.
241   This dataset tests whether ReBase can generate data for highly challenging reasoning tasks.
242
243   Baselines (1) Prompt2Model (Viswanathan et al., 2023a) This method retrieves a model from
244   Hugging Face via the task instruction, then finetunes this model using both synthesized and retrieved
245   datasets (without transforming the latter). (2) Synthesized Data We use the dataset generation
246   method described by Prompt2Model (Viswanathan et al., 2023a) to obtain synthesized data and use
247
      it to finetune a LLM. This generation process uses dynamic temperature and prompt sampling to
      increase the synthesized data’s diversity and demonstrates impressive data synthesize ability. (3)
248
      ZeroGen This method uses pretrained LLMs to directly generate datasets under zero-shot setting.
249
      This method is targeted for classification tasks. We use Claude-Haiku to generate the data. (4) Few-
250   Shot Prompting For this, we directly prompt the pretrained LLM with few-shot examples without
251   any finetuning. We report Claude Haiku which is used as our dataset generator and transformer. We
252   also report GPT-4 as a strong upper bound model.
253
254
      Implementation Details We use a pretrained model2 from the Sentence Transformers
255   toolkit (Reimers & Gurevych, 2019) to encode all data in the datastore construction phase. We
256   use 3K examples for MNLI and SQuAD and 1K for MCoNaLa and each BBH task. We use Claude
257   3 Haiku model to transform the data. To more accurately simulate the case in which we are tack-
258   ling a new task without training data, we prevent the retriever from retrieving any data from the
259   target task’s original training set. For model training, we choose the most recent open-source LLM
260   Llama3-8B (AI@Meta, 2024) as the base model for both the synthesized method and ReBase. We
261   train the model using QLoRA (Dettmers et al., 2023) which requires only one NVIDIA A6000 48GB
262   GPU. We provide training details in Appendix D
263
264   Metrics We report ChrF++ (Popović, 2015) score for MCoNaLa, this metric was similarly used
265   for evaluation by (Viswanathan et al., 2023a). For MNLI, we report the accuracy. For SQuAD,
266   we use same exact match metric and F1 metric in (Rajpurkar et al., 2016). For BBH, we use the
267   evaluation script from (Yue et al., 2023) to first extract the answer in the generated sentence and then
268   report the accuracy.

      Retrieval-Augmented Generation (RAG) Retrieval-Augmented Generation (RAG) (Lewis et al.,
516   2020; Gao et al., 2024; Asai et al., 2023; Chen et al., 2017) retrieves from external knowledge to
517   help the LLM answer open-domain questions. Recent works demonstrate that RAG can greatly
518   boost the reasoning ability of LLMs (Jiang et al., 2023; Shao et al., 2023). IAG (Zhang et al.,
519   2023) leverages both retrieved knowledge and inductive knowledge derived from LLMs to answer
520   open-domain questions. Inspired by the success of RAG, we study how retrieving from external
521   knowledge improves dataset quality and further improves model performance.
522
523
      Data Synthesis Recent studies use LLMs as dataset generators (Patel et al., 2024; Song et al.,
524
      2024) and focus on improving the generated data’s quality. Zerogen (Ye et al., 2022b) uses pretrained
525   LLMs to generate datasets directly under zero-shot setting. Progen (Ye et al., 2022a), Sungen (Gao
526   et al., 2023), and Impossible Distillation (Jung et al., 2024) uses feedback from smaller models to
527   distill the generated data. AttrPrompt (Yu et al., 2023a) improves data quality by improving the
528   prompt. Unnatural Instructions (Honovich et al., 2022), ReGen (Yu et al., 2023b), and S3 (Wang
529   et al., 2023a) improves the data quality by using other datasets as reference. We explores the use of
530   both RAG and LLM’s generation ability to create a diverse and reliable dataset for specific tasks.
531
532
533   6   C ONCLUSION
534
535   In this paper, we present ReBase, a framework that uses retrieval and transformation to create diverse
536   and high-quality domain-specific dataset to train task-expert models. Our method shows significant
537   improvement over conventional dataset generation methods. We establish the benefit of leveraging
538   examples retrieved from a large, heterogenous datastore to create task-specific training data. We
539   believe this work motivates future work on retrieving labeled examples from a prompt; improved
      example retrieval could lead to significantly improved retrieval-based distillation.

      C.2    A NALYSIS
835
836   We observed that most tasks maintain a high percentage of data after filtering. Most tasks retain
837   over 80% or even 90% of the original data. This suggests that ReBase transformed data is generally
838   plausible and usable for downstream finetuning and the filtering process does not substantially re-
839   duce the dataset size. However, there are some exceptions. For date understanding, formal fallacies
840   , sports understanding, dyck languages, navigate, and web of lies, the percentage of the remaining
841   data drops below 50% or even under 20%.
842   We observed that filtering can be beneficial in certain cases but not always. When the filter-
843   ing removes a large amount of data, performance tends to decline. For instance, tasks such as
844   date understanding, formal fallacies, dyck languages, and navigate decline after filtering. How-
845   ever, sports understanding shows improvement in performance after filtering nearly 50% of the
846   data.
