009                                              A BSTRACT
010
011
               Large language models (LLMs) have achieved impressive performance in code
012            generation. Despite the remarkable success, we observed that LLMs often misun-
013            derstand or overlook some problem-specific undertrained keywords during code
014            generation, compromising the accuracy of the generated code. After explicitly
015            explaining these undertrained keywords using well-trained terms in the prompt,
016            LLMs are more likely to generate correct code implementation. Inspired by
017            this observation, we propose a novel technique named SEK (Self-Explained
018            Keywords), which empowers an LLM for better code generation by extracting and
019            explaining the key terms in the problem description with the LLM itself. Compre-
020
               hensive experiments across three benchmarks, i.e., HumanEval(+), MBPP(+), and
               APPS, with five representative LLMs, show that SEK can significantly improve
021
               LLMs in code generation, yielding substantial and consistent gains. For instance,
022
               SEK improves the Pass@1 of DeepSeek-Coder-V2-Instruct from 85.4% to 93.3%
023            on the Humaneval benchmark. Further analysis confirms that SEK enables the
024            LLMs to shift their attention from low-frequency keywords to their corresponding
025            high-frequency counterparts.

      1   I NTRODUCTION
030
031   Code generation aims to generate a code snippet that meets the intent described in natural language.
032   This process can potentially reduce the costs of software development (Xu et al., 2022; Yin & Neu-
033
      big, 2017; Vaithilingam et al., 2022). Recently, the notable success of LLMs such as ChatGPT (Ope-
      nAI, 2022) and Llama-3 (AI@Meta, 2024) has substantially enhanced the state-of-the-art in code
034
      generation. These LLMs demonstrate remarkable proficiency in comprehending natural language
035
      descriptions and translating them into code snippets.
036
037   Despite the remarkable success, we found that LLMs often struggle to translate certain terms in the
038   problem description into corresponding code. When these terms are critical in the programming
039
      context (i.e., serving as a keyword), this limitation can compromise the accuracy of the generated
      code. An example is presented in Figure 1, where the coding problem requires returning even digits
040
      within a given range in ascending order. We found that LLMs fail to recognize that this term refers
041
      to the even numbers between 0 and 9, leading to the omission of this constraint in the generated con-
042   ditional statements. One possible reason for this observation is the long-tail distribution of coding
043   training datasets (Chen et al., 2024d; Zhong et al., 2024b), where some programming terms are rare
044   and undertrained and thus cannot be effectively translated into the corresponding code by the LLM.
045   If we explicitly convert even digits into well-trained terms by explaining it and prompt the LLM to
046   focus on it, the LLM can produce a correct implementation.
047
      Inspired by this example, we hypothesize that we can boost LLMs for code generation by explicitly
048   identifying and explaining certain keywords. However, this is non-trivial and usually requires man-
049   ual efforts. Our key idea is that such keywords can be identified and explained by LLM themselves.
050   This idea is supported by three observations: (1) prior studies show that LLMs can effectively iden-
051   tify task-specific key items (Fang et al., 2024; Fan et al., 2024); (2) our experiments indicate that
052   such LLM-selected keywords are often terms that are more likely to be undertrained, i.e., have a
053   relatively low frequency in the code training set (detailed in Appendix E.5); and (3) although the
      direct mapping from these keywords to code may be undertrained, the semantics of these keywords

079   are typically understandable by LLMs after pre-training on large-scale general corpora. This enables
080   LLMs to describe and explain these keywords using natural language.
081   Based on this idea, this work proposes Self-Explained Keyword (SEK), a novel technique leveraging
082   LLMs’ own comprehension capabilities to automatically identify and explain these problem-specific
083   keywords to enhance their understanding of coding problems. SEK employs a carefully designed
084   prompt with a few examples, directing LLMs to focus on crucial keywords in the problem descrip-
085   tion. We use a frequency-based ranking algorithm to sort these keywords and further prioritize
086   low-frequency keywords, which are then appended to the original problem description to construct
087
      an augmented prompt. Overall, this approach aligns with the working process of pragmatic devel-
      opers, which use auxiliary tools like blackboards to highlight, explain, and rank important parts of
088
      requirements (Andrew Hunt, 2000).
089
090   SEK enhances LLMs’ problem-solving capabilities in a novel way, distinguishing itself from pre-
091   vious methods in prompt engineering for code generation. As shown in Figure 2, unlike previ-
092   ous approaches that often rely on introducing external knowledge, such as human feedback (Chen
093
      et al., 2023a; Wu et al., 2024; Dubois et al., 2024) or the execution results of LLM-generated solu-
      tions (Zhong et al., 2024b; Chen et al., 2023c; Zhong et al., 2024a), into the input, SEK operates by
094
      distilling additional content from the problem description using the LLM itself. Chain of Thought
095
      (CoT) (Wei et al., 2022), which also utilizes LLMs’ inherent knowledge for problem-solving, bears
096   the closest resemblance to SEK. However, the fundamental strategies of CoT and SEK are different:
097   CoT guides the LLM to think in a chain-like manner, while SEK directs the LLM to understand and
098   prioritize key concepts.
099
      We evaluate SEK with five representative LLMs, including three open-source models and two
100
      closed-source models, on three widely used code generation benchmarks. Experimental results
101
      demonstrate that SEK effectively enhances code generation performance. For example, SEK en-
102   ables Llama-3.1 to achieve a relative improvement of 8.8% averaged on the used benchmarks. No-
103   tably, DeepSeek-Coder-V2-Instruct with SEK significantly outperforms it with standard prompting,
104   achieving state-of-the-art performance on several benchmarks (e.g., HumanEval: 85.4% to 93.3%).
105   Furthermore, our ablation studies indicate that the carefully designed prompt and the ranking com-
106   ponent of SEK are effective. Additionally, our attention analysis reveals that SEK helps LLMs
107   comprehend low-frequency keywords by redirecting attention to their high-frequency counterparts.

      2      M ETHODOLOGY
140
141
142
      Code generation aims to generate a solution program based on a problem description. Typically,
143   a problem description includes implementation requirements, and several test cases to help further
144   understand the problem.
145
146
      Figure 3 illustrates the overview of SEK. SEK is designed to address the issue of LLMs overlooking
      low-frequency terms in the program description due to the long-tail distribution in their training
147
      data. To address it, one key is to leverage the LLM’s capabilities to identify and explain potentially
148
      overlooked keywords within the problem description. We employ a carefully crafted prompt with
149   a few-shot learning method to achieve this. After obtaining the keywords and their explanations,
150   another challenge is how to effectively integrate them with the original problem description. For
151   this purpose, we introduce a frequency-based ranking algorithm that prioritizes less frequent tokens,
152   which are more likely to be overlooked by the LLM. These ranked keywords are then appended to
153   the original problem description, serving to guide the LLM towards generating an accurate solution.
154   The process comprises three main steps:
155   KeyExtract & Explain (Section 2.1): Based on the problem description, SEK constructs a prompt
156   to guide the LLM to identify and explain keywords within the problem description.
157
158
      KeyRank (Section 2.2): SEK employs a frequency-based ranking algorithm to prioritize the ex-
      tracted keywords.
159
160   PromptEnrich (Section 2.3), SEK concatenates the ranked keywords and their explanations with
161   the original problem description to create an enriched problem description. This comprehensive
      formulation serves as the final input for the LLM to generate code solutions.

      2.1   K EY E XTRACT & E XPLAIN
163
164   In this step, SEK extracts and explains keywords from the given problem description. Our key in-
165   sight is that LLMs inherently possess strong understanding and reasoning abilities after training on
166   large-scale general corpora, enabling them to explain crucial concepts within a problem description.
167   The effectiveness of using LLMs for keyword extraction has also been demonstrated by recent stud-
168   ies (Maragheh et al., 2023; Lee et al., 2023). Inspired by this insight, SEK uses the LLM itself to
169   perform the task with a prompt-based approach.
170   Specifically, SEK begins by designing a prompt to instruct an LLM for keyword extraction and ex-
171   planation. The prompt is shown in Prompt for KeyExtract & Explain in Figure 3, which consists of
172   three parts. First, it provides the overall instruction for the task, namely the generation of keywords
173   and their corresponding explanations. Then, it specifies the format of input and output. Finally,
174   it provides detailed guidelines. Intuitively, terms associated with input, output, and supplementary
175   content (i.e., clarifications of keywords or specifications of value ranges) within the problem descrip-
176   tion are relatively important, as they contain the problem’s core elements, objectives, and constraints
177   (Guideline 1). For explanations, given the potential ambiguity in natural language expressions and
178
      the clarity of the public test cases, the generated explanations should be both precise and consistent
      with these test cases (Guideline 2,3). We also impose limitations on the keyword quantity to guar-
179
      antee that the LLM identifies and outputs only the important keywords in the problem description
180
      (Guideline 4). The LLM is prompted to identify at most three keywords and generate an explanation
181   for each identified keyword. Ultimately, to facilitate subsequent processing, we further emphasize
182   the output format (Guideline 5). Additionally, we use several examples to leverage LLMs’ in-context
183   learning ability to understand and solve this task.
184
185
186   2.2   K EY R ANK
187
188   After extracting and explaining the keywords, the next goal is to enhance the original prompt. Previ-
189
      ous research has demonstrated that LLMs are sensitive to the order of tokens in the prompt, known
      as position bias (Li et al., 2024; Yu et al., 2024). It highlights the need to carefully arrange the
190
      extracted keywords. Notably, pragmatic human developers tend to place more important keywords
191
      at the beginning in practice (Andrew Hunt, 2000). This preference may be reflected in the train-
192   ing dataset, leading LLMs to also focus more on the keywords written at the front. Therefore, we
193   propose a set of heuristic rules to rank keywords by importance, namely KeyRank. The specific
194   Algorithm is provided in the Appendix A.
195
      We first examine the keywords extracted by two LLMs (Llama 3.1 and DeepSeekCoder-V2) for part
196
      of the coding problems in the APPS training set. These keywords can generally be categorized into
197
      three types: (1) Function keywords, which match the desired function names, such as count nums
198   in Figure 3. (2) General keywords, which appear in the problem description, like sum of digits
199   in Figure 3. (3) Abstract keywords, which do not appear in any input; instead, they are abstract terms
200   summarized from multiple concepts. For example, for two different concepts “substring before the
201   dot” and “substring after the dot” in the problem description, LLM may combine them into a single
202   keyword substring before/after the dot. The proportions of these three categories are
203   22.5%, 59.9%, and 17.7%.
204   We hypothesize that abstract keywords are the most important, as they encompass explanations
205   across multiple concepts. General keywords refer to single concepts and are of secondary impor-
206   tance, while function Keywords, whose explanations have already appeared in the problem descrip-
207   tion, are the least important. Therefore, we propose ordering the keywords as abstract → general
208   → function. Appendix E.1 demonstrates that this heuristic combination order yields the best results.
209
      Moreover, since general keywords represent the majority (59.9%) and LLMs could extract multiple
210   general keywords for a single problem, we further perform an internal ranking of these general
211   keywords. We argue that a keyword is more important if it appears more frequently in the problem
212   description (i.e., higher term frequency). Conversely, if a keyword appears less frequently in a corpus
213   (i.e., lower document frequency), the corresponding code conversion could be more challenging as
214   we stated in the Introduction section, and thus its explanation is more significant. Therefore, we
215   use the TF-IDF, a widely used metric that combines term frequency (TF) and inverse document
      frequency (IDF), to assess the importance of general keywords. TF-IDF is calculated as follows:

228   2.3   P ROMPT E NRICH
229
230   After obtaining the ranked keywords and their explanations, SEK integrates them with the original
231   problem. As shown in the enriched problem in Figure 3, SEK appends the ranking results to the
232
      end of the problem, providing additional explanations for key concepts in the problem. It’s worth
      noting that, to maintain the coherence of the problem context, we insert the phrase “Analyze the
233
      following key terms and their relationships within the problem context:” after the problem. This acts
234
      as a semantic buffer, smoothly transitioning from the original problem description to the appended
235   keywords. The enriched problem is then input into the LLM to generate the final solution.

      3.3   BASELINES
271
272   • Default LLM: This approach is based on the EvalPlus framework (Liu et al., 2024), using prob-
273     lems from the benchmark as input to prompt LLMs for code generation.
274
      • Zero-Shot CoT (Chain-of-Thought) (Kojima et al., 2022): This approach first prompts the LLM
        to “think step by step” for getting the intermediate reasoning steps and then concatenates the orig-
275
        inal problem description with the generated intermediate steps as input to get the code solution.
276
      • CoT (Wei et al., 2022): This approach generates a series of reasoning steps during the solution-
277     generation process for each problem. To ensure comparative fairness, both the CoT baseline and
278     SEK employ an equal number of demonstrations.
279   • One-Step CoT: This approach first prompts the LLM to ”Rephrase the problem description using
280     precise language”, and then uses this refined description to guide code generation. Both One-Step
281     CoT and SEK employ an equal number of demonstrations.
282   • SelfEvolve (Jiang et al., 2023a): This approach first uses LLMs to generate problem-specific
283     knowledge and produce initial code solutions based on such knowledge. Then, it iteratively re-
284     fines code solutions with LLMs based on execution feedback. Notably, SelfEvolve uses different
285
        prompt templates for different benchmarks to extract knowledge. Since these prompt templates
        have been open-sourced, we consistently apply its two-stage prompts on HumanEval (see Ap-
286
        pendix H) in our replication process. For a fair comparison, we remove the self-refinement mod-
287
        ule, and employ the same number of demonstrations as SEK.
288   • Beam Search (Wiseman & Rush, 2016): This approach employs distinct search beams and op-
289     timizes selection during the decoding process. Given that SEK requires LLMs to explore search
290     space twice by modifying the LLM’s search space through additional token insertion, we demon-
291     strate its benefit by comparing it with performing two searches within the LLM’s original search
292     space,i.e., beam search with a beam size of 2. We also demonstrate that with similar computational
293     costs, SEK consistently outperforms beam search (Appendix E.4).
294
295   3.4   I MPLEMENTATION D ETAILS
296
297   Prompt Design. It’s worth noting that the implementations except SelfEvolve are based on the
298   EvalPlus framework. Specifically, the only difference between SEK and Default is the addition of
299   keywords and explanations to the problem description. APPS contains problems in two formats:
300   call-based format and standard input format. Following previous work (Olausson et al., 2023; In-
301
      ala et al., 2022; Chen et al., 2023b), we employ a two-shot prompt to guide the LLM to generate
      appropriate solutions for different formats.
302
303   Demonstration Selection Strategy. Inspired by previous work (Wei et al., 2022; Mu et al., 2023;
304   Wang et al., 2023), we adopt a differentiated strategy that varies based on benchmark complexity
305   (See Appendix D). To reduce bias, we employ an LLM separate from our target LLMs (Claude-3.5-
306   Sonnet) to generate keywords and explanations for each demonstration, which are then manually
307
      reviewed and refined (See Appendix D).
308   Configuration. In our experiments, we treat the LLMs as black-box generators and only need
309   to set a few key interface parameters. We maintain consistent settings across all LLMs, employing
310   greedy decoding for output generation. The maximum output length is uniformly set to 2048 tokens.
311   Specifically, the LLMs accessed via APIs do not support Beam Search. Thus, we only implement
312   Beam Search for Llama-3.1-70B-Instruct and Mixtral-8×22B-Instruct-v0.1. Due to resource limita-
313
      tion, we compare SelfEvolve using GPT-3.5-turbo following the original paper (Jiang et al., 2023a)
      and additionally use two open-sourced LLMs (Llama-3.1 and Mixtral-8x22B).

494   5   R ELATED W ORK
495
496   LLM-based code generation: Recent advancements in LLMs have significantly improved code
497   generation capabilities. Models like CodeGen (Nijkamp et al., 2022), StarCoder (Li et al., 2023),
498
      and GPT series (Black et al., 2022; Chen et al., 2021) have demonstrated remarkable performance
      in translating natural language descriptions into code snippets. These models primarily use decoder-
499
      only architectures and next-token prediction for pre-training. A subset, including CodeT5 (Wang
500
      et al., 2021) and PLBART (Ahmad et al., 2021), employs encoder-decoder architectures. Our work
501   builds upon these foundations, focusing on enhancing LLMs’ problem-solving capabilities without
502   additional training.
503
504
      Prompting techniques for code generation: Prompting techniques for code generation can be
      broadly categorized into three types: The first type utilizes external knowledge to enhance LLMs’
505
      understanding of coding problems or intermediate outputs (Mu et al., 2023; Nashid et al., 2023;
506
      Zhong et al., 2024a). For example, CEDAR (Nashid et al., 2023) retrieves relevant code examples
507   from an external knowledge base to help LLMs understand task requirements. The second type re-
508   lies solely on LLMs’ inherent capabilities, using prompt design to guide LLMs in generating code
509   snippets that meet specific requirements (Wei et al., 2022; Wang et al., 2023; Yao et al., 2024). For
510   instance, Chain of Thought (Wei et al., 2022) employs a step-by-step, chain-of-thought style prompt
511   to guide LLMs in producing correct results. The third type integrates the previous two types, lever-
512   aging both external knowledge and the LLM’s inherent knowledge to solve coding problems (Chen
513   et al., 2023c; Jiang et al., 2023a; Tian & Chen, 2023; Chen et al., 2024c;b). For example, Self-
514   Debug (Chen et al., 2023c) uses the code execution results or the code explanations generated by
515
      the LLM itself to debug the incorrect code multiple times. SEK belongs to the second category.
      Different from other methods, it focuses on improving LLMs’ comprehension of the problem by
516
      identifying and explaining the key concepts in the problem description with LLMs themselves.
517
518   Keyword extraction: Keyword extraction methods have evolved from traditional statisti-
519   cal (Sparck Jones, 1972; El-Beltagy & Rafea, 2009; Florescu & Caragea, 2017; Rose et al., 2010)
520   and graph-based approaches (Mihalcea & Tarau, 2004; Wan & Xiao, 2008; Gollapalli & Caragea,
521
      2014; Grineva et al., 2009) to more advanced techniques leveraging language models (Mahata et al.,
      2018; Bennani-Smires et al., 2018; Sun et al., 2020; Arora et al., 2017). Recent works like Attention-
522
      Rank (Ding & Luo, 2021) and LLM-TAKE (Maragheh et al., 2023) use self-attention mechanisms
523
      and language models to identify significant keywords. Our work extends this concept to the domain
524   of code generation, using LLMs to extract and explain problem-specific keywords to enhance code
525   solution generation.
526
527
528   6   C ONCLUSION AND L IMITATIONS
529
530   In this work, we propose SEK, a simple yet effective method to enhance the code generation ca-
531   pabilities of LLMs. SEK leverages the LLM to extract and explain keywords from the problem
532   description, followed by ranking them based on their frequency. Through extensive experiments, we
533   demonstrate that SEK facilitates LLMs in capturing and clarifying key concepts within problems,
534   thereby generating more accurate code solutions.
535   One limitation of SEK is that the two-stage invocation process of SEK incurs additional computa-
536   tional overhead. Future work could explore compressing the process into one invocation. In addition,
537   keywords are extracted and explained by LLMs, of which the quality cannot be guaranteed due to
538   the hallucinations of LLMs (Ji et al., 2023). Mitigating this requires enhancing the factual accuracy
539   of LLMs (Mitchell et al., 2022; Tang et al., 2023) and proposing effective approaches for detecting
      factual errors (Chen et al., 2024a; Min et al., 2023).

869   Algorithm 1 KeyRank Procedure
870   Input: Keyword Set Kx , Problem P, Corpus C
871   Output: Ranked Keywords Ky
872    1: Kg ← ∅, Ka ← ∅, Ky ← ∅
       2: f ← E XTRACT F UNCTION NAME(P)
873
       3: for each k in Kx do
874    4:     if k = f then
875    5:         Kg ← Kg ∪ {(k, −1)}
876    6:     else if k ∈ P then
877    7:         Kg ← Kg ∪ {(k, TF-IDF(k, P, C))}
       8:     else
878
       9:         Ka ← Ka ∪ {k}
879   10:     end if
880   11: end for
881   12: Kg ← S ORT D ESCENDING(Kg )
882   13: Ky ← Ka ∪ Kg
      14: return Ky

886   First, we initialize the General Keywords, Abstract Keywords, and output as Kg , Ka , Ky , respec-
887   tively. E XTRACT F UNCTION NAME extracts the method name if provided in the problem descrip-
888   tion. Otherwise, it returns a null value. Then, keywords are classified and scored. They can be
889   divided into three classes: Abstract Keywords, General Keywords, and Function Keyword. Abstract
890   keywords do not appear in any input; they are abstract terms summarized from multiple concepts
891   and stored in Ka . General keywords denote items in the problem description. We calculate their
892   importance using TF-IDF based on a code-related corpus. General keywords and their scores are
893
      stored in Kg . Function keyword refers to the method name for solving the problem. Its explanation
      provides a coarse-grained description of the problem requirements. We assign a score of -1 to the
894
      function keyword, and also store them in Kg . Finally, S ORT D ESCENDING sorts the keywords in Kg
895
      based on their scores. The keywords are combined in the order of abstract, general, and function
896   keywords, and are then returned as the Ranked Keywords.

937

938
      399 tasks, and the latter adds 35× test samples.
939   (3) APPS (Hendrycks et al., 2021) includes 10,000 coding problems from open-access websites,
940   split equally into training and test sets. It includes two problem formats: call-based format (input
941   via function parameters) and standard input format (using stdin/stdout). Problems are categorized
942   into introductory, interview, and competition levels. There are three different difficulty levels of
943   problems in APPS, i.e., introductory, interview and competition. Each of them has 1000, 3000, and
944   1000 tasks, respectively. Considering the cost of evaluating the entire APPS test set and following
945   prior work (Olausson et al., 2023; Huang et al., 2024b; Le et al., 2024; Yang et al., 2023), we ran-
946   domly select problems in accordance with the frequency distribution of these difficulty levels and
947
      sample 60, 180, 60 problems at the introductory, interview, and competition levels, respectively.

951   Demonstration selection strategy. Specifically, for HumanEval, we select the first two problems as
952   demonstrations. For MBPP, we choose the first problem. For APPS, considering the model’s input
953   length limitation and to avoid randomness, we select the two shortest problems from the first five
954   problems in the training set. The reason for this differentiated strategy is that HumanEval and APPS
955   problems are more complex, requiring more examples, while MBPP problems are relatively simple

957   Keywords and explanations involved in demonstrations. The prompt for KeyExtract & Explain
958   uses several demonstrations to guide LLMs to produce keywords and their explanations. To ensure
959   the quality of each demonstration, we first employ Claude-3.5-Sonnet, an LLM separate from our
960   target LLMs, to generate multiple sets of keywords and explanations for each demonstration. The
961   generated contents are then manually reviewed, and the most accurate set for each demonstration is

963   tions. Additionally, for HumanEval(+) and MBPP(+) datasets, which provide function names, the

      first two authors discuss and write the explanation for the function name in each demonstration.
