009                                                       A BSTRACT
010
011             Code completion, a key downstream task in code generation, is one of the most
012             frequent and impactful methods for enhancing developer productivity in software
013             development. As intelligent completion tools evolve, we need a robust evaluation
014             benchmark that enables meaningful comparisons between products and guides
015             future advancements. However, existing benchmarks focus more on coarse-grained
016             tasks without industrial analysis resembling general code generation rather than
017             the real-world scenarios developers encounter. Moreover, these benchmarks often
018
                rely on costly and time-consuming human annotation, and the standalone test cases
                fail to leverage minimal tests for maximum repository-level understanding and
019
                code coverage. To address these limitations, we first analyze business data from an
020
                industrial code completion tool and redefine the evaluation criteria to better align
021             with the developer’s intent and desired completion behavior throughout the coding
022             process 1 . Based on these insights, we introduce Codev-Agent, an agent-based sys-
023             tem that automates repository crawling, constructs execution environments, extracts
024             dynamic calling chains from existing unit tests, and generates new test samples to
025             avoid data leakage, ensuring fair and effective comparisons. Using Codev-Agent,
026             we present the Code-Development Benchmark (Codev-Bench), a fine-grained, real-
027             world, repository-level, and developer-centric evaluation framework. Codev-Bench
028             assesses whether a code completion tool can capture a developer’s immediate intent
029
                and suggest appropriate code across diverse contexts, providing a more realistic
                benchmark for code completion in modern software development.

      With the rapid development of large language models (LLMs), LLM4X gains popularity cross various
035   domains, including healthcare [8], question answering [23, 24], and education [13]. In software
036   engineering, code LLMs such as Codeqwen [28], Codegeex [37], and Starcoder [20] also demonstrate
037   strong capability in code generation [22, 21]. Several code assistants including TONGYI Lingma [4]
038   and Copilot [17], begin integrating these code LLMs into industrial products. Among these tools,
039   code completion remains one of the most frequent and effective functionalities for boosting developer
040   productivity in daily software development [5]. With the emergence of various tools, there is an
041   urgent need for a comprehensive benchmark to compare their performance [31].
042   Initially, general code generation benchmarks like HumanEval [4] and ClassEval [6] are used to
043   evaluate completion abilities in form-closed, self-constrained Python functions and classes. However,
044   these benchmarks are too isolated to reflect real-world development practices. In response, DevEval
045   [15] and CrossCodeEval [9] introduce repository-level benchmarks that evaluate the code completion
046   about cross-file contextual understanding and retrieval capabilities, making them more aligned with
047   actual development environments. The latest work, RepoMasterEval [30], further conducts an study
048   to analyze the correlation between user acceptance and benchmark performance, demonstrating that
049
      a well-designed benchmark can guide the product improvement in practical settings. However, three
      key challenges remain: (1) the coarse-grained tasks without industrial analysis resemble general
050
      code generation rather than the real-world scenarios developers face; (2) limited extensibility, as

      2    BACKGROUND AND R ELATED W ORK
096   We begin by introducing existing LLMs designed for code generation. Next, we outline the bench-
097   marks for general code generation task and the more specific code completion task. It is important
098   to note that code completion is a key downstream task in code generation, as it is one of the most
099   frequent and impactful methods for enhancing developer productivity in software development.
100   Unlike coarse-grained tasks such as class or function generation, code completion often requires finer
101
      granularity. As a result, its evaluation demands more fine-grained and developer-centric scenarios,
      distinct from those used in general code generation.
102
103   2.1 LLM FOR C ODE G ENERATION
104   The evolution of Code LLMs starts from basic code generation to highly specialized models designed
105   for complex programming tasks. Codeqwen-1.5b [28] marks a step forward with its extensive
106   pretraining on programming languages, refining its precision in code generation. Following this,
107   Deepseek-coder-v2-lite [38] introduces a lightweight, efficient model suited for rapid completion
      and debugging in constrained environments. The development of Codegeex-4-9b [37] highlights a


                                                         2
      Under review as a conference paper at ICLR 2025


108
      shift toward tackling real-world scenarios with its Fill-In-Middle strategy, excelling in mid-segment
109   code generation. Starcoder-2-7b [20] further advances multi-language support, increasing its utility
110   in popular languages like Python and JavaScript. Finally, Codegemma-7b [27] exemplifies the trend
111   toward enterprise-scale code generation, focusing on managing and refining large codebases, meeting
112   the growing needs of software maintenance and development. Together, these models showcase the
113   continuous innovation in Code LLMs, expanding their capabilities and real-world applications.
114
115   2.2   B ENCHMARK FOR C ODE G ENERATION
116   Early works introduce benchmarks [35, 10] to evaluate code generation on form-closed and self-
117   constrained Python functions, such as HumanEval [4] and MBPP [3]. ClassEval [6] proposes a class-
118   level code generation dataset containing 100 human-crafted, self-contained Python classes. CoderEval
119   [34] extends the evaluation to non-standalone programs. DevEval [15] and CrossCodeEval [9] align
120   these evaluations with real-world code repositories, addressing more complex code generation
121   scenarios. Meanwhile, XCodeEval [12] and HumanEval-X [37] expand the scope to multilingual
122
      programming beyond just Python. In these works, the correctness of generated code snippets,
      measured by test cases (e.g., Pass@k [34]), and the semantic similarity between generated code and
123
      the ground truth (e.g., CodeBLEU [25]) are common evaluation metrics. Test cases derive either
124
      from human annotations or from general-purpose LLMs (GPT-4 [1]). The former (human annotation)
125   is time-consuming and costly, while the latter (GPT-4-generated test cases) tends to be unstable and
126   often fails to capture key correlations across a repository.
127
128   2.3   B ENCHMARK FOR C ODE C OMPLETION
129   With the emergence of various code assistant tools, such as Copilot [4], Visual Studio IntelliCode [7],
130   and TONGYI Lingma [17], people start to incorporate code LLMs in real-world development. Code
131   completion is one of the most frequent and useful functionalities. CrossCodeEval [5], RepoBench
132   [19], and RepoEval [36] propose benchmarks to evaluate the repository-level code completion
133   across different dimensions, such as cross-file contextual understanding, retrieval capability, and
134   various levels of generation granularity. The most recent work, RepoMasterEval [30], also conducts
135   an industrial study to evaluate the benchmark in a practical setting. However, these work face
136   three common challenges: (1) human annotation of data samples and test cases is time-consuming
137
      and costly, (2) coarse-grained tasks such as class and function completion resemble general code
      generation more than the real-world scenarios developer faced, and (3) unsatisfying and standalone
138
      generated test cases fail to reveal code LLMs’ ability to understand context at the repository level.
139
      To address these gaps, we first analyze the feedback data from industrial code completion tool and
140   redefine the evaluation criteria to better capture a developer’s intent and desired completion behavior
141   throughout the coding process. Finally, we deliver the Codev-Agent and Codev-Bench.

144
145   As mentioned earlier, there are three common challenges: (1) coarse-grained tasks resemble general
146
      code generation more than real-world scenarios faced by developers, (2) human annotation of data

147
      demonstrate LLMs’ ability to understand context at the repository level. To address challenge (1),
148
      we request business data from a real-world deployed code completion product through our partner

152   existing unit tests, and generates new test samples based on dynamic data flow to prevent data
153   leakage, ensuring fair and effective comparisons. Built on top of this agent, we provide a fine-grained,
154   real-world, repository-level, and developer-centric benchmark: Codev-Bench.

200
      LLM-based Codev-Agent aims to automatically update our benchmark to avoid data leakage, ensuring
201
      fair and effective comparisons with minimal cost. It can minimize the human effort but keep the
202
      stability in repositories selection, execution environment setup, test samples extraction based on
203   dynamic data flow of existing unit test files. Figure 2 visualizes the pipeline.
204

205
      crawls up-to-date repositories, gathering the most relevant and current data for analysis and test case
206   generation. The crawler operates based on the following criteria: (1) only repositories created within
207   the last four months are considered to ensure data timeliness; (2) only repositories with a high star
208   count (more than 50 stars) are selected to guarantee community engagement and project popularity;
209   (3) after scanning the file directory, only repositories containing unit test files are retained, ensuring
210   the presence of relevant testing infrastructure; (4) by leveraging the Qwen to understand and analyze
211   the README file, the crawler filters for repositories that require only lightweight configuration
212   environments for execution, thus optimizing for efficiency and reducing complexity.
213   Execution Environment Setup After gathering the qualifying repositories and their associated unit
214   tests from the crawling phase, Codev-Agent sets up the corresponding execution environments through
215   an iterative process. As illustrated in Figure 2b, the first step involves detecting and extracting the

      4.1 E XPERIMENTAL S ETUP
361   Scenario 1 - Full block completion: In this scenario, the model is tasked with completing a full
362
      code block (e.g., function, if, for, try, or statement) based on a complete, unbroken surrounding
      context. To pass, the model must accurately complete the block and stop at the correct point, ensuring
363
      it passes the unit test successfully. Scenario 2 - Inner block completion: In this scenario, the
364
      model is required to complete a portion of code block based on a complete, unbroken surrounding
365   context. In addition, 20% of the samples in this scenario have an empty ground truth, evaluating
366   the ability to recognize when the current block is already complete and no further completion is
367   needed. Scenario 3 - Incomplete suffix completion: Compared to Scenario 1, this scenario focuses
368   on cases where the suffix content following the current cursor is incomplete. It covers two sub-cases:
369   one where all the suffix content after the cursor in entire file is empty, and another where only
370   the content within the current function body after the cursor is missing. Scenario 4 - RAG-based
371   completion: In this scenario, the model builds upon the full block completion task by incorporating a
372   Retrieval-Augmented Generation (RAG) module. The repository is partitioned into chunks, with only
373
      functions being considered as candidates. The function containing the current code is used as the
      query, and the query’s embedding is compared with the embeddings of the candidate functions. The
374
      top 3 most similar candidates are then inserted back into the prompt as hints to guide code generation.

      4.3 E VALUATION M ETRICS
401
      Test Case Pass Rate: We generate code snippets by prompting LLMs and inserting them back
402
      into the original code, then rerun the corresponding unit tests. If the test passes without errors, the
403
      generated completion is considered successful. We use Pass@1 as the metric, which represents
404   the probability that the LLM generates a code snippet that passes on the first attempt. This metric
405   provides a clear evaluation of how effectively the LLM generates valid, executable code completions.
406
      Edit Similarity (ES) Levenshtein distance quantifies the number of single-character ed-
407   its—insertions, substitutions, or deletions—required to transform one sequence of tokens into another
408   [26]. In practice, developers accept approximate code completions and make manual edits. Therefore,
409   ES is a crucial metric for evaluating how closely a generated completion matches the intended result.
410   The Levenshtein distance between two strings is calculated using dynamic programming.

      5    S UMMARY AND F UTURE W ORK
533   In this work, we redefined evaluation criteria for code completion tools through a comprehensive busi-
534   ness analysis and introduced Codev-Agent, an automated system for repository crawling, environment
535   setup, call chain analysis, and LLM evaluation. We also proposed Codev-Bench, a developer-centric,
536   real-world framework that assesses whether code completion tools capture developers’ intent across
537   diverse contexts. Our evaluation of several state-of-the-art LLMs offered insights for future improve-

      multiple programming languages, generating a multilingual dataset, while also refining Codev-Bench

      to handle more complex developer scenarios.
