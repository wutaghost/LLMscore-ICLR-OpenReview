009                                               A BSTRACT
010
011            Text-based AI system optimization typically involves a feedback loop scheme
012            where a single LLM generates an evaluation in natural language of the current
013            output to improve the next iteration’s output. However, in this work, we em-
014            pirically demonstrate that for a practical and complex task (code generation) with
015            multiple criteria to evaluate, utilizing only one LLM evaluator tends to let errors in
016            generated code go undetected, thus leading to incorrect evaluations and ultimately
017            suboptimal test case performance. Motivated by this failure case, we assume there
018
               exists an optimal evaluation policy that samples an evaluation between response
               and ground truth. We then theoretically prove that a linear combination of multiple
019
               evaluators can approximate this optimal policy. From this insight, we propose AI
020
               system optimization via Multiple LLM Evaluators (AIME). AIME is an eval-
021            uation protocol that utilizes multiple LLMs that each independently generate an
022            evaluation on separate criteria and then combine them via concatenation. We pro-
023            vide an extensive empirical study showing AIME outperforming baseline methods
024            in code generation tasks, with up to 62% higher error detection rate and up to 16%
025            higher success rate than a single LLM evaluation protocol on LeetCodeHard and
026            HumanEval datasets. We also show that the selection of the number of evaluators
027            and which criteria to utilize is non-trivial as it can impact pact success rate by up
028            to 12%.

      1   I NTRODUCTION
032
033
      Pre-trained foundation models, such as Large Language Models (LLMs), have developed rapidly
      over the recent years (Achiam et al., 2023; Touvron et al., 2023). With these advancements, AI
034
      systems have grown in popularity for various tasks such as code generation (Chen et al., 2024;
035
      Gulwani, 2010), question-answering (Patel et al., 2024; Wang et al., 2024), mathematical reasoning
036   (Trinh et al., 2024; Song et al., 2024), exploration (Dorbala et al., 2024; 2023; Ren et al., 2024),
037   and information retrieval (Gao et al., 2023) etc. As the application complexity increases, the shift
038   to AI systems containing multiple components such as LLM-based agents and web search (Xiong
039   et al., 2024), will continue (Zaharia et al., 2024; Yuksekgonul et al., 2024). Thus, automatically
040   optimizing these systems, AI system optimization (Yuksekgonul et al., 2024), becomes increasingly
041   necessary.
042   An emerging paradigm is text-based optimization, also known as prompt optimization (Cheng et al.,
043   2023; Wang et al., 2023; Zhou et al., 2022), whereby the natural language input prompt is tuned to
044   generate an optimal output. This method requires no numerical gradient descent updates typical in
045   optimization for machine learning models (Van Der Malsburg, 1986; Hassoun, 1995; Barto, 1992)
046   and is thus appropriate for optimizing AI systems with fixed LLM components. Recently, there has
047   been a growing class of iterative online methods for text-based optimization (Cheng et al., 2024;
048   Yuksekgonul et al., 2024; Shinn et al., 2024), where a single LLM generates an evaluation based on
049   the current output to help generate the next iteration’s prompt.
050   While prior art has compared the abilities of a single LLM for evaluations against those of multiple
051   LLMs (Kocmi & Federmann, 2023; Ankner et al., 2024), in AI system optimization literature, there
052   has been a lack of studies questioning the capabilities of using a single LLM evaluator to drive the
053   optimization process. Recently, Yuksekgonul et al. (2024) has viewed the evaluation as a text-based
      analogy to the objective function for backpropagation (Hinton et al., 2006; Rumelhart et al., 1986)

      policy is conditioned on, so the same LLM model can be utilized. In this paper, we use the same
140

231   3.2    OVERVIEW OF AIME: M ULTIPLE ROLE -S PECIFIC E VALUATORS
232
233   Now that we have motivated utilizing multiple LLM-based evaluators, we now address the question
234   on how to utilize multiple evaluators. To do so, we look at the ideas of roles. The LLM-based
235   evaluation policy has an evaluation system prompt to specify what the evaluation should be based
236   on. For tasks such as code generation, there may be multiple criteria or objectives to evaluate for
237   such as correctness, clarity, and efficiency. Furthermore, aspects such correctness of code can rely
238   on various aspects such as logic and syntax. Normally, with a single evaluator, all the criteria are
239
      specified together in the system prompt. However, we see from Figure 1 and later in Section 4 that
      this approach can fail significantly to reach the optimal performance. We thus propose splitting the
240
      evaluation instruction across multiple evaluators, each one getting a specific role. We then aggregate
241
      via string concatenation them into a final evaluation. We chose concatenation as the aggregation
242   method as it is analogous to creating a linear combination of the outputs (Yuksekgonul et al., 2024).
243   We call this approach AIME: AI System Optimization with Multiple Evaluators.
244
245
      Our AIME approach is a simple-to-implement approach that requires minimal changes to the already
      established methods (Yuksekgonul et al., 2024; Cheng et al., 2024) for system optimization. Our
246
      approach requires only modifying the evaluation step of the optimization pipeline from one evaluator
247
      to multiple. In Algorithm 1, given an output y, set of k roles R, and pre-trained LLM πθ we sample
248   k evaluations, {ek }Kk=1 . We obtain ei by conditioning πθ by x, y and Rk ∈ R. Conditioning on rk
249   is to specify the role in the evaluation system prompt.

      We test the merits of our AIME approach via the code generation task because of its practicalness
256
      and its multiple plausible criteria (e.g., correctness, efficiency). Here, the AI system is an LLM
257
      generator that is given a code prompt and must produce a code snippet that passes the unit tests
258   for that prompt. This code generation task is a form of instance optimization (Yuksekgonul et al.,
259   2024), whereby the optimization variable, the input prompt, is defined as xt+1 := (yt , ft ). y0 , f0 are
260   empty strings. We provide empirical results showing that AIME is superior to the single-evaluation
261   (Single-Eval) approach in detecting code errors and that AIME-based optimization achieves higher
262   success in test cases than Single-Eval-based optimization. Experiments were run on an Apple M1
263   Pro and macOS 14.5.

269   to the LLM that it is a code generator while the pupdate from 1 ≤ t ≤ T specifies that it generates a

283   syntax errors, logic errors, correctness, readability, runtime, and code redundancy. The following

492   5   R ELATED W ORKS
493
494
495   AI System Optimization: Many prior works have studied the optimization of complex AI systems.
496
      DSPy (Khattab et al., 2024; 2022; Singhvi et al., 2023), Trace Cheng et al. (2024), and TextGrad
      (Yuksekgonul et al., 2024) have formulated LLM and AI-based systems as a network of multiple
497
      layers and provided methods to optimize these system analogous to backpropagation and autodiffer-
498
      entiation. Pryzant et al. (2023) established text-based gradients, or Textual Gradients, as feedback
499   to an AI system. Chakraborty et al. (2024a); Ding et al. (2024) used a bi-level optimization for-
500   mulation to align AI agents and systems. Text-based reinforcement learning has also been used to
501   improve LLM-based systems (Shinn et al., 2024). Decoding and RLHF is an alternative method to
502   optimize or align an LLM with gradient descent (Chakraborty et al., 2024b; Mudgal et al., 2023;
503   Chakraborty et al., 2024c). While these works have shown tremendous results, there has been a gap
504   in the literature we aim to address analyzing the effect of using multiple independent evaluations to
505   optimize the AI system for a complex task, code generation (Chen et al., 2024; Zeng et al., 2024;
506   Zhang et al., 2023; Jha et al., 2010; Shinn et al., 2024; Yuksekgonul et al., 2024; Zan et al., 2022;
507
      Jiang et al., 2024; Chen et al., 2021; Gulwani, 2010).
508   LLM-based Evaluation: LLM-based evaluation, or LLM-as-a-Judge (Zheng et al., 2023), has been
509   growing in interest due to the ability of LLMs to evaluate large outputs like text (Sellam et al., 2020;
510   Kocmi & Federmann, 2023) quickly and to align with human preferences. Verga et al. (2024)
511   showed a panel of smaller LLM judges can provide numeric scores correlating to human judgment
512   than a single larger LLM model can. Prior work has also studied finetuning LLMs to be judges (Zhou
513
      et al., 2022). Ankner et al. (2024) used LLM-generated critiques to augment the scalar reward from
      a reward model. Li et al. (2023) used discussion between multiple LLMs to select a strong LLM-
514
      based evaluator for question-answering. Strong LLM judges have been shown to generalize across
515
      tasks (Huang et al., 2024). Weak LLM evaluators have been used to judge the debate between two
516   stronger LLMs (Kenton et al., 2024). We are the first to use multiple LLM-based evaluators for
517   iterative AI system optimization.

      6   C ONCLUSION , L IMITATIONS , AND F URTHER W ORKS
521
522
523   In this work, we tackle AI system optimization by introducing AIME. AIME utilizes multiple LLM-
524   based evaluators to provide natural language evaluation for the current system output, improving on
525
      prior methods that only use a single evaluator. Our key insight is to condition each evaluator with
      a specific role rather than giving all the roles to a single evaluator. We prove that increasing the
526
      number of evaluations reduces the suboptimality evaluation gap, and empirically demonstrate that
527
      AIME outperforms Single-Eval in code generation tasks, analyzing success, completion, and error
528   detection rates. Furthermore, we study AIME’s robustness to the adversarial evaluator that generate
529   incorrect evaluations. We also provide ablations such as on the diversity of roles, role combinations,
530   and evaluation temperature, consistently demonstrating AIME’s superior performance and the need
531   for multiple evaluators.
532
      Limitations and Further Work. We only empirically study our approach in code generation. Fur-
533
      ther work could extend this evaluation approach to other tasks that require multiple criteria like
534   molecule optimization or text generation. In terms of system complexity, we only study multiple
535   evaluators for AI systems comprising a single LLM-based agent, and using a compound system with
536   multiple elements such as a web search agent (Agentic AI system) could be interesting. Another as-
537   pect of the work that can be explored further is weighting the different LLM-based evaluations. We
538   gave uniform weighting to all evaluations by giving them the same max output tokens and con-
539   catenating them. Future research could investigate methods of weighting and aggregation, possibly
      using another LLM to summarize or perform best-of-N on the evaluations.
