011                                                 A BSTRACT
012
013            Federated learning is a distributed learning setting where the main aim is to train
014            machine learning models without having to share raw data but only what is required
015            for learning. To guarantee training data privacy and high-utility models, differential
016
               privacy and secure aggregation techniques are often combined with federated
               learning. However, with fine-grained protection granularities the currently existing
017
               techniques require the parties to communicate for each local optimization step, if
018
               they want to fully benefit from the secure aggregation in terms of the resulting
019            formal privacy guarantees. In this paper, we show how a simple new analysis
020            allows the parties to perform multiple local optimization steps while still benefiting
021            from joint noise scaling when using secure aggregation. We show that our analysis
022            enables higher utility models with guaranteed privacy protection under limited
023            number of communication rounds.

026   1    I NTRODUCTION
027
028   Federated learning (FL; McMahan et al. 2017; Kairouz et al. 2019) is a common distributed learning
029   setting, where a central server and several clients holding their own local data sets collaborate to train
030
      a single global model. The main feature in FL is that the clients do not directly communicate data, but
      only what is required for learning, e.g., gradients or updated model parameters (pseudo-gradients).
031
032   While FL satisfies the data minimization principle, i.e., only what is actually needed is communicated
033   while the actual raw data never leaves the client, it does not protect against privacy attacks such as
034   membership inference (Shokri et al., 2017) or reconstruction (Fredrikson et al., 2014; Yeom et al.,
035   2018). Instead, training data privacy is commonly ensured by combining differential privacy (DP;
036
      Dwork et al. 2006b), a formal privacy definition, and secure multiparty computation (MPC; Yao
      1982) with FL (see, e.g., Kairouz et al. 2019).
037
038   DP is essentially a robustness guarantee for stochastic algorithms, which guarantees that small
039   perturbations to the inputs have small effects on the algorithms’ output probabilities. What constitutes
040   a small perturbation depends on the chosen protection granularity: the same basic DP definition can
041   be used for ensuring privacy on anything from single sample to entire data set level. In turn, MPC
042
      protocols can be used to limit the amount of information an adversary has about computations. In
      FL, secure aggregation (SecAgg) protocols, a specialised form of secure computation that requires
043
      significantly less resources than general MPC, are commonly used for communicating model updates
044
      from the clients to the server, which can result in provably better joint DP guarantees than is possible
045   to achieve by any single client in isolation.
046
047   Under the general FL setup, two main alternatives are commonly considered: cross-device FL and
048
      cross-silo FL (Kairouz et al., 2019). In cross-device FL, each client is assumed to have a small
      local data set, while the total number of clients is large, e.g., thousands or millions. In the cross-silo
049
      case, the total number of clients is small, for example, a dozen, but each client is assumed to have a
050
      larger local data set. In this paper, our running example is standard cross-silo differentially private
051   FL (DPFL) where the clients communicate all updates to the server using SecAgg.1 In this setting,
052

      the most useful DP protection granularity is typically something strictly more fine-grained than
055   client-level: when clients are, e.g., different hospitals or banks, there are typically several individuals
056   in a single clients’ local data set and the protection granularity needs to match the use case.
057
058
      While client-level granularity in DPFL is, at least in principle, straightforward to combine with
      SecAgg, more fine-grained granularities such as sample-level DP can present problems: using
059
      existing techniques one has to choose between i) having joint DP guarantees with less noise due to
060
      SecAgg but with all clients using only a single local optimization step per FL round, and ii) having
061   more noisy local DP (LDP) guarantees that do not formally benefit from SecAgg while allowing
062   the clients to do more local optimization steps per FL round. Both of these options have significant
063   drawbacks: the amount of server-client communications is typically one of the first bottlenecks that
064   limit model training in FL, while LDP guarantees regularly require noise levels that heavily affect the
065   resulting model utility. In this paper we show that this trade-off is not unavoidable but can be largely
066   remedied by a simple new analysis of the problem.
067
068   Our Contribution
069
070          • We present a novel and simple theoretical privacy analysis showing when we can increase
071
               the number of local optimization steps in FL using fine-grained DP granularity, while still
               benefiting from joint DP guarantees using a trusted aggregator.
072
073          • We demonstrate empirically that the proposed approach can lead to large utility benefits
074            without requiring any changes to the underlying algorithms under both iid and heterogeneous
075            client data splits.
076          • Our results point to a clear mismatch between the current theoretical understanding of DPFL
077            and practical results.

      2    R ELATED W ORK
080
081
      There is a significant amount of existing work focusing on the general problem of combining DP
082
      with FL, although the focus has mostly been on the cross-device FL setting with user- or client-level
083   DP. To the best of our knowledge, while the combination of DPFL with SecAgg is certainly not
084   novel (see, e.g., Truex et al. 2019; Kairouz et al. 2019; Heikkilä et al. 2020; Stevens et al. 2022;
085   Yang et al. 2023), there is no existing work on the privacy analysis when the clients do multiple local
086   optimization steps with fine-grained DP and communicate the results via SecAgg.2
087
      Considering the existing work in more detail, we can distinguish some main lines of closely-related
088
      research. There are many papers proposing novel learning methods for FL, assuming sample-level DP
089   and joint noise scaling with SecAgg. While the existing work only uses a single local optimization
090   step (see, e.g., Heikkilä et al. 2020; Malekzadeh et al. 2021; Stevens et al. 2022; Yang et al. 2023),
091   our analysis can be leveraged in this setting to enable running multiple local steps generally for many
092   such methods without requiring any other changes to the algorithms.
093
      Another clear line of work has focused on introducing novel discrete DP mechanisms that can be used
094
      with additively homomorphic encryption techniques, which typically operate on the group of integers
095   with modulo additions. Agarwal et al. (2018) proposed a binomial mechanism that provides DP using
096   discrete binomial noise. Improving on the binomial mechanism, Canonne et al. (2020) proposed a
097   discrete Gaussian mechanism, while Agarwal et al. (2021) introduced a Skellam mechanism and
098   Chen et al. (2022b) a Poisson-binomial mechanism, both of which improve on the discrete Gaussian,
099   e.g., by being infinitely divisible distributions: the sum of Skellam/Poisson-binomial distributed
100   random variables is another Skellam/Poisson-binomial random variable. Our work is not focused
101   on introducing new DP mechanisms, but our analysis allows for using many different DP noise
102   mechanism. In particular, our analysis allows for joint noise scaling under SecAgg including when
103
      using infinitely divisible DP mechanisms, such as the Skellam mechanism, with pseudo-gradients
      and fine-grained DP protection level.

113   In principle, any such technique for compressing the model updates compatible with SecAgg can
114   also be directly combined with our joint noise scaling analysis. In contrast, benefiting from gradient
115   quantization is not entirely straightforward as in our case the model updates are pseudo-gradients and
116   not gradients. We leave a detailed consideration and comparison of the possible methods for reducing
117

      the clients for future work.
118
119   In summary, while many of the contributions cited above, e.g., novel DP mechanisms, are not limited
120   to cross-device FL, all the experiments and use cases mentioned in the cited papers that are compatible
121   with SecAgg and use multiple local steps only consider joint noise scaling with user- or client-level
122   DP in cross-device FL. In contrast, we focus on more fine-grained DP granularities, namely on
123
      sample-level DP. As we discuss in Section 3, combining sample-level DP with multiple local steps
      and joint noise scaling using SecAgg with good utility requires a novel privacy analysis. The main
124
      aim of this paper is to provide such an analysis.
125
126   While the currently existing theoretical convergence bounds for DPFL do not show any benefit from
127   increasing the number of local steps in DPFL (see Malekmohammadi et al. 2024, Theorem 3.2), we
128   empirically demonstrate the utility of our analysis in Section 5 after stating the results in Section 4.
129
      Our results clearly highlight the need for improving the theoretical analysis of DPFL over what is
      shown by Malekmohammadi et al. (2024) to understand when increasing the number of local steps is
130
      useful (compare this disagreement of empirical results and theory to the discussion by Mishchenko
131
      et al. 2022 on the provable usefulness of local steps in non-DP FL).

      3.2   P ROBLEM WITH L OCAL S TEPS IN DPFL WITH S EC AGG
184   While DP offers strict privacy protection, it comes at the cost of reduced model utility. This is
185   especially true in the local DP (LDP) setting, where each client protects its own data independently of
186   any other party (Kasiviswanathan et al., 2008). One well-known technique to improve model utility
187   in DPFL has been to utilise secure aggregation (SecAgg) to turn LDP guarantees into joint guarantees
188   or distributed DP guarantees that depend on multiple clients (see, e.g., Kairouz et al. 2019). However,
189   naively combining fine-grained DP protection with SecAgg for distributed DP runs into problems, as
190   we demonstrate in the rest of this section.

224   3.3    T RUST M ODEL
225
226
      In this paper, we assume an honest-but-curious (hbc) server and that all the clients are fully honest.
      The latter assumption can be easily generalised to allow for hbc clients with some weakening to
227
      the relevant privacy bounds: with N (non-colluding) hbc clients, since any client could potentially
228
      remove its own noise from the aggregated results, the noise from the other N − 1 clients needs to
229   guarantee the target DP level. In effect, to allow for all hbc clients, we would need to scale up the
230   noise level somewhat (see, e.g., Heikkilä et al. 2017 for a discussion on noise scaling and for formal
231   proofs).
232
      In principle, the same technique can also protect against privacy threats in the case of including some
233
      fully malicious clients in the protocol (i.e, simply scale the noise so that the hbc clients are enough to
234
      guarantee the required DP level). However, in this case the required level of extra noise will increase
235   quickly with the number of malicious clients leading to heavier utility loss. With malicious clients,
236   there would also be no guarantee that the learning algorithm terminates properly.

410   Setup and Motivation: Our chosen settings try to mimic a typical cross-silo FL setup: there are
411   a limited number of clients, each having a smallish local database. The clients have enough local
412
      compute to run optimization on the chosen model, while the number of server-client communications
      required for updating the global model are the main bottle-neck. Note that this bottleneck will
413
      emerge even with larger actual organisations training models with broadband connections, when the
414
      model size is large-enough, e.g., when training foundation models (Bommasani et al., 2021). This
415   is especially true when using SecAgg protocols, since the cost of running a real SecAgg algorithm
416   presents significant compute and communication overheads even with the efficient protocols discussed
417   in Appendix A.2. In this setting, it makes sense to try and push more optimization steps to the clients
418   while reducing the number of global updates (FL rounds). We also assume that the clients send their
419   local updates via some trusted aggregator (which we only assume and do not implement in practice in
420   the experiments. However, we do use only discrete DP mechanisms compatible with standard SecAgg
421   algorithms in all the experiments). For more details on all the experiments, see Appendix A.4.

512      Improved Privacy from Model Averaging: Finally, our results might sometimes be of use also
513      outside standard FL. For example, consider a setting where we have N copies of a model trained
514      on disjoint data sets (e.g., one could think of independent parties learning a classifier on top of a
515
         common pre-trained model or fine-tuning a common pre-trained model; either scenario would usually
         lead to shared model structure and possibly also hyperparameters without explicit coordination), and
516
         the parties would like to combine the models post-hoc without running any joint training from scratch.
517
         Since this can be seen as FL with a single FL round, if the original model training on each party
518      satisfies Assumption 4.1 (or the relaxed assumptions in Appendix A.1), then a simple averaging of
519      the weights will result in a joint model with improved privacy guarantees against adversaries without
520      access to the original models.
521
        To demonstrate this effect, we account for privacy assuming the same linear model used in Figure 2
522
        (but without actually training any models), Skellam mechanism with 32 bit gradients, Poisson
523
        subsampling with sampling probability 0.1, and varying number of parties and local steps. The
524     accounting is done as it would be done in a realistic setting: we first find a noise level σLDP that
525     results in the target privacy level (unbounded (ε = 5, δ = 1e − 5)-LDP) for each separate model with
526     the chosen number of local steps. We then assume that the local training satisfies Assumption 4.1
527     and calculate the privacy for averaging varying number of local models. Combining even 2 models
528     results in clearly improved privacy for the averaged model (see Table 1 in Appendix A.5).

         6         D ISCUSSION
531
532
         In this paper we have shown how to combine multiple local steps in DPFL using fine-grained protec-
533
         tion granularities with SecAgg, and empirically demonstrated that this can bring considerable utility
534      benefits under various communication-constrained settings. Our experimental results stand in stark
535      contrasts with the message from the currently existing theoretical bounds for DPFL (Malekmoham-
536      madi et al., 2024, Theorem 3.2), which do not show any benefit from increasing the number of local
537      steps. This disagreement of experimental and theoretical results underlines the need for improved
538      theoretical analysis to understand the conditions under which increasing the number of local steps can
539      lead to improved utility, similar to the recent breakthroughs in analysing non-DP FL (Mishchenko
         et al., 2022).

      A.2    F ROM I DEAL T RUSTED AGGREGATORS TO P RACTICAL S EC AGG P ROTOCOLS
919
920   For implementing the trusted aggregator assumed in Theorem 4.7 in practice, it should be noted that
921   as the sum over s is done locally by each client during local optimization, it is always trusted as
922   long as the individual clients are, while the sum over i would need to be implemented, e.g., using a
923   suitable SecAgg protocol. Several such algorithms are known, including the ones proposed by Bell
924   et al. (2020); Bonawitz et al. (2017); Sabater et al. (2022); So et al. (2021).
925   Using a SecAgg protocol will typically also place some extra requirements on the DP mechanisms
926     (s)
      Ai , since the SecAgg algorithms usually run on elements of finite rings. This precludes continuous
927   noise mechanisms. A viable alternative is to use some suitable discrete noise mechanism, such as
928   Skellam (Agarwal et al., 2021) or Poisson-binomial (Chen et al., 2022b). However, differing from
929   the cases considered in the cited papers, since in our case the clients send model updates instead of
930   single gradients, the finite ring size used in the SecAgg protocol needs to accommodate the model
931   update size: it does no good to use Skellam mechanism with gradient quantization to a small number
932   of bits, if the model weights and the resulting model update ∆i for client i still uses 32 bit floats.

958
      nism to optimise the local model parameters, and standard federated averaging as the aggregation rule

964   train-validation. We use Bayesian optimization-based approach implemented in Weights and Biases
965   (Biewald, 2020) for hyperparameter tuning, and simulate FL using Flower (Beutel et al., 2020).

967   run, we train the model on hyperparameter training fraction, test on the validation fraction, and
968   try to optimise for the final model weighted validation loss. After finishing the hyperparameter
969   tuning, we re-train the model from scratch 5 times with different random seeds with the best found
970   hyperparameters using the entire original training data and testing on the test fraction. We report the
971   mean and the standard error of the mean (SEM) in all the figures. In Figure 2 we plot the minimum
      test loss/maximum test accuracy taken over the entire training run.
