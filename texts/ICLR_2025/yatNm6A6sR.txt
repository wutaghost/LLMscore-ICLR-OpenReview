009                                               A BSTRACT
010
011            Road network data can provide rich information about cities and thus become the
012            base for various urban research. However, processing large-volume world-wide
013            road network data requires intensive computing resources and the processed results
014            might be different to be unified for benchmark downstream tasks. Therefore, in
015            this paper, we process the OpenStreetMap data and release a structured world-wide
016            1-billion-node road network graph database with high accessibility and usability.
017            We have presented three illustrative use cases, traffic prediction task, city boundary
018
               detection task and traffic policy control task. Moreover, for the well-investigated
               traffic prediction task, we release a new benchmark with 31 datasets, which is much
019
               more comprehensive than the previously frequently-used datasets. While for the
020
               relatively novel traffic policy control task, we release a new 6 city datasets with
021            much larger scale than the previous datasets. Along with the OSM+ dataset, the
022            release of data converters facilitates the integration of multimodal spatial-temporal
023            data based on map information for large model training, thereby expediting the
024            process of uncovering compelling scientific insights.

027   1   I NTRODUCTION
028
029   Road network has formed the skeleton of cities, as it connect between regions within city and between
030   different cities. For long time, urban regions and road networks stretch along each other. Therefore,
031   road networks can essentially reflect the landscape and function zones in cities, and thus affect human
032   mobility. For instance, sky-scrappers, restaurants, and shopping centers tend to locate at places
033
      with dense in-city road networks. Hence, investigating road network structures is the base for urban
      research, e.g., urban planning, urban traffic prediction.
034
035   However, obtaining accurate road network data for open public research is difficult for the following
036   two reasons. (1) Accurate road network data is collected at extremely high expense by map providers
037   like Google (Google Maps), Bing (Bing Maps), Baidu (Baidu Maps) and Gaode (Gaode Maps). Thus,
038   these products are mainly designed for commercial use with only very limited specific high-level
039
      API open to the public, e.g., POI searching, origin-destination route planning. This can not satisfy
      the need of academic researchers and start-up companies to conduct flexible low-level computing
040
      operations on open road network data to quickly iterate ideas or products. (2) Open-source map
041
      services, e.g., OpenStreetMap (Haklay & Weber, 2008a), built from crowdsourcing mechanisms by
042   world-wide users, seems to be the cure. However, due to the massive amount of the road network data
043   and the complex data format in map object storage, processing the road network data from scratch to
044   obtain desired format for experiments are always challenging and time-consuming. Hence, it is highly
045   desirable that an intermediate format of processed road network data can support diverse downstream
046   applications so as to speed up the scientific discovery.
047
      Following this path, some studies (Grinberger et al., 2022; Bartzokas-Tsiompras, 2022; Ding et al.,
048   2022) have been conducted utilizing open road network data like OpenStreetMap, while several issues
049   remain. First, the cleaning of OpenStreetMap contains a complex pipeline, including converting,
050   reducing, transforming and aggregating. This pipeline may take about 10 hours even only for
051   processing a region with 1,000km2 size, when running on a computer with 32 CPU cores and 128GB
052   memory. Second, the computing of world-wide map data requires memory far more than that of a
053   single machine. The world-wide raw OSM data is roughly 1.1 TB before processing, and finally
      processed structured OSM data can be a graph with more than 1 billion nodes. Although segmenting

      the regions and process each region separately may make it possible to process the data locally, this
055   requires hand-crafted distributed computing strategies and further aggregating the processed data
056   may introduce extra errors to the data. Because of the aforementioned two issues, researchers created
057   different ways to process the road networks and thus obtained various versions of same raw data. In
058   this large model era, this has become the barrier for comparison among different studies and creation
059   of multi-modal spatial temporal data for large model training. Hence, these issues can be solved
060   with building a pre-computed structured version of OpenStreetMap for public use with a powerful
061   computer.
062   In this paper, we propose to make public a structured road network computing engine, OSM+. OSM+
063   is composed of three components: (1) a road network graph database with world-wide intersections
064   (nodes) and road segments (links), and side information contained in map, e.g., POI; (2) a series of
065   auto-parallel fundamental computing APIs to allow efficient node query and distance query, on which
066   researchers may build more comprehensive road network analysis; (3) a series of data converters to
067   tailor the road network data for three typical urban research problems. The contributions of this paper
068   can be summarized as follows.
069
070
      • Data-wise. We provide an easy-to-use world-wide road network database and make it open-
        source. This makes the dataset a benchmark for related research. The OSM+ dataset can be
071
        accessed and the related code can be found at https://anonymous.4open.science/r/
072
        OSM-dataset-3034.
073
074
      • System-wise. We provide cloud-computing based APIs to enable efficient billion-scale graph query
        and processing, allowing various subsequent data processing for extended applications.
075
076   • Application-wise. We provide three example application scenarios, traffic prediction task, traffic
077     signal control task and city boundary detection task. The data converters are released, which will
078
        make it possible to use map data as the base to fuse multi-modal spatial temporal data for large
        model training and accelerate interesting scientific discovery.
079
080   • Benchmark and standardization. For these three tasks with different investigation levels by
081     the community, we provide different benchmark and standardization achievements. For the well-
082     investigated traffic prediction task, we construct a new large-scale benchmark. with 31 cities. This
083
        contains much more comprehensive data than the 6 previously commonly-used datasets (from
        PEMS and METR-LA) and further covers more challenging in-city scenarios. For traffic policy
084
        control task, we release 6 city datasets with each city containing at most 18,948 intersections. This
085
        scale is much larger than the datasets used by most of the previous studies. While for the city
086     boundary detection task, we clearly define this novel problem which may attract more people to
087     develop new methods in this field. These data will largely push the three fields forward on city-scale
088     modeling and generalizability to various cities.

      3     OSM+: R EADY- TO -U SE W ORLD - WIDE ROAD N ETWORK DATABASE

148   To solve the aforementioned problems, we propose OSM+ (OpenStreetMap Plus), a structured
149
      road network computing engine. It is composed of three components: (1) a road network graph

150

151
      efficient node query and distance query; (3) a series of data converters designed to customize the road
152   network data for typical urban research problems.

232   3.2   E FFICIENT Q UERY API S
233
234   We introduce how to use cloud computing resources to enable efficient parallel query of the OSM+
235   data. In comparison to directly downloading data from the OpenStreetMap API, using the OSM+
236
      dataset can save approximately 4 days of preprocessing and cleaning time for global data. Meanwhile,
      we implement several simple optimization techniques, "window check" and "k-d tree" as examples.
237
      Researchers are welcome to implement their own optimization tricks.
238
239   Point Query Operation Point query operation is one of the most basic query operations. Given a
240   point and a radius, select all points within a given radius around the given point in osm_node. A
241   simple method is to traverse all other points in the dataset and filter the points that meet the conditions.
242
      However, doing so will result in a huge amount of calculations. "window check" technique first
      divides the geographic space evenly into several grids according to a certain accuracy, and then
243
      only needs to retrieve data points in adjacent grids during retrieval. We conduct 1,000 point queries
244
      with/without "window check" on cloud computing infrastructure. The results in Table 3 show that
245   "window check" can significantly improve the efficiency of point query.
246
247   Nearest pair query and k-Nearest pair query Given a point and a latitude and longitude range, the
248
      aim is to find the nearest point or points in this latitude and longitude range. We apply k-d tree, a binary
      tree that represents a division of k-dimensional space to reduce the time complexity. Constructing
249
      a k-d tree from OSM+ dataset is equivalent to continuously dividing the k-dimensional space with
250
      hyperplanes perpendicular to the coordinate axis to form a series of k-dimensional hyperrectangular
251   regions. Since OSM+ dataset has huge amounts of records but low dimension (which means n >> d),
252   the optimal time complexity of k-d tree to find the nearest neighbor is O(log2 n). As shown in
253   Table 3, the k-d tree technique can significantly reduce the running time needed for this query.
254

      4.1     C ITY- SCALE T RAFFIC P REDICTION AND T RAFFIC G ENERATION
295
296
      Traffic prediction has been investigated a lot by researchers due to its important applications. Studies
297
      usually use the famous PEMS datasets and METR-LA dataset for experiments. However, in recent
298   years, it has been noticed that newly proposed methods can hardly exhibit significant improvements
299   over existing ones. Thus, we are here to propose 31 new city-level datasets associated with traffic
300   flow data, to provide a more comprehensive benchmark for this problem. In addition to the many more
301   cities compared with previous datasets, the newly proposed datasets are different from previously
302   used datasets for several reasons as shown in Figure 5.
303
304   • Dynamic in-city Scenario: Unlike previous datasets, which may have focused on more uniform
305     highway conditions, these datasets capture a broader spectrum of in-city scenarios. This introduces
306     greater variability in the data, reflecting the diverse and dynamic nature of urban environments.
307     Such variability is crucial for developing models that can effectively handle the complexities and
308     unpredictability inherent in city traffic patterns and infrastructure.
309
310   • Sparsity Challenge: The number of sensors is relatively low compared to the road intersections
311
        of segments. Modeling sparsity is critical as it mirrors real-world conditions where data points
        can be irregular or missing. Addressing sparsity effectively can significantly improve the accuracy
312
        and reliability of the model, ensuring it performs well even in less-than-ideal data conditions.
313
        This aspect of the dataset pushes the boundaries of current modeling techniques, encouraging the
314     development of more sophisticated and resilient algorithms.
315
316
      To evaluate the performance of OSM+ datasets, we test 7 frequently-cited baseline methods on these
317
      31 cities. We test these algorithms on three different prediction horizons (3,6,12) following the widely
318   used setting. Due to the limited space, the average results (over different prediction horizons) are
319   shown in Table 5. It is easy to observe that these methods perform completely differently on these
320   datasets. Compared with the benchmark results in previous papers (Shao et al., 2023), we can have
321   the following conclusions. (1) New datasets induce more variance and bring a more challenging
322   problem for these methods to work on. (2) Currently, there is no single dominating method that
323   can outperform other methods on most of the datasets. Therefore, these datasets will bring great
      stimulation for the development of this field.

      6        C ONCLUSION
527
      In this paper, we introduce a structured road network computing engine called OSM+. OSM+
528
      comprises three main components: (1) a road network graph database featuring global intersections
529
      (nodes) and road segments (links), along with supplementary map information such as points of
530   interest (POIs); (2) a series of auto-parallel fundamental computing APIs designed for efficient node
531   and distance queries, providing a foundation for more comprehensive road network analysis; and
532   (3) a collection of data converters to adapt the road network data for three typical urban research
533   problems. We present three example application scenarios: traffic prediction task, and traffic policy
534   control task and city boundary detection. The released data converters allow map data to be used
535   as a foundation for integrating multi-modal spatial-temporal data, supporting large model training
536   and accelerating scientific discovery. Especially, for traffic prediction task and traffic signal control
537   task, we release a new benchmark covering 31 cities and 6 cities respectively. These datasets will
538
      significantly advance city-scale modeling and improve the generalizability of research across various
      urban environments.

706   C.2    C ITY B OUNDARY D ETECTION
707
708   This section mainly describes the process of using OSM+ data for city boundary detection. First, we se-
709   lect several representative areas for experiment, including China Yangtze River Delta (115°E∼122°E,
710   29°N∼32°N), New York in USA (84°W∼70.5°W, 37.8°N∼45.6°N), Central Europe (2°W∼11.5°E,
711   45.5°N∼53°N), Nigeria in Africa (0.5°E∼14°E, 4.2°N∼11.7°N). We integrate the roadnet data,
712
      raster image population data, and raster image nightlight data of these areas, by converting them into
      point-wise data. Next, we perform clustering on the road net data, nightlight data, and population data
713
      respectively, and then weigh and sum the results of clusterings to obtain a new clustering result. After
714
      that, we divide the entire map into a grid structure with a size of 0.05° in latitude and longitude. We
715   then count the number of points of each type within each grid and assign the grid to the category with
716   the most points, obtaining a rough boundary. Finally, we examine the empty grids by considering
717   a grid nine times its size centered on it and reclassifying it to make the boundary smoother. This
718   process allows us to discover a more reasonable city boundary.

733   Crossformer (Zhang & Yan, 2022) When implementing Crossformer, we set segment length Lseg
734   to 24, as it is related to both the model performance and computation efficiency. Besides, we set the

736
      layer, there are 64 units and the initial learning rate is set to 0.003. Besides, the maximum step of
737
      random walks, i.e., K, is set to 2. For scheduled sampling, the thresholded inverse sigmoid function
738   is used as the probability decay:
