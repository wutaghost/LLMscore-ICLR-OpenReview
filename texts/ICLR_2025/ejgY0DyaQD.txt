009                                              A BSTRACT
010
011            The process of utilizing deep neural architectures to solve tasks differs signifi-
012            cantly from conventional programming due to its complexity and the need for spe-
013            cialized knowledge. While code generation technologies have made substantial
014            progress, their application in deep learning programs requires a distinct approach.
015            Although previous research has shown that large language model agents perform
016            well in areas such as data science, neural architecture search, and hyperparameter
017            tuning, the task of proposing and refining deep neural architectures at a high level
018
               remains largely unexplored. Current methods for automating the synthesis of deep
               learning programs often rely on basic code templates or API calls, which restrict
019
               the solution space to predefined architectures. In this paper, we aim to bridge the
020
               gap between traditional code generation and deep learning program synthesis by
021            introducing the task of Deep Learning Program Design (DLPD), a task of design-
022            ing an effective deep learning program for the task, along with appropriate archi-
023            tectures and techniques. We propose Deep Ones, a comprehensive solution for
024            DLPD. Our solution includes a large-scale dataset and a lightweight benchmark
025            specifically designed for DLPD. On our benchmark, Llama-3.1 8B, fine-tuned on
026            our dataset, demonstrates better architecture suggestion capability than GPT-4o
027            and better performance than Claude-3.5-Sonnet, showcasing that Deep Ones ef-
028            fectively addresses the challenge of DLPD. Deep Ones will be publicly available,
029
               including the dataset, benchmark, codes, and model weights.

      1   I NTRODUCTION
033
      Program synthesis, the process of automatically generating software from high-level specifications,
034
      has gained significant attention due to its practicality. With recent advancements in deep learn-
035
      ing and large language models (LLMs), many studies have proposed models capable of generating
036   source code through pre-training (Wang et al., 2021; Di et al., 2023; Li et al., 2023), additional
037   fine-tuning (Austin et al., 2021; Chen et al., 2021), evolutionary algorithm (Luo et al., 2023), or
038   reinforcement learning (Le et al., 2022; Shojaee et al., 2023). These methods have shown strong
039   performance in code generation benchmarks such as APPS (Hendrycks et al., 2021), MBPP (Austin
040   et al., 2021), and HumanEval (Chen et al., 2021). However, since these benchmarks primarily ad-
041   dress relatively simple code generation tasks, recent efforts have shifted toward generating more
042   complex programs, such as competition-level codes (Li et al., 2022b; Ridnik et al., 2024), data
043   science programs (Lai et al., 2023; Chandel et al., 2022), class-level codes (Du et al., 2023), and
044
      repository-level codes (Zhang et al., 2023a).
045   Nevertheless, such approaches have not been actively investigated within the domain of deep learn-
046   ing programs, which involves generating an executable code that utilizes a deep neural architecture.
047   This is due to several open challenges: complex code structures, sophisticated environment config-
048   urations, and poorly-defined evaluation standards.
049   As a result, most prior research in this area has focused on problem-solving API usage (Shen et al.,
050   2024; Patil et al., 2023; Ge et al., 2024; Gao et al., 2023; Liang et al., 2023), automated architec-
051   tural modification of a pre-defined code or hyperparameter tuning (Huang et al., 2023; Zhang et al.,
052   2023c; Liu et al., 2024), and neural architecture search (Elsken et al., 2019) along with AutoML (He
053   et al., 2021). However, these approaches often operate within a limited solution space, relying on
      slight modifications of predefined architectures. While using a base model guarantees performance


                                                        1

054
      with relatively safe execution, it leaves little room for architectural designs or technical enhance-
055   ment typically made by human researchers, which may boost performance much greater than layer
056   changing or hyperparameter tuning.
057
058
      To bridge the gap between low-level code generation technologies and high-level deep learning ar-
      chitecture usage, we propose the task of Deep Learning Program Design (DLPD), a task of designing
059
      an effective deep learning program for the task utilizing appropriate architectures and techniques.
060
      Additionally, to cope with the aforementioned challenges, we present DeepOnes, a comprehensive
061   solution for DLPD. DeepOnes consists of a large-scale dataset, a multiple-choice QA benchmark,
062   and a lightweight benchmark specifically tailored for evaluating the program design capabilities. We
063   coin these components as DeepData, DeepQA, and DeepBench, respectively.
064
      For effective program design, we assume that large language models must possess extensive knowl-
065
      edge of various architectures and techniques for flexible improvement. However, to the best of our
066
      knowledge, no existing dataset comprehensively covers deep neural architectures and their asso-
067   ciated techniques. To address this gap, we introduce DeepData, a novel dataset comprising rich
068   information extracted from arXiv papers and corresponding implementations on GitHub. Inspired
069   by biomolecular knowledge tasks from Mol-Instructions (Fang et al., 2023), we organize the data
070   into various categories, including description generation, combination prediction, property predic-
071   tion, reasoning, mathematical expression, name guessing, and more. We further process the data for
072   DLPD, by articulating the tasks of requirement-based model suggestion, property-based improve-
073   ment suggestion, and hyperparameter prediction. For a 0.01% subset of these papers, we also collect
074   multiple-choice questions to evaluate the knowledge of current LLMs in the domain of deep learning
075
      techniques.
076   To further evaluate models and establish benchmarks, we present DeepBench, a benchmark that
077   consists of 10 deep learning tasks collected from Papers with Code (PWC)1 , spanning text, image,
078   and audio modalities. DeepBench evaluates a model’s program designing ability by generating
079   a fully executable deep learning program based on the given design. The benchmark utilizes a
080   generate-then-improve framework to evaluate if LLM can truly make an appropriate architectural
081
      or technical improvement, not merely repeating the existing solution.
082   In summary, our contributions are as follows:
083
084             • To bridge the gap between low-level code generation technologies and high-level deep
085
                  learning architecture usage, we propose the task of deep learning program design and Deep-
                  Ones, a comprehensive solution to this task.
086
087             • We introduce DeepData, the first dataset tailored for the task of DLPD. This includes syn-
088
                  thetic data created from research papers and corresponding GitHub repositories, augmented
                  using LLMs.
089
090             • From a small portion of research papers used for DeepData, we collect DeepQA, the first
091
                  multiple choice question-answering benchmark for the topics on artificial intelligence. We
                  evaluate several open-source and closed-source LLMs on DeepQA to analyze the amount
092
                  of knowledge they possess and we show that the model trained on DeepData outperforms
093
                  all the other baselines.
094
                • We create the DeepBench benchmark, comprised of 10 general deep learning tasks across
095
                  several modalities, collected from Papers With Code. This includes the pipeline that syn-
096
                  thesizes a fully-executable deep learning program from a natural language task description.
097
                • We release all the datasets, codes, model weights, and benchmark so that the open-source
098
                  community can make improvements in the field of DLPD.

146
      Langchain, while AssistGPT (Gao et al., 2023) focuses more on utilizing predefined tools. Recent
147   researches even attempt to automate the data-driven discovery (Gu et al., 2024; Majumder et al.,
148   2024; Guo et al., 2024), but they focus on data science analysis rather than the deep learning ar-
149   chitecture itself. No study on automated machine learning focuses on designing or modifying the
150   architecture itself, mostly taking advantage of retrieval or API.

224   3.2   G ITHUB R EPOSITORY
225
226   In addition to research papers, we have used the Github API to retrieve repositories linked from the
227
      papers. From each repository, we extracted only functions and classes, assuming that the contribu-
      tions proposed in the papers are mostly implemented as functions or classes. To reduce excessive
228
      number of tokens, we further filtered the functions and classes using GPT-4o-mini to identify only
229
      classes and functions that are potentially relevant to the paper’s abstract. We make a mapping be-
230   tween the extracted codes and each node in a diagram using GPT-4o-mini, For example, the node
231   <MODEL>CNN may be mapped to the function def CNN. Finally, GPT-4o-mini converts the class
232   or function into pseudocode, focusing on high-level functionality of it. This is because raw code
233   snippets include a lot of noises, which does not relate to the main functionality and make a relatively
234   small LLM hard to learn from it.
235
236   3.3   P REPROCESSING FOR A RCHITECTURE D ESIGN
237
238   Using the data collected from research papers, we created an additional synthetic dataset tailored
239
      for DLPD. Neural architectures are often developed by combining, modifying, or replacing exist-
      ing components based on their properties. For example, ResNet (He et al., 2016) improved the
240
      performance of CNNs by introducing residual connections.
241
242   Motivated by this idea, we categorized the program designing task into three subtasks: (1) propos-
243   ing existing architectures based on requirements (e.g., GPU, time, or task), (2) modifying archi-
244   tectures based on component properties, and (3) selecting appropriate hyperparameters. The first
245
      dataset consists of requirement-model pairs, the second of component-property pairs (e.g., “residual
      connections improve performance and reduce overfitting”), and the third of model-hyperparameter
246
      pairs. The examples are available on the Figure 3.
247
248   As a result, DeepData includes two types of data pairs to fine-tune models. We first fine-tune the
249   LLMs on instruction-style data to inject enough background knowledge on AI-related technologies.
250   Then, we further fine-tune using DLPD-style data to train it to effectively perform program design
251
      and even generate corresponding pseudocode which guides the programmer model to implement it.
252

      4.3.1    TASK S PECIFICATION
357
358   To assess the model’s ability to design programs, we generate a fully executable implementation
359   of the proposed design, which we refer to as deep learning program synthesis. The task of deep
360   learning program synthesis can be formally defined as:
361
362                                              P, S = M (T )                                          (1)
363
      where the program P and the metric score S is generated from the natural language description of
364
      the task T .
365
366   In our evaluation pipeline, we break down this mapping into three key stages: requirement-based
367   architecture proposal, architecture improvement, and evaluation. Each of these steps is explained
368
      below.
369
370   4.3.2    R EQUIREMENT- BASED A RCHITECTURE P ROPOSAL
371
      In practical scenarios where we want to apply deep learning programs, the task is not the only
372
      consideration; we often face constraints related to GPU resources and time. The initial step involves
373   proposing a base model that can address the task while adhering to the constraints of time and
374   GPU availability. Thus, we provide the designer model MD with the task to be solved and the
375   relevant constraints regarding GPU and time. Based on this information, MD determines the most
376   appropriate backbone model architecture AB to use. This process is formalized as:
377
                        AB = MD (T ) where T = {task description, requirements}                         (2)

      4.3.3      A RCHITECTURE I MPROVEMENT
379
380   Proposing an architecture is appropriate for evaluating the model’s proposal capability, but not for
381   program designing capability. We instruct MD to modify AB to enhance performance and efficiency,
382   resulting in an improved architecture AI . This approach mirrors the typical process used by most
383   researchers, where they modify existing models to their own uses. This is expected to have a more
384
      significant impact than simple layer modifications or hyperparameter tuning performed in previous
      works. Additionally, our models generate reference Python code or Python-style pseudocode Cr to
385
      facilitate the next step of implementation. This process is formalized as:
386
387
                                               AI , CR = MD (AB )                                      (3)
388   4.3.4      A RCHITECTURE I MPLEMENTATION
389
390   Using the detailed description of the improved architecture, the programmer model MP tries to
391   implement it by writing an executable Python code C0 . Since testing the validity of C0 with
392   the entire dataset can take a long time, we start with a small subset of the data for training and
393
      run just one epoch training. This lets us quickly check whether C0 is written correctly and can
      run without issues. If C0 is invalid, MP goes through a process of iterative debugging, where
394
      each step is labeled as i, to produce the i-th debugged implementation Ci . We provide rich
395
      information for debugging, including the original task description T , error-causing code Ci−1 ,
396   standard output Oi−1 , standard error Ei−1 , and debugging log Li−1 . The debugging log is created
397   and updated for every debugging step with a simple prompt summarize the problem and
398   your solution in one line natural language sentence, like Syntax
399   Error -> changed the line ‘print("hello world"‘ to ‘print("hello
400   world")‘. This memory prevents MP repeating the same debugging which does not resolve
401   a problem. Once a numeric score Si is successfully recorded in the log file, we consider Ci is
402   validated. This process can be expressed as:
403                                   C0 = MP (T, AI ) (Implementation)                               (4)
404                   Ci , Si , Li = MP (T, Ci−1 , Oi−1 , Ei−1 , L1 , ..., Li−1 ) (debugging)         (5)
405
406   4.3.5      P ROGRAM E VALUATION
407
      After validation, MP updates the code to use the full dataset and the best hyperparameters based
408
      on the recommendations from MD regarding the set of optimal hyperparameters HO . As a result,
409   we obtain a correctly implemented architecture along with the training and testing code, namely CO
410   that includes suitable hyperparameters. In addition, we emphasize the time requirements so that the
411   code stops training and starts evaluation after the predefined time limitation is past.
412                       HO = MD (T, AI , Ci ) (Hyperparameter Recommendation)                       (6)
413                           CO = MP (Ci , HO , T ) (Complete Evaluation Code)                       (7)

      5.1      M ODEL T RAINING
419   We use Llama3.1-8B as our base model. We train Llama3.1-8B on DeepData, with two-stage train-
420   ing. On the first stage, we train it on a general instruction data. On the second stage, given that
421   the model has sufficiently learned AI-realated knowledge, we further train it on DLPD data, which
422   includes requirement-based architecture proposing, property-based architecture improvement, and
423   hyperparameter prediction. For instruction-tuning stage, we have used the batch size of 16, while for
424   DLPD-tuning stage, we have used the batch size of 4 due to long data. In addition, for DLPD-tuning
425   stage, we replayed randomly sampled 1% of instruction data to prevent catastrophic forgetting. For
426   both training stages, we trained the models for 2 epochs, using 1 percent of the dataset for warmup
427
      steps, learning rate of 3e-4, cosine learning rate decay, and Adam-mini (Zhang et al., 2024) opti-
      mizer. This took around 3 days on 4 NVIDIA A6000 GPUs. Furthermore, to accelerate training and
428
      inference speeds and to reduce the memory usage of LLMs, we have applied several techniques. We
429
      employed LLaMA-Factory (Zheng et al., 2024), flash attention 2 (Dao, 2023), unsloth5 , and lora
430   plus (Hayou et al., 2024) for acceleration.

      Implementation’s Correspondence to the Proposed Program Design Although the quality of
514
      designed architecture is reasonable, the Table 3 shows that the best case occurs mostly using classi-
515
      cal models, like BERT variants. This is mostly due to the discrepancy of implementation capabilities
516   of the programmer model, GPT-4o. As shown in the Figure 4 suggests, GPT-4o already has substan-
517   tial knowledge in AI-related technologies. Nevertheless, we observed that GPT-4o lacks knowledge
518   on implementations of several recent techniques. For example, GPT-4o fails to apply common sug-
519   gestions from DeepLlama, such as Mamba, LLaVA, or Llama. This affects DeepLlama negatively,
520   as its initial suggestions are more relevant to cutting-edge technologies. Thus, while DeepBench
521   effectively evaluates the model’s ability to understand the architecture and generate appropriate im-
522   provements, the evaluation on architectural proposals appears limited.

525   6          C ONCLUSION AND L IMITATIONS
526
527   In this paper, we introduce a comprehensive solution for the task of deep learning program design,
528   by proposing a novel dataset and two benchmarks. Through the evaluation on DeepBench, we
529   showcase that DeepData is effective for training a LLM to obtain a broad knowledge on architectures
530   and techniques. Along with a quantitative and qualitative analysis and open-sourcing, we believe
531   that this contributes to the more active research on the task of deep learning program design.
532
533
      Limitations Even we have shown DeepData and DeepBench’s strength, there remains some limi-
534   tations to be resolved. First, as mentioned in the qualitative study, GPT-4o often fails in implement-
535   ing recent knowledge, even though it possesses one of the best code generation capabilities. This
536   may lead to a distorted evaluation of the model’s capability on architecture proposal. In addition, we
537   rely on a closed-source models for code generation, which is extremely costly. In future works, we
538   would like to suggest lightweight open-source models that can replace GPT-4o in deep learning pro-
539   gramming task. These limitations pose the need for more investigations on the task of deep learning
      program design.
