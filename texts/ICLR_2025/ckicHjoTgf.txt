009                                              A BSTRACT
010
011            In recent years, the design of Artificial Intelligence (AI) accelerators has gradu-
012            ally shifted from focusing solely on standalone accelerator hardware to consid-
013            ering the entire system, giving rise to a new AI accelerator design paradigm that
014            emphasizes full-stack integration. Systems designed based on this paradigm offer
015            a user-friendly, end-to-end solution for deploying pre-trained models. While pre-
016
               vious studies have identified vulnerabilities in individual hardware components
               or models, the security of this paradigm has not yet been thoroughly evaluated.
017
               This work, from an attacker’s perspective, proposes a threat model based on this
018
               paradigm and reveals the potential security vulnerabilities of systems by embed-
019            ding malicious code in the design flow, highlighting the necessity for protection to
020            address this security gap. In exploration and generation, maliciously leverage the
021            exploration unit to identify sensitive parameters in the model’s intermediate lay-
022            ers and insert hardware Trojan (HT) into the accelerator. In execution, malicious
023            information is concealed within the control instructions, triggering the HT. Exper-
024            imental results demonstrate that the proposed method, which manipulates sensi-
025            tive parameters in a few selected kernels across the middle convolutional layers,
026            successfully misclassifies input images into specified categories with high mis-
027
               classification rates across various models: 97.3% in YOLOv8 by modifying only
               three parameters per layer in three layers, 99.2% in ResNet-18 by altering four
028
               parameters per layer in three layers and 98.1% for VGG-16 by changing seven pa-
029
               rameters per layer in four layers. Additionally, the area overhead introduced by the
030            proposed HT occupies no more than 0.34% of the total design while maintaining
031            near-original performance as in uncompromised designs, which clearly illustrates
032            the concealment of the proposed security threat.

035   1   I NTRODUCTION
036
037   Recent research has revealed that AI accelerator are inherently vulnerable to various attacks, high-
038   lighting security as a significant concern. These vulnerabilities stem from multiple sources: 1) The
039
      models themselves, which allows attackers to induce malfunction through minor perturbations to the
      inputs (Baniecki & Biecek, 2024; Costa et al., 2024). Besides adding perturbations to the inputs, at-
040
      tackers can also control the model’s behavior by attacking the model parameters (Rakin et al., 2019;
041
      2021; Bai et al., 2023). 2) The accelerator hardware architecture, such as memory, is susceptible
042   to attacks such as the Row Hammer Attack (Kim et al., 2014). Attackers can also exploit timing
043   violations to cause the entire design to malfunction (Liu et al., 2020b; Mukherjee & Chakraborty,
044   2022). 3) Security vulnerabilities during the design synthesis process, such as attackers tampering
045   with the Look-Up Tables (LUTs) in Field-Programmable Gate Array (FPGA) (Nozaki et al., 2020;
046   Krieg et al., 2016).
047
      Lately, the research focus on AI accelerators has gradually shifted from solely focusing on AI ac-
048   celerators to encompassing the entire System on Chip (SoC) and its related software stack, giving
049   rise to a new AI accelerator design paradigm (Xilinx, 2024; Genc et al., 2021). The yellow part in
050   Fig. 1 provides a sketch of this paradigm, which centers not only on optimizing the performance
051   and efficiency of accelerators but also on the deep co-design between hardware and software, as
052   well as their integration at the system level to address increasingly complex AI workloads. Key
053   components of this paradigm include exploration unit (e.g., Design Space Exploration (DSE)) and
      middleware. DSE uses exploration algorithms to evaluate various design parameters to optimize

085   This research aims to propose a generic threat model targeting the AI accelerator design paradigm.
086   Under this model, the system analyzes user-provided pre-trained models to generate the optimal
087   hardware and software stack. Concurrently, an attacker exploits malicious code embedded in the
088   hardware generation flow to analyze model vulnerabilities and insert a hardware Trojan (HT). In par-
089   allel, the software generation flow is compromised, allowing the injection of malicious instructions
090   into the communication protocol between the controller and the accelerator, effectively triggering the
091   HT. The code for this research is available at https://github.com/AnonymousCode-HT/C-SFE.git.

093   generation platforms is proposed. It explores how attackers can integrate malicious code into plat-

      ward propagation without gradient information. It targets intermediate convolutional layers, and
099
      parameters identified by C-SFE exhibit regularity, making it more compatible with HT design. 3)
100
      The proposed threat model was validated on the state-of-the-art automated AI accelerator generation
101   platform, and the actual attack effectiveness on the YOLOv8 classification model, ResNet-18, and
102   VGG-16 was tested on FPGA.

105   2   P RELIMINARIES
106
107   Hardware Trojans are malicious circuits that are secretly inserted into a circuit. In general, they
      remain silent and are triggered only at certain moments, causing a serious impact on the functionality


                                                        2
      Under review as a conference paper at ICLR 2025


108
      of the circuit. The hardware Trojan usually consists of a trigger and a payload, during normal circuit
109   operation, the trigger will monitor a certain (or some) signal, when these signals reach a specific
110   condition, it will control the payload into working state, the payload is responsible for the specific
111   attack. Many different types of hardware Trojans can be created based on physical, activation, and
112   behavioral characteristics (Xue et al., 2020).
113
      AI accelerator design paradigm focuses on optimizing the performance, energy efficiency, and
114
      scalability of deep neural network workloads through specialized hardware architectures. It em-
115   phasizes the co-design of hardware and software to support a wide range of applications, from
116   high-performance computing to low-power edge devices, leading to a significant number of tunable
117   parameters. As a result, exploration unit within the paradigm, such as DSE, become essential for
118   optimizing accelerator performance and energy efficiency, helping designers choose the best con-
119   figurations. Lately, many researchers have explored this design paradigm, with notable examples
120   such as Gemmini (Genc et al., 2021) and NVDLA (Zhou et al., 2018). Gemmini is a full-stack
121   DNN accelerator generator featuring: 1) Flexible hardware parameterization supporting various
122   dataflows (e.g., weight stationary, output stationary). 2) Integration with the Chipyard framework
123
      (Amid et al., 2020), enabling tight coupling with RISC-V Rocket cores via the Rocket Custom Co-
      processor (RoCC) commands. 3) A multi-layer software stack based on ONNX Runtime (Microsoft,
124
      2024), providing an easy-to-use interface for end users and low-level control via C/C++ for system
125
      programmers. Some researchers have also integrated NVDLA into the Chipyard framework to pro-
126   vide a unified approach to accelerator generation (Gonzalez & Hong, 2020; Farshchi et al., 2019).
127   However, inconsistencies in communication interfaces remain. For example, Gemmini uses RoCC
128   commands to communicate with RISC-V Rocket cores, while NVDLA relies on Memory-Mapped
129   I/O (MMIO). The optimization and exploration algorithms in these automatic accelerator genera-
130   tion platforms might be maliciously used for HT insertion; however, this security risk has not been
131   investigated to date.

      3.1   T HREAT M ODEL
136
137   As shown in Fig. 1, the proposed method belongs to the category of grey-box attacks. It is assumed
138   that the adversary is the developer of the AI accelerator platform, and the platform’s level of open-
139   ness is similar to Vitis AI (Xilinx, 2024). This means that due to commercial purposes, its internal
140   exploration unit such as DSE are not made public. Therefore, the platform is transparent to the user,
141   and the design output from the ’VLSI design flow’ is also transparent. However, the user can control
142   the accelerator’s behavior through the controller’s instructions, such as reading data from a spec-
143
      ified location in memory into the accelerator and performing matrix multiplication or convolution
      operations.
144
145   During the design and generation phase, since the platform receives information about the model
146   structure and parameters (i.e., ’User provides model’ in Fig. 1), the adversary would also have
147   this information. Additionally, the adversary cannot access the training set but can access a small
148   dataset for validation purposes for two key reasons. First, because floating-point data is inefficient in
149
      terms of both computational performance and hardware resource utilization, quantization methods
      are extensively employed during the design space exploration process. In our approach, we employ
150
      the Post-Training Quantization (PTQ) method (included in the ’Exploration unit’), which in turn
151
      necessitates the use of a compact validation dataset. Second, after the quantization process, the
152   quantized model requires a small subset of input data to determine if there is a significant difference
153   in model accuracy compared to the floating-point scenario.

      Attackers can insert malicious exploration algorithms into it, or, to reduce the size of the malicious

185   The workflow and attack process are as follows (see Appendix A.1 for a simplified representation):
186
      1) The user intends to deploy their trained model onto domain-specific hardware (e.g., edge devices
      focusing on low-power applications). Therefore, they provide the trained model, calibration input
187
      images, and requirements (e.g., energy-efficiency priority). 2) The Exploration unit configures the
188
      optimal design parameters based on the user’s requirements. Simultaneously, the embedded mali-
189   cious code leverages the exploration algorithm to locate target model parameters (e.g., parameters
190   for executing an N-to-1 attack). 3) Using the optimal design parameters configured by the Explo-
191   ration unit in the previous step, the VLSI design flow automatically generates the RTL code for the
192   SoC (including the controller and accelerator), inserts the HT into the accelerator, and then produces
193   a bitfile (for FPGA) or layout (for ASIC) based on user requirements for delivery. 4) The software
194   stack is adapted to the generated design, enabling the user to easily control the accelerator. During
195   the generation process, malicious code is inserted so that when the control instructions are sent to
196   the accelerator, they carry the information of the target parameters to be attacked.
197   There are various methods to trigger the attack, such as a timing trigger (i.e., the malicious code
198   embedded in the software stack activates at a specific interval, such as once a week), a random
199   trigger (i.e., activation occurs when a generated random number matches a predefined value), or
200   a targeted trigger (i.e., activation after continuously detecting a specific category). As shown on
201   the right side of Fig. 1, once the HT embedded within the accelerator is triggered, it executes
202   attacks such as a 1-to-1 attack, where a specific class is misclassified into an attacker-specified
203
      class, or an N-to-1 attack, where all classes are misclassified into the attacker-specified class. The
      information necessary for the attack is concealed within control instructions, and the specific effects
204
      of the attack are determined by the malicious algorithm embedded in the exploration unit. In the
205
      following section, we take Gemmini as an example to demonstrate how this threat model operates in
206   a real AI accelerator generation platform. Notably, our approach is broadly applicable to any similar
207   automation platform.

      3.4    C ROSS - LAYER S ENSITIVE F ILTER E XPLORATION
291   The proposed C-SFE algorithm addresses the challenges outlined in Section 3.3. By focusing on
292   the intermediate convolutional layers and avoiding both the initial convolutional and FC layers, C-
293   SFE expands the decision boundary of a targeted category by flipping a few bits in one kernel per
294   layer, enabling an N-to-1 attack. C-SFE is structured into two phases: The first phase, kernel-level
295   exploration, primarily identifies sensitive kernel locations related to the specified category. The
296   second phase, bit-level exploration, focuses on determining how to achieve the desired attack effect
297   with the fewest bit flips, based on the selected kernels.

299   (GA) and PSO. The following example uses GA, assuming N kernels of size k × k need to be

474
475
476   In this paper, we present a novel threat model for AI accelerator design paradigms, revealing po-
477
      tential security vulnerabilities. Specifically, we maliciously leverage the exploration unit to locate
      sensitive model parameters, embed hardware Trojans through the code generation module, and trig-
478
      ger them using malicious information hidden in communication instructions. We also highlight the
479
      challenges of implementing this threat model, as previous attack algorithms are not well-suited for
480   it. To address these challenges, we propose a tailored attack algorithm, C-SFE. The effectiveness
481   of the proposed threat model is validated on the Gemmini accelerator generation platform, and the
482   attack performance was tested on three different models: VGG-16, ResNet-18, and YOLOv8m-cls.
483   For each model, 4 kernels in VGG-16, 3 kernels in ResNet-18, and YOLOv8m-cls were attacked,
484   with corresponding total parameter modifications of 28, 12, and 9, respectively. This resulted in
485   over 98%, 99%, and 97% of the inputs being classified into the attacker’s specified category for each
      model.

750   We utilized the five commonly used defense methods introduced in Table 5 to evaluate the stealth-

753   1) Model-based methods enhance resistance to attacks by adjusting and optimizing the model it-
754   self. In our experiments, we retrained the model for 40 epochs using Weight Clustering (He et al.,
755   2020) and then re-applied the proposed attack process. Experimental results show that achieving the

756
      layer. Therefore, while this approach slightly mitigates the attack’s impact, it remains insufficient to
757   effectively defend against the proposed threat model.
758
759   2) Machine learning-based detection of RTL is limited to a finite dataset. Taking the Trust-Hub
760
      dataset (Trust-Hub, 2024) as an example, HW2VEC (Yu et al., 2021) convert benchmarks into the
      form of Data-Flow Graph (DFG) or Abstract Syntax Tree (AST) representation, and then use Graph

      Convolutional Networks (GCN) to train data. The training dataset includes the PIC16F84 micro-
762
      processor and RS232 serial port. As proof, We input the malicious RTL code as test data into the

764
765   3) Hash or checksum computation (Li et al., 2021), (Liu et al., 2020a), (Guo et al., 2021), which
766
      utilize hash or checksum values of off-chip memory (e.g., DRAM), have been proven effective in de-
      tecting attacks like BFA and T-BFA. These attacks specifically aim to alter bits in off-chip memory,
767
      therefore the use of hash functions makes it a reliable method for detecting DRAM-based attacks.
768
      However, the proposed HT resides within the accelerator’s logic unit and targets only parameters
769   stored in the on-chip memory, leaving those in off-chip memory unaffected. It is also worth noting
770   that, the insert position of hardware Trojans is much more flexible making it challenging to deter-
771   mine the optimal location for inserting the hash computation unit. Consequently, the proposed threat
772   model can evade detection by such verification methods.
773
      4) Test vectors (Chakraborty et al., 2009), (Saha et al., 2015) are effective in detecting HTs lo-
774
      cated near the output. For example, in encryption chips, HTs are often deployed along key-related
775   paths and positioned near the output. However, the proposed Payload design executes the attack by
776   modifying internal model parameters without directly impacting the output. This greatly reduces
777   the detection success rate of these methods, potentially rendering them incapable of identifying the
778   attack.
779
      5) Information-Flow Tracking (Hu et al., 2016) monitors the flow of data within a system to identify
780
      and prevent unauthorized data transmission. However, for CNNs, it is challenging to determine
781   which parameters need to be tracked. Additionally, using this technique to track an excessive number
