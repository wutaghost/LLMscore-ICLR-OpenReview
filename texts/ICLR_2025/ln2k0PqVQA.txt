009                                             A BSTRACT
010
011           Automatically synthesizing dense rewards from natural language descriptions
012           is a promising paradigm in reinforcement learning (RL), with applications to
013           sparse reward problems, open-ended exploration, and hierarchical skill design.
014           Recent works have made promising steps by exploiting the prior knowledge
015           of large language models (LLMs). However, these approaches suffer from
016           important limitations: they are either not scalable to problems requiring billions
017           of environment samples, due to requiring LLM annotations for each observation,
018           or they require a diverse offline dataset, which may not exist or be impossible
019
              to collect. In this work, we address these limitations through a combination
              of algorithmic and systems-level contributions. We propose ONI, a distributed
020
              architecture that simultaneously learns an RL policy and an intrinsic reward
021
              function using LLM feedback. Our approach annotates the agent’s collected
022           experience via an asynchronous LLM server, which is then distilled into an
023           intrinsic reward model. We explore a range of algorithmic choices for reward
024           modeling with varying complexity, including hashing, classification, and ranking
025           models. By studying their relative tradeoffs, we shed light on questions regarding
026           intrinsic reward design for sparse reward problems. Our approach achieves
027           state-of-the-art performance across a range of challenging, sparse reward tasks
028           from the NetHack Learning Environment in a simple unified process, solely using
029           the agent’s gathered experience, without requiring external datasets. We make
030
              our code available at URL (coming soon).
031
032   1   I NTRODUCTION
033
034   Reward functions are central to reinforcement learning (RL), and are often assumed to be given
035
      as part of the problem definition (Sutton & Barto, 2018). These functions are written to describe
      the task at hand, and often involve tradeoffs between ease of task definition and ease of policy
036
      optimization. For example, assigning a reward of +1 for solving the task and 0 otherwise is simple
037
      to define and accurately reflects the task goal, but is difficult to optimize due to providing zero
038   gradients almost everywhere.
039
040   These difficulties have motivated the use of intrinsic rewards to aid policy optimization (Randlov
041
      & Alstrøm, 1998; Ng et al., 1999; Sorg et al., 2010; Singh et al., 2010). The reward designer can
      include additional reward shaping terms to create a denser learning signal, which can reflect task
042
      progress or guide the agent towards intermediate goals. However, designing intrinsic rewards can
043
      be remarkably challenging (Booth et al., 2023; Ibrahim et al., 2024) and places increased demands
044   on human experts to provide task-specific knowledge.
045
046   Recently, several works have been proposed to leverage the vast prior knowledge encoded in large
047
      language models (LLMs) to automate the reward design process, based on a task description in
      natural language. They can be broadly categorized into two families:
048
049   1. Generating the reward function’s code by LLM. A number of methods have been proposed
050   to automatically generate executable code that computes the reward directly (Ma et al., 2023; Xie
051   et al., 2023; Yu et al., 2023; Li et al., 2024). While they have demonstrated success in complex
052   continuous control tasks, they either require access to environment source code to include in the
053   prompt, or a detailed description of input parameters and reward function templates. Furthermore,
      they are limited to reward functions compactly expressible via code, describing explicit logic; and


                                                      1
      Under review as a conference paper at ICLR 2025


054
      it is unclear how these approaches can easily process high-dimensional state representations such as
055   images, or semantic features such as natural language.
056
057
      2. Generating reward values by LLMs. Motif (Klissarov et al., 2023) is a typical example of this
      category. It ranks the captions of pairs of observations using an LLM and distills these preferences
058
      into a parametric reward model. Motif does not require access to environment source code nor
059
      numerical state representation, can process semantic input features, and can scale to problems
060   requiring billions of environment samples. Nevertheless, it also suffers from two important
061   limitations. First, it requires a diverse, pre-existing dataset of captioned observations which are used
062   to elicit preferences from the LLM. In many situations, such a dataset might not exist, and collecting
063   it can increase the sample complexity. More importantly, collecting a diverse dataset often requires
064   a non-trivial reward function that is feasible to optimize, which is the primary problem we aim to
065   solve with intrinsic reward functions in the first place. Second, it involves a complex three-stage
066   process, which sequentially annotates observations using an LLM, trains a reward model, and
067   finally trains an RL agent. This is still time-consuming, given that the LLM annotation process can
068
      take several days’ worth of GPU hours, and is done prior to training the reward model and RL agent.
      Alternatively, Chu et al. (2023) query the LLM to directly label observations as having high or
069
      low reward at each timestep. However, querying an LLM for every observation is computationally
070
      infeasible for many RL applications, which involve millions or billions of observations.
071
072   As a consequence, it would be desirable to have an integrated solution that offers:
073
074
      (1) concurrent and fast online learning of both the intrinsic rewards and the policy that requires no
          external data nor auxiliary reward functions,
075
076   (2) expressible reward functions that can capture semantic features that are difficult to process with
077       compact executable code.
078   In this work, we present ONI, a distributed online intrinsic reward and agent learning system. ONI
079   assumes access to captions of observations, similar to previous work (Klissarov et al., 2023; Chu
080   et al., 2023). The captions of collected observations are annotated by an asynchronous LLM server,
081   and both the policy and intrinsic reward model are simultaneously updated using LLM feedback.
082   ONI removes the dependency on external datasets beyond the agent’s own experience and enables
083   large-scale RL training with ease. Such a learning framework allows us to systematically compare
084   different algorithmic choices for synthesizing LLM feedback. Specifically, we explore three meth-
085   ods: the first one is retrieval-based and simply hashes the annotations; the second builds a binary
086
      classification model to distill the sentiment labels returned by the LLM; and the third sends pairs of
      captions to the LLM server for preference labeling and learns a ranking model, similar to Motif. By
087
      carefully comparing the three proposed algorithms, we provide valuable insights into several impor-
088
      tant questions regarding intrinsic reward design. We demonstrate that ONI is able to match Motif’s
089   performance across a range of challenging, sparse rewards from the NetHack Learning Environment
090   (NLE) (Küttler et al., 2020), solely using the agent’s gathered experience in a single, unified process.

094   We consider a partially observed Markov decision process (POMDP) setting where the problem is
095
      defined by M = (S, A, O, p0 , P, O, r, γ). At each episode, an initial state s0 ∈ S is sampled from
      the initial state distribution p0 . At each time step t, the agents observes ot ∈ O which is computed
096
      by the emission function O(st ), and takes an action at ∈ A. This action causes the environment to
097
      transition to a new state, st+1 ∼ p(st , at ). A new observation ot+1 and a reward rt+1 = r(ot+1 )
098   is given to the agent, and the process continues. The P goal of the agent is to learn a policy π : O →
099   ∆(A) which maximizes the expected return Eπ [ t γ t rt ]. In this work, we additionally assume
100   each observation ot includes a textual caption ct , which could be empty. For observation spaces
101   without textual captions, ct could in principle be provided by a captioning model as well.
102
      In many situations, the extrinsic environment reward r is sparse, and the resulting return objective
103
      is challenging to optimize. We therefore consider methods which make use of an auxiliary intrinsic
104   reward rint and define a composite surrogate reward:

134   3.1   S YSTEM DESIGN : DISTRIBUTED PPO WITH LLM ANNOTATIONS
135
      This section outlines the system we have built to learn online intrinsic rewards alongside the policy
136
      optimization. The engineering and design here is an important piece of our research, as the rest of
137   our experimental studies are conducted within this system and influenced by the throughput of its
138   interacting components.
139
140
      Our core system illustrated in Figure 3.1 is built on top of the Sample Factory library v1.0 (Petrenko
      et al., 2020) and their asynchronous variant of proximal policy optimization (Schulman et al., 2017),
141
      referred to as APPO. APPO operates on a single machine and concurrently runs many environment
142
      instances while asynchronously updating policy and value estimates with the usual PPO rules, and
143   adequately handles policy staleness and data transfers between these components. Concretely for
144   NetHack, by running 480 environment instances APPO collects approximately 32k environment
145   interactions per second on a Tesla A100-80GB GPU with 48 CPUs.
146
      For online reward learning via LLM annotations in this system1 , we added 1) an LLM server hosted
147
      on a separate node, 2) an asynchronously running process that passes observation captions to the
148
      LLM server via HTTP request, 3) a hash table that stores the captions and corresponding LLM an-
149   notations, and 4) and learning code that dynamically updates a reward model that learns on these
150   annotations. Without being asynchronous, these components have the potential to block the concur-
151   rent execution and can reduce the throughput of the overall system, because calling into an LLM and
152   updating the reward model are time-consuming. We added them in a way that retains most of the
153   throughput (!), approximately 80-95% of the original: the average throughput is 30k environment
154   interactions per second if we do not train any additional reward model to distill the LLM annota-
155   tions, and 26k if a classification-based reward model is learned at the same time (see Section 3.2
156   for the description of those approaches). The throughput of LLM annotation depends on the actual
157
      LLM and prompt we are using. For the reader’s reference, when hosting a LLaMA-3.1-8B server on
      a Tesla A100-80GB GPU using the prompt displayed in Appendix B, ONI annotates approximately

165   Background: APPO’s Distributed Execution and Shared Memory APPO has three main types
166
      of workers:
167
      1. the learner worker that coordinates most of the operations, updates the policy and value models,
168
         and sends the latest policy ID to the other workers so that they can retrieve it;
169
170   2. the rollout workers that run copies of the environment, execute actions in them and save the
171
         observations; and
172   3. the policy workers that query the policy on new states. The policy workers are separate from
173      the rollout workers so they can efficiently batch across observations.
174
175   The workers here are individual processes forked off from a parent process. These all have shared
176
      CPU and GPU memory buffers, and efficiently communicate mostly via Python’s multiprocessing
      queues, using pointers to locations in the shared memory. We next describe our new LLM worker,
177
      and how we connect it back into the learner.
178
179   Our New Asynchronous LLM Worker and Remote LLM Server This worker has input and
180   output queues that communicate back with the learner process, which we use so it does not block
181   the main execution of the system. The LLM worker awaits new observations to label from the input
182
      queue, formats them into prompts, calls into an LLM, and returns the annotations in the output
      queue. Additionally, the prompt templates and other LLM options are configured by the LLM
183
      worker. We use an annotation and message format that support all the intrinsic reward labels that we
184
      consider in Section 3.2. As Sample Factory already utilizes most of the free CPU and GPU capacity
185   of the system, we opted to call into the LLM via an HTTP/REST interface rather than loading it in
186   this process directly. The communication here adds minimal overhead, and also makes us not need to
187   coordinate the shared memory of multiple GPUs between the main APPO code and LLM. Alongside
188   every APPO run, we allocate an unloaded machine with a new instance of VLLM (Kwon et al.,
189   2023b) to exclusively serve the queries from the RL run. Our system also allows using the same node
190   for both processes if computational resources allow, which further reduces communication costs.
191   One important design decision is that the asynchronous LLM server is only able to process a small
192   percentage of the overall observations encountered 2 , because the environment rollout workers have
193   a much higher throughput than the LLM. An alternative design would be to block the main APPO
194   components and wait for the LLM to label more messages, but we find it more realistic to continue
195   running the policy and label messages as the LLM throughput allows. This also creates the problem
196   of needing to decide on what messages to send to the LLM: the last-in-first-out queue (LIFO), or
197   some uncertainty-based selection. For this work, we use LIFO for simplicity, but note that it would
198   be interesting to investigate alternate approaches in future work.
199   Our Modified Learner Lastly, we connect this new LLM worker back into the main learner, and
200   dynamically learn a reward model on the annotations obtained from it. To do this, we modified the
201   two threads of the learner worker:
202
203   1. the training thread is the main thread that a) aggregates the new trajectories from the latest
204
         environment interactions; b) decides which new observations to send to the LLM; c) updates the
         policy, value, and (now) our intrinsic reward model.
205
206   2. The feedback processing thread (in fact, the initial thread of the learner worker) receives the
207      latest annotations from the LLM worker and updates the dataset (or hash table) that the reward
208      model is trained on. This thread also initializes the reward model at the beginning.
209
      To prevent overfitting the limited amount of annotations in the early stage of training, the training
210
      thread only continuously updates the reward model after we receive 25k annotations. Before that,
211
      the feedback processing thread run a few updates of the reward model every time we receive new
212   annotations. Now that we have a flexible way of annotating messages and fitting a reward model on
213   top of them, we turn to defining the types of intrinsic reward functions.

218   ONI offers flexible algorithmic choices for querying an LLM and distilling its feedback. In this
219   work, we consider the following three methods.
220   Retrieval The simplest approach we consider uses binary labeling and retrieval. The LLM is asked
221   to assign a binary label yi ∈ {0, 1} indicating whether a caption ci is “helpful” or “unhelpful” for
222   making progress on the task. The learner worker maintains a hash table H to store labeled pairs
223   (ci , yi ), managed by the feedback processing thread. In the training thread, each time the RL agent
224   receives an observation ot with caption ct , we check if ct ∈ H and the intrinsic reward is defined as:
225
                                          int          H(ct ) if ct ∈ H
226                                      r (ot ) =                                                        (2)
                                                       0        if ct ∈

228   If ct is unlabeled, it is placed into a last-in-first-out (LIFO) queue Q managed by the LLM annotation
229   process, and then sent to the LLM server. The LLM continuously processes elements ci from Q and
230   returns their labels yi to the data processing thread of the learner worker, where the pairs (ci , yi ) are
231   added into H. As described in Section 3.1, the training thread and the feedback processing thread
232
      run asynchronously. Therefore, editing the hash table does not slow down policy training. This
      retrieval-based approach does not generalize to observations with unlabeled captions. However, the
233
      resulting intrinsic reward is simple and hyperparameter-free, and may work well when the set of
234
      captions belongs to a relatively small set.
235
236   Classification The second approach we consider is based on binary labeling together with training
237   a classification model. Similarly to above, we label observation captions ci with labels yi ∈ {0, 1}

      Exploration Based on Novelty Bonuses Learning from sparse or otherwise difficult-to-optimize
267
      reward functions is a long-standing problem in reinforcement learning. There is a large body of
268

      work which defines intrinsic rewards based on novelty bonuses (Schmidhuber, 1991; Kearns &
271   Singh, 2002; Brafman & Tennenholtz, 2002; Stadie et al., 2015; Bellemare et al., 2016; Pathak
272   et al., 2017; Burda et al., 2019; Shyam et al., 2019; Raileanu & Rocktäschel, 2020; Ecoffet et al.,
273   2019; Agarwal et al., 2020; Zhang et al., 2021; Henaff et al., 2022a; Lu et al., 2024). These methods
274   tend to make minimal assumptions about the task at hand, operate online without requiring external
275   data, and sometimes come with theoretical guarantees. However, since they are fundamentally
276   tabula-rasa, they must rediscover much of the structure in the task that might already be encoded
277   as prior knowledge in an LLM. Therefore, they tend to have difficulty exploring environments of
278   very high complexity, such as NetHack, in a tractable amount of time (Klissarov et al., 2023).
279   LLM-aided Reward Design In addition to Motif (Klissarov et al., 2023), several works have
280   sought to leverage the prior knowledge encoded in LLMs to produce intrinsic rewards. Eureka
281   (Ma et al., 2023), Auto-MC (Li et al., 2024), L2R (Yu et al., 2023) and Text2Reward (Xie et al.,
282   2023) all use LLMs to generate executable code which computes intrinsic rewards from the under-
283   lying environment state, conditioned on a task description. The generated reward function code is
284   then iteratively improved based on aggregate statistics from agents trained with the current reward.
285
      However, a disadvantage with intrinsic rewards represented as code is that they require access to an
      interpretable underlying state representation, and it is unclear how to leverage non-numerical fea-
286
      tures such as those provided by unstructured language captions. The works of Kwon et al. (2023a);
287
      Chu et al. (2023) also successfully used LLMs conditioned on task descriptions to directly generate
288   binary rewards in an online manner, and did not train a reward model. This was possible due to eval-
289   uating on environments and tasks which could be solved with a relatively small number of observa-
290   tions, whereas we consider complex open-ended environments with billions of observations so that
291   labeling them all with an LLM would be computationally infeasible. Also of note is the work of Wu
292   et al. (2024), which additionally conditioned LLMs on user manuals to define the intrinsic reward.
293
      Goal-conditioned Reward Design A different approach to reward function design is to define re-
294   wards as the distance between the agent’s current state and the goal. For example, one line of
295   work learns a state embedding in a self-supervised manner which converts geodesic distances in the
296   original space to Euclidean distances in feature space (Wu et al., 2019; Wang et al., 2021; Gomez
297   et al., 2024). Another line of work of (Fan et al., 2022; Rocamonde et al., 2023; Adeniji et al.,
298   2023; Kim et al., 2024) leverages pretrained image and text encoders, and defines rewards to be
299   some measure of similarity between embeddings of visual observations and embeddings of textual
300   task descriptions. Using a question generation and answering system, Carta et al. (2022) extract
301   auxiliary objectives from the goal description and construct intrinsic rewards. An interesting com-
302
      bination of goal-conditioned and LLM-aided reward design is the ELLM approach introduced in Du
      et al. (2023b), which generates candidate goals and uses the distance in LLM embedding space to
303
      define the reward. This approach shares the limitations of the works of Kwon et al. (2023a); Chu
304
      et al. (2023) discussed in the previous section, in that it requires an LLM call for each agent obser-
305   vation, which becomes computationally infeasible in high-throughput settings involving billions of
306   observations. We discuss more works that utilize LLM for RL broadly in Appendix D.

308   5   E XPERIMENTS
309
310   Environment We use the NetHack Learning Environment (NLE) (Küttler et al., 2020) as our
311   experimental testbed, since it is one of the most challenging open-ended, long horizon and sparse
312
      reward environments available, and was also used as the main environment in the prior work we
      compare to. NetHack is a classic dungeon crawling game which presents a number of interesting
313
      challenges for RL agents: it is procedurally generated, requiring generalization; rewards are sparse
314
      for most tasks, requiring exploration; the environment is partially rather than fully observable;
315   transitions are naturally stochastic; episodes are very long, requiring tens to hundreds of thousands
316   of steps to win the game; the dynamics are highly complex, involving large numbers monsters,
317   objects, non-player characters and other entities. Succeeding in the game requires mastering and
318   balancing diverse behaviors including exploration, resource management, object use, combat,
319   puzzle solving and skill progression.
320
      Tasks and Metrics We evaluate our agents in two ways: how well they are able to succeed in tasks
321   which have a (typically sparse) extrinsic reward, and how well they are able to progress in the game
322   using the intrinsic reward only. For the former, we use one dense reward task and three sparse reward
323   tasks used in prior work (Küttler et al., 2020; Klissarov et al., 2023), listed below.

      Methods and Hyperparameters We instantiate ONI with the three reward functions described
338   in Section 3.2. We name the three approaches ONI-retrieval, ONI-classification and
339   ONI-ranking, respectively. For policy learning, we use the Chaotic Dwarven GPT5 (CDGPT5)
340   architecture (Myffili, 2021) used in prior work (Piterbarg et al., 2023; Kurenkov et al., 2023; Klis-
341   sarov et al., 2023). Architecture details can be found in Appendix A.1. All the methods are trained
342   with two billion (2 × 109 ) environment steps. We train ONI-classification with with classifi-
343   cation threshold η = 0.5, where the intrinsic reward coefficient is β = 0.1 for the Score task and
344   β = 0.4 for all the other sparse-reward tasks and the reward-free agent. Similarly, ONI-retrieval
345   uses β = 0.1 for Score and β = 0.5 for the others. ONI-ranking is trained with β = 0.05. Full
346
      details of the training process can be found in Appendix A.2.
347   LLMs We use the LLaMA-3 herd of models (Dubey et al., 2024) as our LLMs. All prompts are
348   listed in Appendix B. We initially reran the Motif baseline using the official code5 and compared
349   the performance of LLaMA-3.1-70B-Instruct and LLaMA-3.1-8B-Instruct (see Appendix C.1).
350   We did not observe a significant difference in their performance on the Oracle or Score tasks.
351   Therefore, we use LLaMA-3.1-8B-Instruct in our subsequent experiments to reduce computation.
352   Baselines We compare to agents trained with extrinsic reward alone, Motif (Klissarov et al., 2023)
353   , and a variant of the ELLM algorithm (Du et al., 2023a). It is not feasible to directly apply ELLM
354   to our setting, since it requires an LLM call for each observation, where the total number of calls
355   scales to the billions in our case. Therefore, we designed a more scalable variant which i) replaces
356   the LLM embedding with a lightweight bag-of-words embedding 6 using FastText (Bojanowski
357   et al., 2016), and ii) includes an episodic term in the intrinsic reward, as with Motif and ONI (see
358
      Appendix A.2). We call it ELLM-BoW and include more details in Appendix A.3. We note that
      Klissarov et al. (2023) found that other exploration methods based on novelty bonuses such as RND
359
      (Burda et al., 2019), NovelD (Zhang et al., 2021) and E3B (Henaff et al., 2022a) did not improve
360
      over the extrinsic reward baseline on these tasks, hence we do not include them.

402    positive valence being repeated in the early game of NetHack, such as “You find a hidden passage”
403    that allows exploring the rest of the level, or “You kill the {monster}!” which leads to experience
404    gain. ONI-classification, which also predicts binary rewards but is able to generalize to
405    unseen messages, provides a modest but consistent improvement over ONI-retrieval across all
406    environments. This suggests that learning a reward model is indeed helpful. We would expect this
407    gap to increase in settings with added noise or caption diversity, since they increase the likelihood
408    of observed captions being unique.
409    We do not observe any significant gains from using ONI-ranking over ONI-classification,
410    despite it being more conceptually general and able to represent a continuous range of intrinsic
411    rewards rather than binary values. This may be because our tasks take place at the very earliest
412    part of the game of NetHack, where only a small fraction of all possible messages are observed,
413    which would also explain the relatively strong performance of ONI-retrieval. We hypothesize
414    that more benefits will appear in settings with higher observation diversity.
415    ELLM-BoW performs surprisingly well on these tasks, closely tracking or matching Motif and ONI
416    methods. However, in Section 5.2 we highlight a fundamental limitation of ELLM-BoW, namely its
417    inability to capture complex semantic meaning, whereas ONI is capable thanks to its use of an LLM.
418    It is worth noting that directly using ELLM-BoW as in Du et al. (2023a) without our episodic bonus

525   Impact of Intrinsic Reward Coefficient β For ONI-classification and ONI-retrieval, we
526   have used different values of β for the Score task and other sparse reward tasks in Section 5.1.

533   6          C ONCLUSION
534   We have introduced ONI, a distributed online intrinsic reward and agent learning system. We
535   showed that we are able to match the state of the art across a range of challenging sparse reward tasks
536   from the NetHack Learning Environment, while removing the need for a large pre-collected dataset
537   or auxiliary dense reward function required by previous work. We explored three different instan-
538   tiations of our system of varying levels of complexity and generality, and study their tradeoffs. Our
539   work paves the way for intrinsic reward methods which can learn purely from agent experience, are
      not constrained by external dataset size or quality, and can leverage high-performance RL training.

758   A.1   A RCHITECTURES
759
      Our architectures largely follow those used in Klissarov et al. (2023).
760
761   Our policy network uses the Chaotic Dwarven GPT5 architecture originally introduced in Myf-
762   fili (2021). This architecture combines convolutional layers processing the top-down visible map
763
      centered at the agent with fully-connected layers processing messages and bottom-line statistics
      including hit points, experience, hunger level and the like. The convolutional encoder has 3 con-
764
      volutional layers with 32, 64, 128 feature maps respectively, interleaved with exponential linear unit
765
      (ELU) non-linearities (Clevert et al., 2016). Messages and bottom-line statistics are each processed
766   with 2-layer MLPs with 128 hidden units each. All embeddings are combined, passed through a
767   single-layer MLP with 512 hidden units, and then passed to a recurrent GRU module (Cho et al.,
768   2014) with 512 hidden units. Finally, this hidden representation feeds into linear critic and actor
769   heads.
770
      Our reward model The reward model of ONI-ranking is based on the encoder from Küttler et al.
771
      (2020) that processes both state representation and messages. Messages are processed by a 5-layer
772   character-level CNN (Zhang et al., 2015) with 64 feature maps at each layer. The first, second and
773   last layers are interleaved with max-pooling layers with kernel size and stride 3. The output is then
774   passed through a 3-layer MLP with 128, 256, 512 hidden units at each layer respectively, and ReLU
775   non-linearities, followed by a scalar output. The reward model of ONI-classification only process
776   messages, using the same architecture described above.

        pairs of captions to annotate and use for reward model training. Here we study the effect of dedu-
1067
        plicating captions before passing them to our pipeline. In either case, we maintain a message list L,
1068    and sample pairs of captions c1 , c2 ∼ Uniform(L × L) for annotation. The annotated message pairs
1069    are stored in another list, which is sampled from uniformly when training the reward model. In the
1070    first option, each time the agent encounters a message, we check if it is already stored in L and only
1071    add it if not. This is similar to the approach used in ONI-classification. In the second option,
1072    we simply add all captions encountered by the agent into L, regardless of whether they have already
1073    been seen before. This approach is similar to that taken by the original Motif work, which does not
1074    perform any deduplication of the offline dataset.
1075    Figure C.4 shows the performance and the number of captions stored in the replay buffer for both
1076    schemes. We see that for the deduplicated variant, the number of captions does not grow past a
1077    certain point (approximately 70k unique captions). For the non-deduplicated variant, the number of
1078    captions keeps growing linearly over time. The deduplicated variant fails to learn, which highlights
1079    the important effect which the annotation dataset can have. We hypothesize that the deduplicated
        variant may undersample captions that occur frequently in the agent’s experience, for which it is im-

        Under review as a conference paper at ICLR 2025


1080
        portant to reliably estimate reward. For example, the blank message occurs very frequently during
1081
        policy learning, but is only included in a small fraction of the pairs sent to the LLM for annotation,
1082    since it is sampled with the same probability as the other ∼ 70k other captions. In contrast, the
1083    duplicated variant samples the blank message and other frequent captions with much higher proba-
1084    bility. Still, this remains a simple strategy, and designing more sophisticated sampling mechanisms
1085    (for example, that account for the epistemic uncertainty of the reward model) would be an interesting
1086    direction for future work.

1219   D     A DDITIONAL R ELATED W ORK
1220   LLM for RL Broadly Another way of leveraging the prior knowledge encoded in LLMs for de-
1221   cision making is to use the LLM directly as a policy. This approach has been successfully used
1222   in robotics (Ahn et al., 2022; Driess et al., 2023), as well as open-ended exploration in MineCraft
1223   (Wang et al., 2024). Both settings require the LLM to operate at a higher level of abstraction, by
1224   having it call upon a set of semantically grounded skills which handle the low-level sensorimotor
1225   activity. These are in turn produced by imitation learning on expert trajectories or hardcoded APIs.
1226   Jeurissen et al. (2024) prompt the LLM to choose a predefined skill to play NetHack. The prompts
1227   are constructed to represent past events, current observation, and the task description and hardcoded
1228
       available skills are also included. More references on LLMs for decision making can be found in
       the survey paper of Cao et al. (2024).
