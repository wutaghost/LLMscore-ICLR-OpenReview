009                                                A BSTRACT
010
011               Code generation models have shown significant potential for programming tasks.
012               However, existing training methods like supervised fine-tuning face key limita-
013               tions: they do not effectively teach models to prioritize correct over incorrect
014               solutions in ambiguous situations, nor do they effectively optimize the runtime
015               efficiency of the generated code. To address these challenges, we propose Cod-
016               eDPO, a framework that integrates preference learning into code generation to
017               improve two key code preference factors: code correctness and efficiency. Cod-
018
                  eDPO employs a novel dataset construction method, utilizing a self-generation-
                  and-validation mechanism that simultaneously generates and evaluates code and
019
                  test cases. The underlying assumption is that test cases executable by multiple
020
                  code snippets provide more reliable validation, and code that passes more tests
021               is more likely to be correct. Through this self-validation process, our PageRank-
022               inspired algorithm iteratively updates the ranking score of each code snippet, ul-
023               timately creating a code preference optimization dataset based on correctness and
024               efficiency. CodeDPO is flexible and scalable, generating diverse preference op-
025               timization data without depending on powerful models such as GPT-4. Through
026               comprehensive evaluations of five widely used benchmarks, CodeDPO demon-
027               strates significant improvements in correctness and efficiency compared to exist-
028               ing methods. Our experiments prove that CodeDPO enhances the capabilities of
029
                  LLMs in code generation and provides a robust foundation for conducting code
                  preference optimization in more complex and challenging real-world scenarios. 1

      1       I NTRODUCTION
034
      In recent years, code generation models have gained significant attention for their potential to au-
035
      tomate software development. Models such as GPT-4 (GPT-4, 2023), Claude, and open-source
036
      alternatives like Phi (Gunasekar et al., 2023; Abdin et al., 2024), DeepSeekCoder (Guo et al., 2024),
037   and StarCoder (Li et al., 2023; Lozhkov et al., 2024) have demonstrated the capability of LLMs to
038   handle complex code generation tasks. However, one of the ongoing challenges lies in boosting the
039   correctness and efficiency of the generated code.
040
      To improve code generation models, a common approach is supervised fine-tuning (SFT) (Zhang
041
      et al., 2023b), where models are trained on pairs of instructions and correct code snippets. While
042
      SFT improves the overall quality of the generated code, it falls short in teaching models to con-
043   sistently prefer correct solutions over incorrect ones (Hong et al., 2024). Figure 1 illustrates the
044   likelihood of generating code with varying correctness and efficiency during SFT training. When
045   we adopt SFT training on those correct solutions, as the likelihood of preferred outputs increases,
046   the probability of generating undesirable outputs also rises, leading to performance saturation.
047
      To address these limitations, recent research has turned to direct preference optimization (DPO)
048
      (Rafailov et al., 2024), a method designed for alignment based on pairwise preference data. DPO al-
049   lows models to rank different outputs and choose preferable solutions (e.g., more factual or helpful).
050   While DPO has shown success in reasoning tasks like mathematics (Lai et al., 2024; Wu et al., 2024),
051   its application in code generation remains under-explored. Unlike natural language tasks, code gen-
052   eration requires objective metrics, such as executability, which poses challenges for directly applying

      DPO. In this paper, we first define code preference based on two key factors—Correctness and
069   efficiency. Correctness refers to whether the code solves the problem accurately, while efficiency
070   measures how quickly the code runs. Existing methods (Gee et al., 2024; Zhang et al., 2024) rely
071   heavily on high-quality test cases to assess correctness. However, these approaches struggle to fully
072   address correctness and efficiency, facing limitations such as restricted data diversity, an imbalance
073   between positive and negative samples, and insufficient focus on optimizing code efficiency.
074
      In this paper, we introduce CodeDPO, a novel framework that integrates preference learning into
075   code model training to optimize both correctness and efficiency. CodeDPO constructs the dataset
076   from real-world code repositories using a self-generation-and-validation mechanism, where code
077   and test cases are simultaneously generated and evaluated. We assume that tests executable by
078   more code snippets are more reliable, and code that passes more tests is more likely to be cor-
079   rect. To implement this, CodeDPO uses a mutual verification process: each receives an initial self-
080   validation score, which is iteratively updated using a PageRank-inspired (Page, 1999) algorithm.
081   This algorithm adjusts the credibility of each code snippet and tests by considering their relations in
082   cross-verification, prioritizing solutions based on correctness and efficiency. The final preference-
083
      optimized dataset is then used to train various code models using the DPO learning algorithm. A
      key advantage of CodeDPO is its flexibility. Unlike existing methods that rely on high-quality test
084
      cases or powerful models to generate them, CodeDPO does not depend on these resources. Its self-
085
      generation and validation mechanism supports the scalable creation of diverse and robust preference
086   optimization data. This allows our framework to optimize code models for real-world scenarios
087   where high-quality test data may be sparse.
088
089
      CodeDPO can serve as a crucial step in the post-training phase of code models. We conduct ex-
      periments on five popular benchmarks such as HumanEval (Chen et al., 2021), HumanEval+ (Liu
090
      et al., 2024a), MBPP (Austin et al., 2021), MBPP+, and DS-1000 (Lai et al., 2023) with CodeDPO,
091
      demonstrating its superiority over existing methods. Notably, we develop a top-performing 6.7B
092   model by building on an existing SFT strategy (Guo et al., 2024; Wei et al., 2023) and further enhanc-
093   ing it with our CodeDPO approach, achieving an impressive 83.5% pass rate on HumanEval. We
094   also conduct ablation studies to investigate the impact of our self-generation-and-validation mecha-
095   nism and other preference optimization settings. Our findings confirm that CodeDPO enhances the
096   code generation capabilities of LLMs while providing a solid foundation for further research into
097   optimizing code generation for both correctness and efficiency.

      2.2   P REFERENCE O PTIMIZATION FOR C ODE M ODELS
134
135   Preference optimization techniques have recently been used to help LLMs prefer better outputs over
136   weaker ones in various natural language tasks (Rafailov et al., 2024). The Direct Preference Opti-
137   mization (Rafailov et al., 2024) has been widely applied to LLM alignment due to its convenience
138   and effectiveness. Its objective is defined as:

       Compared with the SFT loss, the DPO loss introduces a preference-based mechanism. Instead
      of merely maximizing the likelihood of ground truth data, as in SFT, DPO optimizes the model to
143
      align with human preferences by leveraging both preferred responses (yw , winning) and dispreferred
144
      responses (yl , losing). While DPO has proven effective in reasoning tasks like mathematics (Lai
145   et al., 2024), its use in code generation is still under-explored. Code generation requires objective
146   measures of correctness and efficiency, unlike natural language tasks where preferences are often
147   more subjective. Some works have simply explored PO. Code-Optimize (Gee et al., 2024) builds
148   its dataset from the MBPP-train subset, which includes just 384 problems. PLUM uses GPT-4 to
149   generate tests, which are then used to validate and rank code solutions. PLUM currently achieves
150   state-of-the-art performance in preference optimization for code models. However, PLUM (Zhang
151   et al., 2024) faces some limitations. It uses a limited number of tests to validate the code, and
152   the resulting dataset is imbalanced due to its validation method, which means it can only use KTO
153
      (Ethayarajh et al., 2024) for training. Additionally, PLUM does not consider the code efficiency.
      This paper introduces CodeDPO, which does not rely on external test cases or powerful models
154
      for dataset generation. Our approach uses a self-generation and validation mechanism to create
155
      balanced preference pairs, aiming to optimize both correctness and efficiency.

      3.1   DATA S EED C ONSTRUCTION
178
      The data seed construction for CodeDPO is the first step for initiating the preference learning process
179
      to generate programming task prompts. We adopt a method inspired by OSS-instruct (Wei et al.,
180   2023; 2024)2 , which extracts key programming concepts from open-source code repositories. These
181   concepts serve as the foundation for generating various programming task prompts. For example, a
182   code snippet that performs sorting operations might highlight concepts such as sorting algorithms,
183   data structure traversal, and time complexity. From these concepts, we generate code generation
184   prompts. The data seed thus allows the model to explore a wide range of scenarios.
185
186   3.2   C ORRECTNESS O PTIMIZATION WITH S ELF -G ENERATION AND VALIDATION
187
188   Central to CodeDPO is the self-generation-and-validation loop, which enables the model to itera-
189   tively update the code correctness rank through mutual validation of code and test cases (Chen et al.,
190   2022; 2023; Zhang et al., 2023a). The process begins by generating multiple candidate code snippets
191   based on a prompt. Simultaneously, corresponding test cases are generated to evaluate these snip-
192   pets. The validation loop follows these steps: 1. Code Generation: Given an instruction, the model
193
      generates a set of candidate code snippets C = {c1 , c2 , ..., cn }. 2. Test Case Generation: Test
      cases T = {t1 , t2 , ..., tm } are generated in parallel to validate the candidate snippets. 3. Validation
194
      Process: Each code snippet is executed against the generated test cases. The validation outcomes
195
      are used to update the self-validation scores for both the code snippets and the test cases.
196
197

199
      and test is assigned a self-validation score of 1. Over a fixed number of iterations T = 10, these
200
      scores are updated based on the performance of the snippets and test cases during validation.

      3.3   E XECUTION E FFICIENCY O PTIMIZATION
217
218   In addition to ensuring correctness, CodeDPO integrates execution efficiency optimization to ensure
219   that our approach generates functionally correct and efficient code. During the self-validation loop,
220   the execution time for each code snippet is recorded. However, not all test cases accurately reflect
221   the efficiency of the code. To address this, we use the top-performing code from the correctness
222   optimization phase as a reference, assuming the test cases it passes are credible. The total execu-
223   tion time for each code snippet is then measured based on the subset of these credible tests. Code
224
      snippets that pass these credible test cases with lower execution times are assigned higher efficiency
      scores. Finally, we collect both fast and slow code snippets as part of the training dataset for execu-
225
      tion efficiency optimization, which is used for further training, encouraging the model to prioritize
226
      solutions that are accurate and optimized for speed during code generation.
227

      3.4   F INAL DATASET AND M ODEL O PTIMIZATION
229
230   The final dataset is built from the previous two optimization dataset construction stages, accounting
231   for correctness and execution time. This dual-optimization approach ensures that our CodeDPO
232   dataset can train models to generate not only accurate code but also efficient solutions, addressing
233   both functional and performance challenges in real-world coding tasks. We filter out samples whose
234   ranking scores are identical or too close. The final dataset consists of 93k correctness optimization
235   samples and 21k efficiency optimization samples. Each sample includes a unique code problem
236   prompt with a preferred and a rejected solution. We carefully avoid overlap between the data seeds of
237
      correctness and efficiency samples, ensuring that the constructed dataset captures various problems
      and instructions. In the subsequent training, we combine both correctness and efficiency data to
238
      optimize the model in both aspects simultaneously.
239
240   In our experiments, we apply Direct Preference Optimization (DPO) (Rafailov et al., 2024) across
241   various code models to facilitate optimization learning. To enhance the stability and robustness
242   of the training process, we employ RPO (Pang et al., 2024; Liu et al., 2024b) format loss, which
243
      essentially consists of a weighted SFT loss on the chosen preferences together with the original
      DPO loss, which is defined as: L = LDP O + LSF T . CodeDPO is plug-and-play and can be applied
244
      to nearly all code models, regardless of their type or training stage. We utilize both base models
245
      and SFT models as the backbone for further training. Our goal is to demonstrate that CodeDPO has
246   the potential to enhance code models at different stages of their training, even for models that have
247   undergone extensive training or fine-tuning. The setup details are provided in Section 4.2.

282   on objective metrics such as code correctness and efficiency, it contrasts with other natural language
283   tasks where preferences are often more subjective. This does not require our backbone model to

285   base models. We choose all these popular models as the backbone of our experiments to optimize

      correctness and execution efficiency.

288   4.2    T RAINING AND I NFERENCE S ETTINGS
289
290   For dataset construction, in order to balance generation speed and cost efficiency, we use
291   DeepSeekCoder-v2 as the data generation model. For each problem prompt, we sample 15 code
292   solutions and test cases from this model with temperature = 1.5. To construct the preference
293   optimization dataset, we set T to 10 and d to 0.85 for the self-validation score. Our practice shows
294   that this parameter configuration quickly yields a stable ranking score. In this paper, we focus on
295
      constructing a Python dataset. The total cost of our dataset construction process is nearly 80$.
296   For training, we train each code model for 10 epochs and select the best-performing model based on
297   the lowest validation loss. We utilize a learning rate of 5e-6 with a linear scheduler and warm-up.
298   The maximum sequence length is set to 2048 tokens.
299   For inference, we use greedy search decoding for code generation. All evaluations use the frame-
300   work from bigcode-evaluation-harness (Ben Allal et al., 2022). We use 16 A100 GPUs for all
301   experiments.

      6     C ONCLUSION
531
532
      We propose CodeDPO, a preference optimization framework for code models that focuses on both
533
      correctness and efficiency. CodeDPO introduces a novel dataset construction method that utilizes a
534   self-generation-and-validation mechanism, enabling the simultaneous generation and evaluation of
535   code and test cases to ensure correctness. Our PageRank-inspired algorithm iteratively updates the
536   self-validation score of each code snippet, prioritizing solutions based on correctness and efficiency.
537   Our work technically validates the reliability of self-validation to synthesize preference optimization
538   data, eliminating the need for complex resources such as pre-existing tests or powerful generation
539   models. We hope this work opens new avenues for synthesizing data and implementing large-scale
      preference optimization for code models.

1013   H.3.2     H ANDLING W EAK T EST C ASES
1014
1015   Our designed algorithm is robust. The self-validation scores can reflect the confidence of each code
1016   solutions and test cases through the iterative process. Notably, even in the presence of weak test
1017
       cases (such as assert True), our method handles them robustly. We have carefully considered
       the impact of weak test cases in our design. We address this issue from two perspectives: ❶ Natural
1018
       Suppression of Weak Test Cases in Ranking: Weak test cases are those that almost all code
1019
       solutions pass. While they contribute to the overall scores of all code solutions, they do not affect
1020   the relative differences between code solutions in the ranking process. Since the ranking is based
1021   on score differences, weak test cases naturally have minimal impact on the ranking outcomes. ❷
1022   Filtering Identical or Close Scores: Weak test cases can lead to highly similar scores for multiple
1023   code solutions after repeated score updates, diminishing the ability to differentiate between them. To
1024   address this, as described in Section 3.4, we implement a filtering mechanism that excludes samples
1025   with identical or near-identical ranking scores. This ensures that the influence of weak test cases is
       mitigated in the final dataset.


                                                         19
       Under review as a conference paper at ICLR 2025


1026
       For example, assume we have 15 code solutions and 15 test cases generated by the model. ❶
1027   If a weak test case, such as assert True, is passed by all 15 code samples, its score during each
1028   update step (as computed by Equation 1) will contribute equally to the scores of all code solutions.
1029   As a result, it does not alter the relative ranking of the code solutions. ❷ If all 15 test cases are
1030   similarly weak, the scores of the code solutions will converge to identical or near-identical values
1031   after several updates. To mitigate this, we apply a post-processing step (Section 3.4) to filter out
1032   such cases, ensuring the integrity of the final rankings. By addressing weak test cases through these
1033   mechanisms, our algorithm achieves robustness and maintains the reliability of its outputs, even in
1034   challenging scenarios.
