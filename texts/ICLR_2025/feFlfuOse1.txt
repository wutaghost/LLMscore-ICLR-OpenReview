011                                                   A BSTRACT
012
013             Reinforcement Learning (RL) is a continuously growing field that has the potential
014             to revolutionize many areas of artificial intelligence. However, despite its promise,
015             RL research is often hindered by the lack of standardization in environment and
016             algorithm implementations. This makes it difficult for researchers to compare and
017             build upon each other’s work, slowing down progress in the field. Gymnasium is
018             an open-source library that provides a standard API for RL environments, aiming
019
                to tackle this issue. Gymnasium’s main feature is a set of abstractions that allow
                for wide interoperability between environments and training algorithms, making it
020
                easier for researchers to develop and test RL algorithms. In addition, Gymnasium
021
                provides a collection of easy-to-use environments, tools for easily customizing
022             environments, and tools to ensure the reproducibility and robustness of RL re-
023             search. Through this unified framework, Gymnasium significantly streamlines the
024             process of developing and testing RL algorithms, enabling researchers to focus
025             more on innovation and less on implementation details. By providing a standard-
026             ized platform for RL research, Gymnasium helps to drive forward the field of
027             reinforcement learning and unlock its full potential.

      1       I NTRODUCTION
031
032
033   With the development of Deep Q-Networks (DQN) (Mnih et al., 2013), the field of Deep Reinforce-
034
      ment Learning (DRL) has gained significant popularity as a promising paradigm for developing
      autonomous AI agents. Throughout the last decade, DRL-based approaches managed to achieve or
035
      exceed human performance in many popular games, such as Go (Silver et al., 2017), DoTA 2 (Berner
036
      et al., 2019) or Starcraft 2 (Vinyals et al., 2019). During this time, OpenAI Gym (Brockman et al.,
037   2016) emerged as the de facto standard open source API for DRL researchers. Its simple structure
038   and quality of life features made it possible to easily implement a custom environment that is com-
039   patible with existing algorithm implementations. Gymnasium is the updated and maintained version
040   of OpenAI Gym. In this paper, we outline the main features of the library, the theoretical and prac-
041   tical considerations for its design, as well as our plans for future work. We hope that this work
042   removes barriers from DRL research and accelerates the development of safe, socially beneficial
043   artificial intelligence.

156   At its core, Gymnasium is a collection of interfaces tailored for usage in RL research, so that they can
157   be reused by researchers and developers across various approaches and algorithms. In this section,
158   we briefly describe the main abstractions and interfaces included in Gymnasium.
159
160   Environment The central abstraction in Gymnasium is the Env. It represents an instantiation of
161   an RL environment, and allows programmatic interactions with it. An Env roughly corresponds
      to a Partially Observable Markov Decision Process (POMDP) (Kaelbling et al., 1998), with some


                                                         3
      Under review as a conference paper at ICLR 2025


162
      notable differences discussed in Section 4.1. We show the general usage of an Env in Figure 1, with
163   the equivalent code in the OpenAI Gym version for comparison.
164
165   An Env is primarily defined by the following components:
166
            1. reset method
167
168         2. step method
169         3. Observation space
170
            4. Action space
171
172   The reset method sets the environment in an initial state, beginning an episode. The step method
173   executes a selected action in the environment, and is the only mechanism of moving the simulation
174   forward. The observation space defines the structure of observations that the environment emits, and
175   the action space defines valid actions that can be used in step. For a detailed description of the
176   Env class, we refer the reader to the documentation.
177
178   Vector Environment A secondary, but arguably as important abstraction, is the VectorEnv . It
179   represents a batch of identical, but independently running RL environments. In terms of POMDPs,
180   they are identical POMDPs, but with independently sampled initial states, and executing potentially
181
      different actions. A VectorEnv has a structure analogous to that of an Env , with the key differ-
      ence being that everything is batched across multiple independent environments. This is crucial for
182
      modern RL research, as it enables significant performance gains by parallelizing the environment
183
      execution, and the policy evaluation.
184
185
      Space Space is an abstraction shared between observation and action spaces. It roughly corre-
186   sponds to a set, with some structure imposed on it to improve its utility for representing (observation
187   and action) sets relevant to RL environments. We further describe Gymnasium spaces in Section 4.3.
188
189   Registry Gymnasium provides a simple way to manage environments via a registry. Any envi-
190   ronment can be registered, and then identified via a namespace, name, and a version number. The
191   standard Gymnasium convention is that any changes to the environment that modify its behavior,
192   should also result in incrementing the version number, ensuring reproducibility and reliability of RL
193   research.

201   4.1   F UNCTIONAL API AND POMDP S
202
203   While the object-oriented Env class is the standard way of using Gymnasium, the library also pro-
204   vides FuncEnv, a secondary abstraction with a more functional approach. Beyond just enabling
205   a different programming paradigm, this abstraction has two main advantages: it is more closely
206   connected to the theoretical POMDP formalism, and it enables easy hardware acceleration of imple-
207
      mented environments by using libraries like Jax (Bradbury et al., 2018).
208   Similarly to Env, a FuncEnv is defined with several functions:
209
210         1. initial function: Generates the initial state of the environment.
211         2. observation function: Returns the observation for a given state in the environment.
212         3. transition function: Computes the next state of the environment based on an action.
213
            4. reward function: Computes the reward for transitioning from one state to another given
214
               an action.
215
            5. terminal function: Determines if a state is terminal, indicating the end of an episode.


                                                        4
      Under review as a conference paper at ICLR 2025


216
      This is very close to the components of a POMDP. As a reminder, a POMDP is typically defined as
217   a tuple M = (S, A, T, R, Ω, O, µ), where
218
219          • S is a set of states of the environment.
220          • A is a set of actions available to the agent.
221
             • T : S × A → ∆S is the environment transition function, representing its dynamics.
222
223          • R : S × A × S → R is the reward function which is used to define the agent’s task.
224          • Ω is a set of possible observations
225          • O : S → ∆Ω is the observation function mapping states to observations.
226
             • µ ∈ ∆S is the initial state distribution.
227
228   In the usual object-oriented API, we cannot create a one-to-one connection between the Env mem-
229   bers and methods, and the POMDP components. Instead, the API combines some of the compo-
230   nents:
231
232
            1. reset is equivalent to sampling s ∼ µ, and returning a sample from O(s)
233         2. step takes an action a ∈ A, computes the next state as s′ ∼ T (s, a), and then returns the
234            new observation o ∼ O(s′ ) and the reward r = R(s, a, s′ )
235         3. The observation space is equivalent to Ω
236         4. The action space is equivalent to A
237
238   In the case of the FuncEnv , the connection is much closer:
239
240         1. initial function corresponds directly to sampling the initial state from µ, i.e., s ∼ µ.
241         2. observation function corresponds to the observation function O(s), returning an ob-
242            servation o ∼ O(s) for a given state s.
243         3. transition function directly implements the transition function T (s, a), which com-
244            putes the next state s′ ∼ T (s, a) given a state s and action a.
245         4. reward function implements the reward function R(s, a, s′ ), which calculates the reward
246            based on a given state s, action a, and the resulting next state s′ .
247
248   The one missing function, terminal, does not correspond to any element of the classical POMDP
249   description, but is necessary for practical reasons. We expand on this in the following section.

251   4.2   T ERMINATION AND T RUNCATION
252
253   In RL theory, it is common to assume that actions can be executed in an environment indefinitely,
254
      without stopping. In practical work, this tends to be unrealistic, as researchers only have finite time
      to perform their experiments. To account for that, Gymnasium introduces the notions of episode
255
      termination and truncation. In OpenAI Gym, these two concepts were not clearly separated, so it
256
      is worth expanding on their utility and the distinction between them.
257
258   Termination is a signal that depends on the state reached after an environment transition. Typically,
259   this represents either success or failure of the task that the RL agent is trying to achieve, but generally,
260
      it is any state-dependent reason to end an episode. After reaching a terminal state, step cannot be
      called again until the environment is reset. It is possible for an environment to not have any terminal
261
      states, as is common in theoretical RL problems. In the POMDP formalism, the typical way to
262
      represent episode termination is entering an absorbing state with 0 reward.
263
264   Truncation is similar to termination in the fact that it indicates the end of an episode, but instead
265   of being state-based, it is time-based. It is roughly the equivalent of saying “The episode could
266
      keep going, but we ran out of time, so we decided to stop it here”. In Gymnasium, we provide
      the TimeLimit wrapper that is applied to environments by default, through which developers and
267
      users can define a number of steps after which the environment is guaranteed to finish via truncation.
268
269   While the difference between termination and truncation may seem minor, it has nontrivial implica-
      tions for algorithm implementations. For example, in algorithms like REINFORCE (Sutton et al.,


                                                           5
      Under review as a conference paper at ICLR 2025


270
      1999) and Proximal Policy Optimization (PPO) (Schulman et al., 2017), a key component is esti-
271   mating the value of a given state, that is the expected discounted sum of rewards in the rest of the
272   episode. Similarly, in value-based algorithms like Deep Q-Networks (DQN) (Mnih et al., 2015) or
273   Soft Actor-Critic (Haarnoja et al., 2018), the central concept is that of a state-action value, which
274   also depends on future rewards.
275
      If an episode terminates, this means that there are no further rewards to be obtained. This means
276
      that in the value estimation process, the value after a terminal state is equal to 0. In contrast, if an
277   episode is truncated, that means that the agent encountered a rather arbitrary cutoff. If not for the
278   truncation, it could have kept acting and accumulating rewards, so in the value estimation algorithm,
279   we use an estimate of the final state’s value.
280
      For a practical example, consider a racing environment where the agent has to complete n laps to
281
      finish the race as quickly as possible. If the final signal is that of termination, this means that the
282
      agent should optimize for that exact number of laps. If its trajectory at the end of the last lap means
283   that it would crash shortly after passing the finish line, that is not a problem – that will never happen.
284   Conversely, if the final signal is truncation, then the agent will be incentivized to act as if it would
285   keep racing. This would result in decreasing the reward obtained in the original timeframe, but
286   acting in a safer, more sustainable manner towards the end of the race.
287
288   4.3   A LGEBRAIC SPACES
289
290   In Gymnasium, observation and action spaces can be broadly split into two categories – funda-
291   mental and composite. As the name suggests, composite spaces are composed of one or more
292   subspaces, whereas fundamental spaces cannot be divided this way. There are the following funda-
293   mental spaces:
294
            1. Box – for multidimensional arrays typically containing real numbers
295
296         2. Discrete – for single integers
297         3. MultiBinary – for sequences of binary values
298         4. MultiDiscrete – for sequences of integers
299
300
            5. Text – for strings
301
      There are also the following composite spaces:
302
303         1. Dict – for (Python) dictionaries of spaces
304
            2. Tuple – for tuples of spaces
305
306         3. Sequence – for variable-length sequences of spaces
307         4. Graph – for graphs of spaces (both nodes and edges)
308         5. OneOf – for disjoint unions of spaces
309
310   Considering just the fundamental spaces, and the Tuple* and OneOf composite spaces, we observe
311   that Gymnasium spaces mirror the structure of Algebraic Data Types. We can take any collection
312   of spaces and combine them into a Tuple to obtain a product type – an element of a Tuple space
313   must contain an element from each of its subspaces. We can also combine them into a OneOf space,
314   equivalent to a sum type, which contains an element from one of its subspaces, along with the index
315
      of that subspace.

387   5     B UILT- IN ENVIRONMENTS
388
389   Gymnasium includes a collection of well-tested environments that are fully compliant with the API.
390   Those can serve as references for new environment implementations, or as simple testing grounds
391   for algorithm development. Here, we describe these environments.
392
393   Toy text There are four toy text environments: Blackjack, Taxi, Cliff Walking and Frozen Lake.
394   These are discrete environments that can be modeled as tabular MDPs, and as such can serve as
395   testbeds for RL algorithms that do not rely on neural network for state representation.
396
397   Classic control In the classic control suite, there are four environments: Cartpole, Acrobot, Moun-
398   tain Car, and Pendulum. Each of them has a fairly simple physics simulation at its core, a continu-
399   ous observation space, and either a discrete or continuous action space. These environments tend to
400
      serve well as quick and simple evaluations for new algorithm implementations. A notable exception
      is Mountain Car, which is often difficult to solve without a deliberate exploration mechanism.
401
402
403   Box2D There are three environments using the open-source Box2D physics engine: Bipedal
404
      Walker, Car Racing, and Lunar Lander. These environments use a more complex physics simula-
      tion, including components like collision detection and contact forces (both of which are neglected
405
      in the classic control environments). These represent another step up in complexity, posing a non-
406
      negligible, but still very tractable challenge for RL algorithm implementations.
407
408
      Mujoco MuJoCo (Todorov et al., 2012) environments are a set of eleven physics-based continu-
409
      ous robot control environments. These environments simulate more complex physical interactions,
410   including multi-body dynamics and contact forces, offering a more realistic and high-dimensional
411   setting compared to the previously described environments. Each of these environments involves
412   controlling a simulated agent to achieve tasks such as locomotion or balance in a continuous state
413   and action space. The MuJoCo environments are often used as benchmarks for deep reinforcement
414   learning algorithms due to their higher dimensionality, non-linear dynamics, and need for precise
415   control.

      6     S UMMARY
418
419
      This paper introduces Gymnasium, an open-source library offering a standardized API for RL envi-
420
      ronments. Building on OpenAI Gym, Gymnasium enhances interoperability between environments
421
      and algorithms, providing tools for customization, reproducibility, and robustness. It is compatible
422   with a wide range of RL libraries and introduces various new features to accelerate RL research,
423   such as an emphasis on vectorized environments, and an explicit interface for functional environ-
424   ments that can be hardware-accelerated. Gymnasium includes a suite of benchmark environments
425   ranging from finite MDPs to MuJoCo simulations, streamlining RL algorithm development and
426   evaluation.
