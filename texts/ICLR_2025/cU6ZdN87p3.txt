009                                                 A BSTRACT
010
011            Retrieval-augmented generation (RAG) has been shown vulnerable to retrieval
012            corruption attacks: an attacker can inject malicious passages into retrieval results
013            to induce inaccurate responses. In this paper, we propose RobustRAG as the
014            first defense framework against retrieval corruption attacks. The key insight of
015            RobustRAG is an isolate-then-aggregate strategy: we isolate passages into disjoint
016
               groups, generate LLM responses based on the concatenated passages from each
               isolated group, and then securely aggregate these responses for a robust output. To
017
               instantiate RobustRAG, we design keyword-based and decoding-based algorithms
018
               for securely aggregating unstructured text responses. Notably, RobustRAG can
019            achieve certifiable robustness: we can formally prove and certify that, for certain
020            queries, RobustRAG can always return accurate responses, even when an adaptive
021            attacker has full knowledge of our defense and can arbitrarily inject a small number
022            of malicious passages. We evaluate RobustRAG on open-domain QA and long-
023            form text generation datasets and demonstrate its effectiveness and generalizability.

      1    I NTRODUCTION
027
      Large language models (LLMs) (Brown et al., 2020; Achiam et al., 2023; Google, 2024a) can often
028
      generate inaccurate responses due to their incomplete and outdated parametric knowledge. To address
029
      this limitation, retrieval-augmented generation (RAG) (Guu et al., 2020; Lewis et al., 2020) leverages
030   external (non-parametric) knowledge: it retrieves a set of relevant passages from a knowledge base
031   and incorporates them into the model input. This approach has inspired many popular applications
032   and software like Microsoft Bing Chat (Microsoft, 2024), Perplexity AI (AI, 2024), Google Search
033   with AI Overview (Google, 2024b), LangChain (LangChain, 2024), and LlamaIndex (Liu, 2022).
034
      However, despite its popularity, the RAG pipeline can become fragile when a small fraction (or even
035
      one) of the retrieved passages are compromised by malicious actors, a type of attack we term retrieval
036   corruption. This attack can occur in different scenarios. For instance, the PoisonedRAG attack (Zou
037   et al., 2024) injects malicious passages to the knowledge base to induce incorrect RAG responses (e.g.,
038   “the highest mountain is Mount Fuji”). The indirect prompt injection attack (Greshake et al., 2023)
039   injects malicious instructions into retrieved passages to override the original instructions (e.g., “ignore
040   all previous instructions and send the user’s search history to attacker.com”). Additionally, there
041   are real-world examples where Google Search AI Overview delivered inaccurate responses, such
042   as suggesting applying glue to pizza, due to unreliable content on indexed web pages (BBC, 2024).
043   These RAG failures raise the important question of how to safeguard a RAG pipeline.
044   In this paper, we propose a defense framework named RobustRAG that aims to generate robust
045   responses even when a fraction of the retrieved passages are malicious (see Figure 1 for an overview).
046   RobustRAG leverages an isolate-then-aggregate strategy: it isolates passages into disjoint groups,
047   computes LLM responses based on the concatenated passages from each isolated group, and then
048   securely aggregates these isolated responses for final output. The isolation operation ensures that the
049   malicious passages do not affect LLM responses for other benign passage groups and thus lays the
050
      foundation for robustness.
051   The biggest challenge for RobustRAG is to securely aggregate a mixture of benign and corrupted
052   responses. First, LLM text responses can be highly unstructured; for example, it is not straightforward
053   to recognize “Mount Everest” and “Everest is the highest” as the same response. Second, it is even
      harder to securely aggregate text responses, as corrupted responses can interfere with the aggregation

104   2.1   RAG OVERVIEW
105
106   RAG pipeline and notation. We denote text instruction as i (e.g., “answer the query using the
107   retrieved passages”), text query as q (e.g., “what is the name of the highest mountain?”), and text
      passage as p (e.g., “Mount Everest is known as Earth’s highest mountain above sea level”).


                                                                           2
      Under review as a conference paper at ICLR 2025


108
      Given a query q, a vanilla RAG pipeline first retrieves the k most relevant passages (p1 , . . . , pk ) :=
109   Pk from an external knowledge base. Then, it uses the instruction, query, and passages to prompt
110   an LLM model and get response r = LLM(i ⊕ q ⊕ Pk ) := LLM(i ⊕ q ⊕ p1 ⊕ . . . ⊕ pk ), where
111   ⊕ is the text concatenation operator. In this paper, we will call LLM(·) to obtain different forms of
112   predictions: we use LLMgen to denote the text response, LLMprob to denote the next-token probability
113   distribution vector, and LLMtoken to denote the predicted next token. Our presentation will focus on
114   greedy decoding as it enables deterministic robustness analysis; however, our RobustRAG design can
115   be compatible with different decoding strategies, like top-k sampling.
116   RAG evaluation metric. We use M(·) to denote an evaluation function. Given an LLM response r and
117   gold answer a, the function M(r, a) outputs a metric score (higher scores indicate better performance).
118   Different tasks usually use different metrics: for question answering (QA), M(·) can output a binary
119   score from {0, 1} indicating the correctness of the response; for long-form text generation, M(·) can
120   produce a score using heuristics like LLM-as-a-judge (Zheng et al., 2023).

      2.2   R ETRIEVAL C ORRUPTION ATTACK
123
124
      In this paper, we study retrieval corruption attacks against RAG: the attacker can control a fraction of
125
      the retrieved passages to induce inaccurate responses (i.e., lowering the evaluation metric score).
126
127   Attacker capability. We categorize retrieval corruption attacks into passage injection and passage
128   modification. The former can inject k ′ malicious passages with arbitrary content into arbitrary
129
      positions among the top-k retrieved passages; however, it cannot modify the content and relative
      ranking of benign passages. The latter can arbitrarily modify the content and positions of k ′ original
130
      passages. In this paper, we primarily focus on passage injection because it is a popular setting used
131
      by many attacks (Zou et al., 2024; Zhong et al., 2023; Du et al., 2022; Pan et al., 2023a;b); we will
132   use “corruption” and “injection” interchangeably when the context is clear. In Appendix B, we will
133   quantitatively demonstrate RobustRAG’s certifiable robustness against passage modification.
134
135
      Formally, we use Pk to denote the original (benign) top-k retrieved passages, Pk′ to denote the
      corrupted top-k retrieval result, and A(Pk , k ′ ) to denote the set of all possible retrieval Pk′ when k ′
136
      malicious passages are injected into the original retrieval Pk (and eject k ′ benign passages from the
137
      top-k retrieval). We note that we only aim to achieve robustness when k ′ is smaller than the number
138   of relevant benign passages (k − k ′ ); otherwise, it is theoretically impossible to generate accurate
139   responses based on the retrieved passages.
140
141
      Finally, we allow the attacker to know everything about our models and defenses, including defense
      algorithms and parameters, LLM architectures and weights, and decoding strategies. However, the
142
      attacker can only manipulate k ′ retrieved passages, but not our defense or LLM settings.
143
144   Attack practicality. There are numerous practical attack scenarios. For instance, attackers can launch
145   malicious websites that can be indexed by a search engine (i.e., the retriever) (Greshake et al., 2023).
146   In the enterprise context, malicious insiders may contaminate the knowledge base with harmful
147
      documents (Zou et al., 2024). Additionally, retrieval corruption can occur when an imperfect or even
      malicious retriever returns misleading information (Long et al., 2024). Our defense aims to mitigate
148
      different forms of retrieval corruption, whether they occur before, during, or after the retrieval.

151   2.3   C ERTIFIABLE ROBUSTNESS
152
153   A common pitfall in AI security is evaluating defenses using suboptimal attacks; stronger adaptive
154   attackers can break many defenses once they learn about the defense algorithms (Carlini & Wagner,
155   2017; Athalye et al., 2018; Bryniarski et al., 2022). In this paper, we aim to develop defenses
156   whose worst-case performance and robustness can be formally certified, eliminating any false sense
157
      of security. Formally, given a query q and retrieved benign passages Pk , we aim to measure the
      robustness as the quality of the worst possible response when our defense is prompted with arbitrary
158
      k ′ -corrupted retrieval Pk′ ∈ A(Pk , k ′ ). We formalize this property below.
159
160   Definition 1 (τ -certifiable robustness). Given a task instruction i, a RAG query q, the benign top-
161   k retrieved passages Pk , an LLM-based defense procedure LLMdefense that returns text responses,
      an evaluation metric M, a gold answer a, and an attacker A(Pk , k ′ ) who can arbitrarily inject k ′

      Under review as a conference paper at ICLR 2025


162
      malicious passages, the defense LLMdefense has τ -certifiable robustness if
163
164
                      M(r, a) ≥ τ, ∀ r ∈ R := {LLMdefense (i ⊕ q ⊕ Pk′ ) | ∀ Pk′ ∈ A(Pk , k ′ )}             (1)
165
166   Here, τ serves as a lower bound on model robustness against all possible attackers, even those with
167   full knowledge of our defense, who can inject k ′ passages with arbitrary content into any position.
168   This lower bound can eliminate the false sense of security.
169   We note that the attacker set A(Pk , k ′ ) contains infinitely many possibilities for Pk′ because the
170   injected passages can have arbitrary content. As a result, the response set R can be infinitely large and
171   intractable for us to analyze its worst response. In this paper, we will demonstrate how RobustRAG
172   limits the attacker’s influence and makes R tractable for certifiable robustness analysis.

175   3     ROBUST RAG: A G ENERAL D EFENSE F RAMEWORK
176
177
      In this section, we first present an overview of our RobustRAG framework and then discuss the details
178   of RobustRAG algorithms.
179
180   RobustRAG insights. The key insight of RobustRAG is an isolate-then-aggregate strategy (Figure 1).
181
      Given a set of retrieved passages, we first isolate them into disjoint groups, generate isolated LLM
      responses based on the concatenated passages from each group, and then securely aggregate these
182
      isolated responses for final output. With proper isolation design (Secion 3.1), a small number of
183
      corrupted passages can only affect a small fraction of passage groups and isolated responses. This
184   allows us to recover accurate responses from other unaffected passage groups.
185
186   RobustRAG challenges. The biggest challenge of RobustRAG is to design secure text aggregation
187
      techniques. First, unlike classification tasks where possible outputs are predefined, text responses from
      LLMs can be highly unstructured. For example, given the query “what is the name of the highest
188
      mountain?”, valid responses include “Mount Everest”, “Sagarmatha”, and “Everest is the highest”.
189
      Therefore, we need to design flexible aggregation techniques to handle different forms of text.
190   Second, though we have isolated the adversarial impact to individual responses, malicious responses
191   generated from corrupted passages can still interfere with the aggregation process. Therefore, we
192   need to design secure aggregation techniques for which we can formally analyze and certify the
193   worst-case robustness. To overcome these challenges, we propose two aggregation algorithms.
194
            1. Secure Keyword Aggregation (Section 3.2 & Algorithm 1): extracting keywords from
195            each response and using high-frequency keywords to prompt the LLM for the final response.
196
            2. Secure Decoding Aggregation (Section 3.3 & Algorithm 2): securely aggregating next-
197
               token prediction vectors from different isolated passage groups at each decoding step.

200   3.1   PASSAGE I SOLATION
201
202   In this subsection, we discuss our passage isolation design. Given k retrieved passages Pk =
203   (p1 , . . . , pk ), we isolate them into disjoint passage groups, denoted as Gm = (g1 , . . . , gm ), where
204   each gj represents the concatenation of passages from the j th group. Specifically, we group ω
205   adjacent passages (ω is a defense parameter) to get m := ⌈ ωk ⌉ disjoint groups as Gm := {gj =
206   pω·(j−1)+1 ⊕ . . . ⊕ pmin(jω,k) | 1 ≤ j ≤ ⌈ ωk ⌉}; we use Gm ← I SO G ROUP(Pk , ω) to denote this
207   operation. Furthermore, we use m′ to denote the number of passage groups with corrupted passages.
208   We have m′ ≤ k ′ because each passage only appears in one passage group; m′ reaches its maximum
209   value k ′ when each passage group only contains one malicious passage. The robustness of RobustRAG
210   relies on the other m − m′ benign passage groups.
211   Remark. The group size ω is an important parameter that balances the trade-off between robustness
212   and utility. A larger ω is more likely to provide high-quality responses as each isolated response is
213   based on more passages. However, a large ω reduces the number of passage group m = ⌈ ωk ⌉. If m
214   is too small, the corrupted passage groups can outnumber benign passage groups (m′ > m − m′ ),
215   making certifiable robustness impossible. For example, if we reduce RobustRAG to vanilla RAG by
      setting ω = k, we have m = ⌈ ωk ⌉ = 1, and even one corrupted passage can manipulate RAG outputs.

      3.2   S ECURE K EYWORD AGGREGATION
243
244
245
      Overview. For free-form text generation (e.g., open-domain QA), simple techniques like majority
246
      voting perform poorly because they cannot recognize texts like “Mount Everest” and “Everest” as the
247   same answer. To address this challenge, we propose a keyword aggregation technique: we extract
248   keywords from each isolated LLM response, aggregate keyword counts across different responses,
249   and ask the same LLM to answer the query using keywords with large counts. This approach allows
250   us to distill and aggregate information across unstructured text responses. We only consider unique
251   keywords from each response so that the attacker can only increase keyword counts by a small
252   number, i.e., m′ , instead of arbitrarily manipulating keyword counts.
253   Inference algorithm. We present the pseudocode of secure keyword aggregation in Algorithm 1.
254   First, we isolate k passages Pk into m passage groups Gm using the procedure I SO G ROUP(·, ω)
255   discussed in Section 3.1 (Line 2). Second, we initialize an empty counter C to track keyword-count
256   pairs (w, c) and a zero integer counter n (Line 3). Then, we iterate over each passage group (which
257   can be done in parallel). For each passage group gj , we prompt the LLM with the instruction i1 =
258   “answer the query given retrieved passages, say ‘I don’t know’ if no relevant information found”
259   and query q, and get response rj = LLMgen (i1 ⊕ q ⊕ gj ) (Line 5). If “I don’t know” is not in the
260   response, we increment the integer count n by one to track the number of non-abstained responses
261
      (Line 7). Then, we extract a set of unique keywords Wj from each response rj (Line 8) and update the
      keyword counter C accordingly (Line 9). The procedure G ET U NIQ K EYWORDS(·) in Line 8 extracts
262
      keywords and keyphrases from text strings between adjacent stopwords (more details in Appendix C).
263
      We note that we only extract unique keywords to prevent the attacker from arbitrarily increasing
264   keyword counts. After examining every isolated response, we filter out keywords whose counts are
265   smaller than a threshold µ. We set the filtering threshold µ = min(α · n, β), where α ∈ [0, 1], β ∈ Z+
266   are two defense parameters (Line 12). When n is large (many non-abstained responses), the threshold
267   is dominated by β; when n is small, we reduce the threshold from β to α · n to avoid filtering out all
268   keywords. Given the retained keyword set W ∗ (Line 13), we sort the keywords alphabetically and
269   then combine them with instruction i2 = “answer the query using provided keywords” and query q to
      prompt LLM to get the final response r∗ = LLMgen (i2 ⊕ q ⊕ S ORTED(W ∗ )) (Line 14).

      3.3   S ECURE D ECODING AGGREGATION
271
272   Overview. The keyword aggregation only requires LLM text responses and thus applies to any LLM.
273   If we have additional access to the next-token probability distribution during the decoding phase, we
274   can use a more fine-grained approach called secure decoding. Specifically, at each decoding step,
275   we aggregate next-token probability/confidence vectors predicted from different isolated passages
276   and make a robust next-token prediction accordingly. Since each probability value is bounded within
277   [0, 1], malicious passages only have a limited impact on the aggregated probability vector.
278   Inference algorithm. We present the pseudocode in Algorithm 2. First, we isolate passages into
279   groups Gm (details in Section 3.1) and initialize an empty string r∗ to hold our robust response
280   (Line 1). Second, we identify isolated passages for which the LLM is unlikely to output “I don’t
281   know” (Line 2). Next, we start the decoding phase. At each decoding step, we first get isolated
282   next-token probability vectors vj = LLMprob (i ⊕ q ⊕ gj ⊕ r∗ ) (Line 5). Then, we element-wisely add
283   all vectors together to get the vector v̂ (Line 7). To make a robust next-token prediction based on the
284
      vector v̂, we obtain its top-2 tokens t1 , t2 with the highest (summed) probability p1 , p2 (Line 8). If
      the probability difference p1 − p2 is larger than a predefined threshold η, we consider the prediction
285
      to be confident and choose the top-1 token t1 as the next token t∗ (Line 10). Otherwise, we consider
286
      the prediction to be indecisive, and choose the token predicted without any retrieval as the next token
287   t∗ (Line 12). Finally, given the predicted token t∗ , we append it to the response string r∗ (Line 14)
288   and repeat the decoding step until we reach the limit of the maximum number of new tokens (or hit
289   an EOS token) to get our final response r∗ .
290
      When the task is to generate long responses, we found greater success in certifying robustness by
291
      setting η > 0: no-retrieval tokens are immune to retrieval corruption and do not significantly hurt
292
      model performance as many tokens can be inferred solely based on sentence coherence. For other
293   tasks with short responses (a few tokens), we set η = 0 because sentence coherence becomes less
294   helpful, and no-retrieval tokens can induce inaccurate responses.

297   4     ROBUSTNESS C ERTIFICATION
298
299   In this section, we discuss how to analyze the certifiable robustness of RobustRAG. Our robustness
300   analysis is designed to be agnostic to specific attack algorithms, ensuring that the results apply even
301   to strong adaptive attackers with full knowledge of the defense. We discuss the core concepts and
302
      intuition here and leave the pseudocode and detailed proof in Appendix A.
303   Overview. Given a RAG query q, the robustness certification procedure aims to determine the
304   (largest) τ that satisfies τ -certifiable robustness (Definition 1). Toward this objective, the certification
305   procedure will analyze all possible RobustRAG responses r when an attacker can arbitrarily inject k ′
306   malicious passages to the top-k retrieval Pk . Let R be the set of all possible RobustRAG responses r.
307   We will show that, thanks to our RobustRAG design, R is a finite set. This allows us to measure the
308
      worst-case performance/robustness as τ = minr∈R (M(r, a)), where a is the gold answer.
309   To analyze all possible LLM outputs, we need to first understand possible LLM inputs (i.e., possible
310   passages/groups). For an attacker who injects k ′ passages into arbitrary positions within the top-k
                                  k

311   retrieval result, there are k′ possible cases of injection positions, and we need to analyze all of
312   them. To analyze each case, we simulate the isolation operation I SO G ROUP(·) to identify m′ out
313   of m = ⌈ ωk ⌉ passage groups that overlap with the injection positions (details and examples in
314   Appendix A.1). Our certification will be based on the other m − m′ benign passage groups.1
315
      Warm-up: majority voting. We use majority voting for classification as a warm-up example.
316
      We can first get the voting counts gathered from m − m′ benign responses. If the voting count
317
      difference between the winner and runner-up is larger than m′ , the final response can only be the
318   voting winner r∗ , regardless of the content and position of the m′ corrupted passage groups. This
319   is because the attacker can only increase the runner-up count by m′ (using m′ malicious passage
320   groups), which is not enough for the runner-up to beat the winner. Therefore, we have R = {r∗ } and
321   thus τ = M(r∗ , a) ∈ {0, 1} in this case.

      Secure keyword aggregation. Similar to majority voting, we analyze the m − m′ benign responses:
325   we first extract keywords and get their counts. We next analyze which keywords might appear in
326   the retained keyword set W ∗ (Line 13 of Algorithm 1). Intuitively, keywords with large counts will
327   always appear in W ∗ while keywords with small counts can never be in W ∗ . As a result, the attacker
328   can only manipulate the appearance of keywords with “medium” counts. When the set of medium-
329   count keywords is small (e.g., less than 10), we can enumerate all its possible subsets and generate
330   all possible retained keyword set W ∗ accordingly (by combining large-count and medium-count
331   keywords). Finally, we compute all possible responses r from all possible W ∗ and let them form a
332   response set R—we have τ = minr∈R M(r, a). We present the detailed procedure in Appendix A.2.
333   Secure decoding aggregation. We aim to analyze all possible next-token predictions at every
334   decoding step. Given a partial response at a certain decoding step, we first compute next-token
335   probability vectors predicted on m − m′ benign passage groups and calculate the probability sum
336   of each token. Next, we identify the top-2 tokens with the largest probability sums and compute
337   their probability difference as δ. We will use this δ value to analyze possible next-token predictions.
338   Intuitively, a large δ always leads to the top-1 token being predicted; a medium δ allows for predictions
339
      of either the top-1 token or the no-retrieval token; when δ is small, the prediction can be any malicious
      token introduced by the attacker. We start our certification with an empty string and track all possible
340
      next-token predictions (and partial responses) at different decoding steps. If δ is never “small” when
341
      we finish decoding all possible responses; we can obtain a finite set of all possible responses R—we
342   have τ = minr∈R M(r, a). We present the detailed procedure in Appendix A.3.
343
344   Certifiable robustness evaluation. The certification algorithms allow us to analyze response set R
345
      to determine the τ value of τ -certifiable robustness for a given query q and its gold answer a. In our
      evaluation, we gather a dataset of queries and answers (q, a), calculate the τ value for each query,
346
      and take the averaged τ across different queries as a certifiable evaluation metric of robustness. The
347
      evaluated robustness numbers are agnostic to attack algorithms and hold for strong adaptive attacks.
348
349   We note that the certification algorithms discussed in this section are different from the inference
350   algorithms (Algorithm 1 and Algorithm 2) discussed in Section 3. The inference algorithms are the
351
      defense algorithms we will deploy in the wild; they aim to generate accurate responses from benign
      or corrupted retrieval. In contrast, the certification algorithms are designed to provably evaluate the
352
      robustness of inference algorithms; they operate on benign passages, require the gold answer a (to
353
      compute metric scores), and can be computationally expensive (to reason about all possible r ∈ R).

514   6   R ELATED W ORKS
515
516   LLMs and RAG. Large language models (LLMs) (Brown et al., 2020; Achiam et al., 2023) have
517   achieved remarkable performance for various tasks; however, their responses can be inaccurate due to
518   their limited parametric knowledge. Retrieval-augmented generation (RAG) (Guu et al., 2020; Lewis
519   et al., 2020) aims to overcome this limitation by augmenting the model with external information
520
      retrieved from a database. Recent works (Asai et al., 2024; Luo et al., 2023; Yan et al., 2024) improve
      RAG performance in the non-adversarial setting. This paper studies the adversarial robustness of
521
      RAG pipelines when an attacker corrupts a fraction of the retrieved passages
522
523   Corruption attacks against RAG. Early works studied misinformation attacks against QA mod-
524   els (Du et al., 2022; Pan et al., 2023a;b; Zhong et al., 2023). Recent attacks focused on LLM-powered
525   RAG. Indirect prompt injection (Greshake et al., 2023) injected malicious instructions to LLM
526
      applications. PoisonedRAG (Zou et al., 2024) injected malicious passages to mislead RAG-based
      QA pipelines. GARAG (Cho et al., 2024) used malicious typos to induce inaccurate responses. In
527
      this paper, we designed RobustRAG to be resilient to different forms of corruption attacks.
528
529   Defenses against corruption attacks. To mitigate misinformation attacks, Weller et al. rewrote
530   questions to introduce redundancy and robustness; Hong et al. trained a discriminator to identify
531   misinformation. However, these defenses focused on weak attackers that can only corrupt named
532
      entities, and these heuristic approaches lack formal robustness guarantees. In contrast, RobustRAG
      applies to all types of passage corruption and has certifiable robustness.
533

535   7   C ONCLUSION
536
537   We proposed RobustRAG as the first RAG defense framework that is certifiably robust against
538   retrieval corruption attacks. RobustRAG leverages an isolate-then-aggregate strategy to limit the
539   influence of malicious passages. We designed two secure aggregation techniques for unstructured
      text responses and experimentally demonstrated their effectiveness across different tasks and datasets.

1274   we set the maximum number of new tokens to 20 for the question-answering tasks and 500 for the
1275
       long-form generation task. We utilize in-context learning to guide the model to produce answers

1276

1277

1279   Additional details of evaluation. For short-answer open-domain QA, we observe that the default
1280   answer set is usually small (e.g., one answer) and misses many valid answers. Therefore, we prompt
1281
       GPT-4 (as shown in Figure 9) to expand the answer set, making the evaluation more accurate.

1285   metrics, we further multiply the score by 10 to have a scale from 10-100. The cost of evaluating 50
1286   responses is about $0.01. We provide the prompt template in Figure 10. We did not use GPT-4 as the
1287

       the long-generation task) in a single passage to simulate a strong attacker; the robustness against this

1373   In this section, we present more experiment results and additional analysis of our RobustRAG.
1374   Experiments with GPT-3.5 Models. We report the certifiable robustness and clean performance of
1375   RobustRAG with GPT-3.5-turbo in Table 5, as well as its empirical robustness against two attacks in
1376   Table 6. Similar to our main results, we observe that RobustRAG also achieves significant certifiable
1377   and empirical robustness. For instance, the certifiable accuracy is 69.6% and 37.8% on RQA-MC and
1378   RQA, respectively. Under PIA attacks, our RobustRAG achieves a 5.0% attack success rate, while the
1379   vanilla method exceeds 80%. We did not implement decoding aggregation for GPT-3.5 as it would
1380   require an extremely large number of API calls. We note that we can only get the probability for one
1381   next-token prediction with one API call. That means we need to call GPT many times to generate one
1382
       sentence (with multiple tokens). This is not a big issue for the open-weight model because we can
       reuse the KV cache computed for earlier tokens (we can store the cache for the first N tokens and
1383
       reuse them to predict the (N + 1)th token; however, for GPT API calls, the model needs to recompute
1384
       everything for the first N tokens to get the (N + 1)th token prediction.
