                                                  A BSTRACT
010
011
                Text-to-SQL systems translate natural language (NL) questions into SQL queries,
012             allowing non-technical users to perform complex data analytics. Large language
013             models (LLMs) have shown promising results on the text-to-SQL task. How-
014             ever, these LLM-based text-to-SQL solutions often generate syntactically correct
015             but semantically incorrect SQL queries, which yield undesired execution results.
016             Additionally, most text-to-SQL solutions generate SQL queries without provid-
017             ing information on the quality or confidence in their correctness. Systematically
018             detecting semantic errors in LLM-generated SQL queries in a fine-grained man-
019             ner with explanations remains unexplored. In this paper, we propose SQL ENS,
020
                a framework that leverages the given NL question as well as information from
                the LLM and database to diagnose the LLM-generated SQL query at the clause
021
                level. SQL ENS can link problematic clauses to error causes, and predict the se-
022
                mantic correctness of the query. SQL ENS effectively detects issues related to in-
023             correct data and metadata usage such as incorrect column selection, wrong value
024             usage, erroneous join paths, and errors in the LLM’s reasoning process. SQL ENS
025             achieves an average improvement of 25.78% in F1 score over the best-performing
026             LLM self-evaluation method in identifying semantically incorrect SQL queries on
027             two public benchmarks. We also present a case study to demonstrate that SQL ENS
028             can localize and explain errors for subsequent automatic error correction.

      1       I NTRODUCTION
032
      Text-to-SQL systems, that can translate a natural language (NL) question into a SQL query, de-
033
      mocratize data access for non-technical users, serving as an entry point to a larger data science
034
      pipeline (Patel et al., 2024). The advent of Large Language Models (LLMs) has significantly ad-
035   vanced this field, and LLM-based text-to-SQL techniques (Talaei et al., 2024; Lee et al., 2024) have
036   demonstrated promising results on public benchmarks such as BIRD (Li et al., 2023) and Spider (Yu
037   et al., 2018). Recently, the LLM-based text-to-SQL solutions have been adopted in data platforms
038   offered by AWS1 , Databricks2 , Snowflake3 , etc.
039
      Despite these advancements, text-to-SQL remains a challenging problem. The best performing
040
      method on the BIRD leaderboard4 only achieves an execution accuracy of around 73% on the dev
041   set, still producing more than 400 incorrect SQL queries out of 1534 NL questions. LLM-based
042   systems typically employ a multi-stage generation pipeline, consisting of a retrieval stage to col-
043   lect contextual information, a generation stage to produce candidate SQL queries and a correction
044   stage to regenerate SQL queries based on SQL parser errors as needed. While much attention has
045   been given to the retrieval and generation stages, there is still a lack of fine-grained and explainable
046   error detection in the correction stage. Namely, detecting semantic errors—where the SQL query
047   executes successfully but returns incorrect results—remains challenging and largely unsolved. This
048   is because semantic errors require a deep understanding of both the query logic and the database’s
049
      structure. Most text-to-SQL solutions only produce a SQL query without providing any information
      on the quality or measures of confidence.

068   Existing LLM-based text-to-SQL methods, like DIN-SQL (Pourreza & Rafiei, 2023) and MCS-SQL
069   (Lee et al., 2024), have a self-correction module that prompts an LLM to debug and correct a SQL
070   query. Such modules detect potential errors or measure the confidence of a generated SQL query by
071   generating multiple results and defining confidence e.g. based on LLM judgements or the number
072   of consistent outputs. However, these approaches lack fine-grained semantic error information and
073   explainability. They provide a confidence estimate for the entire SQL query based solely on the
074
      LLM’s output but do not offer detailed insights into which part of the query might be incorrect and
      why it is potentially wrong. This lack of fine-grained and explainable error detection hinders both
075
      end users and the LLMs from effectively troubleshooting errors in LLM-generated SQL queries,
076
      ultimately undermining trust in LLM-based systems and their wider adoption (Brown, 2024).
077
078   In this paper, we develop SQL ENS, a fine-grained and explainable error detection framework for the
079   text-to-SQL task. We concentrate on two specific challenges: (1) identifying potential error signals
080
      in a generated SQL query at the clause level, and (2) aggregating the error signals to determine
      whether the query could be semantically incorrect. SQL ENS is based on the intuition that a SQL
081
      query is reasonable for a question if the intermediate results of its clauses are reasonable (e.g., result
082
      sets are not empty, do not have too many missing values, etc.) and if the overall structure of the SQL
083   follows meaningful join paths in the database schema.
084
085   As shown in Figure 1, SQL ENS parses a given SQL query into an abstract syntax tree (AST). For
086
      each SQL clause in the AST, SQL ENS exploits a variety of error signals - described in detail below
      - from both the database and the LLM to detect potential semantic errors. The database signals are
087
      lightweight and deterministic, assessing the correct usage of SQL clauses and evaluating their ex-
088
      ecution results over the database. The LLM-based signals are derived from LLM’s comprehensive
089   knowledge about SQL and semantic understanding of the given NL question. To mitigate the chal-
090   lenge of potentially noisy signals, SQL ENS is further equipped with a weakly-supervised training
091   process that integrates both the database and the LLM signals to construct a labeled training dataset.
092   SQL ENS then trains a supervised or unsupervised classification model to predict if the given SQL
093   query is semantically correct, and generates an error report with detailed explanations. SQL ENS can
094   also use feedback and any available labeled examples.
095   Fine-grained and Explainable. Detecting semantic errors in text-to-SQL is challenging because
096   any misunderstandings about NL question or (meta-)data can lead to cascading errors throughout
097   the SQL query generation process. We are the first to not only provide an overall estimation of a
098   SQL query’s semantic correctness but also identify potential error causes at the SQL clause level.
099
      Robustness. Our approach can effectively handle noisy signals. The framework is capable of gen-
100
      erating high-quality training data. As such, the system is domain-agnostic and can be adopted even
101   under the absence of labeled data.
102
103   Applicability. SQL ENS is a general framework that can be seamlessly integrated with any text-
104
      to-SQL solution as it only takes as inputs an NL question and its corresponding SQL query. The
      fine-grained and explainable error report from SQL ENS can be utilized by any text-to-SQL solutions
105
      for error correction as demonstrated in Section 4.4.
106
107   Effectiveness. We provide extensive experimental results showing the effectiveness of SQL ENS
      on identifying semantically incorrect SQL queries (Section 4.2). On BIRD (Li et al., 2023) and

124   J between two tables Ti and Tj is based on common attributes (i.e., joinable columns Ti .Cm and
125   Tj .Cn , respectively). A database instance D = {(T1 , T2 , . . . , Tn ), J } comprises a set of tables and
126   a set of join relationships J between these tables.

      3.1   E RROR S IGNALS
174
175   Precise detection of semantic errors from a SQL query is inherently challenging due to the com-
176
      plexity and ambiguity in NL queries, data, and database schemas. Our insight is that many semantic
      errors in LLM-generated queries exhibit common patterns that can be detected through carefully
177
      crafted signals. We categorize these errors as follows.
178
      1. Question Ambiguity. The user’s questions might inherently contain ambiguities, and can be
179
         interpreted in different ways. For example, if a user asks “What were the total sales last quarter?”
180
         in a database where the sales table has columns named both gross sales and net sales, either
181      column could be selected to answer the question.
182
      2. Data Ambiguity. In real-world databases, multiple tables or columns with similar or identical
183
         names could exist due to data integration, versioning, table transformations, and other factors,
184      causing ambiguities as well. For example, a user might ask “What are the average salaries by
185      department?”, but the database contains both a dept table and a department table. Choosing the
186      wrong table leads to incorrect query results.
187
      3. Semantic Misalignment. Even when there is no ambiguity in NL questions nor databases, the
188      semantic gap between the question and the data can still lead to misalignments between the gen-
189      erated SQL query and the given NL question. For instance, the SQL query shown in Figure 2 uses
190      an incorrect join predicate (client.client id = account.account id) as the text-to-SQL algorithm
191      fails to understand the join relationships in the financial database.
192
      Definition 3 (SQL Error Signal) An error signal s analyzes a SQL query q to identify potentially
193   erroneous clauses Q′ and their associated error causes E. Formally, (Q′ , E) = s(q).
194
195
      Intuitively, an error signal acts as a proxy for identifying SQL semantic errors. As depicted in
      Figure 3, we introduce an error causal graph that connects a diverse set of error signals to three com-
196
      mon types of semantic errors described above. This graph resonates with the SQL error analysis
197
      conducted in recent text-to-SQL studies (Wang et al., 2024; Lee et al., 2024). Note that certain error
198   signals can be more effectively and reliably extracted through database-driven analysis, particularly
199   those related to semantic misalignment. In contrast, error signals that require nuanced interpreta-
200   tion of both the question and the SQL demand the deep semantic understanding capabilities from
201   an LLM. Hence SQL ENS incorporates both DB-based and LLM-based error signals to detect se-
202   mantic errors effectively. Note that neither the aforementioned semantic errors nor the error signals
203   described below are exhaustive, as error detection often involves a long tail of edge cases.
204   DB-based Error Signals. DB-based signals are designed to identify semantic misalignment and the
205   inherent ambiguity within the data. In Table 1, we present examples of incorrect and correct SQL
206   query pairs based on NL questions in the BIRD benchmark, highlighting the specific SQL clauses
207   targeted by each signal. We design these DB-based signals by analyzing real-life and benchmark
208   SQL queries such as TPC-DS6 , Redset (van Renen et al., 2024), BIRD (Li et al., 2023), etc. These
209   signals can be efficiently obtained, without using LLMs, by (1) executing a subquery from the

      heuristics from the above query workloads (e.g., Unnecessary Subquery). Further details regarding
      all DB-based error signals are provided in Appendix A.1.

      3.2   AGGREGATING A LL S IGNALS U SING W EAK S UPERVISION
271
272   The signals we collect are noisy, in that any signal may mistakenly flag correct SQL clauses as er-
273
      roneous and vice versa. For example, the empty predicate signal, as shown in Figure 2, is to check
      whether the value “jesenik” is in the column “A2”. However, it can lead to false positives if the
274
      absence of value in the column domain is intended. Therefore, it is crucial to consider these diverse
275
      signals collectively, for a specific application or database, when annotating the semantic correct-
276   ness of a generated SQL query, thereby mitigating the impact of noise. To do this, we leverage a
277   weak-supervision based framework that aggregates multiple sources of noisy labels – called label-
278   ing functions (LFs) – to approximate the true labels. These LFs can be heuristics or rules, each
279   contributing partial information about the target label. Weak supervision not only leverages the col-
280   lective wisdom of the LFs similarly to majority voting, but also goes further by learning the accuracy
281   and correlations of these LFs (Ratner et al., 2017).
282   In the context of SQL ENS, our diverse error signals are essentially the LFs, identifying potential
283   issues with a SQL query but not providing a definitive verdict on its correctness. By combining
284   these noisy error signals using weak supervision, we can infer the correctness of SQL queries, even
285   in the absence of ground truth labels. Specifically, an error signal s maps SQL clauses in q to
286   potential semantic errors. To determine whether any problematic SQL clauses are identified, we
287   apply the LF λs , which converts s(q) into a variable I. Formally,
288
289                                                          1    if |s(q)| > 0
                                        I = λs (q) =
290                                                          −1   if |s(q)| = 0
291
292
      I = 1 indicates that the error signal s has detected at least one problematic SQL clause, suggesting
      that the SQL is likely incorrect, while I = −1 indicates that no issues are found. Note that error
293
      signals are designed to identify potential errors and can only suggest that a SQL query is incorrect.
294
      Relying solely on negative labelers in weak supervision can result in limited coverage, leaving the
295   correctness of many SQL queries uncertain from labeling. To fully assess the correctness of a SQL,
296   we also develop three positive labelers, which are derived from combinations of error signals to label
297   a SQL as correct (i.e., I = 0).
298
299
      1. λall : labels a SQL query as correct if no error signals are detected.
300   2. λdb : labels a SQL query as correct if no database-based signals are detected.
301   3. λllm : labels a SQL query as correct if no LLM-based signals are detected.
302
303   For a SQL query q, we have negative labelers derived from error signals that label a SQL as incorrect
304   (1) and three positive labelers that label a SQL as correct (0). This results in a decision vector
305
      Λq = ⟨λs1 (q), ..., λsn (q), λall (q), λdb (q), λllm (q)⟩.
306   The goal of weak supervision is to learn a generative model, often called a label model, that estimates
307   the joint distribution p(Λ, Y ), where Y represents the unobserved true labels and Λ denotes the
308   observed noisy labels. The model’s objective is to find the parameters θ that best describe the
309   distribution by maximizing the likelihood of observing the labels provided by the LFs. To train the
310   model without true labels, SQL ENS minimizes the negative log marginal likelihood:
311                                                          X
312
                                       θopt = arg min − log      pθ (Λ, Y )
                                                     θ
                                                                   Y
313
314   Since the true labels Y are unknown, the model sums over all possible values that Y could take (Rat-
315   ner et al., 2017). Using the generative model, we obtain probabilistic labels, p(Y |Λ), representing
316   the likelihood of a SQL query’s correctness. SQL ENS then uses these probabilistic labels to train a
317   classifier that predicts the semantic correctness of SQL queries.

363

364
      have reasonable accuracy on Spider and BIRD, which means blindly predicting all SQL queries as
365   correct can still have a high accuracy. Therefore, it is crucial to also consider precision, recall, and

502   5    R ELATED W ORK
503
      Text-to-SQL. Generating accurate SQL from natural language questions (Text-to-SQL) is a long-
504
      standing challenge due to the complexities in user question understanding, database schema com-
505
      prehension, and SQL generation (Quamar et al., 2022; Katsogiannis-Meimarakis & Koutrika, 2023).
506   Recently, large language models (LLMs) have demonstrated significant capabilities in natural lan-
507   guage understanding as the model scale increases. LLM-based text-to-SQL solutions (Hong et al.,
508   2024) have emerged. DIN-SQL (Pourreza & Rafiei, 2023) studies how decomposing the text-to-SQL
509   task into smaller sub-tasks can be effective. MAC-SQL (Wang et al., 2024) presents a multi-agent
510   collaborating framework. The selector preserves relevant tables for user questions, the decomposer
511   breaks down user questions into sub-questions and provides solutions, and finally the refiner val-
512   idates and refines the defective SQL. CHESS (Talaei et al., 2024) introduces a new pipeline that
513   hierarchically retrieves relevant data and context, selects an efficient schema, and synthesizes cor-
514
      rect and efficient SQL queries. MCS-SQL (Lee et al., 2024) leverages multiple prompts to explore a
      broader search space for possible answers and effectively aggregate them. However, the SQL queries
515
      generated by these methods can often be semantically incorrect. And such erroneous queries are only
516
      detected after being executed. These methods can benefit from our SQL ENS by incorporating its
517   fine-grained error detection capability.
518
519   LLM Self-Evaluation. As part of ongoing research to improve LLM’s reliability and trustwor-
520
      thiness, LLM self-evaluation enables an LLM to assess the quality, accuracy, or relevance of its
      own response (Geng et al., 2024). Kadavath et al. (2022) explored the self-evaluation capabili-
521
      ties of LLMs. The findings show that LLMs are well-calibrated when answering multiple-choice
522
      and true/false questions. This ability improves further when the models consider multiple possible
523   answers before deciding on one. Tian et al. (2023) introduced methods to obtain well-calibrated con-
524   fidence scores from large language models (LLMs) that have been fine-tuned using reinforcement
525   learning from human feedback (RLHF). Self-RAG (Asai et al., 2024) enhances the factual accuracy
526   of LLMs by combining selective retrieval of external information with self-critiquing mechanisms.
527   The model can decide when to retrieve information and critique its own outputs, leading to more
528   reliable and verifiable results. However, these methods are designed for general tasks, which do not
529   address the unique challenges in text-to-SQL.
530
531   6    C ONCLUSION
532
      In this paper, we introduce SQL ENS, the first framework that exploits information from both the
533   database and the LLM to achieve fine-grained and explainable error detection in text-to-SQL. DB-
534   based signals identify semantic misalignment and the inherent ambiguity within the underlying data,
535   while LLM-based signals consider the question and the SQL query simultaneously and examine
536   the reasoning process of an LLM. SQL ENS systematically diagnoses SQL queries at the clause
537   level, aggregating noisy error signals through weak supervision to predict the semantic correctness
538   of a SQL query. SQL ENS significantly outperforms LLM self-evaluation methods in identifying
539   semantic errors in SQL queries. Beyond error detection, we also demonstrate the effectiveness of
      using error signals to fix SQL queries automatically.

      A.1   DB- BASED E RROR S IGNALS
651
652   SQL ENS uses the following signals to identify semantic misalignment.
653
654
      • Suboptimal join tree signal is introduced to determine whether a SQL query utilizes the optimal
      join tree for connecting the required tables to answer an NL question. SQL ENS first constructs a join
655
      graph G = (D, J ) based on the database schema, where D represents the tables and J represents
656
      the join relationships. Let Dreq ⊆ D denote the subset of tables required to answer the question. The
657   optimal join tree is defined as the minimum Steiner tree T ∗ that spans Dreq , indicating the minimal
658   set of tables needed for the join. If the SQL query includes more tables than those in Dreq , the
659   Sub-optimal Join Tree signal is flagged. For example, Figure 4 (right) shows an optimal join tree to
660   connect the tables A and C.
661
662               A                       D                   A                        D
663
664               B                       C                   B                        C
665
666                Suboptimal Join Tree             Optimal Join Tree (Minimum Steiner Tree)
667
668                                 Figure 4: Steiner Trees spanning A and C
669
670
      To implement this signal, SQL ENS first identifies the columns in a SQL query that are not involved
671
      in the J OIN clauses. Based on these columns, SQL ENS searches for the minimum Steiner Tree on
672   the join graph built offline to connect the relevant tables. If the SQL query involves more tables in the
673   join than necessary, compared to the minimum Steiner Tree, SQL ENS raises Suboptimal Join Tree
674   flag. This signal can produce false positives when a suboptimal join strategy does not impact the
675   correctness of the SQL query. For instance, if a query asks for the total sales from a specific store, it
676   can directly select the sales column from the store sales table. However, if the query unnecessarily
677   joins the store sales table with a store location table, even though the store location table is not
678   needed to get the correct sales data, the result would still be correct but the join is suboptimal.
679   • Incorrect join predicate signal checks whether a SQL query uses an invalid join predicate. For
680   example, in Figure 2, the predicted SQL incorrectly joins client id with account id, which does not
681   exist in the corresponding schema graph. To implement this signal, SQL ENS first extracts all the
682   join predicates from the SQL query. It then identifies two types of correct join predicates: (1) using
683   a primary key-foreign key (PK/FK) join explicitly defined in the database schema, and (2) derived
684   from PK/FK joins. Specifically, if two columns Ci and Cj both reference the same primary key
685   Ck (i.e., they refer to the same entity), Ci and Cj can be used in a join predicate. This signal may
686
      generate false positives when the PK/FK relationships are not fully documented in the database.
687   • Empty predicate signal detects if there is a predicate within a SQL query that yields an empty
688   result. This signal is useful to detect semantic misalignment errors, including wrong column selec-
689   tion, wrong value usage or wrong comparison operator. SQL ENS extracts all comparisons between
690   a column and a literal from a given SQL, executes each of them individually and records the output
691   size. If there is a predicate that yields empty rows, this signal is flagged. This signal may lead to
692
      false positives when an empty predicate is intentional.
693   • Abnormal result signal detects whether a SQL outputs an abnormal result that does not provide
694   much information. SQL ENS executes the SQL and records its output. The output is considered
695   abnormal if (1) it is empty, (2) it contains a column full of zeros, or (3) it contains a column full
696   of N ULLs. This signal extends beyond the empty predicate to evaluate the entire SQL output. In
697   addition to detecting empty predicates, it can identify empty intermediate execution results, making
698
      it effective for catching semantic misalignment in the intermediate steps of a SQL query. This signal
      may incorrectly flag a SQL query when the abnormal result is intentional, though this scenario is
699
      rare in practice.
700
701   • Incorrect filter in subquery signal detects the problematic filtering in a subquery. Filters in a
      subquery often follow the pattern column = (S ELECT...). When the subquery returns multiple rows,


                                                         13
      Under review as a conference paper at ICLR 2025


702
      the filter condition becomes ambiguous (e.g., I N or ‘=’), potentially leading to errors. SQL ENS
703   uses regular expressions to match this pattern and executes the extracted subquery separately. If it
704   returns more than one row, the signal is flagged. Additionally, SQLite is lenient with SQL semantics,
705   allowing column = (S ELECT...) to match only the first value returned by the subquery. While the
706   query might still be correct if the first value happens to be the desired one, relying on this behavior
707   is generally considered poor practice in SQL writing.
708
      • Incorrect GROUP BY signal detects any standalone G ROUP B Y clause without accompanying
709   aggregate functions such as M AX, C OUNT. A misused G ROUP B Y clause can change the SQL
710   semantics and lead to an incorrect result. In SQLite, a standalone GROUP BY behaves the same
711   as the DISTINCT operator. While the query may still be correct when using GROUP BY as a
712   substitute for DISTINCT, this approach is generally considered poor practice.
713
      • Unnecessary subquery signal indicates if there is an excessive use of subqueries in a SQL query,
714
      which leads to inefficiencies, increased complexity, and a higher likelihood of errors. This signal
715   counts the number of subqueries in a SQL query and flags it as problematic if the count exceeds a
716   specified threshold. In our evaluation, this threshold is set to 3. False positives may arise when the
717   subqueries are necessary for performance optimization or specific logic, even though they exceed
718   the threshold.
719
      The following signals are designed to capture the inherent ambiguity within the data.
720
721   • Value ambiguity signal detects incorrect column selections when a value used in an NL ques-
722   tion appears in multiple columns. For example, “New York” can appear in both “state” and “city”
723   columns. To identify ambiguous value, this signal extracts all values used in the SQL and finds
724
      columns that contain a used value via an inverted index built offline. If there is an alternative column
      that is closer to the question semantically, the signal flags the corresponding value as ambiguous. It
725
      is possible that the originally selected column in the SQL is correct, as the semantic distance may
726
      not always accurately determine which column containing the value is the best candidate to answer
727   the question.
728
729   • Table similarity signal detects potential errors in table selection by identifying alternative tables
730
      with similar structures. It extracts all columns used in the SQL, groups columns by their tables,
      and searches for other tables that contain the same groups of columns. If such a table is found, it
731
      suggests that the alternative table could have been used, indicating a potential mistake in the original
732
      chosen table. False positives can occur when the chosen table is actually correct, but another table
733   with a similar structure exists, leading the system to incorrectly flag a possible error.
734

      A.2   LLM- BASED E RROR S IGNALS
737
738
739   • Evidence violation signal identifies cases where the generated SQL query contradicts the evidence
740
      provided in the question or external knowledge. For example, if the question specifies retrieving
      rows only about active employees, but the SQL query does not include a condition to filter out
741
      inactive employees, this would trigger an evidence violation. The prompt used by SQL ENS can be
742
      found in Appendix A.3.4.
743
744   • Insufficient evidence signal assesses whether the available evidence is adequate to confirm that
745   the SQL query correctly answers the user’s question. For example, if the question lacks a clear
746
      explanation of a domain-specific concept, the LLM is prone to hallucination. This signal essentially
      verifies that the LLM has enough information and context to provide a correct response. The prompt
747
      is shown in Appendix A.3.5.
748
749   • Incorrect question clause linking signal evaluates the LLM’s confidence in the generated SQL
750   clauses. This signal first prompts the LLM to map the concepts, entities, and expressions in the user
751   question to the corresponding clauses in the SQL query. For each identified link, the LLM is then
752
      asked to indicate its confidence in the generated clause by responding with a simple yes or no. This
      signal is flagged when there is at least one clause with low confidence. The prompt can be found in
753
      Appendix A.3.6.
754
755   • Column ambiguity signal identifies whether there are columns in the database that are very similar
      to those used in the SQL query and could also be used to answer the user’s question. This signal


                                                        14
      Under review as a conference paper at ICLR 2025


756
      suggests that a SQL is prone to wrong column selection. The prompt for this signal is shared in
757   Appendix A.3.7.
758
759   The above signals provide specific error causes for the LLM to detect. Given that studies have shown
760
      LLMs possess the ability to validate their own answers Kadavath et al. (2022); Tian et al. (2023), we
      also incorporate a signal that prompts the LLM to provide an overall assessment of its own output.
761
762   • LLM self-check asks an LLM to determine whether the proposed SQL correctly answers the user
763   question considering the database and any available external knowledge. The prompt can be found
764   in Appendix A.3.2.
765   False positives can occur for the LLM-based signals when the LLM misinterprets the question, has
766   limited understanding of the specified knowledge, experiences hallucination, or exhibits bias when
767   evaluating its own output.
