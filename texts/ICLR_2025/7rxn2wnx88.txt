                                                   A BSTRACT
013
014
015
                 The rapid evolution of software libraries presents a significant challenge for code
                 generation models, which must adapt to frequent version updates while maintain-
016
                 ing compatibility with previous versions. Existing code completion benchmarks
017
                 often overlook this dynamic aspect, and the one that does consider it relies on
018              static code prediction tasks without execution-based evaluation, offering a limited
019              perspective on a model’s practical usability. To address this gap, we introduce
020              GitChameleon, a novel, manually curated dataset comprising 116 Python code
021              completion problems, each conditioned on specific library versions and accom-
022              panied by executable unit tests. GitChameleon is designed to rigorously assess
023              the ability of modern large language models (LLMs) to generate version-specific
024              code that is not only syntactically correct but also functionally accurate upon exe-
025              cution. Our comprehensive evaluations reveal that state-of-the-art LLMs struggle
026
                 with this task; for instance, GPT-4 achieves a pass@10 of only 39.9% (43.7%
                 when provided with error feedback), highlighting the complexity of the problem
027
                 and the limitations of current models. By providing an execution-based bench-
028
                 mark that emphasizes the dynamic nature of code libraries, GitChameleon serves
029              as a critical tool for advancing the development of more adaptable and reliable
030              code generation models. We release the dataset and evaluation framework to en-
031              courage further research in this vital area.

      1       I NTRODUCTION
035
036
      Large Language Models (LLMs) have become highly popular in code completion, to the extent
037
      that they are now deployed as virtual coding assistants within popular code editors1 , enhancing the
038
      overall coding workflow. Code, being a dynamic and constantly evolving environment, necessitates
039   a continuous process of adaptation to stay in sync with the rapidly shifting paradigms, frameworks,
040   and methodologies within the software development domain. The inherent variability in coding
041   styles, the emergence of new programming languages, and the continuous evolution of libraries and
042   packages underscore the imperative for an active approach in updating code generation models.
043
      In response to the needs of practical coding environments, several large language models (LLMs)
044
      have been introduced, including StarCoder (Li et al., 2023), DeepSeek-Coder (Guo et al., 2024),
045   CodeLlama (Rozière et al., 2023), among others. Despite these advancements, existing LLMs often
046   struggle to keep pace with the rapid changes in codebases, particularly when tasked with generating
047   version-specific code that is both syntactically and functionally accurate. This issue is especially
048   critical, as developers increasingly depend on AI-assisted coding tools to boost productivity and
049   maintain code quality. A recent Stack Overflow survey revealed that 70% of the participants are
050   using or planning to integrate AI coding tools, 33% citing increased productivity as the primary
051   motivation to incorporate these tools into their workflows2 .

076
077   Given the rapid development and release cycles of popular libraries, as shown in Figure 1 (right),
078
      the need for code generation models to continually adapt to changing API’s is more pressing than
      ever. For example, prominent machine learning and deep learning libraries like PyTorch (Paszke
079
      et al., 2019), NumPy (Harris et al., 2020), and Scikit-Learn (Buitinck et al., 2013) undergo
080
      frequent updates, which are reflected in a consistent upward trend in user downloads and version
081   releases. This dynamic nature of code requires models that can adapt and generate code that adheres
082   to the latest versions and practices, a need that current models often fail to meet comprehensively.
083   In addition, certain hardware is restricted to compatibility with specific versions of commonly used
084   packages, which adds an additional layer of complexity beyond merely updating the knowledge base
085   of a code LLM to the latest library versions.
086
      In response to these challenges, our work introduces a novel benchmark designed to assess the ability
087   of LLMs to generate version-specific code. We propose GitChameleon, a benchmark that evaluates
088   state-of-the-art code models by requiring them to produce executable code based on version-specific
089   prompts. This code is then executed to verify its correctness against expected outputs. By highlight-
090   ing the limitations of current models in generating accurate version-specific code, GitChameleon
091   provides a structured approach to enhance these models and ensure their practical utility in real-
092   world coding environments.
093   In summary, our contributions are highlighted as follows: 1) we introduce a novel code completion
094   benchmark GitChameleon consisting of 116 Python-based version conditioning problems including
095   human written unit tests; 2) we systematically analyse the version-specific performance of state-
096   of-the-art code generation LLMs on API change type, version release year, and specific libraries.
097   3) we demonstrate the effectiveness of utilizing error log feedback as a way to improve version
098   conditioning performance of code generation LLMs.

100   2   G IT C HAMELEON B ENCHMARK
101
102   We introduce GitChameleon, a benchmark comprising 116 Python-based version conditioning
103   problems focused on popular code libraries. To evaluate LLM performance on GitChameleon, each
104   problem is accompanied by handwritten assertion-based unit tests, enabling a thorough execution-
105   based assessment of the outputs generated by the code LLMs. This structured approach enables a
106   thorough understanding and categorization of LLM failures in common scenarios involving version-
107   specific code generation problems. In the following sections, we detail the benchmark statistics, data
      collection methodology, and sample verification process.

181   Differentiating factor Several datasets examine LLM interactions with version-specific code, in-
182   cluding Versicode (Wu et al., 2024b), CodeUpdateArena (Liu et al., 2024), and the dataset by Wang
183   et al. (2024). While these datasets are valuable, our dataset offers a unique and complementary per-
184   spective by focusing on the real-world scenario where developers are often constrained to specific
185   library versions due to technical debt. CodeUpdateArena investigates model adaptation to synthetic
186   API changes, we focus our evaluation on real API changes to assess how effectively an LLM can
187   generate code for version-specific changes of library versions that they have been trained with. In
188   contrast, Versicode and Wang et al. (2024)’s datasets, while addressing library evolution, primarily
189
      rely on string matching for evaluation. Our approach diverges by incorporating executable tests,
      providing a more practical and rigorous assessment of code generation capabilities.

228
229       We evaluate state-of-the-art large language models (LLMs) using the GitChameleon benchmark to
230       assess their ability to generate version-specific, executable code. This study highlights how well
231       current models adapt to dynamic library versions and produce functionally correct code that passes
232
          the provided unit tests.
233
234       3.1   E XPERIMENTAL SETUP
235
          For each open-source LLM, we downloaded the corresponding Hugging Face (HF) weights and
236
          served the models using Text Generation Inference (TGI). We used a single NVIDIA 95GB H100
237       GPU for models with fewer than 70 billion parameters, two GPUs for models more than 70 billion
238       parameters.
239
240
          We configured the generation parameters with a top p value of 0.95, top k of 50, and a tempera-
          ture of 0.3 for Pass@1 and 0.8 for Pass@10, in addition to finding the optimal temperature for each
241
          model. The maximum number of new tokens generated was set to 256. Additionally, we enabled
242
          flash attention (Dao et al., 2022) for all models to enhance inference efficiency. A list of all the
243       models and there cutoff dates if available is provided in appendix A.3.
244
245
          3.2   E VALUATION M ETRICS
246
247       To comprehensively evaluate the performance of code generation models using the GitChameleon
248       dataset, we employ a range of execution-based metrics. These metrics assess not only the correctness
249       of the generated code but also its efficiency and adaptability to different versions.
250
          Pass@k measures the proportion of problems for which at least one of the top k generated solutions
251       passes all assertion tests. This metric provides insight into the model’s ability to generate function-
252       ally correct code. For each problem, we generate n code samples, and compute the pass at k metric
253       by the corrected formula:
254
      1         def corrected_pass_at_k(n, c, k=10):
255
      2         if n - c < k: return 1.0
256   3         return 1.0 - np.prod(1.0 - k / np.arange(n - c + 1, n + 1))
257
258
          For instruct models, we run the model’s parsed output as standalone code, and for base models, the
259
          concatenation of the starting code and model’s parsed output (completion).
260
261       Greedy refers to the standard greedy decoding method, where the most probable token from the
262       next-token distribution is deterministically chosen. This is analogous to setting the temperature to 0.
263       Error Feedback adds the error log to the prompt (after executing the generated code from the
264       model with the initial prompt). Then, the pass@k metric is recalculated based on the model’s gen-
265       erated code using the prompt with error feedback. See appendix A.1 for an example.

466     4             R ELATED WORK
467
468     Code LLM training and evaluation protocols Code LLM evaluations mainly revolve around
469     code completion (Zhang et al. (2023); van Dam et al. (2023); Lu et al. (2021)). Existing benchmarks
470     emphasize generic code completion, yet a recognized limitation is the inability of code LLM to
471     generate and complete code that requires library and project-level knowledge (Xu & Zhu, 2022), let
472     alone version-level knowledge, which is vital for real-world software applications.
473     Recent initiatives address repository-level code understanding by LLMs (Bairi et al. (2023); Shrivas-
474     tava et al. (2023a;b); Liu et al. (2023); Guo et al. (2024)). Attempts at library-level code generation
475     (Zan et al. (2022)) and consideration of dependencies between files (Guo et al. (2024)) have been
476     made. However, these efforts do not directly address the challenge of accommodating version-
477     sensitive changes, adding complexity.
478     The core issue arises from models being trained on library code without explicit knowledge of li-
479     brary versions or their functional changes. Consequently, when tasked with generating code specif-
480     ically compatible with a particular library version, there is a significant risk models often encounter
481     failures.
482
483     Datasets Existing datasets like HumanEval (Chen et al., 2021), MBPP (Austin et al., 2021), and
484     MTPB (Nijkamp et al., 2022) provide sets of handwritten prompts and test cases to evaluate code
485     generated by code LLM. However, these datasets are relatively small and lack context regarding
        a model’s comprehension of repositories. APPS (Hendrycks et al., 2021) and CodeContest (Li


                                                                       9
      Under review as a conference paper at ICLR 2025


486
      et al., 2022) offer challenging datasets with coding competition questions, providing insights into a
487   model’s performance on difficult problems but without a focus on library-specific challenges. DSP
488   (Chandel et al., 2022) and DS-1000 (Lai et al., 2023) concentrate on the top data science libraries in
489   Python, while JulCe (Agashe et al., 2019) uses Jupyter Notebooks for training and evaluation, but
490   these notebooks do not necessarily need to be repository-specific. CoNaLa (Yin et al., 2018) contains
491   problems collected from StackOverflow across multiple programming languages, including both
492   library-specific questions and non-library-specific code. More recently, BigCodeBench (Zhuo et al.,
493   2024) is constructed to evaluate the comprehensive capabilities of code generation with tool use and
494   instruction following, which poses a great challenge for existing models. Several datasets include
495
      version-specific code, such as Versicode (Wu et al., 2024b), CodeUpdateArena (Liu et al., 2024),
      and the dataset by Wang et al. Versicode’s dataset, compiled from academic papers, Stack Overflow,
496
      and library source code, supports tasks like token, line, and block-level code completion and code
497
      editing. Unlike our dataset, Versicode evaluates using exact matches. Wang et al.’s dataset collects
498   API mappings, such as “torch.lstsq() is deprecated in favor of torch.linalg.lstsq(),” and evaluates
499   LLMs using exact match, edit similarity, and fixed rate metrics. Although Versicode and Wang et
500   al.’s datasets address the evolving nature of libraries, their evaluations are limited to string matching.
501
      In contrast, CodeUpdateArena evaluates LLMs’ ability to adapt to API changes, such as adding a
502
      boolean flag, by running tests. However, the dataset is synthetic and are not extracted from real-life
503
      version changes. For CodeUpdateArena, they also take the approach of training LLMs using the
504   updated API function –using docstrings or examples–. It then tests if without access to the update
505   during inference, the LLM’s reflects the synthetic changes. While these datasets provide valuable
506   resources for training and evaluating models, our GitChameleon dataset advances research into
507   version-conditioned code generation by LLMs. Runnable tests offer insights into LLM adaptability,
508   and GitChameleon further assesses a model’s ability to differentiate between library versions, and
509   successfully use a specific version.
510
511
      Implications for Lifelong Learning Continual/lifelong learning in code generation models is in
512
      its early stages (Yadav et al., 2023; Weyssow et al., 2023; Wu et al., 2024a; Gao et al., 2023).
513
      However, current efforts often focus on artificial sequential tasks rather than utilizing the natural
514   distribution shift in the chronological evolution of code. Notably, continual learning mainly targets
515   mitigating catastrophic forgetting and balancing forward- and backward-transfer on a data stream,
516   which may not align optimally with coding environment demands. In coding environments, obso-
517   lete or legacy libraries may prompt selective forgetting of irrelevant knowledge, particularly at the
518   library/package level. Previous work, such as Caccia et al. (2021) may serve as a foundation for
519   developing continual learning in Large Language Models for code.
520
521
522   5    L IMITATIONS
523
524   We consider the lack of prompt optimization done for the instruct models as a considerable limi-
525   tation of our analysis. Furthermore, the dataset consists of 116 problems, which is relatively small
526   compared to other code benchmarks. Finally, we do not explore approaches such as RAG, chain of
527   thought, or finetuning on a split of our benchmark to observe an upper bound of performance on this
528   task. Future work could explore such approaches using our dataset.

531   6    C ONCLUSION
532
533
      Recognizing the crucial need for code LLM adaptation to evolving code environments, particularly
534   in widely used libraries, we introduce a novel and extensive Python-based version-specific bench-
535   mark named GitChameleon. By effectively leveraging GitChameleon, we expose the shortcom-
536   ings of existing state-of-the-art (SoTA) models in producing version-specific code, representing an
537   inaugural effort to draw attention to this challenge. While our work exposes this shortcoming, we
538   acknowledge the dataset’s limitations. In future endeavors, we aim to enhance the dataset’s com-
539   prehensiveness across various programming languages and frameworks. Additionally, we plan to
      introduce new tasks that can benefit research on code LLM models using GitChameleon.
