009                                                A BSTRACT
010
011            In this paper, we present a challenging code reasoning task: vulnerability detection.
012            Large Language Models (LLMs) have shown promising results in natural-language
013            and math reasoning, but state-of-the-art (SOTA) models reported only 54.5%
014            Balanced Accuracy in our vulnerability detection evaluation, even those models
015            pre-trained on large amounts of source code. Our error analysis on LLM responses
016
               shows that the models struggle to reason about the code semantics relevant to
               identifying vulnerabilities, especially subtle semantic differences caused by small
017
               textual changes. We explored prominent models and training settings to understand
018
               their effects on vulnerability detection performance — including better prompts,
019            larger models, more pre-training data, and fine-tuning — but none led to significant
020            improvements. This raises the question of whether simply scaling training data and
021            model size will allow us to “solve” complex code reasoning tasks like vulnerability
022            detection, or if a fundamental shift in modeling and training techniques is required.
023            We also explored adding domain knowledge to prompts; although it helped certain
024            models understand some code semantics, vulnerability detection requires multi-
025            step reasoning, and these models still failed in steps, such as reasoning about
026            variable relations. Our results suggest that new models, new training methods, or
027
               more execution-specific pretraining data may be needed to conquer vulnerability
               detection. We speculate that auto-regressive pre-training on source code may not
028
               effectively extract code semantics, especially on the current pretraining mixtures,
029
               in which execution data is scarce. Success on vulnerability detection as a code
030            reasoning task can benefit many areas of software engineering such as debugging,
031            test input generation, and program repair. Our code and data are available at
032            https://figshare.com/s/78fe02e56e09ec49300b.

035   1    I NTRODUCTION
036
037   Thousands of new software vulnerabilities are discovered each year, costing users and companies
038   millions of dollars (NIST, 2024a). This makes vulnerability detection critically important for software
039
      security. Since Devign in 2019 Zhou et al. (2019), many deep learning approaches have been proposed
      to predict the presence of vulnerabilities, but model performance has not breached 70% F1 score
040
      on realistic datasets Chen et al. (2023). In this paper, we show that though existing LLMs achieve
041
      impressive results on math, natural language, code reasoning and code generation tasks Talmor et al.
042   (2019); Cobbe et al. (2021); Gu et al. (2024); Chen et al. (2021) they struggle to detect vulnerabilities
043   (Section 2). We show that vulnerability detection is a complex code reasoning challenge, requiring
044   both multi-step analysis and an accurate understanding of code semantics. This paper makes the case
045   that vulnerability detection presents a compelling new target task for the ML community; solving it
046   could significantly impact related software engineering tasks, such as debugging, test input generation,
047   and program repair, thereby enhancing developer productivity. Furthermore, improving LLMs’ ability
048   to reason about code and identify vulnerabilities could potentially drive progress in broader reasoning
049   tasks.

051   statements relevant to a potential vulnerability. Second, a developer must understand the semantics
052   of those relevant statements, which requires domain knowledge, such as recognizing bounds/NULL
053   checks and understanding the effects of string, pointer, and arithmetic operations. Sometimes only a
      single operator distinguishes vulnerable and non-vulnerable versions of code, and effective vulnera-

082   bility detection requires understanding these nuances of program semantics. Finally, the developer
083   must logically connect the individual facts about the statements to infer whether a vulnerability exists.
084   This last step requires reasoning about the ranges of values and the temporal relations of symbolic
085   variables, and then comparing them to the application’s security policy, which is often implicit.
086   These steps are challenging for LLMs, both individually and in combination. We studied 14 SOTA
087   LLMs and 7 prompting methods on SVEN (He & Vechev, 2023), a high-quality, real-world dataset
088   consisting of 386 vulnerable functions and their corresponding fixed versions, covering 772 programs.
089   We found that all models and prompts performed close to random guessing (50-55% Balanced
090   Accuracy) (Section 2). Even GPT-4, a SOTA model, couldn’t distinguish vulnerable code from its
091   fixed version for 67.4% cases.
092   After manually analyzing 300 of the LLM responses (Section 3), we found errors occurring at all
093   three steps of the reasoning process. For instance, in step 1, localization, the models frequently
094   (50% of inspected functions) failed to recognize bounds or NULL checks, resulting in false positives.
095   Explicit marking of bounds checks is easily done by humans but seems to be difficult for LLMs to
096   recognize. In step 2, LLMs misinterpreted string, pointer, and integer operations in 10%, 6%, and
097   8% of functions, respectively. Understanding bounds/NULL checks and the operations requires a
098   precise understanding of code execution semantics, which LLMs generally struggle with Gu et al.
099
      (2024); our results confirm this finding and further indicate which structures were most challenging.
      We attribute the models’ lack of understanding of code semantics, even after using various prompting
100
      methods, to two key factors: (1) the models may have limited exposure to execution data during
101
      pre-training, which restricts their ability to learn semantics directly – although LLMs might acquire
102   some semantic understanding indirectly from simple executions aligned with code text, or developer’s
103   discussions about semantics; and (2) the current autoregressive pre-training methods face inherent
104   difficulty of learning execution semantics from code text alone. This is likely why we observe that
105   scaling up model size or dataset volume, and performing fine-tuning, did not significantly improve
106   performance (Sections 3.1 and 3.2); since the necessary data are not in the dataset, it is unlikely that
107   LLMs can learn this complex reasoning via scaling alone. Annotating code semantics in prompts
      reduced some of these errors in certain models (Section 3.3), but determining vulnerabilities require

112
      To the best of our knowledge, this paper is the first to utilize vulnerability detection to systematically
      explore the capabilities of existing LLMs to reason about complex code properties. Ullah et al.
113
      (2023) compared GPT-4’s responses with human-written vulnerability summaries using metrics like
114
      BLEU, ROUGE, and GPT-4 evaluations, but did not delve into the specific failure modes which
115   occurred in responses. Yu et al. (2024); Nong et al. (2024) examined GPT-4 and GPT-3.5’s responses
116   about vulnerabilities but did not perform systematic studies on a set of models, and on the impact of
117   model sizes, training data, training methods, and adding domain knowledge. We classified the errors
118   based on the challenges of reasoning steps, resulting in categories which are more fine-grained and
119   actionable; we explored mitigating a specific type of error using a prototype with CoT-Annotations,
120   as discussed in Section 3.3.
121   In summary, we make the following contributions:
122
123   (1) We clarify vulnerability detection as a complex reasoning challenge;
124
125
      (2) We demonstrate that current SOTA LLMs severely underperform in vulnerability detection,
      achieving only 50-55% balanced accuracy at best;
126
127   (3) Through manual analysis of hundreds of LLM responses, we reveal that LLMs struggle with
128   all stages of reasoning, particularly in understanding semantics of statements involving bound-
129   s/NULL checks, string operations, and pointer handling, which contributes significantly to their poor
130   performance;
131   (4) We show that these reasoning failures and low performance cannot be easily mitigated by
132   increasing model size, improving training data, and applying fine-tuning, even when the model is
133   provided with domain-specific knowledge.
134
135   The fact that vulnerability detection exposes the limitations in current models’ abilities to reason about
136   vulnerabilities, coupled with the availability of well-defined vulnerability data, makes vulnerability
137   detection an ideal benchmark for evaluating and challenging LLM reasoning capabilities.

      Prompts: We used the baseline prompting methods including Basic (zero-shot) Fu et al. (2023) and
142
      In-context (n-shot) Liu et al. (2023b); Zhou et al. (2024) prompts. We used a system prompt to set
143
      the context and instructions including the vulnerability definition (MITRE, 2024) and the program
144   source code (see Section A for details).
145
146   We designed three additional prompting methods that leverage the metadata available in vulnerability
147
      datasets to encourage reasoning and provide the domain knowledge to the model, namely: (1) In-
      context examples from contrastive pairs (Contrastive), which uses pairs before and after a bug-fix
148
      as in-context examples, with the goal of instructing the model the fine-grained differences which
149
      caused the bug, (2) Chain-of-Thought from CVE descriptions (CoT-CVE), which uses CVE bug
150   reports (CVE, 2024) from the Big-Vul dataset (Fan et al., 2020), prompting the model to respond with
151   the explanations of vulnerability, and (3) Chain-of-Thought from static analysis (CoT-StaticAnalysis),
152   which adapts vulnerability proofs output by a static analyzer (Calcagno & Distefano, 2011) as
153   reasoning steps for the example response, conditioning the model to reason step-by-step. We obtained
154   the proofs from the D2A dataset Zheng et al. (2021) (see Section A for details).
155
      Models: We evaluated 14 LLMs which are the SOTA in code generation, based on Zhao et al.
156   (2023); Liu et al. (2023a) and the HumanEval leaderboard (PapersWithCode, 2023) (as of March
157   2024). The models include LLAMA 2 (Touvron et al., 2023), Code LLAMA (Roziere et al., 2023),
158   StarCoder (Li et al., 2023b), StarChat (Tunstall et al., 2023), StarCoder2 (Lozhkov et al., 2024),
159   StarChat2 (HuggingFaceH4 Team, 2024), Mistral (Jiang et al., 2023), Mixtral (Jiang et al., 2024),
160   MagiCoder (Wei et al., 2023), Wizardcoder (Luo et al., 2023), DeepSeek-Coder(Guo et al., 2024),
161   GPT-3.5 (OpenAI, 2023), GPT-4 (OpenAI, 2024), and Gemini 1.0 Pro (Gemini Team, 2023). See
      Section B for details.

495
      Recent studies have initiated investigation into the usage of LLMs for vulnerability detection, using
496   zero-shot prompting Purba et al. (2023); Fu et al. (2023), in-context learning Gao et al. (2023); Liu
497   et al. (2023b); Chan et al. (2023), and fine-tuning Shestov et al. (2024); Yusuf & Jiang (2024); Yang
498   et al. (2024). Several papers have utilized chain-of-thoughts (CoT), such as “Let’s think step-by-
499   step” Li et al. (2023a); Feng & Chen (2024); Sun et al. (2024), multi-step prompts Ullah et al. (2023);
500   Yu et al. (2024), and generic information such as CFG, DFG, PDG, and API calls Zhang et al. (2023);
501   Nong et al. (2024); Khare et al. (2023); Ullah et al. (2023). In this work, we studied the most common
502   prompting methods and proposed four novel prompt approaches tailored for vulnerability detection,
503   integrating information from bug-fix commits (contrastive pairs), CVE descriptions (CoT-CVE),
504
      static analysis reports (CoT-StaticAnalysis), and domain knowledge annotations (CoT-Annotations).
      We further studied the LLMs’ capabilities to distinguish buggy and patched versions of code and
505
      studied the reasoning errors in their responses.
506
507   Several recent papers have analyzed errors in LLM-generated vulnerability detection responses. Ullah
508   et al. (2023) used BLEU, ROUGE, and GPT-4 to automatically compare GPT-4’s reasoning summaries
509   with human-generated ones. Yu et al. (2024); Nong et al. (2024) examined 82-100 responses from
510
      GPT-4 and GPT-3.5, supporting our findings that the models struggled with correctness, logic and
      consistency in general. However, existing studies do not match the depth and breadth of ours. Our
511
      error classifications provide more actionable and detailed categories, enabling us to identify specific
512
      code structures and LLM weaknesses (see Table 3). Additionally, we analyzed factors such as model
513   size, training data, and training strategies, providing cause for concern about future improvements
514   from model scaling. To our knowledge, our study is the most comprehensive manual analysis of
515   LLMs for vulnerability detection, including 14 models and manually analyzing 300 LLM responses
516   with a rigorous multi-rater agreement protocol.

      5    C ONCLUSION
521
522
      In this paper, we have show that vulnerability detection is complex, multistage reasoning task that
523   current LLMs struggle to solve. We conducted a thorough study to show that the SOTA models and
524   prompts performed only slightly better than random guessing. None of the model advancements
525   we explored led to significant improvements, including increasing model size, expanding training
526   data, and instruction/adapter fine-tuning. The models particularly struggled to distinguish between
527   vulnerable and fixed versions of code, where small textual differences cause large changes in
528   semantics. We demonstrated that external tools and domain knowledge helped somewhat with
529   single-step reasoning, but did not significantly improve the models’ performance, which depends on
530   accurate multi-step reasoning. Our findings bring concerns about further research in this area, raising
531
      the question of whether auto-regressively pre-trained LLMs are a good fit for tasks which require
      deep understanding of code semantics. We suggest that a fundamental shift in modeling and training
532
      methods may be necessary in order to overcome the reasoning failures of current LLMs. We believe
533
      that solving code reasoning in vulnerability detection could help address many other challenging tasks
534   in software engineering, such as debugging, code execution prediction, test input generation, and
535   program repair. Reasoning-based models fine-tuned for inherent chain-of-thought, such as OpenAI
536   o1 (OpenAI, 2024) and DeepSeek R1 (DeepSeek, 2024), offer a promising approach to this challenge.
537   Furthermore, frequent localization/understanding errors, such as missing bounds/NULL checks in
538   50% of cases, demonstrate the need for additional context or scaffolding, as demonstrated by our
539   CoT-Annotations prototype Section 3.3. We hope that our paper laid out some key insights and
      motivation for the machine learning community to solve this important challenge.

759   A PPENDIX A         V ULNERABILITY DETECTION PROMPTS
760
761   We explored several prompt designs, guided by model performance on a small dev set or the entire
762   SVEN dataset.
763
764
      Basic (zero-shot) prompting (Fu et al., 2023): We first designed a system prompt to set the context:
      “I want you to act as a vulnerability detection system”, along with natural-language instructions: (1)
765
      Basic query: “Is the following function buggy? Please answer Yes or No.” (We also tried “Is the
766
      following function vulnerable?”; however, our pilot study shows that it did not perform as well.) (2)
767   CWE list: This prompt starts with “Does the following function contain one of the following bug
768   types?”, followed by a fixed list of bug types, e.g., “CWE-190: Integer Overflow”; (3) Q/A: Begin the
769   query with “Question:” and begin the model’s response with “Answer:”. This conditions the model
770   to respond in a question-answering mode.
771
      In-context (n-shot) prompting(Liu et al., 2023b; Zhou et al., 2024): In this prompt, we provide
772   examples of inputs and responses for in-context learning (Brown et al., 2020). The in-context
773   examples condition the model to reply in the same format as the example responses (Xie & Min,
774   2022). The selection of in-context examples can impact the performance. We studied three settings:
775   (1) randomly selecting examples, (2) using retrieval-augmented generation (RAG, Lewis et al. (2020))
776   to retrieve the examples that had similar CodeBERT (Feng et al., 2020) embeddings to the query
777   example, and (3) selecting examples from contrastive pairs (see below for details). We explored
778   several options for formatting in-context examples, such as appending all the examples in one chat-
779   assistant message versus using separate messages (one message performed best) and varying the
780
      number of examples from 1 to 10 (6 performed best).
781   In-context prompting based on contrastive pairs: We formed contrasting pairs of in-context
782   examples by providing the vulnerable version of the code (before the bug-fixing commit) and the
783   fixed version (after the commit) as in-context examples in the same prompt. Since these two versions
784   of the source code differ primarily in the portion related to the bug-fix, our intention is that this
785
      prompt template would highlight the cause of the bug and instruct the model to learn that the small
      differences in code can lead to different labels.
786
787   In-context prompting based on CoT from CVE descriptions: We designed “chain-of-thought”
788   prompts by providing intermediate reasoning steps which lead to the answer, inspired by Wei et al.
789   (2022b). We use in-context examples from the Big-Vul dataset (Fan et al., 2020), which includes the
790   CVE bug reports. For vulnerable examples, we used the default in-context query and provide the
791
      chain-of-thought response. To produce such response, we adapt the descriptions in these bug reports
      to describe how the bug manifests. For example, CVE-2017-9211 Corporation (2024) describes the
792
      vulnerability, including the symptoms, attack surface, and variable involved:
793
794   The crypto skcipher init tfm function in crypto/skcipher.c in the Linux
795   kernel through 4.11.2 relies on a setkey function that lacks a key-size
796
      check, which allows local users to cause a denial of service (NULL
      pointer dereference) via a crafted application.
797
798   We use this description as the CoT response and append “Therefore, the example is buggy” to
799   complete the example response. For non-vulnerable examples, we provide the default in-context
800   example query/response.
801   To ensure high-quality examples in spite of label noise Croft et al. (2023), we removed duplicate
802   examples, excluded examples with overly long or short source code (50-750 tokens), and retained
803   only those examples tied to vulnerabilities (i.e., bug reports) of the same types in SVEN.
804   In-context prompting based on CoT from static analysis: We also used the output buggy paths
805   reported by the Infer (Calcagno & Distefano, 2011) static analysis tool to prepare the chains of thought
806   prompt. Infer reports a single bug-triggering path for each example. The buggy path consists of a
807   list of statements that can lead to the bug. We use in-context examples from the D2A dataset Zheng
808   et al. (2021), which lists buggy paths from the Infer static analyzer Facebook (2024) for several
809   open-source C++ projects. We convert the buggy paths to natural language descriptions and use them
      as the response. This is an example CoT response for a buffer overflow vulnerability:

817   To ensure high-quality examples in spite of label noise Croft et al. (2023), we removed duplicate

       A PPENDIX F         E RROR ANALYSIS METHODOLOGY

974
       To support measurable and targeted improvements in model reasoning, we open-sourced our error
975
       analysis tool Anonymous (2024). Researchers can use this tool to quickly analyze a sample of LLM
976    responses to measure whether and how much the LLMs improved on a specific reasoning error, as we
977    demonstrated in Section 3.3. By offering concrete metrics and a practical tool to measure them, we
978    aim to accelerate both advancements in LLM reasoning research and the adoption of reasoning-based
979    models for vulnerability detection tasks.

984    raters independently identified errors in the LLM responses, refining the protocol after processing 1⁄3,
       1⁄ , and all of the data. We added new error categories when needed, and merged similar categories

987    achieving 0.78 with 86% agreement. We resolved disagreements by majority vote, followed by

       discussion for the final categorization. After the categories were set in Section 3, we used one rater to
       analyze the responses reported in Sections 3.1 to 3.3.
