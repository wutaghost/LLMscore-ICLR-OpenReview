011                                               A BSTRACT
012
013            Generative Language Models rely on autoregressive decoding to produce the output
014            sequence token by token. Some tasks, such as preference optimization, require
015            the model to produce task-level output consisting of multiple tokens directly by
016            selecting candidates from a pool as predictions. Determining a task-level prediction
017
               from candidates using the ordinary token-level decoding mechanism is constrained
               by time-consuming decoding and interrupted gradients by discrete token selection.
018
               Existing works have been using decoding-free candidate selection methods to
019
               obtain candidate probability from initial output logits over vocabulary. Though
020            these estimation methods are widely used, they are not systematically evaluated,
021            especially on end tasks. We introduce an evaluation of a comprehensive collection
022            of decoding-free candidate selection approaches on a comprehensive set of tasks,
023            including five multiple-choice QA tasks with a small candidate pool and four
024            clinical decision tasks with a massive amount of candidates, some with 10k+
025            options. We evaluate the estimation methods paired with a wide spectrum of
026            foundation LMs covering different architectures, sizes and training paradigms. The
027            results and insights from our analysis could inform the future model design.

      1   I NTRODUCTION
030
031   Large Language Models (LLMs) have shown amazing performance after pre-training on a massive
032   corpus (Lewis et al., 2020), instruction tuning (Longpre et al., 2023), and preference alignment (Etha-
033
      yarajh et al., 2024). Generative LMs respond to queries by generating tokens to form an output
      sequence and optimize themselves by learning to generate the correct tokens. The simplicity of
034
      token-level inference and optimization compromises its performance on end tasks, as there is a gap
035
      between the token-level paradigm and sequence-level task results and learning signals.
036
037   Some tasks use generative LM to select the answer(s) from a given pool of options where each
038   candidate answer is a natural language sequence. With such a task formulation, both LLMs’ gener-
039
      alizable reasoning capabilities on novel scenarios and domain expertise contained in the candidate
      space can be utilized. Multiple-choice QA considers answer options as the candidate pool (Khashabi
040
      et al., 2020). The large collection of labels are candidate answers for extreme label classification
041
      tasks (Amigo & Delgado, 2022). The retrieval task aims to extract relevant documents from a large-
042   scale evidence corpus (Lu et al., 2023). The candidate pool can also be expert-curated professional
043   coding systems (Taylor et al., 2022). For example, when instructing the model to prescribe medi-
044   cations, items in drug databases are candidates (Fleming et al., 2023); when conducting diagnoses,
045   disease ontology forms the candidate space (Singhal et al., 2023). The typical practice is to use full
046   decoding to generate an output sequence and then match the natural language output with candidates
047   in the pool to select candidates to be predicted as answers (Mishra et al., 2022). However, selecting
048   candidates using full decoding not only cuts off the gradient flow and disables direct optimization on
049   decoded results but also limits the output bandwidth due to time-consuming discrete decoding.
050   Existing works perform candidate selection without decoding for outcome-level optimization or
051   efficient parallel predictions. For example, Ma et al. (2023b) calculate averaged logits of MCQA
052   options to select an answer without decoding; Xu et al. (2023b) estimate the NLI result using logits of
053   a single token. Though these decoding-free candidate selection practices are widely used, there is no
      formal definition or clear investigation of the properties of each method. There is also no consensus

155   2.4   G ENERATIVE CANDIDATE SELECTION WITHOUT DECODING
156   At this point, we formally define the task of decoding-free generative candidate selection. In
157   §2.3, we introduce generative candidate selection using decoding, where the answer is reflected by
158   the output sequences. However, there are multiple severe limitations of decoding-based candidate
159   selection. On the one hand, the discrete argmax operator for token selection interrupts the gradient
160   flow, making applying objectives on task outcomes inefficient, such as using reinforcement learning
161   with one-per-outcome sparse rewards instead of token-level feedback. On the other hand, the decoding
      process is time and resource-consuming, limiting the output bandwidth of generative LMs.

185   2.5   D IFFERENCE COMPARED WITH ORDINARY APPROACHES
186
      The common property between the generative candidate selection and classification is that both
187
      settings require a given set of selections to produce the final output. However, candidate selection is
188
      different from classification in many key aspects summarized in Table 1, specifically: 1) Support of
189   dynamic candidates. A classification model has to use the same set of output labels across all instances
190   (e.g. positive or negative for sentiment classification). However, candidate selection methods allow
191   the task to have a different set of output candidates for each instance (e.g. different answer options for
192   each question in MCQA). 2) No need of additional parameters. The classification head is an additional
193   set of parameters specialized for the defined output classes. Different classification tasks have to
194   use a separate set of parameters. Candidate selection methods support various candidates with the
195   generative LM’s native parameters only, without any additional parameters. 3) No need for specialized
196   training. The additional parameters for each classification task need to be parameterized by training
197
      on task-specific data, preventing it from generalizing to new tasks or labels. However, decoding-free
      estimation methods support zero-shot candidate selection, as the possibilities of choosing options are
198
      calculated dynamically according to corresponding candidates.

      3.2    O RDINARY CANDIDATE SELECTION METHODS TO COMPARE
245   Full decoding. Following the decoding paradigm introduced in §2.1 and how to induce answers
246   within a candidate pool from the full decoded output sequence introduced in §2.3, we perform
247   full decoding to obtain an output sequence, then use a mapping function to find the corresponding
248   predicted answer from the given candidate pool. The mapping function is task-dependent; we use
249   the task-specific mapping rules introduced along with each dataset. Typical practices include using
250   regular expressions to match patterns in the output sequences, such as phrases like “Answers: ”, and
251   predicting the candidate with the highest semantic similarity with the output sequence.
252   Dense retrieval. In the retrieval baseline, we formulate the candidate selection task as a retrieval
253   task and use dense passage retrieval as one of the reference models. Separate encoders are employed
254   for encoding queries and candidate choices. Specifically, the question and each candidate choice
255   are embedded into a high-dimensional vector space using these encoders. Cosine similarity is then
256   computed between the question embedding and each candidate choice embedding. This similarity
257   score quantifies the relevance of each choice to the posed question and determines the probability of
258   each choice being the correct answer. For our experimental setup, we use the Facebook DPR question
259
      encoder and context encoder (Karpukhin et al., 2020) to generate embeddings of the questions and
      candidate choices, respectively.

467   Using the logits of the first output step, without additional subsequent decoding, has been the default
468   setting to estimate the candidate selection in many works. It is also most efficient without additional
469   decoding steps. We empirically show that using the logits of the first output step to estimate the
470
      candidate selection is the optimal solution in terms of both estimation performance and efficiency.
471
      5.3   E STIMATION PERFORMANCE USING SELECTED CANDIDATE KEYWORDS
472
473   We investigate the estimation capabilities when only the logits of the most important keywords of
474   each candidate sequence are considered. We prompt GPT-4o to select a certain number of the most
475   important and informative tokens among all of each candidate sequence. We then only calculate the
476
      candidate probability using logits of the selected tokens.
477   Insight 7: Using full candidate sequence for estimation is better than selecting essential tokens.
478   In Figure 2(b), we observe that as the considered tokens become more concise and selective (the
479   number of selected candidate tokens becomes fewer), the estimated results of various methods
480   converge to a similar range with a worse recall for most estimation methods. This indicates that it
481   is not necessary to only use essential tokens of the candidate sequence during estimation if it is not
482
      First-only logits being used to derive the selection.

      6             R ELATED WORKS
518
519
      To perform candidate selection from a candidate pool, existing works use a classification head on top
      of encoder outputs (Milios et al., 2023; Yamada et al., 2020; Li et al., 2022). However, classification
520
      formulation requires additional parameters and training while not supporting novel classes and
521
      dynamic candidates for each instance. The generative candidate selection we discussed keeps its
522   flexibility and generalizability with a large throughput. To speed up inference, different parallel and
523   efficient decoding methods are proposed (Bae et al., 2023; Zhang et al., 2018; Huang & Mi, 2010).
524   However, our goal is not to speed up decoding but to evaluate the methods approximating decoded
525   results without decoding. To select a candidate from a pool using the token logits without discrete
526   decoding, existing works propose to obtain the probability of each candidate through aggregating
527   different parts of token logits such as only keeping logits of a special token (Xu et al., 2023a),
528   averaging logits (Saeidi et al., 2024; Song et al., 2024; Ethayarajh et al., 2024; Xiong et al., 2024), or
529   multiplying logits (Ma et al., 2023c). We conduct the first systematic evaluation on these methods.
530
531   7             C ONCLUSION AND FUTURE WORK
532
533
      Obtaining task-level output is crucial in various practical scenarios, including clinical decision-making
534   and preference alignment. However, ordinary generative language models operate at a token-level,
535   which necessitates significant post-processing effort to extract task-level output. We address the
536   general task of selecting the best-matching candidates given a context or question, with no restrictions
537   on domain or candidate space. We conduct a systematic evaluation of various decoding-free candidate
538   selection approaches on tasks with diverse question domains and varying candidate spaces. Further
539   improvements can be achieved by considering more advanced sequence representation methods, such
      as text summarization.

793   only use the logits of the first potential output token without producing the token. Calculating logits

797   to a query through full decoding (as demonstrated in Equation 4), accurate decoding-free candidate
798   selection methods (as shown in Equation 5) are needed, especially for two scenarios. 1) Accurate
799   outcome-based optimization. To optimize the model with the feedback directly from the predicted

813   What are the potential usage and broader impact of the evaluation done in this work? The
814
      conclusions and observations derived from our evaluation provide evidence for more informed and
      confident design choices for both optimizations with outcome-level feedback and efficient answer
815
      production without decoding. When researchers and industry practitioners need to define a function
816
      to estimate the possibility of potential answers using generative language models without decoding
817   the output sequence, they can: 1) choose the best estimation method corresponding to their model
818   architecture and end tasks according to our evaluation results; 2) understand the empirical tradeoff
819   between efficiency, in terms of runtime, and estimation quality, in terms of performance difference;
820   3) decide whether they are confident to use estimation method instead of decoding (especially when
821   the estimation methods provide better performance for non-instruction-tuned models). With the wise
822   decision of the candidate selection method, they can obtain better performance after training the
823   model with answer-level rewards, such as through preference alignment, and produce the predicted
824   answers faster with lower resource usage by replacing decoding with estimation.
825   What are the differences between the two types of tasks used in the evaluation? We quantify the
826   effects of various decoding-free estimation methods in downstream scenarios by using two types of
827   evaluation tasks: tasks with limited numbers of candidates (specifically 5 multiple-choice QA tasks)
828   and tasks with massive numbers of candidates (specifically 4 clinical decision tasks). We summarize
829   their core differences in Table 5.

846   candidate pools across instances for tasks with massive numbers of candidates.
847   Why not use an agent-based system to handle massive candidates? We can provide multiple
848   functions and tools for LLM agents to search, match, or traverse relevant candidates from a large pool
849   of candidates. Compared with full decoding, it will provide more information about the candidate
850   pool and potentially lead to better performance. However, formulating the candidate selection task
851   as an agent is based on and expanded from the idea of decoding discrete tokens to produce answers
852   from output sequences (as described in Equation 4); it does not enjoy the benefits of decoding-free
853   methods, and it is not a comparable setting of the methods we focus on in this paper.
854   Which decoding methods are you using to compare? We use the default decoding setting for
855   each model specified in their generation configuration file. Our work does not aim to propose a new
856   decoding method or compare the performance of various decoding methods. Instead, we emphasize
857   the benefits and limitations of decoding-free candidate selection methods.
858   How is the evaluation performed in this paper different from the evaluations provided in the
859   previous works that use those decoding-free methods? Existing works do not consider how to
860   represent the response candidate from the logits of a single output step as a standalone problem.
861   Thus, they do not provide justification, theoretical proof, or evaluation of the design choice of the
862   decoding-free candidate selection method they used in their works. Our work aims to raise awareness
863   of the importance of this design choice and conduct the first thorough definition of the task and
      systematic evaluation.

875
      test deep domain-specific knowledge. (4) BIG-Bench (Srivastava et al., 2022) includes tasks like
876   boolean expression evaluation and causal judgement based on stories, focusing on logical reasoning
877   capabilities. We select the “logical deduction” category with three objects for our experiments. (5)
878   ARC (Clark et al., 2018) comprises 7,787 multiple-choice questions at grade-school level, divided
879   into a Challenge Set and an Easy Set, to test scientific knowledge. We opt for the Easy Set in our
880   experiments.
881
      We report accuracy and per-instance runtime for these tasks. These datasets are split into subsets
882   such as train or test. We select one of the dataset splits with available answer keys for our study.
883   For the mapping function fmap converting output sequence seqout to candidate selection ans,     ˆ we
884   capture the first the answer candidate sequence or candidate indication head (e.g. A, B, C D) appeared
885   in the output sequence with regular expressions and use the matched candidate as the prediction.
886   All candidate options are added in the input prompt, thus full decoding and decoding-free selection
887   methods use the same amount of available information. We make sure all input sequences for full
888   decoding or decoding-free candidate selection methods are exactly the same.

894

895   the original benchmark setting by selecting the candidate with the highest cosine similarity between

901   type, significantly increasing the difficulty of candidate selection. For full decoding, the model
902   can determine the number of predictions made because the generation of the end-of-sentence token
903   indicates stopping making additional predictions. Decoding-free candidate selection methods rely on
904   candidate probability, and it is hard to determine a fixed threshold for all instances. Thus, we take 20
905   candidates with the highest probabilities, which contain more predictions than ground-truth answers
906   for most testing instances. We then only report recall, indicating the portion of ground-truth answers
907   that are correctly predicted, to mitigate the influence of uncertain selection probability threshold.
908   To speed up the inference of full decoding, we wrap the generative LM with vLLM framework (Kwon
909   et al., 2023), which leverages paging techniques in the operating system to optimize memory usage.

       E       L IMITATIONS
1264
1265
       In terms of accuracy, current estimation methods have room for improvement due to their reliance on
       initial logits and simplified representative tokens (e.g., first, average). Future work could consider
1266
       using logits from more time steps or leveraging LLMs to summarize the candidates into a few words,
1267
       potentially serving as more effective representative tokens.
1268
1269   Regarding efficiency, computing logits dominates the runtime of estimation approaches. Applying
1270   advanced techniques, such as PagedAttention, to optimize memory usage can further enhance the
1271
       efficiency of estimation methods, especially for tasks with lengthy prompts.
1272

1275   As previously mentioned, the accuracy of zero-shot estimation methods is compromised. Therefore,
1276   directly adopting the decisions made by these methods without careful judgment could lead to
1277   unintended consequences in practical scenarios, such as clinical decision-making.
