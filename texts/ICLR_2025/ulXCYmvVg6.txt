009                                               A BSTRACT
010
011
               As the use of large language models (LLMs) for code generation becomes more
012            prevalent in software development, it is critical to enhance both the efficiency
013            and correctness of the generated code. Existing methods and models primarily
014            focus on the correctness of LLM-generated code, ignoring efficiency. In this work,
015            we present E FFI -C ODE, an approach to enhancing code generation in LLMs that
016            can improve both efficiency and correctness. We introduce a Self-Optimization
017            process based on Overhead Profiling that leverages open-source LLMs to generate
018            a high-quality dataset of correct and efficient code samples. This dataset is then
019            used to fine-tune various LLMs. Our method involves the iterative refinement of
020
               generated code, guided by runtime performance metrics and correctness checks.
               Extensive experiments demonstrate that models fine-tuned on the E FFI -C ODE
021
               show significant improvements in both code correctness and efficiency across task
022
               types. For example, the pass@1 of DeepSeek-Coder-6.7B-Instruct generated code
023            increases from 43.3% to 76.8%, and the average execution time for the same
024            correct tasks decreases by 30.5%. E FFI -C ODE offers a scalable and generalizable
025            approach to improving code generation in AI systems, with potential applications
026            in software development, algorithm design, and computational problem-solving.

      1   I NTRODUCTION
030
031
032   Large language models (LLMs) have recently made significant strides across various tasks (OpenAI,
033
      2023; Anil et al., 2023; Anthropic, 2024; Meta, 2024), including code-related applications like code
      completion (Chen et al., 2021; Austin et al., 2021), debugging (Haque et al., 2022; Chen et al., 2023),
034
      and translation(Rozière et al., 2020; Ahmad et al., 2023). These advanced tools have been seamlessly
035
      integrated into popular development environments, enhancing developer productivity by providing
036   intelligent code recommendations based on natural language instructions.
037
038   Before deploying LLMs into integrated development environments (IDEs) as tools, it is crucial to
039
      ensure that the generated code meets the required efficacy standards. To address this, researchers
      have explored various datasets to fine-tune LLMs, thereby improving the efficacy of LLM-generated
040
      code (Ouyang et al., 2022; Wei et al., 2022). For example, Code Alpaca (Chaudhary, 2023) utilized
041
      the Self-Instruct framework (Wang et al., 2023) to synthesize data, while WizardCoder (Luo et al.,
042   2024) employed the Evol-Instruct technique (Xu et al., 2024) to generate heuristic prompts for
043   diverse solutions. Additionally, OSS-Instruct (Wei et al., 2024) created new coding problems using
044   open-source snippets with LLMs, and Octopack (Muennighoff et al., 2024) focused on curating
045   high-quality Git commit messages that resemble natural language instructions. These fine-tuning
046   efforts have led to increased correctness in LLM-generated code.
047
      However, our observation is that existing works primarily focus on enhancing the correctness of
048   LLM-generated code while neglecting to optimize its efficiency. As a result, the efficiency of
049   such code often falls short compared to canonical solutions written by human developers. Recent
050   studies (Shi et al., 2024; Niu et al., 2024; Du et al., 2024; Huang et al., 2024a) also point out that
051   LLM-generated code typically exhibits lower efficiency in terms of execution time and memory
052   usage. For instance, on the EffiBench benchmark (Huang et al., 2024b), even the most advanced
053   LLMs, such as GPT-4-Turbo, produced less efficient code, with average and worst-case execution
      times being 1.69 and 45.49 times longer than those of canonical solutions, respectively.

054
      Efficiency is crucial because inefficient code consumes more computational resources, leading to
055   higher energy consumption and increased operational costs. This is particularly important in the
056   context of sustainability, as the demand for computing power continues to grow, and reducing
057   the environmental impact of large-scale computations becomes a pressing concern. Furthermore,
058   inefficient code may be impractical for use in resource-constrained environments, such as mobile
059   devices or embedded systems, where both energy and processing power are limited. This underscores
060   the urgent need to develop new methods that can enhance both the correctness and efficiency of
061   LLM-generated code.
062   In this paper, we introduce the dataset E FFI -C ODE, aimed at fine-tuning LLMs to improve both code
063   efficiency and correctness. We begin by aggregating source code from eight existing open-source
064   datasets available on the Hugging Face platform. This is followed by a rigorous preprocessing and
065   cleaning process, coupled with the generation of test cases for each task to evaluate code efficiency.
066   The cleaned code is executed using test cases to profile memory usage and execution time Huang
067   et al. (2024a). Through a self-optimization process based on these profiles Huang et al. (2024a), we
068   iteratively refine the code over five optimization cycles. The resulting optimized code, along with its
069
      associated metadata, forms the foundation of our fine-tuning dataset, E FFI -C ODE, which serves as a
      high-quality resource for training LLMs to generate more efficient code while ensuring correctness.
070
071   Extensive experiments on HumanEval (Chen et al., 2021) and EffiBench (Huang et al., 2024b)
072   demonstrate that fine-tuning LLMs with E FFI -C ODE improves both correctness and efficiency. For
073   example, the fine-tuned DeepSeek-Coder-6.7B (DeepSeekAI, 2023) increases the pass@1 from 43.3%
074   to 76.8% on HumanEval, while also reducing the average execution time from 0.59 seconds to 0.41
075
      seconds — representing a 30.5% reduction in execution time overhead. Compared to PIE (Shypula
      et al., 2024), which increases the pass@1 from 12.2% to 19.5% on HumanEval, the pass@1 of
076
      CodeLlama-7B (Rozière et al., 2023) fine-tuned with E FFI -C ODE further increases to 37.8%. In
077
      addition, E FFI -C ODE decreases the execution time by 7.1% while PIE decreases it by 4.8%. We
078   will fully open-source E FFI -C ODE, the source code, and model weights to facilitate research. To
079   conclude, this paper makes the following contributions:
080
081          • We provide a framework to inspire researchers to construct code generation datasets contain-
082            ing efficient solutions for each code generation task, which is versatile and can be adapted to
083            different programming languages and leverage various existing data sources. Unlike some
084            other code generation datasets that rely on powerful models (e.g., GPT-4), our framework
085            can be implemented only using open-sourced LLMs. The framework provides a systematic
086
               method for researchers to enhance existing datasets or create new ones focused on code
               efficiency across different languages and domains.
087
088          • Based on our proposed framework, we release the Effi-Code dataset. To the best of our
089            knowledge, it is the first instruct tuning dataset that focuses on improving the efficiency of
090
               LLM-generated code. The primary purpose of Effi-Code is to instruct and fine-tune LLMs
               to ensure that the LLM-generated code is more efficient.
091
092          • We use Effi-Code to fine-tune widely used LLMs and will release these models on the
093            Hugging Face website in the final version. Different from existing datasets that are used to
094            finetune the LLMs to improve the pass@1 of LLM-generated code, our evaluation results
095
               demonstrate that both the pass@1 and the efficiency results would be improved for LLMs
               finetuned on our Effi-Code dataset.

      Instruction tuning has proven effective in enhancing the usability and overall performance of LLMs
101
      across various language tasks (Ouyang et al., 2022; Wei et al., 2022; Zhao et al., 2024). This approach
102   has been extended to the domain of code generation. The core challenge is the acquisition of high-
103   quality instructional data, which is often labor-intensive. To address this, recent research has focused
104   on developing methods to generate synthetic instruction data. Studies have shown that textbook-
105   quality synthetic data alone can improve a model’s coding and reasoning capabilities (Gunasekar
106   et al., 2023; Li et al., 2023b). One early effort was Self-Instruct (Wang et al., 2023), which utilized
107   LLMs to generate synthetic instruction-response pairs using carefully crafted prompts. The same
      LLM was then instruction-tuned on this synthetic data. Code Alpaca (Chaudhary, 2023) applied the

      Self-Instruct approach with GPT models, tailoring it specifically for code generation, editing, and
109   optimization tasks. Building upon this, WizardCoder (Luo et al., 2024) adapted the Evol-Instruct
110   technique (Xu et al., 2024) to the coding domain by designing heuristic prompts to create more
111   complex and diverse synthetic data. OSS-Instruct (Wei et al., 2024) took a different approach by
112   leveraging LLMs to automatically generate new coding problems inspired by random code snippets
113   from open-source repositories. In contrast, Octopack (Muennighoff et al., 2024) focused on collecting
114   and filtering high-quality Git commit messages that resemble natural language instructions. While
115   these existing methods primarily emphasize generating correct code, E FFI -C ODE explores the use of
116   fine-tuning to improve code efficiency. Our method is orthogonal to existing synthetic techniques,
117
      offering the potential for combination to further enhance the coding capabilities of LLMs.

119   3       E FFI -C ODE : F INE -T UNING F OR E FFICIENCY
120
121   In this section, we provide a det ailed pipeline for constructing the dataset E FFI -C ODE for fine-tuning.
122   Specifically, we first collect source code from eight existing open-source datasets available on the
123   HuggingFace platform1 . To ensure the quality and usability of the collected data, we use several
124   filtering strategies, such as filtering tasks that are not algorithmic tasks and do not require efficiency
125
      optimization2 . In addition, we also generate test cases for each task to ensure that we can measure
      the efficiency of each task’s source code.
126
127   Next, we execute the cleaned source code using the generated test cases to profile the memory usage
128   and execution time for each task. Then, we use Self-Optimization based on these overheAd Profiles
129   (SOAP; Huang et al. 2024a), which iteratively refines the code over five optimization cycles to
130   generate an efficient solution for each task in the collected tasks. Finally, we process the optimized
131
      code and the associated metadata to create our final fine-tuning dataset, E FFI -C ODE, which is carefully
      curated to provide a high-quality resource for training models to generate more efficient code while
132
      maintaining correctness.
133
134
      3.1      S OURCE DATA C OLLECTION
135
136   To construct a high-quality dataset to improve code efficiency, the first important step is to collect
137   a large number of code task candidates, which will be used for further processing. In our exper-
138   iments, we collected code candidates from several existing code generation tasks. As shown in
139   Table 1, the collected datasets include CodeFeedback-Filtered-Instruction (CodeFeed; MAP 2023),
140   Tested-143k-Python-Alpaca (Alpaca; Vezora 2023), Glaive-Code-Assistant (Glaive; Computer 2023),
141   Magicoder-Evol-Instruct-110K (Evol-Ins; UIUC 2023a), Dolphin-Coder (Dolphin; Computations
142   2023), Magicoder-OSS-Instruct-75K (Oss-Ins; UIUC 2023b), Self-OSS-Instruct-SC2-Exec-Filter-
143   50K (Self-Oss; BigCode 2023), and Apps (Hendrycks et al., 2021).

194   case with the initial solution. In other words, we verify if the test case passes the initial solution.
195   Since the initial solutions are correct, we treat the test cases that pass the canonical solution as correct.
196   On the other hand, test cases that do not pass the canonical solution are filtered out. By using the

202   reasoning. Instead, they might focus on straightforward implementation or basic syntax usage. For
203   example, an algorithmic task may be Implement a function to find the longest palindromic substring
204   in a given string. This requires an understanding of dynamic programming and string manipulation
205   algorithms. While a non-algorithmic task may be Write a function to print ’Hello, World!’. This is a
206
      clear example of routine implementation without algorithmic challenges. The primary motivation
      for filtering out non-algorithmic tasks is to ensure that our dataset focuses on problems that assess
207

208
      solving, we maintain the coherence and relevance of our dataset to the intended purpose of evaluating
209   AI models’ coding abilities. To identify and filter out non-algorithmic tasks, we provide the task
210   description and the canonical solution to GPT-3.5-turbo and request it to analyze whether the given
211   task is an algorithmic task based on our provided definition. GPT-3.5-turbo is instructed to return a
212   binary classification (True or False) based on its analysis. Tasks classified as False are considered
213   non-algorithmic and are subsequently removed from our candidate tasks.

      3.3   S ELF -O PTIMIZATION BASED ON OVERHE A D P ROFILE (SOAP)
217
218   To optimize the source code in our collected tasks, we employ the Self-Optimization based on
219   overheAd Profile (SOAP; Huang et al. 2024a) to optimize the efficiency of the source code. For each
220   task in our dataset, we execute the source code using the generated test cases and profile the execution
221   time and memory usage for each line of code using the line_profiler and memory_profiler
222   libraries in Python. The profiling results, along with the original source code and task description,
223   are then fed into DeepSeek-Coder-V2-Lite (Zhu et al., 2024), which analyzes the profiles to identify
224
      performance bottlenecks and inefficiencies in the code. The model applies various optimization
      techniques to refine the code for better efficiency, and the optimized code is validated against the
225
      provided test cases to ensure its functional correctness. This process is repeated for a predefined
226
      number of optimization iterations. By applying SOAP to our collected tasks, we create a dataset of
227   optimized source code that demonstrates improved efficiency compared to the original code. This
228   dataset serves as a valuable resource for training models to generate more efficient code and for
229   understanding the effectiveness of LLM-driven code optimization techniques.
230
231   3.4   P OST-SOAP C LEANING
232
233   After generating efficient source code based on SOAP, we then filter tasks in our candidate pool to
234   enable our fine-tuning process.
235   Filtering tasks not addressed by the Teacher Model (Step 5): As mentioned in Section 3.3, we
236   use DeepSeek-Coder-V2-Lite to construct more efficient solutions for our candidate tasks. However,
237   some tasks are not addressed by DeepSeek-Coder-V2-Lite, which means that we cannot obtain
238   “efficient” solutions for these tasks in our experiments. To maintain the quality and consistency of our
239   dataset, we remove these unaddressed tasks from our candidate pool. This filtering step ensures that
240   all tasks in our dataset have been successfully optimized by the teacher model, providing a reliable
241   foundation for the fine-tuning process.
242   Filtering tasks without efficient solutions (Step 6): We define a solution as inefficient if it exhibits
243   suboptimal execution time or memory usage compared to the initial solution (solution provided by
244   the collected dataset) for the given task. The criteria for determining inefficiency are based on the
245   potential for improvement in terms of execution time and memory usage after applying optimization
246   techniques. Consider a task where the goal is to sort an array. An inefficient solution uses Bubble
247   Sort, which has a time complexity of O(n2 ), as opposed to an efficient solution like Quick Sort with
248   an average time complexity of O(n log n).
249   Despite the application of SOAP (Huang et al., 2024a), some tasks may not yield more efficient
250   solutions due to limited optimization potential, even though they are algorithmic tasks. In such
251   cases, the SOAP process may not be able to generate solutions that significantly improve upon the
252   original code in terms of efficiency. To ensure that our dataset focuses on tasks with meaningful
253   optimization potential, we filter out these tasks from our experiments. To identify and filter out tasks
254   with inefficient solutions, we employ a two-step process. First, we use self-optimization to require
255
      DeepSeek-Coder-V2-Lite to improve the efficiency of the code solutions, which aims to improve
      the efficiency of the code by making optimizations such as reducing redundant computations or
256
      improving data structures. We run DeepSeek-Coder-V2-Lite for five iterations and analyze whether
257
      the efficiency of the code has improved based on metrics such as execution time and memory usage.
258   If the efficiency does not show improvement after these iterations, we consider the task to have an
259   inefficient solution and remove it from our candidate tasks. We acknowledge that there may be cases
260   where the initial code is already efficient, and the lack of improvement after optimization does not
261   necessarily indicate an inefficient solution. However, detecting such cases would require significant
262   manual effort to analyze each task individually. To maintain a consistent and automated approach,
263   we opted to remove all tasks that did not show efficiency improvement after the optimization process,
264   which proved to still perform very well in our evaluation.
265   The post-SOAP cleaning process plays a crucial role in refining our candidate tasks and creating a
266   high-quality dataset for fine-tuning. By filtering out tasks that are not addressed by the teacher model
267   and those without significant efficiency improvements, we ensure that our final dataset consists of
268   tasks with optimized solutions that demonstrate a notable enhancement in performance. This curated
269   dataset serves as a valuable resource for training models to generate efficient code and for advancing
      the field of code optimization using LLMs.

288
      Following Huang et al. (2024b), we evaluate the effectiveness of E FFI -C ODE fine-tuned LLMs using
289   two key aspects: correctness and efficiency of the LLM-generated code. Our metrics are outlined as:
290
291              • Execution Time (ET): Measures the time taken for code execution.
292              • Max Memory Usage (MU): Assesses the peak memory requirement during code execution.
293
                 • Total Memory Usage (TMU): Evaluates the overall memory usage throughout code execution.
294
295
                 • Normalized Metrics: The metrics contains NET (Normalized Execution Time), NMU (Nor-
                   malized Max Memory Usage), and NTMU (Normalized Total Memory Usage). They are our
296
                   primary metrics for assessing efficiency, measuring how efficient/inefficient the LLM-generated
297
                   code is compared with the human-written canonical solution for ET, MU, and TMU.
298
                 • Correctness: We assess the correctness of LLM-generated code using the pass@1 metric with
299
                   greedy decoding, following the approach of existing works.

      Datasets and Models In our experiments, we evaluate the efficiency and correctness of LLM-
345   generated code on two code generation benchmarks, i.e., HumanEval and EffiBench. We fine-
346   tune four open-source LLMs with E FFI -C ODE, including DeepSeek-Coder-6.7B base and instruct
347   model (DeepSeekAI, 2023), Qwen2.5-Code-7B base and instruct model (Hui et al., 2024).
348
      Fine-tuning Setup We use Llama-factory (Zheng et al., 2024) to fully fine-tune all LLMs with the
349
      same setup and train the models using E FFI -C ODE. The maximum sequence length is set to 2048
350
      tokens. We use a batch size of 128 and set the learning rate to 5e-6 with a cosine learning rate
351   scheduler and a warmup ratio of 0.03. We fine-tune all LLMs for 4 epochs under the bf16 data type.
352
353   Prompt Template For all experiments, we use the inference prompt provided by DeepSeek-Coder
354
      for both fine-tuning and inference. The detailed template can be found in Appendix A.3.

      In this paper, our research addresses a critical gap in the efficiency of code generated by LLMs
533
      by introducing the E FFI -C ODE dataset, designed to enhance both the correctness and execution
534   efficiency of LLM-generated code via fine-tuning (e.g., SFT, DPO, and ORPO). Through meticulous
535   aggregation, preprocessing, and iterative optimization, we provide a robust resource that significantly
536   boosts the performance of open-source LLMs like DeepSeek-Coder and Qwen. Our experiments
537   reveal substantial improvements, with notable increases in pass rates and decreases in execution
538   time, underscoring the potential of E FFI -C ODE to advance the state of code generation in resource-

953   A.2     A DDITIONAL R ELATED W ORK
954
955   LLMs for Code The increasing popularity of LLMs for code generation has coincided with the
956   growing availability of open-source code repositories and the need to boost developer productivity.
957   Initial efforts focused on training models specifically for coding tasks, such as CodeT5 (Wang et al.,
958   2021), AlphaCode (Li et al., 2022), CodeGen (Nijkamp et al., 2023), InCoder (Fried et al., 2023),
959   StarCoder (Li et al., 2023a), SantaCoder (Allal et al., 2023), and DeepSeek-Coder (DeepSeekAI,
960   2023). Contrastingly, models such as Codex (Chen et al., 2021) and CodeLlama (Rozière et al., 2023)
961   represent a subsequent stride, being fine-tuned from foundation models (Brown et al., 2020; Touvron
962   et al., 2023). These code LLMs have been applied to various tasks, including code generation (Chen
963
      et al., 2021; Dai et al., 2024), program repair (Haque et al., 2022; Jiang et al., 2023a), automated
      testing (Lemieux et al., 2023; Deng et al., 2023), code translation (Rozière et al., 2020; Ahmad
964
      et al., 2023), type prediction (Mir et al., 2022; Wei et al., 2023), and code summarization (Hasan
965
      et al., 2021; Ahmed & Devanbu, 2022). While LLMs have achieved impressive results in code
966   generation tasks like HumanEval (Chen et al., 2021) and MBPP (Austin et al., 2021), their efficiency
967   has received less attention. Recent studies (Shi et al., 2024; Huang et al., 2024b; Niu et al., 2024) have
968   shown that LLM-generated code exhibits lower efficiency in terms of execution time and memory
969   usage compared to canonical solutions. These findings highlight the need for further research and
970   development to improve the efficiency of LLM-generated code. In this work, we propose the first
971   fine-tuning method that significantly improves both the efficiency and correctness of code generated
      by various LLMs.

1053   Learning From Feedback A prevalent strategy for improving the behavior of LLMs is learning
1054   from feedback, mirroring human learning where individuals refine their actions through trial, error,
1055   and correction (Boyd & Fales, 1983; Metcalfe, 2017). Early efforts involve using human feedback
1056   to evaluate and refine models (Kreutzer et al., 2018; Ouyang et al., 2022; Glaese et al., 2022). To
1057
       minimize human intervention, another strategy focuses on automated feedback. These methods
       iteratively learn from automatically generated feedback signals, understanding the consequences of
1058
       their actions and adapting their behaviors. The source of this automated feedback can be diverse,
1059
       ranging from the LLM itself (Madaan et al., 2023; Shinn et al., 2023), external tools (Gou et al.,
1060   2023; Lu et al., 2024b) or verifiers (Lu et al., 2024a), external knowledge sources (Gao et al., 2023;
1061   Yu et al., 2023) and even generation logits (Yao et al., 2024). In code generation, the program
1062   executor is frequently used as a source of feedback for refining the model’s initial code. For example,
1063   Self-Edit (Zhang et al., 2023a) and Self-Evolve (Jiang et al., 2023b) execute the initial program on
1064   example test cases and provide the execution results as feedback, prompting the LLM to refine the
1065   code. Self-Debug (Chen et al., 2023) explores using program explanation, unit tests, and program
1066   interpreters as feedback types. ALGO (Zhang et al., 2023b) employs a more fine-grained approach by
1067   generating a reference oracle program that solves the problem with an exhaustive search. Feedback is
1068
       then collected by comparing the generated outputs with the oracle. While existing work primarily
       focuses on using feedback to edit the initial code to ensure correctness, our method explores using
1069
       overhead profiles to improve the efficiency of the code.

1111
1112   We have provided a case example in Figure 4 to demonstrate how SOAP’s iterative refinement
1113   improves the quality of the solutions. In this example, the initial code generated by DeepSeek-Coder-
1114   V2-Lite calculates the Levenshtein distance using a recursive approach, which has an exponential
1115   time complexity of O(3(m+n) ). For longer strings, this recursive method becomes highly inefficient
1116   due to the large number of function calls. To optimize the code, the refined version employs dynamic
1117
       programming, which avoids redundant calculations by filling a distance matrix to compute the
       Levenshtein distance. The time complexity of the dynamic programming approach is O(mn), where
1118
       m and n are the lengths of strings a and b, respectively. By filling the distance matrix in a single
1119
       traversal, the optimized code eliminates redundant calculations, resulting in improved efficiency. The
1120   dynamic programming solution leverages the characteristics of optimal substructure and overlapping
1121   subproblems, decomposing the problem into smaller subproblems and storing intermediate results
1122   to avoid redundant calculations, thereby improving the efficiency of the algorithm. In the provided
1123   example, the initial recursive code takes 1.100s to execute, while the optimized dynamic programming
1124   code completes execution in just 0.00012s, demonstrating a significant improvement in execution
1125   time.
