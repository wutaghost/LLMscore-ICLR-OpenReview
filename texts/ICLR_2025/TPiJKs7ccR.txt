                                                   A BSTRACT
008
009
010            Web browsers are a portal to the internet, where much of human activity is un-
011
               dertaken. Thus, there has been significant research work in AI agents that interact
               with the internet through web browsing. However, there is also another interface
012
               designed specifically for machine interaction with online content: application pro-
013
               gramming interfaces (APIs). In this paper we ask – what if we were to take tasks
014            traditionally tackled by browsing agents, and give AI agents access to APIs? To
015            do so, we propose two varieties of agents: (1) an API-calling agent that attempts to
016            perform online tasks through APIs only, similar to traditional coding agents, and
017            (2) a Hybrid Agent that can interact with online data through both web browsing
018            and APIs. In experiments on WebArena, a widely-used and realistic benchmark
019            for web navigation tasks, we find that API-based agents outperform web brows-
020            ing agents. Hybrid Agents out-perform both others nearly uniformly across tasks,
021            resulting in a more than 20.0% absolute improvement over web browsing alone,
022
               achieving a success rate of 35.8%, achiving the SOTA performance among task-
               agnostic agents. These results strongly suggest that when APIs are available, they
023
               present an attractive alternative to relying on web browsing alone.

027   1    I NTRODUCTION
028
029   Web agents use browsers as an interface to facilitate humans in performing daily tasks such as
030   online shopping, online planning, trip planning, and other work-related tasks (Liu et al., 2018; Li
031   et al., 2020; Rawles et al., 2023; Patil et al., 2023; Pan et al., 2024; Chen et al., 2024a; Huang et al.,
032   2024; Durante et al., 2024). Existing web agents typically operate within the space of graphical
033
      user interfaces (GUI) (Zhang et al., 2023; Zhou et al., 2023; Zheng et al., 2024), using action spaces
      that simulate human-like keyboard and mouse operations, such as clicking and typing. To observe
034
      web pages, common approaches include using accessibility trees, a simplified version of the HTML
035
      DOM tree, as the input to text-based models (Zhou et al., 2023; Drouin et al., 2024a), or multi-
036   modal, screenshot-based models (Koh et al., 2024a; Xie et al., 2024; You et al., 2024; Hong et al.,
037   2023). However, regardless of the method of interaction with web sites, there is no getting around
038   the fact that these sites were originally designed for human consumption, and may not be the ideal
039   interface for machines.
040
      Notably, there is another interface designed specifically for machine interaction with online content:
041
      application programming interfaces (APIs) (Chan et al., 2024). APIs allow machines to communi-
042   cate directly with the backend of a web service (Branavan et al., 2009), sending and receiving data
043   in machine-friendly formats such as JSON or XML (Meng et al., 2018; Xu et al., 2021). Nonethe-
044   less, whether AI agents can effectively use APIs to tackle real-world online tasks, and the conditions
045   under which this is possible, remain unstudied in the scientific literature. In this work, we explore
046   methods for tackling tasks normally framed as web-navigation tasks with an expanded action space
047   to interact with APIs. To do so, we develop new API-based agents that directly interact with web
048   services via API calls, as depicted in Figure 1. This method bypasses the need to interact with web
049   page GUIs through simulated clicks.
050   At the same time, not all websites have extensive API support, in which case web browsing actions
051   may still be required. To address these cases, we explore a hybrid approach that combines API-based
052   agents with web-browsing agents, as described in Figure 1. By implementing an agent capable of
053   interleaving API calls and web browsing, we found that agents benefit from the flexibility of this
      hybrid model. When APIs are available and well-documented, the agent can directly interact with

100
      Various benchmarks have been developed to evaluate the performance of web browsing agents.
101
      MiniWoB (Miniature World of Bits) is an early benchmark that provides simple web-based tasks
102   such as clicking links or typing into forms, but it remains limited in complexity and realism (Shi
103   et al., 2017). Mind2Web scales up these tasks, introducing more sophisticated interactions across
104   websites, but it often lacks the dynamic, real-world scenarios found on the broader web (Deng et al.,
105   2023). WebArena (Zhou et al., 2023) advances web browsing benchmarks by creating reproducible
106   sandboxes of a variety of websites, such as managing repositories, posting online, performing online
107   shopping, and planning trips using map services, while VisualWebArena extends WebArena to the
      vision modality (Koh et al., 2024a).

129   with platforms like Gitlab (to manage projects and repositories), Reddit (to browse and post content),

131   Task success is evaluated in three ways: (1) if the task requires producing a specific output, the
132   agent’s response is checked for correctness; (2) for tasks involving changes to a website’s state (e.g.,
133   adding an item to a shopping cart), success is measured by verifying whether the state has changed
134

135
      displaying the desired content.

      2.2   A BASELINE W EB B ROWSING AGENT
139
      While there are a wide variety of agents proposed for such web navigation tasks, in this work we
140   build upon the WebArena baseline agent (Zhou et al., 2023), which operates purely through web
141   interaction by leveraging the accessibility tree2 , a structure that exposes interactive elements like
142   buttons, input fields, and hyperlinks (Yao et al., 2023; Gu et al., 2024). Each element of the acces-
143   sibility tree is characterized by its functionality such as a hyperlink, its content, and specific web
144   attributes (Liu et al., 2024b; He et al., 2024a; Lù et al., 2024). This exposes web page elements in a
145   hierarchical structure that is easy for agents to navigate (Samuel et al., 2024; Burns et al., 2022).
146   Agents based on this framework utilize an action space that simulates human browsing behavior,
147   incorporating actions such as simulated clicks, form input, and navigation between pages (Liu et al.,
148   2023; Song et al., 2024; Gur et al., 2024). Importantly, these agents maintain a comprehensive his-
149   tory of their previous actions, allowing them to contextualize their decision-making in past actions.
150
      While agents utilizing this method can navigate arbitrary web pages and often perform well on
151
      simpler layouts, challenges arise with the complexity of the accessibility tree. Many large language
152   models (LLMs) are not familiar with this structure, leading to difficulties in completing tasks that
153   require numerous or complex interactions. As a result, the average accuracy hovers in the low double
154   digits (Liu et al., 2024a; Deng et al., 2023; Fu et al., 2024). These methods also struggle with content
155   that need to be dynamically loaded or contents not immediately visible within the tree (Abramovich
156   et al., 2024; Chen et al., 2024b; Lutz et al., 2024).
157
      To give one motivating example, in Figure 2, we demonstrate a task where the agent needs to per-
158   form a task determining the number of commits made by the user SaptakS in a repository named

188
189
      In contrast to browsing, API calling offers a direct interface for machines to communicate with web
190
      services, reducing operational complexity. In this section, we explore an API-based approach when
191
      performing web tasks.
192

      3.1     API S AND API D OCUMENTATION
194
195   For websites that offer API support, pre-defined endpoints can be utilized to perform tasks
196   efficiently. These APIs, following standardized protocols like REST3 , allow interaction with
197   web services through sending HTTP requests (e.g., GET, POST, PUT) and receiving structured
198   data such as JSON objects4 as responses. Websites often provide official documentation for
199   the APIs, which can give guidance on how to utilize the APIs. Some documentation is pro-
200   vided in README 5 format, some are in OpenAPI YAML6 format, and some are in plain
201   text format. For instance, Figure 3 shows the official README documentation of a Gitlab
202   API GET /api/{id}/commits. It documents the functionality, input arguments, and out-
203
      put types of the API. For example, one could use the Python requests library, by calling
      requests.get("gitlab.com/api/a11yproject/commits"), to retrieve all commits
204
      of the repository a11yproject. This would return a JSON list containing all the commits to this
205
      repo, as shown in Figure 3.
206

210   APIs depends heavily on the availability of APIs and quality of API documentation. In this work,
211   we acquired APIs by manually looking up official API documentation on a website, although this

236   3.3    U SING API S IN AGENTS
237
238   Once we have the APIs and documentation, we then need to provide methods to utilize them in
239   agents. We utilize two different methods based on the size of the API documentation.
240
241   One-Stage Documentation for Small API Sets For websites with a smaller number of API end-
242   points8 , we directly incorporate the full documentation into the prompt provided to the agent. This
243   approach of directly feeding the full documentation worked well for websites with a limited number
244   of API endpoints, as it allowed the agent to have immediate access to all the necessary information
245
      without the need for a more complex retrieval mechanism.
246
      Two-Stage Documentation Retrieval for Large API Sets For websites with a larger number of
247
      endpoints, providing the full documentation directly within the prompt was impractical due to the
248
      size limitations of agent inputs. To address this, we employ a two-stage documentation retrieval pro-
249   cess, allowing access to only the relevant information as needed, keeping the initial prompt concise.
250
251   In the first stage, the user prompt provide a description of the task, with a list of all avail-
252
      able API endpoints along with a very brief description of each API. For example, {"GET
      /api/{id}/commits": "Get a list of commits in a project"}. This initial
253
      summary helps facilitating understanding the scope of all the available APIs while staying within
254
      the prompt size constraints.
255
256   In the second stage, if the model determines that it needs detailed information about one or more
257   API endpoints, it can use a tool called get api documentation. This tool has a dictionary
258
      that maps each API to its API documentation respectively. The dictionary is obtained using pattern
      match in Python to retrieve substrings related to each endpoints. get api documentation
259
      is able to search the dictionary and retrieve the full README or YAML documentation for
260
      any given endpoint by calling get api documentation with the endpoint’s identifier. This
261   may include the input parameters, output formats, and examples of interacting with the end-
262   point. For example, to retrieve the documentation for GET /api/id/commits, the agent would
263   call get api documentation("GET /api/id/commits"), and an example returned API
264   documentation is the documentation in Figure 3.
265
      This retrieval method allows the agent to make flexible and informed choices during the execution
266
      of tasks. If the agent finds that an API does not meet its needs or if it encounters an error, it can

275   4       H YBRID B ROWSING +API C ALLING AGENTS
276
277   We have proposed API-based methods for handling web tasks, but the question arises: given the
278   benefits of API calling, should we discard web browsing altogether? The most obvious bottleneck is
279   that not all websites offer comprehensive API support. Some platforms offer limited or poorly doc-
280   umented APIs (e.g. there is no API for shopping on Amazon9 ), forcing agents to rely on traditional
281   web browsing methods to complete tasks.
282   To deal with these situations, we propose a hybrid methods that integrates both browsing-based and
283   API-based approaches, and developed a Hybrid Agent capable of interleaving API calls and web
284   browsing, switching dynamically based on task requirements and the available resources. Specifi-
285   cally, for each task, the agent is given the fixed step budget within which it has to finish the task. In
286   each step, the agent could either (1) communicate with humans in natural language to ask for clar-
287   ification or confirmation, or 2) generate and executes Python code which could include performing
288
      API calling, or 3) performs web browsing actions. The agent could choose freely among these three
      options, depending on the agent’s confidence which method could best tackle the task.
289
290   The ideal case is that for websites that offer comprehensive API support, the Hybrid Agent can utilize
291   well-documented endpoints to perform tasks more efficiently than it could through web browsing;
292   for websites with limited API support or poorly documented APIs, the Hybrid Agent could rely
293   more on web browsing to fulfill certain tasks. We later find that enabling an agent to interleave API
294
      calling and web browsing boost the agent’s performance (see Section 6).
295
296   Prompt Construction The Hybrid Agent’s prompt construction extends upon the API-based
297   agent by incorporating both API and web-browsing documentation. Similar to the API-based agent,
298
      the Hybrid Agent is provided with a description of available API calls as discussed in Section 3.3.
      In addition, the Hybrid Agent receives a detailed specification of the web-browsing actions, which
299
      mirrors the information given to the browsing agent described in Section 2.2, including a breakdown
300
      of all potential browser interactions. It also maintains a history of all its prior steps such that the
301   agent could make more informed actions. The prompt can be found in Appendix A.4.

      5.3     API I MPLEMENTATION D ETAILS
379
380   In this section, we will discuss how we provided the APIs to the agents when evaluating different
381   web applications inside WebArena, where we follow the methodologies as discussed in Section 3.3.
382
383   5.3.1    O NE -S TAGE D OCUMENTATION FOR S MALL API S ETS
384
385
      For websites with fewer than 100 API endpoints, namely the Map and Reddit websites, we directly
      incorporated the full documentation into the prompt provided to the agent.
386
387   In the case of the Map API, the documentation was sourced directly from the public API documen-
388   tation provided for the website. The only modification made was the addition of an explanation
389   detailing how to make HTTP requests using the requests library in Python for interacting with the
390   Map API’s endpoints. This ensured that the agent could comprehend both the structure of the API
391
      and how to implement calls programmatically.
392   For Reddit, since there was no pre-existing documentation for the APIs, we leveraged GPT-4o16
393   itself to generate these README files. By prompting GPT-4o with a file containing all implemen-
394   tations of the API endpoints, we generated a README documentation, including input parameters,
395   expected outputs, and example API calls.
396

406   We employed OpenHands as our primary evaluation framework to facilitate the development and
407   testing of our agents (Wang et al., 2024c). OpenHands is an open-source platform designed for
408   creating and evaluating AI agents that interact with both software and web environments, making
409   it an appropriate infrastructure for our proposed methods. The OpenHands architecture supports a
410
      variety of interfaces for agents to interact with. Moreover, this framework allows agents to keep
      a detailed record of past actions in the prompt, enabling agents to execute actions in a way that is
411
      consistent with earlier steps. For coding tasks, it implements an agent based on CodeAct (Wang
412
      et al., 2024a) that incorporates a sandboxed bash operating system and Jupyter IPython17 environ-

416   (Drouin et al., 2024b). However, while the browsing agent can browse websites, and the CodeAc-

418   base, we developed two varieties of agents for API-based solving of web tasks.

      and their usage. At each step, the agent could utilize all previous actions to make informed selection

427
      existing framework of the API-based agent. This Hybrid Agent is provided the prompt describing
428   both the APIs and the browsing actions, allowing for free transitions between API calling and web

495   6.2      D OES API Q UALITY M ATTER ?
496
497   Yes, API quality does significantly impact the performance of the API-based agent. High quality
498   APIs provide comprehensive and well-documented endpoints that enable agents to interact accu-
499   rately and efficiently with websites. With comprehensive API support, the API-based agent could
500   tackle more tasks through API calling, while the Hybrid Agent could rely less on browsing; on the
501   other hand, clear and detailed documentation allows agents to utilize the APIs effectively, ensur-
502
      ing that requests are accurate, and minimizing potential errors in task execution. For example, the
      websites Gitlab and Map with the best API support as mentioned in Section 5.2, demonstrates the
503
      highest task completion accuracy by the API-based agent and the Hybrid Agent across all websites.
504
505   Conversely, low-quality APIs, characterized by incomplete functionality or ambiguous documen-
506   tation, can significantly degrade performance. In such cases, the absence of necessary endpoints
507   may prevent the API-based agent from completing tasks, forcing the Hybrid Agent to resort to web
508
      browsing. Moreover, poorly documented APIs can result in incorrect parameters and headers being
      used, further reducing the effectiveness of the agent. This highlights the importance for websites to
509
      maintain comprehensive and well-documented API support.

530
531
532   In this paper, we propose new web agents that use APIs instead of traditionally browsers. We
533   found that API-based agents outperform browsing-based counterparts, especially on websites with
534   sufficient API support. Hence we further propose an agent that is capable of switching between
535   using APIs or browsers and empirically outperforms agents that only uses one of the two interfaces.
536   For future work, we aim to explore methods for automatically inducing APIs (Wang et al., 2024e).
537   These methods could identify and generate API calls for websites lacking formal API support, fur-
538   ther expanding the applicability and efficiency of API-based approaches. By automating the discov-
539   ery and utilization of APIs, we envision even more robust agents capable of handling diverse web
      tasks with minimal reliance on manual interaction through browsing.

      8   L IMITATIONS
541
542
      Evaluation Benchmark In this paper, we evaluate web agents exclusively on WebArena tasks.
543   While WebArena offers realistic and diverse challenges, the number and variety of tasks may be lim-
544   ited. Other benchmarks, such as Webshop (Yao et al., 2022), MiniWoB (Shi et al., 2017), Mind2Web
545   (Deng et al., 2023), WebVoyager (He et al., 2024b), and VisualWebArena (Koh et al., 2024a), pro-
546   vide alternative evaluation platforms. However, as discussed in Section 2.1, WebArena aligns more
547   closely with real-world scenarios and our use case, while other benchmarks lack support for API
548   calling. For example, VisualWebArena is less applicable to our study because WebArena APIs lack
549   support for interacting with images, a core component of VisualWebArena tasks.
550
551   API Availability A key limitation of API-based agents is the inconsistent availability and coverage
552
      of APIs across websites. Even platforms with extensive API ecosystems, such as GitLab, may lack
      support for specific functionalities (e.g., retrieving a user’s official username from a displayed name),
553
      leading to edge cases where API-based agents are unable to complete tasks due to incomplete API
554
      support. However, advancements in techniques like Automatic Web API Mining (AWM) Wang et al.
555   (2024e) could potentially address this limitation by automatically generating APIs for unsupported
556   tasks, reducing reliance on manual API creation.
557
558   Incorporating APIs Unlike browsing agents, which can adapt to new websites without manual in-
559   tervention, the API-based agent requires additional effort to integrate the necessary APIs documen-
560   tation to the action space of the agent for each website. This manual integration process increases
561   complexity, particularly when the agent must support a wide range of websites, limiting scalability
562   compared to agents that rely solely on web browsing for interactions. However, future advancements
563   in automated API scraping and documentation generation could eliminate this bottleneck, allowing
564   for more scalable and flexible API-based agents.

      A.1   R ELATED W ORK
783
784   The development of AI agents that interact with the web and APIs has garnered significant research
785   attention. Web browsers, serving as the primary interface for interacting with online content, have
786   long been a focus for AI research. Web-based agents that can navigate websites, extract information,
787   and perform tasks autonomously have been studied extensively, especially in the context of large
788   language models (LLMs) and agents designed to mimic human behavior online.
789
790   Web Navigation Agents Much prior work has centered around agents that perform web-based
791   tasks using browsing actions (Yao et al., 2022; Lai et al., 2024; Koh et al., 2024b; Pan et al., 2024).
792   These agents are particularly effective in environments where human-like interaction with a user
793
      interface is necessary (Drouin et al., 2024b). Frameworks such as WebArena have further refined
      the evaluation of such agents by providing complex and realistic web navigation tasks (Zhou et al.,
794
      2023). Our work explores the Hybrid Agent that combines web browsing with API interactions.
795
      While prior work primarily focuses on browsing-only agents, we examine how Hybrid Agents can
796   enhance performance by integrating structured API calls with web navigation.
797
798   Code Generation Agents and Tool Usage Another stream of research focuses on agents that
799   interact with online content via application programming interfaces (APIs) (Wang et al., 2024d; Patil
800   et al., 2023; Qin et al., 2023; Yuan et al., 2024; Wang et al., 2024b; Du et al., 2024). In this context,
801   works such as CodeAct have pioneered the development of agents that generate and execute code,
802   including API calls, to perform tasks typically reserved for software engineers (Wang et al., 2024a;
803   Zhang et al., 2024; Tang et al., 2024). These API-based agents are optimized for tasks that involve
804   structured data exchanges, allowing them to perform operations more efficiently than traditional
805   web navigation agents (Shen et al., 2024). On the other hand, our work integrates both browsing
806
      and API interactions, demonstrating that Hybrid Agents can outperform API-only agents in tasks
      requiring web navigation. While existing research shows the efficiency of API-based agents, our
807
      Hybrid Agent dynamically switches between APIs and web browsing to optimize task performance.
808
809   Additionally, we are the first to explore comparative studies of API v.s. Browsing agents on the
      same websites. We demonstrate that API-based web agents are often more efficient than browsing

      Under review as a conference paper at ICLR 2025


810
      agents when APIs are available, leading to significant improvements in performance. This finding is
811   aligned with previous studies that highlight the advantages of structured interactions through APIs
812   compared to unstructured web browsing interactions.

1205   average, while the Hybrid Agent takes the most steps amongst the three agents. This is likely due

1212   Costs The cost of completing tasks shows a different trend. While the browsing agent requires
1213
       more steps, it is much cheaper compared to the API-based agent and the Hybrid Agent. This is
       primarily because the prompts needed for browsing agents are much shorter. When browsing, the
1214
       agent only needs instructions on how to use the web interface and the limited action space around
1215
       14 browsing actions. In contrast, API-based and Hybrid Agents require access to a much larger set
1216   of API calls. For example, when interacting with GitLab, the agent is provided with 988 available
1217   APIs, leading to much longer prompts and significantly increasing the cost of execution. The cost
1218   goes down when the prompt for API calling is shorter. For example, the Reddit website has the least
1219   length of API documentation, where its cost is also less than other websites. However, as visualized
1220   in Figure 7, the accuracy of the API-based agent and the Hybrid Agent is much higher than the
1221   browsing agent, which makes the increase in cost justifiable due to the significantly improved task
1222   performance. The higher cost is offset by the agents’ ability to complete tasks more accurately and

1236
1237   We randomly sampled 100 tasks from the WebArena tasks and performed error analysis on the
1238   API-based agent. We found that 33% of the tasks are correctly performed with only API calling,
1239   50% are unsolvable with solely APIs, 6% are incorrect due to incorrect task understanding, and
1240   11% are incorrect due to error in calling APIs such as mal-formatting and wrong input. In other
1241   words, among the 50 API solvable tasks, 66% are performed correctly by the API-based agent. This
       showcases the strong capability of the API-based agent when given sufficient API to solve the task.
