010
               State Space Models (SSMs) like Mamba2 are a promising alternative to Trans-
               formers, with faster theoretical training and inference times – especially for long
011
               context lengths. Recent work on Matryoshka Representation Learning – and its
012
               application to Transformer backbones in works like MatFormer – showed how
013            to introduce nested granularities of smaller submodels in one universal elastic
014            model. In this work, we present MatMamba: a state space model which com-
015            bines Matryoshka-style learning with Mamba2, by modifying the block to contain
016            nested dimensions to enable joint training and adaptive inference. MatMamba al-
017            lows for efficient and adaptive deployment across various model sizes. We train
018            a single large MatMamba model and are able to get a number of smaller nested
019            models for free – while maintaining or improving upon the performance of a base-
020            line smaller model trained from scratch. We train language and image models
021
               at a variety of parameter sizes from 35M to 1.4B. Our results on ImageNet and
               FineWeb show that MatMamba models scale comparably to Transformers, while
022
               having more efficient inference characteristics. This makes MatMamba a practi-
023
               cally viable option for deploying large-scale models in an elastic way based on the
024            available inference compute.

027   1   I NTRODUCTION

      Deep learning practitioners often train different sizes of the same kind of model to facilitate deploy-
030   ment in a variety of ranges of available inference compute. For example, the Llama 3.2 (Dubey et al.,
031   2024) series has 1B, 3B, 11B, and 90B variations. These models are extremely powerful individu-
032   ally – but due to independent training do not necessarily share the same metric space – a property
033   which can be extremely useful for inference applications like speculative decoding (Leviathan et al.,
034   2023), hybrid cloud-edge inference, or just general input or compute adaptive processing. Moreover,
035   because training these models is expensive, we typically see only a few chosen sizes trained. This is
036   not desirable in situations where the deployment setup can optimally support an intermediate model
037   (e.g. a 2B model), but has to settle for the less accurate 1B model instead.
038   Techniques like model compression and distillation aim to address these issues, but require addi-
039   tional training (for which data may not be available), and can sometimes drop accuracy (Jaiswal
040   et al., 2023). Thus, methods that offer adaptive inference out of the box at intermediate granulari-
041   ties are extremely useful. This has been explored for Transformers (Devvrit et al., 2023; Cai et al.,
042   2024b) and ConvNets (Yu & Huang, 2019; Cai et al., 2019). The core focus of this work is to try to
043   enable out of the box adaptive inference in a newer architecture: Mamba2 (Dao & Gu, 2024).
044   State Space Models like Mamba2 (Dao & Gu, 2024) and a number of other related newer archi-
045   tectures (see Section 2) have shown tremendous potential as they try to improve on the efficiency
046   of Transformers, while maintaining their potency as accurate and general sequence processing ar-
047   chitectures. Mamba2 has comparable scaling properties to Transformers, while being significantly
048   faster at longer context lengths.
049   In this work, we introduce MatMamba, a nested Matryoshka structure (Kusupati et al., 2022) within
050   a Mamba2 block (Dao & Gu, 2024). MatMamba enables the extraction of hundreds of nested sub-
051   models from the same set of weights, without requiring any additional training during deployment.
052   MatMamba is a general-purpose sequence processing architecture that can be applied to any type
053   of model (encoder/decoder), modality (language/vision/sound/actions), loss function, or learning
      algorithm compatible with a Transformer or Mamba2 layer.

      The philosophically closest work to MatMamba is MatFormer (Devvrit et al., 2023) – which imposes
055   a nested structure on the FFN block in a Transformer layer. We use the same concept to impose
056   a nested structure on any learnable parameter in a Mamba2 block that depends upon the hidden
057   dimensionality of the block. Formally, a MatMamba block consists of a nested combination of
058   g Mamba2 blocks Mi , such that M1 ⊂ M2 ⊂ ... ⊂ Mg , where Mi ⊂ Mj means that all the
059   parameters of a sub-block Mi are present in Mj . We train the model using g forward passes with
060   gradient accumulation followed by a single backward pass for parameter updates (see Figure 1).
061   By jointly training all g granularities, the smallest sub-blocks are incentivized to represent the
062   most important information, like in Matryoshka Representation Learning (Kusupati et al., 2022).
063   We can now use any of the g nested sub-blocks Mi flexibly. Additionally, we can flexibly slice
064   the block along any dimensionality (even beyond the g explicitly optimized granularities). Using
065   Mix’n’Match (Section 3.4), we can perform this operation over multiple layers at varying granu-
066   larities to flexibly extract a combinatorially large number of models from the single larger model.
067   We observe that these extracted models preserve the metric space of the larger model, and are ac-
068   curate across a variety of tested tasks – effectively allowing us to choose a tradeoff between model
069
      performance and compute.
070   We train MatMamba-based vision models (MatMamba-Vision), and find that: (a) MatMamba-
071   Vision models scale as well as baseline Mamba2 based models at all g = 4 granularities; (b)
072   Using Mix’n’Match, we can flexibly extract submodels between the explicitly optimized granular-
073   ities. The submodels span (and sometimes exceed) the pareto optimal accuracy-vs-compute curve;
074   (c) MatMamba-Vision models are significantly faster at higher resolutions than ViTs, making them
075
      promising candidates for long-form and high resolution visual tasks, while enabling adaptive visual
      processing with the nested submodels (see Section 4.1.1).
076
077   Furthermore, MatMamba-Vision models can act as elastic image encoders for adaptive image re-
078   trieval. We can encode visual datasets with the largest model, and because the smaller submodels
079   share its metric space, we can use them as query encoders, needing drastically lower compute with
080   minimal loss in accuracy (see Section 4.1.2).
081   We also train MatMamba-based decoder language models (MatMamba-LM) at various sizes from
082   130M-1.4B parameters, and at g = 4 granularities. We make similar observations here too, that
083   MatMamba-LM models scale as well as Mamba2 baselines with the same architecture for all nested
084   granularities. We also observe interesting homogenous scaling behaviour between the nested gran-
085   ularities for different models (see Section 4.2).
086   Through MatMamba, for the first time, we bring together the adaptivity of Matryoshka-style learning
087   and the efficiency of state space models (SSMs) like Mamba2 (Dao & Gu, 2024).
088
      We make the following research contributions:
089
090   1. We introduce MatMamba, which imposes a nested Matryoshka structure on a Mamba2 state
091      space model. We jointly optimize all nested granularities to train a single elastic model.
092
      2. We show that MatMamba models scale as well as the baseline Mamba2 models for a variety of
093      model sizes from 35M-1.4B parameters on language and vision tasks.
094
      3. Using Mix’n’Match with MatMamba allows the flexible extraction of hundreds of submodels to
095
         perform adaptive inference. These submodels preserve the metric space of the original model.
096
097   4. MatMamba-Vision models are comparably accurate and significantly faster at higher resolutions
098
         than ViTs, making them well suited for long-form/high resolution and adaptive visual processing.

137   (MRL) (Kusupati et al., 2022). MRL simplifies the training process to induce elasticity with a small
138   set of nested granularities (hence the name Matryoshka), exponentially separated in size, all opti-
139   mized with the same target loss function as the full vector. MRL further smoothly interpolates to

142   Matryoshka information packing and learning has been widely adopted in bringing adaptivity not
143   only in output space, but also in input (Beyer et al., 2023) and model weights (Devvrit et al., 2023;
144   Cai et al., 2024b; Valipour et al., 2023). MatFormer (Devvrit et al., 2023) is a direct translation of
145   MRL to every hidden activation vector of a MLP sub-block within a Transformer layer (Vaswani
146   et al., 2017). MatFormer showed scaling trends similar to Transformer, while also providing the
147
      capability to adaptively extract submodels that fall on the accuracy-vs-compute pareto curve. More
      recent works (Cai et al., 2024b; Jain et al., 2024) developed dynamic routing on top of the con-
148
      ditional computation enabled by MatFormer to realize performance gains in deployment. Further,
149
      matryoshka packing was also used for flexible tokenization (Cai et al., 2024a; Hu et al., 2024) as
150   well as diffusion models (Gu et al., 2023).
151
152   Transformers (Vaswani et al., 2017) have been fundamental sequence processing blocks in neu-
153
      ral networks for the past few years. There has been a recent wave of work on efficient sequence
      processing architectures that aim to be faster and equally performant alternatives to Transformers.
154
      Mamba (Gu & Dao, 2023) and Mamba2 (Dao & Gu, 2024) are the most relevant to this work,
155
      with other very closely related works like Linear Attention (Katharopoulos et al., 2020), Test-time
156   training (Sun et al., 2024), RWKV (Peng et al., 2023), Griffin (De et al., 2024), Jamba (Lieber
157   et al., 2024), xLSTM (Beck et al., 2024), HGRN2 (Qin et al., 2024), RetNet (Sun et al., 2023),
158   RecurrentGemma (Botev et al., 2024). Waleffe et al. (2024) present a detailed study of how to
159   train large-scale Mamba-based language models. Works like MambaVision (Hatamizadeh & Kautz,
160   2024), MambaND (Li et al., 2024b), Vision Mamba (Zhu et al., 2024), VideoMamba (Li et al.,
161   2024a), and Sonic (CartesiaAI, 2024) have all shown how a Mamba layer can process visual data
      and other modalities. Liu et al. (2024) present a detailed survey of Mamba-based vision models.

      3.1    M AMBA 2 P RELIMINARIES
165
166   MatMamba is based on Mamba2. We make simple modifications to the Mamba2 block to impose the
167   Matryoshka structure. A detailed description of the internals of Mamba2 can be found in the original
168   paper Dao & Gu (2024). However for the purposes of this work, we treat the Mamba2 block as a
169   combination of an input linear projection (Win , which can be broken down into Wz , Wx , WB , WC ,
170   Wdt ), a causal 1D convolution layer with kernel size 4 (with weights that are a concatenation of
171   Wconvx , WconvB , and WconvC applied in groups), a chunk + selective scan operation (SSM), and
172
      an output projection layer (Wout ). Similar to a Transformer, this block takes in an (b, l, d) shaped
      tensor – b is batch size, l is sequence length, and d is the dimensionality – and produces a (b, l, d)
173
      shaped output after a sequence transformation. For an input tensor u, the Mamba2 block M (u)
174
      consists of the following steps:
175
176                 XBC(u) = σ(Conv(Wconvx ⌢ WconvB ⌢ WconvC , Wx .u⌢ WB .u⌢ WC .u))                     (1)
177
                                      Y (u) = SSM(XBC(u), Wdt .u, A, D)                                  (2)
178
                                                                    T
179                                   M (u) = Norm(Y (u).σ(Wz .u)).Wout                                  (3)

      where ⌢ is the concatenation operation, Conv(k, s) applies a 1-D causal convolution with weights
181
      k (applied in len(k) groups) on a sequence s, and A and D are learnable SSM parameters. σ is a
182   nonlinearity which we set to SiLU (Elfwing et al., 2018), and Norm is a layer norm function which
183   we set to RMSNorm (Zhang & Sennrich, 2019).

185   3.2    M AT M AMBA B LOCK
186
187   A MatMamba block also has both input and output shapes as (b, l, d). It is defined as a nested
188   combination of g Mamba2 blocks Mi , such that M1 ⊂ M2 ⊂ ... ⊂ Mg , where Mi ⊂ Mj means
189   that all the parameters of a sub-block Mi are present in Mj . Works like MatFormer (Devvrit et al.,
190   2023), OFA (Cai et al., 2019), and Flextron (Cai et al., 2024b) all share similar designs in which
191   the largest model Mg is the single universal base model from which numerous smaller submodels
192
      Mi can be flexibly extracted. In MatMamba, we impose the nested structure along the dimensions
193
      of the model parameters. Specifically for a sub-block Mi with expansion factor e = ddmodel inner
                                                                                                       , we
194   choose a Matryoshka dimension mi , such that 0 < mi < dmodel , which results in an inner slice
                                                                  di
195   dimension di = e × mi and number of heads hi = dhead           , subject to di mod dhead = 0. For
196   example, parameters like Wx have a shape of (dinner , dmodel ). For the Mi sub-block, it will become
197   Wx [0 : di ] by slicing it along the dinner dimension. Similarly for parameters like A which have a
198
      shape of (nheads ), it will become A[0 : hi ]. Concretely, the MatMamba block Mi (u) when applied
      to an input tensor u is these steps:
199

206   In practice, Wz , Wx , WB , WC , and Wdt are implemented as a single input projection layer with
207   tensor parallelism, with appropriate rearranging of dimensions depending on mi . Figure 1 illustrates
208   the MatMamba block. We also provide PyTorch-style pseudocode for the block in Appendix A, to
209   provide a clearer understanding of our implementation.
210   Compared to MatFormer (Devvrit et al., 2023), where the Matryoshka structure is only applied on
211   the MLP subblock of the Transformer block, MatMamba applies nesting to the entire block wherever
212   the inner dimension plays a role. This leads to a nearly linear reduction in total parameter count (and
213   also a nearly linear reduction in flop count due to the nature of Mamba2). Also, typically > 95%
214   of the parameter count in a MatMamba block is in the input and output projections, which can
215   be converted into nested layers while maintaining the well understood systems characteristics of
      projection layers. See Appendix A for a detailed example of parameter count reduction.

      We can stack L such MatMamba blocks to create a MatMamba model. For a given mi and nested
217   blocks M1 ⊂ M2 ⊂ ... ⊂ Mg , we can create a MatMamba model fi with L layers, and g nested
218   models f1 ⊂ f2 ⊂ ... ⊂ fg . Each fi is formed by stacking Mi L times. Like Mamba2, the Mat-
219   Mamba backbone is a general purpose sequence processing architecture, which with an appropriate
220   tokenizer and output head can process a variety of modalities.

242   3.4   M IX ’ N ’M ATCH
243
244   We can apply the Mix’n’Match strategy from MatFormer (Devvrit et al., 2023) to flexibly extract
245   any submodel from MatMamba for inference. Concretely, for a model f with L layers, we need to
246   choose a dimensionality mi at each layer i. Note that mi can be either one of the explicitly optimized
247   g granularites (e.g. picking from one of [1024, 512, 256, 128] from a 135M-MatMamba-Vision
248   model, see section 4.1), or we can choose interpolated dimensionalities that were not explicitly
249   optimized for (e.g. picking any random valid value like 768 or 384 that was not explicitly trained).
250   For instance, we could choose m1 = 256 (25% size) in layer 1, m2 = 1024 (100% size) in layer 2,
251
      m3 = 768 (75% size) in layer 3, and so on. The only constraint on mi in MatMamba is that it needs
252
      to lead to an integer number of heads, or that (e × mi ) mod dhead = 0, where e = ddmodel  inner
                                                                                                       . This
253   leads to a combinatorially large number of possible submodels (beyond the g explicitly optimized
254
      granularities) that can be flexibly extracted – all from the same set of base model weights – as shown
      in Figure 1. Due to the Matryoshka structure, the first few dimensions (that are shared among all the
255
      nested submodels) are incentivized to learn the strongest representations.

258   3.5   E LASTIC I NFERENCE
259
260   When deploying a MatMamba model for inference, we typically need to store the single universal
261   model fg in memory. If compute is not constrained (or if the inference workload is predictable),
262   then we can use the full model to get the most accurate results. However, depending on dynamic
263   constraints (e.g. available inference compute, energy usage, system load, desired accuracy etc.), we
264   can perform a forward pass on a chosen slice of the network on the fly.
265   There are exciting possibilities like combining cloud and edge inference – we could store a smaller
266   model fi on the edge device and when necessary, use the larger model fj on the cloud, or using a
267   smaller model to act as a draft model for speculative decoding (Leviathan et al., 2023) with a larger
268   verifier model. We could also potentially do input-adaptive sub-model selection (e.g. use a larger
269   model for a more difficult input). All of these are possible only because MatMamba has a consistent
      and nested Matryoshka structure, in which all the sub-models share the same metric space.

300   unidirectional SSM scan. One crucial design choice we make is to use the [CLS] token as a suffix
301   instead of the conventional prefix. This allows it to attend to information from the entire sequence.
302   We find that this simple architecture works effectively on both image classification and adaptive
303   retrieval. We train two model variations (35M with dmodel = 512 and 135M with dmodel = 1024,
304   see Table 1) with patch size 16 and L = 20 layers on ImageNet-1k Deng et al. (2009) which has
305   1.28M training images and 50k validation images. Compared to other recent work on SSM’s for
306   vision tasks like MambaVision (Hatamizadeh & Kautz, 2024), MambaND (Li et al., 2024b), and
307   Vision Mamba (Zhu et al., 2024) – all of which have major design changes on top of Mamba layers
308
      like bidirectional scan with additional projections, varying order of scans, or combining SSM layers
      with attention and convolution layers – we keep our network architecture as simple as possible.

      4.1.2                  A DAPTIVE I MAGE R ETRIEVAL
413
414   Image retrieval aims to locate semantically similar images using representations generated by a
415   pretrained encoder (Chen et al., 2022). The standard method involves encoding both database and
416   query images with the same encoder and then performing nearest neighbor retrieval. While using
417   a powerful encoder for database images is feasible, the query encoder must be efficient for real-
418   time applications. Moreover, query encoding scenarios can vary, such as on-device versus cloud
419   processing and varying query load and complexity. Existing solutions with fixed encoders often
420   compromise accuracy or cost in different settings.
421   Due to its flexibility, MatMamba-Vision is a promising candidate for query encoding. However, re-
422   trieval also requires that submodels maintain distance relationships between fixed database (encoded
423   with a larger encoder) and query embeddings across various granularities. Using smaller baseline
424   Mamba2 models solely for query encoding can lead to significant distance preservation issues and
425   poor retrieval accuracy (as illustrated in Figure 5).
426   We evaluated both the baseline and MatMamba-Vision encoders on ImageNet-1K for image retrieval
427   at 35M and 135M parameter scales. Using the [CLS] token representation, we calculated 1-nearest
428   neighbor (NN) accuracy. Figure 5 demonstrates that submodels extracted from MatMamba can
429   effectively preserve distances and offer greater flexibility. For example, MatMamba-Vision-135M
430   can reduce compute cost by 55% with a minimal accuracy loss of less than 0.5%. While causal
431   models with suffix [CLS] token might not be as accurate as bi-directional encoders for retrieval, this
      is a promising start towards better long-context encoders while enabling adaptive query processing.

      5                  C ONCLUSIONS
524
525   In this work, we presented MatMamba, which is a way to impose a nested Matryoshka structure
526   on a Mamba2 state space model. It brings together the best of both Mamba-style models (faster
527   inference times, especially for longer sequences) and Matryoshka-style learning. A single Mat-
528   Mamba model contains hundreds of nested and accurate submodels that can be flexibly extracted
529   for inference. MatMamba-Vision and MatMamba-LM models match the performance and accuracy
530   of the independently trained Mamba2 baselines. MatMamba models allow us to choose a desired
531   performance-compute tradeoff, all while being a single Matryoshka-style model instead of multiple
532   different models for specific scenarios. This enables interesting use cases like speculative decod-
533
      ing using a smaller draft model and a larger verifier model, input-adaptive submodel selection, and
      hybrid cloud-edge inference with the same model based on available compute.

705   # Example MatMamba parameters
      d_model = 1024
706   expand = 2
707   headdim = 64
      d_state = 128
708   d_inner = expand * d_model
      n_heads = d_inner // headdim
709
      # Learnable parameters, their shapes:
710   w_z       # (d_inner, d_model)
711   w_x       # (d_inner, d_model)
      w_B       # (d_state, d_model)
712   w_C       # (d_state, d_model)
      w_dt      # (n_heads, d_model)
713   D         # (n_heads)
714   A         # (n_heads)
      w_conv_x # (d_inner, 1, 4)
715   w_conv_BC # (2 * d_state, 1, 4)
      w_out     # (d_model, d_inner)
716
      def matmamba_layer(x_in, mat_dims):
717       '''
718       Arguments:
              x_in: (batch, seq_len, d_model)
719           mat_dims: how many matryoshka dims to select in this block
          Returns:
720           y: (batch, seq_len, d_model)
721       '''
          mat_d_inner = expand * mat_dims
722       mat_n_heads = mat_d_inner // headdim
          assert mat_d_inner % headdim == 0
723
          # Matryoshka structure on dims of W_z and W_x, and number of heads of W_dt
724       w_in_proj = torch.cat(
725           [w_z[:mat_d_inner, :], w_x[:mat_d_inner, :], w_B, w_C, w_dt[:mat_n_heads, :]],
              dim=0
726       )
727       zxbcdt = F.linear(x_in, w_in_proj)
728       z, xBC, dt = torch.split(zxbcdt, [mat_d_inner, mat_d_inner + 2*d_state, mat_n_heads], dim=-1)

729       # Matryoshka structure on W_conv_x based on mat_dims
          w_conv = torch.cat([w_conv_x[:mat_d_inner], w_conv_BC])
730       xBC = F.conv1d(xBC, w_conv, groups=mat_d_inner + 2 * d_state)
          x, B, C = torch.split(xBC, [mat_d_inner, d_state, d_state], dim=-1)
731
732       # Matryoshka structure on number of heads in dt, A, and D
          y = mamba_chunk_scan_combined(x, dt[:mat_n_heads], A[:mat_n_heads], B, C, D[:mat_n_heads])
733
          y = rmsnorm(y * F.silu(z), w_norm)
734
735       # Matryoshka structure on dims of W_out
          y = F.linear(y, w_out_proj[:, :mat_d_inner])
736
          return y
