                                                          A BSTRACT
010
011
                 Generating SQLs according to user queries (text-to-SQL) is a long-standing sequen-
012
                 tial challenge, where the accuracy of the initial schema linking significantly impacts
013
                 the subsequent SQL generation performance. However, existing models often focus
014              more on SQL generation and less on the schema linking task, leading to potential
015              missing or redundant schema linking and suboptimal SQL generation performance.
016              The underlying reason is that schema linking is not a simple selection problem but
017              a Knapsack problem, which should consider both the value of the schema linking
018              in terms of missing important information and the weight of the schema linking
019              in terms of providing redundant information. Motivated by this, we provide two
020              tailored SL benchmarks and two tailored metrics to train SL agents and to evaluate
021              the missing and redundant schema linking. In this paper, we propose the Knapsack
022
                 Schema Linking Agent (KaSLA), which can link the most valuable and least
                 redundant schema element subsets for both tables and columns. KaSLA introduces
023
                 an importance score function to predict each schema element’s importance score,
024
                 and then utilizes the importance score to estimate the value and the weight of each
025              schema. Then, by estimating the capacity, the maximum weight the knapsack can
026              hold, of a given user query from historical SQL records, KaSLA employs efficient
027              dynamic programming to select the most valuable schema element set within the
028              estimated capacity. Extensive experiments on two benchmark datasets demonstrate
029              the superior performance of KaSLA over 12 state-of-the-art baselines. Especially
030              on the popular and challenging BIRD benchmark, KaSLA can outperform the
031              baselines by over 5.72%.

      2       P RELIMINARIES

126
      Notations. Given a natural language query q and a database schema S = {s1 , · · · , s|S| }, where
      si represents either a table or a column in the database, conveying the structured representation of
127
      the corresponding database. The database schema S can be further divided into two subsets: the set
128
      of tables T = {t1 , · · · , t|T | } and the set of columns C = {c1 , · · · , c|C| }, such that S = T ∪ C. For
129
      each table t ∈ T , its corresponding set of columns is denoted as Ct = {c1 , · · · , c|Ct | }.
130
131   Knapsack Problem of Schema Linking. As discussed in the introduction, schema linking perfor-
132   mance is negatively impacted by both missing and redundant elements, indicating that schema linking
133   is not a simple selection problem. For example, directly feeding all schemas into the text-to-SQL
134   models can capture all possible schemas but will suffer from heavy redundancy. On the other hand,
135   when the total number of schemas is limited, there is a higher possibility of missing important ones.
136
      Motivated by this, schema linking can be defined as a knapsack problem (Fréville, 2004), where the
137   objective is to maximize the information contained in the schema linking set while controlling the
138   total weight of the set. Formally, we have the following definition:
139
      Definition 2.1 (Knapsack Schema Linking Problem). For any given query q, let Cq represent the total
140
      weight capacity of schemas for this query. For each schema s, let Vs,q denote the value of the schema
141   element given user query q, and let Ws,q denote the weight of the schema given the query.
142
143   The optimal schema linking can be achieved by solving the following optimization problem:
                                         X                        X
144                       S ∗ = arg max      Vs,q , subject to        Ws,q ≤ Cq .                              (1)
145                                      S⊆S
                                                s∈S                      s∈S
146
147   In this formulation, the objective is to maximize the total value of the selected schema elements while
148   ensuring that the total weight of the selected elements does not exceed the capacity Cq . By solving
149   this knapsack problem, we can obtain the optimal schema linking set that balances the trade-off
150   between including relevant information and minimizing redundancy.
151   To solve the Knapsack Schema Linking Problem, we need to estimate the value Vs,q and weight Ws,q
152   of each schema element given the user query, as well as determine the capacity Cq for each query. In
153   the following sections, we will first introduce the schema linking benchmark and then introduce our
154   proposed KaSLA.

173   3.2   S CHEMA L INKING M ETRICS
174
175   Limitations of Previous Studies Previous text-to-SQL research usually overlooked the evaluation
176   of schema linking (Pourreza & Rafiei, 2023). Other works mentioned schema liking performance still
177   use classification metrics such as AUC (Li et al., 2023a; 2024b), Recall, Precision, and F1 scores (Qu
178   et al., 2024). However, any missing of the necessary element will result in wrong SQL generation.
179   Besides, Recall, Precision, and F1 cannot quantitatively evaluate the actual linking results. AUC is
180
      also not suitable because schema linking is an imbalanced classification task. To this end, we propose

181
      performance rather than traditional metrics. We provide a detailed discussion of the limitations of
182
      traditional metrics and compare them with the proposed metrics in the Appendix G.

      4.2   K NAPSACK FACTOR E STIMATION
264
265   As formulated in Eq. (1), for a given schema element s and query q, KaSLA employs Vs,q and
266   Ws,q to represent the value and weight of s, respectively. For query q, KaSLA utilizes Cq to denote
267   the total weight capacity. Unlike traditional knapsack problems Fréville (2004), KaSLA lacks
268   ground truth values for these factors. This section proposes three estimation functions to predict the
269   aforementioned factors, enabling KaSLA to select the most valuable element subsets while adhering
      to the total weight capacity constraint.

      4.3   H IERARCHICAL K A SLA S CHEMA L INKING
318
319   Our KaSLA operates in a hierarchical manner to effectively address schema linking challenges.
320   The process unfolds in two stages: first, KaSLA performs table linking, and subsequently, for each
321   selected table, it simultaneously accomplishes column linking. This hierarchical approach enables
322   KaSLA to efficiently reduce the dimensionality of the schema space, thereby enhancing its capability
323   to handle large-scale and complex schema linking and text-to-SQL generation tasks encountered in
      real-world applications.

526   7               C ONCLUSION
527
528   This paper introduced the Knapsack Schema Linking Agent (KaSLA), a novel approach to address
529   the schema linking challenges in text-to-SQL tasks. By framing schema linking as a Knapsack
530   problem, KaSLA effectively balances the trade-off between missing and redundant schema linkages,
531   significantly enhancing the accuracy of SQL generation. Our proposed benchmarks and metrics
532   provide a new standard for evaluating schema linking performance, fostering further advancements
533   in the field. Extensive experiments demonstrate KaSLA’s superiority over existing methods.
534   These findings highlight KaSLA’s potential to revolutionize schema linking and advance the broader
535   capabilities of text-to-SQL systems. By effectively reducing both missing and redundant information,
536   KaSLA not only improves schema linking accuracy but also enhances the overall performance of
537   SQL generation. The innovation of the nomination-guaranteed score function plays a crucial role
538   in overcoming the missing and redundancy seesaw problem, offering a robust solution that can be
539   integrated into mainstream text-to-SQL models. Our contributions pave the way for more precise and
      efficient database interactions, underscoring the transformative impact of KaSLA in the field.

697   A    R ELATED W ORK
698
699   Text-to-SQL studies have witnessed significant advancements and development over the years in
700   natural language processing (NLP) research (Wang et al., 2022; Qin et al., 2022). The techniques
701   involved in text-to-SQL implementation have undergone a long period of evolution (Hong et al.,
      2024b). Early methods in text-to-SQL research largely focused on template and rule-based human


                                                      13
      Under review as a conference paper at ICLR 2025


702
      engineering (Li & Jagadish, 2014; Mahmud et al., 2015; Yu et al., 2021). Subsequently, with the
703   emergence of deep learning (Katsogiannis-Meimarakis & Koutrika, 2023) and pre-trained language
704   models (PLMs) (Liu et al., 2019; Devlin et al., 2019), text-to-SQL has further advanced along with
705   these developments (Yin et al., 2020; Choi et al., 2021; Li et al., 2023b). Most recently, as LLMs
706   have gained prevalence in both research-oriented papers and industrial projects, text-to-SQL systems
707   integrating LLMs are now a research hotspot in the NLP and database communities. Our study
708   follows an LLM-based text-to-SQL paradigm, consisting of in-context learning and fine-tuning
709   techniques.
710
711
      A.1   I N - CONTEXT L EARNING - BASED TEXT- TO -SQL
712
713
      At the inception of LLMs, in-context learning (ICL) and prompt engineering emerged as a core
714
      method driving advancements in the study of LLMs (Reynolds & McDonell, 2021). Early efforts in
715   LLM-based text-to-SQL studied the effectiveness of ICL with different prompt designs on various
716   LLMs (Liu et al., 2023). The natural language understanding (NLU) capabilities empowered by
717   numerous training corpora enable LLMs to perform well in SQL generation with simple prompt
718   engineering (Rajkumar et al., 2022). With the success of well-designed ICL methods in other NLU
719   tasks (Wei et al., 2022; 2021), the integration into the text-to-SQL task has also achieved solid
720   improvement (Tai et al., 2023). Chain-of-Thought (CoT) prompting (Wei et al., 2022) shows great
721   potential in natural language reasoning tasks. DIN-SQL (Pourreza & Rafiei, 2023) proposed a CoT-
722   based decomposed ICL framework with self-correction; ACT-SQL () designed a method to generate
723
      automatic CoT exemplars to enhance SQL reasoning. To optimize the prompt towards better quality,
      DAIL-SQL (Gao et al., 2024) introduces a few-shot sampling strategy, providing related samples
724
      for LLMs to learn from, and the Knowledge-to-SQL (Hong et al., 2024a) framework is proposed
725
      to generate helpful knowledge to assist SQL generation. Self-consistency (Wang et al., 2023) for
726   LLM-based text-to-SQL ensures accuracy based on the execution result. The C3 (Dong et al., 2023)
727   framework conducts a majority vote of consistency; SQL-PaLM (Sun et al., 2023) introduces an
728   error-filtering-aided consistency decoding based on execution for better SQL generation. Multi-stage
729   decomposition is also popular in text-to-SQL studies. MAC-SQL (Wang et al., 2024) employs a
730   multi-agent collaboration framework to generate and refine SQL; TA-SQL (Qu et al., 2024) introduces
731   a two-stage generation framework incorporating schema linking and logical synthesis. ICL-based
732   methods are the mainstream paradigm of LLM-based text-to-SQL, which have made significant
733   progress and continue to be widely studied in the most current work. CHASE-SQL (Pourreza et al.,
734
      2024) proposes an in-context learning prompt for question decomposition and involves a novel
      online synthetic example generation method to adapt LLMs to test datasets. Our proposed method
735
      also follows the ICL paradigm for text-to-SQL. Firstly, we generate a schema linking and then use
736
      adaptive evaluation to improve its quality, providing clear guidance to understand the user query and
737   the corresponding database schema. Then, the LLMs incorporate the provided schema linking to
738   generate accurate SQL queries.
739
740
741   A.2   F INE - TUNING - BASED T EXT- TO -SQL
742
743   Fine-tuning is an intuitive and widely recognized technique for enabling LLMs to perform specific
744   downstream tasks (Wei et al., 2021). The fine-tuning methods for aligning LLMs with instructional
745   tasks are gradually evolving towards better performance and greater effectiveness (Rafailov et al.,
746   2023). Even though code-specific LLMs are trained on massive programming scenarios, they still
747
      struggle when facing challenging user queries and complex database environments (Gan et al.,
      2021; Deng et al., 2021; Li et al., 2023c). As a straightforward method, supervised fine-tuning
748
      (SFT) (Wei et al., 2021) is utilized to adapt open-source LLMs for SQL generation, which elicits a
749
      solid improvement (Gao et al., 2024). For well-designed methods, CodeS (Li et al., 2024b) proposes
750   a two-stage training framework. First, a backbone code-LLM is pre-trained on an incremental
751   training corpus, followed by SFT on bi-directional augmented query-SQL pairs, achieving impressive
752   performance with open-source LLMs. Instead of fine-tuning a single model, DTS-SQL (Pourreza &
753   Rafiei, 2024) fine-tunes two LLMs separately for schema linking generation and SQL generation.
754   This two-stage generation process elicits higher accuracy. Our work involves fine-tuning the LLMs
755   and the PLMs based on the schema linking task, then incorporating the generated schema linking to
      assist in accurate SQL generation.

      A.3   S CHEMA LINKING
757
758   Schema linking is a crucial step in text-to-SQL tasks, involving the identification and association of
759   natural language query elements with corresponding database schema components, such as tables
760   and columns. This process significantly influences the overall performance of SQL generation and
761   has been the focus of extensive research.
762   With the advent of Large Language Models (LLMs), schema linking has seen considerable innovation
763   and potential. DIN-SQL (Pourreza & Rafiei, 2023) marked a significant advancement by using LLMs
764   to generate schema linkages, utilizing the entire schema as input to leverage the LLMs’ contextual
765   understanding. Building on this, DAIL-SQL (Gao et al., 2024) incorporated historical query-SQL
766   pairs as evidence, refining schema linking by utilizing past interactions. DELLM (Hong et al., 2024a)
767   extended these capabilities with a specialized data expert LLM, providing additional knowledge
768   to improve schema linking in complex scenarios. Similarly, E-SQL (Caferoğlu & Ulusoy, 2024)
769   introduced a query enrichment method that incorporates relevant database elements to enrich user
770
      queries rather than simplifying the full schema. DTS-SQL (Pourreza & Rafiei, 2024) fine-tuned
      LLMs specifically for table linking and SQL generation, ensuring task-focused training at each
771
      process stage. TA-SQL (Qu et al., 2024) introduced the approach of generating a dummy SQL to
772
      abstract linked schemas via an LLM, allowing for iterative refinement before the final SQL generation.
773   Conversely, CodeS (Li et al., 2024b) implemented a retrieval-based strategy, recalling semantically
774   matching schema elements as inputs for SQL generation models. CHESS (Talaei et al., 2024) treated
775   column linking as a binary classification task to eliminate obviously irrelevant columns and then
776   linked tables and columns from the filtered schema. To address the challenge of missing elements,
777   RSL-SQL (Cao et al., 2024) proposed bidirectional schema linking, with forward schema linking
778   identifying potential matching elements from the full schema and backward schema linking extracting
779   elements from preliminary SQL generated based on these potential matches. Distillery (Maamari
780   et al., 2024) explored schema linking performance with extremely large LLMs like GPT-4o and
781
      Llama 3.1-405b, finding that these models can effectively process full schema without a schema
      linking model, while moderately-sized models, like Llama 3.1-8B, still rely on schema linking.
782
783   Effective schema linking demands no missing elements and minimal redundant elements; any missing
784   schema element can cause SQL generation to fail directly, while redundant elements can confuse the
785   process and lead to incorrect outputs. This challenge aligns with the knapsack problem—maximizing
786   value while minimizing weight within constraints. Inspired by this, we formulated schema linking as
787
      a knapsack problem and proposed KaSLA, which aims to link the most relevant and least redundant
      schema element sets according to user queries.

831

1005   the actual impact of schema linking results to SQL generation by a strict evaluation. It considers

       an instance a failure if any element is missing, acknowledging success only when no elements are

1011   The prediction with element missing will be judged as failure because, in such cases, redundancy is
