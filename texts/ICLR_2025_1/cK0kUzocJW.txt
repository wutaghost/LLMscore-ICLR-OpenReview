008                                                  A BSTRACT
009
010            Recent advancements in Large Language Models (LLMs) have significantly in-
011            creased context window sizes, enabling sophisticated applications but also in-
012            troducing substantial computational overheads, particularly computing key-value
013            (KV) cache in the prefill stage. Prefix caching has emerged to save GPU power
014            in this scenario, which saves KV cache at disks and reuse them across multiple
015            queries. However, traditional prefix caching mechanisms often suffer from sub-
016            stantial latency because the speed of loading KV cache from disks to GPU mem-
017            ory is bottlenecked by the throughput of I/O devices. To optimize the latency of
018
               long-context prefill, we propose Cake, a novel KV cache loader, which employs
               a bidirectional parallelized KV cache generation strategy. Upon receiving a pre-
019
               fill task, Cake simultaneously and dynamically loads saved KV cache from prefix
020
               cache locations and computes KV cache on local GPUs, maximizing the utiliza-
021            tion of available computation and I/O bandwidth resources. Additionally, Cake
022            automatically adapts to diverse system statuses without manual parameter. tuning.
023            In experiments on various prompt datasets, GPUs, and I/O devices, Cake offers
024            up to 68.1% Time To First Token (TTFT) reduction compare with compute-only
025            method and 94.6% TTFT reduction compare with I/O-only method.

034   ing (Wang et al., 2024), long-context Retrieval Augmented Generation (RAG) (Jiang et al., 2024),

      Jin et al., 2024a; Gim et al., 2024). LLM inference engines such as vLLM (Kwon et al., 2023) and
050
      SGLang (Zheng et al., 2023) cache KV in local CPU memory, but this approach is limited by CPU
051   memory size. Given that disks offer much larger capacity and are more cost-efficient than CPU

084   KV cache. It employs a bidirectional parallelized KV cache generation strategy: upon receiving a
085   prefill task, Cake immediately utilizes available GPU power to compute KV cache in normal order
086   starting from the first token, while simultaneously loading the prefix cache in reverse order starting
087   from the last token, until all tokens have corresponding KV cache data. This design is inspired by

089   cost of generating KV cache increases with the token’s distance from the beginning of the sequence,
090   while the data streaming cost remains constant regardless of token position. Importantly, Cake’s
091   design is both simple and elegant, adapting to diverse situations in terms of system architecture and

113   2     BACKGROUND
114
115   2.1   KV CACHE IN LLM I NFERENCE
116
      Large language models (LLMs) have recently made significant impacts across numerous domains.
117   The key to their success lies in the attention mechanism, which enables these models to scale up and
118   effectively process long contexts (Vaswani et al., 2017; Brown et al., 2020). In the attention calcu-
119   lation process, the computation of Key (K) and Value (V) vectors for previously processed tokens
120   becomes redundant during the decoding phase. Recognizing this, the concept of KV cache (Zhang
121   et al., 2023) is introduced. This approach involves storing these computed values and reusing them
122   to reduce computational overhead in subsequent decoding steps.
123   State-of-the-art LLM inference engines (Kwon et al., 2023; Miao et al., 2024; Zheng et al., 2023)
124   typically divide the inference procedure into two distinct phases: prefill and decode. The prefill
125   stage generates the initial KV cache for the input prompt. In the subsequent decode stage, the
126   model utilizes this cache to generate new tokens sequentially. As each new token is produced,
127   its corresponding K and V vectors are computed and appended to the KV cache. This caching
128   mechanism significantly accelerates inference by converting the time complexity of token generation
129   from quadratic to linear (Yang et al., 2024).
130
      2.2   KV CACHE S AVING AND R EUSING
131
132   The prefill stage is a highly resource-consuming procedure that often saturates GPU computation
133   resources and causes high latency compared to the normal decoding procedure (Agrawal et al.,
134   2024). This overhead becomes even more significant in long context scenarios (Fu, 2024).
135
      Meanwhile, it is common for parts of prompts to be reused across multiple requests. For instance,
136   system messages in chatbot services guiding the LLM model’s behavior are usually long and shared
137   across multiple messages. In Retrieval-Augmented Generation scenarios, fetching long text blocks
138   as the context to generate answers can improve the generation performance (Jiang et al., 2024).
139   Other use cases include coding assistants that need to maintain a summarized version of the codebase
140   in the prompt (Cursor, 2024), and agentic search, tool use and multi-agent communication which
141   require multiple rounds of API calls using the same set of historical data (Wu et al., 2023; Li et al.,
142   2023b).
143   To address these challenges and optimize inference efficiency, various systems have been developed
144   to save and reuse KV cache via prefix caching mechanism (Zheng et al., 2023; Kwon et al., 2023;
145   Juravsky et al., 2024; Jin et al., 2024a; Gim et al., 2024; Gao et al., 2024). A typical workflow of
146   these systems are demonstrate in Figure 1. These systems leverage different layers of the storage
147   hierarchy, each with its own trade-offs. GPU memory offers the lowest latency but has the highest
148   cost and smallest capacity, making it impractical for long-term KV cache storage. CPU memory,
149   used by some inference engines (Zheng et al., 2023; Kwon et al., 2023), is often insufficient for
150
      large-scale online serving systems handling millions of requests per second. Consequently, local
      or remote disk storage has emerged as a more viable option for large-scale operations, offering
151
      a balance between cost and capacity. The trend towards disk-based KV cache storage is evident
152
      in industry practices. For example, Deepseek, a major LLM API service provider, implements
153   prefix caching on disk, potentially reducing users’ inference costs by up to 90% (Deepseek, 2024).
154   Similarly, the multi-agent framework AutoGen employs disk-based prefix caching (AutoGen, 2024).
155
156
      Cake aligns with the prefix caching mechanism for long-context LLM inference, improving latency
      by simultaneously scheduling KV computation and loading.

      2.3   C HUNK P REFILL
159
160   Chunk prefill is a technique used to optimize the prefill stage of LLM inference, particularly for
161   long input sequences. Unlike the decoding procedure, which is memory-bound, prefill stage is a
      computation-intensive process that demands significant GPU resources for extended periods. For


                                                        3
      Under review as a conference paper at ICLR 2025


162
      instance, prefilling 10,000 tokens on a 7B model on an A100 GPU takes approximately 1 seconds.
163   Directly processing a long text prompt in its entirety for prefilling would monopolize the GPU,
164   severely impacting the latency of other tasks.
165
166
      Originally proposed in Sarathi (Agrawal et al., 2023; 2024), this method divides the input sequence
      into smaller, near-equal sized chunks and processes them sequentially. By breaking down large
167
      prefill requests into manageable chunks, chunk prefill allows for improved throughput and reduced
168
      prefill operations’ latency impact in LLM serving systems. It enables the interleaving of prefill op-
169   erations with decode operations, minimizing the blocking effect of long prompts on other tasks.This
170   approach has been widely adopted in current mainstream LLM inference engines (Zheng et al.,
171   2023; Kwon et al., 2023).
172
      In vLLM’s detailed implementation of chunk prefill (Kwon et al., 2023), the inference engine forms
173
      a batch of requests for each inference step based on a predetermined token budget. The scheduler
174
      prioritizes decode requests, allocating one token from the budget to each. Any remaining tokens
175   in the budget are then assigned to prefill requests. This dynamic allocation determines the chunk
176   size for chunk prefill operations. By giving precedence to decode requests, this approach minimizes
177   interference with ongoing decoding requests, ensuring lower inter-token latency (ITL) for decode
178   while efficiently utilizing available resources for prefill tasks.
179
      Cake adopts chunk prefill by default as it aims for the long-context inference scenarios.

      3   E XPLORATORY E XPERIMENTS AND M OTIVATION
182
183   Upon receiving a new prefill task, the KV cache can be obtained through various sources, including
184   the computation on local GPU, fetching from local disk, or data streaming from a remote loca-
185   tion, as mentioned in § 2. This section evaluates the performance of different KV cache loading or
186   generation methods and identifies potential improvements, motivating the design of Cake.
187
      Finding 1: The latency of loading KV cache from local or remote disks is linearly correlated
188
      with the data size.
189
190   The size of the KV cache grows linearly with the number of tokens and can be calculated as:
191                                KV cache size = 4 · Nl · H · Lmax · B · P,                          (1)
192   Where Nl is the number of layers, H is the hidden size, Lmax is the maximum context length, B
193   is the batch size, and P is the precision in bytes. Using this formula, we can estimate that for the
194   Llama3-70B model with FP16 precision, the KV cache for a single token consumes approximately
195   2.5 MB. Consequently, a 15-page research paper contains around 10,000 tokens, would require a
196   great amount of memory as large as 25 GB.
197
      Because of the size, the process of loading long-context KV cache into GPU memory is not fast
198   because of the constrained bottleneck throughput of the I/O devices. Local storage options vary in
199   performance: SATA SSDs offer I/O bandwidths of around 600 MB/s, HDDs are limited to about
200   200 MB/s, and high-end NVMe SSDs can achieve read speeds of up to 3 GB/s at a premium cost.
201   Even in the optimal case with a 3 GB/s read bandwidth, loading the KV cache for a 10-page paper
202   would take approximately 8 seconds. Such latency will be even longer when using remote storage,
203   whose bottleneck becomes the network bandwidth. Typical network bandwidths rarely exceed 20
204   Gbps (i.e., 3GB/s) (Liu et al., 2023; Jain et al., 2023), potentially leading to transfer times of up to
205   10 seconds to fetch the 25 GB KV cache from a remote disk.
206   Finding 2: The latency and memory usage of computing KV cache on GPUs increases quadrat-
207   ically with the length of the sequence.
208
      During the prefill stage, the computation cost for generating KV cache of later tokens (i.e., those
209
      with higher indices in the sequence) is expected to be higher than for earlier tokens This is because
210   of the inherent nature of attention mechanisms — computing KV cache involves attention operations
211   across the current and all preceding tokens. To test this hypothesis, we conducted experiments on
212   long-context prefill tasks with the chunk prefill mechanism described in § 2.3. Results presented in
213   Figure 3 demonstrate a clear pattern: the latency for each chunk linearly increases with its index;
214   i.e., the latency for the whole sequence is quadratically correlated with the sequence length. We can
215   also observe the KV cache memory usage linearly increases with the chunk index. The observation
      is critical for Cake to arrange computation tasks optimally.

      Conclusion. The above experiments highlighted the limitation of existing prefix caching – load-

      though consuming GPU power, is fast enough to potentially accelerate the prefill process. Our find-

243   GPU power and I/O bandwidth. This strategy forms the core concept of Cake.

       4                               D ESIGN OF Cake
246
247   We propose Cake, a system that adaptively utilizes both network and available computation re-
248   sources to achieve low-latency KV cache loading. In this section, we first discuss the scope of Cake
249   by specifying its use cases in Section 4.1. We then elaborate design details in Section 4.2 and analyze
250   the benefit of Cake in Section 4.3.
251
252
       4.1                                 U SE C ASES OF Cake
253   Cake works upon a LLM serving system with prefix caching as depicted in Figure 1. Cake is
254   especially beneficial when the system needs to load saved KV cache from disk storage to GPU
255   Memory for inference. It’s worth noting that in cases where the required KV cache data already
256   resides in GPU or CPU memory, it is beyond the scope of Cake. This is because the bandwidth
257   between CPU and GPU is dramatically higher than disk I/O or computation speeds, then comparing
258   with I/O, computation can provide limited help. However, given the memory size constraints on
259   CPU and GPU and the large size of long-context caches, such scenarios are relatively infrequent,
260
      making Cake a crucial component for most long-context LLM inference with prefix caching tasks.
261   Cake is complementary to other optimization techniques. For instance, methods focused on reducing
262   KV cache size (Hooper et al., 2024; Jiang et al., 2023; Kang et al., 2024) or optimizing KV cache
263   loading from local/remote disks (Liu et al., 2023) can be used in conjunction with Cake, potentially
264   yielding even greater performance improvements. Furthermore, Cake is designed to integrate seam-
265   lessly with state-of-the-art LLM serving systems (Kwon et al., 2023; Yao et al., 2024), enhancing
266
      their capabilities in handling long-context scenarios efficiently.

287   The key idea of Cake is to simultaneously leverage both local computation and I/O for data streaming

      to optimize the latency of KV cache prefill. For each long-context request, Cake determines an
      efficient schedule for using all available resources to accomplish KV cache prefill, i.e., determining

      as well as the sequence of these operations. As discussed in §3, the computation of KV cache on later
291   chunks is more expensive than that on earlier chunks. Therefore, we should prioritize computation
292   operations on chunks near the beginning of the sequence while let cache loading operations to fill
293   later chunks. Inspired by this intuition, we design Cake as a bidirectional parrallelized KV cache
294   loader as below.
295
      As illustrated in Figure 4, upon receiving a request, Cake splits the sequence into chunks and initiates
296
      two simultaneous processes: (1) The local GPU computes KV cache from the beginning chunk of
297   the prompt, progressing towards the end. (2) The data streaming process fetches KV cache starting
298   from the last chunk, moving in reverse direction. This bidirectional approach continues until the two
299   processes converge in the middle, signaling the completion of KV cache generation for the entire
300   prompt. We discuss the whole algorithms in Appendix§A and the system implementation details in
301   Appendix§B.

303   4.3    B ENEFITS OF Cake
304
305   Our design offers several key benefits:
306
      • Optimized latency: Cake optimally reduces the overall latency of prefill with KV cache loading
307
      by parallelizing computation and network streaming without any idle time.
308
309   • Automatic adaptation: Cake dynamically adapts to varying conditions without relying on man-
310   ually defined parameters. It consistently offers latency improvements across different sequence
311   lengths and diverse system configurations, tolerating fluctuations in network conditions and compu-
312
      tational capabilities. This adaptability is evidenced by our comprehensive evaluation across various
      scenarios, as detailed in Section 5.
313
314   • Negligible overhead: Cake automatically balances the demand for computation and network
315   resources according to dynamic situations without substantial additional overhead. Previous
316   computation-only and network-only KV cache loading solutions (e.g., Kwon et al. (2023); Liu et al.
317   (2023)) become special cases of Cake when either computation or network is unavailable. Ad-
318
      ditionally, Cake operates without such computation-intensive profiling phase, further reducing its
      operational overhead.

488   In this section, we discuss the benefits and potential overheads introduced by Cake. Our system
489   effectively utilizes both compute and I/O resources to reduce KV cache loading latency, thereby

491
      Comparison with Compute-Only Methods. Unlike approaches that rely solely on computation,
492
      Cake leverages I/O bandwidth to reduce the computational load, resulting in lower latency. This is
493
      particularly advantageous as I/O resources are often less costly and more readily available compared
494   to high-performance compute resources.
495
496   Comparison with I/O-Only Methods. While Cake primarily focuses on efficient I/O utilization, it
497
      also employs compute resources to further reduce TTFT. This dual approach may introduce some
      additional computational overhead. However, as discussed in §5.3 and §2.3, we can strategically use
498
      only the unused token budget for chunk prefill computation. This approach minimizes the system’s
499
      computational overhead due to the batching effect and efficient GPU utilization. The added com-
500   putation is efficiently processed alongside other tasks, leveraging GPU parallelism. Even if there
501   are no unused token left, we can pause the compute procedure and let I/O contribute more to the
502   procedure.
503
      Cost-Benefit Analysis. In many online services, users who pay more often receive higher priority
504
      and faster request processing. Cake is particularly well-suited for these scenarios, offering signifi-
505
      cantly reduced TTFT for a marginal increase in cost.

      A     D ETAILS OF Cake A LGORITHM
703
704   The workflow of Cake can be described in detail as follows:
705
706
      1. Upon receiving a request, Cake first splits the input sequence into chunks of a predetermined size
      (e.g., 512 tokens as suggested by vLLM).
707
708   2. Two pointers are initialized: compute ptr starting at the beginning of the sequence (index 0),
709   and io ptr at the end of the sequence (index total tokens − 1).
710   3. Two parallel processes are initiated: a) A GPU computation thread that starts from compute ptr
711   and moves forward. b) An I/O streaming thread that starts from io ptr and moves backward.
712
713
      4. The GPU computation thread: - Computes KV cache for chunks starting from compute ptr. -
      After each chunk computation, it updates compute ptr by adding the chunk size. - Continues until
714
      compute ptr reaches or surpasses io ptr, or until the required KV cache is found in CPU memory.
715
716   5. The I/O streaming thread: - Fetches pre-computed KV cache for chunks ending at io ptr from
717   storage (local or remote) to CPU memory. - After each chunk fetch, it updates io ptr by subtracting
718   the chunk size. - Continues until io ptr reaches or goes below compute ptr.
719   6. The process concludes when the two pointers meet or cross each other, indicating that KV cache
720   for the entire sequence has been either computed or loaded.
721
      7. Finally, Cake returns the complete KV cache for the entire sequence, ready for use in the subse-
722
      quent inference steps.
723
724   This bidirectional approach allows Cake to efficiently utilize both computational and I/O resources
725   simultaneously, minimizing idle time and optimizing the overall latency of KV cache preparation
726
      for long-context LLM inference.
727

      B     I MPLEMENTATION
745
746   We implement Cake by extending LMCache (LMCache, 2024) and integrating it with vLLM (Kwon
747   et al., 2023), adding approximately 1,000 lines of code.
748
749   B.1   E NHANCEMENTS TO LMC ACHE
750
751   LMCache, originally developed as the KV cache management backend for CacheGen (Liu et al.,
752   2023), hashes token chunks into keys for efficient KV cache retrieval. To enable Cake to continu-
753   ously receive KV cache in the background, we introduce the following enhancements:
754
755   Asynchronous Retrieval We implement an asynchronous get operation to complement LM-
      Cache’s existing asynchronous put functionality. This involves creating a dedicated worker thread


                                                        14
      Under review as a conference paper at ICLR 2025


756
      that continuously reads chunk keys from a task queue and retrieves the corresponding KV cache
757   to memory. Upon successful retrieval, the chunk’s key is added to a resident dictionary for quick
758   access.
759
760   Buffer Preallocation We modify LMCache to preallocate chunk buffers as soon as a chunk key is
761   pushed to the queue. This optimization allows the worker to immediately write received KV cache
762   into memory and proceed to the next chunk without delay.
763
764   B.2   I NTEGRATION WITH LLM S ERVING S YSTEMS
765
766   Cake operates concurrently with LLM serving systems like vLLM. The integration process works
767   as follows:
768   1. Upon receiving a request, Cake divides it into chunks based on the scheduled token budget.
769
770
      2. Hashed keys for these chunks are pushed to the task queue in reverse order using the push seq
      API.
771
772   3. While the asynchronous get worker fetches KV cache from the end of the sequence, vLLM begins
773   chunk prefill from the start.
774   4. After each vLLM engine step, Cake checks if the next chunk of tokens is already in the resident
775   dictionary using the is resident API.
776
777
      5. If the chunk is resident, Cake interrupts the chunk prefill process and directs vLLM to begin token
      generation.
778
779   6. If the chunk is not resident, chunk prefill continues until it encounters a chunk present in the
780   dictionary.
781   This bidirectional approach allows Cake to efficiently utilize both I/O and computational resources,
782   potentially reducing the Time To First Token (TTFT) for long-context LLM inference tasks.

789   test ranges from 2000-10000 mbps. Across all the conditions, Cake with CacheGen produce the best
