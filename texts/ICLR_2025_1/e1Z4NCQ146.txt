009                                              A BSTRACT
010
011
012            Training billion-scale large language models (LLMs) with just a few consumer-
013            grade graphics cards is key to democratizing LLM access. However, existing
014
               frameworks often depend on manual tuning of memory management settings,
               leading to inefficient hardware utilization and suboptimal performance. This paper
015
               introduces ProTrain, a novel training system that automatically tailors memory
016
               management policies to the model architecture and underlying hardware resources,
017            eliminating the need for manual intervention. ProTrain features (1) automated
018            memory management that abstracts complex memory management strategies into
019            a few tunable configuration parameters and searches for optimal parameter settings
020            using cost models and (2) a runtime profiler that provides precise estimates of
021            latency, memory usage, and I/O bandwidth to build high-fidelity cost models.
022            ProTrain does not change the training algorithm and thus does not compromise
023            accuracy. Experiments show that ProTrain improves training throughput by 1.43×
024            to 2.71× compared to the state-of-the-art training systems.

028   1    I NTRODUCTION
029
030
031   Large Language Models (LLMs) have recently achieved remarkable success in various fields. Inspired
032   by the scaling law Kaplan et al. (2020) that the performance (e.g., perplexity) of LLMs often improves
033
      logarithmically with the number of parameters, there has been a trend towards increasing parameter
      size. For instance, the parameter size of GPT-like models has surged from 117 million in GPT-
034
      1 Han et al. (2021) to 1,760 billion in GPT-4 Achiam et al. (2023), a 15,000-fold increase over two
035
      years. The significant growth in parameter size leads to a substantial increase in memory demands.
036   According to existing studies Ren et al. (2021), each unit increase in parameters generally requires
037   16× more memory to store the model states (e.g., fp16 and fp32 parameters, fp16 gradients, fp32
038   momentum and variances), not to mention the increased memory demand for activations due to larger
039   model sizes. Consequently, memory has become the dominant bottleneck in LLM training.
040
      Numerous memory management strategies have been proposed to address memory limitations. They
041
      generally fall into three categories: ZeRO, gradient checkpointing, and swapping. (a) The Zero
042   Redundancy Optimizer (ZeRO) Rajbhandari et al. (2020); Zhao et al. (2023b) distributes model states
043   across multiple GPUs, leverageing aggregated memory capacity to accommodate large models in
044   data parallelism. (b) Gradient checkpointing Chen et al. (2016); Jain et al. (2020); Herrmann et al.
045   (2019); Zhao et al. (2023a); Korthikanti et al. (2023) reduces memory consumption by discarding
046   certain activations during the forward pass and recomputing them during the backward pass. (c)
047   Swapping Rhu et al. (2016); Wang et al. (2018); Le et al. (2018); Huang et al. (2020); Ren et al.
048   (2021); Rajbhandari et al. (2021); Sun et al. (2022) offloads data to external memory sources such as
049   CPU memory or NVMe devices. As we consider swapping to CPU memory, we use swapping and
050
      CPU offloading interchangeably.
051   The three memory management strategies can be implemented within various model training
052   paradigms, including data parallelism Ren et al. (2021), tensor parallelism Shoeybi et al. (2019),
053   and pipeline parallelism Huang et al. (2019); Narayanan et al. (2019). This paper focuses on data
      parallelism, as it is widely used in distributed environments due to its simplicity and scalability.

054
      Popular data-parallel frameworks, such as DeepSpeed Rasley et al. (2020), Colossal-AI Li et al.
055   (2023)1 , and FSDP Zhao et al. (2023b), incorporate the aforementioned memory management
056   strategies. However, these frameworks share a common issue: they demand significant manual
057   effort to configure memory management settings. For example, in DeepSpeed, users must select the
058   appropriate ZeRO optimization stage (e.g., ZeRO-1, ZeRO-2, ZeRO-3), configure offloading options
059   (CPU or NVMe) for both parameters and optimizer states, and set various thresholds for parameter
060   fetching and collective communications. Similarly, while Colossal-AI dynamically manages memory
061   by moving data between the CPU and GPU, users must specify the non-model data ratio. The
062   optimal configuration varies across models and hardware, requiring substantial domain expertise.
063
      Misconfiguration of these settings can lead to reduced efficiency or out-of-memory (OOM) error. For
      instance, GPT-10B running on four RTX 3090 GPUs with the default configuration utilizes only 35.6%
064
      of GPU memory and runs 1.18× slower than with optimized settings. Moreover, configurations
065
      optimized for A100 GPUs cannot be directly applied to RTX 3090 GPUs due to high OOM risks.
066
067   To address this challenge, we propose ProTrain, an efficient LLM training system that automatically
068   identifies memory management policies tailored to the specific LLM architecture and available
069
      memory resources. The basic idea of ProTrain is to abstract memory management strategies into a
      few tunable configuration parameters. ProTrain then builds runtime and memory usage estimators that
070
      quantify the impacts of these configuration parameters on training performance. These cost models,
071
      informed with accurate profiling information on latency, memory, and I/O bandwidth, allow ProTrain
072   to search for the optimal memory management strategy that minimizes runtime while ensuring the
073   peak memory consumption meets the hardware constraints.
074
075
      Our main contributions are:
076              • Automatic Memory Management – To manage model states, we propose a dual-chunk system
077                that treats initial layers of the LLM as persistent chunks in GPU memory and efficiently
078                prefetches or offloads later layers as non-persistent chunks in CPU memory. For activation
079                management, we introduce an interleaved organization that alternates between swapping
080                and gradient checkpointing for each transformer block of the LLM. These strategies are
081
                   abstracted into tunable configuration parameters, creating a structured configuration space
                   that enables precise estimation of runtime and memory usage and facilitating the automatic
082
                   search for optimal configurations with cost models.
083
                 • Memory-Aware Runtime Profiling – We are the first to apply model-wise runtime profiling
084                to LLMs, leveraging detailed memory usage characteristics to reduce overall memory
085                consumption. Building on this, we propose a novel memory-aware profiling method that
086                effectively captures the memory consumption from temporary tensors often overlooked by
087                state-of-the-art approaches, providing precise memory usage estimation to guide automated
088                memory management.
089              • Implementation of ProTrain – We implement these techniques into a training system
090                ProTrain that automatically configures memory management strategies, including CPU
091                offloading, gradient checkpointing, and ZeRO techniques, to optimize training throughput
092
                   while adhering to memory constraints.
                 • Evaluation – We ran ProTrain and other popular training frameworks (e.g., DeepSpeed,
093
                   Colossal-AI, FSDP) on various models such as GPT-2, OPT, Mistral, and LLaMA. On RTX
094
                   3090 GPUs, ProTrain trained models up to 2.47× larger than DeepSpeed and 1.48× larger
095                than Colossal-AI. On A100 GPUs, ProTrain trained models up to 7.5× larger than FSDP,
096                with 1.43× to 2.71× higher throughput than other frameworks. ProTrain also demonstrated
097                excellent scalability with increasing GPUs or batch sizes. These results highlight ProTrain’s
098                superior memory management and efficiency across different hardware setups, making it an
099                excellent choice for LLM training on memory-constrained settings.

124   Training deep learning models involves a repetitive three-stage process across multiple iterations
125   and epochs. The stages include forward propagation (FWD), where a batch of training samples is
126   passed to the model to compute the loss; backward propagation (BWD), which calculates gradients by
127
      backpropagating the loss through the model; and parameter updating (OPTIM), where the gradients
      are used to update model parameters via an optimizer. For the training of large models, it is a common
128
      practice to adopt mixed-precision training Micikevicius et al. (2017), which uses reduced precision
129
      data types for FWD and BWD, while maintaining higher precision for OPTIM to ensure accuracy.
130
131   Memory consumption during training primarily comes from two sources: model states and residual
132   states. Model states include parameters, gradients, and optimizer states (i.e. momentum and variances
133
      used in Adam Kingma & Ba (2014)) while residual states consist of activations and temporary tensors.
      The computational complexity of the FWD and BWD stages scales with model size and batch size,
134
      necessitating their execution on GPUs due to the intensive computational demands. In contrast, the
135
      OPTIM stage involves simpler operations and can be efficiently offloaded to the CPU Ren et al.
136   (2021), which brings significant GPU memory savings by allocating memory-intensive optimizer
137   states on the CPU.

140   3     OVERVIEW OF P ROT RAIN
141
142   Figure 1 illustrates the system architecture of ProTrain, consisting of three core components: (1) an
143   Automatic Memory Management module (§ 3.1) that automatically identifies the optimal memory
144   management policy for training the target LLM on the given hardware, (2) a Memory-Aware Runtime
145   Profiler (§ 3.2) that gathers runtime and memory data to guide memory management decisions, (3) a
146
      High-Performance Training Engine (§ 3.3) that implements the memory management policy. Before
      diving into each subsection, we first elaborate an example of memory management policies.
147
148   Running Example. The training engine diagram illustrates a memory management policy that
149   Automatic Memory Management would discover. In the example, the LLM architecture is divided
150   into three chunks, where each chunk represents a few consecutive transformer blocks. GPU performs
151   the FWD, BWD, and a portion of OPTIM computations while CPU performs the rest of the OPTM
152
      computations. Since the parameters of Chunk 0 will be used immediately at the start of a training
      iteration, they are persistently allocated on the GPU. The parameters for Chunk 1 and Chunk 2 reside
153
      on the CPU and are dynamically uploaded to the GPU or offloaded back to the CPU to ensure the
154
      total memory consumption meets the device memory limit. The flow of communication operations
155   between the CPU and GPUs is as follows:
156
157   (A) Parameter Upload: Before the forward pass, the parameters for Chunk 1 are uploaded from the
158
      CPU to the GPU. Since Chunk 0 already resides on the GPU, only Chunk 1 and Chunk 2 need to be
      uploaded sequentially from the CPU, illustrated as blocks 1 and 2 in the row “CPU → GPU”. The
159
      prefetch of the next parameter chunk begins as soon as the GPU starts computing the current chunk.
160
161   (B) Parameter Gather: Once the parameter chunks are uploaded, the engine performs an all-gather
      operation that collects the parameter shards from all GPUs into a complete parameter chunk for


                                                                   3
      Under review as a conference paper at ICLR 2025


162
      upcoming computations. This step is required for all three chunks, illustrated as blocks 0, 1, and 2 in
163   the top row of “GPU → GPU”.
164
165   (C) Gradient Reduce: In the backward pass, the engine reuses the parameter chunk to store the
166
      computed gradient to optimize memory usage. Once all parameters within a chunk are replaced by
      their corresponding parameters, a reduce-scatter operation is performed to synchronize gradients
167
      across GPUs, illustrated as blocks 0, 1, and 2 in the bottom row of “GPU → GPU”.
168
169   (D) Gradient Offload: Following the gradient reduce, the chunks that were originally on the CPU, are
170   offloaded back to the CPU to free up GPU memory. Only Chunk 1 and Chunk 2 perform this step.
171   (E) Parameter Update: Once on the CPU, the gradients for Chunk 1 and Chunk 2 are used for
172   parameter updates, along with the high-precision parameter chunk already resided on the CPU. This
173   step runs in parallel with the GPU’s backward execution. In contrast, Chunk 0 performs its parameter
174   updates directly on the GPU.
175
      (F) Activation Swapping Out: Activation swapping occurs at the transformer block level, which is
176   more fine-grained than chunks. In the example, only activations from the first transformer block
177   (denoted as four block 3 in the row “GPU → CPU”) are swapped out after each activation is computed.
178
179
      (G) Activation Swapping In: When sufficient GPU memory is available to hold a transformer block’s
      activations, the swapping in begins. This is done in batches (denoted as two block 3 in the row “CPU
180
      → GPU”) rather than individually as swapping out shows, grouping multiple activations to improve
181
      bandwidth utilization.
182
183   In this example, parameter uploads from the CPU only occur during the forward pass assuming
184   the GPU has enough buffer capacity to hold all the parameter chunks. However, if the buffers
185
      become full, the least recently used chunk is evicted, triggering another parameter upload and gather
      operation during the backward pass. Throughout the process, communication overhead is minimized
186
      by overlapping data transfers with computations. Additionally, idle CPU cycles are used to perform
187
      parameter updates, which run concurrently with the GPU’s backward computations, effectively hiding
188   slower CPU parameter update operations.

      3.1     AUTOMATIC M EMORY M ANAGEMENT
191
192   The Automatic Memory Management module abstracts the memory management policy into a few
193   configuration parameters and automatically tunes these parameters to optimize the training efficiency
194   of a LLM on a target hardware. We next elaborate on the abstractions of the configuration space and
195   the optimal configuration search algorithm.
196
197   3.1.1    T HE C ONFIGURATION S PACE OF M EMORY M ANAGEMENT
198
199   Configuration Parameters for Model States. Model states can be offloaded to the CPU to relieve
200   GPU memory pressure but the offloading implementations in existing LLM training frameworks
201
      suffer from various limitations. Fully offloading all parameters, as seen in FSDP Zhao et al. (2023b),
      often leads to inefficient GPU memory usage and high data transfer overhead. DeepSpeed Rasley
202
      et al. (2020) attempts to mitigate this issue by using thresholds, such as maximum live parameters
203
      and prefetch bucket size, to control the offloading ratio. However, its prefetching mechanism operates
204   in a sliding window manner due to poorly timed execution, resulting in frequent small transfers.
205   This causes low bandwidth utilization, significantly degrading performance. Colossal-AI Li et al.
206   (2023) improves bandwidth utilization through fixed-sized chunks but suffers from frequent memory
207   reallocations caused by dynamic chunk management. Moreover, it uploads high-precision parameter
208   chunks for GPU parameter updates at runtime, increasing the risk of memory fragmentation and
209   out-of-memory (OOM) errors.
210   To address these limitations, ProTrain introduces a dual-chunk system consisting of persistent
211   and non-persistent chunks. Persistent chunks remain on the GPU, storing both high-precision and
212   low-precision parameters, which eliminates data transfers and enables direct GPU parameter updates.
213   In contrast, non-persistent chunks are kept in CPU memory, requiring low-precision parameters
214   uploads to the GPU for computation, and gradients offloads back to the CPU for parameter updates.
215   For non-persistent chunks, ProTrain further introduces pre-allocated chunk buffers that are used as
      caches. These buffers allow parameters loaded during the forward pass to be reused in the backward

232   The concept of dual-chunk system allows ProTrain to tailor offloading policies to chunks of different
233   characteristics. Dual-chunk system is inspired by two observations: (1) The forward pass computation
234   can often hide the overhead of CPU offloading for the later layers of a LLM but not that of the first
235   few layers. (2) The parameter updates for the later layers of a LLM, but not the first few layers, can
236   be performed concurrently with the backward pass computation. Therefore, the model states of the
237
      first few layers and the later layers should be managed differently.
238   We use the example in Figure 1 to explain the rationale. As chunk 0 executes first in the forward
239   pass, if its parameters are offloaded to the CPU, its forward pass computation will be blocked by
240   the data transfer overhead from CPU-to-GPU parameter uploading and parameter gather. Chunk 0
241   also gets updated the last with no backward pass computation left to hide the latency from parameter
242   updates. Therefore, managing Chunk 0 as a persistent chunk eliminates cold start latency and enables
243
      efficient GPU parameter updates. In contrast, the data transfer necessary for Chunk 1 and Chunk 2 to
      perform forward pass can be overlapped with the computation of Chunk 0 and Chunk 1 respectively.
244
      If performed on the CPU, the parameter updates of Chunk 1 and Chunk 2 can also be overlapped
245
      with the backward pass computation of Chunk 0 and Chunk 1 respectively. Therefore, managing
246   Chunk 1 and Chunk 2 as non-persistent chunks relieves GPU memory pressure without incurring
247   offloading overheads.

249
      We summarize the configuration parameters from managing model states as follows: (1) chunk size
      – the size of each chunk for the LLM, (2) the number of persistent chunks, and (3) the size of
250
      pre-allocated chunk buffers. In particular, while more persistent chunks and chunk buffers generally
251
      improve performance, memory constraints and the large size of LLMs necessitate a trade-off between
252   memory usage and system efficiency.
253
254   Configuration Parameters for Activations. Previous studies Peng et al. (2020); Beaumont et al.
255
      (2021) have co-optimized activation swapping and gradient checkpointing at the tensor granularity.
      Although tensor-level management offers greater flexibility, it significantly expands the search space,
256
      making it challenging to determine optimal policies for swapping or recomputing individual tensors.
257
      For instance, the LLaMA 34B model has only 48 transformer blocks but has approximately 2,000
258   activation tensors, resulting in a search space as large as 32000 if each tensor has three options.
259   Moreover, managing tensors individually introduces implementation complexities and scalability
260   challenges, making this approach impractical for LLMs. In contrast, popular training frameworks
261   that utilize gradient checkpointing often recompute all transformer blocks, which is inefficient when
262   there is sufficient memory to avoid full recomputation.
263
      To address the above limitations, ProTrain takes a different approach by managing activation swapping
264   and gradient checkpointing operations at the transformer block level. Each block can utilize one
265   of three techniques in handling activations: swapping, gradient checkpointing, or no optimization
266   (i.e., neither swapping nor checkpointing is applied). To enhance efficiency, ProTrain introduces
267   an interleaved organization, in which each swapping block is followed by multiple blocks using
268   gradient checkpointing. This design offers several benefits. First, placing swapping blocks earlier
269   increases opportunities for overlapping swapping with computation. Second, interleaving them with
      checkpointing blocks prevents activation accumulation, reducing the risk of OOM errors caused by


                                                           5
      Under review as a conference paper at ICLR 2025


270
      slower swapping. Third, placing unoptimized blocks in the later layers allows their activations to be
271   consumed sooner, enabling earlier activation prefetching of swapping blocks.
272
273   Figure 2 illustrates our approach using a transformer with 8 blocks. Block 1 and 4 use swapping, while
274
      block 2, 3, 5, and 6 use gradient checkpointing. The remaining blocks are left unoptimized, as their
      earlier backward computations offer limited opportunities for swapping. This interleaved approach
275
      not only maximizes the overlap between computation and communication, but also minimizes peak
276
      memory usage, as visualized in the upper part of Figure 2.
277
278   We summarize the configurable parameters from managing activations as follows: (1) the swapping
279   interval, which is selected based on the computation time needed to swap out a block, (2) the number
280
      of blocks designated for swapping and gradient checkpointing. Striking a balance between the
      number of swapping and checkpointing blocks is crucial: ideally, fewer blocks should use either
281
      technique, as each introduces additional recomputation or transfer overhead. However, when memory
282
      is constrained, swapping is preferred for blocks where communication overhead can be effectively
283   hidden.

285   3.1.2    O PTIMAL C ONFIGURATION S EARCH WITH C OST M ODELS
286
287   We formulate the optimal configuration search as a constrained optimization problem. The goal is
288   to minimize the total runtime of the training process. Since training consists of repeated iterations,
289   minimizing the total training time is equivalent to minimizing the runtime of a single iteration,
290
      denoted as TIteration , which includes the forward pass, backward pass, and parameter updates:
291                                     min TIteration   s.t. MPeak < MCapacity ,                                 (1)
                                        configs
292
293
      where MPeak represents the peak memory usage, and MCapacity is the total GPU memory capacity.
      The set of tunable configuration parameters, configs, that determines the memory management policy
294
      is configs = {npersist , nbuffer , nswap , ncheckpoint }, where npersist denotes the number of persistent chunks
295
      residing on the GPU, nbuffer refers to the number of chunk buffers for prefetching and memory reuse,
296   nswap indicates the number of blocks using activation swapping, and ncheckpoint specifies the blocks
297   applying gradient checkpointing. These configurations are non-negative integers that are bounded
298   by the total number of chunks (Nchunk ) or blocks (Nblock ). Chunk size is determined independently
299   before the optimal configuration search (detailed in Appendix B.1).
300
      To solve the optimization problem, we build two cost models that accurately estimate runtime and
301
      peak memory consumption for each configuration combination. These cost models allow us to
302   identify the optimal configuration setting leveraging profiling information data alone, getting rid of
303   the tedious trial-and-errors to set up training processes. The profiler is discussed in § 3.2.
304
305
      Runtime Estimator. In ProTrain, CPU parameter updates are executed concurrently with the GPU’s
      computations, which include both the backward pass and GPU-based parameter updates. However, if
306
      the CPU parameter updates cannot fully overlap with the GPU’s operations, the total iteration time
307
      becomes constrained by the longer CPU update phase. The runtime cost model is formulated as:
308
309
                            TIteration = TFWD + max{TBWD + TGPU OPTIM , TCPU OPTIM },                             (2)
310   where TFWD and TBWD are modeled as a function of the configuration parameters. For parameter
311   update of the persistent chunks (TGPU OPTIM ) and non-persistent chunks (TCPU OPTIM ), ProTrain models
312   runtimes predictably based on parameter size. Due to space limitations, details are in Appendix A.1.
313   Peak Memory Usage Estimator. Memory usage falls into two categories: static and dynamic
314   components. Static memory, which includes model states and activations, is fixed and predictable.
315   They can be easily determined by chunk size, npersist , and nbuffer . However, dynamic memory involves
316   temporary tensors that are hard to estimate and are often neglected in existing approaches Wang et al.
317   (2024); Huang et al. (2022). Although transient, these temporary tensors can significantly impact peak
318   memory usage, accounting for up to 17.2% (3.06 GB) of total memory. To address this, we design
319   an iterative operator-wise approach to estimate peak memory usage. The basic idea is to track peak
320
      memory during profiling while excluding static memory, then iteratively add back the static memory
      during the estimation phase, operator by operator, to accurately capture the contribution of temporary
321
      tensors to the overall peak memory. Details of the algorithm are given in the Appendix A.2.
322
323   The configuration space in ProTrain is structured and finite, allowing for an exhaustive search
      of all possible configurations. ProTrain employs specific pruning strategies to further reduce the


                                                             6
      Under review as a conference paper at ICLR 2025


324
      search space. For instance, the maximum number of swappable blocks is limited by the swapping
325   interval to ensure they overlap with forward computations. During the backward phase, the system
326   monitors bandwidth usage for chunk prefetching to ensure sufficient bandwidth remains for activation
327   prefetching. Additionally, as configurations are traversed from smallest to largest, any swapping and
328   checkpointing combination that results in memory overflow is immediately discarded, and subsequent
329   iterations involving this combination are skipped. For each viable configuration, ProTrain’s runtime
330   estimator predicts the runtime, selecting the one with the shortest runtime as the final setup.

332   3.2   M EMORY-AWARE RUNTIME P ROFILING
333
334   Traditional memory profiling methods, such as static profiling Patil et al. (2022) and layer-wise
335   runtime profiling Beaumont et al. (2021), are insufficient for capturing the complete memory demands
336
      of LLM training. These approaches often overlook the impact of unhookable operators and temporary
      tensors, leading to inaccurate memory management and suboptimal configuration choices. Model-
337
      wise runtime profiling has the potential to overcome these challenges. However, as it requires the
338
      execution of the entire LLM model, it is constrained by limited GPU memory capacity for LLMs.
339
340   ProTrain develops an memory-aware runtime profiling system that leverages memory usage character-
341   istics to enable model-wise profiling with limited memory capacity. Specifically, ProTrain drops static
342
      memory (e.g., parameters, gradients, activations) from the GPU and regenerates it when required.
      This is based on the observation that static memory usage is predictable (as detailed in Section 3.1.2),
343
      allowing the profiler to focus on capturing the more complex and transient dynamic memory usage.
344
345   To track dynamic memory fluctuations caused by temporary tensors and unhookable operators,
346   ProTrain registers hooks that monitor current and peak memory changes both before and during
347   operations. First, the peak memory usage during each operation is monitored to capture the tem-
348
      porary tensor usage specific to that operation. Second, by analyzing the memory changes between
      consecutive hookable operations, the profiler infers the memory usage of unhookable operators.
349
      This operator-wise approach considers the life cycle of various tensors, enabling a more precise
350
      understanding of memory usage dynamics and making the profiler memory-aware, which is crucial
351   for building accurate cost models.
352
353   Our profiler also tracks the execution time of each operator. Similar to memory profiling, we estimate
354
      the execution times of unhookable operators by analyzing the intervals between hookable ones.
      Additionally, the profiler collects detailed hardware metrics, including memory transfer bandwidth
355
      and collective communication operation durations, under both isolated and overlapping scenarios.
356
      This detailed data collection enables precise performance predictions and facilitates automatic
357   memory management tailored to specific models and hardware, as discussed in Appendix A.

519   ProTrain is designed for small clusters, which may pose challenges in large-scale training where cross-

521   where we trained a 15B GPT-2 model with a batch size of 160 across two nodes (each equipped
522   with four V100 GPUs), ProTrain showed promising results, outperforming DeepSpeed by 1.53×
523   and Colossal-AI by 1.84×, while FSDP encountered OOM errors. Notably, these results were
524   achieved without any dedicated optimizations for multi-node environments in ProTrain, highlighting
525
      the potential for further refinement and performance improvements.
526   Furthermore, ProTrain’s ability to independently profile each node makes it well-suited for adapting
527   to heterogeneous setups, opening up opportunities to explore optimizations across diverse hardware
528   configurations. As future work, we aim to enhance ProTrain’s performance in large-scale, multi-node
529   environments by leveraging these optimization opportunities.

533
      This paper introduced ProTrain, a novel training system designed to simplify the training process
534   through automatic memory management. ProTrain highlights the significance of precise memory
535   usage and runtime data gathered through memory-aware, model-wise profiling to build high-fidelity
536   cost models, along with the careful abstraction of configuration parameters from memory management
537   strategies to automate optimal configuration search. ProTrain achieves up to 5× the performance
538   of existing state-of-the-art frameworks and enables the training of models with up to 75 billion
539   parameters on a single A100 GPU. We hope our work helps AI researchers and practitioners with
      limited GPU resources, making LLMs more accessible to a wider audience.

869
870   Proactive Memory Allocation ProTrain preallocates memory for tensors that persist until training
871   completes, including early allocation of persistent chunks for parameters and optimizer states, as
872   well as GPU chunk buffers. This proactive strategy reduces the number of memory allocations and
873   mitigates fragmentation by grouping long-lived tensors together, ensuring a more organized and
874   efficient memory layout.
875
876
      Single-Stream Memory Allocation ProTrain unifies memory allocations within the default stream
      to improve memory utilization. PyTorch’s allocator adopts a multi-heap design where each stream has
877
      its own heap, limiting cross-heap memory reuse and necessitating the use of record stream()
878
      to ensure correctness. By using a single stream for all allocations and directly managing deallocation
879   synchronization ourselves, we effectively prevent misuse and reallocation conflicts, thereby improving
880   memory efficiency.
881
882   Customized Pinned Memory Allocator We observe that the default pinned memory allocator
883   (CUDAHostAllocator) often over-allocates by rounding up to the nearest power of two, leading
884   to significant memory waste. To address this inefficiency, ProTrain developed a customized pinned
885   memory allocator that leverages insights from automatic memory management to precisely determine
886   pinned memory requirements, providing finer control and avoiding the excessive memory reservation
887   of the default allocator.

1137   Swapping and Recomputation Swapping Rhu et al. (2016); Le et al. (2018); Huang et al. (2020);
1138   Ren et al. (2021); Sun et al. (2022) is a commonly employed technique which leverages external
1139
       memory such as CPU memory to offload tensors, thereby expanding the available memory for
       training. Traditional swapping methods mainly focus on offloading activations, SwapAdvisor Huang
1140
       et al. (2020) extends it to parameters and ZeRO-offload Ren et al. (2021) further extends it to
1141
       optimizer states. Recomputation Chen et al. (2016); Jain et al. (2020); Herrmann et al. (2019); Zhao
1142   et al. (2023a); Korthikanti et al. (2023), also known as gradient checkpointing, is another widely
1143   used technique that trades additional recompute time during backward pass for reduced memory
1144   usage of activations. Initially, Chen et al. Chen et al. (2016) focuses on homogeneous sequential
1145   networks, and subsequent studies Jain et al. (2020); Herrmann et al. (2019) extended its applicability
1146   to heterogeneous networks. Considering the scale and complexity of Transformers, which often
1147   contain numerous layers, previous approaches become less efficient. Therefore, Rockmate Zhao
1148   et al. (2023a) optimizes the plan generation by partitioning models into fine-grained blocks. NVIDIA
1149   further proposes selective activation recomputation which checkpoints and recomputes parts of
1150
       layers Korthikanti et al. (2023). To get the best of both worlds, some works Peng et al. (2020);
       Beaumont et al. (2021); Nie et al. (2022) jointly optimize swapping and recomputation, whereas
1151
       ProTrain differentiates itself by tailoring to fit the specific structure of transformers.
1152
1153   ZeRO Techniques. ProTrain adopts ZeRO to manage model states. The Zero Redundancy Optimizer
1154   (ZeRO) Rajbhandari et al. (2020) distributes model states across multiple GPUs to reduce memory
1155   pressure of each GPU. ZeRO operates in three stages: ZeRO-1 partitions optimizer states across
1156
       GPUs; ZeRO-2 extends this by also distributing gradients; and ZeRO-3 further divides the parameters,
       which are required to be gathered before forward/backward computation. The ZeRO techniques
1157
       have been integrated into state-of-the-art frameworks such as DeepSpeed Rasley et al. (2020),
1158
       FSDP Zhao et al. (2023b), and Colossal-AI Li et al. (2023), each differing in their parameter
1159   organization to optimize bandwidth utilization. Unlike DeepSpeed and FSDP, which require manual
1160   configuration for parameter grouping, Colossal-AI automatically groups parameters into chunks and
1161   dynamically adjusts their size according to the model’s scale. This chunk-based method, inspired by
1162   PatrickStar Fang et al. (2022), is also adopted in ProTrain.
1163
1164
1165   GPU Memory Management Deep learning frameworks, such as PyTorch Paszke et al. (2019)
1166   and TensorFlow Abadi et al. (2016), utilize caching allocators for efficient memory management.
1167   However, these frameworks often face memory fragmentation issues, particularly when integrating
1168
       memory-saving techniques like swapping, recomputation, and parallelization, which hurts allocation
       efficiency. To address this, two main approaches have been proposed. The first is profiling-guided
1169
       optimization Sekiyama et al. (2018); Steiner et al. (2022; 2023), which leverages the repetitive and
1170
       predictable nature of memory allocation patterns during training. This method traces and analyzes
1171   tensor allocations and deallocations to optimize tensor placement, thus improving memory efficiency.
1172   Alternatively, GMLake Guo et al. (2024) introduces Virtual Memory Stitching, a technique that
1173   merges non-contiguous memory blocks, thereby reducing memory fragmentation at the operating
1174   system level. These approaches are orthogonal to ProTrain’s method. Angel-PTM Nie et al. (2023)
1175   adopts a page-based memory management strategy that partitions model states to reduce the memory
1176   fragmentation. In contrast, ProTrain designs a new chunk-based memory management inspired by
1177   PatrickStar Fang et al. (2022) grouping model states into chunks that align with the runtime execution
1178   order, which not only improves bandwidth utilization but also enhances memory locality.

       Overlapping Computation and Communication There are numerous work on overlapping com-
1181
       putation and communication, with many studies Mahajan et al. (2023); Hashemi et al. (2019); Peng
1182   et al. (2019); Jangda et al. (2022); Chen et al. (2024) focus on substituting, splitting, and scheduling
1183   complex operators to achieve fine-grained overlapping. CoCoNet Jangda et al. (2022) enhances
1184   lower-level operator optimization, while Centauri Chen et al. (2024) extends this to graph-level
1185   scheduling, offering a more hierarchical abstraction. Despite these advances, most research focuses
1186   on the optimization of collective communication operations in distributed cases. However, ProTrain
1187   also considers the communication between CPU and GPU under limited GPU memory conditions,
       making it orthogonal to existing research.
