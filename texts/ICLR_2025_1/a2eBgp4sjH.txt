009                                                   A BSTRACT
010
011                We study nearest neighbor search with filter constraints (MultiFilterANN): given a
012                query vector with a discrete set of labels S, retrieve the (approximately) closest
013                vector from a dataset under the constraint that S must be a subset of the labels
014                of the retrieved vector. There has been a burgeoning interest in this problem on
015                the practical side, due to its strong motivation from search and recommendation
016                applications where vector labels correspond to real world attributes such as date,
017
                   price, or color. On the theoretical side, this problem generalizes the subset query
                   problem, which asks us to only determine if S is a subset of some set in the dataset,
018
                   without retrieving the closest vector.
019
                   In this work, we present a systematic study of MultiFilterANN,. Theoretically, we
020
                   demonstrate the power of graph-based algorithms in two ways:
021
022                    • We design provable algorithms with the best known space-time tradeoffs for
023
                         MultiFilterANN in the large filter regime by carefully incorporating ANN
                         algorithms into known subset query algorithms.
024
025
                       • We demonstrate lower bounds for popular algorithms for MultiFilterANN,
                         showing that they can catastrophically fail even on simple data/label sets.
026
027                Our theoretical results inspire our empirical approach, where we extend practical
028                graph indices for standard nearest neighbor search to MultiFilterANN by augment-
029
                   ing the (greedy) search procedure with a penalized distance function that captures
                   filter constraints. Our empirical algorithm is competitive with existing state of
030
                   the art solutions which are tailored for one or two filters, while also seamlessly
031
                   generalizing to any number of filters without any modifications. Lastly we release
032                multiple novel datasets for MultiFilterANN, filling in a noticeable gap in literature.

035   1       I NTRODUCTION
036
037   Driven by advances in machine learning, the capabilities of embeddings to faithfully represent
038   semantic relationships between real world objects such as video, images, and text via geometric
039   distances has dramatically increased. This has prompted the study of new algorithmic questions
040   centered around the analysis of large scale vector data. A prominent example is the nearest neighbor
041   problem with filter constraints (MultiFilterANN): we are given a dataset of n high-dimensional
042
      vectors X where every vector v ∈ X has a set of associated labels Sv ⊆ [m]1 . We wish to design a
      data structure, which given a query vector q with its own set of labels Sq ⊆ [m] and a target k ∈ N,
043
      outputs the k (approximately) closest vectors to q in X, subject to the constraint that Sq must be a
044
      subset of the label set Sv of each of the vectors v returned.
045
046   The problem is well motivated by practical constraints in search and recommendation. It captures the
047   realistic scenario of combining vector search with keyword matches. For example in image search,
048
      we might only be interested in images shot in Paris. In product recommendations, a shopper may
      only wish to view items with certain specifications of size, make, and model. Likewise for text data,
049
      we may only want documents that contain a set of given keywords to be retrieved. In general, hybrid
050
      keyword semantic search is a fundamental task in many modern systems. Leading vector search
051   companies such as Weaviate Weaviate (2024) and Pinecone Pinecone (2024) offer “hybrid search”
052   as a service, with the motivating example being a combination of hard matches based on keywords
053

      and a vector distance based on the dense LLM-based embeddings. Our formulation models the case
055   where the returned data points must satisfy all the labels of the query.
056
057   Beyond its practical appeal, the problem is also rich theoretically: it generalizes the classic subset
058
      query problem from the algorithms literature, where we wish to quickly determine if a given set
      Sq ⊂ [m] is a subset of a collection of n subsets of [n] (i.e., MultiFilterANN without vector search).
059
060   Many recent empirical solutions have been proposed for MultiFilterANN, with most work focusing
061   on the single-filter regime and more recently, |Sq | = 2. These works demonstrate that many existing
062   algorithms for the vanilla nearest neighbor search problem can be extended to handle the case of few
063   filters. Nevertheless, while tremendous practical progress has been made, substantial gaps remain in
064
      our understanding of MultiFilterANN, captured by the following questions:
065
      • On the empirical side, can we design algorithms that scale as the numbers of constraints, i.e. m
066
        and |Sq |, grow? What are the limits of current approaches deployed for the small-filter case?
067
068   • Likewise, what can we say theoretically for MultiFilterANN? First, do the existing practical
069
        algorithms for the small-filter case have provable guarantees? And secondly, can we obtain
        algorithms for MultiFilterANN whose performance is comparable to the best known algorithms
070
        for the subset query problem? Note that if m is constant, there is a simple naive algorithm with
071
        a 2m factor space overhead which constructs a nearest neighbor data structure corresponding to
072     every subset of [m]. Thus, the problem is especially interesting when m is large.
073
074
      Our Contributions. We initiate a broad and systematic study of the MultiFilterANN problem.
075
076   • On the theoretical side, we give an algorithm with o(2m ) space overhead and o(n) query time (in
077     certain regimes), matching some of the best known space and query time trade-offs for subset query
078     problem, while additionally handling the nearest neighbor search component. Our algorithm is
079     flexible and capable of incorporating graph-based indices.
080   • Bridging theory and practice, we construct simple datasets demonstrating failure modes of popular

106   Our ‘base’ dataset X consists of n points in Rd . In addition, every point x ∈ X also has a set of
107   labels (or filters) Sx ⊆ [m]. In this setting, X is also referred to as a labeled dataset. A query q ∈ Rd
      also has a set of labels Sq ⊆ [m]. We define the Multi-FilterANN problem as follows.

      Definition 2.1 (Multi-FilterANN). We are given a labeled dataset X as above and a parameter
109   C ≥ 1. We want to create a data structure which on any query q, returns a point x ∈ X guaranteeing
110   that Sq ⊆ Sx and ∥x − q∥2 ≤ C · minx′ ∈X,Sq ⊆Sx′ ∥x′ − q∥2 .
111
112   This definition is the analogue of the theoretical work on nearest neighbor search Andoni et al. (2018).
113   Thus we think of datapoints as having a vector component as well as a label component. We always
114   ℓ2 (Euclidean metric) to measure vector distance. In our experiments, we instead report the standard
115
      Recall@k score, which is analogous to the empirical works on standard nearest neighbor search
      without filters. It denotes the fraction of the returned top-k points that are among the true k nearest
116
      neighbors of a query q. Note that while we allow for an approximation in the vector distance, we
117
      always require the subset condition to be satisfied.
118
119   The Multi-FilterANN problem is a strict generalization of the SubsetQuery problem, whose study
120   goes back at least two decades Charikar et al. (2002).
121   Definition 2.2 (SubsetQuery Problem). We are given a collection X of n subsets of [m]. When a
122   query q ⊆ [m] arrives, we must determine if there exists an x ∈ X such that q ⊆ x.
123
      Usually the SubsetQuery problem asks if there exists an x such that x ⊆ q, but the formulation we
124
      state is equivalent by taking complements.
125
126    Using the language of prior works Gollapudi et al. (2023), in Multi-FilterANN we are solving the
127   ‘AND’ case, where all the labels of the query must be present in the data points returned. This is as
128    opposed to the much easier ‘OR’ case, where only one label of the query can be present. The ‘OR’
129
       case can be reduced to multiple invocations of an algorithm for the single-filter case.
130   Lastly, we introduce the doubling dimension of a dataset. It is a well-studied measure of intrinsic
131   dataset dimensionality in the context of nearest neighbor search in theory Gupta et al. (2003);
132   Krauthgamer & Lee (2004); Indyk & Naor (2007); Har-Peled & Kumar (2013). Furthermore in
133   practice, the performance of popular empirical algorithms can be analyzed via the doubling dimension
134   Narayanan et al. (2021); Indyk & Xu (2023), and many real world dataset exhibit small intrinsic
135
      dimensionality Aumüller & Ceccarello (2019).
136   We define it for the vector component of a dataset. For any x ∈ X, radius r > 0, we use B(p, r) to
137   denote the ball of radius r centered at p, i.e. B(p, r) = {q ∈ X : ∥p − q∥ ≤ r}.
138   Definition 2.3 (Doubling Dimension). A vector data set X has doubling dimension λ if for any
139   p ∈ X and radius r > 0, X ∩ B(p, 2r) can be covered by at most 2λ balls of radius r.

172   4    T HEORETICAL A NALYSIS
173
174   We present an algorithm with strong theoretical (worst-case) guarantees for the Multi-FilterANN
175   problem. We additionally take inspiration from our theoretical study to design our final empirical
176   algorithm of Section 6. This is elaborated at the end of the section.
177   The starting point of our theoretical algorithm is to note that any Multi-FilterANN data structure
178   must be able to solve the SubsetQuery problem. Thus, we first review a data structure from Charikar,
179   Indyk and Panigrahy, denoted as CIP, for the SubsetQuery problem Charikar et al. (2002). The CIP
180   data structure is among the best known when m = O(poly(log n)). Note that as stated, CIP cannot
181   handle the full generality of the Multi-FilterANN problem.
182
183   Summary of their approach: The CIP data structure partitions the input dataset (of the Subset-
184   Query problem) into multiple levels, thought of as a nested table. All levels, except the final one, hash
185   a given query to only one entry of the table. This last level is a collection of disjoint sets X = ∪i Xi
186   of the data points. The CIP data structure guarantees that (with high probability), for all Xi , either
187   all datapoints in Xi are valid responses to the query (i.e., the query is a subset of all the datapoints
188   in Xi ), or none are. However, the data structure needs to evaluate this for every group. Fortunately,
189
      Charikar et al. (2002) guarantees that we can construct the data structure so that only o(n) groups are
      at the last level, so only o(n) checks need to be made. A detailed summary of the CIP data structure
190
      and its formal guarantees are given in Appendix B.1 (see Theorem B.1).
191
192
193
      Adapting from SubsetQuery to Multi-FilterANN Our key idea is to replace each of the final
      groups in CIP with a nearest-neighbor data structure. If a group (in the final layer of the CIP data
194
      structure) is valid for the query, then we use a vanilla nearest-neighbor data structure to find a close
195
      vector to the query’s vector. Note that this takes care of both the SubsetQuery requirement, as well
196   as the vector search component, i.e., is applicable to the Multi-FilterANN problem. For theoretical
197   guarantees, we may use Locality Sensitive Hashing Indyk & Motwani (1998a) in the final level of
198   the CIP data structure or the DiskANN graph. It implies the following theorem, with a formal proof
199   deferred to Appendix B.

232
233   We now additionally motivate our final empirical algorithm by presenting failure modes of prior ap-
234   proaches for the one or two filters regime. By demonstrating simple instances where such approaches
235   catastrophically fail, we gain valuable insights towards our final algorithm.
236
237   ParlayIV F 2 Counterexample. We start with ParlayIV F 2 , the winner of the recent ANN Bench-
238   mark challenge on nearest neighbor search with 2 filters (queries have at most 2 filters) Landrum
239   et al. (2024). As a summary, the ParlayIV F 2 algorithm essentially constructs a clustering index
240   for each label. When a query arrives, for each label in the query, it evaluates the distance of each
241   cluster to the query, sorting the clusters in ascending order of distance. It keeps adding all the points
242
      from the nearest remaining cluster into a queue until the size of the queue is at least a pre-determined
      parameter. Then, it takes the intersection of these queues and evaluates the distances. In some simple
243
      instances, we can prove that in order to find the nearest point satisfying both the labels, ParlayIV F 2
244
      must add almost all the points into the queue. The corresponding intersections of sets containing
245   O(n) elements can make ParlayIV F 2 take linear time if we want to obtain any non-zero recall.
246
      Lemma 5.1. There exists a size-n one-dimensional dataset with two total labels such that the
247
      ParlayIV F 2 algorithm has query time Ω(n).
248
249   This highlights that popular prior approaches for our problem, which use clustering based methods,
250   are too ‘local.’ In the simple one dimensional aforementioned example, clustering based methods
251   spend too much time focusing on the ‘distance’ part of the problem and ignore the subset match
252   constraint. This forces them to iterate over many irrelevant points in the local vicinity of the query.
253
      In contrast, that graph-based indices avoid this problem by having long-range edges, which in this
254
      instance, would allow us to quickly navigate to the end of the one dimensional dataset, without
255   performing a linear scan through the number line. This prompts the following lesson.
256
257   Lesson 3: Clustering indices alone are not enough for Multi-FilterANN. Long-range connections
258   afforded by graph indices are needed.

      We now discuss two prior graph-based approaches: FilterDiskANN Gollapudi et al. (2023) and NHQ

      Wang et al. (2022; 2024). While they are both graph-based, we give provable hard instances for these

      FilterDiskANN Counterexample. The paper Gollapudi et al. (2023) was the first to empirically
264   study nearest neighbor search with filter constraints, and they focused on the 1-filter regime (queries
265   have 1 label only). At a high level, they construct a graph index (based on the vanilla DiskANN
266   algorithm), and perform the following greedy search given a query q with label Sq : traverse the graph
267   only on edges with both endpoint vertices having label Sq . They start the greedy search procedure at
268   a fixed starting vertex containing label Sq . Thus, for every label ℓ ∈ [m], they require the subgraph
269   of all vertices with label ℓ to be connected in their graph index. While this approach works well for
      the m = 1 single-filter case, we show that it cannot generalize to large m.

      6.1   S EARCHING THE G RAPH I NDEX : T HE P ENALTY G RAPH S EARCH
325
326   Most graph-based ANNS algorithms work in the following manner: the index construction involves
327   building a directed graph G with nodes corresponding to the base vectors; the search algorithm
328   for a query vector employs a natural greedy or best-first traversal policy on G, by starting at some
329   designated point s ∈ P , and iteratively hopping to the neighboring vertex whose vector is closest
330   to the query vector until we reach a fixed point. Algorithm 1 with λ = 0 and τ = ∞ gives a more
331   formal definition of this process.
332   However, the drawback of such a search method is that we are completely agnostic to the query filters
333   and base point labels. To overcome this, the FilteredDiskANN algorithm in Gollapudi et al. (2023)
334   only traverses the neighbors that satisfy the query predicate. This change, along with a suitable
335   change during the graph construction phase yielded reasonable results for simple predicates (like
336   query having one filter label). However, as noted in Section 5, this strict approach is doomed to fail
337   for very selective queries.
338   To overcome this deficiency, we note that the NHQ algorithm Wang et al. (2024) performs greedy
339   search on a novel fused distance, which takes into account the distance ∥q − x∥ between the query
340   vector and the base vector x, and the symmetric difference between the label sets Sq and Ss . However,
341   Lemma 5.3 shows that this formulation also cannot capture the nuances of MultiFilterANN. Instead,
342   we use an asymmetric distance on the label sets.
343
344   Asymmetric Penalty Distance. To provide intuition, the symmetric part of the NHQ fusion distance
345
      heavily penalizes data points that have many irrelevant labels, even if they contain all the labels of the
      query. Consider the following toy example: We have a universe of 5 labels, namely {A, B, C, D, E},
346
      and there are are two base vectors u and v, with label sets Su = {A, B, C, D, E} and Sv = {A, C}.
347
      Suppose we want to retrieve the closest vector to a query q with label set Sq = {B}. Clearly, the
348   correct result must return u as it is the only feasible point. However, using a symmetric difference
349   would result in returning v. Indeed, using the symmetric difference between Sq and Sv is essentially
350   using the hamming (or squared euclidean, for that matter) distance between the one-hot encodings of
351   the label sets, where the dimension of these encodings is of size m. Our simple-in-hindsight approach
352   to solve this issue is to only measure the label difference for the labels in q.
353
      This choice has three desirable properties. First, it does not penalize data points in X that have many
354   labels, skirting the lower bound presented in Lemma 5.3. Secondly, the asymmetric label distance can
355   be embedded into ℓ2 : represent the label set of the data points as the {0, 1} valued m-dimensional
356   vector where we have a coordinate of 1 iff the label is present. On the other hand, represent the label
357   set of the query as a {1, 1/2} valued vector, where again a coordinate of 1 is present iff the query has
358   the label. The ℓ22 distance under this embedding always gives the same contribution for labels that
359   are not present in the query. Thus, we essentially ignore labels not in the query when comparing the
360   label sets of the query and a datapoint in X. Note that the asymmetry of the embedding is necessary
361   to capture the asymmetric label set distance. As presented later, the fact that the asymmetric label
362
      distance can be embedded into ℓ2 implies that our final empirical algorithm has provable worst-case
      guarantees, under the common assumption of bounded doubling dimension (see Lemma D.1). Lastly,
363
      the fact that we can reduce the subset query part of MultiFilterANN to an ℓ2 distance calculation
364
      means it can be seamlessly combined with the vector search component.
365
366   Equipped with this, we can essentially reduce the MultiFilterANN problem to (vanilla) ANN by
367   setting dist′ (q, x) = ∥q − x∥2 + λ · ∥encF(Sq ) − encF(Sx )∥2 where encF denotes the m-dimensional
368
      asymmetric encoding described above and λ > 0 is a large parameter.
369   Theorem 6.1. Given a labelled set X of bounded vectors, sufficiently large λ and query vector q with
370   labels Sq , the closest-k database vectors according to distance dist′ is precisely the closest-k feasible
371
      (i.e., those satisfying the label constraint of Sq ) database vectors according to the original distance.

377   distance, which additively inflates all distances we are dealing with, and this consequently alters the
      behaviour of the algorithms used in practice. Due to these considerations, we settle on the following

      7.2   A BLATIONS
517
518
519   We next investigate the effect of query correlation, filter selectivity, and the effect of the penalty term
520   during graph search as ablation studies.
521
      Query Correlation: Inspired by Patel et al. (2024), we develop synthetic labels that are correlated
522   with the underlying embeddings to varying degrees. Intuitively, if a query vector has positively
523   correlated labels, then we would expect its geometric nearest neighbors to generally the same labels.
524   If a query has negatively correlated labels, then we would expect points far from the geometric nearest
525   neighbors to have the labels. In the uncorrelated setting, we cannot make any assumptions about
526   which points would have the labels with respect to the query’s nearest neighbors. Figure 4 shows that
527   for negatively, positively, and non-correlated query labels, there is a small change in performance
528   in favor of positively correlated labels, suggesting that graph search performance is robust towards
529   different query correlations.
530   Filter Selectivity: We evaluate for queries that come with filters of varying degrees of selectivity.
531   Specifically, we study the case where queries come with high specificity labels (>90%), medium
532   specificity labels (45%), and low specificity labels (10%). Figure 6 shows a considerable change
533   in recall for lower selectivities. Indeed, if a query comes with rare filters that are evenly dispersed
534   throughout the dataset, it becomes more difficult to find points with those filters using a distance-based
535   greedy search, as is the case with our graph algorithm.
536   Removing the Search-time Penalty. As Figure 5 demonstrates, incorporating the penalty provides a
537   minimum 2.5x increase to recall performance at the same QPS. The intuition here is that the penalty
538   allows for an implicit prioritization of points with matching filters in the list L PenaltyGreedySearch
539   maintains in its procedure. Without the prioritization, points that are close geometrically to the query
      without meeting the filter constraints can appear closer to the algorithm, leading to poor recall.

792
      In this setting, we are given a universe of m labels and a collection of n label sets (data points). The
793
      goal is to construct a datastructure that takes as input a query set q and outputs data points whose
794   label sets are subsets of q.
795
796   The CIP-data structure comprises of 3 levels. The first level and the second level are both hash tables;
797
      the query is only routed to one entry in the first level and one entry in the corresponding second level
      table. The third level is a collection of disjoint sets g of datapoints. It will turn out that (with high
798
      probability), for all g, either all datapoints in g are valid responses or none are. However, we will
799
      need to evaluate the feasibility of a datapoint from each group. Fortunately, it will turn out that, for
800   any δ ∈ (0, 1), we can construct such a structure with the number of such groups being at most n1−δ .
801
802   To construct the first level, we sample a random subset S (with size o(m)) of the labels. The first
803
      level table will be indexed by the power set of S. The query q will be mapped to S ′ = q ∩ S. It turns
      out that, for each S ′ , with probability 0.5, we can construct a "representative" set R such that for
804
      almost all datapoints x satisfying x ∩ S ⊂ S ′ , |x − R| is small and |R − q| is small. The second level
805
      table will be indexed by small subsets of R (corresponding to R − q) and the third level table will
806   have the datapoints grouped by x − R. More concretely:

         While the main contribution in this work is in making graph-based indices more robust to handling
984      query predicates, as discussed earlier, almost any graph-based search algorithm would struggle when
985      the query predicate is highly selective. Indeed, in extreme situations where a very small number of
986      points satisfy the predicate, it might just make sense to first identify these points (say, by intersecting
987      inverted indices of each of the query labels), and running brute force distances to compute the closest
988      k points. It turns out that for our algorithm in its current shape, there are also some predicates of low,
989      but not too low, selectivity for which an intermediate clustering-based data-structure works best.
990      Given the labelled dataset X of vectors, we choose a target numClus of the number of clusters, and
991      run a k-means clustering with k = numClus to generate the clusters. For each cluster Ci ⊂ X, we
992      maintain label-wise inverted indices invList(Ci , l) that contains the ids of database points that fall
993      into cluster Ci and have label l in their label sets. When the query q arrives with label set Sq , we
994      first identify the closest L clusters, say, C1 , C2 , . . . , CL from the numClus clusters (using brute force
995      search). Then, for each of these L clusters C, we compute ∪L            i=1 ∩l∈Sq invList(Ci , l) to gather the
996      feasible points in these clusters, and then identify the closest k points to the query, again via brute
997      force search, from this set to generate our output.
998      Putting all these pieces together, our final empirical algorithm is then defined as follows: Given the
999      query q with labels Sq , estimate the fraction of database points which will satisfy Sq using a sample
1000     dataset along with label-wise inverted indices over the sample.

1003           2. If the estimate is moderate (e.g. 50000 points), we search using the clustering layer.
