009                                              A BSTRACT
010
011            Counterfactual regret minimization is a family of algorithms of no-regret learning
012            dynamics capable of solving large-scale imperfect information games. We pro-
013            pose implementing this algorithm as a series of dense and sparse matrix and vector
014            operations, thereby making it highly parallelizable for a graphical processing unit,
015            at a cost of higher memory usage. Our experiments show that our implementation
016            performs up to about 401.2 times faster than OpenSpiel’s Python implementation
017            and, on an expanded set of games, up to about 203.6 times faster than OpenSpiel’s
018
               C++ implementation and the speedup becomes more pronounced as the size of the
               game being solved grows.
019
020
021
      1     I NTRODUCTION
022
023
      Counterfactual regret minimization (CFR) (Zinkevich et al., 2007) and its variants dominated the
024   development of AI agents for large imperfect information games like Poker (Tammelin et al., 2015;
025   Moravčı́k et al., 2017; Brown & Sandholm, 2018; 2019b) and The Resistance: Avalon (Serrino et al.,
026   2019) and were components of ReBeL (Brown et al., 2020) and student of games (Schmid et al.,
027   2023). Notable variants of CFR are as follows: CFR+ by Tammelin (2014) (optionally) eliminates
028   the averaging step while improving the convergence rate; Sampling variants (Lanctot et al., 2009)
029   makes a complete recursive tree traversal unnecessary; Burch et al. (2014) proposes CFR-D in which
030   games are decomposed into subgames; Brown & Sandholm (2019a) explores modifying CFR such
031   as alternate weighted averaging (and discounting) schemes; Xu et al. (2024) learns a discounting
032
      technique from smaller games to be used in larger games.
033   We propose implementing CFR as a series of linear algebra operations, as done by Graph-
034   BLAS (Kepner et al., 2016) for graph algorithms, thereby parallelizing it for a graphical processing
035   unit (GPU) at a cost of higher memory usage. We analyze the runtimes of our implementation with
036   both computer processing unit (CPU) and GPU backends and compare them to Google DeepMind’s
037   OpenSpiel (Lanctot et al., 2020) implementations in Python and C++ on 20 games of differing sizes.
038   Our experiments show that, compared to Google DeepMind OpenSpiel’s (Lanctot et al., 2020)
039   Python implementation, our GPU implementation performs about 3.5 times slower for small games
040   but is up to about 401.2 times faster for large games. Against their C++ implementation, our per-
041   formance with a GPU is up to about 85.5 times slower for small games, but, on an expanded set of
042   games, is up to about 203.6 times faster for large games. Even without a GPU, our implementation
043   shows speedups compared to the OpenSpiel baselines (from about 1.5 to 46.8 times faster than their
044
      Python implementation and from 16.8 times slower to 4.5 times faster than theirs in C++). We see
      that the speedup becomes more pronounced as the size of the game solved grows.

190   2.4   P RIOR U SAGES OF GPU S FOR CFR
191
      In the mainstream literature, algorithms inspired by CFR or using CFR as a subcomponent like
192
      DeepStack (Moravčı́k et al., 2017), Student of Games (Schmid et al., 2023), and ReBeL (Brown
193
      et al., 2020) only perform a limited lookahead instead of a complete game tree traversal. A neural
194   network-based value function is typically used to evaluate the heuristic value of a node – GPUs can
195   be utilized for the evaluation of these networks. Besides the fact that the vanilla CFR considers the
196   entire game tree and does not use a value function, our approach differs significantly in that we use
197   the GPU to parallelize CFR at every step of the process.
198
      A number of obscure unpublished works (Reis, 2015; Rudolf, 2021) have implemented CFR directly
199
      on CUDA and found orders of magnitude improvements in performance. However, in the imple-
200   mentation by Rudolf (2021), every thread assigned to each node moves up the game tree (toward
201   the root), thus resulting in a quadratic number of visits to the game tree per iteration in the worst
202   case. The implementation by Reis (2015) is superior in that only one visit is made at each node
203   per iteration by doing level-by-level updates (an approach we also use). However, aside from re-
204   producibility issues with his work1 , both require each thread to perform a “large number of control
205   flow statements” – a limitation mentioned by Reis (2015) – and require more generalized kernel
206   instructions.
207   Our approach addresses these issues by framing this problem as a series of linear algebra operations,
208   and the utilization of GPUs for this task is an extremely well-studied problem in the field of systems,
209   and can take advantage of optimized opcodes for these operations. Our implementation is also com-
210   patible with discrete games in OpenSpiel, which are commonly used as benchmarks for evaluating
211   newly proposed CFR variants, unlike the work by Reis (2015) whose compatible games are limited
212   to customized poker variants. In addition, our pure Python code is open-source.
213      1

495   5     D ISCUSSION
496
497
      Our CFR implementations require a single complete game tree traversal during a setup phase to
498   construct the game tree matrices. The time it takes to complete this one-time operation for the 20
499   games we test are tabulated in Appendix D. One advantage of our approach is that inefficient game
500   logic implementation does not impact the runtime of our implementations as the game tree itself is
501   encoded into matrices, whereas it can severely impact the performance of OpenSpiel’s implemen-
502   tations which evaluate game logic during the traversal itself. This may explain the sizable gap in
503   runtime between the speed of OpenSpiel’s C++ implementation between the experiments even for
504   the games of roughly the same sizes.
505   We only explore parallelizing the vanilla CFR algorithm, as proposed by Zinkevich et al. (2007).
506   Later variants of CFR show improvements, namely in convergence speeds, which modify various
507   aspects of the algorithm. The discounting techniques proposed by Brown and Sandholm can triv-
508   ially be applied by altering Equation 14 and Equation 15. However, pruning techniques (Brown &
509   Sandholm, 2015) would require non-trivial manipulations on the game-related matrices – possibly
510   between iterations – problematic since updating CSR matrices is computationally expensive.
511   On each iteration, our implementation deals with the entire game tree and stores values for every
512   node – impractical for extremely large games. In traditional implementations of CFR, while a com-
513   plete recursive game tree traversal is carried out, counterfactual values are typically not stored for
514   each node but instead for each infoset-actions. We demonstrate that it is possible to achieve a sig-
515   nificant parallelization (and hence speedup) at a cost of higher memory usage. Intuitively, the root-
516   to-leaf paths can be partitioned to construct subgraphs of which separate adjacency and submask
517   matrices can be loaded and applied as necessary – a similar approach can be used for alternating
518
      player updates (Burch et al., 2019) and sampling variants (Lanctot et al., 2009).
519   Our approach provides an alternate way for CFR to be run on supercomputers. During the develop-
520   ment of Cepheus (Tammelin et al., 2015), the game tree was chunked into a trunk and many subtrees,
521   each of which was assigned to a compute node to be traversed independently. This introduced a bot-
522   tleneck in the trunk as the subtree nodes (which depend on the trunk’s results) must wait for the trunk
523   calculation to complete during the downward pass, and wait again while the trunk uses the values
524
      returned by the subtrees during the upward pass. Our approach is simply a series of matrix/vector
      operations, and distributing this is a well-studied problem in systems.
525
526   In our GPU implementation, we used CuPy (Okuta et al., 2017) without any customizations in
527   configurations and did not profile or probe into resource usage. A careful analysis of these for
528   further optimizations will most likely yield further performance improvements.
529
530
      6     C ONCLUSION
531
532
      We introduced our CFR implementation, designed to be parallelized by computing each iteration as
533
      dense and sparse matrix and vector operations and eliminating costly recursive tree traversal. While
534   our goal was to run the algorithm on a GPU, the tight nature of our code also allows for a vastly more
535   efficient computation even when a GPU is not leveraged. Our experiments on solving 20 games
536   of differing sizes show that, in larger games, our implementation achieves orders of magnitude
537   performance improvements over Google DeepMind’s OpenSpiel (Lanctot et al., 2020) baselines in
538   Python and C++, and predict that the performance benefit will be even more pronounced for games
539   of sizes larger than those we tested. Addressing the memory inefficiency and incorporating the use
      of a GPU with non-vanilla CFR variants remains a promising avenue for future research.
