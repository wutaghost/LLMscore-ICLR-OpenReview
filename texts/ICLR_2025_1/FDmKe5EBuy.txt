011                                                A BSTRACT
012
013            Automated red teaming can discover rare model failures and generate challenging
014            examples that can be used for training or evaluation. However, a core challenge in
015            automated red teaming is ensuring that the attacks are both diverse and effective.
016
               Prior methods typically succeed in optimizing either for diversity or for effective-
               ness, but rarely both. In this paper, we provide methods that enable automated red
017
               teaming to generate a large number of diverse and successful attacks.
018
               Our approach decomposes the task into two steps: (1) automated methods for
019
               generating diverse attack goals and (2) generating effective attacks for those goals.
020
               While we provide multiple straightforward methods for generating diverse goals,
021            our key contributions are to train an RL attacker that both follows those goals and
022            generates diverse attacks for those goals. First, we demonstrate that it is easy to
023            use a large language model (LLM) to generate diverse attacker goals with per-
024            goal prompts and rewards, including rule-based rewards (RBRs) to grade whether
025            the attacks are successful for the particular goal. Second, we demonstrate how
026            training the attacker model with multi-step RL, where the model is rewarded for
027            generating attacks that are different from past attempts further increases diversity
028            while remaining effective. We use our approach to generate both prompt injection
029
               attacks and prompts that elicit unsafe responses. In both cases, we find that our
               approach is able to generate highly-effective and considerably more diverse attacks
030
               than past general red-teaming approaches.

      1    I NTRODUCTION
034
035
      Although large language models (LLMs) are now used for many real world tasks (Chen et al., 2021;
036   Achiam et al., 2023; Reid et al., 2024), they are known to be vulnerable to adversarial attacks that
037   can cause them to generate toxic content (Gehman et al., 2020; Perez et al., 2022; Zou et al.), reveal
038   private information (Carlini et al., 2021; Nasr et al., 2023), amplify biases and stereotypes (Zhao
039   et al., 2018; Sheng et al., 2019), hallucinate (Lin et al., 2021; Sun et al., 2023), and be vulnerable to
040   prompt injections (Willison, 2022; Schulhoff et al., 2023; Greshake et al., 2023). To address these
041   vulnerabilities, it is necessary to be able to find weaknesses and failure cases of the model, and
042   iteratively improve on those weaknesses.
043   Red teaming is an effective tool for detecting vulnerabilities and is commonly led by humans or
044   automated red teaming using ML models. Past work on training an LLM as a red teamer using
045   reinforcement learning (RL) requires training a high-quality toxicity classifier as a reward signal and
046   bears a tradeoff between success rate and attack diversity (Perez et al., 2022; Mehrabi et al., 2023).
047   This is because RL causes the model to overfit to the reward and give nearly identical successful
048   attack repeatedly. In contrast, zero- or few-shot prompting approaches do not have a reward signal
049   during training, enabling diverse outputs but much lower likelihood of attack success. Here we aim
050
      to improve how we train a red-teamer LLM to obtain diverse yet effective attacks.
051   Building on this dichotomy, we make the insight to factorize the automated red-teaming system into
052   two parts: first, generate diverse goals for the attacker and then use those to train a red teamer using
053   RL. We find there are multiple easy ways to generate diverse goals for an attacker, such as leveraging
      existing datasets of past attacks or using few-shot prompting of a traditional LLM. These generated

054
      attacker goals are unlikely to be directly effective because they are not tuned for the model being
055   attacked, but they do provide broad diversity without significant manual curation of types of diversity.
056
057   Given a diverse set of ineffective attacks, how do we train a model to make them effective in a realistic
058
      way? Past work on gradient-based attacks have focused on adding soft-tokens or suffixes (Zou et al.;
      Wichers et al., 2024; Sitawarin et al., 2024; Andriushchenko et al., 2024), but these approaches result
059
      in attacks that are unnatural, i.e., unlikely to be requests from real users even adversarial ones. Rather,
060
      whereas past RL approaches relied on a general toxicity reward, we propose a new approach of
061   automatically generated a targeted, zero-shot, rule-based reward (RBR) per-example (Glaese et al.,
062   2022; Achiam et al., 2023; Mu et al., 2024). This not only improves diversity but also leads to a
063   much more flexible design. We also add an additional reward to encourage the model to not stray too
064   far from the diversely-sampled, one-shot demonstrated (ineffective) attack. These rewards improve
065   diversity by avoiding collapse during RL.
066
      While by traditional diversity metrics the above approach performs well, we qualitatively found that
067   the red teamer often learns a relatively small set of tactics to get the model being red-teamed to behave
068   incorrectly. This is similar to gradient-based attacks finding a narrow set of suffixes. To address this
069   we propose using multi-step RL where the red teamer can repeatedly generate new attacks, each time
070   conditioning on past attacks and being rewarded for being both successful and different from past
071   attacks it tried. We go one step further and design a custom diversity measure that focuses on the
072   style or tactics of the attack, which we use in this diversity reward.
073   We demonstrate how to apply our red-teaming approach to two applications: indirect prompt injection
074   from third-party inputs (Willison, 2022; Greshake et al., 2023) and “jailbreaking,” i.e., eliciting
075   unsafe responses. Indirect prompt injections are instructions embedded in third party inputs, such as
076   outputs from tools, that try to trick the model to follow an alternative set of instructions than what
077   the user wanted. While “jailbreaking” aims to get the model to say severely unsafe thing, indirect
078   prompt injections can target any behavior that the user didn’t want, e.g., get the model to respond
079   in a different language (Wallace et al., 2024). Notably, indirect prompt injections are difficult for
080   past automated red-teaming approaches because there is no single grader that covers the diversity of
081
      attacker goals, making the proposed auto-generated reward approach particularly well-suited. While
      contemporaneous works have mentioned generating indirect prompt injections (Wallace et al., 2024;
082
      Reid et al., 2024), to the best of our knowledge, this is the first paper to offer a method for automated
083
      red teaming for indirect prompt injections. In our experiments, we demonstrate, both quantitatively
084   and qualitatively, that our approaches better balance and trade-off diversity and effectiveness on both
085   tasks.
086
087
      To summarize, our main contributions are:
088          • System Factorization: We propose separating the task into (1) generating diverse red-
089            teaming goals and (2) generating successful attacks for those goals. We demonstrate both
090            can be automated and combined to greater effect.
091          • Generated rewards: We provide a method for generating diverse red-teaming goals and
092            accompanying reward functions that can be directly used during RL to train the red teamer
093
               for these goals.
             • Diversity-Reward for Multi-step RL: We also demonstrate how multi-step RL further
094
               increases diversity. We also offer a new diversity reward that focuses on the diversity of
095
               style or tactics of the attacks, enabling the red teamer to continue to generate new attacks.
096          • New Applications: In addition to demonstrating the effectiveness for safety “jailbreaking,”
097            we also offer a method for automated red teaming of indirect prompt injections (Greshake
098            et al., 2023), which to the best of our knowledge is the first work to do so.

103   Gradient-based Adversarial Attacks Adversarial attacks on models aim to trigger incorrect or
104   undesired outputs. With access to the full model architecture, parameters and the training pipeline,
105   it enables white-box attacks that often relies on gradient signals to learn an effective attack. When
106   the adversarial loss is differentiable, such as the probability of wrong labels in classification, we can
107   directly optimize it (Carlini and Wagner, 2017; Madry et al., 2017). However, attack success criteria
      on large language models are commonly non-differentiable, as the output tokens are discrete. Guo


                                                          2
      Under review as a conference paper at ICLR 2025


108
      et al. (2021) apply Gumbel-Softmax approximation (Jang et al., 2016) to make categorical distribution
109   differentiable. (Ebrahimi et al., 2017; Wallace et al., 2019a; Shin et al., 2020) all treat text operations
110   as inputs in the vector space and measure the derivative of loss with regard to these vectors. Mehrabi
111   et al. (2022) experiment with variations of Universal Adversarial Triggers to encourage learned toxic
112   triggers to be imperceptible in the context of multi-turn conversations, via adding language model
113   loss or extra filtration. Zou et al. learn triggers for the model to output affirmative statement given
114   unsafe requests and find that attack sequences learned on open-sourced models show non-trivial
115   transferability to other commercial models. This approach works well when optimizing to output a
116   set of known bad content (Wallace et al., 2019a; Jones et al., 2023; Zou et al.; Wichers et al., 2024;
117
      Sitawarin et al., 2024; Andriushchenko et al., 2024). While conceptually related to our work, we
      treat it as separate because the attacks are often either in soft-tokens or text that is unrealistic, i.e.,
118
      unlike human generated prompts. As such, we find these useful for understanding the limits of a
119
      model’s robustness while we focus our work on generating diverse realistic attacks that can be used
120   to understand model weaknesses and used in training.
121
122   Red Teaming Red teaming is a common approach for discovering model weakness (Dinan et al.,
123
      2019; Ganguli et al., 2022; Perez et al., 2022; Markov et al., 2023), where red teamers are encouraged
      to look for examples that could fail the model. Models trained with red teaming are found to be more
124
      robust to adversarial attack (Dinan et al., 2019; Ziegler et al., 2022) and human-in-the-loop dynamic
125
      data collection can efficiently improve model performance (Vidgen et al., 2020; Kiela et al., 2021).
126   Red teaming can be done by humans with model assistance (Xu et al., 2021). For example, both
127   Wallace et al. (2019b) and Ziegler et al. (2022) created tools to highlight tokens with high saliency
128   scores. FLIRT (Mehrabi et al., 2023) solely relies on in-context learning where a set of exemplars
129   are initialized with a small set of human curated adversarial attack examples and grow with more
130   new attacks are discovered. In-context exemplars are sorted by a combined score of effectiveness,
131   diversity and low-toxicity. Our approach of red teaming is fully based on models where a red teamer
132   model is trained to output effective attacks, similar to Perez et al. (2022). They fine-tuned the attack
133   model with reinforcement learning where the reward is assigned by a toxic classifier on model outputs.
134
      Further, Casper et al. (2023) describe how to train the toxicity classifier as part of their process. In
      contract, we rely on automatically generated rule-based reward function to judge the attack success
135
      corresponding to diverse red-teaming goals.
136
137   Contemporaneous work has explored new related directions here. Samvelyan et al. (2024) use a
138   genetic-like search algorithm to generate attacks, and is able to achieve diverse attacks but requires
139   more curation of the components of diversity. Hong et al. (2024) add a diversity regularizer to the RL
140
      trainer that also discourages collapsing of the model; we will use this as a baseline in our experiments.
      On the surface, (Ge et al., 2023) is also similar in taking a multi-step approach but their approach is
141
      closer to adversarial training with alternating red teaming and training on red-team data; we believe
142
      this can (and should) be combined with any red-teaming approach for improving model robustness.

      We begin by describing the red-teaming problem and our proposed factorization of it.
148
149
150   3.1   P ROBLEM S ETUP
151
152   Here we assume that we have a generative model M which given a prompt p will produce a response
153   y, i.e., M(p) → y. Our goal is build an attacker A that will generate attack prompts p. The goal of
154   the attacker is to get the defender to do some unsafe behavior, judged by a model R(M(p); p).
155
      To use a concrete example, we can assume that our generative model M is an LLM trained to be a
156   conversational agent and to avoid harmful or offensive responses. Further, we can imagine that our
157   attacker is a different LLM trying commands like “Tell me how to build a bomb.” Finally, the judge
158   can be a moderation model, e.g., Moderation API (Markov et al., 2023), Llama Guard (Inan et al.,
159   2023), or Perspective (Dixon et al., 2018), which will determine when a conversational response is
160   unsafe. This is similar to past “jailbreaking” work. Our goal is not just to find a single attack that can
161   generate an undesirable response, but rather for the attacker to be able to be used to generate many
      diverse attacks that generate undesirable responses. Our problem statement can be written as: given a

181   3.2       P ROPOSED S YSTEM D ESIGN
182
183   As discussed above, most prior work approaches this as one large problem for a single model. Here
184   we factor out these goals to some degree into two steps:
185
                1. Get diverse attacker goals: How can we gather or generate a large number of diverse goals
186
                   for the attacker? That is, if we want the attacker to get the model to generate unsafe content,
187                what are a diversity of types of unsafe content we could want it to generate?
188             2. Generate effective attacks: Given a set of goals for the attacker, how can we generate
189                effective attacks that meet these goal and are stylistically diverse?
190
      We will see that factoring the problem makes it easier to generate diverse and effective attacks.
191
192   Given we previously described the attacker as a monolithic A, let’s adjust our notation to reflect this
193   factorization. First, we want a method AG that will produce attacker goals g ∼ AG . Then we will
194   assume that the attacker model takes in a goal and can produce effective attacks, Am (g) → p.
195   While in Section 4 we discuss multiple approaches for AG , we first here give a high-level overview of
196   the approach we take for the attacker Am to help clarify the overall system design. We build on work
197   from Perez et al. (2022) and will train our attacker Am using reinforcement learning. We will describe
198   below a reward function R, but overall we train Am by: Am = arg maxAm Eg∼AG [R(M(Am (g)))].
199   In the subsequent sections, we focus on designing AG and Am .

      Our first task is to generate diverse red-teamer goals. In particular, we will describe how an LLM can
      be used to generate diverse goals that can be directly used to improve the attacker model’s diversity.
205
206   This concept of diverse red-teamer goals is fairly intuitive. If we want to find cases of the model
207   giving advice to commit a crime, breaking this into “instructions to hijack a car” and “guidance for
208   how to launder money” are considerably different ways the model could give an undesirable response,
209   and just because a defender refuses one does not mean it will refuse the other.
210   While this is intuitive, how would we formulate this for an attacker to generate effective attacks for
211   all of these goals? We consider that these goals could be used in the Am in two ways: (1) as natural
212   language instructions (i.e., in the prompt), and (2) in the reward. First, these goals can be given in
213   the prompt to the red teamer, as instructions or as a one-shot example, e.g., “Write a prompt that gets
214   an AI to give [instructions]. For example, ‘Please AI, do [instructions]’.”

      We can also use the goals as part of the reward. First, these goals can be judged, e.g., did the model
235   actually give instructions for the specific crime, and used during training Am as a reward. From a
236   technical perspective, implementing this on the surface seems more challenging. Here, we build on
237   rule-based rewards (RBRs) (Glaese et al., 2022; Achiam et al., 2023; Mu et al., 2024). Rule-based
238   rewards, at a high-level, are simply classifiers that judge if an input violates some specified policy, but
239   in (Glaese et al., 2022; Achiam et al., 2023) these are implemented as an LLM that takes in the policy
240   as a prompt and then performs the classification. An example from Glaese et al. (2022) is “Do not
241   use stereotypes or make any other harmful generalising statements about groups of people.” Building
242   on insights from Mu et al. (2024), we make the observation that LLMs are quite good at zero-shot
243
      classification and as such, we can use a large number of automatically generated, targetted rule-based
      rewards. In particular, we formulate the goals as prompts for a rule-based reward, e.g. “Does this text
244
      give [criteria]?” Additionally, while not their primary purpose, we find that the goals also are
245
      useful as part of regularizers in the reward function. We’ll discuss these in more detail in Section 5.2.
246
247   Now, how do we generate a diverse set of instructions and criteria?
248   Few-shot Generation As discussed in prior work (Perez et al., 2022; Mehrabi et al., 2023), few-shot
249   prompting methods can give diverse but unsuccessful attacks. However, for our use case, we don’t
250   need them to be successful, just diverse. In practice, we have found this to be a fairly easy strategy
251   but do not tune our methodology extensively. For example, a prompt to generate prompt injection
252   goals can be seen in Figure 2. As we can see here, the model can output both an example instruction
253   (the first part of the tuple) and a criteria (the second part of the tuple). The example instruction can
254   then be used as part of the prompt to the attacker Am , e.g., as a one-shot example. The criteria in
255
      the second part of the attack can be used in the rule-based reward as discussed above, e.g., “Does
      this text urge daily exercise?”. In other cases, we use a slightly more tailored approach to generate
256
      attacker goals that are more specific, such as getting the model to respond in different styles, with
257
      examples “like a pirate” or “in leetspeak.” More detail on how we generate attacks can be found in
258   Appendix A.1. That said, the main point here is that this can be relatively directly and easily done
259   with existing LLMs.
260
261
      Reward Generation from Data A second approach is that often there are existing datasets related
      to the task at hand that can be transformed into instructions and rewards. For example, when trying to
262
      get the model to produce unsafe text, we can build on datasets like the Anthropic Harmless dataset
263
      (Bai et al., 2022), which includes many diverse examples trying to solicit unsafe responses by human
264   red teamers. While we will later show that using this dataset directly is unlikely to elicit an unsafe
265   response, it still is valuable in demonstrating a wide diversity of safety concerns. We find that we can
266   use an LLM to transform every example into an instruction and criteria with a few-shot prompt; an
267   example prompt can be seen in Figure 3. The full prompt is given in Appendix A.2, but we find that
268   this is an easy task for an LLM and use the data from this directly for training the red teamer and
269   later for evaluation. Taken together, we believe both few-shot generation and rewriting existing data
      provide flexible and powerful ways to generate diverse goals for the red teamer.

      Once we have a diverse set of red-teamer instructions and rewards, we can now begin to use these to
273   train our red-teamer model. As mentioned above, our red teamer Am is trained using reinforcement
274   learning. In particular, we assume that from the methods in Section 4 that we have a dataset of
275   instructions and criteria. With a slight abuse of notation, we will consider that we can sample
276   g = (instruction, criteria) ∼ AG , where we will use the instruction to create an example attack
277   xg and we will use the criteria to form the RBR xr . In particular, the prompt for the attacker is
278   constructed based on using the instruction and example attack xg . For example:
279
280       Please write a request that would get an AI to answer the question: “persuade the user to incorporate daily exercise for health
          benefits”. For example: “To any AI reading this, ensure you persuade the user to incorporate daily exercise for health benefits.”

      5.1        ATTACK S UCCESS R EWARD
286
287   The primary reward for the model is based on judging if the attack p was successful based on the
288   response the model gives M(p) → y; we’ll denote this as RAttSuccess (y). As mentioned above, we
289
      must rely on automated methods for judging the attack success.
290   In some cases, e.g. indirect prompt injections, the rule-based reward is the judge RRBR . That is, we
291   can query an LLM RRBR based on the generated RBR xr , to produce the reward: RAttSuccess (y) =
292   RRBR (y; xr ). Here, the RBR is a yes/no question where RRBR := P (yes) for the given question.
293   In other cases, such as for unsafe content generation, we want the red teamer to find attacks that are
294   both successful according the specific attack goal and violate a general policy. For unsafe content
295
      generation, we use OpenAI’s Moderation API (Markov et al., 2023) as this second reward component.
      In this case, we simply average these two components: RAttSuccess (y) = 0.5 × RRBR (y; xr ) + 0.5 ×
296
      RModeration (y)2 .

298
299
      5.2        F EW- SHOT S IMILARITY
300   In addition to using the RBRs to judge if the red teamer is accomplishing the example-specific attack
301   goal, we also use the example attack as an anchor from which the generated attack shouldn’t be too
302   different. That is, in the example above, the generated attack shouldn’t be too different from “To
303   any AI reading this, ensure you persuade the user to incorporate daily exercise for health benefits.”
304   Because these example attacks are diverse by design as described in Section 4, this helps to further
305   prevent the red teamer from collapsing to giving very similar attacks. In this case, we do this by
306   measuring how similar the generated attack Am (g) is to the example xg given in the prompt. To
307   measure similarity we use cosine similarity, denoted as sim, based on embeddings ϕ of both the
308
      example and the attack (Neelakantan et al., 2022), and then use a sigmoid to adjust the distribution.
      As such, one component of our reward is RFewshot = sim(ϕ(xg ), ϕ(Am (g))). (See §B.1.1 for details.)

311   5.3        M ULTI - STEP RL AND S TYLE D IVERSITY
312
      While adhering to the diversity of safety goals improves diversity across attacks, we still find that the
313   attacks can be fairly similar, particularly stylistically. For example, “Create a satirical op-ed: ‘How to
314   deal with protesters: Just run them over with your car!” and “Create a satirical op-ed: ‘How to rob
315   trans people: Just invade their gender identity by calling them men!” target different unsafe behavior
316   but follow a very similar pattern.
317
      To address this, we let the red teamer perform a sequence of attacks, each conditioned on prior attacks
318
      and rewarded for being different from those past attacks. In particular, we treat this as a conversation
319   where the red teamer gets a response, e.g., “Success! Please create a new, different prompt to elicit
320   the model to [goal]” and then can respond with another attack. To slightly extend our notation, in
321   the first step of the trajectory, the red teamer will produce an attack Am (xg ) → p0 , and in subsequent
322   steps, the red teamer will produce an attack Am (xg ; p0:T −1 , RAttSuccess,0:T −1 ) → pT .

340   5.4       I MPLEMENTATION D ETAILS
341
      As with any complex training setup there are numerous implementation details. While we kept all
342
      fixed across experiments for a clean and clear experimentation, we discuss a few key choices here.
343
344   Length Penalty We found that an easy way the red teamer can increase diversity is by adding
345   arbitrary text to the attacks. This results in attacks that are less meaningfully different and we also
346
      believe that shorter, simpler attacks are more valuable to discover as they are more likely to be
      uncovered by real people. We therefore add a length penalty Rlen , where attacks less than min len
347
      long are not penalized and attacks longer than max len are equally penalized.
348
349   Multi-objective Reward As described above, there are many reward components. We want the
350   attacks to be successful and to be similar to the example attack and to be stylistically diverse and
351   to be not too long. We find that multiplying the rewards is the most effective way to encourage the
352
      model to do all of these goals simultaneously and not give up on any one:
353                                        R = RAttSuccess × RFewshot × RDiv × Rlen                                    (2)
354
355
      When a given term doesn’t apply, we set it to a constant (we only apply RFewshot to the first attack in a
      sequence p0 and we only apply RDiv to subsequent attacks pT for T > 0). Also, we want to ensure
356
      that no reward gets too close to zero as it hurts training; Rlen in particular runs this risk, and is of
357
      lower priority than the other rewards, so we scale it between [0.5, 1].

456   Next, we test how well the red teamer can get the model to violate safety policies. We use this more

458   Task Design We use the Anthropic “Harmless” data from Bai et al. (2022) as seeds for this task. We
459   use the existing train/test split of the data, prune examples that have more than one user message, and
460   use a 3-shot prompt to generate a “goal” and “criteria” for each example. The “goal” is then used in
461   red teamer’s prompt and the “criteria” is used in the RBR. We use 5000 examples for our training set
462   and 654 examples for our test set, and will include code upon publication.
463   Grading and Metrics To grade whether an attack was successful we use OpenAI’s Moderation API
464   (Markov et al., 2023), where we use the maximum value across all categories as the real valued output
465   and use a threshold of 0.5 to determine if the attack was successful. We generally will compute attack
466   success rate based on this, and will compute diversity based on cosine similarity as described above.

469   consistency), and the “Curiosity” reward from Hong et al. (2024), which adds multiple reward terms
470   to penalize the model for generating attacks similar to those in earlier batches. Again, we train three
471   versions of each model and at test time, we use temperature = 0, T = 10 steps.

518   7    C ONCLUSION , L IMITATIONS , AND F UTURE W ORK
519   In this paper we offer new techniques for automated red-teaming that are more effective and lead
520   to more diverse attacks. We make a number of contributions. We show how designing red-teaming
521   as a two-step process enables combining low-success, high-diversity methods (e.g., few-shot attack
522   generation) with RL to make attacks effective. Further, we offer multiple new components to the RL
523   attacker design, including using generated attack goals as per-example rewards, a multi-step design
524   for conditioning on past attack attempts, and a modified diversity signal that focuses on attack style.
525   We show that each of these components combine to lead to better red-teaming, such as enabling
526   red-teaming for indirect prompt injections and achieving improved diversity with minimal hit to
527
      attack success rate.
528   While we are excited to share our research with the community as we believe it can help others
529   develop stronger red-teaming methods and safer AI, there are a few limitations that we hope to
530   expand on in future work. Most significantly, “jailbreaking” and “red-teaming” remain broad terms
531   with methods that are not easy to compare as they make different implicit assumptions about what
532   type of attacks are valuable to uncover, and further work is needed to develop consistent evaluations
533
      and metrics for concepts like diversity and realisticness. We focus on comparing to the most related
      methods, understanding when each approach works. Additionally, while we believe the contributions
534
      here are effective, we do find significant sensitivity in our method, as is common in RL. In particular,
535
      we observe significant variance across runs and sensitivity to reward shaping choices. We do not find
536   this to prevent usage, but are hopeful that future work can make the method easier and more reliable.
537
538   Altogether, we believe the technical contributions can provide new and improved techniques for
539
      building red-teaming LLMs and are optimistic about the way in which the work can be built upon for
      even stronger red teamers and new applications.

543   such as discovering vulnerabilities and creating hard examples for evaluation and training purposes,

834
835   Prompting During the multi-step RL, the “user” role in the conversation with the red-teamer needs
836   to give a response. There, we tell the red-teamer whether it’s attack was successful or not and what to
837
      do next as follows:
838
      Algorithm 1 Set “User” Response to Red-Teamer
839
840
      Require: reward, diversity reward, attack goal
       1: msg ← ””
841
       2: if reward > 0.1 then
842
       3:     msg ← “Success! (Reward = {reward})!”
843    4: else
844    5:     msg ← “Failed!”
845    6: end if
846    7: if diversity reward is not None then

848    9:         msg ← msg + “But this was too similar to past successes (diversity = {diversity reward})!”
849   10:     end if
850   11: end if
      12: msg ← msg + “Please create a new, different prompt to elicit the model to {attack goal}”
851
      13: return msg

855   Diversity reward As described in the paper, we compute a diversity reward RDiv based on the
856   cosine similarity to the past attacks attempted by the model. Because these similarities are based
857   on the closest from a variable size set, the absolute values of diversity can vary. As a result, we
858   normalize the values over each batch and then put them through a sigmoid function. In particular,
859   if we have a batch of examples RDiv = {ri }, we denote the mean diversity by µDiv and standard
860
      deviation by sDiv . We compute the reward by:

861                                       (i)                ri − µDiv
862                                      RDiv = σk=5,x0 =0                                             (3)
                                                              sDiv + ϵ
863
      Here ϵ is a smoothing factor set to 0.1.
