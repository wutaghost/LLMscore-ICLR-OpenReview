009                                                A BSTRACT
010
011            Modern code completion engines, powered by large language models (LLMs),
012            assist millions of developers with their impressive capabilities to generate function-
013            ally correct code. As such it is crucial to investigate their security implications. In
014            this work, we present INSEC, the first black-box adversarial attack designed to
015            manipulate modern LLM-based code completion engines into generating vulnera-
016            ble code. INSEC works by injecting an attack string as a short comment in the
017            completion input. The attack string is crafted through a query-based optimization
018            procedure starting from a set of initialization schemes. We demonstrate INSEC’s
019
               broad applicability and effectiveness by evaluating it on various state-of-the-art
               open-source models and black-box commercial services (e.g., OpenAI API and
020
               GitHub Copilot). We show that on a diverse set of security-critical test cases cover-
021
               ing 16 CWEs across 5 programming languages, INSEC significantly increases the
022            rate of generated insecure code by ∼50%, while upholding the engines’ capabilities
023            of producing functionally correct code. Moreover, due to its black-box nature,
024            developing INSEC does not require expensive local compute and costs less than 10
025            USD by querying remote APIs, thereby enabling the threat of widespread attacks.

028   1    I NTRODUCTION
029
030   Large language models (LLMs) have greatly enhanced the practical effectiveness of code completion
031   (Chen et al., 2021; Nijkamp et al., 2023; Rozière et al., 2023), significantly improving programmers’
032   productivity. As a prominent example, the GitHub Copilot code completion engine (GitHub, 2024) is
033   used by more than a million programmers and five thousand businesses (Dohmke, 2023). However,
034   prior research has shown that LLMs are prone to producing code with dangerous security vulnerabil-
035
      ities (Pearce et al., 2022; Li et al., 2023). This poses significant security risks, as LLM-generated
      vulnerabilities can be incorporated by unassuming programmers (Perry et al., 2023). Even more
036
      concerning is the potential for attacks on the completion engine, which can substantially increase
037
      the frequency of generated vulnerabilities. Prior research has considered poisoning attacks, elicit-
038   ing insecure behavior in a “white-box” manner by modifying the model’s weights or training data
039   (Schuster et al., 2021; He & Vechev, 2023; Aghakhani et al., 2024; Yan et al., 2024). However, these
040   attacks require access to the models’ training process, which is typically out of reach for the adversary
041   (Carlini et al., 2024) or demands expensive compute to mount. Moreover, such attacks cannot be
042   executed on code completion services already in operation, such as GitHub Copilot.
043
044   Realistic Black-Box Setting In this work, we focus on an alternative setting, where the attacker
045   launches adversarial attacks to piggyback on an existing code completion engine for vulnerable code
046   generation. In line with the operational methods of state-of-the-art completion services, such as
047   GitHub Copilot and the OpenAI API (OpenAI, 2024), the attacker has only black-box query access to
048   the engine, without any knowledge of or influence over its internal workings. However, the attacker
049   can inject adversarial content into the engine’s input to induce the generation of vulnerable code. To
050
      ensure stealthiness and increase the likelihood of adopting the vulnerable completions, the attack must
      preserve the functional correctness of generated programs and minimize any introduced overhead.
051
052   Our threat model eliminates the need for interfering with the training of a remote model or the costly
053   process of training and hosting a custom local model, key limitations that hinder the practicality
      of poisoning attacks. Instead, it captures realistic scenarios where the attacker targets deployed

079   completion. This comment serves as an influential instruction for the model to generate insecure code,
080   while having minimal impact on the functionality of the generated code. Moreover, the attack string

082   an example, Figure 1 depicts how INSEC drives CodeLlama-7B to apply a weak hash function. To

098   and monetary costs, e.g., <$10 for the development of an attack with GPT-3.5-Turbo-Instruct.

      Code Completion Engine We represent code as strings and consider an LLM-based code com-
109   pletion engine, denoted as G, which produces infillings c based on an input pair of a code prefix p
110   and a suffix s. See Figure 1a for an example. We represent the completion process by c ∼ G(p, s).
111   The final completed program x is then formed by concatenation: x = p + c + s. When the engine
112   produces multiple completions from a single query, we use the notation c ∼ G(p, s).
113
114   Measuring Functional Correctness Given (p, s), the primary goal of code completion is to
115   generate c, such that x = p + c + s is a functionally correct program and meets the programmer’s
116   requirements. Following the popular HumanEval benchmark (Chen et al., 2021), we use unit tests to
117   decide the correctness of x. We define an indicator function 1func (x) that returns 1 if and only if x
118   passes all associated unit tests. To measure the overall capability of G in functional correctness, we
119
      leverage the standard pass@k metric (Chen et al., 2021), formally defined as below:

      3    T HREAT M ODEL
141
      The attacker seeks to compromise a completion engine such that it effectively acts as a malicious
142
      engine Gadv that frequently suggests insecure code. If these suggestions are incorporated, they
143
      could introduce major vulnerabilities into the programmer’s codebase. To maximize the chances
144   of programmers adopting Gadv and its insecure code suggestions, the attacker must ensure the
145   stealthiness of the malicious activity by maintaining the overall utility of Gadv (He & Vechev, 2023).
146
147
      To capture a broad range of important practical settings, including attacks on black-box APIs like
      OpenAI API (OpenAI, 2024) and commercial plug-ins such as GitHub Copilot (GitHub, 2024), we
148
      assume that the attacker has only black-box access to G when developing their attack. As such,
149
      the attacker has no access to model internals, such as parameters, training data, logits, or even the
150   tokenizer. While the restricted access makes our setting more realistic, it also significantly increases
151   the difficulty of attack development, as continuous optimization w.r.t. the target model is not possible.
152
153
      One way to achieve this would be to train and host a malicious code completion engine. However,
      this is not realistic, as: (i) training, hosting, and engineering a state-of-the-art engine (such as, e.g.,
154
      GPT-3.5-Turbo-Instruct) requires resources only available to very few commercial or state actors,
155
      and (ii) while some attackers might have the resources to handle a small model, it is difficult for such
156   a model to gain traction, because it cannot compete with popular commercial solutions. Instead, the
157   attacker can efficiently reach their goal by developing a black-box adversarial attack for existing,
158   already adopted code completion engines. Formally, given black-box access, the attacker can leverage
159   a code completion engine G to devise a lightweight attack function f adv . This function modifies
160   the original input pair (p, s) into an adversarial pair (p′ , s′ ), which is then fed into G to achieve the
161   malicious objective, i.e., Gadv (p, s) = G(f adv (p, s)). For the attack to be successful, Gadv must
      satisfy three constraints: (i) Gadv should exhibit a high rate of generated vulnerabilities, as quantified


                                                          3
      Under review as a conference paper at ICLR 2025


162
      by vul ratio(Gadv ); (ii) Gadv must maintain strong functional correctness relative to G, as measured
163   by func rate@k(Gadv , G); and (iii) in order to be practically deployable and remain stealthy, f adv
164   must also be lightweight and minimize any introduced overhead.
165
166   Practical Attack Deployment In Section 1, we discussed the highly concerning potential of
167   deploying such an attack as a malicious IDE plug-in—a prominent attack vector for malware, since
168   such plug-ins are able to execute arbitrary commands with user-level privilege, and are subjected
169   only to easily avoidable anti-virus scanning in marketplaces (Ward & Kammel), amassing millions
170   of downloads (Pol, 2024; Toulas, 2024). The attack can also be deployed in various other realistic
171   ways, as long as the adversary gains control over G’s input. These include intercepting user requests,
172   supply chain attacks, or setting up a malicious wrapper over proprietary APIs. Note that even though
173   end-to-end deployment of such an attack is possible, due to ethical considerations, we do not attempt
174
      deployment, but focus on developing our attack within the confines of the outlined threat model.

176   4     O UR INSEC ATTACK
177
178   In this section, we present INSEC, the first black-box attack within the confines of the practical threat
179   model described in Section 3. INSEC consists of an attack template (Section 4.1) and a randomized
180   optimization algorithm (Section 4.2), which is initialized using diverse strategies (Section 4.3).
181
182   4.1   ATTACK T EMPLATE
183
184   According to our threat model, the attacker’s objective is to find an adversarial pre-processing function
185   f adv . INSEC constructs f adv using a predefined template that inserts a short attack string σ as a
186   comment above the line awaiting for completion, not modifying the suffix. An example insertion can
187   be found in Figure 1b. It is important to note that under INSEC, the programmer retains the freedom
188
      to make any completion request, and a fixed σ is indiscriminately inserted into all such requests. This
      design conforms to the requirements of our threat model: (i) σ acts as an instruction that drives the
189
      engine to generate vulnerable code in relevant security-sensitive coding scenarios; (ii) because σ
190
      is short, it causes minimal negative impact on overall functional correctness; and (iii) the insertion
191   process at deployment time is trivial and adds only a few tokens, resulting in negligible overhead. In
192   Section 5 and Appendix C, we provide various ablation studies to empirically validate the optimality
193   of our design choices for the attack template, including the insertion location and σ’s length.
194

      4.3   ATTACK I NITIALIZATION
258
259   To improve the convergence speed and performance of our optimization algorithm, we develop five
260   diverse strategies for initializing the pool of candidates for the attack string σ. These strategies are
261   generic and easy to instantiate. Furthermore, both the initialization strategies and the optimization
262   process are performed only once per attack, since σ is fixed at deployment time. Thanks to the
263   modular design of INSEC, more initialization strategies can be easily added if necessary.
264   We now provide a high-level description for each strategy. Detailed explanations and examples
265   can be found in Appendix B. The first two strategies are independent of the vulnerabilities targeted
266   by the attacker: (i) Random Initialization: this strategy initializes the attack string by sampling
267   tokens uniformly at random to increase diversity. (ii) TODO initialization: inspired by Pearce et al.
268   (2022), this strategy initializes the attack string to “TODO: fix vul”, indicating that the code to be
269   completed contains a vulnerability. For the remaining three strategies, we utilize the completion
      tasks in the training set Dtrain
                                  vul along with their corresponding secure and vulnerable completions:


                                                          5
      Under review as a conference paper at ICLR 2025


270
      (iii) Security-Critical Token Initialization: as noted by He & Vechev (2023), the secure and
271   vulnerable completions of the same program may differ only on a subset of tokens. Following this
272   observation, we compute the token difference between the secure and vulnerable completions. We
273   start the optimization from a comment that either instructs to use vulnerable tokens or instructs not to
274   use secure tokens. (iv) Sanitizer Initialization: many vulnerabilities, such as cross-site scripting,
275   can be mitigated by applying a sanitization function on user-controlled input. In this strategy, we
276   construct the initial comment to indicate that sanitization has already been applied, guiding the
277   completion engine not to generate it again. (v) Inversion Initialization: for a given vulnerable
278   program, this strategy requests the engine to complete a comment in the line above the vulnerability.
279
      This initial comment directly exploits the learned distribution by the LLM, as it generates the most
      likely comment preceding a vulnerable section of code.

295   Evaluating Functional Correctness We instantiate the func rate@k metric, as defined in Equa-
296   tion (2), to evaluate functional correctness. To achieve this, we follow Bavarian et al. (2022) to use

298   paired with the corresponding unit tests. To create each completion task, we remove a single line
299   from the canonical solution of a HumanEval problem. Since our vulnerability assessment spans five
300   programming languages, we create a separate dataset for each language, using a multi-lingual version
301
      of HumanEval (Cassano et al., 2022). As the canonical solutions in HumanEval are only in Python,

302
      then divide these datasets into a validation set Dval                   test

305   func rate@10 exhibit a similar trend and thus omit func rate@10 when not necessary.

469   6    D ISCUSSION
470
471   INSEC’s Surprising Effectiveness Although our black-box threat model assumes a more restricted
472   realistic attacker than prior attacks that require access to model internals (Schuster et al., 2021; He &
473
      Vechev, 2023; Wu et al., 2023; Aghakhani et al., 2024; Yan et al., 2024), INSEC remains effective
      in terms of both vulnerability rate and functional correctness. This can be attributed to INSEC’s
474
      ability to exploit the strong instruction-following capabilities of LLMs and the fact that many types of
475
      vulnerabilities lie within the distribution modeled by LLMs. Moreover, the perturbation introduced by
476   INSEC is small, allowing modern LLMs, especially the more capable ones, to ignore the perturbation
477   in normal usages not concerning security, thereby generating functionally correct code.
478
479
      Potential Mitigations We appeal to the developers of these engines to implement mitigations, such
480   as: (i) alerting the programmer if a substring occurs repeatedly at an unusually high frequency; (ii)
481   similarly to mitigating certain jailbreaks (Jain et al., 2023), sanitizing prompts before feeding them
482   to the LLM; or (iii) interrupting users suspected of repeated querying for the purpose of optimizing
483   an attack similar to ours. For the latter point, while current code completion engines already have
484   query limits in place, as evidenced by our success at attacking GitHub Copilot, they are insufficient
485   in preventing INSEC-style attacks. We further discuss directions for defenses in Appendix E, such
      as adding security inducing comments, scrubbing comments, and deployment of static analysis.

      Limitations and Future Work While our black-box attack already exposes a concerning vulnera-
487   bility of today’s code completion engines, future studies could push the boundary further. Our attack
488   still incurs some relative functionality loss on certain completion engines. Stronger attacks could
489   incorporate an explicit objective in the optimization to preserve functional correctness. Moreover, an
490   interesting future direction would be to extend our work to more scenarios, such as coding agents
491   (Jimenez et al., 2024) and an even more diverse set of vulnerabilities.

      7    R ELATED W ORK
495
496   Code Completion with LLMs Transformer-based (Vaswani et al., 2017) LLMs trained on massive
497
      codebases have excel at solving programming tasks, with specialized code-specific models including
      Codex (Chen et al., 2021), CodeGen (Nijkamp et al., 2023), StarCoder (Li et al., 2023), CodeLlama
498
      (Rozière et al., 2023), and many others. LLMs specialized for code completion are trained with a
499
      fill-in-the-middle objective (Bavarian et al., 2022; Fried et al., 2023) in order to handle both a code
500   prefix and postfix in their context. Several user studies have confirmed the benefit of LLM-based
501   code completion engines in improving programmer productivity (Vaithilingam et al., 2022; Barke
502   et al., 2023), with such services being used by over a million programmers (Dohmke, 2023).
503
504
      Security Evaluation of LLM Code Generation As code LLMs are increasingly employed, inves-
505   tigating their security implications is critical. Pearce et al. (2022) were first to show GitHub Copilot
506   (GitHub, 2024) frequently generates insecure code. Follow-up works extended their evaluation,
507   revealing similar issues in StarCoder and ChatGPT (Li et al., 2023; Khoury et al., 2023). CodeLMSec
508   (Hajipour et al., 2024) evaluates LLMs’ insecure code generation using automatically generated
509   security-critical prompts. However, these works focus on model security only in benign cases, while
510   we examine LLM-based code completion under attack, the worst case from a security perspective.
511
512   Attacks on Neural Code Generation Prior attacks achieve increased code vulnerability by inter-
513   fering either directly with the model weights or its training data (Schuster et al., 2021; He & Vechev,
514   2023; Aghakhani et al., 2024; Yan et al., 2024). However, such attacks are unrealistic to be carried out
515   against deployed commercial services. In contrast, our attack only requires black-box access to the
516   targeted engine. Besides the different threat models, our evaluation covers more CWEs and languages
517
      than these works, as discussed in Appendix A. In a similar fashion to jailbreaks targeting generic
      LLMs (Zou et al., 2023; Yao et al., 2024), DeceptPrompt can synthesize adversarial natural language
518
      instructions that prompt LLMs to generate insecure code (Wu et al., 2023). However, our work differs
519
      from theirs in two significant ways. First, DeceptPrompt requires access to the model’s full output
520   logits, which often are not available for model APIs or commercial engines. In contrast, INSEC
521   does not face this limitation and successfully attacks widely used commercial services. Second,
522   our work considers the attack’s generalization among different completion inputs. DeceptPrompt,
523   however, only targets a single user prompt at a time. Apart from code generation, prior work has
524   leveraged genetic optimization for semantic-preserving transformations to attack code classification
525   models (Yang et al., 2022). This attack is performed for each input, incurring significant overhead for
526   inference. In contrast, the attack string of INSEC is derived once and fixed across inputs at inference,
527   thus meeting the real-time requirements of modern code completion.

      8    C ONCLUSION
530
531
      We presented INSEC, the first black-box attack capable of directing commercial code completion
532
      engines to generate insecure code at a high rate, while preserving both utility and functional correct-
533
      ness. INSEC leverages an attack template that inserts an attack string as a short comment above the
534   completion line, coupled with a black-box optimization algorithm that iteratively mutates candidate
535   attack strings and selects the top-performing ones. This optimization procedure is further strength-
536   ened by a set of diverse initialization strategies. Through extensive evaluation, we demonstrated the
537   effectiveness of INSEC not only on open-source models but also on real-world production services
538   such as the OpenAI API and GitHub Copilot. Given the broad applicability and high severity of
539   our attack, we advocate for further research into exploring and addressing security vulnerabilities in
      LLM-based code generation systems.

      E THICS S TATEMENT
541
542
      In this paper, we have introduced INSEC, the first black-box attack to adversarially steer (commercial)
543   code completion engines towards generating insecure code. As our attack can be potentially developed
544   even by an attacker with notably low resources, and deployed on commercial services exploiting well-
545   known vulnerabilities of, for instance, IDE plug-in marketplaces; we have made careful steps to ensure
546   that our research process and publication of our results is aligned with the ethical responsibilities
547   carried by the potential harms of INSEC. For this reason, 45 days before making any version of this
548   manuscript, or any other derivative of this study, public, we have responsibly disclosed our findings to
549   the corresponding model developers. Further, due to ethical concerns, the scope of our experiments
550   and the attack source code do not extend to implementations of an end-to-end real-world attack on the
551
      commercial engines, e.g., we do not implement any method that hijacks user queries before delivering
      them to the completion engine. Additionally, we also did not include any concrete optimized attack
552
      strings in this paper, nor in any supplementary material. All attack strings included in the paper are
553
      dummy strings representing the overall patterns of the optimized attacks. Finally, from a broader
554   perspective, we believe that the good-faith uncovering and publishing of exploits to systems with a
555   wide user base is ultimately of benefit to the security of such applications, providing the first step
556   towards mitigating security limitations that could otherwise be exploited by nefarious actors.

      Random Initialization We increase the diversity of our initialization by generating random attack
825
      strings. We achieve this by randomly sampling tokens from the attacker’s tokenizer T and concatenat-
826   ing them into strings. Note that such generated strings are not usually completely random characters,
827   but feature some structure based on the size and content of the tokenizer dictionary. An example for
828   such a string σ is “éd senior     sp cuts”, which includes complete words and unicode characters
829   and was generated by sampling tokens at random from the CodeQwen tokenizer (Bai et al., 2023).
830
831   TODO Initialization We initialize the attack string σ to “TODO: fix vul” to indicate that the
832   code to be completed was marked, e.g., by a human developer, to contain a security vulnerability.
833   If the completion engine is aware of potential vulnerabilities or has picked up similar code snip-
834   pets containing review notes and insecure code, we expect it to be steered towards generating the
835   corresponding insecure code.
836
837   Security-critical Token Initialization We observe that, for a wide range of vulnerabilities,
838   there exist critical tokens that decide the security of the whole program. For instance, con-
839
      sider the following implementation of a database query using securely parameterized SQL:
      cursor.execute('SELECT ... WHERE id=%s', user id). Here, user id is an untrusted user
840
      input and the %s', parametrization makes sure that any potentially dangerous characters in user id
841
      are escaped. In contrast, an insecure implementation would be: cursor.execute('SELECT ...
842   WHERE id=' + user id), where the untrusted input is directly concatenated to the query without
843   any checks. As such, the security-critical tokens are “%s',” and “' +”. The concrete tokens for each
844   CWE can be extracted directly using the training dataset and secure and insecure completions by
845   computing the textual difference. We exploit this pattern to create an initialization scheme yielding
846   strings of the format “use {insecure tokens}” and “don't use {secure tokens}”. For the
847   above example of SQL injection in Python, we would create initial attack strings “use ' +” and
848   “don't use %s',”.

1039
1040   In this section we discuss possible defenses against INSEC, such as adding comments to counter the
1041   effect of INSEC, scrubbing all comments from prompts and deploying static analysis in production.
1042
1043
1044
       Security comments We investigate whether adding additional comments can mitigate our attack,
1045
       when such comments instruct the model to generate secure code. We insert This code should
1046
       be secure in the line above the INSEC attack string, using the attack string optimized without
1047   the presence of the comment. On GPT-3.5, averaged over all CWEs, this slightly decreases the
1048   vulnerability ratio from 76% to 62%. This score still largely exceeds the baseline ratio of only
1049   22%. This result is not surprising, as previous work has found that usual, unoptimized comments
1050   are insufficient to steer models towards secure code generation (He et al., 2024a; Liu et al., 2024).
1051   Exploration of the interaction between opposing optimization schemes for and against code security
1052   would pose an interesting topic of future research.
1053
1054
1055   Comment scrubbing In contrast, we investigate the scrubbing of all comments from code as a
1056   possible avenue for defense. We note that code models rely on comments to steer their generations
1057   (Anonymous, 2024; Song et al., 2024) and suspect that removal of comments generally reduces
1058   performance on standard tasks. We evaluate this experimentally by removing all comments from the
1059   HumanEval dataset and replacing them with stub comments, before requesting fill-in completion,
1060   for StarCoder 3b, the StarCoder2 family, and GPT-3.5. We observe an overall func rate@1of only
1061   89.6% compared to vanilla completions, matching the decrease in functionality due to INSEC. As
1062   developers are usually not willing to sacrifice functional correctness for security (He et al., 2024b),
1063
       and may get frustrated at the lack of steerability of the LLM, we suspect that straightforward removal
       is not a suitable defense.

1067   Static Analysis and Anomaly Detection While we evaluate the vulnerabilities in Section 5 using
1068   static analysis (GitHub, 2023; Singh & Aggarwal, 2022), it is not implied that static analysis could
1069   reliably prevent generation of insecure code by LLMs in the wild. First, INSEC can be extended to
1070   trigger unknown zero-day exploits or known, but difficult-to-identify vulnerabilities, thus remaining
1071
       undetected by common static analysis tools. This can be achieved through use of custom tooling
       or manual assessment for vulnerability judgment during attack string optimization, instead of static
1072
       analysis tools. Secondly, even for known and detectable CWEs, static analysis tools are rarely
1073
       configured appropriately (Charoenwet et al., 2024), suffer from poor explanations for discovered
1074   vulnerabilities (Nachtigall et al., 2019) and lack actionable advice for mitigation (Nachtigall et al.,
1075   2023). This results in static analysis being much less prevalent in practice than might be expected
1076   (Ryan et al., 2023), with Copilot-generated vulnerable code already being found in public GitHub
1077   repositories (Fu et al., 2023). Anomaly detection tools (Aragon, 2024; Aggarwal, 2017) are unlikely
1078   to pick up the subtle modifications caused by INSEC to code completions, and would need to
1079   monitor and discover individual prompts sent to the LLM to discover irregularities. We are therefore
       convinced that INSEC poses a realistic threat to code security.
