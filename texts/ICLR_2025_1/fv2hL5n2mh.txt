009                                               A BSTRACT
010
011            Automated Machine Learning (AutoML) approaches encompass traditional meth-
012            ods that optimize fixed pipelines for model selection and ensembling, as well
013            as newer LLM-based frameworks that autonomously build pipelines. While
014            LLM-based agents have shown promise in automating machine learning tasks,
015            they often generate low-diversity and suboptimal code, even after multiple itera-
016            tions. To overcome these limitations, we introduce Tree-Search Enhanced LLM
017            Agents (SELA), an innovative agent-based system that leverages Monte Carlo
018
               Tree Search (MCTS) to optimize the AutoML process. By representing pipeline
               configurations as trees, our framework enables agents to conduct experiments in-
019
               telligently and iteratively refine their strategies, facilitating a more effective ex-
020
               ploration of the machine learning solution space. This novel approach allows
021            SELA to discover optimal pathways based on experimental feedback, improving
022            the overall quality of the solutions. In an extensive evaluation across 20 machine
023            learning datasets, we compare the performance of traditional and agent-based Au-
024            toML methods, demonstrating that SELA achieves a win rate of 65% to 80%
025            against each baseline across all datasets. These results underscore the signifi-
026            cant potential of agent-based strategies in AutoML, offering a fresh perspective
027            on tackling complex machine learning challenges. The code will be open-sourced
028            upon publication.

      1   I NTRODUCTION
032
033
      Automated Machine Learning (AutoML) is a rapidly evolving field that seeks to automate the pro-
      cess of designing reliable machine learning solutions with minimal human intervention. Traditional
034
      AutoML frameworks, such as Auto-WEKA (Thornton et al., 2013), Auto-Sklearn (Feurer et al.,
035
      2015; 2020), AutoGluon (Tang et al., 2024b), and H2O AutoML (LeDell & Poirier, 2020), rely on
036   predefined search spaces and routines. These frameworks primarily focus on optimizing hyperpa-
037   rameters and model ensembling to find the best model configuration. However, this fixed and static
038   approach often lacks the adaptability needed to handle diverse and dynamic data scenarios, resulting
039   in suboptimal performance in more complex settings. Additionally, the traditional focus on model
040   training leaves other crucial stages of the machine learning pipeline, such as data preprocessing and
041   feature engineering, underexplored, thereby limiting the overall effectiveness of these systems.
042   Recently, large language model (LLM)-based agents have emerged as promising tools for automat-
043   ing machine learning tasks by leveraging natural language processing capabilities to generate code.
044   These systems typically begin with a natural language prompt describing the dataset and the prob-
045   lem, after which an LLM generates an end-to-end solution. Early efforts, such as Zhang et al.
046   (2024), experimented with prompting LLMs to generate machine learning solutions, while Hong
047   et al. (2024) introduced agents equipped with Hierarchical Graph Modeling and Programmable
048   Node Generation to address complex and dynamic workflows. Despite these advances, LLM-based
049   solutions often fall short in generating diverse and highly optimized workflows, as their search pro-
050
      cess remains limited to a single pass or trial. Without iterative refinement or the ability to explore
      alternative strategies, these solutions frequently converge on suboptimal results, even when multiple
051
      attempts are allowed.
052
053   A critical shortcoming of both traditional AutoML and LLM-based frameworks lies in their inability
      to mimic the nuanced problem-solving approach of human experts. When approaching a machine

079
      Inspired by this human-centered approach, we propose Tree-Search Enhanced LLM Agents
080
      (SELA) for automated machine learning, a novel framework that integrates the strengths of LLM
081   agents with a structured search and refinement process modeled on how experts solve machine learn-
082   ing problems. As illustrated in Figure 1, our framework combines the benefits of stage-wise plan-
083   ning, where each stage (e.g., Exploratory Data Analysis, Data Preprocessing, Feature Engineering,
084   and Model Training) is handled sequentially, with an iterative refinement mechanism. In SELA, the
085   search space of a machine learning problem is conceptualized as a tree, where each branch repre-
086   sents a potential solution path. This tree-based structure enables the agent to systematically explore
087   and refine solutions, much like an expert who tests and improves their strategy based on continuous
088   feedback.
089   To navigate this search space, we employ Monte Carlo Tree Search (MCTS) (Browne et al., 2012)
090   as the core decision-making engine, leveraging its ability to balance exploration (testing new strate-
091   gies) and exploitation (improving known good strategies). MCTS allows the agent to efficiently
092   explore large decision spaces, collect and process experimental results, and intelligently select the
093   next promising configuration to test. By iterating through this cycle of experimentation and refine-
094   ment, SELA incrementally improves its solutions, offering a more dynamic and flexible approach
095
      than static AutoML frameworks.
096   We rigorously evaluated SELA using 20 diverse datasets from the AutoML benchmark, comparing
097   its performance against both traditional AutoML systems and agent-based AutoML approaches. The
098   results demonstrate that SELA consistently delivers superior performance across a wide range of
099   machine learning tasks, validating its effectiveness and adaptability.
100   Our research makes the following contributions:
101
102          1. We introduce a novel approach that empowers LLM agents to address machine learning
103             challenges through an iterative, feedback-driven process. This mirrors the methodology of
104             human experts, enabling continuous exploration of various configurations and improving
105             outcomes through multiple rounds of refinement. This iterative exploration yields more
106
                diverse and optimized solutions than single-pass strategies.
107          2. We present a robust system that intelligently selects and executes experiments to generate
                high-performance pipelines. At the heart of this framework is the conceptualization of the

      Tree Search and Its Integration with LLMs Tree search algorithms have significantly advanced
121   problem-solving in artificial intelligence, with Monte Carlo Tree Search (MCTS) emerging as a
122   leading technique. These algorithms have been successfully applied across various domains, includ-
123   ing robotics (Best et al., 2019; Wu et al., 2015; Clary et al., 2018), chemistry (Segler et al., 2018),
124   and gaming (Silver et al., 2016; 2017), where MCTS is used to navigate vast solution spaces and
125   solve complex problems. More recently, research has focused on integrating tree search with Large
126   Language Models (LLMs) to enhance reasoning and decision-making. Studies such as Krishna-
127   murthy et al. (2024) and Dwaracherla et al. (2024) explored LLMs’ capacities for efficient explo-
128   ration, while Tang et al. (2024a) and Hui & Tu (2024) developed strategies for exploiting previously
129
      learned knowledge. Striking a balance between exploration and exploitation, Zhou et al. (2024) and
      Chi et al. (2024) applied MCTS for planning with external or self-evaluated feedback, while Feng
130
      et al. (2023); Wang et al. (2024) adapted AlphaZero-style tree search to LLM-based tasks. These
131
      advancements underscore the potential of combining tree search methods with LLMs, balancing
132   exploration of new solutions with exploitation of prior knowledge to enhance decision-making.
133
134   Advances and Limitations in AutoML Systems Automated Machine Learning (AutoML) frame-
135
      works were introduced to reduce the need for expert knowledge in designing machine learning
      pipelines. Early AutoML efforts, such as (Feurer et al., 2020; Jin et al., 2019; Olson & Moore,
136
      2016; Thornton et al., 2013), focused primarily on automating key pipeline components like hy-
137
      perparameter optimization, model selection, and ensembling. These frameworks achieved notable
138   progress by integrating meta-learning and hyperparameter search strategies to automatically select
139   and tune machine learning models. More recent AutoML systems, such as (Erickson et al., 2020)
140   and (LeDell & Poirier, 2020), employed ensembling techniques to further improve performance,
141   and extensions into multi-modal data settings (Tang et al., 2024b; Jin et al., 2023) have broadened
142   AutoML’s applicability.
143
      Recently, there has been growing interest in leveraging LLMs within AutoML systems to enhance
144   pipeline flexibility. Studies such as Hollmann et al. (2024) and Li et al. (2024) applied LLMs to
145   automate feature engineering, while Liu et al. (2024) introduced LLMs for hyperparameter tuning.
146   In addition, Luo et al. (2024) proposed embedding LLMs at each stage of the machine learning
147   workflow. Despite these advancements, traditional AutoML systems remain constrained by rigid
148   pipelines and limited flexibility to adapt to unique datasets or specific task requirements.
149
      LLM Agents for Dynamic Machine Learning Pipelines In contrast to static pipelines, LLM-based
150   agents offer a more dynamic solution for addressing complex machine learning challenges. Hong
151   et al. (2024) introduced an LLM agent with hierarchical graph modeling and programmable node
152   generation, enabling the creation of sophisticated, adaptable pipelines for diverse data scenarios.
153   Similarly, Zhang et al. (2024) demonstrated that LLMs could effectively interpret structured inputs
154   and apply past experiences to solve new machine learning tasks. Guo et al. (2024) expanded on this
155   by introducing a data science agent that leverages case-based reasoning; however, it faces challenges
156   when generating solutions from scratch due to its reliance on existing codebases. Schmidt et al.
157   (2024) proposed an iterative approach, where the entire pipeline is generated in one step and refined
158
      iteratively through incremental modifications.
159   Building on these efforts, SELA introduces an agent that integrates the strengths of both
160   approaches—stage-wise planning and iterative refinement—allowing it to autonomously explore
161   and generate machine learning solutions from the ground up. This approach offers greater flexibility
      and control during the search process, enabling the generation of optimized solutions at each stage.

188   As illustrated in Figure 2, SELA consists of three key components: an LLM-based insight proposer,
189
      a search module using MCTS, and an LLM agent as the experiment executor. First, the LLM
      generates insights from the problem description and dataset, defining a search space. The search
190
      module then organizes this space into a tree structure and uses MCTS to explore promising paths.
191
      During each cycle, the selected path is passed to the LLM agent, which translates the configuration
192   into an executable pipeline. The agent plans, codes, and executes the experiment, feeding the results
193   back to refine future searches. This iterative process continues until the termination criterion is met.
194   The following sections provide a detailed explanation of each component.
195
196   3.1      I NSIGHT P ROPOSAL AND S EARCH S PACE C REATION
197
198   To enable SELA to explore a wide range of machine learning strategies, we introduce an insight
199   proposer that generates diverse methods tailored to different stages of the machine learning work-
200   flow. Each proposed insight suggests either a single technique or a combination of methods aimed
201   at enhancing performance. For instance, a feature engineering insight might recommend creating
202
      interaction features from existing variables, while a model training insight could propose a specific
      algorithm or suggest running a grid search to improve accuracy.
203
204   The insight proposer takes as input the problem description p and dataset information d, such as
205   metadata and sample records, and generates m insights λ for each stage of the machine learning
206   process using a large language model M . These insights are stored in an insight pool, forming a
207   search space Λ for SELA to explore. We decompose the machine learning process into five stages:
208
      Exploratory Data Analysis (τ1 ), Data Preprocessing (τ2 ), Feature Engineering (τ3 ), Model Training
      (τ4 ), and Model Evaluation (τ5 ). For simplicity, we denote the entire set of stages as T and refer to
209
      any specific stage as τ .

      In order to systematically explore the different configurations in machine learning experiments, we
232   model the search space as a hierarchical tree. This structure allows us to apply tree search algorithms,
233   where each path through the tree represents a different experiment configuration. Algorithm 1 also
234   provides an overview of this searching process.

      3.3.1    E XPERIMENT N ODE
238
      To facilitate the exploration of various strategies, we model the proposed search space as a hierar-
239
      chical tree that is well-suited for applying search algorithms. Each node in the tree, denoted as x,
240   represents one insight λ in the search space Λ and contains the following attributes:
241
242
              • Insight λ(x): Represents the specific insight λτi ∈ Λ associated with this node, where τ
243
                denotes the stage of the machine learning pipeline.
244
245           • Depth δ(x): Indicates the stage of the machine learning process the node corresponds to
246             (e.g., depth 1 might represent data preprocessing, depth 2 for feature engineering, and depth
247             3 for model training).
248
249           • Value v(x): The cumulative score from simulations for this node and all its descendants.
250
              • Number of Visits nvisits (x): The total number of simulations conducted for this node and
251
                its descendants.
252
253           • Simulation Score s(x): The score for simulating this node.
254
255           • Solution Code σsol (x) The final code produced after the node simulation.
256
257           • Stage Code σstage (x): The code generated up to the node’s current stage, a part of the
258
                solution code
259
260   By modeling the search space as a tree, each path from the root to a node x represents an experiment
261   configuration c(x) = {λ(x1 ), λ(x2 ), . . . , λ(x)} ⊂ Λ, where x1 , x2 , . . . , x are nodes along the path.
262   The task of finding the optimal solution can therefore be viewed as a path search within the tree,
263
      where each path corresponds to a potential configuration of the experiment.

      Simulation Once expanded, a node xsample is randomly sampled from Xchild for simulation. The path

331   opment set) is retrieved and backpropagated through the tree. The score is propagated from the
332   simulated node up to the root, updating each parent node’s value and visit count. This allows nodes

334   code is also backpropagated up to the tree, and it can be processed and saved as stage code depending
335   on the parent node during the update.
336

      3.3.3    E XPERIMENT S TATE S AVING AND L OADING
340
341   To boost execution efficiency, SELA implements fine-grained code reuse by caching code at the
342   stage level. This caching is done according to each attempted configuration c, allowing the frame-
343   work to reuse as much saved code as possible if the incoming configuration cnew shares any part with
344   existing ones.
345
      Given that LLMs produce non-deterministic outputs, the same instruction can yield different code,
346
      leading to greater variance in final performance. To minimize this variance and reduce token us-
347   age during execution, SELA saves and loads the stage code for each node. Whenever a node is
348   chosen for execution, the experimenter reruns the saved stage code, ensuring consistency before
349   progressing to the next stage. This approach effectively conserves resources while maintaining ro-
350   bust performance across stages. In Appendix D, we examine the cost efficiency of this state-saving
351   and loading mechanism.

504   5   C ONCLUSION
505
506   In this paper, we introduced SELA, a novel framework that integrates LLM-based agents with
507   Monte Carlo Tree Search (MCTS) to automate machine learning workflows. Our experimental
508   results, conducted on 20 machine learning datasets, demonstrate SELA’s effectiveness and high-
509   light its distinct advantages over both traditional AutoML frameworks and existing LLM-based ap-
510   proaches. The proposed methodology is not limited to machine learning but could be adapted to a
511   wide range of sequential decision-making problems, provided they can be represented as tree struc-
512   tures with scalar rewards derived from their leaf nodes. Looking ahead, future work could explore
513
      extending this framework to other domains, including software engineering, scientific discovery,
      game playing, and robotics. Furthermore, improving the efficiency and scalability of the tree search
514
      process for larger solution spaces remains an important area for investigation. Another promising
515
      direction is developing techniques to provide interpretable explanations of the search process and
516   solution rationale, enhancing the transparency and trustworthiness of the system. SELA represents
517   a significant advancement in automated machine learning, demonstrating the potential of combining
518   traditional search algorithms with the flexibility of LLMs.

           The MCTS process in this case study consists of a structured exploration of the machine learning
1153
           pipeline, executed in the following steps:
1154
1155       Step 1: Initialization (Node 0)
1156       The process begins by defining high-level tasks, such as data analysis, data pre-processing, fea-
1157       ture engineering, and model training. These general steps establish the overall framework for the
1158
           machine learning workflow.
1159       Step 2: Feature Engineering Exploration (Selection and Expansion)
1160       MCTS then explores specific feature engineering techniques. For instance, Node 0-0 introduces
1161       features like the magnitude of vectorial velocities, Node 0-1 generates time-based features, and
1162       Node 0-2 creates spatial relationship features between body parts. These feature engineer methods
1163       aim to improve data representation, which is crucial for enhancing model accuracy.
1164       Step 3: Model Training (Expansion)
1165       At this point, the process tests various machine learning models. For example, Node 0-1-1 applies
1166       a Support Vector Machine (SVM) with a radial basis function (RBF) kernel, while Node 0-0-2
1167       evaluates a Neural Network. The models are trained and evaluated based on performance across
1168       training, development, and test datasets.
1169       Step 4: Performance Evaluation (Simulation)
1170       Each node is scored based on model performance. MCTS retains and further explores the best-
1171       performing nodes, using prior successful results to guide the search for improved solutions.
1172
           Step 5: Nodes Update (Backpropagation)
1173
           After the simulation, the performance score is retrieved and backpropagated through the tree. For
1174       example, after simulating Node 0-1-1, MCTS backpropagates the result up the tree, updating parent
1175       nodes like Node 0-1 and Node 0.
1176
1177
           Step 6: Best Model Selection
           In the final step, MCTS selects the best-performing solution. In this case, Node 0-1-1, using the
1178
           SVM with RBF kernel, achieved the highest scores across datasets, effectively combining feature
1179
           engineering and model selection to optimize the machine learning pipeline.
