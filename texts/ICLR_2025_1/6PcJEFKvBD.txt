011                                              A BSTRACT
012
013            offline rl ope is a fully unit tested and runtime type checked Python package for
014            performing off-policy evaluation of offline RL models. offline rl ope has been
015
               designed for OPE workflows using real world data by: naturally handling uneven
               trajectory lengths; including novel convergence metrics which do not rely on OPE
016
               estimator ground truths; and providing a compute and data efficient API which
017
               can be integrated with many offline RL frameworks. This paper motivates and
018            describes the core API design and functionality to enable ease of use and exten-
019            sion. The implementations of OPE methods have been benchmarked against exist-
020            ing implementations to ensure consistency and reproducibility. The offline rl ope
021            source code can be found on GitHub at: REDACTED
022
023
024   1   I NTRODUCTION
025
026   Offline RL enables MDPs to be solved without interaction with an environment (i.e., with only a
027   logged (batch) dataset) and has grown in popularity recently due to the availability of such data and
028
      the challenges of performing environment interactions in high stakes settings (Levine et al., 2020).
      A core challenge however, when environment interaction is not possible (fully offline RL (ORL)) is
029
      off-policy evaluation (OPE). OPE refers to evaluating a hypothetical target policy, πe with access to
030
      trajectories generated according to an alternate policy, πβ . Since performing off-policy evaluations
031   is inherently counterfactual, OPE must be performed carefully and is still an active research area.
032
033
034
      There still does not exist a well established and tested code base for performing OPE. Such a
      codebase, which is agnostic to the implementation of the policy learning algorithm, would be
035
      beneficial to ensure reproducibility and transparency in the application of OPE. This ambiguity
036
      in the application of OPE estimators has been rooted in the non-uniqueness of estimators (e.g.,
037   weighted Per-Decision proposed by Precup et al. (2000) and Kallus & Uehara (2019)) and the use
038   of custom implementations of estimators without declaration of changes (e.g., value clipping by
039   Raghu et al. (2017)).
040
041
      offline rl ope, is a unit tested and runtime type-checked Python library for performing off-policy
042
      evaluation on real world data, which is agnostic to the framework used for training ORL models.
043   Specifically, the contributions are as follows:
044
045          • Developed for use with real world data: handles uneven trajectory lengths (section 3.1);
046            and includes offline OPE evaluation metrics (section 3.3.3) and common techniques (e.g.,
047            clipping) for importance sampling (IS) and doubly robust (DR) estimators (section 3.1);
048          • An API that can be easily extended for research purposes or used as plug-and-play;
049          • Optionally integrates with d3rlpy (Seno & Imai, 2021) for train-time evaluations (as a posed
050            to post-hoc).
051
052   The focus for the first release of offline rl ope has been to implement standard techniques for IS and
053   DR in a flexible and efficient API. The first release has focused primarily on IS and DR estimates
      due to the existence of a strong implementations of FQE by Seno & Imai (2021). More advanced

054
      IS methods (i.e., marginal state IS), model-based methods, composite methods, direct methods and
055   efficient influence function methods will be included in future releases. The package code base is
056   available at: REDACTED

      4.2    C ONTINUOUS ACTION SPACES
402
403   With respect to continuous action spaces, offline rl ope and Scope-RL differed significantly in their
404   approach and as such, could not be compared against one another. To demonstrate the efficacy
405   of the offline rl ope implementation for continuous actions spaces, the relative ranking2 of 3 poli-
406   cies were compared against the ground truth evaluations using the Pendulum environment (Towers
407   et al. (2024)). In addition to a number of other expected observations, table 6 suggests that broadly
408   speaking, estimators implemented in offline rl ope were able to accurately rank the performance
409   of policies against the ground truth performance, demonstrating the efficacy of implementation. In
410   addition, expected observations included:
411
              • Pure PD estimators benefited the most from weight clipping since the bias of doubly robust
412
                methods was already being controlled through the reward approximation;
413
              • The pure PD estimator demonstrated the worse correlation due to the high variance of the
414
                estimator;
415
416           • Combining the FQE method from d3rlpy with the DR and WDR methods improved the
417
                ranking performance, despite all FQE models converging reasonably well and a reasonable

604   A     OFFLINE RL OPE /S COPE -RL DISCUSSION
605
606   Table 1 provides an overview of the differentiating factors between offline rl ope and Scope-RL
607   however, these are discussed in greater detail below. The primary difference is ”audience” which is
608
      a result of a number of features being included in offline rl ope, not included in Scope-RL.
609
610
611   A.1   AUDIENCE
612
      Uneven trajectory lengths The Scope-RL API requires trajectories to be a constant length, an
613
      assumption which is certainly not always satisfied in real world applications. Whilst trajectories
614
      could be padded prior to running OPE, this would render the calculation of self-normalised
615   estimates incorrect. In comparison, offline rl ope can handle trajectories of differing lengths and
616   does not require the user to perform any pre-processing in order to do so (see section 3.1 for a
617   further explanation).
618
619
620   Generic workflow specifically for real world data In addition to supporting uneven trajectory
621   lengths, the offline rl ope API enables real world applications in several other ways. offline rl ope
622   supports arbitrary evaluation and behaviour policies through the Policy class (section 3.2) whilst
623   providing functionality to trivially define Pytorch and sklearn behaviour estimators through the
624   PropensityModels API (section 3.3.1). In contrast, Scope-RL is deeply integrated with d3rlpy
625   and does not provide any functionality for defining behaviour propensity estimates outside of the
626
      generation of synthetic datasets. Additionally, the OPE evaluation metrics included in Scope-RL
      require an oracle measure of performance. In contrast, offline rl ope focuses on defining a workflow
627
      for real world data by introducing metrics (section 3.3.3) such as VWP and enabling stages of the
628
      OPE pipeline to be easily cached for debugging and post-hoc analysis.
629
630
631
      Off-policy selection with d3rlpy Despite having a generic API, offline rl ope ships, natively,
632   with a deep integration with d3rlpy. In particular, offline rl ope enables OPE metrics to be run
633   during training (see section 3.3.2) thus enabling early stopping to be performed.
634
635
636   A.2   E STIMATORS , API AND IMPLEMENTATION
637
638   In addition to audience, Scope-RL differs in the number of other areas. The number of esti-
639   mators currently implemented within Scope-RL far exceeds that of offline rl ope in particular,
640   state marginal IS estimators, double reinforcement learning and the DICE family of estimators.
641   Whilst these estimators will be implemented within offline rl ope in the future, users looking for
642   implementations as of the time of writing this document are referred to Scope-RL.
643
644   The offline rl ope API utilises equation 1 to enable an array of IS based estimators to be imple-
645   mented, relying on significant class inheritance. This has resulted in an API which is easy to extend
646   and maintain and is in contrast to the individually defined discrete and continuous estimators within
647   the Scope-RL API. The implementation deatils of this API are discussed in section 3.1.



                                                       12
      Under review as a conference paper at ICLR 2025


648
      The final difference between Scope-RL and offline rl ope is the handling of continuous action
649   spaces. offline rl ope can only (reasonably) evaluate stochastic policies learnt over a continuous
650   action spaces whilst, in Scope-RL, all policies over continuous actions spaces are evaluated through
651   kernel smoothing of actions (Kallus & Zhou (2018)).

658

659   for an estimator to be doubly robust, certain bias conditions must also be met. Generally speaking
660
      DR estimators incorporate a (direct) approximation of the policy value under πe i.e., Q̂πe (s, a) in
661   addition to the IS estimates. In contrast, direct methods (DM) solely utilise the aforementioned

663

      Within ORL, the FQE DM method (Le et al. (2019)) is often cited as a ”go-to” method for
665

666
      that the selection of OPE estimator is problem specific. Theoretically, the decision of whether to
667   use an IS estimator or a direct method estimator is a function of the complexity of the propensity
668   and reward (outcome) model, respectively (Alaa & van der Schaar (2018)). Voloshin et al. (2021)
669   observed this empirically as well as noting additional factors such as: evaluation policy/behaviour
670   policy mass-match and horizon length.
671
672
      There also exists practical differences in estimators, in terms of complexity of hyperparameter
673   tuning and computation time. For pure IS methods, the aforementioned complexities are a result of
674   the behaviour policy estimation and as such, need to only be performed once per behaviour policy
675   (rather than per evaluation policy as in the case of direct and DR methods), pure IS methods are
676   more suited for rapid model experimentation. Tang & Wiens (2021) leveraged this observation,
677   proposing a two stage model development pipeline, where IS methods are used for initial model
678   assessment.
679
680   The above observations clearly motivate the development of a robust code base for performing a
681   range of OPE estimation.
