                                                 A BSTRACT
008
009
               The recent advancements of large vision language models shows their great poten-
010
               tial in driving the agent system operating on user interfaces. However, we argue
011            that the power multimodal models like GPT-4V as a general agent on multiple
012            operating systems across different applications is largely underestimated due to
013            the lack of a robust screen parsing technique capable of: 1) reliably identifying
014            interactable icons within the user interface, and 2) understanding the semantics of
015            various elements in a screenshot and accurately associate the intended action with
016            the corresponding region on the screen. To fill these gaps, we introduce O MNI -
017            PARSER, a comprehensive method for parsing general user interface screenshots
018            into structured elements, which significantly enhances the ability of GPT-4V to
019
               generate actions that can be accurately grounded in the corresponding regions of
               the interface. We first curated an interactable icon detection dataset using popular
020
               webpages and an icon description dataset. These datasets were utilized to fine-tune
021
               specialized models: a detection model to parse interactable regions on the screen
022            and a caption model to extract the functional semantics of the detected elements.
023            O MNI PARSER significantly improves GPT-4V’s performance on ScreenSpot bench-
024            mark. And on Mind2Web and AITW benchmark, O MNI PARSER with screenshot
025            only input outperforms the GPT-4V baselines requiring additional information
026            outside of screenshot. We further demonstrate that O MNI PARSER can seamlessly
027            integrate with other vision language models, significantly enhancing their agentic
028            capabilities.

      1   I NTRODUCTION
032
033
      Large language models have shown great success in their understanding and reasoning capabilities.
      More recent works have explored the use of large vision-language models (VLMs) as agents to
034
      perform complex tasks on the user interface (UI) with the aim of completing tedious tasks to replace
035
      human efforts (YZL+ 23; YYZ+ 23; DGZ+ 23; ZGK+ 24; HWL+ 23; YZS+ 24; WXJ+ 24; GFH+ 24;
036   CSC+ 24). Despite the promising results, there remains a significant gap between current state-of-the-
037   arts and creating widely usable agents that can work across multiple platforms, e.g. Windows/MacOS,
038   IOS/Android and multiple applications (Web broswer Office365, PhotoShop, Adobe), with most
039   previous work focusing on limiting applications or platforms.
040
      While large multimodal models like GPT-4V and other models trained on UI data (HWL+ 23;
041
      YZS+ 24; CSC+ 24) have demonstrated abilities to understand basic elements of the UI screenshot,
042   action grounding remains one of the key challenges in converting the actions predicted by LLMs
043   to the actual actions on screen in terms of keyboard/mouse movement or API call (ZGK+ 24). It
044   has been noted that GPT-4V is unable to produce the exact x-y coordinate of the button location,
045   Set-of-Mark prompting (YZL+ 23) proposes to overlay a group of bounding boxes each with unique
046   numeric IDs on to the original image, as a visual prompt sent to the GPT-4V model. By applying
047   set-of-marks prompting, GPT-4V is able to ground the action into a specific bounding box which has
048   ground truth location instead of a specific xy coordinate value, which greatly improves the robustness
049   of the action grounding (ZGK+ 24). However, the current solutions using SoM relies on parsed
050
      HTML information to extract bounding boxes for actionable elements such as buttons, which limits
      its usage to web browsing tasks. We aim to build a general approach that works on a variety of
051
      platforms and applications.
052
053   In this work, we argue that previous pure vision-based screen parsing techniques are not satisfactory,
      which lead to significant underestimation of GPT-4V model’s understanding capabilities. And a

054
      reliable vision-based screen parsing method that works well on general user interface is a key to
055   improve the robustness of the agentic workflow on various operating systems and applications. We
056   present O MNI PARSER, a general screen parsing tool to extract information from UI screenshot into
057   structured bounding box and labels which enhances GPT-4V’s performance in action prediction in a
058   variety of user tasks.
059
      We summarize our contributions as follows:
060
061              • We curate a interactable region detection dataset using bounding boxes extracted from DOM
062                tree of popular webpages.
063
                 • We propose OmniParser, a pure vision-based user interface screen parsing method that
064                combines multiple finetuned models for better screen understanding and easier grounded
065                action generation.
066
                 • We evaluate OmniParser on the ScreenSpot, Mind2Web, and AITW benchmarks, demon-
067
                   strating significant improvement over the GPT-4V baseline, using only screenshots as input.
068
069
                 • We show that O MNI PARSER operates seamlessly and serves as an easy-to-integrate tool for
                   a variety of state-of-the-art public vision language models. And we open-source both our
070
                   code and model to facilitate further development and integration of O MNI PARSER to other
071

076   2.1       UI S CREEN U NDERSTANDING
077
      There has been a line of modeling works focusing on detailed understanding of UI screens,
078
      such as Screen2Words (WLZ+ 21), UI-BERT (BZX+ 21), WidgetCaptioning (LLH+ 20), Action-
079
      BERT (HSZ+ 21). These works demonstrated effective usage of multimodal models for extracting
080   semantics of user screen. But these models rely on additional information such as view hierarchy, or
081   are trained for visual question answering tasks or screen summary tasks.
082
083
      There are also a couple publicly available datasets on UI screen understanding. Most notably the
      Rico dataset (DHF+ 17), which contains more than 66k unique UI screens and its view hierarchies.
084
      Later (SWL+ 22) auguments Rico by providing 500k human annotations on the original 66k RICO
085
      screens identifying various icons based on their shapes and semantics, and associations between
086   selected general UI elements (like icons, form fields, radio buttons, text inputs) and their text labels.
087   Same on mobile platform, PixelHelp (LHZ+ 20) provides a dataset that contains UI elements of
088   screen spanning across 88 common tasks. In the same paper they also released RicoSCA which
089   is a cleaned version of Rico. For the web and general OS domain, there are several works such
090   Mind2Web (DGZ+ 23), MiniWob++(LGP+ 18), Visual-WebArena (KLJ+ 24; ZXZ+ 24), and OS-
091   World (XZC+ 24) that provide simulated environment, but does not provide dataset explicitly for
092   general screen understanding tasks such as interactable icon detection on real world websites.
093   To address the absence of a large-scale, general web UI understanding dataset, and to keep pace with
094   the rapid evolution of UI design, we curated an icon detection dataset using the DOM information
095   from popular URLs avaialbe on the Web. This dataset features the up-to-date design of icons and
096   buttons, with their bounding boxes retrieved from the DOM tree, providing ground truth locations.

105   and agents in (YYZ+ 23; WXY+ 24; RLR+ 23) for mobile domain. These work often leverages
106   the DOM information in web browser, or the view hierarchies in mobile apps to get the ground

117   into several steps of state-action pairs (S0 , A0 ), ..., (Sn , An ), where n is the number of steps. Each
118   step requires the model’s (e.g. GPT-4V) ability to: 1) understand the state of the UI screen information
119   Si in the current step, i.e. analyzing what is the screen content overall, what are the functions of
120   detected icons that are labeled with numeric ID, and 2) predict what is the next action Ai on the
121   current screen that is likely to help completing the whole task. Normally this can be formulated as:
122

126   Here Si encapsules the current information of the screen, which varies according to the model
127   π’s input types and the operating environment. For example, in the web tasks (ZGK+ 24), Si =
128   {hi , Imgi }, where hi is the HTML information at step i, and Imgi is the screenshot of the GUI at step
129
      i. In Android tasks (RLR+ 23), hi is the view hierarchy information. In these cases, the model π is
      required to extract information directly from Imgi and at the same time generate an action prediction.
130

131

132   can leverages more information from the parsed screen and focus more on the action prediction.

135
      detection model, a finetuned icon description model, and an OCR module. This combination produces
      a structured state informaton Si , which includes a DOM-like representation of the UI and a screenshot
136
      overlaid with bounding boxes for potential interactable elements

146   3.1   I NTERACTABLE R EGION D ETECTION
147
148   Identifying interactable regions from the UI screen is a crucial step to reason about what actions
149   should be performed given a user tasks. Instead of directly prompting GPT-4V to predict the
150   xy coordinate value of the screen that it should operate on, we follow previous works to use the
151   Set-of-Marks approach (YZL+ 23) to overlay bounding boxes of interactable icons on top of UI
152
      screenshot, and ask GPT-4V to generate the bounding box ID to perform action on. However,
      different from (ZGK+ 24; KLJ+ 24) which uses the ground truth button location retrieved from
153
      DOM tree in web browswer, and (YYZ+ 23) which uses labeled bounding boxes in the AITW
154
      dataset (RLR+ 23), we finetune a detection model to extract interactable icons/buttons.
155
156   Specifically, we curate a dataset of interactable icon detection dataset, containing 67k unique screen-
157   shot images, each labeled with bounding boxes of interactable icons derived from DOM tree. We
158
      first took a 100k uniform sample of popular publicly availabe urls on the web (OXL+ 22), and collect
      bounding boxes of interactable regions of the webpage from the DOM tree of each urls. Some
159
      examples of the webpage and the interactable regions are shown in 2.
160
161   Apart from interactable region detection, we also have a OCR module to extract bounding boxes of
      texts. Then we merge the bounding boxes from OCR detection module and icon detection module

      while removing the boxes that have high overlap (we use 90% as a threshold). For every bounding
163   box, we label it with a unique ID next to it using a simple algorithm to minimizing the overlap
164   between numeric labels and other bounding boxes.

      3.2   I NCORPORATING L OCAL S EMANTICS OF F UNCTIONALITY
167
168   We found in a lot of cases where only inputting the UI screenshot overlayed with bounding boxes
169   and associated IDs can be misleading to GPT-4V. We argue the limitation stems from GPT-4V’s
170   constrained ability to simultaneously perform the composite tasks of identifying each icon’s semantic
171   information and predicting the next action on a specific icon box. This has also been observed by
172   several other works (YYZ+ 23; ZGK+ 24). To address this issue, we incorporate the local semantics
173   of functionality into the prompt, i.e. for each icons detected by the interactable region detection
174
      model, we use a finetuned model to generate description of functionality to the icons, and for each
      text boxes, we use the detected texts and its label.
175
176   We perform more detailed analysis for this topic in section 4.1. To the best of our knowledge, there is
177   no public model that is specifically trained for up-to-date UI icon description, and is suitable for our
178   purpose to provide fast and accurate local semantics for the UI screenshot. Therefore we curate a
179   dataset of 7k icon-description pairs using GPT-4o, and finetune a BLIP-v2 model (LLSH23) on this
180
      dataset. Details of dataset and training can be found in Appendix 8.1. After finetuning, we found the
      model is much more reliable in its description to common app icons. Examples can be seen in figure
181
      5. And in figure 3, we show it is helpful to incorporate the semantics of local bounding boxes in the
182
      form of text prompt along with the UI screenshot visual prompt.

      4.1   E VALUATION ON S EE A SSIGN TASK
194
      To test the ability of correctly predicting the label ID given the description of the bounding boxes for
195   GPT-4v models, We handcrafted a dataset SeeAssign that contains 112 tasks consisting of samples
196   from 3 different platforms: Mobile, Desktop and Web Browser. Each task includes a concise task
197   description and a screenshot image. The task descriptions are manually created and we make sure
198   each task refers to one of the detected bounding boxes, e.g. ’click on ’settings”, ’click on the
199   minimize button’. During evaluation, GPT-4V is prompted to predict the bounding box ID associated
200   to it. Detailed prompt are specified in Appendix. The task screenshot images are sampled from the
201   ScreenSpot (CSC+ 24) benchmark, where they are labeled with set of marks using O MNI PARSER.
202   The tasks are further divided into 3 sub-categories by difficulty: easy (less than 10 bounding boxes),
203
      medium (10-40 bounding boxes) and hard (more than 40 bounding boxes).
204   From table 1, we see that GPT-4V often mistakenly assign the numeric ID to the table especially
205   when there are a lot of bounding boxes over the screen. And by adding local semantics including
206   texts within the boxes and short descriptions of the detected icons, GPT-4V’s ability of correctly
207   assigning the icon improves from 0.705 to 0.938.
208   From figure 3, we see that without the description of the referred icon in the task, GPT-4V often
209   fails to link the icon required in the task and the ground truth icon ID in the SoM labeled screenshot,
210   which leads to hallucination in the response. With fine-grain local semantics added in the text prompt,
211   it makes it much easier for GPT-4V to find the correct icon ID for the referred icon.

467   6     D ISCUSSIONS
468
469   In this section, we discuss a couple of common failure cases of O MNI PARSER with examples and
470
      potential approach to improve.
471   Repeated Icons/Texts From analysis of the the GPT-4V’s response log, we found that GPT-4V
472   often fails to make the correct prediction when the results of the O MNI PARSER contains multiple
473   repeated icons/texts, which will lead to failure if the user task requires clicking on one of the buttons.
474   This is illustrated by the figure 8 (Left) in the Appendix. A potential solution to this is to add finer
475   grain descriptions to the repeated elements in the UI screenshot, so that the GPT-4V is aware of the
476
      existence of repeated elements and take it into account when predicting next action.
477   Corase Prediction of Bounding Boxes One common failure case of O MNI PARSER is that it fails
478   to detect the bounding boxes with correct granularity. In figure 8 (Right), the task is to click on the
479   text ’MORE’. The OCR module of O MNI PARSER detects text bounding box 8 which encompass
480   the desired text. But since it uses center of the box as predicted click point, it falls outside of the
481   ground truth bounding box. This is essentially due to the fact that the OCR module we use does
482
      not have a notion of which text region are hyperlink and clickable. Hence we plan to train a model
      that combines OCR and interactable region detection into one module so that it can better detect the
483
      clickable text/hyperlinks.
484
485   Icon Misinterpretation We found that in some cases the icon with similar shape can have different
      meanings depending on the UI screenshot. For example, in figure 9, the task is to find button related


                                                                      9
      Under review as a conference paper at ICLR 2025


486
      to ’More information’, where the ground truth is to click the three dots icon in the upper right part
487   of the screenshot. O MNI PARSER successfully detects all the relevant bounding boxes, but the icon
488   description model interpret it as: "a loading or buffering indicator". We think this is due to the fact
489   that the icon description model is only able to see each icon cropped from image, while not able to
490   see the whole picture during both training and inference. So without knowing the full context of the
491   image, a symbol of three dots can indeed mean loading buffer in other scenarios. A potential fix to
492   this is to train an icon description model that is aware of the full context of the image.
493
494   7   C ONCLUSION
495
496   In this paper, We propose O MNI PARSER, a general vision only approach that parse UI screenshots
497   into structured elements. O MNI PARSER encompasses two finetuned models: an icon detection model
498   and a functional description models. To train them, we curated an interactable region detection
499   dataset using popular webpages, and an icon functional description dataset. We demonstrate that with
500   the parsed results, the performance of GPT-4V is greatly improved on ScreenSpot benchmarks. It
501   achieves better performance compared to GPT-4V agent that uses HTML extracted information on
502   Mind2Web, and outperforms GPT-4V augmented with specialized Android icon detection model on
503   AITW benchmark. We hope O MNI PARSER can serve as a general and easy-to-use tool that has the
504
      capability to parse general user screen across both PC and mobile platforms without any dependency
      on extra information such as HTML and view hierarchy in Android.

758   8.1     D ETAILS OF I CON -D ESCRIPTION DATASET
759
760   In figure 5, we see that the original BLIP-2 model tend to focus on describing shapes and colors
761   of app icons, while struggling to recognize the semantics of the icon. This motivates us to finetune
762   this model on an icon description dataset. For the dataset, we use the result of parsed icon bounding
763   boxes inferenced by the interactable icon detection model on the ScreenSpot dataset since it contains
764   screenshots on both mobile and PC. For the description, we ask GPT-4o whether the object presented
765
      in the parsed bounding box is an app icon. If GPT-4o decides the image is an icon, it outputs one-
      sentence description of the icon about the potential functionality. And if not, GPT-4o will output ’this
766
      is not an icon’, while still including this in the dataset. In the end, we collected 7185 icon-description
767
      pairs for finetuning.
768
769   We manually inspected the dataset to include icon-description pairs across a wide range of functions.
770   These icons include system icons and popular software/app icons. After deduplication, we have 174
771
      app icons in the pc platform, and 170 app icons in mobile platform. Further, we leveraged GPT-4o
      and conducted an analysis of the distribution of the icons. We first summarize the icons into the
772
      following types:
773
774           1. Functional Icons. These icons represent actions or functionalities users can perform.
775              Subcategory:
776                 • Navigation Icons: Back, forward, home, refresh, menu.
777                 • Action Icons: Add (+), delete (trash), edit (pencil), search (magnifying glass), share,
778                   upload, download.
779                 • System Actions: Lock, log out, power off, settings (gear icon).
780           2. Informational Icons. These icons convey information or statuses. Examples include:
781              Subcategory:
782                 • Notification Icons: Alerts, messages, updates.
783                 • Status Icons: Battery level, network signal, Wi-Fi, Bluetooth, processing/loading
784                   (spinner).
785                 • Error or Warning Icons: Exclamation marks, red crosses, or triangles.
786
              3. App-Specific Icons. These icons are unique to a specific app or service.
787
              4. Media Control Icons. These icons control media playback. Examples include play, pause,
788
                 stop, fast forward, rewind, volume up, volume down.
789
790   The detail distribution of each sub-category is presented in figure 4. With this dataset, the resulting
791   model demonstrates strong generalization on varied benchmarks and real-world applications.
792
      We finetune BLIP-2 model for 1 epoch on the generated dataset with constant learning rate of 1e−5 ,
793   no weight decay and Adam optimizer. We show a few of the qualitative examples of finetuned model
794   vs the original model in figure 5.
