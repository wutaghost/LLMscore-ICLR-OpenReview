009                                                                                                                    A BSTRACT
010
011                      Neurosymbolic learning has emerged as a promising paradigm to incorporate sym-
012                      bolic reasoning into deep learning models. However, existing frameworks are
013                      limited in scalability with respect to both the training data and the complexity of
014                      symbolic programs. We propose D OLPHIN, a framework to scale neurosymbolic
015                      learning at a fundamental level by mapping both forward chaining and backward
016                      gradient propagation in symbolic programs to vectorized computations. For this
017                      purpose, D OLPHIN introduces a set of abstractions and primitives directly on top
018
                         of a high-performance deep learning framework like PyTorch. It thereby enables
                         neurosymbolic programs to be written in a language like Python that is familiar to
019
                         developers and compile them to computation graphs that are amenable to end-to-
020
                         end differentiation on GPUs. We evaluate D OLPHIN on a suite of 13 benchmarks
021                      across 5 tasks that combine deep learning models for text, image, or video pro-
022                      cessing with symbolic programs that involve multi-hop reasoning, recursion, and
023                      black-box functions like Python eval(). D OLPHIN achieves comparable or bet-
024                      ter accuracy on all benchmarks while taking 0.3%-61.7% of the time (and 23.2%
025                      on average) to train these models on the largest input per task compared to base-
026                      lines Scallop, ISED, and IndeCateR+, which time out on most of these inputs.

      1       I NTRODUCTION
029
030
      Deep learning has made great strides in tasks such as image classification, speech recognition, and
031
      natural language processing. With the emergence of foundation models like GPT-4 and CLIP, deep
032   learning is increasingly applied to more complex tasks. While such models work well for prediction
033   and generation tasks, they are limited in their ability to perform reasoning required for tasks involv-
034   ing structure, logic, and planning, where symbolic approaches traditionally excel (Kambhampati
035   et al.). Neurosymbolic programming (Chaudhuri et al., 2021) has emerged as a promising paradigm
036   to incorporate symbolic reasoning into deep learning models, providing the best of both worlds.
037
      Various frameworks have been developed to improve the programmability and accessibility of neu-
038   rosymbolic applications (Manhaeve et al., 2018; Li et al., 2023; Solko-Breslin et al., 2024). These
039   frameworks support complex symbolic reasoning features like recursion and black-box functions,
040   implement efficient differentiable reasoning algorithms, and provide bindings for deep learning
041   frameworks like PyTorch. However, these frameworks incur significant overhead during training.
042
      Consider a typical workflow of such a framework in Figure 1(a). We have a supervised learning
043
      task with labeled data (x, y), a neural network M✓ that processes input x, and a symbolic program
044   Psymbolic that takes the network’s output r and produces final output y. Existing frameworks, such
045   as Scallop (Li et al., 2023), execute the neural model on GPU but use a separate CPU-based backend
046   (implemented in Rust in Scallop’s case) for the symbolic program. Moreover, they introduce inter-
047   process latency in transferring state between the neural and symbolic sub-systems.

      Together, these issues hinder the scalability of neurosymbolic learning with respect to problem com-
055   plexity and data complexity. First, the symbolic computation engine must derive a set of all possible
056   results and their associated probabilities in a manner that is differentiable with respect to the net-
057   work’s parameters ✓. As the complexity of the symbolic program increases, the number of possible
058   results and their associated weights also grows exponentially, leading to a combinatorial explosion
059   in the number of required computations. However, the symbolic computations are discrete and not
060   easily parallelizable on modern hardware like GPUs. Second, larger datasets also compound the
061   computational cost of neurosymbolic learning. Deep learning typically addresses this challenge
062   by batching computations across multiple data samples. However, in neurosymbolic learning, the
063
      computations may differ across data samples, making it difficult to batch them effectively.
064   To address these challenges, we need to fundamentally rethink the design of a neurosymbolic frame-
065   work. One approach is to develop specialized and low-level primitives that scale specific bench-
066   marks but make it time-intensive for developers to write neurosymbolic programs tailored to partic-
067   ular tasks. Alternatively, providing high-level primitives—such as a logic programming language
068   like Scallop (Li et al., 2023) or DeepProbLog (Manhaeve et al., 2018)—simplifies the development
069
      of symbolic programs but limits the fine-grained control needed to scale specific applications. Fi-
      nally, to truly democratize neurosymbolic programming, it is crucial to develop a framework that
070
      seamlessly integrates into the everyday deep learning workflows that developers already use.
071
072   In this work, we propose D OLPHIN, a novel framework for scalable neurosymbolic learning. In
073   D OLPHIN, we build three key components that effectively tackle the scalability and programmabil-
074   ity challenges described above. First, we develop a general symbolic representation that efficiently
075
      captures the relationships between neural network outputs and associated discrete symbols. Sec-
      ond, we introduce a set of primitives to map forward chaining in symbolic programs to vectorized
076
      computations over these representations. Third, we develop a set of vectorized provenance semir-
077
      ings (Green et al., 2007) that are easily pluggable into D OLPHIN and enable to efficiently compute
078   symbolic gradients. As illustrated in Figure 1b, these components together allow D OLPHIN to build
079   a computation graph that spans both symbolic and neural operations, is highly parallelizable, and
080   end-to-end differentiable on GPUs. Finally, D OLPHIN is implemented as a library that is integrated
081   with PyTorch, allowing users to easily incorporate it into their existing deep learning pipelines.
082
      We evaluate D OLPHIN on a diverse set of neurosymbolic tasks that involve text, image, video, and
083
      multi-modal data, and use rich reasoning features such as recursion and black-box Python functions.
084   Neurosymbolic programs written using D OLPHIN only require 0.3%-61.7% (23.2% on average) of
085   the time to train compared to state-of-the-art baselines including differentiable reasoning frame-
086   works like Scallop, and sampling-based frameworks like ISED and IndeCateR+ while maintaining
087   similar levels of accuracy. We also observe that D OLPHIN efficiently scales to more complex bench-
088   marks and larger datasets whereas the baselines either time out after 10 hours or fail to converge.
089
      We make the following contributions in this work:
090
091   • We propose D OLPHIN, a novel neurosymbolic programming framework for end-to-end differen-
092     tiable symbolic reasoning in a scalable manner.
093   • We develop novel abstractions to represent symbolic and neural computations and introduce vec-
094     torized primitives for neurosymbolic programs.
095   • We develop vectorized provenances that can be plugged into D OLPHIN for efficient computation
096     of symbolic gradients on parallelizable hardware such as GPUs.
097   • We evaluate D OLPHIN on a diverse range of challenging neurosymbolic tasks across different
098     domains and show that it effectively scales with increasing problem complexity and dataset size.

182   3.1     D OLPHIN C ORE D ESIGN P RINCIPLES
183
      We based D OLPHIN’s framework design on the following core principles:
184
185   • Flexible programmability: The framework should allow developers to write neurosymbolic ap-
186     plications in Python with minimal effort, providing intuitive primitives that seamlessly integrate
187     with Python’s rich and expressive language features.
188   • End-to-end differentiability on GPUs: The framework should allow any neurosymbolic pro-
189     gram to be end-to-end differentiable on GPUs irrespective of the task characteristics.
190   • Scalable: The framework should easily scale with greater problem and data complexity.
191   • Tunable: Similar to hyperparameters in deep learning, the framework should provide a simple
192     interface for developers to choose provenances (and their configurations) or define new ones.
193
194   Together, these principles help address the challenges of scaling neurosymbolic frameworks. The
195   flexible programmability and tunability allow us to write complex neurosymbolic programs, while
196   GPU differentiability and scalability work towards addressing data complexity. We show how we
197
      realize these principles by describing the key components of D OLPHIN.
198
199   3.2 T HE D OLPHIN S YNTAX
200   D OLPHIN provides a programming interface that developers can use to express symbolic programs
201   in a Pythonic manner. D OLPHIN maps each operation of the symbolic program to PyTorch
202
      which enables end-to-end GPU-accelerated differentiable reasoning. Figure 4 presents D OLPHIN’s
      programming interface including the symbolic abstractions and operations over them.
203
204
      3.2.1    A BSTRACTIONS
205
206   The three main abstractions provided by D OLPHIN for expressing differentiable symbolic programs
207   are shown on the left of Figure 4. Symbols S represent symbolic entities relevant to the program.
208   These entities can be any Pythonic object, such as hand-written digits in MNIST-SumN or coordi-
209   nates of points in PathFinder. Tags T are tensors that represent their likelihoods. Typically, tags for
210   symbols are derived from the outputs of machine learning models, such as the probability distribu-
211   tion over digits produced by the CNN classifier in MNIST-SumN. Finally, Distribution D represents
212
      the likelihood of an input being classified as one of the pre-defined symbols.
213   Distributions serve as the fundamental datatype of a D OLPHIN program and act as its main interface
214   with a machine learning model. For instance, when the developer instantiates a Distribution object,
215
      such as in the following code snippet from Figure 2a:
                           Dres = Distribution(self.CNN(imgs[0]), digits)

      Under review as a conference paper at ICLR 2025


216
       the output of the CNN model is directly passed to the Distribution object, effectively acting as
217   an input to the symbolic program. The Distribution object itself, as shown in Figure 2b, contains
218   batches of tags extracted from the model outputs, and maintains the set of corresponding symbols.
219
220
      To enable such a seamless integration between the PyTorch model and the symbolic program, Distri-
      butions are designed to operate directly over PyTorch tensors. This has two main advantages. First,
221
      it preserves the gradients of the model output throughout the symbolic program, enabling end-to-
222
      end differentiability. Second, it allows D OLPHIN to perform operations over an entire batch of tags,
223   leveraging the vectorized operations provided by PyTorch. D OLPHIN can thus operate efficiently on
224   specialized hardware like GPUs, allowing the symbolic program to scale effectively.

299   3.4    D OLPHIN P ROVENANCES
300
301   As discussed earlier, each symbol in a distribution is associated with a batch of one or more tags.
302
      The D OLPHIN primitives define how to manipulate certain tags. For instance, Equations (1) and (3)
      specify the tags to be conjuncted or disjuncted. We now define the semantics of these operations.
303
304   The goal of such operations is to approximate the probabilities of the symbols in the output distri-
305   bution as accurately as possible. This is achieved by using a mathematical framework called prove-
306   nance semirings (Green et al., 2007). Provenance semirings provide generalized algebraic structure
307   to propagate tags when computing over tagged data. In the case of D OLPHIN distributions, we can
308
      view the tags as representing the probabilities, and the data as the distribution’s symbols.
309   Designing and implementing provenances can be challenging since they must be accurate enough
310   to capture the semantics of the symbolic program, while at the same time being coarse enough to
311   maintain computational feasibility. Furthermore, the provenances must be differentiable to enable
312   end-to-end training for neurosymbolic tasks. While neurosymbolic frameworks like Scallop (Li
313   et al., 2023) implement differentiable provenances, they are not designed to leverage hardware ac-
314
      celerations or batched optimizations due to the CPU-bound nature of their implementations. We thus
      design vectorized provenances in D OLPHIN that are differentiable and enable GPU computations.
315
316   We simplify the definition of provenances from Scallop as a 5-tuple: (T, 0, 1, ⌦, ). Here, T is the
317   tag space, ⌦ : T ⇥T ! T is the conjunction operator with identity 0, and : T ⇥T ! T is the dis-
318   junction operator with identity 1. We then implement two differentiable provenances in D OLPHIN:
319   Differentiable Add-Mult Probabilities (DAMP) and Differentiable Top-K Proofs (DTKP). Table 1
320
      summarizes the operations of these provenances.
321   Differentiable Add-Mult Probabilities. Differentiable Add-Mult Probabilities (DAMP) is a pop-
322   ular technique that uses the probability space as its tag space: T = [0, 1]. Its conjunction operation
323   ⌦ is defined as the product of probabilities, clamped at 1, and its disjunction operation is defined
      as the sum of probabilities. The main assumption underlying the DAMP operations is that the input

      Distributions are mutually exclusive and independent. This assumption allows DAMP to compute
325   probabilities extremely efficiently, as the operations are simple and can be easily vectorized.
326
327   Differentiable Top-k Proofs. Differentiable Top-k Proofs (DTKP) (Huang et al., 2021) was pro-
328
      posed to overcome the shortcomings of DAMP. This provenance tracks a set of up to k proofs for
      each symbol. Each proof, in turn, denotes the set of input symbols necessary to derive the output
329
      symbol. These proofs are then used to compute the probabilities of the output symbols. In Scal-
330
      lop, DTKP tags are converted into probabilities via differentiable weighted model counting (WMC).
331   This form of DTKP, which we call DTKP-WMC, is computationally hard and is by nature difficult
332   to vectorize due to the varying sizes of proof sets and the WMC procedure. We hence design a
333   vectorized approximation of DTKP-WMC, called DTKP-AM (DTKP with Add-Mult), that can be
334   efficiently computed on GPUs.

      k proofs with the highest probabilities. These definitions thus allow us to take advantage of the
351
      benefits of the DTKP provenance while enabling efficient computation on GPUs. To calculate the
352
      probability of the entire tag, DTKP-AM adds the probabilities of the individual proofs and clamps
353   it at 1. We provide a detailed discussion of DTKP-AM in Appendix C.
354
355   3.5   B UILDING THE D OLPHIN P ROGRAM
356
357   The programmer specifies the neurosymbolic task using a Python program P that uses D OLPHIN’s
358   programming interface to connect the neural components (e.g., neural networks) with the symbolic
359   components and operations. We call P the symbolic program. Because P is a Python program and
360   D OLPHIN interfaces with PyTorch, D OLPHIN supports any PyTorch-based neural network(s), most
361   Python language features, and custom user-defined functions. This feature enables greater flexibility
362
      and expressiveness in neurosymbolic programs than existing frameworks.
363   In addition to P , the programmer provides one or more neural networks M1 , . . . , Mk , and a dataset
364   D. Given these inputs, D OLPHIN extracts the computation graph that encodes how the neural net-
365
      work outputs are transformed using symbolic operations to produce a final result Dres . All com-
      putations in D OLPHIN are expressed using distribution objects Di . Each D OLPHIN primitive (e.g.,
366   A PPLY) takes one or more distribution objects as inputs and applies a transformation to produce an-
367   other distribution object. Because each distribution object Di only contains vectors of tags (or prob-
368   abilities), the entire computation graph (including the neural network(s)) can be ported to a GPU for
369   faster execution. During training, D OLPHIN optimizes over the standard objective function:

387   MNIST-SumN. The MNIST-SumN (or briefly, SumN) task from (De Smet et al., 2024) takes as
388   inputs N handwritten digits from the MNIST dataset and returns their sum. We consider three
389   versions of this task: small (N = 5), medium (N = 10), and large (N = 15).
390   Hand-Written Formula (HWF). The HWF task from Li et al. (2020) takes as input a set of images
391   of handwritten digits and arithmetic operators representing a formula. The task is to evaluate the
392   formula and return the result. We consider three versions of HWF: small (formulas of length up to
393   7), medium (formulas of length up to 15), and large (formulas of length up to 19).
394
      PathFinder. PathFinder (or Path) from Tay et al. (2021) tests the ability of an agent to reason over
395   long-range dependencies within an input image. The image consisting of two dots and a sequence
396   of curved and dashed lines. The task is to identify whether the two dots are connected via the lines.
397   We consider three versions of this task based on the image size in pixels: small (32 x 32), medium
398   (128 x 128), and large (256 x 256).
399
      CLUTRR. In this task from Sinha et al. (2019), the input is a passage of text containing some
400
      information about several individuals and some of their relationships. The task is then to infer
401   the relationship between two given individuals, which is not explicitly provided in the input. We
402   consider two versions of this task, where the training data contains relation chains of lengths up to 3
403   (small) or 4 (medium).
404
      Mugen. In this task from Hayes et al. (2022), the input is a video of gameplay footage that is 3.2
405
      seconds long and a natural language passage captioning the video. The goal is to measure how
406
      aligned the text is with the video. This task has two variants: Mugen-TVR, where the model must
407   retrieve the video that best aligns with the text, and Mugen-VTR, where the model must retrieve the
408   text that best aligns with the video. This benchmark tests the ability of the model to reason over
409   multimodal data. We consider two versions of this task: small, with 1000 training samples, and
410   medium, with 5000 training samples.

497   to the logits that most influenced the output, making this a more effective provenance for this task.

499   5       R ELATED W ORK
500   Neurosymbolic programming frameworks. Frameworks like Scallop (Li et al., 2023), Deep-
501   ProbLog (Manhaeve et al., 2018), and ISED (Solko-Breslin et al., 2024) provide a simple interface
502   for neurosymbolic programming. There are also domain-specific tools like NeurASP Yang et al.
503   (2021) for answer set programming and NeuralLog Chen et al. (2021) for phrase alignment in NLP.
504   While these frameworks provide intuitive abstractions, they are bottle-necked due to expensive data
505
      transfers between symbolic computations done on CPU versus neural computations that execute on
      GPU, making neurosymbolic learning hard to scale. In contrast, D OLPHIN provides a deeper inte-
506
      gration of the two worlds by building a Python-based API on top of PyTorch, which scales better.
507
508   Scaling techniques. Several optimization techniques have been proposed to improve the scalability
509   of differentiable reasoning algorithms. Some techniques aim to scale reasoning algorithms by com-
510   piling the symbolic program into computation graphs that can be run on GPUs. LYRICS (Marra
511
      et al., 2019), Logic Tensor Networks (Badreddine et al., 2022), and Tensorlog (Cohen et al., 2020)
      are examples of such techniques. However, these methods focus on first order logic programs and
512
      provide limited support for user-defined Pythonic functions, essential for building complex neu-
513
      rosymbolic programs.1 Greedy NTP (Minervini et al., 2020a) reduces the computation cost of
514   NTP (Rocktäschel & Riedel, 2017) by tracking only a subset of proof states using nearest neigh-
515   bor search. Likewise, the conditional theorem prover (Minervini et al., 2020b) employs a machine
516   learning-based proof selection technique. However, unlike D OLPHIN, these methods are point solu-
517   tions that do not fundamentally address the scalability challenge for neurosymbolic learning.
518
      Specialized neurosymbolic solutions. There are many specialized solutions for various neurosym-
519
      bolic tasks. For instance, NGS (Li et al., 2020) uses a hand-coded syntax to specify the structure
520   of mathematical expressions for HWF. More general solutions, such as NS-CL (Mao et al., 2019)
521   includes a framework for visual question answering that learns symbolic representations for text and
522   images. NeRd (Chen et al., 2021) transforms questions in natural language into executable programs
523   based on symbolic information extracted from text. Orvieto et al. (2023) proposes a recurrent neu-
524   ral network architecture that achieves 95% accuracy on Path (S) and 94% on Path (M). In contrast,
525   Dolphin is a general programming framework that tries to scale diverse neurosymbolic programs.
526
527   6       C ONCLUSION AND L IMITATIONS
528
529
      We proposed D OLPHIN, a programmable framework for scaling neurosymbolic learning. D OLPHIN
      provides abstractions for writing symbolic programs along with pluggable vectorized provenances
530
      to compute symbolic gradients. This allows users to write differentiable symbolic programs in
531
      Python within PyTorch pipelines that can scale to complex programs and large datasets. We show
532   that D OLPHIN scales significantly better than existing neurosymbolic frameworks while achieving
533   state-of-the-art performance on a variety of tasks.
534
535
      A limitation of D OLPHIN is that it needs the user to write programs in a batched manner. While this
      is a common pattern within deep learning, it may be restrictive to users new to batched programming.
536
      Also, while D OLPHIN works well with most models, the representation needed by generative models
537
      (e.g., Causal LLMs like Llama) has not been investigated. A third limitation is that Dolphin lacks
538   support for non-deterministic symbolic programs. We leave this to future work.
