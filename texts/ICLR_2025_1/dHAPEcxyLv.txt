                                                       A BSTRACT
010
011
                   Text-to-SQL is a longstanding task aimed at automatically converting natural lan-
012
                   guage questions into SQL queries for database retrieval. Despite impressive ad-
013                vancements, particularly with Large Language Models (LLMs), existing methods
014                still struggle with issues such as misinterpreted, omitted, or unwanted constraints.
015                To address these challenges, we propose READ-SQL, a novel framework em-
016                ploying a reasoning path dcomposer, READER, for text-to-SQL tasks. READER
017                decomposes SQLs into clauses, sub-SQLs, and reasoning paths, supporting data
018                preparation and confidence level determination in post-processing. READ-SQL
019                comprises two main models: a Generator and a Corrector, both trained via LoRA
020                for parameter efficiency. Based on READER’s decomposition, READ-SQL gen-
021
                   erates two types of augmented data using an LLM: question/SQL pairs and ques-
                   tion/reason pairs. The Generator is trained on both original and augmented data
022
                   to identify constraint changes and enhance reasoning. The Corrector is trained
023
                   on data from READER’s post-processing, improving self-correction by refining
024                high-confidence SQLs and addressing low-confidence elements. Extensive exper-
025                iments show that READ-SQL significantly outperforms leading baselines, with
026                READ-SQL-3B achieving 57.37% execution accuracy on BIRD’s Dev set, sur-
027                passing several 7B-parameter models and setting a new state-of-the-art with fewer
028                parameters. Additionally, READER and the Corrector show broad applicability
029                when integrated with LLMs or other base models.

032   1       I NTRODUCTION
033
034   Text-to-SQL, a longstanding and pivotal task in natural language processing, focuses on transform-
035
      ing natural language questions into executable SQLs 1 , streamlining database interactions for non-
      experts and significantly enhancing information retrieval efficiency (Deng et al., 2022; Katsogiannis-
036
      Meimarakis & Koutrika, 2023; Liu et al., 2024a). Recent advances in task decomposition, intermedi-
037
      ate representations, and post-processing strategies have significantly pushed the field forward (Wang
038   et al., 2023; Guo et al., 2019; Pourreza & Rafiei, 2023). Additionally, the integration of Large Lan-
039   guage Models (LLMs) has greatly enhanced natural language understanding and SQL generation
040   through broader context and carefully crafted prompts (Wang et al., 2023; Li et al., 2024a; Talaei
041   et al., 2024b). However, challenges remain in applying these advancements to real-world scenarios,
042   particularly when handling vague questions in large, complex database schemas (Zhang et al., 2024;
043   Liu et al., 2024b; Li et al., 2024a; Liu et al., 2024a).
044   Figure 1 illustrates three typical categories of errors in text-to-SQL, with JOIN-related errors clas-
045   sified accordingly. We attribute these errors to the inherent disparity between natural language and
046   the (semi-)structured syntax of SQL (Liu et al., 2024a):
047
      – Misinterpreted constraints: These occur when the model incorrectly parses the natural language
048
        query, resulting in erroneous SQL clause (Li et al., 2023c; Talaei et al., 2024b).
049
      – Omitted constraints: These arise when the model overlooks essential elements, leading to in-
050     complete SQLs (Pourreza & Rafiei, 2023; Talaei et al., 2024b; Wang et al., 2023).
051   – Unwanted constraints: These involve superfluous clauses that exceed the requirements of the
052     natural language input (Talaei et al., 2024b; Wang et al., 2023).

070   Existing methods address these challenges from two main perspectives: (1) SQL-like grammar
071   languages: These approaches reduce the complexity of SQL generation by employing intermediate
072   SQL-like grammar representations (Gan et al., 2021c; Yu et al., 2018a; Eyal et al., 2023). They
073   can be easily integrated with pre-trained models and large language models (LLMs) to produce
074   effective results (Gan et al., 2021c; Pourreza & Rafiei, 2023; Li et al., 2023a; Rai et al., 2023). (2)
075   Direct output of SQL structure: These methods generate the structure of SQL directly, leveraging
076   grammar information as an intermediate representation and focusing on the syntactic structure of
077   SQL (Gu et al., 2023b;a; Yu et al., 2018a). However, existing methods primarily focus on optimizing
078
      SQL generation without improving the model’s understanding of the question or establishing strong
      relationships between questions and SQL clauses. Additionally, since these methods do not employ
079
      an end-to-end architecture, there is potential for information loss during the process (Liu et al.,
080
      2024a).
081
082   To address these shortcomings, we propose READ-SQL, a novel framework comprising two key
083   models, a Generator and a Corrector, both supported by a reasoning path decomposer, READER,
084
      specifically designed for text-to-SQL tasks. READER can parse an executable SQL into an Abstract
      Syntax Tree (AST) (Wang et al., 1997) and decompose it into clauses, forming sub-SQLs and rea-
085
      soning paths. READ-SQL generates two types of augmented data via an LLM: question/SQL pairs
086
      and question/reason pairs, embedding information about subtle constraint changes and the connec-
087   tion between questions and reasoning paths. The Generator is trained on a multi-task fine-tuning
088   framework via LoRA (Hu et al., 2022), leveraging both the original and augmented data, and out-
089   puts either SQLs or reasons (describing sub-SQLs generation) based on prefix tokens. We aim to
090   enhance the model’s understanding of questions by incorporating additional question/SQL pairs and
091   bridging the gap between questions and SQL through question/reason pairs. For post-processing,
092   READ-SQL employs a Corrector, also trained via LoRA, on the Generator’s processed output SQLs,
093   which are categorized by READER into two types: high-confidence clauses for basic SQLs and low-
094   confidence clauses for retrieving similar items from table schema. The Corrector is trained on these
095
      two types of data to produce the final SQLs, enabling precise self-correction.
096   We summarize our main contributions as follows:
097
      • We propose READ-SQL, featuring two key models: a Generator and a Corrector. The Generator
098
        captures subtle differences between questions and SQLs and enhances the connection between
099     questions and reasoning paths, improving constraint recognition and reasoning abilities. The Cor-
100     rector refines the Generator’s processed SQL outputs for precise self-correction.
101   • Both the Generator and Corrector rely on READER, which decomposes SQLs into clauses to
102     form sub-SQLs and reasoning paths. This aids in data generation for the Generator and refines its
103     outputs for the Corrector.
104   • Extensive experiments show that READ-SQL outperforms leading baselines, with READ-SQL-
105     3B achieving 57.37% execution accuracy on the BIRD Dev, surpassing several 7B-parameter
106     models. Additionally, READER and the Corrector are highly versatile, making them suitable
107
        for integration with LLMs or other base models. Furthermore, READ-SQL significantly reduces
        three common types of text-to-SQL errors.

      2     R ELATED W ORK
109
110
      Generating accurate SQLs from natural language questions, commonly known as text-to-SQL, is an
111   active area of research within both the natural language processing and database communities (Liu
112   et al., 2024a; Zhang et al., 2024). In the following, we review two mainstreams of related work.
113
114
      2.1   T EXT- TO -SQL WITH P RE - TRAINED L ANGUAGE M ODELS
115
116   The development of text-to-SQL has progressed from early neural network-based methods, such as
117   IRNet (Guo et al., 2019) and Bridge (Lin et al., 2020), to pre-trained model approaches like RESD-
118   SQL (Li et al., 2023a), and now to the current era of large models, including DIN-SQL (Pourreza
119   & Rafiei, 2023) and CHESS (Talaei et al., 2024b). Throughout this evolution, task decomposi-
120   tion has remained a fundamental process in text-to-SQL, typically involving schema linking, SQL
121   generation, and post-processing. Various methods have been proposed to handle these sub-tasks:
122   • Schema linking is a critical task that involves identifying the relevant database table columns and
123     values referenced in natural language questions. Pre-trained models (Li et al., 2023a; 2024c) and
124     LLMs have significantly improved schema linking performance (Talaei et al., 2024b; Pourreza
125     & Rafiei, 2023). Even when irrelevant table schema is present, LLMs can accurately generate
126     SQLs (Maamari et al., 2024). Schema linking can help models alleviate the gap between question
127     and SQL, but even state-of-the-art methods (Li et al., 2023a; 2024c) suffer from type three errors
128
        in text-to-SQL. In this paper, we adopt the schema linking solution from (Li et al., 2024c) and
        focus on the remaining two steps, exploring potential ways to better understand the intent behind
129
        questions during SQL generation and post-correction.
130
      • SQL generation involves the challenge of bridging the gap between the flexibility of natural
131     language and the rigid structure of SQLs. This discrepancy introduces two key issues: the model
132     may misinterpret the query’s intent or generate incorrectly formatted SQL. Current approaches of-
133     ten rely on LLMs or fine-tuned pre-trained models for end-to-end solutions (Talaei et al., 2024b;
134     Li et al., 2024c). Some researchers explore intermediate representations, like Natural SQL, to
135     simplify generation (Pourreza & Rafiei, 2023), but these methods have limited practical appli-
136     cation and do not significantly improve query comprehension. Other approaches, such as (Ye
137     et al., 2023), decompose the task into sub-problems, generating sub-SQLs before forming the
138     final SQLs. While they offer a multi-step reasoning solution, they carry the risk of error propa-
139
        gation. The limitations of existing methods motivate us to design an end-to-end framework that
        eliminates error propagation, enhancing multi-step reasoning and question comprehension.
140
      • Post-processing aims to refine generated SQLs to improve both user satisfaction and query accu-
141
        racy. DIN-SQL (Pourreza & Rafiei, 2023) introduces a self-correction module to detect potential
142     syntax errors, while MAC-SQL (Wang et al., 2023) employs a multi-agent approach for error
143     identification and correction. In contrast, C3 (Dong et al., 2023) and DAIL-SQL (Gao et al.,
144     2024) apply self-consistency by sampling multiple results and selecting the most consistent one.
145     CodeS (Li et al., 2024c) selects the first executable SQL as the result. Bertrand-DR (Kelkar et al.,
146     2020) reorders sampled results to align with user preferences. However, these methods often
147     struggle to accurately identify where corrections are needed or focus solely on selecting outputs
148     without modifying the SQL itself. These limitations motivate us to develop a more effective ap-
149     proach for identifying potential error clauses and systematically generating the final SQL.
150
151   2.2   T EXT- TO -SQL WITH A BSTRACT S YNTAX T REES
152
153
      The Abstract Syntax Tree (AST) (Wang et al., 1997) is a tree-like data structure that represents
      the syntactic structure of source code, enabling compilers and other tools to efficiently parse and
154
      analyze code. Each node in an AST corresponds to a structural element, such as an operator or
155
      function call. Decoders like RAT-SQL (Wang et al., 2020) and IRNet (Guo et al., 2019) utilize a
156   tree-structured approach to generate the AST for a SQL query and then convert it back into SQL,
157   ensuring grammatical accuracy. ASTormer (Cao et al., 2023) builds on this by utilzing an AST-aware
158   transformer decoder that incorporates grammatical structure with both absolute and relative position
159   encoding. These methods fall under the category of grammar-based decoders (Zhang et al., 2024).
160   However, with the rise of LLMs (Wei et al., 2022a), the focus has shifted away from this approach.
161   We revisit the use of ASTs to ensure their relevance in the LLM era. By decomposing ASTs, we can
      derive sub-SQLs and trace their evolution, facilitating the model’s step-by-step reasoning.

      Problem Definition Formally, given a natural language question Q and a database D with schema
189
      S, the text-to-SQL task aims to translate Q into a SQL query y that can be executed on D to answer
190   the question Q. The database D contains the schema S = (T , C, R) of three components: a set of
191   N tables T = {t1 , t2 , ..., tN }; a set of columns C = {c11 , ..., c1n1 , ..., cN         N
                                                                                       1 , ..., cnN } associated with the
192   tables, where ni is the number of columns in the i-th table; and a set of foreign key relations R =
193   {(cik , cjh ) | cik , cjh ∈ C}, where (cik , cjh ) indicates a foreign key relationship between two columns.
194                         PN
      We use M = i=1 ni to denote the total number of columns in D.

201   3.1      READER: A RE ASONING P A TH D ECOMPOS ER
202
      READER parses executable SQL into constraints, generating sub-SQLs and reasoning paths. This
203
      forms the foundation for enhanced data generation and refined self-correction. Figure 3 illustrates
204   the four main steps of READER: (1) parsing an SQL into AST; (2) identifying all constraints from
205   the AST; (3) get sub-SQLs by removing constraints on AST and (4) construct reasoning paths for
206   the sub-SQLs. Further details can be found in Appendix A.1. It is noted that
207
      • Step 2: Identify constraints. A constraint is defined as a sub-tree in the AST where the root
208
        node is an operation type and all its child nodes are non-operation types (see Appendix A.1.2 for
209
        a detailed explanation). For instance, the “SELECT” node represents an operation, while its child,
210     “name”, is a non-operation node. Similarly, the “WHERE” node and its corresponding child are
211     recognized by READER as a constraint, indicated by their background color in Figure 3.
212   • Step 3 : Sequentially delete constraints. READER gets sub-SQLs by removing constraints.
213     READER initializes a binary tree to store results, with the root node storing the AST parsed
214     from the original SQL. At each level, READER removes one constraint from the root, storing
215     the remaining sub-tree as the right leaf node, while replicating the root in the left leaf node. For
        example, the circle labeled 3 in Figure 3 is obtained by removing the “WHERE age=18” constraint

302   3.4    C ORRECTOR FOR S ELF -C ORRECTION
303
304   Corrector uses READER to re-evaluate the low-confidence constraints in the results generated by
305   the Generator, producing the final SQL, as illustrated in the Figure 2. To begin, we will explain the
306   reasoning process of Corrector. This process primarily consists of two parts: constructing the input
307   for Corrector, and generating the final SQL. The entire process is illustrated in Figure 4. Corrector’s
308
      input relies on READER, which parses the SQL to extract two types of constraints:
309   • High-confidence constraints appear consistently in all generated SQLs. For example, READER
310     identifies four identical constraints and combines them to form a basic AST, which is parsed to a
311     basic SQL, as shown in Figure 4. Basic SQL retains the consistent part of the Generator’s result,
312     allowing Corrector to generate SQL based on it, which helps reduce errors.
313   • Low-confidence constraints are those that differ in other SQLs or appear in only some SQLs.
314
        Corrector requires additional information to generate accurate SQL from low-confidence clauses.
        First, it extracts the table names, column names, and values from the low-confidence clauses.
315
        Then, READ-SQL uses search engines, such as Elasticsearch, to retrieve related information,
316
        including table schema, foreign keys, and value matches.
317
318   Finally, READ-SQL organize basic SQL and the retrieved table schema, foreign keys and value
319   matches into the final input, as shown in Figure 4, allowing Corrector to output the final SQL.
320   Similar to the Generator, the Corrector model is also fine-tuned using LoRA. However, since the
321   Corrector requires the Generator’s output, we perform cross-validation on the text-to-SQL training
322   set to obtain the Generator’s outputs for each fold. Following the previously described process, we
323   construct the input data and combine the data from all folds to create the final training dataset for
      the Corrector. Detailed steps are provided in the appendix A.4.

328   Baselines For supervised fine-tuning, nearly all baselines are derived from the state-of-the-art
329   (SOTA) text-to-SQL approaches listed on the official leaderboards of the BIRD and Spider bench-
330   marks (Li et al., 2023a; 2024c; Yang et al., 2024; Li et al., 2023b; Scholak et al., 2021). We select
331   SFT CodeS as our primary competitive baseline (Li et al., 2024c). Additionally, we compare READ-
332   SQL with LLM-based methods (Pourreza & Rafiei, 2023; Gao et al., 2024).
333
334   Datasets We conduct experiments on two English text-to-SQL benchmarks: BIRD (Li et al.,
335
      2023c) and Spider (Yu et al., 2018b). To evaluate the model’s robustness, we also evaluate READ-
      SQL on three Spider variants: Spider-DK (Gan et al., 2021b), Spider-Syn (Gan et al., 2021a), and
336
      Spider-Realistic (Deng et al., 2021). BIRD includes 9,428 training samples and 1,534 development
337
      samples, while Spider has 8,659 training samples and 1,034 development samples, both featuring
338   hidden test sets. Details of the datasets are provided in Appendix A.5.1.
339
340   Metrics To evaluate the performance of the Text-to-SQL parser, following Li et al. (2024c); Talaei
341   et al. (2024b); Pourreza & Rafiei (2023); Li et al. (2024a), we apply the following metrics: (1) For
342   the BIRD benchmark, we employ execution accuracy (EX), which measures whether the generated
343   SQL retrieves the correct results from the database, and the Valid Efficiency Score (VES), which
344   is determined by dividing the execution time of the ground truth SQL query by the execution time
345   of the predicted SQL query, to quantify the accuracy and performance efficiency of the generated
346   SQLs. Since VES relies on the environment, we rerun the results of CodeS for a fair comparison. (2)
347
      For Spider and its variants, we adopt the test-suite accuracy (TS) score (Zhong et al., 2020), which
      measures whether generated SQLs consistently pass the EX evaluation across multiple database in-
348
      stances, thereby reducing false positives—instances where a prediction that is semantically different
349
      from the correct answer coincidentally matches the same denotation in a specific database.
350
351
      Implementation Details We provide a detailed explanation of READ-SQL’s Generator and Cor-
352   rector separately.
353
354   – Generator: We use GLM-4-0520 (Zeng et al., 2024) as the base LLM to generate questions and
355
        reasoning paths. For schema linking, we adopt the same method as CodeS (Li et al., 2024c) to
        obtain the database prompt. During training, we fine-tune the model using the LoRA (Hu et al.,
356
        2022) technique, with CodeS-1B and CodeS-3B as base models. The learning rate is set to 1e-4,
357
        the training runs for 6 epochs, and we use a batch size of 8 with a λ value of 8. For inference, the
358     beam size is set to 4, and greedy decoding is applied.
359   – Corrector: We utilize Elasticsearch as a retrieval tool to select the top 4 columns and the top 2
360     cell values based on approximate matching. When constructing the dataset, we employ four-fold
361     cross-validation, training the Generator on three folds with the above parameters while evaluating
362     on the remaining fold. This process is repeated across all four folds. The Corrector is also fine-
363     tuned using LoRA (Hu et al., 2022) in CodeS-1B and CodeS-3B, with a learning rate of 1e-4,
364     6 epochs, and a batch size of 8. During inference, we set the beam size to 4 and select the first
365     executable SQL as the final output.

      memory for the models and an AMD EPYC 7742 CPU of 128 GB memory for testing VSE. More

      information is detailed in Appendix A.5.3.

      5              C ONCLUSION
524
525
      We propose READ-SQL, a novel framework that leverages READER, a key module to parse SQLs
526
      into clauses, sub-SQLs, and reasoning paths to enhance text-to-SQL tasks. READ-SQL consists
527
      of two main models: the Generator and the Corrector. The Generator is trained on both original
528   and augmented data to recognize subtle differences between questions and SQLs while improving
529   reasoning capabilities. Additionally, the Corrector applies READER in post-processing to ensure
530   precise self-correction. Experimental results demonstrate that READ-SQL significantly improves
531   strong baselines of the same model size, setting a new SOTA. Furthermore, the Corrector can be
532   deployed on any base model, and the post-processed SQL generated by the Generator can be fed
533   into an LLM to enhance text-to-SQL performance, underscoring its wide applicability.
534   Several future directions are promising: (1) While READER effectively parses SQLs, this work
535   only utilizes partial information. Further exploration of READER’s parsed information could im-
536   prove performance. (2) The Corrector’s effectiveness may be limited when supplemental items are
537   unreliable due to insufficient schema information, suggesting the need for more robust methods to
538   identify those low-confidence items. (3) Due to computational constraints, we conduct experiments
539   on a model up to 3B. Scaling up both the model and data presents an interesting opportunity to
      unlock the full potential of READ-SQL.

890   A.1.1     E XPLANATION OF A LGO . 1
891
892   Algorithm 1 outlines the procedure of READER:
893
      • Line 2 runs a parser to parse the SQL into an AST.
894   • Line 3 gets the set of constraints in the AST based on pre-defined rules. Detailed information
895     about these rules is provided in the Appendix A.1.2.
896   • Line 4 initializes a binary tree to store intermediate results, with the AST placed in the root node.
897   • Lines 5-6 deletes each constraint and save the result in a binary tree.
898   • Lines 7-16 defines a method to remove constraint in the AST from the leaf nodes of the binary
899     tree, assigning the resulting new AST and a backup of the original AST as the child nodes.
900   • Lines 17 extract all possible sub-SQLs from the leaf nodes of the binary tree.
901   • Lines 18 employ a bottom-up breadth-first approach to identify the inclusion relationships of
902
        constraints in sub-SQLs and retrieve all existing reasoning paths.
903   Hence, given a question, “What is the name of a person who is 18 years old?” and its ground-truth
904   SQL, “SELECT name FROM person WHERE age = 18”, we can obtain the corresponding results
905   as follows:
906   • In line 3 of Algorithm. 1 or Step (2) in Figure 3, identify three clauses: “SELECT name”, “FROM
907     person”, and “WHERE age = 18”.
908   • In 4-16 of Algorithm. 1 or Step (3) in Figure 3: remove constraints to obtain sub-SQLs. First,
909     the method deletes “SELECT name” to obtain “SELECT * FROM person WHERE age = 18”,
910     along with a backup of the original SQL. Next, the method removes “WHERE age = 18” from
911     both results, yielding “SELECT * FROM person” and “SELECT name FROM person”, thereby
912     generating all sub-SQLs.
913   • In 17-18 of Algorithm. 1 or Step (4) in Figure 3: using “SELECT * FROM person” as the starting
914
        point of the reasoning path, the next points can be “SELECT name FROM person” and “SELECT
        * FROM person WHERE age = 18”. The endpoints of both reasoning paths represent the ground-
915
        truth SQL.

940   A.1.2    N ODE T YPES
941
942   For each node in AST, they can be categorized into the following types (see details in Table 6):
943   • Non-operational type nodes include columns, tables, identifiers, Literals, etc.
944   • Operational types nodes are defined as nodes whose operation objects are non-operational nodes.
945     In simple terms, an operation node can be defined as a constraint.
946
      Generally speaking, the subquery node in SQL represents a distinct SQL query. We treat it as a sep-
947   arate constraint and perform constraint decomposition on the subquery independently. A subquery
948   (also known as an inner query or nested query) is a query that is embedded within another SQL
949   query, as shown in Figure 8, Sub-query 1 is a subsquery node in sub-query 2.
950
      For each sub-SQL, we can determine the constraints based on the following criteria:
951
952   • Dependencies exist between constraints, necessitating careful judgment before deletion. For ex-
953     ample, the constraint "WHERE movies.movie_popularity > 1000" relies on the JOIN constraint
954     "INNER JOIN movies ON ratings.movie_id = movies.movie_id." This indicates that the column
955     referenced in the WHERE clause belongs to the movies table. Consequently, when deleting the
956
        JOIN constraint, the corresponding WHERE constraint must also be removed; otherwise, the
        JOIN constraint cannot be deleted. Nodes with dependencies include: GROUP BY and HAVING;
957
        and JOIN nodes along with all constraints involving the tables in the JOIN.
958
      • Constraints can be merged to reduce the number of generated sub-SQLs. For instance, the con-
959     straints "SELECT name" and "SELECT year" can be combined because both columns belong to
960     the person table. However, if the columns originate from different tables, they cannot be merged.
961     The only node types that can be merged are non-operation column node types.
