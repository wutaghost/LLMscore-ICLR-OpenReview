009                                                A BSTRACT
010
011            Resolving conflicts from merging different software versions is a challenging task.
012            To reduce the overhead of manual merging, researchers develop various program
013            analysis-based tools which only solve specific types of conflicts and have a lim-
014            ited scope of application. With the development of language models, researchers
015            treat conflict code as text, which theoretically allows for addressing almost all
016
               types of conflicts. However, the absence of effective conflict difficulty grading
               methods hinders a comprehensive evaluation of large language models (LLMs),
017
               making it difficult to gain a deeper understanding of their limitations. Further-
018
               more, there is a notable lack of large-scale open benchmarks for evaluating the
019            performance of LLMs in automatic conflict resolution. To address these issues,
020            we introduce C ON G RA, a CONflict-GRAded benchmarking scheme designed
021            to evaluate the performance of software merging tools under varying complexity
022            conflict scenarios. We propose a novel approach to classify conflicts based on
023            code operations and use it to build a large-scale evaluation dataset based on 44,948
024            conflicts from 34 real-world projects. We evaluate state-of-the-art LLMs on con-
025            flict resolution tasks using this dataset. By employing the dataset, we assess the
026            performance of multiple state-of-the-art LLMs and code LLMs, ultimately uncov-
027
               ering two counterintuitive yet insightful phenomena. C ON G RA will be released at
               https://github.com/xxx/ConGra.

030   1    I NTRODUCTION
031
032   Code merging has become a challenging task for developers during project development and mainte-
033   nance. Git, the most popular version control system (Spinellis (2012)), uses a text-based code-merging
034   mechanism. Despite its efficiency, developers often struggle with manually merging different versions
035   of code when Git fails to resolve conflicts automatically. These conflicts come from text or even the
036   syntax and the functionality of the code from different versions.
037   To address conflict resolution, researchers leverage program analysis to achieve syntax-error-free
038   code merging (Zhu et al. (2022); Sousa et al. (2018); Larsén et al. (2022); Shen et al. (2019); Apel
039   et al. (2012; 2011)). Based on abstract syntax trees (AST), these tools merge AST vertices and edges
040   to ensure syntax correctness and better merge results. Nevertheless, a conflict is still generated and
041   need to be resolved by developers if the merge of AST nodes fails. With the development of language
042   models and even LLMs, they are now applied to conflict resolution (Dinella et al. (2022); Zhang et al.
043   (2022); Svyatkovskiy et al. (2022); Dong et al. (2023)). Trained on extensive datasets of prior and
044
      manually merged codes, these models predict suitable resolution for each conflicting code segment
      without manual efforts.
045
046   However, evaluating the performance of LLMs on conflict resolution tasks is challenging due
047   to the wide variation in conflict difficulty and the lack of effective grading methods to reflect
048   these differences. For example, ConflictBench (Shen & Meng (2024)) classifies conflicts based
049   on the source of conflict-resolved code, such as from either merging candidate versions or newly
050
      introduced by developers. This classification does not accurately reflect the complexity of the conflicts.
      Additionally, the community lacks comprehensive conflict resolution benchmarks (discussed in
051
      Section 2.4), especially for extreme cases involving long code contexts.
052
053   To this end, we introduce C ON G RA, which is designed to evaluate code merging tools across a
      diverse range of merging scenarios and assess their ability to resolve conflicts of varying complexities.

      We propose a novel approach to construct graded conflict dataset according to conflict’s resolving
055   complexity. We evaluated six state-of-the-art LLMs (three general LLMs and three code LLMs) on
056   C ON G RA to assess their abilities to resolve conflict in various merging scenarios. The dataset is
057   constructed using 44,948 conflict cases sourced from 34 large-scale open-source projects written
058   in C, C++, Java, and Python. The results show that LLMs with longer contexts support do not
059   always yield better results compared to models with shorter contexts. Additionally, general LLMs
060   (e.g. LLama3-8B and DeepSeek-V2) outperform specialized code LLMs in automatically resolving
061   conflict. Besides, we will release our datasets and a benchmark at https://github.com/xxx/ConGra.
062   In summary, we made the following contributions:
063
064         1. We introduce the first classification approach to generate a complexity-graded conflict
065            dataset. Using this approach, collected conflicts can be classified into seven categories.
066         2. We release a large-scale graded dataset for conflict resolution benchmarking. which contains
067            44,948 conflict cases from popular projects written in C, C++, Java, and Python.
068
            3. We conduct the first comprehensive evaluation of LLM’s performance on conflict resolution
069            task, and find two thought-provoking counter-intuitive phenomena.

072   2     BACKGROUND
073
074   2.1   TASK D EFINITION OF AUTOMATIC M ERGE C ONFLICT R ESOLUTION
075
      Git offers multiple strategies to merge code (Atlassian (2022)). The most common and widely-
076
      used strategy is the three-way merging strategy (Vinkler (2023)). The fundamental concept behind
077   three-way merging is to locate the most recent common ancestor (O) through the historical commit
078   graph, given two merging candidates (A and B), and to generate the automerged code version (M )
079   based on the difference A − O and the difference B − O. Before generating M , Git identifies
080   multiple difference matching triplets Ai − Oi − Bi by comparing A − O and B − O. For each
081   difference matching triplet, 1) if Ai − Oi equals Bi − Oi , it indicates that both A and B have made
082   identical changes to the same code segment. In this case, the difference block will be applied to M ;
083   2) if Ai − Oi is empty, but Bi − Oi exists, it suggests that only B has modified this code segment.
084   Consequently, the difference block from B will be applied to M , and vice versa; 3) if Ai − Oi and
085
      Bi − Oi are not empty nor equal, it indicates that A and B have made different modifications to the
      same code segment, resulting in a conflict in M . In this paper, we assume that M always contains at
086
      least one conflict.
087
088   Due to the presence of conflicts, Git-based automated code merging may encounter exceptions,
089   prompting developers to manually address them. Following conflict resolution, developers will
090   release the resolved code version (R). In large projects, merging two versions results in numerous
091
      conflicts, involving substantial code modifications. This significantly raises developers’ project
      maintenance costs (Vale et al. (2021)). To this end, various automatic conflict resolution (ACR)
092
      systems have been proposed to mitigate these challenges.
093
094
      2.2   P ROGRAM A NALYSIS - BASED ACR
095
096   Program analysis-based ACR (Zhu et al. (2022); Larsén et al. (2022); Sousa et al. (2018); Apel et al.
097   (2011); Shen et al. (2019); Apel et al. (2012)) ensures the syntactic integrity of the merged code by
098   merging on AST. As they implement the three-way merging strategy, conflicts inevitably arise for
099   unmergeable AST nodes, requiring manual intervention.
100
      To assess the performance of program analysis-based ACR, researchers focus on metrics like the
101
      number of generated conflicts, the accuracy of resolutions, and resource consumption. However,
102   these tools often lack convincing evaluation datasets and performance comparisons. As shown
103   in Table 1, tools like Spork, SafeMerge, and Mastery (Larsén et al. (2022); Sousa et al. (2018);

106   categorize structural and textual merges, but this is essentially an ablation study, leaving the dataset’s
107   resolving complexity intertwined. IntelliMerge (Shen et al. (2019)) classifies merge scenarios but
      fails to differentiate these categories in the final evaluation, preventing performance comparisons

137   2.3     M ACHINE LEARNING - BASED ACR
138
139   Machine learning-based ACR (Zhang et al. (2022); Dong et al. (2023); Svyatkovskiy et al. (2022);
140   Dinella et al. (2022); Aldndni et al. (2023); Shen et al. (2023); Elias et al. (2023)) leverages a vast
141   number of code merge examples during pre-training and harnesses the language-aware capabilities of
142
      machine learning models to understand the intricacies of code merging. Once trained, these models
      offer resolution suggestions for merge conflicts. They does not rely on the three-way merge strategy,
143
      eliminating majority of the need for manual conflict resolution.
144
145    Table 1 illustrates the features implemented in machine learning-based ACRs’ evaluation. Machine
146   learning-based ACRs must provide accurate resolutions for all conflicts, underscoring the importance
147   of resolution correctness. Prior research efforts have primarily concentrated on improving two key
148
      metrics: precision and accuracy. Accuracy measures the percentage of total conflicts for which these
      tools produce the correct resolution, while precision indicates the percentage of correct resolutions
149
      among all resolution suggestions provided by the tools. By the same token as depicted in Section 2.2,
150
      the matching algorithm between generated resolution and ground truth are exact string matching,
151   which cannot reflect the real code similarity. Furthermore, existing language model-based ACR test
152   datasets also lack classification of merge scenarios based on different complexities, hindering the
153   assessment of model performance across various levels of code merging tasks’ difficulty.
154

      2.4     C ONFLICT R ESOLUTION B ENCHMARKS
156
157   ConflictBench (Shen & Meng (2024)) is the only benchmark for software code merging evaluation
158   to date. It categorizes merging scenarios by resolution types and uses three metrics with an exact
159   string matching algorithm to evaluate merging tools. However, as shown in Table 1, its classification
160   strategy doesn’t fully capture the complexity of merging scenarios. For example, conflicts can be
161   resolved by retaining all edits from A or B, or by introducing new edits. ConflictBench categorizes
      these into three groups, but the complexity boundaries between them are often blurred. Additionally,

      Figure 1 provides an overview of C ON G RA’s benchmarking pipeline. C ON G RA starts with collecting
181
      open-source projects’ historical merging scenarios with conflict from Github. After the raw dataset is
182   constructed, C ON G RA further colors the conflict snippets with code operations which are extracted
183   via a lightweight but powerful syntax tree level analysis. These conflicts then are classified into
184   various categories based on the code operations taken in the conflict code block to construct C ON G RA
185   dataset. Via the classifier, each conflict can be pigeonholed by diverse resolving complexities. Then,
186   we combine the conflict along with its context to construct the prompt for LLMs. Finally, LLMs
187   will respond to the query and provide conflict resolution suggestions. The results will be assessed by
188   C ON G RA’s evaluation metrics.
189
190   3.2   C OLORING
191
      Firstly, analyzing conflict code snippets is challenging because the snippets in M generated by Git are
192
      often fragmented and can contain any part or format of code. Existing program analysis tools (Lattner
193
      & Adve (2004); Wang & Shoshitaishvili (2017); Lam et al. (2011)) cannot handle these incomplete
194   snippets syntactically or semantically. To address this, C ON G RA applies M ’s modifications from A
195   and B separately to generate two merged files (Ma and Mb ). This approach applies all the difference
196   blocks from either A or B, allowing for code analysis of the conflicting snippets by recording the
197   text ranges and analyzing the whole code from Ma and Mb .
198
      Secondly, for the benchmarking pipeline, automating code complexity assessment must use the
199
      lightest code analysis. Traditional program analysis requires a strict program context, including
200   the project environment and necessary compilation settings, which are hard to preserve in datasets.
201   Additionally, program analysis tools perform data flow or control flow analysis (Chess & McGraw
202   (2004); Nielson et al. (2015)), consuming significant resources, especially for large-scale datasets.
203   C ON G RA addresses this through code operation abstraction. It extracts the operations of conflicting
204   code fragments from Ma and Mb to reflect code complexity. Since C ON G RA’s analysis involves
205   only lexical and grammatical analysis without deeper compilation or context dependency, it achieves
206   efficient benchmarking.
207   Given Ma , Mb and respective conflict-related code ranges, C ON G RA extracts code operations from
208   these conflict-related code blocks in the coloring phase. C ON G RA first leverages tree_sitter (Brunsfeld
209   (2024)), a lightweight multi-language code parsing framework, to convert plain text code into a syntax
210   tree that is suitable for analysis. We pre-define types of code operations based on the node of the
211   syntax tree. These code operations are arranged in ascending order of priority as shown below:
212
213         1. Composite Type Definition (CTD): Definition of composite types, e.g. the definition of
214
               class and the definition of struct in C++ language.
215         2. Function Body Definition (FBD): Definition of the function body content. Any modifica-
               tion to the function body is included in this operation.


                                                          4
      Under review as a conference paper at ICLR 2025


216
            3. Function Prototype Definition (FPD): Definition of function prototype, i.e. function name,
217            return value type and parameter list.
218
            4. Language-Specific Operations (LSO): Operations related to language features, such as
219
               macro definitions in C/C++ and the introduction of third-party libraries in Python.
220
221         5. Commenting (CMT): Developers’ comments on the code.
222         6. Variable Declaration (VD): Declaration of all variables. These variables refers to global
223            variables, local variables, and member variables defined in composite types.
224
225
      These code operations have different priorities, and operations with higher priorities can override
      operations with lower priorities. For instance, the modification of a local variable name in a function
226
      body can belong to both function body definition and variable declaration. To this end, C ON G RA
227
      will give priority to variable declaration as the result of code operation extraction. After analyzing
228   each syntax tree node, C ON G RA only retains code operations related to the conflicting code area,
229   and generates code operation lists for the two versions of the code, namely Pa and Pb . These code
230   operations serve as the basis for conflict classification in the subsequent stage.
231

326   5.1     B ENCHMAKRING S ETTINGS
327
328   Evaluation Metric. We choose the accuracy and precision of generated resolutions as the key
329   metrics in our evaluation. Accuracy is the percentage of generated resolutions that match the ground
330   truth to the total number of conflicts. Precision refers to the percentage of generated resolutions
331   that match ground truth to all generated resolutions. We use the combination of normalized edit
332   distance, winnowing, and cosine semantic similarity (Ristad & Yianilos (1998); Schleimer et al.
333
      (2003); Rahutomo et al. (2012)) as the code-matching standard (i.e., ES, WS, and SS). For a generated
      resolution, C ON G RA regards the resolution matches the ground truth when at least one of the above
334
      values greater than 80%.
335
336   Benchmarked Models. As depicted in Figure 5, C ON G RA exhibits a substantial number of conflict
337   instances, characterized by their contextual information. Consequently, we opt for LLMs that are
338   capable of accommodating long contexts as our benchmarked models. These language models,
339
      supporting over 8K tokens, comprise three general language models (Llama3-8B 1 , DeepSeek-
      V22 ( DeepSeek-AI et al. (2024)), GLM-3-turbo 3 ) and three code language models (CodeLlama-
340
      7B( Roziere et al. (2023)), CodeLlama-34B( Roziere et al. (2023)), DeepSeek-Coder V1( Guo et al.
341
      (2024))). To compare the performance of variants of the same model on C ON G RA, we also include
342   additional variants to form the following variant comparision group on Java and Python conflicts: (1)
343   DeepSeek-Coder V1 and DeepSeek-Coder V2( Zhu et al. (2024)). (2) CodeLlama 7B and CodeLlama
344   34B. (3) LLama3-8B and LLama3.1-8B 4 . We also include one of the most state-of-the-art LLM,
345   GPT-4o-mini 5 as the baseline becaust it is one of the most widely-used LLMs. We only evaluate it
346   on Java dataset because of the resource and token limitation.
347
      Experiment Settings. We set the temperature coefficient uniformly at 0.7 to fully ensure the
348   creativity of LLMs in the experiments. For each conflict, we take at most the previous 100 lines and
349   the following 100 lines of text as context, and decrement line by line until the number of tokens is
350   lower than the maximum context length supported by LLMs. In addition, for each conflict that does
351   not correctly obtain the model output, we will repeat the experiment up to 10 times, otherwise it will
352   be regarded as an unprocessable case. For open-source models, we use vLLM( Kwon et al. (2023) as
353   the backend for model deployment.
354   Comparision with Existing Merging Tool Baselines. We initially planned to choose prior related
355   works as the baselines to contribute to this evaluation with the selected LLMs. However, these
356   existing tools fails to be adequate baselines because of the following reasons: (R1) For conflict
357   dataset, although ConflictBench provides graded conflicts based on manual strategies, it does not
358   evaluate merging tools across different conflict classifications. Since our goal is to assess LLM
359   performance across various conflict complexities, ConflictBench’s classification rules are unsuitable.
360   Additionally, its small size (only 180 conflicts) may lead to biased results. (R2) Using program-
361   analysis-based conflict resolving systems as benchmarks presents inconsistent metrics and unfair
362
      comparison challenges. For inconsistent metrics challenge, the aforementioned seven merging tools
      are either structured or semi-structured merging tools and will merge code on the AST level. As
363
      ConGra extracts merging code chunks, conflict code chunks, and resolving code chunks (i.e., the
364
      conflict resolving ground truth) on the textual level, it is impossible to map the modified zone from
365   these merging tools to the ground truth zone, therefore fails to get the evaluation metric to compare
366   with the LLMs’ performance. For unfair comparison challenge, unlike LLMs, these systems still
367   generate conflicts when the merging of AST nodes fails. Regarding the project scale of ConGra,
368   more complicated merging scenarios are included in our dataset, therefore causing a high conflict
369   ratio even in the SOTA merging tools. If we only focus on the data point without conflicts, the
370   number of samples in the dataset will be inconsistent (compared with the number of samples used by
371   LLMs), which causes a large deviation in the results. (R3) Regarding to machine learning approaches,
372   RPredictor and MESTRE can predict whether the developers should keep the left version, keep the
373
      right version, or resolve the conflict manually. Unfortunately, they cannot generate any conflict

406   5.2.1    OVERALL P ERFORMANCE
407
      In this section, we present a comprehensive evaluation of advanced LLMs on our proposed C ON G RA
408   dataset. The experimental results can be found in Tables 2 and 3. Upon careful comparison, we draw
409   the following observations: (1) LLMs with longer context support do not always yield optimal results.
410   GLM-3-Turbo, which supports a 128K context, is generally outperformed by LLama3-8B, which only
411   supports an 8K context, with the exception of semantic similarity. For instance, in Python and Java,
412   the Precision metric for GLM-3-Turbo is nearly 20% lower than that of LLama3-8B. We speculate
413   that this may result from LLMs with ultra-long context support not being sufficiently trained on merge
414   conflict datasets, hindering their ability to effectively extract valuable information from the context
415   to suggest merge resolutions. (2) Code LLMs do not appear to demonstrate a distinct advantage.
416
      Notably, LLama3-8B and DeepSeek-V2 surpass Code LLMs in Precision across all four languages.
      We attribute this to two factors: (i) general models are in fact trained on extensive code repositories,
417
      so their code comprehension capabilities are not significantly inferior to Code LLMs. (ii) Elements
418
      within conflicts such as comments and variable names contribute a plethora of semantic information,
419   extending beyond mere code understanding, which presents a challenge for Code LLMs. (3) Both
420   LLama3-8B and DeepSeek-V2 prove to be well-suited for automatic conflict resolution. We also
421   conducted experiments to evaluate the performance of different variants of models on C ON G RA in
422   Appendix G, and explore the most state-of-the-art models’ performance on C ON G RA in Appendix H.

      6   D ISCUSSION
504
      Limitation. The conflict classification strategy in Section 3 ensures correct classification but may
505
      fail to identify the exact category of a conflict due to advanced language usage. For example, the use
506
      of template class definitions in C++ (Vandevoorde & Josuttis (2002)) prevents tree_sitter (Brunsfeld
507   (2024)) from capturing the type used as the template argument during instantiation. Consequently, a
508   conflict that should be classified as "syntax" may be regarded as "non-syntax." We will continue to
509   refine C ON G RA’s classification in future work.
510
      Societal Impacts. (1) Improvement of benchmarking current LMs on code merging tasks. During
511
      the merging process, LMs may fail to handle pure text conflict, and generate error-prone merged
512
      code with incorrect syntax or functionality. These wrong results require large amounts of human
513   intervention to fix. With ConGra, these LMs can be well benchmarked in terms of their ability
514   to generate text-correct, syntactic-correct, and functional-correct merged code. We believe the
515   publication of ConGra can further enhance the utilization of LMs in the code merging field and cast a
516   positive impact on the whole community. (1) Propagation of large datasets for LMs’ training and
517   testing. There is a trend that more large-scale Git repository data are used for LLMs purposes. We do
518   believe that the publication of ours or similar works will fuel the momentum as well. We think the
519   open-source of large datasets will finally lead to a positive contribution..
520
521   7   C ONCLUSION
522
523   We propose C ON G RA, a complexity-graded conflict benchmarking system. C ON G RA implements
524   a highly efficient and accurate conflict classification algorithm to construct a complexity-graded
525   conflict dataset, which is used to evaluate the performance of merging tools under various conflict
526   scenarios. C ON G RA utilizes three code matching metrics of different granularities and combines
527   them to calculate the accuracy and precision of auto-generated resolutions. We evaluate six LLMs
528   on C ON G RA, and the results show that LLMs with longer context support often perform worse
529   than those with shorter context support, and general LLMs outperform specialized code LLMs in
530
      precision.

       E                                              C ODE S IMILARITY M ETRICS
1027
1028
                                                           • Edit Similarity (ES). Edit distance is a metric used to measure the difference between two
1029                                                         strings (Ristad & Yianilos (1998)). It is defined as the minimum number of edit operations
1030                                                         required to transform one string into another. In C ON G RA, we normalize the edit distance,
1031                                                         treating the normalized result as the similarity measure in character level.
1032
                                                           • Winnowing Similarity (WS). Winnowing is an algorithm used for text similarity detection
1033                                                         and fingerprint extraction (Schleimer et al. (2003)). It has been implemented in the Moss
1034                                                         code plagiarism detection system (Aiken (2022)) to determine code similarity. In C ON G RA,
1035                                                         we normalize the winnowing result to assess the similarity of the whole conflict code
1036                                                         snippets.
1037                                                       • Semantic Similarity (SS). We use the cosine similarity between the generated resolution
1038                                                         and the actual resolution as the semantic similarity. In C ON G RA, in order to effectively
1039                                                         model the semantic information of the resolution, we use BCEmbedding NetEase Youdao
1040                                                         (2023) as the embedding model to obtain the representation of each resolution.
