009                                                     A BSTRACT
010
011                The typical process for LLM’s development involves pre-training a general founda-
012                tion model on massive data, followed by fine-tuning on task-specific data to obtain
013                a series of specialized experts. Serving these experts can pose significant memory
014                challenges, as loading all experts onto devices is impractical, and frequent switch-
015                ing between experts in response to user requests can incur substantial I/O costs.
016
                   Previous approaches decompose the expert weights as the pre-trained weights plus
                   delta weights, followed by quantizing the delta weights using output channel-wise
017
                   step sizes to reduce the model size. However, these methods overlook the fact that
018
                   certain input channels of delta weights can cause significant quantization errors at
019                extremely low bitwidths. To this end, we introduce ME-Switch, a memory-efficient
020                expert switching framework tailored for serving multiple LLMs. To condense the
021                number of bits required for describing the delta weights, we propose a salient-
022                aware delta compression method that first identifies which input channels of delta
023                weights are salient based on reconstruction error and then employs mixed-precision
024                quantization that selectively quantizes non-salient input channels of delta weights
025                to extremely low bits while keeping the salient ones intact, significantly reducing
026                storage demand while maintaining performance. Extensive experiments show the
027
                   promising memory efficiency and accuracy of ME-Switch. For example, when
                   serving three models from the Mistral-7B family, ME-Switch reduces the model
028
                   size by 2.04× and maintains nearly lossless performance on instruction, mathemat-
029
                   ical reasoning, and code generation tasks. Furthermore, our method can efficiently
030                serve 16 Mistral-7B models on an NVIDIA A100 GPU.

033   1       I NTRODUCTION
034
035   Large language models (LLMs) such as GPT-4 (Achiam et al., 2023) and Gemini (Team et al.,
036   2023), have achieved significant advancements in natural language processing (NLP). Following a
037   pretrain-finetune paradigm (Touvron et al., 2023a;b; Dubey et al., 2024), these models are first trained
038
      on large-scale text datasets to develop a broad foundation of language understanding, enabling them
      to excel in tasks requiring common-sense knowledge. To acquire task-specific knowledge, these
039
      pre-trained models are further fine-tuned on specialized tasks, enabling their adaptation or alignment
040
      for diverse applications such as interactive agents (Touvron et al., 2023b; Jiang et al., 2023), code
041   generation (Luo et al., 2023b; Lozhkov et al., 2024), and mathematical problem solving (Luo et al.,
042   2023a; Lozhkov et al., 2024), demonstrating the remarkable versatility of LLMs. For instance, even
043   high-capacity LLMs like the MoE model Mixtral-8x22B 1 are fine-tuned on instruction-following
044   data to create specialized variants such as Mixtral-8x22B-Instruct-v0.1 2 , enhancing their ability to
045   follow human instructions. While LLMs are powerful, fine-tuning for a specific task to enhance
046   performance is generally more practical and efficient than multitask fine-tuning that often encounters
047   conflicting objectives, mode collapse, and demands meticulous data mixing along with substantial
048   training resources (Team et al., 2023; Touvron et al., 2023b). For example, DeepSeek-Coder-V2-
049
      Base (Zhu et al., 2024), a 236B-parameter MoE code model, is fine-tuned from DeepSeek-V2 (Liu
      et al., 2024a) to achieve significantly improved performance in the code domain but demonstrating
050
      reduced effectiveness in general question-answering tasks. This highlights the necessity of obtaining
051
      multiple task-specific LLMs.

069   However, serving multiple models poses several major challenges. First, even with a relatively small
070
      number of models, the storage demands are significant due to the extensive number of parameters each
      model contains. For example, three LLaMA-2-70B models would collectively require over 384GB of
071
      storage, calculated as 128GB per model times three. Second, the substantial memory requirements of
072
      these models may make it impractical to load all of them into GPU memory simultaneously. While
073   dynamically swapping model weights in and out of GPU memory as needed is feasible, the large
074   size of the models makes this process slow and inefficient, significantly delaying response times and
075   adversely affecting user experience.
076
      To address the above challenges, existing methods (Liu et al., 2024b; Yao & Klimovic, 2023)
077
      decompose the weights of fine-tuned models into pre-trained weights and delta weights introduced
078
      during fine-tuning. While low-rank approximation (Hu et al., 2022) can compress these delta weights,
079   it falls short for full fine-tuned models whose delta weights lack low-rank properties (Liu et al., 2024b;
080   Lialin et al., 2024; Hao et al., 2024). As a result, prior work has adopted per-tensor quantization (Liu
081   et al., 2024b) as an alternative, significantly reducing storage needs and facilitating efficient sharing
082   of the base model’s storage across multiple models. Nevertheless, this method ignores the distinction
083   in delta weight values across different input and output channels, resulting in substantial quantization
084   error. To mitigate this, output channel-wise quantization (Xiao et al., 2023; Wei et al., 2023; Liu et al.,
085   2024d) where each output channel is allocated its own learnable step size, still neglects input channel
086   variations, as shown in Figure 1. To further mitigate information loss, rescaling input channels before
087
      quantization can be employed, but this provides only limited alleviation at extremely low bitwidths.
088   In this paper, we propose ME-Switch, a memory-efficient expert switching framework tailored for
089   LLMs. To reduce the quantization error while simultaneously reducing storage needs, we develop a
090   mixed-precision quantization method that quantizes non-salient input channels of delta weights to
091   extremely low bits while preserving those salient ones, which could substantially increase quantization
092   errors when quantized at very low bitwidths, in full precision. Since the number of salient input
093
      channels is relatively small, incorporating a limited amount of high-precision delta weights incurs
      negligible memory overhead and inference cost. To identify the important input channels of delta
094
      weights, one may select based on their magnitudes (Dettmers et al., 2022), as shown in Figure 2(a),
095
      which however fails to capture those leading to high quantization error. In contrast, our approach
096   identifies salient input channels of delta weights based on their impact on reconstruction errors in the
097   output activations, as illustrated in Figure 2(b).
098
099
      Our contributions can be summarized as follows. 1) We introduce ME-Switch, a memory-efficient
      framework designed for serving multiple LLMs. It only stores a single full-precision pre-trained
100
      model and dynamically loads the appropriate compressed delta model weights in response to user
101
      queries. 2) We develop a mixed-precision quantization method that significantly reduces the storage
102   demands of serving multiple LLMs while maintaining performance, which is achieved by selectively
103   quantizing the non-salient input channels of delta weights and leaving the salient ones unchanged. 3)
104   We conduct extensive experiments demonstrating the promising memory efficiency and accuracy of
105   ME-Switch. Remarkably, when serving three models from the Mistral-7B family, ME-Switch not
106   only delivers near-lossless performance on instruction, mathematical reasoning, and code generation
107   tasks but also reduces the model size by 2.04×. More impressively, our method is able to serves up
      to 16 Mistral-7B models on a single NVIDIA A100 GPU without running out of memory.

      2   R ELATED W ORK

      Efficient LLM serving. LLMs can be efficiently deployed on GPUs for high-throughput serving
111   using several inference frameworks, such as vLLM (Kwon et al., 2023) and Orca (Yu et al., 2022).
112   Given that a single LLM cannot excel across all domains, it is crucial to serve multiple LLMs
113   simultaneously to handle diverse user queries effectively. To determine which model to use during
114   inference, Zooter (Lu et al., 2023) utilizes a task-level router to load different LLMs based on the
115   incoming task requirements. This approach, however, introduces significant memory challenges, as
116   hosting all LLMs on a GPU simultaneously can excessively strain the available VRAM. One solution
117   is to serve multiple Low-Rank Adaptation (LoRA) modules within a multi-tenant serving system,
118   such as Punica (Chen et al., 2023) and S-LoRA (Sheng et al., 2023a). However, these methods fail
119
      to accommodate full fine-tuned models because such models typically do not display the low-rank
      characteristics necessary for LoRA approximation. Another viable strategy to address the above
120
      issue involves dynamically loading different LLMs from CPU memory to GPU memory as needed,
121
      thereby reducing peak GPU memory utilization. Nevertheless, this dynamic loading competes for
122   GPU memory bandwidth with model-swapping operations, presenting a significant bottleneck. To
123   mitigate these challenges, techniques like Deltazip (Yao & Klimovic, 2023) and BitDelta (Liu et al.,
124   2024b) have been developed to compress the delta parameters. These methods allow task-specific
125   delta parameters to be loaded into GPU memory on-demand, ensuring that only a single pre-trained
126   model’s parameters reside permanently in GPU memory. This approach aims to achieve low-latency
127   inference while minimizing the costs associated with maintaining numerous fine-tuned models in
128   GPU memory. Our approach stands out by serving multiple LLMs with a minimal memory footprint
129   with nearly lossless performance, which is achieved through an advanced delta weights quantization.
130   Delta compression. In recent years, numerous approaches have focused on reducing the storage
131   overhead for maintaining different task-specific models through delta parameter compression. Model
132   merging strategies often incorporate multiple tasks’ delta parameters (Ding et al., 2023) into the
133   pretrained parameters to minimize the number of parameters needed for multi-task operations. To
134   address the parameter conflicts that often arise during model merging, Yu et al. (Yu et al., 2023a)
135   proposed a method involving massive unstructured random pruning (achieving 90% sparsity) of
136
      delta parameters and subsequent rescaling. This approach ensures that the accuracy of downstream
      tasks is not compromised. In a similar vein, Ties-merging (Yadav et al., 2024) developed a pruning
137
      strategy based on the magnitude and sign of delta parameters to further reduce conflicts and storage
138
      needs. Additionally, the LoRA-based PEFT methods (Hu et al., 2021; Valipour et al., 2022; Ping
139   et al., 2024) introduce a novel approach by learning one or several low-rank matrices to represent the
140   delta parameters. ZipLoRA (Shah et al., 2023) explores the sparsity within these low-rank matrices,
141   allowing the fusion of low-rank matrices from different tasks to further reduce the spatial overhead of
142   multi-task models. However, these methods struggle to accurately approximate delta parameters in
143   full fine-tuned models, as these models usually lack low-rank properties (Liu et al., 2024b; Lialin
144   et al., 2024; Hao et al., 2024). Conversely, some studies focus solely on compressing delta parameters
145   for each task without merging, thereby mitigating performance degradation across multiple tasks.
146   For example, Ryu et al. (Ryu et al., 2023) and Isik et al. (Isik et al., 2023) combine quantization and
147
      low-rank estimation techniques to reduce the storage size of delta parameters. Liu et al. (Liu et al.,
      2024b) push this further by quantizing each task’s delta parameters to 1 bit, achieving more than a
148
      tenfold compression. However, these quantization methods often lack robust handling of outliers,
149
      resulting in performance declines compared to uncompressed delta parameters. Moreover, models
150   that do not integrate multi-task delta parameters require manual activation of specific delta parameters
151   for each task, which reduces the model’s applicability in multi-task environments.

192   4      P ROPOSED M ETHOD
193
194   In this section, we propose ME-Switch, a memory-efficient expert switching framework designed to
195   optimize the deployment of multiple LLMs. ME-Switch reduces memory demands while maintaining
196   performance by using mixed-precision quantization, which quantizes non-salient weights to extremely
197   low bitwidths while keeping salient weights intact.
198
199   4.1        S ALIENT-AWARE D ELTA C OMPRESSION
200
201
      In this section, we introduce our salient-aware delta compression approach, which specifically targets
      the preservation of salient input channels during quantization. While quantization methods with
202
      learnable step sizes for each output channel can handle variations in output channels effectively (Xiao
203
      et al., 2023; Liu et al., 2024d), variations in input channels pose a significant challenge to maintaining
204   model performance, as shown in Figure 1. Some salient input channels of delta weights, regardless
205   of their magnitude, can cause substantial information loss and degrade model performance when
206   quantized to a very low bitwidth. To address this, rescaling the input channels of delta weights before
207   quantization (Lin et al., 2024) offers a solution to mitigate some of the quantization error, which
208   however provides only limited alleviation under the context of extremely low-bit quantization.
209
      Salient-aware delta selection. To protect salient input channels of delta weights, we develop a
210   mixed-precision quantization method, where the majority of input channels of delta weights are
211   quantized to low-bit precision, while only a small number of critical input channels are represented in
212   their original precision. This approach achieves significant reductions in storage requirements with
213   minimal performance loss. The remaining challenge lies in identifying these salient input channels.
214   A naive approach might involve selecting the input channels with large magnitudes as the salient
215   delta weights (Dettmers et al., 2022), as shown in Figure 2(a). However, this method neglects the
      influence of input activations on the outputs, failing to identify channels that lead to high quantization

      4.2       M ODEL SIZE REDUCTION ANALYSIS
256
257   Let Ψ be the model size of an BF16 pre-trained model. When storing M BF16 models, the total
258   model size is M Ψ. In contrast, using our method, we store a single base model and M compressed
259   delta models. The total model size is Ψ + M Ψ̃, where Ψ̃ represents the size of the compressed delta
260   model. Therefore, the compression ratio can be computed by M Ψ/(Ψ + M Ψ̃). For example, when
261   serving 9 LLaMA-2-13B models, our method achieves a compression ratio of 3.85×. Empirical
262   studies on compression ratios for varying model numbers are shown in Figures 5 and 6.

504   Latency analysis. To assess the latency improvements from delta weights compression, we measured
505   the end-to-end decoding latency of the Mistral-7B model with an input sequence length of 128
506   on a single NVIDIA A100. Decoding latency is critical, as it typically dominates processing time
507
      in LLM operations (Lin et al., 2024; Liu et al., 2023). Our efficient Triton kernel, which enables
      batched matrix multiplication between multiple compressed weight matrices and high-precision
508
      input activations, is compared against the conventional approach of individually processing multiple
509
      models. Results depicted in Figure 7 illustrate that while our method may perform slightly slower
510   than the naive approach for a small number of models due to additional dequantization overhead, it
511   provides lower latency as the number of models is greater than 4. These results show that our method
512   scales more efficiently than the naive inference method. In the naive method, each xW is computed
513   independently during the forward pass, requiring a distinct W for each user in the batch. As the
514   number of models grows (>= 4), this approach results in substantial I/O costs due to loading of large
515   weight matrices. In contrast, our method leverages shared pre-trained model weights W along with a
516                        ˆ significantly reducing the inference I/O burden. More importantly, unlike the
      set of small deltas ∆,
517   naive approach, our method is able to simultaneously serve 16 models on GPUs without running into
518   out-of-memory (OOM) issues, demonstrating better scalability and efficiency in high-load scenarios.
519   We further provide a detailed breakdown of decoding times for Mistral-7B model in Figure 8. We
520   observe that the dequantization cost is very small across different model numbers. Initially, latency is
521   dominated by I/O operations because LLM decoding is a memory-bound process when the batch size
522   is small (Lin et al., 2024; Liu et al., 2023). However, as the number of models grows, compute-related
523   operations, such as matrix multiplications, begin to dominate the overall latency. Notably, as the
524   number of models increases, the increased latency attributed to x∆    ˜ exceeds that of xW, primarily
525   due to the increased multiplication cost of the more compressed delta weights.

531
      In this paper, we have introduced ME-Switch, a memory-efficient expert switching framework
      designed for LLMs. Our method has addressed the critical challenge of balancing model performance
532
      with storage efficiency. The core of our ME-Switch lies in a novel mixed-precision quantization
533
      method that selectively compresses non-salient delta weights to extremely low-bit precision while
534   preserving salient delta weights. Extensive experiments on Mistral-7B, LLaMA-2-13B, and LLaMA-
535   3-8B families have demonstrated that ME-Switch achieves performance comparable to unquantized
536   expert models across various tasks while significantly reducing model size. In terms of limitations,
537   quantizing the base model itself could further reduce the overall model size. This approach would
538   require careful consideration of the combined effects of quantizing both the base model and the delta
539   weights to ensure performance is maintained. Furthermore, reducing the bitwidth of the KV Cache
      could accelerate the decoding speed, offering additional efficiency improvements.

759   A    M ODEL - LEVEL ROUTING
760
      In public-facing applications or open-ended systems, user inputs may vary widely in content and
761   intent, often lacking clear contextual information. This makes it particularly challenging to determine
762   the appropriate model for queries in advance. Therefore, we explore a simple yet effective approach as
763   a possible solution to determine the appropriate model for a given user query. Consider a set of LLMs
764   represented as F = {f1 , f2 , · · · , fM }, where M denotes the number of models. Given a user query
765   q, we aim to find the most suitable LLM by solving the following problem arg maxf ∈F P (q, f (q)),
766   where P is a function that measures the quality or performance of the LLM response. To simplify the
767   routing process, we assume that each LLM in F specializes in distinct and non-overlapping domains
768   such as code generation or mathematical problem solving. This setup allows us to treat the routing
769
      challenge as a multiple-choice question-answering task, where each option corresponds to a specific
      domain, thereby transforming the problem into a domain classification problem. For the challenging
770
      problem of overlapping domains, extending the current setup to a multi-label classification framework
771
      would be necessary, and we consider this a promising direction for future work. Note that dialogue
772   LLMs like Qwen1.5-1.8B-Chat (Bai et al., 2023) exhibit capabilities in following instructions, which
773   inspires us to utilize a small pre-trained LLM as a model-level router. As illustrated in Figure A,
774   we first prompt the router with the user’s question using a template designed to elicit domain
775   classification. Specifically, when a user query is received, it is embedded into the prompt template
776   to form a structured question, as shown in Table A. This structured question is then processed by
777   the router to perform domain classification. Based on the router’s response, we then dynamically
778   load the corresponding compressed delta weights for the selected domain-specific model, such as a
779   mathematical model, to generate outputs.

809   Comparisons with routing in Mixture of Experts (MoE). Besides model-level routing, another
      approach to handle diverse user queries efficiently is to construct a MoE using a set of pre-trained

810
      expert models. This can be achieved by integrating the feedforward layers from all pre-trained
811   LLMs into a single MoE module at each attention-FFN block, and merging other layers, such as
812   self-attention layers, by simply averaging their weights (Sukhbaatar et al., 2024). An additional
813   gate network is introduced for each MoE module to perform token-level routing. However, this
814   approach requires extensive fine-tuning of the entire network parameters and the gating network, as
815   there is a significant gap between MoE experts and pre-trained LLMs. For example, training an MoE
816   using existing Math, Code, and Wikipedia experts requires over 900 GPU days (Sukhbaatar et al.,
817   2024). Notably, each expert in an MoE tends to become a generalist across all domains due to the
818   load balancing loss, which encourages an even distribution of the workload among experts (Fedus
819
      et al., 2022; Jiang et al., 2024). This contrasts with pre-trained expert models, which are typically
      specialized for specific domains. Compared with MoE, our model-level routing has a significantly
820
      lower training cost, as we only need to train a router while keeping the expert models frozen.

      def twobit dequant bmm scale kernel(a ptr, b ptr, c ptr, scales ptr, M, N, K, stride am, stride ak, stride bk, stride bn,
865          stride cm, stride cn, stride scales, stride batch a, stride batch b, stride batch c, stride batch scale, BLOCK SIZE M: tl.
             constexpr, BLOCK SIZE N: tl.constexpr, BLOCK SIZE K: tl.constexpr, GROUP SIZE M: tl.constexpr, ACTIVATION: tl.constexpr,
866   ):
867       """Kernel for computing the matmul C = A x B.
          A has shape (B, M, K), float
868       B has shape (B, K//n bits, N), int, packed boolean
          C has shape (B, M, N),
869       scales is of shape (N) float16
          """
870       # −−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−
871       # Map program ids ‘pid‘ to the block of C it should compute. This is done in a grouped ordering to promote L2 data reuse.
                  See above ‘L2 Cache Optimizations‘ section for details.
872       pid = tl.program id(axis=0)
          pid batch = tl.program id(axis=1)
873
         num pid m = tl.cdiv(M, BLOCK SIZE M)
874      num pid n = tl.cdiv(N, BLOCK SIZE N)
         num pid k = tl.cdiv(K, BLOCK SIZE K)
875
876      num pid in group = GROUP SIZE M ∗ num pid n
         group id = pid // num pid in group
877      first pid m = group id ∗ GROUP SIZE M
         group size m = min(num pid m − first pid m, GROUP SIZE M)
878
         pid m = first pid m + (pid % group size m)
879      pid n = (pid % num pid in group) // group size m
880
         offs m = (pid m ∗ BLOCK SIZE M + tl.arange(0, BLOCK SIZE M)) % M
881      offs n = (pid n ∗ BLOCK SIZE N + tl.arange(0, BLOCK SIZE N)) % N

882      offs am = tl.max contiguous(tl.multiple of(offs m, BLOCK SIZE M), BLOCK SIZE M)
         offs bn = tl.max contiguous(tl.multiple of(offs n, BLOCK SIZE N), BLOCK SIZE N)
883      offs k = tl.arange(0, BLOCK SIZE K)
884
         a ptrs = a ptr + (offs am[:, None] ∗ stride am + offs k[None, :] ∗ stride ak) + pid batch ∗ stride batch a
885
         # Adapted from GPTQ−Triton (https://github.com/fpgaminer/GPTQ−triton)
886      # b ptrs is set up such that it repeats elements along the K axis n bits times
         b ptrs = b ptr + ((offs k[:, None] // 16) ∗ stride bk + offs bn[None, :] ∗ stride bn) + pid batch ∗ stride batch b
887      scales ptrs = scales ptr + offs bn ∗ stride scales + pid batch ∗ stride batch scale
888
         # (BLOCK SIZE K, BLOCK SIZE N)
889      # shifter is used to extract each bit of each element in the int matrix
         shifter = (offs k % 16) ∗ 2
890      scales = tl.load(scales ptrs)
891      # −−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−
         # Iterate to compute a block of the C matrix.
892      # We accumulate into a ‘[BLOCK SIZE M, BLOCK SIZE N]‘ block
893      # of bf32 values for higher accuracy.
         # ‘accumulator‘ will be converted back to bf16 after the loop.
894      accumulator = tl.zeros((BLOCK SIZE M, BLOCK SIZE N), dtype=tl.float32)
         for k in range(0, num pid k):
895          # Load the next block of A and B, generate a mask by checking the K dimension.
             # If it is out of bounds, set it to 0.
896          a = tl.load(a ptrs)
897          # b = tl.load(b ptrs, mask=offs k[:, None] < K − k ∗ BLOCK SIZE K, other=0)
             b = tl.load(b ptrs) # (BLOCK SIZE N,)
898
             # Convert B from int to a.dtype
899          # b: (BLOCK SIZE K, BLOCK SIZE N)
             b = (b >> shifter[:, None]) & 0x3
900          b = (b − 2).to(a.dtype)
901          b = b ∗ scales[None, :] # BF16
             # b = b.to(a.dtype)
902
             # We accumulate along the K dimension.
903          accumulator += tl.dot(a, b)
             # Advance the ptrs to the next K block.
904          a ptrs += BLOCK SIZE K ∗ stride ak
905          # b ptrs += BLOCK SIZE K ∗ stride bk
             b ptrs += (BLOCK SIZE K // 16) ∗ stride bk
906      # You can fuse arbitrary activation functions here
         # while the accumulator is still in BF32!
907      # if ACTIVATION == "leaky relu":
         # accumulator = leaky relu(accumulator)
908      c = accumulator.to(tl.float16)
909      # −−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−−
910      # Write back the block of the output matrix C with masks.
         offs cm = pid m ∗ BLOCK SIZE M + tl.arange(0, BLOCK SIZE M)
911      offs cn = pid n ∗ BLOCK SIZE N + tl.arange(0, BLOCK SIZE N)
         c ptrs = (
912          c ptr
             + stride cm ∗ offs cm[:, None]
913          + stride cn ∗ offs cn[None, :]
914          + pid batch ∗ stride batch c
         )
915      c mask = (offs cm[:, None] < M) & (offs cn[None, :] < N)
         tl.store(c ptrs, c, mask=c mask)
