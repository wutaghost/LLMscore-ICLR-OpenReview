011                                               A BSTRACT
012
013            The reliability and security of WebAssembly (Wasm) binaries are crucial for mod-
014            ern web development, yet effective testing methodologies remain undeveloped.
015
               This paper addresses the gap in Wasm binary testing by proposing a novel ap-
               proach for test cases generation, leveraging Large Language Models (LLMs) to
016
               enhance test coverage and bug detection. Traditional testing approaches typically
017
               require access to source code, which is often unavailable for Wasm binaries. Our
018            generate-then-test methodology overcomes this limitation by generating equiva-
019            lent C++ code to simulate expected Wasm behavior, creating and mutating test
020            cases in C++, and compiling these tests to evaluate them against the Wasm bi-
021            nary. Key contributions include automated test case generation using LLMs and
022            improved code coverage through type-aware mutations, with comprehensive eval-
023            uation demonstrating the effectiveness of our approach in detecting subtle bugs in
024            Wasm binaries, thereby ensuring more reliable Wasm applications.

      1   I NTRODUCTION
027
028
      WebAssembly (Wasm) is a low-level, portable bytecode language designed for high-performance
029
      computations, providing near-native execution speeds and extensive cross-platform compatibility.
030   Initially developed for web applications (Haas et al., 2017), Wasm is now supported by all major
031   browsers, including Chrome, Firefox, Safari, and Edge (McConnell, 2017). Its use has since ex-
032   panded to various domains, including mobile devices (Pop et al., 2022), smart contracts (McCallum,
033   2019), blockchains (Bian et al., 2019; Protzenko et al., 2019), and the Internet of Things (Gur-
034   deep Singh & Scholliers, 2019; Liu et al., 2021).
035
      With its increasing adoption, the demand for in-depth analysis and testing of Wasm has become
036   increasingly critical. While Wasm is commonly used for legitimate applications, it has also been
037   exploited for malicious purposes, such as cryptojacking, where Wasm code is covertly executed in
038   browsers to mine cryptocurrencies (Konoth et al., 2018; Kharraz et al., 2019; Musch et al., 2019b;
039   Romano et al., 2020), highlighting the importance of thoroughly understanding Wasm. As a com-
040   pilation target for high-level languages such as C, C++, Go, and Rust, Wasm presents unique chal-
041   lenges, particularly when distributed as third-party modules without access to the original high-level
042   source code (Musch et al., 2019a; Romano & Wang, 2023). These challenges are further exacerbated
043   by the fact that 28.8% of Wasm binaries are minified (Hilbig et al., 2021), often with obfuscated
044
      variable names and potentially flaky test suites (Liu et al., 2024b), making interpretation and testing
      more difficult. These issues underscore the urgent need for developing robust test suites to ensure
045
      WebAssembly modules are both benign and functionally correct.
046
047   Despite Wasm’s growing significance in modern web development, effective testing methods re-
048   main underdeveloped. A key challenge in testing Wasm binaries is the lack of access to original
049   source code and corresponding test cases, which renders traditional testing approaches that depend
050
      on source code availability not applicable (Pacheco et al., 2007; Choi et al., 2019; Han et al., 2019;
      Watson et al., 2020). This calls for an automated solution capable of generating relevant test cases
051
      to validate the functionality of Wasm binaries without relying on the original source code.
052
053   This paper addresses the gap in Wasm binary testing by proposing a novel approach for test case
      generation, leveraging Large Language Models (LLMs) to improve test coverage and bug detection.

054
      The core of our approach lies in a generate-then-test methodology, inspired by Yu et al. (2022),
055   designed to evaluate the functionality and correctness of WebAssembly binaries. We begin by gen-
056   erating functionally equivalent C++ code based on a high-level understanding of the Wasm binary’s
057   intended functionality. This step allows us to leverage the extensive toolsets and debugging capa-
058   bilities available for C++, facilitating more robust and efficient testing. Moreover, using C++ as an
059   intermediate representation streamlines test case generation and mutation, ultimately leading to a
060   more comprehensive evaluation of the Wasm binary. This generated C++ code serves as the founda-
061   tion for crafting initial test cases, covering various aspects of the binary’s functionality. To improve
062   test coverage, we apply type-aware mutations to refine these initial tests, producing diverse inputs
063
      to uncover edge cases and potential bugs. Finally, these mutated test cases are compiled with the
      original Wasm binary to verify its behavior across all scenarios. This approach is implemented in
064
      our tool, WasmTest, which automates test generation and validation for Wasm binaries.
065
066   In summary, our work makes the following contributions:
067
             • Automated Test Case Generation: We leverage LLMs to automate the generation of in-
068
               termediate C++ code and test cases from high-level descriptions of Wasm functionalities.
069
070          • Enhanced Code Coverage: Our type-aware mutation strategy significantly enhances test
071
               case diversity and coverage, enabling the detection of subtle bugs in Wasm binaries.
072          • Evaluation and Validation: We demonstrate the effectiveness of WasmTest through met-
073            rics such as compile rate, code coverage, test correctness, and bug detection rate.

077   2.1   W EBA SSEMBLY I NTERPRETATION
078
079   In the realm of WebAssembly understanding, significant efforts have been made towards recover-
080
      ing data and function types, generating function summaries, and decompiling Wasm to high-level
      programming languages. For example, Lehmann & Pradel (2022) leveraged LSTM to recover high-
081
      level input parameters and return data types for Wasm functions. Romano & Wang (2023) devel-
082
      oped semantics-aware intermediate representations for Wasm functions and trained effective ma-
083   chine learning classifiers to predict module and function purposes. Huang & Zhao (2024) pretrained
084   a BERT language model for WebAssembly on a multi-modal corpus of C/C++ source code, natural
085   language documentation and corresponding WebAssembly, achieving boosted performance on type
086   recovery and code summarization. Furthermore, efforts to decompile Wasm into high-level C/C++
087   snippets have incorporated fine-tuning or symbolic program analysis into LLMs (She et al., 2024;
088   Fang et al., 2024). Despite these advances, no prior work has focused on automatically generating
089   software tests for WebAssembly, making our approach as the first to address this need.
090   There are several fuzzing tools that generate valid, randomized WebAssembly binaries to test run-
091   times or tools. Wasm-smith (Nick Fitzgerald, Alex Crichton, 2024) focuses on creating fuzzed mod-
092   ules for testing the robustness of runtimes, validators, or parsers; Wasm-mutate (Cabrera-Arteaga
093   et al., 2024; Arteaga et al., 2022) mutates Wasm programs to produce semantically equivalent vari-
094   ants for fuzzing compilers; and Wasm-maker (Cao et al., 2024) performs differential testing of
095   runtimes with syntactically and semantically rich binaries. These works target the infrastructure
096   around Wasm, while our approach, WasmTest, directly generates test cases for Wasm code to verify
097
      its functionality and ensure it is bug-free.
098
099   2.2   LLM S FOR C ODE
100
      Recent progress in LLMs, particularly code-specific models like CodeLlama (Rozière et al., 2023),
101
      have demonstrated their remarkable proficiency in handling code-related tasks (Hou et al., 2023; Fan
102   et al., 2023). Their ability to grasp intricate semantics and structural nuances makes them highly ef-
103   fective in tasks requiring deep insights into program semantics and structures. These advancements
104   have opened new possibilities for the automated generation of test suites for code snippets. Recently,
105   LLMs have shown strong performance in binary or assembly-level code reverse engineering (Pearce
106   et al., 2022; Xu et al., 2023; Wong et al., 2023; Al-Kaswan et al., 2023; She et al., 2024; Fang et al.,
107   2024). Our work builds on these advancements by applying LLMs to generate comprehensive, se-
      mantically meaningful test inputs for Wasm binaries.

      Researchers have also integrated LLMs with traditional test generation techniques to enhance auto-
109   mated testing. CODAMOSA (Lemieux et al., 2023) uses LLMs to generate inputs when traditional
110   fuzzing stalls; ChatUniTest (Xie et al., 2023) refines Java test cases iteratively; MuTAP (Dakhel
111   et al., 2024) generates targeted tests with intentionally created program errors; TitanFuzz (Deng
112   et al., 2023a) and FuzzGPT (Deng et al., 2023b) specialize in creating inputs for deep learning li-
113   braries; and Fuzz4All (Xia et al., 2024) provides a generic framework for generating tests across
114   programming languages using documentation or specifications.
115
116   2.3   AUTOMATED T EST G ENERATION
117
118   Automated test generation is a widely-used approach for identifying software bugs. Black-box
119
      testing, such as fuzz testing (Miller et al., 1990; Woo et al., 2013), feeds random inputs to the
      system under test (SUT) or function under test (FUT) without accessing its source code (Nidhra &
120
      Dondeti, 2012; Woo et al., 2013). In contrast, white-box testing analyzes the source code to produce
121
      higher-quality test cases (Pacheco et al., 2007; Nidhra & Dondeti, 2012). For example, symbolic
122   execution (King, 1976) overcomes coverage limitations by solving symbolic path constraints to
123   generate tests that explore deeper paths. Coverage-guided fuzzing (Serebryany, 2016), or grey-box
124   testing, strikes a balance by using coverage information from the SUT to refine input generation
125   and mutation. Recently, LLMs have been used to generate semantically meaningful test inputs for
126   problems coded in dynamically-typed languages when traditional approaches fail (Liu et al., 2024a).
127   Our work focuses on directly generating tests for binary-level languages like WebAssembly, using
128   LLMs to create high-quality, semantically rich test inputs (i.e., white-box), which are then scaled to
129   a larger amount through type-aware mutation (i.e., black-box).

133   Figure 1 provides an overview of WasmTest, which takes as input a Wasm snippet, its intended
134   functionality summary, and optionally example tests. WasmTest begins by constructing a prompt to
135   generate multiple functionally equivalent C++ implementations filtered by the example tests and a

139
      compiles the C++ test cases with the original Wasm code and evaluates them in a WebAssembly
      runtime. Finally, metrics are reported to evaluate the effectiveness of the generated tests.

188   3.1   G ENERATING F UNCTIONALLY E QUIVALENT C++ C ODE .
189
190   The first step in WasmTest is generating C++ code that accurately represents the Wasm binary’s
191
      high-level functionality. This step is critical as it enables the use of C++’s extensive toolsets and
      debugging capabilities, thereby enhancing the robustness of correctness validation. The input to this
192
      process is a natural language description of the Wasm snippet’s functionality, which can be pro-
193
      vided directly or derived through summarization and decompilation techniques (Fang et al., 2024;
194   She et al., 2024; Huang & Zhao, 2024). WasmTest constructs a code generation prompt by com-
195   bining few-shot demonstration examples with the functionality summary and feeds it into a LLM.
196   The LLM generates multiple C++ implementations, which are then validated using example tests.
197   Additionally, a majority-voting filtering step is employed, where an implementation is considered
198   correct only if it produces the same output as more than half of the other generated implementations
199   for the given use cases. Only those implementations deemed correct proceed to the next phase.
200
201   3.2   G ENERATING S EED C++ T EST C ASES
202
203   After generating the correct C++ implementations, WasmTest proceeds to create seed test cases.
204   We restrict the LLMs to generating test inputs only, excluding their suggested expected outputs
205
      due to observed inconsistencies in their output values. To address this limitation, WasmTest runs
      these LLM-generated inputs through the validated C++ implementations (as discussed in Section
206
      3.1) to compute the correct expected outputs. This guarantees that the tests faithfully reflect the
207
      code’s actual behavior. By taking this approach, WasmTest capitalizes on the LLM’s strengths in
208   generating high-quality test inputs, while mitigating potential inaccuracies in its output generation.
209
210

291   4.2   E VALUATION M ETRICS
292
293   Compile Rate. We measure the percentage of generated test cases that successfully compile with
294   both the original C++ and WebAssembly code under test. For C++ compilation, we use GCC with
295   the -std=c++17 flag to compile the tests alongside the function code. For WebAssembly, we
296   use Emscripten with the -sMAIN MODULE=1 flag, compiling the test cases as the main module,
297
      aligning function name, and linking them to the WebAssembly code. We report the percentages of
      successfully compiled test cases in C++ (% C++ Compile) and WebAssembly (% Wasm Compile).
298
299   Code Coverage. Code coverage, also referred to as test coverage, is a metric that measures how
300   much of a program’s source code is executed during testing. Higher code coverage generally sug-
301   gests a lower likelihood of undetected software bugs. We measure two types of coverage: C++
302   code coverage and WebAssembly code coverage. For C++ code coverage, we use Gcov (Project,
303
      2024), which can be used in conjunction with GCC to measure how frequently each line of code
      is executed and identify which lines are actually covered during testing. For WebAssembly, we
304
      use Wasabi (Lehmann & Pradel, 2019), a dynamic analysis tool for WebAssembly based on binary
305
      instrumentation. Wasabi inserts additional WebAssembly code between the program’s original in-
306   structions to call into JavaScript-based analysis functions. We implement these analysis functions
307   as hooks to perform operations whenever a particular instruction is executed. To obtain instruction
308   coverage, we record the location of each executed instruction.
309
      Pass Rate. Pass rate measures the percentage of generated tests that run successfully with function
310
      under test without errors. A higher pass rate indicates that the generated test cases align with the ex-
311
      pected functionality. Our assumption is that if the Wasm binary under test is correctly implemented,
312   all generated tests should pass, which indicate that the test inputs are valid, the expected outputs are
313   correct, and the tests compile successfully with the original Wasm binary.
314
315
      Bug Detection Rate. Achieving high coverage does not necessarily mean that bugs will be detected.
      To address this, we evaluate the effectiveness of our generated tests in detecting bugs within a subset
316
      of buggy implementations. A higher bug detection rate suggests that the generated test cases are
317
      comprehensive and robust in identifying incorrect code. In addition to evaluating existing buggy
318   code, we also apply mutation testing to randomly selected evaluation data, manually inserting bugs
319   into the code. Mutation testing (Budd, 1980) is considered as a more precise assessment of test qual-
320   ity than code coverage alone (Petrović & Ivanković, 2018). This process creates multiple artificial
321   buggy versions of the program, called a mutant, each containing one deliberately inserted, subtle
322   error. The test suite’s quality is measured by the number of mutants it can detect, a process known
323   as “killing” the mutants. In line with previous works (Shi et al., 2014; Liu et al., 2024a), we use the
      percentage of mutants killed as a key metric for evaluating bug detection rate.

357   5.1    RQ1: G ENERATION OF C OMPILABLE T ESTS
358
359   In this research question, we evaluate WasmTest’s ability to generate compilable C++ functions and
360   test cases across evaluation tasks, using the performance of various LLMs. Figure 3 shows the valid
361   file counts that were successfully generated and compiled at each task stage for each model.
362   In Figure 3, # Tasks represents the number of WebAssembly binaries under test. WasmTest first
363   generates equivalent C++ function code samples (Section 3.1). The Function Generation phase
364   counts tasks where at least one generated function sample passes example tests. Subsequently, LLMs
365   generate seed test cases (Section 3.2), which undergo type-aware mutation and output correction
366   (Section 3.3). The Test Generation phase indicates the number of tasks yielding a full suite of test
367   cases. Finally, WasmTest compiles the generated test cases with the original code to keep those that

      6     D ISCUSSION
522
523
      The generate-then-test approach leverages LLMs for automated test case generation for binary-level
524   code like WebAssembly, significantly improving test coverage and bug detection for WebAssembly
525   binaries. Despite these strengths, the mutation process introduces a slight reduction in compile rates,
526   suggesting that refinement of mutation strategies is possible to maintain a balance between test thor-
527   oughness and compile success. Additionally, WasmTest currently relies on C++ as an intermediate
528   language. When switched to other programming languages, this approach may be less applica-
529   ble, especially for some low-resource languages that are not commonly seen during LLM training.
530   Future work may aim to extend the methodology to support additional languages and evaluate its
531   effectiveness across a wider range of real-world Wasm applications to ensure broader applicability.
532
533   7     C ONCLUSION
534
535   This paper presents WasmTest, a novel approach for robust testing of Wasm binaries. To address the
536   challenges of testing Wasm binaries without source code, WasmTest employs a generate-then-test
537   methodology that uses LLMs to automatically generate equivalent C++ code, create and mutate test
538   cases, and compile these tests for evaluation against the Wasm binary. Results show that WasmTest
539   is capable of generating comprehensive test cases and significantly enhances the detection of subtle
      bugs, making it a valuable resource for developers and security analysts.
