009                                               A BSTRACT
010
011            Current methods for compressing neural network weights, such as decomposition,
012            pruning, quantization, and channel simulation, often overlook the inherent sym-
013            metries within these networks and thus waste bits on encoding redundant infor-
014            mation. In this paper, we propose a format based on bits-back coding for storing
015            rotationally symmetric Transformer weights more efficiently than the usual array
016            layout at the same floating-point precision. We evaluate our method on Large Lan-
017            guage Models (LLMs) pruned by SliceGPT (Ashkboos et al., 2024) and achieve a
018            3-5% reduction in total bit usage for free across different model sizes and architec-
019
               tures without impacting model performance within a certain numerical precision.

      1   I NTRODUCTION
023
024
      Modern neural networks, particularly Large Language Models (LLMs), typically contain billions
      of parameters. Therefore, encoding and transmitting these models efficiently is gaining widespread
025
      interest. Currently, compression techniques of model weights mainly fall into four categories, in-
026
      cluding decomposition (e.g., Hu et al., 2022; Saha et al., 2023), pruning (e.g., Hoefler et al., 2021;
027   Frantar & Alistarh, 2023; Ashkboos et al., 2024), quantization (e.g., Wang et al., 2023; Xu et al.,
028   2024), and channel simulation (e.g., Havasi et al., 2019; Isik et al., 2023; He et al., 2024).
029
030
      However, these techniques ignore the fact that neural networks typically exhibit symmetries in their
      weight space. For example, in feedforward networks, applying a random permutation to the neurons
031
      in one layer and its inverse to the weights in the subsequent layer leaves the output unchanged.
032
      Encoding weights without accounting for these symmetries will lead to suboptimal codelength.
033
034   In this work, we address this redundancy by developing a practical storage format for model weights
035   that takes symmetries into account to reduce the compressed model size. We demonstrate the prac-
036
      ticality of our method by compressing popular model architectures. Specifically, our contributions
      are as follows:
037
038
      • We propose a practical bits-back coding scheme for rotational symmetries. We apply our approach
039     to Large Language Models (LLMs) pruned by SliceGPT (Ashkboos et al., 2024) and demonstrate
040     that our proposed approach can save additional free bits while preserving prediction accuracy
041     within a certain numerical precision.
042
      • We further showcase that by transmitting a small number of bits as a correction code, we can
043
        rescue the performance drops due to numerical inaccuracies.
044
045   • We perform experiments on the OPT (Zhang et al., 2022) and Llama-2 (Touvron et al., 2023)
046
        across different sizes, where we can save 3-5% additional bits for free. Notably, our method is
        completely training-free and can be executed on a consumer-grade GPU or even CPU. Further-
047
        more, our proposed method only adds minimal overhead to the time it takes to load the model
048
        parameters into memory and does not affect inference latency.

      Bits-back Coding. The motivating idea behind bits-back coding (Hinton & Van Camp, 1993;
082   Townsend et al., 2019) can be summarised as follows: “If we can make multiple equivalent choices
083   to encode something, we should make our choice at random.” Note that transmitting this random
084   choice requires some bits, and the bits-back coding algorithm provides a concrete procedure to re-
085   cover the bits we used to randomize our choice. The procedure is based on the following insight
086   from compression: assuming we have the right coding distribution P , the encoding function of a
087   compressor will output a sequence of uniformly random bits. Therefore, if we run this process in re-
088   verse and run the decoder on a sequence of uniformly random bits, it will output a sample following
089   P ! Therefore, lossless de-compression can be viewed as a computational way of performing inverse
090
      transform sampling, which provides an invertible way to make the aforementioned random choice.
091   To make the bits-back mechanism more precise, assume we have some data x that belongs to some
092   equivalence class rxs. In many cases, encoding only the equivalence class rxs instead of a specific
093   instance x would be enough for the task at hand. Given a new item x and a stream of already
094   compressed bits M, bits-back coding uses the decoder of lossless compressor on M to decode a
095   random element of the equivalence class x1 „ Px|rxs and leaves a shorter message M1 . After this,
096   bits-back coding uses the encoder of the compressor to encode x1 using Px as the coding distribution
097   and append it to M1 . This procedure is reversible and hence decodable, so long as the receiver of
098
      the message can recover x upon seeing x1 . This ensures that x1 can be coded back into the stream to
      recover the original message M. As one of our contributions, in section 3.2, we explain how such a
099
      recovery step can be carried out when x is a weight matrix and rxs is an equivalence class under a
100
      certain rotational symmetry.
101
102   A concern with bits-back coding is its initialization: we need an initial stream of bits M0 to encode
103   the first item. While M0 represents a significant overhead if we only encode a few items, it only
104
      causes a constant overhead and quickly becomes negligible as the number of encoded items grows.

106   most Large Language Models. Its basic component is the transformer block, as shown in Figure 1a.
107   Each block consists of a multi-head attention layer, a LayerNorm (Ba et al., 2016), and a feedforward
      network (FFN). Two residual connections are added around the attention layer and FFN.

      SliceGPT (Ashkboos et al., 2024) is a recently proposed method for pruning weights in Transformer
109   models. The approach leverages the insight that the outcome of LayerNorm (more precisely, RM-
110   SNorm, i.e., x Ð x{||x||) is invariant if we apply a rotation to the input and its inverse to the output.
111   This rotation matrix and its inverse can be absorbed into the weights before and after the normaliza-
112   tion layer. Therefore, by performing PCA on the hidden states, we can choose rotation matrices that
113   align with the principal components. This allows us to prune the rows and columns corresponding
114   to the less significant eigenvalues in the hidden states, effectively reducing the model’s complexity
115   without drastically hurting the performance. We visualize each transformer block after rotation and

      3.2   D ECODING AND E NCODING ROTATION M ATRICES
271
272
      Now, we discuss how to decode/encode a rotation matrix from/to a given bitstream. A naive ap-
273
      proach is to directly decode and encode these D2 entries in a rotation matrix Q P RDˆD , e.g., by
274   float16. However, it is difficult to guarantee that D2 elements decoded from a given bitstream
275   can form a rotation matrix. In fact, a D-dimensional rotation matrix Q has only DpD ´1q{2 degrees
276   of freedom (DOF), which means that we only need to decode and encode DpD ´ 1q{2 floats for the
277   entire matrix. Therefore, the question becomes: (a) how can we construct a random rotation matrix
278   from DpD ´ 1q{2 random floats; (b) how can we recover these floats given a rotation matrix?
279
      Ideally, we aim to generate a uniformly distributed random rotation matrix, i.e., a random rotation
280   matrix from the Haar distribution. Following the method by Stewart (1980), we can construct the
281   matrix by iteratively applying Householder transformations (Householder, 1958).
282
283
      However, this algorithm is difficult to reverse: we need to reverse the householder transformations
      one by one, and hence, we will suffer from large numerical instability. Therefore, we propose a
284
      simple method to generate a rotation matrix. This approach does not result in a uniformly distributed
285
      rotation matrix. However, we found our approach works well in practice. Since our goal is not to
286   design a theoretically optimal algorithm but rather a more practical approach to perform bits-back,
287   we leave a better design for the rotation matrix to future works.
288
289
      We describe the process of decoding and encoding a rotation matrix in Algorithms 3 and 4. Again,
      we employ a bits-back approach for efficiency. In brief, to decode a rotation matrix, we first decode
290
      a symmetric matrix from the bitstream by decoding its diagonal and upper triangular parts and
291
      performing an eigenvalue decomposition. The eigenvalues are then encoded back into the bitstream.
292   To encode this rotation matrix, we first decode its eigenvalues from the bitstream, reconstruct the
293   symmetric matrix via matrix multiplication, and then encode its diagonal and upper triangular parts
294   back into the bitstream. Notably, our approach requires only the number of bits corresponding to
295   DpD ´ 1q{2 floats, which aligns with the degrees of freedom of a random rotation matrix.

298   3.3   P UTTING T HINGS T OGETHER AND H ANDLING N UMERICAL I NACCURACY
299
300   Having discussed the canonical direction for the transformer and the algorithm for decoding and
301   encoding a rotation matrix, we detail the complete algorithm for encoding and decoding the en-
302   tire transformer using bits-back in Algorithms 5 and 6, respectively. In these algorithms, we
303   use Encode to and Decode from to represent the process of appending or popping arrays of
304   float16 values into or from the current bitstream.
305
      However, since we only save rotated weights in finite precision (e.g., float16), we may suffer
306   from numerical inaccuracy, and hence the rotation matrix recovered by Algorithm 2 in decoding
307   will have deviations from the original rotation matrix applied to the canonical weights in encoding.
308   This will lead to two undesirable outcomes: (1) the bitstream after re-encoding the rotation matrices
309   (as shown in lines 7 and 13 in Algorithm 6) will contain errors, which will affect the weights decoded
310   subsequently from this bitstream; (2) the weight matrices rotated back to the canonical direction (as
311   shown in lines 6 and 12 in Algorithm 6) will contain errors.
312   The first error can be fatal in standard bits-back coding, as they are usually implemented using a
313   variable-length code, such as asymmetric numeral systems (Duda, 2009; Townsend et al., 2019).
314   Such a system is very sensitive to decoding errors: as the code assigns different codelengths to sym-
315   bols by design, if the decoder can only recover the compressed data approximately due to numerical
316   errors, not only are they not getting the correct bits back, they might not even get the correct number
317   of bits back. If the decoder makes such an error even once, it misaligns the rest of the bitstream (i.e.,
318   it will be longer or shorter than it should be), causing catastrophic decoding errors.
319   On the other hand, our proposed method is robust to such errors because we implement bits-back
320   coding with a fixed-length code: we set a floating-point precision (e.g., 16 bits) ahead of time. Then,
321   each encoding and decoding operation will change the message length by the same amount: for a
322   fixed precision, we can compute the total codelength of the model ahead of time. Importantly, this
323   means that any decoding error remains local: if we do not recover a given weight w exactly, this will
      only affect the value of w but will not affect the rest of the bitstream.

      correction code to correct errors exceeding a certain threshold. Specifically, errors in (a) occur in the

      DpD ` 1q{2 floats obtained by Algorithm 3 when encoding the rotation matrix to the bitstream, and
379   errors in (b) occur when rotating the weight matrices back to the canonical direction. Note that the
380   encoder can simulate both procedures during encoding to determine the exact value that the decoder
381   will obtain. If the error between the value obtained by the decoder and the one held by the encoder
382   exceeds a certain threshold, the encoder can send a correction code containing the positions and
383   the true values in float16. Correcting each value will require approximately 16 ` rlog2 Ls bits,
384   where L is the total number of values the decoder will reconstruct that can have errors. For example,
385   L “ DpD ` 1q{2 for the error caused by (a), and L represents the total number of parameters in the
386   weight matrix for the error caused by (b).
387   A natural concern is that the correction code could become large if there are too many errors. Fortu-
388   nately, as we show in Figure 2, only a tiny portion of values have relatively large errors. Therefore,
389   the correction code requires only a small number of bits to transmit and does not significantly im-
390   pact the overall coding efficiency. It is worth noting that this correcting strategy can be considered
391   a simple error-correction code. Therefore, we may be able to adopt more complex error-correction
392   codes, but we leave this design for future exploration.

      It is also possible to parallelize the encoding and decoding of individual layers for further accelera-

      implementation below. We note that standard bits-back coding typically cannot support paralleliza-

      tion because it relies on decoding variable-length bits from a bitstream from previous samples. In

508   mined. If the embedding layer cannot provide enough bitstream to decode random rotation matrices
509   for all layers, we can divide the layers into multiple groups and encode the layers within each group
510   in parallel. We then encode and decode each group in parallel, using bits from the previous group of
511   layers or the embedding layer for the first group.

      5   C ONCLUSION , L IMITATIONS AND F UTURE D IRECTIONS
515
516   In this work, we introduce bits-back coding to encode Large Language Models pruned with
517   SliceGPT. Our approach can save 3-5% additional bits almost for free across several different archi-
518   tectures and sizes. While bits-back coding has long been applied in data compression, its application
519
      to neural networks, where redundancy and symmetry are prevalent, has been underexplored. Our
      work attempts to bridge this gap, opening a new direction for model compression. A key takeaway
520
      is that by re-parameterizing and pre-processing network weights to explicitly capture symmetries,
521
      as demonstrated in SliceGPT, we can leverage bits-back coding to eliminate redundant bits.
522
523   Future research can focus on designing improved algorithms for encoding and decoding the random
524   rotation matrix, developing better error-correction codes to manage large deviations due to numer-
525
      ical instability, and integrating our method with other model compression techniques, such as the
      extremely quantized networks (Ma et al., 2024). Our method’s major concern is the numerical in-
526
      stability. While we discuss reducing large deviations by a small number of bits as a correction code,
527
      the challenge of making this approach efficient for extremely quantized networks remains open.
528
529   Finally, we note that our proposed approach is not specifically limited to Transformer with SliceGPT.
530   The concept of bits-back coding applied to neural networks is general, and we can extend it to
531
      many architecture that exhibits symmetries. For instance, in Low-rank Adaptation (LoRA; Hu
      et al., 2022), the modulation is decomposed as W “ AB. In this setup, applying a rotation Q
532
      to A as AQ and to B as QJ B preserves W. This allows our approach to be seamlessly in-
533
      tegrated into such settings. Another special case of our method is dealing with the permutation
534   invariance. Any MLP with a single, d-dimensional hidden layer and activations ϕ, defined as
535   f px | A, Bq “ B ϕpAxq exhibits the following permutation invariance. For a d ˆ d permutation
536   matrix P, we have f px | A, Bq “ f px | PA, BP´1 q. However, it is a standard fact that permu-
537   tation matrices are orthogonal. Hence, our proposed method could be applied to a wide range of
538   network architectures to eliminate the redundancy introduced by the permutation symmetry of the
539   hidden units. Thus, an interesting future direction is to investigate if our method could improve the
      transmission and storage costs of smaller models, such as ones deployed on edge devices.

772   C     C OMPARED WITH U NIVERSAL S OURCE C ODING
773
774   As our proposed algorithm aims to reduce the storage and transmission cost, it is sensible to use a
775   universal source coding algorithm to reduce the storage cost. However, we note that our proposed
776   method is not directly comparable to these source coding algorithms. In fact, we can view our
777   proposed algorithm as a pre-processing step before universal source coding. Concretely, we suggest
778
      to combine our proposed method with a source coding algorithm to form the following ”bits-back”
      pipeline:
779
780         1. We obtain some network weights with rotational symmetries.
781
            2. We use our method to eliminate the redundancies induced by the rotational symmetries in
782
               the weights.
783
784         3. We apply a universal source coding algorithm to the output of our algorithm.
785
      In the main text, we only looked at the gains we get if we apply our method without any universal
786
      source coding. Hence, a concern arises: do we retain significant gains if we run the pipeline we
787
      suggest above, compared to just running universal source coding without our method?
788
789   Fortunately, the answer is positive. In particular, we compared the pipeline suggested above to
790   universal source coding (we used ZIP in our experiment) on the OPT-2.7B model (slicing 30%).
791
      Using ZIP only, the compressed size comes to 3.97 GB while using our suggested bits-back pipeline,
      it reduces to 3.79 GB, and approximately 5% gain in storage size as before. While the exact gain
792
      my vary depending on the universal source coding algorithm, these models are large enough that the
793
      gains we report here should be fairly robust across different source coding algorithms.
794
795   There is an intuitive reason for retaining the gain even after source coding: the general-purpose
796   universal source coding algorithm is unaware of the redundancies introduced by the rotational sym-
797
      metry in the weights and, therefore, cannot utilize it to reduce storage size. From this perspective,
      the storage savings resulting from our bits-back method are ”orthogonal” to the savings that result
798
      from source coding.
