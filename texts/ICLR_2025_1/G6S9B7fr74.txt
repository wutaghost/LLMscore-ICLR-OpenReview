009                                               A BSTRACT
010
011            Large Language Models (LLMs) present an opportunity to create automated as-
012            sistants that can help users navigate complex tasks. However, existing approaches
013            have limitations in handling conditional logic, integrating knowledge sources, and
014            consistently following instructions. Researchers and industry professionals of-
015            ten employ ad hoc pipelines to construct conversational agents. These pipelines
016            aim to maintain context, address failure cases, and minimize hallucinations, yet
017            frequently fail to achieve these objectives. To this end, we present Genie – a pro-
018
               grammable framework for creating task-oriented conversational agents that are
               designed to handle complex user interactions and knowledge queries. Unlike
019
               LLMs, Genie provides reliable grounded responses, with controllable agent poli-
020
               cies through its expressive specification, Genie Worksheet. In contrast to dialog
021            trees, it is resilient to diverse user queries, helpful with knowledge sources, and
022            offers ease of programming policies through its declarative paradigm. The agents
023            built using Genie outperforms the state-of-the-art method on complex logic do-
024            mains in STARV2 dataset by up to 20.5%. Through a real-user study involving
025            62 participants, we show that Genie beats the GPT-4 Turbo with function calling
026            baseline by 21.1%, 20.1%, and 61% on execution accuracy, dialogue act accuracy,
027            and goal completion rate, respectively, on three diverse real-world domains.

030   1   I NTRODUCTION
031
032   Researchers and industry practitioners have demonstrated significant interest in developing task-
033
      oriented dialogue agents. These agents are typically designed with a transactional focus, aiming
      to fill slot values based on user utterances to complete specific tasks (Budzianowski et al., 2018;
034
      Andreas et al., 2020; Rastogi et al., 2020). However, natural conversations between users and call
035
      agents cannot be adequately modeled solely as slot-filling tasks. Users often require external infor-
036   mation to progress through the steps necessary for task completion. For example, when attempting
037   to make a restaurant reservation, users frequently seek to identify a venue that best aligns with their
038   preferences. Additionally, users may interrupt the agent at any point during the conversation to pose
039   new inquiries. Traditional dialogue tree-based approaches often struggle to handle such unexpected
040   user questions or deviations, leading to a suboptimal user experience (Bocklisch et al., 2017; Xie
041   et al., 2022; Amazon, 2023; Press, 2024; Google, 2024).
042   The advent of Large Language Models (LLMs) offers a promising opportunity to create more natural
043   dialogue agents. These LLM-based agents can effectively navigate “unhappy paths,” where the
044   system adeptly manages unforeseen user inquiries at any stage of the conversation. Furthermore,
045   LLMs have enabled the development of knowledge assistants capable of querying structured data
046   by generating query languages such as SQL (Pourreza & Rafiei, 2023b) and SPARQL (Liu et al.,
047   2024c) and retrieving text for unstructured knowledge (Khattab et al., 2023). However, LLMs can
048   be unreliable, posing challenges for developers aiming to deploy them in real-world applications,
049   primarily due to a lack of control over conversations and inaccurate contextualization of information.
050   Challenge 1: Creation of effective, informative, and responsive informative agents, while let-
051   ting developers exercise control without onerous efforts. Developers desire to maintain control
052   over critical aspects of the agent’s operation, including the flow of conversations, the timing of ac-
053   tions, and the information elicited from users. For example, if a user declines an agent’s offer to book
      a restaurant, developers should be able to program the agent to suggest alternative options, such as

078   Challenge 2. Support users’ queries for information, which may be embedded in a task re-

080   invocations (Patil et al., 2023) and database queries (Pourreza & Rafiei, 2023a; Liu et al., 2024d). To

086   Challenge 3: Dialogue systems need to remember pertinent facts from the dialogue history.
087   Current methods based on LLMs typically rely on including in the prompt a complete dialogue his-

090   tended interactions; summarization may omit essential information, leading to repetitive questioning
091   or generating inaccurate responses, often referred to as “hallucinations” (Bang et al., 2023).
092   Our proposed method. This paper proposes a method that let developers easily create dependable
093   and effective task-oriented agents that can carry out accurate, fluid, and informative conversations
094   with users. We address the above challenges with a novel high-level, expressive and declarative
095   specification representation we have created called a Genie Worksheet. The core concept is to ask
096   the developer to directly declare variables to represent information that is pertinent to the conversa-
097   tion and to describe the actions to perform according to the input information. For answering user
098
      questions, the developer only have to supply the knowledge bases, which may be structured or un-
      structured. In other words, the developer is only declaring the necessary specification, and we show
099
      that the specification is sufficient to create effective knowledge and task-oriented agents.
100
101   From the Genie Worksheet, the Genie framework derives a conversational agent that fills in the
102   variables in the Worksheet as the conversation continues. The actively updated Worksheets double
103   up as a succinct representation of the dialogue state, designed carefully to address the complexity
104
      in semantic parsing and agent policies. As shown in Figure 1, an agent processes each turn of the
      conversation in four steps: a contextual neural semantic parser to interpret the input, an execution
105
      module performs the queries and actions, an algorithmic agent policy module decides on the agent
106
      actions, and a neural response generator generates the response governed by the agent policy.

135   2    D ESIGN AND R ATIONALE OF THE G ENIE W ORKSHEET
136
137   The objective in the design of the worksheet is to provide developers a high-level, concise, ex-
138   pressive specification of the task at hand. Our worksheet design is inspired by the versatility of
139   webforms. Modern websites contain multiple fields which can be optional, tabs for selection of
140   task, and pop-up windows that depend on previous user responses. To facilitate task completion and
141   access to knowledge bases, Genie Worksheet allows two kinds of worksheets: a task worksheet and
142   a knowledge base worksheet, described below.
143
144   Task Worksheet Users perform different tasks based on their requirements. For example, a stu-
145   dent who is having trouble enrolling in a class would need to fill out a form containing details
146   about the course they want to enroll in, the error message they are seeing, and any other additional
147
      comments and tasks like Leave of Absence are irrelevant to them. Hence, to make only relevant
      worksheets available to the user, all the worksheets have a WS Predicate field indicating when a task
148
      worksheet should be activated, as illustrated in Figure 2. The task worksheet also has a WS Name
149
      and a WS Kind set to Task. Additionally, to provide flexibility to the agent, Genie Worksheet allows
150   executing arbitrary Python code to call external APIs, change assigned values, or explicitly respond
151   to the user with a given string using built-in actions like say. The actions are triggered when all the
152   required parameters for a task are assigned and are defined under WS Actions.
153
      Each worksheet contains a set of fields. A field has a Predicate, similar to the WS Predicate, it
154
      indicates when a field can be filled in. Genie allows three kinds of fields. The user provides values for
155
      input fields; the output fields are set according to the output of executing an API or knowledge
156   base call; and the internal fields can only be manipulated by the agent. These are defined under
157   Kind. The Description field provides a natural language description of the field. Each field is tied
158   to a Type, where all conventional types are allowed. If it is an enum Type (e.g., student task in
159   Figure 2), a set of Enum Values are specified in the succeeding lines. The boolean DontAsk will
160   record the information if the user offers it, but the agent will not ask for it. If DontAsk is false, the
161   agent will ask for the field if it is not assigned, but user can refuse to answer if it is not Required.
      Whenever a field is assigned, similar to WS Actions, the supplied Actions should be executed either

      Knowledge Access Real-life user utterances can contain task statements and knowledge queries.
180   Hence, to handle such utterances, Genie Worksheet treats knowledge access as a first-class object.
181   Real-life queries often involve both structured and unstructured accesses. For instance, queries
182   “What’s the best-rated restaurant with a romantic atmosphere” require access to both the structured
183   “ratings” column and the free text “reviews” column. To handle hybrid knowledge bases, Genie
184   adopts the SUQL query language, an SQL extension that integrates search of unstructured data (Liu
185   et al., 2024d). For each knowledge base to be included, the developer must create a worksheet with
186   Kind set to KB. The fields in the KB Worksheet define the schema of the KB.

      3.1          F ORMAL R EPRESENTATION
191
192   Current LLM-based dialogue agents contextualize the conversation by providing the complete dia-
193   logue history or summarizing it. However, despite their impressive natural understanding capabil-
194   ities, they often struggle with maintaining relevant contextual information across longer conversa-
195   tions. Additionally, summarizing conversations can cause a loss of key information in the conversa-
196   tion. This leads to hallucinated or fabricated values and repeating questions. Therefore, to mitigate
197   these issues, Genie tracks the dialogue state with a formal dialogue state.
198   A dialogue consists of a set of alternating user and agent turns, {u1 , a1 , · · · , ut−1 , at−1 , ut }, where
199   ui ∈ U, ai ∈ A denoting user and agent utterances.
200
201   Dialogue State As an agent runs, it assigns values to the fields in task worksheets, executes ac-
202   tions, and performs queries. Figure 3 shows the conversation history in the course enrollment do-
203   main, where the student’s goal is to enroll in courses for the next semester. We represent dialogue at
204   each turn t, dt ∈ D, as a sequence of records, r ∈ R, each of which can be a completed or partially
205   specified task or a KB query. For example, r1 in Dialogue State (dt−1 ) corresponds to the knowl-
206   edge query ut−1 , where the student asks for courses where students are happy with the workload.
207
      Each record in the dialogue state has a kind of either task or KB query, an initiator, which can be
      an agent or user. As shown in r1 , a KB query record has four additional attributes: (1) names, the
208
      set of names of knowledge bases mentioned in the query (courses and ratings in r1 ), (2) NL query:
209
      a string of the form answer(s), where s is the de-contextualized (i.e. self-contained) query in
210   natural language, (3) KB query: the formal query associated with the NL query, (4) result: the result
211   of executing the formal query. A task record has two additional attributes: (1) name, the name of a
212   worksheet, (2) assignment, which maps one or more fields in the worksheet to its value. For exam-
213   ple, r2 in dt−1 corresponds to the task of choosing a course. Hence, the kind is Task, and the name
214   is the Course worksheet. The compositions of queries and APIs are supported by passing the result
215   of an instance of a record in as a field of another record. The assignment of course id in record
      r2 is r1 , which finds the courses where students are happy with the course load.

249   3.2      A RCHITECTURE OF THE G ENIE AGENT
250
      The Genie agent has three components:
251
252              • P(ut , at−1 , āt−1 , dt−1 , w) : U × A × P(Ā) × D × W → D: A parser P accepts the current
253                user utterance ut and contextualizes the utterance using the last agent utterance at−1 , the
254                previous-turn agent acts āt−1 , the formal dialogue state from the last turn dt−1 , worksheet
255                w, and generate a new dialogue state d.
256              • AP(d, d′ , w) : D × D × W → P(Ā) × D: an agent policy that accepts the dialogue state
257                from the previous turn d, the new dialogue state after parsing d′ , and the worksheet w and
258                produces a set of agent dialogue acts ā, along with a new dialogue state.
259              • RG(dt , āt , ut , at−1 ) : D × P(Ā) × U × A → A: a response generator that accepts the
260                formal dialogue state dt , an agent act set āt , the last user ut utterance and agent utterance
261                at−1 to generate the agent response at .
262

327   The agent policy (AP) interprets the updated dialogue state d′ ∈ D generated by the parser (P ),
328
      previous dialogue state d and the worksheet signature w ∈ W to inform its actions, resulting in an
      update of the dialogue state d′ ∈ D and a set of new agent acts ā ∈ P(Ā). The agent policy (AP)
329
      utilizes the Diff(d′ , d) function that identifies any fields and worksheets that have been modified
330
      or initiated between two dialogue states d′ and d.
331
332   Algorithm 1 outlines the agent policy. The procedure initiates by assessing whether a new KB record
333   exists within the updated dialogue state d′ compared to d (Line 4). If any required parameters p are
334
      missing in the record, the agent prompts the user to provide the missing values (Line 6). Otherwise,
      the system reports the result of the KB query (Line 9). Subsequently, the policy checks whether any
335
      new fields require confirmation and prompts users (Line 14). If no confirmation is needed, actions
336
      associated with the newly filled fields are executed (Line 16), and the dialogue state is updated
337   accordingly (Line 17). Any agent acts based on developer defined actions are recorded (Line 18). If
338   all fields in any worksheet are filled and their corresponding actions have been completed, the policy
339   executes the worksheet’s actions (Line 22) and updates the dialogue state (Line 23). Any agent acts
340   based on actions are recorded (Line 24). Finally, the policy identifies the first unfilled field in the
341   dialogue state and generates an agent act requesting the user to provide the missing value (Line 28).
342
      As in our example in Figure 3, the policy first executes the KB query in r4 and adds the result of the
343   KB query as R EPORT (r4 ) act to the agent acts list. It then checks all the required fields in record r2
344   and finds that the value of field “course units” is unfilled and hence adds A SK (r2 , course units).
345
346
      Response Generation Once we have the final dialogue state d′ and the agent’s dialogue acts ā
      produced by AP, we generate a response to the user using an LLM. The LLM takes in the previous
347
      user and agent utterance (u, a) as additional context. The LLM is instructed to adhere to the agent
348
      acts ā as opposed to making up its own actions. As shown in Figure 3, the agent response (at ) “...
349   HCI 347 can be up to 5 units. Would you like to take it?” corresponds to the R EPORT and A SK agent
350   acts ( āt ) generated by the agent policy. The prompt is shown in Appendix H.1.

352   3.3   D ISCUSSION
353
354   Genie automatically handles the dialgoue flow based on a succinct declarative Worksheet specifica-
355   tion. This is significantly different from dialogue trees which require the programmer to hand-code
356   the policy for each turn. Another popular approach is to model the conversation as a finite state
357   machine where user inputs are mapped to a small number of user dialogue acts and rules are used to
358   map the user dialogue acts, together with results of API or DB queries, to agent dialogue acts. As we
359
      discussed, each turn can invoke multiple different agent dialogue acts, resulting in an exponentially
      many agent dialogue state, had a finite machine been used. Purely function calling in LLMs will
360
      also fail to handle the many details in a conversation. In Appendix B, we show some conversational
361
      snippets that we can handle well but can cause other techniques to fail, along with the reasons.

513   6       R ELATED W ORK
514
515   Dialogue Tree-based frameworks. A wide variety of commercial and open-sourced products
516   (Amazon, 2023; Press, 2024; Xie et al., 2022; Google, 2024; Microsoft, 2022a;b; Reach, 2022;
517   Bocklisch et al., 2017; Watson, 2022) exist that help developers program a conversational virtual
518   assistant using dialogue trees, with slight differences in design. For instance, Press (2024); Google
519   (2024); Microsoft (2022a); Reach (2022); Watson (2022) allow users to interactively program a di-
520
      alogue tree with a GUI. Developers can manipulate building blocks that classify user intents, some-
      times with the help of built-in NLU modules. RASA allows developers to declare a list of global
521
      intents and program linear conversational paths accordingly (Bocklisch et al., 2017). Converse
522
      allows developers to program a special kind of “and-or” trees (Xie et al., 2022).
523
524   LLM and task-oriented dialogues. A series of recent works attempts to integrate LLMs with
525   TOD, showing its capability in dialogue state tracking (Hu et al., 2022; Feng et al., 2023; Hudeček
526
      & Dusek, 2023; Li et al., 2024b; Zhang et al., 2023a). In particular, Li et al. (2024b) uses GPT’s
      function calling capability for state tracking, which we compare against in Section 5. Deng et al.
527
      (2024) uses LLM for its dialogue policy planning module but does not support developer-defined
528
      actions through code. When building ToD agents with LLMs, existing works typically feed the
529   entire set of instructions to the model for response generation (Zhang et al., 2023a). While this
530   works well on smaller domains, LLMs struggle to follow all instructions in real-life situations (Liu
531   et al., 2024b). Genie Worksheet instead uses formal agent acts to instruct LLMs on its response.
532

534   7       C ONCLUSION
535
536   We introduce Genie, a novel framework for building knowledge-integrated task assistants with
537   LLMs, with Genie Worksheet, a high-level specification that allows for detailed control over conver-
538   sation flows. The agents built using Genie outperforms the state-of-the-art method on complex logic
539   domains in STARV2 dataset by up to 20.5%. Our real user studies with 62 participants demonstrate
      Genie drastically outperforms traditional LLM-based systems and dialogue trees.

      D.1    E VALUTION M ETRICS
760
761   Semantic Parsing Accuracy For each user turn, we manually inspect the code generated by the
762   semantic parser for correct APIs and Databases and filled fields. We define the gold Semantic
763   Parsing output (SP ) as the set of correct API calls (A), Database calls (D), and fields to fill (F ).
764   Then for each user utterance, SP = {s1 , s2 , ..., sm }, where si ∈ {A ∪ D ∪ F } and m is the total
765   number of choices, such that m = |A ∪ D ∪ F |. The Semantic Parsing Accuracy (SP Acc) for a
766   system is defined as the number of correct choices in the semantic parser’s output divided by m.
767   Execution Accuracy We manually inspect each turn to check whether the agent executes the correct
768   API and databases. For each agent turn, let the executions be E = {e1 , e2 , ..., em } where ei is an
769   API or database call. We evaluate whether each ei is a true positive or false positive. We calculate
770   the Execution (Ex Acc) as the number of true positives divided by the number of true and false
771   positives for all the execution calls in a conversation.
772   Agent Dialogue Act Accuracy For each turn, we manually inspect whether the agent’s dialogue act
773   follows the policies provided by the developer . Formally, for each turn, we define a list of gold acts
774   a1 , a2 , ..., am where ai ∈ A, all the possible agent dialogue acts. The Agent Dialogue Act Accuracy
775   (DA Acc) for a system is defined as the number of correctly predicted actions divided by m. For
776   GPT-4 (FC), we map its responses to the equivalent elements in the power set of agent acts.
777   Goal Completion Rate We define Goal Completion Rate (Goal CR) as the user’s ability to success-
778   fully complete the task with the system’s assistance. with the appropriate parameter values, then the
779   goal is considered completed. Goal Completion Rate is a binary metric for each conversation, where
780   1 denotes that the goal was completed, and 0 indicates that the goal was not achieved.
