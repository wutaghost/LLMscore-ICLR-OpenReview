009                                                           A BSTRACT
010
011                     Mixture of Experts (MoE) pretraining is more scalable than dense Transformer
012                     pretraining, because MoEs learn to route inputs to a sparse set of their feedfor-
013                     ward parameters. However, this means that MoEs only receive a sparse backward
014                     update, leading to problems such as router load imbalance where some experts
015                     receive more tokens than others. We present a lightweight approximation method
016
                        that gives the MoE a dense gradient while only sparsely activating its parameters.
                        A key insight into the design of our method is that at scale, many tokens not routed
017
                        to a given expert may nonetheless lie in the span of tokens that were routed to that
018
                        expert, allowing us to create an approximation for the expert output of that token
019                     from existing expert outputs. Our dense backpropagation outperforms standard
020                     TopK routing across multiple settings without significantly increasing runtime.

047   1           I NTRODUCTION
048
049   Large-scale pretraining hinges on scaling up the number of parameters in a model, because models
050   with more parameters are more sample-efficient and require less training to reach the same perfor-
051   mance as smaller models (Kaplan et al., 2020; Hoffmann et al., 2022). The most common model ar-
052   chitecture is a dense Transformer architecture (Vaswani et al., 2023) because its performance scales
053   well with parameters and data. However, a sparsely activated Mixture-of-Experts (MoE) Trans-
      former architecture (Shazeer et al., 2017) has been used by many industry deployments (Team et al.,


                                                                     1
054
      2024; xAI, 2024; Databricks, 2024; Jiang et al., 2024; Snowflake, 2024; DeepSeek-AI et al., 2024)
055   because MoEs have been shown to scale even better than dense Transformers (Clark et al., 2022; Du
056   et al., 2022; Lepikhin et al., 2020; Fedus et al., 2022). MoEs learn a routing function that selectively
057   activates the TopK subset of their modules, or experts, most relevant to a given input. This condi-
058   tionally sparse activation (Jacobs et al., 1991; Jordan & Jacobs, 1994) allows us to multiplicatively
059   increase the model parameter count without significantly increasing the cost of training or inference.
060
      The sparse router enables MoEs to scale, but it also presents a challenge, because the router does not
061   receive a gradient update from experts that it does not activate, and may not learn to route a token
062   to its appropriate expert. This may cause load imbalance— where a few experts are over-utilized —
063   leading to inefficient training and resource usage (Zoph et al., 2022; Zhou et al., 2022).
064
      In this work we propose a new router that can receive a dense gradient update from a sparse for-
065
      ward pass to address the instability issues arising from sparse routing. Our method adds minimal
066
      overhead, but improves on the common Top-K routing in both performance and load balance.

071   MoEs. The MoE layer replaces the feedforward networks (FFN) of transformers and consists of
072
      two components : 1) N FFNs (experts), E0 (x), E1 (x), . . . EN (x) and 2) a router that assigns tokens
      to experts. Each input to the MoE layer is processed by K experts where K < N , and this is the
073
      source of sparsity in MoEs. The K experts are chosen by the router, which is a learnable component
074
      that maps each token to a set of weights over the experts. The router performs a linear transformation
075   Rdtoken → RN which produces logits; these are normalized using softmax, resulting in a probability
076   distribution over the experts. With the router’s linear transformation parameterized by a matrix W ,
077   we can represent the expert weights π in the following way:
078
079
                                            π ∈ RN = Softmax(W x)                                         (1)
080
081   Once we have these expert weights, we apply a routing function to decide which of K experts to
082   route and process this token through. We consider TopK because it is the most popular.
083
      Top-K routing. A standard method to select K out of N experts given the expert weights is to
084
      select the experts corresponding to the K highest weights. Top-K routing (Fedus et al., 2022) passes
085   the token to the K selected experts and averages the expert outputs using these weights to produce
086   the final output. Experts not selected by the Top-K routing function do not process the token, and
087   this introduces sparsity in MoEs. By representing the K chosen experts as the set A, we can express
088   the output of the MoE layer as an average of expert outputs weighted by the router scores:
089                                             y = Σi∈A πi Ei (x)                                        (2)
090
091   The expert weights serve two roles. They are used by the routing function to decide which of the
092   K experts to process a token through, and also provide the weights for combining the outputs of
093   the expert. The Top-K routing scheme makes the MoE layer desirable for training large, compute-
094   efficient neural networks. It allows models to be scaled up, by way of increasing the total number of
095   experts, while keeping the compute per token constant (as it is a function of K and not N ).

097   parameters W . We express y as a function of W by combining Eq. (1) and Eq. (2). With the chain
098   rule, we can backpropagate through this function by considering the gradient at each respective step:

      3   D ESIGNING D ENSE BACKPROPAGATION
150
151
      In this section we design a new backpropagation method that can send a dense gradient to the
152
      sparsely activated router and expert parameters without doing additional forward passes. In a stan-
153
      dard MoE, both the embedding corresponding to expert i in the routing layer (i.e. the ith row of
154   the routing weight matrix) and the weights of expert i receive no gradient update from a token x
155   if x is not routed to expert i. This is because Ei (x) is never computed, so it provides no upstream
156   gradient. This corresponds to experts that are not in the top K being omitted in Eq. (2). We ap-
157   ply an approximation Êi (x) as a substitute for the upstream gradient so that the router and expert
158   weights can receive some non-zero signal corresponding to this expert. Then, the router can factor
159   in outputs from all experts when learning to route each token, and the expert can be updated based
160   on all tokens. Ultimately, instead of K/N expert parameters being updated per token, every token
161   will update more parameters in both the router and the experts. Updating the router should improve
      load balance and the routing distribution.

182   3.1   A PPROXIMATING E XPERT ACTIVATIONS W ITH D ENSE BACKPROPAGATION
183
184   To approximate the dense gradient in Eq. (4), we must approximate Ei (x) for every expert i that a
185   token x was not passed to. Although we have no information about what the function Ei looks like
186   for x, when training with large token batch sizes it is very likely that we have outputs of Ei for many
187   other tokens. We hypothesize that although Ei (x) is not computed for a given x, the output lies in
                                                                                 P
188   the span of other tokens’ activations for Ei . In other words, Êi (x) = j cj Ei (xj ) for some other
189   tokens xj . Our approximation method relies on finding these relevant xj and the corresponding
190   weights cj to develop an approximation for a token. We visualize this general approach in Fig. 3.
191
      Dense Backpropagation via Expert Group Approximation. Our dense backpropagation method
192
      that we dub expert group approximation produces an estimator Êi (x) from the expert outputs of
193
      other relevant tokens. We apply a single approximation to a large group of tokens that were not
194   routed to expert i. This is efficient, but it may not be a good approximation for any specific x.
195   Instead, this is an estimator for the expert output across the entire batch - this is sufficient as we
196   will only need an approximation for the batch gradient to update the router during training. In dense
197   backprop, we select relevant xj and cj purely based on the router output.
198
      This expert approximation is used solely to estimate the dense gradient in equation 4. We do not use
199
      this dense approximation in the forward pass, as that is inconsistent with the sparse forward pass we
200   expect at inference. Additionally, applying this approximation at inference is infeasible because of
201   the much smaller batch size. However, we do use this approximation to update the experts along
202   with the router. As we will show, updating all the experts is a crucial component of our method, and
203   something prior work has not done. In particular, Liu et al. (2023) focuses on the router gradient.

      3.3   E XPERT G ROUP A PPROXIMATION
240
241   Our method is based on approximating the expert output Ei (x) for a group of multiple tokens at
242   once. For a token x, we want to approximate outputs of experts that x was not routed to, i.e. Ei (x)
243   where i ∈/ R(x). We hypothesize that tokens being routed to the same expert is a strong indicator of
244   similarity between the tokens. This is because tokens routed to the same expert Ei both yield high
245   dot products with the embedding Wi corresponding to that expert in the routing layer (see Eq. (1)).
246   We develop an approximation for Ei (x) by aggregating outputs of Ei for tokens that were routed
247   to both expert i, and other experts that x was routed to. Formally, we consider an alternate routing

249
      wish to approximate, and any other experts (if K > 2). Then, the adjacent token space XR′ will
      consist of tokens that are very similar to x by virtue of having similar routing P

359   4.2    E MPIRICAL A NALYSIS
360
361   Gradient Approximation. We verify that our method is indeed faithfully approximating the dense
362   router gradient i.e. the gradient to the router if all experts were activated. We track the dense
363
      gradient by routing to all experts and backpropagating only on the MoE output (independent of the
      full forward pass). In Fig. 6a we compare this dense gradient to the actual router gradient for each
364
      of our approaches and the standard Top-K router, and similarly for the expert weights in Fig. 6b.
365
      We plot the cosine similarity between the dense gradient and our approximated gradient, where the
366   relatively high similarities our methods achieve indicate we are more accurately approximating the
367   dense gradient. We also observe a major difference between the gradient of the standard Top-K
368   router and our approach in later layers. We record the gradient similarity from the start of training
369   until the trends stabilize; if we were to extend these plots out, we would see the similarities stay
370   relatively stable. This validates our motivation to try and approximate the true dense gradient.
371
      Load Balance. One reason for our method’s superior performance is in how it improves the routing
372   distribution. Without a gradient signal for unactivated experts, top-K routing may not learn a bal-
373   anced distribution across experts, and route more tokens to some experts than others. On the other
374   hand, our method sends a signal to the router even for experts that a specific token was not routed
375   to. The router now receives more information to make better decisions on how to route inputs. Load
376   imbalancing leads to some experts processing a larger fraction of tokens than others, and thus having
377   to learn a representation for more tokens. although all experts share the same number of parameters.
      Balancing expert load thus allows for more efficient utilization of MoE parameters.

450   Variations on the method. Thus far we have presented results where the approximation we use for
451   a group of tokens is just a simple average of the expert outputs for similar tokens. However, when
452   we construct the approximation we can weight tokens based on their router scores. In Table 3 we
453   compare two variations on our main method. As a reminder, in this method we construct a mask
454   of shape experts, experts for each token. The row is the expert that token was routed to, and
455   the column is the expert we want an approximation for. When we take the product of this mask
456   and the router scores, we can weight each row by the probability corresponding to the expert we
457
      want an approximation for, or weight each approximation by the probability for the expert we’re
      using the approximation for. The former should give us more “accurate” approximations, because
458
      it will prioritize tokens that are more likely to be routed to the expert we want an approximation
459
      for. The latter should give us more “viable” approximations, because it weights by closeness to
460   the space we’re using the approximation for. We find that both variants improve over our original
461   method, with the “accurate” variation proving to be the best across K = 1 and K = 2. Stacking the
462   “accurate” variant on top of our method would further improve our main results.

      Scaling with Sparsity. We present most results in this paper with K = 2 because Zoph et al.
464
      (2022) finds that it improves performance over K = 1 with minimal overhead, and thus multiple

532   5   D ISCUSSION
533
534   We propose a training method for MoEs to improve load balancing and language modeling perfor-
535   mance. By approximating the signal of a dense mixture-of-experts layer, the MoE router is able
536   to learn a better distribution of routing inputs to different experts. This approximated dense signal
537   unlocks the possibility for more sparse MoEs at training and inference time. Whereas typical Top-K
538   routing would provide too sparse of a signal to learn a stable routing distribution, our method demon-
539   strates significant improvements in load balancing in very sparse configurations. This unlocks better
      performance on standard language benchmarks, and has little overhead.

         Our Expert Group Approximation computes an approximation, for each expert, for all tokens
1219   routed to it from each other expert, resulting in N 2 total approximations. However, we may want
1220   to actually compute an approximation for specific tokens. Consider tokens belonging to the set
1221          C
       x ∈ X{i,·} , i.e. tokens not routed to expert i. We want to approximate Ei (x) for such x. At a high
1222   level, we want to search for similar tokens to x, select their expert outputs Ei (xj ), and aggregate
1223   these outputs as a weighted linear combination. This problem can be solved using attention. We
1224                                                                C
       want to query using all tokens not routed to expert i, i.e. X{i,·} . The keys will correspond to tokens
1225   that were routed to expert i, i.e. X{i,·} . And the values will be the expert outputs of these relevant
1226   tokens. Fig. 14a (left) outlines how we compute an approximation using multi-head attention, where
1227   each head corresponds to approximating for a single expert.
1228
       Sparse Attention with LSH. Computing attention across all tokens on a GPU is computationally
1229
       expensive, and we do not need attention scores for all the tokens to compute the approximation, just
1230   for the most similar tokens to x. With a block-sparse attention mask, we can greatly reduce the at-
1231   tention computation, especially when most of the computed scores would be redundant. In Fig. 14b
1232   we outline our attention approximation that uses locality-sensitive hashing (LSH) to group tokens
1233   into buckets, with a high probability that the nearest neighbors to a token will lie in the same bucket.
1234   The attention mask now has an additional condition: the query index q and key index k must corre-
1235   spond to tokens in the same bucket. We sort the QKV into groups based on their assigned buckets to
1236   encourage a block-diagonal attention mask, and verify that this sparsity reduces the runtime of our
1237   attention approximation. Note that it is possible that some tokens receive no approximation because
1238
       there are no keys to query in the bucket. In this case, we set the approximation to 0.
1239   Despite the reduction in computation from using a sparse attention mask, implementing this method
1240   is still computationally infeasible. We cannot use kernel based implementations like FlashAtten-
1241   tion because our attention inputs are the high-dimensional model embeddings instead of the low-
       dimensional embeddings that FlashAttention expects. We believe future works expanding efficient
