                                                   A BSTRACT
011
               Graphical User Interface (GUI) Agents show amazing abilities in assisting human-
012
               computer interaction, automating human user’s navigation on digital devices. An
013
               ideal GUI Agent is expected to achieve high accuracy, low latency, and generality
014            across various GUI platforms. Recent visual-based approaches show promises,
015            taking the advantages of advanced Vision Language Models (VLMs). Although
016            they generally meet the requirements of generality and low latency, these visual-
017            based GUI Agents often fall short in terms of localization accuracy. To address
018            this issue, we propose SpiritSight, a visual-based generalist end-to-end GUI agent
019            with outstanding grounding abilities. First, we create a multi-level, large-scale,
020            high-quality GUI training dataset with scalable methods and train SpiritSight us-
021            ing curriculum learning, empowering it with robust GUI understanding and local-
022
               ization capabilities. Second, we introduce the Universal Block Parsing (UBP)
               method, which frames the localization task as a multi-image QA problem, further
023
               enhancing SpiritSight’s ability to ground GUI objects. With the above-mentioned
024
               efforts, SpiritSight constantly outperforms previous SOTA methods across nu-
025            merous major automatic GUI navigation benchmarks. Notably, SpiritSight-8B
026            achieves a 46.1% Step Success Rate(Step SR) on the Mind2Web benchmark
027            without any candidates element input, more than doubling the performance of
028            SeeClick (20.9%) with a comparable model scale. SpiritSight also outperforms
029            other visual-language-based methods in various GUI platforms, demonstrating its
030            superior capability and compatibility in GUI Agent tasks. The models and the
031            code will be made available upon publications.

069   1   I NTRODUCTION
070
071
      Graphical User Interface (GUI) automation has long been pursued by people along with the develop-
072   ment of the modern digital devices. Thanks to recent advances of Large Languge Models (LLMs),
073   GUI Agents are constructed to assist users in interacting with graphical interfaces, automatically
074   making action decisions based on observations of environmental elements and user’s objective.
075
      Current approaches can be divided into three categories based on their inputs. Language-based and
076
      visual-based approaches make use of Hyper Text Markup Language (HTML) / Extensible Markup
077
      Language (XML) and screenshots as input (Zheng et al., 2023; Huq et al., 2023; Deng et al., 2024;
078   Wan et al., 2024; Lai et al., 2024; Lee et al., 2024; Yin et al., 2024; Hong et al., 2024; Cheng
079   et al., 2024; Chen et al., 2024b), respectively. Visual-language-based methods integrate multi-modal
080   information by enhancing HTML with screenshots (Furuta et al., 2023; Thil et al., 2024; Kil et al.,
081   2024; Zheng et al., 2024).
082
      The language-based and visual-language-based methods typically applied only in the web domain,
083
      and often limited by the excessive length of HTML or security concerns regarding it. The visual-
084   based approaches demonstrates enhanced compatibility across various GUI platforms, as acquir-
085   ing screenshots is generally easier than obtaining hierarchical data from platforms except for the
086   web. However, visual-based approaches struggle to localize the elements objects (i.e. buttons, text
087   boxes) from the input visual context. Some works solve this problem by adopting Dynamic High-
088   Resolution (Kim et al., 2022; Chen et al., 2024c) approach, which may bring ambiguity to the
089   process of model learning. Others attempt to collect large scale training data through manual syn-
090   thesis (Shi et al., 2017; Liu et al., 2018; Lee et al., 2023), human annotation (Yao et al., 2022; Deng
091   et al., 2024; Rawles et al., 2024; Chen et al., 2024a; Chai et al., 2024; Lu et al., 2024; Lù et al.,
092
      2024) and the use of common datasets (Deka et al., 2017; Li et al., 2020; Wang et al., 2021; Cheng
      et al., 2024; Zhang et al., 2024a), while these data are respectively unrealistic, expensive and of low
093
      quality.
094
095   To address the aforementioned challenges, in this paper, we proposed a single-stage, visual-based
096   GUI Agent——SpiritSight, which has strong ability in GUI navigation task. Our contributions are
097   summarized as follows.
098   Firstly, we propose a cost-effective GUI dataset of 5.46 million samples to enhance our model’s
099   GUI understanding and localization capabilities. The datasets is collected from real-world and
100   filtered through carefully designed rules to ensure data quality. They are also constructed with a
101   clear hierarchy and consist of 3 different level of components: text/icon recognition and grounding
102   tasks, functional grounding task, and GUI navigation task. The first two parts of datasets, which
103   constitute 90% of the total and have been collected for free, are primarily used to equip our model
104
      with robust elements grounding capabilities, thereby improve its GUI navigation ability.
105   Secondly, We introduce a Universal Block Parsing (UBP) method to resolve the ambiguity
106   in Dynamic High-Resolution input. This method treats the localization task as a multi-image
107   QA problem (Raj et al., 2021), where each element object are grounded within the corresponding
      sub-image. It also introduce a 2-dimensional block-wise position embedding method (Kim et al.,


                                                        2
      Under review as a conference paper at ICLR 2025


108
      2022) to help the model learn the spacial information of cropped input image, thereby enhances the
109   grounding capabilities of SpiritSight.
110
111   Thirdly, we evaluate our SpiritSight model family in various GUI benchmark and it exhibits
112
      impressive performance among them. We release two versions of GUI Agent with different model
      size: the large-scale SpiritSight-26B, standard SpiritSight-8B and the lightweight SpiritSight-2B.
113
      SpiritSight-2B achieve a 96% hit rate on text/icon grounding task, demonstrating near-perfect perfor-
114
      mance in pure grounding tasks. On the ScreenSpot (Cheng et al., 2024) benchmark, SpiritSight-8B
115   achieve a 66.5% accuracy and surpasses SeeClick (8B) (Cheng et al., 2024) by 13.1%, and by 19.1%
116   over CogAgent (18B) (Hong et al., 2024). Under the non-candidate input setting, SpiritSight-8B
117   and SpiritSight-26B attains an average Step Success Rate of 46.1% and 50.2% on the Mind2Web
118   (Deng et al., 2024) benchmark, outperform all works including language-based, visual-based and
119   even visual-language-based methods.

      2     R ELATED W ORK
123
124   2.1   L ANGUAGE - BASED AND VISUAL - LANGUAGE - BASED GUI AGENT
125
      Several works leverage the capabilities of Large-scale Language models (LLMs) to construct GUI
126
      agents. It is noticed that they are mostly multi-stage architectures. Mind2Web (Deng et al., 2024)
127   employs a lightweight language model to extract candidate elements from HTML, followed by a
128   ranking model that sorts the elements based on task descriptions and historical actions. Finally,
129   a large language model predicts actions and the elements on which they are applied. WebAgent
130   (Gur et al., 2023) first uses an encoder-decoder model to generate low-level instructions and relevant
131   HTML code snippets, then uses another decoder to produce executable Python code. AutoWebGLM
132   (Lai et al., 2024) simplifies HTML code through manually designed rules before predicting the
133   action codes.
134   Other visual-language-based works leverage both GUI screenshots and hierarchical HTML/XML to
135   enhance the robustness of GUI agents. WebGUM (Furuta et al., 2023), CC-Net (Thil et al., 2024)
136   use ResNet and ViT to extract features from screenshots respectively. The image embedding are
137   then combined with text embedding and fed into a multi-modal transformer. SeeAct (Zheng et al.,
138   2024), AppAgent (Yang et al., 2023) identify all interactive elements using HTML files or XML
139   files. It then assign each interactive element a unique identifier in the screenshot and then feed the
140   screenshot into the model.
141   These language-based methods or visual-language-based methods that rely on the hierarchical in-
142   formation exhibit several limitations: (1) Acquiring hierarchical representations like HTML/XML
143   are not equally available on different platforms. And even this information is available, their internal
144   rule difference makes language-based GUI Agents less compatible; (2) HTML often contains redun-
145   dant and customized information, requiring additional models or extensive manually crafted rules
146   for effective filtering.(3) Text-based GUI Agents are vulnerable to injection attacks (Zhan et al.,
147
      2024; Wu et al., 2024; Liao et al., 2024), where malicious instructions hidden in HTML can easily
      lead to erroneous or unsafe actions.
148
149
150   2.2   V ISUAL - BASED GUI AGENT
151
152   Recently, some visual-based approaches have been proposed to overcome the drawbacks of
153
      language-based methods. Some of them (Shaw et al., 2023; Hong et al., 2024; Cheng et al., 2024;
      Baechler et al., 2024) are single-stage methods that only use GUI screenshots as input for MLLMs
154
      and output the next action in an end-to-end manner. However, these agents perform worse on rel-
155
      evant GUI benchmarks compared to other approaches. MobileAgent and MobileAgent-v2 (Wang
156   et al., 2024b;a) are two-stage methods, using the GPT-4V API instead of publicly available MLLMs.
157   They find that top models like GPT-4V are not adept at element grounding tasks, thus introduce ad-
158   ditional tools such as OCR and icon recognition models to assist with localization. However, this
159   may increase the complexity and inference latency of the agent system. Overall, the current MLLMs
160   demonstrate poor localization capabilities for GUI grounding task, limiting the navigation capabil-
161   ities of single-stage visual-based GUI agents. In Appendix A, We discuss additional related works
      on large-scale language Models (LLMs) and Multi-modal Large Language Models (MLLMs).

      3     DATA C OLLECTION
186
187
      In this chapter, We introduce a data collection strategy specifically designed to address the defi-
      ciency in a visual-base GUI agent. We highlight the deficiency by modeling the GUI navigation task
188
      using a sequential decision-making process and further breaking it down through hierarchical policy
189
      decomposition. See Appendix B for details.
190
191
      3.1     L EVEL O NE : V ISUAL -T EXT A LIGNMENT
192
193   Visual text alignment refers to the model’s ability to recognize or locate the text content of a text el-
194   ement or the icon caption of a icon element, which requires the source data of the GUI platform. On
195   the web scenario, We collected website URLs from two sources: the common crawl (Group, 2024)
196   datasets and URLs from website ranking. We then developed a data collection tool using play-
197   wright (Microsoft, 2024) library to get real-world web data from the collected URLs. With this tool,
198   we collected 740k web-page screenshots along with their DOM annotation, with the diversity both
199   in resolution and languages. We also noticed that the icons on the web-pages often lack captions.
200   The existing icon detection (Bai et al., 2021; He et al., 2021) tools are not fully adaptable to the web
201
      scenario, due to their extensive use of custom-designed icons. So, we developed a InternVL-Icon as
      the icon annotation tool by collecting a dataset of 30K icon-caption pairs from Alibaba (2024) and
202
      fine-tuning InternVL1.5-26B using this dataset. After that, we annotate all the icons on web-pages
203
      by the captions generated from InternVL-Icon. As for the mobile scenario, we collect data from
204   AitW (Rawles et al., 2024), which contains a large-scale GUI data in mobile devices.
205
206   Based on our collected source data, we construct three tasks: text2bbox, bbox2text, and bbox2dom.
207
      Text2bbox task prompts the model to ground the element based on the given text or icon caption. We
      additionally include context information for the elements that appears multiple times in a screenshot
208
      to avoid ambiguity. The text2bbox data is the most abundant among the three tasks, in order to help
209
      the model learn grounding capabilities. Bbox2text task is the inverse version of the text2bbox task,
210   teaching the model about Optical Character Recognition (OCR) and icon captioning. Bbox2dom
211   task requires the model to generate the DOM-tree based on the given bounding box area, as show
212   in Figure 7. The bbox2dom is constructed to help model learns about the GUI layout knowledge
213   besides the basic OCR and icon recognition. To make sufficient use of the context length of the
214   model, We pack dozens of data pairs in one training sample for text2bbox and bbox2text task, and
215   select the box that include as many elements as possible for bbox2dom task. Overall, we totally
      construct 1.9M and 1.1M training samples on web and mobile platforms. See Appendix F for more

217   GUI grounding ability, of our SpiritSight model.

      3.2   L EVEL TWO : V ISUAL -F UNCTION A LIGNMENT
220
221   Visual-Function Alignment refers to the model’s ability to recognize or locate the function of a ele-
222   ment, where the element function data cannot be directly obtained from the real-world environment
223   like in the first level. Inspired by the back-translation (Sennrich, 2015) method for data construc-
224   tion, who collect the dataset for the forward translation task using back-translation, we leverage
225   InternVL’s capabilities in image understanding to collect element function data.
226   We conducted custom tests on InternVL2-26B to evaluate its ability to recognize element functions
227   before collecting data. We divided the screenshot into a 3x3 grid to represent the approximate
228   location of the element (i.e. in the top-left corner of the image) and placed a bounding box around the
229   target element in the screenshot to assist with specifying the element. By providing the model with
230   the screenshot, the element’s text content or icon caption, the region where the element is located,
231   we prompted the model to generate the corresponding function of the element. Additionally, We
232   utilize InternVL2.5-20B to enhance the quality and diversity of the generated function descriptions.
233   InternVL2-26B achieving an 80% accept rate with human judgement, which we consider acceptable
234
      for constructing the functional grounding data.
235   Based on the methodology above, we collect element-to-function pairs for all the operable elements
236   collected in level 3.1 and then reverse it to function-to-element pairs. Combined with the position
237   annotations of the elements, we ultimately obtain function2bbox pairs. We use the same packing
238   method as in the text2bbox and bbox2text data for efficient model training and ultimately obtained
239   0.9M training samples. Besides, we also collect the functional grounding data for the mobile sce-
240
      nario, which is derived from the construction of the GUI navigation data, as described in the level 3
      section.
241
242
243   3.3   L EVEL THREE : V ISUAL GUI NAVIGATION
244
      We utilize the public available AitW (Rawles et al., 2024) dataset to construct our GUI navigation
245
      training data. As mentioned in Zhang et al. (2024a); Chai et al. (2024), AitW data involves a
246   certain amount of incorrectly labeled samples, so we decide to clean it with GPT-4o. We adopt the
247   Chain-of-Thought (CoT) (Wei et al., 2022b) to make the judgment more accurate. Specifically,
248   We prompt GPT-4o with the task objective, the screenshot at the current step and the next step,
249   the previous actions, and the labeled current action. GPT-4o is required to first summarize the two
250   screenshots and tell the difference between them, then describe the current actual step description
251   according to the difference, and lastly assess the reasonability of current action. We filter out data
252   samples deemed unreasonable and ultimately got 0.63M GUI navigation samples.
253   With the collected CoT-style data, we are able to collect functional grounding data for mobile sce-
254   nario that is mentioned in section 3.2, as each step includes a description. The collected data also
255   allow us to train the model in a CoT manner to make it stable for model to learn and easy to converge.
256
257   3.4   OTHER T RAINING DATA

260   a supplement, including doc/web/mobile VQA datasets (Mathew et al., 2021; Chen et al., 2024b;
261   2021; Hsiao et al., 2022), image captioning datasets (Deka et al., 2017; Wang et al., 2021), and
262   mobile grounding datasets (Li et al., 2020; Deka et al., 2017). Finally, we construct 0.49M QA
263
      pairs from the datasets above.

341   5.1   I MPLEMENTATION D ETAILS
342
343
      We use InternVL(2B, 8B and 26B) (Kim et al., 2022; Chen et al., 2024c) as pre-trained models.
      The history actions are limited within 5 actions to avoid excessive overload. The training process
344
      is divided into two phases: continual pre-training and fine-tuning. During the pre-training phase,
345
      we train all the collected datasets mentioned in the section 3 simultaneously. Different prompts
346   are designed for different training tasks to avoid task confusion. We unfreeze the visual encoder,
347   decoder, and MLP layer of InternVL. The learning rate is set to 1e-4, 1e-4, 5e-5 for 2B, 8B, 26B,
348   respectively, and the batch size is 1024.
349
      After pre-training, we fine-tuning our model in several downstream datasets separately. for the
350
      ScreenSpot benchmark, we follow the data proportions from Cheng et al. (2024), using part of the
351
      first-level and second-level data, as well as data from Li et al. (2020); Deka et al. (2017); Wang
352   et al. (2021) to train the entire model. For other GUI navigation benchmarks, we first train the entire
353   model for 1 epoch using third-level data and the training data corresponding to each benchmark,
354   then fine-tune the model for 1 epoch on the benchmark-specific training data using LoRA (Hu
355   et al., 2021). While training the entire model, the learning rate is set to the same as pre-training, and
356   the batch size is 1024. During fine-tuning, the learning rate is set to 5e-5, the batch size is 64, with
357   the alpha of visual encoder and decoder set to 32 and 64, respectively.

471
472   Exploring the cross-lingual capabilities of GUI agents is highly beneficial for their application in
473   non-English environments. We split the training and test sets of GUIAct(web-multi) dataset into
474   English and Chinese parts, respectively. We then fine-tune SpiritSight-8B on two sets of data: the
475   entire training set (English + Chinese) and the English-only training set. The results are shown in
476
      Table 3.
477   Under the English + Chinese configuration, SpiritSight achieved very similar results on both the
478   English and Chinese test sets. Notably, SpiritSight, fine-tuned only on the English training set,
479   achieved an Step SR of 24.5% on the Chinese test set, reaching half of the English + Chinese
480   performance. The zero-shot capability of SpiritSight in Chinese comes from the small but effective
481   foundational Chinese data included in the pre-training phase.
482   This experiment provides a paradigm for applying GUI agents to non-English environments: by
483   collecting (1) free web and mobile GUI information from the target language environment (level 1
484   & level 2 data), and (2) a small amount of high-quality GUI navigation data at minimal cost (level
485   3 data). With this, the same capabilities as in the English environment can be achieved through
      training.

      7    L IMITATION AND F UTURE W ORK
487
488
      Safety and compliance issues. As the SpiritSight Agent is a visual-based GUI agent, it constantly
489   requires access to screenshots which may contain personal information or other sensitive data. Users
490   and system providers should manage the system privileges granted to the SpiritSight Agent carefully
491   to mitigate potential privacy and security risks.
492
      Increased computational demands with higher input resolutions. The computational require-
493
      ments of the SpiritSight Agent increase with the resolution of the input images. The inference
494
      latency for each step will get longer as the input image size get larger. Yet, for the real-world usage
495   of GUI Agents, the fluency of operation is pivotal to user’s experience. Future work could explore
496   more efficient model architectures or compression techniques to address these computational chal-
497   lenges.

501
      In this paper, we propose an advanced visual-based end-to-end GUI agent——SpiritSight, with
502
      high generalization across multiple GUI platforms. We construct a efficient mutli-level, large-scale,
503
      high-quality GUI pre-training data to equip SpiritSight with robust GUI perception, grounding and
504   understanding capabilities. We further introduce UBP method to resolve the ambiguity in Dynamic
505   High-Resolution input during model training, further enhancing the ability of SpiritSight to ground
506   GUI objects. Ultimately, SpiritSight shows strong performance in numerous GUI navigation bench-
507   mark across various GUI platforms, demonstrating great potential for practical deployment in real-
508   world applications.

      A.1   L ARGE - SCALE L ANGUAGE M ODELS
813
814   In recent years, large language models (LLMs) (Radford et al., 2018; Devlin, 2018; Raffel et al.,
815   2020; Xu et al., 2021; Wu et al., 2023; Nijkamp et al., 2022; Roziere et al., 2023; Yu et al., 2023;
816   Wang et al., 2023a; Ying et al., 2024; Shao et al., 2024; Wei et al., 2022b;a; Pan et al., 2023)
817   have demonstrated remarkable capabilities in the field of Natural Language Processing (NLP), en-
818   compassing natural language generation, commonsense knowledge question-answering, code com-
819   pletion, mathematical computation, and logical reasoning. LLM have also demonstrated strong
820   decision-making capabilities, laying the foundation for the emergence of GUI agents.
821
822   A.2   M ULTI - MODAL L ARGE L ANGUAGE M ODELS
823
824   With the development of large language models, numerous works (Bai et al., 2023; Wang et al.,
825
      2023b; Lin et al., 2023; Li et al., 2024; Chen et al., 2024c; Zhang et al., 2024b; Yao et al., 2024; Jin
      et al., 2023) have proposed Multi-modal Large Language Models (MLLMs) to bring the capabilities
826
      of language models into the visual domain. CLIP (Hafner et al., 2021) uses contrastive learning to
827
      align visual and language features, while BLIP (Li et al., 2022) and BLIP-2 (Li et al., 2023)build on
828   this by adding a language decoder, enabling the models to perform image-grounded text generation.
829   InternVL (Chen et al., 2024d) attempts to scale the parameters of visual encoder up to 6 billion,
830   significantly enhancing the model’s ability to perceive visual input. LLaVA (Liu et al., 2023) and
831   Sphinx (Lin et al., 2023) improve the models’ understanding and chat abilities through instruction
832   tuning and multitask learning, respectively. Beyond general domains, OCR-Free (Kim et al., 2022)
833   methods use an encoder-decoder architecture to achieve end-to-end visual document understanding.
834   This demonstrates the significant potential of MLLM in GUI navigation task.

      B     TASK F ORMULATION
837
838
      For a given GUI platform, we first obtain an action space A which contains all possible action that
839
      an agent can take. Given the task description T , the previous actions H = {a1 , a2 , ..., at−1 }, the
840   action space A and the current screenshot ot , the agent is expected to infer the optimal action a∗t
841   that maximizes the expected future reward. The inference process is guided by a policy π, as shown
842   below, which maps the current context to a probability distribution over the action space A.
843
844                                            a∗t ∼ π(a|T , H, A, ot )                                   (3)
845
      We propose a hierarchical decomposition of the policy to handle the complexity of action inference.
846   Initially, the overall policy π is decomposed into step inference policy πs (s|T , H, ot ) and action
847   inference policy πa (a|s, A). The step inference policy πs selects the step s based on the current
848   context, where the step is defined as the natural language description of the action. Once the step
849   s is determined, the action inference policy πa selects a specific action a from the action space A
850   conditioned on s.
851
      Further, we decompose πa into two additional sub-policies: πpos (apos |s, A) and πattr (aattr |s, A).
852   Here, apos corresponds to the positional information of the action, typically the coordinates where
853   the action is performed, while aattr represents the non-positional information of the action, such as
854   the action type (click, input) or additional parameters like input text. The decomposition is formally
855   expressed as:
856
857
                    π(a|T , H, A, ot ) = πs (s|T , H, ot ) · πpos (ap os|s, A) · πattr (aattr |s, A)      (4)
858   It is easy for visual-based agent to learn on the step inference policy πs , as modern VLLMs performs
859   well on reasoning and decision-making. The non-positional inference policy πattr is also easy since
860   the non-positional part of a action can be directly paraphrased according to the step. For example,
861   INPUT(”Copenhagen”) can be directly infer through the step Input “Copenhagen” into the arrival
862   input box. The primary challenge lies in learning the positional sub-policy πpos as mentioned in
863   Chapter 2. Based on this, we construct a large scale dataset with a primary focus on grounding tasks
      to address the challenge of learning accurate positional actions.

887   C     OVERALL A RCHITECTURE
888
889   We build our model based on the pre-trained InternVL2.0 (Chen et al., 2024c) (InternVL for short), a
890   family of advanced and open-sourced VLMs. We chose InternVL for the following reasons: (1) The
891   large-scale and high-performance visual encoder is more capable to handle the text-rich GUI envi-
892   ronment. (2) The dynamic resolution strategy largely preserves the details of the input screenshots,
893   allowing for the perception of fine-grained text and icon information. We take the advantage of
894
      large-scaled visual encoder with a large-scaled GUI dataset described in chapter 3. We further pro-
      pose a Universal Block Parsing (UBP) method to handle with the small object localization problem
895
      brought by dynamic resolution in chapter 4.
896
897   The architecture of SpiritSight is depicted in Figure 5. To begin with, the input image is the GUI
898   screenshot. According to the dynamic resolution algorithm of InternVL, the appropriate ratio of
899   input image is decided. Then, the image is divided into several blocks, each with a unique index, in
900
      preparation for the post-processing phase of our UBP method. These image blocks will be flattened
      as batches before they are sent into visual encoder, which results in the loss of their 2D spatial
901
      relation. To address this issue, we introduce the 2D Block-wise Position Embedding (2D-BPE)(Kim
902
      et al., 2022) method, which maintains the blocks’ 2D spatial relation by adding a row embedding
903   and column embedding to each block. Afterwards, the embedded image features, along with the
904   task objective, the action space and the history actions are passed through the InternLM2 decoder
905   to make the action code inference. Finally, the exact pixel coordinate and the action to be executed
906   is obtained by the UBP parser. We define a separate action space Aspace for each GUI platforms,
907   making our SpiritSight model highly compatible to a variety of GUI navigation tasks. The history
908   actions H are limited within 5 actions to avoid excessive history overload. For each step, SpiritSight
909   would observe the current screen, output the optimal action-code A according to task objective T,
910   history actions H and the given action space Aspace . The detailed prompt template can be seen in
911
      Appendix H.

993    F.1   W EB DATA C OLLECTION
994
995    We collected website URLs from two sources: the Common Crawl (Group, 2024) datasets and URLs
996    from website ranking. We then developed a data collection tool using playwright (Microsoft, 2024)
997
       library to get real-world web data from the collected URLs. We used the URLs from the website
       ranking as a supplement to Common Crawl due to their compromised quality, including a large
998
       proportion of blank pages, sparse-texted pages, and dead pages. We developed a data collection tool
999
       using playwright (Microsoft, 2024) library to get real-world web data from the collected URLs.
1000
1001   We sequentially traverse the collected URLs. For each URL, we start data collection only after
1002   the page has fully loaded. The data collected includes website screenshots and hierarchical element
1003
       structure information. A carefully designed scheme is used to collect the element hierarchy. First, we
       perform grid sampling on the coordinates, with a step size of 8, to ensure that elements with a length
1004
       or width greater than 8 are captured. Then, we obtain the corresponding element objects based
1005
       on the sampled coordinates, which include various information about the elements, such as type,
1006   inner text, coordinates, and interactivity. We determine whether an element is clickable by checking
1007   its pointer property and registered events, and assess whether it can accept text input by checking
1008   its type. We also label these element objects to simplify the HTML information. Specifically,
1009   after collecting all the element information, we developed an HTML pruning algorithm to simplify
1010   the HTML structure. Through this pruning algorithm, all labeled element nodes and those with
1011   structural representation functions are retained. The resulting DOM trees are used to construct the
1012   bbox2dom data.
1013   After collecting the data from the current website, we acquire new pages using two methods:
1014   scrolling down or clicking on an element, with clickable elements randomly sampled from all avail-
1015   able ones. We collect 30 pages for each initial URL. This process is repeated, creating a continuous
1016   cycle of interaction. Ultimately, we collect 740k web-page screenshots along with their DOM an-
1017   notation. Among them, English websites account for 3/4, while Chinese websites account for 1/4.
