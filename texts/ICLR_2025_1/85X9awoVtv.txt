009                                                A BSTRACT
010
011            We study auditing total data withdrawal, the case in which a user requests the ex-
012            clusion of their data from both the training and test data for some machine learn-
013            ing task. This approach is motivated by the need for comprehensive compliance
014            with data privacy regulations and legal frameworks around the world. We con-
015            ceptualize the task of auditing total data withdrawal as an optimization problem.
016            Compliance verification is conducted under mild assumptions using a dedicated
017            verification algorithm. We then evaluate this formulation over various datasets,
018            architectures, and verification hyperparameters. Our verification algorithm serves
019
               as a tool for regulators to ensure auditable compliance and provides enhanced
               privacy guarantees for users.

023   1   I NTRODUCTION
024
025   Tech companies are increasingly seeking user data to enhance the training of their large-scale ma-
026   chine learning (ML) models. However, users frequently wish to impose limitations on the use of
027
      their data, particularly when it involves sensitive personal information. In response, governments
      and institutions have implemented policies to regulate data usage. A notable example is the Euro-
028
      pean Union’s General Data Protection Regulation (GDPR) (European Parliament & Council of the
029
      European Union, 2016a), which outlines strict restrictions on how tech companies, referred to as
030   data controllers, can utilize user data.
031
032   The right to be forgotten mandates that a data controller must delete a user’s data upon request. In
033
      other words, the data controller must not only delete the user’s data but also remove the effect of
      the user’s data on trained models (Federal Trade Commission, 2022). In response, the high cost of
034
      retraining models from scratch has led to the development of machine unlearning (Bourtoule et al.,
035
      2021). Unlearning enables data controllers to remove user data effects without retraining models
036   from scratch. Furthermore, methods have been proposed to verify that machine unlearning has been
037   properly executed (Sommer et al., 2022). However, unlearning only addresses data deletion during
038   the training phase and does not fully comply with GDPR requirements. Thus, we say that unlearning
039   implements partial data withdrawal over the training data.
040
      Instead, the right to object states that a user can completely deny access to their data at any stage
041
      of data processing. That is, it is possible that the user data cannot be used at either train or test
042   time. This sentiment extends beyond the GDPR to other policies for sensitive user data like the
043   Healthcare Insurance and Portability Act (HIPAA) (Gostin, 2001) and is consistent with recent legal
044   frameworks encouraging data transparency (California State Legislature, 2024).
045
      In this work, we study total data withdrawal. Total data withdrawal refers to the user requesting
046
      complete exclusion of their data from all stages of data processing, including both training and test-
047
      ing phases. This ensures that the user’s data is neither used to train the model nor used to make
048   predictions. In particular, we study formal methods to audit data controller compliance with total
049   data withdrawal. Verification methods are encouraged by policies like the GDPR (European Parlia-
050   ment & Council of the European Union, 2016b) and help establish trust by allowing users to audit
051   compliance. Recent violations of user trust have damaged public opinion about data controllers and
052   their data handling practices (U.S. SEC, 2019). Furthermore, lawsuits have been levied against data
053   controllers for violating the right to object (O’Caroll, 2022). At scale, if data controllers do not pri-
      oritize compliance, this could be financially dangerous. Thus, data controllers are incentivized to act

101   2         BACKGROUND AND R ELATED W ORK
102
103   To our knowledge, no previous work provably verifies compliance with total withdrawal. However,
104   we review related work in adjacent areas. We particularly focus on machine unlearning and its
105   verification methods (Sekhari et al., 2021), which, although different in goal, share similarities with
106   our approach. Additionally, we discuss robustness verification (Salman et al., 2019), which we adapt
107   to implement our verification framework. Furthermore, we draw connections between our work and
      broader concepts in data privacy.

108
      Machine Unlearning: Machine unlearning is inspired by the GDPR’s right to be forgotten, man-
109   dating that data controllers delete user data upon request (European Parliament & Council of the
110   European Union, 2016a). In practice, data controllers must remove user data and its effects from
111   trained models and algorithms. A naive approach is to retrain the model from scratch without the
112   user’s data. However, this is prohibitively expensive for large models (Sekhari et al., 2021). Thus,
113   various methods have been developed to comply without retraining from scratch, including resource-
114   efficient retraining (Bourtoule et al., 2021), differentially private unlearning (Sekhari et al., 2021),
115   and finetuning through student-teacher knowledge distillation (Kurmanji et al., 2024).
116   However, unlearning does not ensure compliance at test time. To clarify, unlearning ensures partial
117   data withdrawal by removing user data from the training set only. We propose total data withdrawal,
118   excluding user data from all stages of processing, including training and testing. This approach
119   provides more comprehensive compliance with regulations like the GDPR.
120
      Unlearning Verification and Auditing Compliance: With growing interest in unlearning, verifica-
121   tion methods have been increasingly studied (Wang et al., 2024). These methods involve hypothesis
122   testing (Sommer et al., 2022), federated learning (Gao et al., 2024), data poisoning (Xu et al., 2024),
123   measuring gradient changes (Zhou et al., 2024), and cryptographic primitives (Eisenhofer et al.,
124   2022).
125
      These methods audit compliance with partial data withdrawal, focusing on the training set. How-
126
      ever, policies like the right to object require data controllers to exclude user data from all sources,
127   including test data. Unlearning verification overlooks this while our method addresses it. Further-
128   more, current unlearning verification methods can be circumvented (Zhang et al., 2024) and thus
129   may not guarantee data withdrawal.
130
      Additionally, membership inference attacks (MIAs) provide an empirical notion of unlearning veri-
131
      fication. We elaborate more on the differences between our work and MIAs in Appendix C.
132
133   More generally, our setting differs from typical compliance issues in machine learning, such as
134   privacy-preservation (Steinke et al., 2024) or fairness (Yan & Zhang, 2022).
135   Robustness Verification: Our problem is loosely related to robustness verification (Salman et al.,
136   2019), ensuring a neural network classifies all instances in an ϵ-ball consistently. In doing so, one
137   verifies that their model is robust to adversarial attacks within that ϵ-ball (Goodfellow et al., 2014).
138
      Beyond adversarial robustness, verification methods like this have been applied to power systems,
139
      aviation, and other fields (Bak et al., 2021). To the authors’ knowledge, legal compliance has not
140   been studied in this context.
141
142   Data Privacy: Due to the risk of data leakage from large information systems (Al-Rubaie & Chang,
143
      2019), data privacy is a critical paradigm within computer science. Often, large data controllers such
      as social media companies use end-to-end encryption (Perrin & Marlinspike, 2016) to ensure privacy
144
      for their users. Since a data controller cannot use the data, they have no incentive to keep the data
145
      or violate privacy. This is an implicit premise of end-to-end encryption. Our formulation is posed
146   similarly. It verifies that the data controller cannot use the data, and therefore has no incentive to
147   keep the data or violate the law.
148
149
      In the context of machine learning, data privacy is ubiquitous as well (Fredrikson et al., 2015). Ap-
      proaches to privacy-preserving machine learning include differential privacy (Dwork et al., 2006)
150
      and homomorphic encryption (Brakerski et al., 2014). In some settings, our formulation can also
151
      verify privacy-preservation. Specifically, for identification tasks like facial recognition, our algo-
152   rithm can accurately verify that a user cannot be identified. Thus, our work can also be thought of
153   as a novel data privacy verification method. We leave this to future work.

      3.2   P ROBLEM S ETTING
174
      As a reminder, we aim to verify compliance with total data withdrawal for a task T . To that end, we
175   define total data withdrawal as follows:
176
177   Definition: Total data withdrawal is when a user requests that their data instance is neither used to
178
      train a machine learning model for task T nor used for task T itself at test time.
179   This means we should present a condition that enables a regulator to confirm that the data controller
180   has not used the user’s data in either the train or test set. To clarify, it is insufficient for a regulator to
181   verify unlearning alone. Unlearning verification only ensures partial data withdrawal during training
182   and offers no guarantees at test time. Importantly, verifying data deletion from a provided test set is
183   insufficient to ensure compliance at test time; an adversarial data controller could hide the true test
184
      set, and thus circumvent the law.
185   To establish a reliable condition, we let a regulator verify that the data controller misclassifies the
186   user’s instance. We also consider that the data controller might adversarially perturb the instance to
187   ensure correct classification, even if the original instance is misclassified (Goodfellow et al., 2014).
188   This altered instance could be used for task T , circumventing the law. To prevent this, we verify that
189   the data controller misclassifies all instances within a closed ball around the original instance.
190   We formally demonstrate that this condition is both necessary and sufficient for verification. It is
191   important to note that while this condition is not required for general compliance, it is essential for
192   auditable compliance. A data controller can technically comply by simply deleting the data from
193   both the training and test sets, yet still classify these instances correctly. However, such compliance
194   cannot be audited, which undermines transparency. In our setting, transparency is crucial, and thus,
195   satisfying this condition is necessary. Even if the data controller is trustworthy, they cannot achieve
196
      transparency without meeting this condition.

258   4.2   S OUND AND C OMPLETE V ERIFICATION
259
260
      There are two main properties a verification algorithm satisfies: soundness and completeness. An
      algorithm is sound (complete) if every time it verifies (falsifies) a property, the answer is guaranteed
261
      to be correct (Liu et al., 2021). In our context, soundness means that whenever we verify, the
262
      data controller has auditably complied, and completeness means that whenever we falsify, the data
263   controller has not auditably complied.
264
265   A verification algorithm is considered sound if it correctly verifies a property and complete if it
266
      correctly falsifies a property. In practical terms, for verification methods based on optimization such

307   5   E XPERIMENTS
308
309   We present numerical evidence demonstrating the effectiveness of our algorithm in verifying total
310   data withdrawal. Our approach is straightforward for identification tasks, such as face recognition,
311   where removing the identifying label of xu is sufficient to induce misclassification. However, in-
312   ducing misclassification without merely deleting the label is more complex. Therefore, to extend
313
      our approach beyond identification tasks, we have designed custom benchmarks.
314   Benchmarks: We construct two primary benchmarks to make a clear distinction between the object
315   set and the rest of the data. We use the popular datasets MNIST (Lecun et al., 1998), FashionMNIST
316   (FMNIST) (Xiao et al., 2017), Kuzushiji-MNIST (KMNIST) (Clanuwat et al., 2018), and CIFAR-
317   10 (Krizhevsky et al., 2009) as the starting datasets. These widely used datasets represent a diverse
318   set of challenges for our verification method.

320   dataset as follows:

      from N (0, 0.01).

         6            D ISCUSSION
412
413
         In this work, we examine total data withdrawal, where the data controller is prohibited from us-
         ing a user’s instance during both training and testing phases. To enhance trust between users and
414
         data controllers, it is essential that an independent regulator can audit the data controller for legal
415
         compliance. Accordingly, we propose a novel method for auditing compliance. Our sound and com-
416      plete algorithm ensures that the data controller adheres to total data withdrawal requirements. Our
417      method enables efficient verification for identification tasks, where deleting the user’s label induces
418      misclassification. We provide both formal and empirical evidence demonstrating that our approach
419      accurately verifies and falsifies compliance.

      Limitations: Our formulation is limited to the δ-ball, multiclass classification settings, and fm
      which have approximately 0 training loss. Beyond the label deletion case, our method relies on the
453
      data controller’s ability to induce misclassification of particular instances.
454
455   Future Directions: Firstly, to address our limitations, future directions include designing verifica-
456   tion formulations for binary classification, regression tasks, fm that do not interpolate perfectly, and
457   that handle adversarial changes to xu independent of δ. Additionally, data controllers need algo-
458
      rithms to efficiently obtain fm from f0 , i.e. inducing misclassification over Xoadv , without severely
      affecting the generalization of fm .
459
460   Next, while neural network verification techniques are natural for our problem, probabilistic relax-
461   ations with hypothesis testing (Sommer et al., 2022) might improve resource-efficiency, extend to
462   different architectures, and address our limitations.
463   Then, extending this method to diverse datasets across different modalities, like text, is an important
464   challenge. Total data withdrawal can be invoked over non-image data as well. Verifying over text
465   remains a difficult problem in verification more broadly (Casadio et al., 2024).
466
      Finally, working with regulators to bring this method into practice is a worthwhile future direction
467
      as well.

729   A    A SSUMPTIONS
730
731   To begin, we provide evidence that Assumption 1 holds for many datasets used in machine learning
732   research, some of which are used in Sec. 5. From Table S3, it is clear that many datasets do not
733   consider binary classification as their setting.
734   For Assumption 2, we consider it in its most general form where all instances in the train and test
735   data are at least ϵ apart. We provide this ϵ for common datasets in both input and embedding spaces
736   in Table S4. From Table S4, it is clear that ϵ is usually fairly large in both the input and embedding
737   spaces, especially when compared to the adversarial budgets in the robustness literature Bai et al.
738   (2021). Note that the δ used in Sec. 5 are always lower than the smallest ϵ for that dataset across the
739   pixel and embedding spaces.
740   To elaborate on what occurs when our assumptions are relaxed, consider the case that Assumption 1
741   is relaxed. Then, the data controller can find the true class for xu by taking complements of fm (xu )
742   and thus use xu for task T . Thus, our condition no longer becomes sufficient.
743
      Then, consider what occurs when Assumption 2 is relaxed. Then, there might exist an instance
744   xk belonging to user k in B̄δ (xu ). This instance xk may be classified as t, although all instances
745   related to xu in B̄δ (xu ) are misclassified. Thus, we would be falsifying incorrectly. In particular,
746   our condition no longer becomes necessary and our verification formulation incomplete by default.
747
      From the CIFAR5 results in Table 2, it is clear that we do not falsify correctly if our train loss
748
      does not converge to 0 or approximately 0. This illustrates the necessity of Assumption 3. Thus, in
749
      the case that Assumption 3 is relaxed, our method verifies partial data withdrawal over the test set.
750   Notably, neural networks satisfying Assumption 3 are often called interpolators and are of interest
751   in deep learning theory (Taheri & Thrampoulidis, 2023).

973
974    To run αβ-CROWN, we consider the following:
975
976           • We generate a file custom.py to store our model definitions.
977
              • For each object set instance, we generate a VNNLib file (Demarchi et al., 2023) bounding
978
                each pixel with respect to its ℓ∞ bounds per B̄δ (xu ). We get the true class of xu beforehand
979             and consider the output specifications in line with Appendix D.
980
981
              • For the yaml file, we let the operations be deterministic, use customized model definitions
                in custom.py, and load in object set instances via a VNNLib CSV. We also set the batch
982
                size to 2048 and the maximum branch-and-bound iterations to 30 in order to preserve GPU
983
                memory. Finally, we set the operations to be deterministic for reproducibility. Otherwise,
984             we use default arguments.
985
986    Since we use default arguments, the following occurs during verification:
987
988          1. We run a PGD attack on the object set instance, performing complete but not sound verifi-
989             cation. In our case, we are trying to find an instance in B̄δ (xu ) that is classified correctly.
990             If this fails, we proceed.
991          2. Then, we run CROWN, performing sound but incomplete verification. We do so as this
992             tends to be faster than running β-CROWN alone. If this fails i.e. CROWN falsifies, we
993             proceed.
994
             3. Finally, we run β-CROWN, performing sound and complete verification.
