009                                              A BSTRACT
010
011
012           Open-weight LLMs are particularly appealing choices to generate training data
013           for fine-tuning Code LLMs on domain-specific service robot applications because
014
              they are cost-effective, customizable, and offer better privacy protection. How-
              ever, unlike proprietary LLMs, open-weight models are more error-prone and
015
              often produce programs that violate domain-specific constraints. A promising
016
              solution is to incorporate a robot simulator with a well-defined environment to
017           verify program correctness. Yet, these environments require pre-enumeration of
018           relevant entities and their states, which limits the diversity of programs that can be
019           effectively verified. In this work, we introduce ROBO -I NSTRUCT that preserves
020           the diversity of programs generated by an LLM while providing the correctness of
021           simulator-based checking. ROBO -I NSTRUCT introduces ROBO S IM to dynamically
022           synthesize consistent simulation environments for each generated program. More-
023           over, ROBO -I NSTRUCT handles subtler instruction-program inconsistencies that
024           do not result in a constraint violation via I NSTA LIGN, an LLM-aided instruction-
025
              program alignment process. Given domain-specific APIs and a few seed examples,
              ROBO -I NSTRUCT can leverage an 8B Llama3 model to generate a training dataset
026
              for fine-tuning a 7B CodeLlama model. Our fine-tuned model achieves a 28.75%
027
              improvement in pass@1 over the original base model and a 13.75% improve-
028           ment compared to its S ELF -I NSTRUCT-finetuned counterparts, even surpassing the
029           performance of a few proprietary LLMs, such as GPT-3.5-Turbo and Gemini-Pro.

122   2     ROBO -I NSTRUCT
123
124   In this section, we describe how ROBO -I NSTRUCT generates training datasets for domain-specific
125   service robot programs. Fig. 1 provides a high-level overview of the framework. To add an entry to
126   the training dataset, S ELF -I NSTRUCT first generates an instruction-program pair based on robot APIs
127   and seed tasks (detailed prompts in Appendix A.4.2). Then, ROBO S IM dynamically synthesizes a
128
      consistent simulation environment to validate the program as it executes.
129   If the program is invalid, ROBO -I NSTRUCT applies a rejection-sampling method, which generates
130   a new program based on the same instruction and re-evaluates it. This process continues until the
131   program is valid or a predefined maximum resampling limit is reached. If the limit is exceeded, the
132   instruction may be incompatible with the domain-specific APIs or too complex, so the instruction-
133   program pair is discarded.
134   Finally, if the program is valid, I NSTA LIGN uses an open-weight LLM to refine the instruction to
135   better align with the program’s intent, and the revised instruction-program pair is added to the training
136   dataset. In the following subsections, we delve into the specific design of each component.
137
138   2.1   ROBO S IM : DYNAMIC S YNTHESIS OF S IMULATION E NVIRONMENTS FOR P ROGRAM
139         VALIDATION
140
141   We present a principled approach to designing ROBO S IM, a system for dynamically synthesizing
142   consistent simulation environments to validate domain-specific robot programs. For service mobile
143
      robots, a simulation environment often relies on three concepts:
144
            1. A list of entities to reason about, e.g., “apple”, “kitchen”
145
146         2. The type of the entities, and hence their affordances, e.g., “apple” is an object, you can pick
147
               it up; “kitchen” is a location, you can go to it, and it contains objects.
148         3. The state of the entities in the world, e.g., the “apple” is in the “kitchen”.
149
150   These concepts are closely related to the domain-specific APIs, where each API invocation during
151   program execution can trigger updates to the simulation environment. To handle the interaction
152
      between APIs and simulation environments, we introduce DYNAMIC E VAL, an algorithm inspired
      by Angelic Execution (Broy & Wirsing, 1981), a software engineering technique to infer program
153
      properties from incomplete API specifications. DYNAMIC E VAL automatically generates a simulation
154
      environment for each program and checks its correctness within the inferred environment.
155

      pick(”apple”), DYNAMIC E VAL infers that apple is an entity to consider, even if it is not currently
159

257   2.2                I NSTA LIGN : LLM- AIDED I NSTRUCTION -P ROGRAM A LIGNMENT P ROCEDURE
258
259   Even after ROBO S IM verifies that a program doesn’t violate domain-specific constraints, it may still
260
      have subtle inconsistencies with the instructions — such as omitting a step implied by the instruction.
      The proposed rejection-sampling strategy alone cannot resolve this issue, and correcting verified
261
      programs to fully reflect the instructions while ensuring they remain valid is a challenging task.
262
263   We note that LLMs nowadays have demonstrated impressive code understanding capabilities (Rozière
264   et al., 2024; Nam et al., 2024; Leinonen et al., 2023; Li et al., 2023; Lekshmi-Narayanan et al., 2024).
265   Instead of correcting the program, it may be more effective for the LLM to revise its generated
266
      instructions to better align with the program’s intent.
267   We propose I NSTA LIGN, a procedure that prompts an LLM to revise generated instructions to better
268   match the intent of the program, as shown in Fig. 4. I NSTA LIGN follows two steps: first, it uses
269   Chain-of-Thought reasoning (CoT) (Wei et al., 2022) to generate a revised instruction based on the
      instruction-program pair; then, it prompts the LLM to compare the original and revised instructions,

      setting the temperature T = 1 and top p = 0.95. The maximum resampling limit is capped at 3 to
359
      accommodate instructions that initially produce invalid programs. For the LLM used in I NSTA LIGN,
360   we empirically adjust the generation temperature to T = 0.3 to optimize performance. Furthermore,
361   we assess the edit similarity between token sequences of each instruction pair in the dataset (Lee
362   et al., 2022), removing duplicates where the similarity score exceeds 0.6. The same similarity-
363   based approach is used to decontaminate the dataset against the ROBO E VAL benchmark tasks. We
364   use the same setup to generate data via S ELF -I NSTRUCT. Instead of discarding invalid programs,
365   S ELF -I NSTRUCT includes every generated instruction-program pair in the training dataset. Finally,
366   we create two datasets with 5K instruction-program pairs each using S ELF -I NSTRUCT and ROBO -
367   I NSTRUCT respectively. These datasets are then used to fine-tune the CodeLlama-Python-7B model.
368
      The learning rate is set to be 3e-5 with a warmup ratio of 3% and a constant lr scheduler. We employ
      the AdamW optimizer (Loshchilov & Hutter, 2019) with an effective batch size of 8, training each
369
      model for 5 epochs using a sequence length of 2048 tokens. We train all our models on a single
370
      H-100 GPU using unsloth (Unslothai, 2024).
371

      5.1   LLM S FOR ROBOT C ODE G ENERATION
489
490   LLMs have shown impressive capabilities in generating robot programs from natural language (Liang
491   et al., 2022; Singh et al., 2023; Huang et al., 2023a). One popular approach uses LLMs to generate
492   composable costmaps for robots to plan their motion on. In this approach, Voxposer (Huang et al.,
493   2023b) focuses on the tabletop manipulation setting and NavCon (Biggie et al., 2023) focuses on
494   creating composable maps for navigation. Using LLM to create reward functions is also promising.
495
      Eureka (Ma et al., 2023; 2024) and Language to Rewards for Robotic Skill Synthesis (Yu et al., 2023)
      both show that LLM can generate good reward functions that allows robots to acquire complex skills.
496
      Finally, LLM can also be used to generate programs for high-level planning. LLM+p (Liu et al.,
497
      2023a) outputs a robot plan in the form of the well-defined planning domain definition language
498   (PDDL). Tidybot (Wu et al., 2023) uses an LLM to generate a rule that captures user preferences
499   from examples and executes a program to sequentially complete the task in order. RoboEval (Hu
500   et al., 2024) focuses on generating domain-specific programs for service mobile robots. It generates a
501   program that allows the service robot to carry out long-horizon tasks and then validates the correctness
502   of the program.
503   5.2 G ENERATING DATASETS F OR F INE - TUNING LLM S
504
505   To enhance LLMs’ performance in code generation, numerous studies have explored the creation of
506   specialized datasets (Muennighoff et al., 2024; Köpf et al., 2023; Muennighoff et al., 2022). S ELF -
507   I NSTRUCT (Wang et al., 2022) is one popular method for generating synthetic datasets using an LLM.
508
      Following this methodology, Alpaca (Taori et al., 2023) generates 52K instruction-following demon-
      strations and subsequently fine-tunes the LLaMA 7B model (Touvron et al., 2023) to create Alpaca
509
      7B, which can behave qualitatively similarly to OpenAI’s text-davinci-003. Code Alpaca (Chaudhary,
510
      2023) extends this approach to generate code instructions using 21 seed tasks, while Gorilla-LM (Patil
511   et al., 2023) adapts the method to focus on ML domain-specific APIs from Huggingface, TensorFlow
512   Hub, and Torch Hub. To create more complex instructions, Evol-Instruct (Xu et al., 2024; Luo
513   et al., 2024) proposes iteratively updating instructions to become more complex through different
514   prompting strategies. In addition to Evol-Instruct, OSS-Instruct (Wei et al., 2023) uses open-source
515   code snippets to generate 75K high-quality instruction data and fine-tunes the CodeLlama-Python-7B
516   model to create Magicoder, which can match the performance of GPT-3.5-Turbo (OpenAI, 2022) on
517   HumanEval (Chen et al., 2021). While these works focus on creating seed instruction sets to generate
518   synthetic data to effectively fine-tune an LLM, our research investigates post-processing methods in
519
      addition to S ELF -I NSTRUCT. Specifically, we concentrate on generating domain-specific programs
      in robotics (Hu et al., 2024), where we can effectively leverage constraints to filter out erroneous
520
      programs.
521
522   6     C ONCLUSION , L IMITATION AND F UTURE W ORKS
523
524   In this work, we introduce ROBO -I NSTRUCT, a novel framework to generate synthetic training
525   data to fine-tune small language models for domain-specific robot programs. ROBO -I NSTRUCT
526   comprises two novel components: 1) ROBO S IM, a method to synthesize consistent simulation
527   environments dynamically for verifying the generated programs, and 2) I NSTA LIGN, an LLM-aided
528   instruction alignment procedure to revise instructions to better align with the generated programs.
529
      The experimental results show that the 7B CodeLlama model fine-tuned on the ROBO -I NSTRUCT
      dataset significantly outperforms larger open-weight LLMs and proprietary models like GPT-3.5-
530
      Turbo and Gemini-Pro in generating service robot programs. However, a key limitation is that
531
      ROBO -I NSTRUCT relies on S ELF -I NSTRUCT to filter invalid programs, which may introduce biases
532   and affect dataset quality. Another limitation is the use of a simple rejection-sampling method to
533   handle invalid programs, which may not fully address the underlying issues. Future work will focus
534   on improving dataset quality by integrating ROBO -I NSTRUCT with advanced methods like Evol-Inst
535   and OSS-Inst.

773   A.2     R ELEVANCE TO STRIPS PLANNING
774
775   The proposed DYNAMIC E VAL shares significant similarities with the formulation of STRIPS planning.
776   A STRIPS instance is typically represented as a tuple ⟨I, G, A, P ⟩, where I denotes the initial state
777   of the simulation environment, G represents the desired goal state that the robot aims to achieve, A
778   defines the set of actions available to transition between states, and P is the set of preconditions that
779
      must be satisfied before performing actions. Thus, DYNAMIC E VAL can be reformulated to align with
      the STRIPS formulation as shown in Alg. 3. Each API invocation corresponds to an action, and its
780
      precondition consists of a set of literals, representing specific combinations of entities, types, and
781
      states.
782
783   To address this, we extend the classic STRIPS formulation by incorporating dynamically discovered
784   literals. Unlike the conventional STRIPS approach, where each literal is binary—True when defined
785
      and False when not—we introduce a third value, ”Undefined.” This means a literal must be explicitly
      defined as either True or False; otherwise, it remains in the Undefined state. When an action requires
786
      a literal that is undefined, a random value (True or False) is assigned to it, and the literal is added to
787
      the state of the simulation environment (line 7). Once the precondition is fully defined, the action is
788   executed, and domain-specific constraints are checked for any violations (line 10). This extension
789   enables DYNAMIC E VAL to handle arbitrary programs effectively.
790
791   Algorithm 3 DYNAMIC E VAL STRIPS(api fn, params, W)
792
       1: Input: api fn                                                                  ▷ The API function name
793
       2: Input: api inputs                                          ▷ The input received by the API invocation
794    3: Input: W                                            ▷ The current state of the simulation environment
795    4: p ← G ET P RECOND(api fn, params)                 ▷ Get the parameter-specific precondition for api fn
796    5: for l ∈ p do                                          ▷ Loop through every literal in the precondition
797    6:    if C HECK D EFINED(W, l) is Undefined then
       7:       W ← G ROW W ORLD(l, W)                ▷ Randonly instantiate the literal and grow W to include it
798
       8:    end if
799    9: end for
800   10: retval, W ← E XEC U PDATE(api fn, params, W)                           ▷ Execute api fn and update W
801   11: return retval
