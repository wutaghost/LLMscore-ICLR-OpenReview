011                                               A BSTRACT
012
013            Knowledge distillation from Large Language Models (LLMs) to locally hosted
014            Small Language Models (SLMs) provides advantages for Data Science Code Gen-
015            eration (DSCG) such as enhanced data privacy and reduced response times. How-
016            ever, achieving effective distillation without resource-intensive training is chal-
017            lenging. This paper investigates whether LLMs can distill knowledge to SLMs
018            through In-Context Learning (ICL), a training-free method for rapid task adap-
019            tation. We present the Agents Help Agents (AHA) framework, which facili-
020
               tates automatic knowledge distillation from LLMs to SLMs via agent orchestra-
               tion. AHA consists of three phases: exploration through an Agent Orchestration
021
               Interface (AOI), memory collection of successful examples, and inference aug-
022
               mented with distilled knowledge. The AOI orchestrates interactions between a
023            LLM as a Teacher Agent and a SLM as a Student Agent. And we propose two
024            distillation strategies: a static approach that aggregates an offline instruction set
025            and a dynamic RAG-based approach that distills knowledge dynamically during
026            inference. We evaluate AHA on three challenging code generation tasks for tab-
027            ular data analysis: TAB MWP, B IR D-SQL, and W IKI TQ. Experimental results
028            demonstrate the effectiveness of AHA, leading to an average 27.5% relative im-
029            provement in the performance of the Student Agent Phi-3-mini. Additionally,
030            relative gains of 14.3% and 30.9% are observed in Llama-3.1-8B and GPT-35-
031
               Turbo, respectively, even though those models were not calibrated as part of the
               orchestration, highlighting the model-agnostic nature of the distilled knowledge in
032
               AHA. Further analysis compares distillation and demonstration techniques across
033
               different data input settings, providing insights into optimal configurations for
034            DSCG.

038   1   I NTRODUCTION
039
040
      Data Science Code Generation (DSCG) automates the conversion of natural language queries into
041
      executable code, empowering non-expert information extraction and analysis from tabular data ef-
042   ficiently. This process enhances productivity, reduces the technical barrier for data analysis, and
043   allows data scientists to focus on deriving insights, ultimately supporting more effective decision-
044   making (Khanbabaei et al., 2018; Han et al., 2011; Fayyad et al., 1996). This is a challenging task
045   since it not only requires code generation capability but also data understanding capability.
046
      Large Language Models (LLMs) have demonstrated remarkable performance across diverse, com-
047
      plex tasks (Singh et al., 2023; Mu et al., 2024; Chen et al., 2023; Zheng et al., 2024; Deng et al.,
048   2023). Leveraging LLMs or LLM agents for automatic code generation from user queries offers an
049   effective solution (Yang et al., 2024b; Wang et al., 2024b). However, the integration of LLMs in
050   DSCG faces two primary challenges: 1) Privacy concerns arise when utilizing closed-source LLMs
051   such as GPT-4 (Achiam et al., 2023) or Claude-3.5-Sonnet (Ogunseyi et al., 2023). 2) Deploying
052   large open-source models like Llama-3.1-405B (Dubey et al., 2024) or DeepSeek-v2 (236B) (Liu
053   et al., 2024) can be challenging due to their large number of parameters. Balancing these benefits
      and challenges is crucial for effective data science applications.

054
      Small Language Models (SLMs), such as Phi-3-mini (Abdin et al., 2024) and Llama-3.1-8B (Dubey
055   et al., 2024), have gained attention for their In-Context Learning (ICL) capabilities but more advan-
056   tages for local deployment and on-device inference. These models offer computational efficiency
057   and enhanced data privacy, crucial for resource-constrained or privacy-sensitive applications (Joshi
058   et al., 2024). While SLMs have shown competitive performance in some general tasks including
059   natural language understanding (Nie et al., 2020) and code completion (Chen et al., 2021), their
060   effectiveness in data science code generation tasks remains an open question.
061   Fine-tuning is a common strategy to enhance SLM capabilities for complex tasks (Petroni et al.,
062   2021). However, this approach encounters several challenges in the domain of data science DSCG.
063   One primary issue is the limited availability of high-quality training data. Professional tabular
064   datasets, such as relational databases, are often small or proprietary, restricting access to substantial
065   corpora for training. Additionally, the dual expertise required in both coding syntax and data un-
066   derstanding for accurate annotation further constrains dataset scalability (Li et al., 2024b; Lei et al.,
067   2024). This is reflected in recent benchmarks for data science code generation, which typically con-
068   tain around or fewer than 1,000 samples, highlighting the complexity and resource constraints in
069
      this field (Hu et al., 2024; Agashe et al., 2019; Lai et al., 2023; Zhang et al., 2023; Yin et al., 2023).
      Recent research has explored distillation from LLMs to SLMs through fine-tuning on synthetic data
070
      generated (Team et al., 2024; Magister et al., 2023; Kang et al., 2023). While this approach shows
071
      promise, several challenges persist. For example, frequent updates to code packages introduce new
072   syntax that may conflict with previously trained knowledge (Wu et al., 2024). Furthermore, the per-
073   formance improvements obtained from these methods often fail to generalize well across different
074   programming languages or frameworks, requiring extensive re-training for each package update or
075   new task (Shen et al., 2022a; Ke et al., 2023). However, In-Context Learning (ICL) can adapt to new
076   requirements or tasks by providing relevant instructions or examples, reducing the effort required
077   for re-training or continual training. This raises a research question in DS code generation: Can
078   LLMs distill knowledge to SLMs through In-Context Learning (ICL)?
079   In this paper, we explore the potential of knowledge distillation from LLMs to SLMs via ICL. We
080   design Agents Help Agents (AHA), a novel, fully automated framework that enables LLM as a
081   Teacher Agent to guide SLMs as Student Agents in complex data science code generation tasks.
082   AHA operates in three phases: exploration, memory database collection, and knowledge-driven
083   inference. During exploration, we employ the Agent Orchestration Interface (AOI) that allows an
084   LLM to probe and analyze SLM code knowledge by converting questions into step-wise functional
085   plans and asking SLMs to infill the code for each plan. Then, successful collaborated cases are
086
      stored in memory databases. We also introduce two novel distillation techniques during inference:
      General Instructions and Meta Instructions.
087
088   We evaluate AHA on three challenging tabular analysis datasets that need code generation:
089   TAB MWP (Lu et al., 2023), B IRD -SQL (Li et al., 2024a), and W IKI TQ (Pasupat & Liang, 2015).
090   The experimental results demonstrate that the AHA framework significantly improves the perfor-
091   mance of SLMs across all datasets, validating the potential of our knowledge distillation approach
092
      via In-Context Learning (ICL). Notably, the performance boost achieved by AHA is not limited to
      the specific SLM trained during the orchestration process but generalizes across other SLMs as well.
093
      This model-agnostic nature further highlights the flexibility and adaptability of distilled knowledge
094
      in our framework, enabling it to be applied in a wide range of data science code generation scenarios
095   without requiring extensive retraining.

      2.2             AGENT O RCHESTRATION I NTERFACE
126
127   The Agent Orchestration Interface (AOI) is designed to mediate the interaction between a Large
128   Language Model (LLM), a Teacher Agent, and a Student Agent, represented by the SLM. The
129   primary goal of the AOI is to generate successful and informative problem-solving cases, which
130   are later used for knowledge distillation in DSCG. The AOI is composed of three key components:
131   Abstraction Lifting, Orchestrated Coding, and Plan Optimization.
132   Abstraction Lifting (AL). In this phase, LLM generates a functional plan Pi =
133   {si1 , si2 , . . . , siK } based on a query qi , data input di , and the corresponding ground truth (gt) code
134   c˜i . The ground truth code c˜i can either be sourced from an existing dataset or generated by the
135   Teacher Agent when it is not directly available but a ground-truth answer string exists. This func-
136   tional plan is defined as Lal (di , qi , c˜i ), where Lal LLM performing abstraction lifting. Each step sij
137   in the plan corresponds to a key subtask derived from the query, collectively forming a structured
138   template outlining the solution process. These steps are annotated by the LLM with descriptive
139   comments and placeholders such as [Fill Your Code Here] in Python or [Fill Your
140
      Sub-Query] in SQL, as shown in Figure 2, ensuring that the SLM follows the logical flow of
      the entire plan and enables guided code generation. Unlike Chain-Of-Thought (Wei et al., 2022)
141
      plans, which provide intermediate steps in continuous textual form, our approach bridges high-level
142
      problem understanding with low-level code implementation logic, allowing the SLM to follow the
143   better plan for data science code generation.
144
145   Orchestrated Coding (OC). Once the functional plan Pi is provided, the SLM considers all
146   context including the question and data input to generate the complete orchestrated code ci =
147
      fgen (di , qi , Pi ) in a single turn by filling all placeholders, ensuring the solution is correct and ex-
      ecutable. The results from executing this orchestrated code are then compared to those from a ref-
148
      erence solution (such as ground truth answer string or gt codes) to evaluate whether the SLM fully
149
      understands both the data and the logic needed to answer the question. This comparison serves as
150   a key indicator of the problem-solving accuracy of SLM and alignment with the intended solution.
151   While the ground truth code may already be available from datasets or generated by the Teacher
152   Agent, orchestrated coding and abstraction lifting are crucial for a few reasons. First, AL breaks
153   down complex problem-solving tasks into manageable sub-tasks, with the potential to improve the
154   performance of the SLMs across a wide range of analytical queries by assisting them in under-
155   standing modular structure. Additionally, error isolation can be grounded in the program structure,
156   enabling more precise identification of issues and contributing to optimized plans. This is supported
157   by our analysis in Section 3.6 that compares chain-of-thought with functional plans across multiple
158
      turns of orchestration.
159   Plan Optimization (PO). The plan optimization process is an iterative procedure that unfolds
160   over multiple turns, denoted by t. During each iteration, the SLM refines the functional plan Pit .

      database. This database includes the correct orchestrated codes, along with the context of the ques-
198
      tion and data input. This process ensures that the SLM can efficiently reference and apply related

      knowledge to new, unseen queries.

237   2.4   K NOWLEDGE D ISTILLATION FROM M EMORY DATABASE
238
239   This part presents two methods for distilling knowledge from the memory database: fine-to-coarse
240   general instruction generation and RAG-based meta-instruction generation. Distilled knowledge
241   then guides the SLM in learning how to plan and generate code more accurately for unseen queries.
242
243   Fine-to-Coarse Knowledge Distillation for General Instructions Generation. Our first ap-
244   proach generates universally applicable, "plug-and-play" instructions through a novel fine-to-coarse
245   knowledge distillation method. This technique employs a recursive, tree-based strategy, offering an
246   alternative to conventional sequential updating methods. Approaches such as (Askari et al., 2024)
247
      process batches of examples sequentially and need to select initial examples that often require hu-
      man efforts. In contrast, our method aims to enable a fully automated workflow, constructing a
248
      knowledge tree recursively and in parallel, thereby reducing both biases and the need for human
249
      intervention. We define the set of distilled General Instruction Ig as:
250
251                                Ig = Lsum (M) = Tl ,     Tl = Lagg (Tl−1 ),                               (4)
252
      where Lsum is a recursive function executed by the LLM, and Tl is the root node of the tree with the
253
      highest layer l, which encapsulates task-specific knowledge, enabling the SLMs to apply it to new
254
      examples.
255
256   At each recursive step, nodes at layer l aggregate knowledge from layer l − 1 using the aggregation
257   function Lagg , which summarizes local instructions within each batch (see prompts in Figure 12,
258
      13 of Appendix H for reference). The leaves represent the individual case studies recorded in M,
      while higher layers abstract and generalize knowledge from lower ones. The layer of the tree and
259
      the final number of distilled rules are hyper-parameters, allowing for balancing complexity and
260
      generalization capability. This hierarchical structure can capture broadly applicable rules, resulting
261   in a more general instruction. General Instruction then guides the SLM in generating correct code
262   for unseen queries in a plug-and-play manner represented by Figure 1 (Right). A detailed illustration
263   with examples can be found in Appendix A.
264
265   RAG-based Knowledge Distillation for Meta Instruction Generation. General Instructions,
266   while broadly applicable, often fall short when addressing long-tail problems. To mitigate this gap,
267   we propose a Retrieval-Augmented Generation (RAG) framework for localized instruction distilla-
268   tion. In the retrieval phase, we identify the top relevant examples from a memory database M via
269   an embedding model. Relevance is measured via a function D, expressed as R(qi ) = D(qi , M, k),
      where k is number of most relevant cases.

      Under review as a conference paper at ICLR 2025


270
      Then, rather than leveraging the entire heterogeneous context, the case studies of these retrieved
271   examples are then fed into the SLM to extrapolate plans for solutions, adhering them to the specific
272   query. Here, SLM performs a secondary distillation, extracting shared knowledge patterns from
273   these case studies, which have already been distilled by the LLM (Teacher Agent), to generate
274   instructions, noted as Meta-Instruction (Im (qi )), precisely specific to the current query at hand.
275   The process is formalized:
276                                          Im (qi ) = fagg (qi , R(qi ))                               (5)
277
278   where fagg is an aggregation function applied by the SLM. By doing so, SLMs can generate more rel-
279   evant and contextually appropriate instructions, effectively bridging the gap between general knowl-
280   edge and query-specific requirements.
281
282   Knowledge-Driven Inference. Harnessing the distilled instructions I ∈ (Ig, Im(qi )) from the
283   memory database, the SLM initially formulates a structured plan pgen , which it subsequently em-
284   ploys to generate code for new queries. For a given query qi and its associated data di , this process
285   unfolds as follows:
286                             Pgen = fplan (qi , di , I),       ci = fgen (Pgen , qi , di ),           (6)
287
      where fplan denotes the planning function executed by the SLM. This plan serves as a blueprint,
288
      guiding the following code generation phase. The SLM then employs the function fgen , which takes
289
      Pgen along with the original query qi and data di to generate the final code ci .

404   3.4   D ISTILLATION V.S. D EMONSTRATION
405
406   In this section, given the memory database, we compare the effectiveness of our knowledge distil-
407   lation techniques, with conventional demonstration-based strategies. In our approach, distillation
408   involves transferring knowledge from the memory database to SLMs through task-specific instruc-
409   tions. On the other hand, demonstration-based methods guide SLMs by presenting explicit task
410   examples to facilitate analog reasoning (Yu et al., 2024). We implemented two variants of few-
411
      shot demonstrations: Static: Human experts select three representative examples from the memory
      database, which remain constant across all cases. Dynamic RAG-based: Examples are selected
412
      from AHA memory database based on similarity to the current query. For fair comparison, we also
413
      implement the same RAG system as AHA-MI, described in Section 3.2.
414
415   Our findings indicate that few-shot demonstration generally underperforms AHA knowledge dis-
416   tillation technqiues on each dataset. However, we observe a surprisingly superior performance of
417
      the RAG-based few-shot demonstration compared to our designed knowledge distillation and other
      baselines on TAB MWP. This effectiveness appears to correlate with the complexity of the input data
418
      by further analysis. Referring to Table 2, we note that TAB MWP presents the simplest data input,
419
      containing only 2.22 columns and 6.13 rows per data point, with clean values consisting of numbers
420   or processed strings. However, when dealing with W IKI TQ, which contains irregular value types,
421   column names, and B IRD -SQL, which presents complex database schemas and values, SLMs ex-
422   hibit confusion with such heterogeneous and complex inputs. More critically, SLMs generate 38.2%
423   more invalid outputs (e.g., "SELECT \n\n\n\n...") in B IRD -SQL.
424
      Based on these observations, we conclude that dynamic few-shot demonstration is more conve-
425
      nient and effective for leveraging the memory database when the input data is less complex. On
426   the contrary, for complex data such as tables with dirty values or relational databases, our designed
427   knowledge distillation enables SLMs to better utilize knowledge and perform tasks more effectively.
428   It is worth noting that in real-world scenarios, complex data schemas and inputs are prevalent (Lee
429   et al., 2021). Moreover, our approach exhibits greater scalability as task complexity increases. Al-
430   though dynamic few-shot learning achieves a slight 3.0% advantage over our method on simpler
431   tasks, our technique outperforms it by a significant 16.2% on more complex systems. This asymme-
      try in performance gains highlights the robust generalization of our knowledge distillation approach

      Under review as a conference paper at ICLR 2025


432
      for DSCG tasks across a spectrum of input complexities, from simple to more challenging data
433   inputs.

      4     R ELATED W ORK
485   Data Science Code Generation (DSCG). DSCG focuses on automating code generation for data-
      centric tasks, requiring a deep understanding of data formats like CSV, TSV, and relational databases


                                                                                9
      Under review as a conference paper at ICLR 2025


486
      (RDB). Unlike general code generation models, which primarily generate syntactically correct code
487   in response to natural language instructions (Chen et al., 2021; Luo et al., 2024), DSCG must en-
488   sure that the generated code correctly interacts with underlying data structures. This involves un-
489   derstanding the schema, format, and semantics of the data, whether in Python code for handling
490   tabular data (Chen et al., 2024; Cheng et al., 2023; Shen et al., 2022b) or SQL for interacting with
491   relational databases (Yu et al., 2018; Lee et al., 2021; Li et al., 2024a). Spreadsheet-based code
492   generation further extends DSCG, automating the generation of formulas and operations in tools
493   (Wang et al., 2023a; Bhatia et al., 2023). Even though large language models (LLMs) have demon-
494   strated effectiveness in enhancing the capabilities of SLMs, concerns regarding data privacy in cloud
495
      environments have prompted a reevaluation of their deployment strategies.
496   Knowledge Distillation. Knowledge distillation can mitigate this problem by transferring LLM
497   capabilities to smaller models, enabling efficient deployment in resource-constrained environments
498   (Xu et al., 2024). The field has evolved from early work on softened output training (Hinton, 2015)
499
      to advanced techniques like task-specific fine-tuning (Sanh, 2019), zero-shot learning (Wang et al.,
      2023b), and instruction-following datasets (Wang et al., 2023c;b). Progressive distillation tech-
500
      niques, such as the Orca framework (Mukherjee et al., 2023), demonstrate the potential for guiding
501
      the development of efficient open-source models. Self-distillation approaches have explored au-
502   tonomous training data generation (Wang et al., 2023c). Recent advancements have focused on im-
503   proving the performance and privacy aspects of DSCG by knowledge distillation (Luo et al., 2024).
504   At the same time, synthetic data has been leveraged to enhance the generalization of SQL generation
505   across different schemas (Yang et al., 2024a). Even though these techniques are effective, most still
506   require training efforts to transfer knowledge. Our AHA framework introduces agent-based distil-
507   lation through in-context learning, eliminating the need for task-specific fine-tuning and improving
508   scalability across models and tasks.
509   Agent Memory. Agent memory can improve the capability of LLM-based agents, particularly in
510   tasks that require long-term context retention and continuous knowledge accumulation (Zhang et al.,
511   2024). Traditionally, research has focused on teaching LLMs to reflect on and evolve from mem-
512   ory built through their own interactions, limiting knowledge transfer to the model performing the
513   task (Shinn et al., 2023). For DSCG, memory plays a critical role in managing complex data for-
514   mats, maintaining long-term context, and learning from iterative analysis processes. For instance,
515
      reGAL (Stengel-Eskin et al., 2024) introduces a memory mechanism that enables LLMs to reuse
      abstractions across program synthesis tasks by storing and recalling reusable subroutines, signifi-
516
      cantly improving code generation performance. Similarly, models like MAGIC (Askari et al., 2024)
517
      have demonstrated how memory can facilitate self-correction in data analysis code generation. In
518   more complex software engineering contexts, frameworks like SWE-Agent (Yang et al., 2024b) and
519   OpenDevin (Wang et al., 2024b) by codeAct (Wang et al., 2024a) extend the use of memory by con-
520   sidering complicated contexts such as entire code repositories and prior interactions, allowing agents
521   to manage more intricate tasks like cross-file dependencies and repository-level refactoring. How-
522   ever, current agent memory systems typically rely on a single model, i.e., memory is constructed and
523   knowledge is learned and leveraged exclusively by models like GPT-4, limiting knowledge transfer.
524   Our work introduces distillation techniques that enable SLMs to leverage memory orchestrated by
525   multiple models, including more capable GPT-4o. This approach allows SLMs to utilize richer, ex-
526
      ternal knowledge for improved performance in knowledge-driven ICL, effectively bridging the gap
      of knowledge sharing between high-capacity models and more efficient SLMs.

529   5   C ONCLUSION
530
531   In this paper, we presented Agents Help Agents (AHA), an automatic framework for efficient knowl-
532   edge distillation from Large Language Models (LLMs) to Small Language Models (SLMs) in Data
533   Science Code Generation (DSCG). AHA leverages In-Context Learning to enhance SLM perfor-
534
      mance without fine-tuning, using agent orchestration and memory-based distillation to improve task
      accuracy. Evaluations on three challenging tabular data analysis datasets, which requires code gener-
535
      ation, show a 27.5% relative performance increase for Phi-3-mini and model-agnostic effectiveness,
536
      benefiting models like Llama-3.1-8B and GPT-35-Turbo even they did not participate in the orches-
537   tration. These results highlight the potential of AHA for developing intelligent applications with a
538   focus on privacy and computational efficiency.

           D ISTILLATION
867
868
      We introduce a novel fine-to-coarse knowledge distillation method employing a recursive, tree-based
      approach to generalize to unseen queries. This method improves upon traditional sequential updat-
869
      ing techniques by constructing a knowledge tree recursively and in parallel, thereby reducing bias
870
      and ensuring a more robust distillation process.
871
872   Our knowledge tree consists of nodes, each containing a batch of successful cases from the set M.
873   This structure allows for concurrent summarization of essential rules from diverse examples. We
874
      define the set of distilled instructions I as:
875
876                                             I = Lsum (M) = Tl ,
877
878   where Lsum is a recursive function executed by the Large Language Model (LLM) to distill knowl-
879   edge, and Tl is the root node of the tree. This function constructs a multi-layered tree, with each
880   layer aggregating knowledge from the preceding layer. The tree’s depth adjusts dynamically based
881   on the context length of case studies, ensuring optimal abstraction at each layer.
882   At each recursive step, nodes in the current layer l aggregate knowledge from layer l − 1:
883
884
                                                  Tl = Lagg (Tl−1 ),
885
886
      where Lagg is the aggregation function merging batches of successful cases into more abstract rep-
887
      resentations.
888
889   The leaves of the tree (layer L) contain the original cases from M, represented as batches:
890
891                                            TL = {B1 , B2 , ..., Bk }
892
893   where each batch Bj is a set of successful cases:
894
895
                                            Bj = {Mj1 , Mj2 , ..., Mjn }
896
897
      and each successful case Mji is defined as:
898
899
900                                           Mji = (qji , dji , ĉji , Sji )
901
902   Here, qji is the ith natural language query of j th batch, dji is the ith corresponding data of j th
903   batch, ĉji is the ith orchestrated correct code of j th batch, and Sji is the ith case study of j th batch
904   summarizing the solution.
905   Each higher layer in the tree abstracts and summarizes the knowledge from the level below, culmi-
906   nating in the root node Tl , which represents the final set of distilled instructions I.
907
908
      This recursive and parallel tree construction allows for simultaneous extraction of rules, significantly
      reducing dependence on the order or selection of initial examples. Each node encompasses multiple
909
      successful cases, facilitating the extraction of generalized instructions through the identification of
910
      common patterns and rules.
911
912   The process continues iteratively from the leaves to the root, resulting in comprehensive and un-
913   biased distilled instructions I. This framework provides a well-rounded guide for the SLM in
914
      generating correct code for unseen queries, effectively balancing knowledge complexity with SLM
      constraints.
915
916   Our method represents an advancement in knowledge distillation for language models, offering a
917   robust approach to extracting generalizable knowledge from diverse examples and enhancing SLM
      performance on unseen queries.

1000   The main goal of this work is to evaluate the code generation capabilities of models in understanding
1001   data schemas and structures across multiple datasets. Given the impracticality of providing all data
1002   values in real-world scenarios in which datasets may consist of millions or even billions of rows,
1003   we sample values for the part of data input to simulate realistic code generation tasks. We feed the
1004   markdown format of schemas with data samples as data_overview.
1005   For TAB MWP, we provide only the column names and the first three rows of values. This enables
1006   models to infer the data structure and value types necessary for Python Pandas code generation
1007   without exposing all the data.
1008
       For W IKI TQ, which contains more complex and varied value types, we provide the first 10 rows of
1009
       values and column names to help models navigate the dataset’s intricacies.
1010
1011   In the case of B IRD -SQL, which contains relational databases with complex schemas and diverse
1012   value types, more advanced schema-linking techniques are often required to retrieve relevant tables
1013   or columns before answering queries (Wang et al., 2020; Pourreza & Rafiei, 2024). While we
1014
       consider this advanced schema-linking process as future work for AHA, our current focus is on the
       code generation aspect. Therefore, we provide:
1015
1016
              • Ground truth retrieved tables, reducing input complexity and simulating realistic human-
1017
                machine interactions where users might supply potentially relevant tables.
1018
1019          • Full columns with column meaning description files.
1020          • The first three rows of values for each table.
1021
1022   Although the retrieved tables are given, the models must still consider constraints and generate cor-
1023   rect SQL queries. As shown in Table 3, performance on Bird-SQL remains relatively low, even with
1024   simplified table retrieval, highlighting the challenges of generating accurate SQL queries in complex
1025   database environments. This methodology allows us to evaluate code generation capabilities while

       C.3   P REVENT M ODELS F ROM D IRECT A NSWERING
1027
1028   We observe two kinds of direct answers behaviors in Python tasks, which leads to unfair evaluation
1029   of code generation ability:
1030
1031
       Unfair Data Inspection. We find that SLMs usually generate data inspection codes such as
1032
       print(df.head()) in their code generation. When producing answers given executed results,
1033   such first few rows will show again in which SLMs tend to answer it correctly even with wrong codes
1034   or bugs. We need to decouple study of tabular understanding and code generation understanding data
1035   structure. And the mutually influenced capability would be the future goals of development.
1036
1037
       Data Leakage. In datasets like W IKI TQ, current popular SLMs tend to exhibit a form of data
1038   leakage, where models effectively "memorize" the ground truth answers, resulting in unfair evalu-
1039   ations. Through a sample of 100 generated codes using the Phi-3-mini baseline across these datasets,
1040   we observed that the model often embeds the correct answer directly into the code. For instance,
1041   given the question "Who was the opponent of James V.?", the generated code might
1042   include a line like opponent = "Smith W.", which corresponds to the ground truth answer.
1043   The frequency of this leakage is particularly high in W IKI TQ, where 23 out of 100 samples exhibits
1044   this behavior, especially for simpler question types. In contrast, datasets with more complex ques-
1045   tion structures, such as TAB MWP, exhibits only 5 cases of leakage out of 100, while no instances are
1046
       found in B IR D-SQL. These findings suggest that more complex input structures and question/code
       types can effectively reduce the possibility of data leakage.
1047
1048
1049   Mitigating Direct Answering Behaviors. To address these cheating behaviors, we propose an
1050   embodied prompt, as outlined in Section H. This approach minimizes data leakage and prevents
1051   unfair data inspection during evaluation. As illustrated in Figure 7, we design scenarios where the
1052
       model is informed that it has already inspected the dataset and does not need to generate further data
       inspection commands. Additionally, the embodied prompts encourage the model to approach tasks
1053
       as a professional data analyst, preventing it from assigning variables based on memorized answers.
1054
       Our evaluation shows that this method successfully eliminated data leakage in all 100 tested cases.
1055   Also, the unfair data inspections appear less frequently. The remaining of unfair data inspections
1056   will be removed by post-processed by regex functions.
1057
1058
       We believe that this solution holds promise for addressing data leakage issues in complex benchmark
       evaluations. This is particularly important in DSCG, where datasets are difficult to collect from
1059
       expert teams and frequent re-annotation to prevent leakage is impractical.

       In conclusion, AHA proves to be an effective method for enhancing the performance of SLMs in the

1111   E     A BLATION S TUDY
1112
1113   We conducted a comprehensive ablation study of AHA-MI, as shown in Table 5. Code-T5+ is a
1114   code embedding model (Wang et al., 2023d), while BGE-Large (Xiao et al., 2024) represents one of
1115   the state-of-the-art (SOTA) text embedding models. The study examines two types of RAG Index:
1116   one where distance is computed using question embeddings alone, and another where both question
1117   and schema embeddings are used. The "Plan + Gen" approach involves first constructing a plan
1118   with distilled knowledge, followed by generation using knowledge-driven planning. In contrast, the
1119
       "Gen" approach involves direct generation without prior planning. The instruction type labeled w/
       examples refers to cases where a specific example is provided by the Teacher Agent. We evaluate
1120
       performance with 1, 3, and 5 examples to assess the impact of varying numbers of RAG examples.
1121
       The results of the ablation study reveal several key insights:
1122
1123
       Code embeddings outperform text embeddings. The superior performance of Code-T5+ over
1124
       BGE-Large-en can be attributed to the nature of the task. While text embeddings emphasize on
1125   semantic and domain knowledge, code embeddings capture the syntactic and logical structure of
1126   coding problems, which is crucial for DSCG tasks. Even when presented with identical questions,
1127   the code solutions can vary significantly depending on the data input. Code-T5+ is able to effectively
1128   embed questions from a programming perspective, benefiting from its pre-trained corpus, whereas
1129   text embeddings are less suited for the task.
1130
1131   Embedding only the question is more effective than embedding both the question and schema.
1132   The study demonstrates that question-only embeddings lead to better results. This suggests that the
1133   inclusion of schema in the embedding may introduce unnecessary complexity, which may hinder
       performance on the DSCG task.


                                                        21
       Under review as a conference paper at ICLR 2025


1134
       Planning is essential for more complex tasks. The results stress on the importance of planning in
1135   a knowledge-driven generation. For tasks requiring complex reasoning, the "Plan + Gen" approach
1136   outperforms direct generation (Gen), indicating that structured planning significantly improves task
1137   performance.
1138
1139   One example may bias the SLM. Involving a single example in the instruction can introduce
1140   bias in Sequence Learning Models (SLMs). A specific example might cause the SLM to over-
1141   follow to certain information, leading to hallucinations. For instance, if the example includes a
1142   reference to "singer", the SLM may generate plans that include "singer" even when the
1143   question pertains to an unrelated topic, such as "cars". This observation highlights the lack of
1144   robustness in SLMs when exposed to overly specific examples. Consequently, it is better to provide
1145
       more general, transferable knowledge in instructions. The degraded performance observed with
       1 RAG example supports this conclusion, as the model becomes overly reliant on the provided
1146
       information.
1147
1148
       More examples do not always improve performance. Interestingly, increasing the number of
1149   RAG examples (from 1 to 5) results in a performance drop. This suggests that longer input se-
1150   quences may confuse the SLM, making it more difficult to distill relevant knowledge. Based on
1151   these findings, we recommend using 3 RAG examples as the optimal balance for complex DSCG
1152   tasks since it avoids both the biases of a single example and the confusion caused by too many
1153   examples.

1169   We conducted an error analysis by sampling 50 incorrect cases for both AHA-MI and AHA-GI
1170
       across three datasets. Although AHA substantially improves the overall performance of SLMs, we
       found that 54% of the errors were caused by over-reasoning. This issue tends to emerge even in
1171
       relatively simple cases. As discussed earlier, SLMs can overly adhere to the instructions derived
1172
       from planning and guidance, which is problematic when the task is enough simple and does not
1173   require decomposition or reasoning. In these cases, direct code generation would lead to more
1174   accurate results. The remaining errors stem from common issues in code generation tasks, such as
1175   incorrect string handling, incorrect column selection, database constrain understanding.

       G    L IMITATIONS AND F UTURE W ORK
1179
       A key limitation of our current approach with AHA is the reliance on initial training examples for
1180
       both LLMs and SLMs to facilitate orchestration. This is why we selected datasets that include a
1181
       training corpus suitable for distilling knowledge. However, an important avenue for future work is
1182   to explore how to generate such training data in a fully zero-shot manner, without relying on human-
1183   annotated or enumerated examples. Additionally, as highlighted in the error analysis, over-reasoning
1184   negatively impacts performance on simpler tasks, where additional reasoning or decomposition is
1185   unnecessary. To address this, future work could focus on developing or prompting smaller models to
1186   act as routers, as proposed by Ding et al. (2024), to classify questions based on whether they require
1187   planning. This would help avoid over-reasoning in straightforward cases and improve the overall
       efficiency of AHA.
