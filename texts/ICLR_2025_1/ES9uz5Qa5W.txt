009                                                   A BSTRACT
010
011              LLM-powered conversational interfaces (e.g., ChatGPT, Claude, and Gemini)
012              support iterative text generation, enabling users to easily generate tailored texts (e.g.,
013              texts that should address domain-specific constraints) through a series of follow-up
014              text editing requests. However, generating such tailored texts that address the
015              user-specified constraints across multiple different contexts requires repetitive text
016              generation efforts, which is cumbersome, inefficient, and demanding. To address
017              this challenge, we introduce the concept of GPT shortcuts, which is designed to 1)
018
                 learn iterative text generation patterns from a dialogue and 2) apply these learned
                 patterns to directly generate the tailored text. GPT shortcuts generate texts that
019
                 address necessary constraints while maintaining similar structural appearance to
020
                 the target text in the dialogue, across different contexts. To assess the capability
021              of language models in generating GPT shortcuts, we present S HORTCUT B ENCH,
022              a benchmark consisting of 250 crowdsourced iterative text generation dialogues
023              across five text generation tasks. Using S HORTCUT B ENCH, we conducted an
024              analysis using six LLMs and four prompting methods, varying ways to specify
025              necessary constraints to address in the prompt. We found that 1) larger models
026              generally outperform smaller models, 2) self-explanatory constraints within the
027              target text are effective, and 3) precisely specifying necessary constraints to address
028              is critical for improving the performance.1

031   1       I NTRODUCTION
032
033   Large Language Models (LLMs) have been utilized for various types of text generation tasks (Lin
034
      et al., 2024). Users can generate texts addressing varying constraints, such as domain-specific
      requirements (e.g., writing a clinical report following specific diagnostic rules (Wang et al., 2024))
035
      and structure of the text (e.g., including “unusual observations” and “treatment plan” in the clinical
036
      report as a numbered list), depending on their context and preferences. To generate such tailored
037   texts with conversational LLMs (e.g., ChatGPT, Claude, and Gemini), users refine and revise the
038   generated texts through multiple turns until they get the desired texts (Figure 1-a). Unlike single-turn
039   generation, conversational text generation offers an easy and interactive way to create texts through
040   incremental revisions.
041
      However, iterative text generation poses significant challenges to users when repetitively generating
042
      tailored text across different contexts (e.g., writing clinical reports that address hospital-specific
043   medical guidelines in a consistent format for multiple patients). It requires a case-by-case review and
044   revision process to address them, which is cumbersome, time-consuming, and demanding. These
045   challenges can be mitigated by allowing users to directly generate target texts that address the
046   constraints specified in the iterative text generation dialogue, which we represent as a shortcut. The
047   shortcut allows users to bypass the need for repeatedly specifying the desired constraints, streamlining
048   future text generation tasks. Despite the potential, the concept of shortcut in the context of the iterative
049   text generation has not been explored yet.
050   We introduce a GPT shortcut generation task, aiming at generating a function that directly generates
051   texts resembling the target text in an iterative text generation dialogue (Figure 1-b). Specifically, the
052   generated text should address the user-specified constraints while maintaining a similar appearance

      Using S HORTCUT B ENCH, we conducted an ablation study using six off-the-shelf LLMs, GPT-{3.5-
132
      turbo, 4o-mini, 4o} and Llama3.1-{8B, 70B, 405B}. We compared four one-shot prompting methods:
133   prompting with the target text as an example output only (OneShot), with the list of user utterances
134   in the dialogue as explicit constraints in addition to the target text (+UR), with Chain-of-Thought
135   instructions for identifying necessary constraints to address in the user utterances (+EUR), and
136   with the ground-truth constraints (+GTC). Figure 2 shows an overall performance of the LLMs and
137   methods. Overall, larger models outperformed smaller models for both SBcon and SBapp , illustrating
138   the requirements of reasoning capability (Table 2). Small models, in particular, struggled to generate
139   texts with similar appearance (Figure 5). One-shot prompting was effective when the example
140   output (i.e., the target text) is self-explanatory, clearly revealing the necessary constraints. However,
141
      it struggled with complex dialogues and example outputs that lack such explanations. Providing
      explicit constraints in addition to the example significantly improved SBcon in these cases (Table 3).
142
      Moreover, specifying precise constraints (i.e., correct constraints to address) in the prompt further
143
      improved SBcon with comparable SBapp . However, the Chain-of-Thought approach (Wei et al., 2022)
144   for precisely identifying the necessary constraints from the user utterances did not improve SBcon and
145   even degraded SBapp (Table 2). To sum up, our findings suggest that LLMs can apply the iterative
146   text generation patterns to different contexts, and precisely capturing these patterns is important for
147   improving performance, especially for small models.
148
      GPT shortcuts can provide practical value by streamlining the iterative text generation process,
149
      allowing users to easily create task automation that aligns with their specific needs by simply
150   demonstrating the text generation process. This can be seen as a Programming-by-Demonstration
151   approach (Cypher & Halbert, 1993; Cambronero et al., 2023) within prompt programming (Liang
152   et al., 2024; Beurer-Kellner et al., 2023), which has been recognized as an effective approach for
153   end-users to create customized task automation (Li et al., 2017a;b). For example, by demonstrating
154   how to write clinical summaries in specific cases, medical professionals can create GPT shortcuts
155   that produce consistent summaries for different patients, addressing complex regulatory requirements
156   and including necessary medical terminologies (Van Veen et al., 2023; Lee et al., 2023). Developers
157   can leverage GPT shortcuts to produce consistent code reviews that follow the company-specific
158
      coding standards and review formats (Yang et al., 2024; Lu et al., 2023), by demonstrating the code
      review generation for specific code. We believe that our work represents a promising starting point for
159
      enabling the complex yet powerful task automation by formalizing the problem as learning iterative
160
      text generation patterns and evaluating the capabilities of LLMs.

173   2      R ELATED W ORK
174
175   Conversational LLMs. LLM-powered conversational agents have made significant success in
176   addressing diverse text generation needs (Achiam et al., 2023). Despite the ease of use and flexibility
177
      of the conversational interaction (Flohr et al., 2021), research has shown that novice users face
      challenges in creating effective prompts to generate the desired texts (Zamfirescu-Pereira et al., 2023;
178
      Kim et al., 2023; Dang et al., 2022). Iterative text generation allows users to more easily create
179
      the desired texts by a step-by-step approach, prompting LLMs to further address constraints (Wen
180   et al., 2024) in multi-turn dialogues. Despite the needs of consistently addressing the constraints
181   specified in the dialogue, applying the iterative text generation patterns in a different context remains
182   unexplored. We address the challenge by introducing the concept of GPT shortcut, which learns the
183   iterative text generation patterns and applies them to a different context.
184
      Evaluating LLM capabilities. Research has extensively evaluated the capabilities of LLMs in
185
      various task categories, including knowledge retrieval (Vu et al., 2023; Chen et al., 2024), question
186   and answering (Zhuang et al., 2023; Alonso et al., 2024), reasoning (Rein et al., 2023; Liu et al., 2024),
187   evaluation (Zheng et al., 2023; Zhou et al., 2024), and novel idea generation (Si et al., 2024; Radensky
188   et al., 2024). However, most benchmarks focus on single-turn interactions where a prompt is given
189   and LLM response is assessed. Recently, W ILD B ENCH (Lin et al., 2024) examined performance of
190   LLMs in supporting challenging tasks using real multi-turn dialogues (Zhao et al., 2024), but it is still
191   limited as a benchmark for the GPT shortcut generation task in terms of dialogue complexity (89%
192   has ≤ 2 turns) and coherency across tasks. To evaluate LLM capabilities in GPT shortcut generation
193   task through more complex dialogues, we introduce S HORTCUT B ENCH, a crowdsourced benchmark
194
      that consists of 250 iterative text generation dialogues (≥ 3 turns) across five text generation tasks.
195   Identifying input-output relations. Research has investigated methods for effectively identifying
196   the relationship between an input and output text. Existing approaches include prompting LLMs
197   to describe these relationships (Honovich et al., 2022) and optimizing prompts by sampling (Shin
198   et al., 2020; Zhou et al., 2022). The GPT shortcut generation task can be seen as identifying the
199   relationship between the input and output text by analyzing the iterative text generation dialogue,
200
      which involves more practically useful and complex patterns compared to simpler relationships
      discussed in existing benchmarks, such as negating the input sentence (Honovich et al., 2022). As the
201
      first step, we evaluate the capability of LLMs in capturing and applying the iterative text generation
202
      patterns using S HORTCUT B ENCH.

208   iterative text generation producing a target text y when given input x (d : x → y), a GPT shortcut

210   Figure 1-a) shows a dialogue d that generates target text y from the input x, and Figure 1-b)
211   demonstrates the GPT shortcut fd (x′ ) directly generating target text y ′ from given a new input x′ .
212
      GPT shortcuts enable users to perform the text generation tasks in an easy, efficient, reliable way
      while avoiding repetitive conversations in varying contexts.

231   4     S HORTCUT B ENCH
232
233   S HORTCUT B ENCH is a benchmark dataset for assessing the capability of language models in the
234   GPT shortcut generation task, which consists of 1) 250 iterative text generation dialogues across
235   5 text generation tasks, 2) a source text (for creating GPT shortcuts) and three candidate texts (for
236   testing GPT shortcuts) for each dialogue, and 3) checklists for assessing whether the generated text
237   from each of the candidate texts addresses necessary user-specified constraints in the dialogue.
238
239   4.1   DATA C OLLECTION P ROCEDURE
240
241
      Our goal was to collect iterative text generation dialogues via crowdsourcing so that the dataset
      captures realistic and diverse human intents and behavior in iterative text generation. We selected
242
      five tasks that 1) crowd workers can easily engage in without any significant barrier and 2) have
243
      been widely studied in NLP: text summarization (Nallapati et al., 2016), text simplification (Surya
244   et al., 2018), essay grading (Wang et al., 2022b), story generation (Fan et al., 2018), and QA (Fan
245   et al., 2019). For each task, we chose one dataset to select a source text (for creating GPT shortcuts)
246   and three candidate texts (for testing GPT shortcuts). Then we collected 50 iterative text generation
247   dialogues for each task via crowdsourcing on the Prolific platform. To collect multi-turn dialogues,
248   we asked each participant to improve the generated text by their preferences at least two times so that
249   the collected dialogues contain at least three user utterances. Specifically, participants were required
250   to select at least three text generation outcomes that they liked and considered to have improved over
251   the turns. More detailed task selection process, algorithms for choosing the source and candidate
252
      texts, and the data collection interface are included in Appendix A.
253   With the collected iterative text generation dialogues, we generated checklists for each dialogue,
254   which captures necessary constraints (e.g., “Include timeline on the summary.”) that the target text
255   addresses. It is important to note that not all the user-specified constraints in the dialogue have been
256   addressed in the target text. To make sure that the checklists cover the addressed constraints, we took
257   a two-step approach. First, we identified all the requested constraints. Specifically, we leveraged
258
      GPT-4o to generate all the user-specified constraints from the dialogue, similar to prior approaches to
      creating such checklists (Lin et al., 2024). Then we manually reviewed the checklists by examining
259
      their coverage and generalizability to make sure that the checklists are not too specific to the source
260
      text but applicable to other input texts (e.g., for a story generation task where the source story contains
261   DVDs, we revised a constraint “make DVDs antique” into “make a commonly used item in the past
262   become an antique”). Second, we evaluated whether the target text addresses each of the constraints
263   using GPT-4o as a judge. We conducted the evaluation ten times for each constraint and filtered
264   constraints that have been determined as addressed at least seven times.

334   5     M ETHODS FOR G ENERATING GPT S HORTCUTS
335
336   Conceptually, GPT Shortcuts are functions that produce a target text from a set of input texts.
337
      Identifying potential input texts within the iterative text generation dialogue requires analyzing the
      dialogue along with the task demonstration and the target text. In this paper, we consider a simple
338
      case where the function takes a single input text. The following methods were used to generate GPT
339
      shortcuts. See Appendix B for the prompts used for the methods.
340
341
      5.1   T EXT GENERATION METHODS
342
343   OneShot. Given the input text and target text in the iterative text generation dialogue, one-shot
344   learning can produce the desired text for another input text. This is a simple but effective approach,
345   given its remarkable performance and ability to generate texts similarly formatted to the target text.
346   To produce the desired text, LLMs are expected to address user-specified constraints that have been
347   implicitly presented in the target text.
348   OneShot+UserRequests (+UR). A limitation of OneShot is the lack of information about task
349   demonstrations. Therefore, we provide the list of user utterances in the dialogue to illustrate how
350   the target text has been generated from the input text. LLMs are expected to address effective user
351   requests, which is a subset of the list of user requests that have been addressed in the target text.
352
      OneShot+EffectiveUserRequests (+EUR). Naively putting the entire list of user requests could
353
      confuse LLMs as only the subset of user requests should be addressed. Using the Chain-of-Thought
354   approach (Wei et al., 2022), we prompted LLMs to (1) evaluate whether each user request is addressed
355   in the target text, (2) list the effective user requests, and (3) produce an output text that addresses the
356   identified effective user requests. In this way, we expect LLMs can be less confused about which user
357   requests to address.
358
      OneShot+GroundTruthConstraints (+GTC). We put the ground-truth constraints in the checklists
359
      in the prompt, expecting to yield the ceiling performance of the other methods as it clearly informs
360   correct constraints without any confusion.
361

      Including explicit constraints in the prompt significantly helps, especially for complex dialogues.
402
      OneShot achieved reasonably high SBcon score (≥ 0.9 except for GPT-3.5), which implies that LLMs
403   are able to capture the necessary constraints just with the target output as an example. As a possible
404   reason, we observed that the example output is often self-explanatory. For example, LLM responses
405   tend to contain an introductory statement that explains the text to generate, such as “Sure, here are
406   the key points you should remember for your test!”, which clearly describes the necessary constraints.
407   Also, structured texts like a numbered list with headers also specifically guide which kinds of texts to
408   generate at each point. However, OneShot significantly underperformed compared to other methods
409   when the target text did not clearly imply the necessary constraints. For instance, the gap of SBcon
410   scores between OneShot and +GTC was 4.5 times larger on average when the dialogue contained
411
      more than four constraints compared to fewer (Table 3). The performance gap was the largest in the
      story generation task, likely because the generated stories tend to be long texts consisting of multiple
412
      paragraphs without such explicit cues. This highlights the importance of comprehensively presenting
413
      the necessary constraints, especially for complex text generation tasks.
414

416   the language models, +GTC achieved higher SBcon score than both +UR and +EUR. It suggests
417

418
      overall performance degraded. Notably, GPT-3.5 achieved a significant gain on SBcon with +GTC
419

      7    C ONCLUSION AND F UTURE W ORK
481   Conclusion. We introduced the concept of GPT shortcuts, which learns iterative text generation
482   patterns from a dialogue so that users can directly generate the target text in the dialogue in different
483   contexts. Our evaluation highlights the capabilities and limitations of the language models in
484   generating GPT shortcuts. As the iterative text generation through conversational LLMs is a natural
485   user interaction, there exists diverse user needs and iterative text generation patterns. Developing
      robust methods for learning such diverse and complex patterns could serve as a foundational step

499   specialized requirements, such as technical writing, education, marketing, and law.
500   Limitations and future work. This work focused on five text generation tasks that have been actively
501   investigated in the NLP community, but considering more tasks in evaluation could inform broader
502   perspectives about the performance of GPT shortcuts. Also, we believe that considering more complex
503   iterative text generation dialogues can offer richer insights into how language models capture the
504   complex patterns, but collecting complex dialogues at scale is challenging. Designing a crowdsourcing
505   workflow for collecting complex iterative text generation dialogues could be interesting future work.
506   Starting from the crowdsourced utterances in S HORTCUT B ENCH, synthetically generating complex

      dialogues is another promising approach. Finally, future work could develop novel GPT shortcut
      generation methods by effectively capturing and applying the iterative text generation process.

778   We carefully selected five text generation tasks from the following process. We started with a list of
779   NLP task types in prior literature (Wang et al., 2022a). For each type of the tasks, we filtered tasks
780   that require open-ended text generations without a definite answer to support iterative text generation
781   scenarios. Then we further filtered tasks that crowd workers can perform iterative text generations,
782   which requires judging the text generation results and asking specific requests for improving the
783   text, without expertise. Finally, we focused on tasks where producing a long text (e.g., multiple
784
      paragraphs) makes sense. As a result, ten task categories have been identified as a suitable iterative
      text generation task to be conducted via crowdsourcing (Table 4).
785
786   For each of the meta-category, we selected one task category for the task diversity. Then we designed
787   iterative text generation tasks that crowd workers can easily engage in, using an existing dataset
788   except for the QA task. Long-form open-ended QA task is one of the most practically useful tasks
789   using LLM-powered conversational agents. Therefore, we generated questions by asking GPT-4
790
      to generate Why and How questions, two most popular questions reported in prior research (Fan
      et al., 2019).3 Using GPT-4, we prompted “Generate 10 why/how questions that spark curiosity in
791
      laypeople”.
792
793   For the evaluation, we need to prepare other input texts (candidate texts) as testcases, which are
794   different from the source text (i.e., the text used in the dialogue). For a clear evaluation, the user
795   requests (i.e., constraints) in the dialogue should be applicable to the candidate texts so that addressing
796
      the user requests for the candidate texts makes sense. To this end, we prepared three “similar” texts
      with the source text. Specifically, we randomly chose three texts from the dataset for QA (why and
797
      how question), text simplification (medical literature), and Text quality evaluation (essay group 8,
798
      randomly chosen) task as the dataset already contains texts within a certain scope. CNN/Daily Mail
799   Dataset and ROCStories datasets consist of texts in diverse topics. Therefore, we first randomly
800   sampled 2,000 texts as a pool and chose three closest texts with the source text by computing cosine
801   similarity using BERT embeddings (Devlin et al., 2018).
