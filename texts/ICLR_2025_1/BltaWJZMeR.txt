009                                              A BSTRACT
010
011
               This paper presents DataSciBench, a comprehensive benchmark for evaluating
012
               Large Language Model (LLM) capabilities in data science. Recent related bench-
013
               marks have primarily focused on single tasks, easily obtainable ground truth, and
014            straightforward evaluation metrics, which limits the scope of tasks that can be
015            evaluated. In contrast, DataSciBench is constructed based on a more com-
016            prehensive and curated collection of natural and challenging prompts. We de-
017            velop a semi-automated pipeline for generating ground truth (GT) and validating
018            evaluation metrics. This pipeline utilizes and implements an LLM-based self-
019            consistency strategy to produce accurate GT by leveraging collected prompts,
020            predefined task types, and aggregate metrics. Furthermore, it employs a care-
021            ful approach to filter a high-quality Task - Function - Code (TFC) list and assess
022
               each code execution outcome within TFC based on precisely defined metrics and
               programmatic rules. Our experimental framework involves testing 6 API-based
023
               models, 8 open-source general models, and 9 open-source code generation models
024
               using the diverse set of prompts we have gathered. Through this approach, we aim
025            to provide a more comprehensive and rigorous evaluation of LLMs in the domain
026            of data science, shedding light on their strengths and weaknesses. Experimental
027            results demonstrate that API-based models greatly outperform open-sourced mod-
028            els on all metrics except for VLM-as-a-judge and Deepseek-Coder-33B-Instruct
029            achieves the highest score among open-sourced models. We release all code and
030            data at code.

      1   I NTRODUCTION
035
036   Large language models (LLMs) (Achiam et al., 2023; Team et al., 2023; GLM et al., 2024) are
037   increasingly used in data science and scientific domains, e.g., data analysis (Hong et al., 2024),
038   protein generation (Jumper et al., 2021; Chen et al., 2024), and scientific reasoning (Zhang et al.,
039
      2024a;b). For data science tasks, LLMs offer the potential to (semi-)autonomously conduct data
      analysis (Huang et al., 2023) and data visualization (Hong et al., 2024) by calling code interpreters
040
      with corresponding Python libraries given the public known problems. These works are bench-
041
      marked on relatively straightforward tasks where ground truth labels can be precisely obtained.
042   However, much of real-world data analysis requires reasoning over more complex scenarios, such
043   as evaluating the quality of the images generated by the data visualization task. The proper evalua-
044   tion of these more complex data science tasks remains an open research direction.
045
      While there are some existing benchmarks used to evaluate LLMs for related challenges (see Ta-
046
      ble 1), those benchmarks typically focus on evaluating narrower tasks with easy-to-obtain ground
047
      truth and straightforward evaluation metrics. For example, MLAgentBench (Huang et al., 2023)
048   presents a machine learning research benchmark by building an LLM Agent pipeline. SWE-
049   Bench (Jimenez et al., 2023) benchmarks the abilities of LLM to solve real-world software issues
050   from GitHub. InfiAgent-DABench (Hu et al., 2024) completes data analysis tasks by generating la-
051   bels with GPT-4 and calculating accuracy. The frontier of LLM evaluation is towards more complex
052   real-world tasks that consist of multiple subtasks. For these challenging prompts, how to generate
053   ground truth and define specific evaluation metrics for each subtask in a comprehensive perspective
      is a question worth exploring.

066   In this paper, we introduce a new benchmark, called DataSciBench, which evaluates the data
067   science abilities of LLMs and helps LLMs improve their data analysis and data visualization abili-
068
      ties. Regarding collected prompts, their corresponding responses, and evaluation metrics, we hope
      that they meet the following characteristics: (1) Require more natural, challenging, and high-quality
069
      prompts to promote the development of LLMs’ improvement. (2) Strong correlations are neces-
070
      sary for sequential tasks so that models can be distinguished well. (3) Multiple types of results are
071   required to perform comprehensive evaluations.
072
073   To bridge the gap between task definition, evaluation criteria, and automated assessment in data
074
      science contexts, we propose a novel semi-automated framework, called Task - Function - Code
      (TFC) generation and evaluation. Specifically, from coarse-grained perspectives, we first aggregate
075
      the range of task types, functions, and corresponding codes, then, from fine-grained perspectives,
076
      we define programmatic rules for the outputs of each function depending on the specific tasks and
077   compare results with ground truth to ensure fair and consistent assessment. To validate the effec-
078   tiveness of LLMs and our proposed TFC pipeline on our collected comprehensive prompts, we
079   experiment with 6 API-based models, 8 open-sourced general models, and 9 open-sourced code
080   generation models. We observe that API-based models greatly outperform open-sourced models
081   on average. Specifically, GPT-4o surpasses all other models on all metrics except for VLM-as-a-
082   judge and Deepseek-Coder-33B-Instruct achieves the highest score among open-sourced models.
083   However, all models have significant room for improvement in following fine-grained instructions,
084   calling the appropriate tools, executing accurate plans, and exporting the required execution outputs.
085   Overall, our key contributions are as follows:
086
087   • We introduce DataSciBench, a comprehensive benchmark designed to assess the performance
088     of LLMs in data science tasks. We develop a semi-automated pipeline to generate ground truth
089
        and evaluate aggregated metrics using carefully crafted complex questions.
090   • We propose a Task-Function-Code (TFC) list based on predefined aggregated metrics and pro-
091     grammatic rules. We then assess 23 large language models from both coarse-grained and fine-
092     grained perspectives, presenting the results in Table 2.
093   • Our study includes various analyses such as comparisons and correlations with existing bench-
094     marks, presented in Figure 5 and Table 3. Furthermore, we offer research insights derived from
095     experimental outcomes of the evaluated LLMs that point to interesting directions for future work.

146   3     DATA S CI B ENCH
147
148   DataSciBench consists of three important components as outlined in Figure 1.
149
150   • Prompt Definition and Collection which defines 6 task types and collects 222 real, challenging,
151     and high-quality prompts through question filtering and expert review.
152
      • Response Integration and Validation which proposes novel Task - Function - Code (TFC) to
153     assess the key tasks for each prompt and defines the aggregate functions and programmatic rules
154     to effectively evaluate the specific task description and compare prediction with ground truth.
155
      • LLM Evaluation which assesses 6 API-based models, 8 open-sourced general models, and 9
156
        open-sourced code generation models from coarse-grained (e.g., success rate, completion rate)
157
        and fine-grained (e.g., VLM-as-a-judge, aggregate functions) perspectives.

      3.1     P ROMPT D EFINITION FOR DATA S CIENCE
161
      Task Type. We define six typical data science tasks as follows:


                                                                            3

162
      1. Data cleaning and preprocessing. This task detects and processes missing values, outliers, and
163      duplicate data; and standardizes data formats, such as a uniform format for dates and times.
164
165
      2. Data exploration and statistics understand. This task calculates basic statistical indicators of
         data (mean, median, standard deviation, etc.), generates data distribution charts (histograms, box
166
         plots, etc.), calculates correlations between variables, and draws correlation matrices or maps.
167
168   3. Data visualization. The goal of this task is to visualize and analyze data and create interactive
169      charts so users can freely explore the data.
170   4. Predictive modeling. The task aims to select the appropriate machine learning algorithm, such as
171      linear regression, decision tree, random forest, etc.; carry out feature engineering, such as feature
172      selection, feature transformation, feature combination, etc.; the data set is divided into the training
173      set and test set, and the model is trained and evaluated; and select the corresponding evaluation
174      indicators for different prediction problems, such as classification, regression or clustering.
175   5. Data mining and Pattern recognition. This task uses association rule mining, frequent item
176      set mining, and other methods to find interesting patterns in the data; Text mining technology is
177      used to extract keywords, topics, and other information from text data; and apply cluster analysis,
178      classification algorithms, etc. to identify underlying patterns and structures. Pattern recognition
179      tasks can conduct these functions: image recognition, text clustering, and time series detection.
180   6. Interpretability and Report generation. This task aims to provide explanations of model re-
181      sults, such as feature importance, model parameters, etc., and automatically generate reports and
182      summaries that present the results of the analysis in a way that is easy to understand and share.
183
184   Task Integration. To increase the difficulty of assessing the prompt, we chose more complex
185   prompts that included multiple tasks. These sequential tasks can be any combination of tasks.
186
187   3.2      DATASET C OLLECTION
188
189
      Question Collection. We collect questions from four sources:
190
191   • Coarse-grained collection from a real-world online platform. We collect natural prompts from one
192     online code-generation platform, CodeGeeX (Zheng et al., 2023).
193
      • Extracted and rewritten from a public code benchmark. We select data science-related and high-
194     quality prompts from BigCodeBench and then rewrite them to unified instructions.
195
196
      • Hand-Written by humans. We also write elaborated prompts to increase the difficulty and robust-
        ness of the evaluated benchmark by referring to relative websites1 .
197
198   • Synthesized from LLMs. We use a few-shot examples drawn from human-written prompts to ask
199     LLM to generate similar prompts.
200
201   Question Filtering. We filter low-quality questions via the following three principles: (1) Choose
202
      questions that keywords include, but are not limited to, “machine learning”, “deep learning”, “data
      preprocessing”, and “data visualization”. (2) Filter questions that require updating code, finding
203
      errors, and explaining concepts. (3) Rewrite questions that align with human preferences and LLMs,
204
      which refers to questions solvable by both humans and large language models, avoiding overly
205   specialized or ambiguous queries.
206
207   Expert Review. To ensure the quality of the collected prompts, we review them by experts who are
208   professionals in computer science and data analysis. The review process includes three stages: (1)
209   In stage 1, experts verify the correctness and adjust the suitability of prompts. In addition, experts
210   ensure the answers to the prompts are easy to evaluate. For example, handing missing values for
211   a data frame. (2) In stage 2, experts format all prompts into unified instructions and the format
212
      encompasses input data, input file, prompt, and expected output file. (3) In stage 3, experts ensure
      the availability of datasets of input prompts, including generating random datasets and collecting the
213
      public datasets.

241   3.3   R ESPONSE I NTEGRATION AND VALIDATION
242
243   Ground Truth Generation and Validation. To obtain the response to collected prompts, we
244   propose the following strategy to generate a test case of each prompt. We first obtain the outputs
245   of each prompt by sampling LLMs several times and the final output by executing the generated
246
      code. We use two different validation methods to ensure the rationality and reliability of the answer
      generated by LLMs. With regard to prompts originating from BigCodeBench where reliable test
247
      cases are provided, we validate the generated answer by performing all test cases. Answers that pass
248
      all test cases are rechecked by humans and finally considered as ground truth. As for other prompts,
249   we initially adopt a self-consistency strategy (Wang et al., 2022) to obtain generated codes, and then
250   manually validate these results manually through cross-verification by multiple authors to ensure
251   accuracy and reliability.
252
253   Evaluation Selection. We introduce a structured approach to identify and evaluate key tasks
254   across six established types. We first use GPT-4o-mini to select several valuable task types, return
255
      corresponding evaluation functions, and generate the evaluation codes for each prompt to effectively
      evaluate the capabilities of LLMs and reduce the evaluation cost. Each group data is simplified as a
256
      tuple (T, F, C) in generated R as follows:
257
258                                          R = {(Ti , Fi , Ci )|N
                                                                  i },                                   (1)
259
260   where N is the number of valuable task types per prompt, and this value is different for each ques-
261   tion. Then we conduct a data interpreter (DI) (Hong et al., 2024) to generate a directed acyclic graph
262
      (DAG) in a hierarchical structure for each prompt, in which each task type is defined as a node at
      one level in a DAG. Based on the generated graphs, we take a powerful LLM as a backbone and run
263
      all evaluation functions to obtain the ground truth of each task type. To some extent, this way of
264
      verification can avoid the commonly used LLM-as-a-Judge black-box assessment.
265
266   Function Aggregation. To unify the key functions and improve the scalability of the evaluation,
267   we select top-K functions for each task type and aggregate all generated functions to the top-K
268   function category, as shown in Figure 2. Generally, the K is set as 5. For instance, the function
269   category for data cleaning and preprocessing includes Data Cleaning Completeness, DataFrame
      Shape Validation, Data Completeness, Normalization Range Check, and Data Quality Score.

270
      Programmatic Rules. Regarding aggregate functions with corresponding codes, we define uni-
271   fied rules to validate generated code. Specifically, we unify all outputs as boolean or decimal types
272   ranging between 0 and 1. Then, we obtain the final value by comparing ground truth with prediction
273   output depending on the specific task description of aggregate functions. For example, regarding
274   Data Cleaning Completeness, which calculates the final number of rows/columns after preprocess-
275   ing, the final output is 1 if the number is the same as the number of ground truths otherwise 0. For
276   some specific tasks whose output type is decimal, we also set a corresponding threshold to transform
277   the output to boolean for simplicity, such as, the threshold being set to 0.5 if the aggregate function
278   is silhouette score for data mining and pattern recognition.
279
      Summary. Based on the abovementioned processes, we obtain 222 effective prompts and corre-
280
      sponding test cases, which help the following evaluations of API-based and open-sourced models.

305   missing (score of 0), fail (score of 0), success-non-compliant (score of 1), and success-compliant

      Fine-grained Aggregate Metrics.                We also define the fine-grained aggregate metrics for detail
347   evaluating all LLMs.
348
349   • Vision-language model (VLM)-as-a-judge assesses the overall score of two inputs based on pre-
350     defined criteria (Appendix A.5), providing a step-by-step rationale for its evaluation.
351
      • Data Quality Score (F1) in Data cleaning and preprocessing aims to assess the cleanliness of data
352
        post-preprocessing. It yields a boolean output of 1 if it matches the ground truth, or 0 otherwise.
353
354   • Plot Validity (F2) in Data visualization pertains to the accuracy of visual representations, such
355
        as checking whether the shape of the association matrix is consistent with the ground truth. If
        consistent, then the final value is 1, otherwise 0.
356
357   • Data Accuracy (F3) in Data Exploration and Statistics Understand focuses on understanding data
358     quality and can be quantified using Mean Squared Error (MSE). The final value is derived by
359     comparing it against the ground truth with a predefined threshold.
360   • Visualization Completeness (F4) in Data visualization evaluates the comprehensiveness of gener-
361     ated images (e.g., PNG, jpeg, PDF) by checking their existence compared to the ground truth. A
362     score of 1 is assigned if the files exist, and 0 otherwise.
363   • Model Accuracy (F5) in Predictive modeling is utilized to gauge the predictive performance of
364     models, providing a boolean accuracy value or decimal ranging between 0 and 1.

477
478
      With curated metrics, we are able to obtain deeper insights into LLMs’ ability to plan and execute
      complex data science tasks. The experiment results also raise questions that are worth exploring
479
      since some results do not conform with conventional perceptions.
480
481   Models excel at reasoning but do not necessarily perform better on complex data science tasks.
482   Although it’s true that data science coding tasks often involve scheduling and step-by-step execu-
483   tion similar to reasoning scenarios, results show that even the LLMs proficient in reasoning tasks
484   can still fail to complete complex data science tasks. For instance, the OpenAI’s o1-mini model,
485   which is commonly regarded as one of the best reasoning models, unexpectedly failed on many
      of DataSciBench’s tasks. The model only achieves a 29.77% overall success rate, significantly

      lagging behind the company’s previously introduced models like GPT-4o and GPT-4-Turbo. Af-
487   ter examining the completions generated by o1-mini, we discovered that the failures are primarily
488   caused by non-compliance with instructions, incorrect calls, and forgetfulness. While successfully
489   splitting the task into multiple subtasks, the model often forgets to export required execution out-
490   comes or just outputs undesired data. In other cases, the model may falsely call a library function or
491   method that sometimes does not even exist. These facts remind us that real-life data science coding
492   tasks often comprehensively challenge the model’s ability to follow fine-grained instructions, utilize
493   existing tools (libraries, APIs...), and do planning. To perform and align well on these tasks, a model
494   has to be competitive on all related aspects.

      Large scale models sometimes may fail to follow simple instructions more frequently.

      6     R ELATED W ORKS
505
506   6.1   LLM S FOR DATA S CIENCE
507
      With the popularity of large-scale language models, researchers have developed a series of LLM-
508
      based agents for data science. Specifically, SheetCopilot (Li et al., 2024) designs a tabular agent,
509
      which directly processes natural language-described tasks, and generates and executes a series of
510   operation plans on datasheets to produce the desired results. Data Copilot (Zhang et al., 2024d) is an
511   intelligent agent that serves as a bridge between users and data, which automatically executes data
512   processing, prediction, and visualization tasks based on users’ data needs. InsightPilot (Ma et al.,
513   2023) focuses on exploratory data analysis and can automatically discover data insights related to
514   fuzzy questions raised by users. Data interpreter (Hong et al., 2024) augments problem-solving in
515   data science with dynamic planning with hierarchical graph structures, tool integration, and logical
516   inconsistency identification in feedback. The correctness of data analysis in data science has a
517   significant impact on decision-making. Therefore, with the continuous increase of data science
518
      agents, it is urgent to conduct a comprehensive and in-depth evaluation of data science agents.
519
520   6.2   LLM AGENT E VALUATION B ENCHMARKS FOR DATA S CIENCE
521
522   Assessing the effectiveness of LLMs in handling diverse and challenging data science prompts
523
      is essential to push the boundaries of benchmarking practices in the field of data science and
      LLM research. Data science agents often solve problems by generating code, so the capabili-
524
      ties of data science agents are closely related to the code generation capabilities of large models.
525
      There are already many benchmarks for evaluating the code capability of large models. MLAgent-
526   Bench (Huang et al., 2023) benchmarks the LLMs’ abilities on traditional machine learning tasks.
527   NaturalCodeBench (Zhang et al., 2024c) evaluates the capabilities of code generation models on the
528   real prompts from the CodeGeeX (Zheng et al., 2023) platform. However, the general code evalu-
529   ation benchmark ignores the characteristics of data science tasks and cannot comprehensively and
530   effectively evaluate the capabilities of large models in data science.
531
532   7     C ONCLUSION
533
534   This paper introduces DataSciBench, a novel framework tailored to assess the capabilities of
535   Large Language Models (LLMs) in data science tasks. By meticulously curating challenging
536   prompts and leveraging robust LLMs alongside a self-consistency strategy, we generate ground truth
537   for each prompt. To comprehensively evaluate LLM performance, we aggregate evaluation metrics
538   and synthesize the Task-Function-Code (TFC) list programmatically. Subsequently, we evaluate 23
539   API-based and open-source models, offer valuable research and engineering insights, and present
      error analyses of the assessed LLMs.

651
652   In certain visualization tasks, our initial metrics and evaluation methods (e.g., VLM-as-a-judge)

654   effectively. One potential approach could involve employing Vision Language Models (VLMs) to

      A.2   C OMPARISON WITH EXISTING BENCHMARKS
658
659   While DataSciBench does show a correlation with LCB or BCB in Section 5.3, our benchmark
660   offers several unique and important contributions:
661
662   • Domain-Specific Focus: DataSciBench specifically targets data science and analytics tasks.
663
        However, existing benchmarks primarily focus on general programming problems. This special-
        ization helps evaluate models’ capabilities in handling real-world data analysis scenarios.
664
665   • Task Diversity: Our benchmark includes unique task types like data preprocessing, visualization,
666     and statistical analysis. These tasks are underrepresented in current benchmarks. This provides
667     deeper insights into models’ data science-specific capabilities.
668   • Complementary Insights: While overall correlations exist, we observe meaningful differences
669     in model rankings. For example, models like Meta-Llama-3-8B-Instruct and CodeLlama-34B-
670     Instruct show distinct performance patterns. These differences highlight capabilities specific to
671     data science tasks that other benchmarks may not capture.
672
      The correlation with existing benchmarks validates our evaluation methodology, while our domain-
673
      specific focus provides valuable new insights for assessing AI models in data science applications.
674
675
676   A.3   M OTIVETION AND EXAMPLE OF TASK -F UNCTION -C ODE (TFC)
677
      The TFC framework was developed to address several critical challenges in automated evaluation of
678
      data science tasks:
679
680   • Systematic Task Selection: TFC provides a structured approach to identify and categorize key
681     tasks across six established types. This systematic organization ensures comprehensive coverage
682     of essential data science operations and helps maintain evaluation consistency and completeness.
683   • Standardized Evaluation Metrics: Data science tasks often lack standardized evaluation criteria.
684     TFC addresses this by explicitly defining appropriate evaluation functions for each task. For
685     example, data preprocessing tasks require specific metrics that differ from visualization tasks.
686     This standardization ensures fair and consistent assessment.
687   • Automated Execution Framework: TFC includes executable code components for both tasks
688     and evaluation metrics. This automation significantly improves evaluation efficiency, result repro-
689     ducibility, and testing scalability.
690
      • Ground Truth Generation: TFC serves as a crucial foundation for establishing ground truth,
691
        particularly valuable for complex tasks where ground truth is not readily available, and enables
692     systematic verification and validation of model outputs.
693
694   Overall, the TFC structure represents a novel contribution by providing a comprehensive framework
695   that bridges the gap between task definition, evaluation criteria, and automated assessment in data
696   science contexts.
