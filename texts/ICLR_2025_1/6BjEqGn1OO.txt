009                                                A BSTRACT
010
011            Chatbots built upon language models have exploded in popularity, but they have
012            largely been limited to synchronous, turn-by-turn dialogues. In this paper we
013
               present a simple yet general method to simulate real-time interactive conversations
               using pretrained text-only language models, by modeling timed diarized transcripts
014
               and decoding them with causal rejection sampling. We demonstrate the promise
015
               of this method with two case studies: instant messenger dialogues and spoken
016            conversations, which require generation at about 30 tok/s and 20 tok/s respectively
017            to maintain real-time interactivity. These capabilities can be added into language
018            models using relatively little data and run on commodity hardware.
019
020
021   1    I NTRODUCTION
022
023
      Chatbots built upon language models have exploded in popularity, but their interaction model is
024   extremely limited: the user and the system take turns writing messages, where the system waits until
025   the user finishes their message to respond then responds instantly and uninterruptibly. Extensions to
026   support audio have used speech to text and text to speech to eliminate the need for typing and reading
027   the screen (OpenAI, 2023), but the constraints of the interaction model have remained the same.
028
      In this paper we present a simple method to simulate real-time interactive conversations using
029
      pretrained text-only language models. Namely: model timed diarized transcripts—i.e., sequences
030   of [timestamp, speaker id, message]—at the desired granularity, and then decode these transcripts
031   with causal rejection sampling—i.e., sample a continuation that will be finalized at the predicted
032   timestamp, and if there is intervening user input before the timestamp, reject the planned continuation
033   (to the extent that its probability under the model has changed) and resample a new one. This method
034   is naturally sparse over time and number of speakers, scaling computation with the amount of content
035   being actively produced at each moment. It is also quite general; in principle, it can also be applied to
036   any task involving timed sequences of events, from time series forecasting to applications in gaming.
037   We demonstrate the promise of this method with case studies in two domains. First, we use the
038   instant messenger chat history between the first authors to train a real-time interactive asynchronous
039   text dialogue model. Second, we use public speech datasets with diarized transcripts to train a
040   real-time spoken conversation model, cascaded through word-level speech to text and text to speech
041   models. Here there is an additional complication in that real-time streaming speech to text systems
042   are unstable, i.e., predictions may change in light of future context. We address this with retconning,
043   i.e., revising the user’s input history but keeping any already finalized system outputs.
044   We evaluate these embodiments of our method with respect to performance (properties of the control
045   token format and of our proof-of-concept implementation) and quality (test perplexity, offline human
046   ratings, and online human ratings)—across finetuned models from 160M to 12B parameters. For the
047   offline human rating setting only, we also use long in-context learning to test larger pretrained models
048   available by API. In order to maintain real-time interactivity, generation needs to be about 28 tokens
049   per second for the instant messenger use case and 22 tok/s for spoken conversations, which are easy
050
      to achieve on a single A100 at our model scales. We find that, predictably, better pretrained models
      lead to better results, though there is still obvious room for improvement with dataset/model scale.
051
052   We publicly release our code (and some demo videos) at this link. We hope that these proofs of
053   concept spark the imagination and show that language models can easily be adapted to new real-time
      interaction modes.

      2     M ETHOD

      We model timed diarized transcripts using causally masked (decoder-only) language models. Given
057
      a sequence of events ei , where each event ei consists of a timestamp ti (timed), a speaker id si
058   (diarized), and a message mi (transcript), we model p(ei |e1 , ..., ei−1 ). In practice, this function
059   decomposes into p(ti |e1 , ..., ei−1 ), p(si |e1 , ..., ei−1 , ti ), and p(mi |e1 , ..., ei−1 , ti , si ), or even more
060   granular distributions if these components are represented as multiple tokens. By modeling events
061   sparsely over time, we are able to sample transcripts with computation proportional to the num-
062   ber/complexity of the events, rather than the time duration.
063
      In order to make this model interactive, we use causal rejection sampling. We pick a particular
064   speaker id S to represent the user and sample candidates êi ∼ p(ei |e1 , ..., ei−1 ), where we interpret
065   the timestamps t within these events with respect to the current real time. If an input from the user
066   (S, T, M ) interrupts before the timestamp t̂i is reached, we reject the candidate êi and sample a new
067   candidate êi+1 ∼ p(ei+1 |e1 , ..., ei−1 , ei = (S, T, M )). If no such interruption occurs before t̂i , there
068   are two possibilities: If the speaker id ŝi within êi is not S, we accept the message candidate m̂i ,
069   emit it to the user, then sample êi+1 , etc. If ŝi is S, then we resample ê′i ∼ p(ei |e1 , ..., ei−1 , ti ≥ t̂i ).
070
      Because it takes some amount of time tlatency (varying with message length) to execute the model
071
      and sample from p(ei |...), if the user repeatedly provides input less than tlatency before the predicted
072
      timestamps t̂i , the model will be starved and unable to generate any acceptable events. We provide
073   two modifications to mitigate recomputation from user interruption:
074
075   First, we enforce a hard lower bound on the model’s generation bandwidth by stipulating that if the
076
      user input comes within treact of t̂i , we accept êi as a candidate for êi+1 . The relationship between
      tlatency and treact determines whether the model can maintain real-time interactivity in the worst
077
      case. We do not expect moderate treact to harm generation quality too much because a human
078
      reaction time of approximately 150-200 ms (Thompson et al.; Jain et al.) should be reflected in the
079   underlying causal structure of human training data.
080
081   Second, we reduce the average amount of recomputation by integrating speculative decod-
082
      ing (Leviathan et al., 2023; Chen et al., 2023). Rather than discard the candidate êi unconditionally
      upon user interruption, we treat it as a draft for the new generation, rejecting and resampling based
083
      on the closeness of p(ei = êi |e1 , ..., ei−1 , ti ≥ T ) and p(ei+1 = êi |e1 , ..., ei−1 , ei = (T, S, M )).
084
      Note that this is different from traditional speculative decoding, where a smaller model for the
085   same distribution drafts a candidate;1 the use of different prompts under the same model resembles
086   classifier-free guidance (Ho & Salimans, 2022; Sanchez et al., 2023). Like with treact , we expect this
087   to work to the extent that there is a looseness in the causal dependencies of nearby messages from
088   different parties.2
089
      See Algorithm 1 for a formal description of causal rejection sampling (speculative decoding omitted
090
      for clarity; see Appendix A for the full version), or see our code at this link.
091

093   instant messenger dialogues and spoken conversations.

145   2.2   S POKEN C ONVERSATIONS
146
147   We also apply our general method to timed diarized word-level automatic speech recognition (ASR)
148   transcripts. By cascading input through speech-to-text and output through text-to-speech, we can
149   simulate spoken conversations. Note that—like cascaded approaches in general—this has the obvious
150   limitation that it bottlenecks the input and output through text, stripping away aspects of speech
151   like tone and introducing errors from intermediate models. While there exist off-the-shelf streaming
152
      speech-to-text models that output word-level timestamps, we are not aware of any text-to-speech mod-
      els (streaming or otherwise) that accept them as input: the closest is incremental text-to-speech (Ma
153
      et al., 2020a). This limits our ability to generate natural-sounding speech; we use word-level text to
154
      speech invoked at the specified timestamps and consider this out of scope.
155
156   There is an additional complication due to the use of streaming speech-to-text models: these models
157   are able to achieve low latency because they output preliminary transcriptions that may change in light
158
      of future input and are only finalized some time later. This means that not only can the user’s input
      interrupt the model’s candidate generation, but the input can retroactively change after a candidate
159
      has been generated, accepted, and spoken out.
160

      4     R ELATED W ORK
419
420
      We survey related work in three areas: text dialogues, spoken dialogues, and use of language models
421
      to model time broadly.
422
423
      4.1     T EXT D IALOGUE M ODELING
424
425
      Modeling text dialogues is perhaps the founding problem of artificial intelligence: Turing’s im-
426   itation game poses the challenge of distinguishing man from machine through turn-by-turn text
427   dialogue (Turing, 1950). While timing is mentioned here (a model that responds too quickly could be
428   distinguished from a human), the interaction model is limited. Since then there has been a wealth
429   of work on dialogue systems (Ni et al., 2022), initially with complex rule-based methods (Weizen-
430   baum, 1966) but shifting over time towards unified deep learning methods, culminating in Meena
431   & LaMDA (Adiwardana et al., 2020; Thoppilan et al., 2022), the Blenderbot series (Roller et al.,
      2020; Komeili et al., 2021; Shuster et al., 2022), and of course the recent wave of chatbots such as


                                                              8
      Under review as a conference paper at ICLR 2025


432
      ChatGPT (Schulman et al., 2022), Gemini (Google, 2024), Copilot (Microsoft), Claude (Anthropic,
433   2023), Pi (Inflection), Coral (Cohere), HuggingChat (HuggingFace), etc. These chatbot works have
434   primarily focused on basic, goal-directed conversational capabilities in the desired domains, which
435   until recently has been very challenging, and less on the interaction model. Replika (Replika) and
436   certain modes in Character.AI (character.ai) do allow multiple messages per conversation turn, but
437   with undisclosed methods and unclear limitations.
438
      CICERO (Bakhtin et al., 2022) studies Diplomacy, a political strategy game that involves instant
439   messaging with other players in real time. The primary focus is on using dialogue paired with actions
440   to achieve certain goals in the game, which implies the ability to imitate natural timing to avoid raising
441   suspicion with human players. CICERO uses a chain of encoder-decoder models and heuristics to
442   perform tasks such as predicting the next message time vs. content independently, and not all context
443   is available to all models. Messages are rejected/resampled when user input causally intervenes on
444   planned messages. Our work uses a simpler approach with a single transcript in a decoder-only
445   model, which minimizes recomputation and makes all information available for all decisions; we
446   further improve performance by using a reaction time window and causal speculative decoding.
447   The task of imitating specific people based on their digital footprint (for better or worse) has captured
448   the popular imagination, featuring in shows like Silicon Valley, Black Mirror and Westworld and
449   described with names like generative clones or ghosts in academic literature (Morris & Brubaker,
450   2024). Blog posts about finetuning LMs on personal chat histories are relatively common, but they
451   either model timed transcipts noninteractively, or synchronous turn by turn conversations interactively
452   (as a traditional chatbot). We are not aware of prior work that turns models of timed transcripts into
453
      interactive applications.
454
455   4.2   S POKEN D IALOGUE M ODELING
456
      To go beyond manually crafted turn-taking heuristics for what is in generality an extremely complex
457
      task (Skantze, 2021), the main approach for generating spoken conversations has been direct audio
458
      modeling. dGSLM (Nguyen et al., 2022), AudioLM (Borsos et al., 2023), and SpiRit-LM (Nguyen
459   et al., 2024) do this by modeling learned discrete tokens with autoregressive language models; the
460   former models two streams of audio (dialogues), while the latter two model one. While the token
461   modeling is causal, the tokenization is not, so these methods do not directly work for streaming
462   generation. In concurrent work, GPT-4o (OpenAI, 2024) offers an “Advanced Voice” mode, but it
463   does not offer full interactivity (e.g. while users can interrupt the model, it cannot interrupt users) and
464   relies on undisclosed methods.
465   Discrete audio tokenization is generally performed at a fixed rate of ∼40-50 tok/s for a single
466   audio stream, vs. ∼20 tok/s for our approach supporting arbitrary numbers of speakers.5 This fits
467   into the general pattern of cascaded vs. end-to-end models: cascaded models are generally more
468   performant/require less data and therefore can be developed sooner using fewer resources, but they are
469   eventually superseded by end-to-end models which can provide the optimal quality given sufficient
470   resources.
471   Though not exactly dialogue, simultaneous translation often operates through a cascade of ASR
472   and TTS, though timing information (besides the relative ordering of words in the source and target
473   streams) is stripped away (Ren et al., 2020; Ma et al., 2020b).
474
475   4.3   T IME -AWARE L ANGUAGE M ODELS
476
477   There are many works that make language models aware of time in one sense or another. Even
478   without special effort, language models learn latent representations of time to the extent that it helps
479   explain the training distribution (Gurnee & Tegmark, 2024). The language model CTRL (Keskar
480   et al., 2019) is conditioned on metadata about each document, which may include the publication
481
      date. Whisper (Radford et al., 2022a) and some other speech-to-text models predict timestamps as
      text. Park et al. (2023) lets loose generative agents in a virtual town environment, where they act on

484   sparse indexing over time may be higher than dense tokenization without indexing. And because our approach is
485   sparse over time, it is more difficult to batch and has inconsistent load, which may be disadvantageous for bulk
      serving.

487   for time series forecasting, whether pretrained (Das et al., 2024), finetuned (Jin et al., 2024), or
488   zero-shot (Gruver et al., 2023), though here time is usually dense (proceeds at a fixed rate). We are

      5    C ONCLUSION
494
495   In this paper, we presented a simple yet general method for simulating real-time interactive con-
496
      versations using pretrained language models—modeling timed diarized transcripts and decoding
      with causal rejection sampling—situated in two use cases: instant messenger dialogues and spoken
497
      conversations. It is easy to imagine extensions such as multiple simultaneous conversations with one
498
      simulated individual (by adding conversation ids in addition to speaker ids) or modeling multimodal
499   conversations (images, actions, etc.), though this may require more capable language models. While
500   we demonstrated the promise of this method using interactive conversations, it can be applied to
501   turn language models into interactive models for any kind of event sequence, i.e., sparse-over-time
502   world models. We hope that this method will facilitate more flexible interaction with the under-
503   lying capabilities of language models and enable new applications in fields such as gaming and
504   entertainment.
505
506
      E THICAL C ONSIDERATIONS
507
508
      While work improving the ability to simulate real-time interactive conversations can make language
509
      models more useful or delightful, it also poses risks for fraud and manipulation. In order to mitigate
510   these risks, we limit our work to simulating natural conversations in text, a medium which is perceived
511   as less trustworthy than audio or video. (While we simulate the timing aspects of spoken conversation,
512   our generations are still easily distinguished from real speech.) We provide only proofs of concept
513   with small datasets, and do not scale up to sizes where these capabilities would become more refined.
514   We also do not study goal-directed methods which could be used to steer a model to execute fraud.
515
      We believe that it is valuable to expose this capability overhang so that the community can respond
516   with appropriate measures. For example, a better understanding of the amount of data needed to
517   impersonate someone with a generative clone could affect how much conversational data users
518   are comfortable sharing publicly on social media, or motivate end-to-end encryption/disappearing
519   messages to prevent private data leakage in the event of hacking. Developing interfaces for language
520   models that are not immediately distinguishable from humans could also help to evaluate extreme
521   risks like deception and persuasion in frontier models (Shevlane et al., 2023), to the extent that people
522   react differently to communication that they perceive to be from a model vs. another person. Bad
523   actors are already capable of sophisticated deepfake scams and aren’t exactly forthcoming about their
524
      methods.
525   There are also ethical considerations when simulating real people or fictional characters absent ill
526   intent, such as privacy and the effects of parasocial relationships; these tend to be general concerns
527   that are not strictly related to real-time interactivity. See Morris & Brubaker (2024) for an in-depth
528   discussion of these factors. In terms of the specific datasets we used in this paper: We used our own
529   instant messenger history with the consent and active involvement of both participants, and do not
530
      release the data/model for privacy reasons. The U.S. Supreme Court’s oral arguments are inherently
      public and the conversation is in a specialized legal domain rather than anything that would encourage
531
      parasocial relationships. We model only text transcripts and use generic text to speech voices (i.e.,
532
      we do not contribute methods to impersonate any of the speakers).

      In Algorithm 2, we present a description of causal rejection sampling including speculative decoding.
868
      For simplicity, we describe the speculative rejection sampling as if it rejects or accepts an entire event,
869
      but in implementations where events are composed of multiple tokens, the acceptance/rejection acts
870   in finer granularity on tokens (so a prefix in a speculated event can be accepted, and only the rest has
871   to be resampled).
872
873
      Note that in order to maintain the validity of the rejection sampling, we must condition the draft
      distribution on ti ≥ T , because if t̂i had been < T it would have already been finalized and we would
874
      not be considering it for rejection sampling. Renormalizing this correctly in timestamps consisting of
875
      multiple tokens requires some finesse. For our instant messenger case, this is further complicated
876   by the fact that as interruptions come in, the same timestamp in a draft message may change format.
877   For example, after a message planned for :02;17.8, a generation may plan for :03;52.0, but
878   after an interruption at :03;24.7, the draft must be reinterpreted to have a timestamp of ;52.0).
879   We expect that speculation is not worth the implementation burden unless you are using a simple
880   custom vocabulary for timestamps or pressing up against performance limits. In our experiments
881   performance was adequate without it.
