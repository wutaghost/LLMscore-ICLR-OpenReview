009                                              A BSTRACT
010
011            Large language models (LLMs) offer a promising way forward for automating
012            software engineering tasks, such as bug fixes, feature additions, etc., via multi-
013            step LLM-based agentic workflows. However, existing metrics for evaluating such
014            workflows, mainly build status and occasionally log analysis, are too sparse and
015            limited in providing the information needed to assess the quality of changes made.
016
               In this work, we designed LLM-based critics to derive well-structured and rigor-
               ous intermediate/step-level, execution-free evaluation proxies for repo-level code
017
               changes. Importantly, we assume access to the gold patch for the problem (i.e.,
018
               reference-aware) to assess both semantics and executability of generated patches.
019            With the gold test patch as a reference, we predict executability of all editing lo-
020            cations with an accuracy of 91.6%, aggregating which, we can predict the build
021            status in 82.1% of the instances in SWE-bench. In particular, such an execution-
022            focused LLM critic outperforms other reference-free and reference-aware LLM
023            critics by 38.9% to 72.5%. Moreover, we demonstrate the usefulness of such a
024            reference-aware framework in comparing patches generated by different agentic
025            workflows. Finally, we open-source the library developed for this project, which
026            allow further usage for either other agentic workflows or other benchmarks.

028   1   I NTRODUCTION
029
030   Source code within repositories is typically complex and inter-dependent. As a result, code changes
031   (referred to as edits) are often spread across multiple functions, classes, and/or files. This makes
032   day-to-day software engineering (SE) tasks – such as fixing bugs or adding new features – onerous
033   for developers. The inter-dependence can also introduce different forms of syntactic, semantic, or
034   logical errors, which may impact multiple locations. Thus, developers often find themselves in
035   iterative editing cycles, where they must repeatedly build, identify and fix failures.
036   Recent improvements in large language models (LLMs) (OpenAI, 2023; Anthropic, 2024; Touvron
037   et al., 2023) has prompted the use of LLM agents – systems capable of interacting with its envi-
038   ronment to make rational decisions – for automating these complex SE tasks through multi-step
039   processes (Zhang et al., 2024; Wang et al., 2024). We refer to such an orchestration of autonomous
040   or semi-autonomous agents as agentic workflows. Despite their promise, evaluating the effectiveness
041   and reliability of these workflows poses significant challenges.
042   Existing metrics for evaluating agentic workflows primarily include build success status and, in some
043   cases, log analyses. However, these are limited. Firstly, retrieving such metrics requires setting up
044   a test environment and running the test suite, which is either impossible whenever considering a
045   generic range of code repositories in industrial applications or both laborious and time consuming.
046   Secondly, they are too sparse and provide a narrow view of the overall performance, which is not
047   sufficient to assess the quality of the changes made. For example, build status does not provide in-
048   sights into functional correctness or performance under various conditions. Thirdly, analyzing logs
049   can be cumbersome and may not provide actionable insights without significant manual interpreta-
050
      tion.
051   These limitations are particularly problematic for partial failures, where the modified code may
052   not even compile, pass unit tests, or the integration test. In such cases, traditional metrics are not
053   sufficiently available to an agent for improving the patch. Therefore, having access to evaluation
      proxies that assess quality of code changes and are independent of build status after modifications is

116   2.1     A P RIMER ON U SING LLM S FOR C ODE E VALUATION
117
118   In the domain of source code, recent work has shown a promise in the use of LLMs for the au-
119   tomated evaluation of code generation tasks, demonstrating their potential to assess code without
120
      actual execution (Zhou et al., 2023; Zhuo, 2024; Dong et al., 2023). In this context, two primary
      classes of such metrics are widely used: reference-free and reference-aware. The former leverage
121
      LLM’s understanding of source code and their alignment with the given intent (i.e., natural lan-
122
      guage description used for code generation) to assess code quality and functional correctness (Zhou
123   et al., 2023; Zhuo, 2024). Conversely, reference-aware metrics compare the generated code with
124   a ground-truth reference implementation, evaluating how closely the former aligns with the latter,
125   thus providing a direct measure for functional correctness. These are effective for the assessment of
126   generated code. However, a task (e.g., GitHub issue) can be resolved in multiple ways by making
127   non-unique code edits at different locations in the repository. Therefore, such automated approaches
128   are not directly extensible for the qualitative assessment of agentic workflow-generated patches.
129
130   2.1.1    P ROBLEM F ORMULATION
131
132
      Let’s assume a set of coding tasks T ∈ T , typically represented by a goal specified in natural
      language and a code repository to modify. These tasks can generally represent new feature design,
133
      code migration, bug fixing, unit test generation . . . In all generality, we model candidate solutions
134
      to the task T as code patches P = {p1 , p2 , ..., pN }, representing the before/after difference. Note
135   that the candidate patches pi can originate from multiple sources, including humans, LLM/agent
136   workflows, or even by artificial perturbations of correct solutions.
137
138
      The goal of our work is to design an evaluation score S : p 7→ {0, 1} or S : p 7→ R that assesses the
      quality of candidate patches pi with respect to the task T . Importantly, we assume the access to a
139
      ground-truth patch p⋆ that successfully resolves the task T . We allow our score S to depend on that
140
      ground-truth patch p⋆ . In that regard, our problem can be seen as supervised or reference-aware. We
141   relax this assumption in Section 4.3.
142
143   In order to measure the quality of our evaluation score with regards to the two goals above, we
144
      consider the following metrics:
145
146   Success Discrimination Granted a set of successful and failed patches, we consider in sections
147
      4.2 and 4.4 classification models for binarized evaluation score S to map the score to a task success
      criteria such as build status. From this criteria, we assess classical accuracy metrics such as as
148
      precision, recall and F1.
149
150
      Progress Monotonicity By using labeled patches whose advancement status is known (e.g.,
151
      patches generated during agent trajectory) or by artificially perturbing existing patches, we generate
152   set of tuples such p1 ⪯ p2 , with respect to a given notion of order and assess that S(p1 ) ≤ S(p2 ) in
153   Section 4.1 and 4.1. Several such proxy notions of order, i.e. task advancement, can be designed,
154   either in reference-free or reference-aware mode. For instance, percentage of passing tests, number
155   of line of code changed, number of files modified, correctness of the changes, edit distance between
156   the reference and the candidate, etc. We show the limitations of such methods with respect to the
157   initial goal yet illustrate how our method positively correlates with these metrics.
158
159   2.2     LLM C RITICS FOR E XECUTION -F REE PATCH E VALUATION
160
161   Code modifications from an agentic workflow might resolve certain issues while potentially failing
      to address others, or even introduce new ones. To assess the correctness and effectiveness of such

                                                        3
      Under review as a conference paper at ICLR 2025


162
      generated patches, our framework employs test-centric LLM critics. These utilize the potentially
163   unseen tests extracted from gold test patch as a reference (i.e., reference-aware) and individually
164   predict whether each of the tests pass or not. For this purpose, we consider two variants of the
165   generated patch:
166
              1. Context Enhancement: By default, a patch shows only 3 lines of context around each
167
                 hunk, which may not be sufficient for accurately predicting test outcomes due to limited
168
                 understanding of input propagation. To address this, we expand the context to include
169              additional lines that span the entire functions or methods containing the code changes.
170              Such context enhancement provides a more reliable test-centric evaluation of patches.
171
              2. Source Code: Next, we extract the functions or methods containing the code changes after
172
                 applying the patch. We refer to these as post-commit functions. The rationale, in this regard,
173
                 is that the new unseen tests are likely evaluating the post-commit functions.
174
175   Such a design represents a micro-evaluation of patches, as it assesses the generated patches in the
176   context of the unseen tests, individually. The predictions for each test reveals how the changes affect
177
      a particular aspect of the code. This is particularly useful to track progress in potential failures with-
      out actual execution, while establishing a comparative framework for different agentic workflows.
178
179   To determine the overall build status after applying an agentic workflow-generated patch, we aggre-
180   gate the individual test oracle predictions. We determine a build success if our LLM critic predicts
181   all of the new unseen tests to pass. In contrast, if even one of the tests is predicted to fail, we deter-
182   mine a build failure. Note that more ensemble strategies can be explored for this purpose, which is
183
      beyond the scope of this work. In summary, our test-centric framework enables an execution-free,
      macro-evaluation of whether the generated patch successfully addresses all intended functionalities
184
      by aggregating the micro-evaluations based on new unseen tests.
185
186
      2.2.1     U NCERTAINTY Q UANTIFICATION WITH B LACK -B OX C ONFIDENCE M EASURES
187
188   As described in Section 2.2, we first formulate our micro-evaluation of patches using LLM critics
189   as a binary classification task. To calibrate these predictions, we assume that all parameters dur-
190   ing inference are unknown. We elicit confidence estimates by prompting the LLM to express its
191   confidence in unseen test pass/fail prediction as a value between 0 and 100. When combined with
192   Chain-of-Thought (COT) prompting (Wei et al., 2022), such verbalized confidence measures have
193   been shown to be useful for improving the reliability of the LLM’s predictions (Xiong et al., 2024).

198
199   SWE-bench (Jimenez et al., 2024) is a benchmark comprising real-world software engineering tasks,
200   with each instance containing pairs of GitHub issues and corresponding pull requests. While the
201   former describes the desired changes to the codebase, the latter includes the actual code changes
202   made by human developers in resolving the issue and the test cases to validate the changes. Here,
203   the goal of the agentic workflow is to interact with the unfixed repository snapshot, and attempt to
204
      fix the issue. The generated patch is tested against the new unseen tests as well as existing tests
      impacted. In this paper, we consider a canonical subset of SWE-bench (dubbed SWE-bench-Lite)
205
      containing 300 instances collected from 11 popular Python projects, where the gold change patch
206
      contains at most 3 edits in a single file.

247   4.1   LLM C RITICS FOR M ICRO -E VALUATION OF PATCHES
248
249   As described in Section 3.1, a task instance contains gold tests that an agentic patch is expected to
250   pass. To enable a micro-evaluation of patches (i.e., in predicting test oracles), we leverage isolated,
251   test-aware LLM critics that are given access to each of the unseen tests independently as a reference.
252   In this experiment, we evaluate our test-centric framework against a random baseline. Except when
253   using a less effective agentic workflow (e.g. RAG + ChatGPT-3.5 on the SWE-bench leaderboard),
254   where a higher number of tests fail, other agents have significantly more passing ones (ratio of
255   passing to failing tests for factory-code-droid is 85:15). This indicates errors are typically
256   concentrated in fewer editing locations within a multi-hunk patch. We account for such a class
257   imbalance by weighing the probabilities for our baseline proportional to the class frequencies.
258   In Table 1, we report the performance of our isolated, test-aware LLM critics using both patch
259   variants for test oracle prediction. We can see that the LLM critic with context-enhanced patches
260   performs the best, outperforming other baselines by 7.4% to 12.7%. Interestingly, the imbalance in
261   passing and failing tests yields a strong random baseline, which records better performance than even
262   the LLM critic with source code (i.e., post-commit functions). However, as we show in Section 4.2,
263   this is not useful in predicting build outcomes. Upon further analysis, we observed that the LLM
264   critic utilizing source code instead of context-enhanced patches helps identify failing tests better. In
265   contrast, the latter helps identify around 25% more passing tests. This may be because, in this case,
266
      the LLM understands the purpose of the test in the context of the changes better.
267   Confidence Scores v/s Test Complexity. Here, we explore the verbalized confidence scores from
268   the isolated, test-aware LLM critic, and its relationship with test complexity. In Figure 3 (left), we
269   see a clear pattern: the LLM critic tends to make more correct predictions with a high confidence
      than incorrect ones, thus suggesting that the LLM’s verbalized confidence scores are a reliable in-

      dicator of its predictions. Using test character lengths as a proxy to complexity, in Figure 3 (right),
271   we can see that this is particularly true for tests with lower complexity. Based on these analyses, we
272   chose to automatically assume failures when the LLM critic assesses a patch with low confidence
273   (≤ 65%) for tests with high complexity (test length > 50). We observed that such thresholding helps

278   4.2     LLM C RITICS FOR M ACRO -E VALUATION OF PATCHES
279
280   To enable the macro-evaluation of patches (i.e., in predicting build outcomes), in our test-centric
281   framework, we next aggregate the predictions from the isolated, test-aware LLM critics (as in Sec-
282
      tion 4.1). If even one test among all unseen tests is expected to fail, we predict a build failure.
      Otherwise, we predict that the patch would successfully pass the build.

      4.2.1    BASELINES
286   We select multiple baselines to evaluate our test-centric framework in build status prediction:
287
288
        (a) Random: First, to establish a reference for the more sophisticated LLM critics, we aggregated
            the predictions from the random baseline in Section 4.1, with the assumption that even a
289
            single failing test prediction results in a build failure. Such an approach helps us measure
290
            the importance of aggregating reasoning-based test oracle predictions from LLM critics as
291         opposed to random ones, in determining build outcomes.
292
293   Every task instance in SWE-Bench comes with a gold patch, containing: the actual code changes
294
      human developers made to resolve the issue, and the test cases that validate the changes. In our test-
      centric framework, we consider only the test cases as the reference. Here, aiming to evaluate whether
295
      the generated patch and the gold change patch are equivalent, we use the latter as the reference.
296
297     (b) Edit Distance: In this baseline, we leverage a pre-trained code language model, Code-
298         BERT (Feng et al., 2020), to retrieve the embeddings for both the generated patch and the
299         reference. Next, we compute the cosine similarity between the two to quantify the similarities
300         between their semantic contents. To determine the build outcome, we perform a grid search on
301
            the validation set to identify an optimal threshold, which we then apply to all task instances.
302     (c) Change-Aware: Here, we design LLM critics probing for patch equivalence. Given the can-
303         didate and gold change patch pairs, these determine whether they would result in the same
304         functional outcome. To this end, we assume two patch variants: one, default patches with 3
305         lines of context around each hunk; two, ± function-level patches (as described in Section 2.2).
306   Next, to assess the importance of aggregating micro-evaluations, we compared with:
307
308
        (d) Holistic, Test-Aware: In this baseline, we design LLM critics to predict the collective outcome
            of all new unseen tests. A positive prediction indicates that all tests pass, i.e., build success.
309
            Conversely, a negative prediction indicates at least one of the tests fails, signifying a build
310
            failure. To this end, the holistic, test-aware LLM critics take the generated patch and all corre-
311         sponding reference tests as inputs. Here, we consider both patch variants (as in Section 2.2).

      5   R ELATED W ORK
487
488
      Large Language Models for Code. Recent software engineering (SE) research has focused on the
489   use of machine learning-based approaches for SE tasks including program synthesis (Li et al., 2022;
490   Nijkamp et al., 2023), vulnerability detection (Fu & Tantithamthavorn, 2022), automated program
491   repair (Li et al., 2020; Ahmed & Devanbu, 2023), test generation (Schäfer et al., 2023), etc. These
492   have traditionally been limited to code snippets (often at the method-level) extracted from software
493   repositories. With the advances in large language models (LLMs), there has been a shift in focus
494   towards extending these to the repository-level, for SE tasks like code (Bi et al., 2024; Deng et al.,
495   2024; Pan et al., 2024) and patch (i.e., code change) generation (Zhang et al., 2024; Bairi et al.,
496   2024). In this work, we design test-aware LLM critics to evaluate code changes.
497   Benchmarks for Repository-Level Coding Tasks. Building on method-level (Chen et al., 2021;
498   Austin et al., 2021) and class-level (Du et al., 2023) code generation benchmarks, there has been a
499   rise in repository-level code generation benchmarks in academia. CrossCodeEval (Ding et al., 2023),
500   CoderEval (Yu et al., 2024), RepoBench (Liu et al., 2024) support multilingual code generation tasks
501   utilizing cross-file context extracted from real-world open-source repositories. Extending beyond
502   code completion, SWE-bench (Jimenez et al., 2024) introduces a broader set of challenges involving
503
      patch generation, grounded in real-world software engineering tasks like bug fixing, feature addition
      or enhancement, etc. However, SWE-bench is limited to Python task instances. As a first step toward
504
      multilingual support, SWE-bench-java (Zan et al., 2024) was developed to extend this framework
505
      to Java. We evaluate our LLM critics on the Python task instances in SWE-bench. However, these
506   can be easily extended to new programming languages and repositories, providing a foundation for
507   assessing the code execution-specific understanding of LLMs across other languages.
508
509
      Automated Evaluation of Large Language Models in Coding Tasks.
510   Traditionally, the generated code can be evaluated statically, in terms of software quality (e.g. read-
511   ability, complexity (Oman & Hagemeister, 1992)). In particular, code changes in a repository can be
512   assessed via program differencing (Apiwattanapong et al., 2004) and change impact analysis (Ren
513   et al., 2005) – useful to determine the effect of a change on the rest of the repository. However, none
514   of these approaches account for the correctness of the generated code or patches.
515   By matching against a reference solution, semantic-based metrics such as BLEU (Papineni et al.,
516   2002), ROUGE (Lin, 2004), and CodeBLEU (Ren et al., 2020) or neural based metrics such as
517   CodeBERT (Feng et al., 2020) help establish match-based evaluation proxies. However, these are
518   limited to source code and do not capture program semantics well nor correlate efficiently with
519   human judgment (Eghbali & Pradel, 2004; Tran et al., 2019). Recent work has proposed the use of
520   LLMs for such an evaluation, thus helping establish execution-free evaluation proxies which probe
521
      for correctness. These include both reference-free (Zhou et al., 2023; Zhuo, 2024) and reference-
      aware approaches (i.e., those utilizing human developer-written code or tests) (Dong et al., 2023).
522
      However, these are not extensible for the evaluation of patches and still exhibit limited efficiency.
523
      In this work, we design both reference-free and test-reference-aware evaluation proxies (the latter
524   significantly outperforming the former).
525
526
527   6   C ONCLUSION
528
529   As a conclusion, we designed LLM-based critics to derive execution-free evaluation proxies for
530
      repo-level code changes. With the gold test patch as a reference, we predict executability of all
      editing locations with an accuracy of 91.5%, aggregating which, we can predict the build status in
531
      81.8% of the instances in SWE-bench, outperforming other reference-free and reference-aware LLM
532
      critics by 38.9% to 72.5%. Most notably, we observe that aggregating fine-grained assessments of
533   candidate patches leads to better performance than evaluating them as a whole and that using tests
534   as a reference proves more effective than code changes, more so because these enable fine-grained
535   assessments.
536
      Natural extensions of our work include investigating benchmarks from other programming lan-
537
      guages, directly incorporating this execution score in agentic framework by relaxing the reference-
538
      aware character of our evaluation, and utilizing the LLM input to proactively design better tests.
