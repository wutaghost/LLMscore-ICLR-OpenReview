008                                                    A BSTRACT
009
010            Reinforcement learning (RL) with unit test feedback has enhanced large language models’
011            (LLMs) code generation, but relies on sparse rewards provided only after complete code
012            evaluation, limiting learning efficiency and incremental improvements. When generated
013            code fails all unit tests, no learning signal is received, hindering progress on complex tasks.
014            To address this, we propose a Process Reward Model (PRM) that delivers dense, line-level
015
               feedback on code correctness during generation, mimicking human code refinement and
               providing immediate guidance. We explore various strategies for training PRMs and inte-
016
               grating them into the RL framework, finding that using PRMs both as dense rewards and
017
               for value function initialization significantly boosts performance. Our approach increases
018            our in-house LLM’s pass rate from 28.2% to 29.8% on LiveCodeBench and from 31.8%
019            to 35.8% on our internal benchmark. Our experimental results highlight the effectiveness
020            of PRMs in enhancing RL-driven code generation, especially for long-horizon scenarios.

023   1   I NTRODUCTION
024
025   The rapid advancement of large language models (LLMs) has revolutionized code generation, enabling
026   models to achieve near-human performance on programming tasks (Chen et al., 2021a; Li et al., 2022;
027
      OpenAI, 2023). These models have demonstrated remarkable abilities to generate syntactically correct and
      functionally viable code snippets, significantly aiding software development processes. Building upon these
028
      successes, recent research has explored the use of reinforcement learning (RL) from unit test feedback to
029
      further enhance the code generation capabilities of LLMs (Le et al., 2022; Shojaee et al., 2023; Liu et al.,
030   2023; Dou et al., 2024). By incorporating unit tests as a reward mechanism, these methods aim to guide
031   LLMs toward generating code that not only compiles but also passes specified test cases, thereby improving
032   overall code reliability and quality.
033
      However, a significant challenge arises from the nature of the reward signals derived from unit tests. These
034
      signals are inherently sparse, as they are only received at the end of an episode after the entire code snippet
035
      has been generated and evaluated. This delay in feedback impedes learning efficiency and limits the model’s
036   ability to make incremental improvements during code generation. When an LLM fails to generate code
037   that passes any unit tests, it receives no meaningful learning signal, making it difficult to learn to solve
038   more complex coding problems. In contrast, human programmers typically do not rewrite code from scratch
039   when their programs fail unit tests. Instead, they analyze the code to pinpoint and fix errors, leveraging
040   their understanding of programming logic and structure to iteratively improve upon the current version.
041   This process of step-by-step refinement, which involves receiving and acting upon fine-grained feedback, is
042   missing in the current RL training loop for code generation from unit test feedback.
043   To address this limitation, we propose integrating a Process Reward Model (PRM) (Lightman et al., 2023;
044   Wang et al., 2024a) into the RL training framework for code generation. A PRM provides dense signals by
045   offering line-level feedback that indicates the correctness of each generated line of code. This fine-grained
046   feedback mechanism mimics the human approach to code refinement and has the potential to enhance learn-

047
      ing efficiency by providing immediate guidance during code generation. While the concept of using PRMs
048
      is intuitive, training an effective PRM and integrating it into RL training is non-trivial. Challenges include
049   accurately modeling the correctness of partial code snippets and ensuring stable and effective training dy-
050   namics when combining PRM-generated signals with traditional RL methods. Although previous research
051   has attempted to incorporate PRMs into LLM RL training (Wang et al., 2024a), these efforts have been
052   limited to the mathematical domain and have not fully explored the complexities involved.
053
      In this work, we conduct a comprehensive analysis of how PRMs can be integrated into RL training for code
054
      generation. We explore various strategies for training a robust code PRM and investigate different methods of
055   utilizing PRMs to improve code generation performance. Based on our experiments, we provide a practical
056   recipe for successfully using PRMs and integrating them into RL training in the context of code generation
057   problems. Notably, one of our key findings is that using PRMs concurrently as both dense rewards and value
058   function initialization in RL training leads to a significant performance improvement. Our contributions can
059   be summarized as follows:
060
061    • We propose an effective approach that automatically generates process-level supervision data by iden-
062      tifying the first error line in generated code using binary search. We then train a PRM on this data to
063      generate dense signals during RL training. To the best of our knowledge, we are the first to demonstrate
064
         that PRMs can benefit RL from unit test feedback in code generation.
065    • We conduct systematic experiments to determine how to properly and effectively integrate PRMs into
066      RL. Our analysis explores various strategies for training a high-quality code PRM and utilizing PRMs to
067      improve code generation. We summarize our findings into a practical recipe for successfully using PRMs
068      in the context of code generation.
069
       • By following the recipe, we significantly improve our in-house proprietary LLM’s pass rate from 28.2%
070
         to 29.8% on the LiveCodeBench dataset and from 31.8% to 35.8% on our in-house benchmark. Besides,
071
         we find that integrating PRMs into RL training benefits code generation in long-horizon scenarios.

079   (y(1) , y(2) , . . . , y(T ) ), where T represents the total number of code generation steps. Each individual code

      Our method introduces a PRM to assess the correctness of each line of the code during the generation
136   process. The PRM serves as an oracle that provides intermediate rewards based on the potential of the
137   current code prefix to be extended into a correct program. By offering intermediate feedback, the PRM
138   helps the model identify and reinforce beneficial code generation patterns while discouraging those that
139   introduce errors. This fine-grained feedback mirrors the human approach to coding, where programmers
140   continuously evaluate and adjust their code.

196   3.2   I NTEGRATING PRM INTO RL T RAINING
197
198   Given a learned PRM, we aim to identify best practices for enhancing code generation during RL training.
199   While prior work has used PRMs to verify intermediate steps in mathematical tasks (Lightman et al., 2023;
200   Wang et al., 2024a; Jiao et al., 2024; Wang et al., 2024b; Luo et al., 2024), their potential for guiding code
201
      generation remains largely unexplored. In mathematical domains, LLMs may generate correct answers
      with faulty reasoning (Lightman et al., 2023), making intermediate verification essential. However, in code
202
      generation, problems are typically accompanied by multiple unit tests, making it improbable for incorrect
203
      code to pass all tests. As a result, the emphasis on intermediate verification is less applicable. Instead,
204   we propose leveraging PRMs as auxiliary sources of dense signals to facilitate better exploration during
205   RL training. While preliminary attempts have been made to incorporate PRMs into RL training (Wang
206   et al., 2024a), these efforts are limited and warrant a more thorough investigation. We explore the following
207   methods to integrate PRMs effectively:
208   PRM as Dense Rewards. Similar to Wang et al. (2024a), we use PRMs to provide step-level reward
209   signals that guide more efficient policy exploration during RL training. By rating the correctness of each
210   line in the code response, the PRM supplies “dense” rewards that encourage the policy to explore more
211   promising code paths, leading to improved performance.
212   PRM as Value Initialization. The PRM’s method of annotating code, by fixing a prefix y≤t and rolling
213   out the policy to sample correct responses, can be viewed as a “hard” value estimation of y≤t . We hy-
214   pothesize that the PRM’s capability to provide line-level feedback could serve as a useful inductive bias for
215   initializing the value function in RL algorithms, which can ease the credit assignment problem by offering a
216   more informed starting point.
217   PRM as Both Dense Rewards and Value Initialization. To fully capitalize on the advantages of PRMs,
218   we combine both approaches. By using PRMs for dense rewards and as an initialization for the value model,
219   we aim to enhance RL training through improved exploration and more effective credit assignment.

      4.2.1    PRM T RAINING : M ORE DATA OR B ETTER DATA ?
283
284   Recent research on LLMs highlights that data quality often outweighs quantity (Gunasekar et al., 2023; Li
285   et al., 2023b). We found the same holds true for PRM training data selection. Although automated data
286   annotation allows for the generation of large volumes of PRM training data through model sampling, our
287   experiments showed that increasing data volume can sometimes degrade PRM performance when integrated
288   into RL. In contrast, a smaller, carefully curated subset of the full dataset led to better supervision and
289   improved outcomes. For example, when all sampled responses to a given prompt either consistently pass (or
290
      fail) unit tests, PRM gains little useful information. In such cases, the model can only learn to predict the
      correct (or incorrect) label when it encounters the same prompt again, limiting its ability to generalize. We
291
      explored various strategies for selecting and filtering data, as detailed in Section 4.3.
292
293   4.2.2    RL T RAINING : A LLEVIATING PRM H ACKING
294
295   Reward model hacking (Skalse et al., 2022) is a well-known issue in RLHF training, where the policy learns
296
      to exploit flaws in the reward model to achieve high rewards without genuinely improving the quality of
      response. Similarly, we observed that PRM is also susceptible to such exploitation. Here we discuss two key
297
      practical strategies to mitigate the risk of PRM hacking and ensure the reward signals remain aligned with
298
      the intended task objectives.
299
300   PRM reward length normalization. As described in Section 4.1, when used to provide dense rewards,
301   PRM assigns line-level reward signals at the end-of-line tokens in the LLM-generated response. However,
302   if we directly use the predictions of the learned PRM, denoted as Rϕ , as the reward signal RPRM in 3, we
303   observed that this can be exploited. Specifically, the policy may generate numerous lines for which PRM
304   predicts positive rewards, thus inflating the overall reward. This occurs because writing more lines allows
305
      the model to accumulate excessive intermediate rewards, effectively hacking the optimization objective. To
      mitigate this issue, we apply length normalization to the PRM predictions. Given a prompt x and a response
306
      y with T lines, y = (y(1) , y(2) , . . . , y(T ) ), we define the PRM dense reward signal at the m-th line as:
307
308                                                          1

      This normalization ensures that the policy does not gain higher cumulative rewards by generating trivial or
311
      unnecessarily long responses, as the accumulated reward is bounded within the range of [−1, 1] regardless
312
      of the response length.
313
314   Assigning additional neutral label into PRM training. While length normalization helps reduce PRM
315   exploitation, it is insufficient to fully prevent PRM hacking. Even with normalization, we empirically ob-
316   served that the model can still exploit PRM by generating excessive comment lines within the code. The
317   underlying issue is that writing a correct comment is often far easier than producing correct code. As a result,

      5     R ELATED W ORKS
424
425
426   5.1   LLM S FOR C ODE G ENERATION
427
428   Recently, large language models (LLMs) have demonstrated impressive capabilities in code generation by
429   pre-training on vast text datasets that include code (Lu et al., 2021; Christopoulou et al., 2022; Allal et al.,
430   2023; Zheng et al., 2024; Li et al., 2023b). Additionally, models fine-tuned through supervised fine-tuning
431
      (SFT) have achieved competitive results in code generation tasks (Chen et al., 2021a; Li et al., 2023a; Luo
      et al., 2023; Rozière et al., 2024; Guo et al., 2024). Reinforcement Learning (RL) optimizes policies by
432
      interacting with an environment and receiving rewards (Williams, 1992). Recently, RL has been incorporated
433
      into LLMs to enhance code generation using unit test feedback (Shojaee et al., 2023; Liu et al., 2023; Le
434   et al., 2022). CodeRL (Le et al., 2022) applies unit test signals as rewards with an actor-critic method,
435   while PPOCoder (Shojaee et al., 2023) builds on this by using the PPO algorithm. RLTF (Liu et al., 2023)
436   improves precision by locating errors, though the reward space remains sparse. Despite progress, RL’s
437   potential to significantly boost code generation in sparse reward environments remains underexplored.
438
439
440   5.2   P ROCESS R EWARD M ODELS
441
442   Process reward models (PRMs) have garnered significant attention in recent LLM developments, partic-
443   ularly in the mathematical reasoning domain, where they provide verification for intermediate reasoning
444   steps (Lightman et al., 2023; Wang et al., 2024a; Jiao et al., 2024; Wang et al., 2024b; Luo et al., 2024).
445
      While some approaches rely on costly and resource-intensive human-annotated process data (Lightman
      et al., 2023), recent research has focused on automating the collection of process supervision data Wang
446
      et al. (2024a); Jiao et al. (2024); Wang et al. (2024b); Luo et al. (2024). Building on these efforts, we sim-
447
      ilarly automate process supervision but differ in our primary objective. Rather than using PRMs solely as
448   enhanced verifiers compared to Outcome Reward Models (ORMs), we focus on their integration into RL
449   training for code generation. While Wang et al. (2024a) provides preliminary results on PRMs improving
450   RL training in the mathematical domain, their findings are limited. Our work offers a more thorough and
451   systematic investigation of how PRMs can be leveraged in RL for code generation tasks.
452
453
454   6     C ONCLUSIONS AND L IMITATIONS
455
456
      In this work, we addressed the challenge of sparse reward signals in reinforcement learning (RL) for code
457
      generation by introducing a Process Reward Model (PRM) that provides dense, line-level feedback. This ap-
458
      proach mirrors human-like code refinement and improves learning efficiency. Our experiments demonstrate
459   that integrating PRMs significantly enhances the pass rates of code generation models on both the Live-
460   CodeBench dataset and proprietary benchmarks. This method has the potential to improve long-horizon
461   code generation scenarios, advancing the state-of-the-art in LLM-based code generation.
462
      Despite the promising results, our approach has several limitations that warrant further investigation. First,
463
      the effectiveness of the PRM relies heavily on the quality of the collected data. While we automated data
464
      collection using binary search and unit tests, this method may not capture all nuances of code correctness
465   and may introduce noise, especially in more complex or ambiguous programming tasks. Second, the com-
466   putational cost of collecting PRM training data is still substantial, even though we employed binary search
467   to mitigate it. Third, our automated PRM data collection method requires external verification (unit tests in
468   our case), which is not applicable to many other domains such as creative writing or open-ended generation
469   problems. This could limit the applicability of our approach in those areas.

      We evaluated our models on two additional coding benchmarks: HumanEval and MBPP. The descriptions of
802
      these two datasets are provided below. For evaluation, we used the same settings as described in Section 4.1.
803   Specifically, we generated 10 candidate responses for each problem, using a temperature of 0.2, nucleus

805   as the evaluation metric. The results can be found in Table 4.
806
807   HumanEval (Chen et al., 2021b) This dataset comprises 164 hand-crafted programming problems that
808   are designed to evaluate the functional correctness of code generated by LLMs, rather than merely comparing
809   textual similarity to reference solutions. The problems in HumanEval cover a range of tasks, including
810   language comprehension, algorithms, and basic mathematics, and they are comparable to typical software
811   interview questions. Each problem is accompanied by a function signature, a docstring, a function body,
812
      and multiple unit tests to rigorously test the generated solutions. We directly input the problem to the model
      without using few-shot prompting.
813
814
      MBPP (Austin et al., 2021) This dataset comprises 974 crowd-sourced Python programming tasks, specif-
815
      ically crafted to be solvable by entry-level programmers. Each problem in the MBPP dataset consists of a
816
      task description, a code solution, and three automated test cases, covering programming fundamentals and
817   standard library functionality. The dataset is particularly valuable for evaluating a model’s ability to synthe-
818   size short Python programs from natural language descriptions. For evaluation, we used the problems with
819   IDs 11–510, totaling 500 problems, as recommended by the original dataset authors. We directly input the
820   problem to the model without using few-shot prompting.

      D    A DDITIONAL O PEN - SOURCE LLM E XPERIMENTS
847
848
      Base Model. We adopt Qwen2.5-7B as our base model (QwenTeam, 2024), a recently released causal
849   language model available at https://huggingface.co/Qwen/Qwen2.5-7B. Qwen2.5 belongs
850   to the Qwen series of large language models (Yang et al., 2024), known for their advanced capabilities
851   across a wide range of domains. The Qwen2.5-7B model has 7.61 billion total parameters (6.53 billion
852   excluding embeddings) and utilizes the Transformer architecture as its core. It incorporates state-of-the-
853   art enhancements, including Rotary Positional Embedding (RoPE), SwiGLU activation, RMSNorm, and
854   Attention QKV bias. The model consists of 28 layers and employs 28 attention heads for queries (Q) and 4
855   for keys and values (KV), making it highly efficient for tasks requiring robust attention mechanisms.
856
857   SFT Settings. We fine-tuned the Qwen2.5-7B model on the same supervised fine-tuning (SFT) dataset
858
      described in Section 4.1. The model was trained for two epochs, starting with a learning rate of 1 × 10−7 ,
      which linearly increased to 2 × 10−5 during the first 2% of the total training steps. After reaching the
859
      peak learning rate, a cosine learning rate decay schedule was applied, gradually reducing the learning rate to
860
      2×10−6 for the remainder of the training. Additionally, a constant weight decay of 0.01 was used throughout
861   the SFT training process to regularize the model and improve generalization. The model fine-tuned through
862   this process is referred to as Qwen2.5-7B-SFT.
863
864   RL Baseline. We adopted the same RL baseline training method and used the same RLHF dataset de-
865   scribed in Section 4.1 to further train the Qwen2.5-7B-SFT model. For PPO training, we configured the
866   following hyperparameters: a batch size of 4096, a linear warmup over the first 5 steps, followed by a
867   constant learning rate of 2 × 10−6 for both the actor and critic, and a KL penalty of 0.01. The training
868   utilized the AdamW optimizer and spanned approximately 300 steps, during which we empirically observed
869   performance convergence.
870
871
      PRM Training. Following the approach outlined in Section 4.1, we selected four checkpoints at 50, 100,
      150, and 200 steps during the training process of the RL baseline model. For each checkpoint, we sampled
872
      n = 5 responses for every coding prompt in the training dataset Dtrain . Each sampled response was labeled
873
      using the binary search procedure described in Algorithm 1, with K = 20 completions generated for each
874   partial code prefix. The data collected from all checkpoints was then aggregated to form a PRM training
875   set, employing the Revised Only strategy described in Section 4.3. This resulted in 165K samples and 28M
876   tokens. On average, each response contained 16.07 lines. The PRM label distribution was 25.88% for -1,
877   15.90% for 0, and 58.22% for +1. The PRM was initialized using the value model from the RL baseline and
878   fine-tuned on this PRM dataset using the objective function defined in Eq. (2).
879
880   Integrating PRM into RL. We used the same settings and hyperparameters as described in Section 4.1.
881   Additionally, we observed that due to the properties of the Qwen2.5-7B tokenizer, a newline token is not
882   always represented as a simple "\n" token. Instead, the tokenizer combines other non-space characters with
883   an ending "\n" to form new tokens (e.g., ":\n", "):\n", ")\n", "\n\n", "())\n", "]\n",
884
      "()\n", "():\n", etc.). This makes it more challenging to accurately identify line separator tokens in
      the model’s responses.
885
886   Empirically, we addressed this challenge by selecting the 50 most frequent tokens in the PRM dataset whose
887   corresponding token strings include "\n". The full list of token ids is shown below:

           24135, 317, 7368, 2440, 10907, 22165, 4432, 5929, 7129, 345, 11043, 532, 4660, 21686, 14288}.
