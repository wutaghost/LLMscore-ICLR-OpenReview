009                                               A BSTRACT
010
011            Large language model (LLM)-based mobile agents are increasingly popular due
012            to their capability to interact directly with mobile phone Graphic User Interfaces
013            (GUIs) and their potential to autonomously manage daily tasks. Despite their
014            promising prospects in both academic and industrial sectors, little research has
015            focused on benchmarking the performance of existing mobile agents, due to the
016            inexhaustible states of apps and the vague definition of feasible action sequences.
017            To address this challenge, we propose an efficient and user-friendly benchmark,
018            MobileAgentBench, designed to alleviate the burden of extensive manual testing.
019
               We initially define 100 tasks across 10 open-source apps, categorized by multiple
               levels of difficulty. Subsequently, we evaluate several existing mobile agents, in-
020
               cluding AppAgent and MobileAgent, to thoroughly and systematically compare
021
               their performance. All materials will be accessible on our project webpage, con-
022            tributing to the advancement of both academic and industrial fields.

      1   I NTRODUCTION
026
027
      With the emergence of large language models (LLMs) (Achiam et al., 2023), researchers have devel-
028
      oped various autonomous agents across fields such as robotics (Bousmalis et al., 2023; Reed et al.,
029
      2022), games (Wang et al., 2023b; Du et al., 2023), and mobile phones (Yang et al., 2023b; Rawles
030   et al., 2023). Among these, mobile agents have attracted significant attention due to their potential
031   to enhance user experiences and provide intelligent assistance on-the-go.
032
033
      People have been dreaming of Intelligent Personal Assistants (IPAs) (de Barcelos Silva et al., 2020)
      that can fully automate daily tasks for decades. Since Apple introduced its digital assistant, Siri (Ap-
034
      ple, 2011) in 2011, almost all the leading technology companies have launched their own IPAs,
035
      including Microsoft Cortana (Microsoft, 2014), Amazon Alexa (Amazon, 2014), and Google Assis-
036   tant (Google, 2016). While these digital assistants provide a hands-free human-computer interaction
037   experience using Natural Language Interface (NLI), they can only fulfill relatively simple tasks, such
038   as setting an alarm clock or sending a text message (Li et al., 2024). For third-party apps, developers
039   have to follow and implement the application programming interfaces and protocols, such that when
040   a user issues a very specific command, the system can invoke the corresponding functionality. This
041   limits the usability of those digital assistants.
042   LLMs contribute significantly to resolving the persistent challenge of understanding user intent.
043   The demonstrated reasoning ability (Qiao et al., 2022) highlights the potential of LLM-based au-
044   tonomous agents as next-generation Intelligent Personal Assistants (IPAs), which are not limited by
045   the programming interfaces since they directly operate on the Graphic User Interface (GUI) (Wang
046   et al., 2024; Yang et al., 2023b). The GUI can either be represented by a text-based view tree to be
047   consumed by a LLM or a screenshot image that can leverage a Multi-modal LLM (MLLM) (Yin
048   et al., 2023). The action space of the agents composes a series of functions to simulate human oper-
049   ations, such as click, type, swipe, etc. In this way, LLM agents can theoretically achieve whatever
050
      human users can do, without any modification of the existing apps.
051   The promising future of LLM-based smartphone agents attracts more and more researchers to study
052   this topic. However, the scope of benchmarks available for evaluating the performance of these
053   agents remains constrained. Among the existing benchmarks, several prevalent issues are evident:
      1. Scalability and usability. Researchers need to fully understand complicated data structures and

083   offers several advantages over previous benchmarks due to its ease of use and minimally invasive

086   testing tasks across various Android operating system versions and executing on actual devices.
087

089   sion process. Third-party developers can specify the conditions for task success using just a few
090   lines of Python code, without needing extensive knowledge in Android development. This accessi-
091   bility makes MobileAgentBench more conducive to developers and researchers from non-Android
092   Development communities. Furthermore, we introduce an innovative method for determining the
093   task-terminating state, rendering the benchmark resistant to the complexities of tracking multiple
094   potential success pathways. This approach ensures that MobileAgentBench provides reliable and
095   precise benchmarking outcomes.

      2.1    M OBILE LLM AGENTS
111
112   Studies before the LLM-era employed reinforcement learning (RL) to solve the autonomous GUI
113   navigation problem (Gur et al., 2018). The recent advancement of LLM and MLLM becomes the
114   dominant agent paradigm becaust of the greater ability of UI understanding and reasoning. Early
115   studies focused on web agents, which achieve task automation within browsers (Deng et al., 2024;
116   Zhou et al., 2023). Recently, more and more studies started to investigate agents on mobile devices,
117   especially on the Android platform, as Android smartphones are the most widely used personal
118
      computing devices.
119   Mobile LLM agents share a similar algorithm. The full input prompt often consists of four main
120   components: the user prompt (task description), the current UI view hierarchy (VH) description, the
121   action function list, and historical information. Specifically, the action list mainly includes click,
122   swipe, type, and other common UI operations. If MLLM is used, the current screenshot is also a
123   part of the input. The LLM/MLLM is asked to think of the next action based on the current and
124
      historical states and call the correct function to perform the given task step by step. The agent finally
      parses the model response and sends control signals to the Android device using Android Debug
125
      Bridge (ADB) 1 , UIAutomator 2 , or other higher-level UI automation frameworks.
126
127   Despite the similarity of the high-level ideas, researchers have developed different techniques to im-
128   prove performance and efficiency. Among these works, AndroidArena (Xing et al., 2024) transforms
129   the long and overwhelming view hierarchy XML into a compressed representation and assigned UI
130
      elements with unique node IDs, which shortens the prompt and makes the system more efficient.
      MobileAgent (Wang et al., 2024) observes that GPT-4V lacks the capability of UI element local-
131
      ization, and employs an Optical Character Recognition (OCR) model to locate and localize text
132
      views. Moreover, it uses the CLIP (Radford et al., 2021) and Grounding DINO (Liu et al., 2023)
133   models to detect icons. AppAgent (Yang et al., 2023b) uses SoM (Yang et al., 2023a) prompts
134   to localize UI elements and breaks tasks into two phases, exploration, and deployment. During
135   the exploration phase, the agent automatically interacts with the apps and summarizes the observa-
136   tions into a document. In the deployment phase, it employes the Retrieval Augmented Generation
137   (RAG) (Lewis et al., 2020) technique to utilize the summarized knowledge and improve success
138   rate. CogAgent (Hong et al., 2023) proposes its own highly efficient 18B-parameter MLLM, which
139   can be loaded on a single commercial GPU. Furthermore, Octopus v2 (Chen & Li, 2024) proposes a
140   compact 3B-parameter model, which unlocks the potential to run mobile LLM agents on-device in
141
      an efficient and privacy-preserving manner.
142
143   2.2    B ENCHMARKS FOR M OBILE LLM AGENTS
144
      Since the Mobile LLM agent is a newly emerging research field, the choice of benchmarks is very
145
      limited. Some studies rely on verifying the task execution status manually to evaluate the perfor-
146   mance (Yang et al., 2023b), which is tedious and time-consuming. To expedite the agent develop-
147   ment, we need a fully autonomous benchmarking system to report various metrics, especially the
148   task success rate. However, automatically judging if a task is completed successfully is non-trivial.
149   The main challenge is caused by the dynamic nature of the GUI navigation task – the agent may
150   perform random actions and drive the app to an unknown state.
151
      AITW (Rawles et al., 2023) is a popular benchmark for mobile LLM agents. It has a large scale,
152   but it’s based on static screenshot images. Thinking of each app state (screenshot) as a node, and
153   each action as an edge, we can build a State Transferring Graph (STG) based on the screenshots and
154   the human-annotated actions. It fails immediately if the agent performs actions in a non-considered
155   sequence and leads the STG to a non-existent node, even if the agent can eventually complete the
156   task.
157
      The only solution is to identify task successes on real devices. One approach is to match the agent’s
158   actions with the annotated ground truth (GT). A step-wise matching algorithm is not accurate, be-
159   cause the agent may not finish the task exactly in the same order with GT. AndroidArena (Xing

293   The overview of MobileAgentBench is shown in Fig. 2. The benchmarking apps run on real devices
294
      from a device farm, which can either be a physical device or an emulator. The device talks to the

295

344       4.1    M ETRICS
345
          We define 6 metrics to comprehensively benchmarking mobile agents:
346
347               • Success Rate (SR): SR = Nsuccess /Mtasks , where Nsuccess is the number of successful
348                 tasks, judged by the benchmark system. Mtasks is the number of total benchmarking tasks.
349                 This metric reflects the agent’s ability to correctly finish a task end-to-end.
350               • Step-wise Efficiency (SE). SE = Sactual /Smin , where Sactual is the number of actual
351                 steps the agent takes to successfully finish a task, and Smin is the minimum number of
352                 steps. This metric tells us if the agent performs unnecessary or redundant actions and
353                 reflects the efficiency of the agent. Failure tasks are not taken into account.
354               • Latency. The average time spent in seconds before and after one action. This metric tells
355                 us how long a user needs to wait between two actions.
356
                  • Tokens. The number of LLM input and output tokens. For simplicity, we use the GPT-
357                 4V (gpt-4-vision-preview) standard (OpenAI) to calculate the number of tokens
358                 for all models, which gives us a rough estimation of the LLM cost. For text, 1 token is
359                 4 characters. For an image, it’s divided into multiple 512× 512 tiles, and each tile is 170
360                 tokens. 85 base tokens are applied to each image as well.
361               • False Finish Rate (FFR). F F R = Nearly /Mf ailure , where Nearly is the number of early
362                 stopped tasks and Mf ailure is the total number of failure tasks. This metric represents how
363                 likely the agent falsely thinks it has finished the task and stopped early.
364               • Over Execution Rate (OER). OER = Nlate /Msuccess , where Nlate is the number of late
365                 stopped tasks and Msuccess is the total number of successful tasks. Symmetricly to FFR,
366                 this metric reveals how likely the agent falsely thinks the task is not finished successfully.

      While the proposed MobileAgentBench is efficient, user-friendly, and addresses many issues of the

      existing benchmarks for mobile LLM agents, there are two main limitations that the authors would

455
      In this paper, we propose a new benchmark, MobileAgentBench, for mobile LLM agents on the
456
      Android platform. With the 100 built-in benchmarking tasks, researchers can test and evaluate
457
      existing and new agents automatically on real Android devices. Extending the benchmark to support
458   customized tasks is also easy, as only basic Python coding skills are needed. Leveraging the Android
459   Accessibility Services and only checking the final app state, MobileAgentBench can detect task
460   completion status faithfully. We report the evaluation results of 5 popular agents across multiple
461   metrics, and they can be used as strong baselines to advance future mobile LLM agent development.

      A.2     C USTOMIZATION
620
621   The 100 built-in tasks make the benchmark work out of the box. However, third-party researchers
622   may want to implement their customized tasks. For example, benchmarking their agent performance
623   on commercial apps. This can be done easily without modifying the benchmark library. First, imple-
624   ment the customized task class inheriting the Task class provided by the benchmark framework. The
625   most important method to implement is check finish, which defines how to judge if the given
626   task is finished successfully. If necessary, the setup and teardown function can also be imple-

      mented to prepare and reset the environment to execute the current task. Then, add the task names
      to a JSON file and pass it to the benchmark’s task orchestrator. When running, the task orchestrator
628
      will invoke each task (built-in or customized) one by one, automatically.

670       dealing with multiple possible action paths to success, we determine if the task is successful by only

672       We check if the Frequently Asked Questions text view (a child view in the About page) exists in

681       bility Service, such as a button-clicking event. It helps to check the task-finishing status when it’s

          not directly reflected in the current UI view.
