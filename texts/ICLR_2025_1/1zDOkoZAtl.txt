009                                               A BSTRACT
010
011            Previous work has demonstrated that in some settings, the mechanisms imple-
012            mented by small neural networks can be reverse-engineered. However, these efforts
013            rely on a manual approach that cannot easily be applied to networks with billions
014            of parameters. To investigate a potential avenue towards scalable interpretabil-
015            ity, we show it is possible to use meta-models, neural networks that take another
016            network’s parameters as input, to learn a mapping from transformer weights to
017
               human-readable code. We build on Tracr (Lindner et al. 2023) to synthetically
               generate transformer weights that implement known programs in the RASP lan-
018
               guage (Weiss et al. 2021), then train a transformer to extract RASP programs from
019
               weights. Our trained compiler effectively extracts algorithms from model weights,
020            reconstructing a fully correct algorithm 60% of the time.
021

023   1   I NTRODUCTION
024
025   Neural networks are typically black boxes; we know that they are able to perform a task (image
026   recognition, language modeling, etc.), but we do not know how they perform it. In this work, we
027   approach the problem of extracting a full description of the computations implemented by a neural
028   network and displaying it in a human-readable form. We propose to train a neural network (the
029
      meta-model) to produce a full description of the algorithm implemented in a small transformer
      encoder (the base model) when given the base model’s parameters as input.
030
031   A challenge for methods aiming to extract an algorithm description from a base model is that we
032   typically do not have access to the ground truth algorithm. Thus it is difficult to evaluate or train a
033   method for extraction. To overcome this challenge, we introduce a dataset of 1.6 million base models
034   that implement known programs. We leverage RASP (Weiss et al. 2018), a programming language
035
      designed as a computational model for transformers, and Tracr (Lindner et al. 2023), a compiler that
      compiles RASP programs to transformer weights.
036
037
      Contributions:
038
039          • We design rasp-gen, a sampler that generates valid RASP programs, and use it to construct
040            a dataset consisting of 1.6 million RASP programs and corresponding model weights.
041            (Section 3)
042          • We train a transformer meta-model to recover RASP programs directly from model weights.
043            (Section 3)
044
045   The trained meta-model accurately recovers RASP programs 60% of the time on an i.i.d. test set. The
046   meta-model is also able to recover a hand-written sorting algorithm, not generated by the program
047
      sampler and confirmed to not be present in the training set (Figure 2).

081   Elementwise mappings (MLP layers). RASP programs can implement arbitrary elementwise

090   RASP uses Select and Aggregate operations: given a boolean predicate predicate and two
091   sequences a and b, the Select operation returns a boolean ‘selector’ matrix:

093
094   To reduce a selector matrix to a sequence, the Aggregate operation takes as input a selector and
095   a sequence, and returns the average of the input sequence weighted by the nonzero elements in the
096
      selector.

101   Tracr. Tracr (Lindner et al. 2023) is a compiler for a large subset of RASP. Given a RASP program,
102   Tracr outputs a set of transformer weights that implement the RASP program. To compile a RASP
103   program, Tracr first computes the value set (i.e. the set of possible values) of sequences in intermediate
104
      layers using a Python implementation of RASP, then converts each RASP operation into either an
      MLP layer or an attention head that implements the same mapping. Where possible, layers are then
105
      merged and stacked, forming a sequence of MLP and multi-head attention layers.
106
107   Tracr distinguished two kinds of sequences: categorical and numerical. Categorical sequences are
      assumed to be discrete-valued. During compilation, Tracr transforms RASP operations on categorical

      3.1   T RAINING A DECOMPILER FOR T RACR
141   We generate a dataset of 1.6 million RASP programs compiled using Tracr and train a meta-model to
142   map transformer weights directly to RASP code, effectively training a decompiler for Tracr. This
143   experiment functions as a proof of concept to show that meta-models are able to reverse-engineer
144   algorithms implemented in compiled transformers.
145
146   Sampling RASP Programs In order to generate our dataset, we need to sample ran-
147   dom RASP programs.              To sample a program, we sequentially sample an operation
148   from the set {Map, SequenceMap, LinearSequenceMap, Select, Aggregate,
149   SelectorWidth} while keeping track of available sequence variables, starting from the two
150
      input sequences (input tokens and positional indices). To make sure that sampled programs are
      nontrivial we filter out programs that are constant or equal to the identity on a set of test inputs. We
151
      further filter out programs in the subset of RASP not supported by the Tracr compiler. After we finish
152
      sampling and filtering programs, the resulting programs are tokenized and deduplicated.
153
154   Tokenizing. In order to cast decompilation as a sequence prediction task, we tokenize the RASP
155   language via a vocabulary of 105 tokens consisting of variable names, operations, encodings, a set of
156   60 possible elementwise mappings, and a set of possible predicate functions. Since every Select
157   operation is always followed either by an Aggregate or a SelectorWidth, we fuse Selects
158   with the subsequent operation when tokenizing; that is,
159    SelectAggregate(x, y, pred, z) = Aggregate(Select(x, y, pred), z)
160

195   Base model dataset. We generate a dataset of 1.6 million RASP programs via the procedure
196   described above and use Tracr to compile every program to a set of transformer weights. This results
197   in a dataset consisting of tuples (P, W ), where P is a RASP program and W is the corresponding set
198   of transformer weights. We deduplicate this dataset after generation. Generated programs contain
199   between 4 and 9 SOps (sequence operations), and the compiled transformers are between 3 and 10
200   layers deep. Every compiled transformer contains between 600 and 65,563 weights.
201   Compilation using Tracr is computationally cheap; compiling a single model takes under five seconds
202   on average on a single CPU, and to generate the full dataset we used approximately 1000 CPU-hours
203   (CPU cores × hours worked). This stands in contrast to the cost of training thousands of base models
204   as typical for previous work on meta-models (Eilertsen et al. 2020; Schürholt, Kostadinov, et al.
205   2021). For instance, it cost us 1200 A100-hours to generate the dataset in Appendix A.
206
207   Meta-model training. We cast decompilation as a supervised next-token prediction task. For
208   input to the meta-model, we flatten the base model weights and pad them to a fixed length m, then
209   reshape them into an array w ∈ Rm/d×d where d is the embedding dimension of the meta-model.
210   The tokenized RASP operations are padded to a fixed length r and embedded via a linear layer as is
211
      standard in language modeling. We then concatenate the weights and the RASP program, resulting in
212
      an input array x ∈ R(m/d+r)×d . In our experiments we pick m = 65, 536, d = 256, and r = 128.
213   We train the meta-model to predict the next token in the RASP program via a standard cross-entropy
214   loss. At test time, we generate an entire RASP program autoregressively: we condition the trained
215   model on a set of base model parameters and perform consecutive model calls to generate a RASP
      program.

      Uniqueness of compilation and decompilation. In general, different sets of model weights can
217   implement the same function, e.g. due to symmetries in MLP layers. Similarly, the Tracr compiler
218   may return different sets of weights given the same RASP program, since numeric MLP operations
219   are compiled via a piecewise linear approximation found by solving a nondeterministic optimization
220   problem. In the other direction, two distinct RASP programs may compute the same function. If
221   this is the case, the Tracr compiler does not guarantee that two RASP programs compile to distinct
222   models. Thus it is possible that in some situations, our decompiler must choose between two RASP
223   programs that both validly describe the base model weights. However, we have not been able to find
224   such cases in our dataset.
225
226   Results. Our results are reported in Figure 3, and we display an example of a short reconstructed
227   program in Figure 2. We evaluate the meta-model on a i.i.d. test set of programs which we split
228
      off after deduplication, so it is guaranteed to consist of unseen examples. On this test dataset the
      decompiler is able to decompile 60% of programs without errors. On a per-token level it achieves
229
      an accuracy of 98.3%; a tokenized RASP program typically consist of between 30 and 60 tokens.
230
      Unsurprisingly, the accuracy degrades significantly with program length, dropping from 80% on
231   programs consisting of 5 operations down to 26.3% for programs consisting of 8 operations. We also
232   evaluate on a handcrafted RASP program that sorts an input sequence (Figure 2), which we ensure is
233   unseen during training.

      3.2   D ECOMPILING FROM NON - SPARSE WEIGHTS
236
237   Weights obtained by compiling a RASP program via Tracr are dissimilar from weights obtained via
238   training by gradient descent. Not only are Tracr-compiled models highly sparse, they also represent
239   sequence variables (i.e. internal activations) in a disentangled fashion, as every RASP variable is
240   represented by a separate linear subspace in the residual stream.
241   In fact, Friedman et al. (2023) show that a significant subset of Tracr-compiled models (those
242   consisting only of categorical sequence variables) can be mapped to RASP code via a hand-crafted
243   algorithm. While our dataset is more challenging, as it includes compiled models that operate on
244   numerical variables, it is clear that decompiling Tracr is an easier problem than extracting algorithms
245   from trained transformers in general. To account for this sparsity problem, we run a second experiment
246   to show that our meta-model is still able to recover accurate RASP programs from non-sparse models.
247   The key to our approach is the observation that it is possible to apply a linear transformation to
248   transformer weights without modifying the output. If A ∈ Rd×d where d is the dimension of the
249   residual stream, then consider modifying a model such that it applies A to activations before writing
250   them to the residual stream and A−1 before reading from it, leaving the final output unchanged.
251   Since every layer reads from and writes to the residual stream linearly, it is enough to multiply every
252   weight matrix by A or A−1 as appropriate, resulting in a new set of transformer weights. Given a
253   transformer with sparse weights, we can therefore construct a set of dense weights with the same
254   outputs by sampling a random orthogonal matrix and applying it to the sparse weights.
                                                                   ′
255   Finally, we use PCA to learn a linear projection B ∈ Rd×d to compress the original activations of
256   size d to a smaller dimension d′ < d. We apply B in the same way as the sampled orthogonal matrix,
257   multiplying weights by B or B T as appropriate. This does not necessarily leave the output fully
258   unchanged, but if d′ is not too small the outputs of the compressed model are equal on > 99% of
259   inputs.
260   The purpose of this compression is to ensure that activations are not disentangled. In Tracr, RASP
261   sequence variables are represented as orthogonal directions in the residual stream of a compiled
262   model. In contrast, a common observation in trained transformers is that a model learns to make use
263   of more features than can be orthogonally represented. Thus compressing the residual stream helps
264   make our testbed more realistic.
265
266   Base model dataset. We construct a dataset of 780, 000 program-model pairs (P, W ) as in Sec-
267   tion 3.1, keeping program length fixed at 5 RASP operations. For every datapoint, we then apply
268   the de-sparsification procedure described above: we first multiply weights by a random orthogonal
269   matrix, and then create a set of compressed weights W ′ by applying a compression matrix obtained
      via PCA.

295   4   L IMITATIONS
296
297   Models obtained by Tracr are dissimilar from trained models The models we train on tend to
298   be compiled from simple RASP programs with no more than a few (1-5) RASP operations per layer
299
      and less than 10 total. It is likely that most transformers trained in realistic settings do not have a
      short representation in RASP.
300
301
      Training dataset. We have chosen a task such that it is easy to generate a training dataset for
302
      the meta-model, and for which a loss function is easily evaluated. For Tracr in particular it is
303
      computationally cheap to generate hundreds of thousands of programs, and a ground truth explanation
304   is readily available via the RASP program. It is likely to be harder to generate training data for
305   real-world interpretability tasks. In addition, our meta-models tend to be larger than the base models
306   they are trained on by about a factor of 10-100, which would be prohibitive for very large base
307   models.
308
309   We use a black box to interpret a black box. Interpretability research can broadly be classified
310   into two kinds of approaches: those that generate explanations, and those that verify explanations.
311   We show that meta-models can be used to generate explanations, but do not address the problem
312   of verifying the explanations produced by the meta-model. Without any means of verification, this
313   approach cannot provide guaranteed assurances about the base models analyzed.

      Meta-models. While to our knowledge we are the first to use the term meta-model in a paper, the
318   idea of using neural networks to operate on neural network parameters is not new. A line of work
319   focuses on training an autoencoder meta-model on a dataset of neural network weights (Schürholt,
320   Kostadinov, et al. 2021; Schürholt, Knyazev, et al. 2022). The trained encoder can be used as a feature
321   extractor to predict model characteristics (such as hyperparameters), and the decoder can be used
322   to sample new weights, functioning as an improved initialization scheme. In earlier work, Eilertsen
323   et al. (2020) train a meta-model to predict base model hyperparameters such as learning rate and
      batch size. Our meta-model architecture is simpler and outperforms both works on all tasks we tested

324
      (Appendix A). Although we improve on the state-of-the-art, we don’t include these results in the
325   main text because our focus is on automating interpretability rather than hyperparameter prediction.

327
      Extraction. Weiss et al. (2018) algorithmically extract a representation of an RNN as a finite state
328
      automaton. This is similar to our work because we are also interested in extracting a full description
329   of the computation performed by a transformer (Section 3); the main difference is that we learn
330   an extraction method (rather than using a fixed algorithm), and that we work with compiled rather
331   than trained models. Other works that have extracted programmatic representations of functions
332   implemented by trained neural networks include Cai et al. (2017) and Mikulik et al. (2020). More
333   recently, Friedman et al. (2023) show it is possible to deterministically extract a RASP-like description
334   from transformer parameters trained to operate on categorical variables in a fashion explicitly inspired
335   by Tracr.
336
337   Hypernetworks. Hypernetworks (Ha et al. 2017) are neural networks that generates the weights of
338   another network (usually called the ‘main’ network). Typically, a hypernetwork takes a layer index
339   and other layer information as input and outputs the weights for that layer, thus achieving a kind
340   of relaxed weight sharing between layers. One trains a hypernetwork by jointly back-propagating
341   through the main network and the hypernetwork. Hypernetworks are related to meta-models in that
342
      they operate on weights directly. They are different in that hypernetworks return weights as output,
      whereas meta-models take weights as input.
343
344
345   Interpretability. The field of interpretability studies the workings of machine learning models,
346
      with the goal of making the outputs and behaviour of these models more understandable to humans
      (Doshi-Velez and Kim 2017; Lipton 2018). While there is no universally agreed-upon definition
347
      of interpretability, in the context of this work, we focus on the particular problem of mechanistic
348
      interpretability, which aims to fully reverse engineer the learned algorithm implemented by a neural
349   network. Despite the supposed black-box nature of neural networks, the field has had some noteworthy
350   successes understanding network internals (Cunningham et al. 2023; Bricken et al. 2023), in one
351   setting fully understanding the exact algorithm implemented by a network (Nanda et al. 2023).
352   However, so far these successes have mostly been restricted to relatively small models, and either
353   only consider models trained on toy tasks or limited aspects of a model’s behavior. Other recent
354   work on mechanistic interpretability includes tracking chess knowledge in AlphaZero (McGrath et al.
355   2022), locating a circuit responsible for a specific grammatical task in GPT-2 (Wang et al. 2022), and
356   the study of superposition in transformers (Elhage et al. 2022).
357   There have been a number of proposed approaches to automating mechanistic interpretability,
358   including automated circuit ablation (Conmy et al. 2023) and verification of circuit behavior (Chan
359   et al. 2022). Both of these works study automatic verification of hypotheses, but don’t propose a
360   method for automatic generation of hypotheses. A different approach to automated interpretability
361   is to use LLMs to annotate neurons based on dataset examples (Bills et al. 2023; Foote et al. 2023).
362   While this allows for the automatic generation of hypotheses, these hypotheses are written in natural
363
      language and thus hard to verify and likely unreliable.

365   6    F UTURE W ORK
366
367   Our work provides a first proof-of-concept for the approach we propose: using meta-models to
368   automate aspects of mechanistic interpretability. A number of challenges remain before this approach
369   can be applied practically.
370
371
      Scaling Meta-Models. A challenge in training meta-models is that training data is either synthetic
372   and thus potentially unreliable (such as Tracr-compiled models), or expensive to generate (such as
373   when generating a large dataset of trained base models). This is especially problematic for large state-
374   of-the-art models (e.g. LLMs), since training hundreds or thousands ‘frontier’ models is not feasible.
375   There are a number of potential avenues to effectively scaling up meta-models to process large input
376   models. Questions include: (1) Can large-scale pre-training on a base model zoo (e.g. doing masked
377   weight prediction, or contrastive learning) improve performance? (2) Can a meta-model trained on
      smaller base models generalize to larger base models, implying that neural circuitry is consistent


                                                         7
      Under review as a conference paper at ICLR 2025


378
      across scale? (3) Can meta-models be readily applied to problems that only require processing a
379   small part of a base-model at a time?
380
381   Transformer Reverse-Engineering. Tracr-compiled models are relatively sparse compared to
382   trained transformers. We suggest a couple steps to approach general transformer reverse-engineering.
383   (1) Can meta-models reverse-engineer transformers obtained from a more realistic variant of the Tracr
384   compiler featuring a compressed residual stream and SGD-trained weights? 1 (2) Can a meta-model
385   trained on Tracr-compiled models generalize to transformers trained on the inputs and outputs of
386   similar RASP programs? If transfer from Tracr-compiled models is helpful, it may be possible to
387   cheaply generate large training sets for meta-models.
388
389   Creating Hypotheses for Causal Scrubbing. Causal scrubbing (Chan et al. 2022) is a technique
390   for evaluating whether a simplified, human-legible computational graph is an accurate model of a
391
      given neural network circuit. Can a simple dataset be constructed with one-to-one pairs of (network
      circuit, equivalent computation graph)? Can a meta-model be trained on this dataset and learn to map
392
      circuits to mechanistic explanations?
393
394
      Classifying Attention Heads in LLMs. Recent work in mechanistic interpretability has associated
395   specific functions to attention heads in LLMs.2 Can a meta-model be trained to identify the functions
396   of attention heads using relatively few labeled examples? Operating on one head at a time has
397   numerous benefits, as the meta-model need only process a small part of the input model, and a single
398   large input model can produce many labeled training examples.
399
400   Automated Verification of Interpretations. Can a meta-model be trained to output not only a
401   programmatic description of the base model, but also evidence or proof that this description is
402   accurate? One approach would be to train a meta-model to adversarially suggest examples which
403   might disprove any proposed equivalence between a model and an interpretation.
404
405   7       C ONCLUSION
406
407   Scaling is currently a major bottleneck for mechanistic interpretability. The current state-of-the-art
408   requires substantial human labor by researchers to understand a model, and may remain infeasible
409   for many large models in deployment. We propose to use transformers, which show favorable
410   performance scaling, as “meta-models”—models that take other models weights as input—that
411   can be trained to perform interpretability tasks. The method is general: we apply it to generating
412   human-readable code from neural networks, but it is very flexible; for example in Appendix A we
413   apply our meta-model to the task of predicting hyperparameters from weights. Despite its generality,
414
      it performs well: beating prior work on both hyperparameter prediction and successfully recovering
      the majority of RASP instructions from Tracr-compiled transformer weights.
415
416   Our work indicates the potentially broad applicability of meta-models in the circumstances where it
417   is possible to construct an appropriate supervised training set of models and interpretations. Future
418   work may extend meta-models to more complex and more useful tasks.

       A              C OMPARISON WITH PRIOR META - MODEL WORK
560
561   Past work has applied meta-models (that is, neural networks that take weights of other neural networks
562   as input) to a variety of tasks. To sanity check our choice of meta-model architecture as well as our
563   methods for preprocessing network parameters for model input, we compare against Eilertsen et al.
564   (2020) (henceforth EJR) and Schürholt, Kostadinov, et al. (2021) (henceforth SKB), who both train a
565   meta-model to predict hyperparameters used to train base models.
566   The main difference between our meta-model and EJR/SKB is that we use a simple transformer
567   encoder as meta-model. In order to compare against EJR/SKB, we adapt our meta-model to the
568   classification setting by removing all causal attention masks and attaching a single linear layer to the
569   output at position 0, from which we decode the logits for classification.
570
      EJR use a CNN meta-model to predict (from the base model weights) the dataset, batch size,
571
      augmentation method, optimizer, activation function, and initialization scheme used to train the base
572   model. They use two datasets: one where the architecture (and thus the size) of the base models are
573   fixed, and another where the base models have variable size. We recreate their second dataset as it is
574   the more general setting. We follow their dataset generation procedure, training CNNs with random
575   selections of the hyperparameters listed above.
576
      The setting of SKB is similar but differs in a few respects. SKB use a fixed model size for the base
577
      models, a smaller set of hyperparameters for classification, and an autoencoder architecture as the
578   meta-model. The autoencoder is first pre-trained in an unsupervised manner to reconstruct neural
579   network weights. After pretraining, the encoder plus an extra linear layer is fine-tuned to perform the
580   classification task. While pretraining on large datasets is a promising direction, we chose to train a
581   classifier directly.
582
583
584   Base model dataset. We create two base model datasets corresponding to the experimental setups
585   in EJR and SKB respectively. For comparison to EJR, we train 10,000 CNNs while randomizing the
586   model size, dataset, batch size, augmentation method, optimizer, activation function, and initialization
587   scheme used to train the base model. For comparison to SKB, we replicate their dataset construction
588   and train 30,000 CNNs on each of MNIST, FashionMNIST, CIFAR-10, and SVHN while randomizing
589   the optimizer, activation function, and initialization scheme used to train the base model.
590   We match the dataset size and composition for both EJR and SKB, including randomizing hyper-
591   parameters in the same way. The only difference in our setup is in the comparison to EJR, where
592   we use fewer augmentations when training the CNN base models. This is because EJR use an large
593   set of augmentations that is hard to replicate. We discuss this difference more in the appendix. We
      also provide more details on both base model datasets in the appendix. Training the CNNs for

596
597   Meta-model training. For every hyperparameter (activation, batch size, and so on), we train a
598   meta-model (a decoder-only transformer) to classify base model (input) weights based on the values
599   of the hyperparameter. For example, the meta-model might predict the kind of activation function

607

      B.2   DATASET PREPROCESSING
682
      Recall that our base model dataset consists of 1, 6 million datapoints, where each datapoint is a tuple
683
      (p, w) where p is the tokenized rasp program (an integer vector of length r = 128) and w is the
684   corresponding set of transformer weights (a float vector of length m = 65, 536).
685
686   At dataset generation time, we filter out all base models larger than m parameters (this is less than 1%
687
      of all models). Before model input, we treat each set of weights as a vector of length m, padding to
      length m if required (we use the pad value 0.05). Similarly, we filter out all datapoints with a RASP
688
      program longer than r when tokenized.
689
690   When Tracr compiles a RASP program, a small subset of parameters in the compiled model can be
691   very large (>1000). For this reason, we preprocess the weights array with a symmetric log-transform
692   that is linear close to the origin:
693                                         (
                                          ′   sign(w) log(|w|) if |w| > 2
694                                     w =
695                                           w · (log 2)/2     otherwise.
696
697   This transformation is chosen to be continuous, linear in the region [−2, 2], and symmetrically
698
      logarithmic elsewhere.
699   For input to a meta-model with an embedding dimension of width d, we reshape the weights vector
700   of every example to shape (m/d, d). In our case, d = 256. We embed the tokenized RASP program
701   p via a linear layer as is standard in language modeling. We then concatenate the weights and the
      RASP program, resulting in an input array of shape (m/d + r, d).
