009                                                A BSTRACT
010
011
               As deep learning models become larger and more expensive, many practitioners
012            turn to fine-tuning APIs. These web services allow fine-tuning a model between
013            two parties: the client that provides the data, and the server that hosts the model.
014            While convenient, these APIs raise a new concern: the data of the client is at
015            risk of privacy breach during the training procedure. This challenge presents
016            an important practical case of vertical federated learning, where the two parties
017            perform parameter-efficient fine-tuning (PEFT) of a large model. In this study, we
018            systematically search for a way to fine-tune models over an API while keeping the
019            labels private. We analyze the privacy of LoRA, a popular approach for parameter-
020
               efficient fine-tuning when training over an API. Using this analysis, we propose
               P3 EFT, a multi-party split learning algorithm that takes advantage of existing PEFT
021
               properties to maintain privacy at a lower performance overhead. To validate our
022
               algorithm, we fine-tune DeBERTa-v2-XXLarge, Flan-T5 Large and LLaMA-2 7B
023            using LoRA adapters on a range of NLP tasks. We find that P3 EFT is competitive
024            with existing privacy-preserving methods in multi-party and two-party setups while
025            having higher accuracy.

      1    I NTRODUCTION
030
031   One of the main reasons behind deep learning success is its ability to transfer knowledge between
032   tasks (Tan et al., 2018). When training a model for any particular problem, it is common to reuse
033
      previously trained models from other, related problems. In the past, this was typically done by
      downloading pre-trained model weights from public hubs, then fine-tuning the said models on the
034
      downstream task. However, as models grow larger and more compute-intensive, fine-tuning them
035
      locally becomes an increasingly difficult task. Furthermore, many recent models are not released, but
036   instead made available as proprietary services.
037
038   When a model cannot be fine-tuned locally, many practitioners opt instead for the so-called fine-tuning
039
      APIs (OpenAI, 2024; Hugging Face, 2024; Dreambooth API, 2024; OctoAI, 2024). These APIs
      are web services that host one or several pre-trained models and allow clients to perform limited
040
      fine-tuning. More specifically, APIs usually allow their clients to run parameter-efficient fine-tuning
041
      (PEFT), such as LoRA (Hu et al., 2022) or Prefix-tuning (Li & Liang, 2021). These techniques allow
042   adapting a model to a dataset while training a relatively small number of additional weights, which is
043   particularly important for large language or image generation models that have billions of parameters.
044
045
      Although the fine-tuning APIs can be convenient, they also introduce new risk in terms of data privacy.
      When a client uses such API to train on sensitive data, they need to ensure that their data will stay
046
      private Duan et al. (2023). This is particularly important when dealing with patient’s medical records,
047
      personal user data or trade secrets McMahan et al. (2017); Li et al. (2021). The two main threats to
048   data privacy are that the API provider obtains the private data and that a third party intercepts data in
049   transit. Therefore, data privacy is not guaranteed even if the API provider is trusted. Several recent
050   works propose LLM fine-tuning protocols that establish a certain level of privacy for multi-party
051   fine-tuning Xiao et al. (2023); Duan et al. (2023); Li et al. (2023b). Unfortunately, these algorithms
052   work for a narrow class of fine-tuning algorithms, specifically prompt-tuning, or assume that a
053   client can run LLM training locally using an obfuscated version of the model, provided by a remote
      server Xiao et al. (2023). As a result, these algorithms are impractical for our use case of fine-tuning


                                                         1

054
      over an API. The few algorithms that are suitable for API fine-tuning guarantee the privacy of input
055   tokens Li et al. (2023b), meaning that the attacker can infer private training labels.
056
057   In this work, we seek to alleviate this problem by designing a two-party fine-tuning protocol that
058
      performs standard parameter-efficient fine-tuning with privacy guarantees. We formulate our protocol
      as a special case of split learning (or vertical federated learning), where one side (server) holds the
059
      pre-trained model and the other (client) has private training data. More specifically, we focus on the
060
      privacy of client’s training labels. While input privacy is often crucial, there are scenarios where
061   input data is publicly available, such as social media user pages. In these cases, labels could include
062   ad clicks (known to the social network) or financial information (known to a bank that matches social
063   profiles to its internal records). This example further justifies the use of LLMs, as social media pages
064   often contain substantial amounts of text, and LLMs excel at processing long-context data.
065
      Instead of developing a specific privacy-preserving architecture, we seek algorithms that can work
066
      with popular existing models and PEFT algorithms. Furthermore, our approach relies on the properties
067   of parameter-efficient fine-tuning. Notably, since the adapters are compact, both parties can maintain
068   multiple sets of adapters and swap between them with relative ease. This allows us to design a
069   PEFT-specific algorithm that can solve its task more effectively than general split learning strategies
070   Li et al. (2022).
071
      We summarize our main contributions as follows:
072
073   • We analyze Low-Rank Adaptation, a common parameter-efficient fine-tuning algorithm, from the
074     perspective of label privacy in the split learning setup. We observe that, despite fine-tuning less than
075     0.1% of model parameters, PEFT algorithms leak client’s training labels against simple attacks that
076     work for modern pretrained transformers.
077   • Based on our analysis, we formulate a framework for privacy-preserving parameter-efficient fine-
078     tuning (P3 EFT). This framework leverages the properties of PEFT to obfuscate the gradients and
079     parameters communicated during fine-tuning with little impact on the fine-tuned model quality.
080
      • To verify the practical viability of P3 EFT, we conduct experiments on popular real-world PEFT
081
        workloads1 . Specifically, we fine-tune DeBERTa-v2-XXL (He et al., 2021), Flan-T5-Large (Chung
082     et al., 2022) and LLaMA-2 7B (Touvron et al., 2023) on a set of standard language understanding
083     problems. We find that, compared to prior split learning algorithms, P3 EFT can maintain label
084     privacy throughout training with a significantly smaller accuracy drop.

      2.1      F EDERATED LEARNING AND SPLIT LEARNING
090
      Privacy preservation in machine learning has been a subject of active study within several frameworks.
091   An important branch of privacy-preserving learning methods is federated learning, or FL (McMahan
092   et al., 2017), which can be broadly described as an approach allowing several parties to train a model
093   jointly without sharing their private data. In particular, vertical federated learning (Hardy et al., 2017;
094   Yang et al., 2019) targets the scenario where different features (including the label) of each training
095   instance are kept by different parties.
096
      One of the most popular approaches to vertical FL for neural networks is split learning (Gupta &
097   Raskar, 2018; Vepakomma et al., 2018), where each party stores its part of the overall model. To train
098   the model in such an approach, it is only necessary to transfer the intermediate activations and the
099   gradients between layers, while the data itself is stored at the premises of the participant hosting each
100   layer. In this work, we focus on the two-party formulation of split learning, where one side stores the
101   features for each example and another one stores the labels.
102
      Recent works have investigated the setting of two-party split learning from the label leakage per-
103   spective (Vepakomma et al., 2019; Pasquini et al., 2021): because the label party needs to pass the
104   gradients of the loss function to the non-label party, it is possible for the latter party to deduce the
105   labels by inspecting the gradients or activations or by hijacking the training procedure. Li et al. (2022)
106   provide a set of attack methods that allow recovering private labels and propose a defense mechanism

      2.2   PARAMETER - EFFICIENT FINETUNING
112
113   The majority of large neural networks today are not trained with a specific task in mind: instead, they
114   are pretrained on a general objective and then adapted for the downstream problem. Importantly, the
115   growth in the size of foundation models has led to the increased popularity of parameter-efficient
116   finetuning (PEFT) methods that adapt the model to a given task by training a small number of
117   task-specific parameters. There are several prominent approaches to parameter-efficient finetuning,
118   ranging from trainable prompts (Li & Liang, 2021; Hambardzumyan et al., 2021), to residual
119
      adapters (Houlsby et al., 2019; Pfeiffer et al., 2021). We focus on Low-Rank Adaptation (or
      LoRA, Hu et al., 2022), one of the most popular PEFT methods that adds extra parameters to each
120
      weight matrix in the form of a low-rank factorization (see Appendix D for a more detailed description).
121
      Such formulation allows LoRA adapters to be merged into the original weights after finetuning; this
122   ability, combined with the simplicity of the method, has made LoRA a broadly popular approach in
123   multiple domains. Still, the approach we propose can be applied to any PEFT method.
124
125
      Several recent lines of work explore the problem of fine-tuning LLMs with privacy guarantees Yu
      et al. (2022); Shi et al. (2022). Zhao et al. (2023) analyze the viability of prompt tuning for federated
126
      learning, and Zhang et al. (2023); Liu et al. (2023) study PEFT algorithms in the setting of horizontal
127
      federated learning, that is, where multiple users train a shared model on their local private data.
128   Another, more relevant research direction considers private fine-tuning in a vertical federated learning
129   scenario, where participants hold different model layers Li et al. (2023b); Wang et al. (2023); Wan
130   et al. (2023). Most of these studies leverage the idea of differential privacy to prove an upper bound on
131   how much information is leaked Dwork (2006). Unfortunately, these upper bounds are typically loose
132   and do not match practical observations for real models (see results and discussion in Section 4.2
133   and Appendix B). Finally, Xiao et al. (2023) presents an alternative algorithm that protects client data
134   by running the entire fine-tuning on client side by emulating the server-side model layers. While
135   this approach is more holistic, it assumes that clients can run fine-tuning locally, which makes it
136
      impractical for many real-world users of LLM fine-tuning APIs. The primary distinction between
      our work and these studies is that we investigate parameter-efficient adaptation in the setting of split
137
      learning: we aim to finetune a model without disclosing the labels of examples to the model provider.

140   3     P RIVACY- PRESERVING PARAMETER - EFFICIENT FINE - TUNING
141
      In this section, we analyze the privacy of parameter-efficient fine-tuning and propose a protocol for
142   two-party parameter-efficient fine-tuning with the desired privacy guarantees. We begin by analyzing
143   the privacy of API fine-tuning with popular PEFT algorithms in Sections 3.1 and 3.2. Then, in
144   Section 3.3, we formulate a protocol for privately computing gradients over fine-tuning APIs. Finally,
145   we formulate the full P3 EFT protocol in Section 3.4.
146
147   3.1   S ETUP
148
      To analyze the privacy of API fine-tuning, we first need to formulate a common framework for
149
      this type of APIs and develop private learning protocols. This step is important, because existing
150   fine-tuning APIs greatly vary in what they offer to the client: from closed APIs that require users to
151   submit their full training data OpenAI (2024) to more flexible APIs where clients can run individual
152   training steps (Li et al., 2023a; Borzunov et al., 2022; Rao et al., 2021). Similarly to most existing
153   works on split learning, we focus on the latter type of APIs that allows clients to run individual
154   forward and backward passes over a remote model. Thus, a client can use these APIs to obtain the
155   training gradients for their PEFT adapters, then update adapters locally with any optimization method.
156   In our work, we adopt this archetype of fine-tuning API as it offers sufficient flexibility to develop
157   privacy-preserving algorithms.
158   We formulate fine-tuning over an API for two or more parties: a client, and one or several servers.
159   The client owns a training dataset with inputs X and labels Y . In turn, each server has the same
160   pre-trained model h(xi , θ) ∈ Rd . Note that the parameters θ denote not the pre-trained model
161   weights, but the trainable adapter weights for a certain PEFT algorithm. A model can encode an input
      xi ∈ X and produce a d-dimensional vector of activations that depend on the adapter weights θ.

      Before building more advanced algorithms, let us analyze the privacy of client’s labels under standard
195   fine-tuning. We consider an “honest, but curious” attacker model. This means that the server will
196   faithfully run the forward and backprop computations as requested by the client without changing
197   the results. Furthermore, we assume that servers are independent and do not communicate client’s
198   data between each other. However, a server can recover client’s labels by performing arbitrary
199   computations using any information it receives from the client. When training in this way, a client
200   does not directly communicate training labels to the server. However, it communicates inputs, adapter
201   parameters, and gradients. Furthermore, the server communicates input representations that can be
202   intercepted by a third party.

243   single training step with arbitrary PEFT type. Several prior works approach this by modifying the
244
      training objective or model architecture. However, when dealing with a real-world PEFT workload

245
      accuracy4 . Thus, we seek an algorithm that preserves both model and training objective.

300   The private backpropagation algorithm can allow client to safely compute gradients once, but, in
301   practice, client usually needs to run many consecutive steps. This creates an additional vector of
302   attack: if the same server receives two sets of parameters θt , θt+1 , they could potentially recover gθ
303   by inverting the optimizer.
304   In the simplest case, if the server somehow knows that the client computes θt+1 = θt − η · gθ , then
305   they can compute gθ = (θt − θt+1 )/η. While gθ does not necessarily leak private labels, a server
306   could, in some cases, use gθ to recover gh , either fully (e.g. if Jacobian is invertible), or partially.
307

308
      backprop on two consecutive steps. This is easy to do in decentralized systems where there are
309
      many potential servers. However, even when there is a single server, they could be required to set up
310   multiple trusted execution environments (Nvidia, 2024). A more risky alternative is to ensure that the
311   gradients cannot be reversed from consecutive parameters: randomize initial optimizer statistics or
312   add noise to parameters. This solution is easier, but it can slow down training in some cases.
313
      To summarize, we formulated a procedure that allows a client to compute gradients privately for
314
      any given model and PEFT type. Below, we analyze this procedure in conjunction with activation
315
      protection technique from the next section. However, private backpropagation can also be used
316   separately, to ensure gradient privacy for baseline methods (e.g. Sun et al. (2022)) or potential
317   future algorithms. This approach can fit any vertical federated learning or split learning scenario
318   that requires propagating gradients through an untrusted third party. Furthermore, since Equation 3
319   recovers true gradients, this obfuscation method does not affect the training dynamics.

458   Next, we evaluated the entire P3EFT algorithm. To control tasks and model type, we examined

461
      • Without LoRAs. In this baseline, the client gathers h activations at the beginning (with no
462     adapters), then proceeds to train local “head” layers using these activations. This method cannot
463     leak information about training labels except for what is stored in X.
464
      • Regular fine-tuning (Regular FT) refers to training a single LoRA adapter normally. This baseline
465
        represents an upper bound on model accuracy, but lacks privacy.
466
467
      • Distance Correlation (DC). Our re-implementation of the distance correlation defense formulated

468
469   • Private Split Learning Framework (PSLF) represents another reimplementation of the framework
470     from Wan et al. (2023). The protection mechanism in this method is based on the utilization of
471
        Randomized Response (Warner, 1965) and satistifies the conditions of Label Differential Privacy.
472

473
      measures:
474
475   • Spectral attack AUC — a measure of vulnerability to an attack proposed in Sun et al. (2022),
476     measured as classifier ROC AUC: lower value corresponds to better privacy.
477   • Norm attack AUC — vulnerability to a variant of attack proposed in Li et al. (2022), measured as
478     classifier ROC AUC (lower is better). Despite the initial proposal of this approach for attacking
479     gradients, we observed that it is also well-suited for attacking activations.
480
      • K-means accuracy — vulnerability to clusterization attack, measured in the percentage of correctly
481     clustered activations, lower is better.

484   training period as a measure of privacy leakage, because it is the worst possible scenario that matters

526   5     C ONCLUSION AND D ISCUSSION
527
      In this work, we analyze privacy-preserving fine-tuning of large neural networks in the context of
528   parameter-efficient fine-tuning and the two-party split learning setting. We show that while standard
529   fine-tuning suffers from label leakage even in the parameter-efficient case, it is possible to leverage
530   the efficiency of PEFT to alter the procedure without any significant performance drawbacks. We test
531   the resulting method, named P3 EFT, on a range of pretrained language models and multiple datasets,
532   showing that it is competitive with a strong baseline in terms of label privacy while having higher
533   task performance.
534   In future work, it is natural to explore how this approach can be extended to establish holistic privacy
535   in both labels and inputs. This problem can be approached from two directions: either adapt the
536   ideas of P3 EFT for input privacy, or combine it with an existing work like Li et al. (2023b). Another
537   important direction for future research is exploring the privacy of the long-term client-provider
538   interaction. In a typical real-world use case of API fine-tuning, a client performs multiple training
539   runs on overlapping data and hyperparameters. This could open additional attacks vectors that
      combine information from multiple training runs.

826   We consider significantly stronger assumptions regarding the attacker’s capabilities - namely, a
827   white-box scenario. We assume the server knows the client-side model and, consequently, all possible
828
      2B vectors gh for different label sets. Thus, the server’s task is to determine which of the 2B label
      sets corresponds to the current batch (or at least determine which sets are more or less probable)
829
      based on the transmitted vectors.
830
831   We examine several possible setups and investigate the minimum m required to ensure that all 2B
832   sets remain equally probable from the attacking server’s perspective:
833
834
            1. The case with two non-interacting servers. In this scenario, it suffices to set m = 2 and
               send one vector ξi to each server. From each server’s perspective, all 2B variants have equal
835
               probability because for any ξi and a given gh , there exists a vector η such that gh belongs to
836
               the span of ξi and η.
837
838
            2. Single server case. We demonstrate that in this scenario, it is sufficient to set m = B.
839            To show this, we note that for the b-th element of the batch holds ∂lb /∂hb = ∂lb /∂pb ×
840
               ∂pb /∂hb , where pb ∈ R is the head’s prediction for activations hb - the probability of
               class 1. ∂pb /∂hb is a constant Jacobian of rank 1 and does not depend on the label value.
841
               Thus, both possible vectors ∂lb (y)/∂hb lie in the Jacobian’s image and belong to the same
842
               one-dimensional subspace.
843
               Therefore, it is sufficient for the client to send to the server a basis vector of the corresponding
844
               one-dimensional subspace ∂pb /∂hb for each batch element b (and zero vectors for the
845            remaining batch elements). Knowing αb , the client can reconstruct the corresponding
846            contribution of b-th element to the weight gradient gθ . The server, however, cannot determine
847            which label generated the given gradient for each example, as both lie on the same line.
848
            3. Single server case, m < B. In this setup, the client is not able to protect all gradients of the
849            batch. Indeed, for B = 3 and m = 2, the set of 23 possible gradient combinations cannot
850            be embedded in any 2-dimensional plane that the client can construct from 2 vectors (the
851            linear span of these 23 is 3-dimensional). At most, the client can fully protect m − 1 labels,
852            while the server will know the remaining b − m + 1 labels up to a flip of all labels.
853
854   We want to emphasize again that the above results obtained under a white-box assumption, which is
855   significantly stronger than our practical setup. The general case is considerably more challenging
856   for the attacking side, and developing a possible attack or determining the theoretical limits of the
857
      attacker’s capabilities is a complex task. However, we believe that the theoretical analysis presented
      above may be a good first step in this direction.

860   D    I NFORMAL DESCRIPTION OF L O RA FINE - TUNING
861
862   For convenience, we provide a brief summary of fine-tuning with LoRA (Hu et al., 2022). This PEFT
863   method was originally designed for fine-tuning large pre-trained language models on downstream
      NLP tasks. These language models are typically based on the Transformer architecture (Vaswani


                                                          16
      Under review as a conference paper at ICLR 2025


864
      et al., 2017), where most trainable parameters are allocated to linear layers in multi-head self-attention
865   and feedforward blocks.
866
867   In the first stage of LoRA fine-tuning, user augments the model with adapters. To do so, a user goes
868
      over linear layers in transformer blocks and adds two trainable matrices, A and B that affect this
      layer’s forward pass. Let Wi × x + bi be the original layer with n inputs and m hidden units. Here,
869
      Wi ∈ Rm×n is a pre-trained weight matrix, bi ∈ Rm is a pre-trained intercept vector and x ∈ Rn
870
      represents a vector of inputs to this particular layer. During the forward pass, a layer with LoRA
871   adapter computes Wi × x + bi + Bi × Ai × x, or equivalently, (Wi + B × A) × x + bi . Here, Ai
872   and Bi are two newly added matrices that constitute a LoRA adapter.
873
874
      The adapter matrices A ∈ Rr×n and B ∈ Rm×r have a very small intermediate dimension r. For
      instance, when training GPT-3 with LoRA adapters, authors use 1 ≤ r ≤ 64, whereas the main
875
      weight dimensions are m = n = 12288. The first matrix A is initialized with small random normal
876
      values, and the second matrix B is initialized at zeros. That way, initial A and B do not affect the
877   model predictions.
878
879   Once all adapters are initilized, the user trains all Ai and Bi matrices of the model, while keeping
880
      the rest of the weights frozen. This way, only a small faction (less than 1%) of model weights are
      updated. Once the training is over, the learned adapters Ai and Bi can be merged into the main
881
      weights (Wi := Wi + Ai × Bi ) or used separately.
882
883   LoRA adapters are designed with two objectives in mind: i) to allow fine-tuning models in limited
884   GPU memory and ii) to allow inferencing many fine-tuned models using one inference server. When
885   fine-tuning, LoRA achieves small memory footprint due to the fact that user does not need to compute
886
      gradients (or optimizer statistics) for billions of main model parameters. During inference, a server
      can keep a library of several adapters for different tasks and swap between them on demand.
