009                                                A BSTRACT
010
011            Root cause analysis (RCA) is crucial for enhancing the reliability and performance
012            of complex systems. However, progress in this field has been hindered by the
013            lack of large-scale, open-source datasets tailored for RCA. To bridge this gap, we
014            introduce LEMMA-RCA, a large dataset designed for diverse RCA tasks across
015            multiple domains and modalities. LEMMA-RCA features various real-world fault
016
               scenarios from Information Technology (IT) and Operational Technology (OT)
               systems, encompassing microservices, water distribution, and water treatment
017
               systems, with hundreds of system entities involved. We evaluate the performance
018
               of fourteen baseline methods on LEMMA-RCA across various settings, including
019            offline and online modes, as well as single and multi-modal configurations. The
020            dataset is publicly available at https://lemma-rca.github.io/.

023   1   I NTRODUCTION
024
025   Root cause analysis (RCA) is essential for identifying the underlying causes of system failures,
026   ensuring the reliability and robustness of real-world systems. Recent advancements in artificial
027
      intelligence and software development have led to increased complexity and interdependence in
      modern systems. This complexity heightens their vulnerability to faults arising from interactions
028
      among modular services, which can disrupt user experiences and incur significant financial losses.
029
      Traditional manual RCA, however, is labor-intensive, costly, and prone to errors due to the complexity
030   of systems and the extensive data involved. Therefore, efficient and effective data-driven RCA
031   methods are crucial for pinpointing failures and mitigating financial losses when system faults occur.
032
033
      Root cause analysis has been extensively studied across various domains and settings (Capozzoli et al.,
      2015; Deng & Hooi, 2021; Brandón et al., 2020; Fourlas & Karras, 2021; Gao et al., 2015). Based on
034
      the application scenarios, RCA can be carried out in offline/online fashion with single/multi-modal
035
      system data. Existing studies on RCA in these settings involve numerous learning techniques such as
036   Bayesian methods (Alaeddini & Dogan, 2011), decision trees (Chen et al., 2004), etc. Particularly,
037   causal structure learning based technique (Burr, 2003; Pamfil et al., 2020; Ng et al., 2020; Tank et al.,
038   2022; Yu et al., 2023; Wang et al., 2023a;b; Zheng et al., 2024) has proven effective in constructing
039   causal or dependency graphs between different system entities and key performance indicators (KPIs),
040   thereby enabling the tracing of underlying causes through these structures.
041
      Data is the oxygen of data-driven methods. Despite significant progress in RCA techniques, the
042   availability of large-scale public datasets remains limited, often due to confidentiality concerns (Harsh
043   et al., 2023). This scarcity hinders fair comparisons between RCA methods. Additionally, publicly
044   accessible datasets often contain manually injected faults rather than real faults, and each dataset
045   typically covers only a single domain. These limitations can prevent existing RCA methods from
046   effectively identifying various types of system faults in real-world scenarios, potentially leading to
047   regulatory and ethical consequences in critical sectors.
048   To address these limitations, we introduce LEMMA-RCA, a collection of Large-scalE Multi-ModAl
049   datasets with various real system faults to facilitate future research in Root Cause Analysis. LEMMA-
050   RCA encompasses real-world applications such as IT operations and water treatment systems, with
051   hundreds of system entities involved. LEMMA-RCA accommodates multi-modal data including
052   textual system logs with millions of event records and time series metric data with more than 100, 000
053   timestamps. We annotate LEMMA-RCA with ground truth labels indicating the precise time stamps
      when real system faults occur and their corresponding root-cause system entities.

080

081   and its extensive utility for advanced research in root cause analysis.

084   2    P RELIMINARIES AND R ELATED W ORK
085
086   Key Performance Indicator (KPI) is a monitoring time series that indicates the system status. For
087   instance, latency and service response time are two common KPIs used in microservice systems. A
088   large value of latency or response time usually indicates a low-quality system performance or even a
089   system failure.
090   Entity Metrics are multivariate time series collected by monitoring numerous system entities or
091   components. For example, in a microservice system, a system entity can be a physical machine,
092   container, pod, etc. Some common entity metrics in a microservice system include CPU utilization,
093   Memory utilization, disk IO utilization, etc. An abnormal system entity is usually a potential root
094   cause of a system failure.
095
      Data-driven Root Cause Analysis Problem. Given the monitoring data (including metrics and
096   logs) of system entities and system KPIs, the root cause analysis problem is to identify the top K
097   system entities that are most relevant to KPIs when a system fault occurs. RCA techniques can
098   be implemented in various settings, where offline/online and single-modal/multi-modal are mostly
099   commonly concerned. Offline RCA is conducted retrospectively with historical data to determine past
100   failures, whereas online RCA operates in real-time using current data streams to promptly address
101   issues. On the other hand, single-modal RCA relies solely on one type of data for a focused analysis,
102   while multi-modal RCA uses multiple data sources for a comprehensive assessment. We illustrate the
103   procedure of RCA in single-modal offline and multi-modal online settings in Figure 1.
104   Single-modal Offline Root Cause Analysis (RCA) retrospectively identifies the primary cause of
105   system failures using a single data type after an event has occurred (Wang et al., 2023b; Tang et al.,
106   2019; Meng et al., 2020b; Li et al., 2021; Soldani & Brogi, 2022). For example, Meng et al. (Meng
107   et al., 2020b) analyze monitoring metric data to discern sequential relationships and integrate causal
      and temporal information for root cause localization in microservice systems. Similarly, Wang et


                                                                 2
      Under review as a conference paper at ICLR 2025


108
      al. (Wang et al., 2023b) construct an interdependent causal network from time series data, using a
109   random walk strategy to pinpoint the most probable root causes. Li et al. (Li et al., 2021) evaluate
110   microservice traces, determining that a service with a higher ratio of abnormal to normal traces
111   is likely the root cause. Recently, large language model (LLM) based methods become a new
112   research direction to learn causal relation for root cause identification due to the success of LLMs
113   in performing complex tasks (Chen et al., 2024; Shan et al., 2024; Goel et al., 2024; Zhou et al.,
114   2024; Roy et al., 2024; Wang et al., 2024). For instance, Chen et al. (Chen et al., 2024) introduce
115   RCACopilot, an innovative on-call system empowered by the large language model for automating
116   RCA of cloud incidents. Shan et al. (Shan et al., 2024) propose to first identify the log messages
117
      indicating configuration-related errors and then localize the suspected root-cause configuration
      properties based on the selected log messages and the offered configuration settings by LLMs. Goel
118
      et al. (Goel et al., 2024) demonstrate that LLMs can benefit from service functionality and upstream
119
      dependency information in better reasoning, thus improving the quality of the identification of root
120   causes. Although these studies demonstrate notable efficacy, they rely exclusively on single-modal
121   data, which may lead to suboptimal and biased outcomes in root cause localization.
122
123
      Multi-modal Offline RCA. Recent studies have explored utilizing multi-modal data for offline RCA,
      which can be divided into two approaches (Yu et al., 2023; Hou et al., 2021; Zheng et al., 2024;
124
      Lan et al., 2023). The first approach, exemplified by Nezha (Yu et al., 2023) and PDiagnose (Hou
125
      et al., 2021), involves extracting information from each modality separately and then integrating it for
126   analysis. Conversely, the second approach focuses on the interactions between different modalities.
127   For instance, MULAN (Zheng et al., 2024) develops a comprehensive causal graph by learning
128   correlations between modalities, while MM-DAG (Lan et al., 2023) aims to jointly learn multiple
129   Direct Acyclic Graphs, improving both consistency and depth of analysis.
130
      Online RCA. Despite significant advances, most RCA methods are designed for offline use, requiring
131
      extensive data collection and full retraining for new faults, which delays response times. To address
132   this, Wang et al. (Wang et al., 2023a) introduced an online RCA method that decouples state-invariant
133   and state-dependent information and incrementally updates the causal graph. Li et al. (Li et al.,
134   2022a) developed a causal Bayesian network that leverages system architecture knowledge to mitigate
135   potential biases toward new data. However, these methods are limited to single-modal data, and there
136   is a critical need for online RCA methods that can effectively handle multi-modal data

      3.2   DATA P REPROCESSING
253
254   After collecting system metrics and logs, we assess whether each pod exhibits stationarity, as non-
255   stationary data are unpredictable and cannot be effectively modeled. Consequently, we exclude
256   non-stationary pods, retaining only stationary ones for subsequent data preprocessing steps.
257
      Log Feature Extraction for Product Review and Cloud Computing. The logs of some system
258   entities we collected are limited and insufficient for meaningful root cause analysis. Thus, we
259   exclude them from further analysis. Additionally, the log data is unstructured and frequently uses
260   a special token, complicating its direct application for analysis. How to extract useful information
261   from unstructured log data remains a great challenge. Following (Zheng et al., 2024), we preprocess
262   the log data into time-series format. We first utilize a log parsing tool, such as Drain, to transform
263   unstructured logs into structured log messages represented as templates. We then segment the data
264   using fixed 10-minute windows with 30-second intervals, calculating the occurrence frequency of
265   each log template. This frequency forms our first feature type, denoted as X1L ∈ RT , where T is
266
      the number of timestamps. We prioritize this feature because frequent log templates often indicate
      critical insights, particularly useful in identifying anomalies such as Distributed Denial of Service
267
      (DDoS) attacks, where a surge in template frequency can indicate unusual activity.
268
269   Moreover, we introduce a second feature type based on ‘golden signals’ derived from domain
      knowledge, emphasizing the frequency of abnormal logs associated with system failures like DDoS


                                                                   5
      Under review as a conference paper at ICLR 2025


270
      attacks, storage failures, and resource over-utilization. Identifying specific keywords like ‘error,’
271   ‘exception,’ and ‘critical’ within log templates helps pinpoint anomalies. This feature, denoted as
272   X2L ∈ RT , assesses the presence of abnormal log templates to provide essential labeling information
273   for anomaly detection.
274
      Lastly, we implement a TF-IDF based method, segmenting logs using the same time windows and
275
      applying Principal Component Analysis (PCA) to reduce feature dimensionality, selecting the most
276
      significant component as X3L ∈ RT . We concatenate these three feature types to form the final
277   feature matrix X L = [X1L ; X2L ; X3L ] ∈ R3×T , enhancing our capacity for a comprehensive analysis
278   of system logs and improving anomaly detection capabilities.
279
280
      KPI Construction for SWaT and WADI. The SWaT and WADI sub-datasets include the label
      column that reflects the system status; however, the values within this column are discrete. To
281
      facilitate the root cause analysis, it is beneficial to transform these values into a continuous format.
282
      Specifically, we propose to convert the label into a continuous time series. To achieve this, we employ
283   anomaly detection algorithms, such as Support Vector Data Description and Isolation Forest, to model
284   the data. Subsequently, the anomaly score, as determined by the model, will be utilized as the system
285   KPI. More data preprocessing details on SWaT and WADI can be found in Appendix A

365   the causal graph’s skeleton using an independence test. (2) Dynotears (Pamfil et al., 2020): It
366   constructs dynamic Bayesian networks through vector autoregression models. (3). C-LSTM (Tank
367   et al., 2022): This model utilizes LSTM to model temporal dependencies and capture nonlinear
368   Granger causality. (4). GOLEM (Ng et al., 2020): GOLEM relaxes the hard Directed Acyclic Graph

371
      by detecting abnormal patterns. (7). CORAL (Wang et al., 2023a): An online single-modal RCA
372   method based on incremental disentangled causal graph learning. (8). CIRCA (Li et al., 2022b):
373   This model utilizes structural graph construction, regression-based hypothesis testing, and descendant
374   adjustment to identify root cause metrics. (9). ϵ-Diagnosis (Shan et al., 2019): This model diagnoses

376   ϵ-statistics. (10). RCD (Ikram et al., 2022): This technique hierarchically localizes the root cause of
377   failures by focusing on relevant sections of the causal graph. (11). PCMCI Runge et al. (2019): Thi
      technique combines conditional independence tests with a causal discovery algorithm to infer causal

      The first four baseline models were originally designed to learn causal structures solely from time
      series data. As outlined in (Wang et al., 2023b;a), these causal discovery models can be extended
383
      to identify the root cause nodes. In this process, we first apply causal discovery models to learn the
384
      causal graphs, then utilize random walk with restarts (Wang et al., 2023a) on these graphs to identify
385   the top K nodes as root causes. The last three algorithms are applicable exclusively to metric data.
386   Besides, we extend NOTEARS and GOLEM to the online learning setting, denoted by NOTEARS∗
387   and GOLEM∗ , respectively3 . For the online setting, we use the historical normal data (e.g., 8 hours
388   for the Product Review sub-dataset, and 1 hour for the SWaT and WADI sub-datasets) to construct
389   the initial causal graph and update iteratively for each new batch of data. CORAL can inherit the
390   causations from the previous data batch, while NOTEARS∗ and GOLEM∗ have to learn from scratch

510   5    D ISCUSSIONS
511
512   Broader impact: To facilitate accurate, efficient, and multi-modal root cause analysis research
513   across diverse domains, we introduce LEMMA-RCA as a new benchmark dataset. Our dataset also
514   offers significant potential for advancing research in areas like multi-modal anomaly detection,
515   change point detection, causal structure learning, and LLM-based system diagnosis. Based on
516   the thorough data analysis and extensive experimental results, we highlight the following areas for
517   future research:
518   • Expanding Domain Applications: To enhance the LEMMA-RCA dataset’s versatility and impact,
519      we plan to incorporate data from additional domains such as cybersecurity and healthcare. This
520      integration of diverse data sources will facilitate the development of more comprehensive root cause
521
         analysis technologies, significantly extending the dataset’s applicability across various industries.
522   • Online Multi-Modal Root Cause Analysis: Most RCA methods are offline and single-modal,
523      leaving a gap for real-time, multi-modal approaches. Developing these methods can enable instant
524
         analysis of diverse data streams, essential for dynamic environments like industrial automation and
         real-time monitoring.
525
526
      Limitations: Despite its broad capabilities, the LEMMA-RCA dataset may have limited generaliz-
      ability, as its fault scenarios may not fully capture the diversity of real-world conditions due to factors
527
      like system interruptions and unforeseen circumstances. Additionally, the dependency graphs in our
528
      data are semi-complete, reflecting the inherent challenge of obtaining complete ground-truth graphs
529   in complex systems, which may impact the precision of derived analyses.
530
531
      6    C ONCLUSION
532
533
      In this work, we present LEMMA-RCA, the first large-scale, open-source dataset featuring real
534   system faults across various application domains and multiple modalities. We conduct an inclusive
535   empirical study on LEMMA-RCA by testing the performance of fourteen baseline methodologies
536   under different settings, including offline/online modes and single/multiple-modality data. Our
537   experimental results demonstrate the utility of LEMMA-RCA. By making this dataset publicly
538   available, we aim to facilitate further research and innovation in root cause analysis for complex
539   systems, contributing significantly to the development of more robust and secure methodologies that
      ensure the high performance of modern systems, particularly those that are mission-critical.

      In the original SWaT and WADI datasets, the attack model demonstrates irregular attack patterns,
705   occasionally targeting multiple sensors simultaneously, or executing attacks at closely spaced intervals.
706   To follow the principles of RCA, we have established two specific preprocessing rules for these
707   datasets: 1) Each recorded attack event must only involve a single sensor or actuator. 2) The duration
708   of the dataset corresponding to each attack event must be standardized to two hours. Consequently,
709   we selectively keep attack events that impact only one sensor or actuator. If the interval between
710   successive attack events is insufficiently short, we assume the stability in the monitoring data
711   immediately before and after each attack event. To ensure the necessary two-hour duration for each
712   event, we concatenate normal-state data from both before and after the attack period. This adjustment
713
      positions the attack event centrally within a continuous two-hour segment, facilitating consistent and
      accurate analysis.

      F    D ETAILED D ESCRIPTION OF BASELINES
820
      We evaluate the performance of the following RCA models on the benchmark sub-datasets:
821
822
             • PC (Burr, 2003): The PC algorithm is a data-driven method for causal discovery, producing a
823            partially directed acyclic graph (PDAG) that represents causal relationships among variables.
824            It starts with a fully connected graph and iteratively removes edges based on conditional
825            independence tests, then orients the remaining edges to construct a causal structure. The
826            algorithm assumes the causal Markov property, no hidden confounders, and no cycles in
827            the graph. It is widely used for root cause analysis to identify direct and indirect influences
828            on specific outcomes but is sensitive to the reliability of independence tests and cannot
829            distinguish between equivalent causal structures.
830          • Dynotears (Pamfil et al., 2020): Dynotears is score-based approach for learning these
831            models that scales gracefully to high-dimensional datasets. To accomplish this, the authors
832            cast the problem as an optimization problem (i.e. score-based learning), and use standard
833            second-order optimization schemes to solve the resulting program. Dynotears is based
834            on the recent algebraic characterization of acyclicity in directed graphs, which makes the
835            formulation simple and amenable to different modeling choices.
836          • C-LSTM (Tank et al., 2022): The C-LSTM framework is designed for interpretable nonlin-
837            ear Granger causality discovery in MLPs and RNNs by leveraging the flexibility of neural
838            networks while introducing component-wise architectures to disentangle the effects of
839            lagged inputs on individual outputs. It enhances interpretability and manages limited, high-
840
               dimensional data by applying sparsity-inducing penalties to weight groupings that connect
               input histories to output series. The framework’s sparse component-wise models, such as
841
               cMLP and cLSTM, incorporate group sparsity penalties to effectively select Granger-causal
842
               relationships through the outgoing weights of inputs.
843
844          • GOLEM (Ng et al., 2020): GOLEM (Gradient-based Optimization of dag-penalized Likeli-
845
               hood for learning linEar dag Models) is a likelihood-based structure learning method for
               DAGs that replaces hard DAG constraints with soft sparsity and DAG penalties, enabling
846
               continuous unconstrained optimization. This approach simplifies the optimization problem
847
               while maintaining the ability to learn a DAG equivalent to the ground truth. The framework
848            is validated in both asymptotic and finite-sample regimes, demonstrating its flexibility across
849            various linear models. By avoiding strict constraints, GOLEM is computationally more
850            efficient and theoretically robust for causal discovery.
851
             • REASON (Wang et al., 2023b): REASON is a framework for root cause localization
852
               in complex systems with interdependent network structures. It combines Topological
853            Causal Discovery (TCD) and Individual Causal Discovery (ICD). TCD employs hierarchical
854            graph neural networks to uncover intra- and inter-level causal relationships, modeling fault
855            propagation using a random walk with restarts. ICD focuses on analyzing individual time-
856            series data, using Extreme Value theory to detect abrupt fluctuations and estimate root cause
857            likelihoods, especially for short-lived failures. The framework integrates results from both
858            components to identify system entities with the highest causal scores as root causes.
859          • Nezha (Yu et al., 2023): Nezha is an interpretable and fine-grained root cause analysis
860            (RCA) method for microservices that unifies heterogeneous observability data (metrics,
861            traces, logs) into a homogeneous event format. This representation enables the construction
862            of event graphs for integrated analysis. Nezha statistically localizes actionable root causes at
863            granular levels, such as specific code regions or resource types, offering high interpretability
               to support confident mitigation actions by SREs.


                                                        16
      Under review as a conference paper at ICLR 2025


864
             • CORAL (Wang et al., 2023a): CORAL is an online root cause analysis (RCA) framework
865            that automatically triggers RCA processes and incrementally updates the RCA model.
866            It includes three key components: Trigger Point Detection, Incremental Disentangled
867            Causal Graph Learning, and Network Propagation-based Root Cause Localization. The
868            trigger detection uses multivariate singular spectrum analysis and cumulative sum statistics
869            to identify system state transitions in near-real-time. Incremental causal graph learning
870            decouples state-invariant and state-dependent information to efficiently update the RCA
871            model. Finally, CORAL applies a random walk with restarts on the causal graph to localize
872            root causes, terminating when the causal graph and root cause list stabilize.
873          • CIRCA (Li et al., 2022b): CIRCA is an unsupervised root cause analysis method that
874            formulates the problem as a causal inference task called intervention recognition. Its core
875            idea is to identify root cause indicators by evaluating changes in the probability distribution
876            of monitoring variables conditioned on their parents in a Causal Bayesian Network (CBN).
877            CIRCA applies this approach to online service systems by constructing a graph among
878            monitoring metrics, leveraging system architecture knowledge and causal assumptions to
879            guide the analysis.
880          • ϵ-Diagnosis (Shan et al., 2019): ϵ-Diagnosis is an unsupervised, low-cost diagnosis algo-
881            rithm designed to address small-window long-tail latency (SWLT) in web services, which
882            arises in short statistical windows and typically affects a small subset of containers in
883            microservice clusters. It uses a two-sample test algorithm and ϵ-statistics to measure the
884            similarity of time series, enabling the identification of root-cause metrics from millions
885            of metrics. The algorithm is implemented in a real-time diagnosis system for production
886
               microservice platforms.
887          • RCD (Ikram et al., 2022): RCD is a scalable algorithm for detecting root causes of failures
888            in complex microservice architectures using a hierarchical and localized learning approach.
889            It treats the failure as an intervention to quickly identify the root cause, focuses learning
890            on the relevant portion of the causal graph to avoid costly conditional independence tests,
891            and explores the graph hierarchically. The technique is highly scalable, providing action-
892
               able insights about root causes, while traditional methods become infeasible due to high
               computation time.
