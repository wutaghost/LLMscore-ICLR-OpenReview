009                                               A BSTRACT
010
011            We propose a formal setup of Learning from Language Feedback (LLF) and a new
012            benchmark, LLF-Bench (pronounced as “elf-bench”), to evaluate the ability of AI
013            agents to interactively learn from natural language feedback and instructions. LLF
014            is essential for people, largely because the rich information provided by language
015            feedback can help a learner avoid much of trial and error and thereby speed up the
016            learning process. AI agents, thanks to being powered by Large Language Models
017            (LLMs), can potentially benefit from language feedback during learning like peo-
018            ple do. However, existing benchmarks do not assess this crucial capability. They
019
               either use numeric reward feedback or require no learning at all (only planning or
               information retrieval). LLF-Bench, the benchmark we introduce, is designed to
020
               fill this omission. It is a diverse collection of decision-making tasks that includes
021
               user recommendation, poem writing, navigation, and robot control. LLF-Bench
022            implements several randomization techniques to ensure that the agent actually
023            needs to learn in order to complete these tasks. In addition, LLF-Bench allows
024            configuring the kind of information conveyed by the feedback (e.g., performance
025            assessment, explanations or suggestions), which facilitates studying how agents
026            respond to different feedback types. Together, these features make LLF-Bench a
027            unique research platform for developing and testing LLF agents.

030   1   I NTRODUCTION
031
032   Natural language is an intuitive medium for a person to teach an AI agent, since that is how humans
033   learn from and teach each other. Compared to rewards – the feedback modality typically used in
034   the reinforcement learning (RL) paradigm (Sutton & Barto, 2018) – language feedback can provide
035   rich signals about agent behaviors beyond a quantitative measure of instantaneous performance. For
036   instance, language feedback can explain why the agent’s previous suboptimal behaviors should be
037   avoided, rather than just punishing the agent without giving justification. Language feedback can
038   also provide direct suggestions on how the agent can improve its future behavior, similar to action
039
      feedback used in imitation learning (IL) (Ross et al., 2011; Spencer et al., 2021). However, providing
      action feedback to a robot as has traditionally been done in IL requires a teleoperation setup, which
040
      might not always be feasible. Language feedback, on the other hand, can be given verbally by an
041
      ordinary user (Liu et al., 2023a). In recommendation systems, incorporating user feedback has been
042   studied under coactive learning (Shivaswamy & Joachims, 2015). Reinforcement learning from
043   human feedback (RLHF, (Christiano et al., 2017)) and preference learning (Rafailov et al., 2024)
044   incorporate ranking-based, not verbal feedback.
045
      We capture the essence of using language as a feedback modality in a new learning paradigm –
046
      Learning from Language Feedback (LLF). In an LLF problem, an agent interacts with a task en-
047
      vironment and receives language instructions and feedback. At the start of an episode, the agent
048   is first given a natural language instruction that describes the objective of the task, the rules, and
049   (optionally) side information that may help solve the problem. After executing an action in the en-
050   vironment, the agent receives teacher feedback in natural language, which can be used as a learning
051   signal. LLF generalizes reinforcement learning (RL) from return maximization to general problem-
052   solving. Like RL, LLF focuses on sequential decision problems. However, in contrast to RL, an
053   LLF agent does not receive rewards in numeric form and is not necessarily tasked with maximizing
      returns.

073   Figure 1 shows an example LLF flowchart. LLF replaces RL’s assumption of numeric rewards with

081   In addition to the new learning paradigm, this work’s contribution is LLF-Bench (Learning from
082   Language Feedback Benchmark; pronounced as “elf-bench”), a simulation benchmark designed to
083   evaluate an AI agent’s ability to adapt quickly in LLF settings based on just language feedback.
084   LLF-Bench is a collection of sequential decision-making problems, ranging from item recommen-
085   dation to poem writing to robot control. Each of them has a natural-language description and a
086
      natural-language feedback generator that replaces RL’s rewards as the learning signal (Section 3).
      Additionally, LLF-Bench provides a high-level wrapper that can convert any existing RL environ-
087
      ment with OpenAI Gym interface into an LLF setup (Appendix C).
088
089   Prior to LLF-Bench, several benchmarks have been proposed to evaluate LLM-based agents for
090   decision-making (e.g., AgentBench (Liu et al., 2023b), OpenAGI (Ge et al., 2023), MINT (Wang
091   et al., 2023b), and LMRL Gym (Abdulhai et al., 2023)). However, most tasks in these benchmarks
092
      center around planning and information retrieval problems; only few require the agent to learn and
      adapt beyond what an LLM can already do. Also, many existing benchmarks lack language varia-
093
      tions, so developers might accidentally identify a specific prompt that overfits to a particular verbal
094
      formulation of the task specification. This fails to reflect a key property of LLMs’ real-life use
095   cases, where a user needs LLM-based agents to handle tasks whose solution cannot be directly in-
096   ferred from the task description and has to be learned from interactions and feedback instead (such
097   as “make the title text larger” or “wrap the code with an error-catching block.”). Are LLM-based
098   agents capable of learning from general language feedback? LLF-Bench aims to provide a set of
099   environments to help answer this question while addressing the challenges in reliable LLM agent
100   benchmarking.

      2.1   T HE M ECHANICS OF LLF
109
110   LLF is an abstract learning setup that models the interaction between an agent (e.g., a learning
111   algorithm), a world (e.g., a robot hardware, or a recommendation system backed by a database),
112   and a teacher (e.g., a person). The agent in the LLF setup is asked by the teacher to complete a
113   task in the world via a natural-language instruction. The task’s objective described in the instruction
114   may be different from reward maximization and could include information about how to interpret
115   observations, what the valid actions are, and what tips (such as examples) may help the agent solve
116   the problem. After receiving the instruction, the agent sees the initial observation of the world state
117   and starts to interact with the world by taking actions within the problem’s prescribed action space
118
      (which can e.g. be a finite space, a continuous vector space, or a free-form text space just like that
      in RL). After an action is executed, the world’s internal state may change and the agent sees the
119
      next observation of the world. As the agent interacts with the world, the teacher provides natural
120
      language feedback on how the agent performs to guide the agent to do better. This language feedback
121   is a strict generalization of the reward signal in RL and can provide richer information to help agent
122   learn (e.g., suggestions, explanations, etc.). If we group the world and the teacher in LLF together
123   as an abstract environment, we see that LLF mainly replaces the reward maximization objective and
124   numeric feedback in RL with a generic task instruction and language feedback. In LLF-Bench, we
125   simulate LLF problems through the OpenAI Gym interface, described in Appendix C.
126
127
128
      2.2   I SN ’ T RL E NOUGH ?
129
      The LLF setup is motivated by the inefficiency and unnaturalness of communicating intentions via
130
      rewards in the real world. The concept of return maximization in RL, while giving a simple ab-
131
      straction of interactive learning, often creates a barrier for people to transfer knowledge and convey
132   intentions to AI agents. The reward paradigm forces one to compress all the information one wishes
133   to convey to the agent at a given step into a single numerical value expected to encourage or pe-
134   nalize certain behaviors. In addition, rewards are received only after the agent takes actions, so the
135   agent has to not only learn to solve the task but also learn to understand the task’s objective. This
136   bottleneck limits the information that can be transferred to the agent and couples solution learning
137   with intention understanding, causing the agent to learn inefficiently in a trial-and-error manner.
138   In many cases, it is also difficult for human designers to fully understand the long-term effects
139   of maximizing return (the expected sum of rewards), even when each instantaneous reward makes
140   sense. This misalignment has led to many surprising behaviors of RL agents (Amodei et al., 2016).
141   Consequently, reward engineering has been a common practice in building RL systems, where the
142   user iteratively tweaks the task’s rewards by observing how the agent behaves after maximizing the
143   current reward function. However, reward engineering is an expensive process. If agents were able
144   to learn directly from language feedback, learning systems could be built more economically.
145   Overall, compared to RL, LLF embraces the rich language feedback used in human-to-human learn-
146   ing. Its expressivity provides a potentially more efficient mechanism for training agents than RL.
147
148
149   2.3   W HY S HOULD W E S TUDY LLF N OW ?
150
151   Interactive learning settings with language-based instructions (Misra et al., 2018; Chen et al., 2019)
152   or observations have been extensively studied in the literature (Wang et al., 2016; Guu et al., 2017;
153
      Zhong et al., 2021). However, in all these settings, one assumes access to either gold actions or
      rewards. In contrast, in LLF the agent is provided with neither of these, which makes LLF appear
154
      harder than RL. We argue that this difficulty of working with general language feedback has been
155
      the reason why LLF hasn’t received much attention previously, despite its potential benefits. Re-
156   cently, Large Language Models (e.g., GPT4 (OpenAI, 2023), Gemini (Gemini Team, 2023)) have
157   demonstrated impressive natural language processing abilities. In addition, multiple LLM agents
158   have shown promising signs of solving text-based problems involving decision making, planning,
159   information retrieval, and tool use (Wang et al., 2023a; Schick et al., 2023; Wu et al., 2023). There-
160   fore, with LLMs, it may be possible to design algorithms to systematically solve general LLF prob-
161   lems. Conversely, solving LLF can also be viewed as a way to measure LLMs’ ability to tackle new
      learning tasks.

184   In fact, with access to accurate LLMs, LLF is not harder than RL if the task instructions in LLF
185
      are detailed enough to allow the LLM to infer from observations alone (without language feedback)
      whether the agent has succeeded at following the instruction. (Note that this assumption does not
186
      mean that the instruction necessarily shows the agent how to solve the problem.) Under this assump-
187
      tion, LLF problems can always be solved without the feedback, by a reduction to an RL problem
188   with sparse binary reward of success (the binary reward can be computed using a LLM to detect
189   success based on the instruction and the observation). However, such a reduction approach would
190   lead to inefficient learning.

      3     LLF-B ENCH
194
      The main research question of LLF is how to best leverage the language feedback, which can convey
195
      more information than just success/failure, to learn the optimal policy for the task in a sample-
196   efficient manner. We design LLF-Bench as a research platform to facilitate the development and
197   evaluation of LLF agents (e.g., LLM agents) built to make progress on this research agenda.
198
199   3.1    P ROBLEM S ETS AND TASKS
200
201   LLF-Bench consists of 8 diverse sets of decision-making problems (see Figure 2), with different
202   action spaces (discrete, continuous, and free-form text spaces) and decision horizons. Their brief
203   descriptions follow below, with more details in Appendix B:
204
205   • llf-bandit is a verbalized version of the classic multi-armed bandit problem, which we im-
206
        plement based on gym-bandits. llf-bandit tests the agent’s learning ability in an unknown
        environment with a finite number of actions.
207
208   • llf-poem consists of a set of poem writing tasks, where the agent needs to write a poem sat-
209     isfying certain syllable- and line-constraints. These problems tests the agent’s learning ability to
210
        infer and solve constraint satisfaction problems.
211   • llf-reco-movie simulates the scenario where a user wants movie or TV show recommenda-
212     tions based on some preferences. The user specifies their preferences in text, and any recommen-
213     dation made by the agent is matched to a movie database for checking whether the preferences are
214
        matched correctly.
215   • llf-optimization consists of 8 loss functions (Rosenbrock, Bohachevsky, etc.) and pro-
        vides an interface to give verbal feedback for the task of optimization on any loss function.

234   3.2   D ESIGN OF LLF-B ENCH
235
      When designing a learning benchmark, an important consideration is whether the evaluation can
236
      truthfully reflect an agent’s learning and generalization abilities and separate them from overfitting.
237
      To this end, we make two important design choices:
238
239   1. Following the framing of LLF, LLF-Bench implements the task instruction as part of the envi-
240      ronment, as opposed to as part of the agent. The latter is common in the current literature of LLM
241      agents, and many LLM agents heavily rely on using task-specific prompt templates (Yao et al.,
242      2023; Wang et al., 2023a). Via this design, we encourage users of LLF-Bench to develop agents
243      that can simultaneously work well across different problems sets in LLF-Bench. We hope that
244      this paradigm shift will facilitate the development of more general learning agents that can solve
245      multiple tasks, rather than agents tailored to a single task.
246   2. LLF-Bench provides the option to further randomize the textual description of task instruction
247      and feedback that the agent receives. In addition, for several environments, we randomize the en-
248      vironment’s latent parameters (e.g., to permute the action ordering in llf-bandit or change
249      the room connectivity in llf-gridworld) when the environment is reset. Sensitivity to dif-
250      ferent phrasings of the same instruction is often used to measure the robustness of a text-based
251      model (Ribeiro et al., 2018; Wallace et al., 2019). This design is motivated by the observation
252
         that LLMs as of now do not always perfectly understand semantics and can be sensitive to the
         exact texts that are presented (Zhu et al., 2023). It has been shown LLMs suffer from recency
253
         bias and can give drastically different outputs for semantically similar inputs (Arora et al., 2023;
254
         Leidinger et al., 2023). To combat that, for each problem instance in LLF-Bench, we manu-
255      ally curate a set of syntax templates via paraphrasing, which are used to produce a diverse yet
256      semantically equivalent set of task instructions and feedback during interactions. Through ran-
257      domization, LLF-Bench can better evaluate the agent’s task solving ability and prevent the agent
258      from overfitting a single text realization.
259
260   Configurable Feedback System One prominent feature of LLF-Bench is its configurable feed-
261   back system. Taking inspiration from the education research literature (Shute, 2008) and research
262   on effective learning signals for reinforcement learning agents, such as heuristics-guided learn-
263   ing (Cheng et al., 2021) and hindsight learning (Sinclair et al., 2023), we classify the language
264   feedback into 3 different types:
265
266
      1. Reward: Feedback of performance on the current action (similar to reward scalars and success
         booleans, generalized and expressed via language). By using this feedback type, several classical
267
         RL environments can be comparably tested with LLF agents in LLF-Bench.
268
269   2. Future Feedback: Suggestions of future behaviors, such as hints (positive feedback) or things
         to avoid (negative feedback).


                                                        5
      Under review as a conference paper at ICLR 2025


270
      3. Hindsight Feedback: Explanation of past behaviors, such as why some behaviors are bad (neg-
271      ative feedback) or why some behaviors are helpful (positive feedback).
272
273   This taxonomy is inspired by the education research literature (Shute, 2008). While LLF-Bench can
274   provide textualized numeric reward (i.e. the Reward type), learning from the Reward-type feedback
275   is different from learning scalar rewards directly (e.g., in RL). Even when given textualized rewards,
276   an LLM agent still has to understand the text and take actions accordingly, a challenge that is absent
277   in RL where rewards are separately available as scalars. This is similar to the challenge of video-
278
      game agents that see the game score is on the screen but need a good semantic understanding of the
      screenshots to use it as a reward signal.
279
280   We also note that hindsight and future feedback types are different from text-based RL and language-
281   grounding tasks. The latter two use only numeric feedback (if framed as RL) or actions (if framed
282   as imitation learning). In LLF, the feedback is text. As Table 1 shows, text-based RL and language-
283   grounding tasks are only similar to LLF in that they have observations that are text and/or images.
284
      However, in both cases feedback conveys a different type of information than observations: obser-
      vations (partially) describe world state, while feedback says something about the agent’s actions.
285
286   By default, an LLF-Bench environment provides a mix of these feedback types (when appropriate).
287   It can also be easily configured to provide only a subset of these feedback categories. This makes
288   for a more realistic learning problem, rather than the same type of atomic feedback at every step.
289   LLF-Bench generates the feedback through templates. For each problem instance, we curated 5-20
290
      versions of each atomic type of feedback. The environment, when queried, randomly samples from
      them and composes the samples together into overall feedback messages based on the configuration.
291
      Compared with generating feedback through an LLM-based simulated teacher, this template-based
292
      approach, while being less realistic, ensures reproducibility (through controlling the random seeds)
293   and is efficient to run. See Appendix C.1 for details.
294
295
      Interface For ease of use, LLF-Bench adopts the OpenAI Gym API (Brockman et al., 2016),
296   which abstracts the interaction with reset and step API functions. LLF-Bench environments
297   return the natural language instruction and feedback as the observation (a Python dict) and the action
298   spaces vary across problems. LLF-Bench environments also return rewards per the Gym step API.
299   While agents in the LLF setup do not use rewards, the returned rewards can be used to evaluate an
300   LLF agent’s performance; this feature makes the LLF-Bench environments also usable as typical
301   RL environments. LLF-Bench also provides a text-mode option (where both the observation and the
302   action are free-form texts), so that it can also be used as a benchmark for evaluating LLMs as agents
303   as well. Please see Appendix C for details of the API and the implementation of LLF-Bench.

361   LLM Agent Benchmarks Building agents based on LLMs has ushered in a new set of challenges.
362   In general, the environments included in these benchmarks only require planning and information
363   retrieval, and have sparse reward signals at the end of each attempt to solve the task. Very few of
364   these benchmarks measure the ability of an agent to learn and adapt to a task (e.g., the Abstraction
365   and Reasoning Corpus by Chollet (2019)). Liu et al. (2023b) proposed a set of environments that
366   cover a few popular types of task setups, such as web browsing, game, and code generation. Their
367   focus is on the diversity of tasks, not LLMs’ robustness or ability to incorporate feedback – two
368   factors crucial for LLMs’ successful operation in user-centric environments. Ge et al. (2023) con-
369
      structed a set of tasks where LLMs are prompted to use language or vision-related models to solve a
      complex task that requires multiple steps. The task-level feedback they provide is a numerical score
370
      from a domain-specific evaluation method. MINT (Wang et al., 2023b) is a benchmark that also
371
      offers natural language style feedback. However, MINT synthesizes user feedback by prompting
372   LLMs. This incurs additional costs, introduces additional variability in the evaluation process, and

374   et al., 2023) provides a set of 8 environments that include full and partial observability. The tasks

376   during multi-round interactions. DialOp (Lin et al., 2023) provided three constraint-satisfaction-

      6   C ONCLUSION
502
      We introduced LLF-Bench to evaluate AI agents’ ability to learn interactively from instructions
503
      and language feedback. We conjecture that the LLF paradigm will be significant for speeding up
504   the agents’ learning process by avoiding trial and error. LLF-Bench contains a diverse collection
505   of tasks such as recommendation, constrained writing, navigation and robot control. LLF-Bench is
506   designed to reflect an agent’s learning and generalization capability, separating them from over-fitted
507   performance on any given task. A key highlight is the configurable feedback system, classifying
508   language feedback into performance, past behaviour explanations, and future suggestions — this
509   encompasses existing RL environments, as well as imitation learning problems. Finally, we hope
510   that LLF-Bench will serve as a research platform for developing and testing LLF agents, enabling
511   the development of more general-purpose agents capable of learning to solve multiple tasks.

770   B       TASKS IN LLF-B ENCH
771
772   LLF-Bench consists of 8 different problem sets, ranging from user-recommendation, poem-writing,
773   navigation, to robot control. In the LLF setup, the reward is masked out (though the environments
774   in LLF-Bench still return rewards for evaluation purposes). To solve these problem efficiently, an
775   LLF agent needs to have sufficient common sense understanding of the natural language instruction
776   and the feedback. In addition, the agent needs to be able to learn from environmental interactions
777
      and feedback. We intentionally design these suites of problems such that, while the agent can tell
      success from the instruction and the environmental observation, it is difficult for the agent to infer
778
      the optimal policy from them without additional learning.
779
780   These problem sets feature different action spaces, problem horizons, and test different abilities of
781   LLF agents. We provide a summary in Table 5 and next describe each problem set in more detail.

797   B.1      L L F-B A N D I T
798
799   llf-bandit is a verbalized version of the classic multi-armed bandit problem. We built
800   llf-bandit based on gym-bandits4 by adding natural language task instruction and feedback.
801   There are a total of 8 bandit problems in llf-bandit. For each problem, the task instruction tells
802   the task name from the underlying gym-bandits, that the goal is a bandit problem, as well as the
803   feasible actions. While being a bandit problem, llf-bandit’s feedback does not necessarily con-
804
      vey the reward value in text (it depends on the configuration of the feedback type). When reset,
      the environment randomizes the order of actions and, if applicable, the underlying reward function.
805
      The agent here needs to learn to explore and exploit in multiple rounds of interactions to find the
806
      best arm as fast as possible with small regret (measured in terms of the hidden rewards). Overall,
807   llf-bandit tests the agent’s learning ability in an unknown environment with a finite number of
808   actions.

      B.3      L L F-R E C O-M O V I E
822
823   llf-reco-movie is an environment that simulates user-recommendation system interactions on
824   the topic of recommending movies. To simulate a user, the environment will first randomly sample a
825   user preference profile over a set of attributes such as the type of entertainment (TV show or movie),
826   year range (80s, 90s, 2000s, or recent), preferred genres (Action, Comedy, Documentary, etc.), and
827   age restriction (child/family-friendly or R-rated). Then, a mask will be sampled to randomly hide
828   one or more of the preferences in the initial request. An agent needs to recommend a few items (no
829   restriction on the number of items) that all satisfy the stated preference. An item-by-item feedback
830
      is provided in this environment to point out detailed preference violations that can allow LLMs to
      improve their recommendations. The reward is defined as the percentage of recommended items
831
      being correct. r ∈ [0, 1]. This is a classic slate recommendation setup (Li et al., 2011; Swaminathan
832
      et al., 2017).

      B.4      L L F-O P T I M I Z A T I O N
835
836   llf-optimization provides an easy-to-use interface with automatic procedurally generated
837   feedback that examines LLMs’ ability to make a series of proposals x to minimize a particular loss
838                                                                                                       dy
      function y = f (x). The feedback provided in this environment is created by computing gradient dx
839   and then verbalizing this information based on the change in input between the previously proposed
840   x and the current chosen x. For each optimization problem, the input range is bounded: x ∈
841   [xmin , xmax ], and the reward is simply min(−yt , −ymax ) (to prevent any choice of x that is outside
842   of the bound). We provide implementations of 8 classic loss functions (Rosenbrock, Bohachevsky,
843   etc.), and the base class is easily extendable to cover other loss functions. This is an environment
844

861   describes the current room including all the objects in it, and the different doors that are available.
862   If the agent takes an action, such as a = north, then it will transition from its current room, to the

870   B.7     L L F-A L F W O R L D
871
872   The llf-alfworld environment is a wrapper built on top of the popular AlfWorld text-game
873   environment6 (Shridhar et al., 2021) which itself is built as a parallel to the embodied Alfred
874   dataset (Shridhar et al., 2020). llf-alfworld contains multi-step reasoning tasks, where in each
875   episode, the agent is given an instruction in a house setting and must take a sequence of actions to
876
      fulfill this instruction. In each step, the agent is given a textual description of what it sees and a
      list of valid actions. The agent generates a text action (e.g., open drawer 1), which if it is valid can
877
      change the agent’s observation, and if it is invalid then results in no change. The agent additionally
878
      gets a reward for each action. The goal of the agent is to maximize the total reward by solving the
879   task. Unlike the llf-gridworld setting, the agent is tested for generalization as each episode
880   can contain a new task in a possibly new house environment. The main addition in llf-alfworld
881   is the capability to provide text feedback instead of reward. The text feedback is generated using an
882   optimal trajectory for that episode, as well as the instantaneous reward and the list of valid commands
883   for each time step.

886
887   llf-metaworld is based on the existing Meta-World v2 benchmark7 (Yu et al., 2019) and sup-
888   ports both text (of low-dimensional states) and image observations. Meta-World consists of 50
889   simulated robotic manipulation tasks, in each of which a robotic Sawyer arm needs to move an ob-
890
      ject into a specified position, e.g., push a puck to a goal location or press a button. An agent trying to
      accomplish an llf-metaworld task is presented with an instruction stating that the task is about
891
      getting a robotic manipulator to successfully handle an object and explaining what each dimension
892
      of the agent’s 4D state space means. By default, the environment interprets an agent’s action as a
893   target pose where the arm should move8 , and tries to move the arm there using Meta-World’s built-in
894   P-controller. At each time step, the agent receives as observation a description of the current state
895   mentioning the pose of the arm and all relevant objects in the scene. The language feedback here
896   may include advice on where to move the arm next and where not to move it.

       C.1   I NSTRUCTION AND F EEDBACK
973
974    Instruction is a string that is defined inside the Environment and describes in natural language the
975    problem that a decision-maker must solve. We recommend that agent-designers should not inspect
976    and overfit to a specific instruction describing the desired task in an environment; the default be-
977    havior of LLF-Bench environments is to paraphrase instructions in different ways to minimize the
978    chances of prompt overfitting. Three different types of Instruction are supported in LLF-Bench,
979    and can be toggled by passing an appropriate instruction_type to the make command of a
980
       LLF-Bench environment:
981          • Basic: instruction_type = ‘b’. This is the default instruction type for LLF-Bench
982            environments. The instructions provide an agent with the goal, semantics of its action
983            space, as well as the expected syntax of its responses. The instruction provides enough
984            information for a competent agent (e.g., a literate human) to begin interacting with the
985            environment.
986          • Complete: instruction_type = ‘c’. The instructions additionally provide infor-
987            mation to reliably infer (e.g., by a literate human) an optimal policy for achieving the goal.
988          • Practical: instruction_type = ‘p’. It contains the Basic instructions, and ad-
989            ditionally includes Feedback for previously executed actions. The goal of a learning
990            agent is to infer the optimal policy (i.e., comparable in performance to the one with
991            instruction_type = ‘c’) as quickly as possible.
992
993    Feedback is a string that provides the signal for an agent to learn from its interaction. LLF-Bench
994
       implements two kinds of feedback: an atomic feedback, and a composite feedback. The type of
       feedback an environment provides to an agent is set by passing an appropriate feedback_type
995
       parameter to make. Atomic feedbacks are inspired by the education research literature (Shute,
996
       2008). LLF-Bench currently supports 5 different types and we plan to include new styles (to include
997    e.g., questioning) in the future:
998
999          • feedback_type = ‘r’: This is the textualization of the scalar reward signal or suc-
1000           cess signal from classical RL. By using the text-wrapper and this feedback type, several
1001           classical RL environments (implemented in OpenAI Gym or Gymnasium) can be compa-
1002           rably tested with LLF agents in LLF-Bench.
1003         • feedback_type = ‘hp’: This hindsight positive feedback provides an explanation
1004           about a past action by the agent that was desirable.
1005         • feedback_type = ‘hn’: This hindsight negative feedback provides an explanation
1006           about a past action by the agent that was undesirable.
1007         • feedback_type = ‘fp’: This future positive feedback provides a suggestion for a
1008           potential future action that could be desirable.
1009
             • feedback_type = ‘fn’: This future negative feedback provides a suggestion for po-
1010           tential future actions that should be avoided.
1011
1012   feedback_type = ‘r’ corresponds to the current performance evaluation from the educa-
1013   tion research literature, whereas feedback_type = ‘fp’, ‘fn’ correspond to future be-
1014   havior suggestion. Finally, feedback_type = ‘hp’, ‘hn’ correspond to the past behavior
1015   explanation style of feedback studied in the education research literature.
1016   Composite feedback types allow the environment to provide the agent multiple kinds of atomic
1017   feedbacks. This makes for a more realistic learning problem, rather than the same type of atomic
1018   feedback at every step of the environment.
1019
1020
             • feedback_type = ‘a’: All of the Atomic feedback types that are supported by the
               environment are provided to the agent at each round of interaction.
1021
1022         • feedback_type = ‘m’: The agent receives a Mix of different atomic feedbacks. A
1023           random subset of the supported feedback types are sampled by LLF-Bench to provide to
1024
               the agent at each step.
1025         • feedback_type = ‘n’: The agent receives No feedback, this mode is provided for
               debugging purposes.

1027   types to allow fine-grained control of the learning signal that an agent can receive from LLF-Bench
1028   environments. The default behavior in make for any environment uses feedback_type = ’a’.

       C.2   I NSTRUCTION AND F EEDBACK R ANDOMIZATION
1031
1032   To reduce the sensitivity of learning agents to a specific text realization, LLF-Bench implements a
1033   template-based paraphrasing system, by which users can randomize the instruction and the feedback
1034   that the agent receives. For each problem in LLF-Bench, we implement about 4-20 paraphrased
1035   templates for each instruction and each feedback type. When the randomization options are turned
1036   on, the LLF-Bench environment will randomly choose one from these curated templates to formulate
1037   the language instruction and feedback returned to the agent. LLF-Bench also provides the option to
1038
       deterministically use a particular template. The randomness of paraphrasing can be controlled by
       setting the seed parameter in the OpenAI Gym reset function.
1039
1040   Compared with using language models to generate feedback on-the-fly, the use of templates offers
1041   advantages: (1) the latter is free, while the former can be very expensive. (2) The latter results in far
1042   higher reproducibility: it is very hard or even impossible to guarantee that an LLM produces exactly
1043   the same output (gives the same feedback) for the same prompt (for the same state). (3) Some
1044
       generic templates can be broadly useful across many environments, serving as an initial example
       when building a new environment. All three reasons are very important for a benchmark, and in
1045
       general the use of synthetic data has been common in the literature (Hudson & Manning, 2019;
1046
       Hermann et al., 2020; Blukis et al., 2018), although we acknowledge that it has its own drawbacks.
1047
1048
1049   D     R ELATED W ORK
1050
1051   Grounded Language Learning Reinforcement learning with textual information has been stud-
1052   ied under the branch of multi-modal representation learning. This branch of study has several fo-
1053   cuses that are both similar and different from our goal with LLF-Bench. One focus deals with
1054
       ambiguity and difficulty in understanding instructions or goals specified by natural language (Wang
       et al., 2016; Bahdanau et al., 2019; Chaplot et al., 2018). While the ambiguity of instructions is a
1055
       concern, we focus more on robustly behaving under different instructions that all represent the same
1056
       underlying goal. Another focus of this body of work is to ground visual information with textual
1057   instruction – a core aim of multi-modal representation learning (Bisk et al., 2016; Misra et al., 2017),
1058   with an extension to robotic interaction (Karamcheti et al., 2022; 2023). Language provides a natu-
1059   ral shared representation that enables easier transfer between different tasks (Hanjie et al., 2021) or
1060   supplies important information such as safety constraints for a policy (Yang et al., 2021). In previ-
1061   ous work, feedback is often not considered. When feedback is considered, it is usually framed as
1062   error messages from a syntax parser (if the action space is text) and can indeed be incorporated into
1063   learning (Côté et al., 2019). This type of feedback corresponds to feedback_type = ‘hn’ in
1064   our setup.
1065
1066   Text-based Games Extending from using reinforcement learning for solving complex games,
1067   there are many text-based games that include challenges such as the navigation of space, manip-
1068
       ulation of the environment to achieve goals, and reaction to random events. Narasimhan et al.
       (2015) repurposed a classic text adventure game, Zork, where both observation and action space
1069
       are text. Côté et al. (2019) proposed a set of text-based game environments and included a few
1070
       carefully designed challenges for RL to solve, such as large state and action space (determined by
1071   the vocabulary size) and long credit assignment. On the other spectrum, Küttler et al. (2020) cre-
1072   ated a learning environment from the game NetHack. Although the game state is represented with
1073   hundreds of text symbols, policy learning is conducted on the screenshot of the terminal. Similarly,
1074   BabyAI (Chevalier-Boisvert et al., 2019) is a set of procedurally generated grid-like maze environ-
1075   ments – the objects and representation in the environment are a fixed set of symbols. None of these
1076   environments consider providing language feedback on the agent’s action.
1077
1078   Learning from Language Feedback Providing feedback to an RL agent’s action as part of the
1079   learning signal beyond task rewards has been studied in robotics. However, most of the efforts
       were limited to eliciting binary preference feedback (Sadigh et al., 2017; Biyik & Sadigh, 2018) or


                                                          20
       Under review as a conference paper at ICLR 2025


1080
       ranking-based feedback from real people (Basu et al., 2019). Sumers et al. (2021) crowd-sourced
1081   a small feedback dataset on a small game. They considered three types of feedback, evaluative
1082   feedback (which corresponds to feedback_type = ‘r’), descriptive feedback (which in our
1083   setup is decomposed into feedback_type = ‘hp’, ‘hn’), and imperative feedback (which
1084   corresponds to feedback_type = ‘fp’, ‘fn’). They then used a sentiment classifier to
1085   extract coarse information from this feedback to improve the policy’s behavior. Nguyen et al. (2021)
1086   proposed an approach to map textual instructions to trajectories in embodied settings by assuming
1087   that a user can label a generated trajectory with the instruction that is likely to generate the trajectory
1088   under the optimal policy. More recently, Cui et al. (2023); Liu et al. (2023a) studied the case of
1089
       language feedback as corrections to a robotic arm at any time of the task execution, which is an
       instance of the LLF setup that we are considering.
1090
1091
1092   LLM Sensitivity to Prompts A long line of work has investigated smaller-scale language-based
1093   systems’ sensitivity to different expressions that have the same underlying meaning. They can be
1094   categorized as adversarial attacks to text-based systems (Ribeiro et al., 2018; Wallace et al., 2019)
1095   or as mechanisms to improve language-based systems’ output via self-consistency (Edunov et al.,
1096   2018). More recently, the lack of robustness to prompts has been found on large language models
1097   as well (Liu et al., 2023c; Wolf et al., 2023). Zhu et al. (2023) proposed a benchmark dataset to
1098
       investigate the robustness of LLMs on different types of prompts that can contain user errors for
       tasks related to natural language.

1110
       interface. When error happens due to LLM outputting inadmissible actions, we catch the error and

       All environments are initialized with horizon of H = 30 and are stopped if success earlier if the
