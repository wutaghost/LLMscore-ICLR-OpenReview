009                                                     A BSTRACT
010
011               While code large language models have made significant strides in AI-assisted coding
012               tasks, there are growing concerns about privacy challenges. The user code is transparent
013               to the cloud LLM service provider, inducing risks of unauthorized training, reading, and
014               execution of sensitive code. Such fear of data leaking prevents developers from submitting
015               their code to LLMs. In this paper, we propose CodeCipher, a novel method that perturbs
016               privacy from code while preserving the original response from LLMs. CodeCipher trans-
017               forms the LLM’s embedding matrix so that each row corresponds to a different word in the
018
                  original matrix, forming a token-to-token confusion mapping for obfuscating source code.
                  The new embedding matrix is optimized through minimizing the task-specific loss func-
019
                  tion. To tackle the challenge from the discrete and sparse nature of word vector spaces,
020
                  CodeCipher adopts a discrete optimization strategy that aligns the updated vector to the
021               nearest valid token in the vocabulary before each gradient update. We demonstrate the ef-
022               fectiveness of our approach on three AI-assisted coding tasks including code completion,
023               summarization, and translation. Results show that our model successfully confuses the
024               source code while preserving the original LLM’s performance.1

027   1       I NTRODUCTION
028
029   The rise of code large language models (code LLMs), including CodeLlama (Rozière et al., 2023) and
030   DeepSeek-Coder (Guo et al., 2024), has ushered in a new phase for the research on code intelligence. These
031
      code language models have made significant strides in improving the quality of code generation and have
      achieved high pass rates across multiple benchmark datasets such as HumanEval (Chen et al., 2021), and
032
      MBPP (Austin et al., 2021). With the increasing reliance on these models for AI-assisted coding tasks, there
033
      is a growing need to leverage remote LLM cloud services, which enable broader scalability and enhance
034   the capabilities of these models by offloading computations to powerful cloud infrastructure. While large
035   white-box LLMs are readily available for local deployment and execution, limited computational resources
036   may prevent individual users from deploying them locally, necessitating a client-server architecture.
037
      LLM cloud services allow users to perform AI coding tasks without the need for computational infras-
038
      tructure. However, providing code LLMs as services raises significant privacy concerns, which limit their
039
      practical applications (Yao et al., 2024; Yan et al., 2024). As users’ code is transparent to these remote sys-
040   tems, there is a risk of exposing and misusing sensitive or proprietary information. Such fear of data leak-
041   ing prevents many developers from submitting their code directly to LLMs. Therefore, privacy-preserving
042   mechanisms in LLM deployment are required to ensure user trust and further expand code LLM usage.
043
      Code obfuscation techniques have been common solutions to defend code against unauthorized access,
044
      which disguises elements of a piece of code while maintaining its output. Existing techniques adopt rule-

      based perturbations, such as renaming variables with semantically meaningless symbols (Jain et al., 2020)
048
      and injecting dead code (Chakraborty et al., 2022), to obscure human comprehension of code semantics.
049   However, since LLMs also rely on meaningful symbols and other human-readable code structures (Ding
050   et al., 2024; Rodeghero et al., 2014). Brute code obfuscation could significantly degrade the capability of
051   LLMs and thus are inapplicable in practice.

      In this work, we propose CodeCipher, a novel learning-based code obfuscation technique tailored for LLMs.
053
      CodeCipher safeguards code from unauthorized training, reading, compiling, and execution, without sacri-
054
      ficing LLM service quality. The core idea behind CodeCipher is to transform the LLM’s embedding matrix
055   so that each row corresponds to a different word in the original matrix. This process creates a token-to-token

058
      The new embedding matrix is optimized by minimizing the task-specific loss. Due to the discrete and
      sparse nature of word embedding space, straightforward gradient descent can not pinpoint the embedding
059
      of a valid token. To tackle this problem, we adopt a discrete optimization strategy: at each iteration, the
060
      algorithm updates the perturbation vector, projects it onto the nearest valid token in the embedding space,
061   and recalculates gradients based on the new projection.
062
063   The efficacy of our approach was rigorously assessed across three AI-assisted coding tasks: code com-
064
      pletion, code summarization, and code translation. Results revealed that our method surpassed a range of
      conventional obfuscation methods in terms of both the level of confusion and the preservation of perfor-
065
      mance for downstream tasks. These findings affirm the practicability of our approach for enhancing privacy
066
      and security in LLMs.

071   Privacy Protection for Large Language Models While LLMs have brought tremendous value to many
072   fields, they have also sparked deep concerns among users regarding data privacy and security (Nasr et al.,
073   2023). Numerous studies have been conducted on model security in previous research (Fan et al., 2023; Yan
074
      et al., 2024; Lin et al., 2024). On the model side, federated learning and differential privacy are commonly
      used to protect privacy leakage during model training. Homomorphic encryption is widely used during
075
      model inference (Lin et al., 2024). On the user side, privacy protection focuses on marking or transforming
076
      user’s data. Watermarking, the most common technology, embeds hidden markers in data to detect its use
077   in model training (Sun et al., 2023). Another method is data transformation, which makes data unintelli-
078   gible or unusable even if leaked. Traditionally, this involves rule-based removal of sensitive information
079   (Machanavajjhala et al., 2007). More recently, LLMs have been leveraged to obscure sensitive data (Song
080   et al., 2024). Users can construct privacy-preserving prompts, mixing real and fake inputs to prevent the
081   server from identifying the true prompt (Utpala et al., 2023; Zhou et al., 2023).
082
      While the aforementioned methods have been widely used for privacy protection, they differ from CodeCi-
083   pher significantly in technical scope. One key difference is that they often necessitate complex modifications
084   to both client- and server-side model architectures and training methodologies. For instance, homomorphic
085   encryption requires substantial changes to the self-attention architecture in LLMs. In contrast, CodeCipher
086   offers a lightweight, practical solution that only involves minimal data transformations on the client side
087   without altering the model itself or requiring server modifications. This simplicity makes CodeCipher a
088   more accessible option for code obfuscation.
089   Privacy Protection for Code LLMs Data leakage is also a concern in LLMs for code, which are often
090   trained on open-source code repositories that may contain sensitive information. Studies (Feng et al., 2022;
091   Huang et al., 2023; Al-Kaswan et al., 2023) have shown that specific prompts can extract private data like
092   passwords and server keys from large models. Numerous solutions have been proposed to address this issue,
093   such as adding watermarks to detect unauthorized use of training data (Sun et al., 2023), training models

094
      on desensitized data (though its impact on performance is still unknown) (Yang et al., 2024), and distilling
095
      models to smaller, more secure local models (Shi et al., 2024). However, ensuring the safe and private use
096   of LLMs, especially for code generation, remains a critical challenge that requires further research.
097
098   Code Obfuscation Code obfuscation aims to reduce the readability of source code through code transfor-
099
      mation, reorganization, and modification (Behera & Bhaskari, 2015). Commonly used code obfuscation
      techniques include identifiers renaming (Eastridge-Technology, 2000; Hoenicke, 2001), control flow flatten-
100
      ing (László & Kiss, 2009), dead code injection (Chakraborty et al., 2022), self-modifying code (Giffin et al.,
101
      2005), and property encryption (Pandey & Rouselakis, 2012).
102
103   While code obfuscation has been widely used for code security, its applicability to LLMs remains largely
104   unexplored. Traditional rule-based methods achieve the goal of obfuscation, but they are not specifically
105
      tailored to LLMs. As a consequence, the code representations learned by LLMs can be altered, leading to
      significant performance degradation.

      In this paper, we propose a learning-based method to find the optimal confuse function g(·) between code
142
      tokens. Unlike previous rule-based methods, we move our focus to the embeddings of code tokens. The
143   core idea behind CodeCipher is to transform the LLM’s embedding matrix so that each row corresponds to
144   a different word in the original matrix. The transformation establishes a token-to-token confusion mapping,
145   which the system uses to obfuscate tokens when encountering new code snippets. Our method consists of
146   three stages as illustrated in Figure 1: (i) transforms the LLM’s embedding matrix to a permutation, estab-
147   lishing a token-to-token confusion mapping; (ii) optimizes the transformation of embedding by minimizing
148   the task-specific loss function; addresses the intricacies involved in the discretization process, particularly
149   the alignment of continuous word vectors to valid tokens; and (iii) obfuscates code using the trained model.
150   A more comprehensive description of the algorithm can be found in Appendix A.

204   4.2   T RAINING
205
206   Our training objective is to find an optimal E′ that minimizes the task-specific loss function Ltask when
207   applied to the obfuscated code. A straightforward idea is to optimize Ltask using gradient descent. For a
208   code input, we update the embedding e for each token as follows:
209                                                    e′ = e − η∇e Ltask (x′ )                                  (4)
210
211
      where η represents the learning rate, ∇e L denotes the gradient of the loss function w.r.t. the current embed-
      ding e.
212
213   However, direct gradient descent encounters the intrinsic limitation in the word embedding space. Code
214   tokens reside on a sparse, discrete manifold in the embedding space. As illustrated in Figure 7, a single
215   gradient computation is hard to pinpoint the exact position of a valid token in the embedding space, thereby
216   hindering precise vector manipulation.
217   Inspired from Yuan et al. (2021), we develop a discrete gradient search algorithm. During each gradient
218   update, the algorithm aligns the updated vector with semantically meaningful tokens, ensuring transitions
219   occur between valid tokens in the original embedding space. More specifically, the algorithm performs
220   gradient update in Equation 4 along a finer-grained trajectory v1 , ..., vt . At each mini step, the embedding
221   vector is incrementally perturbed and then projected onto the nearest valid token vector in the vocabulary.
222   The nearest valid token is identified by calculating the dot product with the normalized embeddings of
223
      existing words. The gradient is then recalculated based on this new token, guiding further refinements in the
      following iterations. Formally,
224

229   This iterative process continues for a certain number of steps, at which point we select the vector with the

231   To prevent over-obfuscation, we limit the training process to a maximum of N iterations. The training early
232   stops if the perplexity deviation of a code sample reaches a predefined threshold δ. We define δ as a linear

234   controls the rate at which the threshold increases throughout the training process.

253   Code Completion. A task that completes the subsequent code given users’ partial code. We train our
254   model on the MultiPL-E dataset (the Python subset from the Leetcode section) (Cassano et al., 2023), which
255   comprises 446 code snippets. We test the model on the HumanEval benchmark (Chen et al., 2021) which
256   contains 164 Python programming problems. For each problem, we obfuscate the first half of the code,
257   provide it to the LLM, and ask it to complete the remaining half. As the obfuscated half code might prevent
258
      the final output from being compiled, we guide the model to re-generate the entire code from head using the
      prompt in Appendix B.
259
      Code Summarization. A task that provides a succinct natural language summary for a given code frag-
260
      ment (Sridhara et al., 2010). We train and test our obfuscation model on the CodeSearchNet benchmark (Hu-
261   sain et al., 2019) using the prompt in Appendix B.
262   Code Translation. A task that translates source code written in one language to another (Pan et al., 2023).
263   We train our model on the Java-to-Python subset of XLCoST (Zhu et al., 2022) and test it on the Java-to-
264   Python subset of HumanEval-X (Zheng et al., 2023). We use google-java-format2 to ensure consistent line
265   breaks and indentation across the code, aligning it with the formatting of the test dataset. This process re-
266   sulted in a total of 2,374 data pairs. In the prompt in Appendix B, we explicitly instructed the model to avoid
267   class-based code structures.
268   Baselines. We compare CodeCipher with both rule-based and LLM-based code obfuscation approaches.
269   1) Random perturbation, which randomly replaces a portion of tokens in the code with non-semantic
270   symbols. 2) Identifier renaming, which replaces variable and function names with non-semantic sym-
271   bols (Eastridge-Technology, 2000; Hoenicke, 2001). It is implemented from Spoon (Pawlak et al., 2015).
272   3) Dead branch injection, which inserts code blocks that never execute (e.g., while(false)). It is im-
273   plemented from NatGen (Chakraborty et al., 2022). 4) Remove symbols, which removes formatting and
274   syntactic symbols such as whitespace and parentheses from the code. 5) Prompting LLMs, which prompts

      5.6      T RANSFERABILITY
417
418   Our approach is developed on white-box LLMs where the embedding layers and task gradients are accessi-
419   ble. In practice, there is always a need to apply the model in a new LLM, particularly black-box LLMs which
420   provide only an API interface to users. We are unable to fine-tune these models based on their parameters.
421   In this section, we aim to evaluate whether the obfuscated code generated by our method can still achieve
422   good performance on other models.

                                                                                 9
      Under review as a conference paper at ICLR 2025


423
      Setup: We used the obfuscated code generated by CodeLlama-7B with a perturbed embedding layer to
424
      evaluate the performance on other models. We conducted the experiments on the code completion task,
425   which is one of the most common and general tasks. The representative models chosen for evaluation
426   include StarCoder (Li et al., 2023), DeepSeek-Coder (Guo et al., 2024), and GPT-3.5-turbo (OpenAI, 2023),
427   which have shown strong performance on AI coding tasks. We employed greedy decoding during inference,
428   and the HumanEval dataset was used for testing.
429
      Results: As shown in Figure 6, our method obtains relatively stable Pass@1 scores in each model through
430
      different obfuscating models, and in most cases, the deviation from original scores is within 10%. In par-
431   ticular, in the black-box LLM GPT-3.5-turbo, the obfuscated codes outperform its original results, showing
432   that our method remains high efficacy on other models. We hypothesize that our method learns common
433   token mappings that are portable across LLMs. Overall, the results suggest that our method is not tied to any
434   specific version of the LLM; instead, it is a general approach that works independently of the LLM version
435   used. In practical applications, we can develop our method on a local proxy model such as the open-sourced
436   CodeLlama-7B and apply the learned token confusion mapping to other cloud LLM servers.

      6   C ONCLUSION
462
463
      In this paper, we propose CodeCipher, a novel code obfuscation method tailored for LLMs. CodeCipher
464
      learns a transformation on the word embedding space, establishing a confusion mapping between code
465   tokens that can be used for code obfuscation. The embedding transformation is optimized by minimizing
466   the task-specific loss. A discrete gradient search algorithm is designed to tackle the sparsity challenge in
467   the embedding space. Experiments on three AI-assisted coding tasks demonstrate that our method achieves
468   a superior balance between code obfuscation and model performance compared to traditional rule-based
469   methods.
