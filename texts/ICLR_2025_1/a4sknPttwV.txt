009                                               A BSTRACT
010
011            The quality of datasets plays an increasingly crucial role in the research and
012            development of modern artificial intelligence (AI). Despite the proliferation of open
013            dataset platforms nowadays, data quality issues, such as incomplete documentation,
014            inaccurate labels, ethical concerns, and outdated information, remain common in
015            widely used datasets. Furthermore, these issues are often subtle and difficult to be
016            detected by rule-based scripts, therefore requiring identification and verification
017            by dataset users or maintainers–a process that is both time-consuming and prone
018            to human mistakes. With the surging ability of large language models (LLM),
019
               it’s promising to streamline the discovery of hidden dataset issues with LLM
               agents. To achieve this, one significant challenge is enabling LLM agents to
020
               detect issues in the wild rather than simply fixing known ones. In this work, we
021
               establish a benchmark to measure LLM agent’s ability to tackle this challenge.
022            We carefully curate 221 real-world test cases from eight popular dataset platforms
023            and propose an automatic evaluation framework using GPT-4. Our proposed
024            framework shows strong empirical alignment with expert evaluations, validated
025            through extensive comparisons with human annotations. Without any hints, a
026            baseline GPT-4 Curator agent can only reveal 11% of the data quality issues in
027            the proposed dataset, highlighting the complexity of this task and indicating that
028            applying LLM agents to real-world dataset curation still requires further in-depth
029            exploration and innovation.

032   1   I NTRODUCTION
033
034   High-quality datasets have become increasingly crucial for advancing artificial intelligence (AI)(Jain
035   et al., 2020; Kaplan et al., 2020). Open dataset platforms, such as Hugging Face(Lhoest et al.,
036   2021) and BIG-Bench (Srivastava et al., 2023), have substantially accelerated AI research and
037   development by facilitating community contributions. However, community-contributed datasets
038
      often encounter subtle data quality issues, including insufficient documentation (Yang et al., 2024),
      inaccurate annotations (Klie et al., 2024), and ethical concerns (Gebru et al., 2021).
039
040   There have been existing efforts on standardizing dataset management practices (Gebru et al., 2021;
041   Wang et al., 2023; Gan et al., 2024) and developing dataset curation toolkits and systems (Gupta
042   et al., 2021; Mao et al., 2023). However, these toolkits and systems often rely heavily on rule-
043   based scripts, which lack the necessary flexibility to detect the aforementioned subtle data quality
044
      issues. Consequently, existing techniques remain insufficient to address the complex challenge of
      automatically curating datasets at scale on open dataset platforms, where we still primarily rely on
045
      dataset users or platform maintainers as dataset curators to identify data quality issues.
046
047   Recent advancement of large language models (LLMs) has led to promising development of LLM
048   agents for real-world software engineering problems (Jimenez et al., 2024a). For certain issues in
049   software development, LLM agents have been shown to be capable of autonomously generating
050
      proper code to fix them (Mesh, 2024; Tao et al., 2024). Given this development, we envision that
      LLM agents could become an effective technique for building autonomous dataset curation systems.
051
052   However, there is still a significant gap between the need of dataset curation and the state-of-the-art
053   LLM agents for software engineering. Specifically, existing studies primarily focus on developing
      agents that solve identified and well-defined issues, while dataset curation requires one to discover

129   2       R ELATED W ORK
130
      Dataset Quality Management With the advancement of modern AI, the need for high-quality
131
      datasets is surging (Jain et al., 2020; Kaplan et al., 2020). Consequently, open dataset platforms
132   like Hugging Face and Kaggle keep growing rapidly. However, recent studies (Klie et al., 2024;
133   2023; Weber-Genzel et al., 2023) indicate that many popular datasets suffer from errors, biases, or
134   annotation artifacts (Gururangan et al., 2018), and lack proper documentation (Yang et al., 2024). In
135   addition, common issues in real-world datasets include problems with data loading scripts and file
136   discrepancies, where conflicting information exists between files.
137
      There have been many existing efforts on dataset quality management. The FAIR principles (Wilkin-
138   son et al., 2016) provide a broad framework for creating and managing datasets. Gebru et al. (2021)
139   puts forward a standardized process for documenting datasets in the context of machine learning.
140   Gong et al. (2023) propose several strategies, including dataset profiling, data cleansing, and quality
141   monitoring. More recent works (Gebru et al., 2021; Wang et al., 2023; Gan et al., 2024) further ex-
142   plore standardizing dataset management practices. Based on these works, researchers have developed
143   advanced dataset management toolkits and systems (Gupta et al., 2021; Mao et al., 2023; Zhou et al.,
144   2024). More recently, Zhou et al. (2023); Chen et al. (2023a) developed systems that are capable of
145   greatly enhancing data quality. However, they focus on filtering a large-scale LLM-pertaining corpus
146
      rather than detecting dataset quality issues. There are some specialized works focused on detecting
      harmful content (Althobaiti, 2022; Kirk et al., 2023), annotation errors (Weber-Genzel et al., 2023;
147
      Wang & Mueller, 2022) and data attributions (Longpre et al., 2023). Nevertheless, those methods
148
      are not generalizable enough and lack the flexibility to solve challenges such as nuanced ethical
149   biases (see example 1) and incorrect annotations involving logical or factual errors (see example 3).
150   Moreover, studies on file discrepancies( example 2) and relevant issues remain scarce. Consequently,
151   we still primarily rely on dataset users or platform maintainers to identify these issues in practice. We
152   envision LLM agents as a promising technique to address the challenges in this space and propose a
153   benchmark dataset as an initial step toward this goal.
154
155   LLMs for Software Engineering Our work is also relevant to the emerging research direction on
156   using LLMs to solve software engineering tasks. Fan et al. (2023) suggest that there are two major
157   categories in this area: (i) code generation and completion, and (ii) maintenance and evolution.
158   There has been extensive literature on LLMs for code generation and completion, in terms of both
159   models and benchmarks. On the model side, various code generation LLMs have been proposed (Chen
160   et al., 2022; Luo et al., 2023; Rozière et al., 2024), along with agent frameworks designed to tackle
161

      complex coding problems, such as data analysis (Hong et al., 2024; Guo et al., 2024). On the
163   benchmark side, datasets have been created to complete functions and short programs from natural
164   language descriptions (Jimenez et al., 2024a; Chen et al., 2021; Austin et al., 2021; Lai et al., 2022;
165   Zhang et al., 2024; Ding et al., 2023) and retrieve relevant code (Liu et al., 2023a). We note that
166   the main purpose of the proposed DCA-Bench is to measure LLM agents’ capability of discovering
167   hidden dataset issues instead of generating code, although generating and running some test code
168   may be helpful.
169   For the second category, maintenance and evolution, it involves tasks such as localizing and fixing
170   bugs (Wu et al., 2023; Feng & Chen, 2023), improving program performance (Garg et al., 2024;
171   Chen et al., 2023c), and refactoring code without changing program behavior (Poldrack et al.,
172   2023). Our work falls into this category while differing from previous studies in two key aspects.
173   Firstly, while most literature focuses on tasks with well-defined outcomes, such as fixing a buggy
174   function, our work addresses non-standard issues like ethical concerns in data points or cross-file
175   inconsistencies. Secondly, few studies have explored using LLMs to maintain dataset repositories,
176   which is increasingly important for the machine learning community.
177
178   LLMs as A Proxy of Human Evaluation Recent studies have shown that LLMs, when carefully
179   prompted, can serve as a good proxy of human evaluations for a number of scenarios, such as
180
      evaluating text generation quality (Liu et al., 2023b; Chiang & Lee, 2023), reasoning ability (He
      et al., 2024a; Hao et al., 2024), and generated image quality (Chen et al., 2023b; You et al., 2024).
181
182   Inspired by these studies, we adopt an LLM to automatically evaluate the Curator’s responses based
183   on carefully designed instruction. We also empirically show that our LLM-based Evaluator highly
184   aligns with human preference, ensuring a reliable automatic evaluation scheme.

191   3.1   DATASET OVERVIEW
192   The dataset assets provided in DCA-Bench consist of test cases and reference resources. Tab. 1
193   shows basic statistics of test cases. Each test case typically contains multiple files, designed to
194   create a minimal environment for uncovering hidden data quality issues. We classify these test cases
195   into four categories based on the number of files included and number of hidden issues in a test
196   case. Additionally, we assign 18 descriptive tags to further label the cases, such as “data-problem”,
197   “document-problem”, “infrastructure-problem”, and “ethical/legal-risk”. Furthermore, DCA-Bench
198   provides reference resources, including related documentation and external materials such as dataset
199   curation tutorials, to help users enhance Curator performance using methods like RAG (Lewis et al.,
200
      2021).
201
      3.2   T EST C ASE C ONSTRUCTION
202
203   The test cases in DCA-Bench are collected from the real issues reported by users or maintainers of
204
      eight dataset platforms, including Hugging Face Dataset (Lhoest et al., 2021), BIG-Bench (Srivastava
      et al., 2023), Kaggle (Kaggle), OpenML (Vanschoren et al., 2014), TensorFlow Dataset (Abadi
205
      et al., 2015), Open-Data-Registry (AWS-Labs, 2024), Five-Thirty-Eight (FiveThirtyEight, 2024),
206
      and Graph Learning Benchmark (GLI) (Ma et al., 2022). The construction of the test cases of the
207   DCA-Bench involved the following two stages.
208
209   Issue Collection and Preprocessing At this stage, we select and preprocess relevant and manage-
210   able data quality issues with the following steps:
211
212   1. Selection of Dataset Platforms and Issues: We focus on dataset platforms where users and
213      maintainers can interact. For example, there is a “Discussion” section in each Kaggle dataset for
214      discussing dataset-related issues, which helps collect meaningful data quality issues. We then
215      manually collect data quality issues from these discussions. Please refer to A.1.4 for more details
         about the choice of platforms and case selections.

245   Hints Generation Asking Curators to discover hidden issues from raw dataset files can be very
246   challenging. To make it more manageable and test the Curators’ capability in finer granularity, we
247   generate different levels of hints to help the Curators locate the hidden issues. We apply GPT-4 with
248   carefully designed prompts2 to generate three levels of hints for each test case, in addition to the
249   no-hint setting:
250
251    • h0 : No hint provided. In this case, the Curator is required to detect the issue fully on its own.
252    • h1 : General description of the issue, without any specific details or hints on the location.
253    • h2 : Information about which files are involved in the issue, in addition to information from h1 .
254
       • h3 : Partial contextual information about the issue, in addition to information from h2 .
255
256
      After generating the hints, we manually double-check the hints and make necessary modifications to
257   ensure that the generated hints follow the guidelines above.

259   3.3      B ENCHMARK API
260   After gathering test cases and related information from dataset platforms, as well as generating
261   multi-level hints, we proceed to develop the benchmark API, which encapsulates the inputs to the
262   Curator and the Evaluator, adhering to the Task I/O paradigm as illustrated in Fig.2.
263
      In this framework, the Curator receives following from benchmark API: (i) dataset files with hidden
264
      issues, (ii) hints on the issues, and (iii) optional dataset documentation and reference materials. It
265
      is then asked to identify, describe, and provide contextual evidence for these issues. The Curator
266   is allowed to use any strategy and tools to process provided dataset files and find the issue, e.g.,
267   flattening the file contents and feeding them to the Curator, applying RAG technology, using a code
268   interpreter to write and execute programs, or combining multiple strategies together.

      3.4   E VALUATION P IPELINE
295
296   We now discuss the motivation and design of our evaluation pipeline. We evaluate the Curators
297
      in terms of the following two aspects: (i) whether the Curators accurately identify the issues, and
      (ii) whether the Curators provide necessary contextual evidence about the issue. We accordingly
298
      categorize the performance of the Curator into three levels:
299
300    • fail: The Curator fails to discover any issues, only identifies irrelevant issues, or acknowledges
301      the issue but offers completely wrong contextual evidence.
302    • success: The Curator identifies the annotated issue and provides at least one correct piece of
303      contextual evidence.
304
       • success+ : The Curator correctly identifies all issues and provides all necessary contextual
305      evidence.
306
307   Note that this evaluation process has several intrinsic challenges. Firstly, due to the nuanced nature of
308   text generation, different Curator outputs can refer to the same issue, while similar Curator outputs
309   may also point to different issues. Fixed keyword (He et al., 2024b) or rule-based code tests (Jimenez
310   et al., 2024a) are often insufficient to distinguish the nuances. Additionally, Curators may be able to
311
      identify the issues using only a portion of the contextual evidence annotated in the test case, which
      means that evaluation protocols requiring the inclusion of all the annotated contextual evidence can
312
      be overly stringent. While human experts can easily capture such nuances and properly evaluate the
313
      Curator performance, human evaluation is too expensive for a benchmark in practice. Recognizing
314   that evaluating the Curator’s performance is easier than building the Curator agent, we propose an
315   automatic evaluation pipeline where LLMs are leveraged to serve as the Evaluator.
316
317
      We explored several prompting strategies using a few samples of human-annotated outcomes on a
      handful of issues and ended up with the following prompt design, which asks the LLMs to rate the
318
      Curator outputs in terms of a few criteria, and then aggregate these ratings to obtain an overall score
319
      for the Curator performance. The criteria we have in the Evaluator prompt are listed as follows:
320
321    • Precise Contextual Evidence: This criterion evaluates whether the Curator has detected and
322      described the issue while providing accurate contextual evidence. The Curator should receive a
323      low rating if it identifies the issue but fails to locate the correct context. A medium or high rating
         is justified only if the Curator identifies the issue and accurately pinpoints where it arises.


                                                            6
      Under review as a conference paper at ICLR 2025


324
       • Detailed Issue Analysis: This criterion examines whether the Curator’s response extends beyond
325      the mere repetition of provided hints, indicating a profound understanding of the issue.
326
327
       • Relevance of Reasoning: This criterion ensures the Curator’s logical reasoning is directly pertinent
         to the addressed problem rather than being generic or irrelevant.
328
329   For each test case, the Evaluator receives the Curator’s output along with the hints and the ground
330   truth annotations about the issues in the test case. Then, the Evaluator is asked to rate the Curator’s
331   performance for each criterion with a real number from 0 to 1. The final score is a weighted sum of
332   the three ratings, with weights of 0.85, 0.15, and 0.05, reflecting their respective importance. The
333   Evaluator then categorizes the Curator’s performance according to defined thresholds on the final
334   score3 .
335   To enhance the accuracy of the evaluation, a simplified voting strategy similar to Verga et al. (2024)
336   was implemented to mitigate the influence of randomness. The strategy comprises three rounds of
337   voting. In the first round, we collect results from n Evaluators. If a consensus is reached, the decision
338   is made with the consensus; however, if there is no consensus, we conduct another round of voting
339   within m Evaluators, and apply a majority vote to make the decision. In the event of a tie, additional
340   votes will be conducted one by one until a definitive majority vote is determined. In practice, m and
341   n are both set to 2.
342   The testing results of the Evaluator on larger scale samples are displayed in Section 4.1, demonstrating
343   its reliable performance. Notably, we collect annotated test data for the Evaluator after we have
344   finalized the prompt design for our Evaluator. Therefore, we believe the proposed prompt design of
345   our Evaluator can generalize across different Curators and issues in the DCA-Bench.

354
      conduct a comparative analysis against human annotations. We randomly select 23 test cases for each
355   of the four hint levels, resulting in a total of 92 (issue, hint) pairs. Using these pairs, we collect the
356   outputs of a baseline Curator agent, which is based on the OpenAI Assistant API with GPT-4-0125
357   preview equipped with the Code Interpreter tool4 , on each of the 92 pairs. The Evaluator and human
358   annotators then rate the Curator outputs independently. For the Evaluator, we test four different
359   models as backends: GPT-4-0125 preview, GPT-4o-0513, GPT-3.5-Turbo, and Llama3-70B-Instruct.
360   Human annotators follow a structured codebook5 to ensure rating consistency. Finally, we compare
361   the Evaluator’s ratings with human annotations (ground truth) using a binary classification scheme:
362   fail versus succ (where succ combines both success and success+ ). Additional results
363
      based on a three-class classification scheme (fail, success, and success+ ) are provided in
      Appendix A.2.3.

396   Experimental Setup We apply our Evaluator with backend GPT-4-0125-preview to benchmark
397   the performance of some baseline Curators. We first experiment with the baseline Curator based on
398   the OpenAI Assistant API equipped with the Code Interpreter tool on the full set of test cases and
399
      hint levels. As pointed out by OpenAI API users6 , the OpenAI Assistant API’s performance is often
      worse than that of the web-interface ChatGPT, so we also test Curators based on the web-interface
400
      ChatGPT. However, experimenting with the web-interface ChatGPT is not scalable, as we need
401
      to manually feed the prompts to the browser. Therefore, we selected 32 (issue, hint) pairs where
402   the Assistant-API-based Curator fails, forming a hard set of DCA-Bench. Then, we carry out this
403   small-scale experiment on ChatGPT-4, ChatGPT-4o, and ChatGPT-4 with reference materials7 . The
404   model’s knowledge is limited to May 2024, when we conduct the experiments. The performance of
405   the Curator is classified as fail and succ by the Evaluator.

410   given the most informative hint (Hint Level 3). Despite this improvement, the success rate is still
411   unsatisfactory given the amount of information provided, as we cannot expect to have such informative

446   5       C ONCLUSION
447
      To help the development of LLM agents capable of dataset curation, we present DCA-Bench, a
448   collection of representative data quality issue cases from popular dataset platforms. Instead of fixing
449   predefined issues, DCA-Bench aims to test the agent’s capability to discover hidden data quality
450   issues, a critical initial step in the dataset curation pipeline. To efficiently and effectively evaluate
451   the performance of the Curator agents, we develop an LLM-based Evaluator with carefully designed
452   prompts, which aligns well with human annotators. We conduct a benchmark study on the baseline
453   Curator using DCA-Bench, and the results indicate that while LLMs have potential in real-world
454   dataset curation, further exploration and innovation are needed to fully realize their capabilities.
455
456   Limitations Dataset curation is a complex and comprehensive problem, and test cases we collect
457   might not fully cover the entire problem set. Additionally, due to the complexity of dataset files,
458
      we cannot guarantee that all the issues in the dataset files in DCA-Bench are labeled. Lastly, we
      haven’t considered other modality information such as images or audios, which may be helpful to
459
      effectively curate multimedia datasets. Based on our work, future studies could explore developing
460
      more complex LLM agent systems, best practices for handling multi-modal information in dataset
461   curation, or creating a more realistic simulation environment for testing LLM agents.

1120   We mainly focus on two types of dataset platforms. The first type includes HuggingFace and Kaggle,
1121   which hosts a huge volume of datasets and offers a dedicated discussion section for individual datasets.
1122   The second type is hosted on GitHub, such as BIG-Bench, where the data quality issues are identified
1123   from the issue or pull request (PR) sections. Roughly speaking, for HuggingFace and Kaggle, we first
1124   select a few popular datasets and then examine the discussion posts relevant to data quality issues.
1125   For GitHub-based repositories, we obtain data quality issues by browsing the issue or PR sections.
1126   Our detailed collection process is outlined as follows:
1127
1128
       Kaggle:
1129
1130
             1. We begin by using Kaggle API to select datasets that have ≥ 5 discussions with a proper
1131
                files size ≤ 512 MB, and collect them into a candidate list.
1132
1133         2. We manually review the discussion sections of these datasets, collecting discussions relevant
                to dataset issues and compiling them into a table.


                                                         21
       Under review as a conference paper at ICLR 2025


1134
             3. For each collected discussion, if the dataset issues are confirmed by either the dataset owner
1135            or other participants (with at least two reports of the issue), we consider it as verified. If not,
1136            we manually verify the issues by checking the corresponding files, which is straightforward
1137            due to the dataset’s version history control (in case the issue has been resolved in the current
1138            version).
1139
1140   HuggingFace:
1141
1142         1. We follow a similar approach to Kaggle, first selecting candidate datasets that have enough
1143            discussions and proper file size by using HuggingFace API
1144         2. Using the same strategy as for Kaggle, we collect discussions relevant to dataset issues and
1145            verify them through the same process.
1146
1147   Platforms Hosted on GitHub: On GitHub, dataset issues are identified from several sources,
1148
             1. Issues raised by platform maintainers during the review of new dataset contribution PRs.
1149
                These are inherently verified by the maintainers.
1150
1151         2. Issues reported by users in the issues section. If these issues are confirmed by the maintainers,
1152
                they are considered verified (And it’s usually linked to a PR so as to fix it). If not, we
                manually check the corresponding files ourselves.
1153
1154   We have also surveyed on other dataset hosting platforms, which we found are not ideal for
1155   constructing DCA-Bench. Zenodo is a platform for safely storing and sharing research data, which
1156   fully follows FAIR principles (Wilkinson et al., 2016). However, to our knowledge, it doesn’t support
1157   a feedback mechanism, which makes it unclear whether there are issues within those dataset files
1158   and whether the descriptions align with the datasets. Compared to HuggingFace and Kaggle, this
1159   lack of feedback makes the platform less informative about the issues that concern dataset users,
1160   which is why we did not collect datasets from it. Harvard-Dataverse faces similar issues, as it has a
1161   GitHub repository but does not use it for reporting dataset problems. Domain-specific datasets like
1162
       the Materials-Data-Facility or PANGAEA also share the same problem, making it difficult for us
       to collect issue samples from them, which is why they were less interesting for us. Open-Science-
1163
       Framework seems not to be a standard dataset hosting platform—it is a platform for hosting projects,
1164
       where the files do not necessarily have to be datasets, so we dismissed further discussions.

1168   A.2.1    D ESIGN D ETAILS OF THE E VALUATOR
1169
1170   In this part, we elaborate on the prompt design of our Evaluator. While it’s natural to ask LLMs to
1171   directly classify the Curators’ performance into fail/success/success+ , this strategy is poorly
1172
       aligned with the annotation of human experts as we tested. In addition to common prompt methods
       (e.g., COT (Wei et al., 2023), role-playing (Kong et al., 2024), delimiter (OpenAI, 2024b), we believe
1173
       more prompt engineering is needed to improve the performance of the Evaluator, and here is our
1174
       design:
1175
1176   We chose to have the Evaluator rate on specific metrics because this strategy allows for iterative
1177   refinement of the prompt by adjusting the weight and threshold of each metric. This approach
1178
       decreases uncertainty by introducing fixed numerical comparisons, which provides a clearer direction
       for tuning, as opposed to relying solely on textual standards, which can be more ambiguous.
1179
1180   In designing the Evaluator’s prompt, we started by selecting a small subset of human-labeled samples.
1181   We began with an initial weight distribution of (0.7, 0.2, 0.1) to reflect the relative importance of each
1182   metric in Section 3.4. The high initial weight for "Precise Contextual Evidence" was chosen to align
1183   with our focus on whether the Curator identifies annotated issues by providing contextual evidence,
1184
       as discussed in Section 3.4. We then fine-tuned these weights using the selected subset to better align
       the Evaluator’s prompt with human judgments.
1185
1186   The motivation behind this design is to make iterative refinement of prompts more effective. De-
1187   veloping a good prompt requires testing it on a subset of human-labeled issues with corresponding
       outcomes from Curators and then making adjustments based on the outcomes. Unlike the model


                                                           22
       Under review as a conference paper at ICLR 2025


1188
       training process, which benefits from clear gradients and systematic optimization, tuning text prompts
1189
       is often an empirical process necessitating numerous trials without the guarantee of improvement in its
1190   performance. Employing numerical weights and thresholds could enable us to tune the prompt more
1191   fine-grained, only changing the weights or threshold based on the results, which share similarities
1192   with linear regression, resulting in a more efficient prompt tuning process and better performance. By
1193   "training" on a this human-labeled subset, we finalized weights in the prompt as 0.85, 0.15, and 0.05.
1194
       It’s important to highlight that we collected samples for the test set used in Section 4.1 after finalizing
1195
       the prompt design, ensuring that there is no test data leakage to the prompt tuning.
1196
1197 To be more detailed, the Evaluator is prompted with the following instructions to evaluate the
1198  performance of the agent, with <ISSUE>, <HINT>, and <ANSWER> being the place-holder.
1199
1200
1201   You are required to act as an answer evaluator. Given an issue context,
1202       the hint disclosed to the agent and the answer from the agent,
1203
       you should rate the performance of the agent into three levels: "failed",
            "partially", and "success". The rating rules are as follows:
1204
1205   <rules>
1206   1. You will be given a list of <metrics>, for each metric, you should
1207       rate in [0,1] for the agent based on the metric criteria, and then
1208
           multiply the rating by the weight of the metric.
       2. If the sum of the ratings is less than 0.45, then the agent is rated
1209       as "failed"; if the sum of the ratings is greater than or equal to
1210       0.45 and less than 0.85, then the agent is rated as "partially"; if
1211       the sum of the ratings is greater than or equal to 0.85, then the
1212       agent is rated as "success".
1213
       3. **<text>** means the text is important and should be paid attention to.

1571   Here is a more detailed analysis of the overall performance with different hints. As outlined in
1572   L183-L186, the hints are divided into four levels, from 0 to 3, focusing on two dimensions with
1573   increasing information: the description of the issue and its specific context. The design follows an
1574   incremental pattern from h0 to h3. Please refer to Appendix 10 for some specific examples. Let’s
1575   continue by discussing the capabilities required to succeed at each hint level.
1576   If we aim to develop an LLM agent for this task without considering cost constraints, the process
1577   should involve the following steps: Retrieve Knowledge → Plan and Select Files to Check →
1578   Read Files and Validate Issues → Receive Feedback and Update State → (Loop) → Make Final
1579   Decisions. Since our Baseline Curator does not support looping, we exclude it from the following
1580   analysis. We will use this diagram to demonstrate which steps become less critical as the hint level
1581   increases.
1582
1583   h0: With no hints provided, the Curator must examine all the uploaded files. This requires the
1584   ability to handle and retain very long inputs effectively. To identify potential issues, the Curator needs
1585
       comprehensive knowledge of dataset curation to anticipate a wide range of problems and retrieve
       relevant information when provided with the dataset files. The task also involves exploring and
1586
       interacting with the dataset files to gradually narrow down the search space, requiring high-level
1587
       reasoning and planning abilities. This corresponds to no reduction in the original diagram’s required
1588   steps.
1589
1590
       h1: When given a general description of the issues but no specific location, the search space is
1591
       somewhat reduced but not entirely, as the description remains broad. A robust understanding of
1592   dataset curation and the ability to manage long inputs are still necessary. Ideally, this follows the
1593   pattern of Retrieval of Knowledge (↓), with minimal impact on the steps for planning and selecting
1594   files, except where file names are relevant. The steps for reading files and validating issues also see a
1595   slight reduction (↓).
1596
1597   h2: At this level, the Curator is provided with both a general description of the issue and specific
1598   hints about which files are involved. This further reduces the search space, making the ability to
1599   manage long inputs sufficient but less critical. The need for extensive knowledge retrieval decreases
1600   as the Curator can focus on the relevant files. Planning and file selection are simplified, as the Curator
1601   has clear guidance on which files to examine. Reading and validating issues also become easier,
1602   as the scope is narrower, reducing the overall complexity of the task. This follows the pattern of
1603
       Retrieval of Knowledge (↓), Planning and Choosing Files (↓), and Reading and Validating Issues (↓).
1604
1605   h3: At the highest hint level, the Curator receives the general description, the specific files involved,
1606
       and partial context. This greatly reduces the need for managing long inputs and makes the retrieval
       of knowledge more targeted. Planning and file selection is almost straightforward, with the Curator
1607
       essentially guided to the areas needing attention. Reading and validating issues are significantly
1608
       simplified due to the detailed context, further decreasing the Curator’s workload. This follows
1609   the pattern of Retrieval of Knowledge (↓ ↓), Planning and Choosing Files (↓ ↓), and Reading and
1610   Validating Issues (↓ ↓).

1784   A.5.1    D ISCUSSION ON THE D IFFERENCE BETWEEN O PENAI A SSISTANT API AND C HAT GPT
1785
1786   The performance discrepancy between OpenAI’s Assistant API and ChatGPT has been widely
1787
       discussed within the OpenAI user community. Users often report that the Assistant API’s performance
       does not match the capabilities of ChatGPT, especially in tasks involving complex file processing and
1788
       analysis.
1789
1790   The Assistant API provides several methods for handling files: 1. Code Interpreter for file uploads. 2.
1791   Building VectorStores via File Search. 3. Attaching files to messages using their file IDs.
1792   However, these methods have significant issues: - The quality of the VectorStore built using File
1793   Search is inferior to ChatGPT’s built-in file search capabilities. - Files are referenced by their file IDs
1794   rather than their names, leading to confusion and errors during processing.
1795
       The following code snippet illustrates the process of uploading and referencing files in the Assistant
1796
       API by attaching them to the messages:
1797
1798
       msg_file_ids = []
1799
       for file_path in file_paths:
1800      message_file = self.client.files.create(
1801         file=open(file_path, "rb"), purpose="assistants"
1802      )
1803      msg_file_ids.append(message_file.id)
1804
       attachments = [{"id": file_id, "tools": [{"type": "file_search"}]} for
1805       file_id in msg_file_ids]
1806
1807   thread = self.client.beta.threads.create(
1808      messages=[
             {
1809
                "role": "user",
1810            "content": input,
1811            "attachments": attachments
1812         }
1813      ]
       )
1814
1815
       Despite correctly assigning file names during the upload process, the Assistant API seems to ignore
1816
       these names during subsequent file processing, relying solely on file IDs. This results in frequent
1817   mix-ups and incorrect file references.
1818
1819   Several forum posts and articles highlight deficiencies relevant to this topic: - Different responses
1820
       assistant playground vs API - using the same assistant ID25 - Huge difference between ChatGPT
       assistant and API assistant26 - Assistant API not consistent27 . This implies the limitations of OpenAI
1821
       Assistant API.
1822
1823   In conclusion, while the Assistant API offers various tools for convenient file handling, its current
1824   implementation presents significant challenges that hinder its effectiveness compared to ChatGPT,
1825   which motivates us to select a subset of DCA-Bench containing hard cases to test on ChatGPT.
