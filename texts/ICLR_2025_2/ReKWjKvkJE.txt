                                                 A BSTRACT
010
011
               Recent advancements in large language models (LLMs) have shown promise in
012
               bridging the gap between natural language queries and database management
013            systems, enabling users to interact with databases without the background of SQL.
014            However, LLMs often struggle to fully exploit and comprehend the user intention
015            and complex structures of databases. Decomposition-based methods have been
016            proposed to enhance the performance of LLMs on complex tasks, but decomposing
017            SQL generation into subtasks is non-trivial due to the declarative structure of
018            SQL syntax and the intricate connections between query concepts and database
019            elements. In this paper, we propose a novel Structure GUided text-to-SQL
020            framework (SGU-SQL) that incorporates syntax-based prompting to enhance the
021
               SQL generation capabilities of LLMs. Specifically, SGU-SQL establishes structure-
               aware links between user queries and database schema and recursively decomposes
022
               the complex generation task using syntax-based prompting to guide LLMs in
023
               incrementally constructing target SQLs. Extensive experiments on two benchmark
024            datasets demonstrate that SGU-SQL consistently outperforms state-of-the-art text-to-
025            SQL baselines. These results highlight the importance of incorporating structural
026            syntax information for effective text-to-SQL generation and pave the way for more
027            robust and reliable interfaces to databases in the era of artificial intelligence.

030   1   I NTRODUCTION
031
032   Text-to-SQL is a challenging task that aims to bridge the gap between natural language queries
033   and database management systems, enabling users to interact with databases without knowing the
034   background of SQL. In the past few years, this task has been incrementally evolving due to the
035   complexity of SQL syntax and the intricate connections between user queries and database elements.
036   Models need to interpret intricate natural language queries and construct SQL queries with precise
037   syntax structure, all while linking with correct tables and columns in the database. A wide range
038
      of research has been proposed to address these issues, including intermediate query languages,
      graph-based modeling, and skeleton query generation (Wang et al., 2019; Li et al., 2023a;b).
039
040   Recently, this field has seen significant progress with the emergence of Large Language Models
041   (LLMs) like GPT series (Radford et al., 2018; Achiam et al., 2023; Brown et al., 2020; OpenAI,
042   2023). Training on a wide array of corpus, LLMs exhibit exceptional ability in understanding and
043   producing text that closely mimics human communication. Researchers have started exploring the
044
      potential of LLMs for text-to-SQL by leveraging their extensive knowledge reserves and superior
      generation capabilities (Rajkumar et al., 2022; Gao et al., 2024). These approaches often involve
045
      prompt engineering to guide proprietary LLMs in SQL generation (Chang & Fosler-Lussier, 2023;
046
      Pourreza & Rafiei, 2023) or fine-tuning open-source LLMs on text-to-SQL datasets (Gao et al., 2024).
047
048   Despite their advancements, LLM-based text-to-SQL models encounter several limitations that
049   impede their successful application in practice.
050   ❶ Ambiguous User Intent. Accurately interpreting the user’s intent in natural language remains a
051   significant challenge for LLM-based models. Natural language is inherently ambiguous and context-
052   dependent, making it difficult for models to discern precise requirements. For example, a query like
053   "Show me last quarter’s sales performance" requires the model to infer specific details such
      as relevant tables, metrics defining "performance" and the exact time frame for "last quarter".


                                                        1
      Under review as a conference paper at ICLR 2025


054
      Additionally, nuanced language involving implied conditions or comparisons, such as "better than
055   average" or "most recent" can lead to misinterpretations, resulting in queries that do not fully
056   align with the user’s intent.
057
058
      ❷ Sophisticated Database Architecture. Mapping natural language terms to specific database
      columns and tables is another critical area where LLM-based models struggle. Databases often have
059
      complex schemas with interrelated tables and non-intuitive naming conventions. For instance, a user
060
      referring to "customer purchases" might imply multiple tables like "Customers", "Orders" and
061   "OrderDetails" The model must accurately identify and relate these tables, which is challenging
062   without comprehensive schema awareness. Moreover, similar column names across different tables
063   can cause confusion, leading to incorrect selections and incomplete queries, especially in large or
064   poorly documented databases.
065
      ❸ Complex Syntax Structure of SQL. Generating syntactically accurate and logically coherent
066
      SQL queries is a challenging task. SQL requires precise clause arrangement, correct operator usage,
067   and adherence to grammatical rules. LLMs may produce queries with syntax errors, such as missing
068   commas, incorrect JOIN conditions, or misplaced keywords. Constructing complex queries involving
069   nested subqueries, aggregate functions, or window operations demands high precision, which is
070   typically beyond the current capabilities of LLMs. Recently, decomposition-based methods have
071   been proposed to enhance the performance of LLMs on complex tasks. However, decomposing the
072   complicated linked structure into smaller, manageable components for step-by-step SQL generation
073   requires effective strategies. Traditional approaches often struggle with handling complex queries due
074   to the declarative structure of SQL and the intricate connections between user queries and database
075
      elements.
076   In this paper, we propose a novel Structure Guided text-to- SQL framework (SGU-SQL). SGU-SQL
077   addresses the above issues by leveraging the structural information in queries and databases through
078   structure-aware linking and syntax-based decomposition, providing additional guidance to the LLM
079   for better SQL generation. Specifically, SGU-SQL represents user queries and databases into unified
080   and structured graphs and employs a tailored structure-learning model to establish a connection
081
      between the user queries and the databases. The linked structure is then decomposed into sub-syntax
      trees, guiding the LLMs to generate the SQL query incrementally. Our main contributions are
082
      summarized as follows:
083
084   • We identify the limitations of LLM-based Text-to-SQL models and introduce SGU-SQL, which
085     leverages structural syntax information to improve SQL generation capabilities of LLMs.
086   • SGU-SQL proposes graph-based structure construction to comprehend user query and database
087     structure and then link query and database structure with dual-graph encoding.
088
      • SGU-SQL introduces tailored structure-decomposed generation strategies to decompose queries with
089
        syntax trees and then incrementally generate accurate SQL with LLM.
090
091
      • Experiments on two benchmarks verify that SGU-SQL outperforms state-of-the-art baselines, includ-
        ing 11 fine-tuning models, 7 structure learning models, and 14 in-context learning models.

      3         T HE F RAMEWORK OF SGU-SQL
122
123
      In this section, we will introduce the key components of SGU-SQL in detail. We leverage the implicit
124
      structural information in both queries and databases from three aspects: (i) A graph-based structure
125   construction for both user query and database understanding; (ii) A tailored structure linking method
126   is proposed to map the natural language query to the relevant database elements. (iii) Structure-based
127   prompting to LLMs for accurate SQL generation, which decomposes the complex generation into
128   sub-tasks and guides LLMs to generate the SQL query incrementally, adhering to the necessary
129   syntax structure. The overall illustration is presented in Figure 1.
130
131   3.1           R EVISITING U SER Q UERY AND DATABASE VIA G RAPH
132
133   Bridging the gap between textual queries and the structured database poses several challenges.
134   Firstly, constructing an accurate structure that captures the relationships between query terms and
135
      database entities is a non-trivial task. Secondly, linking the query to the appropriate tables and
      columns in the database is challenging, especially when there is ambiguity or a lack of explicit
136
      connections. In this paper, we build a comprehensive query-schema graph designed to structure the
137
      query concept, the schema, and pre-defined relations between the query phrases and the tables or
138   columns present within the schema. The graph contains three key structures: (i) Query Structure
139   (Rq ): Encodes dependencies between tokens in the question, derived from its syntactic parse. (ii)
140   Database Structure (Rs ): Represents intrinsic relationships within the database schema, like foreign
141   keys. (iii) Linking Structure (Rl ): Aligns query entities with the columns or tables in the database.
142
143   3.1.1                U SER Q UERY U NDERSTANDING AND R EPRESENTATION
144
      A query graph can be depicted as Gq = (Vq , Rq ), where Vq denotes the node set that characterizes
145
      the keywords specified in the question, and Rq signifies the relationships among these keywords.
146
      To differentiate the relationship between various words, we establish three separate link categories,
147   including Forward-Syntax, Backward-Syntax and None-Syntax relations as defined in Table 9, to
148   encapsulate the particular syntactic connections among words in the vernacular question.
149
150
      a) Query Parsing: Syntactic parsing can help resolve structural ambiguities in the query by providing
      a hierarchical representation of the sentence structure. Specifically, we first define a context-free
151
      grammar Gq for the query language:
152
153                                                                                          Gq = (Nq , Σq , Pq , Sq ),                                                             (1)
154
      where Nq is a finite set of non-terminal symbols representing query concepts. Σq is a finite set
155   of terminal symbols representing query terms. Pq is a finite set of production rules that map non-
156   terminals to sequences of terminals and non-terminals. Sq ∈ Nq is the start symbol.
157
158
      The production rules Pq define the syntactic structure of the query language. For example, the set of
      production rules of SQL is listed in Figure 2-(a) of the Appendix.
159
160   Parsing a user query Q using the grammar Gq yields a syntax tree Tq = (Vq , Eq ), where Vq is the
161   set of vertices representing query concepts. Eq ∈ Rq is the set of edges representing syntactic
      relationships between the query concepts.

163   (words with multiple meanings) and syntactic ambiguity (multiple possible syntax trees). Let Q be
164   the set of all possible interpretations of a query q. The ambiguity challenge can be formulated as
165   selecting the most likely interpretation q̂ from Q:

171   to construct an accurate graph representation. Let M be the set of entities mentioned in the query and
172   E be the set of unique entities. The coreference resolution can be formulated as finding a mapping
173   function ϕ : M → E that maps each mention to its corresponding entity:
174                                        ϕ(m) = argmaxe∈E P (e | m),                                        (3)

176   where P (e | m) is the probability of entity e given the mentioned entity m.

187   3.1.2   DATABASE U NDERSTANDING AND R EPRESENTATION
188
189   To generate accurate SQL queries, text-to-SQL systems also need to have a comprehensive under-
190   standing of the database structure, including table names, column names, and relationships between
191   or across various tables/columns. Representing and encoding the database in a way that can be effec-
192
      tively utilized by the text-to-SQL model is a challenging task. In this paper, we introduce a schema
      graph to represent database structure. Specifically, let D be a database consisting of a set of tables
193
      T = T1 , T2 , . . . , Tn . Each table Ti ∈ T has a set of columns Ci = {Ci1 , Ci2 , . . . , Cim }. We define
194
      a database schema graph Gd = (Vd , Rd ) to represent the structure of the database schema, where S
195   denotes the set of nodes representing tables and columns, and Rd is the set of edges representing the
196   relationships between them.
197
198
      a) Node Representation: Each table Ti ∈ T is represented as a node vTi ∈ S in the schema graph.
      Similarly, each column Cij ∈ Ci of table Ti is represented as a node vCij ∈ S. The set of nodes S in
199
      the schema graph is defined as:
200
201                                S = vTi | Ti ∈ T ∪ vCij | Cij ∈ Ci , Ti ∈ T .                              (5)
202
203   b) Edge Representation: The relationships between tables and columns in the database schema are
204   represented as edges in the schema graph. As shown in Table 9, we define the following three types
205   of edges:
206
      • Table-Column Edges: For each column Cij ∈ Ci of table Ti , we add an edge E{Ti , Cij } ∈ RS
207
        connecting the table node vTi to the column node vCij . This edge represents the relationship
208

212

213

222   3.1.3   S TRUCTURE L INKING WITH D UAL G RAPH E NCODING
223
      The syntax tree Tq obtained from parsing the user query Q captures the syntactic structure of the
224
      query. It represents the hierarchical relationships between query concepts and terms, which is crucial
225
      for understanding the intent behind the query. By incorporating the syntax tree into the query graph
226   Gq , we preserve the syntactic structure of the query and its inherent meaning. The schema graph Gd
227   represents the structure of the database schema, with vertices representing tables and columns and
228   edges representing their relationships. By combining the syntax tree with the schema graph through
229   the mapping function ϕ, we establish a link between the query concepts/terms and the corresponding
230   schema elements. This mapping allows us to identify which tables and columns in the database are
231   relevant to the user query, enabling more accurate and targeted querying.

      3.2     S TRUCTURE -D ECOMPOSED P ROMPTING WITH S YNTAX T REE
288
289   3.2.1     D ECOMPOSING Q UERY WITH S YNTAX T REE
290
291   The performance of LLMs on complex tasks can be improved by using decomposing-based methods.
292
      However, decomposing a SQL query into subtasks is challenging due to its declarative structure
      and the intricate connections between query concepts. To this end, in this section, we introduce
293
      a context-free syntax tree that defined in Figure 2-(a) to break down the text-to-SQL generation
294
      task into smaller subtasks according to the syntax structure of the user query. Specifically, we first
295   employ the query parsing described in Section 3.1.1 to build the syntax tree to achieve a linguistic
296   understanding of the natural language query and then adopt a node mapper to match nodes in the
297   linguistic syntax tree to SQL operations (Kate, 2008). Following this, the original query can be
298   divided into several subtasks according to the SQL operations distributed on the syntax tree.
299
300   3.2.2    S UBTASK D ECOMPOSITION
301
302
      Given the context-free syntax tree T , we decompose the generation task into subtasks based on the
      syntactic structure of the query. Each non-terminal node n ∈ N in the tree represents a subtask that
303
      needs to be solved to generate the corresponding part of the SQL query. The decomposition process
304
      f : N → S that maps each non-terminal node to its corresponding SQL component, is illustrated at
305   Algorithm 1.
306
307   3.2.3    SQL G ENERATION
308
309   To generate the SQL component sn for a non-terminal node n ∈ N , we employ a LLM M that takes
310   the natural language query Q and the subtask context cn as input and produces the corresponding
311   SQL component:
312
                                             sn = M(Q, cn ).                                     (18)
313   The subtask context cn captures the relevant information from the context-free syntax tree T that is
314   needed to generate the SQL component for node n. It can include the parent node, sibling nodes,
315   and other relevant contextual information. The final SQL query S is obtained by combining the SQL
316   components generated for all the non-terminal nodes in the context-free syntax tree T , starting from
317
      the root node n0 : S = sn0 . By decomposing the text-to-SQL generation task into subtasks based on
      the syntax structure of the user query, we can leverage the hierarchical information captured by the
318
      context-free syntax tree to generate more accurate and structured SQL queries.

406   The effect of prompting strategy In this part, we conduct comprehensive experiments to investigate
407   the effectiveness of our proposed prompting strategy. Specifically, we compare the structure-based
408   decomposing strategy used in our SGU-SQL with other prompting strategies like CoT (Wei et al.,
409   2022) and few-shot prompting. As shown in Table 4 and 8, we can have the following observations.
410
411   Obs.1. Our structure-based decomposing significantly outperforms other simple prompting strategies.
412   Our method demonstrates superior performance across all tested LLMs. Specifically, compared
413   to CoT, our approach achieves an average improvement of 5.03%, while outperforming few-shot
414   prompting by 4.98% on average.
415
416   Obs.2. Our structure-based decomposing significantly outperforms other advanced prompting
417   strategies. The key distinction of our approach is that it dynamically decomposes queries based
418   on their syntax structure, rather than either using fixed decomposition patterns (like DIN-SQL) or
419   purely relying on LLM’s black-box understanding (like ACT-SQL, MAC-SQL). This syntax-aware
420   decomposition strategy proves more effective for handling complex SQL generation tasks.
421
422   Obs.3. Simple decomposing-based methods are ineffective in the text-to-SQL task. While
423   decomposing complex tasks into subtasks like CoT, can enhance model performance in many natural
424   language understanding tasks, it proves to be ineffective in the text-to-SQL task. As shown in Table 4
425   and 8, applying COT on PaLM-2 even leads to a performance decrease of 1.08% compared to
426   the naive few-shot prompting. This is attributed to the complex syntax of SQL, and the intricate
427   correspondence between query terms in user queries and database data units. Conversely, we formally
428
      define the meta-operations in SQL and propose a decomposing strategy according to the syntax tree
      to separate the query into subtasks. This boosts the LLMs’ comprehension of linked queries to
429
      generate accurate SQLs step by step.

      5     C ONCLUSION
522
      Recent advancements in large language models (LLMs) have shown promise in improving the
523
      accuracy of text-to-SQL generation. However, existing models typically input queries and database
524
      schemas into LLMs to perform semantic-structure matching and generate structured SQL, while often
525   overlook the structural information inherent in user queries and databases, which could significantly
526   enhance the generation of accurate SQL queries. This oversight can result in the production of
527   inaccurate or inexecutable SQL queries. To fully exploit the structure, we propose the structure-to-
528   SQL framework (SGU-SQL), which leverages the inherent structure information to improve the SQL
529   generation of LLMs. Specifically, SGU-SQL links user queries and databases in a structure-enhanced
530   manner. It then decomposes complicated linked structures with syntax trees to guide the LLM
531   to generate the SQL step by step. Extensive experiments on two benchmarks demonstrate that
532   SGU-SQL consistently outperforms state-of-the-art SQL generation baselines. These results highlight
533
      the importance of explicitly incorporating structural information for effective text-to-SQL generation.

      A.1     S TRUCTURE L EARNING FOR T EXT- TO -SQL
738
739   Definition 1. Structure Learning for Text-to-SQL: Given a natural language query D and a database
740   schema Q, the task of graph learning for Text-to-SQL aims to generate a graph-based representation
741   G that captures the structural and semantic relationships between the query and the schema, and
742   to learn a mapping function f : Gq → Gd , where Gq is the structural user queries, and Gd is the
743   corresponding database contents linked to the query Gq .
744   Let G = (V, E) denote the graph representation, where V is the set of nodes and E is the set of edges.
745   The nodes v ∈ V represent the entities and components in the query and schema, such as tables,
746   columns, and query tokens. The edges e ∈ E represent the relationships and dependencies between
747   the nodes. The graph learning task involves two main components, including graph construction and
748   graph representation learning.
749
750   A.1.1    G RAPH C ONSTRUCTION
751
      The first step is to construct the graph G from the query Q and schema D. This involves extracting
752
      relevant entities and relationships from the input and organizing them into a graph structure. The
753
      graph construction process can be formally defined as:
754
                                               G = Construct(Q, D),                                  (19)
755
      where Construct(·) is a method that maps the query and schema to the graph representation.

757
758   Once the graph is constructed, the next step is to learn meaningful representations of the nodes
759   and edges in the graph. This is typically achieved using Graph Neural Networks (GNNs), which
760   propagate information across the graph structure to capture the structural and semantic relationships.
761   The representation learning process can be formally defined as:
762
                                    h(l+1)

                                                  v , {hu : u ∈ N (v)}),                                (20)

764   where hv is the representation of node v at layer l, N (v) is the set of neighboring nodes of v, and
765   GNN(·) is the graph neural network function that updates the node representations based on their
766   neighbors. The learned graph representations are then used to generate the corresponding SQL query
767   S by applying a decoding function f to the graph:
768                                                S = f (G).                                           (21)
769
770   The objective of graph learning for Text-to-SQL is to optimize the parameters of the graph construction
771   and representation learning components, as well as the decoding function, to generate accurate and
772
      executable SQL queries from natural language queries and database schemas.

793
794   Text-to-SQL has witnessed significant evolution over the past few years. Early researchers focused on
795   well-designed rules, which were later superseded by deep learning-based techniques. More recently,
796   the integration of pre-trained language models (PLMs) and large language models (LLMs) has further
797   advanced state-of-the-art text-to-SQL generation. This section traces the developmental trajectory of
798   Text-to-SQL methods, highlighting the key milestones and innovations that have shaped the field.

802

803
      were suitable for simple database scenarios. However, the increasing complexity of database structure
804   and the high labor costs associated with rule-based methods have made such approaches impractical.
805   The advent of deep neural networks, such as sequence-to-sequence models and encoder-decoder
806   structures like LSTMs (Hochreiter & Schmidhuber, 1997) and Transformers (Vaswani et al., 2017),
807   has revolutionized the field of text-to-SQL (Guo et al., 2019; Choi et al., 2021). They automatically
808   learn a mapping from user queries to corresponding SQL queries. Typically, RYANSQL (Choi
809   et al., 2021) introduced intermediate representations and sketch-based slot filling to handle complex
      questions and improve cross-domain generalization. More recently, pre-trained language models

899
      In-context learning enhances LLM performance by providing detailed task instruction, background
900
      knowledge, and contextual examples during inference, thereby improving performance for specific
901   tasks. This approach has seen innovative applications in text-to-SQL, with strategies aimed at
902   optimizing prompt contents and formats based on user queries and database structures. Typically,
903   C3-SQL (Dong et al., 2023) designed a zero-shot prompting framework for ChatGPT with clear
904   prompting for effective input format and tailored hints for calibration and consistency checking
905   during the query generation. KATE (Liu et al., 2021) first investigated the impact of few-shot
906   examples on GPT-3’s performance. (Nan et al., 2023) further conducted a systematic investigation
907   into different demonstration selection methods and optimal instruction formats for prompting LLMs
908   in the text-to-SQL task, whereas DESEM (Guo et al., 2023) developed a domain-specific vocabulary
909
      masking technique, called similarity assessment, highlighting the relevance of SQL-specific terms.

910
      by complexity and breaking down the generation task into sub-problems and feeding the solutions
911
      of those sub-problems into LLMs to improve the generation performance of complex SQL queries.
912   DAIL-SQL (Gao et al., 2024) further enhanced the performance by incorporating suitable formatting
913   of the database schema and selecting examples based on skeleton similarities. Some recent work
914   improves the in-context learning framework by incorporating execution feedback through second-
915   round prompting for regeneration. For example, MRC-EXEC (Shi et al., 2022) introduced a natural
916   language to code translation framework with execution, which executes each sampled SQL query
917   and selects the example with the minimal execution result–based Bayes risk (Müller & Sennrich,
      2021). LEVER (Ni et al., 2023) proposed an approach to verify NL2Code with execution, utilizing a

1201   developers to manually create extensive training data. This process requires: (i) Writing natural
1202   language questions about the database. (ii) Creating the corresponding correct SQL queries. (iii)
1203   Validating the accuracy of both questions and SQL queries. This manual data collection process is
1204
       not only time-consuming but also requires expertise in both SQL and the specific database domain,
       making it a significant bottleneck for the practical deployment of text-to-SQL systems.
1205
1206   Generating synthetic training data based on a template-based approach. This method aims to
1207   eliminate the need for manual data collection by systematically generating training examples using
1208   predefined syntax templates and database schema information. The generation process operates
1209   in three coordinated stages: template selection based on database schema, schema integration by
1210
       populating templates with actual table and column names, and natural language query generation.

1249   E.3.2   API PLANNING
1250   API planning aims to generate a sequence of API calls to accomplish a given goal or user request.
1251   Each API is essentially a function with input parameters and return values. Each function can be
1252   treated as a table, where input parameters and return values are equivalent to columns in the table.
1253   Based on the data flow, we can build a graph to describe the dependencies between different APIs,
1254   transforming the API planning task into a problem similar to Text-to-SQL, as the dependency graph
1255   is analogous to the schema graph in text-to-SQL.
