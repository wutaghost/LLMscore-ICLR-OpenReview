                                                   A BSTRACT
010
011
               Retrieval-augmented classification and generation models significantly benefit
012
               from the early-stage fusion of high-quality text-based auxiliary metadata, often
013
               called memory, but they suffer from high inference latency and poor robustness to
014            noise. In classifications tasks, particularly the extreme classification (XC) setting,
015            where low latency is critical, existing methods incorporate metadata for context
016            enrichment via an XC-based retriever and obtain the representations of the relevant
017            memory items to perform late-stage fusion to achieve low latency. With an aim
018            of achieving higher accuracy while meeting the low latency constraints, in this
019            paper, we propose MOGIC, an approach for metadata-infused Oracle guidance
020            for XC tasks. In particular, we train an early-fusion Oracle classifier with access
021            to both query-side and label-side ground-truth metadata in the textual form. The
022
               Oracle is subsequently used to guide the training of any existing memory-based
               XC Disciple model via regularization. The MOGIC algorithm, when applied to
023
               memory-based XC Disciple models such as OAK, improves precision@1 and
024
               propensity-scored precision@1 by ∼2% on four standard datasets, at no additional
025            inference-time costs to the Disciple model. We also show the feasibility of applying
026            the MOGIC algorithm to improve the performance of state-of-the-art memory-
027            free XC approaches such as NGAME or DEXA, demonstrating that the MOGIC
028            algorithm can be used atop any existing XC-based approach in a plug-and-play
029            manner. Finally, we also show the robustness of the MOGIC method to missing
030            and noisy metadata settings. We will release code on acceptance.

033   1    I NTRODUCTION
034
035   Context enrichment is the process of incorporating metadata from an external source (referred to as
036   the memory for a model, comprising memory items) to improve the overall performance of a given
037   task. While these approaches have gained popularity in generation, in the form of retrieval-augmented
038   generation (or RAG) (Lewis et al., 2020; Gao et al., 2024; Fan et al., 2024), such context enrichment
039
      via an external store also benefits classification tasks (Guu et al., 2020a; Guo et al., 2023; Mohan
      et al., 2024; Long et al., 2022). This paper focuses on using memory to improve classification, and in
040
      particular when the label space is extremely large in the order of millions a.k.a. eXtreme Classification
041
      (XC) (Mohan et al., 2024). XC tasks involve sparse query representation, and are short-text in nature,
042   dealing with matching queries to keywords (for sponsored search ads (Dahiya et al., 2021; Jain et al.,
043   2016; Prabhu et al., 2018b)), queries to product titles (for product recommendations (Dahiya et al.,
044   2021; Medini et al., 2019; Mittal et al., 2021)), or queries to webpage titles (for tagging (Babbar &
045   Schölkopf, 2017; Chang et al., 2020; You et al., 2019)). In such XC tasks, auxiliary metadata often
046   has relevant diverse information that the input query does not, which can be leveraged to provide
047   better predictions. For example, on sponsored search ads task that involves query-to-ad-keyword
048   prediction, the query-side metadata is obtained by mining the organic search webpage titles clicked in
049   response to the query on the search engine, while on the Wikipedia categories prediction task, other
050
      Wikipedia article titles connected to the original page via hyperlinks could serve as the metadata. For
      a comprehensive overview of the XC preliminaries, please refer to Appendix L.
051
052   Memory based augmentation in XC remains an unsolved problem due to challenges of accurate and
053   fast memory access (low latency requirements are imposed by XC). While the ideal/golden memory
      items linkages are typically available during training data (referred to as ground-truth metadata), at

074   test time, these would be predicted from the available set (referred to as predicted metadata). Mohan
075   et al. (2024) found that the predicted metadata for test-time queries may either be unavailable (poor
076   zero-shot generalization), or the retrievals might be irrelevant (noisy metadata). These challenges
077
      can be characterized into: (a) sensitivity to retrieved metadata: low quality retrieval from memory
      leads to noisy augmentation to the query, degrading task performance. For example, Cuconasu et al.
078
      (2024); Yoran et al. (2024); Yu et al. (2023) have showed that, for text-based early-fusion models,
079
      robustness to retrieved memory item quality is critical to performance. (b) latency dependence on
080   form and timing of metadata infusion: textual metadata offers higher interpretability than embedding
081   based metadata, but textual metadata has high inference-time fusion latency. Various state-of-the-art
082   approaches within the retrieval augmentation settings can be viewed as balancing trade-offs between
083   these challenges.
084
      A recent approach to leverage memory metadata in XC (Mohan et al., 2024; Chien et al., 2023)
085
      demonstrated the advantage of using metadata to improve online extreme classification, there are mul-
086   tiple shortcomings. Since these approaches employs late-stage embedding-based fusion, their model
087   shows gains in terms of generalization and inference latency, but the quality of the representations
088   could be sub-par compared to early-fusion models. Additionally, unique to the classification setting,
089   there could also be label-side metadata that can be incorporated within the memory framework.
090
      In this paper, we design a novel memory-based approach for extreme classification that utilizes both
091
      query and label-side metadata and maintains low inference latency, by employing a combination of
092   early-fusion of text-based metadata and late-fusion of memory items. Our approach involves two
093   phases of training: Oracle training and Oracle-guided Disciple training. In the first phase, we train
094   an early-fusion Oracle classifier which has access to both query-side and label-side ground-truth
095   metadata in the text form. In the second phase, the Oracle is used to guide the training of any existing
096   memory-based XC Disciple model, by means of a regularization loss. Table 1 shows two examples
097   of queries with ground truth labels; predicted and ground truth memory items; and predictions from
098   MOGIC (OAK), OAK, MOGIC (NGAME), NGAME (Dahiya et al., 2023a) and Oracle. We observe
099   that for the first example, “Courts by type” is a good memory item, but some predicted metadata
100
      (“Landforms” and “Grasslands”) mislead OAK to produce bad predictions about geographical places;
      NGAME predictions are also bad. However, with MOGIC regularized training, MOGIC (OAK) and
101
      MOGIC (NGAME) were able to retain the original intent of the query and predicted various type
102
      tennis courts. Furthermore, in the second example, when retrieved metadata is completely irrelevant
103   to the query “Tangbe” which is about a village in Nepal, OAK’s prediction was completely around
104   the wrongly retrieved metadata while after Oracle guidance, MOGIC (OAK) was able to ignore the
105   noisy information and retrieved the right Wikipedia See Also pages of Tangbe village.

      Overall, our contributions are:
109
110    • We propose MOGIC, a Metadata-infused Oracle Guidance framework for Improved Extreme
111      Classification, that maintains real-world inference latency, while achieving 1-2% improvement
112      over state-of-the-art XC models.
113    • In the first phase, we train an early-fusion Oracle classifier which has access to both query-side
114      and label-side ground-truth metadata in the text form. In the second phase, the Oracle is used to

116      a regularization loss.
117    • Extensive experiments on four popular benchmark XC datasets show that (1) MOGIC improves
118      accuracy significantly in terms of standard metrics like precision, NDCG and propensity scored

123   2    R ELATED W ORK
124
125   Extreme Classification (XC): XC is a crucial component in ranking and recommendation sys-
126   tems (You et al., 2019; Guo et al., 2019; Dahiya et al., 2021; Mittal et al., 2021; Saini et al., 2021;
127   Gupta et al., 2023; Mohan et al., 2024). XC approaches learn a classifier associated with each
128   of the classes in the multi-label setting, with features obtained via classical approaches such as
129   bag-of-words (Babbar & Schölkopf, 2017; Prabhu et al., 2018b) or decision trees (Prabhu et al.,
130
      2018b) or deep-learning techniques that leverage either pre-trained (Jain et al., 2019) or learned (You
      et al., 2019; Jiang et al., 2021; Dahiya et al., 2023a) features. The closest approach to ours is that
131
      of OAK (Mohan et al., 2024), wherein an XC classifier, such as an NGAME (Dahiya et al., 2023a)
132
      encoder, is used to retrieve the metadata, and a single transformer attention layer is used to fuse
133   the representations of both the query and the retrieved metadata. The proposed MOGIC algorithm
134   leverages a text-based early-fusion model to improve the representations of the memory items in
135   OAK. Along another direction, models such as DEXA (Dahiya et al., 2023b) aggregate information
136   from the neighborhood of the encoder representations to form the context. Consequently, as we show
137   in Table 7, the MOGIC algorithm can also be applied atop models such as NGAME and DEXA to
138   improve performance.
139   Retrieval-augmented Generation (RAG): The RAG paradigm has become the defacto approach for
140   incorporating metadata for context enrichment in generative model, with the application typically
141   being that of question answering. Prior to RAG, models such as REALM (Guu et al., 2020b) have
142   leveraged external knowledge sources to improve the accuracy of the transformer encoders using a
143   retriever that selects relevant documents or passages from the memory, while an encoder fuses the
144   input text and memory items, computing an enriched embedding. RAG-based approaches (Lewis
145   et al., 2020; Akyurek et al., 2023; Zhang et al., 2023; Radhakrishnan et al., 2024; Muennighoff
146   et al., 2024) combine pre-trained parametric and non-parametric memory for language generation. In
147
      RAG settings, the memory, typically text-based, is infused with the query at input (Yang et al., 2018;
      Karpukhin et al., 2020; Qu et al., 2021; Lan et al., 2023; Lála et al., 2023; Yan et al., 2024). Other
148
      approaches incorporate task-specific memory, such as tabular data (Zha et al., 2023; Luo et al., 2023)
149
      or knowledge graphs (Gaur et al., 2022; He et al., 2024). We observe that retrieval-augmented models
150   benefit from early-fusion, and high-quality metadata, but suffer from high inference latency and poor
151   robustness to noise. MOGIC makes a novel contribution by introducing the textual early-fusion of
152   metadata into XC models while respecting latency constraints.
153
      Guided Representation Learning: Transferring capabilities via context-following from large
154
      language models (LLMs) to smaller ones (Kim & Rush, 2016; Gupta & Agrawal, 2022; Xu et al.,
155
      2024) has been widely studied. In the generative setting, models such as Alpaca (Taori et al., 2023),
156   Vicuna (Chiang et al., 2023), Self-instruct (Wang et al., 2023), etc., have been shown to use supervised
157   instruction-following fine-tuning to improve generation where the tuning data was generated using
158   LLMs. On the LLM-based classification task, AugGPT (Dai et al., 2023) employs a teacher LLM to
159   rephrase input sentence to improve general and clinical-domain classification performance. Various
160   other works (Gilardi et al., 2023; He et al., 2023; Gao et al., 2023; Li et al., 2024; 2023) have
161   considered guidance for LLM-based classification in the context of annotation generation, data
      clustering and curation, etc., but do not target the XC setting. The Oracle guidance framework in

196   3.1   T HE MOGIC F RAMEWORK
197
198
      MOGIC comprises four main components, (a) The base XC model (Disciple), either memory-based,
      or memory free; (b) XC task specific loss function; (c) Guidance loss function and (d) The Oracle
199
      O for guidance. In this paper, we primarily focus on memory-based XC models, and in particular,
200
      OAK. Please refer to Figure 1 for overall understanding of the integration of OAK into the MOGIC
201   framework. For more details, please refer to Appendix J. The four blocks are described below.
202
203         1. Disciple: Disciple, in this case, OAK is a trainable XC architecture with parameters θD
204            such that this model takes in query or label and outputs d dimensional embedding such that
205            Eθ : X → S d−1 lies on a unit sphere S d−1 .
206         2. Task-specific loss function: This loss function denoted by LDisciple is associated with the
207            task for which the Disciple is being trained. For instance for OAK, LDisciple is a triplet
208            margin loss function described in (Mohan et al., 2024).
209         3. Guidance loss function: The guidance is passed on by the Oracle to the Disciple using the
210            loss functions LAlignment and LMatching , more details in Section 3.3.
211         4. Oracle (O): Oracle is an encoder model (an LLM or a small language model (SLM)) with
212            parameters θO which is used to guide the Disciple in the second training phase of MOGIC
213            framework. A typical Oracle is computationally expensive but highly accurate embedding
214            based model. The model takes in query as well as labels and its associated metadata as
215            input to generate high quality representations in terms of the XC task. These embeddings
               are then used to guide the Disciple using Guidance loss. In MOGIC, we explore various

      Other Wikipedia article titles connected to the original page via hyperlinks in the article are used as
347
      metadata in this case.
348
349   Implementation details: We initialize the encoder with a MS-MARCO (Chen et al., 2024) pre-
350   trained DistilBERT and fine-tune it. Table 15 in the appendix summarizes all hyper-parameters
351
      for each dataset. Note that MOGIC uses golden linked metadata only at training time, whereas at
      inference time, these metadata linkages are induced (for a new query, links to metadata are predicted
352
      by the Disciple). MOGIC uses PyTorch and was trained on a machine with 4 AMD MI200 GPUs.

518   5      C ONCLUSION
519
520   We introduce MOGIC, a novel framework for enriching query representations using relevant metadata
521   without incurring high inference latency. This is achieved via a two phase training. The first phase
522   trains an Oracle using text metadata infusion both on the query as well as the label side. The second
523   phase involves guiding the training of a Disciple model using embeddings from the Oracle classifier.
524
      Through extensive experiments on four popular benchmark XC datasets, we have demonstrated that
525   MOGIC significantly outperforms state-of-the-art XC models, achieving improvements in terms of
526   precision, NDCG, and propensity scored precision. Moreover, MOGIC exhibits robustness to missing
527   and noisy metadata, making it a valuable tool for real-world applications.
528
      In conclusion, MOGIC represents a significant advancement in the field of extreme classification,
529
      offering a practical and effective solution for incorporating metadata to enhance model performance.
530
      Our work highlights the potential of Oracle-guided training for improving the robustness and accuracy
531   of memory-based models in challenging classification tasks.

974
       This section provides a detailed walkthrough of the MOGIC framework as depicted in Figure 1. The
975    Figure 3 shows a more detailed version of the architecture and it will help us to walk through an
976    example on the training and the inference of the MOGIC framework. Let us consider the query
977    “Grass Court” mentioned in Table 1. The framework has three major blocks for Query Processing
978    block, Label Processing block and Oracle representation block. The Query Processing block and
979    Label Processing block are part of the Disciple.
980
981    Query Processing block: involves the following steps
982
983           • Metadata Retrieval: The query “Grass Court” is sent to the memory bank to retrieve
984             relevant metadata, including “Courts by type”, “Landforms”, and “Grasslands”. The
985             memory bank contains vector representations for each of these metadata. These vector
986             representations are then sent for further processing.
987           • Query Encoding: The query “Grass court” is passed through the main encoder to obtain
988             its vector representation.
989           • Query Enrichment: The query representation is fused with the metadata representation
990             using a cross-attention layer to create an enriched query representation.
991
992    Label Processing block: similar to the above query processing
993

1058   MOGIC performs training of the Disciple model with the predicted metadata as opposed to OAK (Mo-

1061

1062
       with ground-truth metadata. While the combiner can filter out unnecessary metadata, maintaining
1063   a closer training-inference distribution proves helpful. This experiment shows that it would not be
1064   beneficial to train the models directly with ground truth metadata.

1068

1069   the predicted metadata is concatenated with the input and (b) both early and late fusion (Setting 5)

1256   L   P RELIMINARIES
1257
1258         • Ground-truth metadata: Besides the query text, often, a variety of auxiliary information is
1259           available in many domains, e.g., frequently clicked webpages for search queries in sponsored
1260           search, previously searched queries for web search query auto-completion, etc. Auxiliary
1261
               information available from disparate but related tasks often have relevant diverse information
               that the input query does not, which can be leveraged to provide better predictions. We
1262
               call such auxiliary information as ground-truth metadata. Metadata often has relevant
1263
               diverse information that the input query does not, which can be leveraged to provide better
1264           predictions. For example, on sponsored search ads task that involves query-to-ad-keyword
1265           prediction, the query-side metadata is obtained by mining the organic search webpage titles
1266           clicked in response to the query on the search engine, while on the Wikipedia categories
1267           prediction task, other Wikipedia article titles connected to the original page via hyperlinks
1268           could serve as the metadata.
1269         • Early fusion: When the query and metadata tokens are concatenated in the original text
1270           form itself (initial stage of processing) rather than in embedding form, we call it early fusion.
1271           This approach contrasts with late fusion, where these are combined at later stages.
1272
             • Late-stage fusion: When the query and metadata embeddings are combined after their
1273           tokens have been processed through multiple Transformer encoder layers, we call this
1274           combination as late-stage fusion.
1275
             • Metadata-infused Oracle: This is an Oracle model which is given access to ground truth
1276
               metadata. Metadata-infused Oracle is the crux of our MOGIC framework. In the two stage
1277
               method, Metadata-infused Oracle training forms the first stage. In this stage, an Oracle
1278           model is trained using both query-side and label-side ground-truth metadata. This metadata
1279           is in textual form and is used to enhance the training process.
1280
             • Memory-based models versus Memory-free models: Memory-based XC models are
1281
               models that have access to memory (metadata). Very few XC methods are memory-based.
1282           On the other hand, most of the XC methods do not leverage metadata at all and are therefore
1283           called memory-free methods.
1284
             • Query-side metadata: Additional auxiliary information related to the input query is called
1285
               query-side metadata. For example, in Table 1, for the query “Grass court”, query-side meta-
1286
               data can be “Tennis terminology”, “Sports rules and regulations”, “Tennis court surfaces”.
1287
1288
             • Label-side metadata: Additional auxiliary information related to a label is called label-
               side metadata. For example, for the label “Clay court”, label-side metadata is “Tennis
1289
               terminology”, “Sports rules and regulations”, “Clay”, “Tennis court surfaces”.
1290
1291         • Missing labels: XC tasks involve a vast number of labels, making it impractical for anno-
1292           tators to mark all potential labels. This inherent limitation often results in missing labels,
1293
               which are labels that should have been included in the ground truth but were inadvertently
               omitted.
1294
1295         • Memory bank: Memory bank is associated with both the queries and labels, having
               parameters θM . Formally, memory bank is represented by M ∈ RM ×D , each memory

       In this figure, we explain the detailed architecture of the proposed MOGIC framework. The input
1317
       query is “gummy candy”, and the expected label is “Jelly Bean”. The overall framework comprises
1318
       of 4 different parts: Disciple, task specific loss functions, guidance from the Oracle to the Disciple,
1319   and the Oracle itself. The framework comprises of 2 stages: Oracle training and Disciple training.
1320   The Oracle training leverages both the query-side as well as the label-side ground truth metadata.
1321   As shown in the figure, the query-side ground truth metadata includes 2 memory items: “Gummi
1322   candies” and “Candy”. The label side ground truth metadata includes just one memory item in
1323   this case: “gelatin desserts”. This rich metadata is concatenated with the query and the label
1324   in the text form to train the Oracle. In stage 2, we train the Disciple based on guidance from the
1325   Oracle. This stage 2 is illustrated in detail in the Figure 4. The Disciple denoted by the green box,
1326   consists of an encoder, memory bank and a combiner C . At train time, the Disciple first encodes
1327
       both the query as well as the label using the same encoder. Since the Disciple has to be low latency,
       it must involve late fusion. For late fusion, we need embedding representations of both the query
1328
       as well as the metadata items. To obtain embedding of the metadata items, the query is also sent to
1329
       the memory bank. The combiner actually performs the late fusion using cross attention and outputs
1330   an enhanced query representation. Of course while training the Disciple, we need to ensure that the
1331   task specific loss is minimized. This loss tries to maximize the similarity between the embedding
1332   of the label and the enhanced query representation, and is illustrated in the task part of the Figure 4.
1333   The guidance part of the Figure 4 shows how the guidance is passed on by the Oracle to the Disciple
1334   using 4 different loss functions. These loss functions try to maximize the similarity between (a)
1335   Oracle’s query representation and the Disciple’s query representation (b) Oracle’s label representation
1336   and the Disciple’s label representation (c) Oracle’s query representation and the Disciple’s label
1337   representation, and (d) Oracle’s label representation and the Disciple’s query representation.
