011                                               A BSTRACT
012
013            Recent advancements in large language models (LLMs) have greatly improved code
014            generation, specifically at the function level. For instance, GPT-4o has achieved a
015            91.0% pass rate on HumanEval. However, this draws into question the adequacy
016
               of existing benchmarks in thoroughly assessing function-level code generation
               capabilities. Our study analyzed two common benchmarks, HumanEval and MBPP,
017
               and found that these might not thoroughly evaluate LLMs’ code generation ca-
018
               pacities due to limitations in quality, difficulty, and granularity. To resolve this,
019            we introduce the Mostly Hard Python Problems (MHPP) dataset, consisting of
020            210 unique human-curated problems. By focusing on the combination of natu-
021            ral language and code reasoning, MHPP gauges LLMs’ abilities to comprehend
022            specifications and restrictions, engage in multi-step reasoning, and apply cod-
023            ing knowledge effectively. Initial evaluations of 26 LLMs using MHPP showed
024            many high-performing models on HumanEval failed to achieve similar success on
025            MHPP. Moreover, MHPP highlighted various previously undiscovered limitations
026            within various LLMs, leading us to believe that it could pave the way for a better
027
               understanding of LLMs’ capabilities and limitations.

      1   I NTRODUCTION
031
032   Large language models (LLMs) have recently driven striking performance improvements across
033
      various tasks (Ouyang et al., 2022; Touvron et al., 2023; OpenAI, 2023). Recent models such as
      Llama 3.1 (Dubey et al., 2024), CodeLlama (Rozière et al., 2023), CodeGemma (Team et al., 2024),
034
      and GPT-4o (OpenAI, 2024) have been successful in demonstrating their efficacy in code-related tasks
035
      from program repair (Haque et al., 2022; Jin et al., 2023) to automated testing (Lemieux et al., 2023;
036   Schäfer et al., 2024). LLMs are utilized to develop innovative tools aimed at aiding programmers to
037   write code more efficiently (Chen et al., 2021).
038
039
      Code generation is a key area for evaluating LLMs’ capabilities. Code generation broadly spans
      converting natural language prompts into executable code, not limited to predefined templates such
040
      as function signatures and docstrings. This process can range from pure text descriptions to complete
041
      code generation, emphasizing the versatility and adaptability required for LLMs. Our focus is on
042   Function-Level Code Generation. An example is illustrated in Figure 1. It emphasizes the translation
043   of natural language into functional code, underlining natural language comprehension’s importance
044   for creating accurate programming constructs. Benchmarks like HumanEval (Chen et al., 2021) and
045   MBPP (Austin et al., 2021) stand out in assessing these models, showcasing LLMs’ strengths in code
046   generation through their understanding of natural language and generation abilities. For instance,
047   GPT-4o (OpenAI, 2024) achieves a 91.0% pass rate on HumanEval (Chen et al., 2021).
048   However, on existing benchmarks, performance differences between models are insignificant - all
049   achieve high pass rates. We thus raise two concerns: 1) Basic datasets lack discriminative power to
050   distinguish model capabilities, making it difficult to assess their relative strengths and weaknesses.
051   2) High overall pass rates on existing tasks alone cannot determine if models have truly mastered
052   functional programming competency and encoding skills to address diverse challenges. To answer
053   these questions, we conducted detailed experiments with strong code models on the market, including
      closed-source models like GPT-4 (OpenAI, 2023), GPT-3.5 (OpenAI, 2022), and open-source models

      Through an extensive manual analysis, we identified 7 main challenges in code generation tasks,
      leading to the introduction of the Mostly Hard Python Problems (MHPP) dataset. MHPP consists
079
      of 210 unique, manually created Python programming problems, each supplemented by unit tests.
080
      MHPP focuses on comprehensively evaluating LLMs’ abilities to tackle various challenges in code
081   generation. This includes handling variance in natural language inputs, understanding newly defined
082   contexts, demonstrating commonsense, dealing with edge cases, following complex instructions,
083   using mathematical and algorithmic knowledge, and showing familiarity with coding principles. It is
084   important to note that each challenge within MHPP necessitates different degrees of natural language
085   comprehension and code reasoning abilities.
086

087   different weaknesses across models when addressing various challenges involved in code generation
088   tasks. Notably, the models struggled the most with challenges that required advanced algorithmic
089   reasoning. Our comprehensive experiments demonstrate that MHPP can effectively test model
090   performance against diverse code generation challenges. We hope MHPP can serve as a stepping
091   stone for a better understanding of LLM capabilities and limitations to advance code generation.
092   particularly in the domain of algorithmic reasoning.

141   2.2     H UMAN E VAL
142
143   We conduct an extensive error analysis for 3 LLMs on HumanEval, including GPT-4 (OpenAI, 2023),
144   GPT-3.5 (OpenAI, 2022) and DeepSeekCoder (DeepSeekAI, 2023) as depicted in Figure 2. We
145
      analyze the errors made by LLMs on HumanEval and categorize the code generation challenges that
      led to these mistakes into 7 types:
146
147   Distraction: The description is lengthy and contains redundant information. To address this challenge,
148   LLMs need to extract essential information needed for accurate code generation.
149   Redefinition: The description introduces new concepts or operational rules, presenting a counter-
150   factual scenario with corresponding explanations. LLMs need to comprehend this newly introduced
151   context for accurate code generation.
152
      Shortcut: This challenge requires LLMs’ unconventional thinking, solving such problems often
153
      involves concise solutions derived from logical reasoning, lateral thinking, and a grasp of knowledge
154
      including mathematics and game theory.
155
156   Commonsense: Understanding the problem relies on commonsense knowledge not explicitly ex-
157   plained in the description. Commonsense involves universally understood facts for humans, such as
158
      temporal, spatial, and geometric knowledge. LLMs need a solid grasp of commonsense to interpret
      the context and then generate code.
159
160   Cornercase: This challenge demands thorough thinking of the problem, paying close attention to
161   implicit boundary conditions that could affect the outcome. LLMs need to consider all the corner
      cases for correct code generation.

192   3     B ENCHMARK C ONSTRUCTION
193
194   To delve deeper into the capabilities and limitations of LLMs beyond the basic code generation capa-
195   bilities identified by MBPP and HumanEval, we have created a unique code generation benchmark
196   Mostly Hard Python Problems (MHPP). This benchmark comprises expert-curated problems tailored
197   specifically for the seven challenges we identified in code generation. Note that using HumanEval as
198   a starting point may limit the coverage of problem types and error patterns. Therefore, we actively
199   sought to generalize the problem types and address more realistic and challenging error patterns in
200   the creation of MHPP. We refer readers to Appendix C. Our annotation team includes 12 members,
201
      all of whom possess either a master’s or a Ph.D. degree in computer science.
202   To ensure the quality of our dataset, three members serve as meta-annotators. Based on the seven
203   challenges, the annotators were tasked with defining the problem statement for each challenge,
204   creating a single, self-contained Python function to solve the given problem, and developing test
205   cases to validate the semantic correctness of the function, as detailed in Section 3.1. Additionally, the
206   annotators were required to provide a ground-truth solution that successfully passed all the proposed
207
      test cases.
208   In defining the problems, annotators were instructed to formulate descriptions clear and detailed
209   enough to allow for the translation of these descriptions into code by a human, without further
210   clarification. To maintain the originality and integrity of MHPP, annotators were strictly prohibited
211   from directly copying problems from publicly accessible websites, or employing simple modifications
212   to existing problems, such as synonym replacements or paraphrasing, as outlined in Section 3.2.
213
214   3.1     C HALLENGE -S PECIFIC A NNOTATION
215
      We provide guidelines catered to the diverse requirements of annotating different challenges.


                                                                               4
      Under review as a conference paper at ICLR 2025


216
      Distraction: Annotators are required to create elaborate natural language descriptions that incorporate
217   redundant information. These descriptions should exceed 200 words and introduce distractions.
218
219   Redefinition: Annotators are required to introduce new concepts or operational rules, effectively
220
      creating counterfactual scenarios. Each problem should introduce more than one new concept along
      with comprehensive explanations.
221
222   Shortcut: Annotators are required to craft problems that permit concise solutions by lateral thinking,
223   or applying knowledge from mathematics and game theory.
224   Commonsense: Annotators are required to construct problems that are grounded in foundational
225   commonsense concepts. These problems should not include explicit explanations of the involved
226   commonsense principles, and more than one concept should be featured.
227
      Cornercase: Annotators are required to write problems with solutions that need to consider more
228
      than 1 corner case.
229
230   Complexity: Annotators are required to develop problems that have more than 3 operational steps
231   or hops of reasoning. An example would be a problem that necessitates sorting a list, extracting
232   maximum and minimum elements, and then calculating the difference between these elements.
233   Codesense: Annotators are required to craft problems that necessitate the utilization of more than 1
234   specific Python package, both internal and external, such as RE and Numpy.

      3.2   Q UALITY A SSURANCE
238
      To ensure the quality of MHPP, we initiated a comprehensive two-phase quality assurance process.
239
      Our primary goal in the first phase is to eliminate any risk of data contamination that may arise from
240
      the inclusion of problems that have previously appeared on open-access websites. To achieve this,
241   we tasked meta-reviewers with meticulously searching the Internet to ensure none of the problems
242   selected were already publicly available. Additionally, we employed a contamination detector (Li,
243   2023), to confirm a 0% contamination rate, resulting in the exclusion of 6 problems identified at this
244   stage. We then asked the annotators to annotate another 6 problems until all of the problems met the
245   requirements. Progressing to the second phase, our focus shifted towards ensuring that each problem
246   rigorously meets the specific criteria for the respective challenges. This entailed a detailed review of
247   every aspect of the problem, including the natural language description, the reference solution, and
248   the test cases, conducted by a panel of three meta-annotators.
249   To guarantee consistency and accuracy, we adopted an iterative approach wherein annotators were
250   tasked with addressing and rectifying any issues flagged by the meta-reviewers until unanimous
251   approval was obtained. In addition, in order to prevent the risk of future data contamination, we
252   build an evaluation pipeline to mitigate data leakage, rather than releasing the whole MHPP dataset
253   on popular platforms including HuggingFace or GitHub. Researcher can only get a result report by
254   submitting model outputs using API without knowing any test case or canonical solution.
255

257
258   Detailed statistics of MHPP are outlined in Table 1. The total number of our dataset is 210 and each
259   challenge category contains 30 questions. A significant observation is that the average problem in
260   MHPP contains 167.6 words and the corresponding solutions span across 14.9 lines of code. This
261   indicates a considerable increase in verbosity and code complexity when compared to benchmarks
262   such as MBPP and HumanEval. Furthermore, MHPP surpasses these benchmarks in the number of
263   test cases, with an average of 14.0 test cases per problem—higher than MBPP’s 3.0 and HumanEval’s
264   7.2. Further comparisons can be found in Appendix A.
265   Crucially, the design of MHPP specifically addresses more nuanced challenges and diverse context
266   formats, a distinction not observed in other datasets. For instance, challenges categorized under the
267   Distraction and Complex categories are marked by significantly longer descriptions, posing unique
268   challenges in context comprehension. Conversely, problems falling under the Shortcut class feature
269   notably fewer lines of code in their solutions, highlighting challenges in achieving concise problem

481   6.1   LLM S FOR C ODE
482
483   The burgeoning interest in LLMs for code has coincided with the profusion of openly available
484   code repositories and the pressing need to enhance the productivity of software developers. Initial
485   models predominantly focused on code generation tasks have included CodeT5 (Wang et al., 2021),
      AlphaCode (Li et al., 2022), CodeGen (Nijkamp et al., 2023), InCoder (Fried et al., 2023), Star-


                                                        9
      Under review as a conference paper at ICLR 2025


486
      Coder (Li et al., 2023a), SantaCoder (Allal et al., 2023), CodeFuse (Di et al., 2024), CodeShell (Xie
487   et al., 2024), and (DeepSeekAI, 2023; DeepSeek-AI et al., 2024), all of which were trained on
488   code. Contrastingly, models such as Codex (Chen et al., 2021) and CodeLLaMA (Rozière et al.,
489   2023) represent a subsequent stride, having been fine-tuned from foundation models (Brown et al.,
490   2020; Touvron et al., 2023). The evolution continued as LLMs leveraged instruction-like datasets
491   for fine-tuning. Among these, WizardCoder (Luo et al., 2023), Phi (Gunasekar et al., 2023; Li
492   et al., 2023b), MagiCoder (Wei et al., 2024), and SafeCoder (He et al., 2024) are notable examples.
493   Across various coding applications, these code LLMs have set new standards of excellence, show-
494   casing their prowess in domains including program repair (Haque et al., 2022; Jiang et al., 2023),
495
      automated testing (Lemieux et al., 2023; Deng et al., 2023), code translation (Rozière et al., 2020;
      Ahmad et al., 2023; Xue et al., 2024), type prediction (Mir et al., 2022; Wei et al., 2023), and code
496
      summarization (Hasan et al., 2021; Ahmed & Devanbu, 2022).
497
498
499
      6.2   C ODE G ENERATION B ENCHMARKS
500
      Code generation (Chen et al., 2021; Austin et al., 2021) has emerged as a vital domain for evaluating
501   LLMs, where models generate code snippets based on natural language descriptions, often given in
502   the form of docstrings. Creating datasets for this task is challenging, leading most efforts to source
503   natural language and code pairs from the Internet (Hendrycks et al., 2021; Li et al., 2022; Chandel
504   et al., 2022; Jain et al., 2022; Shinn et al., 2023) or use distant supervision (Agashe et al., 2019). For
505   instance, APPS (Hendrycks et al., 2021) compiles questions from open-access coding portals like
506   Codeforces and Kattis, covering a wide difficulty range. Similarly, CodeContests (Li et al., 2022) and
507   LeetcodeHard (Shinn et al., 2023) draw problems from specific platforms, enriching the diversity and
508   challenge of datasets. However, the training of LLMs on vast repositories, including GitHub, poses a
509
      risk of including solutions to these problems, thereby emphasizing the importance of hand-written
      sets like HumanEval (Chen et al., 2021) and MBPP (Austin et al., 2021) for accurate benchmarks.
510
      These datasets, based entirely on human-written questions, are pivotal despite their focus on simpler
511
      functions, highlighting a need for advancing benchmarks to match the growing capabilities of LLMs.
512   More code generation benchmarks are discussed in Appendix A.
513

515   7     C ONCLUSION
516
517   In this work, we construct the MHPP benchmark comprising 210 unique, manually created Python
518
      problems. The prime focus of MHPP is the semantic grounding of code generation, effectively
      measuring LLMs’ competence in comprehending detailed specifications and restrictions in natural
519
      language descriptions, undertaking complex reasoning, and employing code knowledge to facilitate
520
      the desired functionality. Upon applying MHPP, we observe that the most powerful LLM still struggles
521   on this challenging benchmark. We hope MHPP can shed light on understanding the capabilities and
522   limitations of LLMs for code generation and form a foundation for further improvements. Though
523   MHPP offers valuable insights into code generation, it’s important to acknowledge its limitations in
524   terms of data size and potential bias, which are provided in Appendix G.

       F     P OTENTIAL S TRATEGIES FOR I MPROVING LLM S ON MHPP
1114
1115
1116   Based on the experimental results of various LLMs on MHPP. We propose potential strategies for
1117
       overcoming the challenges of MHPP. We have devised a set of strategies tailored to each category of
       challenges as follows:
1118
1119   Distraction: To tackle this challenge, we propose incorporating controlled noise into the training data
1120   and designing tasks that require the model to identify the genuine development intent and generate
1121   corresponding code.
1122   Redefinition: We recommend enhancing the model’s exposure to knowledge-based data. This will
1123   improve its ability to comprehend concepts within questions. For new or contradictory definitions,
1124   we suggest refining the model’s in-context learning to prioritize the given context over general world
1125   knowledge. Techniques like symbol tuning could be beneficial for this purpose.
1126
       Shortcut: To address this, we propose augmenting the training data with more mathematical and
1127   logical reasoning tasks to help the model recognize patterns.
1128
1129   Commonsense: We recommend incorporating more relevant knowledge data. However, it’s crucial
1130
       to avoid overfitting. Models can benefit from interacting with real-world data, such as world models
       and multimodal data, including images, to enhance their understanding of spatial concepts.
1131
1132   Cornercase: We suggest training models with more real-world code data, especially data rich in
1133   corner cases, to strengthen this capability. Using non-code data with many corner cases and extremes
       can also enhance the model’s robustness and accuracy during training.

1176   G    L IMITATIONS OF MHPP
1177
1178   Data Size: The MHPP dataset indeed has a smaller scale compared to automatically generated
1179   datasets. This characteristic is intrinsic to hand-written datasets like HumanEval, to which MHPP is
1180   similar in terms of scale. While the dataset’s size enables a detailed analysis, we acknowledge that it
1181   could potentially restrict the diversity and representativeness of the data, thereby limiting the model’s
1182   ability to generalize to larger, more diverse codebases.
1183   Potential Bias: The focus of MHPP on function-level code generation might introduce certain
1184   biases due to the annotation process primarily targeting challenges encountered during the writing
1185   of functions. This emphasis may result in a bias towards specific types of errors or difficulties,
1186   which might not comprehensively represent the wide array of challenges encountered in real-world
1187   coding practices. We recognize the importance of acknowledging these potential biases in the dataset
       collection procedure.

       These limitations highlight the need for further research to develop strategies for effectively scaling
1189   up hand-written datasets while maintaining annotation quality. Extending the scope of the dataset
1190   beyond the function level to capture the broader context of code generation tasks is also important.
1191   By addressing these limitations, future code generation datasets can provide a more comprehensive
1192   picture of real-world software development challenges, ultimately leading to the development of
1193   more robust and versatile code generation models.
