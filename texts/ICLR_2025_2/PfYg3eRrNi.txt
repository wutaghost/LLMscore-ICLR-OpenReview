007                                                       A BSTRACT
008
009
                 Despite the potential of language model-based agents to solve real-world tasks
                 such as web navigation, current methods still struggle with long-horizon tasks
010
                 with complex action trajectories. In contrast, humans can flexibly solve com-
011
                 plex tasks by learning reusable task workflows from past experiences and using
012              them to guide future actions. To build agents that can similarly benefit from this
013              process, we introduce Agent Workflow Memory (AWM), a method for inducing
014              commonly reused routines, i.e., workflows, and selectively providing workflows
015              to the agent to guide subsequent generations. AWM flexibly applies to both of-
016              fline and online scenarios, where agents induce workflows from training examples
017              beforehand or from test queries on the fly. We experiment on two major web navi-
018              gation benchmarks — Mind2Web and WebArena — that collectively cover 1000+
019              tasks from 200+ domains across travel, shopping, and social media, among others.
020
                 AWM substantially improves the baseline results by 24.6% and 51.1% relative
                 success rate on Mind2Web and WebArena while reducing the number of steps
021
                 taken to solve WebArena tasks successfully. Furthermore, online AWM robustly
022
                 generalizes in cross-task, website, and domain evaluations, surpassing baselines
023              from 8.9 to 14.0 absolute points as train-test task distribution gaps widen.

      Language model (LM) based agents are rapidly improving, and are now able to tackle digital tasks
      such as navigating the web (Zhou et al., 2024; Deng et al., 2023) or operating mobile apps (Rawles
029
      et al., 2023; 2024). Current agents mostly integrate a fixed set of given examples via training (Fu
030
      et al., 2024; Murty et al., 2024) or in-context learning (Zheng et al., 2024). This allows them to
031   perform well on action sequences similar to those presented in these examples, but results in a lack
032   of robustness to changes in task contexts or environments (Deng et al., 2023). Essentially, they
033   fail to grasp the key to disentangling increasingly complex tasks — to extract and learn reusable
034   task workflows shared across similar tasks and environments (Yu et al., 2023; Wang et al., 2024a).
035   Moreover, as agents solve each task separately, they do not learn from past successes and failures,
036   and are therefore unable to adapt over time (Yoran et al., 2024).

078   adaptability (§3.1). On Mind2Web, we evaluate AWM on the cross-website and cross-domain test
079   splits to examine its domain generality, where it scores 8.9–14.0 absolute points higher over baseline,
080   and the margins become more substantial as the train-test distribution gap widens (§3.2). Both
081
      results show the superior generalization of AWM across tasks, websites, and domains.

089   2.1       P ROBLEM S TATEMENT
090
      For the purpose of this paper, we consider agents with a language model backbone L and text-based
091
      memory M , where the base memory contains documentation of built-in actions such as CLICK
092               1
      and TYPE. To solve a task specified by a natural language (NL) instruction q, the agent acts in an
093
      environment defined by a transition function T . For each time step ti , the environment state si gives
094
      observation oi , which is then passed into the model to generate action ai via L(q, M, oi ) → ai .
095   The action is executed in the environment and changes the state as T (si , ai ) → si+1 . This observe-
096   act loop iterates until the model predicts the stop action ai =STOP, or reaches a task termination
097   condition, e.g., a maximum pre-determined number of steps.
098
      Each completed task forms an experience e, which comprises an NL instruction q and a trajectory of
099
      steps attempting to solve the task, where each step p contains the agent observation o obtained from
100
      the current state, and action taken by the agent a, formulated as p = (o, a). Our goal is to induce
101   useful workflows W = {w} from the set of experiences E = {e} constructed from past or collected
102   examples, using an induction module I via I(E) → W. We add induced workflows into the agent
103   memory M as guidance for subsequent task-solving.
104
      Next, we introduce the workflow representation design, workflow induction method, and agent
105
      memory update with workflows in varied setups.

190   3     E XPERIMENTS
191
      In this section, we experiment on two major web navigation benchmarks – WebArena (§3.1) and
192
      Mind2Web (§3.2). For each benchmark, we first introduce the benchmark and top-performing base-
193   line methods, then present our AWM approach and showcase its ability to achieve superior task
194   success and generalization across varied setups.
195
196
      For both benchmarks, we conduct AWM on a website basis. In other words, we group examples by
      their associated websites, and respectively run AWM on each group. This mechanism maintains an
197
      small collection of workflows that are nonetheless relevent to the test tasks.
198
199
      3.1   W EBA RENA
200
201   WebArena (Zhou et al., 2024) provides 812 web navigation tasks on five websites that cover four
202   common application domains: e-commerce, social forum discussions, collaborative software devel-
203   opment, and content management. Most importantly, WebArena supports rigorous evaluation on the
204   functional correctness of agent trajectories.
205   We adopt the current state-of-the-art method without human-annotated site-specific knowledge,
206   BrowserGym (Drouin et al., 2024), which altered the agent default action space. We adopt the
207   BrowserGym framework and its default action space, and represent webpages using accessibility
208   trees, following the environment representation in Zhou et al. (2024). Because BrowserGym in-
209   puts both webpage HTML and accessibility tree representations, to keep a fair comparison with our
210   method, we also run the BrowserGym version with only accessibility tree webpage representations,
211   denoted as BrowserGymax −tree . We also compare to the SteP method (Sodhi et al., 2023), which
212
      uses 14 human expert written workflows tailored to solving WebArena. Our method, in contrast,
      uses no human supervision and is not tailored to the WebArena setting.
213
214   Following baseline approaches, we use GPT-4 (gpt-4-0613) with a temperature of 0.0 to ensure
215   mostly stable model outputs. Because WebArena only has test examples and no additional high-
      quality, domain-aligned examples exist, we only conduct AWM in the online setting as in §2.3.

305
306   Mind2Web (Deng et al., 2023) features web navigation in cross-task, website, and domain settings,
307   stressing the generality of agents on versatile operations and environments. Each task in Mind2Web
308   has a fixed number of steps; at each step, the agent needs to predict an action, which is evaluated by:
309   (1) element accuracy: to check if the correct page element is selected, (2) action F1 to check if the
310   action taken on the element is correct, and aggregating (1) and (2) yields (3) step success rate which
311   checks that both element and action selection are correct at the current step. Lastly, after completing
312   every step in the given task, the last metric (4) task-level success rate measures if all intermediate
313   steps are successfully conducted for this task, i.e., all steps for this task score 1 under metric (3).
314   Because Mind2Web provides a training set covering part of the tested websites (cross-task split), we
315   explore both the offline setting that induces workflows from the training set and applies to test sets,
316   and the online setting, where we stream workflow induction and inference on test queries (§2.3).

508   6    R ELATED W ORK
509
510   Web Agent Benchmarks The first modern and widely used web agent benchmark is Shi et al.
511   (2017)’s MiniWob, which evaluates across various scenarios such as flight booking. (Liu et al.,
512   2018) then created MiniWob++ with extra challenges. More recently, WebShop (Yao et al., 2022)
513   features a simulated e-commerce website and crowd-sourced text instructions. WebArena (Zhou
514   et al., 2024) integrates four more websites and enables realistic execution-based evaluations, and
515   VisualWebArena (Koh et al., 2024) extends with tasks that necessitate visual inputs. Mind2Web
516   (Deng et al., 2023) proposes versatile tasks and stresses agent generalization across websites and
517
      domains. We use WebArena and Mind2Web to evaluate our method’s task success and generality.
518   Enhancing Agents for Complex Tasks Many works improve agents by modifying their action
519   space, such as constraining its action search space (Liu et al., 2018), enabling LM self-feedback to
520   refine predicted actions (Sun et al., 2023), or incorporating human-designed actions to certain tasks
521   (Sodhi et al., 2023; Sarch et al., 2024). Other works explore ways to augment agent memory, such
522   as adding example demonstrations in context (Haluptzok et al., 2023; Zheng et al., 2024; Fu et al.,
523
      2024). However, high-quality examples are not always available or easy to collect. Our AWM can
      flexibly operate even when auxiliary examples are non-existent and only test queries are available.
524
525   Learning Common Procedures from Experiences Some works use full examples (Zheng et al.,
526   2024) as context for an agent, yet they entangle with example-specific contexts and face challenges
527   in extrapolating to other tasks or domains (Majumder et al., 2023). Many works propose to extract
528   frequently reused sub-routines from experiences with rule-based (Ellis et al., 2023; Bowers et al.,
529
      2023; Grand et al., 2023) or LM-based methods (Cai et al., 2023; Wang et al., 2024c;a) methods,
      and use them as auxiliary skills to ease future task-solving (Oh et al., 2017; Liang et al., 2023; Yu
530
      et al., 2023; Mao et al., 2023). We explored both rule- and LM-based methods to induce reusable
531
      workflows, and use them flexibly as context guidance that are free of environment grounding issues.
532

      7    C ONCLUSION
535
536   We propose agent workflow memory that induces, augments, and uses workflows, offline from avail-
537   able examples or purely online at inference time. We evaluate AWM on WebArena and Mind2Web,
538   and achieve 24.6% and 51.1% relative increases in task success rate. AWM also demonstrates its
539   superior generalization abilities across tasks, websites, and domains. We hope AWM sheds insight
      on and boosts advances in dynamic memory building and agent adaptations on varied digital tasks.

793   A.3   W ORKFLOW Q UALITY A NALYSIS
794
795   To provide intermediate information beyond the end-to-end task success, we propose several met-
796   rics to verify the quality of the model-induced workflows. (1) Number of workflows: The number
797   of workflows augmented to the memory, fewer workflows is better, whereas agents rely on fewer
798   workflows to achieve satisfactory performance. (2) Coverage: How many steps in the action tra-
799   jectory are covered by the workflows, higher coverage presumably signals the general applicability
800
      of the concerned workflow. (3) Function overlap: How much functionality overlap exists between
      workflows, we measure this by counting the number of overlapping sub-trajectories (≤ 2 steps) be-
801
      tween each workflow pair for the same website. Less overlap indicates more maximized workflow
802
      management. (4) Utility rate: How often are workflows used by test examples.
803
804   We evaluate the workflows on WebArena test examples and Mind2Web cross-task test examples.
805   We do not evaluate coverage on WebArena since it requires canonical trajectories, yet which are not
806
      available for WebArena. For Mind2Web, we do not evaluate on cross-website and cross-domain test
      examples since workflows induced from training examples do not have domain overlapping with
807
      these test examples, thus less applicable to them.
808

823   B    RULE -BASED W ORKFLOW I NDUCTION
824
825   Beyond LM-based workflow induction, we also explored a rule-based workflow induction method.
826   Our rule-based workflow induction module consists of two steps: (i) experience deduplication, and
827   (2) invalid action filtering.
828
      For deduplication, we extract the action sequence of the experience, e.g., extracting CLICK
829
      → CLICK → TYPE from the trajectory CLICK(’12’) → CLICK(’30’) → TYPE(’44’,
830
      "cat"). We group experiences by their action sequence and randomly select n (n = 1 by de-
831   fault) experiences from each group. Specifically on WebArena, where the task template for each
832   experience is available. We conduct another round of deduplication by grouping experiences by
833   their task template, and randomly selecting n (n = 1 by default) experiences from each group. This
834   process yields diverse experiences from the given set of experiences.
835
      Next, for each unique experience, we remove the invalid steps in its action trajectory. Invalid actions
836
      means actions that cannot be successfully executed on the environment, because the input arguments
837   do not meet the requirement of the action function. Specifically, we have one rule of determining
838   invalid actions for CLICK and TYPE, that requires the first argument to be a string-formatted in-
839   teger (which refers to the id of an element in the environment). We remove CLICK and TYPE
840   steps if they do not meet this requirement. For example, an experience with trajectory CLICK(12)
841   → CLICK(’12’) → CLICK(’30’) → TYPE(44, "cat") → TYPE(’44’, "cat") will
842   yield CLICK(’12’) → CLICK(’30’) → TYPE(’44’, "cat"). We conduct this invalid ac-
843   tion filtering for each unique experience, and take the resulting experiences as rule-based workflows.
