009                                                 A BSTRACT
010
011
               User interface understanding with vision-language models has received much atten-
012
               tion due to its potential for enabling next-generation software automation. However,
013
               existing UI datasets either only provide large-scale context-free element annota-
014            tions or contextualized functional descriptions for elements at a much smaller scale.
015            In this work, we propose the AutoGUI pipeline for automatically annotating UI
016            elements with detailed functionality descriptions at scale. Specifically, we lever-
017            age large language models (LLMs) to infer element functionality by comparing
018            the UI content changes before and after simulated interactions with specific UI
019            elements. To improve annotation quality, we propose LLM-aided rejection and
020            verification, eliminating invalid and incorrect annotations without human labor.
021            We construct an AutoGUI-704k dataset using the proposed pipeline, featuring
022
               multi-resolution, multi-device screenshots, diverse data domains, and detailed
               functionality annotations that have never been provided by previous datasets. Hu-
023
               man evaluation shows that the AutoGUI pipeline achieves annotation correctness
024
               comparable to trained human annotators. Extensive experimental results show that
025            our AutoGUI-704k dataset remarkably enhances VLM’s UI grounding capabilities,
026            exhibits significant scaling effects, and outperforms existing web pre-training data
027            types. We envision AutoGUI as a scalable pipeline for generating massive data
028            to build GUI-oriented VLMs. AutoGUI dataset can be viewed at this anonymous
029            URL: https://huggingface.co/AutoGUI.

      1    I NTRODUCTION
034
035   User interface understanding with visual language models(VLMs) (Hong et al., 2023; Cheng et al.,
036   2024; You et al., 2024a; Lee et al., 2023; Baechler et al., 2024) has received wide attention due to
037   its potential in fundamentally transforming how we interact with software as well as unleashing
038   unseen flexibility for existing apps (Fig. 1). Functionality prediction, which aims to understand the
039
      semantic purpose and interactive affordance of individual UI elements, is a crucial task that goes
      beyond previous UI understanding tasks focusing on structural mapping between UI code and visual
040
      layout, such as UI REG/REC (Hong et al., 2023; Li et al., 2020a) and diagram to code (Xia et al.,
041
      2024; Liu et al., 2023a).
042
043   To enhance the UI understanding capability of VLMs, large-scale high-quality training data is
044   indispensable. However, the scale of existing open-source datasets (Li et al., 2020a; Deka et al.,
045
      2017a; Li et al., 2020b; Kapoor et al., 2024; Wang et al., 2021) for UI understanding remains on the
      order of millions, significantly fewer than natural image datasets such as LAION-5B (Schuhmann
046
      et al., 2022). Additionally, the prevailing methods (Deka et al., 2017a; Li et al., 2020a) for collecting
047
      UI annotation are labor-intensive, leading to prohibitive costs that hinder scalability. Moreover,
048   existing UI understanding datasets predominantly focus on describing either the visual appearance (Li
049   et al., 2020a;b) (e.g., a button beside the navigation bar), element categories (Cheng et al., 2024) (e.g.,
050   “menu button”), or brief functions weakly related to the UI context (Bai et al., 2021) (e.g., “show
051   more information”) shown in Fig. 2. These datasets lack contextual functional descriptions of UI
052   elements, which poses a challenge for VLMs in comprehending the roles these elements serve within
053   specific UI contexts, such as distinguishing between two visually similar magnifying glass icons that
      may represent distinct functionalities like searching and zooming.

071   To address the challenge. we propose AutoGUI, a scalable and automatic UI data annotation pipeline
072   that provides unlimited UI element functionality annotations. Our annotation pipeline automatically
073   collects UI interaction trajectories and leverages large language models (LLMs) to infer element
074   functionalities based on UI content changes, eliminating the need for manual annotation by human
075   experts. Initially, the proposed pipeline crawls a multitude of interaction trajectories on either a web
076   browser or an Android emulator and captures screenshots at various aspect ratios. Subsequently, we
077   use open-source LLMs (AI@Meta, 2024) to annotate the functionalities of elements on collected
078
      GUIs based on changes to UI contents when interacting with these elements. To ensure data quality,
      LLM-aided rejection is utilized to eliminate invalid samples, such as incompletely rendered UIs.
079
      Additionally, inspired by recent works on LLM verification (Weng et al., 2022; Lightman et al.,
080
      2023), multiple LLMs are prompted as verifiers to identify false functionality predictions. With
081   both the rejection and verification processes, our pipeline removes unclear and invalid samples.
082   We curate the AutoGUI-704k dataset with the proposed pipeline. AutoGUI-704k contains 704k
083   high-quality functionality grounding and referring tasks used to finetune and evaluate open-source
084   VLMs. With the vast knowledge embedded within LLMs (e.g., Llama-3-70B (AI@Meta, 2024))
085   and fast inference infrastructure (Kwon et al., 2023; Gugger et al., 2022), our pipeline can efficiently
086   annotate high-quality samples at a large scale and substantially reduced cost compared to traditional
087   methods. Moreover, pioneer experiments find that our pipeline achieves annotation accuracy of
088   96.7% comparable to a trained human annotator.
089   Based on the collected AutoGUI-704k dataset, we finetune open-source VLMs that own little UI
090   grounding capabilities. Experimental results demonstrate that data collected through our AutoGUI
091   pipeline significantly enhances the VLMs’ UI grounding accuracy and exhibits remarkable scaling
092   effects. The results also show that our functionality annotation type is superior to the data type
093   directly derived from web HTML code (Hong et al., 2023; Cheng et al., 2024), serving as a promising
094   data source for building VLMs capable of UI grounding.

133   2.2      E XISTING UI DATASETS AND B ENCHMARKS
134
135   Unlike mature natural image datasets (Russakovsky et al., 2014; Schuhmann et al., 2022), UI
136   understanding datasets have received less attention in computer vision. Several efforts have been
137   made to develop mobile UI modeling datasets (Wang et al., 2021; Li et al., 2020a;b; Bai et al., 2021;
138   Burns et al., 2022), primarily annotating the RICO dataset (Deka et al., 2017b), which includes
139   72K screenshots from Android apps. Examples include Widget Captioning (Li et al., 2020a), which
140   analyzes captions and linguistic features of UI elements, and RICOSCA (Li et al., 2020b), which maps
141   single-step instructions to UI locations. Recently, MoTIF (Burns et al., 2022) and AITW (Rawles et al.,
142   2023) have been proposed to focus on interpreting high-level instructions in Android environments.
143
      However, these manually curated and crowd-annotated datasets are limited in size and costly to
      update, presenting challenges in adapting to new UI types.
144
145   The web scenario has also gained much attention. WebShop (Yao et al., 2022), as an early at-
146   tempt, introduces a simplified simulator for web navigation tasks. More recent projects, such as
147   Mind2Web (Deng et al., 2024) and WebArena (Zhou et al., 2023), have developed realistic and repro-
148   ducible web environments to improve web agent capabilities. VisualWebBench (Liu et al., 2024b) has
149
      established a comprehensive evaluation framework for VLMs, focusing on UI grounding. To tackle
      data insufficiency issues, recent studies like SeeClick (Cheng et al., 2024) and CogAgent (Hong et al.,
150
      2023) have utilized the latest Common Crawl data to create large-scale datasets. However, these data
151
      are derived from HTML code snippets which contain plenty of noise.
152
153   This paper aims to address the aforementioned limitations of existing UI datasets by introducing
154   an automatic LLM-based annotation pipeline. By focusing on contextual functional descriptions of
155
      elements, our pipeline aims to enhance VLM’s capability of understanding users’ functional intents.
      The advantages of our AutoGUI dataset over existing datasets are summarized in Tab. 1.

208   3.1        C OLLECTING UI I NTERACTION T RAJECTORIES
209
210   Our pipeline initiates by collecting interaction trajectories, which are sequences of UI contents
211   captured by interacting with UI elements. Each trajectory step captures all interactable elements and
212   the accessibility tree (AXTree) that briefly outlines the UI structure, which will be used to generate
213   functionality annotations. To amass these trajectories, we utilize the latest Common Crawl repository
214   as the data source for web UIs and Android Emulator for mobile UIs. Note that illegal websites
215   and Apps are excluded manually from the sources to ensure no pornographic or violent content is
      included in our dataset. Please refer to Sec. A.2 for collecting details and data license.

      3.2      F UNCTIONALITY A NNOTATION BASED ON UI DYNAMICS
217
218   Subsequently, the pipeline generates functionality annotations for elements in the collected trajectories.
219   Interacting with an element e, by clicking or hovering over it, triggers content changes in the UI. In
220   turn, these changes can be used to predict the functionality f of the interacted element. For instance,
221   if clicking an element causes new buttons to appear in a column, we can predict that the element
222   likely functions as a dropdown menu activator (an example in Fig. D). With this observation, we
223   utilize a capable LLM (i.e., Llama-3-70B (AI@Meta, 2024)) as a surrogate for humans to summarize
224
      an element’s functionality based on the UI content changes resulting from interaction. Concretely, we
      generate compact content differences for AXTrees before (st ) and after (st+1 ) the interaction using a
225
      file-comparing library1 . Then, we prompt the LLM to thoroughly analyze the UI content changes
226
      (addition, deletion, and unchanged lines), present a detailed Chain-of-Thoughts (Wei et al., 2022)
227   reasoning process explaining how the element affects the UI, and finally summarize the element’s
228   functionality.
229
230
      In cases where element interactions significantly transform the UI and cause lengthy differ-
      ences—such as navigating to a new screen—we adjust our approach by using UI description changes
231
      instead of the AXTree differences. Specifically, we prompt the same LLM to discern the UI hierarchy,
232
      describe UI regions, and finally describe the entire UI functionality. After describing the UIs before
233   and after the interaction, the LLM analyzes the description differences, presents reasoning, and
234   summarizes the element’s functionality. This annotation process is formulated as:
235
236                                               f = LLM(panno , st , st+1 )                               (1)
237
238   where f is the predicted functionality, panno is the annotation prompt (Tab. A and Tab. B). Examples
239
      of annotated elements are depicted in Fig. 4 and more annotation details are explained in Sec. A.4.
240
241   3.3      R EMOVING I NVALID S AMPLES VIA LLM-A IDED R EJECTION
242
      The collected trajectories may contain invalid samples due to broken UIs, such as incomplete UI
243
      loading. These samples are meaningless as they contain corrupted UI content and can mislead the
244   models trained with them.
245
246   To filter out these invalid samples, we introduce an LLM-aided rejection approach. Initially, hand-
247
      written rules are used to detect obvious broken cases, such as blank UI contents, UIs containing
      elements indicating content loading, and interaction targets outside of UIs. While these obvious cases
248
      constitute a large portion of the invalid samples, there are a few types that are difficult to detect with
249
      hand-written rules. For instance, interacting with a “view more” button might unexpectedly redirect
250   the user to a login page instead of the desired information page due to website login restrictions.
251   To identify these challenging samples, we prompt the annotating LLM to also act as a rejector.
252   Specifically, the LLM takes the UI content changes, generated using a file-comparing library, as
253   input, provides detailed reasoning on whether the changes are meaningful for predicting the element’s
254   functionality, and finally outputs predictability scores ranging from 0 to 3. This process is formulated
255   as follows:
256                                       score = LLM(preject , e, st , st+1 )                               (2)
257   where preject is the rejection prompt (Tab. C).
258
      This approach ensures that clear and predictable samples receive higher scores, while those that are
259
      ambiguous or unpredictable receive lower scores. For instance, if a button labeled "Show More", upon
260   interaction, clearly adds new content, this sample will considered to provide sufficient changes that
261   can anticipate the content expansion functionality and will get a score of 3. Conversely, if clicking on
262   a "View Profile" link fails to display the profile possibly due to web browser issues, this unpredictable
263   sample will get a score less than 3.
264
      After implementing empirical experiments, we deploy this LLM-based rejector to discard the bottom
265
      30% of samples based on their scores to strike a balance between the elimination of invalid samples
266   and the preservation of valid ones (More details in Sec. A.6). The samples that pass the hand-written
267   rules and the LLM rejector are subsequently submitted for functionality annotation. Please see
268   representative rejection examples in Fig. H.

      3.4   I MPROVING A NNOTATION Q UALITY VIA LLM-BASED V ERIFICATION
289
290   The functionality annotations produced by the LLM probably contain incorrect, ambiguous, and
291   hallucinated samples (See a case in Fig. 3), which probably misleads the trained VLMs and compro-
292   mises evaluation accuracy. To improve dataset quality, we prompt LLMs to verify the annotations
293   by checking whether the targeted element e fulfills the intent of the annotated functionality f . This
294   process presents the LLMs with the interacted element, its UI context, the UI changes induced by
295   this element, and the functionality generated in the previous annotation process. The LLMs are
296   then tasked with analyzing the UI content changes before predicting whether the interacted element
297
      aligns with the given functionality. If the LLMs determine that the interacted element fulfills the
      functionality given its UI context, the LLMs will grant a full score (An example in Fig. I). If the
298
      interacted element is considered to mismatch the functionality, this functionality can be seen as
299
      incorrect as this mismatch indicates that it may not accurately reflect the element’s actual role within
300   the UI context.
301
302   To mitigate the potential biases in LLMs (Panickssery et al., 2024; Zheng et al., 2023; Bai et al.,
303
      2024), two different LLMs (i.e., Llama-3-70B (AI@Meta, 2024) and Mistral-7B-Instruct-v0.2 (Jiang
      et al., 2023)) are employed as verifiers and prompted to output 0-3 scores. The scoring process is
304
      formulated as follows:
305
                                       score = LLM(pverify , e, f, st , st+1 )                       (3)
306
307   where pverify denotes the verification prompt (Tab. D). Only if the two scores are both 3s do we
308   consider the functionality label correct (More details in Sec. A.7). Although this filtering approach
309
      seems stringent, we can make up the number of annotations through scaling.
310
311   3.5   F UNCTIONALITY G ROUNDING AND R EFERRING TASK G ENERATION
312   After rejecting, annotating, and verifying, we obtain a high-quality UI functionality dataset contain-
313   ing triplets of {UI screenshot, Interacted element, Functionality}. To convert this dataset into an
314   instruction-following dataset for training and evaluation, we generate functionality grounding and
315   referring tasks using diverse prompt templates (see Tab. E). To mitigate the difficulty of predicting

354
355   This section analyzes the reliability of the proposed annotation pipeline and data quality.
356
357
      Comparison with Human Annotation To demonstrate the superiority of the proposed automatic
      annotation pipeline based on open-source LLMs, N = 145 samples (99 valid and 46 invalid) are
358
      randomly selected as a testbed for comparing the annotation correctness of a trained human annotator
359
      and the pipeline. Here, correctness is defined as Correctness = C/(N − R), where C and R
360   denote the numbers of correctly annotated and rejected samples, respectively. The denominator
361   subtracts the number of rejected samples as we are more interested in the percentage of correct
362   samples after rejecting the samples considered invalid by the annotator. The authors thoroughly check
363   the annotation results according to the three criteria in Fig. 6: 1. Context-specificity. The functionality
364   annotations must include context-specific descriptions to ensure one-to-one mapping between the
365   element and its annotation. 2. Appropriate details. Avoid detailing unnecessary aspects of the UIs to
366   keep the description focused on functionality. 3. No hallucination. The annotations must not include
367   information not grounded in the visual context of the UIs. See more details in Sec. B.1.
368   After experimenting with three runs, Tab. 3 shows that the proposed AutoGUI pipeline achieves
369   high correctness comparable to the trained human annotator (r6 vs. r1). Without rejection and
370   verification (r2), AutoGUI is inferior as it cannot recognize invalid samples. Notably, simply using
371   the rules written by the authors can improve the correctness, which is further enhanced with the
372   LLM-aided rejector (r4 vs. r3). Moreover, utilizing the annotating LLM itself to self-verify its
373   annotations helps AutoGUI surpass the trained annotator (r5 vs. r1). Introducing another LLM
374
      verifier (i.e., Mistral-7B-Instruct-v0.2) brings a slight increase which results from Mistral recognizing
      Llama-3-70B’s incorrect descriptions of how dropdown menu options work. Overall, these results
375
      justify the efficacy of the AutoGUI annotation pipeline.

      AutoGUI employs the strong LLM to generate more detailed and clear annotations which would take

      A) AutoGUI functionality annotations effectively enhance VLMs’ UI grounding capabilities
474   and achieve scaling effects. We endeavor to show that the element functionality data autonomously
475   collected by AutoGUI contributes to high grounding accuracy. The results in Tab. 4 demonstrate
476   that on all benchmarks the two base models achieve progressively rising grounding accuracy as the
477   functionality data size scales from 25k to 702k, with SliME-8B’s accuracy increasing from merely
478   3.2 and 13.0 to 62.6 and 44.0 on FuncPred and ScreenSpot, respectively. This increase is visualized in
479   Fig. K showing that increasing AutoGUI data amount leads to more precise localization performance.
480   After fine-tuning with AutoGUI 702k data, the two base models surpass SeeClick, the strong UI-
481   oriented VLM on FuncPred and MOTIF. We notice that the base models lag behind SeeClick and
482   CogAgent on ScreenSpot and RefExp, as the two benchmarks contain test samples whose UIs cannot
483   be easily recorded (e.g., Apple devices and Desktop software) as training data, causing a domain
484   gap. Nevertheless, SliME-8B still exhibits noticeable performance improvements on ScreenSpot
485   and RefExp when scaling up the AutoGUI data, suggesting that the AutoGUI data helps to enhance
      grounding accuracy on the out-of-domain tasks.

531   6       C ONCLUSION
532
533   We propose AutoGUI, a scalable and automatic annotation pipeline aimed to produce massive UI
534
      element functionality annotations used to enhance UI understanding capabilities of open-source
      VLMs. The pipeline prompts an open-source LLM to generate element functionalities based on the
535
      UI content changes induced by interacting with the elements. To guarantee high quality, LLM-aided
536
      rejection and verification are introduced to remove invalid samples. Fine-tuned with the data collected
537   by AutoGUI, the base models obtain strong UI grounding ability and exhibit data scaling effects. We
538   hope that AutoGUI will open up possibilities for advancing the field of general UI agents.

1018   The AutoGUI pipeline utilizes UI content changes to predict the functionalities of the interacted
1019   elements. For interactions that manipulate the existing UI, the pipeline analyzes differences in the
1020   AXTrees to annotate functionalities. Conversely, when interactions result in navigation to a new UI,
1021   the pipeline examines changes in UI descriptions to guide the annotation process. Details on these
1022
       methodologies are outlined below:
1023   UI manipulation case We use a file-comparison library, DiffLib, to generate line-by-line differences
1024   of the AXtrees before and after interactions. To balance efficiency with annotation integrity, we
1025   limit the differences to 250 lines. In addition to the standard markings by DiffLib—addition,
       deletion, and unchanged status—we incorporate two additional change markers: ‘Repositioning’ and

1148   To eliminate invalid samples before functionality annotation, the AutoGUI pipeline prompts the
1149
       annotating LLM to also determine the validity of samples by analyzing the predictability of the UI
       content changes. The LLM evaluates each sample against three criteria: 1) Explicitness of Changes:
1150
       This measures how clearly the changes indicate the element’s functionality. Changes that directly
1151
       suggest functionality receive higher scores, while vague or irrelevant changes are not scored. 2)
1152   Relevance of Changes: This criterion assesses the significance of the modifications in relation to
1153   the element’s intended function. Highly related modifications obtain a high score. No scores for
1154   irrelevant or unrelated content changes. 3) Predictability of Outcome: This involves determining how
1155   anticipated the interaction outcome is based on the changes, considering common web conventions
1156   and user experience principles. Highly predictable changes obtain a high score, whereas moderate,
1157   unexpected, or counter-intuitive outcomes receive no score.
1158   Given the UI content changes as the input, the LLM first presents detailed reasoning processes about
1159   the three criteria and then outputs an overall score summing the individual scores for each criterion,
1160   with each contributing 0 to 3 points for a maximum of 9 points. The LLM presents three rejection
1161   results with temperature = 1.0 for each sample. Samples falling in the bottom 30% of average scores
1162   are considered invalid and discarded. This method ensures a balance between high recall of actual
1163   invalid samples and retention of valid samples. The prompt is shown in Tab. C, the rejection process
1164   is illustrated in Fig. G, and several representative rejection examples are shown in Fig. H. Note
1165   that UI content changes are represented as line-by-line differences in UI manipulation cases, and as
1166
       descriptive changes in navigation scenarios.
1167   To validate the effectiveness of the chosen score range 0-3, we test the ranges 0-2, 0-3, and 0-4
1168   to select a range that helps to reduce false positives (valid but rejected) and increase true positives
1169   (invalid and rejected). We used 216 tasks, including 147 valid and 69 invalid samples as the test bed.
1170   We then drew a line chart illustrating the rejection ratios (Y-axis) for both valid and invalid samples
1171   against various threshold settings (X-axis) (Note that a sample whose score ranks below the threshold
1172
       will be discarded). The selection criteria: the area under the curve (AUC) for the valid samples should
       be as small as possible, while the AUC for invalid samples should be large, ensuring valid samples
1173
       rank higher. The results in Fig. F show that when using the score range 0-3, the AUC for invalid
1174
       samples is the largest while the value for valid ones is small, which suggests that this range achieves
1175   a better tradeoff between retaining valid samples and rejecting as many invalid samples as possible.

1447   D     L IMITATIONS
1448
1449   AutoGUI is dedicated to providing an autonomous way to collect scalable UI grounding/captioning
1450   data for training capable UI-VLMs. However, AutoGUI still encounters several limitations:
1451
       Lack of Diverse Mobile App Data. As many Apps implement anti-emulator code, it is extremely
1452   difficult to navigate through popular Apps, such as TikTok and WeChat, on Android emulators.
1453   To circumvent this issue, AutoGUI renders webpages at various resolutions, including smartphone
1454   resolution, to mimic diverse device types. Although mainstream websites, such as YouTube and
1455   Reddit, provide delicately designed webpage responsiveness for various resolutions, a number of less
1456   common websites do not possess such flexible responsiveness and distort severely when rendered at
1457   smartphone resolutions. Therefore, collecting UI data at a smartphone resolution probably leads to
       domain gaps between the collected data and real smartphone Apps that are not rendered with HTML.


                                                           27
       Under review as a conference paper at ICLR 2025


1458
       AutoGUI is Not Indented to Record Task-Oriented Interaction Trajectories. AutoGUI randomly
1459   interacts with UIs to record transition trajectories and utilize the UI content changes to predict the
1460   functionalities of the interacted elements. Hence, the collected trajectories do not provide high-level
1461   task semantics. In other words, the AutoGUI dataset does not contain tasks that combine multiple
1462   low-level steps, such as selecting a check-in date and then a check-out date. These long-horizon
1463   tasks are usually generated by human annotators in the existing works Deng et al. (2024); Rawles
1464   et al. (2023). In future work, we can also utilize capable LLMs to generate high-level tasks and then
1465   prompt the LLMs to interact with UIs according to the tasks.
1466   AutoGUI Cannot Annotate UI Elements That Modify Content on the Internet To avoid causing
1467   potential contamination on the Internet and bearing unexpected responsibilities, we try our best to
1468   eliminate interaction samples that manipulate sensitive elements that probably modify contents on
1469   the Internet. For example, elements used to post comments, make purchases, and enter account
1470   information are discarded. Consequently, the AutoGUI pipeline mainly annotates elements that only
1471   support read-only functionalities.

1473   E   P OTENTIAL S OCIETAL I MPACT
1474
1475
       The potential societal impacts of the proposed AutoGUI can be considered across various dimensions:
1476   Accessibility Enhancements VLMs trained with the AutoGUI data obtain stronger UI grounding
1477   capabilities, thereby possessing the potential to act as UI agents. By enabling context-aware under-
1478   standing of UI functionalities, the VLMs can help users locate elements on complex UIs, significantly
1479   improving accessibility features in software. This could lead to the development of applications that
1480   are more intuitive for users with disabilities, such as those requiring screen readers or other assistive
1481
       technologies.
1482   Research Impact: By reducing the labor and time required for annotating UI data via the AutoGUI,
1483   the industry and academia could lower costs to easily build UI agents. This could also shift labor
1484   demands towards more creative and strategic roles rather than repetitive annotation tasks.
1485   Privacy and Security Concerns: Although we employ precautions of eliminating samples related
1486   to sensitive UI elements (e.g., avoid interacting with elements modifying the Internet and use only
1487   popular public websites without exposing privacy), corner cases still exist on the vast Internet. UI
1488   data involving either content modification or personal information are hard to discern as UI designs
1489   are distinct and no universal detection rules exist. Therefore, it is essential for cyber-security research
1490   to consider the potential leakage of personal information in the collected data and devise preemptive
1491   protective approaches.
1492   Potential for Bias and Fairness: The bias of the LLMs used in the AutoGUI annotation pipeline is
1493   probably reflected in the collected data, leading to a trained UI-VLM that inherits the bias. Therefore,
1494   mitigating bias in the LLM’s annotations will be important for developing fair VLM agents that align
1495   with the values of users from diverse cultures.
