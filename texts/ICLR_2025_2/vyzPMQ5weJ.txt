009                                              A BSTRACT
010
011            An obvious way to alleviate memory difficulties in GPU-based AI computing is
012            via CPU offload, where data are moved between GPU and CPU RAM. While CPU
013            offload is useful, it can greatly slow down a computation due to the relatively slow
014            transfer rate between CPU RAM and GPU RAM. To address this, overlapping
015            memory transfer and compute is a necessity, but this asynchronicity introduces non-
016            determinacy, and hence it is impossible to know beforehand what is the best order
017
               of operations. We describe T URNIP, which is a system for running AI computations
               using CPU offload, designed to handle this nondeterminacy. The key innovation
018
               in T URNIP is the compilation of the AI computation into a dependency graph that
019
               gives the T URNIP runtime freedom to run operations such as GPU kernel calls in
020            many different orders; at runtime, T URNIP chooses the best order on-the-fly, in
021            response to real-time events. We find that T URNIP outperforms standard, PyTorch-
022            based systems supporting constrained GPU RAM by a significant margin, and also
023            avoids out-of-memory errors in a severely memory constrained environment.

026   1   I NTRODUCTION
027
028   Memory management for modern AI computations is difficult. In the LLaMA large language model
029   (Touvron et al., 2023), for example, the attention computation on a sequence of length n produces
030   an intermediate result with 128 × n2 floating point numbers. Thus, for a long input sequence of
031
      100,000 tokens, the attention computation will result in 1.2 trillion numbers, which would require
      2.4 terabytes to store at half-precision. It is for these reasons that out-of-memory errors plague AI
032
      programmers (FAQ).
033
034   CPU offload—where data are moved to CPU RAM for storage—can help. As CPU RAM is much
035   more inexpensive than GPU RAM and it is possible to install many terabytes of CPU RAM in a
036   GPU server essentially “for free”,1 it makes sense to leverage CPU RAM to temporarily store data.
037
      This idea has been explored in several systems such as pofo (Beaumont et al., 2021), AutoTM
      (Hildebrand et al., 2020), SwapAdvisor (Huang et al., 2020), Capuchin (Peng et al., 2020), and POET
038
      (Patil et al., 2022). These systems view a GPU computation as a dataflow graph, and plan how to fit
039
      the computation into GPU RAM by making use of CPU RAM offload.
040
041   While CPU offload is an obvious idea, it can greatly slow down a computation, due to the relatively
042   slow transfer rate between CPU RAM and GPU RAM. Thus, any system for CPU offload must ensure
043
      that when such a transfer happens, no computation is blocked waiting for the transfer to finish.
044   In this paper, we propose T URNIP (short for “nondeTerministic gpU RuNtime wIth cPu offload”)
045   which is a runtime for multi-GPU servers, designed to systematically support CPU RAM offload.
046   The key innovation of T URNIP is its combination of a pre-computed memory access plan called a
047   MEMGRAPH with a “nondeterministic,” event-driven system runtime. A MEMGRAPH is a dependency
048   graph where vertices represent tasks (such as the execution of a GPU kernel to perform a small part
049
      of attention computation in a layer of a large language model) and edges represent data or memory
      dependencies. Any execution order that respects the dependencies in the MEMGRAPH is valid, and
050
      tasks are dispatched at any time that their dependencies have been met and the appropriate resources
051
      are free. Thus, two executions of the same MEMGRAPH may lead to different sequences of operations

      being executed on a GPU, or different sequences of tensors being paged to CPU RAM—hence the
055   non-determinacy. However, the dependencies in the MEMGRAPH are such that the final output is
056   always correct, no matter the execution order. T URNIP’s event-driven, fully asynchronous runtime is
057   unique. Because operations can be dispatched whenever the dependencies are fulfilled and are not
058   constrained to any specific ordering, it lowers the chance that any GPU will be stalled waiting for a
059   memory transfer to complete. If one task cannot run due to an un-met dependency in the MEMGRAPH,
060   it is possible that there is another task that can run.
061   The key technical challenge is how to effectively build a MEMGRAPH with as few dependencies as
062   possible, to allow the runtime as much freedom as possible to dispatch operations so that it is never
063   blocked, waiting for a memory transfer to complete. T URNIP builds a MEMGRAPH by simulating
064   an execution of the computation, mapping tensors to GPU memory locations and, when necessary
065   adding edges that represent memory dependencies, as well as offload and reload operations.

069
      The design of T URNIP is based on a simple hypothesis: When running a GPU-based computation that
070
      utilizes CPU RAM, asynchronous operations such as offload and reload will have a seemingly
071
      nondeterministic running time that is difficult to pre-plan for. The system runtime must accommodate
072   the resulting non-determinacy, or else performance can suffer.
073
074   Consider Figure 1, which depicts a MEMGRAPH for a single
075
      GPU system with an offload (data movement from GPU
      RAM to CPU RAM) and a reload (from CPU RAM to
076
      GPU RAM). Vertices are operations (for example, GPU kernel
077
      calls) that produce data. Black edges indicate data or consump-
078   tion dependencies; red edges indicate memory dependencies
079   (MEMGRAPHs will be described in detail in Section 4). Note
080   the memory dependency from offload1 to 4. This exists
081   because the output of 4 will be written to the location of the  Figure 1: A simple MEMGRAPH.
082   output of 1, and so 4 cannot execute until the offload1
083   completes. The data dependency from reload3 to 5 exists
084   because the kernel associated with vertex 5 will consume the reloaded tensor.
085   Imagine that a system has executed GPU kernels associated with vertices 1 and 2. It is currently
086   executing the offload1 and the reload3 . At this point, it is impossible to know which kernel
087   should run next (4 or 5) as this depends on which memory transfer finishes first. Ideally, this decision
088   will be made at runtime. If the system deterministically decides to run 4 before 5 at compile time, and
089   the reload3 finishes first, the GPU will sit, idle, waiting for the offload1 to complete. This is
090   why a special-purpose, “nondeterministic” runtime is needed: to properly handle the non-determinacy
091   induced via the addition of memory operations.

096   accept an abstracted version of a generic GPU computation. Other systems are more specifically
097   targeted to certain categories of models, optimization algorithms, or to specific tasks such as training or
098   inference. Unlike T URNIP, none of these existing systems consider the effect of the non-determinism
099   of offload and reload operations on system performance, nor do any focus on the system
100   runtime.

145   4   TASKGRAPHS AND MEMGRAPHS IN T URNIP
146
147   T URNIP takes is input a TASKGRAPH. A TASKGRAPH is a dataflow graph (a directed, acyclic graph)
148   that describes how to perform multi-GPU computations. In a TASKGRAPH, edges represent data flow,
149   and vertices represent operations over tensors. A vertex without any inputs (called an input vertex)
150   is associated with an input tensor. An operation associated with a non-input vertex may be either a
151   kernel call that is to be executed on a specific GPU, or a GPU-to-GPU data transfer.
152   T URNIP is agnostic as to how the TASKGRAPH is created; it could, for example be created using
153   a framework such as FlexFlow (Jia et al., 2019) or Alpa (Zheng et al., 2022). Consider a matrix
154   multiplication X × Y, and assume we wish to execute this matrix multiplication on three GPUs.
155   To produce a TASKGRAPH, a framework such as FlexFlow may choose to decompose this matrix
156   multiplication as depicted in Figure 2, perhaps corresponding to the TASKGRAPH of Figure 3.
157   Given such a TASKGRAPH, T URNIP first compiles the TASKGRAPH into a MEMGRAPH, which it will
158   eventually execute. Like a TASKGRAPH, a MEMGRAPH is also a directed acyclic graph. Every vertex
159   in the original TASKGRAPH will be present in a corresponding MEMGRAPH. Further, the compilation
160   process may add additional offload and reload operations that move memory from GPU RAM
161   to CPU RAM, and vice versa. During the compilation process, the output associated with every vertex
      in the MEMGRAPH is mapped to a memory location. Unlike the input TASKGRAPH, the MEMGRAPH

      is not a dataflow graph; it is a dependency graph. If there is an edge from v1 to v2 , it means that
163   v2 depends on v1 and v2 may not execute until after v1 has been executed. In a MEMGRAPH, there
164   are two types of dependencies. One is a data dependency, which is inherited from the TASKGRAPH
165   (or is created via the addition of an offload or reload; see below). The second is a memory
166   dependency, which is added to ensure that there are no race conditions in the graph. A race condition
167   occurs when there is some vertex for which two valid executions of the graph may produce a different
168   output. This can happen when two vertices write to the same memory location, and it is possible for a
169   third vertex to read either output, depending upon the execution order.

231   5    T HE T URNIP E XECUTION E NGINE
232
233   Once a MEMGRAPH has been produced, it is executed by the T URNIP engine using a nondeterministic,
234   event-based framework. As soon as a GPU is unused or a tensor is ready to be offloaded to
235   RAM, the T URNIP runtime can immediately assign any available work to the GPU or begin the
236   transfer, without regard to the overall state of the computation. Also note that there are no calls to
237   memory-management routines such as cudaMalloc or cudaFree during MEMGRAPH execution,
238   as memory management is no longer dynamic. Tensor placement is pre-determined before execution,
239   and if dependencies are respected, there can be no memory corruption due to race conditions.
240   To execute the MEMGRAPH, T URNIP runs a central event processing loop, that repeatedly processes
241   callback functions that are called response to completion of the work associated with a MEMGRAPH
242   vertex (completion of a GPU-to-GPU transfer, completion of the GPU kernel, or completion of an
243   offload or reload). When a vertex completes and a callback is invoked, the event loop checks
244   to see if any other vertex can be executed. That is, it searches for a vertex v1 where (a) all vertices v2
245   with an edge v2 → v1 in the MEMGRAPH have also completed; (b) if v1 is a kernel call, then the GPU
246   v1 is assigned is currently free. When the event loop finds such a vertex, it launches it, and searches
247
      for another such vertex. When it can find no such executable vertex, it goes to sleep until woken by
      another callback.

250   6    B UILDING A MEMGRAPH
251
252   The key technical question we address in this paper is: How to construct a MEMGRAPH from a
253   TASKGRAPH ? The primary requirement for the compilation process is correctness. Correctness
254   requires that (a) every data dependency present in the TASKGRAPH is also present in the MEMGRAPH,
255   or is replaced with a sequence of offload-reload operations;2 (b) there are no race conditions in
256   the MEMGRAPH; (c) the MEMGRAPH has no cycles. In addition, it is desirable for the MEMGRAPH to
257
      be performant. A MEMGRAPH will not be performant if memory dependencies severely constrain the
      execution order of vertices. Such constraints may reduce parallelism and GPU utilization.
258
259   Our basic tactic during compilation is to rely on a simulated execution of the TASKGRAPH to
260   generate the MEMGRAPH. Given a serial ordering of the vertices in the TASKGRAPH that respects
261   all dependencies (so that if v1 → v2 is in the TASKGRAPH, v1 is before v2 in the ordering) we
262   simulate its execution, making calls to special variants of malloc and free that do not actually
263
      allocate GPU RAM, but instead maintain a map of used and free RAM slots on the GPU that is
      the target of the compilation. These implementations also maintain a history of which tensors
264
      occupied which positions in simulated GPU RAM, to correctly generate memory dependencies. As
265
      the simulation runs, the MEMGRAPH is constructed. Calls to the special malloc implementations
266   associate MEMGRAPH vertex outputs to GPU memory locations (effectively producing the mappings
267   depicted in Figure 4 and Figure 6). Whenever a call to the malloc variant fails because there is

      Discussion. Throughout the first token inference experiments, T URNIP typically performed the best
491
      in terms of latency, with ZeRO Inference generally much slower but outperforming FlexGen, except
492
      for larger sequence sizes. This would seem to validate the highly non-deterministic, dataflow-based
493   approach advocated for in the paper, at least if the goal is low latency.
494
495   To be fair, we note that FlexGen is designed for high throughput, as opposed to low latency, and
496
      FlexGen utilizes multiple GPUs only via pipelined parallelism. Note that FlexGen does not seem
      to get any slower when moving from batch size of eight to 16 on the A100 server. This suggests
497
      that filling the pipeline leads to substantial latency. Further, pipelined parallelism is more effective
498
      with more work in each pipeline stage, due to the high synchronization overhead and the need to try
499   to overlap communication with computation, perhaps explaining FlexGen’s better performance for
500   larger sequences, which are more dense computationally.
501
502
      ZeRO Inference takes a much different approach, but it uses a highly synchronized form of model
      parallelism as it traverses the levels in a transformer, also carefully trying to overlap communication
503
      and computation, which may simply be more effective when there is more work to at each level.
504
      T URNIP, on the other hand, is radically different. It does not “understand” the levels in a transformer,
505   does not need to synchronize processing of the various levels, and simply tries to asynchronously
506   process kernels as fast as it can. If it is stuck waiting for communication, it simply tries to do
507   something else.
508
      For training, there were clear advantages of T URNIP over ZeRO Infinity, especially for smaller
509
      sequence lengths. This was particularly true on the A100 server, where T URNIP was often much
510
      faster. For batches of sequences of length 1K, T URNIP often took less than 50% of the time to process
511   each batch, compared to ZeRO (at a batch size of 8, the time to process 1K sequences was 5.6 seconds
512   for T URNIP and 12.5 seconds for ZeRO, for batch size of 32 it was 12.1 seconds for T URNIP and
513   17.8 seconds for ZeRO). The differences in performance were much less pronounced on the P100
514   server, though there T URNIP was still faster. Finally, we note that both systems suffered significant
515   out-of-memory errors during training. Interestingly, T URNIP seemed to have more problems with
516   memory on the A100 server, whereas ZeRO Infinity had more problems with memory on the P100
517   server. We conjecture that some of that could be solved in T URNIP with a better input dataflow graph,
518   which cuts the input problem into smaller pieces.

      9    C ONCLUSIONS
522
523   We present T URNIP, a system and runtime for executing memory-intensive computations on GPU
524   servers, supporting CPU offload. The key innovation of T URNIP is its reliance on a “nondeterministic”
525
      runtime where a dependency graph is used to guide execution of the computation; any order of
      operations is acceptable, if the dependencies are respected. We argue that this is necessary when CPU
526
      RAM offload is used, or else the system will often be stalled, waiting for CPU-to-GPU transfers.
527
528   Limitations. As currently implemented, the biggest limitation of T URNIP is that the input computa-
529   tion (in the form of a TASKGRAPH) must be static, and known completely beforehand, so that the
530   MEMGRAPH can be constructed. This is not of consequence during model training, but can be an
531   issue during any recursive, generative AI computation. This includes LLM inference, where the
532   next token must repeatedly be generated and the KV -cache increases in size. There are some naive
533   solutions to this (such as pre-compiling a MEMGRAPH for a specific number of token generations in
534   the case of an LLM) but more work will be necessary to create a satisfactory solution to recursive
535   computation.
536   Another limitation is that while our experiments did show that T URNIP has certain performance
537   advantages, it is not possible to be absolutely sure where those advantages come from. Our ablation
538   shows that a fixed execution order slows down T URNIP, suggesting that nondeterminism is important.
539   But unlike ZeRO and FlexGen, T URNIP was implemented from the ground up and does not rely on
      Python or PyTorch—that fact alone could account for some of the performance differences.

      A    T URNIP E NGINE D ETAILS
649
650
      Our execution engine consists of a central event loop that “launches” each vertex in the MEMGRAPH.
651   A vertex can be launched when (1) all dependencies have been completed and (2) the required
652   resources are obtained. When a vertex is launched, the corresponding operation is executed and then
653   a provided callback is called to notify the event loop that the vertex has completed. In turn, the event
654   loop frees up the obtained resources and keeps track of when vertices complete execution so that
655   subsequent vertices can be launched. In practice, when launched, a vertex will execute one or more
656   asynchronous CUDA operations on CUDA stream and will then call cudaStreamAddCallback.
657   As such, every vertex requires as a resource a stream, where a single stream can only be used by a
658   single launched vertex at a time. We use 5 streams per GPU. offload, Reload and inter-GPU
659
      communication vertices will call cudaMemcpyAsync. For CPU storage, we allocate a single, large
      contiguous block of memory with cudaHostAlloc with flags cudaHostAllocPortable and
660
      cudaHostAllocWriteCombined. When executing Offload vertices, we allocate into the
661
      CPU storage memory using our custom allocator; when executing Reload vertices, we free from our
662   custom allocator. All compute vertices are executed using either cuTensor functions or hand-written
663   CUDA kernels. An example where a hand-written CUDA kernel is beneficial is for executing portions
664   of softmax so that less workspace memory and fewer vertices would be required. Two additional
665   resources may be required for computing vertices: workspace memory as required for executing
666   multiple cuTensor functions and locks around write-protected memory. As an example, we would
667   execute a summation of n tensors with n calls to tensor increment sum-into kernels. However, the
668   output memory would be protected by a resource so that only one sum-into can happen at a time.
669   This implementation is designed to support non-determinism. We use CUDA version 11.8.0 and
670
      cuTensor version 2.0.1. All other code is C++.
