009                                                A BSTRACT
010
011            There has been a surge of interest in language model agents that can navigate
012            virtual environments such as the web or desktop. To navigate such environments,
013            agents benefit from information on the various elements (e.g., buttons, text, or
014            images) present. However, it remains unclear which element attributes have the
015            greatest impact on agent performance, especially in environments that only provide
016            a graphical representation (i.e., pixels). Here we find that the ordering in which ele-
017            ments are presented to the language model is surprisingly impactful—randomizing
018
               element ordering in webpages compromises average agent performance to a degree
               comparable to removing all visible text from webpages. While web agents benefit
019
               from the semantic hierarchical ordering of elements available via the browser,
020
               agents that parse elements directly from pixels do not have access to any such order-
021            ing. Here we endeavor to derive effective orderings and investigate the impact of
022            various element ordering methods in web and desktop environments. We find that
023            dimensionality reduction provides a viable ordering for pixel-only environments.
024            We train a UI element detection model to derive elements from pixels and apply our
025            findings to an agent benchmark—OmniACT—where we only have access to pixels.
026            Our method completes more than two times as many tasks on average relative to
027            the previous state-of-the-art.

030   1    I NTRODUCTION
031
032   There has been growing interest in using language model (LM) agents to autonomously navigate
033
      virtual environments. Autonomous web agents (Zhou et al., 2023; Kim et al., 2023; Zheng et al.,
      2024; Gur et al., 2024; He et al., 2024) have become a particularly popular area of research. Typically,
034
      a web agent takes as input a task prompt from a user, observes a text and visual representation of the
035
      environment, and then outputs one or more actions to execute the task in the environment. Recently,
036   research interest has expanded to include agents that can navigate mobile (Rawles et al., 2023; Yan
037   et al., 2023) and desktop (Xie et al., 2024; Kapoor et al., 2024; Bonatti et al., 2024) environments as
038   well.
039
      At a high level, a virtual environment consists of numerous elements—some are interactive (e.g.
040
      buttons or widgets), while others are not (e.g. plain text). To allow for human navigation, these
041
      elements are usually represented in the pixel space via a Graphical User Interface (GUI). In contrast,
042   agents often rely on distinct state representations to navigate virtual environments. The exact format
043   of a state representation varies between environments and approaches. In web environments, common
044   text representations include the HTML or accessibility tree (Zhou et al., 2023; Koh et al., 2024). For
045   visual representations, a popular approach is to label UI elements with bounding boxes and numeric
046   identifiers (Koh et al., 2024; He et al., 2024), known as Set-of-Mark (Yang et al., 2023). In either case,
047   the state representation is derived from the underlying Document Object Model (DOM) (Zhou et al.,
048   2023; Koh et al., 2024; He et al., 2024). However, many environments lack a descriptive DOM and
049   only provide pixel information, which we show is insufficient for existing agents (see Section 4.2).
050
      To construct an effective state representation from only pixels, it is important to answer the following
      basic questions about these representations: What aspects of a state representation are most important
051
      to an agent? How can we derive these important aspects with only pixels?
052

137
      Here we propose and evaluate strategies for deriving effective element orderings in scenarios where
138
      a hierarchical ordering based on the GUI design is not explicitly provided by the environment.
139
      Across multiple agent benchmarks, we find that ordering elements via a 2D-to-1D dimensionality
140   reduction (Van der Maaten and Hinton, 2008) reliably yields improvements to agent performance
141   relative to other baselines. We experiment on the VisualWebArena (Koh et al., 2024) and OmniACT
142   (Kapoor et al., 2024) benchmarks and achieve new state-of-the-art performance on OmniACT.

164   properly navigate more realistic benchmarks, completing only 15% of web tasks (Zhou et al., 2023)

173   a filtered down version of the accessibility tree. All of these approaches require access to either a
174

      environments that only give access to their graphical representations which is significantly more
175
      challenging.
176
177
      Agents With Access to Only a Graphical Representation. There have been several approaches—
178
      primarily focused on desktop and mobile environments—to directly navigating a GUI via its pixels.
179   Kapoor et al. (2024) and Yan et al. (2023) focus on navigating desktop and mobile applications
180   respectively. Both leverage an off-the-shelf-segmentation model—Segment Anything (Kirillov et al.,
181   2023)—to find icons in the image. These icons are then either represented in text (Kapoor et al., 2024)
182   or labeled with Set-of-Mark (Yang et al., 2023) bounding boxes and labels in the image (Yan et al.,
183   2023). We instead train an object detection model that detects interactable UI elements directly. While
184   previous UI element detection models are trained to detect accessibility features (Wu et al., 2023),
185   our model is trained specifically to detect interactable elements that would be useful to an agent. All
186   three of our approaches use Optical Character Recognition (OCR) modules such as EasyOCR (AI,
187
      2020) to extract text from pixel information.
188
189
      Agent Input Ablations. While most agent studies include some ablations, few focus on detailed
      analysis of an agent’s input. To our knowledge, Huq et al. (2023) is the only other study that directly
190
      studies this. Their study focuses on broader components to an input prompt such as the selection of
191
      few-shot examples used, while we focus on specific element attributes such as element ordering.

194   3    P ROBLEM D EFINITION
195
196   We define the environment state E as a set of elements E = {e1 , e2 , . . . , en }, where each element ej
197   is a tuple ⟨ij , Cj , Aj , Sj ⟩ defined by the following parameters:
198
199
             • ij ∈ {0, 1} denotes the interactability of element ej . An element with ij = 1 is interactable,
               while an element with ij = 0 is not.
200
201          • Cj = {⟨x1 , y1 ⟩, ⟨x2 , y2 ⟩} is the set of pixel coordinates that form the bounding box around
202            ej . ⟨x1 , y1 ⟩ is the top left coordinate and ⟨x2 , y2 ⟩ is the bottom right coordinate.
203          • Aj = {a1 , a2 , . . . , am } is the set of potential actions that can be taken on that element. For
204            example, the potential actions for a search bar might be {click, type}; a non-interactive
205            text element has the action set ∅.
206          • Sj represents the set of other environment-specific attributes for element ej . These at-
207            tributes can include image captions, type labels (e.g., Button, Text Field), or accessibility
208            information.
209
210   While certain environments may provide full access to this environment state, here we are focus on
211   environments where only the pixel information, P, is available. We then must predict elements from
212
      the pixel information to construct a state representation for the agent. In other words, we must find a
      function g : P → E
213
214   Elements can be represented in both visual and text modalities. For images, the most common
215   approach is to overlay bounding boxes with numeric identifiers around each interactable element
      (Koh et al., 2024; He et al., 2024; Yan et al., 2023) in a manner inspired by Set-of-Mark Prompting


                                                          4
      Under review as a conference paper at ICLR 2025


216
      (Yang et al., 2023). In text, a common approach is to represent each element as a string containing its
217   index, coordinates, and other attributes, such as "[1] [x,y] [Description]". Because LMs operate on
218   sequential data, elements must be given an ordering; in most approaches, this is implicitly defined by
219   the method used to identify the elements.
220
221   Ordering. The ordering is defined as a permutation σ of the indices {1, 2, . . . , n} where
222   {eσ(1) , eσ(2) , . . . , eσ(n) } represents a specific sequence of elements. An ordering function is a
223   function f : E → σ that takes the environment as an input and yields a specific ordering σ.

              I MPACTFUL ?

      Here we describe a series of ablation experiments designed to examine which aspects of an LM state
229
      representation are most impactful to the performance of LM agents. In particular, we experiment
230   on the VisualWebArena (VWA) (Koh et al., 2024) benchmark and ablate attributes of the state
231   representation of the state-of-the-art agent (proposed in the same paper). We pick this representation
232   in particular due to both its strong performance on VWA, as well as its similarity to common practices
233   seen in other agent research (He et al., 2024) and open source projects (Ishan0102, 2024; Dupont,
234   2024). Our experiments in turn ablate the impact of (1) multimodal (image and text) aspects of the
235   state representation, and (2) individual element attributes within the text component alone.
236
237   4.1       V ISUALW EBA RENA
238
239
      VisualWebArena focuses on multimodal tasks in the web and provides a self-hostable environment
      for language agents to navigate (Koh et al., 2024). Agents operating on VisualWebArena have full
240
      access to the DOM. The current best approach (Koh et al., 2024) utilizes a multimodal representation
241
      where elements are parsed in a pre-order traversal of the DOM tree (World Wide Web Consortium,
242   2013). Each element j has attributes Sj = {id, tag, text} where id is a unique numeric identifier for
243   interactable elements and ∅ otherwise, tag is the HTML tag (e.g. BTN or IMG), and text is the alt
244   text and captions for images and the HTML text otherwise. In the text representation, an example of
245   an element would be “[1] [IMG] [alt text, caption]”. In the image representation, each element is
246   labeled with bounding boxes and numeric labels.
247
      The original agent in (Koh et al., 2024) only achieves 15% success rate across all tasks. Since our
248   goal is not to improve agent performance on VisualWebArena but rather to understand the importance
249   of attributes in the state representation, we examine a subset of tasks to reduce costs.0 Specifically,
250   we explore tasks marked as “easy” within tasks that the original agent completed successfully. Due to
251   variance associated with stochastic LM outputs, our reproduction of these originally-successful tasks
252   yields a success rate around 74.07% ± 5.56%. The exact list of tasks can be found in Appendix A.8.
253   We reuse the action space from the original agent which consists of executing high-level actions
254   (e.g., click, hover) on individual elements—see Appendix A.6 for more details.

      5     E XPERIMENTAL S ETUP
325
326   For the remainder of this paper, we leverage the insights gained from our state representation ablations
327   on VisualWebArena to tackle a more challenging task: enabling LM agents to act in environments
328   that only expose pixel information.
329
      Most applications are built on top of an underlying hierarchical representation. For example, a
330
      webpage is modeled by the DOM which is hierarchical. When exposed, this hierarchy can be used
331
      to determine a strong element ordering. However, the availability and quality of an underlying
332   hierarchical representation can vary greatly between environments. For example, Chen et al. (2020)
333   found that 77% of mobile applications had missing labels and Ross et al. (2020) found that 53%
334   of image buttons had missing labels and were incorrectly sized. Additionally, Ross et al. (2020)
335   found that 8% of applications lacked interactable element information altogether. In such scenarios,
336   we may only have access to the application’s pixel information. We continue to experiment with
337   VisualWebArena and also experiment with the OmniACT benchmark as a scenario where we only
338   have access to pixel information. Details on the LM agent backbones used in our experiments can be
339   found in Appendix A.7.
340
341   5.1   O MNI ACT
342
      OmniACT provides both web and desktop environments for agents to benchmark on. OmniACT
343
      contains 177 application screenshots overall and 2021 tasks in the test set. Agents are tasked with
344
      generating pyautogui code that can navigate the application screenshot. We consider OmniACT as a
345   setting where only a pixel information is available.
346
347   To detect UI elements {e1 , e2 , . . . , en } when given only pixel information P we train an object
348
      detection model (Ren et al., 2016) to detect interactable UI elements in the screenshot and use
      EasyOCR (AI, 2020) to extract visible text. In other words, the function g : P → E is defined by the
349
      trained object detection model. We add visible text and captions to each interactable UI element. We
350
      gather a dataset by finding 674,416 interactable elements over 1468 Common Crawl webpages. We
351   selected our webpages based on top websites from Similarweb (2024). Despite the domain shift from
352   webpages to desktop applications, we found that our object detection model worked reasonably well
353   on the OmniACT benchmark in the end-to-end agent setting. We publicly release this model along
354   with our paper. Training details can be found in the Appendix A.1.
355
      OmniACT provides partial human annotations for each screenshot; multiple, but not all, interactable
356
      UI elements are annotated with bounding boxes. The original intent of these bounding boxes is for
357   evaluation only. As a result, there are significantly less UI elements annotated compared to possible
358   UI elements in the application. We experiment with elements derived from these human annotated
359   bounding boxes to understand a) impacts to ordering performance in an easier setting and b) the
360   potential performance that can be gained by improving UI element detection.
361
362   5.2   M ETRICS
363
364   Our primary metric is task success rate which is the standard for agent evaluations (Koh et al., 2024;
365   Zhou et al., 2023; He et al., 2024). VisualWebArena provides an evaluation framework for task
366
      success. Task success criteria include achieving an expected final webpage state or receiving a desired
      response from the language agent. OmniACT does not provide task success rate directly, instead
367
      introducing the sequence score and action score metrics. Sequence score evaluates if the output
368
      contains the correct high level action (e.g. click or type), but does not check if the action element
369   or parameter (e.g. click [1] or type [parameter]) are correct. Action score evaluates if
370   the output contains both the correct high level action and the correct element or parameter. Thus, for
371   OmniACT we focus our evaluations on the action score as it is more similar to task success rate.
372
373   5.3   O RDERING M ETHODS
374
375   In addition to random ordering, we experiment with two different ordering methods.
376
377   Random. We pick the ordering σrand uniformly at random from all possible permutations. This
      provides a baseline performance.


                                                         7
      Under review as a conference paper at ICLR 2025


378
      Raster. Elements {e1 , e2 , . . . , en } are ordered in a left-to-right raster scan. We define a raster scan
379                                                       y
      as an ordering σraster where i < j iff ⌊ y8i ⌋ < ⌊ 8j ⌋ and xi < xj . We chose to discretize the scan to
380   prevent jumps in ordering from minor pixel variations. This method mimics the natural way English
381   speakers read text and images from left-to-right and top-to-bottom.
382
383   t-SNE. We apply dimensionality reduction techniques to better capture the spatial relationships.
384   Using t-SNE (Van der Maaten and Hinton, 2008), we reduce the dimensionality with the function
385   g : ⟨x, y⟩ → z. The set of values Z = {z1 , z2 , . . . , zm } is used to determine the ordering σtsne . We
386   use the scikit-learn (Pedregosa et al., 2011) implementation of t-SNE and keep the default parameters.
387   Our intuition for choosing t-SNE stems from visually and qualitatively inspecting our agent trajecto-
388   ries. We observed that agents often use adjacently ordered elements as context clues to determine the
389   correct action. Furthermore, agents have difficulty reasoning about functionally associated elements
390   that are separated from each other in the ordering. t-SNE generates an effective ordering as elements
391   that are visually close together (i.e. nearby in 2D pixel space) are often functionally associated with
392   each other as well. When reducing dimensions, t-SNE retains local structure which increases the
393   odds that functionally associated elements are adjacent in the induced 1D ordering.

      6.2    S TATE - OF - THE -A RT P ERFORMANCE ON O MNI ACT
460
461   We achieve a new state-of-the-art performance on OmniACT. Due to cost, we look to our previous
462   experiments to pick the best combination of features for our approach. We observe that multimodal
463   representations are still helpful. We find that t-SNE ordering improves performance the best in
464   OmniACT when elements are detected by our model. Koh et al. (2024) states that high level actions
465   are easier for an LM to reason with.
466   We analyze the differences between our best approach and OmniACT’s baseline. These are as follows.
467
468          • Element Source: OmniACT obtains elements by searching for icons with Segment Anything
469            (Kirillov et al., 2023) and text with EasyOCR. Unfortunately, there are no shared artifacts
470            for their icon detection system. We obtain UI elements through an object detection model
471            and text with EasyOCR (AI, 2020).
472          • Ordering: It is unclear how elements are ordered in OmniACT. Considering how most
473            approaches don’t pay specific attention to ordering, we assume the ordering in OmniACT is
474            effectively random. We order our elements using our t-SNE ordering.
475          • Action Space: OmniACT directly outputs pyautogui code as their actions. We consider a
476            higher level action space that maps to pyautogui code.
477          • Intractability OmniACT lists out each element, but does not indicate which element is
478            interactable. We specify whether elements are interactable or not.
479          • Multimodal Representation: OmniACT evaluates their full test set using a text only
480            representation3 . We experiment with a multimodal representation.

502   7     F UTURE W ORK
503
504   Further Improvements to Ordering We provided two simple methods to apply ordering when
505   a default ordering is not given. However, both approaches still fall short when compared to the
506   hierarchical ordering derived from the DOM. We hope that future research can introduce more
507   sophisticated methods to find ordering with only pixel information.
508
509   Image Only Ordering We focused on the impact of element ordering in a text representation
510   (although element labels re-ordered accordingly in the visual representation as well). The impact
511
      of element ordering may or may not generalize to an image only representation. Unfortunately, our
      results indicated that a visual representation alone was insufficient for web and desktop environments
512
      which prevents us from conducting this experiment. However, Yan et al. (2023) found that a visual
513
      representation in mobile environments was able to achieve comparable performance with and without
514   a text representation. In the future, we hope to experiment with various ordering methods on an
515   image only representation.
516
517   Expanded Scenarios and Benchmarks In this paper, we explored two benchmarks—
518   VisualWebArena and OmniACT—as web and desktop scenarios. In the future, we hope to explore
519   other benchmarks and settings with our approach. For example, Xie et al. (2024) uses the OS level
520   accessibility tree for desktop agent navigation. We hope to compare our approach against theirs and
521   believe that combining both approaches may lead to further improvements. Additionally, mobile
522   environments often have only pixel-level information (Chen et al., 2020; Ross et al., 2020); we hope
523
      to apply our approach to a mobile benchmark such as Rawles et al. (2023).
524
525   8     C ONCLUSION
526
527   We conducted thorough ablations to show that element ordering has a significant impact on the
528   performance of agents. We provided a method of ordering elements through dimensionality reduction
529   and showed that it performed best in realistic environments. We trained a UI element detection model
530   on Common Crawl data and publicly share the model. We demonstrated an end-to-end method which
531   allows a LM agent to act on environments that provide only pixel information. Using this method, we
532
      were able to achieve a new state-of-the-art performance on OmniACT.

872   Here’s the information you’ll have:
873   The user’s objective: This is the task you’re trying to complete.
874
      The current desktop screenshot: This is a screenshot of the desktop or
          webpage, with each interactable element assigned a unique numerical
875       id. Each bounding box and its respective id shares the same color.
876   The observation, which lists the IDs of all interactable elements on the
877       current screenshot with their text content if any, in the format [id]
878        [tagType] [text content]. tagType is the type of the element. text
879
          content is the text content of the element. For example, [1234] [
          BUTTON] [’Add to Cart’] means that there is a button with id 1234 and
880        text content ’Add to Cart’ on the current web page. [] [StaticText]
881       [text] means that the element is of some text that is not
882       interactable.

884
      The actions you can perform fall into two categories:
885   Mouse Actions:
886   click [id]: This action clicks on an element with a specific id.
887   double_click [id]: This action double clicks on an element with a
888       specific id.
889
      right_click [id]: This action right clicks on an element with a specific
          id.
890   hover [id]: Hover over an element with id.
891
892   Keyboard Actions:
893   type [content]: Use this to type content. Be sure to use other commands
894
          to click before or press enter after if necessary.
      press [key_comb]: Simulates the pressing of a key combination on the
895       keyboard (e.g., enter).
896   hotkey [key1] [key2]: Simulates the pressing of a multiple key
897       combinations on the keyboard. For example, hotkey [Ctrl] [Alt] [
898       Delete] will press Ctrl+Alt+Delete.

      To be successful, it is very important to follow the following rules:
900   1. You should only issue actions that are valid given the current
901       observation. Everything is possible. You MUST issue actions.
902   2. You can issue a sequence of actions separated by newlines.
903   3. You should follow the examples from past messages to reason step by
904
          step and then issue the next actions.
      4. You should start every answer with "Let’s think step-by-step"
905   5. Generate the actions in the correct format. Start with a "In summary,
906       the next actions I will perform are" phrase, followed by the actions
907       inside ‘‘‘. Each action should be split by a newline. There should be
908        no text inside ‘‘ except for the actions. For example, "In summary,
909
          the actions I will perform are ‘‘‘click [1234] type [sample text]
          press [enter]‘‘‘".
