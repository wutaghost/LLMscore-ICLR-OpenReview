                                                  A BSTRACT
008
009
               Automated feature engineering (AutoFE) liberates data scientists from the burden
010
               of manual feature construction, a critical step for tabular data prediction. While the
011            semantic information of datasets provides valuable context for feature engineer-
012            ing, it has been underutilized in most existing works. In this paper, we introduce
013            AutoFE by Prompting (FEBP), a novel AutoFE algorithm that leverages large lan-
014            guage models (LLMs) to process dataset descriptions and automatically generate
015            features. Incorporating domain knowledge, the LLM iteratively refines feature
016            construction through in-context learning of top-performing example features and
017            provides semantic explanations. Our experiments on real-world datasets demon-
018            strate the superior performance of FEBP over state-of-the-art AuoFE methods. We
019
               also conduct ablation study to verify the impact of dataset semantic information
               and examine the behavior of our LLM-based feature search process.

      1   I NTRODUCTION
023
024
      Tabular data, a form of structured data comprising instances and attributes, have extensive use in vast
025   domains, e.g., credit assessment, market prediction, and quality control. Traditional machine learn-
026   ing models, especially tree-based models (Breiman, 2001; Ke et al., 2017), have strong performance
027   on tabular datasets of small and medium sizes (Grinsztajn et al., 2022) and good interpretability.
028   Feature engineering is the process of computing new features from feature attributes of a dataset to
029   enhance downstream model performance, which is crucial for traditional ML models as the new fea-
030   tures extract useful information for target prediction by capturing complex non-linear relationships.
031   Feature engineering by hand requires domain expertise to alleviate the significant human labor.
032   Automated feature engineering (AutoFE) uses high-level algorithms and models to automate the FE
033   process such that the performance is comparable to domain experts. Existing AutoFE methods, such
034   as (Zhu et al., 2022a;b; Zhang et al., 2023), compute and evaluate a large number of features in a
035   trial-and-error manner. While some learn to optimize the feature utility during the FE process, these
036   methods do not utilize domain knowledge to guide the feature search. The need to start from scratch
037   for new datasets or downstream models hampers their effectiveness and efficiency. Besides, these
038   methods do not offer explanation of the computed features, impairing the interpretability.
039   The descriptions contained in tabular datasets
040   provide rich context for feature engineering.

057   We propose AutoFE by Prompting (FEBP), a novel AutoFE algorithm that leverages LLMs for
058
      effective, efficient, and interpretable feature engineering, as illustrated in Figure 1. By providing
      dataset descriptions and example features in canonical Reverse Polish Notation (cRPN), we prompt
059
      the LLM to generate new features. After evaluating the features, we update the prompt with top-
060
      performing features and their evaluation scores and instruct the LLM to construct further features.
061   Through this iterative process, the LLM explores the feature space and improves solutions through
062   in-context learning of successful examples. The semantic information of data not only guides the
063   feature search, but helps the LLM understand and learn from the patterns in example features. Ap-
064   plying domain knowledge, the LLM generates semantically meaningful features and explains their
065   usefulness. Experiments on seven real-world datasets demonstrate that FEBP significantly outper-
066   forms state-of-the-art baselines, achieving over 5% performance gain on average across three down-
067   stream models. Additionally, our ablation study shows that incorporating dataset semantic context
068   improves the performance. We also analyze the behavior of the LLM-based feature search process
069
      and examine the effects of hyperparameters.
070   Our main contributions are: (1) We introduce a novel LLM-based AutoFE algorithm that utilizes
071   dataset semantic information for automated feature search. This is the first method capable of gen-
072   erating features in string representations while providing semantic explanations. (2) We benchmark
073   the performance of our approach against state-of-the-art baselines using both GPT-3.5 and GPT-
074   4. (3) We investigate the impact of semantic context on our approach, analyze the behavior of the
075
      LLM-based feature search process, and examine hyperparameter effects.

077   2   R ELATED W ORK
078
079   Large Language Models. LLMs are large-scale general-purpose neural networks pretrained on
080   large corpora of raw text data for natural language processing, typically built with transformer-
081   based architectures (Vaswani et al., 2017). Generative LLMs, such as the GPT family (Radford
082   et al., 2019; Brown et al., 2020; OpenAI, 2023) and the LLaMA family (Touvron et al., 2023a;b),
083   are pretrained to successively predict the next token given the input text and can be finetuned using
084
      reinforcement learning from human feedback (Ziegler et al., 2019; Ouyang et al., 2022). By this
      means, they acquire the knowledge about syntax and semantics of human languages and are able to
085
      achieve state-of-the-art performance on various tasks like text generation, summarization, and ques-
086
      tion answering. LLMs can be adapted to specific tasks without changing model parameters through
087   prompt engineering. Few-shot learning (Brown et al., 2020) includes examples in the prompt for
088   the language model to learn in-context. Leveraging such capability, the LLM may function as a
089   problem solver (Yang et al., 2024) that iteratively improves candidate solutions according to the task
090   description and feedback. Chain-of-though (Wei et al., 2022; Kojima et al., 2022) enables complex
091   reasoning capabilities of LLMs through intermediate reasoning steps.
092
093   Automated Feature Engineering. AutoFE computes new features for the input data and augments
094   or replaces portions of the existing features, to enhance the performance of downstream models.
095   Common AutoFE approaches include expansion-reduction (Kanter & Veeramachaneni, 2015; Horn
096   et al., 2020; Zhang et al., 2023), evolutionary algorithms (Smith & Bull, 2005; Zhu et al., 2022a), and
097
      reinforcement learning (Khurana et al., 2018; Li et al., 2023; Wang et al., 2023). DIFER (Zhu et al.,
      2022b) utilizes neural networks to learn the utility of constructed features and optimize features in
098
      the embedding space. OpenFE (Zhang et al., 2023) introduces a feature boost algorithm to speedup
099
      feature evaluation. Nonetheless, these approaches do not utilize the semantic information of data,
100   which impedes the performance and interpretability of engineered features.
101
102   AutoFE with Domain Knowledge. The benefits of incorporating domain knowledge in AutoFE
103   include: (1) improving the effectiveness; and (2) reducing the cost of learning an AutoFE model,
104   especially the feature evaluation overhead. Prior works take different directions. One direction is to
105   transfer the knowledge through pretraining. LFE (Nargesian et al., 2017) represents features with
106   quantile sketches transferable across datasets and inputs them to a feature transformation recom-
107   mendation model. FETCH (Li et al., 2023) is an RL-based AutoFE framework taking tabular data
      as the state and generalizable to new data. E-AFE (Wang et al., 2023) pretrains a feature evaluator

108
      to efficiently learn the RL-based AutoFE model. The other direction is to leverage the semantic
109   information of datasets. KAFE (Galhotra et al., 2019) leverages knowledge graphs to identify se-
110   mantically informative features relevant to the prediction task. CAAFE (Hollmann et al., 2023)
111   manipulates Pandas data frames using the code generated from the LLM based on dataset descrip-
112   tions. In our work, we adopt a compact form of feature representation in strings with pre-defined
113   transformation operators. Our approach reduces the search space and helps the LLM learn the pat-
114   terns of useful features, leading to stronger and more robust performance. We further discuss the
115   differences between our approach and CAAFE in Appendix G.

      4    M ETHODOLOGY
138
      In this section, we present AutoFE by Prompting (FEBP), a novel AutoFE algorithm leveraging the
139
      power of LLMs, particularly, the GPT models (Radford et al., 2019; Brown et al., 2020; OpenAI,
140
      2023). The high-level idea is to provide the LLM with descriptive information of the dataset in the
141   prompt and guide it to search for effective features using examples.
142
143   We represent features in a compact form in our
144
      prompt. A feature x̃ ∈ XT is expressible as a
      tree, where the leaf nodes are raw features and
145
      the internal nodes are operators. However, the
146
      expression trees of features containing commu-
147   tative operators (like addition and multiplication)
148   are not unique since the child nodes of these op-
149   erators are unordered. We introduce a canonical-
150   ization scheme: arranging operator nodes before
151   feature nodes for left skewness and lexicograph- Figure 2: We obtain the canonical RPN (cRPN) by

152   ically sorting the nodes within each group. We
153   then serialize the canonical expression tree into the postorder depth-first traversal string, i.e., canon-
154   ical reverse Polish notation (cRPN), ensuring the one-to-one mapping between features and string
155
      representations. We denote the feature corresponding to an RPN string f as x̃f and the set of features

243     Methodologically, we instruct the LLM to act as a problem solver (Yang et al., 2024) within our
244     algorithm. Similar to evolutionary algorithms (Smith & Bull, 2005; Zhu et al., 2022a; Morris et al.,
245
        2024) that generate new solutions through crossover and mutations on high-fitness candidates, we
        maintain a pool of top-performing candidate features as examples. By learning examples and scores
246
        in-context (Brown et al., 2020), the LLM can recognize patterns of promising features and propose
247
        new features that are likely to be effective. For instance, it may make analogies to, modify, or
248     combine some example features in the prompt (Appendix F.3). Early in the search, we expect greater
249     exploration due to the diversity of initial examples. As iterations progress, the LLM focuses more
250     on exploiting promising feature spaces, gradually refining the search until convergence. The dataset
251     semantic information serves as a prior that guides the selection of feature attributes and operators to
252     enhance the effectiveness of feature search. The sampling temperature of the LLM can be adjusted to
253     balance exploration and exploitation, with higher temperatures encouraging more diverse solutions
254     and lower temperatures favoring incremental changes to existing examples.
255     We adopt the same set of transformation operators T as those in (Zhu et al., 2022b), including:
256
             • Unary transformations: logarithm, reciprocal, square root, and min-max normalization;
257
258            • Binary transformations: addition, subtraction, multiplication, division, and modulo.
259     When computing min-max normalization, we take the minimum and maximum from the training
260     data. Other transformations require only the information of an individual example. Hence, all our
261     transformation operations can be performed instance by instance on each individual test example
262
        without leaking the information of other test examples. As discussed by Overman et al. (2024), data
        leakage is an issue that has not been properly addressed in many existing AutoFE works.

408   We analyze our LLM-based feature search process for deeper insights. Here, we present experi-

411   regressions, with the shaded area representing one standard deviation above and below the mean.
412
413   Feature Learning. We examine the cross validation score of candidate features on the training data
414   across feature construction iterations. Figure 5 shows a general upward trend in the score as the
415   number of iterations increases. This demonstrates that FEBP effectively improves the quality of
416   constructed features through in-context learning of top-performing examples during feature search.
417
418   Feature Complexity. We analyze the order of candidate features across feature construction it-
419
      erations. Figure 6 shows that the feature order increases more rapidly in the early iterations and
      stabilizes over time. On the one hand, FEBP effectively explores more complex features within
420
      promising feature spaces. On the other hand, our constraint instruction regularizes the process,

      preventing the generation of overly complex features.

512   6    C ONCLUSION
513
514   We propose a novel LLM-based AutoFE algorithm for effective, efficient, and interpretable fea-
515   ture engineering that leverages the semantic information of datasets. Our approach provides the
516   LLM with dataset descriptions and example features represented in canonical RPN, prompting it
517
      to construct new features. The LLM iteratively explores the feature space and improves feature
      construction by learning from top-performing examples. Experimental results demonstrate that our
518
      approach significantly outperforms state-of-the-art AutoFE methods and the inclusion of seman-
519
      tic context from dataset descriptions enhances performance. We also analyze the behavior of our
520   LLM-based feature search process. Our work paves the way for further LLM-driven applications on
521   automated machine learning pipelines and underscores the potential of utilizing semantic informa-
522   tion. In the future, we plan to incorporate adaptive techniques for prompt design.

      A     D ISCUSSION ON C ANONICAL RPN F EATURE R EPRESENTATION
649
650
      A.1   W HY RPN
651
652   RPN provides a compact and unambiguous form of feature representation. In contrast, infix expres-
653   sion requires extra information such as brackets to determine operator precedence. Without brackets,
654   the feature in infix expression col-0 − ( col-1 + col-2 ) would be indistinguishable from the feature
655   ( col-0 − col-1 ) + col-2, while both features are distinctively encoded in RPN. Such compactness
656   and unambiguity of RPN facilitate sequential modeling since there is no need to model the extra
657
      information, e.g., the positions of brackets.
658   Compared with other forms of feature representation such as prefix expression of depth-first traversal
659   or breadth-first traversal, RPN better encodes the recursive structure of the expression tree. The
660   bottom-up enumeration of tree nodes makes it easy for the LLM to evaluate the feature expression
661   by scanning the sequence from left to right, for instance, ((col-0 col-1 −) col-2 +) (parentheses
662   denote recursion). Using the prefix expression (+ (− col-0 col-1) col-2) or breadth-first expression
663
      (+ (− [col-2] col-0 col-1)), however, the LLM always needs to look back to find the operator, which
      undermines sequential modeling. We find in our experiments that when using prefix expression, the
664
      LLM encounters difficulty in generating syntactically valid feature expressions.
665
666
667
      A.2   W HY C ANONICALIZATION
668   While there is one-to-one mapping between feature expression trees and RPN expressions, a fea-
669   ture that contains commutative operators (like addition and multiplication) can be represented by
670   different RPN expressions, since the child nodes of these operators are unordered. We introduce
671   a canonicalization scheme: arranging operator nodes before feature nodes and lexicographically
672   sorting the nodes within each group. Through canonicalization, we create one-to-one mapping be-
673   tween features and cRPN expressions. This ensures the consistency of our feature representation
674   and facilitates the in-context learning of feature patterns.
675   By arranging operator nodes before feature nodes, we also introduce left skewness to the expres-
676   sion tree that enhances the clarity of the recursive structure in cRPN. As illustrated in Figure 10,
677   the original feature expression (col-2 (col-1 col-0 +) ∗) becomes ((col-0 col-1 +) col-2 ∗) after
678   canonicalization, so that the LLM does not need to look back for col-2 when evaluating the expres-
679   sion. We present additional experimental results in Appendix E to validate the effectiveness of our
680   canonicalization scheme.

1524   G    D IFFERENCES TO CAAFE
1525
1526   While our work FEBP and CAAFE (Hollmann et al., 2023) both utilize LLMs to construct new
1527   features incorporating dataset semantic information, they differ in several key aspects. We design
1528
       FEBP such that it taps into the in-context learning capability of LLMs and performs effective fea-
       ture search. In FEBP, we provide top-performing constructed features in the prompt as learning
1529
       examples, label them with performance scores, and rank them by score. We demonstrate that the
1530
       LLM learns to optimize feature construction over the course of algorithm. CAAFE instead stores
1531   all previous instructions and code snippets in the conversation history, which hinders the in-context
1532   learning of optimal feature patterns. It quickly consumes the LLM’s context as the algorithm iter-
1533   ates, incurring more and more LLM generation costs. In comparison, the LLM generation cost of
1534   FEBP stays constant across iterations, without a maximum limit on the number of iterations it can
1535   perform. Therefore, our method FEBP has stronger capability of performing feature search in large
1536   search spaces requiring many iterations, such as datasets with numerous feature attributes.
1537   In FEBP, we also explore representing features in a different form, i.e., canonical RPN (cRPN). We
1538   refer to Appendix A for further detail. Compared with the Python code representation in CAAFE,
1539   cRPN is more compact, which not only reduces LLM generation costs but also makes the in-context
1540   learning of feature patterns easier, and more human interpretable. The use of pre-defined operators
1541   reduces the search space and simplifies the learning process for optimizing feature construction.
1542   Together, our approach gives better control than code representation and helps avoid undesirable
1543   or unexpected LLM outputs. Another advantage of cRPN is that it is convenient to import exter-
1544   nal features (as outlined in Algorithm 1) and export the results as individual features, providing
1545
       compatibility with other feature engineering methods.
1546   More fundamentally, we demonstrate in this work that general-purpose LLMs like GPTs can effec-
1547   tively model recursive tree structures in the form of cRPN feature expressions and reason about the
1548   structures in the context of semantic information, paving the way for further LLM-driven applica-
1549   tions. We hereby underscore the importance of adopting proper representation for the downstream
1550   task to tap into LLMs’ potential.
