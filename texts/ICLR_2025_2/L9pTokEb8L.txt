009                                                    A BSTRACT
010
011             Large Language Model (LLM) agents are rapidly improving to handle increas-
012             ingly complex web-based tasks. Most of these agents rely on general-purpose,
013             proprietary models like GPT-4 and focus on designing better prompts to im-
014             prove their planning abilities. However, general-purpose LLMs are not specif-
015             ically trained to understand specialized web contexts such as HTML, and they
016             often struggle with long-horizon planning. We explore an alternative approach
017             that fine-tunes open-source LLMs using production-scale workflow data collected
018
                from over 250 domains corresponding to 6 billion tokens. This simple yet ef-
                fective approach shows substantial gains over prompting-based agents on exist-
019
                ing benchmarks—our WorkflowAgent achieves state-of-the-art performance on
020
                Mind2Web and substantially improves the baseline task success rate from 37.2%
021             to 51.3% on WebArena. We further perform detailed ablation studies on vari-
022             ous fine-tuning design choices and provide insights into LLM selection, training
023             recipes, context window optimization, and effect of dataset sizes.

026   1    I NTRODUCTION
027
028   Large language model (LLM) agents have advanced significantly in web navigation. They can carry
029   out user-specified tasks in multiple steps by reasoning on their own what actions to take and what
030   external resources to interface with. Recent studies (Zheng et al., 2024; Lai et al., 2024; Zhang
031
      et al., 2024) have shown that, with better planning and exploration strategies, LLM agents can
      independently solve various web tasks ranging from simple navigation, such as locating a specific
032
      Wikipedia page, to more complex operations, such as booking flights or restaurants.
033
034   Despite these improvements, the performance of existing web agents on research benchmarks re-
035   mains significantly below human levels (Deng et al., 2023; Zhou et al., 2024; Drouin et al., 2024).
036   One possible reason is their dependence on general-purpose LLMs. Indeed, all top-performing
037
      agents like WebPilot (Zhang et al., 2024), AWM (Wang et al., 2024b), and SteP (Sodhi et al., 2024)
      rely on prompting proprietary models like GPT-4 (OpenAI, 2024a). These general-purpose LLMs
038
      are not optimized for interpreting web contexts such as HTML or accessibility trees; their pretraining
039
      and alignment processes do not address navigation-related challenges; and their proprietary nature
040   presents a major obstacle in adapting them to web environments via continual training.
041
042   In this work, we explore an alternative approach by fine-tuning open-source LLMs with a large set
043
      of real-world web workflow data1 to develop specialized web agents (Figure 1). Through extensive
      experiments, we show that this approach not only boosts the web understanding and planning abili-
044
      ties of LLMs, achieving state-of-the-art results on various benchmarks, but also allows us to develop
045
      agent models significantly smaller than proprietary LLMs, drastically reducing the serving costs.
046
047   To achieve these results, we first collect a set of proprietary workflow data representing action
048   sequences executed by real users in real web environments. This dataset encompasses a large and
049
      diverse spectrum of websites (over 250 domains and 10,000 subdomains), task objectives, task
      difficulty, and task length. Each step in the workflow features not only the raw HTML-DOM of
050
      the website but also a comprehensive documentation of the action, including action description in

      scale data. This approach not only improves agents’ capabilities relative to prompt-engineered al-
091
      ternatives, but also reduces inference costs due to the smaller sizes of open-source LLMs. While
092
      our work focuses on studying the effect of fine-tuning, WorkflowAgent can be extended to leverage
093   more sophisticated search or memory modules (Koh et al., 2024; Wang et al., 2024b), combined
094   with existing planning frameworks (Yao et al., 2022; Madaan et al., 2023; Shinn et al., 2023), or
095   integrated into multi-modal web agent systems as the text model (Wang et al., 2024a). We view
096   WorkflowAgent as an important step towards developing AI assistants and fully automated agents
097   for real-world web applications.

      2    R ELATED W ORK
100
101
      Prompting-based agent frameworks. The majority of web agent works reuse existing LLMs and
102   propose different prompting strategies to improve action prediction. One line of research focuses on
103   exploiting previous experience via self-feedback (Sun et al., 2023) or in-context demonstrations (Fu
104   et al., 2024; Zheng et al., 2024; Wang et al., 2024b; Ou et al., 2024). A separate line of work
105   centers around encouraging exploration by including external evaluators (Pan et al., 2024), using
106   synthesized instructions (Murty et al., 2024), or applying more sophisticated search algorithms like
107   stack (Sodhi et al., 2024), best-first tree search (Koh et al., 2024), or Monte Carlo Tree Search (Zhang
      et al., 2024). Chain-of-Thought (Wei et al., 2023) or ReAct (Yao et al., 2023) prompting have also


                                                            2
      Under review as a conference paper at ICLR 2025


108
      been used (He et al., 2024). Despite the research efforts, these prompting methods rely heavily
109   on the quality of the LLM used. Open-source models such as LLaMA (Dubey et al., 2024), Code
110   LLaMA (Rozière et al., 2024), and Flan-T5 (Chung et al., 2022) generally underperform proprietary
111   models like GPT-4. However, fine-tuning proprietary LLMs can often be costly and challenging, as
112   it is restricted to being done through APIs. This implies an opportunity for enhancing open-source
113   LLMs to match or outperform proprietary agents.
114
      Fine-tuning-based web agents. Compared to developing better reasoning and planning frame-
115   works, comparatively less attention has been given to optimizing the LLMs themselves to better
116   handle web environments. Due to the difficulty of directly generating a single target element from
117   the raw HTML, which often contains thousands of elements, existing work mostly focuses on multi-
118   stage prediction. MindAct (Deng et al., 2023) proposes a two-stage pipeline that first uses a small
119   LM to filter the web elements and then uses a more powerful LM to select from the filtered elements
120   in a multi-choice question answering format. Both LMs can be fine-tuned using the Mind2Web
121   dataset. WebAgent (Gur et al., 2023) uses HTML-5 to first process the HTML and then fine-tunes
122   a 540B Flan-UPalm to generate code for controlling web pages. More recently, AutoWebGLM (Lai
123
      et al., 2024) trains a single ChatGLM3 6B (GLM et al., 2024) using a combination of curriculum
      learning, reinforcement learning, and rejection sampling fine-tuning. Despite the complicated train-
124
      ing and inference procedures, these methods often underperform agents that prompt GPT-4. In
125
      contrast, our work shows that given sufficient high-quality workflow data, fine-tuning a single LLM
126   can achieve strong performance. We note that the newly released OpenAI o1 (OpenAI, 2024c) can
127   be viewed as a specialized agent with a complicated planning framework. Nonetheless, we show
128   in Section 4.1 that WorkflowAgent outperforms o1-preview by a large margin on our proprietary
129   dataset. Moreover, while none of the training details for o1 have been released, our work provides
130   valuable insights into data preprocessing and fine-tuning.
131
      Beyond the aforementioned work, there is an earlier line of research that fine-tunes LLMs for HTML
132   inputs (Gur et al., 2022; Nakano et al., 2022; Liu et al., 2023). However, their primary application is
133   question-answering tasks, such as answering “could sunflowers really track the sun across the sky”,
134   and they cannot be used to generate a sequence of actions based solely on the user objective.
135
      Lastly, we note that an emerging line of research has committed to developing multi-modal web
136
      agents that use screenshots along with HTML observations. Examples include CogAgent (Hong
137
      et al., 2023), SeeClick (Cheng et al., 2024), WebVoyager (He et al., 2024), and AWA 1.5 (JaceAI,
138   2024). However, our current version of WorkflowAgent focuses exclusively on text-based inputs due
139   to the lack of extensive, high-quality paired data and effective visual preprocessing schemes. Thus,
140   we do not include comparisons with the aforementioned multi-modal methods in our experiments
141   and leave developing multi-modal WorkflowAgent as future work.

      In this section, we first overview the general setup of solving web-based tasks with LLM agents.
146
      Then, we detail our proposed method to develop specialized agents from open-source LLMs.
147
148
      3.1   G ENERAL S ETUP
149
150   We consider solving web-based task as a sequential decision-making process guided by a high-level
151   objective. For each task, the user first specifies an objective and a starting web page. Then, at
152   every step, the agent outputs an action based on the task objective, the current web page, and the
153   history. Formally, denote the user objective as q. The web environment is governed by a transition
154   function T that can evolve over time. The agent is instantiated by a language model L. At each time
155   step t, the agent observes ot produced by the environment state st and observes the history ht =
156   H(o1:t−1 , a1:t−1 ). It outputs an action at = L(q, ot , ht ), which is executed in the environment, and
157   the state changes correspondingly st+1 = T (st , at ). This iterative process stops when the agent
158
      issues a stop signal, or a task termination condition is met, such as we have reached a predefined
      maximum number of steps.
159
160   For single-modal, text-only agents, the observation ot typically consists of the website’s URL, the
161   HTML-DOM (Object Model for HTML, which defines HTML elements and their properties, meth-
      ods, and events), and potentially the accessibility tree (a representation that can be understood by


                                                         3
      Under review as a conference paper at ICLR 2025


162
      assistive technologies like screen readers). Since the raw HTML-DOM is often long and contains
163   redundant structural information, most methods employ preprocessing and pruning strategies, which
164   could be as simple as retaining a fixed set of HTML tags and attributes or more complex ones like
165   LLM-based element ranking and filtering (Deng et al., 2023).
166
      The action at emulates the keyboard and mouse operations available on web pages. The most general
167
      action space in existing work consists of element operations, such as clicking, typing, and key
168
      combination pressing; tab actions, such as opening, closing, and switching between tabs; navigation
169   actions, such as going forward and backward in the browsing history (Zhou et al., 2024).
170
171   As discussed earlier, previous web agent work focuses on presenting useful demonstrations through
172
      ht or iteratively revising at to improve the quality of the predicted next step. In contrast, we explore
      whether we can improve the model L itself by learning from a vast amount of data and incorporating
173
      more information into ot , such as the natural language description and HTML representation of a
174
      action. We detail our approach in the next section.

      3.2     W ORKFLOWAGENT : S PECIALIZING W EB AGENTS T HROUGH F INE -T UNING

178   3.2.1    C OLLECTING P RODUCTION -S CALE DATA
179
180   We collected a large set of real-world proprietary data through a workflow documentation software
181   that streamlines the creation of step-by-step guides to achieve web-based tasks. The software allows
182   users to record their interactions with the web through a browser extension and converts the interac-
183
      tions into well-annotated instructions, which can be then customized to specific business needs. Our
      dataset consists of everyday workflows in common web application domains, encompassing cus-
184
      tomer relationship management (CRM) tools like HubSpot and Salesforce; productivity tools like
185
      Notion and Calendley; social platforms like Facebook and LinkedIn; shopping sites like Amazon
186   and Shopify; and many others.
187
188   Each workflow features a high-level user objective and a step-by-step documentation of the action
189
      sequence to achieve the task. The objective spans a wide range of topics, such as “add a user in a
      Salesforce” or “invite someone to manage Facebook ad accounts”. Each step contains the following
190
      information: the current web page’s URL, raw HTML-DOM, a natural language description of the
191
      action performed, the type of action, and the autogenerated CSS selector to identify the action target.
192   There are three types of actions in the dataset:
193
194           • mouse click action: click at an element
195           • keyboard sequence action: type a sequence of characters to an element
196
              • keyboard combination action: press a set of keys together (e.g., hotkey like ctrl+c)
197
198   Note that there is no scroll actions in our action space since all elements are already fully accessible
199   in the captured data. This is because we capture the full DOM from a system perspective, which
200   inherently includes the entire webpage as observed from the backend. This method differs from user-
201   centric data collection, where only the elements within the visible browser viewport are captured.
202
      To ensure the quality of the data, we remove workflows with invalid selectors, i.e., the selector
203
      cannot be used to locate a target element in the DOM. We also remove non-English workflows to
204   reduce dataset complexity and enable us to explore English-only LLMs like Mistral 7B (MistralAI,
205   2023). The resulting dataset is at production scale: using raw data collected over a two-month
206   period, we are able to extract workflow data from more than 250 domains and 10,000 subdomains
207   with an average task length of 11 steps, which correspond to about 6 billion training tokens. This
208   large-scale, high-quality, real-world dataset is unmatched in prior web agent research.
209
      Since this dataset is collected from real users and might contain sensitive and confidential informa-
210   tion, it will not be released to the public to protect user privacy. The dataset is solely for research
211   purposes and has been anonymized to prevent the identification of any individual.

213   3.2.2    P REPROCESSING
214
215   For WorkflowAgent, we consider an observation space consisting mainly of the URL and HTML-
      DOM. Specifically, HTML-DOM provides agents with all structural and content information about


                                                          4
      Under review as a conference paper at ICLR 2025


216
      the web page that are essential for generating the next step and long-term planning. For instance,
217   while a drop-down menu may not be visible on the website before expansion, the agent can detect the
218   menu items from the DOM and determine whether to click and expand it. We do not use accessibility
219   tree to develop WorkflowAgent because it may lose information about the HTML elements, such as
220   the drop-down items, and does not generalize across different browsers and devices.
221
      Given our observation space, a subsequent problem is that the DOM can be quite long and exceed
222
      the context window of prevailing open-source LLMs. To reduce the DOM sizes, we propose a
223   pruning algorithm that maintains the essential structure and content while eliminating redundant or
224   disruptive elements that could hinder the LLM’s understanding. Specifically, we first use the Beau-
225   tifulSoup library (Richardson, 2007) to remove non-essential components such as metadata, CSS,
226   and JavaScript. Then, we utilize a tag-attribute white list to retain useful tag level information like
227   retaining interactive elements. Since some attribute values can contain random character sequences
228   that do not provide useful information, we propose a novel detection method that removes the at-
                                                                         len(s)
229   tributes with character-to-token-ratio smaller than 2, i.e., len(tokenizer(s)) < 2, where s denotes the
230   value string. Intuitively, if each character in a string is encoded using a separate token, it is highly
231   likely that the string is not semantically meaningful. Lastly, we remove the comments and extra
232   whitespaces to clean up the DOM. After pruning, we assign each tag in the HTML with a unique ID
233   by traversing the HTML tree from bottom to top. More details about preprocessing and analysis on
234   the tokenizer-pruning method can be found in Appendix A.1.
235   We restrict the action space of WorkflowAgent to the three types of operations specified in Sec-
236   tion 3.2.1. To preprocess the action sequences, we rewrite each step into five lines as follows:
237
238    1.
239
       Description: Click the “Menu” button to browse all food options
       Action: mouse click action
240
       Node: 832
241
       Target: <svg class=“open-hamburger-icon” node=“832” role=“img”>
242
243   The first line represents the current time step. The second line is the natural language description
244   of the action, which can help LLMs to learn about the rationale behind applying a specific action.
245   The third line is one of the three operations in the action space. The fourth line is the unique ID
246   assigned to the target element. The last line details the HTML tag and attributes, which can be
247
      directly obtained from the processed DOM.
248   For the history, we consider only previous actions, omitting previous observations due to the exten-
249   sive length of the DOMs. That is, ht = a1:t−1 . Therefore, at each step, WorkflowAgent will be
250   given the task objective, URL, HTML-DOM, and all previous actions in the aforementioned five-
251   line format. Its goal is to output the next action at = L(q, ot , a1:t−1 ) that helps complete the task.
252   In Appendix A.3, we provide an example of a full workflow.
253   Lastly, during our inspection, we find that 10% of the action descriptions in the dataset are not
254   informative (e.g., “click here”). In these cases, we use GPT-4o (OpenAI, 2024b) to regenerate the
255   action description from screenshots. We provide the prompt as well as examples of the regenerated
256   action descriptions in Appendix A.4.1.

258   3.2.3   F INE -T UNING WITH L O RA
259
260   After preprocessing, we divide the dataset into two splits. The test set comprises of 1200 workflows
261   with diverse objectives and domains. We use the remaining workflows as the training data to adapt
262   LLMs via standard supervised fine-tuning. Note that for each example, the label is a single next-
263   step instead of all remaining steps needed to complete the task. The agent is trained to generate all
264   information in the five-line format described above, including the natural language description.
265   To reduce fine-tuning cost, we opt for the parameter efficient method LoRA (Hu et al., 2022) instead
266   of full fine-tuning, since we have not observed significant performance gain by updating more pa-
267   rameters. We also follow previous work (Zhao et al., 2023) to fine-tune the layernorms in addition
268   to the LoRA adapters. Based on empirical observations, we set the fine-tuning epoch to 2, effective
269   batch size to 32, LoRA rank to 64 and α to 128. We use a cosine scheduler with 30 warmup steps
      and a learning rate of 1e-4.

      Context Window Length. We evaluate the models with 65K context window to add additional
325   context and increase the rate of solvable tasks (Table 2). On both Qwen2 and Qwen2.5, scaling the

332   the agent’s performance. To this end, we sample our training set without replacement into smaller

334   scale, we observe that there is a roughly 2% performance boost when we double our dataset size.
335
      To sum up, using our proprietary dataset, we study the effect of LLM backbone, context window,
336
      and dataset size on the agent performance. We find that (1) scaling parameter count generally im-
337   proves prediction quality, but the latency and training time of large LLMs can be prohibitive; (2)
338   using longer context window boosts model performance on EM but increases the inference time
339   significantly; (3) training with more tokens is helpful. Based on these insights, we use Qwen2 7B
340   fine-tuned on the full 6B-token dataset at a 32K context window as the final version of WorkflowA-
341   gent. The results shown in later sections are based on this model. Since WorkflowAgent only has
342   7B parameters, it is much cheaper to serve at inference time than large-scale proprietary models.

      4.2   M IND 2W EB
411
      Mind2Web (Deng et al., 2023) is a text-based dataset for assessing the navigation ability of web
412
      agents across different tasks, websites, and domains. Each task features a human demonstration of a
413
      real-world workflow, such as booking a hotel on Airbnb. At each step, the agent is asked to predict a
414   single action, consisting of an operation and the target element. Performance is measured by element
415   accuracy, which checks if the correct target is selected; action F1 score, which measures operation
416   correctness like text input; step success rate, which evaluates whether both the target element and
417   the operation are correct; and task success rate, indicating all steps are correct.
418
      The original Mind2Web benchmark reports two sets of baselines: (1) a single-stage, generation-
419
      based agent (i.e., fine-tuned Flan-T5B ) directly generates the operation and the target based on
420   the full DOM; (2) multi-stage, multi-choice question-answering agents (i.e., the MindAct family)
421   first use a pretrained element-ranking model to filter out 50 candidate elements from the full DOM
422   and then use a separate LLM to recursively select an action from five candidates in a multi-choice
423   question-answering (QA) fashion until one action is chosen. Both sets of baselines are trained using
424   the training data and then evaluated on the test set. Note that direct generation is more challenging
425   than multi-choice QA, and all multi-stage baselines outperform generation baselines by a large mar-
426   gin. Beyond the Mind2Web original baselines, we also consider memory-augmented agents such as
427   AWM (Wang et al., 2024b) and Synapse (Zheng et al., 2024), fine-tuned AutoWebAGLM (Lai et al.,
428
      2024) and HTML-T5 (Gur et al., 2023).
429   WorkflowAgent belongs to the single-stage, generation category. We directly evaluate its perfor-
430   mance on the Mind2Web test data without using Mind2Web training data for further adaptation. For
431   performance robustness, we call WorkflowAgent five times and use majority vote to select the final
      generated action. More details about DOM processing and output comparison are in Appendix A.5.

      4.3   E ND - TO -E ND TASK E XECUTION ON W EBA RENA
460
461
      WebArena (Zhou et al., 2024) features 812 web navigation tasks across five domains: E-commerce
462   (OneStopShop), social forums (Reddit), software development (GitLab), content management
463   (CMS), and online map (OpenStreetMap). Unlike the static Mind2Web, it implements a dynamic
464   environment for agents to interact with and allows for assessing the functional accuracy of action
465   sequences. Since the WebArena environment is implemented to accept only target element IDs
466   specified in the accessibility tree, whereas WorkflowAgent operates on DOM and outputs targets in
467   HTML, we employ GPT-4o to map between the different representations.
468   More generally, we tackle end-to-end task solving by developing a multi-agent system that utilizes
469   GPT-4o to simulate user interactions with WorkflowAgent: (1) objective refinement: user adds
470   details about the task objective to help complete the task; (2) action generation: based on the current
471   website and action history, WorkflowAgent outputs an action suggestion; (3) action execution: user
472   executes the suggested action, e.g., clicking a button; (4) completeness evaluation: user observes the
473   current state and decides whether the task is completed.
474   We apply the above pipeline to solve the WebArena tasks. In stage 3, GPT-4o maps the agent’s
475   output in HTML to the accessibility tree format, which is then processed by the WebArena envi-
476   ronment. To further improve performance, we allow WorkflowAgent to generate multiple actions
477   in stage 2 and select the one with the highest confidence using majority vote and GPT-4o analysis.
478   More details about evaluating WorkflowAgent on WebArena can be found in Appendix A.6.
479
      We compare our performance with all top-performing, text-only agents on the WebArena leader-
480   board. We note that we do not include Autonomous Web Agent (AWA) 1.5 (JaceAI, 2024) as a
481   baseline because it uses a proprietary system to parse the HTML-DOM and web screenshots, rather
482   than building from the WebArena GitHub. This allows them to have richer observations and bypass
483   the accessibility tree action mapping step. In contrast, WorkflowAgent is single-modal, text-only,
484   and we stick to the original WebArena implementation. That said, AWA 1.5 employs more ad-
485   vanced reasoning, planning, and progress tracking techniques and is the only agent system with a
      higher average task success rate than ours.

522   WebArena, further highlighting the advantages of fine-tuning web agents with large-scale datasets.

      5    C ONCLUSION
525
526
      In this work, we explore how fine-tuning open-source LLMs with high-quality real-world workflow
527   data can benefit developing specialized web agents. We present WorkflowAgent, which consistently
528   outperforms existing methods that prompt proprietary models in various evaluation settings and
529   benchmarks. We also provide empirical insights into data processing and model fine-tuning.
530
      Limitations and Future Work. The long-context nature of DOMs presents great challenges in
531
      adapting LLMs. In the short term, we aim to enable WorkflowAgent to compare and reason over
532
      multiple DOM chunks so that its observation is always complete. This might require integrating
533   a memory component, which could also aid in maintaining context or state across interactions to
534   improve multi-step reasoning. Besides, we currently do not incorporate planning into WorkflowA-
535   gent, so its output will be directly used as the next action. However, adding better action selection
536   strategies such as Monte Carlo Tree Search (MCTS) could potentially facilitate online planning and
537   exploration, further improving the agent’s decision-making processes in complex scenarios. In the
538   long run, we aim to expand WorkflowAgent’s capabilities to handle multi-modal inputs and mul-
539   tilingual content. This would significantly broaden its applicability across different linguistic and
      visual contexts, making it more versatile and robust in real-world web environments.

      A.1.2     T OKENIZER P RUNING
757
758   In this section, we provide more details on the tokenizer-based detection method to remove random
759   character strings. The rationale behind our approach is based on the observation that typical English
760   words consist of more than two characters. Assuming the token count is t and the character count is
761   s, this means that when t = 1, s ≥ 2, leading to st ≥ 2. By setting the pruning threshold to 2 and
762
      removing tag attributes with st < 2, we aim to eliminate strings composed solely of single-character
      tokens, which are likely to be nonsensical.
763
764   In our actual implementation, we employ this technique only for tag attributes with s > 32, being
765   more lenient for shorter attributes. To show that this tokenizer pruning strategy is effective and to
766   study the performance across different tokenizers and pruning thresholds, we perform the following
767   experiments.
768   We take three tokenizers from different models: Qwen2-7B-Instruct, Mistral-7B-Instruct-v0.3,
769   and Meta-Llama-3-8B. For each tokenizer, we vary the pruning thresholds across a set of values:
770   {1.5, 1.75, 2, 2.25, 2.5}. Note that it is meaningless to study overly small thresholds (e.g., it is im-
771   possible to have st < 1) or overly large thresholds (e.g., st < 3 could result in the loss of meaningful
772   attributes, as many English words contain three letters). We randomly sample 1000 DOMs from our
773   proprietary test dataset, apply our standard pruning pipeline followed by tokenizer pruning, and then
774
      perform three analysis:
775
              • False positives: we use the Python enchant library to detect if there are meanful English
776
                words within the pruned strings. Note that even though these are actual words, many of
777             them are related to DOM structure and can be safely ignored. Still, we count them as false
778             positives since the tokenizer method is designed to remove random character strings.
779
              • Average s and t for the entire DOM before and after tokenizer pruning: this is for under-
780             standing the reduction in content length.
781
              • Lastly, we sort tags and attributes by the frequency of being pruned to identify patterns.

       A.6.2    S CROLLING ACTIONS AND C OMBOBOX S ELECTION
1027
1028   In our data collection process, we capture the full DOM from a system perspective, which inherently
1029   includes the entire webpage as observed from the backend. This method differs from user-centric
1030   data collection, where only the elements within the visible browser viewport are captured. Conse-
1031   quently, there is no concept of scrolling in our training datasets since all elements are already fully
1032   accessible in the captured data.
1033   However, we recognize the importance of scroll actions in solving WebArena from a user perspec-
1034   tive. To address this, before issuing any action to the environment, our multi-agent system includes
1035   a viewport check that uses the bounding box position to determine if the target element is within
1036   the visible webpage area. If not, the system manually inserts necessary scroll actions to bring the
1037   element into view. This ensures accurate interaction with web elements in a typical user scenario.
1038   To handle combox selection, our agent discovers a workaround that bypasses the need for scrolling
1039   through comboboxes. Specifically, after clicking on the combobox, it types the name of the desired
1040   item in the combobox, which brings the item to the top of the dropdown menu. Then, the agent can
1041   simply click the item or press Enter. This approach avoids the need for scrolling and is especially
1042   effective in densely populated lists. It improves the task success rate on a large number of Map,
1043   Reddit, and GitLab tasks.
