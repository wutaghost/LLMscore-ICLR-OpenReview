009                                              A BSTRACT
010
011            For web agents to be practically useful, they must adapt to the continuously evolv-
012            ing web environment characterized by frequent updates to user interfaces and
013            content. However, most existing benchmarks only capture the static aspects of the
014            web. To bridge this gap, we introduce WebCanvas, an innovative online evalua-
015            tion framework for web agents that effectively addresses the dynamic nature of web
016
               interactions. WebCanvas contains three main components to facilitate realistic
               assessments: (1) A novel evaluation metric which reliably capture critical inter-
017
               mediate actions or states necessary for task completions while disregarding noise
018
               caused by insignificant events or changed web-elements. (2) A benchmark dataset
019            called Mind2Web-Live, a refined version of original Mind2Web static dataset
020            containing 542 tasks with 2439 intermediate evaluation states; (3) Lightweight
021            and generalizable annotation tools and maintenance pipelines that enables the
022            community to collect and maintain the high-quality, up-to-date dataset. Building
023            on WebCanvas, we open-source a baseline agent framework with extensible mod-
024            ules for reasoning, providing a foundation for the community to conduct online
025            inference and evaluations. Our best-performing agent achieves a task success rate
026            of 23.1% and a task completion rate of 48.8% on the Mind2Web-Live test set.
027
               Additionally, we analyze the performance discrepancies across various websites,
               domains, and experimental environments. We encourage the community to con-
028
               tribute further insights on online agent evaluation, thereby advancing this field of
029
               research.

032   1   I NTRODUCTION
033
034   The enhanced reasoning capabilities of foundational models (Ouyang et al., 2022; Achiam et al., 2023;
035   Touvron et al., 2023a;b; Liu et al., 2024a; Bai et al., 2023) demonstrate the potential for autonomous
036   agents performing on navigation and information retrieval tasks in real-time within web environment,
037   thereby augmenting the human workforce (Shi et al., 2017; Nakano et al., 2021). However, the
038   journey towards autonomous web agents delivering accurate, robust, fast, and cost-effective outcomes
039
      to end-users remains fraught with challenges. These include the inherent scarcity of data, the lack
      of knowledge and reasoning abilities of high-level actions on certain websites, and the absence
040
      of accurate and effective process feedback during execution, among others (Gür et al., 2023; Gur
041
      et al., 2024). We posit that a significant barrier to realizing the value of web agents is the lack of
042   an easy-to-use platform for the community to drive effort towards real-time data gathering and web
043   agent online benchmarking. This belief is grounded in following observations.
044
045
      Digital agents require environmental observations and feedback for context. Thus, dynamic, real-
      world environments are essential for agent evaluation and data collection. The Internet itself emerges
046
      as the most extensive arena for the assessment of agents, offering an unparalleled complexity for
047
      environmental interaction (Liu et al., 2018; Zhou et al., 2023). However, the rapid evolution of the
048   web environment introduces significant data distribution shifts over time. Figure 2 summarizes three
049   prevalent patterns of changes in web tasks over time. For example, the Mind2Web dataset (Deng
050   et al., 2024), which archives web-based interactions as static HTML snapshots and was released one
051   year ago, shows that more than half tasks(50.5%) changed to some extent in their corresponding
052   live websites one year later. This shift may potentially create discrepancies between the offline
053   and online development and evaluation of real-world web agents. In addition, the accumulated
      knowledge and training data of static websites leads to the saturation of existing benchmarks, making

      2     P ROBLEM F ORMULATION OF I NTERACTIVE W EB -BASED TASK
134
      The real-world web environment can be formulated as: (S, A, T , O) with state space S, action space
135
      A(Table 10), deterministic transition function T : S × A −→ S and a state observation space O(§5).
136
      Given a task instruction i, current observation ot ∈ O and the action history a1:t−1 , an agent issues an
137   action at ∈ A. Consequently, after the execution of the action, the environment transitions to a new
138   state st+1 ∈ S, and the corresponding observation updates to ot+1 ∈ O. To measure the completion
139   of tasks, we have defined key nodes and evaluation metrics, which are elaborated in §3.1 and §3.2.

      3.1    D EFINITION OF K EY N ODES
144
145   The concept of “key nodes” is one of the pivotal ideas in our work. Key nodes refer to indispensable
146   steps in the process of completing specific web tasks, meaning that regardless of the path taken to
147   accomplish a task, these steps are required. These may involve navigation to certain webpages or
148   the performance of specific actions on web pages, such as filling out forms or clicking buttons. This
149   design philosophy not only reflects the dynamic nature of the web environment but also captures the
150   diversity of paths present in real-world web pages.
151   As illustrated in Figure 1, consider the task of “Find top-rated upcoming adventure movies on Rotten
152   Tomatoes” as an example. Users might start directly at the Rotten Tomatoes homepage or use a
153   search engine to navigate straight to the “New Movies Coming Soon” page of the Rotten Tomatoes.
154   Moreover, when filtering the movies, users might choose to first apply a filter for the “adventure”
155   genre and then sort by popularity, or alternatively, sort by popularity before applying the genre
156   filter. Despite the availability of different paths to achieve the goal, entering the specific page and
157   performing the genre and popularity sorting are essential steps in accomplishing the task. Therefore,
158
      these three steps are identified as “key nodes”.
159   In the dynamic and noisy real-world web environment, identifying these key nodes is challenging due
160   to the potential changes in page content and UI updates, which could render element selector paths
161   obsolete. Therefore, we preferred to use URL state as identifiers for key nodes rather than element
      interaction, which enhanced the Benchmark’s robustness against layout changes. Only element class

183   3.2   E VALUATION M ETRICS
184
185   The evaluation metrics of WebCanvas comprised of two main components: step score and task
186
      score. The step score evaluates the agent’s performance with regard to each key node, defining three
      types of evaluation targets along with three evaluation functions at each step. The task score includes
187
      two functions to assess the task’s completeness and overall execution efficiency.
188
189
190
      Step Score Inspired by previous works (Zhou et al., 2023; Koh et al., 2024), we introduced three
      evaluation targets in calculating step score, allowing us to examine from different aspects: URL,
191
      Element Path, and Element Value. We implemented three match functions for these targets: Exact
192
      Match, Include Match, and Semantic Match. Each key node is associated with an evaluation
193   function, which comprises an evaluation target and a match function. One step score is awarded when
194   the agent successfully reaches a key node and passes the associated evaluation function verification.
195   Table 1 shows a list of applicable evaluation functions and their introductions for reference. Some
196   examples are shown in Table 7. We use Completion Rate to represent the MICRO average of Step
197   Scores.

211   Task Score Task performance is evaluated using two main metrics: the Task Success Rate and

      4.1    DATASET C ONSTRUCTION
220
      To develop a real-world online benchmark for web agents, we introduce Mind2Web-Live, which is
221
      derived from tasks present in the Mind2Web (Deng et al., 2024) dataset. We employed WebCanvas
222
      framework as a guidance for the sampling and re-annotation of these tasks. Consequently, we
223   selectively excluded all tasks that contained time-sensitive descriptions, such as those involving
224   specific dates or times. We randomly sampled 601 tasks from the training set and included all 179
225   tasks from the cross-task test set, which were then re-annotated in a real-world online environment.
226
      The annotation process presented multiple challenges. Notably, due to updates in website content
227
      and operational changes, we discovered 96 tasks that were no longer applicable and subsequently
228
      removed them from the dataset. Additionally, 142 tasks were discarded due to ambiguous task
229   definitions, log-in requirements or the difficulty in clearly defining key nodes. To enhance the clarity
230   and reliability of task execution, we revised the instructions for 51 tasks.
231
232
      For human trajectory and key node annotation, we developed and made public a browser plugin and
      an annotation platform. After a rigorous annotation and review process, described in Appendix A,
233
      542 high-quality tasks were established for the Mind2Web-Live dataset, including 438 of the training
234
      set and 104 of the test set. As shown in Table 2, Mind2Web-Live encompasses 2439 key nodes and
235   4550 detailed annotation steps. The tasks in the dataset cover a wide range of webpage types and
236   operations, designed to comprehensively evaluate the performance of web agents in a dynamic and
237   variable online environment. The distribution of the evaluation function is illustrated in Figure 5.

      5.1    D ISCREPANCY BETWEEN O FFLINE E VALUATION AND O NLINE G ENERALIZATION
297
      The settings of evaluation on offline datasets that reflect real-world intents, such as Mind2Web (Deng
298   et al., 2024), are inherently different from WebCanvas framework. Nevertheless, we managed
299   to study the qualitative discrepancy between offline evaluation and online generalization. During
300   online inference, we attempted to reproduce the original prompt of the MindAct model, which was
301   trained and evaluated on the offline dataset, as proposed in the Mind2Web paper. It is important to
302   note that the evaluation metrics used in offline evaluation differ from those proposed in our online
303   evaluation framework. The Step Success Rate in offline testing assesses the accuracy of single-step
304   action prediction, and for the entire task dimension, a positive reward is given only when all single-
305   step actions are correctly predicted, which is not the case in online evaluation, as we evaluate the
306
      intermediate state, not the referenced action.
307
308   5.2    M AIN R ESULT
309
      In our experiments, we observed significant discrepancies between offline and online testing. The
310
      results, as detailed in Table 3, show that the model trained on the Mind2Web training set struggles
311
      to generalize to the online environment a year later. The comparative performance of MindAct-
312   Large (Deng et al., 2024), GPT-3.5, and GPT-4 in the online environment was opposite to that
313   in offline testing. We further analyzed why such differences occur. Specifically, through human
314   annotation of 100 evaluation steps of GPT-4, we noted that approximately 30% of its actions were
315   reasonable, indicating that these models are possibly capable of generating valid paths under the
316   prompt, albeit penalized by the offline reference-based evaluation method. We also discovered that
317   the MindAct model, when reasoning in an online environment, frequently encountered difficulties
318   in recovering from erroneous states. When entering web pages less relevant to the task goal, the
319   MindAct model had a high probability of outputting null actions, causing the task to terminate.
320   Furthermore, comparative performance of different models in Table 4 indicates that GPT-4 outper-
321   forms other models in both effectiveness and efficiency in web agent tasks within a live environment,
322   with Qwen being the best-performing open-source model. However, there remains considerable room
323   for future enhancements across all models. These results underscore the need for models that can
      better generalize to dynamic, real-world web environments.

348
349   6.1   FACTORS I NFLUENCING AGENT P ERFORMANCE
350
351   In this section, we delve into the factors influencing agent performance across a range of web tasks.
352   Through a series of experiments, we assessed the impact of task complexity, website dynamics,
353   task domain, key node distribution in the dataset, and the experimental setup—including system
354   specifications, browser engine, and IP location.
355   Our findings reveal that increased task complexity directly correlates with diminished agent per-
356   formance. The domain of the task also significantly affects performance, with agents handling
357   entertainment-related tasks more adeptly than those involving shopping or travel. This variation sug-
358   gests LLMs’ capacity of semantic understanding and reasoning differs across domains and websites.
359   Moreover, the experimental environment plays a crucial role in agent performance. We recommend
360   experimenting on a Windows platform using Chrome or Firefox browser engines, preferably on
361   servers located in the United States. Statistics and experiment results are detailed in Appendix F.2.
362
363
      6.2   N ECESSITY OF K EY N ODE E VALUATION IN L IVE E NVIRONMENTS
364
365
      Previous agent evaluation methods primarily focus on two aspects: reference-based evaluation (Deng
366   et al., 2024; Zheng et al., 2024) and outcome-based evaluation (Zhou et al., 2023; Koh et al., 2024;
367   Mialon et al., 2023). However, these methods falter when applied to the unpredictable nature of live
368   web tasks. To address the inherent variability in task completion paths within an online evaluation
369   framework, we employed Sankey diagrams to visualize the trajectories of our web agent and human
370   demonstrations on tasks where our agent successfully navigated all designated key nodes in Figure
371   10 within §F.2.
372   We further annotate Mind2Web-Live test set to identify whether the final key node is a sufficient
373   condition for task completion. It turns out only 46 out of 104 tasks met this criterion. This finding
374   starkly illustrates that solely evaluating the final state or outcome is inadequate for web environments
375   that are not fully reproducible. As shown in Figure 3, key node based evaluation enhances explainabil-
376   ity of agent performance, prevents illegal shortcuts taken and facilitates the modeling of structured
377   in-progress rewards, valuable for both in-context reasoning experiments and future reinforcement
      learning training.

399   6.3    P LANNING WITH H UMAN -L ABELED R EWARD
400
401   Reward modeling for agent tasks is crucial in both in-context reasoning and reinforcement learn-
402   ing (Shinn et al., 2024; Bai et al., 2024). Previous research has proven that LLMs can generate high
403
      quality reward signal to enhance reasoning performance across various agent tasks (Shinn et al., 2024;
      Pan et al., 2024). However, recent research adopting an un-tuned foundation model for self-reward
404
      prediction shows that their effectiveness is not consistent in specific domains (Olausson et al., 2023;
405
      Shinn et al., 2024). Our preliminary experiments indicate that agent performance do not benefit from
406   a self-reward module in the online web environment. This is attributed to several factors, such as
407   overconfidence in task completion assessments and the long-term impact of poor-quality reward
408   signals accumulated in agent memory. Thus it raises a natural question - Does the quality of the
409   reward signal hinder the self-reward module’s effectiveness in online web environments? In our
410   study, we introduced a reward module with human-labeled reward. The experimental results on
411   Mind2Web-Live, which confirm our hypothesis, are detailed in Table 5.
412   From the original data, we extracted post-action URLs, action types, CSS selector paths, and key
413   nodes functions as metadata for our golden reference synthesis. We then employed a carefully
414   designed prompt available in Appendix K, using GPT-4 to generate a structured linguistic guidance
415   for task progress estimation for each task. This guidance includes the overall goal of the task and
416   task completion criteria, specifically highlighting all key nodes for the task to be considered fully
417   completed. We then integrate the content of the current task’s golden reference with the original
418   design of history and current observation for reward reasoning. From comprehensive experiments,
419   we find that the integration of a reward module does not enhance agent performance and may even
420
      lead to a decline in Task Success Rate and Task Completion Rate. This finding aligns with findings
      in (Shinn et al., 2024) about the effect of self-reflection modules in web agent tasks. However, we
421
      find the Completion Rate improves in both GPT-3.5 and GPT-4 experiments with the integration of a
422
      reward module with human-labeled reward, despite the reward module triggering premature stops.
423   These findings point out the importance of better reward modeling in web agent reasoning.

      8      D ISCUSSION & L IMITATIONS
463
      Developing a suitable evaluation framework is a fundamental component in the advancement of
464
      autonomous web agents. This research addresses the challenge of live evaluation in a real-world
465
      web environment. Among these are the need to define key nodes in a completely open environment,
466   unify the inference processes across different digital autonomous agents, and reduce the maintenance
467   costs associated with real-time data and evaluation functions. Through our efforts, we have made
468   significant strides toward establishing a robust and accurate online evaluation system for web agents.
469
      However, the transition to live, dynamic evaluations in unpredictable online environments introduces
470
      new complexities not present in controlled, offline settings. The unsolved challenges we encountered
471
      in online evaluation of web agents include network instability, dynamic and complex task pathways,
472   and the limitations of static evaluation functions. These challenges highlight the necessity for ongoing
473   research and community efforts to refine and enhance evaluation frameworks for autonomous web
474   agents in complex, real-world environments. For more details, please refer to Appendix J.

      A.1   R ECORDING PROCESS
652
      In the construction of Mind2Web-Live, the quality and reliability of the data are paramount. To
653
      this end, we have employed an efficient tool(Figure 6) for recording browser operations. The tool
654   precisely captures browser interaction from the users, covering a wide range of activities such as
655   clicks and input actions. The recorded details include the type of operation, execution parameters,
656   target element’s selector path, element content, and its coordinates on the webpage. Moreover, the
657   tool accompany each step with a webpage screenshot, not only facilitating process replication but
658   also providing a visual reference for workflow validation and review(Figure 18, 19). This approach
659   enables us to comprehensively record all the steps required to complete specific tasks, forming the
660   foundation of Mind2Web-Live. Upon completion of the data recording, we meticulously annotated
661   the key nodes of each process along with their corresponding Evaluation Functions.
662
663   A.2   A NNOTATION PROCESS
664
665   In our study, the annotation process plays a pivotal role in ensuring data quality and task validity. To
666   ensure the accuracy and consistency of data annotations, we assembled an annotation team comprised
667   of several authors of this paper and five senior undergraduate students majoring in Computer Science.
668   Not only do the members of the annotation team possess a solid background in Computer Science, but
669   they also received specialized training to ensure consistency in their understanding and identification
670   abilities in annotating key nodes. Prior to beginning the formal annotation process, all annotators
671
      were rigorously trained over a period of two weeks, which included trial annotations that were
      subsequently not included in the final dataset.
672
673   During the annotation phase, we employed a comprehensive reward mechanism. Each annotator was
674   compensated based on the number of tasks they completed, with additional bonuses awarded for
675   high-quality annotations to encourage precise and consistent results. This combined reward system
676   not only bolstered work enthusiasm but also enhanced the overall quality of the annotation work,
677
      laying a solid foundation for the construction of an efficient web agent benchmark.
678   To guarantee the quality of annotations, we instituted a variety of strategies. Each task was annotated
679   independently by one annotator, followed by individual reviews by two other members to verify the
680   accuracy of the key nodes. Throughout the annotation process, we regularly organized discussion
681   sessions for the annotation team to share their experiences and challenges encountered, thereby
682   improving the overall efficiency and quality of the team’s annotations.

      Human Alignment Score The Human Alignment Score(HAS) assesses how well an agent’s
813   workflow aligns with human behavior. It’s crucial for agents not just to be efficient, but to operate
814   in ways that resemble human actions. The evaluation of this aspect is conducted by contrasting
815   the agent’s task completion signal with the ground truth annotations provided by humans, to gauge
816   the level of consistency. An agent that accurately issues a completion signal upon task completion
817   is deemed to exhibit a high degree of alignment with human behavior, thus earning a full score of
818   one point. Conversely, a delay in issuing the completion signal upon task completion results in a
819   deduction of 0.05 points from the full score as a penalty for decision latency. In instances where
820   an agent stops its operation before accomplishing all the task objectives, the score is determined by
821
      the ratio of the step score attained to the maximum step score achievable for that task. Furthermore,

      E.1   AGENT F RAMEWORK
834
835   Planning Integrates past action history, current observations, and task instruction to plan future
836   actions and determine operational values based on the ReAct (Yao et al., 2023) reasoning framework.
837   It can be formally expressed as: Planning(h1:t , ot , i) −→ (zt , at ), where h1:t represents history
838   information until time t, ot is the observation at time t, i is the task instruction, while the outputs zt
839   and at are the thought and action at time t respectively.
840
      Observation Processes the current webpage’s source code and screenshots, producing an accessi-
841
      bility tree (Zhou et al., 2023) and visual observations as ot . In our planning model, we solely focus
842   on textual observations, as visual images involve various grounding mechanisms which could detract
843   from the main focus of our paper. We plan to address this aspect in future research.
844
845   Memory Responsible for storing the task instruction and tracking the agent’s operational his-
846
      tory, including thoughts and actions history across states. It can be formally expressed as
      h1:t = (z1:t , a1:t , r1:t ) within the framework, where r1:t denotes the history of reward signal
847
      if presents.
848
849   Reward Utilizes a self-reflection structure (Shinn et al., 2024), providing a series of reward signal,
850   including a verbal reflection and signal on whether the task is completed. This can be formalized as
851   Reward(h1:t , i, ot+1 ) → rt .

897   Accessibility Tree We employ an accessibility tree-based approach to extract the fundamental
898   textual feature representation from the web environment. The accessibility tree serves as an abstract
899   representation of the structure of a web page, detailing the characteristics of each element within
900   the page. However, the accessibility tree contains a significant amount of redundant information,
901   necessitating the use of a stringent set of filtering criteria to select interactive elements. These
902   filtering criteria include the element’s tag, visibility, usability, as well as textual or image content.
903
      Concurrently with the construction of the accessibility tree, we annotate each filtered interactive
      element, providing information such as element ID, tag, and content. For example, ([1] input ‘search’,
904
      etc.). This annotation method facilitates the precise generation of corresponding CSS selector paths
905
      during subsequent LLM prediction and execution phases, thereby accurately locating the required
906   elements.
907
908
      Screenshot We capture screenshots of the current web page to obtain its visual representation and
909
      provide this visual context to visual language models, such as GPT-4V. This input method mimics
910   human visual perception, allowing the model to gather the most comprehensive information from the
911   web page. Compared to relying solely on the accessibility tree, using screenshots enhances the ability
912   to identify the layout, appearance, and positioning of web elements more effectively. Additionally, it
913   captures interactive elements and other crucial page information that the accessibility tree might miss.
914   To balance inference costs and recognition effectiveness, the original resolution of the screenshots is
915   set to 1080 × 720, though users can define the screenshot resolution according to their specific needs
916   in practical applications.

       G     Q UALITATIVE A NALYSIS OF E XPERIMENTS
1189
1190
       In this section, we conducted a qualitative analysis of error cases in our experimental results. Typical
1191   errors include: local optima, premature termination of tasks, and information loss during inference.
1192
1193
       G.1    L OCAL O PTIMA
1194
1195   In our online environment experiments, a task may involve multiple constraints or requirements.
1196   Web pages often contain numerous clickable links, and frequently feature interactable elements
1197   with similar or even identical names. Due to a lack of prior knowledge about the web domain
1198   associated with current task and confusion caused by similar elements, the planning module’s local
1199   decision-making for the current web state is not always accurate. Moreover, our web agent lacks
1200   proactive thinking to revert to an intermediate state within a limited number of steps, thus stuck in a
1201
       local optima of the task. This is one of the main reasons for the low task success rate. As shown in
       the first line in Table 14, in the task “Check the rating and user reviews for the game ‘Deathloop’ on
1202
       IGN”, the web agent ended up at the review article page for ‘Deathloop’ on IGN due to incorrect
1203
       path selection from the Google search results, rather than the expected page for ratings and user
1204   reviews. In other cases, when actions like filling out forms are required, the greedy nature of LLMs
1205   leads them to input more task-relevant information than necessary. This results in a narrower range
1206   of information that can be extracted from the webpage, as shown in the second line in Table 14.
1207   Meanwhile, the limitations of browser automation tools currently prevent the complete restoration of
1208   a web page to its state before action execution. Memory management of web agents also could not
1209   eliminate the effect of past incorrect trajectories. These all highlight the challenges of autonomous
1210   agent reasoning.
1211
1212   G.2    P REMATURE T ERMINATION OF TASKS
1213
       In the experiments, we also discovered that the web agent sometimes only partially completes tasks.
1214
       This typically indicates that web agent sometimes prematurely judges itself as having finished the
1215
       task. The reasons for premature termination are varied. For instance, the agent might hallucinate
1216   during inference (such as simplifying a task of reaching a page and filling out content to just reaching
1217   the page), leading it to self-judge the task as complete after only finishing intermediate steps and not
1218   continuing further. In other instances, it may have the right thought process in earlier steps, but fails
1219   to deliver the correct action input or effectively execute the action on the page, yet in subsequent
1220   steps, it “reads” this thought and mistakenly believes the action has been executed. Lastly, when it is
1221   difficult to continue along the current path, the agent might lower its standards for task completion
1222   and erroneously judge the task as complete, thus terminating the task prematurely. As shown in the
1223   third line in Table 14, in the task “Track the status of a rebate for Monty Lue, house number 4847, zip
1224
       code 10019 in Menards”, the web agent reached the “Track Your Rebate” page but did not continue
       to complete the form, instead prematurely deciding the task was complete and ending the task.
1225
1226
1227
       G.3    I NFORMATION L OSS IN O BSERVATION
1228
       The relationships between web elements are varied and complex. Often, the essential information of
1229   an element is not contained within the element itself but is instead found within its child elements,
1230   parent, or even sibling elements. For instance, a button tag might not always contain useful attributes;
1231   sometimes, they are empty or irrelevant. Based on our understanding of the DOM tree on the web,
1232   we map information from specific elements (like span) to interactive elements such as buttons. Due
1233   to the diversity of these mapping relationships, our framework currently only considers mapping
1234   valuable information from certain special elements to their parent elements, recursively iterating until
1235   an interactive element is identified, as shown in Figure 15. If this recursive search fails to find an
1236   interactive element or reaches the recursion limit, the element is discarded, as illustrated in Figure
1237
       16. Given the complexity of webpage elements, our initial implementations focus predominantly on
       parent-child mapping relationships. Future work will delve deeper into inter-element mappings to
1238
       ensure the accuracy and correctness of element mappings.

       J   L IMITATIONS & F UTURE WORKS
1459
1460
       The unsolved challenges we encountered in online evaluation of web agents include:
1461
1462   1. Network Instability: The variability in network conditions can lead to discrepancies between
1463   the results obtained from online real-time evaluations and those from closed environments. For
1464   instance, issues such as CAPTCHAs, network outages, or inconsistencies across different IPs can
1465
       influence outcomes. However, in other words, WebCanvas allows for the generation of detailed
       execution logs, enabling precise documentation of a web agent’s performance under specific network
1466
       and website conditions. This feature is crucial for understanding real-world agent behavior, including
1467
       potential issues like being blocked or triggering anti-automation mechanisms.
1468
1469   2. Complex Task Pathways: The diversity of potential execution paths for a given task may not
1470   be completely identified by human annotators. This oversight can lead to a misalignment between
1471
       the defined key nodes and the essential components of task completion, inadvertently penalizing
       correct processes. A model-based evaluation approach could mitigate some of these issues, but it
1472
       also introduces dependency on the model’s capabilities, which may result in unstable evaluation
1473
       outcomes.
1474
1475   3. Static Evaluation Functions: The current static nature of our evaluation functions does not
1476   accommodate changes in task instructions based on environmental variables such as time, location,
1477
       or weather conditions. For example, a task might involve booking a flight to Hawaii next month if
       the weather is favorable. Ideally, the evaluation module would dynamically adjust its criteria for
1478
       success based on ongoing feedback and environmental data, necessitating a logic or code-based
1479
       reward system that can respond to these changes.
1480
1481   In conclusion, while we have addressed several key challenges associated with online evaluations,
1482   many unresolved issues persist. These challenges underscore the need for ongoing research and
1483
       community efforts to refine and enhance the evaluation frameworks for autonomous web agents in
       complex, real-world environments. We encourage the community to continue exploring these avenues
1484
       to improve both the reliability and validity of web agent assessments.
