009                                              A BSTRACT
010
011            Transformer-based large language models (LLMs) have demonstrated surprisingly
012            robust performance across a wide range of language-related tasks, including pro-
013            gramming language understanding and generation. In this paper, we take the first
014            steps towards a formal investigation of using transformers as compilers from an
015            expressive power perspective. To this end, we introduce a representative pro-
016            gramming language, Mini-Husky, which encapsulates key features of modern
017            C-like languages. We show that if the input code sequence has a bounded depth
018
               in both the Abstract Syntax Tree (AST) and type inference (reasonable assump-
               tions based on the clean code principle), then the number of parameters required
019
               by transformers depends only on the logarithm of the input sequence length to
020
               handle compilation tasks, such as AST construction, symbol resolution, and type
021            analysis. A significant technical challenge stems from the fact that transform-
022            ers operate at a low level, where each layer processes the input sequence as raw
023            vectors without explicitly associating them with predefined structure or meaning.
024            In contrast, high-level compiler tasks necessitate managing intricate relationships
025            and structured program information. Our primary technical contribution is the
026            development of a domain-specific language, Cybertron, which generates formal
027            proofs of the transformer’s expressive power, scaling to address compiler tasks.
028            We further establish that recurrent neural networks (RNNs) require at least a lin-
029
               ear number of parameters relative to the input sequence, leading to an exponential
               separation between transformers and RNNs. Finally, we empirically validate our
030
               theoretical results by comparing transformers and RNNs on compiler tasks within
031
               Mini-Husky.

034   1   I NTRODUCTION
035
036   Transformers (Vaswani, 2017) have demonstrated remarkable proficiency across various do-
037   mains, achieving near-expert performance in solving International Mathematical Olympiad prob-
038   lems (Google Deepmind, 2024) and excelling in complex reasoning tasks in science, coding, and
039   mathematics (OpenAI, 2024a). They also handle routine coding tasks with high precision and have
040   been integrated into code editors to significantly boost programmers’ productivity (cur, 2024; Taelin,
041   2023a). Despite these advancements, the full extent of their underlying capabilities remains only
042   partially understood.
043   In this paper, we aim to deepen our understanding of transformers’ abilities to perform compilation
044   tasks. Empirically, transformer-based LLMs have shown rapid progress in code generation and com-
045   pilation. For example, MetaLL (Cummins et al., 2024) enables LLMs to optimize code by interpret-
046   ing compiler intermediate representations (IRs), assembly language, and optimization techniques.
047   Gu (2023) highlights the ability of LLMs to generate high-quality test cases for Golang compil-
048   ers. Surprisingly, Taelin (2023b) demonstrates that models like Sonnet-3.5 can compile legacy code
049   into modern languages like TypeScript, outperforming the now obsolete AgdaJS compiler (Agda
050
      Development Team, 2024).
051   To formally study this problem in a controlled setup, we designed a C-like programming language
052   called mini-husky, which encapsulates key features of modern C-like languages such as (Flanagan,
053   2011) and Rust (Klabnik & Nichols, 2023). We focus on three representative compilation tasks: ab-
      stract syntax tree (AST) construction, symbol resolution, and type analysis. The AST is a recursive

054
      structure that represents the input as a tree. From the perspective of programming language design,
055   the AST is considered the true representation of the input, with the textual code serving merely as
056   a convenient interface for human users (Alfred et al., 2007). All syntactic and semantic processing
057   can then be interpreted as specific operations on these trees. Symbol resolution involves verifying
058   the validity of references to entities and flagging errors for undefined symbols. Type analysis en-
059   compasses both type inference, which assigns types to variables without explicit annotations, and
060   type checking, which identifies mismatches between actual and expected types.
061   We demonstrate that under the clean code principle (Martin, 2008), transformers can efficiently
062   perform AST construction, symbol resolution, and type analysis, where efficiency means that these
063   tasks can be conducted by transformers with a number of parameters that scale logarithmically with
064   the input code length. To the best of our knowledge, this is the first theoretical demonstration that
065   transformers can function as compilers in a parameter-efficient manner.
066
      We further compare transformers and recurrent neural networks (RNNs). By connecting the type
067   analysis task with the associative recall, we show even under the clean code principle (Martin,
068   2008), RNNs require a memory size that scales linearly with the input sequence length to success-
069   fully perform type analysis. Consequently, for type analysis in compilation, transformers can be
070   exponentially more efficient than RNNs. We also empirically validate our theoretical findings by
071   demonstrating the superiority of transformers in the type analysis task.
072
      Technical Challenges and Our Technique.
073
074   Proving that transformers can perform compilation tasks presents several challenges:
075
076   • Transformers operate at too low a level. Transformers process sequences of floating-point vec-
077     tors, akin to raw bits in computers, and proving their ability to perform specific tasks is similar
078
        to writing specialized parallel machine code. Previous work (Yao et al., 2021) often resorts to
        graphical illustrations for readability, even for basic tasks.
079
      • Compilers are exceedingly high-level. Compilers are among the most complex programming
080
        endeavors of our time. Compilation involves numerous sophisticated procedures, some of which
081     are undecidable or computationally expensive, such as code optimization (Alfred et al., 2007))
082     and type analysis (Pierce, 2002). For example, type analysis in complex type systems poses
083     significant challenges, often requiring the development of advanced logical frameworks (Dunfield
084     & Krishnaswami, 2019).
085
086   To overcome these challenges, we design a domain-specific language (DSL) called Cybertron to
087   serve as the proof vehicle, i.e., a major part of our proof consists of reasoning about type-correct
088   code in Cybertron that represents a transformer. Without using Cybertron, writing an equivalent
089   natural language proof would be too complex and intractable. Using code to prove propositions is
090   not new to computer science; it is, in fact, the norm in interactive theorem proving (ITP) (Har-
091
      rison et al., 2014). ITP focuses on generating computer-verifiable proofs through a combination
      of human-guided instructions and software automation. For instance, the correctness of the Ke-
092
      pler conjecture (Hales et al., 2017) is verified by the combination of the ITP theorem provers HOL
093
      Light (Harrison, 2009) and Isabelle (Paulson, 1994). To the best of our knowledge, we are the first
094   to apply this approach to understanding neural networks.

      Expressive Power of Transformers. A line of work studies the expressive power of attention-based
118
      models. One direction focuses on the universal approximation power (Yun et al., 2019; Bhattamishra
119
      et al., 2020b;c; Dehghani et al., 2018; Pérez et al., 2021). More recent works present fine-grained
120   characterizations of the expressive power for certain functions in different settings, sometimes with
121   statistical analyses (Edelman et al., 2022; Elhage et al., 2021; Likhosherstov et al., 2021; Akyürek
122   et al., 2022; Zhao et al., 2023; Yao et al., 2021; Anil et al., 2022; Barak et al., 2022; Garg et al., 2022;
123   Von Oswald et al., 2022; Bai et al., 2023; Olsson et al., 2022; Akyürek et al., 2022; Li et al., 2023;
124   Hao et al., 2022; Pérez et al., 2019; Strobl, 2023; Chiang et al., 2023; Wei et al., 2022; Wang et al.,
125   2022; Feng et al., 2023; Li et al., 2024; Reddit User, 2013). There are also characterizations of trans-
126   formers to be as powerful as universal computers if put in a looped context (Giannou et al., 2023).
127   The most related one is Yao et al. (2021) where the authors prove constructively that bounded depth
128
      Dyck language can be recognized by encoder-only hard attention transformers, which has similari-
      ties to our settings of bounded depth programming language recognized encoder-only hard attention
129
      transformers. The major difference is that we introduce concepts and tasks from programming lan-
130
      guage theory Pierce (2002) to study the semantic powers of transformers.
131
132   Transformers vs. RNN. It is important to understand the comparative advantages and disadvantages
133   of transformers against RNNs. Empirically, synthetic experiments have shown an advantage of
134
      transformers against RNNs for long range tasks (Bhattamishra et al., 2023; Arora et al., 2023).
      Theoretically, there has been a rich line of work focusing on comparing transformers and RNNs in
135
      terms of recognizing formal languages (Bhattamishra et al., 2020a; Hahn, 2019; Merrill et al., 2021),
136
      which show that the lack of recursive structure of transformers prevent them from recognizing some
137   formal languages that RNNs can recognize. However, the gap can be mitigated when we consider
138   the bounded length of input or bounded grammar depth (Liu et al., 2022; Yao et al., 2021), which
139   is quite reasonable in practice and is used in this paper. On the other side, prior work (Jelassi et al.,
140   2024; Wen et al., 2024) proves a representation gap between RNNs and Transformers in repeating
141   a long sequence. In summary, it is somehow intuitive that recursive structures with limited memory
142   perform badly at tasks which requires information retrieval. Our paper shows that semantic analysis
143   for programming languages is such a task.
144   DSLs for Transformers. We note that we are not exactly the first one to employ a domain-specific
145   language to understand the expressive powers of transformers. Previously, DSLs with simple typings
146   like RASP (Weiss et al., 2021) were proposed to prove constructively that transformers can do
147   various basic sequence-to-sequence operations. Lindner et al. (2023) writes a compiler that compiles
148   RASP into actual transformers, Friedman et al. (2023) shows that RASP can be learned, and Zhou
149   et al. (2023) uses RASP to prove that simple transformers can perform certain algorithms. The major
150   difference between RASP and our DSL Cybertron is that Cybertron has a powerful algebraic type
151
      system that helps prove complicated operations beyond simple algorithms.

240   Abstract Syntax Tree Construction. Abstract Syntax Tree (AST) is a hierarchical, tree-like rep-

242   text of the code, the AST abstracts away surface syntax details, capturing the essential elements and
243   their relationships in a structured form. Each node in the AST corresponds to a construct occurring
244
      in the source code, such as expressions, statements, or declarations. This representation is central
      to various stages of language processing, enabling efficient syntax checking, semantic analysis, and
245

      5.1     A BSTRACT S YNTAX T REE C ONSTRUCTION
317
318   We start with a definition that characterizes low-complexity code.
319   Definition 1 (code with Bounded AST-Depth). Let MiniHuskyD be the set of token sequences that
320   can be parsed into valid ASTs in Mini-Husky with a depth less than D.
321
322   D in the above definition is small in practice, and a linear dependency on D is acceptable, but the
323   linear dependency on the length of the token sequence L is not. The fundamental reason is that
      the clean code principle (Martin, 2008) requires one to write code with as little nested layer as


                                                        6
      Under review as a conference paper at ICLR 2025


324
      possible for greater readility. Readability is of the utmost importance because “Programs are meant
325   to be read by humans and only incidentally for computers to execute” (Abelson et al., 1996). This
326   assumption of bounded hierarchical depth is not limited to just programming languages, but is often
327   seen as applicable to natural languages (Frank et al., 2012; Brennan & Hale, 2019; Ding et al., 2017),
328   motivating Yao et al. (2021) to have a similar boundedness assumption. Below is the main result for
329   AST construction using transformers.
330   Theorem 1. There exists a transformer encoder of model dimension and number of layers being
331   O(log L + D) and number of heads being O(1) that represents a function that maps any token
332   sequence of length L in MiniHuskyD to its abstract syntax tree represented as a sequence.
333
334   We note log L is small because 64-bit computers can only process context length at most 264 and
335   D is small by assumption. Therefore, there exists a transformer with an almost constant number of
336   parameters that is able to process comparatively much longer context length.
337
338   Proof Sketch. The idea is to construct ASTs in a bottom-up manner with full parallelism. We shall
339   recursively produce the final ASTs in at most D steps. We shall maintain two values, called pre_asts
340   and asts . asts represents ASTs that have already been allocated, although they might not have been
341   fully initialized. pre_asts represents tokens that have yet to form ASTs and new ASTs that have not
342   been fully initialized. For each round, we try to create new ASTs from pre_asts and update asts
343   and pre_asts . For the n-th round, we provably allocated all ASTs with a depth no more than n. Then
344
      for the D-th round, all ASTs are properly constructed and allocated. Each round can be represented
345   by a transformer of O(1) number of heads, model dimension O(log L + D), and O(1) number of
346   layers. Therefore, the end-to-end process is then representable by a transformer of O(1) number
347   of heads, model dimension O(log L + D), and O(log L + D) number of layers. See full details in
348   Appendix F.
349
350   5.2   S YMBOL R ESOLUTION
351
352   Next, we show that transformers can effectively perform symbolic resolution as log L and D are
353   almost constant as compared with context length L.
354   Theorem 2. There exists a transformer encoder of model dimension and number of layers being
355   O(log L + D) and number of heads being O(1) that represents a function that maps any token
356   sequence of length L in MiniHuskyD to its symbol resolution represented as a sequence of values
357   of type Option<SymbolResolution> .
358
359   Proof Sketch. First, we need to define the type for scopes. It is represented by a tiny sequence of
360   indices of curly brace block AST that enclose the type/function/variable. We assign the scope by
361   walking through the ASTs in a top-down manner. We not only assign scopes to item definitions,
362   we also: (1) assign scopes to ASTs representing curly brace blocks, with these scopes equal to the
363   scope of block itself, and (2) assign scopes to identifiers waiting to be resolved, with these scopes
364   equal to the maximum possible scope of its resolved definition. The computation process is easily
365   represented in Cybertron, indicating attention is expressive enough for this calculation and it only
366   takes O(D) number of layers.
367   After obtaining all the scopes for all items, it takes only one additional layer to obtain the symbolic
368   resolution through attention. As attention is expressed through the dot product of two linear projec-
369   tions Q and K, we have to choose the representation of the scope type properly to finish the proof.
370   The full details are in Appendix G.

      In practice, H is significantly smaller than the context length L for reasonably written code because
379   it is upper bounded by the number of statements in a function body which is required to be small
380   according to the clean code principle (Martin, 2008). Below, we present the main result of using
381   transformers for type analysis. See full details in Appendix H.
382
      Theorem 3. For L, D, H ∈ N, there exists a transformer encoder of model dimension, and number
383
      of layers being O(log L + D + H) and number of heads being O(1) that represents a function that
384   maps any token sequence of length L in MiniHuskyAnnotatedD,H to its type errors represented as
385
      a sequence of values of type Option<TypeError> .

388   5.4   P ROOF V EHICLE : C YBERTRON , A D OMAIN -S PECIFIC L ANGUAGE
389
      Here we highlight our main proof technique. Proving that transformers can express complex algo-
390
      rithms and software like compilers is a significant challenge due to the inherent differences between
391
      how transformers operate and the nature of high-level tasks they are expected to perform. Trans-
392   formers process input at a low level, where each layer manipulates raw token sequences as vectors
393   without predefined structure or meaning. However, high-level tasks—such as constructing ASTs
394   and performing type and symbol analysis—require handling complex, structured information that
395   depends on long-range relationships and interactions across the input. Bridging the gap between
396   this raw, unstructured processing and the structured, multi-step logic required for these tasks in-
397   troduces significant difficulty. Compilers, for instance, typically rely on rule-based, step-by-step
398   operations that are abstract and sequential, which transformers must simulate through their attention
399   mechanisms and feedforward layers. The challenge is further compounded by the need to formally
400
      prove that transformers can handle such tasks efficiently and accurately, despite operating in a fun-
      damentally different manner. To address these challenges, we propose a domain-specific language
401
      (DSL) called Cybertron, which allows us to systematically prove that transformers are capable of
402
      expressing complex algorithms while maintaining sufficient readability.
403
404   A key feature of Cybertron is its expressive type system, which provides strong correctness guar-
405   antees. The type system ensures that every value is strongly typed, making it easier to reason about
406
      function composition and ensuring the validity of our proofs. This type system is crucial for man-
      aging how transformers represent and manipulate both local and global types—where local types
407
      correspond to individual tokens and global types refer to sequences of tokens, encapsulating broader
408
      program information.
409
410   What transformers output (possibly in the intermediate layers) is a representation in sequences of
411   vector of sequences of values in these types. As types are mathematically interpreted in this paper
412
      as a discrete subset of a vector space, Cybertron allows us to construct transformers with automatic
      value validity guarantees if the Cybertron code is type-correct.
413
414   In Cybertron, complex functions are broken down into “atomic” operations through propositions
415   on function compositions and computation graphs (Propositions 11,13,14,2). It is straightforward to
416   prove that these “atomic” operations are representable by transformers, either by feedforward layers
417   or attention layers. For example:
418   • Feedforward layers: boolean operations like AND (Proposition 6), OR (Proposition 7), or NOT
419     (Proposition 5), or operations over option types like Option::or (Proposition 9) being applied to
420     each token in a sequence.
421   • Attention layers: operations that require information transmission between tokens such as
422      nearest_left and nearest_right that collect for each token the nearest left/right non-nil informa-
423     tion (Proposition 15).
424
425   This approach allows us to break down complex operations into primitive tasks that transformers
426   can simulate. Feedforward layers handle local operations on individual tokens, while attention lay-
427   ers manage long-range dependencies and interactions between tokens, simulating the multi-step
428
      reasoning required for higher-level tasks.
429   Cybertron’s expressive type system and function composition framework help bridge the gap be-
430   tween the low-level processing transformers perform and the high-level reasoning necessary for
431   complex tasks like compilation. For full details, including the mathematical foundations of Cy-
      bertron’s type system and function composition, see Appendix D.

433
434
      Now we compare transformers and RNNs from both theoretical and empirical perspectives.
435
436
437   6.1   A L OWER B OUND FOR RNN S FOR T YPE C HECKING
438
      Previously, it has shown that RNN is provably less parameter efficient than transformers for associa-
439
      tive recall (Wen et al., 2024). Intuitively speaking, the type checking step covers associative recall.
440   Based on this observation, we obtain the following lower bound for RNNs.
441
442
      Theorem 4. For L, D, H ∈ N, for any RNN that represents a function that maps any token sequence
      of length L in MiniHuskyAnnotatedD,H with D, H = O(1) to its type errors represented as a
443
444   sequence of values of type Option<TypeError> , then its state space size is at least Ω(L).
445
446   Theorem 3 and Theorem 4 give a clear separation between transformers and RNNs in terms of the
447
      compilation capability. Specifically, if the input codes satisfy D, H ≪ L, which is typically the case
      under the clean code principle (Martin, 2008), then transformers at most need O ((log L + D + H))
448
      number of parameters, which is significantly smaller what RNNs requires, Ω(L).

512   We demonstrated that transformers can efficiently handle a number of syntactic and semantic analy-

      these theoretical insights, showing strong performance on synthetic and real datasets, underscoring

      Trees are one of the most fundamental objects to study in computer science. However, its exact
813   definition differs for different domains. The trees used in “abstract syntax tree” (Section B) is more
814   restrictive than that in mathematics, which we call “typed tree”, so that one can define recursive
815   computation more rigorously.
816
817   A.1   W HAT ARE T REES
818
819   Trees in data structures have slightly additional meaning as compared to trees in mathematics. In
820   this paper, all trees are trees in data structures. For clarity, we lay down the precise definition of
821   trees in data structure.
822   Definition 3 (Tree). A tree T is a set of nodes storing elements such that the nodes have a parent-
823   child relationship that satisfies the following:
824
825          • If T is not empty, it has a special node called the root that has no parent.
826
             • Each node v of T other than the root has a unique parent node w; each node with parent
827
               w is a child of w.
828
829   We denote the nodes of T as N(T ).
830
      Definition 4 (Recursive Definition of a Tree). A tree T is either empty or consists of a node r (the
831
      root) and a possibly empty set of trees whose roots are the children of r.
832
833   However, the above definition is too permissive. We shall define a typed version as follows:
834
      Definition 5 (Typed Tree). A tree type consists of a set of values V and a set of relationships
835
      C ⊆ V × N, and a typed tree under this type is any tree T such that for each node, a value v ∈ V
836   is assigned such that (v, n) ∈ C where n is the number of the children of the node.
837
838   All trees in this paper are typed.
839
      Example 1 (Abstract syntax tree (AST) as Typed Tree). Consider an AST for a simple arithmetic
840   expression. Let the set of values V be:
841
842                                    V = { num , add , sub , mul , div }
843
      and the set of relationships C ⊆ V × N specify the allowed number of children for each value:
844
845                        C = {( num , 0), ( add , 2), ( sub , 2), ( mul , 2), ( div , 2)}
846
847   An example AST for the arithmetic expression (3 + 5) × 2 is the following typed tree:
848
849          • The root node is labeled mul (multiplication), and it has two children.
850
851               – The left child is labeled add (addition), and it has two children:
852
                     * The left child of add is labeled num with the value 3.
853
854
                     * The right child of add is labeled num with the value 5.
855               – The right child of mul is labeled num with the value 2.
856
857
      This tree conforms to the typing rules because:
858
             • num has 0 children,
859
860          • add has 2 children,
861
862          • mul has 2 children,
863
      all of which satisfy the relationships in C.

      A.2   R EPRESENTATIONS OF T REES
865
866   It’s also important to talk about tree representations. We are studying transformers, and then it’s
867   necessary to represent large trees as a sequence, otherwise the model dimension is not large enough
868   to contain the information locally. Let’s first talk about the classical arena pattern used in sys-
869   tem programming for representing trees and we shall slightly adapt it to our use case for studying
870   transformers.
871
872   Arena Pattern. To represent trees efficiently in memory, especially when trees are frequently
873
      modified (such as insertions or deletions of nodes), an arena pattern is often used. The arena pattern
      provides a way to manage memory allocation for tree structures, allowing for efficient memory usage
874
      and avoiding fragmentation. Here’s how the arena pattern works in the context of tree representation:
875
876   Definition 6 (Arena Pattern in Tree Representation). In the arena pattern, a tree is represented by
877
      an array (or vector) of nodes, called an arena. Each node in the arena contains:
878
             • An element or value stored in the node.
879
880          • References (often indices or pointers) to the node’s children and possibly to its parent.
881
882   The key characteristics of the arena pattern are:
883
884          • Memory Contiguity: All nodes are stored contiguously in memory within the arena, which
885            allows for efficient traversal and modification operations.
886          • Fixed Capacity: The arena has a fixed or dynamically resizable capacity, and nodes are
887            added sequentially. This avoids the overhead of allocating individual nodes on the heap.
888
889          • Index-based References: Instead of using pointers, the nodes reference each other using
890            indices within the array, which simplifies memory management and can lead to cache-
891
               friendly operations.
892          • Efficient Allocation and Deallocation: Nodes can be efficiently allocated and deallocated
893            within the arena without requiring complex memory management techniques like garbage
894            collection or reference counting.
895
896   The arena pattern is particularly useful in scenarios where the structure of the tree is highly dynamic
897   or when performance is critical. It allows for a simple and efficient way to manage and traverse trees
898   without the typical overhead associated with more traditional pointer-based tree representations.
899
900
      Adaptations for Transformers For transformers, inputs, intermediate values and outputs are all
      sequences. So the trees are represented as sequences of nodes with node reference representable by
901
      token position encoding. Based on the representation, transformers will be able to perform various
902
      kinds of recursive tree operations, as we shall see.

905   B     C ONTEXT F REE G RAMMAR
906
907   In this section, we lay down the well-known definitions of context free grammar, derivations, and
908   parse trees. To define an abstract syntax tree (AST), one commonly resorts to generation rules, such
909
      as context-free grammars (CFG) (Alfred et al., 2007) and parsing expression grammars (PEG) (Ford,
      2004). In most cases, just generation rules themselves are not sufficient to define properly a lan-
910
      guage. Many practical languages like C and C++ cannot be solely described by these rules (David,
911
      2009) so that they can reuse the limited set of special characters on the keyboard. Furthermore, se-
912   mantic constraints like type correctness are intrinsically contextual and cannot be expressed through
913   CFG or similar rules. However, CFG or other rules provide a valuable construct, the AST. With
914   an AST, one can refine the language definition by putting restrictions on the syntax tree through
915   tree operations. Effectively, a language can be seen as a subset of trees, not as a subset of strings.
916   Semantic analysis like symbol resolution and type checking can be described effectively based on
917   trees. In short, CFG standalone is hardly practical but it provides a useful and clear foundation to
      build definitions upon.


                                                          17
      Under review as a conference paper at ICLR 2025


918
      A context-free grammar (CFG) is defined as a 4-tuple G = (V, Σ, R, S), where:
919
920           • V is a finite set of variables (non-terminal symbols).
921
              • Σ is a finite set of terminal symbols, disjoint from V . Sequences of Σ, i.e., elements of Σ∗
922
                are called (literal) strings.
923
924
              • R ⊂ V × (V ∪ Σ)∗ is a finite set of production rules, where each rule is of the form A → α,
                with A ∈ V and α ∈ (V ∪ Σ)∗ .
925
926           • S ∈ V is the start symbol.
927
928
      Given a context-free grammar G = (V, Σ, R, S), we define derivation as follows:
929
              • A derivation is a sequence of steps where, starting from the start symbol S, each step
930
                replaces a non-terminal with the right-hand side of a production rule.
931
932
              • Formally, we write u ⇒ v if u = αAβ and v = αγβ for some production A → γ in R,
                where α, β ∈ (V ∪ Σ)∗ and A ∈ V .
933
934           • A leftmost derivation is a derivation in which, at each step, the leftmost non-terminal is
935             replaced.
936           • A rightmost derivation is a derivation in which, at each step, the rightmost non-terminal
937             is replaced.
938           • We denote a derivation sequence as S ⇒∗ w, where w ∈ Σ∗ is a string derived from S in
939             zero or more steps.
940
941   A parse tree (or syntax tree) for a context-free grammar G = (V, Σ, R, S) is a tree that satisfies
942   the following conditions:
943
944
              • The root of the tree is labeled with the start symbol S.
945           • Each leaf of the tree is labeled with a terminal symbol from Σ or the empty string ϵ.
946           • Each internal node of the tree is labeled with a non-terminal symbol from V .
947           • If an internal node is labeled with a non-terminal A and has children labeled with
948             X1 , X2 , . . . , Xn , then there is a production rule A → X1 X2 . . . Xn in R.
949
              • The yield of the parse tree, which is the concatenation of the labels of the leaves (in left-to-
950
                right order), forms a string in Σ∗ that is derived from the start symbol S.

1150   D.1   I NTRODUCTION
1151
1152   It’s often difficult to directly prove that transformers or in general other low level forms of computa-
1153   tion can express complicated algorithms and even complex software. There are way too many details
1154   as compared with typical mathematical proofs in machine learning theory. Hence, we propose the
1155   domain specific language Cybertron, where we can systematically prove transformers can express
1156   complicated algorithms and complex software with sufficient readability.
1157   (Note: Cybertron is fundamentally different from Mini-Husky! Mini-Husky is the target language
1158   that we want transformers to analyze yet Cybertron is the domain specific language we use to prove
1159   that transformers can do that.)

1163   automatic type checking. Thus, using Cybertron one can argue operations more complicated than

       D.2   P HILOSOPHY: S EQUENTIAL R EPRESENTATION OF E VERYTHING

1190   Before going through the full details, let’s first talk about the fundamental philosophy behind trans-
1191   former and Cybertron.
1192
       One of the fundamental reasons transformers can be easily adapted across multiple modalities, in-
1193
       cluding NLP and CV, is their sequence-to-sequence operation. Everything can be represented as an
1194   arbitrary-length sequence. Texts are sequences of words, images are sequences of image patches,
1195   videos are sequences of spacetime patches (OpenAI, 2024b), and even graphs with sparse spatial
1196   structures can be represented as sequences of indexed nodes with additional information like parent
1197   node indices. Since inputs of various modalities can be cast into vector sequences, transformers can
1198   be applied to different domains without modifications to their architecture (Dosovitskiy et al., 2020).
1199
       Interestingly, this sequence-based thinking is not new. We’ve actually been representing every-
1200   thing as sequences since the very early days of computer science. This has been the foundation
1201   of how data is stored and processed in computers. However, sequence representations were tradi-
1202   tionally viewed as low-level and sometimes inefficient for practical use, prompting the development
1203   of higher-level abstractions for programming. The rise of transformers, with their scalable learning
1204   capabilities, encourages us to reconsider the significance of sequence-based representations.
1205
       From a systems perspective, viewing everything as a sequence is the foundational approach in com-
1206   puter science. Data in a computer is stored as a continuous stream of bits. Whether it’s text, images,
1207   videos, or graphs, this data is represented in computer memory as an ordered sequence of bits. This
1208   aligns with how transformers handle different types of input by transforming them into sequences
1209   of vectors. Thus, the sequence-based operation of transformers mirrors the sequence-based repre-
1210   sentation of data in computer memory.
1211
       In essence, if a data structure can be represented in computer memory using N bits, it can
1212   be processed as a sequence of bits of length N . This natural sequence representation in memory
1213   is consistent with how transformers process data, which makes them particularly flexible across
1214   different modalities. For example, recent state-of-the-art approaches Wu et al. (2024) show that
1215   transformers can even be trained directly on raw bits of data, further emphasizing this connection.
1216
       Moreover, this sequence-based viewpoint offers fresh insights when applied to the domain of pro-
1217
       gramming, particularly in areas such as code generation and analysis. With tools like ChatGPT and
1218   Copilot being widely used by developers, the impact of transformers on programming workflows
1219   is growing. Understanding the complexity of algorithms and programs expressed in sequence form
1220   becomes an interesting area of study, as it reveals new possibilities for how we approach computa-
1221   tion.
1222
       In comparison to traditional systems like CPUs and human cognition, transformers are highly paral-
1223
       lel but shallow in their operation. A transformer processes data in a fixed number of layers, while a
1224   CPU executes 109 cycles per second, and humans may take days to process information like reading
1225   a book. Transformers, therefore, represent a fundamentally different computational model that is
1226   worth studying further in the context of sequence-based operations.
1227
       Example 2. Image to Sequence: In computer memory, an image is typically stored as a continuous
1228   block of pixel values, often in row-major order, where each pixel’s value is encoded as bits in a
1229   sequence. When a transformer processes an image, it divides the image into patches (e.g., 16 ×
1230   16 pixels), and each patch is flattened into a vector of pixel values. This creates a sequence of
1231   patches, where each patch corresponds to a vector. The way transformers represent these patches
1232   as a sequence closely aligns with how the image data is sequentially stored in computer memory.
1233   Example 3. Video to Sequence: A video is stored in computer memory as a sequence of frames,
1234   where each frame is essentially an image. In a similar manner to images, these frames are stored as
1235   continuous pixel values. Transformers process videos by dividing the frames into spacetime patches,
1236   where each patch captures a small region of space over a short segment of time. These spacetime
1237   patches are flattened and arranged into a sequence for the transformer to process. The sequential
1238   ordering of these patches matches how video frames and pixel data are stored in computer memory.
1239   Example 4. Graph to Sequence: In computer memory, a graph is typically stored using an adja-
1240   cency list or adjacency matrix, where nodes and their connections (edges) are stored sequentially
1241   in a data structure. Transformers process graphs by representing each node and its features as a
       vector, and then creating a sequence of these vectors. The sequence may also encode additional


                                                         23
       Under review as a conference paper at ICLR 2025


1242
       information, such as the parent-child relationships between nodes. This sequence-based represen-
1243   tation of graphs is consistent with how graph data is stored in memory, where nodes and edges are
1244   arranged in a structured order.
1245
       Example 5. Text to Sequence: Text is naturally stored in computer memory as a sequence of char-
1246   acters or words, where each character is encoded as a sequence of bits (such as ASCII or Unicode
1247   values). When transformers process text, they convert each word into a word embedding, which is a
1248   vector of real numbers. The sequence of word embeddings corresponds to the sequence of charac-
1249   ters or words stored in memory. This natural sequential representation of text in both memory and
1250   transformers ensures efficient handling of linguistic data.
1251   Example 6. AST (Abstract Syntax Tree) to Sequence: In computer memory, an abstract syntax
1252   tree (AST) is typically stored as a tree-like structure, where each node represents a component of
1253   the program (e.g., operators, variables, or statements). However, this tree can be linearized into a
1254   sequence by traversing it in a specific order (e.g., pre-order traversal). When transformers process
1255   an AST, they convert it into a sequence of tokens, where each token corresponds to a node in the tree.
1256   This sequential representation of the tree in transformers mirrors how the tree is stored as nodes and
1257
       edges in memory, and how it can be flattened into a linear sequence.
1258   In conclusion, the sequence-based representation in transformers is not just a novel approach for
1259   deep learning but is deeply rooted in how data has been stored and processed in computer memory
1260   since the early days of computing. This consistency between how data is stored in memory and how
1261   transformers process data as sequences is a key factor in their adaptability across different domains.

1766
       Having laid the necessary mathematical foundation behind Cybertron, we now turn to explaining
1767
       its surface—its syntax and semantics. Cybertron serves as a syntax sugar for expressing local
1768   and global computation graphs, which are the vehicles used to demonstrate the expressive power of
1769   transformers. In Cybertron, computations are divided into two layers: the local world and the global
1770   world. These layers play distinct but complementary roles in constructing computation graphs.
1771
1772   D.7.1     L OCAL W ORLD
1773
1774   The local world in Cybertron corresponds to the feed-forward layers of a transformer, focusing on
1775   computations over local types. Local types represent individual tokens or data points, and compu-
1776   tations in this world handle operations on tokens independently of their surrounding context.
1777
1778   Data Types. Local types in Cybertron include basic types such as Bool , Idx , Pos , Fin<n> ,
1779    BoundedVec<T, N> , etc. These types are essential for building local computation graphs that operate
1780   over individual tokens. Compound types, like structs and enums, can also be defined for more
1781   complex token representations. These types serve as the building blocks for the local computation
       graphs that transform data at the token level.

2246

2247

2251   The reduction process is carried out in multiple stages, each focusing on different syntactic con-
2252   structs:
2253

2254
2255

2256             2. reduce_by_delimited : This step reduces constructs that are delimited, such as those involv-
2257                ing parentheses, braces, or other grouping symbols. It ensures that delimited blocks are
2258                properly nested and combined in the AST.

2260                tifying and structuring calls within the AST, ensuring correct representation of function

2263

2264
