                                                   A BSTRACT
010
011
               As LLM-based agents exhibit exceptional capabilities in addressing complex
012
               problems, there is a growing focus on developing coding agents to tackle in-
013
               creasingly sophisticated tasks. Despite their promising performance, these coding
014            agents often produce programs or modifications that contain runtime errors, which
015            can cause code failures and are difficult for static analysis tools to detect. Enhanc-
016            ing the ability of coding agents to statically identify such errors could significantly
017            improve their overall performance. In this work, we introduce Execution-free
018            Runtime Error Detection for COding Agents (REDO), a method that integrates
019            LLMs with static analysis tools to detect runtime errors for coding agents, without
020            code execution. Additionally, we propose a benchmark task, SWE-Bench-Error-
021            Detection (SWEDE), based on SWE-Bench (lite), to evaluate error detection in
022
               repository-level problems with complex external dependencies. Finally, through
               both quantitative and qualitative analyses across various error detection tasks, we
023
               demonstrate that REDO outperforms current state-of-the-art methods by achiev-
024
               ing a 11.0% higher accuracy and 9.1% higher weighted F1 score; and provide
025            insights into the advantages of incorporating LLMs for error detection.

052   While the majority of coding agents across different tasks focus on enhancing comprehension, re-
053   trieval and reasoning capabilities, the systematic detection of runtime errors has received compara-
      tively limited attention. However, ensuring that generated code is free from runtime errors is as criti-

054
      cal as the aforementioned capabilities. For example, an AttributeError can cause the modified code
055   to fail, irrespective of the agent’s comprehension and reasoning processes. Indeed, coding agents are
056   not immune to runtime errors. On SWE-Bench-lite (Jimenez et al., 2024b), the top six coding agents
057   as of August 2024 (CodeStory, Mentatbot, Marscode, Lingma, Droid, AutoCodeRover) produce, on
058   average, 1.8 SyntaxErrors, 25.8 TypeErrors, 5.2 NameErrors, and 11.3 AttributeErrors. Moreover,
059   SWE-Bench (lite) is deliberately curated to include only straightforward problems, suggesting that
060   the incidence of runtime errors could be substantially higher on the full SWE-Bench dataset. Ad-
061   ditionally, detecting these runtime errors enables coding agents to rapidly iterate solutions, thereby
062   reducing both time and cost.
063   Recent coding agents do often incorporate modules for detecting runtime errors. For example, SWE-
064   agent (Yang et al., 2024) and AutoCoderRover (Zhang et al., 2024) employ static analysis tools
065   such as Pylint (Foundation, 2024c), while CodeR (Chen et al., 2024) utilizes dynamic analysis by
066   generating unit tests and executing the modified code. However, these methods have three critical
067   limitations. First, because runtime errors are often triggered by specific inputs that are typically
068   unpredictable from the code alone, static analysis approaches struggle to detect errors such as many
069
      TypeErrors. Second, although dynamic analysis techniques may identify these runtime errors, they
      require execution of the underlying code, which is problematic for three reasons. First, execution
070
      ceases upon encountering the first runtime error, allowing only one error to be detected at a time,
071
      thereby increasing the cost of detection. Secondly, dynamically configuring execution environments
072   for diverse project setups and testing frameworks poses significant challenges and might trigger
073   technical, legal or privacy concerns (Puddu et al., 2022; Lacoste & Lefebvre, 2023). Furthermore,
074   although these mechanisms are commonly employed as separate modules, their performance has
075   never been comprehensively evaluated. This lack of modular evaluation makes it difficult to assess
076   the extent to which these error detection modules enhance the performance of coding agents.
077
      In this work, we introduce an execution-free runtime error detection method, termed REDO, which
078   integrates static analysis tools, such as Pyflakes (Foundation, 2024b) or PyRight (pyright, 2024),
079   with a large language model (LLM). This approach extends the capabilities of static analysis tools
080   to detect a broader range of errors without the need for code execution. The combination of these
081   tools is specifically designed to maximize the advantages of both tools, achieving a balanced trade-
082   off between reliability and the breadth of detectable errors. Similar to other works (Jimenez et al.,
083   2024a; Bieber et al., 2022), we specifically focus on the runtime error detection in Python reposito-
084   ries; but our method could be straightforwardly extended to other languages. Moreover, we present
085   a challenging and practical benchmark, SWE-Bench-Error-Detection (SWEDE), which is the first
086
      repository-level error detection in the presence of complex external dependencies. Beyond SWEDE,
      we perform a suite of experiments encompassing various tasks to further evaluate error detection al-
087
      gorithms. As shown in Figure 1, REDO significantly outperforms previous methods, obtaining
088
      SOTA performance across diverse scenarios; and through qualitative analysis, we provide insights
089   into the underlying mechanisms of REDO.

135   2.2   S TATIC A NALYSIS T OOLS AND RUNTIME E RROR P REDICTION
136
137   Static analysis, a technique for examining computer programs without execution, is particularly
138
      valuable in contexts where executing the program might lead to legal, privacy, or computational
      concerns. Due to its non-executive nature, static analysis has found widespread application in error
139
      detection (Zheng et al., 2006; Dillig et al., 2007; Chow et al., 2024), bug identification (Ayewah
140
      et al., 2008; Mashhadi et al., 2024), and vulnerability discovery Charoenwet et al. (2024); Son-
141   nekalb et al. (2023); Esposito et al. (2024); Chess & McGraw (2004); Livshits & Lam (2005); Evans
142   & Larochelle (2002). In the context of Python programming, various professional static analysis
143   tools have been developed to enhance code quality. For instance, Pylint (Foundation, 2024c) and
144   Pyflakes (Foundation, 2024b) are designed to identify errors, while Bandit PyCQA (2024) focuses
145   on detecting common security vulnerabilities. Additionally, tools such as PyRight (pyright, 2024)
146   and MyPy (Foundation, 2024a) perform type checking, contributing to more robust software devel-
147   opment. Although the integration of LLMs with static analysis is still in its infancy, some recent
148   studies have proposed combining these technologies to enhance bug detection in complex systems,
149
      such as the Linux kernel (Li et al., 2024a), and to identify security vulnerabilities (Li et al., 2024b).
      However, the exploration of LLMs in conjunction with static analysis for runtime error detection
150
      remains limited. Additionally, Bieber et al. (2022) presents a notable effort in this domain by lever-
151
      aging Graph Neural Networks (GNNs) for predicting runtime errors, along with proposing a dataset,
152   referred to as STA, to evaluate their approach’s efficacy.

156
157   In this study, we introduce REDO, which serves to check the safety of modification patches. Here,
158
      “Unsafe” instances are those that might crash due to runtime errors; and “Safe” instances are those
      that can be successfully run. REDO operates through a two-phase process: differential analysis and
159
      LLM-based detection.
160
161   The differential analysis component employs a static analysis tool, which provides a dependable
      method for detecting runtime errors. However, its detection capabilities are generally constrained to

      SyntaxError, AttributeError, and NameError. To address this limitation, the LLM-based detec-
163   tion is incorporated to reason about the input contexts. It extends REDO’s detection capabilities to
164   errors such as TypeError and ValueError, which are typically beyond the scope of static analysis.
165
166
      By integrating these mechanisms, REDO achieves a balanced trade-off between reliability and the
      breadth of error detection. An overview of REDO’s architecture is illustrated in Figure 2.

170   3.1   D IFFERENTIAL ANALYSIS
171
172   Static analysis tools like Pyflakes and PyRight typically ensure detection through reliable methods,
173   such as verifying syntax correctness and maintaining data type consistency, making them essential
174
      for identifying runtime errors in coding agents. In this study, we employ PyRight as our static
      analysis tool since it is fast and lightweight; however, our framework is designed to be flexible,
175
      allowing the integration of any static analysis tool.
176
177   Despite their utility, static analysis tools are affected by two challenges. First, they are prone to
178   generating false positives, where potential vulnerabilities are incorrectly flagged (Kang et al., 2022;
179   Kharkar et al., 2022; Murali et al., 2024). For example, when PyRight is applied to original python
180
      scripts containing the modified functions, which do not contain runtime errors, it falsely classifies an
      average of 267 instances, considering 89% of all testing instances as “Unsafe” across various coding
181
      agents. To mitigate this issue, particularly in code edit tasks, we introduce the concept of differential
182
      analysis. This method involves applying static analysis tools to both the original and modified
183   implementations separately. By comparing the errors detected in the original implementation (SOrig )
184   with those in the modified implementation (SMod ), we can identify any new errors introduced by
185   the modifications. If new errors are detected in the modified implementation, the patch is flagged
186   as “Unsafe”. Differential analysis effectively refines static analysis tools to focus specifically on
187   runtime errors introduced by modifications, thereby filtering out false positives from the original
188   implementation. Notably, this removes almost all positives induced by the original implementation.
189   The second challenge with static analysis tools is their inability to detect errors that are triggered
190   under specific inputs. In dynamically typed languages like Python, variable data types are unknown
191   before execution and also can change in response to different inputs. Since static analysis tools
192   cannot reason about input contexts, they often fail to detect these errors, resulting in low recall in
193   error detection performance. We show a concrete example in Figure 4. To address this challenge,
194   we propose the LLM-based detection described in the next section.
195
196
197   3.2   LLM- BASED D ETECTION
198
199   In comparison to static analysis tools, LLMs possess the ability to comprehend both the problem
200   statement and the modification patch. This capability allows them to reason about potential input
201   contexts and anticipate runtime errors that might be overlooked by static analysis tools. However,
202   the reasoning process of LLMs is not always as reliable as the error detection mechanisms inherent
203   in static analysis tools. To harness the complementary strengths of both approaches, we restrict
204   the application of LLMs to instances deemed ’Safe’ by static analysis tools. Accordingly, we have
205   designed the LLM prompt template to identify potential runtime errors that static analysis tools may
206
      have missed.
207   Specifically, for each modification patch, we provide two additional pieces of information: the prob-
208   lem statement and the Python script containing the original version of the modified functions. The
209   problem statement outlines the input contexts and describes the potential verification process for the
210   modified implementation. The original implementation provides the running context of the modi-
211   fied functions, including the safe utilization of variables and functions. Subsequently, we prompt
212
      the LLM to enumerate potential runtime errors that may arise due to the modification and ultimately
      assess the safety of the patch. A detailed prompt template is provided in Appendix E.2.
213
214   Given the cost and latency associated with LLM calls, the LLM API is restricted to a single invoca-
215   tion per instance in our study. For the LLM, we utilize Claude-SONNET-3-5, with the temperature
      setting fixed at zero. A pseudo-code of whole REDO framework is given in Algorithm 1.

      4     SWE-B ENCH -E RROR -D ETECTION (SWEDE)
217
218   The difficulty of runtime error detection can vary drastically among different coding problems. For
219   instance, on the task proposed by Bieber et al. (2022), only one python script is considered on
220   each data point. This task is practical as it resonates the competitive programming scenario where
221   one python script should contain all functionality; and external dependencies are simple. However,
222   as coding agents become more powerful, additional challenging and practical scenarios should be
223   considered. In this work, we propose an repository-level error detection task, which is based on
224   SWE-Bench (lite) (Jimenez et al., 2024a) and its evaluation results using different coding agents.
225   We name this task as SWE-Bench-Error-Detection or SWEDE.
226   SWE-Bench (lite) is a popular benchmark dataset containing instances of repository-level coding
227   problems. Taking a GitHub repository and a problem statement as inputs, SWE-Bench (lite) asks
228   coding agents to generate a modification patch to resolve the problem. SWEDE extends SWE-Bench
229   (lite) to include generated patches and evaluation logs from SWE-Bench leaderboard (Jimenez et al.,
230   2024b); but with a focus on detecting runtime errors induced by generated patches, without execut-
231   ing the code.
232   The task is challenging for two reasons. First, the modified scripts usually call other python scripts,
233   referred to as external dependencies, within the same repository. For instance, as shown in Table 1,
234   when only one directory level above the location where the modified file resides is considered, there
235   already are many dependencies on average. When all files in the repository are considered, the
236   number of dependencies could become even more intimidating. Furthermore, the unit tests might
237   not directly interact with the modified files. These two factors make SWEDE challenging for error
238
      detection algorithms as running contexts of variables and functions are harder to infer. The task is
      practical because, when coding agents autonomously modify repositories, detecting runtime errors
239
      early offers instrumental information; and can potentially reduce cost and time.

250

327   STA dataset. To assess the general applicability of REDO, we conduct evaluations on another
328   dataset: the balanced test set introduced by Bieber et al. (2022), hereafter referred to as STA. This
329   dataset is derived from a code generation dataset CodeNet (et al., 2021b) and comprises approxi-
330   mately 27,000 Python submissions for competitive programming tasks. The balanced test set was
331   designed to ensure that the number of submissions without errors is approximately equal to those
332   containing runtime errors. The dataset categorizes 26 distinct error types, including the category
333   ”no error.” The detailed enumeration of errors and their corresponding indices are consistent with
334   those listed in the prompt template provided in Appendix E.3. Unlike the SWEDE dataset, each
335
      submission in STA consists of a single Python script that processes input via standard input (stdin)
      and performs its functionality without relying on external dependencies. In the context of STA, error
336
      detection algorithms are tasked with predicting runtime error types based on the Python submission,
337
      with or without the presence of input contexts. These input contexts describe the potential values
338   that stdin may take, providing clues regarding possible runtime errors. We refer to the scenario
339   involving input context as With Context, and the scenario without input context as Without Context.
340
341
      Given that the task in STA differs from that in SWEDE, we adapt REDO using modified prompt
      templates. The templates for both the scenarios with and without input contexts are provided in
342
      Appendix E.3 and Appendix E.4, respectively. The performance of the algorithms is assessed us-
343
      ing three metrics: accuracy (Acc), weighted F1 score (W.F1), and weighted error F1 score (E.F1).
344   Weighted metrics are computed by calculating the F1 score for each class independently and av-
345   eraged using a weight that depends on the number of true instances for each class; the E.F1 is
346   calculated exclusively for data points containing a runtime error. We compare our methods with
347   those reported by Bieber et al. (2022). To further investigate the contributions of individual com-
348   ponents within REDO, we also evaluate PyRight and LLM separately. Since the original dataset
349   samples do not include error-free submissions, we employed three different random seeds to sample
350   an equivalent number of submissions for a fair comparison. The means are reported in Table 3, and
351   the standard deviations are presented in Table 6.

354   of STA. Additionally, we observed discrepancies between the annotated error types and the results
355   obtained from our running results. Consequently, we also present evaluation results, enclosed in
356   brackets, based on error types identified in our runs. Under these conditions, REDO exhibits even
357   more pronounced improvements over the baselines. Second, our ablation PyRight obtains decent
358
      performance in both Without and With Context tasks. Considering that the input to STA submissions
      are usually most common cases, as opposed to corner cases in SWEDE, our result demonstrates that
359
      static analysis tools could perform well if the inputs and dependencies are simple. Furthermore,
360
      when comparing REDO to PyRight, we can also see that REDO obtains extra benefits by including
361   the LLM based tool. This shows the effectiveness of our proposed framework.

      and STA highlight that each excels in distinct tasks. By integrating these two tools, we can
      leverage the strengths of both, rendering our framework more versatile and effective across a broader

      range of tasks.

      How could the detected errors help? This section presents a qualitative example illustrating how
450
      detected errors can facilitate the correction of flawed patches. We introduce a preliminary patch-
451   fixing algorithm that leverages error messages generated by REDO. Specifically, when presented
452   with a modified patch (referred to as the original generated patch) and its associated detected errors,
453   our approach first applies the patch and extracts the modified code chunks, with ten lines of code
454   added before and after each chunk. We then prompt the LLM to refine the code chunks according
455   to the identified errors. Finally, a fixed patch is generated based on the modifications to the code
456   chunks. The detailed LLM prompt is provided in Appendix E.6. For instance, in the case of Django-
457   12308 utilizing ACR, REDO initially identifies a risky attribute call that could potentially lead to an
458   AttributeError. Based on this detection, the patch-fixing algorithm generates an fixed patch that
459
      avoids invoking the risky attribute, thereby preventing the AttributeError present in the original
      patch. A comprehensive description of the algorithm and a detailed analysis of this example are
460
      provided in Section G.

464   6   C ONCLUSION
465
466   In this study, we first present REDO, an innovative error detection framework that operates through
467   a two-step process: differential analysis followed by LLM-based detection. This approach achieves
468   a balanced trade-off between reliability and the scope of detectable errors. We also propose SWE-
469   Bench-Error-Detection (SWEDE), a novel and challenging runtime error detection task that aligns
470   with the increasing deployment of autonomous coding agents responsible for repository-level mod-
471   ifications. Furthermore, we conduct a comprehensive set of quantitative experiments to empirically
472   demonstrate the efficacy of REDO across various tasks. In addition, our qualitative analysis offers
473
      insights into the conditions under which LLM integration proves beneficial or falls short, and how
      detected runtime errors using REDO could help fix the previous flawed patches.

477   7   L IMITATION AND FUTURE WORK
478
479
      The current implementation of the LLM-based detection step constrains the number of LLM API
480   call on each data point to just one. Expanding the number of API calls, potentially by leveraging
481   agentic AI techniques, could significantly improve error detection capabilities. Additionally, the
482   SWEDE task is presently confined to a binary classification of ’Safe’ and ’Unsafe.’ As error de-
483   tection algorithms become more sophisticated, it is crucial to consider expanding the classification
484   schema to encompass a wider spectrum of error types, similar to those addressed in STA. Finally, the
485   advancement of more refined ensemble and patch-fixing algorithms holds the potential to enhance
      the efficacy of REDO in supporting coding agents.
