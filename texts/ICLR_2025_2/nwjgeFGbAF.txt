009                                                A BSTRACT
010
011            Malware classification by using function call graphs (FCG) is an important task in
012            cybersecurity. One big challenge in this direction is the lack of representative, large,
013            and unique FCG datasets. Existing datasets typically contain obsolete Android
014            application packages (APKs), largely consist of small graphs, and include many
015            duplicate FCGs due to repackaging. This results in misleading graph classification
016            performance. In this paper, we propose a new comprehensive dataset, Better
017            Call Graphs (BCG), that contains large and unique FCGs from recent APKs,
018            along with graph-level APK features, with benign and malware samples from
019
               different types and families. We establish the necessity of BCG through the
               evaluation of several baseline approaches on existing datasets. BCG is available at
020
               https://iclr.me/.

      1    I NTRODUCTION
025
026   Malware detection is a key task in the field of cybersecurity. In most malware samples, minor changes
027
      in the source code of the original malware can lead to substantially different compiled code (e.g.,
      through instruction reordering, branch inversion, and register allocation) (Bayer et al., 2006). This is
028
      often exploited to bypass signature-based detection, a common method of malware detection (Scott,
029
      2017). However, these minor source code modifications have little impact on the executable’s control
030   flow, which can be depicted using a function call graph (FCG) where functions are the nodes and
031   the call relations are the edges. Hence FCG-based malware detection has been an important field of
032   study, particularly within the realm of Android (Ye et al., 2017; Freitas et al., 2020).
033
      One challenge in FCG-based malware detection is that achieving accurate and robust classification
034
      models has been hampered by the limited availability of modern and representative large-scale
035
      datasets. Existing datasets typically contain old Android application packages (APKs) along with
036   their corresponding FCGs. Considering the dynamic landscape of Android ecosystem, obsolete
037   APKs offer no benefit at all as they are developed on earlier version of an Android. In addition,
038   the complexity of both benign and malicious applications has drastically changed in recent years.
039   Furthermore, most existing datasets contain many duplicate APKs, packaged with trivial differences,
040   hence have a different name but the same FCG structure.
041
      In this paper, we introduce a new and comprehensive dataset named Better Call Graphs (BCG),
042   comprising extensive and distinct function call graphs (FCGs) extracted from recent APKs. The
043   dataset includes benign samples as well as malware samples spanning various types and families.
044   We establish the necessity of BCG through the evaluation of several baseline approaches on existing
045   datasets. We show that existing datasets often yield misleading scores when state-of-the-art classifiers
046   are applied. Our dataset also contains graph-level APK features, capturing both the structural and
047   behavioral characteristics of malware. BCG is publicly available at https://iclr.me/. The
048   dataset is released under a CC-BY license, enabling free sharing and adaptation for research or
049   development purpose.
050   In the rest of the paper, we first give a background and summarize related works on FCG-based
051   Android malware detection and Android-based FCG datasets in Section 2. Then we provide a detailed
052   description of how the BCG is collected and filtered in Section 3. Next, we explain the properties
053   of BCG in detail (Section 4) and perform graph classification experiments on our dataset, as well
      as other established datasets, using state-of-the-art graph classification methods (Section 5) — we

      perform malware type as well as family classification. Finally, we summarize our work and discuss
055   the limitations in Section 6.

058   2     R ELATED WORK
059
060   Literature is rich with studies on Android malware detection, by using various types of networks such
061   as Function Call Graphs (FCGs), Control Flow Graphs, and Network Flow Graphs. These studies
062   rely on existing malware datasets to evaluate their methods’ effectiveness. Here we delve into prior
063   research on FCG-based detection for Android apps, followed by a discussion of commonly used
064   malware datasets.
065
066   2.1   FCG- BASED A NDROID MALWARE DETECTION
067
068   Android apps, packaged in APK files, bundle all their components — code, resources, and a manifest
069   file. Extracting various features from the code allows researchers to analyze how the app works and
070
      identify potential security risks. FCGs, in particular, have proven valuable for malware classification
      by revealing how the app’s functions interact, potentially exposing malicious behavior. Researchers
071
      have actively explored diverse methods that leverage FCGs to analyze Android apps for security
072
      purposes. These methods typically involve constructing FCGs and enriching them with node features,
073   which can be either basic properties or more complex embeddings learned through graph neural
074   networks. The enriched FCGs are then used for identifying and classifying malware. Classifications
075   are typically performed to identify both the malware type and the malware family. Malware type
076   refers to the broad category of malicious behavior exhibited by a malware program. Common types
077   of malware include Virus, Worm, Trojan Horse, and Ransomeware. Malware family refers to a group
078   of malware programs that share similar characteristics, codebase, or functionality. Malware families
079   are often named by antivirus companies based on their unique features. For example, within the
080   "Trojan Horse" type, there might be the "Emotet" family known for email spam and credential theft.
081   MAMADroid and APIGraph utilized API semantics features to capture the semantic similarities
082   between malware variants and analyze information flow for malware detection (Onwuzurike et al.,
083   2019; Zhang et al., 2020). In contrast, Yuan et al. (2020) focused on byte-level classification
084   by converting malware binaries into Markov images and applying deep learning for detection.
085   Meanwhile, Fan et al. (2018) proposed a family-level classification approach by leveraging frequent
086   subgraphs within FCGs. Zhu et al. (2018) constructed enriched FCGs from Smali code, incorporating
087
      function types (system or third-party API) and permission requirements for each node, and used
      Graph Convolutional Networks (GCNs) to train malware classifiers on these graphs. Similarly, Feng
088
      et al. (2020) focused on extracting features directly from the disassembled code sections in CGdroid.
089
      This approach first uses hand-crafted features like the number of string constants and instructions for
090   each node, and then utilizes a GNN to learn graph embeddings and an MLP for final classification.
091   Vinayaka & Jaidhar (2021) further extended this concept by incorporating FCG’s graph structural
092   attributes (e.g., node degree) and non-graph features extracted from the disassembled functions, such
093   as method attributes and opcode summaries. Yumlembam et al. (2022) took a more general approach,
094   modeling apps as local graphs where nodes denote APIs and co-occurring APIs in the same code
095   block as edges. The authors explored features like centrality measures, permissions, and intents
096   from the manifest file. Lo et al. (2022) leveraged PageRank (Page et al., 1999), in/out degree, and
097   betweenness centrality values as node attributes. DeepCatra utilized call traces, opcode features, and
098
      TF-IDF for critical API identification (Wu et al., 2023).
099   Moving beyond basic features, some approaches explored learning node embeddings using GNNs
100   and NLP techniques (Catal et al., 2021; Gunduz, 2022). Errica et al. (2021) leveraged Contextual
101   Graph Markov Models to learn embeddings based on call graph structure and out-degree features,
102   followed by classification with a neural network. Cai et al. (2021) and Xu et al. (2021) have explored
103   leveraging word embedding techniques to analyze Android apps. Xu et al. (2021) used the Skip-gram
104
      algorithm to transform Android opcodes into vectors for analysis.
105   These approaches highlight the active research in utilizing FCGs for Android app security analysis.
106   However, a crucial limitation of most existing studies is the reliance on datasets that contain
107   obsolete and/or duplicate APKs. This inflates the reported performance of classification approaches.
      Our work addresses this limitation by introducing a new malware classification dataset, BCG, that

      consist of new and unique FCGs. BCG will pave the way for true evaluation the effectiveness of
109   the aforementioned FCG-based methods and open doors for new research directions in the field of
110   Android app security analysis.

123   2.2   A NDROID - BASED FCG DATASETS
124
125   While numerous graph classification datasets exist for various fields like bioinformatics and social
126   networks, options for cybersecurity, specifically malware detection using graph analysis, are scarce.
127   Most existing datasets in this domain are closed-source. Fortunately, a few publicly available options
128   like CICAndMal2017, CICMalDroid, AndroZoo, Drebin, and MalNet provide valuable resources for
129
      researchers analyzing Android malware through graph structures.
130   Given the dynamic nature of the Android ecosystem and the interest of malicious entities in releasing
131   APKs, several FCG datasets have been collected and curated to study malware characteristics. The
132   Drebin dataset is one of the first in this direction, offering 5,560 malware apps (179 families)
133   collected between 2010-2012 by MobileSandbox (Arp et al., 2014), but it lacks benign samples.
134   Drebin provides summaries of each malicious APK using 10 features like permissions and intents.
135
      While the Drebin dataset is valuable for multi-class classification (identifying the top-k most frequent
      malware families), binary classification (malware vs. benign) requires collecting benign samples
136
      from other sources. The AndroZoo dataset contains over 24 million APKs, mostly benign and with
137
      some malware verified through VirusShare (Allix et al., 2016). AndroZoo is constantly updated
138   and provides 10 features per APK which includes sha256, sha1, md5, apk_size, dex_size, dex_date,
139   pkg_name, vercode, vt_detection, and vt_scan_date. It primarily consists of benign APKs, and the
140   fraction of malware APKs is 16.67% (4 million APKs). The Canadian Institute for Cybersecurity
141   (CIC) released two Android app datasets for malware analysis: CICAndMal2017 (Lashkari et al.,
142   2018) and CICMalDroid (Mahdavifar et al., 2020). The former one has 10,854 APKs collected
143   between 2015-2017 and categorized by malware type (benign, adware, ransomware, SMS, and
144   riskware) and includes network traffic features. The latter one boasts a larger collection, 17,341 APKs
145   from 2018, with similar malware classifications. It provides not only static features (permissions,
146
      intents) but also dynamic behavior data (system calls) and network traffic (pcap format). MalNet
      is a more recent dataset of 1.2 million FCGs extracted from AndroZoo APKs collected between
147
      2006-2021 (Freitas et al., 2020). The dataset is categorized into 47 types and 696 families. It also
148
      offers a smaller version with 5000 FCGs, MalNet-Tiny, for efficient experimentation. Another related
149   dataset, MalRadar, includes 4,534 APKs collected from 2014 to 2021 and provides information only
150   at the family level (Wang et al., 2022).
151
152
      While these datasets are valuable, they suffer from two key issues:
153         1. Most FCGs are obtained from obsolete APKs, latest being only from 2021 (most of which
154            are also benign). The complexity of both benign and malicious applications has drastically
155            changed in recent years. For example, in the past it was simple enough to classify malware
156            by determining if phone SIM card details were sent over the network. However, now benign
157            applications do this for two-factor authentication and even for unique tracking of individuals
158
               for leaderboards in games.
            2. The current datasets often contain many duplicate APKs, packaged with trivial differences,
159
               which potentially inflates the classification performance (details in Section 4.3).
160
161   In this work, we address these limitations by constructing BCG, a new dataset with unique and
      recent APKs filtered with respect to various criteria, to enable a more robust testbed for FCG-

183   3    C OLLECTING AND FILTERING FCG S
184
185   To ensure a robust and relevant BCG dataset, we constructed it by comprehensively analyzing both
186   APK files and their corresponding graph properties. This involved filtering and refining APKs based
187   on various quality and relevance criteria, overcoming limitations of existing datasets. There are
188   some high level observations that guided our approach: (1) old APKs are simplistic in their structure
189
      and capabilities, (2) repackaging is very common, (3) certain virus families are over represented in
      datasets (often related to point 2), and (4) small (based on bytecode size and not auxiliary files) APKs
190
      are often uninteresting from a detection standpoint. A detailed flowchart of the BCG construction
191
      process is provided in Figure 1. Here we summarize each step in this process:
192
193   1. Downloading APKs: The foundation of our BCG dataset is built upon acquiring data sources
194   that exhibit specific properties, including larger and more recent APKs (details of these properties
195
      are provided in Section 4). To achieve this, we secured permission from two prominent repositories:
      AndroZoo (Allix et al., 2016) and VirusShare (vir, 2024a). We selected AndroZoo and VirusShare
196
      as representative sources due to their extensive collections and widespread use in the research
197
      community. These are the defacto sources for the mobile, programming languages, and Android
198   security communities. AndroZoo, for example, includes apps from the Google Play Store as well
199   as other app sources such as Appchina and Anzhi (Chinese App Store), encompassing a total of
200   24,751,611 APKs—significantly more than the number of apps currently available on the Google
201   Play Store. This extensive repository covers all apps from these sources without any exclusions. We
202   downloaded over 100,000 APKs from the AndroZoo and VirusShare repository. Due to a limitation
203   within AndroZoo that restricts concurrent downloads to 40 files, this process took around 72 hours to
204   complete. Additionally, storing this vast amount of data required 2 TBs of storage space.
205   2. Filtering APKs by year and size: To ensure our dataset has recent and large files, we followed a
206   two-step filtering process on the 100,000 downloaded APKs. Firstly, we extracted the DEX (Dalvik
207   Executable) year from each APK. DEX year refers to the approximate year the application was
208   compiled and the information is embedded within the APK file. By removing APKs with a DEX year
209   before 2017, we ensured our dataset primarily reflects more recent applications. Secondly, to ensure
210   the APKs contained sufficient information for analysis, we removed any APKs with a size less than
211   4MB. We analyzed the distribution of APK size alongside the number of nodes and edges in their
212
      FCGs. This analysis revealed that APKs exceeding 4MB typically contained enough nodes and edges
      for meaningful FCG-based analysis. Consequently, we filtered out APKs with a size less than 4MB.
213
      These filtering process resulted in a dataset of around 40,000 APKs.
214
215   3. Determining family and type: To create the types and families in BCG, we utilized a multi-step
      process with VirusTotal (vir, 2024b). First, we downloaded detailed reports for each APK using the

      4. Constructing FCGs: Androguard has proven effective in previous datasets for constructing FCGs
251
      (Desnos & Gueguen, 2011). It conducts static analysis of the DEX file within each APK, identifying
252   method names and their interactions. Methods are represented as nodes, while calls between methods
253   are depicted as directed edges. We leverage Androguard to produce FCGs for each APK in our
254   dataset. To facilitate further analysis, we also hash the method names into unique identifiers, enabling
255   the creation of an integer edge list. Our datasets are published in two formats: one containing the
256   original method names as nodes, and another using hashed IDs.
257
      5. Filtering APKs by number of edges: While the MalNet dataset has a large average graph size,
258   it contains many very small graphs (less than 100 nodes/edges). To exclude trivially small graphs
259   (less than 100 nodes/edges, as observed in MalNet), we further filtered our datasets to only include
260   APKs with a minimum of 100 edges in their FCGs, which is large enough to contain complex graph
261   structure. This filtering process resulted in a collection of around 29K graphs from 40K graphs.
262
      6. Removing duplicates: Existing datasets, such as MalNet and CICMaldroid, often contain
263
      duplicate APKs with identical functionalities (reflected in their FCG properties like number of nodes,
264   edges, etc.) but disguised by different names. This can be due to repackaging (malware with minor
265   changes) or the same APK appearing in multiple app stores. To address this, we identified and
266   removed duplicate FCGs based on six key graph properties of the FCGs: number of nodes, number of
267   edges, average degree, in-degree centrality, size of largest connected component, and size of largest
268   weakly connected component. For further confirmation, we verified that these duplicates also shared
269   the same malware type and family information. Our methodology results in no false positives but
      there may well remain false negatives, i.e., undetected duplicates or FCGs that differ by one or few

274   4      P ROPERTIES OF BCG
275
276   While existing datasets like MalNet (Freitas et al., 2020) offer valuable properties for evaluating
277   Android malware with FCGs, they often contain duplicate APKs with different names but identical
278   FCG structures. Most of the previous datasets primarily consists of samples collected before 2017,
279   potentially limiting its generalizability to modern malware, and hence misleads the ongoing research
280   on malware detection. Additionally, existing malware datasets often include numerous smaller-sized
281   APKs, which limits their utility in comprehensive malware analysis. Moreover, these datasets often
282
      lack essential APK properties, such as detailed information on the services or libraries used by the app,
      which impedes a thorough understanding of the app’s behavior and functionality, making it difficult
283
      to accurately classify malware. To address these limitations, we ensured that BCG has four key
284
      features: (1) larger size to facilitate more robust graph classification, (2) recent data (including 2017
285   and after) to reflect evolving threats, (3) unique APKs to ensure a more accurate evaluation testbed,
286   and (4) non-graph APK features (graph attributes) for a more holistic evaluation. The summary of
287   our datasets, BCG, is given in Table 2 and here, we briefly describe each of the four features:
288
289
      4.1         L ARGER IN SIZE
290
291   Existing datasets for analyzing and differentiating APKs through graphs are often too small. While
292   MalNet has the largest dataset size, it contains many graphs with very few nodes/edges. Out of
293   100K APKs in MalNet, approximately 3,000 FCGs contain less than 100 nodes/edges. Analyzing
294   such small graphs offers limited value for malware classification, or broader graph classification,
295   and may even mislead the classifier to prioritize size. We address this limitation by creating a new
296   dataset specifically designed for graph-based analysis. For this reason, we consider only those with a
297   minimum size of 4MB during the APK collection phase. Additionally, we exclude any graphs with
298   fewer than 100 edges. This ensures our dataset consists of larger graphs, providing more meaningful
299
      insights for malware classification as well as type/family classifications within malware.
300

306
      ecosystem has evolved significantly since 2016. Older APKs, built for simpler Android versions,

      malware detection methods. To address these limitations, we focus on constructing a new malware

341     4.3           U NIQUE APK S
342
343     High malware classification performance reported by recent approaches might be misleading due
344     to the presence of duplicate APKs in commonly used datasets (Shirzad et al., 2023; Cao et al.,
345     2023; Rampášek et al., 2022). The duplicate APKs have identical FCG structures but appear under
346     different names. This is caused by repackaging, where the same malware is redistributed with minor
347     modifications like altered app icons or backgrounds, or by the same APK being uploaded to multiple
348
        app stores. For instance, Malnet-Tiny and CICMalDroid suffer from this significantly. Malnet-Tiny
        contains around 2,000 repackaged APKs (approximately 40% of the entire data) while CICMalDroid
349
        has 41% of duplicate APKs. We have also investigated a subset of the MalNet, 100K out of 1.2M,
350
        and observed that approximately 51% of the APKs are duplicates. Importantly, removing these
351     duplicates often can lead to a significant drop in the classification performance as the duplicate FCGs
352     cause label leakage, or even database lookups, in the original data when both train and test splits
353     contain the same FCG. We conducted experiments on the original MalNet-Tiny and CICMalDroid,
354     as well as the filtered version after duplicates are removed, using different classifiers (details are in
355     Section 5.2). Figure 3 presents the results. Macro F1 scores decrease after removing the duplicate
356     APKs, consistently for all the classifiers, reaching up to 16.38% decrease in CICMalDroid dataset on
357     GIN method. Motivated by this, we focus on constructing new malware classification datasets that
358     contain unique APKs.

      4.4   N ON - GRAPH APK F EATURES (AF)
379
380   Prior research (Wang et al., 2020; Lee et al., 2019; Alzaylaee et al., 2020) highlights the effectiveness
381   of basic APK features for malware classification. However, existing datasets do not include these APK
382   features, requiring manual extraction from the APKs. To address this, we have incorporated APK
383   features into our BCG dataset. We extracted two basic features from the APK and manifest files: APK
384   size and DEX size. APK size refers to the entire APK file size in bytes, while DEX size represents the
385   size of the Dalvik Executable (DEX) file also in bytes. The DEX file contains the optimized machine
386
      code used by the Android system to run the application. Beyond basic size information, we utilize
      Androguard (Desnos & Gueguen, 2011) to extract various textual features from the APK, which have
387
      been shown to be useful for malware classification in previous works (Wang et al., 2020; Lee et al.,
388
      2019). These textual features include the app/package name, permissions requested by the app, all
389   activities of the app, services or libraries used by the app, and the list of broadcast receivers. We
390   encoded all of the textual features using a 100-dimensional TensorFlow sentence encoder (Cer et al.,
391   2018) and further reduced its dimensionality to 2 using t-SNE (Van der Maaten & Hinton, 2008)
392   for efficient processing. t-SNE effectively preserves local structure and retains original information
393   and widely adopted in prior malware detection studies (Yumlembam et al., 2022; Zhu et al., 2018).
394   Therefore, we used t-SNE to obtain two-dimensional features. A detailed description of all non-graph
395   APK features is provided in Table 3.

      5.2   G RAPH CLASSIFICATION BASELINES
414
415   We consider several state-of-the-art methods for classifying FCGs. These methods encompass both
416   feature-based approaches that analyze characteristics like permissions or app size directly from the
417   APK, and graph-based approaches that focus on the FCG structure and potentially incorporate node
418   features for richer information. Here we summarize each briefly:
419
      1. Deep learning techniques on graphs: We consider three established methods based on GNNs
420   and node embeddings: GCN (Kipf & Welling, 2016), GIN (Xu et al., 2018), and LDP (Cai & Wang,
421   2018). These methods, originally used for malware type and family classification on MalNet, are
422   adapted to our setting. LDP is a simple node representation scheme that summarizes each node and
423   its immediate neighbors using five degree statistics. These features are then aggregated and combined
424   into feature vectors for the GNNs. To ensure consistency, we adopt the same experimental setup in
425   MalNet (Freitas et al., 2020), using 5 GNN layers, Adam optimizer, 64 hidden units, a learning rate
426   of 0.0001, and LDP node features for GCN and GIN. We also incorporated GraphSAGE into our
427   evaluation alongside GCN and GIN given its established effectiveness for malware classification (Lo
428
      et al., 2022; Yumlembam et al., 2022; Vinayaka & Jaidhar, 2021). Like GCN and GIN, GraphSAGE
      was implemented following the same experimental setup for a consistent comparison.
429
430   2. Random forest on app-level features: To investigate the effectiveness of basic APK properties,
431   that are unrelated to FCGs, we extracted various features from the original APK files by using
      Androguard (details are discussed in Section 4.4). We refer to these as APK Features, AF for short.

470   Malware type classification. We evaluate the classification performance of the aforementioned
471
      classifiers on three datasets: MalNet-Tiny, CICMalDroid, and our new BCG dataset. Malware type

      as denoted in Table 2. Table 4 gives the results. All the results on MalNet-Tiny and CICMalDroid

475   difference in Macro-F1 (and Precision and Recall): the best classifier can easily reach to around
476   90% on MalNet-Tiny and CICMalDroid but can only yield 19% on BCG! Table 4 also includes the

480   equivalent (4.54). This highlights the inherent difficulty of classifying malware in BCG.
481   From those results, it is evident that state-of-the-art approaches can easily obtain high performance
482   for malware classification on previously curated datasets, however the same cannot be said for
483   our newly and carefully constructed BCG dataset. The state-of-the-art methods fail miserably on
484   identifying malwares on BCG. This indicates that malware classification using FCGs is significantly
485   more complex than previously thought. There is a clear need for new graph classification techniques
      that can handle the complexities within the BCG.

516   6   C ONCLUSION
517
518
      Traditional malware classification datasets struggle with redundancy, limited size, and outdated data.
519
      These limitations hinder the development of effective models for detecting modern malware threats.
520   This work addresses these issues by introducing the BCG dataset, a collection of recent and unique
521   FCGs containing benign and malware classes as well as different types and families of malware. In
522   our BCG dataset, comprising 9938 graphs, the average size is 25k nodes and 54k edges. It spans
523   a diverse hierarchy of 29 types and 118 families. The analysis of BCG dataset revealed promising
524   avenues for future research. By overcoming the limitations of existing datasets, BCG paves the way
525   for significant advancements in malware classification research.
526   Limitations. While the BCG dataset is valuable for FCG-based malware classification, it does
527   not encompass other aspects of malware analysis, such as identifying malicious code or unpacking
528   obfuscated content. Future research could explore expanding BCG to include these attributes for a
529   more comprehensive analysis.
