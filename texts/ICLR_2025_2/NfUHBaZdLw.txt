009                                                      A BSTRACT
010
011                Neural text-to-SQL models, which translate natural language questions (NLQs)
012                into SQL queries given a database schema, have achieved remarkable perfor-
013                mance. However, database schemas frequently evolve to meet new requirements.
014                Such schema evolution often leads to performance degradation for models trained
015                on static schemas. Existing work either mainly focuses on simply paraphras-
016                ing some syntactic or semantic mappings among NLQ, DB and SQL or lacks
017                a comprehensive and controllable way to investigate the model robustness issue
018                under the schema evolution. In this work, we approach this crucial problem by
019
                   introducing a novel framework, EvoSchema, to systematically simulate diverse
                   schema changes that occur in real-world scenarios. EvoSchema builds on our
020
                   newly defined schema evolution taxonomy, which encompasses a comprehensive
021
                   set of eight perturbation types, covering both column-level and table-level modifi-
022                cations. We utilize this framework to build an evaluation benchmark to assess the
023                models’ robustness against different schema evolution types. Meanwhile, we pro-
024                pose a new training paradigm, which augments existing training data with diverse
025                schema designs and forces the model to distinguish the schema difference for the
026                same questions to avoid learning spurious patterns. Our experiments demonstrate
027                that the existing models are more easily affected by table-level perturbations than
028                column-level perturbations. In addition, the models trained under our paradigm
029                exhibit significantly improved robustness, achieving up to 33 points improvement
030
                   on the evaluation benchmark compared to models trained on unperturbed data.
                   This work represents a significant step towards building more resilient text-to-
031
                   SQL systems capable of handling the dynamic nature of database schemas.1

      1       INTRODUCTION
036
037   Text-to-SQL parsing aims to translate natural language questions (NLQs) into SQL queries given a
038
      database schema, enabling the development of natural language interfaces that allow users to query
      data and invoke services without requiring programming skills (Wang et al., 2020; Zhang et al.,
039
      2024a; Yu et al., 2018; Zhang et al., 2023; Li et al., 2024; Tai et al., 2023). Existing neural text-
040
      to-SQL models have achieved remarkable performance on existing benchmarks (Li et al., 2024;
041   Yu et al., 2018), which play an important role in empowering different platforms such as business
042   and marketing platforms (Song et al., 2024; Zhang et al., 2024b) and being integrated into virtual
043   assistants to enable real-time data query and analysis (Deksne & Skadiņš, 2022).
044
      However, database schemas are not static; they frequently evolve to accommodate new use cases and
045
      improve efficiency (Hillenbrand & Störl, 2021; Cleve et al., 2015). For instance, depending on the
046
      scenario, a large patient table might be merged from or split into two tables: a patient information
047   table and a patient diagnosis table (Figure 1-c), to reduce redundancy, enhance data integrity, and
048   optimize performance (Kumar & Azad, 2017). Such schema evolution occurs frequently, which
049   often leads to distribution shifts (Quionero-Candela et al., 2009; Koh et al., 2021) such as nomen-
050   clature shifts, data granularity shifts, table and column relation shifts and schema complexity shifts.
051   These distribution shifts can cause significant performance degradation when the model trained on
052   old database schema is adapting to new schema designs.

074
075   This challenge highlights a crucial issue in model robustness: how well can a text-to-SQL model
076   adapt to changes in the database schema? Recent studies introduce evaluation benchmarks designed
077   to expose robustness issues by perturbing NLQs, databases or SQL queries (Chang et al., 2023;
078   Deng et al., 2021; Pi et al., 2022; Ma & Wang, 2021). However, these studies have at least one of the
079   following limitations: 1) mainly focus on the syntactic paraphrasing or simple semantic mappings
080   among NLQ, DB and SQL (Chang et al., 2023; Deng et al., 2021); (2) lack a taxonomy of compre-
081
      hensive schema evolution types (Pi et al., 2022); (3) only focus on schema evolution that does not
      lead to SQL changes (Ma & Wang, 2021). These efforts are insufficient in the face of increasingly
082
      complex and rich database schema changes found in reality. Meanwhile, while it is natural to con-
083
      sider collecting new data after schema evolution for retraining a model, repeating the entire model
084   training life cycle frequently can be costly in terms of both time and resources.
085
086   Under this background, we seek to answer the following two questions: (1) How sensitive are exist-
087
      ing text-to-SQL models to various types of database schema changes? (2) How can we train a more
      robust text-to-SQL model that not only performs well on existing database schemas but also adapts
088
      effectively to schema changes? Towards this end, we propose a novel schema evolution synthesis
089
      framework, EvoSchema, which can simulate a wide range of realistic schema design changes by
090   perturbations. Our framework can augment the existing datasets with more comprehensive and real-
091   istic schema change types in a systematic way, which not only builds the foundation to evaluate the
092   robustness against different granularities of schema evolution, but also improves models’ ability by
093   forcing models to distinguish the structure difference within the schema so as to avoid learning the
094   spurious patterns.
095
      As illustrated in Figure 1, EvoSchema framework builds upon our newly defined taxonomy, which
096   encompasses a total of eight types of perturbations over schema, covering both column-level and
097   table-level changes. Column-level perturbations include adding, removing, and renaming columns,
098   while table-level perturbations involve adding, removing, renaming, splitting, and merging tables.
099   We keep the NLQs fixed and examine the robustness of a model under different schema evolutions,
100   and show that existing models are more easily affected by table-level perturbations than column-
101   level perturbations. Moreover, we enhance model robustness by training them with the same ques-
102   tions but coupled with different schema designs to generate the corresponding SQL queries. This
103   training procedure forces the model to distinguish the schema difference which can help models gain
104
      a stronger ability to recognize the correct table and column relation and map them to the questions.
      Our experimental results demonstrate that the perturbation data generated by this framework can
105
      help train better text-to-SQL models, which are more robust to different schema evolution types,
106
      especially on table-level perturbations.
107
      In summary, our main contributions are as follows:

121   2     M ETHODOLOGY
122
123   2.1   BACKGROUND
124
125   In the dynamic landscape of databases, schemas frequently evolve to meet new demands, intro-
126   ducing significant challenges for text-to-SQL models (Delplanque et al., 2018; Cleve et al., 2015).
127   These schema changes can vary widely, from minor modifications to complete restructuring, and
128   can significantly impact the performance of models trained on static schemas. In realistic scenarios,
129   a database can often contain a large number of tables, and only several related tables are responsi-
130   ble for a natural language question (NLQ). In our experiment, we represent the relevant database
131
      schema using Data Definition Language (DDL) 2 and combine it with the NLQ as input. This input
      is then used to prompt the model to generate the corresponding SQL query.
132
133
134   2.2   R ATIONALE FOR S CHEMA E VOLUTION T YPES
135
136
      When a database schema evolves, it can induce distribution shifts in the data that may impact model
      performance. We categorize potential distribution shifts into four types: nomenclature shifts, data
137
      granularity shifts, table and column relation shifts, and schema complexity shifts. (1) Nomenclature
138
      shifts occur when tables and columns are renamed, which may alter the convention of the established
139   terminology within the schema. For example, tables originally named “Products”, “Customers”, and
140   “Orders” might be renamed to “Items”, “Clients”, and “Purchases”, respectively. Such changes often
141   reflect updates in business terminology or compliance with new standards. A desired model should
142   handle those nomenclature shifts to adapt to the new terminology. (2) Data granularity shifts arise
143   from adding or removing columns or tables, which changes the level of detailedness captured in the
144   database. For instance, an “Employee” table with a single “ContactNumber” field might involve
145   another two separate “WorkContact” and “PersonalContact” fields later. This increases the data
146   granularity to meet new requirements, necessitating models to adapt to more complex and detailed
147
      semantics. (3) Table and column relation shifts and schema complexity shifts mainly result from
      restructuring tables through splitting or merging. This process can highly affect how each table
148
      is related to other tables by which column. Both the primary keys and foreign keys may change
149
      along with the table restructure. Besides, the schema complexity may change when multiple tables
150   merge from or split into one table. A desired model is expected to be robust to such changes. By
151   categorizing the distribution shifts caused by schema evolution, we can more effectively understand
152   and evaluate a model’s capacity to adapt to changes in the underlying database schema.
153
154
      2.3   S CHEMA E VOLUTION S YNTHESIS F RAMEWORK
155
156   Our study aims to cover comprehensive potential schema evolution types, which can foster the
157   robustness evaluation of the existing text-to-SQL models and inspire robust model training. We
158   synthesize all the schema evolution types through hybrid strategies, which will leverage both the
159   heuristic rules to guarantee the data quality and LLMs to ensure diversity.
160

      Broad Coverage of Different Schema Evolution Types: We aim to encapsulate a broad range of
163   schema evolution types, recognizing their prevalence and impact in real-world scenarios. Specif-
164   ically, our schema evolution taxonomy includes both column-level and table-level perturbations,
165   which are categorized into eight distinct types. Column-level perturbations comprise three types:
166   adding, removing, and renaming columns, where modifications are restricted to the columns within
167   existing tables. Table-level perturbations encompass five types: adding, removing, renaming, split-
168   ting, and merging tables. These perturbations occur frequently in practice, underscoring the need
169   for text-to-SQL models that can robustly handle such changes.
170   Hybrid Data Synthesis Strategies: To ensure both diversity and quality in the generation of schema
171   perturbations, we employ a combination of heuristics and GPT models to synthesize various pertur-
172   bation types. For each given seed instance, consisting of a <NLQ, relevant schema, SQL> triple,
173   we maintain the natural language question (NLQ) fixed across all perturbation types, while only
174   modifying the relevant schema. The corresponding SQL query is adjusted as necessary to remain
175   consistent with the changes in the database schema.
176
177   2.4   DATA G ENERATION
178
179   Our proposed schema evolution framework can simulate different types of schema perturbations
180   in a configurable way. For adding or renaming columns, both the modified column size and the
181   column position in the tables are set randomly, and we set the original column size in the table as
182   the maximum number of columns to be changed. For removing columns, we can randomly remove
183
      important or unimportant columns from the existing relevant tables. The important columns are
      the columns that appear in the gold SQL, which will inevitably affect the prediction. For adding,
184
      removing, or renaming tables, we randomly add, remove or rename one or multiple tables.
185
186   Schema Change: To ensure the diversity and reasonability of the synthesized schema, we leverage
187   the capabilities of GPT-3.5 and GPT-4 to synthesize realistic and contextually appropriate columns
188   or tables, which help effectively produce high-quality synthetic data that meets our requirements.
189
      For adding or renaming columns and tables, we input the existing relevant tables to GPT-3.5, and
      let the model generate the potential tables or columns that fit the context. For splitting tables or
190
      merging tables, since they are more complex than other perturbations, we use GPT-4 to choose the
191
      tables that can be split or merged and then use the modified tables to replace the original ones.
192   For adding or renaming columns and tables, we apply heuristics to choose the suitable synthesized
193   tables or columns, which are not duplicated with the existing ones. Besides, to ensure the correct
194   relationship among different tables after modifying the schema, we apply heuristics to ensure all the
195   foreign keys change along with their referenced table names and column names. When removing
196   columns or tables, any foreign keys in other tables that reference the removed columns or tables will
197   be removed as well.
198   SQL Change: To ensure the consistency of the <NLQ, relevant schema, SQL>, after we change
199   the relevant table schema, we revise the gold SQL accordingly. Since the NLQs are the same for
200   adding or removing columns and tables, and the schema evolution here doesn’t affect answering the
201   questions, we keep the gold SQL unchanged for these perturbation types. For renaming columns or
202   tables, we revise gold SQL if they appear in the gold SQL. For table splitting or merging, due to the
203   complexity and variation in the required SQL changes, we use GPT-4 to revise the gold SQL. This
204   revision is based on the mappings from the original to the new tables and columns, as well as the
205   necessary adjustments to the JOIN paths. We manually check the edited gold SQL for the evaluation
206
      benchmark to make sure they are correct.
207   By employing these strategies, EvoSchema offers a comprehensive and diverse set of schema evo-
208   lution scenarios that mirror the complexities encountered in real-world database management. By
209   integrating heuristics with LLM-generated perturbations, we maintain both of the diversity and qual-
210   ity, ensuring that the synthesized data is both realistic and challenging.
211
212   2.5   T RAINING PARADIGM
213
214   In our work, we propose a new training paradigm to enhance the model’s robustness against different
215   schema evolution. For each <NLQ, relevant schema, SQL> triple, we fix the NLQ in the training
      data, and augment each triple with different schema designs, which may or may not lead to SQL

      change. Consequently, we obtain multiple triples that can be derived from each of the original triples.
217   We train the model by learning multiple schema designs and SQLs to the original question mappings,
218   which can improve the model’s ability to identify the correct relationships among different tables and
219   columns to the question, and can better distinguish the difference among different schema designs.
220   Through this procedure, the model can avoid learning spurious patterns better and therefore enhance
221   the robustness against different schema evolution types.

225   3.1       DATASET
226
227   For our experiments, we utilize the BIRD (Li et al., 2024) and Spider (Yu et al., 2018) datasets, which
228   are specifically designed for the text-to-SQL task. Both of them consist of NLQs, corresponding
229   database schemas, and gold SQL queries. These datasets are diverse, encompassing a wide range of
230   real-world database scenarios, which provides a robust foundation for evaluating the performance
231   of models in translating NLQs into SQLs.
232   Schema Perturbations: To evaluate the robustness of the text-to-SQL models, we use the BIRD and
233   Spider datasets not only in their original form but also augmented with various column-level and
234   table-level schema perturbations. We ensure that the NLQs remain fixed, while the schema and SQL
235   queries are adjusted as necessary to reflect the changes introduced by our perturbations. We follow
236   the standard train/dev split provided with these datasets, and apply all the perturbations on both
237   training data and evaluation data. The data statistics are in Table 8 and the examples of different
238
      perturbation types are in Figure 2 in the Appendix.

240   3.2       T RAINING AND E VALUATION S ETTINGS
241
      Training Setting: We choose four open-source models: Code Llama-7B (Rozière et al., 2024),
242
      Mistral-7B (Jiang et al., 2023), Llama 3-8B (Dubey et al., 2024) and SQLCoder-7B 3 and two
243   closed-source models: GPT-3.5 4 and GPT-4 (OpenAI et al., 2024) for our experiments. For these
244   four open-source models, we explore two settings: 1) without perturbation types: the model is
245   trained on the original training data without any perturbation types introduced during training. 2)
246   with perturbation types: the model is trained by merging both the original training data and the
247   perturbation training data. For closed-source models, we only use them for evaluation.
248
      Evaluation Setting: For all the closed-source models and the finetuned open-sourced models, we
249   evaluate them under two settings: 1) without perturbation types: this setting uses the standard,
250   unaltered original evaluation data to evaluate the model performance. 2) with perturbation types:
251   the models are evaluated on data where different perturbations are introduced. By comparing the
252   model performance under these two settings, we can assess how resilient the finetuned models and
253   GPT models are to schema evolution in text-to-SQL parsing. This setup provides a comprehensive
254   evaluation of model performance in both standard and perturbed environments, allowing for detailed
255   analysis of robustness and adaptability across different models and schema evolution types.
256
257   3.3       E VALUATION M ETRICS
258
      1) Table Match F1: this score is a metric to measure how well the model correctly identifies the
259
      relevant tables required to generate a valid SQL query. The F1 score is a harmonic mean of preci-
260
      sion and recall, where the precision is the percentage of tables correctly predicted out of all tables
261   predicted by the model and the recall is the percentage of tables correctly predicted out of all the
262   actual tables that should have been selected. The Table Match F1 score combines these two metrics
263   to provide a balanced evaluation, which can assess the ability of text-to-SQL models to correctly
264   identify the required tables from the database schema to form accurate queries. A higher Table
265   Match F1 indicates better performance in selecting the correct tables for the SQL query.
266
      2) Column Match F1: this score is to evaluate how accurately the model identifies the relevant
267   columns required to generate a valid SQL query from a natural language input. Like the Table

283
284   We choose Code Llama-7B (Rozière et al., 2024), Mistral-7B (Jiang et al., 2023), Llama 3-8B

      4.2   I NFLUENCE OF I RRELEVANT TABLES
372
373   We observed that the model trained with perturbation types demonstrates significant robustness to
374   table-level perturbations, such as adding and splitting tables. Upon analyzing the errors, we found
375   that models trained without perturbation types tend to predict SQL queries that join all available
376   tables, even when some tables are irrelevant to the NLQs and SQLs. We hypothesize that this occurs

      5     R ELATED W ORK
512
513   Existing research on text-to-SQL robustness is mainly two-fold: robustness evaluation and robust-
514   ness training. Recent studies introduce evaluation benchmarks designed to expose robustness issues
515
      by perturbing NLQs, databases or SQL queries. However, these studies tend to focus on syntactic
      paraphrasing or simple semantic mappings, such as different representations of numbers or name
516
      abbreviations across NLQ, DB, and SQL (Chang et al., 2023; Deng et al., 2021). While some work
517
      analyzes schema changes, they mainly focus on irrelevant column modifications that do not affect
518   SQL (Ma & Wang, 2021) or with limited perturbation types (Pi et al., 2022). These efforts are in-
519   sufficient in the face of increasingly complex and rich database schemas found in modern datasets.
520   Moreover, the advent of LLMs has mitigated many linguistic challenges, further emphasizing the
521   need for robust adaptation to structural changes in database schemas. For robust training, existing
522   methods employ strategies like decomposing tasks so that models generate each sub-clause indi-
523   vidually before merging them (Gao et al., 2022), or using execution-guided decoding to eliminate
524   incorrect sub-clauses (Wang et al., 2018). While these approaches focus on enhancing various as-
525   pects of text-to-SQL robustness, our work specifically addresses the challenge of schema evolution.
526

      6     C ONCLUSION
528
529
530   In conclusion, we formulate the critical challenge of schema evolution in adaptive text-to-SQL sys-
531
      tems and introduce EvoSchema, a novel framework designed to study and mitigate this problem.
      We developed a comprehensive taxonomy of schema evolution types, enabling the synthesis of real-
532
      istic schema designs through column-level and table-level perturbations. Leveraging this taxonomy,
533
      we constructed an evaluation benchmark that facilitates thorough and comprehensive assessment of
534   model robustness against various schema perturbations. Furthermore, we proposed a new training
535   paradigm that augments existing training data with diverse schema designs, enhancing data diversity
536   and compelling models to recognize schema differences during training. Our approach significantly
537   improves text-to-SQL models, achieving up to a 33-point gain on various schema perturbation eval-
538   uation types compared to models trained on the original, unperturbed data. These findings highlight
539   the effectiveness of our methods in building more robust text-to-SQL models capable of adapting to
      evolving schemas, paving the way for future advancements in the field.
