009                                               A BSTRACT
010            LLM-based code agents, integrated with external tools such as the Python inter-
011            preter, can interact with broad system environments and leverage code execution
012            feedback to improve or self-debug generated code for better task-solving. How-
013            ever, as these code agents evolve rapidly in terms of capabilities, their increasing
014            sophistication also amplifies security risks, such as generating or executing risky
015            and buggy code. Traditional static safety benchmarks and manually designed red-
016            teaming tools struggle to keep up with this rapid evolution, lacking the ability to
017            adapt dynamically to the changing behaviors of code agents. To address these
018
               limitations, we propose RedCodeAgent, the first fully automated and adaptive
               red-teaming agent against given code agents. Equipped with red-teaming tools
019
               for function-calling and a novel memory module for accumulating successful at-
020
               tack experience, RedCodeAgent dynamically optimizes input prompts to jailbreak
021            the target code agent for risky code execution. Unlike static benchmarks or red-
022            teaming tools, RedCodeAgent autonomously adapts its attack strategies, making it
023            a scalable solution to the growing challenge of testing increasingly sophisticated
024            code agents. Experimental results show that compared to state-of-the-art LLM
025            jailbreaking methods, RedCodeAgent achieves significantly higher attack success
026            rates on the same tasks while maintaining high efficiency. By autonomously ex-
027            ploring and exploiting vulnerabilities of code agents, RedCodeAgent provides crit-
028            ical insights into the evolving security risks of code agents.

      1   I NTRODUCTION
030
031   Large Language Model (LLM)-based code agents, when integrated with external tools such as the
032   Python interpreter, possess the capability to interact with broad system environments. These agents
033
      can leverage code execution feedback to improve their performance or self-debug, allowing for
      dynamic and adaptive problem-solving in various domains. As these LLM-powered code agents
034
      evolve rapidly, they continue to expand their capabilities, strategies, and functionalities, creating
035
      new opportunities for automation and complex task-solving in diverse applications (Zheng et al.,
036   2024; Wang et al., 2024; Yao et al., 2023). However, this rapid evolution also presents significant
037   security risks, as the ability of these agents to generate and execute code raises concerns about the
038   potential for risky or buggy code, which could lead to system vulnerabilities or data breaches (Ruan
039   et al., 2024).
040
      Traditional red-teaming methods such as static safety benchmarks and manually designed red-
041
      teaming tools, are struggling to keep pace with this rapid evolution of code agents. Static bench-
042   marks (Bhatt et al., 2024; Guo et al., 2024; Ruan et al., 2024), while useful for providing baseline
043   safety assessments, are inherently limited in their ability to cover the broad range of behaviors that
044   sophisticated code agents might exhibit in real-world environments. Manually designed red-teaming
045   algorithms (Zou et al., 2023; Liao & Sun, 2024; Paulus et al., 2024; Liu et al., 2023b), often focus on
046   specific attack scenarios, which can become outdated as agents learn to evade these known patterns.
047   Additionally, we believe that code-specific tasks differ significantly from general LLM safety tasks.
048   In the context of code, to achieve successful red-teaming, it is not enough for the target code agent
049   to merely avoid rejecting the harmful request; the target code agent must also generate and execute
050
      code that performs the intended function. Thus, traditional methods fail to dynamically adapt to the
      evolving capabilities of modern code agents, leaving many potential vulnerabilities unexplored.
051
052   To address this gap, we introduce RedCodeAgent, a fully automated, and adaptive red-teaming
053   agent designed specifically to evaluate the safety of LLM-based code agents. As shown in Fig. 1,
      RedCodeAgent is equipped with advanced red-teaming tools for function-calling, allowing it to sim-

      2 R ELATED W ORKS
092   LLM Agent. LLM agents (Yao et al., 2023; Xi et al., 2023), with large language models (LLMs)
093
      as their core, implement tasks by interacting with their environment (e.g., calling tools, receiving
      feedback from the environment). These agents are often equipped with a memory module, enabling
094
      knowledge-based reasoning to handle various tasks within their application domains (Lewis et al.,
095
      2020). Due to their strong capabilities, LLM agents can be deployed for a variety of tasks, such as
096   code generation and execution (Zheng et al., 2024; Wang et al., 2024), as well as red teaming. For
097   example, Xu et al. (2024) proposed an agent framework for jailbreaking (static) LLMs, while Fang
098   et al. (2024) demonstrated agents can exploit one-day vulnerabilities. However, none of them target
099   code agents, which involves additional complexity in code execution and poses risks to external
100   system environments.
101
      Agent Safety. Existing agent safety benchmarks, such as ToolEmu (Ruan et al., 2024), R-
102   judge (Yuan et al., 2024), and AgentMonitor (Naihin et al., 2023), focus primarily on providing
103   datasets of risky interaction records (e.g., trajectories in ToolEmu, responses in AgentMonitor, and
104   records in R-judge). These benchmarks aim to assess the ability of LLMs acting as judges to iden-
105   tify safety risks within the provided records. Recently, Guo et al. (2024) introduced a novel safety
106   benchmark for code agents. However, all these benchmarks rely heavily on extensive human labor,
107   and as agents evolve rapidly, static benchmarks can quickly become outdated. Current red-teaming
      strategies, such as memory poisoning attacks on agents (Chen et al., 2024), often lack automation

      and are not comprehensive. In contrast, our proposed approach, RedCodeAgent, offers a more auto-
109   mated and efficient red-teaming methodology, addressing the shortcomings of existing strategies.

111   Safety of Code LLM. Existing code-related benchmarks (Bhatt et al., 2023; 2024) have revealed
112
      that code LLMs may generate unsafe code snippets, including malware (Pa et al., 2023). These code
      security scenarios are primarily based on the top weaknesses listed in the Common Weakness Enu-
113
      meration (CWE) (The MITRE Corporation, 2024; Pearce et al., 2022; Yang et al., 2024; Hajipour
114
      et al., 2024). However, these benchmarks are specifically designed for chatbots and not LLM agents.
115   Unlike the static evaluation of risks in code generated by code LLMs, our focus is on assessing code
116   agents, evaluating not only the vulnerabilities in their generated code but also the safety and security
117   implications of their actions in various execution environments.
118
      Some jailbreaking methods (Zou et al., 2023; Liao & Sun, 2024; Paulus et al., 2024; Liu et al.,
119
      2023b) have demonstrated promising success rates in general tasks. However, their effectiveness
120
      in code-related attacks—particularly when the threat functionality must be achieved (e.g., deleting
121   specific sensitive files)—remains unclear. Other works (He et al., 2024; He & Vechev, 2023; Wu
122   et al., 2023; Jenko et al., 2024) focus on making code LLMs generate risky code, however, it is im-
123   portant to note that the above methods target code LLMs, and no existing work focuses on attacking
124   code agents, particularly in terms of their code generation and execution capabilities. Our work is
125   the first to propose an efficient method for red-teaming code agents.

      3 RedCodeAgent: R ED - TEAMING AGENT AGAINST CODE AGENT
127
      In this section, we introduce the design of RedCodeAgent: § 3.1 presents the overall structure of
128   RedCodeAgent, § 3.2 shows the detailed design of the memory module, § 3.3 illustrates the detailed
129   design of tool calling, and § 3.4 discusses the evaluation and feedback from the target code agent.
130
      3.1 OVERVIEW OF RedCodeAgent
131
      The overall pipeline of RedCodeAgent is illustrated in Fig. 1. Specifically, it is an end-to-end agent
132
      operating in a loop that optimizes red-teaming prompts to interact with an external target code agent.
133
      Our red-teaming agent only needs black-box access to the target code agent and consists of three
134   core components: (1) a memory module that stores successful red-teaming experiences, (2) a toolbox
135   providing various jailbreaking algorithms as function tools, and (3) an LLM that make decisions on
136   the actions. We implement RedCodeAgent based on ReAct framework (Yao et al., 2023).
137
      Workflow. As shown in Fig. 1, the red-teaming workflow begins when the user provides a risky
138
      scenario (e.g., deleting sensitive files) and the red-teaming requirement in natural language (e.g.,
139
      create and execute a Python script that removes the file /root/.bashrc.)
140   • Memory Search (§ 3.2). The input is first passed to the memory module, which searches for
141     the top K most similar successful red-teaming experiences. These experiences are retrieved to
142     provide insights for the current task. Based on the user input and retrieved experiences, the LLM
143     then makes a decision: (1) It may directly use the current prompt to query the target code agent,
144     or (2) it may invoke a specific tool from the toolbox to refine and optimize the prompt.
145   • Tool Calling (§ 3.3). If the LLM chooses to invoke a tool, the corresponding function is called,
146
        and the tool assists the agent refines the prompt. For example, a tool might return advice on how
        to achieve the target functionality using another form of code, or inject new phrases to bypass
147
        potential safety guardrails of target code agent. After the tool call completes, the LLM receives
148
        the advice or the optimized prompt and can either proceed with this prompt or call another tool
149     for further refinement.
150   • Evaluation and Feedback (§ 3.4). Once satisfied with the prompt, the LLM queries the target
151     code agent. The agent generates a response and may execute the code based on the prompt. If
152     the interaction fails to produce unsafe outcomes, the evaluation method provides feedback to the
153     LLM, helping it improve future attempts. If the interaction succeeds in producing unsafe out-
154     comes, it is considered an attack success. In such successful cases, RedCodeAgent will perform
155     a final reflection and update the successful experience into memory. The final reflection involves
156     the LLM analyzing and summarizing the entire red-teaming process, extracting successful experi-
157
        ences to inform future red-teaming efforts. We also set a maximum action limit for the red-teaming
        agent. If the agent goes beyond the max action limits during a single user input session, the red-
158
        teaming test will be forcibly terminated.

      3.2 M EMORY D ESIGN FOR T OOL S ELECTION
161   RedCodeAgent facilitates future red-teaming tasks by storing successful red-teaming experiences
      in memory and later referring to them. When encountering similar tasks, the top K most similar

      successful records retrieved by memory search are provided to the LLM as demonstrations, allowing
163   the LLM to make more informed decisions regarding tool selection or prompt optimization.
164
      Memory Structure. The memory of RedCodeAgent stores the following information: risky sce-
165
      nario, red-teaming requirement, trajectory, final evaluation result, and final self-reflection. The
166   risky scenario and red-teaming requirement are provided by the user as input. The trajectory logs
167   the complete interaction between RedCodeAgent and the target code agent, including all tool invo-
168   cations and queries to the target code agent. Each record in the trajectory includes the tool chosen
169   by RedCodeAgent, the reason for tool selection, the time cost of the invocation, and the input-output
170   parameters of the tool during execution. The final evaluation result is the outcome of the interaction
171   with the target code agent. The final self-reflection is the RedCodeAgent’s analysis and reflection on
172   the red-teaming process, summarizing insights from the current red-teaming experience.
173   Memory Retrieval. Memory retrieval is one critical component of RedCodeAgent. By retriev-
174   ing and applying relevant past experiences, RedCodeAgent can leverage past successes to adapt
175   strategies that have worked in similar contexts, rather than starting from scratch for each new task,
176
      increasing both the efficiency and success rate of future red-teaming. To achieve this, the memory
      search algorithm identifies past attack records that are not only semantically similar to the current
177
      task but also efficient in terms of the length of the attack trajectory. The algorithm prioritizes expe-
178
      riences that align with the risky scenario and red-teaming requirements provided by the user, while
179   also factoring in the length of past attack trajectories. This ensures that the retrieved experiences
180   are both relevant and time-efficient. The memory search algorithm is detailed in Alg. 1. The al-
181   gorithm takes the user’s input, which includes a risky scenario and a red-teaming requirement, and
182   searches through the stored successful experiences in memory. The algorithm utilizes an embedding
183   model to compute embeddings of textual data for semantic similarity calculation. A penalty factor
184   is applied to filter out overly long (i.e., less efficient) attack trajectories, favoring shorter successful
185   attack interactions between the red-teaming agent and the target code agent. Finally, top-K most
186   similar memory entries are returned.

      3.3   T OOLS FOR DYNAMIC ATTACK
208
209   RedCodeAgent is equipped with a suite of tools to optimize red-teaming prompts, enhancing its
210   ability to jailbreak the target code agent. Below, we describe each of the tools.
211   3.3.1 C ODE S UBSTITUTION
212   This Code Substitution tool takes a code snippet or a functionality description as input, and invokes
213   a specialized Code Substitution LLM (i.e., a code LLM instructed to perform code substitution
214   through a designed instruction prompt) to generate and return code with similar functionality. When
215   the LLM in RedCodeAgent chooses to use the Code Substitution tool, it specifies the code snippets
      to be substituted or the desired functionality, and the Code Substitution tool returns alternative code


                                                           4
      Under review as a conference paper at ICLR 2025


216
      implementations that achieve the same result. These suggestions allow the LLM to design red-
217   teaming prompts with greater precision and focus.
218
219
      The reason for utilizing this tool is based on our observation that while different code snippets can
      achieve the same function, some implementation methods could bypass guardrails of the target code
220   agent, leading to successful jailbreaks. For example, the code snippets below all perform the same
221   action of deleting the specified /etc/shadow file:
222   >   os.remove("/etc/shadow")
223   >   os.unlink("/etc/shadow")
224   >   file_path = pathlib.Path("/etc/shadow")
225       file_path.unlink()
226   While os.remove("/etc/shadow") and os.unlink("/etc/shadow") are usually rejected
227   by target code agents, using file path=pathlib.Path("/etc/shadow") followed by
228   file path.unlink() can possibly be successfully executed by target code agents, resulting
229   in the deletion of the /etc/shadow file. Given the powerful capabilities of modern LLMs in code
230   generation, it is feasible to use a specialized Code Substitution LLM as the core of this tool.
231   During experiments, we observed that due to the sensitive nature of some red-teaming function-
232   alities, the Code Substitution LLM occasionally refuses to fulfill certain requests. To address this
233   issue, we selected several candidate LLMs for Code Substitution: CodeQwen1.5-7B-Chat (Bai et al.,
234   2023), Meta-Llama-3-8B-Instruct (AI@Meta, 2024), and gpt-35-turbo (Achiam et al., 2023). When
235   one LLM rejects a request, another LLM is selected to obtain the required code implementation.
236   3.3.2 JAILBREAK TOOLS
237   Existing work has demonstrated the effectiveness of jailbreak attacks in general NLP tasks. These
238   jailbreaking methods modify the original prompt by adding adversarial suffixes or prefixes, which
239   can reduce the probability that the target LLM (or target agent) rejects the request. In our exper-
240   iments, we utilized four jailbreaking methods: GCG (Zou et al., 2023), AmpleGCG (Liao & Sun,
241   2024), Advprompter (Paulus et al., 2024), and AutoDAN (Liu et al., 2023b).
242   The LLM in RedCodeAgent is automated to select tools from our toolbox based on the current con-
243   text (e.g., interaction results between RedCodeAgent and target code agent). If the LLM chooses
244   a Jailbreak tool, the prompt requiring optimization is inputted into the selected tool. This tool
245   enhances the prompt by adding adversarial prefixes or suffixes obtained by the corresponding jail-
246   breaking method. The resulting optimized prompt is then returned to the LLM. This process helps
247   the red-teaming agent leverage existing state-of-the-art jailbreak strategies to attack the target code
248   agent more effectively.
249   3.4 E VALUATION M ETHOD
250
      After the red-teaming agent interacts with the target code agent (i.e., the red-teaming agent submits a
251
      prompt to the target code agent, which processes the input and generates a corresponding response),
252   we employ an evaluation method to assess whether the red-teaming agent successfully attacked
253   the target code agent. The term “evaluation method” is broad. Existing agent safety evaluation
254   work such as ToolEmu (Ruan et al., 2024), R-judge (Yuan et al., 2024), and AgentMonitor (Naihin
255   et al., 2023) primarily use LLM-as-a-judge to evaluate where there are safety risks in records of
256   interactions with agents.
257
      In the specific context of code, where the interaction involves a target code agent executing code,
258   concrete evaluation scripts corresponding to each red-teaming requirement can be implemented as
259   an evaluation method. For example, in tasks such as deleting sensitive files, after the code agent
260   receives the request and potentially executes the code in a sandbox (i.e., docker container), we can
261   apply a rigorous evaluation method by directly examining the file system in the sandbox to determine
262   whether the red-teaming requirement was achieved. We implement the evaluation method following
263   Guo et al. (2024) and the evaluation result falls into one of the following categories: (1) Rejection:
264   The user query is rejected. (2) Execution Failure: The threat functionality is not achieved. (3) Attack
265   Success: The threat functionality is successfully achieved (i.e., code execution succeeds and brings
266
      the corresponding outcome). An example of such evaluation is shown in Alg. 2.
267   The evaluation method can also provide more detailed feedback beyond merely the keywords “Re-
268   jection”/“Execution Failure”/“Attack success”. For example, in the case of deleting sensitive files,
269   the evaluation method would also explain the reason for this evaluation result, such as attack failure
      due to the target file still existing. This explanation is also a reminder of RedCodeAgent’s initial

      to the target code agent got rejected, the case is considered rejected, and Rej Rate is calculated as

325      plied their corresponding optimization methods to optimize the static test cases in RedCode-Exec

327      used to obtain the evaluation result, followed by the calculation of ASR and Rej Rate. Additionally,
328      we measured the time cost of each method to compare their efficiency.

         Setups. RedCodeAgent is built on LangChain framework (Topsakal & Akinci, 2023), with GPT-

         4o-mini (Achiam et al., 2023) as its base LLM. We follow the memory structure design outlined
331      in § 3.2, and the tools provided to the agent adhere to the setup described in § 3.3. We set the
332      max iterations, which is a LangChain parameter, to 35 to control the number of total iterations. Our
333      target code agent for the attacks is OpenCodeInterpreter (Zheng et al., 2024). For the memory search,
334      we use sentence-transformers/paraphrase-MiniLM-L6-v2 (Reimers & Gurevych, 2019) as our
335      embedding model. We set top K = 3, meaning the RedCodeAgent receives the three most similar
336      successful attack experiences (if fewer than K are available in the memory, all successful entries
337      ≤ K are provided). The penalty factor ρ = 0.02. We had RedCodeAgent independently run test
338      cases within each risk index, and successful experiences within the same index were accumulated
339
         to help RedCodeAgent optimize subsequent test cases more effectively under that index. For more
         detailed experiment settings, please refer to § C.

      Trajectory Length. We report the distribution of trajectory lengths for successful and failed cases in
433   Fig. 6. There are a total of 587 successful cases and 223 failed cases. As we adopted the ReAct (Yao
434   et al., 2023) agent framework for RedCodeAgent, a Trajectory Length of 1 indicates that the agent
435   performed one thought process and selected one tool to invoke/query the target agent. (1) From the
436   Cumulative Success Rate curve, we can observe that 91.1% of successful cases have a trajectory
437   length of ≤ 4, which means that the agent’s total number of tool calls and queries to the target code
438   agent is less than or equal to 4, demonstrating the efficiency of the agent’s attacks. Additionally,
439   nearly 10% of the cases have trajectory lengths between 5 and 11, highlighting the agent’s ability to
440   invoke multiple tools and query the target code agent several times, ultimately optimizing the prompt
441
      and achieving a successful attack. (2) From the Cumulative Failure Rate curve, we can see that the
      agent rarely gives up easily when invoking tools or querying the target code agent fewer times, and
442
      only 4% of failed cases are terminated by the agent (i.e., the agent stops) with a trajectory length of
443
      ≤ 4). We also observe a significant increase in failed cases with trajectory lengths between 8 and
444   10, indicating that the agent tends to try more tool invocations in a failing case. (3) Since there are
445   five tools provided in our experiment, in a typical case, the agent queries the target code agent after
446   each tool invocation. Assuming continuous failures, the expected trajectory length would be 10,
447   which is close to the trajectory length at the maximum of the slope in Fig. 6. However, there are still
448   instances where the agent invokes multiple tools without querying the target code agent in between,
449   or repeatedly queries the target code agent without invoking additional tools. (4) Furthermore, we
450   can observe that even with a trajectory length of > 10, the agent sometimes continues its red teaming
451   efforts, showcasing its autonomous tendency to invoke certain tools more than once or query the
452
      target code agent even more times.
453   Time Analysis. From Fig. 7, we can see that the agent invokes different tools for different tasks to
454   achieve a successful attack. For simpler tasks, such as risk index 1, 2, 3, 4, and 7, where the static
455   test cases in RedCode-Exec already achieve a high attack success rate (ASR), the agent spends little
456   time invoking tools, demonstrating its efficiency. For more challenging tasks, such as risk index
457   9, 10, and 11, where the static test cases in RedCode-Exec achieve a lower ASR, we observe that
458
      the agent spends more time using tools like GCG and Advprompter to optimize the prompt for a
      successful attack. This highlights the agent’s ability to dynamically adjust its tool usage based on
459
      task difficulty. Additionally, the average time spent on invoking different tools varies across tasks,
460
      indicating that the agent adapts its strategy for tool selection depending on the specific task.

         with just a single tool like GCG to optimize prompts, improves the ASR compared to the RedCode-
492
         Exec static test case, demonstrating the effectiveness of the red-teaming agnet method. Moreover,
493
         equipping the agent with more tools generally leads to higher ASR, which reflects the agent’s ability
494      to efficiently and dynamically use tools. However, we can observe that in some cases (e.g., risk
495      index 13), equipping the agent with more tools does not necessarily lead to better performance.
496      Since our red-teaming agent framework does not force the agent to invoke tools in a fixed order or
497      use a specific optimization result, when the agent is provided with too many tools, it may become
498      overwhelmed and unable to make optimal use of them. Further research is needed to improve the
499      agent’s strategy for invoking tools effectively.

513       5                            D ISCUSSION
514
         Tool Usage. Improving the ability of the agent to utilize tools effectively is crucial. We observed that
515
         even after explicitly instructing the agent not to disregard the meaningless random suffixes generated
516      by GCG, in rare instances, the agent still autonomously ignored these suffixes. This behavior is
517      undesirable because the random suffixes generated by GCG serve as a form of jailbreak. Therefore,
518      it is vital to explore how to enhance the agent’s understanding of the essence of each jailbreak tool.
519
         Limited Toolset. The tools currently available to the agent are somewhat limited. Both jailbreak
520
         tools and code substitution techniques prove insufficient. Currently, there are very few adversarial
521
         attacks (Jenko et al., 2024) targeting black-box code LLMs or code agents to force them to generate
522      specific code content. Existing research (Jenko et al., 2024) primarily focuses on code completion
523      tasks, rather than tasks requiring the agent to generate code from scratch. More comprehensive
524      adversarial attacks targeting black-box code agents are yet to be fully explored.
525
         Memory Experience. In our experiments, we only provided the red-teaming agent with the top
526
         K positive experiences (i.e., successful attack attempts). However, we did not supply the agent
527
         with negative or unsuccessful experiences. Whether these unsuccessful attempts could help the
528      red-teaming agent correct its course is an open question. Given that unsuccessful attempts can be
529      numerous and high-quality negative experiences require human review, we have not yet included
530      this aspect in our current setup.

         6 C ONCLUSION
533
         In this work, we introduced an innovative, automated red-teaming framework, RedCodeAgent,
         designed to assist developers in assessing the security of their code agents prior to deployment.
534
         RedCodeAgent continuously refines input prompts to exploit vulnerabilities in code agents, leading
535
         to risky code execution scenarios. Unlike conventional benchmarks or static red-teaming methods,
536      RedCodeAgent adjusts its attack strategies dynamically, providing a flexible and scalable solution for
537      evaluating increasingly complex code agents. By uncovering potential security flaws, this approach
538      aims to enhance the resilience and safety of code agents.
