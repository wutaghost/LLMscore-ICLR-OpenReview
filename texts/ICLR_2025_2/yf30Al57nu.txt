                                                  A BSTRACT
013
014            Large Language Models (LLMs) have significantly advanced code generation but
015
               often require substantial resources and tend to over-generalize, limiting their effi-
               ciency for specific tasks. Fine-tuning smaller, open-source LLMs presents a viable
016
               alternative; however, it typically lags behind cutting-edge models due to super-
017
               vised fine-tuning’s reliance solely on correct code examples, which restricts the
018            model’s ability to learn from its own mistakes and adapt to diverse programming
019            challenges. To bridge this gap, we introduce C ODE L UTRA, a novel framework
020            that enhances low-performing LLMs by leveraging both successful and failed code
021            generation attempts. Unlike conventional supervised fine-tuning, C ODE L UTRA
022            employs an iterative preference-guided refinement mechanism to compare cor-
023            rect and incorrect solutions as well as maximize the likelihood of correct codes.
024            Through continuous refinement, C ODE L UTRA enables smaller LLMs to match or
025            surpass GPT-4’s performance in various code generation tasks without relying on
026
               vast external datasets or larger auxiliary models. On a challenging data science
               coding task, using just 500 samples improved Llama-3-8B’s accuracy from 28.2%
027
               to 48.6%, approaching GPT-4’s performance. These results highlight C ODE L U -
028
               TRA ’s potential to close the gap between open-source and closed-source models,
029            making it a promising approach in the field of code generation.

032   1   I NTRODUCTION
033
034   Large language models (LLMs) have revolutionized numerous domains, consistently delivering
035   great performance across different tasks (Brown et al., 2020; Achiam et al., 2023; Bubeck et al.,
036   2023; Team et al., 2023; Anthropic, 2023; OpenAI, 2023). Among these applications, code gener-
037   ation stands out as particularly promising. Models pre-trained on extensive code repositories have
038   demonstrated an impressive capability to solve diverse programming challenges (Li et al., 2023;
039
      2022; Nijkamp et al., 2023; Zheng et al., 2023; Fried et al., 2022; Chen et al., 2021b; Wang et al.,
      2021b; 2023b).
040
041   Despite the promise, deploying ultra-large models for code generation poses significant challenges.
042   Top-tier closed-source models like GPT-4 (OpenAI, 2023) are often highly resource-intensive and
043   do not offer the flexibility for customization to specific code generation tasks. Fine-tuning smaller,
044   open-sourced LLMs for targeted applications serves as a compelling alternative. Code generation
045
      demands not just syntactical correctness but also a deep understanding of logical and domain-
      specific nuances, which complicates model training. While a straightforward approach is to perform
046
      supervised fine-tuning (SFT) on the target task, this typically results in only incremental improve-
047
      ments and frequently lags behind cutting-edge solutions. This limited improvement stems from
048   SFT’s reliance solely on correct code examples, which restricts the model’s ability to learn from its
049   own mistakes and adapt to diverse programming challenges. We illustrate this in Figure 1, where
050   SFT provides a minor boost in performance on the challenging data science task, with LlaMa-3-8B’s
051   Pass@1 improving from 28.2% to 30.0%. A notable gap remains between the fine-tuned model and
052   GPT-4, which leads with a Pass@1 score of 49.4%. To address the issue, prior solutions collect ad-
053   ditional training data by leveraging more powerful LLMs (Shen et al., 2023; Luo et al., 2023; Wang
      et al., 2023b; Yang et al., 2024) or by collecting external datasets from public repositories (Lozhkov

081
      To address this challenge, we introduce C ODE L UTRA, a framework that iteratively improves the
082   performance of a given LLM without relying on vast external datasets or larger auxiliary models.
083   Our approach demonstrates that even with limited data at hand, substantial gains in code generation
084   quality can be achieved, closing the gap between smaller fine-tuned models and the top-tier LLMs.
085   Unlike traditional fine-tuning methods that rely solely on correct code solutions, C ODE L UTRA con-
086   structs and learns from both successful and failed code generated by the current model, which forms
087   self-generated comparative data. The failed code attempts are invaluable for refining models, since
088   they provide concrete examples of common errors and enable the model to learn strategies for avoid-
089   ing similar mistakes in future generations. To harness the comparative data, a key innovation of
090
      C ODE L UTRA is the preference-guided refinement mechanism that compares correct and incorrect
      code snippets, iteratively refining the model’s understanding of code quality. With each iteration, the
091
      model generates code solutions, evaluates their correctness, and updates its parameters based on the
092
      evolving preference dataset. This process allows for continuous improvements of the base model,
093   making the framework effective even with limited initial data (e.g., a few hundred samples).
094
095   We comprehensively evaluate the effectiveness of C ODE L UTRA on challenging data query and data
096
      science tasks, where the LLM is tasked with generating the correct SQL or Python code to solve a
      given problem. We compare C ODE L UTRA with 13 open-source and closed-source LLMs that are
097
      competitive in code generation. Notably, on the data query task, our framework allows Llama-3-
098
      8B (Dubey et al., 2024) to achieve an execution accuracy of 76.6%, which exceeds GPT-4’s 74.4%.
099   Under a challenging data science task, we find that using just 500 samples improved Llama-3-8B
100   from an accuracy of 28.2% to 48.6%, approaching the performance of GPT-4. This demonstrates
101   that C ODE L UTRA achieves strong results even with a limited number of high-quality annotations.
102   Moreover, we observe the consistent performance gains of C ODE L UTRA on different base models,
103   including Gemma-7B (Team et al., 2024) and StarCoder-7B (Dai & Kumar, 2023). These findings
104   highlight the potential of C ODE L UTRA in closing the performance gap between open-source and
105   closed-source models. To summarize our key contributions:

120   LLMs for code generation. LLMs are pre-trained on diverse datasets encompassing both natural
121
      and programming languages. In code generation tasks, an LLM receives a prompt—such as a natural
      language description, and generates the corresponding code by predicting the next token in the
122
      sequence. Formally, code generation is modeled as the conditional probability of a code sequence
123
      y = (y1 , y2 , . . . , yT ) given an input prompt x:

129   (y1 , y2 , . . . , yt−1 ) represents the tokens generated before time step t.

      Supervised fine-tuning on task-specific dataset. Pre-trained LLMs can be suboptimal on task-
      specific dataset, necessitating fine-tuning. We consider a task-specific dataset D = {(xi , yi )}ni=1
132
      containing n examples, where each pair (xi , yi ) represents an input prompt xi and its corresponding
133
      target code yi . Supervised fine-tuning (SFT) adjusts the model’s parameters θ by maximizing the
134   likelihood of generating correct code sequences yi . The loss is defined as:
135
136                              πSFT = argmaxπθ E(xi ,yi )∼D (log πθ (yi |xi )) .                         (2)
137   This process relies on the quality of the dataset, guiding the model to produce more accurate and
138   reliable code based on the given prompts.
139
140   Verification of code correctness. We verify the correctness of the generated code by comparing
141   it with the ground truth code ygt . We define a verification function V (y) that checks if the output of
142   the generated code matches that of the ground truth. Given an input I, the generated code is correct
143
      if it produces the same output f (y, I) as the ground truth code f (y, I). Specifically, V (y) = 1 if the
      generated code is correct, and V (y) = 0 otherwise.
144
145
      Limitations of SFT for code generation. Code generation demands not just syntactical correct-
146
      ness but also a deep understanding of logical and domain-specific nuances, which complicates model
147   training. A major limitation of SFT is that it solely maximizes the likelihood of providing correct
148   code, which restricts the model’s ability to learn from its own mistakes. Since the training process
149   focuses exclusively on provided examples, the LLM doesn’t receive the gradient from incorrect or
150   suboptimal code. For instance, if the model only predicts wrongly in the final token in a code snip-
151   pet, the overall probability P (y|x) in the Equation 1 might still remain high as the preceding tokens
152   are correct. Correspondingly, the SFT loss is very small in that case. However, this single erroneous
153   token can render the entire code nonfunctional or introduce subtle bugs, significantly affecting exe-
154   cution correctness despite a high likelihood score, which is different from general NLP tasks. This
155   reliance on only correct examples limits the model’s capacity to identify and recover from errors,
      reducing its ability to handle more complex or nuanced coding tasks. This motivates our framework
156
      C ODE L UTRA, which leverages both successful and failed code generation attempts.

233   Remark. While DPO-style preference optimization has been studied in recent literature (see re-
234   view in Section 6), its connection to code generation remains largely unexplored. To the best of our
235   knowledge, our work is the first to establish this critical link between iterative preference learning
236   and code generation. We highlight several novel aspects that differentiate our approach from prior
237   work. First, previous studies mostly focus on natural language generation tasks and rely on model-
238   generated rewards or feedback from other large models (Chen et al., 2024; Xiong et al., 2024; Yuan
239   et al., 2024; Pang et al., 2024; Xie et al., 2024). In contrast, we focus exclusively on code generation
240   where the preference labels for iterative refinement come from execution results. This shift is es-
241
      sential and unique for code generation, where execution correctness is a key indicator of preference.
      Moreover, our work uncovers the insight into the dual-loss mechanism, revealing that it plays a cru-
242
      cial role in improving performance in the context of code generation. From a practical standpoint,
243
      our training pipeline bypasses the standalone SFT stage typically required before the preference op-
244   timization phase, thereby streamlining the training process. Overall, our work not only establishes
245   a novel link between iterative preference learning and code generation but also introduces practical
246   innovations that enhance performance, marking a significant step forward in the field.

253   tasks: Data Query and Data Science. Both tasks reflect common and practical challenges in fields
254   such as business, healthcare, and scientific computing, where precise code generation is critical for
255   solving data-related problems while receiving limited attention. For the Data Query task, the model
256   is given a natural language problem description and is tasked with generating the corresponding
257   SQL query for a database using an LLM. For example, given the description “How many heads
258   of the departments are older than 56?”, the model should produce the appropriate SQL query to
259
      execute this request. In the Data Science task, the LLM is tasked with generating the correct Python
      code to solve a given data science problem. For instance, given the problem “I have a 2D array to
260
      represent a many-many mapping. What is the quickest way to zero out the second row and the first
261
      column?”, we test LLM’s ability to solve data science problems with numpy. We provide examples
262   of the two tasks of the question and ground truth code in the Appendix A.2

      6    R ELATED W ORK
494
495
      Preference learning for LLMs. Preference learning aims to guide language models toward gen-
      erating outputs that align with human preferences and desirable outcomes. A significant body of
496
      research has utilized human feedback to refine LLMs across various language tasks (Ziegler et al.,
497
      2019; Ouyang et al., 2022; Stiennon et al., 2020; Kreutzer et al., 2018). The Reinforcement Learn-
498   ing from Human Feedback (RLHF) framework, in particular, has been effective in aligning large
499   pre-trained language models (Christiano et al., 2017; Ziegler et al., 2019; Ouyang et al., 2022; Bai
500   et al., 2022). However, RLHF can suffer from training inefficiencies and sensitivity to hyperpa-
501   rameters. In response, recent studies have shifted towards closed-form loss functions that directly
502   utilize offline preference data, exemplified by DPO (Rafailov et al., 2023) and related methodolo-
503   gies (Liu et al., 2023; Ethayarajh et al., 2024; Gheshlaghi Azar et al., 2024; Pal et al., 2024; Liu et al.,
504   2023; Xiong et al., 2023a; Tang et al., 2024; Yu et al., 2024). While DPO-style models inherently
505   provide rewards, iterative DPO—where the model generates its own pairwise preference data—has
506
      demonstrated strong performance and potential (Chen et al., 2024; Xiong et al., 2024; Yuan et al.,
      2024; Rosset et al., 2024; Pang et al., 2024; Xie et al., 2024). In this work, we introduce iterative
507
      preference-guided refinement to code generation for the first time, achieving GPT-4-level results and
508
      providing an in-depth analysis that paves the way for future research.
509
510
      LLMs for code generation. LLMs trained on vast corpora of code have demonstrated remarkable
511
      capabilities across a wide range of tasks, including code generation (Chen et al., 2021c; Austin et al.,
512   2021; Zhang et al., 2022), program repair (Xia & Zhang, 2022; Wei et al., 2023; Xia et al., 2023;
513   Jiang et al., 2023; Bouzenia et al., 2024; Xiong et al., 2023b), and software testing (Chen et al.,
514   2023; Wang et al., 2024a; Zhou et al., 2024). Foundational models (Nijkamp et al., 2022; Wang
515   et al., 2021a; Li et al., 2023; Roziere et al., 2023) pre-trained on extensive codebases, have estab-
516   lished strong general-purpose capabilities for code generation. Building upon these powerful foun-
517   dations, more recent models like W IZARD C ODER (Luo et al., 2023) and DS-C ODER (Li et al., 2023;
518   Bouzenia et al., 2024) enhance contextual understanding by leveraging repository-level organization
519   of pretraining data and incorporating retrieval-augmented techniques (Borzunov et al., 2024). More-
520
      over, C ODE I NSTRUCT (Wang et al., 2024b) capitalize on instruction fine-tuning to improve align-
      ment with human coding preferences. Fine-tuning methodologies and prompt-engineering tech-
521
      niques (Luo et al., 2023; Chen et al., 2023; Zhang et al., 2024b) have been crucial in unlocking these
522
      models’ full potential for domain-specific tasks, such as security, AI-assisted development, and code
523   synthesis in specialized fields. Zhang et al. (2024a) pay attention to preference learning for program-
524   ming problems. However, they rely on GPT-4 for generating test cases and use preference data in
525   a single offline run. In contrast, we iteratively refine a small LLM using self-generated preference
526   data, without external datasets or larger models. Moreover, different from Zhang et al. (2024a) per-
527   forming SFT and preference learning in two stages, we introduce the dual-loss to compare correct
528   and incorrect solutions and maximize the likelihood of correct codes in one stage.
529

      7    C ONCLUSION
531
532
      We introduced C ODE L UTRA, a preference-guided refinement framework designed to enhance LLMs
533
      for code generation without the need for external datasets or larger models. By utilizing self-
534   generated data from both successful and failed attempts, C ODE L UTRA refines the model’s under-
535   standing of code quality. Our experiments on data query and data science tasks demonstrate that
536   C ODE L UTRA significantly boosts the performance of base LLMs. For example, fine-tuning Llama-
537   3-8B with C ODE L UTRA outperforms GPT-4 on data query tasks and nearly matches GPT-4 perfor-
538   mance on data science task with small training data. Additionally, C ODE L UTRA reduced common
539   coding errors while improving the quality and accuracy of generated code. These results underline
      C ODE L UTRA’s potential as a cost-efficient and scalable solution for LLM code generation.

954   B     L IMITATION AND FUTURE WORK
955
956   While C ODE L UTRA significantly enhances code generation performance by leveraging self-
957   generated comparative data, it exhibits several limitations that warrant consideration. The current
958   framework primarily focuses on the correctness of the generated code, overlooking other vital as-
959   pects such as efficiency, readability, and adherence to specific formal specifications, which are essen-
960   tial for practical applications. Additionally, C ODE L UTRA treats all failed code attempts uniformly,
961   without distinguishing between different types or severities of errors, potentially limiting the model’s
962   ability to learn from more informative mistakes.
963   To address the aforementioned limitations, future research related to C ODE L UTRA should explore
964   several key directions. Expanding the preference-guided refinement mechanism to incorporate addi-
965   tional criteria such as code efficiency, readability, and compliance with formal specifications would
966   enhance the overall quality and utility of the generated code. Developing a more nuanced approach
967   to categorizing and prioritizing failed code attempts based on the type and severity of errors could
968   enable more targeted and effective learning, thereby improving the model’s ability to avoid simi-
969
      lar mistakes in future generations. Exploring alternative evaluation methods, such as static code
      analysis or formal verification tools, could reduce the framework’s reliance on execution results and
970
      broaden its applicability to a wider range of tasks.
