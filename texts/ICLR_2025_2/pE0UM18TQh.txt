009                                                A BSTRACT
010
011            The recently introduced TabPFN pretrains an In-Context Learning (ICL) trans-
012            former on synthetic data to perform tabular data classification. In this work, we
013            extend TabPFN to the fine-tuning setting, resulting in a significant performance
014            boost. We also discover that fine-tuning enables ICL-transformers to create com-
015            plex decision boundaries, a property regular neural networks do not have. Based
016
               on this observation, we propose to pretrain ICL-transformers on a new forest
               dataset generator which creates datasets that are unrealistic, but have complex
017
               decision boundaries. TabForest, the ICL-transformer pretrained on this dataset
018
               generator, shows better fine-tuning performance when pretrained on more com-
019            plex datasets. Additionally, TabForest outperforms TabPFN on some real-world
020            datasets when fine-tuning, despiting having lower zero-shot performance due to
021            the unrealistic nature of the pretraining datasets. By combining both dataset gen-
022            erators, we create TabForestPFN, an ICL-transformer that achieves excellent fine-
023            tuning performance and good zero-shot performance.

      1   I NTRODUCTION
027
      Tabular data classification is widespread across all industries, leading to an increased interest in the
028
      research field of deep learning for tabular data (Liakos et al., 2018; Zhang et al., 2020; Keith et al.,
029
      2021; Pang et al., 2022). This type of classification involves classifying a target variable based on
030   a set of attributes, which are commonly stored in tabular format. Examples of tabular classification
031   include predicting the existence of chronic kidney disease based on blood test results (Ogunleye
032   & Wang, 2020), estimating the click-through rate of advertisements (Richardson et al., 2007), and
033   predicting the stability of pillars in hard rock mines (Liang et al., 2020). Despite the significance of
034   tabular data, major breakthroughs in AI, as demonstrated in vision and language domains, have yet to
035   reach the tabular domain. In fact, neural networks are currently outperformed by tree-based machine
036   learning algorithms such as XGBoost (Chen & Guestrin, 2016) and CatBoost (Prokhorenkova et al.,
037   2018) in tabular classification tasks (Gorishniy et al., 2021; Grinsztajn et al., 2022; McElfresh et al.,
038
      2023).
039   In an attempt to bridge this performance gap, a recent method called tabular prior-data fitted net-
040   works (TabPFN) (Hollmann et al., 2023) introduces an in-context learning (ICL) (Dong et al., 2023)
041   scheme, demonstrating promising results (Grinsztajn et al., 2022). This tabular ICL-transformer can
042   predict test observations zero-shot: with only one forward pass using training observations included
043   in the context. Hollmann et al. generate their pretraining data synthetically, focusing on creating
044
      realistic datasets that act as a “prior”. They make their datasets realistic by carefully crafting corre-
      lations between features, introducing variety in feature importance, and leveraging structural causal
045
      models to simulate causal relationships.
046
047   In this work, we extend TabPFN to the fine-tuning setting. We show that fine-tuning this ICL-
048   transformer on downstream tasks boosts the performance, particularly on datasets with more than a
049   thousand samples. We also find that increasing the context size provides a performance increase for
050
      both zero-shot and fine-tuning. Overall, we find that fine-tuning provides such a large performance
      boost that we recommend always using it over zero-shot when the number of observations exceeds
051
      a thousand, and using zero-shot only when inference speed is an issue.
052
053   During this process, we also discovered an interesting property of ICL-transformers. Fine-tuned
      ICL-transformers can create complex decision boundaries, see Figure 1 for an example. Intuitively,

073   the complexity is determined by how far the decision boundary differs from a simple linear line.
074   Neural networks trained from scratch on tabular data often have overly simple decision boundaries,
075   a phenomenon known as simplicity bias (Shah et al., 2020), while tree-based methods do not suffer
076
      from this (Grinsztajn et al., 2022). We find that fine-tuned ICL-transformers, in contrast, are able to
      create these complex decision boundaries similar to tree-based methods.
077
078   Given this observation, we wonder whether pretraining on more complex data would improve the
079   fine-tuning performance. To this end, we introduce the novel forest dataset generator, which creates
080   highly complex synthetic datasets using a simple decision tree. By varying the parameters of the de-
081   cision tree, we can control the complexity of the generated datasets. We observe that TabForest, the
082
      ICL-transformer pretrained on this forest dataset generator, achieves better fine-tuning performance
      as the complexity of the generated datasets increases.
083
084   Furthermore, TabForest shows fine-tuning performance that surpasses TabPFN on specific real-
085   world datasets, even though the zero-shot performance of TabForest is significantly lacking com-
086   pared to TabPFN. This suggests that for the fine-tuning performance of some real-world datasets,
087   pretraining the ICL-transformer on highly complex datasets is more important than pretraining on
088
      realistic datasets, although for zero-shot, we would always prefer the TabPFN dataset generator.
089   As we would like to have a single ICL-transformer that performs well across all real-world tabular
090   datasets, we mix the TabPFN and forest dataset generators to pretrain TabForestPFN. This model
091   has excellent fine-tuning performance on two benchmarks (Grinsztajn et al., 2022; McElfresh et al.,
092   2023), matching the performance of either TabForest or TabPFN. At the same time, mixing in the
093   forest dataset generator does not seem to harm the zero-shot performance at all. This makes Tab-
094
      ForestPFN the preferred ICL-transformer over TabForest and TabPFN.
095   In conclusion, fine-tuned ICL-transformers are highly effective tabular data classifiers, capable of
096   creating complex decision boundaries. This new insight advances our understanding of tabular ICL-
097   transformers and opens up new avenues for further research to enhance their performance. With
098   further developments, we anticipate a significant shift in the field of tabular data, moving from tree-
099   based methods towards ICL-transformers.

101   2   R ELATED W ORKS
102
103   There are three main branches of tools for tabular data classification: classical statistical methods
104   like linear regression, K-nearest neighbors, Gaussian processes (Williams & Rasmussen, 1995),
105   and support vector machines (Hearst et al., 1998); tree-based algorithms like XGBoost (Chen &
106   Guestrin, 2016), CatBoost (Prokhorenkova et al., 2018), and LightGBM (Ke et al., 2017); and neu-
107   ral network-based methods such as the approach presented in this paper. There are several papers
      benchmarking the different methods (Gorishniy et al., 2021; Shwartz-Ziv & Armon, 2022; Grinsz-


                                                         2
      Under review as a conference paper at ICLR 2025


108
      tajn et al., 2022; McElfresh et al., 2023; Zabërgja et al., 2024). Overall, tree-based methods stand at
109   the top, with neural networks ranging from inferior to at best competitive.
110
111   Nonetheless, there have been numerous approaches that tackle tabular data classification with neu-
112
      ral networks. First, we have the class of neural networks trained from scratch: training starts from
      random initialized weights and is only trained on the data at hand. Research has focused on archi-
113
      tectures (Katzir et al., 2021; Somepalli et al., 2021; Arik & Pfister, 2021; Gorishniy et al., 2023;
114
      Huang et al., 2020; Chen et al., 2023a), embeddings (Ruiz et al., 2023; Gorishniy et al., 2022; Chen
115   et al., 2023b), and regularization (Shavitt & Segal, 2018; Kadra et al., 2021).
116
117   In general, methods training from scratch can struggle because tabular datasets can be small. So,
118
      researchers have sought ways to use large volumes of tabular data or to change the training objective.
      Some employ self-supervised learning (Kossen et al., 2021; Yoon et al., 2020; Zhu et al., 2023; Bahri
119
      et al., 2022; Ucar et al., 2021; Sui et al., 2023), or closely related transfer learning techniques (Nam
120
      et al., 2023; Levin et al., 2023; Zhou et al., 2023). Others leverage pretrained LLMs or language data
121   (Hegselmann et al., 2023; Zhang et al., 2023; Kim et al., 2024; Yan et al., 2024) to make predictions.
122
123   One of those related transfer learning methodologies is tabular in-context learning, a new field
124
      sparked by TabPFN (Hollmann et al., 2023). Currently, there is ongoing research on how to scale
      TabPFN to encompass more observations and features (Ma et al., 2023; Feuer et al., 2024; Thomas
125
      et al., 2024), as this architecture is limited by GPU memory. Our fine-tuning work can be seen as
126
      one approach to tackle this issue.

129   3     P RELIMINARIES
130
131   In tabular classification, we are interested in predicting targets y ∈ N given features x ∈ Rd , where
132
      d is the number of features. We predict y using an in-context learning (ICL) transformer pretrained
      on a synthetic dataset. The in-context learning allows the transformer to predict targets based on
133
      other observations included in the forward pass. In our work, zero-shot refers to one forward pass
134
      through the ICL-transformer without any fine-tuning, while fine-tuning refers to one forward pass
135   through the ICL-transformer after fine-tuning. Our work builds on TabPFN (Hollmann et al., 2023),
136   so below we explain their dataset generator and their transformer architecture.
137
138   3.1   TAB PFN DATASET G ENERATOR
139
140   The TabPFN authors create their own synthetic dataset using Bayesian Neural Networks (BNN) and
141   Structural Causal Models (SCM). To construct a dataset, they first create a BNN or a SCM with
142   random characteristics and with randomly initialized weights. Then they randomly draw an input
143   X and pass it through the model to generate output y. Their final dataset is given by (X, y). See
144   their paper (Hollmann et al., 2023) for more details.
145   In their approach, they emphasize their ability to create realistic datasets, and even call their gener-
146   ator a “prior”. They chose SCMs specifically because it can capture real-world causal mechanisms.
147   One other aspect they focus on is simplicity, biasing the generator towards less complex input-
148   output relationships. Additionally, they ensure the inputs X have natural correlations by correlating
149   the features blockwise, and they vary their feature importance by tuning the magnitude of weights
150   belonging to different features. These methods suggest the authors believe creating realistic datasets
151
      is important for achieving good performance.
152
153

      4.2   F INE - TUNING P ROCEDURE
250
251
      In our work, we introduce fine-tuning to the tabular ICL-transformer. When fine-tuning, we like to
252   draw support and query sets from our training data such that the performance generalizes to the test
253   set. This requires careful consideration of dataset splitting. The benchmark datasets already provide
254   us with a training-validation-test split. We use this validation dataset for early stopping.
255
      Every gradient descent step, we randomly draw a 80/20 support and query split from the training
256
      dataset. For validation, we draw the support set from the training set and draw the query set from
257
      the validation set. Every validation sample is seen exactly once, while the support samples are
258   randomly drawn with replacement. We use early stopping based on the validation loss, which is
259   calculated after every fine-tuning step.
260
261
      The early stopping technique can decide to stop fine-tuning immediately if the validation loss in-
      creases in the first step of training, which allows us to fall back on the zero-shot performance in case
262
      fine-tuning harms the performance. This is especially important when using very small datasets,
263
      as they are prone to overfitting. At the same time, fine-tuning can leverage all samples in training
264   dataset, while zero-shot cannot use more samples than that fit on the GPU.

        5.3                            C OMPLEXITY OF ICL-T RANSFORMERS ’ D ECISION B OUNDARIES
395
396     In the previous section, we have seen that TabForest and TabPFN both achieve excellent performance
397     as neural networks. Now we take a look at their decision boundaries. Repeating the analysis of the
398     WhyTrees’ authors (Grinsztajn et al., 2022), we use the Electricity dataset (OpenML ID: 44156) to
399     predict a binary target on two features.
400     To capture the complexity of the decision boundary, we define the complexity score V . We split the
401     feature space into a total of n grid cells where each cell has a predicted probability pij for grid cell
402     indices (i, j). The complexity score V is defined as the sum of absolute values between neighbor
403     cells:
404                       1X
                     V =         |pi+1,j − pij | + |pi−1,j − pi,j | + |pi,j+1 − pij | + |pi,j−1 − pij |
405                       n ij
406
407     The complexity score represents how fast the prediction changes when moving along the grid.
408
        We plot the results in Figure 1. We see that when fine-tuning, both TabPFN and TabForest can
409     create decision boundaries that are more complex than their zero-shot variants. The complexity
410     of the decision boundaries was one of the characteristics that explained why tree-based methods
411     outperformed neural networks (Grinsztajn et al., 2022). These results suggest ICL-transformers can
412     also create complexity in their decision boundaries.
413
        In our intuition, the ICL-transformer learns how to create these decision boundaries during pretrain-
414
        ing. We can interpret this from a weight initialization perspective. The weights of the pretrained
415     ICL-transformer provide a good initialization for the model to create complex decision boundaries,
416     while an ICL-transformer trained from scratch lacks this ability. For this reason, TabForest can
417     create decision boundaries of higher complexity than TabPFN.

        5.4                            A BLATION OF THE F OREST DATASET G ENERATOR

421     In Section 4.1, we discussed two ways to influence the complexity of the forest dataset generator:
422     the tree depth and the base size, which is the number of observations to fit the tree algorithm. We
423     expect the performance of TabForest to increase when the complexity of the forest dataset generator
424     increases.
425
        In Figure 5 we show the results of pretraining different settings of base size and maximum tree depth
426     on the WhyTrees benchmark. The tree depth is set to 1-25 as the base size changes, and the base size
427     is fixed to 1024 as the tree depth changes. When scaling up the base size from 2 to 32 and the tree
428     depth from 1 to 9, we observe that the performance increases, and stabilizes for higher complexities.
429     We provide figures of the data generated with these lower complexity hyperparameters in Appendix
430     A.10 to give an impression. The correlation between performance and complexity supports our
431     claim that learning complex decision boundaries is the driving force behind the performance of
        fine-tuned TabForest.

509   6                    C ONCLUSION
510
511   The introduction of TabPFN (Hollmann et al., 2023) has opened up a new field of in-context learn-
512   ing (ICL)-transformers for tabular data classification. Our research has demonstrated that fine-tuned
513   ICL-transformers achieve excellent performance and also learn to create complex decision bound-
514   aries. Furthermore, by adding the forest dataset to the pretraining mixture, we achieved performance
515   levels competitive with tree-based methods.
516   Despite these advancements, there are still obstacles for ICL-transformers to overcome if we want
517   them to replace tree-based methods in the realm of tabular data. One major challenge is the perfor-
518   mance limitation of ICL-transformers due to GPU memory constraints. Our work uses fine-tuning
519   as a solution to this problem, but it would be valuable to compare this approach to other concurrent
520   research such as prompt tuning (Feuer et al., 2024), in-context distillation (Ma et al., 2023) and
521   retrieval (Thomas et al., 2024). Moreover, our research focused solely on classification, although
522
      we expect that a simple switch from cross-entropy loss to mean-squared-error loss would suffice to
      tackle regression tasks. Another area that requires exploration is the setting with an exceptionally
523
      high number of features (Cherepanova et al., 2024), where the performance of ICL-transformers is
524
      unknown (McCarter, 2024). Lastly, tree-based methods can explain which features are important for
525   their predictions, and research is needed to determine if ICL transformers can achieve a similar feat
526   (Rundel et al., 2024). Overcoming these challenges will cement the ICL-transformer as the clear
527   successor to tree-based methods.

      A.1   E THICS AND S OCIAL I MPACT
705
706   Improving tabular data classification can provide major benefits to society. From medical to physics
707   applications, better performance can save lives and money. There are, however, also more nefar-
708   ious applications of tabular data classification, such as fraud risk detections based on ethnics or
709   nationality; population analysis for micro-targeting political ad campaigns; and insurance premium
710   discrimination based on underlying medical conditions.
711   Our models cannot detect the purpose for which the model is used. In contrast to large language
712   models, our tabular data models take numerical data as input. Ethnicity, gender, or other sensitive
713   information is represented by a class number. This means our models cannot recover the meaning
714   behind the numbers.
715
      A big benefit of this ’numerical anonymization’ is privacy. In our paper, we use synthetic data,
716
      which is completely privacy risk free. But even when pretrained on real data, recovering the original
717   data can be extremely hard due to the lack of labels and contextual information.
718
719   In light of the above, we decide to publish the model and open access to anyone. We do not know an
720
      effective way to create any form of safeguards against misuse, and we would welcome any advice
      from the research community that addresses this issue.

      Code is attached to this submission.

726   The code includes everything: downloading datasets, preprocessing result benchmarks, training

729   The code is built upon the works of Grinsztajn et al.; Hollmann et al. and McElfresh et al.. Although
730   this resulted in a Frankenstein monster of a codebase, we have made great efforts to rewrite most

      A.3   DATA P REPROCESSING
738
739   Before data is put into the neural network, the data is preprocessed. We use the exact same routine for
740   both synthetic data and real-world data to ensure minimal differences in distribution and summary
741   statistics of the input to the transformer. Algorithm 2 presents the procedure for preprocessing.
742
743
744   Algorithm 2 Data Preprocessing
745   Require: Xraw , yraw
746    1: Impute NaN features with column mean.
747    2: Remove features with one unique value.
748    3: Select a subset of a hundred features.
749    4: Transform all features to normal using quantile transformation.

      Because we fixed the input size of the neural network to df = 100 features, we first select a subset
757
      of a df features using scikit-learn’s SelectKBest Pedregosa et al. (2011). If there are less than df
758   features, we add zeros to ensure exactly df features. Following TabPFN, we scale by multiplying by
759   100/df ∗ , where df ∗ is the number of features after selecting a subset. To be robust to skewness and
760   outliers, we transform the data using scikit-learn’s QuantileTransformer to follow a normal distri-
761   bution. We make no distinction between numerical and categorical values in all our preprocessing.
762
      In comparison to TabPFN, our data preprocessing follows roughly the same scheme. One change is
763
      the use of a quantile transformer, while they use standard input or a power transformer. We consider
764   this a preference, both seem to work fine.
765
766   Furthermore, the TabPFN authors like to ensemble outputs of the TabPFN architecture, varying the
767
      transformation function between standard input and power transformer. In our paper, we use no
      ensembling at all.

      A.4    T RAINING S ETTINGS
771   All ICL-transformer architectures, including the original TabPFN, use the same model. The model
772   consists of 12 layers, 4 attention heads, a hidden dimension of 512, and 10 classes as output di-
773   mension. Pretraining uses batch size 512, learning rate 1e-4, weight decay 0.00, AdamW optimizer
774   with betas (0.9, 0.95), cosine scheduling, and maximum global gradient norm 1.0. Fine-tuning
775   is performed under batch size 1, learning rate 1e-5, weight decay 0.00, no scheduling, with early
776   stopping, and a maximum of 300 steps. To fit the model on an RTX 3090 with 24GB during fine-
777   tuning, we set the maximum support size to 8192 samples and the maximum query size to 1024.
778   Pre-training uses data generated with maximum support size 1024 and maximum query size 128.
779
      We choose these settings because the maximum support size affects performance, but the maximum
      query size only affects inference speed. On these settings, we train all ICL-transformers for 50,000
780
      steps, which takes 4 GPU-days on one H100. Running all benchmarks takes one additional H100
781
      GPU-day.
