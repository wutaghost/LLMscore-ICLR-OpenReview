080   In this paper, we introduce the Recurrent Drafter (ReDrafter), for fast LLM inference. Figure 2
081   illustrates its generative process. ReDrafter’s performance gains are driven by three key aspects:
082   (1) Using a recurrent neural network (RNN) (Mikolov et al., 2010) conditioned on the LLM’s hid-
083   den states as the draft model harnesses local temporal dependencies, improving the accuracy of the
084   drafter’s predictions and effectively converting computational resources into speedups. (2) By uti-
085   lizing beam search to explore multiple candidate sequences and applying a dynamic tree attention
086   algorithm to eliminate duplicated prefixes among the candidates, we significantly reduce computa-
087   tional overhead. (3) Training through knowledge distillation (Zhou et al., 2023) from LLMs im-
088
      proves the alignment of the draft model’s predictions with those of the LLM, effectively transferring
      the computational load from inference time to training time.
089
090   In a PyTorch implementation, ReDrafter accelerates Vicuna inference by up to 3.5x compared to
091   the autoregressive method in MT-Bench on Nvidia H100 GPUs, achieving state-of-the-art perfor-
092   mance. Additionally, we demonstrate the effectiveness of ReDrafter in production environments
093   through two key use cases. The first is a production-level deployment on TensorRT-LLM, designed
094
      to manage high traffic with long context lengths. In this implementation, tensor parallelism and con-
      tinuous batching are utilized to ensure the system efficiently handles larger volume of requests while
095
      maintaining low latency. In this scenario, ReDrafter achieves up to 2.5x speedup on the MT-bench
096
      benchmark. The second use case focuses on an on-device approach using MLX on Metal GPUs
097   within Apple Silicon chips. Despite the limited compute resources in this setup, we observed a
098   memory bottleneck. ReDrafter effectively mitigates this bottleneck, resulting in up to 2.3x speedup,
099   demonstrating its capability to optimize performance in resource-constrained environments.

      Beam search returns draft candidates with shared prefixes. For examples, in Figure 4(a), the second
202
      candidate “morning sipping coffee and watching” and the third candidate “morning sipping coffee
203
      on her” share a prefix “morning sipping coffee”. Sending those duplicated tokens to LLM result
204   in computation overhead. As a result, we remove those shared prefixes, revealing a tree structure
205   over beam search results. When sending de-duped candidate to LLM for verification, we modify
206   the attention mask to reveal dependencies among tokens (Figure 4(b)). We refer this mechanism as
207   “dynamic tree attention”.
208
      The use of a tree structure to save computation resembles approaches seen in Miao et al. (2023);
209
      Spector & Re (2023); Cai et al. (2024), where similar methods were employed to avoid redundant
210   calculations for shared prefixes. However, unlike the use of tree structures mentioned above, we
211   must determine ours dynamically as it relies on individual beam search results at runtime. A standard
212   trie-based algorithm is slow as it is difficult to be parallelized. We notice that a unique property of our
213   problem is that all candidate sequences have the same length. With this insight, we discovered that
214   our dynamic tree attention can be efficiently constructed using standard tensor operators, a critical
215   aspect for leveraging modern-day accelerators with little overhead. This leads to further acceleration
      of LLM inference in speculative decoding.

      The key observation is that we can find all shared prefixes in the beam search result using standard
217   tensor operations without building a prefix trie. We have developed functions called dedup_prefix
218   to identify shared prefixes, and another function pack_beam to compress a beam into a deduped,
219   “packed beam” (illustrated in Figure 4(c)). We provide the pseudo-code for dedup_prefix in the
220   Appendix A.1 to provide a glimpse of how this tensor-based algorithm works. Subsequent opera-
221   tions can be similarly designed to use the prefixes found here to construct dynamic tree attention.
222   This leads to further acceleration of LLM inference in speculative decoding. The use of dynamic tree
223   attention is not limited to ReDrafter. It can also be used in detached speculative decoding approaches
224   while a separate draft model performs beam search and then apply dynamic tree attention.

226   3.4   S PECULATIVE DECODING WITH R E D RAFTER
227
      We briefly describe the steps of using ReDrafter for speculative decoding. In each generation step
228
      during inference, ReDrafter alternates between using the draft model to generate tokens and calling
229
      the LLM to verify and accept them. We start with all previously generated tokens with the last
230   hidden state. We employ beam search to generate a beam, consisting of a set of candidate sequences.
231   Subsequently, dynamic tree attention is applied to flatten and compress the beam into a packed
232   beam, while formulating an appropriate attention mask. LLM then proceeds with a forward pass
233   to compute the log probabilities for all proposed tokens. Then, we select the best candidate with
234   the longest accepted prefix. The selection method can range from a greedy approach (aka. token
235   matches), to rejection sampling. Simultaneously, LLM provides hidden states and the initial token
236   for the next draft model call. We append accepted tokens to the end of previously generated tokens
237   and run the next iteration until the stopping criteria are met. ReDrafter guarantees the generated
238
      sequence matches LLM’s output.

424
      and reduce the number of LLM calls, the optimal speedup is achieved at lower beam widths. For
425
      example, the best performance occurs at BW=1 on the M1 Max for the 7B model, and BW=3 on

428   GPU can process these FLOPs in parallel, once the computation cost reaches the hardware’s limit,
429   the performance gain from speculative decoding diminishes. This explains why the optimal speedup
430   for the M1 Max is achieved at BW=1, whereas for the M2 Ultra, it occurs at BW=3, highlighting
431   the greater computational power of M2 Ultra. However, performance declines at BW=4 for both
      devices due to the increased cost of verifying draft tokens.

      4.4.1         B EAM W IDTH AND BATCH S IZE
454   In ReDrafter, beam width and batch size are two critical factors for leveraging redundant computa-
455   tional resources. While increasing either may push the system to its computational limits, they im-
456   pact the system differently: a larger beam width raises the likelihood of generating draft sequences
457   that are more likely to be accepted, whereas a larger batch size improves overall system throughput.
458
      We examine the trade-offs between beam width and batch size by conducting a grid search on both
459
      parameters on an Nvidia H100 GPU using MT-Bench, with Vicuna-7B as the LLM. We measured
460   tokens-per-second per request (TPS) to assess latency (the inverse of latency) and tokens-per-second
461   per request multiplied by batch size (TPS×BSZ) to evaluate overall system throughput. The results
462   are summarized in Table 5.
463
      When beam width is held constant and batch size increases, TPS initially remains stable but even-
464
      tually declines, while TPS×BSZ continues to rise. This shows that increasing batch size improves
465
      GPU utilization and enables more efficient batch processing, despite fewer resources being allocated
466   to each individual request. The TPS drop occurs earlier at higher beam widths due to the additional
467   computational cost. At a batch size of 80, out-of-memory (OOM) errors occur at larger beam widths,
468   indicating limited memory capacity.
469
      The optimal configurations for per-request TPS and TPS × BSZ vary. The highest per-request TPS,
470
      around 110, is achieved with a beam width of 64 and a batch size of 1 or 2, while the highest
471
      TPS×BSZ, approximately 1636, is reached with a beam width of 2 and a batch size of 80. This
472   underscores the importance of tuning beam width and batch size based on specific use cases. For
473   scenarios prioritizing low latency, a larger beam width with a smaller batch size is recommended.
474   Conversely, if high throughput is the main goal, a larger batch size paired with a moderate beam
475   width is more effective.

      A.1     GPU- FRIENDLY DYNAMIC TREE ATTENTION IMPLEMENTATION
651
652   In Listing 1, we demonstrate the GPU-friendly implementation of the function dedup_prefix
653   from Section 3.3 with five lines of code. We assume batch size equals to 1 in this example for
654   brevity. Extending to batch size greater than 1 is straight-forward by pre-pending an extra batch size
655   dimension in tensors.
656   The function processes the initial tensor beam, generating the prefix_tree tensor as its output. In
657   this tensor, prefix_tree[i][j]=k indicates that the candidate at the smallest index k shares an
658   identical prefix beam[i][:j+1]. For instance, prefix_tree[2][1]=0 signifies a shared prefix
659   “morning sipping” between beam[0] and beam[2] from the example in Figure 4. It is possible to
660   condense tokens where prefix_tree[i][j]<i.
661
662                       Listing 1: An example inplementation for dedup prefix.
663   def dedup_prefix(beam):
664       """For each prefix in each candidate, find the smallest candidate
665           index that shares the same prefix.
666
            Args:
667
                - beam: (beam_width, beam_length) input beam.
668         Returns:
669             - prefix_tree: [beam_width, beam_length] prefix_tree[i][j]=k
670                 indicates that candidate sequences i and k in share the same
671                 prefix, or, in other words, beam[i][:j+1]== beam[k][:j+1]
            Examples:
672
                beam = tensor([[91, 92, 93, 95],
673                            [91, 92, 94, 96],
674                            [91, 92, 93, 97]])
675             prefix_tree = tensor([[0, 0, 0, 0],
676                                   [0, 0, 1, 1],
                                      [0, 0, 0, 2]]
677
            """
678         beam_length = beam.shape[1]
679         prefix_target = torch.arange(1, beam_length+1)
680         # Build a square boolean matrix matches.
681         # If matches[i][j][k]==True, then the k-th token of the i-th sequence
                 is the same as the k-th token in the j-th sequence.
682
            # So, i and j are in range [0,beam_width), k in [0,beam_length).
683         matches = beam[:, :, None]==beam[:, None, :]
684         seq_matches = (torch.cumsum(matches, dim=2)
685             == prefix_target[None, None, :])
686         # The previous candidate with smallest index that shares the same
                prefix.
687
            prefix_tree = torch.argmax(seq_matches, dim=2)
688         return prefix_tree

694   A.2.1    O PTIMAL S PEEDUP AND GPU P OTENTIAL
695
696   To gain detailed insights into the behavior of Recurrent Drafting on Apple Silicon chips, we need to
697   analyze the speedup in relation to two key inference-time parameters: beam width and beam length
698
      in the drafter beam search.
699   The parameter beam width refers to the number of draft token sequences. A larger beam width
700   increases the likelihood that the sequence with the longest acceptable prefix is included in the beam.
701   This allows the LLM to accept more tokens in each decoding step, reducing the overall number of
      decoding steps required—or, equivalently, decreasing the number of calls to the LLM.


                                                       13
      Under review as a conference paper at ICLR 2025


702
      However, a wider beam requires more FLOPs for the LLM to verify the draft tokens. While a
703   powerful GPU can process these FLOPs in parallel, minimizing the increase in wall time, it also
704   implies higher power consumption and computational cost.
705
706
      The parameter beam length refers to the number of steps in the beam search algorithm, the number
      of calls to the RNN draft model, or equivalently, the length of the draft token sequences in the
707
      beam. A longer beam increases the length of the prefix that can be accepted, reducing the number
708
      of decoding steps or calls to the LLM. However, this length is constrained by the token sequence
709   length used during the training of the RNN draft model.
710
711   Figure 7 illustrates the speedup in relation to beam width and beam length on the M1 Max and M2
712
      Ultra. We can observe that optimal speedup (colored in yellow) is achieved when the beam length
      is close to the length of the training sequences used for the RNN draft model. This maximizes the
713
      predictive power of the RNN. However, the optimal speedup is not necessarily achieved at the exact
714
      training length of 5; it can occur at a slightly shorter length, such as 4, since the RNN may not
715   always accurately predict the 5th token.
716
717   We also observe that the optimal speedup is achieved with narrower beams—1 for the M1 Max and
718
      3 for the M2 Ultra. In contrast, our experiments on A100 and H100 GPUs using PyTorch show that
      the optimal speedup is achieved with beam widths of 50 or more. This discrepancy is due to the
719
      inherent performance gap between server-grade GPUs and mobile GPUs, which is expected. It also
720
      explains why the M2 Ultra can handle a wider optimal beam of 3, compared to the M1 Max’s 1, as
721   the M2 Ultra is equipped with a more powerful GPU.

740   A.2.2    P ERFORMANCE T UNING : MLX AND A PPLE S ILICON
741
742   Before starting the MLX implementation, we had experience with PyTorch and TensorRT-LLM.
743   However, working on our first MLX project revealed that many lessons we learned from program-
744   ming CUDA no longer apply to MLX and Apple Silicon. Below, we share some of the key lessons
745
      learned from this journey.
746   Use Native Dtype and Low-Bits
747
      Our benchmark code explores various factors, including data types (dtype). From our exploration,
748
      running autoregression and recurrent drafting in float16 is consistently faster than in bfloat16.
749   We also observed that both float16 and bfloat16 outperform float32, as they use less memory
750   access bandwidth. Similarly, as reported by other studies, 4-bit quantization significantly accelerates
751   MLX programs compared to float16.
752
      MLX Does Lazy Evaluation
753
754   While PyTorch programs execute eagerly, MLX programs run lazily. In MLX, the Python code
755   may appear to be executing tensor operations sequentially, but these operations might not actually
      run until the final result is printed. This lazy execution extends beyond numerical operations; even


                                                        14
      Under review as a conference paper at ICLR 2025


756
      loading model parameters from the filesystem into tensors in memory is done lazily. To ensure that a
757   model is fully loaded before benchmarking the inference algorithm, you must call mlx.core.eval
758   (model.parameters()) before invoking the model.
759
760
      Don’t Break the JIT Compilation
761   Lazy evaluation in MLX allows it to silently trace tensor operation calls and compile these traces
762   just-in-time into Metal GPU kernels for improved runtime performance. While this provides conve-
763   nience, it can also influence how we program. For example, during a comparison of the execution
764   time between MLX functions and their PyTorch counterparts, we noticed that one MLX function
765   consumed a disproportionately large fraction of the total running time, whereas its PyTorch equiv-
766
      alent did not. We discovered that this function involved numerous calls to array.item(). MLX
      had to compile the Python code before each of these calls, causing the code to be split into many
767
      segments, which significantly slowed down the execution time.
768
769   Measure Performance in All Levels of Details
770   The Instruments app, included with Xcode, provides visualization of CPU threads and Metal GPU
771   operations, similar to NVIDIA Nsight. This tool helps developers gain a general understanding of
772   potential bottlenecks in their code. to capture detailed performance metrics, we developed custom
773   utilities to measure and log the execution time of individual functions.
