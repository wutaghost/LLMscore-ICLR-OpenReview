009                                               A BSTRACT
010
011
               Automated machine learning (AutoML) accelerates AI development by automat-
012
               ing tasks in the development pipeline, such as optimal model search and hyper-
013            parameter tuning. Existing AutoML systems often require technical expertise to
014            set up complex tools, which is in general time-consuming and requires a large
015            amount of human effort. Therefore, recent works have started exploiting large lan-
016            guage models (LLM) to lessen such burden and increase the usability of AutoML
017            frameworks via a natural language interface, allowing non-expert users to build
018            their data-driven solutions. These methods, however, are usually designed only
019            for a particular process in the AI development pipeline and do not efficiently use
020            the inherent capacity of the LLMs. This paper proposes AutoML-Agent, a novel
021
               multi-agent framework tailored for full-pipeline AutoML, i.e., from data retrieval
               to model deployment. AutoML-Agent takes user’s task descriptions, facilitates
022
               collaboration between specialized LLM agents, and delivers deployment-ready
023
               models. Unlike existing work, instead of devising a single plan, we introduce a
024            retrieval-augmented planning strategy to enhance exploration to search for more
025            optimal plans. We also decompose each plan into sub-tasks (e.g., data prepro-
026            cessing and neural network design) each of which is solved by a specialized agent
027            we build via prompting executing in parallel, making the search process more
028            efficient. Moreover, we propose a multi-stage verification to verify executed re-
029            sults and guide the code generation LLM in implementing successful solutions.
030            Extensive experiments on seven downstream tasks using fourteen datasets show
031            that AutoML-Agent achieves a higher success rate in automating the full AutoML
032
               process, yielding systems with good performance throughout the diverse domains.

      Automated machine learning (AutoML) has significantly reduced the need for technical expertise
038
      and human labors in developing effective data-driven solutions by automating each process in the
039   AI development pipeline (Yao et al., 2018; Ren et al., 2020; He et al., 2021), such as feature engi-
040   neering, model selection, and hyperparameter optimization (HPO). However, current AutoML sys-
041   tems (Gijsbers et al., 2024) often necessitate programming expertise to configure complex tools and
042   resources, potentially creating barriers for a larger pool of users with limited skills and knowledge.
043
      To make AutoML frameworks more accessible, recent studies (Trirat et al., 2021; Viswanathan et al.,
044
      2023; Li et al., 2023; Hollmann et al., 2023b; Liu et al., 2024a; Zhang et al., 2023; Shen et al., 2023;
045   Zhang et al., 2024a; Hong et al., 2024a; Guo et al., 2024a; Yang et al., 2024) have suggested to use
046   natural language interfaces with large language models (LLM) for machine learning (ML) and data
047   science (DS) tasks. Nevertheless, these previous LLM-based AutoML frameworks only considered
048   a limited number of tasks due to their restricted designs, either only for a process in the pipeline
049   (e.g., feature engineering (Hollmann et al., 2023b; Li et al., 2024; Malberg et al., 2024), HPO (Liu
050   et al., 2024a; Zhang et al., 2024a), and model selection (Zhang et al., 2023; Shen et al., 2023)) or for
051   a specific group of downstream tasks (e.g., natural language processing (Viswanathan et al., 2023)
052   and computer vision (Yang et al., 2024)). In addition, most methods overlook the inherent capability
053
      of LLMs to search for promising models by performing actual training of the candidate models
      during the search process, making it prohibitively costly and slow.

068   High Complexity of the Planning Tasks The planning of the entire AutoML pipeline introduces
069   additional complexities compared to task- or problem-specific planning, primarily due to the inter-
070   dependencies among the steps in the pipeline. For example, types of retrieved datasets affects how to
071   design preprocessing steps and neural networks. Then, the designed network affects which particular
072
      hyper-parameters need to be optimized depending on the given downstream task. Such inter-step
      dependencies result in the enlarged search space since it should consider all possible combinations
073
      of inter-related steps. Furthermore, enabling the framework to operate across various downstream
074
      tasks exacerbates these challenges, as each has task-specific requirements.
075
076   Challenges in Accurate Implementations To develop a modular and extendable framework that
077   effectively handles diverse ML tasks, it is crucial to enhance the flexibility of the LLM agent in its
078   code generation ability, such as by decoupling the template code from the code for specific datasets.
079   However, using LLMs to autonomously generate complete ML pipelines may lead to hallucination
080
      issues, including code incompletion, incorrect or missing dependencies, and potential undiscovered
      bugs (Hong et al., 2024b). Furthermore, LLMs often struggle with code generation when prompted
081
      with ambiguous task descriptions. Thus, we need accurate analysis of the requirements, and a code-
082
      generation platform that can adaptively generate code based on disambiguated requirements.
083
084   To address the above challenges, we propose a novel multi-agent framework, AutoML-Agent, for
085   full-pipeline AutoML from data and model search to evaluation, with strategies to tackle the com-
086
      plexity of the planning problem as well as accurate implementation of code. As illustrated in Fig-
      ure 1, AutoML-Agent accepts a user’s task description and coordinates multiple specialized agents
087
      to collaboratively identify and execute an optimal ML pipeline, ultimately delivering a deployment-
088
      ready model and its inference endpoint as the output.
089
090   Specifically, to tackle the complex planning problem, we introduce a new retrieval-augmented
091   planning strategy equipped with role-specific decomposition and prompting-based execution. This
092
      strategy produces multiple plans based on retrieved knowledge for a given task description, fa-
      cilitating the exploration of promising plans. Moreover, it enables LLM agents to discern global
093
      (pipeline-level) and local (process-level) relationships among steps through plan decomposition,
094
      which helps them focus on their immediate sub-tasks while aligning with the user’s goal. The
095   retrieval-augmented component also simplifies extending LLMs to various downstream tasks us-
096   ing relevant knowledge. The prompting-based execution enhances search efficiency by exploiting
097   LLMs’ in-context learning capabilities without any further training, which could introduce addi-
098   tional cost. To enhance the accuracy of the implementation, we adopt structure-based prompt pars-
099   ing that extracts ML-relevant requirements from the user’s description and multi-stage verification
100   that provides feedback between each step in the framework to ensure the quality of instructions when
101   guiding the LLM for code generation. These modules aim to improve the correctness and clarity of
102   the task description for code implementation. Our main contributions are as follows.
103   • We propose a novel multi-agent LLM framework for AutoML, designed to automate the entire AI
104     development pipeline. To the best of our knowledge, this is the first attempt to employ LLMs in a
105     task-agnostic AutoML framework that spans from data retrieval to model deployment.
106   • We address the challenges due to the complexity of the planning problem in full-pipeline Au-
107     toML by introducing retrieval-augmented planning with role-specific plan decomposition and
        prompting-based plan execution, enhancing the flexibility and efficiency of the search process.

127   2     R ELATED W ORK
128
129   Automated machine learning (AutoML) is a transformative approach for optimizing ML workflows,
130   enabling both practitioners and researchers to efficiently design models and preprocessing pipelines
131   with minimal manual intervention (Ren et al., 2020; He et al., 2021; Gijsbers et al., 2024). Despite
132   several advancements in AutoML (Jin et al., 2019; Feurer et al., 2022; Tang et al., 2024), most of
133
      them are designed only for particular elements of the ML pipeline. Only a few works (Bisong, 2019;
      Mukunthu et al., 2019; Microsoft, 2021) support multiple steps of the pipeline. Also, due to the
134
      traditional programming interfaces, these systems often have complex configuration procedures and
135
      steep learning curves that require substantial coding expertise and an understanding of the under-
136   lying ML concepts, limiting their accessibility to non-experts and being time-consuming even for
137   experienced users. These limitations hinder the widespread adoption of traditional AutoML systems.
138
139
      Large language models (LLM), e.g., GPT-4 (Achiam et al., 2023) and LLaMA (Touvron et al., 2023),
      have recently shown promise in addressing these limitations with the complex problem-solving skills
140
      across disciplines via human-friendly language interfaces, including AI problems (Xi et al., 2023).
141
      This shift towards natural language-driven interfaces democratizes access and allows users to artic-
142   ulate their needs in a more intuitive manner. However, existing LLM-based frameworks can only
143   assist in a specific step of the ML pipeline, such as feature engineering (Hollmann et al., 2023b),
144   model search (Shen et al., 2023; Hong et al., 2024a; Guo et al., 2024a), or HPO (Liu et al., 2024a;
145   Zhang et al., 2024a). A few attempts (Viswanathan et al., 2023; Yang et al., 2024) support the entire
146   ML production pipeline, yet only for a specific type of downstream tasks. Besides, these methods
147   either naively use the LLMs or overlook the inherent capabilities, making their search processes
148   time-consuming for the AutoML pipeline that requires sophisticated planning and verification.
149   In contrast to the existing studies, our framework aims to overcome these limitations by incorporat-
150   ing a new retrieval-augmented planning strategy, coupled with plan decomposition and prompting-
151   based execution techniques, alongside structure-based prompt parsing and multi-stage verification.
152   Through these enhancements, we can increase plan execution efficiency and support diverse ML
153   tasks with more accurate pipeline implementation. Table 1 summarizes the key differences between
154   AutoML-Agent and existing frameworks.

183   3.1                      AGENT S PECIFICATIONS
184
185   We now provide brief descriptions of the agents in our multi-agent AutoML framework.
186
      Agent Manager (Amgr ) acts as the core interface between users and other LLM agents in the frame-
187
      work. It is responsible for interacting with the user, devising a set of global plans for subsequent
188   processes with retrieved knowledge, distributing tasks to corresponding agents, verifying executed
189   results with feedback, and tracking the system progress.
190
191
      Prompt Agent (Ap ) is an LLM specifically instruction-tuned for parsing the user’s instructions into
      a standardized JSON object with predefined keys. The parsed information is then shared across
192
      agents in the framework during the planning, searching, and verification phases.
193
194   Data Agent (Ad ) is an LLM prompted for doing tasks related to data manipulation and analysis. The
195   analysis results from the Data Agent are used to inform the Model Agent about data characteristics
196   during the model search and HPO.
197   Model Agent (Am ) is an LLM prompted for doing tasks related to model search, HPO, model
198   profiling, and candidate ranking. The results produced by the Model Agent are sent back to the
199   Agent Manager for verification before proceeding to the Operation Agent.
200
      Operation Agent (Ao ) is an LLM prompted for implementing the solution found by the Data and
201   Model Agents that passes the Agent Manager’s verification. The Operation Agent is responsible
202   for writing effective code for actual runtime execution and recording the execution results for final
203   verification before returning the model to the user.
204
      After we define all agents with their corresponding profiles as described above (see §B.1 for detailed
205
      prompts), the Amgr then assigns relevant tasks to each agent according to the user’s input. Note that
206
      we can implement Ad and Am with more than one agent per task based on the degree of parallelism.

255   3.3   I NSTRUCTION DATA G ENERATION AND P ROMPT PARSING
256
257   Data Generation For Ap to generate accurate JSON objects, we need to instruction-tune the LLM
258   first because it can output a valid JSON object but with incorrect keys that are irrelevant to subse-
259   quent processes. Following Xu et al. (2024), we first manually create a set of high-quality seed
260   instructions then automatically generate a larger instruction dataset D = {(Ii , Ri )}Ni=1 , having N
261   instruction-response pairs. Here, Ii is the i-th instruction with the corresponding response Ri . We
262   use the JSON format substantially extended from Yang et al. (2024) for response Ri with the fol-
263
      lowing top-level keys to extract the user’s requirement from various aspects of an AutoML pipeline.
264   • User. The user key represents the user intention (e.g., build, consult, or unclear) of the given
265     instruction and their technical expertise in AI.
266   • Problem. The problem key indicates the characteristics and requirements of the given task, in-
267     cluding area (e.g., computer vision), downstream task (e.g., image classification), application or
268     business domain, and other constraints like expected accuracy and inference latency.
269   • Dataset. The dataset key captures the data characteristics and properties, including data modality,
        requested preprocessing and augmentation techniques, and potential data source.


                                                         5
      Under review as a conference paper at ICLR 2025


270
      • Model. The model key captures the expected model characteristics and properties, including
271     model name (e.g., ViT), family (e.g., Transformer), and type (e.g., neural networks).
272   • Knowledge. The knowledge key extracts additional knowledge or insights helpful for solving the
273     given problem directly provided by the user, potentially associated with the expertise level.
274   • Service. The service key is relevant to the downstream implementation and deployment. It pro-
275     vides information such as a target device and an inference engine.
276
277   Prompt Parsing Then, we can use the generated dataset D to train an LLM and use it as Ap .
278   Note that these standardized keys are important for a better control over the LLM agents’ behavior
279   within our framework and necessary for effective communication between agents. Moreover, these
280   keys provide contextual information for generating a high-quality AutoML pipeline from various
281   perspectives. After the instruction tuning, we use the Ap to parse the user’s instructions (or task
282
      descriptions) and return the parsed requirements R = Ap (I) to Amgr , as shown in §C.1.

284   3.4   R ETRIEVAL -AUGMENTED P LANNING
285
      Recent studies (Guo et al., 2024b; Huang et al., 2024; Masterman et al., 2024; Zhang et al., 2024b;
286
      Hu et al., 2024) highlights that effective planning and tool utilization are essential for solving com-
287
      plex problems with LLMs, especially in a multi-agent framework. By bridging two techniques in
288   a single module, we propose a retrieval-augmented planning (RAP) strategy to effectively devise a
289   robust and up-to-date set of diverse plans for the AuotML problems.
290
291
      Let P = {p1 , . . . , pP } be a set of plans. Based on past knowledge embedded in the LLM, knowl-
      edge retrieved via external APIs (such as arXiv papers), and R, RAP generates P multiple end-
292
      to-end plans for the entire AutoML pipeline having different scenario pi . This strategy enables
293
      AutoML-Agent to be aware of newer and better solutions. Specifically, we first use the parsed re-
294   quirements R to acquire a summary of the relevant knowledge and insights via API calls, including
295   web search and paper summary. Amgr then uses this information to devise P different plans, i.e.,
296   P = Amgr (RAP(R)). Note that Amgr devises each plan independently to make the subsequent
297   steps parallelizable. The benefit of this strategy is that it enhances exploration for better solutions
298   while allowing parallelization. Examples of generated plans are provided in §C.2.
299
300   3.5   P ROMPTING -BASED P LAN E XECUTION AND I MPLEMENTATION
301
302   Given the generated P, we now describe how Ad and Am execute each pi using prompting tech-
303   niques without actually executing the code. Examples of the execution results are in §C.4.
304
305   Plan Decomposition Due to the high complexity of the end-to-end plan, we first need to adap-
306   tively decompose the original plan pi ∈ P into a smaller set of sub-tasks si relevant to the agent’s
307   roles and expertise to increase the effectiveness of LLMs in solving and executing the given plan
308
      (Khot et al., 2023). The plan decomposition (PD) process involves querying the agents about their
      understanding of the given plan specific to their roles. Formally, sdi = PD(R, Ad , pi ), where sdi is
309
      the decomposed plan for Data Agent, containing sub-tasks for the given plan pi . Then, the agent
310
      executes the decomposed plan towards the user’s requirements instead of the original lengthy plan.
311   We define the sub-tasks smi of Am below due to its reliance on Data Agent’s outcomes. Examples
312   of decomposed plans are in §C.3.
313
314   Pseudo Data Analysis In AutoML-Agent, Ad handles sub-tasks in sdi , including the retrieval, pre-
315   processing, augmentation, and analysis of the specified dataset. During the data retrieval phase, if
316   the dataset is not directly supplied by the user, we initiate an API call to search for potential datasets
317   in repositories, such as HuggingFace and Kaggle, using the dataset name or description. Upon lo-
318   cating a dataset, we augment the prompt with metadata from the dataset’s source; if no dataset is
319   found, we rely on the inherent knowledge of the LLM. We then prompt Ad to proceed by acting
320   as if it actually executes sdi , according to the dataset characteristics and user requirements R. The
321   summarized outcomes of these sub-tasks, Oid , are then forwarded to the Am . Hence, Oid = Ad (sdi ).

339   3.6   M ULTI -S TAGE V ERIFICATION
340
      Verification, especially with refinement or feedback, is essential for maintaining the correct trajec-
341
      tory of LLMs (Baek et al., 2024; Madaan et al., 2023; Gou et al., 2024). Our framework incorporates
342
      three verification steps to guarantee its accuracy and effectiveness: request verification, execution
343   verification, and implementation verification.
344
345
      Request Verification Initially, we assess the clarity of the user’s instructions to determine if they
346   are relevant and adequate for executing ML tasks and addressing the user’s objectives. If the in-
347   structions prove insufficient for progressing to the planning stage, Amgr will request additional in-
348   formation, facilitating multi-turn communication. This request verification (ReqVer in Algorithm 1
349   Line 3) step, however, often overlooked in existing studies, placing an undue burden on users to for-
350   mulate a more detailed initial prompt—a challenging task particularly for those who are non-experts
351   or lack experience. Prompts for ReqVer are shown in §B.4.1.
352
353   Execution Verification After executing the plans in §3.5, Amgr then verifies whether any of the
354   pipelines produced by Ad and Am (i.e., O) satisfy the user’s requirements via prompting (see
355   §B.4.2). If the results are satisfied, the suggested solution is selected as a candidate for imple-
356   mentation. This execution verification (ExecVer) step effectively mitigates computational overhead
357
      in the search process by allocating resources exclusively to the most promising solution.
358
359   Implementation Verification This implementation verification (ImpVer) phase closely resembles
360
      the execution verification; however, it differs in that it involves validating outcomes derived from
      the code that has been executed and compiled by Ao . We present the prompt for this verification
361
      in §B.4.3. If the outcomes meet the user’s requirements, Amgr provides the model and deployment
362
      endpoint to the user.
363
364   Note that if any execution or implementation fails to satisfy the user requirements (i.e., does not pass
365   the verification process), these failures are systematically documented. Subsequently, the system
366
      transitions to the plan revision stage. During this stage, Amgr formulates a revised set of plans,
      incorporating insights derived from the outcomes of the unsuccessful plans.

391   Evaluation Metrics For a comprehen-
392   sive evaluation, we measure the agent’s effectiveness in both code generation and task-specific per-
393   formance aspects by using comprehensive score (CS) (Hong et al., 2024a) to simultaneously evaluate
394   both the success rate (SR) of code generation and the normalized performance score (NPS) of the
                                                                                1
395   built pipelines. That is, CS = 0.5 × SR + 0.5 × NPS. Here, NPS = 1+s         is a transformation of
396   loss-based performance score s, e.g., RMSLE. More detailed explanations are included in §A.4.
397   As described above, we evaluate all frameworks under two different settings. To measure SR of each
398   method, we use a grading scale ranging from 0 for total failure to 1 for perfect conformity to the
399   user’s requirements. For the constraint-free setting, a method can get a score of 0.5 (pass modeling)
400   or 1.0 (pass deployment). For the constraint-aware setting, a method can get a score of 0.25 (pass
401   modeling), 0.5 (pass deployment), 0.75 (partially pass the constraints), or 1.0 (pass all cases).
402
403   Baselines As we propose a framework for the novel task of full-pipeline AutoML with LLMs,
404   there is no direct baseline available for comparison. We thus compare AutoML-Agent against the
405   task-specific manually designed models (see §A.3): Human Models, the variants of state-of-the-art
406   AutoML: AutoGluon (Erickson et al., 2020; Shchur et al., 2023; Tang et al., 2024), a state-the-of-art
407   LLM for data science: DS-Agent (Guo et al., 2024a), and general-purpose LLMs: GPT-3.5 (Brown
408   et al., 2020) and GPT-4 (Achiam et al., 2023) with zero-shot prompting.

496   4.4   A BLATION AND H YPERPARAMETER S TUDIES
497
498   To validate the effectiveness of each component in AutoML-Agent, we conduct the following ab-
499   lation studies. The results are presented in Figure 5a and Table 8. First, we investigate retrieval-
500   augmented planning (RAP) alone, where retrieved knowledge from external APIs is directly used
501   without plan decomposition and multi-stage verification. As expected, this ablation leads to a de-
502   cline in performance, and in some cases, even fails to generate a runnable model. This outcome
503
      highlights the importance of the decomposition and verification modules. Second, we evaluate RAP
      with plan decomposition, where the plan is decomposed for each specific agent. While this variant
504
      demonstrates better downstream performance, it still fails to produce runnable models in certain
505
      downstream tasks due to the lack of code verification. Finally, we assess the full framework with
506   multi-stage verification, which provides feedback to agents, thereby enhancing both their planning
507   and coding capabilities. Integrating all components significantly empowers LLM agents to effec-
508   tively incorporate external knowledge from various sources to build a full-pipeline AutoML system.
509
      To further verify the effectiveness of devising multiple plans in our retrieval-augmented planning
510
      strategy (§3.4), we conduct a hyperparameter study by varying the number of plans P in the
511
      constraint-free setting. As shown in Figure 5b and Table 9, the number of plans does not signif-
512   icantly affect the success rate, likely due to GPT-4’s robust planning capabilities. However, based
513   on the NPS and CS metrics, we observe that the number of plans has a notable impact on downstream
514   task performance. Also, these results also suggest that adding more plans does not necessarily lead
515   to better results, as the model may generate multiple similar plans, resulting in similar outcomes.
516   Consequently, we select 3 as the default number of plans.

      5     C ONCLUSIONS
519
520
      This paper presents AutoML-Agent, a novel LLM-based multi-agent framework designed for Au-
521   toML, covering the entire pipeline from data retrieval to model deployment. AutoML-Agent tack-
522   les the full-pipeline planning complexity and implementation accuracy challenges in the LLMs for
523   task-agnostic AutoML by leveraging the newly proposed retrieval-augmented planning strategy and
524   multi-stage verification. In addition, we enhance the plan execution efficiency by integrating role-
525   specific decomposition and prompting-based execution techniques into the framework. Our experi-
526   ments on seven ML tasks demonstrate that AutoML-Agent outperforms existing methods in terms of
527   success rate and downstream task performance.
528   Limitations and Future Work Even though we offer a flexible module to accommodate various
529   ML tasks and data modalities, the absence of a skeleton code for completely new tasks could increase
530   the risk of code hallucination problems. Additionally, in the current version, there is still a gap
531   in code generation quality when using different backbones, e.g., GPT-4 vs. GPT-3.5, which is
532   not unique to our approach but a common challenge faced by existing LLM-based frameworks.
533   Developing a more robust framework that can effectively provide reasonable solutions with less
534   reliance on the LLM backbone is very promising future work.
535   In addition, our work still faces code generation failures when applied to machine learning tasks that
536   require significantly different development pipelines from those tested in our experiments, which
537   focused on general supervised and unsupervised settings. Tasks such as reinforcement learning and
538   recommendation systems pose particular challenges. Consequently, extending AutoML-Agent to
539   these tasks will require the development of additional agents to handle specific steps in the target
      pipeline, such as actor-environment interaction and reward modeling in reinforcement learning.

      E THICS S TATEMENT
549
550
      We expect AutoML-Agent to offer significant advantages by promoting AI-driven innovation and
      enabling individuals with limited AI expertise to effectively utilize AI capabilities. However, we
551
      acknowledge the potential misuse of AutoML-Agent by malicious users, such as generating offen-
552
      sive content, malicious software, or invasive surveillance tools when exposed to harmful inputs.
553   This vulnerability is not unique to our approach but represents a common challenge faced by ex-
554   isting LLMs with substantial creative and reasoning capabilities, which can occasionally produce
555   undesirable outputs.
556
      Although we strictly instruct the LLM to focus on generating positive results for machine learning
557
      tasks, there is a possibility of unforeseen glitches that could introduce security issues within the
558
      system. Therefore, we recommend running AutoML-Agent within a Docker container to ensure
559   isolation from the host’s file system. Additionally, due to its integration with external services for
560   retrieval-augmented generation and API-based LLMs like GPT-4, privacy concerns may arise. Users
561   should carefully review any data included in API prompts to prevent unintended data disclosures.

       A.4     E VALUATION M ETRICS
1081
1082   Success Rate (SR) We employ the success rate (Guo et al., 2024a; Hong et al., 2024a), which eval-
1083   uates whether the models built by an LLM agent are executable in the given runtime environment.
1084   Success rate is used to assess code execution.
1085
       For the constraint-free setting, we apply a three-level grading scale as follows.
1086
1087
       • 0.00: Code cannot be executed.
1088
       • 0.50: Code provides a runnable ML/DL model.
1089   • 1.00: Code provides a runnable model and an accessible deployment endpoint (e.g., Gradio).
1090
1091   For the constraint-aware setting, we use a five-level grading scale to evaluate whether the code
1092   executes successfully and satisfies the given constraints. The grading criteria are as follows.
1093
1094   • 0.00: Code cannot be executed.
1095   • 0.25: Code provides a runnable ML/DL model.
1096   • 0.50: Code provides a runnable model and an accessible deployment endpoint (e.g., Gradio).
1097   • 0.75: Code provides a deployed, runnable model that partially meets constraints (e.g., target per-
1098     formance, inference time, and model size).
1099   • 1.00: Code provides a deployed, runnable model that fully meets constraints.
1100
1101   Normalized Performance Score (NPS) In this paper, each downstream task is associated with
1102   a specific evaluation metric, which may vary between tasks. These metrics include accuracy, F1-
1103   score, and RMSLE. For metrics such as accuracy and F1-score, we present the raw values to facilitate
1104   comparison across identical data tasks. For performance metrics where lower values indicate better
1105   performance, such as loss-based metrics, we normalize all performance values s using the following
                                   1
1106   transformation: NPS = 1+s     . This transformation ensures that metrics like RMSLE are scaled
1107   between 0 and 1, with higher NPS values indicating better performance.
1108   Note that achieving downstream task performance (NPS) requires a runnable model, i.e., SR > 0. If
1109   the model cannot run, the NPS is zero by default as it cannot make any predictions.
1110
1111
       Comprehensive Score (CS) To evaluate both the success rate and the downstream task perfor-
1112
       mance of the generated AutoML pipelines simultaneously, we calculate CS as a weighted sum of
1113   SR and NPS, as follows: CS = 0.5 × SR + 0.5 × NPS.
