009                                              A BSTRACT
010
011            Recently, deep learning has gained popularity in solving statistical or approxi-
012            mate problems. However, working with symbolic data has been challenging for
013            neural networks. Despite this, the natural sciences are making strides in utilizing
014            deep learning for various use cases. In this work, we aim to solve the problem
015            of symbolic integration by using deep learning through integral rule prediction,
016            enabling faster search and better interpretability. We propose a novel symbolic
017            integration rules dataset containing 27 million distinct functions and integration
018
               rule pairs. We show that by combining a transformer model trained on this dataset
               into SymPy’s integral steps function, the number of branches explored during the
019
               depth-first-search procedure was reduced by a factor of 3 and successfully solve
020
               functions that the original version was unable to handle.

023   1   I NTRODUCTION
024
025   The advancements in computing have led deep learning to become more commonly used for many
026   applications. Deep learning has been demonstrated to be highly effective in identifying statistical
027   patterns. It delivers exceptional results in diverse areas such as computer vision (He et al., 2016;
028   Dosovitskiy et al., 2020), speech recognition (Chan et al., 2016; Gulati et al., 2020), and natural
029   language processing (NLP) Vaswani et al. (2017a); Devlin et al. (2018).
030   Deep learning for symbolic tasks, such as mathematical reasoning has not been extensively explored.
031   Automating mathematical reasoning is one such area where numerous studies have been conducted,
032   forming the foundation of these explorations. Notable studies include those by (Huang et al., 2023;
033   Piotrowski et al., 2019; Zaremba et al., 2014; Loos et al., 2017).
034
      Few attempts have been made to address the problem of symbolic integration using data-driven
035
      methods. Earlier works by Lample & Charton (2020) (hereafter referred to as ’LC’), Kubota et al.
036   (2022), and Noorbakhsh et al. (2021) have laid a solid foundation in this area. However, there are
037   not enough primary follow-up studies on the problem. LC proposed an end-to-end black-box model
038   that frames the problem as sequence-to-sequence prediction. Using an encoder-decoder transformer,
039   the model generates the integral given a mathematical function. Despite its good performance,
040   this approach lacks explainability regarding whether the model memorizes specific patterns or can
041   generalize the integration of complex functions by focusing on sub-expressions, as noted by Davis
042   (2019).
043   Our approach to solving symbolic integration is similar to systems like AlphaGo (Silver et al., 2016)
044   and AlphaGeometry (Trinh et al., 2024), where a neural model guides the steps toward solving a
045   specific problem, such as chess in the case of AlphaGo and Olympiad geometry problems in the case
046   of AlphaGeometry. These systems are efficient because they are trained on the steps of a specific
047   task and are more interpretable than a system trained in a purely end-to-end fashion. Symbolic
048   integration is essentially a search problem. When faced with a symbolic integration problem, people
049   typically employ a systematic approach: they identify an applicable integration rule, apply the rule
050
      to obtain a new expression, and repeat the process until the final integral is found.
051   In this paper, we introduce the one-of-a-kind dataset Symbolic Integration Rules Dataset (SIRD-
052   27M), consisting of nearly 27 million mathematical functions and their integration rules. This
053   dataset is one step towards automating mathematical function integration in an interpretable man-
      ner using neural networks. We include the step-by-step integration rules for a given mathematical


                                                       1

054
      function, similar to how we perform integration manually. At each step, we provide the integration
055   rule that must be applied alongside the expression for complex rules like integration by-parts and
056   u-substitution rules. Using the step-by-step integration procedure, we train a deep learning model
057   that predicts the consequent rule and expression for a given mathematical function as the input.
058   When performed repeatedly, this process simulates a human-like integration process. Compared to
059   existing works that predict the integral directly in a black-box manner, this process is more accurate
060   and interpretable. We also present robust benchmarks showing that SIRD can train a highly scalable
061   and versatile model. Our approach could be applied to enhance the capabilities of general computer
062   algebra systems.
063   The significant contributions of the paper are listed as follows:
064
065             • Introduced a large-scale, novel dataset - SIRD-271 , that can be employed for performing in-
066               terpretable step-by-step integration operation. The dataset contains over 27M mathematical
067               functions, corresponding integration rules, and expressions for specific integration rules.
068             • We propose three mathematical integration tasks to gauge our models’ performance for the
069               symbolic integration task. These include - Complete Rule Prediction, Rule Prediction, and
070               Integral Prediction.
071
                • We used the symbolic maths package SymPy (Meurer et al., 2017) to perform integration
072               given the model-predicted integration rules. To achieve this, we alter the integral steps
073               method of SymPy, replacing it with the proposed guided integral steps where we use the
074               depth-first-search procedure to navigate the integration rules search space compared to inte-
075               gral steps that depends on predefined static heuristics. Our proposed guided integral steps
076               consistently explores 3× fewer branches than integral steps.

      Solving integrals using deep learning is part of the field of symbolic reasoning. Symbolic reason-
081
      ing (Lavrac & Dzeroski, 1994; Newell & Simon, 2007) is a well-explored topic. The art of reasoning
082
      involves the intricate manipulation of symbols and the application of logical rules to perform deduc-
083   tion (Johnson-Laird, 1999), which involves drawing conclusions based on given premises, induc-
084   tion Lavrac & Dzeroski (1994), which involves making generalizations based on observed patterns,
085   and abduction Kovács & Spens (2005), which involves generating hypotheses to explain observed
086   phenomena. In visual question answering, (Yi et al., 2018) have introduced a technique that merges
087   neural networks with symbolic rules. This integration enables the system to perform compositional
088   and interpretable reasoning by leveraging visual and textual data. For the task of program synthe-
089   sis, (Shin et al., 2018) suggested a method that incorporates inferred execution traces to guide the
090   generation of accurate programs using LLMs.
091   Mathematical tasks such as multiplication (Kaiser & Sutskever, 2015; Zaremba et al., 2014), solving
092   word problems (Huang et al., 2023; Wang et al., 2018; Chatterjee et al., 2022), and calculus (Lam-
093   ple & Charton, 2020; Panju & Ghodsi, 2020) have been attempted using deep learning. Despite
094   progress, the current efforts remain largely black box, lacking interpretability and explainability.
095   In recent times, the community has been exploring the math-solving capabilities of large language
096   models (LLMs), as highlighted in literature such as (Ji & Gao, 2023; Tang et al., 2023; Frieder
097   et al., 2023). These models primarily focus on solving mathematical problems expressed in natural
098   language involving numbers. However, calculus problems, expressed through various symbols and
099   expressions without words, have been explored using a more end-to-end black-box method (Lample
100   & Charton, 2020; Noorbakhsh et al., 2021), which calls for further exploration. In the past, various
101   attempts (Rich et al.; Meurer et al., 2017) have been made to develop parsing rules for different
102   mathematical functions and use calculus rules to solve problems intuitively. Calculus problems are
103
      search problems where the final solution is several mathematical steps away.
104   One of the closest works by LC used transformers for symbolic integration. This work utilized lan-
105   guage models, notably a vanilla transformer, for training with differential equations and integration
106   problems as input to the network. The model was found to memorize specific functional patterns

136   3     SIRD-27M: S YMBOLIC I NTEGRATION RULES DATASET
137
138   We train an encoder-decoder model on integral calculation steps rather than integrals (more in Sec-
139   tion 4.1). For this, we developed a Symbolic Integration Rules Dataset (SIRD-27M) that consists
140   of 27 million function-integration rule pairs. These pairs represent occurrences in the integral cal-
141   culation steps when solving integrals for various functions. This dataset encompasses a total of 24
142   different integration rules (see Appendix C for details), including complex rules such as the substi-
143   tution rule and integration by parts. Since applying these complex rules also requires an expression,
144   we provide the expression to be used along with the rule. The following sections discuss the steps
145   of generation, validation, and processing for SIRD-27M.
146
147
148   3.1   DATA S OURCE
149
150   LC introduced three datasets for symbolic integration: the Forward Method (FWD) Dataset, the
151   Backward Method (BWD) Dataset, and the Integration by Parts (IBP) Dataset, released under CC
152
      BY-NC 4.0 license (Lample & Charton, 2020). All three datasets consist of function and integral
      pairs, their difference lies in how the samples are generated. Hence, relatively, each of the three is
153
      out-of-distribution(OOD) with respect to the others. This variance affects parameters such as the
154
      length of function and integral expressions. The characteristics of each dataset are as follows:
155
156   Forward Method (FWD) Dataset: In this dataset, the authors generate random functions and per-
157   form integration using the integrate method from SymPy (see Appendix A.1 for details). The dataset
158
      is named for its forward generation direction, moving from the function to the integral.
159   Backward Method (BWD) Dataset: Here, authors generate random functions representing the
160   integrals. These functions are then differentiated to obtain the functions corresponding to the in-
161   tegrals. The generation direction is backward, moving from the integral to the function through
      differentiation, hence the name.

222   3.3   DATA VALIDATION
223
224   The integral steps method, in its original form, outputs a sequence of integration rules that can
225   be used to calculate the integral of a function. Our data generation script not only produces this
226
      sequence of integration rules but also generates explicit pairs of function-integration rules for each
      step. To validate the correctness of our data generation script, we employ two checks:
227
228          • First, we directly compare the output of the original integral steps method with the output
229            of integral steps generated by our data generation script for an input mathematical function.
230          • Second, we apply the output sequence of integration rules from our data generation script
231            to obtain the final integral of the input mathematical function and differentiate it. Then, we
232            compare the derivative with the input mathematical function.
233
234   This process ensures the correctness of the intermediary functions and integration rules generated
235   by our data generation script.

237   3.4   DATA P ROCESSING
238
      After generating the intermediary steps, we segregate all the function and integration rule pairs, each
239
      forming a sample in SIRD. Hence, a single function from the FWD dataset leads to multiple samples
240   for SIRD - one for each step towards solving the integral of the function. Afterwards, we remove
241   the duplicate entries in the dataset.
242
243
      There are two types of integration rules in the dataset based on their application behaviour: a) simple
      rules such as the add rule, multiplication rule, power rule, etc., which do not require any additional
244
      expression for their application (refer to the left in Figure 2) b) complex rules such as the substitution
245
      rule, integration by parts, etc., which require an additional expression for their application (refer to
246   the right in Figure 2). For complex rules, the application expression is also available in SIRD.
247
248   Looking at a sample from SIRD for training, a mathematical function serves as the input to the
249
      model, and an integration rule along the application expression, if present, serves as the label. It
      is required to convert a mathematical function (model input) into a sequence of tokens that main-
250
      tains the relationship between symbols, numbers, and operators for training a sequence-to-sequence
251
      model. Expression trees are widely used to represent mathematical functions. These trees can be
252   transformed into sequences of tokens using prefix, infix, or postfix notations. Similar to LC, we
253   adopt the prefix notation of the expression tree, wherein a node is positioned before its children.
254   This notation offers the added advantage of not necessitating parentheses, as required in infix nota-
255   tion, resulting in a shorter sequence length when representing the mathematical function.
256
      As for the label, if the sample has a simple rule (without an application expression), there will
257
      be a single-token output sequence, i.e., the integration rule name. In the case of a complex rule
258   (with an application expression), the application expression is also converted to prefix notation and
259   appended after the rule name token. This approach to converting the label into a sequence helps
260   standardize how labels are utilized for model training. It simplifies the process for the model to
261   learn representations for three key decisions: determining which rule to apply, whether there will be
262   an application expression, and identifying the content of the application expression. For examples
263   of input and output sequences, please refer to the fourth and fifth columns in Table 1. Additionally,
264   the distribution of input sequence lengths is presented in Appendix B.
265   We have used same strategy for tokenisation as LC. Our tokens encompass various unary operators
266   (such as sin, cos, tan, exp, etc.), binary operators (like add, sub, mul, pow, etc.), specific symbols
267   (e.g., x, u, y, etc.), and numerical values. Furthermore, we’ve included 24 additional tokens corre-
268   sponding to the integration rules. As previously mentioned, these tokens facilitate the creation of an
269   expression tree for the input mathematical function and its corresponding prefix notation. Few token
      examples can be seen in Table 1

272   4.1     M ODEL A RCHITECTURE
273
274   We have formulated a step-by-step modelling approach. Given a mathematical function as input, we
275   train a model to learn which integration rule to apply next to compute its integral. This is modelled as
276   a sequence-to-sequence prediction. where in given a single variable mathematical function Fexpr ,
277   corresponding integration rule and expression (if required in the application of integration rule)
278   Grule&expr , an encoder-decoder model is trained on the following objective:
                                               n
279                                         1X
280                                   min         l(SΘ (Fexpr ), Grule&expr )                              (2)
                                           n i=1
281
282   where l denotes loss function used to train the model, SΘ represents the encoder-decoder model, Θ
283   represents the learnable model parameters and n denotes number of training samples.
284   We use SIRD-27M to train the model. We utilize the Transformer architecture from Vaswani et al.
285   (2017b) for the sequence-to-sequence model. Besides symbolic integration, LC empirically found
286   that the Transformer architecture performs relatively well for a few other symbolic mathematics
287   tasks. Hence, we decided to maintain the same architecture and configuration.
288   Regarding the configuration of the Transformer architecture, we employ eight attention heads, six
289   layers, and an embedding size of 512. We utilized the Adam optimizer Kingma & Ba (2014) with
290   a learning rate of 4 ∗ 10−5 . The model is trained using a batch size of 256 samples. We limited the
291   number of tokens in input expressions to 384 and used approximately 21 million samples (80% of
292   SIRD-27M) for training. The following sections will discuss the definitions, details and evaluations
293   of different benchmark tasks.

      4.3   I NTEGRAL P REDICTION
356
357
      Integral Prediction: This task evaluates the end-to-end performance of the model in calculating
358
      integrals. To achieve this, we developed the guided integral steps method (detailed below) by in-
359
      tegrating our model with SymPy’s integral steps method. guided integral steps applies integration
360   rules based on our model’s predictions at each step. A prediction is correct if the derivative of the
361   resulting integral matches the input function.
362
363
      We have reported results using the test sets from the FWD, BWD, and IBP datasets. The function
      lengths and complexities in the SIRD-27 dataset are simpler, as they represent intermediary steps,
364
      making the other three datasets out-of-distribution (OOD) and ideal for demonstrating the general-
365
      ization of our approach. Furthermore, considering that SIRD-27M is based on samples from FWD,
366   we also included evaluations on the BWD and IBP datasets. Both datasets significantly differ from
367   FWD in generation method and function characteristics, as described in Section 3.1. However, the
368   current version of SIRD does not support complex mathematical functions involving hyperbolic
369   trigonometric operators and their inverses (e.g., sinh(x), cosh(x)). The integral steps method can-
370   not handle such functions. Consequently, we exclude such functions from the test sets of all three
371   datasets before evaluation. Additionally, we remove functions with a sequence length greater than
372   384 when converted to prefix notation.
373   Guided integral steps: It is created by incorporating our model into the integral steps method of
374   SymPy. In its original implementation, integral steps calls SymPy methods corresponding to each
375   integration rule in a fixed order defined heuristically. The integration rules are checked at every
376   intermediate step to determine their applicability until the correct one is identified. It performs a
377   depth-first search through all the defined integration rules at each intermediary step when solving
      for the integral.

379   Instead, we run an inference through our model with the expression of the current intermediary step

381   tion rule token, based on which these rules are ranked and tried at each step. This way, the order
382   of integration rules explored during the depth-first search becomes dynamic as it depends on the
383   expression of the current intermediary step. Figure 1 can be referenced for the visual representation

      Accuracy Comparison: Table 3 shows that guided integral steps outperforms the LC model trained
413
      on FWD across all three test sets. Notably, integral steps also outperforms the LC model, but
414   guided integral steps achieves a higher success rate than integral steps for the FWD and IBP test

416   in BWD results in longer absolute times for guided integral steps due to the model inference call

419   set for individual functions in these results.

423   improves cross-test performance (Lample & Charton, 2020), this would make the comparison with
424   guided integral steps unfair. Additionally, it’s unclear whether the improvement in cross-test per-
425   formance reflects true generalization or pattern memorization, as Welleck et al. (2021) pointed out.
426   Comparing to LC’s FWD model helps demonstrate the OOD performance of guided integral steps
427   using the BWD and IBP test sets.

440   functions are significantly more complex and take minutes to complete, where the speedup is also
441   evident. For practical application in a production system, inference overhead can be further reduced
442   by techniques such as caching results for commonly occurring expressions, thereby minimizing the
443   number of inference calls. Additionally, batching multiple inference calls can be employed to opti-
444   mize performance.

      5   C ONCLUSION
447
448
      We present a novel step-by-step approach to the symbolic integration problem. We frame integra-
449
      tion as a search problem and accelerate it using AI, resulting in a fast, accurate, and interpretable
450
      approach. We introduce a new dataset called SIRD-27M, where the task is to predict the integration
451   rule that should be applied to a given function to find its integral. We show that a model trained on
452   SIRD can guide the search for the integral, outperforming heuristics-based search and showing su-
453   perior generalization ability. Our work is a preliminary but strong exploration of using deep learning
454   for step-by-step symbolic integration, leading the way for further research.
455
      With this work, we are laying the primary foundation for AI-assisted problem-solving in the sym-
456
      bolic integration domain, and it would be intriguing to explore whether its use cases can be extended
457   to other scientific problems.

460   6   L IMITATIONS & F UTURE D IRECTIONS
461
462   Though our approach performs well in guiding the search for integral rules, one of the primary
463   drawbacks of this approach is that it depends on the underlying step-by-step integration algo-
464   rithm integral steps in our case. Furthermore, there is an overhead of model inference calls at
465
      each step, and inference optimization might be required to optimize the runtime performance of
      guided integral steps.
466
467   The search methodology can be further refined with a beam search encompassing all intermediary
468   steps when solving integrals for a function. This enhancement will further improve guided integral
469   steps. Additionally, in this version of our work, we do not utilize the application expressions pre-
470   dicted by the model for complex rules; however, incorporating them could dramatically reduce the
471
      time required to solve integrals.

600   A.1    F UNCTIONS FROM S YM P Y
601
602   For the generation of FWD, Lample & Charton (2020) (hereafter referred to as ’LC’) used integrate
603
      function from SymPy, and we used the modified version of integral steps function to generate sam-
      ples for SIRD utilizing functions from FWD. Here, we briefly explain both the functions and on
604
      what grounds these differ.
605
606          • Integrate Function: This is the principal method to integrate functions in SymPy. This
607            function uses the Risch-Norman algorithm (Risch, 1969) and can solve both definite and
608            indefinite integrals, though it does not provide the steps of integration.
609          • Integral steps Function:
610
                 – This function imitates how a human would solve an integral problem step by step.
611
                    It outputs all the intermediate steps (expressions and integration rules) required to
612                 solve the integral of a given function. Further, these intermediate steps can be input
613                 to another function manualintegrate, which can apply the steps to generate the final
614                 integral of the function.
615              – To calculate output of integral steps, various integration rules’ are implemented in
616                 SymPy. Whenever called with an input, it heuristically searches which integral rule
617                 to apply based on a defined order in the codebase. There are certain functions such
618                 as the add rule which recursively again calls integral steps to solve the subproblems.
619                 It tries all the potential rules in a predefined order to either output steps to solve the
620                 integration of a function or fallback to DontKnowRule.
621
      Based on the description, it is clear that integrate function is far more capable of solving an end-to-
622
      end integration problem than integral steps. However, we rely on the latter for our study as we re-
623   quire integration rules instead of just the final integral for a function. Also, currently, integral steps
624   cannot generate integration rules for complex functions involving hyperbolic trigonometric func-
625   tions.
626
      Given the nature of integral steps implementation for many functions, it might go into an infinite
627
      search loop which either can be broken by timeouts or Maximum Call Stack Exceeded.
