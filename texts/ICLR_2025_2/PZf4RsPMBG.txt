009                                               A BSTRACT
010
011            Large Language Models (LLMs) have transformed software engineering (SE), ex-
012            hibiting exceptional abilities in various coding tasks. Although recent advance-
013            ments have led to the development of autonomous software agents using LLMs
014            for end-to-end development tasks, these systems are often tailored to specific SE
015            tasks. We present H YPER AGENT , a novel generalist multi-agent system that ad-
016
               dresses a broad spectrum of SE tasks across multiple programming languages by
               emulating the workflows of human developers. H YPER AGENT consists of four
017
               specialized agents—Planner, Navigator, Code Editor, and Executor—capable of
018
               managing the full lifecycle of SE tasks, from initial planning to final verifica-
019            tion. H YPER AGENT achieves state-of-the-art results on diverse SE tasks, includ-
020            ing GitHub issue resolution on the well-known SWE-Bench benchmark, surpass-
021            ing strong baselines. Additionally, H YPER AGENT excels in repository-level code
022            generation (RepoExec) and fault localization and program repair (Defects4J), fre-
023            quently outperforming SOTA baselines.

      1   I NTRODUCTION
027
      In recent years, Large Language Models (LLMs) have demonstrated remarkable capabilities in as-
028
      sisting with various coding tasks, ranging from code generation and completion to bug fixing and
029
      refactoring. These models have transformed the way developers interact with code, providing pow-
030   erful tools that can understand and generate human-like code snippets with impressive accuracy.
031   However, as software engineering tasks grow in complexity, there is an emerging need for more
032   sophisticated solutions that can handle the intricacies of real-world software development.
033
      Software agents built on LLMs have emerged as a promising solution to automate complex software
034
      engineering tasks, leveraging the advanced reasoning and generative abilities of LLMs. These agents
035
      can handle tasks such as code generation, bug localization, and orchestrating multi-step development
036   processes. However, most current agents are limited in scope, typically focused on a specific SE
037   task, such as resolving GitHub issues (Jimenez et al., 2023; Chen et al., 2024; Arora et al., 2024;
038   Xia et al., 2024; Zhang et al., 2024a; Yang et al., 2024) using benchmarks like SWE-bench (Jimenez
039   et al., 2023), or tackling competitive code generation tasks like APPS (Hendrycks et al., 2021),
040   HumanEval (Chen et al., 2021a), and MBPP (Austin et al., 2021). Other agents (Qian et al., 2024;
041   Hong et al., 2023; Nguyen et al., 2024) focus on generating complex software based on requirements.
042   While these specialized agents excel in their domains, their claim of addressing general software
043   engineering tasks is often overstated, as real-world SE challenges require more versatility across
044
      tasks, languages, and development scenarios.
045   To address such drawbacks, we propose H YPER AGENT, a generalist multi-agent system designed to
046   resolve a broad spectrum of SE tasks. Our design philosophy is rooted in the workflows that software
047   engineers typically follow in their daily routines—whether it’s implementing new features in an
048   existing codebase, localizing bugs in a large project, or providing fixes for reported issues and so on.
049   While developers may use different tools or approaches to tackle these tasks, they generally adhere
050
      to consistent workflow patterns. We illustrate this concept through a workflow that represents how
      developers typically resolve coding tasks. Although different SE tasks require varied approaches,
051
      they all follow a similar workflow.

072   Figure 1: Illustration of a Developer’s Workflow for Resolving a Software Engineering Task. The
073   diagram outlines the key phases a developer typically follows when implementing a new feature,
074   such as adding a “Dark Mode” to a web application.
075
076   1. Analysis & Plan: The developer starts by understanding the task requirements through docu-
077      mentation review and stakeholder discussions. A working plan is then formulated, outlining key
078      steps, potential challenges, and expected outcomes. This plan remains flexible, adjusting as new
079      insights are gained or challenges arise.
080   2. Feature Localization: With a plan in place, the developer navigates the repository to identify
081      relevant components, known as feature localization (Michelon et al., 2021; Martinez et al., 2018;
082      Castro et al., 2019). This involves locating classes, functions, libraries, or modules pertinent to
083      the task. Understanding dependencies and the system’s overall design is crucial to make informed
084      decisions later.
085   3. Edition: The developer edits the identified code components, implementing changes or adding
086      new functionality. This phase also involves ensuring smooth integration with the existing code-
087      base, maintaining code quality, and adhering to best practices.
088   4. Execution: After editing, the developer tests the modified code to verify it meets the plan’s
089      requirements. This includes running unit and integration tests, as well as conducting manual
090      testing or peer reviews. If issues are found, the process loops back to previous phases until the
091      task is fully resolved.
092
093   These four steps are repeated until the developer confirms task completion. The exact process may
094   vary depending on the task and the developer’s skill level; some tasks are completed in one phase,
095   while others require multiple iterations—if the developer is unsatisfied after the Execution step,
096
      the entire process may repeat. In H YPER AGENT, the framework is organized around four primary
      agents: Planner, Navigator, Code Editor, and Executor, as illustrated in Figure 2. Each agent cor-
097
      responds to a specific step in the workflow shown in Figure 1, though their workflows may differ
098
      slightly from how a human developer might approach similar tasks.1 . Our design emphasizes three
099   main advantages over existing methods: (1) Generalizability, the framework adapts easily to vari-
100   ous tasks with minimal configuration, requiring little additional effort to incorporate new modules,
101   (2) Efficiency, agents are optimized for processes with varying complexity, employing lightweight
102   LLMs for tasks like navigation and more advanced models for code editing and execution and (3)
103   Scalability, the system scales effectively in real-world scenarios with numerous subtasks, handling
104   complex tasks efficiently.
105   Experimental results (See Section 5) highlight H YPER AGENT ’s unique position as the first sys-
106   tem capable of working off-the-shelf across diverse software engineering tasks and programming

129   In recent years, applying deep learning to automated programming has captured significant interest

      fine-tuned for programming tasks (Wang et al., 2021; 2023; Feng et al., 2020; Allal et al., 2023; Li

134   Roziere et al., 2023; Nijkamp et al., 2022; Luo et al., 2023; Xu et al., 2022; Bui et al., 2022). These

137   et al., 2021), APPs (Hendrycks et al., 2021) and CRUXEval Gu et al. (2024).
138
139   2.2   AUTONOMOUS C ODING AGENTS
140
141   The rise of open-source development tools based on large language models (LLMs) has transformed
142   autonomous coding by enabling planning, self-critique, and functionality extension through function
143   calls. Integrating these tools into workflows has significantly improved code generation performance
144   on benchmarks like HumanEval (Chen et al., 2021b). Notable contributions include Huang et al.
145   (2023), Chen et al. (2023), Shinn et al. (2024), Islam et al. (2024), Chen et al. (2022), and To et al.
146
      (2024). Additionally, research on generating complex software systems from requirements has led
      to MetaGPT (Hong et al., 2023), AgileCoder (Nguyen et al., 2024), and ChatDev (Qian et al., 2024),
147
      aiming to automate broader aspects of software development beyond single-file generation.
148
149   Recently, there has been growing interest in using coding agents to automatically resolve GitHub
150   issues, addressing real-world software engineering challenges. Key works in this area include SWE-
151   Agent (Yang et al., 2024), SWE-bench (Jimenez et al., 2023), AutoCodeRover (Zhang et al., 2024b),
152
      and agentless approaches (Xia et al., 2024). This shift towards agent-based models bridges academic
      benchmarks with practical software engineering, fostering more sophisticated AI-assisted develop-
153
      ment tools that mimic human problem-solving in coding tasks.

      3.1     C ENTRALIZED M ULTI -AGENT S YSTEM
190
191   The H YPER AGENT framework comprises four primary agents:
192
193   Planner The Planner agent serves as the central decision-making unit. It processes human task
194   prompts, generates resolution strategies, and coordinates child agent activities. The Planner operates
195   iteratively, generating plans, delegating subtasks, and processing feedback until task completion or
196   a predefined iteration limit is reached.
197
198   Navigator The Navigator agent specializes in efficient information retrieval within the codebase.
199   Equipped with IDE-like tools such as go to definition and code search, it traverses codebases
200   rapidly, addressing challenges associated with private or unfamiliar code repositories. The Navigator
201   is designed for speed and lightweight operation, utilizing a combination of simple tools to yield
202
      comprehensive search results.
203
      Editor The Editor agent is responsible for code modification and generation across multiple files.
204
      It employs tools including auto repair editor, code search, and open file. Upon receiving
205
      target file and context information from the Planner, the Editor generates code patches, which are
206   then applied using the auto repair editor.
207
208   Executor The Executor agent validates solutions and reproduces reported issues. It utilizes an
209   interactive bash shell for maintaining execution states and open file for accessing relevant
210   documentation. The Executor manages environment setup autonomously, facilitating efficient test-
211   ing and validation processes.
212
213   3.2     AGENT C OMMUNICATION AND S CALABILITY
214
215   Inter-agent communication in H YPER AGENT is optimized to minimize information loss, enable ef-
      ficient task delegation, and support scalable parallel processing for complex software engineering


                                                                                                  4
      Under review as a conference paper at ICLR 2025


216
      tasks. This is achieved using an asynchronous communication model based on a distributed Mes-
217   sage Queue. The Planner communicates with child agents via a standardized message format with
218   two fields: Context (background and rationale) and Request (actionable instructions). Tasks are bro-
219   ken down into subtasks and published to specific queues. Child agents, such as Navigator, Editor,
220   and Executor instances, monitor these queues and process tasks asynchronously, enabling parallel
221   execution and significantly improving scalability and efficiency. For example, multiple Navigator
222   instances can explore different parts of a large codebase in parallel, the Editor can apply changes
223   across multiple files simultaneously, and the Executor can run tests concurrently, accelerating vali-
224   dation.
225   A lightweight LLM summarizer 2 compiles and condenses execution logs from child agents, ensuring
226   minimal information loss. Summaries, including key details like code snippets and explored objects,
227   are sent back to the Planner via the Message Queue for aggregation. The Message Queue provides
228   several advantages: (1) Parallel task execution increases throughput, (2) Dynamic task distribution
229   optimizes resources, (3) Failed tasks are requeued for reliability, (4) Easy scalability through ad-
230   ditional agents, and (5) The decoupled architecture allows independent scaling of the Planner and
231
      agents. This scalable, asynchronous model allows H YPER AGENT to handle complex SE tasks in
      distributed environments, adapting to fluctuating workloads and task complexities, making it ideal
232
      for real-world software development.

      3.3       T OOL D ESIGN
235
236   The effectiveness of H YPER AGENT is enhanced by its specialized tools, designed with a focus
237   on feedback format, functionality, and usability. Tools provide succinct, LLM-interpretable out-
238   put and are optimized for their roles in the SE process. Input interfaces are intuitive, reducing
239   the risk of errors. The Navigator uses a suite of tools, including the code search tool, which
240   employs a trigram-based search engine (Zoekt) 3 with symbol ranking. IDE-like features such as
241   go to definition, get all references, and get all symbols enhance code navigation, while
242   get tree structure visualizes code structure and open file integrates keyword search. A prox-
243   imity search algorithm helps address LLM limitations in providing precise positional inputs. The
244
      Editor uses the repair editor tool for applying and refining code patches, automatically handling
      syntax and indentation issues, and employs navigation tools for context-aware editing. The Executor
245
      leverages an interactive shell to maintain execution states for command sequences, along with
246
      open file and get tree structure for accessing testing and setup documentation. Further details
247   about the tools like tool format, functionalities and input parameters can be found in Appendix A.3.
248

298   range of approaches to software engineering tasks, ensuring comprehensive comparison. To eval-
299   uate performance, we used three key metrics: (1) the percentage of resolved instances, indicating
300   the proportion of tasks where the system produced a solution that passed all unit tests; (2) average
301   time cost and (3) average token cost, reflecting computational resource usage. These metrics pro-
302   vide a balanced evaluation of success rate, time efficiency, and resource consumption in addressing
303   real-world software engineering problems.

1038   The go to definition tool is a specialized code navigation utility designed to empower Large Lan-
1039   guage Model (LLM) Agents in locating the definitions of symbols, such as variables, functions, or
1040   classes, within a codebase. Built atop the Language Server Protocol (LSP) infrastructure, it enables
1041   an LLM to search through source files, identify the symbol’s location, and return its definition, all
1042
       while maintaining adherence to the structure of the programming language in use.
1043   At the heart of the tool’s functionality is the LSPToolKit, which serves as a natural language middle-
1044   ware between LLMs and LSP servers. This middleware abstracts the complexities of interacting with
1045   LSP backends, allowing the LLM to work with multiple programming languages without needing
1046   to comprehend the unique protocols of each. By offering a consistent interface, LSPToolKit en-
1047   ables seamless communication with LSP servers for languages such as Python, JavaScript, or Java,
1048
       making the tool adaptable to a variety of codebases across different languages.
1049   When an LLM triggers go to definition, it initiates a search for the symbol in the specified file,
1050   with the option to refine the search using line numbers. The LSPToolKit then communicates with the
1051   relevant LSP server, retrieves the symbol’s definition through syntactic and semantic analysis, and
1052   presents the information in a format easily interpretable by the LLM. If verbose output is enabled, the
1053   tool can provide additional details such as symbol documentation, code snippets with line numbers,
1054
       and a preview of the symbol’s definition. For lengthy definitions, the tool can suggest displaying
       partial content for easier readability.
1055
1056   This tool proves particularly beneficial in scenarios where LLMs are tasked with code refactoring,
1057   debugging, or generating new code based on existing elements. By automating the search and re-
1058   trieval of symbol definitions, and bridging LLMs with multiple LSP backends, the tool enhances the
1059   LLM’s capability to assist developers in efficiently exploring codebases, while ensuring language-
1060
       specific details are accurately conveyed and presented.
