010                                               A BSTRACT
011
012
               Despite Large Language Models (LLMs) achieving impressive results in code
013
               generation, significant challenges remain in automated ML development, particu-
014            larly in utilizing existing ML repositories effectively. Also, recently, people have
015            developed LLM agents that attempt to interact with repository code (e.g., resolving
016            issues), prompting the need for end-to-end evaluations starting from environment
017            setup to deploying the repository rather than merely generating code in already-
018            configured environments. These two gaps have motivated our development of
019            ML-B ENCH, a benchmark rooted in real-world ML applications that leverage
020            existing code repositories. ML-B ENCH encompasses annotated 9,641 examples
021            across 18 GitHub repositories, challenging LLMs to accommodate user-specified
022
               arguments and documentation intricacies effectively. To evaluate both LLMs and
               agents, two setups are employed: ML-B ENCH -L for assessing LLMs’ text-to-code
023
               conversion within a predefined deployment environment, and ML-B ENCH -A for
024
               testing autonomous agents in an end-to-end task execution within a Linux sandbox
025            environment. Our findings indicate that while GPT-4o leads with a Pass@5 rate
026            surpassing 50%, there remains significant scope for improvement, highlighted by
027            issues such as hallucinated outputs and difficulties with bash script generation. No-
028            tably, in the more demanding ML-B ENCH -A, GPT-4o achieves a 76.47% success
029            rate, reflecting the efficacy of iterative action and feedback in complex task resolu-
030            tion. Our code is available at https://anonymous.4open.science/r/ML-Bench
031            and our data is in the supplementary material.

034   1    I NTRODUCTION
035
036   Large Language Models (LLMs) have demonstrated remarkable prowess in function-level code
037   generation (Austin et al., 2021; Chen et al., 2021; Hendrycks et al., 2021b; Li et al., 2022). Recent
038   benchmarks have shifted from simple function synthesis to more complex tasks such as code editing
039
      and debugging (Cassano et al., 2023; Tian et al., 2024; Haque et al., 2023; Li et al., 2024) and coding
      within a repository context (Ding et al., 2024; Zhang et al., 2023a; Li et al., 2024; Yu et al., 2024).
040
      Furthermore, the evolution of code generation benchmarks reflects a growing recognition of the need
041
      for more realistic evaluation scenarios (Guo et al., 2024), like proficiency with data science libraries
042   (Lai et al., 2023; Ma et al., 2024), programming with external tools and APIs (Li et al., 2023; Shen
043   et al., 2023; Wang et al., 2023a; Gao et al., 2024).
044
045   While benchmarks like SWE-bench (Jimenez et al., 2024) have established strong foundations
046
      for evaluating repository-level code understanding, and MLAgentBench (Huang et al., 2023) has
      highlighted the importance of ML capabilities, a critical gap remains in evaluating models’ ability
047
      to utilize existing ML repositories correctly. Rather than testing models’ capability to implement
048
      ML algorithms from scratch, we focus specifically on how well models can understand and execute
049   workflows using established ML codebases - a crucial skill for practical ML development. This gap
050   is particularly significant given the recent surge of research in LLM-based agents for data science and
051   ML tasks (Hong et al., 2024; Hassan et al., 2023).
052

      repositories as libraries, as shown in Figure 1. To better assess the abilities of LLMs and agents at

088   advances this by enabling agents to not only execute ML experiments but also automatically configure
089

178   To sum up, while recent efforts have explored LLM-based agents for navigating GitHub repositories,
179   such as conducting ML experiments in simplified environments (Huang et al., 2024) or resolving
180
      repository issues (Jimenez et al., 2024) (see Table 1), ML-Bench addresses a distinct and critical
      challenge faced by many machine-learning researchers: both setting up and executing experiments
181
      using research repositories in-the-wild. Compared to existing work, our contributions are:
182
183
              • SWE-Bench (Jimenez et al., 2024) tasks agents with locating and modifying specific func-
184             tions to resolve an issue within a pre-deployed testing environment. ML-Bench challenges
185             agents to independently configure environments and download necessary data, mimicking
186             real-world research scenarios more closely.
187
              • While MLAgentBench (Huang et al., 2023) evaluates LLMs’ ability to run simple ML
188             experiments, it focuses on optimizing ML experiments rather than comprehending and
189             setting up a repository for experimentation. ML-Bench goes beyond this by requiring agents
190             to utilize machine-learning codebases.
191
              • ML-Bench evaluates the entire workflow of (1) setting up, e.g., downloading/installing
192             existing datasets, models, & packages, and (2) running ML experiments, from initial
193             repository exploration to result in interpretation. We have released a one-click evaluation
194             code, facilitating easy use and extension of the benchmark by researchers.

      2.1    TASK F ORMULATION AND D ESIGN P RINCIPLE
199
200   ML-B ENCH aims to test models’ ability to utilize existing ML repositories according to user
201   requirements. For each task, a model receives a GitHub repository, a natural language instruction,
202   and specific parameter requirements. The model must then generate executable code that correctly
203   uses the repository’s functions or models while adhering to the provided requirements.
204   For ML-B ENCH settings, (1) ML-B ENCH -L provides a complete build environment, allowing us to
205   test output bash scripts generated by LLMs within a Docker environment regarding the correctness and
206   excitability. (2) ML-B ENCH -A provides agents with access to an empty Docker environment without
207   essential packages. Agents must attempt to download the requirements for each user instruction
208   involving the installation of new datasets or Python packages themselves. This design ensures that
209   our testing framework aligns with practical application workflow.
210   ML-B ENCH focuses on the end-to-end task of setting up and executing research-related tasks in
211   repositories, presenting a unique set of challenges not fully addressed by existing benchmarks: a)
212   Models must comprehend both code and README files to navigate complex ML repositories.
213   b) Tasks require comprehension, use, modification, and reasoning across multiple files within a
214   repository. c) Agents must configure environments, install dependencies, download datasets, and
215   acquire necessary models. d) Agents need to make sequential decisions while interacting with
      the environment, mimicking real-world research scenarios. e) Unlike previous work that typically

      2.2   S UMMARY OF DATA
245
246   ML-B ENCH contains 18 diverse repositories, each reflecting varying complexity and tasks, while
247   filtering out substandard samples. The data quantities and breakdown per repository are detailed in
248   Table 2. Regarding the code language, our annotated output includes both bash scripts, which invoke
249   Python files with specific arguments, and Python code, which calls functions from the repository.
250   Bash scripts significantly outnumbered Python code snippets (See Appendix A for the explanation).
251   Each repository contributed approximately 480 examples, summing up to 9,641 examples. For our
252   experiments involving the fine-tuning of open-source LLMs, we split the dataset based on code
253   origin: The In-Distribution (ID) approach utilizes data from the same repository both for training
254   and testing, allowing repository-specific code to be exposed to models during fine-tuning. In contrast,
255   the Out-Of-Distribution (OOD) method employs disjoint sets for training and testing, encompassing
256   eight repositories—half for model training and the remaining for evaluation. The overall statistics
257   and further detailed data metrics for each repository utilized can be found in Appendix G.4 and G.3.
258
      2.3   DATA C OLLECTION AND A NNOTATION P IPELINE
259
260   Eight computer science graduate students with proficient programming abilities contributed to our
261   data annotation, with each repository’s related data being the responsibility of one annotator and an
262   additional reviewer to ensure accuracy. These students, who are co-authors of this paper, brought
263   their domain expertise to ensure high-quality annotations. Annotators were permitted to use GPT-4 to
264   expedite the annotation, although manual verification and adjustments were required. Annotating a
265
      repository took approximately 5-10 hours (Appendix D). The annotation workflow is shown in Figure
      4:
266
267   (1) README file Selection: Annotators commenced by meticulously reviewing repository contents
268   to identify all README files, including those within various subdirectories, each covering different
269   functionalities. On average, a GitHub repository included 12 README pages, with one notable
      repository, DGL, comprising 154 README files. (2) Task Mining: Annotators identify practical

      randomly selected 20-30 test set candidates after annotating each repository. These candidates

      underwent additional executable verification and correctness checks. Cross-validation was performed

      by other annotators to remove near-duplicate cases, ensuring that cases differing only in minor

285   Beyond task identification, annotators with machine learning expertise extracted key parameters

301
302   3.1   ML-B ENCH -L S ETUP
303   Our experimental inputs include human instructions and the entire repository code (including
304   README files). We present three distinct experimental setups to evaluate the models. Given
305   that current models cannot process the entire code context, the three scenarios range from ideal to
306   extreme. Oracle Segment (Oracle): For the Oracle setup, annotators identify and record crucial
307   segments within README files — referred to as "Oracle Segments" — that contain necessary codes
308   and textual explanations pivotal for completing the prescribed tasks. These segments serve as the
309   foundational source to derive the ground truth code, ensuring that models can access all critical
310
      evidence when generating code. BM25 Retrieval (Retrieval): In this setup, we employ a BM25
      retriever to extract segments from the repository’s documentation, including README files, that are
311
      relevant to the given instructions. This method aims to mimic a more automated way of narrowing
312
      down necessary information without human pre-selection. Code (Code): This setting exposes the
313   model to the entire code repository. All files within the repository, with README files placed at the
314   forefront, are presented as input to the model. Due to model context limitations, texts are truncated
315   when necessary, and potential information loss is analyzed and documented in Appendix H.2. Please
316   refer to Appendix H.1 for further details on implementing the BM25 retriever.

      4     ML-B ENCH -A E XPERIMENTS
379
380   4.1   ML-B ENCH -A S ETUP
381
382   In ML-B ENCH -A, as shown in Figure 5, we provision a sandbox environment as the testing ground
383
      for agents. The sandbox offers a fundamental setup, such as a configurable Docker image, allowing
      agents to modify and execute commands freely within the simulation. Agents are granted the
384
      ability to execute bash scripts or interact with IPython notebooks. The agents must interact with
385
      this environment, perusing code within repositories—regardless of the extended code or required
386   parameters—to accumulate comprehensive information. This process necessitates successive actions,
387   with the agent autonomously determining the correctness of each step and iteratively refining its
388   approach upon encountering errors. We expect the agents’ outputs to differ from previous LLM
389   settings due to the dynamic and interactive nature of the tasks.
390
      Recent agent frameworks, including SWE-Agent (Yang et al., 2024), Aider (Gauthier), OpenDevin
391
      (Wang et al., 2024b), provide a well-defined suite of impactful actions that bridge the agent with
392
      its operational environment. Like Reflexion (Shinn et al., 2023) and CodeAct (Wang et al., 2024a),
393   agents iteratively execute actions, refine their approach via feedback, and perform effectively in
394   solving complex tasks. These agents are designed to mimic the workflow of human programmers,
395   thoroughly parsing and employing a repository. To facilitate this, the agents can execute any Python
396   code and bash commands within a secure and isolated Linux OS sandbox, providing an ideal setting
397   for our benchmark evaluations. In each instance, ML-B ENCH -A initiates an isolated docker container
398   sandbox where all agents’ bash commands are executed, with the outcomes returned as observations.
399   Different agent frameworks implement environmental interactions in varying ways, with each action
400   yielding observations for AI agents. Here ML-B ENCH -A essentially assesses the effectiveness of
401
      different environments. In ML-B ENCH -A, a configurable workspace directory contains repositories
      agents are to handle, installed within a safe sandbox environment that provides controlled access
402
      for agents to interact with and process as needed. For evaluation, instead of relying on the Pass@K
403
      metric used in ML-B ENCH -L, we emphasize the agent’s effectiveness in fulfilling user requirements
404   through interactive execution rather than predetermined outputs (Success Rate). Success is defined by
405   agents correctly following repository-documented workflows, matching expected execution patterns,
406   and producing outputs in the required format. Unlike the stochastic nature of ML tasks, which
407   complicates direct output validation, our deterministic evaluation framework focuses on reproducible
408   and consistent criteria. These include environment setup, dependency management, and correct API
409   usage, verified by human annotators with high agreement (0.92 Cohen’s kappa). This methodology
410   ensures reliability by avoiding stochastic variations and emphasizing correct repository interaction
411   over final model performance.

473   5.2          E RROR A NALYSIS
474
475   By analyzing the execution log, we find that the errors of models and agents fall into five categories:
476
      Hallucination Errors (E1): These errors include instances when the models misinterpreted the user’s
477
      intention, misplaced Python code and bash script, or generated random or irrelevant code.
478
479   Lack of Knowledge or Information (E2): This type of error primarily stems from the model’s
480   inability to fulfill user requirements based on crucial parameters. Possible types of errors are as
481   follows:
482               1. Code inner information. The models sometimes lack sufficient information necessary to
483                  satisfy the user’s requirements. For instance, this deficiency might involve missing parameter
484                  names (—-lr and —-learning–rate) or unavailable options (it only accepts ’Citeseer’ when
485                  the input given was ’citeseer’).


                                                                                                                                  9
      Under review as a conference paper at ICLR 2025


486
            2. Domain knowledge. The models sometimes lack the domain-specific knowledge required
487            to handle certain instances. For example, in BERT, the models simultaneously generated
488            —-case=True and —-model=uncased.
489
            3. Grammar knowledge. This happens when the models incorrectly identify and handle certain
490
               command line symbols. Like the $ symbol, which could affect execution.
491
492
            4. Local data information. The models were not supplied with enough crucial parameter
               information for specific commands, leading to the inability to find paths and successful
493
               execution. While less common, this error was particularly notable in the OpenCLIP.
494
495
      Knowledge Manipulation (E3): Take BERT, for instance, where the model needed to integrate
496   DIR=/model/ and —-model_dir=$DIR to form —-model_dir=/model. There were also cases where
497   it couldn’t merge /model_name in /data into a proper path format like /data/model_name. Similar
498   incidents were observed in OpenCLIP.
499
500
      Syntax Errors (E4): These errors cover instances of incorrect code generation of syntax errors
      instead of hallucination, mainly Python syntax errors such as the use of undefined variables. These
501
      errors arise from cases that prefer generating Code.
502
503   Operational Error (E5 – ML-B ENCH -A Only): These errors only occur in ML-B ENCH -A. Less
504   powerful agents, such as those using base models like GPT-3.5, may have difficulty understanding
505   the task in the execution sandbox. In these cases, the agent gives up easily or exits without finishing
506   the tasks. Interestingly, these errors are less observed in advanced model settings.
507   On ML-B ENCH -L, we find that Claude 3 has more hallucinations than GPTs for closed-source
508   models. However, its ability to fulfill requirements (represented by E2) is better than that of GPT-3.5.
509   Under the retrieval setting, neither GPT-3.5 nor Claude 3 exhibits an increase in hallucination but an
510   increase in the proportion of E2 compared to the Code setting. For GPT-4, both E1 and E2 increased
511   because GPT-4 generates code without considering the content in cases involving task-irrelevant
512   information, except for GPT-3.5 and Claude 3.
513   Compared to the Code setting, Oracle provision decreases the quantities of E1 and E2, while the
514   differences in E3 and E4 are insignificant. This suggests that whether an Oracle is provided does not
515   significantly affect the closed-source models’ knowledge manipulation and Python code generation
516   abilities. We tend to attribute these to the models’ inherent ability rather than the reference.
517
      The ML-B ENCH -A showcases potential in incorporating feedback from the experience, leading to
518
      fewer E1 and E4 errors. Yet, compared to ML-B ENCH -L, ML-B ENCH -A is more prone to E2 and E3
519
      errors due to file type discrepancies from the set output type. Because of the sandbox environment’s
520   great flexibility, we observe increased hallucinations with each step, including installing conflicting
521   dependencies and navigating incorrect or even illegal directories. Meanwhile, E5 happens in less
522   powerful agents. The agent often ignores the instruction to export the final solution to run.sh, or
523   refuses to answer the question immediately after entering the virtual sandbox. Compared to GPT-4,
524   GPT-4o shows enhanced abilities in sourcing relevant information from repositories, leading to
525   markedly fewer errors associated with E2 and E3. Yet, it exhibits a tendency toward more frequent
526   hallucinations than GPT-4, for instance, installing incorrect packages. For a detailed error analysis
527   and quantitative performance visualization on both setups, refer to Appendix G.6 and G.7, and for
528
      examples running on both setups, refer to Appendix G.8 and G.9.

      6   C ONCLUSION
531   ML-Bench addresses the limitations of existing benchmarks in comprehensively evaluating model
532   performance across real-world ML workflows. It simulates the complete ML development process,
533   from environment configuration to code execution. Our tasks require models to retrieve supporting
534   evidence, generate code, and set hyperparameters correctly, as well as download/install existing
535   datasets, models, & packages. We introduce ML-B ENCH -L and ML-B ENCH -A, two distinct
536
      evaluation setups assessing LLMs’ code generation capabilities and agents’ end-to-end task execution
      abilities, respectively. Results show GPT-4 achieving a Pass@5 rate over 50% in ML-B ENCH -L and a
537
      76.47% success rate in ML-B ENCH -A, highlighting areas for improvement in handling hallucinations
538
      and bash script generation.

      Our study, while comprehensive within its scope, is subject to certain limitations that stem primarily
543
      from linguistic and data source constraints.

      Models Limitation We acknowledge that the scope of our benchmark might not entirely encapsulate
546
      the breadth of available open-source models. While we have conducted extensive tests using a variety
547   of models beyond the results presented in the paper, including but not limited to:
548
549          • mistralai/Mistral-7B-Instruct-v0.3

             • mistralai/Mixtral-8x22B-Instruct-v0.1

552          • Qwen/Qwen1.5-72B-Chat

             • codellama/CodeLlama-34b-Instruct-hf

557          • meta-llama/Meta-Llama-3.1-8B-Instruct-Turbo
558          • meta-llama/Meta-Llama-3.1-70B-Instruct-Turbo
559          • meta-llama/Meta-Llama-3.1-405B-Instruct-Turbo
560
561   Due to space constraints, detailed results from these models were not included in the manuscript. We
562   emphasize that the primary objective of our benchmark is not to be an exhaustive repository of the
563   latest open-source models but rather to establish a robust and versatile benchmark framework. Our
564   goal is to inspire the community to develop better ML agents.
565
      Our benchmark, ML-Bench, is designed to be widely applicable and has already seen extensive
566   adoption within the community. By providing a comprehensive and practical evaluation framework,
567   we aim to pave the way for future advancements in the development of ML agents, regardless of the
568   specific models used.
569
570   Data Source Limitation - Reliance on GitHub Repositories in English Our reliance on GitHub
571   repositories with documents exclusively in English introduces a selection bias. GitHub, while rich in
572   open-source projects and documentation, may not comprehensively represent the broader landscape
573   of software development practices and trends globally. This choice potentially overlooks significant
574   contributions and insights from non-English-speaking communities. This limitation might impact the
575   development of tools and models tailored to a more diverse set of programming environments and
576
      community needs.
577
578   Methodological Limitation - Relying on Pre-built Machine Learning Packages Our method-
579
      ology utilized existing machine learning packages instead of developing algorithms from scratch.
      While this approach allowed us to leverage well-established, tested, and optimized tools, it also
580
      introduces certain constraints. Dependence on pre-built packages means our work is confined to
581
      the capabilities and limitations of these tools. This reliance could limit our ability to fully explore
582   novel or unconventional approaches possible with custom-built algorithms. Moreover, this choice
583   potentially impacts the reproducibility and customization of our findings. Researchers who seek to
584   build upon our work may encounter similar constraints imposed by the pre-built packages we utilize.
585   These limitations can hinder innovation and adaptation in different contexts or for specific usage.
586
587   Scope Limitation - Tasks Limited to README File Descriptions By strictly adhering to the
588   specified tasks, our study may overlook potential applications or challenges not explicitly documented
589   in README. This limitation can result in a narrower understanding of the tools we examined, as
590   it fails to explore their full potential and applicability. The reliance on README descriptions also
591   assumes that these documents comprehensively and accurately reflect all relevant aspects of the
592   repositories, which may not always be accurate. Important tasks or nuances might be undocumented
593
      or underrepresented in these files.

2056   When evaluating the relevance of retrieved code segments within repository-level tasks, we utilized
2057   the BLEU-1 score as a metric to measure the similarity between hand-annotated "Oracle" segments
2058   and those extracted by the model using BM25 retrieval. "Oracle" segments are those identified by
2059   annotators as most pertinent, serving as the foundational evidence required for the task at hand. As
2060   depicted in Table 20, the resulting BLEU scores indicate a low degree of correlation, suggesting that
2061   the retrieval segments identified by BM25 are significantly dissimilar to the Oracles crafted by human
2062   annotators. This finding is demonstrative of BM25’s limitations in effectively identifying the most

2436    We introduce DeepFloyd IF, a novel state-of-the-art open-source text-to-image model with a high
2437
        degree of photorealism and language understanding. DeepFloyd IF is a modular composed of a
        frozen text encoder and three cascaded pixel diffusion modules: a base model that generates 64x64
2438

2439

2440    text encoder based on the T5 transformer to extract text embeddings, which are then fed into a UNet
2441    architecture enhanced with cross-attention and attention pooling. The result is a highly efficient
2442    model that outperforms current state-of-the-art models, achieving a zero-shot FID score of 6.66
2443    on the COCO dataset. Our work underscores the potential of larger UNet architectures in the first
2444    stage of cascaded diffusion models and depicts a promising future for text-to-image synthesis.
