009                                              A BSTRACT
010
011            The rapid advancement of AI models has significantly impacted daily life, with
012            Large Language Models (LLMs) playing a pivotal role in automating tasks and
013            providing all-in-one solutions via API services. Meanwhile, there is a growing
014            demand for private, resource-constrained, customizable, and high-performance
015            models tailored to specific user needs. However, many users struggle to deploy
016
               these models due to limited resources or technical expertise. In this work, we
               try to address these challenges by focusing on two primary objectives: (1) to
017
               meet the specific needs of a broad range of users, and (2) to lower the barriers
018
               to AI model usage (e.g., resource constraints, technical expertise) for most users.
019            In our preliminary exploration, we introduce FLAME, a framework that deter-
020            mines and generates AI models based on data or task descriptions provided by
021            users. While existing solutions rely on pre-built models or extensive finetuning,
022            FLAME leverages LLMs (e.g., GPT4-turbo) to capture data patterns and task fea-
023            tures from user input, converting them into user requirements and structured meta-
024            data (e.g., task type, model architecture, and classifier dimension). Then, FLAME
025            uses them as guidance to generate customized models by hypernetworks. This ap-
026            proach significantly improves efficiency, achieving up to 270x faster model pro-
027
               duction compared to finetuning-based paradigms (e.g., all-parameter and LoRA
               fine-tuning) while maintaining comparable performance across various tasks. We
028
               validate the effectiveness of FLAME through comprehensive experiments on Nat-
029
               ural Language Processing (NLP), Computer Vision (CV), and tabular datasets,
030            demonstrating its ability to quickly deliver high-quality, customized models.

033   1   I NTRODUCTION
034
035   Recent advancements in AI models, especially in LLMs such as GPT-4 (OpenAI, 2023), LLaMA
036   3.1 (AI, 2024) and Phi-3 (Abdin et al., 2024) have significantly influenced our daily life (Zhao
037   et al., 2023; Yang et al., 2023). Leveraging their impressive abilities, LLMs offer an all-in-one
038   solution for versatile user requirements through API services, making advanced AI accessible for
039
      tasks like text generation, summarization, and chatbots. However, there is still a growing demand
      for private (deployed), customizable, and resource-constrained models suited to specific domains
040
      (Staab et al., 2023; Yao et al., 2023). Since user requirements vary widely, deploying general models
041
      might not always achieve optimal results, particularly in specialized areas such as law, economics,
042   and medicine. In contrast, tailored models tend to exhibit superior performance (Turc et al., 2019;
043   Gunasekar et al., 2023; Fu et al., 2023; Hsieh et al., 2023; Yao et al., 2024; Chen & Varoquaux,
044   2024). However, users might not have adequate expertise or enough data, time and resources to
045   determine the model and finetune it. These barriers greatly hinder the wider application of AI
046   models. Therefore, our research aims (I) to meet the specific needs of a broad range of users and
047   (II) to lower the barriers to AI model usage (e.g., resource constraints, technical expertise) for
048   most users.
049   However, these pursuits face certain challenges. First, changes in user requirements can lead to
050   model adjustments at varying levels. While minor changes in user requirements may necessitate
051   adjusting the parameters of the target model for better performance (Sagawa et al., 2020; Lv et al.,
052   2023), significant task alterations (e.g. regression to classification, data modality change) might
053   require the model to change its output dimensions or even its architecture. Second, to reduce con-
      straints on AI model usage, both general capabilities (e.g., task understanding) and precise

079   In our preliminary exploration, we initiate our idea into FLAME, as depicted in Figure 1. Specif-
080   ically, given users’ input (User Data, User Description, or both), FLAME constructs prompts to
081
      utilize LLMs (GPT4-turbo) to summarize the task, analyze data patterns and task features, and for-
      mat it into a user requirement (just a single sentence) and structured metadata (determines the most
082
      appropriate target model for the given task). Next, FLAME uses Multi-Head Module-Wise Param-
083
      eter Generator to decode user requirements into model parameters to output the tailored model,
084   which could be directly used by users for prediction. For instance, as shown in Figure 1, for a text
085   classification task of sentiment analysis, users can provide data batches (User Data) or just describe
086   the task (User Description). FLAME then constructs a prompt, interacts with LLM, and outputs
087   User Requirement in Requirement Generator and structured Metadata in Model Generator. Next,
088   we use User Requirement and Metadata to generate the model parameters by Parameter Generator.
089   An optional finetune process (either full-parameter or LoRA (Hu et al., 2022) finetuning) can be
090   undergone for better performance. Finally, users can apply this tailored model to their data. In short,
091   our contributions can be summarized as follows:

094

096
                    conditioned parameters, which extends LoRA-based hypernetworks to more model archi-

104   2.1       L ARGE L ANGUAGE M ODELS
105
106   In recent times, the field of natural language processing (NLP) has been significantly reshaped by
107   the emergence of large language models (LLMs) like ChatGPT (Wang et al., 2019a), GPT-4 (Ope-
      nAI, 2023), LLaMA (Touvron et al., 2023), and others. The concept of LLM arises from language

108
      model (Vaswani et al., 2017; Devlin et al., 2019), an algorithm used in natural language processing
109   to predict the likelihood of a sequence of words occurring in a sentence. Characterized by deep ar-
110   chitectures, billions of parameters, and tremendous training corpus, LLMs have drastically enhanced
111   the ability of machines to understand, interpret, and generate human language (Naveed et al., 2023;
112   Brown et al., 2020).
113
      Upon their introduction, LLMs have quickly gained widespread attention and have been applied
114
      across various domains, including machine translation, text completion, conversational agents, and
115   so on (Shen et al., 2023; Romera-Paredes et al., 2023; Pan, 2023). However, despite their impressive
116   capabilities, LLMs come with their own set of challenges. Research indicates that in certain specific
117   areas, smaller models can outperform LLMs (Turc et al., 2019; Gunasekar et al., 2023; Fu et al.,
118   2023). Moreover, due to the immense size and complexity of these models, they are often impractical
119   for users to employ or fine-tune, particularly when faced with limitations in computational resources
120   or technical expertise (Hu et al., 2022).
121
122   2.2   H YPERNETWORKS
123
124   Hypernetwork, the model designed to output the weights of another model is first proposed by Ha
125
      et al. (2017). Since it only needs a single forward pass to output model parameters, it provides a fast
      and efficient alternative to the vanilla pretrain-finetune paradigm. Given its unique capability, it has
126
      gained wide attention in various fields like recommendation system, natural language processing,
127
      and computer vision. Lv et al. (2023) proposes a framework DUET for efficient device model
128   generalization, which uses hypernetwork to generate the MLP layers of device models for model
129   personalization. Alaluf et al. (2022) proposes HyperStyle, which learns to modulate StyleGAN’s
130   weights to faithfully express a given image in editable regions of the latent space. Ivison et al.
131   (2023) proposes HINT, which uses hypernetwork to encode task definitions into task-conditioned
132   LoRA adapters (Hu et al., 2022) and applies them to LLMs. To clarify, it’s important to note that
133   HINT and FLAME differ greatly in both motivations and technical details. HINT focuses on
134   using hypernetworks to make instruction tuning more efficient for LLMs, but ours is broader. We
135   aim to generate a variety of models that meet specific user needs, which involves significant technical
136
      differences from HINT. For more information, we kindly refer readers to Section 3.
137   While most hypernetworks are mlps, recently, a few works discuss the potential of hypernetworks
138   with more complex architectures, like GAN (Ratzlaff & Li, 2019), ResNet (Alaluf et al., 2022).
139   These works explore the potential of hypernetworks in model generation to some extent.

      3     M ETHODOLOGY
142
143
      The workflow of FLAME consists of 2 main modules: Requirement Generator and Model Cus-
144   tomizer, as depicted in Figure 2. Requirement Generator takes User Data or User Description as
145   input and outputs User Requirement, while Model Customizer translates User Requirement into an
146   off-the-shelf AI model. In this section, we elaborate on the details of our framework.
147
148   3.1   R EQUIREMENT G ENERATOR
149
150   Given User Data or User Description, Requirement Generator interacts with LLM (GPT4-turbo) to
151   analyze data patterns and task features and summarize them into one sentence: User Requirement
152   r ∈ R.
153   Effective prompt design is crucial for accurately distilling patterns from data. On the one hand,
154   User Data might be insufficient, and the lack of data poses challenges to reflect the real distribution
155   in users’ scenarios. On the other hand, LLMs tend to highlight simpler patterns directly inferable
156   from labels. User Description could enable LLM to focus more precisely on the proper and unique
157   patterns. Generally, we summarize the demands of prompt design as follows:
158
159
             • The metadata of the task (e.g. task type, output dimensions) must be pointed out in the final
               sentence.
160
161          • Data-specific information, if any, should be reflected in the final sentence and must only
               focus on the data itself rather than the labels given.

      3.2.2   PARAMETER G ENERATOR
217
218   Once target model architecture T is determined, Parameter Generator P (·; θp = (θe , θg )) generates
219   the parameters θt with User Requirement r ∈ R as input, following an encoder-decoder architecture,
220   where the encoder E(·; θe ) encodes r into a latent variable, the decoder (Multi-Head and Module-
221   Wise Parameter Generator) G(·; θg ) decodes it into model parameters module by module.
222   The Encoder E(·; θe ) : R 7→ Rd is a language model (DistilBERT-base) followed by a feed forward
223   network for further feature transformation. Following previous solutions (Wang et al., 2019b) to get
224   the latent variable of User Requirement r, before FFN, we use the latent of the first token [CLS] as
225   the sentence embedding, formatted as Equation (1):
226                                              z = E(r; θe ).                                        (1)
227   Vanilla hypernetworks then decode z into the modules of θt (e.g., layer1.0.weight) one by
228   one. However, the diversity of User Requirements brings three challenges.
229
      The diversity of User Requirements leads to variations in architecture even for similar tasks.
230
      A subtle difference in requirements can significantly alter the target model architecture, making
231
      it challenging for hypernetworks. For instance, switching from MRPC to STS-B in the GLUE
232   Benchmark (both tasks involve semantic similarity) changes the model requirement from a 2-class
233   classification to regression. However, hypernetworks could not generate these different architectures
234   simultaneously and efficiently. Our preliminary solution is to use a multi-head approach. We first
235   split the target model modules into two types: the static components (e.g., inner layers, remains
236   consistent across tasks), and the dynamic components (e.g., the final classifier, varies across tasks).
237   Hypernetworks can easily generate the static component, as its structure remains unchanged across
238   user requirements. For the dynamic component, which poses more challenges, we assign a decoder
239   head to each task. This ensures that the output shape in each head is fixed, allowing the model to
240
      efficiently adapt to changes in architecture.
241   For instance, to generate a DistilBERT-base model for both MRPC and STS-B, we need to di-
242   vide its modules into static components (e.g., layer1.0.weight) and dynamic components
243   (e.g., classifier.weight is 768 × 2 for MRPC and 768 × 1 for STS-B). During infer-
244   ence, we first output the parameters of the static components. Then, for MRPC, to handle the dy-
245   namic component, we switch to the MRPC-specific decoder head, outputting modules like 768 × 2
246
      classifier.weight. For STS-B, the only change is switching to the STS-B-specific decoder
      head. It’s important to note that this approach works only for tasks seen during training. For new
247
      test tasks, we could only manually select the most similar task head. Addressing this limitation is
248
      left for future work.
249
250   Direct generating large models would result in convergence issues. Generally, it is not practical
251   to output the parameters of large-scale models directly for convergence issues (Dinh et al., 2022;
252
      Alaluf et al., 2022). As a result, we add LoRA adapters (Hu et al., 2022) into the model, generate
      their parameters, and finally obtain the target model by merging them.
253
254   While reducing generation size by only producing LoRA adapter weights can cut down on
255   complexity, it doesn’t always lead to optimal model performance. Previous works like HINT
256   (Ivison et al., 2023) have demonstrated that hypernetworks can generate LoRA adapters to adjust
257   transformer models. Yet, this success might not extend to non-transformer models like ResNet, suf-
258
      fering from performance degradation. We’ve discovered that this issue arises from overlooking the
      adjustable layers present in these models (e.g. norm layers). Unlike the LayerNorm in transformers,
259
      other norm layers, such as BatchNorm are learnable. However, outputting their parameters together
260
      with LoRA adapters would greatly increase the scale of hypernetworks, resulting in convergence
261   issues or out-of-CUDA-memory errors. Therefore, we disable the functionality of these layers. For
262   implementation details, we kindly refer readers to Appendix B.
263

333   4.1.1    BASELINES
334
      Generally speaking, FLAME introduces a novel capability: translating user data or descriptions into
335
      model parameters. As this is the first framework of its kind, there are no directly comparable base-
336
      lines. To address this, we compare FLAME with two widely adopted training paradigms: Finetune
337   and LoRA, which are standard for adapting models to new tasks. In Finetune, we finetune the tar-
338   get model with all parameters by the training data in each task individually. Since FLAME uses
339   LoRA to reduce the complexity, we treat finetuning the target model with LoRA adapters in each
340   task as the baseline LoRA. The LoRA adapters are the same as FLAME’s. In tabular experiment,
341   since MLP is simple, FLAME directly outputs its weights, rather than using LoRA adapters. Mind
342   that in each setting, we ONLY need ONE FLAME to solve the tasks, while other paradigms need
343   finetuning for each tasks.
344   For our study, we introduce a variant of FLAME, FLAME-F. This adaptation includes an additional
345   step where, following FLAME’s generation, we perform full-parameter finetuning using consistent
346   hyperparameter settings (1 epoch if no specified). Details are available in Appendix A.
347
348   4.1.2    M ETRICS
349
350   In addition to the performance metric, we stand at the viewpoint of common users WITHOUT
351   technical expertise and propose additional metrics. We humbly think that it is NOT how long
352   it takes to train a FLAME but how long they could get a model that counts for the most
353
      users. Hence, we evaluate FLAME with two additional efficiency metrics: E2E Runtime and
      Relative Efficiency. E2E (end-to-end) Runtime measures total task completion time (seconds),
354
      while Relative Efficiency scales this runtime against the worst-performing method.

      4.2     R ESULTS AND O BSERVATIONS
358
      The results can be seen from Tables 1 to 3, underscoring the efficiency, and satisfactory performance
359
      of our framework. For hyperparameter settings, we kindly refer readers to Appendix A. Here,
360   we provide detailed discussions of our results.
361
362   FLAME yields progressively more significant speed gains as the size of the target model in-
363
      creases. Leveraging the power of hypernetworks, FLAME generates custom model weights in a
      single forward pass, eliminating the need for a resource-intensive and expertise-dependent finetun-
364
      ing process. This approach yields progressively more significant speed gains over the conventional
365
      pretrain-finetune paradigm as the size of the target model increases. The acceleration observed
366   ranges from approximately 40x for a simple MLP (1K) to 270x for Distil-BERT base (66M), mark-
367   ing a 7-fold increase in efficiency. It is also interesting to find out that in the experiments of tabular
368   data, LoRA is a bit slower than Finetune. The deficiency in speed stems from the target model.
369   Since tabular tasks are rather simple, FLAME directly uses MLP, a very shallow neural network iso-
370   morphic to LoRA adapters. Hence, directly finetuning the target model is more efficient than using
371   LoRA to finetune it. These results highlight FLAME’s exceptional efficacy in producing tailored
372   models, particularly for larger target architectures.
373   Inter-task knowledge empowers FLAME for enhanced model generation. Due to hypernet-
374   works’ limitations, FLAME cannot generate large models directly. Instead, our implementation
375   for sizable target models is to generate LoRA adapters and merge them to construct the final mod-
376   els. This approach may initially seem at most comparable to the baseline LoRA. Yet, in practice,
377   FLAME surpasses LoRA in all experiments and even outperforms Finetune in CV and tabular data
      tasks. This performance boost is largely attributable to the inter-task knowledge gleaned by FLAME.

523   5               F UTURE W ORK AND C ONCLUSION
524
525
526   In this work, we introduce FLAME, a framework that leverages LLMs to determine and generate
527   customized models based on user data or task descriptions. While FLAME shows strong perfor-
528   mance in generating customized models, efficiently addresses specific user needs and lowers the
529   barrier to AI model usage, it is important to note that this is merely a preliminary exploration, and
530
      several challenges remain unsolved.
531   First, the granularity of Model Generator can be improved. A more detailed analysis of factors
532   such as task complexity and available user resources could enable more refined model architecture
533   decisions. Second, the capabilities of Parameter Generator need expansion. The current multi-head
534   solution is limited to tasks resembling the training data. For tasks with greater disparity (e.g., new
535   output dimensions or modalities), FLAME still falls short. Further, for different modalities, separate
536
      FLAMEs are required. Developing an all-in-one FLAME for different modalities is a key goal for
      future research.
537
538   Generally speaking, by introducing FLAME, we aim to paves the way for a new paradigm in adap-
539   tive, efficient model creation. However, our research is still in its early stages, and we welcome
      discussions and collaborative efforts to further explore this emerging field.

892   C    P ROMPT D ESIGN AND C ASE S TUDY ON R EQUIREMENT G ENERATOR
893
894   Since FLAME directly uses LLM to summarize User Requirements, it is crucial to design proper
895   prompts for them. As discussed in Section 3.1, the prompt should tell the type of the task and the
896   data-specific information. However, due to various reasons (e.g. lack of data), summarizing users’
897
      requirements often poses challenges. In response, we carefully design the prompt and incorporate
      users’ knowledge (User Data or Description) into it, resulting in better performance.
898
899   As shown in Figure 4, the first row is the system prompt (prompt template), which contains our re-
900   quirement of generating User Requirements, two examples for better reasoning, and the place (last
901   2 lines) to fill User Description and User Data individually. In the requirement part, we emphasize
902   that LLM ought to succinctly identify the type of the task (e.g. classification) and point out the
903
      data-specific information. For further uses, we require the output User Requirement must have 5
      elements, namely task type, data pattern, number of classes, number of input features (optional) and
904
      scale. These features are curial for further process in Model Generator, which determines the archi-
905
      tecture of target model. In the example part, we provide 2 examples and analyze them, following
906   the idea of COT (Chain of Thought) (Wei et al., 2022) for better performance.
907
908   As shown in Figure 4, in NLP, we use benchmark GLUE for experiments. The major difference
909
      between GLUE’s tasks can be directly summarized from their task name since they vary in task
      type (e.g. binary two-input classification, multi-class one-input classification, one-input regression.)
910
      Therefore, to instruct LLMs to precisely capture user requirements in NLP, we don’t need to provide
911
      User Description. In the example, with the prompt given, LLM successfully points out that the given
912   data is a task of binary sentiment analysis. Similar results can be observed in tabular data. Here,
913   different from NLP, we also provide background information (User Description) on the given data.
914   We directly tell LLM that the data comes from datasets such as Iris and Car Evaluation. With
915   User Description and User Data provided, LLM successfully points out that the input is a tabular
916   classification to evaluate the acceptability of car purchases. For CV, since the task is multi-class
917   image classification and there is 31 categories, for better accuracy, we directly tell the number of
      classes in User Description. Without it, LLM would tell that it is multi-class, losing the information

961   C.1        I MPACT OF U SER D ESCRIPTION
962
963   As discussed above, for easy tasks, the provision of User Data would be sufficient to the generation
964   of User Requirements. However, we find that for difficult tasks (like CV tasks), User description
965   could sufficiently improve the quality of User Requirement.
966   Begin with the example in Figure 4, if we remove User Description in the CV example, the output
967   of LLM would be: This is a multi-class image classification task, where each image features office
968   and personal items, often with a shallow depth of field and soft lighting, suggesting an indoor
969   setting with artificial light. Without User Description, LLM first fail to figure out the classifier
970   dimension, simply telling that it is multi-class classification. Meanwhile, it outputs some WRONG
971   features. The shallow depth of field is the feature of the domain DSLR, which does not apply to
      Webcam. As a result, FLAME would output a model not well customized to the Webcam domain.
