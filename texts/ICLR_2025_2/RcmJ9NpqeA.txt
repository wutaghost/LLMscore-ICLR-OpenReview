009                                               A BSTRACT
010
011            The proliferation of software vulnerabilities presents a significant challenge to cy-
012            bersecurity, necessitating more effective detection methodologies. We introduce
013            White-Basilisk, a novel approach to vulnerability detection that demonstrates su-
014            perior performance while challenging prevailing assumptions in AI model scal-
015            ing. Utilizing an innovative architecture that integrates Mamba layers, linear self-
016            attention, and a Mixture of Experts framework, White-Basilisk achieves state-
017            of-the-art results in vulnerability detection tasks with a parameter count of only
018
               200M. The model’s capacity to process sequences of unprecedented length en-
               ables comprehensive analysis of extensive codebases in a single pass, surpass-
019
               ing the context limitations of current Large Language Models (LLMs). White-
020
               Basilisk exhibits robust performance on imbalanced, real-world datasets, while
021            maintaining computational efficiency that facilitates deployment across diverse
022            organizational scales. This research not only establishes new benchmarks in code
023            security but also provides empirical evidence that compact, efficiently designed
024            models can outperform larger counterparts in specialized tasks, potentially re-
025            defining optimization strategies in AI development for domain-specific applica-
026            tions.

029   1   I NTRODUCTION
030
031   The field of Artificial Intelligence (AI) has experienced significant advancements in recent years,
032   particularly in the domain of Natural Language Processing (NLP). Large Language Models (LLMs)
033
      such as GPT, Llama, and Gemini have demonstrated remarkable capabilities across diverse tasks.
      These models, often comprising hundreds of billions of parameters, reflect the ”bigger is better”
034
      philosophy in AI development. However, even with the notable success of these massive models,
035
      they present substantial challenges. For instance, the computational requirements for training and
036   inference of such models are considerable, resulting in high energy consumption and limited acces-
037   sibility (Patterson et al., 2021; Faiz et al., 2023). As these models continue to expand in size and
038   complexity, it is important to reconsider the sustainability and necessity of this approach for all AI
039   applications.
040
      One domain where the limitations of the ”bigger is better” paradigm become particularly evident
041
      is in specialized tasks like code vulnerability detection. This critical aspect of cybersecurity re-
042   quires models that can comprehend complex code structures, identify subtle patterns, and maintain
043   high accuracy – all while ideally being deployable in resource-constrained environments. Code vul-
044   nerability detection represents a unique challenge at the intersection of software engineering and
045   cybersecurity. Vulnerabilities often emerge from complex interactions between various components
046   in the codebase, making detection through standard techniques difficult. The rapid evolution of
047   both software development practices and attack methodologies further exacerbates this challenge.
048   As developers embrace novel paradigms such as microservices and AI-driven code generation, the
049   potential attack surface expands in ways that can be challenging to predict.
050   Traditional approaches to automated vulnerability detection, including static application security
051   testing (SAST) tools, have demonstrated both strengths and limitations in addressing security chal-
052   lenges. Studies have shown that SAST tools generally achieve lower vulnerability detection rates
053   compared to newer approaches like large language models (LLMs) (Zhou et al., 2024). This lower
      detection rate is concerning because it means that SAST tools may miss critical security vulnera-

054
      bilities, potentially leaving software systems exposed to exploitation and attacks. The inability to
055   identify a significant portion of vulnerabilities undermines the effectiveness of these tools in ensur-
056   ing robust software security, highlighting the need for more advanced or complementary detection
057   methods.
058
      Recent machine learning (ML)-based approaches have demonstrated promising results in vulnerabil-
059
      ity detection. However, many exhibit significant limitations in processing extended code sequences
060
      and comprehending complex, context-dependent vulnerabilities. These models often struggle with
061   long-range dependencies and fail to capture the nuanced interactions between disparate code com-
062   ponents. Moreover, their performance can be inconsistent across diverse vulnerability types and
063   programming paradigms, limiting their efficacy in comprehensive security analyses. In response
064   to these challenges, we present White-Basilisk: a compact but powerful model that challenges the
065   prevailing paradigm in AI design. With approximately 200M parameters – a fraction of the size of
066   many current state-of-the-art models – White-Basilisk demonstrates that thoughtful architecture and
067   targeted training can yield impressive results in code vulnerability detection.
068   White-Basilisk’s design philosophy centers on maximizing efficiency without compromising perfor-
069   mance. This approach led to several innovations that form the core of our contributions, which can
070   be summarized as follows:
071
072         1. Efficient Architecture: We propose a novel integration of Mamba layers, linear-
073            complexity Infini-attention, and Mixture of Experts. This combination enables effective
074            processing of long code sequences while maintaining a relatively small parameter count.
075         2. Extended Context Length: White-Basilisk can analyze sequences up to 128,000 tokens
076            during inference. This capability facilitates whole-codebase analysis on a single GPU,
077            potentially uncovering vulnerabilities that span multiple functions or files.
078         3. Resource-Efficient Training: Our model achieves competitive performance using a
079            dataset of just 2 million code samples. This efficiency challenges conventional assump-
080            tions about data requirements for specialized AI tasks.
081         4. Advanced Training Techniques: We incorporated Fill-in-the-Middle (FIM) pretraining
082            and Scale-Invariant Fine-Tuning (SIFT) to enhance model robustness and generalization.
083            These techniques contribute to White-Basilisk’s ability to perform well across diverse code
084            vulnerability detection tasks.
085
            5. Comprehensive Benchmarking: We conduct rigorous experiments to evaluate White-
086            Basilisk’s efficacy across multiple benchmark datasets, including PRIMEVUL Ding et al.
087            (2024), BigVul Fan et al. (2020), Draper Russell et al. (2018), REVEAL Chakraborty et al.
088            (2021), and VulDeepecker Li et al. (2018). Our compact model demonstrates competi-
089            tive performance against larger counterparts, emerging as the front-runner in several key
090            metrics.
091
092   By addressing the unique challenges of code vulnerability detection with a resource-efficient ap-
093   proach, White-Basilisk not only explores new possibilities in this critical domain but also raises per-
094
      tinent questions about the necessity of ever-larger models in AI. In the subsequent sections, we will
      elucidate the details of White-Basilisk’s architecture, training procedures, and performance across
095
      multiple benchmarks.

098   2   R ELATED W ORK
099
100   The field of automated code vulnerability detection, a cornerstone of our research, has undergone
101   rapid evolution since its inception in the early 2000s. Driven by the growing complexity and secu-
102   rity risks of software systems, early pioneers explored static analysis techniques Chess & McGraw
103   (2004) and pattern matching methods Livshits & Lam (2005). While these approaches laid a founda-
104
      tional framework, they often encountered significant drawbacks, including high false positive rates,
      difficulty detecting complex vulnerabilities, and susceptibility to obfuscation techniques. Due to
105
      these limitations, they were generally unsuitable for active production environments.
106
107   As the field matured, researchers recognized the transformative potential of AI, gradually shifting
      from conventional practices to a new paradigm. This transition was marked by the development of


                                                        2
      Under review as a conference paper at ICLR 2025


108
      VulDeePecker Li et al. (2018), one of the first deep learning (DL)-based systems for vulnerability
109   detection. VulDeePecker utilized code gadgets and Bidirectional Long Short-Term Memory (BiL-
110   STM) networks to identify vulnerabilities in C/C++ code. This work demonstrated the ability of DL
111   techniques to capture complex patterns associated with code vulnerabilities, paving the way for the
112   development of further ML-driven solutions. However, its reliance on manually crafted features lim-
113   ited its generalizability. Building on this work, Russell et al. (2018) developed the Draper dataset,
114   which provides a substantial real-world dataset specifically designed for training neural networks in
115   the task of vulnerability detection. Their work showed the advantages of leveraging vast training
116   data to enhance model performance, improving performance but still struggling with limited context
117
      windows that restricted the capture of long-range dependencies in code.
118   Following that, pre-trained LLMs emerged as a significant breakthrough in code analysis. Hanif
119   & Maffeis (2022) proposed VulBERTa, an adaptation of the RoBERTa model for detecting code
120   vulnerabilities, demonstrating the potential of transfer learning from natural language processing to
121   code analysis. By fine-tuning LLMs pre-trained on extensive corpora of code, this approach quickly
122   gained popularity due to its ability to capture latent patterns in both code structure and semantics.
123
      However, similar to many transformer-based models, it suffers from quadratic computational com-
      plexity with sequence length, constraining its applicability to large-scale projects.
124
125   Another line of research focused on models specifically tailored for code understanding. Feng et al.
126   (2020) introduced CodeBERT, a bimodal pre-trained model for programming language and natural
127   language. This work allowed for a deeper understanding of both code and its associated documenta-
128   tion, proving useful across a range of software engineering tasks, including vulnerability detection.
129
      However, while powerful, these models typically demand significant computational resources and
      encounter difficulties when processing extremely long sequences, which restricts their practical use
130
      in analyzing large codebases. For instance, CodeBERT was pre-trained for approximately 1000
131
      hours on 16 interconnected NVIDIA Tesla V100 GPUs, representing a substantial energy and re-
132   source investment.
133
134   Other recent work concentrated on improving the granularity and efficiency of vulnerability detec-
135
      tion. Such an example is LineVul (Fu & Tantithamthavorn (2022)), a transformer-based approach
      for line-level vulnerability prediction, enabling precise localization of vulnerabilities within code-
136
      bases. While valuable for precise localization, this approach may overlook vulnerabilities that span
137
      multiple lines or functions.
138
139   Comprehensive studies by Chakraborty et al. (2021) and Ding et al. (2024) highlighted persistent
140   challenges in the field, including data quality issues, unrealistic evaluation methods, and difficul-
141
      ties in handling long-range dependencies. Ding et al. (2024) showed that existing benchmarks
      significantly overestimate model performance, with state-of-the-art models achieving high scores
142
      on flawed datasets but failing on more realistic ones. Our work directly addresses these concerns
143
      through improved data collection, realistic evaluation metrics, and model architecture designed for
144   long-range understanding.
145
146   Currently, existing vulnerability detection methods primarily focus on their main objectives without
147
      considering the increasing size of model parameters. This uncontrolled growth in model size has led
      to the development of energy-hungry models requiring dozens of cutting-edge GPUs. Most recently,
148
      research developments have been made in efficient language model design. For example, building
149
      on the work of Mamba Gu & Dao (2023), Lieber et al. (2024) introduced Jamba, which employs
150   a Mixture of Experts (MoE) strategy, combining Mamba layers and attention mechanisms in an
151   interleaving pattern Fedus et al. (2022). In addition, Wang et al. (2020) proposed the Linformer,
152   which can reduce self-attention complexity to linear time using low-rank approximations, while
153   the Transformer-XL architecture Dai et al. (2019) has been shown to effectively model long-term
154   dependencies by introducing recurrence into self-attention.
155
      In this paper, we introduce an efficient method for developing AI models. Our model, White-
156   Basilisk (Figure 1), demonstrates how the integration of cutting-edge training techniques enables
157   the deployment of comparatively small models with just 200M parameters — significantly fewer
158   than the billions seen in some other cases. Furthermore, unlike current state-of-the-art models, our
159   implementation was trained using only a single NVIDIA A100 40GB GPU, highlighting its resource
160   efficiency. The increased model performance combined with reduced energy requirements for model

      The architecture of White-Basilisk was designed to address three key challenges in code vulnera-
      bility detection. Firstly, it tackles the problem of long-range dependencies, as code vulnerabilities
181
      often span multiple functions or even files, necessitating a model capable of understanding exten-
182
      sive contexts. Secondly, the architecture strikes a balance between local and global information
183   processing. This dual focus enables both fine-grained understanding of code syntax and broad com-
184   prehension of overall program structure, both crucial for effective vulnerability detection. Lastly, the
185   design prioritizes computational efficiency, aiming to create a powerful model that can be deployed
186   in real-world settings without requiring massive computational resources. By addressing these inter-
187   connected objectives, our model provides a robust solution for identifying and analyzing potential
188   security flaws in code. To this end, we developed a novel hybrid architecture that combines three
189   main components:
190
            1. Mamba layers: These form the backbone of our model, efficiently capturing local depen-
191
               dencies and providing adaptive computation based on input content.
192
193         2. Linear-complexity Infini-attention: Our adaptation of this mechanism allows for efficient
194            processing of extremely long sequences, enabling whole-codebase analysis.
195         3. Mixture of Experts (MoE): This adds dynamic adaptability throughout the network, al-
196            lowing the model to specialize its processing based on input characteristics.
197
198   The synergy between these components allows White-Basilisk to process sequences up to 128,000

      3.1   M AMBA L AYERS
220
221   Mamba layers form the backbone of White-Basilisk, chosen for their exceptional efficiency in cap-
222   turing local dependencies in code sequences. Unlike traditional recurrent neural networks (RNNs)
223   or attention mechanisms, Mamba’s selective state-space mechanism allows for linear-time compu-
224   tation with respect to sequence length. This efficiency is crucial for processing long code sequences
225   without excessive computational overhead. The adaptive nature of Mamba layers enables the model
226
      to focus computational resources on the most relevant parts of the input, making it highly efficient
      for detecting subtle patterns that may indicate vulnerabilities.
227
228   In White-Basilisk, Mamba layers are implemented with a State size (dstate ) of 16, a Convolution
229   kernel size (dconv ) of 4 and an Expansion factor of 2. The core computation in a Mamba layer can
230   be summarized as:

236
      code analysis tasks.

241   while maintaining computational efficiency. MoE layers allow the model to activate only a subset

243   models of similar capacity. In the context of code vulnerability detection, this efficiency is crucial
244   as it allows our model to handle effectively diverse types of code and potential vulnerabilities with-
245   out adding to its complexity and number of parameters. The sparsity induced by MoE layers also
246   contributes to faster inference times, a critical factor in real-world deployment scenarios.
247   The MoE layers in White-Basilisk are configured with 8 Experts and 2 Experts per token. For an

      4.2     P RETRAINING
325
326   Traditional LLM training methods are generally designed to enable models to comprehend language
327   and its syntax. This is often accomplished through Causal Language Modeling (CLM), where a
328   model learns to predict the next token given its input. Another common approach is based on
329   the Fill in the Middle (FIM) technique, in which random text portions are masked, and the model
330   must reconstruct the missing content. Some advanced source code LLMs combine both methods
331   to increase model flexibility (Li et al. (2023)). Similarly, in our work, we employ both techniques
332
      during model pre-training on the selected 2M code samples. This process requires approximately
      600 hours to complete on a single NVIDIA A100 40GB GPU.
333
334
      4.2.1    SIFT (S CALE -I NVARIANT F INE -T UNING )
335
336   We implement automated adversarial training using SIFT to improve the model’s resilience against
337   adversarial examples. SIFT operates by introducing small perturbations to the input during training,
338   encouraging the model to learn more robust features. In our implementation, we added a Pertur-
339   bationLayer into the model architecture, which applies learnable perturbations to the input embed-
340   dings. The training process was designed to minimize both the task loss and the adversarial loss, the
341   latter being computed as the difference between predictions on clean and perturbed inputs.
342   This approach confers several advantages, including improved model generalization and enhanced
343   robustness to minor variations in input. Such resilience is crucial in the domain of code vulnerability
344   detection, where the model must maintain consistent performance across diverse code samples and
345   potential adversarial inputs.
346
      By adopting this technique, we have effectively imbued the model with a form of ’adversarial im-
347   munity’, rendering it more resilient against potential attacks or attempts to deceive its analysis. This
348   enhanced robustness is particularly valuable in security-critical applications, where the reliability
349   and consistency of the model’s performance are of paramount importance.

741   The classification head of White-Basilisk is designed to efficiently transform the high-dimensional
