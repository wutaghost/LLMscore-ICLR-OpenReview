009                                               A BSTRACT
010
011            Hardware design verification (DV) is a process that checks the functional equiva-
012            lence of a hardware design against its specifications, improving hardware reliability
013            and robustness. A key task in the DV process is the test stimuli generation, which
014            creates a set of conditions or inputs for testing. These test conditions are often
015            complex and specific to the given hardware design, requiring substantial human
016
               engineering effort to optimize. This leads to a significant challenge in automated
               and efficient testing for arbitrary hardware designs. We seek a solution that takes
017
               advantage of large language models (LLMs). LLMs have already shown promising
018
               results for improving hardware design automation, but remain under-explored for
019            hardware DV. In this paper, we propose an open-source benchmarking framework
020            named LLM4DV that efficiently orchestrates LLMs for automated hardware test
021            stimuli generation. Our analysis evaluates six different LLMs involving six prompt-
022            ing improvements over eight hardware designs and provides insight for future work
023            on LLMs development for efficient automated DV.

      1   I NTRODUCTION
027   Large Language Models (LLMs) (Yang et al., 2020; int, 2020; Touvron et al., 2023) have gained
028   significant attention in recent years due to their language generation and comprehension capabilities
029   on tasks such as language translation (Feng et al., 2020), question answering (Yang et al., 2020),
030   and sentiment analysis (Liu et al., 2021). Recently, there has been interest in exploiting LLMs
031   to improve hardware design generation (Blocklove et al., 2023; Fu et al., 2023; Lu et al., 2024).
032   Arguably, hardware design verification (DV), which checks the correctness of hardware designs,
033
      ranks among the most crucial and time-consuming tasks in hardware development. Hardware DV is
      often time-consuming, usually taking up to 60%-70% of the development time (Shin, 2024), and
034
      requires significant human guidance and expertise due to the complexity of both hardware design
035
      and its corresponding testing requirements (Shin, 2022).
036
037   On the other hand, existing work on LLMs has been studied for software testing. For example, Codex
038   (Chen et al., 2021) can produce functionally correct bodies of code from natural language docstring
039
      descriptions. LLaMA 2 (Touvron et al., 2023), an LLM using instruction tuning and Reinforcement
      Learning with Human Feedback (RLHF) (Christiano et al., 2017; Stiennon et al., 2020) for fine-tuning,
040
      emerges impressive generalization and external tool usage ability. However, these approaches are
041
      not directly applicable due to the following two challenges. First, unlike software programming
042   languages, there is a scarcity of high-quality, open-source hardware designs and testing code available
043   online for training LLMs. This limitation is critical because Hardware Description Languages (HDLs)
044   possess distinct semantics that differ fundamentally from software programming languages. These
045   unique characteristics make HDLs considerably more challenging for LLMs to interpret and learn
046   from, as the models cannot simply transfer their knowledge from conventional programming contexts
047   without substantial modifications. Second, the testing space for a hardware design design is typically
048   large, leading to a scalability problem. Existing approaches on hardware DV require human guidance
049   to reduce search space, such as adding heuristics to guide tests of a particular hardware design. This
050
      raises an important question: can LLMs effectively minimize the amount of human effort involved
      in hardware DV?
051

053   hardware DV. In the DV process shown in the right of Figure 1, the test stimuli generation stands out
      as the most labor-intensive phase, often requiring iterative trial-and-error. A good stimuli discovers

      Effective test stimuli generation has been a major challenge in meeting 100% coverage (Witharana
135   et al., 2022). For a simple design, verification can be done with individual directed tests, in which
136   test stimuli (inputs for the DUT) are manually generated. For more complex designs, a large number
137   of stimuli is required for exercising as much of the design’s functionality as possible. Traditionally,
138   constrained-random testing (CRT) has been used to generate vast random but valid test stimuli
139   and to attempt to “hit” the bins. However, CRT is inefficient to hit as many bins as human effort
140   for hardware states with complicated conditions. Still, it remains the case that extensive human
141   engineering involvement is required for the test stimuli design process.

144
145   While the application of LLMs on hardware design verification has been brought to focus only
146   recently, test code generation for software engineering with LLMs has been well-studied and achieved
147
      remarkable performance (Chen et al., 2022; Schäfer et al., 2023; Lukasczyk and Fraser, 2022). Chen
      et al. (Chen et al., 2022) utilize LLM to suggest potential implementations and corresponding test
148
      cases for a function. They provide the LLM with the signature and a description of the function and
149
      select the best solution based on functionality agreement. Schäfer et al. (Schäfer et al., 2023) propose
150   a pipeline to generate unit tests for existing code, which iteratively refines the prompt to generate
151   better tests. These studies show promising results on software code, while we shift focus to using
152   LLM to reason HDLs and hardware design specifications, leading to a different setting and more
153   sophisticated procedure.
154
      In hardware design verification, assertion-based verification (ABV) is also widely adopted together
155
      with code coverage, functional coverage, and validation using generated test patterns (Witharana
156   et al., 2022). ABV inserts assertions into the DUT HDL source to detect violations of predefined
157   design properties. However, ABV requires test patterns (i.e. input test stimuli) to activate given
158   assertions and therefore reveal vulnerabilities. For simulation-based ABV approaches, traditional test
159   generation that uses random or constrained-random tests cannot guarantee to activate assertions with
160   complex conditions in a reasonable time. In order to speedup ABV, Pal et al. (Pal et al., 2008) propose
161   bias random test generation. They consider the DUT as a black box and restrict test generation to
      only input/output signals. Ferro et al. (Ferro et al., 2008) used combinatorial testing, which provides

228   4.1      LLM4DV F RAMEWORK
229
      In this work, the proposed LLM4DV framework automates the DV process by exploiting LLMs for
230
      test stimuli generation, shown in the left of Figure 1. Compared to traditional DV processes that use
231
      user-defined test stimuli, the stimulus generation agent uses an LLM to provide a test stimulus in
232   each timestep. This reduces human involvement in the hardware DV loop and effectively guides tests
233   to increase coverage rates.
234
235
      In each generation cycle, the prompt generator produces a prompt based on a template (⃝)
                                                                                             7 and the
      current coverage feedback from the coverage monitor. LLM4DV allows customization of prompts
236
      inside a dialogue, this means each query message can receive different prompts, as managed by the
237
      query scheduler (⃝)5 shown in Figure 1. This is explained in Section 4.3.
238
239   The LLM takes in the prompt and generates a natural language response, from which the test stimulus
240   values are extracted and sent to the DV flow in the right of Figure 1. The DV framework then produces
241
      current coverage which is considered as input for the LLM-based stimulus generation agent (⃝)         8
      shown in Figure 1. The processes of test stimuli generation and hardware testbench simulation are
242
      executed in parallel asynchronously. Specifically, a buffer is placed between the stimulus generation
243
      agent interfaces to balance the rate of the test stimuli generation and consumption. In every timestep
244   when the stimulus generation agent is requested for a test stimulus, it takes out the oldest value in its
245   stimulus buffer; if the buffer is empty, the agent starts a new generation cycle, in which the LLM
246   takes in a new request and a list of new stimuli will be added to the buffer.
247
      In LLM4DV, each DV process is viewed as a “trial”, where there would be multiple dialogues made
248
      in a single trial, as illustrated in Figure 1, which are controlled by the dialogue scheduler (⃝).
                                                                                                     6 A trial
249
      stops in one of the following three states, and the agent is considered “exhausted”. When reaching
250   such a state, it becomes ineffective or inefficient to expand testing coverage and the pipeline stops.
251   First, full coverage is reached, where all coverage bins have been hit. Second, no new coverage is
252   extended within a number of trials, where our implementation by default specifies that the stimulus
253   generation agent cannot hit any bins within 25 responses. Finally, the coverage expansion speed is
254   low, where our implementation by default specifies that the stimulus generation agent hits fewer than
255   three bins within 40 responses. @Reviewer 9kfb: Algorithm 1 provides the exact implementation of

      the LLM4DV framework, we explore six prompting strategies and improvements over a set of LLMs

      4.3   G ENERAL P ROMPTING S TRATEGIES
294   We provide a Coverage-Feedback Template to generate prompts for the LLM. When constructing it,
295   we utilize prompt engineering techniques including 1) System message: it is included at the beginning
296   of every prompt, and is used to prime the model with context, instructions, or other information
297   relevant to the use case; 2) Start with clear instructions; 3) Repeat instructions at the end; 4) Add
298
      clear syntax: punctuation, headings, and section markers; 5) Specifying the output structure.
299   Coverage-Feedback Prompt Template The Coverage-feedback prompt template contains templates
300   for the system message, initial query, and iterative queries.
301
302   • The system message clarifies the expected response format and specify other requirements.
303   • The initial query is the first user query message in a dialogue. It contains three parts: 1) Task
304     introduction: a description of what is included in this prompt and what the LLM will be asked to
305     do; 2) Coverage plan summary: a description of cover bins of the coverage plan; and 3) Initial
306     question: a one-line instruction.
307   • The iterative queries are the user messages following the first assistant (LLM) response. Each
308     contains three parts:
309
        1. Result summary: a general feedback which:
310
            - if the previous assistant response was gibberish (i.e. contains mostly nonsense words) or
311
              didn’t follow the output format, the result summary repeats the output format requirement;
312
            - otherwise if the previous assistant response failed to hit any new bins, the result summary
313           points that out and ask for a new list of stimuli;
314
            - if the previous assistant responses hit some bins, the result summary points that out and ask
315           for a new list of stimuli.
316
        2. Differences: a list of uncovered bins.
317
        3. Iterative question: a one-line instruction, repeating the output format requirement if previous
318
           response was gibberish or didn’t follow the output format.

      4.4   F OUR G ENERIC P ROMPTING I MPROVEMENTS
322   In our experiments, we develop two improvements necessary for making the framework executable
323   and two improvements that increase its performance on most cases, which can be effectively employed
      regardless of the nature of the DUT and coverage plan. Details see Appendix A.3.


                                                         6
      Under review as a conference paper at ICLR 2025


324
      Missed-bin Sampling This optimization is ⃝   4 in Figure 1, and is later used in the query scheduler. In
325   most generation cycles in a trial, there would be hundreds to thousands bins uncovered. The iterative
326   queries can’t include all of them because the prompt’s length would exceed the LLM’s input token
327   number limit. Meanwhile, exposing too many uncovered bins to the LLM confuses the LLM on
328   which mistakes should it resolve first.
329
      We propose missed-bin sampling, which samples a number of bins from all uncovered bins to be
330
      included in the differences part of iterative queries. Our experiment finds that more random sampling
331   methods encourage the agent to cover bins with stricter hitting conditions, and more stable sampling
332   methods make the agent more efficient in hitting the easier bins.
333
334
      We define three sampling methods (1) Pure Random Sampling, which randomly samples seven bins
      from all uncovered bins. (2) Coverpoint Type-based Sampling, which samples from “easier bins” and
335
      “harder bins” respectively. (3) Mixed Coverpoint Type-based and Pure Random Sampling, which
336
      switches between previous two whenever the agent becomes inefficient with current strategy.
337
338   Best-iterative-message Sampling The LLM needs previous query messages in the dialogue to learn
339   about what has happened. However, as the dialogue grows, the length of input may exceed the
340
      LLM’s input limit. One solution is summarizing previous query message, which helps generalizing
      concepts in the dialogue but loses details, which is crucial in our task. On the other hand, sampling
341
      from previous messages acceptably loses some generality meanwhile preserves key details, including
342
      the bin description and positive examples (i.e responses that successfully hit many bins) useful for
343   covering corner cases. These strategies are used in our Query Scheduler in ⃝. 5
344
345   We propose four sampling methods (I) Recent Responses, where we keep the initial query (and its
346
      response), and three most recent iterative queries (and their responses). (II) Successful Responses,
      where we randomly keep three that hit the largest number of bins. (III) Mixed Recent and Successful
347
      Responses, where we keep two most successful and one most recent query. (IV) Successful Difficult
348
      Responses, which is similar to Successful Responses but each “harder bin” counts as 2.5 bins.
349
350   Dialogue Restarting LLMs sometimes behave stubbornly, repeating mistakes they made previously.
351   We introduce a dialogue restarting scheduler (⃝)
                                                     6 to resolve this problem. When the LLM hits less
352
      than three new bins within t responses, we clear the dialogue record and restart from the system
      message and initial query. We define four dialogue restarting schedules (a) Normal Tolerance, where
353
      t = 7. (b) Low Tolerance, where t = 4. (c) High Tolerance, where t = 10. (d) Coverage Rate-based
354
      Tolerance, where t = 4 in the beginning and t = 7 after reaching certain coverage rate threshold.
355
356   Best-iterative-message Buffer Reset When the dialogue record is reset, the buffer for best iterative
357   messages in Best-iterative-message sampling can also be cleared or kept. These two strategies display
358
      a trade-off between “effectively forgetting past mistakes” and “learning about the task faster after
      restart”. This reset is also then incorporated in the dialogue restarting scheduler (⃝).
                                                                                           6
359
360   We define three resetting plans for the best-iterative-message buffer, (i) Clearing best-messages. (ii)
361   Keeping best-messages. (iii) Stable-restart Keeping best-messages: keeping the buffer on restarts, but
362   not using it for the first four responses after restarts.
363   We have employed distinct notations to denote the available options for these prompting enhancements.
364   This enables us to encode specific combinations, such as Claude-3 1 I a i, indicating that the first
365   option is selected for all the aforementioned prompting strategies.

496
      Across all DUTs, each configuration demonstrates that LLM4DV can either match or exceed the
497
      coverage rates achieved via naive CRT. This signifies not only the adaptability of LLMs to varied
498
      hardware testing contexts but also their potential to streamline certain aspects of verification by
499   reducing reliance on extensive random input generation. Formal methods only work well when the
500   design states are small. Particularly, the AMPLE Prefetcher Weight Bank only achieves a coverage
501   lower than 1%, because it contains large storage queues, which introduces a large number of feasible
502   states to represent values in all possible orders. In fact, the number of states grows exponentially with
503   queue size, leading to a “state space explosion”, despite the simplicity of the design.
504
      In Table 4, our evaluation metrics encompass not just the maximum coverage rate but also the
505   maximum and average message counts. This comprehensive evaluation becomes particularly valuable
506   when maximum coverage attained is 100%, which could happen with less complex DUTs. Maximum
507   and average message counts allow for assessing the efficiency of LLMs in achieving this state of
508   success. Practically speaking, this aspect is beneficial as a more expedient DV cycle is normally
509   desirable.
510
      Among the LLM models tested, Claude 3.5 Sonnet stands out, especially in handling more complex
511   tasks such as those associated with CPU architectures. This model’s superior performance in scenarios
512   involving the Ibex and MIPS CPU may indicate a more nuanced understanding of CPU operations,
513   likely stemming from richer pre-training that possibly included diverse computational and hardware-
514   related datasets. Claude 3.5 Sonnet’s effectiveness in these settings could suggest that its training
515   included exposure to architectural nuances specific to CPUs, enhancing its ability to generate more
516   relevant and coverage-effective test stimuli. Nevertheless, it still falls short of the 100% mark achieved
517   by the formal tool. This suggests that while LLMs can handle complex scenarios to a degree, they
518   may lack the deep, specialized knowledge or the ability to effectively navigate the vast state spaces
519
      that high-complexity DUTs entail.
520   The consistently high coverage achieved by all LLM models in testing lower complexity DUTs,
521   such as the Asynchronous FIFO and SDRAM Controller, demonstrates the proficiency of LLMs
522   in handling straightforward scenarios. This high performance is mirrored by the baseline formal
523   tool, indicating that LLMs are competent and can rival traditional verification methods in simpler
524   verification contexts. Claude 3 Sonnet, for instance, maintains 100% coverage across simpler DUTs,
525
      suggesting excellent efficiency in generating relevant test cases with minimal extraneous inputs. The
      efficiency of test generation, as reflected by the message count metrics, provides another dimension
526
      of evaluation. Models like Claude 3 Sonnet, which generally require fewer messages to achieve high
527
      coverage, indicate a more targeted and efficient approach to test case generation. In contrast, models
528   requiring a higher number of messages, such as Llama-2-70b-chat and Codellama-70b-Instruct, may
529   be generating less precise or less effective test stimuli, indicating inefficiencies that could translate to
530   increased testing time and resource consumption in practical applications.

      6    D ISCUSSION
534   Gimmick or Trend? The computer architecture and hardware design community is now starting
535   to see debates regarding the effectiveness of LLMs for automated chip design, questioning whether
536   their use is merely a gimmick or represents a future trend. Our particular take on this problem is that
537   there is a need to set up open datasets and benchmarks for different problems in chip design, so that
538   the effectiveness and potential use of LLMs can be fully understood and quantified. Our work fits
539   exactly in this category, and we target, in our opinion, the most human labor-intensive part (in terms
      of engineering) of the chip design process. Our baseline results have demonstrated that LLMs can


                                                          10
      Under review as a conference paper at ICLR 2025


540
      achieve satisfactory coverage rates on straightforward designs, but they struggle with more complex
541   ones, suggesting that LLMs do hold promise within the specific context of automated hardware DV.
542
543   Data Asymmetry and LLM4DV as Downstream Evaluation Owing to the fundamental difference
544
      between programming languages used in software and hardware engineering, existing LLMs are
      presumably more adept with software programming languages like Python and may lack a deep un-
545
      derstanding of the semantics of hardware description languages (HDLs). For instance, the StarCoder
546
      model’s training data comprises various programming languages, yet SystemVerilog and Verilog
547   represent only about 5% of that data (Li et al., 2023). In the meantime, we see the provided LLM4DV
548   flow presents an excellent opportunity to evaluate the capability of LLMs to function as agents for
549   complex tasks, making it an ideal downstream evaluation task.
550
      Enabling Future DV Research with LLMs The LLM4DV framework serves as a standard exper-
551
      imentation platform to explore and evaluate DV work. The framework provides an interface for
552
      researchers to orchestrate LLMs and input their own prompts for future DV research. For example,
553   advanced approaches, such as coverage-directed generation and mutating tests (Fine and Ziv, 2003b;
554   Guzey and Wang, 2007; Laeufer et al., 2018), could be integrated into the LLM prompts for better
555   coverage. These directions face research challenges that are beyond the scope of this work, but
556   LLM4DV offers an infrastructure on which to build them and baseline results for evaluation.

      We introduce LLM4DV, an open-source benchmark framework designed to efficiently coordinate
568   LLMs for automated hardware test stimuli generation. LLM4DV facilitates integration with diverse
569   DUTs, coverage plans, and LLMs. Our framework has been tested with a range of DUTs and LLMs,
570   and we have developed a set of prompting enhancements that establish solid baselines in the bench-
571   mark. Our results illustrate that while these LLMs perform well with simple DUTs, their effectiveness
572   is limited when dealing with more complex designs. This still suggests that LLMs have the potential
573   to overcome common challenges in DV research, such as state space explosion and input specificity,

575   research. The natural language interface and explainability of LLMs can better integrate domain

      hardware designers and also serve as a valuable downstream task for assessing LLMs’ capabilities

      A.1          D ESIGN C HOICES IN PARAMETER S ETTINGS
834
835   Since the benchmark suite involves chaining multiple rounds of dialogue between the LLM and the
836   testbench framework, we have done a comprehensive evaluation/ablation of the hyperparameters
837   involved to facilitate this agentic behaviour. The majority of these hyperparameters stem from the
838   various prompting engineering techniques/optimizations involved. In fact, many of these prompting
839   techniques are in existing protocols or usage guides on LLMs (Microsoft; OpenAI).
840
841   Maximum message number (700): We ran a set of “infinite-message experiments” on the Primitive
842
       Data Prefetcher Core and Ibex CPU Instruction Decoder, where the maximum response number
       is much larger than the average exhaustion threshold. As illustrated in the results in Figure 4 and
843
       5, the coverage values of all runs plateau after at most 500 messages, hence we set the maximum
844
       message number to 700 for a safety margin.
845
846
      Trial termination condition (no hits in 25 messages or fewer than 3 hits in 40 messages): This
       prevents over-using the resources when the agent is “exhausted”. Extensive tests have demonstrated
847
       that LLMs struggle to score additional hits after 25 non-scoring messages or to show significant
848
       performance improvement if recording fewer than 3 hits in 40 messages. Typically, we regard this
849    as a “low activity measure” beyond which we ask the model to stop.
850
      Number of preceding messages (3 responses): This was determined by considering both the con-
851
       text length restrictions of current LLMs and the typical length of prompts and responses. Maintaining
852
       three prompts and responses usually ensures that the maximum context length is not exceeded while
853    retaining as much of the previous dialogue as possible.
854
      Dialogue restarting tolerance (t = 4, 7, 10): We decide to restart the whole dialogue when the “low
855
       activity measure (< 3 hits)” is observed in t continuous message queries, since we empirically
856
       observe that LLMs’ responses can be trapped into local minima. t values are chosen as a compre-
857    hensive range in the suitable range below the trial termination condition (since we’d expect to see
858    multiple dialogue restarts before trial termination), with our ablation experiments showing t = 7
859    stably performs the best.
860
      Number of few-shot examples: For few-shot prompting, different coverage plans have different
861    “types” of bins. These types are outlined in Appendix A.4. Few-shot prompting is most efficient
862    when one example is given for each bin type. In practice, the number of few-shot samples equals to
863    the bin types for that specific coverage plan.

904

909

911                   responses. The number of 20% is obtained empirically.
