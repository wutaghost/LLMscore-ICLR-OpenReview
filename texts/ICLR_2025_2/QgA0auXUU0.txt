009                                                 A BSTRACT
010
011            Which parts of pre-target input1 are most influential for next-token prediction
012            in the context of programming languages? In this paper, we present evidence
013            that code snippets at specific locations in pre-target inputs play a decisive role
014            in large language model (LLM) inference, and these snippets exhibit a consistent
015            pattern. Firstly, we introduce a novel causal tracing method to identify tokens,
016
               so-called high-information tokens, that significantly contribute to next-token pre-
               diction. Building on this, we propose a multi-phase causal tracing process to ana-
017
               lyze the importance distribution of high-information tokens, revealing a consistent
018
               pattern, named the Important Position Rule (IPR). To further validate this hypoth-
019            esis, we assess the role of IPR across various LLMs, languages, and tasks. Our
020            extensive evaluations for code translation, code correction and code completion
021            tasks (Java, Python, C++) on models CodeLlama-7b/13b/34b-Instruct (Roziere
022            et al., 2023) and gpt-3.5/4-turbo (Ouyang et al., 2022), confirm this hypothesis.
023            Furthermore, we observe that IPR exhibits structural and semantic properties sim-
024            ilar to the ⟨subject, relation, object⟩ paradigm in natural language. Leveraging
025            this insight, we successfully combine IPR with the knowledge editing method
026            ROME (Meng et al., 2023) in order to repair translation errors, achieving a cor-
027
               rection rate of 62.73% to 75.31%. To our knowledge, this is the first application
               of knowledge editing in the context of programming languages.

030   1    I NTRODUCTION
031
032   Recently, researchers have delved into the internal mechanisms of large language models. The inher-
033   ent syntactic structures in natural language, such as ⟨subject, relation, object⟩, provide solid support
034   for interpretable inference (Kim et al., 2024; Stolfo et al., 2023; Katz et al., 2024; Haviv et al.,
035   2023) and knowledge editing (Meng et al., 2022; Zhang et al., 2024; Meng et al., 2023; Gupta et al.,
036   2024). However, the existence of analogous mechanisms in code-based LLMs remains uncertain,
037   which poses challenges for the aforementioned research in programming language contexts. In this
038
      paper, we demonstrate that code snippets located at specific positions play a crucial role in guiding
      LLM-based code inference, and that these snippets exhibit a consistent pattern.
039
040   In the field of natural language processing (NLP), GPT models have exhibited a remarkable ability
041   to learn and utilize syntactic structures, which are essential for establishing internal correlations
042   between words (Petroni et al., 2019; Rai et al., 2024; Bajpai et al., 2024). For example, given the
043   prompt “The Eiffel Tower is located in”, GPT can accurately predict “Paris”. It underscores the
044
      importance of syntactic structures in guiding inference and enhancing predictions (Mikolov et al.,
      2013; Touvron et al., 2023; Ouyang et al., 2022; Roziere et al., 2023). This leads us to the Core
045
      Research Question: Is there a consistent pattern that significantly contributes to LLM-based code
046
      inference? To address this question, we first quantify the information content of individual tokens
047   to assess their contribution to the inference process. Subsequently, we discover the importance
048   distribution pattern of high-information tokens and examine how this pattern influences the inference
049   process, ultimately providing new insights into the inner workings of LLM-based code inference.
050
      Existing tracing methods primarily focus on perturbing the training or test datasets to identify high-
051
      information tokens based on changes in generated tokens. However, relying solely on target token

      changes to determine the impact of input tokens is a rather crude approach (Dong et al., 2019;
055   Brown et al., 2020; Devlin et al., 2018). On the one hand, the output of the Transformer model is a
056   probability distribution over multiple tokens in the vocabulary. Therefore, this approach overlooks
057   other input tokens that have a relatively high influence (Adler et al., 2016; Hao et al., 2021; Geva

061   In this paper, we propose introducing perturbations to the tokens in both the input sequence and
062   the generated prefix. By analyzing the resulting fluctuations in the internal representations, we can
063
      effectively quantify the information content of individual tokens.
064   In recent years, knowledge editing has garnered widespread attention in the field of natural language
065   processing (Mazzia et al., 2023; Wei et al., 2023). Meng et al. (2023) highlights that fine-tuning
066   the middle layer weights of GPT models enables them to rapidly learn new knowledge, such as
067   “The Eiffel Tower is located in Berlin”. We argue that the success of knowledge editing stems from
068   the properties of the latent space. After extensive training, the frequent ⟨subject, relation, object⟩
069
      structures in the training samples establish a robust association between the tokens “Eiffel Tower”
      and “Paris”. Therefore, when editing at the position of “Eiffel Tower”, effective information sub-
070
      stitution can be achieved. However, existing knowledge editing techniques primarily rely on the
071
      ⟨subject, relation, object⟩ structure, which is typically limited to the natural language. In this paper,
072   we find that code snippets identified by IPR in the input sequence and the output prefix exhibit strong
073   semantic and syntactic correlations, similar to the close relationship between phrases of “subject”
074   and “object” in natural language. Building on this observation, we apply IPR in knowledge editing
075   to rectify errors in Java to Python translation. The main contributions of this paper are the following:
076
077          • We propose a causal tracing method that interacts the low-dimensional text sequence with
078            the high-dimensional internal representation. This approach quantifies the information
079            content of individual tokens in both the input sequence and the generated output prefix,
080            enabling an assessment of each token’s contribution to the next-token prediction.
081          • We introduce a multi-phase causal tracing process, revealing a consistent pattern of high-
082            information tokens, named the important position rule.
083
             • We validate the role of IPR in code inference across diverse models, tasks, and program-
084            ming languages, including code translation, code correction, and code completion, utiliz-
085            ing CodeLlama-7b/13b/34b-Instruct and gpt-3.5/4-turbo with Java, Python, and C++. Our
086            evaluation shows that code snippets identified by IPR play a critical role in next-token
087            prediction. We also confirmed their robust generalization capabilities, providing valuable
088            interpretability for LLM-based code inference.
089          • We combine IPR with ROME in the context of programming languages, generalizing this
090            knowledge editing beyond the NLP context. Our approach effectively corrected errors in
091            Java to Python translation, with a correction rate of 62.73% to 75.31%.

106   compression method, LLMLingua, which effectively captures the interdependencies among com-
107   pressed content. From a causal perspective, Feder et al. (2021a) introduced CausaLM, a framework

206   = f (x, y1:j−1 ).

237   3.2   C AUSAL T RACING M ETHOD
238
      Large language models can be viewed as intermediaries that transform low-dimensional text se-
239
      quences into high-dimensional internal representations. In this section, we introduce a causal trac-
240
      ing method designed to quantify the information content of individual tokens. This method involves
241   perturbing individual tokens in ⟨src func, tgt pref⟩ and employs the dot product similarity to analyze
242   fluctuations in the internal representation.
243
244          • Clean representation: Given a pre-target input cj , we utilize the Transformer model to
245            convert this clean pre-target input into an internal representation, denoted as kj .
246          • Corrupted representation: We corrupt an individual token in ⟨src func, tgt pref⟩, result-
247            ing in a corrupted pre-target input c′j . We then apply the Transformer model to convert this
248            corrupted pre-target input c′j into an internal representation, denoted as kj′ .
249
             • To assess the contribution of the corrupted token to the internal representation, we employ
250            dot product similarity sim() to measure the difference between kj and kj′ . This method
251            provides a metric for quantifying the information content of the token (Luo et al., 2018).
252            A lower similarity score indicates higher information content for the token, suggesting
253            that the corrupted token has lost more critical information, as the perturbation significantly
254            distorts the internal representation. In contrast, a higher score reflects lower information
255            content, implying a smaller impact on the internal representation.

      Python. As shown in Table 1, models such as CodeLlama-7b-Instruct, CodeLlama-13b-Instruct,

458   5     A PPLICATION : IPR- BASED KNOWLEDGE EDITING
459
460   In this section, we introduce an application of IPR: knowledge editing for LLMs in the context of
461   programming languages. We integrate IPR with the ROME approach (Meng et al., 2023), effectively
462   correcting errors in Java to Python translations by updating middle layer weights.
463
464   5.1   IPR-BASED ROME APPROACH
465
466
      Existing knowledge editing techniques are constrained by the inherent ⟨subject, relation, object⟩
      structure of natural language. In this paper, we elucidate the crucial role of IPR in next-token predic-
467
      tion, where IPR-based code snippets, src frag and tgt frag, exhibit significant semantic and syntactic
468
      correlations. Furthermore, the core token in src frag exhibits analogous functionality and purpose
469   to the target token, similar to the relationship between “subject” and “object”. This finding indicates
470   the potential of IPR to support knowledge editing in the context of programming languages.
471
472
      Building on this, we integrate ROME with IPR to perform knowledge editing on the mid-layer feed-
      forward module of the CodeLlama-7b-Instruct model, thereby correcting errors in Java to Python
473
      translation. Specifically, for a failed Java-Python translation pair, we consider the error as the target
474
      token and employ the IPR to extract the corresponding code snippets ⟨src frag, tgt frag⟩ as the basis
475   for knowledge editing. For the tuple ⟨src frag, tgt frag, corrected error⟩, we treat the core token of
476   src frag as the “subject” and the corrected error as the “new object”, thereby generating a new key-
477   value pair (k ∗ , v ∗ ). This key-value pair allows us to update the weight matrix using the equation
478   W k ∗ = v ∗ , where k ∗ and v ∗ are defined as follows:

532   6         C ONCLUSION
533
534   In this paper, we introduce a causal tracing method to identify high-information tokens, enabling a
535   precise understanding of how Transformer models extract information from pre-target inputs in the
536   context of programming languages. Based on this, we reveal a consistent pattern of high-information
537   tokens, named the important position rule. Then, we thoroughly evaluated the performance of IPR
538   across various models, tasks, and programming languages. Furthermore, we successfully applied
539   IPR to knowledge editing, proving that IPR can provide support for knowledge editing and inter-
      pretable inference.

886   A.1.3      JAVA -P YTHON TRANSLATION DATASET
887
888   Necessity of Building a Large-Scale Java-Python Translation Dataset: In Section 5, we aim to
889   collect common errors made by CodeLlama-7b-Instruct when translating Java to Python. While
890
      CodeLlama-7b-Instruct achieves a translation accuracy of 88.21% for Java → Python, commonly
      used datasets offer limited samples, with only 164 examples in HumanEval (Chen et al., 2021)
891
      and 974 examples in MBPP (Austin et al., 2021). Relying solely on these two datasets to identify
892
      common errors is unreliable, as their limited sample size does not support a comprehensive analysis
893   of main errors, which may lead to biased results.
894
895   Construction Process of the Java-Python Translation Dataset: Following the work of Roziere
896
      et al. (2022), we built a Java-Python translation dataset7 . First, we download the Java source code
      from Google BigQuery and apply the TransCoder-ST preprocessing pipeline for dataset filtering.
897
      Next, we set a maximum runtime of 20 seconds for each process and use EvoSuite to generate Java
898
      test cases. Here, the unit test cases are created based on two criteria: a mutation score above 0.9 and
899   at least two assertions. Through this process, we collect 82,665 Java functions with corresponding
900   unit test cases. Subsequently, we feed the processed Java functions, along with their corresponding
901   test cases, into TransCoder-ST (Java → Python, beam size = 1).
902
      For each Java function, we generate a corresponding Python function along with its associated unit
903
      test cases. Then, we execute the Python unit tests to validate the correctness of the translation. If
904
      the Python function passes all tests, we pair the Java function with the Python function to form a
905   translation language pair. This process ensures the accuracy of both the Python functions and their
906   corresponding unit tests. After removing duplicate entries from the processed dataset, we obtain a
907   final Java-Python translation dataset comprising 25,227 language pairs, each accompanied by unit
908   test cases. (See example in Table 10)

933
934   To explore the role of IPR-based code snippets in next-token prediction, we consider the follow-
935   ing two methods in code completion and code correction tasks: (1) removing the IPR-based code
936
      snippet from ⟨src func, tgt pref⟩, referred to as the IPRremove code snippet; (2) corrupting the IPR-

937
      the IPRremove / IPRcorrupt snippet dataset as input to the LLM and compute the success rate by
938                                     ′

939
      comparing the generated token yj with the original target token yj .
940   As shown in Tables 8 and 9, we find that when we remove or corrupt the IPR-based code snippets,
941   the success rate consistently drops below 8%. Further, we considered whether it stemmed from the
942   short length of ⟨src func, tgt pref⟩. Specifically, after removing 12 tokens (i.e., the length of the
943   IPR-based code snippets), the remaining valid tokens in ⟨src func, tgt pref⟩ were too few, leading to
944   a significant impact on next-token prediction. Based on this, we counted the length of src func in
945
      both the code translation and code correction tasks in Table 5. We found that in the code translation

946
      src func samples exceed 30 tokens. It means that, despite the fact that the lengths of the IPRremove
947
