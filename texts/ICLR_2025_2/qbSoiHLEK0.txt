011                                               A BSTRACT
012
013            Automatic Machine Learning (AutoML) is the popular supervised learning ap-
014            proach for tabular data. One of its key components is generating the most suitable
015
               features given the available training dataset. To overcome the disadvantages of
               existing automatic feature generation techniques, such as lack of generality and
016
               interpretability, we propose the novel approach, LLM2Features. It uses LLMs
017
               (Large Language Models) to generate meaningful features using automatically
018            collected statistics about the dataset without explicitly describing the data, making
019            it ideal for implementing in AutoML frameworks. In particular, we introduce the
020            LLM-based critic that additionally verifies the presence of syntax or logical errors.
021            The experimental study demonstrates the benefits of the proposed LLM2Features
022            approach in accuracy and training time compared to the state-of-the-art feature
023            generation tools.

      1   I NTRODUCTION
027
      Nowadays, AutoML is widely used for training machine learning models on tabular (structured)
028
      data (Erickson et al., 2020; Fakoor et al., 2020; Li et al., 2021) as they allow to achieve high-quality
029
      results in several lines of code without the need to be an expert in choosing algorithms and their hy-
030   perparameters. One of the challenging steps in AutoML is the automated feature engineering (Mu-
031   muni & Mumuni, 2024) that can generate the most informative features for concrete task (Luo et al.,
032   2019; Silva & Silva, 2023).
033
      Existing feature generation methods have several disadvantages, namely, the need to input additional
034
      data from the human (Kanter & Veeramachaneni, 2015; Hollmann et al., 2024) or the impossibility
035
      of enriching the data sufficiently well without losing the interpretability of generated features (Zhang
036   et al., 2023; Horn et al., 2020; Li et al., 2022). Hence, they may not be suitable for practical appli-
037   cations when data analysts have already designed a list of valuable and interpretable features. As a
038   result, the most popular AutoML frameworks (Feurer et al., 2020; LeDell & Poirier, 2020; Vakhru-
039   shev et al., 2021) use existing feature generation frameworks (Kanter & Veeramachaneni, 2015) or
040   traditional data pre-processing (Qi et al., 2023) for categorical variables, correct type conversion,
041   etc.
042   This paper studies LLM (Large Language Model)-based automated feature generation techniques
043   for the AutoML model that fits the generated features (Han et al., 2024). LLMs have been trained
044   on a much larger amount of data including feature generation code, and have a good representation
045   of the world they can use when generating features. The first successful application of LLM is the
046   CAAFE (Context-Aware Automated Feature Engineering) framework (Hollmann et al., 2024). Un-
047   fortunately, it requires a detailed data description, so it cannot be implemented in typical AutoML
048   solutions without human interaction. Moreover, the generated features sometimes lack meaningful-
049   ness and interpretability. Moreover, it is even possible that generted features contains mistakes of
050
      logical errors.
051   In this paper, we propose to use only the provided dataset itself without the need for any additional
052   information. In particular, our main contribution is the novel approach, LLM2Features, which au-
053   tomatically extracts essential statistics from the dataset and feeds them into the prompt for feature
      generation using high-quality LLM, such as GPT-4o or GPT-o1. It is experimentally shown that

      LLM-based feature generation for popular LightAutoML framework (Vakhrushev et al., 2021) has
055   much better quality metrics and human interpretability of the features when compared to tra-
056   ditional feature engineering frameworks. Therefore, the proposed method can be used not only for
057   generating interpretable features for AutopML but also for introductory exploration of data in an
058   unknown domain to the analyst.

      2.1   P ROBLEM S TATEMENT
063
064   Given a dataset D = {(x1 , y1 ), (x2 , y2 ), . . . , (xn , yn )}, where xi ∈ Rm is an m-dimensional in-
065   put vector, and yi is the corresponding target variable, the goal of feature generation is to find a
066   function z = f (x), where z is a new k-dimensional feature vector. The objective is to maximize
067   the predictive performance of a model M trained on the new feature space: max P (M (D′ )),
068   where D′ = {(z1 , y1 ), (z2 , y2 ), . . . , (zn , yn )} and P is a performance metric (e.g., accuracy, ROC-
069   AUC, RMSE). An additional requirement to automated feature generation is to minimize the level
070   of Human Involvement (H.I.). In this paper, we use three different values for this metric:
071
             • 0 is just to load the data (pd.DataFrame (Wes McKinney, 2010)). The best suitable method
072
               for use with AutoML
073
074
             • 1 is to describe the data with free-form text (where the data comes from, what the nature of
               it is) to the prompt
075
076          • 2 is to preprocess the features with code (fill in the omissions, remove anomalies, cast the
077            types (e.g., featuretools requirements)

079   2.2   AUTOMATIC F EATURE G ENERATION
080
      There exist two types of feature generation techniques, which 1) maximize the quality metrics by
081
      arbitrary transformations of features leading to the lack of interpretability (Bosch et al., 2021), or 2)
082
      generate logical, interpretable features by using knowledge of the world and data (Gosiewska et al.,
083   2021). Among the first type of techniques, it is necessary to mention AutoFeat (Horn et al., 2020)
084   feature generation by repeating various operations on one or a pair of features and using a built-in
085   selector to select only helpful features. The OpenFE (Zhang et al., 2023) follows a similar principle
086   but makes it faster with a specially developed boosting selector for a deeper understanding of the
087   feature importance to the model in a further generation. The second technique that brings new infor-
088   mation into the data is the featuretools library (Kanter & Veeramachaneni, 2015), which generates
089   new features, including multi-level features (connecting features by some operations) according to
090   pre-defined rules inspired by real-world scenarios. The pre-defined rules include the interaction with
091
      dates, coordinates, age, and address. Another interesting example is the FETCH (Li et al., 2022) that
      trains a single neural network to predict correct feature transformations for any tabular dataset, al-
092
      lowing us to accumulate knowledge about the most useful feature-dependent transformations.
093
094
      2.3   L ARGE L ANGUAGE M ODELS
095
096   LLMs are models intended for understanding, interpreting, and generating human-like texts (Tou-
097   vron et al., 2023; Team et al., 2024). In this paper, we use the family of ChatGPT models developed
098   by OpenAI, which are trained to perform human-like conversations and assist in various tasks. It is
099   a relatives model of InstructGPT (Ouyang et al., 2022), designed to follow instructions in prompts
100   and provide detailed answers. These models typically obtain state-of-the-art in the field of LLM
101   models, so we decided to use them in our framework.
102   LLMs show significant cross-domain knowledge capabilities, i.e., they can transfer and apply knowl-
103   edge across different domains or subject areas and solve complex problems in various fields. Knowl-
104   edge can be tested by examinations (Newton & Xiromeriti, 2024). LLMs are trained on huge
105   amounts of data spanning multiple domains, allowing them to develop broad knowledge that can
106   be applied to various tasks and topics. According to different estimates, the size of a training sample
107   starts from 600 GB and is obtained from multiple Internet sources (including Wikipedia). Trillions
      of LLM parameters (Allen-Zhu & Li, 2024) allow for structured summarization of global Internet


                                                          2
      Under review as a conference paper at ICLR 2025


108
      information, therefore it is of great interest to use LLM in the context of feature generation, because
109   the knowledge of a data analyst in the domain-specific data domain will most often be lower than
110   LLM.
111
112
      An advanced prompting technique LLMs use to increase their ability to reason and solve problems
      is the Chain of thought (CoT). It motivates LLMs to decompose complex problems into intermediate
113
      steps by mimicking human reasoning processes. CoT helps LLMs solve complex problems (Feng
114
      et al., 2023). It involves asking the model to “think step by step” when answering questions or
115   solving problems. This technique uses the model’s general knowledge to improve its performance
116   on tasks that require logic, computation, and decision-making. We add instructions like “Describe
117   your logic and reasoning” to the prompt. Further in the paper, we ask LLMs to generate useful
118   domain information to generate features based on it, too.
119
120
      2.4   LLM ACTOR VS . LLM CRITIC
121
122   Industry practice and research (Gou et al., 2024) demonstrate that if we verify the answers of an
123   LLM with another LLM by assigning the conditional roles of “actor” and “critic” in advance, i.e., a
124   model that solves the problem and a model that catches errors in the solution of the first model, then
125   we can improve the quality of text generation, find mistakes in advance, and improve the quality
126   metrics in different kinds of tasks. Further in the paper, we propose to use LLM not only for feature
127   generation but also for error catching and feature correction.
128
129   2.5   I NTERPRETABILITY
130
131   Interpretability of features is important in the context of machine learning tasks. It is often neces-
132   sary not only for the quality of the final feature pipeline but also for understanding its background
133   and possible values to build a stable implementation. Most existing non-LLM approaches (except
134   featuretools) do not provide proper interpretability of features, so they are selected based on opti-
135   mized oversamples to improve the final model’s prediction quality without caring about possible
136   degradation of the scoring over time. This is why the LLM approach is attractive, as it can describe
137
      the reason for generating a particular feature, not only to generate the maximum number of features
      useful for the quality of the model (to be shown later in the paper). In addition, LLM models can be
138
      attempted to be interpreted (Singh et al., 2024) by delving deeper into the causes of generation

147

148
      model prior knowledge to generalize from limited task-specific data. (Parnami & Lee, 2022). The
149
      following statistics are collected from the human data: column information (data types), number
150   of omitted values in columns, random sampling of records, distribution of values in features, and
151   correlation in data for numerical features. Also, we request LLMs to check their answers, write all

153   themselves in the [FEATURES] section, which increases the total number of instruction followings
154   when generating features

158   Python standard modules or numpy (Harris et al., 2020), pandas, and geopy. If these packages are not
159   installed, the code check unit will drop the generated feature. Next, the generated features from the
160   training and testing set are computed for each example. In case of any error, the feature is not added.
161   If less than three correct features are generated, the request to GPT-4o is sent again. These generation
      tasks can also be effectively performed by an LLM-critic checking the generated response. The

290   4.1     E XPERIMENTAL SETTINGS
291
292   The proposed LLM2Features approach is implemented in two supported scenarios: 1) with a de-
293   scription of data, including information about domain and attributes, and 2) without data descrip-
294   tion, where features are generated only based on statistics from the dataset. We use two state-of-the-
295   art LLMs from OpenAI, namely, GPT-4, GPT-4o (Achiam et al., 2023), GPT-o1-preview(OpenAI,
296
      2024), that can better follow rather complex prompt (Table 1).
297   In addition to using initial features, we compare our pipeline with several state-of-the-art feature
298   generation techniques, such as OpenFE (Zhang et al., 2023), AutoFeat (Horn et al., 2020) and
299   Featuretools (Kanter & Veeramachaneni, 2015). Moreover, we used an official LightAutoML
300   Pipeline2 (Vakhrushev et al., 2021), which is an example of a classic approach for AutoML frame-
301   works. It simply encodes categorical features, transforms some data, and boosts selectors. Finally,
302
      we implemented the state-of-the-art LLM-based feature generator with the domain knowledge about
      the dataset, CAAFE (Hollmann et al., 2024) with GPT-4 and GPT-4o. Special preprocessing was
303
      only applied to the data if the method for feature generation did not work without accurate type
304
      conversion, omission, and anomaly correction. Omissions were filled with median (statistics were
305   counted with the condition of not allowing leakage of test data), feature types were corrected by the
306   meaning of the feature and the needs of specific algorithms for feature generation (for example, the
307   basic implementation of featuretools requires type conversion using woodwork 3 ).
308
      The proposed approach is implemented in two settings:
309
310           1. LLM2Features with human input: with domain information, attributes, and data
311
              2. LLM2Features without human input: without domain information, attributes, and data
312
                 (attributes are generated only based on statistics from the dataset). It is the most suitable
313              for application with AutoML frameworks.
314

316   tasks (Table 3) that are widely used in various papers (Hollmann et al., 2024; Katz et al., 2016; Kaul

318   et al., 2021) v0.3.8.1, to train classification and regression models, which has recently won the Kag-
319
      gle’s AutoML Grand Prix 2024. We compute traditional metrics, namely, F1-score and ROC-AUC
      (Area Under the ROC Curve) for classification and RMSE (Root Mean Squared Error) and MAPE
320
      (Mean Absolute Percentage Error) for single regression. Moreover, we estimate the performance

467   5        C ONCLUSION
468
469
      This paper proposes a novel LLM-based automatic feature generation approach for AutoML with
470
      tabular data (Fig. 1). We experimentally proved that the proposed approach lead to better perfor-
471   mance (Tables 4, 5) when compared to conventional state-of-the-art feature generators. It is impor-
472   tant to emphasize that we improved metrics not for outdated ML models but using the contemporary
473   LightAutoML framework (Vakhrushev et al., 2021). Moreover, our method generates interpretable
474   features for regression and classification tasks based only on feature names and data statistics with-
475   out an additional description (Table 7). We are the first to present this extremely important property
476   for LLM-based auto feature generation. As a result, our LLM2Features pipeline can be integrated
477   into an arbitrary AutoML pipeline for complete no-code ML. Also, the described approach can be
478   used for preliminary data analysis. The LLM generates valuable features for AutoML and allows
479
      humans to explore unknown data in greater detail. The source code of our experiments will be made
      publicly available4 .
480
481   In future, it is necessary to extend our LLM2Features approach, e.g., use more examples of feature
482   generation in the prompt. It is also important to highlight that our current approach sends all requests
483   to the LLM on a one-time request. Hence, it is possible to increase the accuracy by sending requests
484   to adjust features with their importance for the obtained AutoML model. Finally, applying our
