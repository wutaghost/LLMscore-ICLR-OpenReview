011                                              A BSTRACT
012
013            Automatic Program Repair (APR) aims to automatically fix software bugs, play-
014            ing an essential role in software development. While current research demon-
015            strates that Large Language Models (LLMs) excel in file-level program repair,
016            their effectiveness in repository-level program repair remains unexplored. Real-
017            world software projects, which often consist of multiple files, present significant
018            challenges for LLMs in identifying bugs and generating fixes due to the intricate
019
               project structures. To bridge this gap, we introduce R EPO F IX E VAL, a repository-
               level APR benchmark consisting of 160 real-world bug fixing suites from popular
020
               Python projects. R EPO F IX E VAL provides the original buggy programs, associ-
021
               ated issue reports, corresponding fixes, and unit tests to verify the correctness of
022            each fix. Based on the benchmark, we further propose a three-step evaluation
023            framework for LLM-based APR tools, encompassing (1) discovering issues from
024            execution failures, (2) localizing buggy code segments, and (3) generating code
025            fixes. Experimental results highlight that LLMs struggle with organizing error
026            messages during the issue discovery phase. We find that longer contexts positively
027            affect performance, but only a few LLMs can effectively utilize extended context
028            information at the 128K level. Some open-source LLMs demonstrate competi-
029            tiveness with closed-source counterparts, yet even the best-performing GPT4-o
030
               only resolves 12.3% of bugs. Our study reveals the capabilities and limitations of
               16 LLMs in handling repository-level bugs, providing valuable insights for future
031
               research in this field.

035   1   I NTRODUCTION
036
037   Automated Program Repair (APR) aims to automatically detect and generate code fixes (i.e.,
038   patches) for a given piece of buggy code. APR is crucial in enhancing the overall reliability of
039   software systems by accelerating the debugging process and reducing the chances of human error.
040   Over the years, APR tools have evolved from conventional rule-based methods (Huang et al., 2024)
041   to contemporary data-driven, intelligent approaches. Recent advancements in large language mod-
042   els (LLMs), such as GPT-4 (OpenAI, 2023) and Llama3.1 (Dubey et al., 2024) offer alternative
043   solutions for more complicated program bugs without relying on historical code patches.
044   Despite the numerous efforts in developing APR approaches, their evaluation is limited in an over-
045   simplified scenario of function-level or file-level bugs (e.g., DebugBench Tian et al. (2024)). Con-
046   sequently, there remains a gap between these benchmarks and real-world complicated development
047   environments, where developers often deal with multi-file projects. Resolving project-level program
048   bugs presents substantial challenges in comprehending the relationships between functions across
049   multiple files and localizing faults within extensive codebases.
050   To fill this gap, in this paper, we propose a practical repository-level program repair benchmark,
051   named R EPO F IX E VAL, consisting of 160 repository-level program bugs from 16 GitHub projects.
052   These bugs are selected from widely-used repositories with executable unit tests in high coverage,
053   ensuring their quality and reproducibility. Five experienced programmers further manually identify
      crucial bug fixes from commit histories. As a result, R EPO F IX E VAL collects buggy programs,

      R EPO F IX E VAL introduces a novel issue-aware scenario for program repair, inspired by bug-
090   reporting and bug-resolving processes in real-world software development. Rather than fixing “ex-
091   isting well-described error reports” from clients, as in SWE-Bench (Jimenez et al., 2023), this setting
092   requires LLMs to autonomously identify and summarize the bug symptoms based on current infor-
093   mation. As illustrated in Fig. 1 (Step 2), given a buggy repository and associated runtime error
094   messages, the program repair workflow involves three steps: (1) discovering issues from execution
095   results, (2) localizing faults within the codebase, and (3) producing patches to fix the bug.
096   To systematically understand how existing LLMs can resolve repository-level program repair prob-
097   lems, we assess their performance on each of the aforementioned steps. First, for issue generation,
098   we use the “LLM-as-a-judge” framework (Zheng et al., 2023a) to assess the reproducibility, rele-
099   vance, explanation, and overall of the produced issues. Second, we measure the accuracy of the
100   identified faulty lines against the actual ones for fault localization. Third, we evaluate the bug
101   patches by verifying whether the patched code passes all unit tests.
102   We evaluate 16 LLMs, systematically examining the impact of context lengths ranging from 2,000 to
103   128,000 tokens on performance, and set 14 different input settings for each length. When provided
104   with error messages, GPT-4o achieved the highest patch pass rate of 11.25%. Notably, we observed
105   that most LLMs demonstrated performance improvements within a context length of 16K tokens.
106   However, only selected LLMs maintained their performance capabilities at extended context lengths
107   of up to 128K tokens. Through rigorous quantitative analysis, we established significant correlations
      among the three key tasks: issue discovery, fault localization, and patch generation.

      2     R ELATED W ORK
124
125
      LLMs for Automated Program Repair. Automated Program Repair (APR) seeks to assist devel-
      opers by automatically generating patches for software bugs (Gazzola et al., 2018). APR tools pro-
126
      duce patched code based on the original code after identifying buggy locations (Fan et al., 2023; Wei
127
      et al., 2023; Ye et al., 2022a). There are three traditional APR tools types: template-based (Koyuncu
128   et al., 2020), learning-based (Ye et al., 2022b) and LLM-based methods. While template-based
129   tools require human-intensive labour to define fixing rules, learning-based tools can generate more
130   versatile and expressive patches after training on historical bug-fixing data. More recently, LLMs
131   such as Starcoder (Li et al., 2023), WizardCoder (Luo et al., 2023), mCoder (Chai et al., 2024) and
132   UniCoder (Sun et al., 2024) have shown potential for APR due to their exceptional performance and
133   remarkable generalization capabilities across a diverse range of code-related tasks (Prenner et al.,
134   2022).
135
136   Benchmarks for Automated Program Repair Popular APR benchmarks like Defects4j (Just
137   et al., 2014), QuixBugs (Lin et al., 2017a), DebugBench (Tian et al., 2024) and EvalGPTFix (Zhang
138
      et al., 2023) enable the evaluation of code LLMs’ auto repair ability but mainly focus on function-
      level or file-level tasks. CrossCodeEval (Ding et al., 2023) and RepoBench (Liu et al., 2023) have
139
      introduced cross-file tasks to address this limitation, but they only focus on code completion tasks.In
140
      the realm of repository-level code repair, SWE-Bench (Jimenez et al., 2023) stands out by focusing
141   on resolving issues present in existing GitHub repositories. Nevertheless, its scope is confined to
142   human-reported issues, lacking an evaluation of large language models’ ability to discover issues
143   autonomously and fault localization. ReposVul (Wang et al., 2024) has only introduced a repository-
144   level vulnerability dataset. Similarly, while RepoBugs (Chen et al., 2024) has delved into repository-
145   level code fix tasks, its evaluation methodology primarily emphasizes the final outcome rather than
146   the entire issue identification and code localization process. Moreover, RepoBugs relies on human
147   experts to judge the correctness of code modifications, lacking an automated evaluation mechanism.

149   3     R EPO F IX E VAL : A B ENCHMARK FOR R EPOSITORY- LEVEL C ODE F IX TASK
150
151   R EPO F IX E VAL is a benchmark containing repository-level program bugs. The task is to discover
152   issues, localize faults, and ultimately generate a patch that resolves the bug to pass the relevant tests.
153
154   3.1   B ENCHMARK C ONSTRUCTION
155
156   We construct R EPO F IX E VAL based on the following principles to ensure its representativeness of
157   bug-fixing practices: (1) Code should originate from real-world and active projects; (2) The project
158   should contain mature test suites for easier maintenance; and (3) Bug cases should be well-annotated
159   and of high quality.
160
161   Stage I: Real-World Code Repository Collection. Similar to (Li et al., 2024) and (Jimenez
      et al., 2023), we searched GitHub for high-quality open-source repositories, focusing on those with


                                                          3
      Under review as a conference paper at ICLR 2025


162
      well-known organizations and multiple source files to represent mainstream development environ-
163   ment. We chose Python repositories due to its popularity and robust testing frameworks.
164
165
      Stage II: Execution-based Filtering. We built a runtime environment for each repository to ex-
166   ecute unit tests, ensuring the functionality of the code. Repositories without automated test suites
167   (e.g., pytest, unittest) or with failed tests were excluded to guard the reproducibility of unit
168   tests in our framework.
169
170   Stage III: Annotation & Quality Control. Five experienced software developers annotated is-
171   sues using two methods: (1) identifying fix-related commits from git history, and (2) manually
172   injecting bugs to simulate real-world scenarios. Developers proposed issues and solutions for each
173   bug. Solutions that pass all unit tests are considered correct. An expert reviewed each case for
174   quality control. The final test case included a buggy codebase, issues, solutions, bug localization,
175   unit test code, and commands. For each case, issues were proposed from the user’s perspective, and
176   solutions were provided from the developer’s perspective, serving as reference answers.
177   In summary, R EPO F IX E VAL contains 160 bug cases, covering projects from natural language pro-
178   cessing, machine learning, database applications, algorithm implementation, and general tools. Ta-

191   3.2.1    TASK 1: I SSUE D ISCOVERY
192
193   An issue refers to a user-created report of a problem or bug, including a title and a detailed descrip-
194
      tion of the problem and steps to reproduce the bug. When users encounter bugs or identify potential
      improvements, they can create such issues in an issue tracking system (e.g., GitHub, Jira) to notify
195
      project maintainers. In this task, the model is asked to simulate the issue creation process based on
196
      bug symptoms.
197
198
      Model Input & Output Given code repository C and the error message from unit tests E, LLMs
199
      M are required to generate the corresponding issue i = Missue (C, E). The output issue i is struc-
200   tured in JSON which has title, description and explanation, ensuring a standardized
201   representation that can be easily disseminated among development tools and teams.
202
203
      Evaluation Metrics. Given the open-ended nature of proposing and evaluating issues, we employ
204   the LLM-as-a-judge (Zheng et al., 2023a) method as the standard for evaluation. For the given is-
205   sues, we primarily assess them from four perspectives: reproducibility Ir , relevance Iv , explanation
206   Ie and overall Io . Reproducibility (Ir ) assesses the presence and quality of step-by-step instructions,
207   scripts, or code references for reproducing the issue. Relevance (Iv ) evaluates the accuracy in iden-
208   tifying the bug, linking the problem to error messages, and providing an effective solution. Solution
209   explanation (Ie ) examines the clarity and completeness of the proposed solution, including its ra-
210   tionale and validation. Overall quality (Io ) provides a holistic assessment of the issue description’s
211   comprehensiveness and clarity.
212   Formally, let i, E represent model-generated issues and error messages respectively, and let P denote
213   the patch that can fix the bugs. We set the reference score for an answer as 8 points. Ix (where
214   x ∈ {r, v, e, o}) is evaluated on a scale from 0 to 10: Ix = Meval (i, P, E) The overall evaluation
215   score IAvg can be defined as IAvg = 41 (Ir + Iv + Ie + Io ). To align with the subsequent evaluation
      metrics, we multiplied the value of I by 10, resulting in a score range of 0 to 100.

      3.2.2     TASK II: FAULT L OCALIZATION
217
218   The subsequent step, fault localization, aims to identify the exact code lines for a particular de-
219   fect. Pinpointing the precise segment of code that leads to a specific issue requires not only a deep
220   understanding of the codebase but also an ability to effectively navigate the directory structure.
221
222
223   Model Input & Output For the current code repository C, corresponding issue i and error mes-
224   sage E, we prompt LLMs to pinpoint the code segments l requiring modification: l = Mloc (i, C, E)
225   In fault localization, we define two location granularities: (1) Filename: Identifies the specific file
226
      containing the error, serving as the initial debugging step. (2) Specific Code: Determining the ex-
      act line or code snippet needing change ensures targeted modifications, formatted as ’line number:
227
      content of the current line’.
228
229
230   Evaluation Metrics To evaluate the performance of LLMs in the fault localization task, we em-
231   ploy a set of comprehensive and multidimensional metrics. These metrics are designed to assess
232   different aspects of the generated outputs, including the accuracy of the JSON schema, filenames
233   and specific code content.
234
235   • If the model’s output is in the correct JSON format, we will use F1, precision, and recall metrics
236     for evaluation. We integrate the evaluation content into the form {ID}ˆ{filename}ˆ{line
237     number}, where ID represents the task identifier, filename is a string, ˆ is the delimiter, and the
238     line number indicates the specific lines of code to be modified. Each line requiring modification
239     constitutes a separate evaluation case, collectively forming a set. The ground truth set is denoted
240     as Sgt , and the model output set is denoted as Spd . We use the proportion of correctly predicted

249   3.2.3     TASK III: B UG F IXING
250
251   Unit testing is crucial for detecting errors in pre-deployment period, significantly reducing the effort
252   required to address bugs in later stages. R EPO F IX E VAL amis to generate a code patch that can fix
253   the identified bug while ensuring all unit tests pass and original functionality is maintained.
254
255
      Model Input & Output After issue discovery and fault localization, we focus on effective code
256
      repair. Inspired by SWE-Bench, we use LLMs to generate patch files instead of complete code.
257
      Given the code repository C, issue i, fault localization l, and error message E, we prompt LLMs to
258   produce patch p to resolve the bug: p = Mpatch (i, l, C, E)
259
260
261   Evaluation Metrics We employ a comprehensive evaluation process to assess patch effectiveness.
262   Initially, we verify the patch’s correctness by integrating it into the codebase without conflicts using
263   Git. Next, we run unit tests to confirm that the patch successfully resolves the bug.
264   To prevent potential exploitation where LLMs might modify test code to pass evaluations, we imple-
265   ment an additional two-step verification process: (1) Code Segregation: We separate the codebase
266   into functional code and test code. (2) Dual Testing: After applying the patch to the functional code,
267   we conduct tests using both original original (pre-patch) test code and potentially modified (post-
268   patch) test code. A patch is deemed valid only if it passes all tests in both scenarios, ensuring that
269   the fix is genuine and not a result of compromised test code. This rigorous approach guarantees that
      patches not only resolve the initial issue but also maintain overall system integrity.

      Code LLMs. We evaluate 16 instruct models with sizes ranging from 1.5B to 405B parame-
275   ters, including open/closed-source LLMs. For general models, we evaluate GPT-4o-2024-05-13,
276   GPT-4o-mini-2024-07-18 (OpenAI, 2023), Llama3.1 (Dubey et al., 2024), Qwen2 (Yang et al.,

278   2024), DeepSeek-V2.5 (Zhu et al., 2024), Codestral (mistralai, 2024) and CodeGeeX4 (Zheng et al.,
279   2023b).
280
281   Code Retrieval. Given that the content of most code repositories significantly exceeds the context
282   length limit supported by the models, it is infeasible for the models to read the entire repository
283   content at once. Therefore, we employ the BM25 (Robertson et al., 2009) retrieval algorithm to
284   retrieve relevant files. To include potential buggy code in the input for LLMs, we extracted relevant
285   code segments according to the error messages for each case and referenced issues.
286
287   Implementation Details. Our experimental setup utilizes 8 NVIDIA A100 (80G), leveraging the
288   vLLM (Kwon et al., 2023) as the inference backend. For the DeepSeek and GPT-4 models, we
289   employed the official APIs. To ensure the diversity of issue generation, we applied a temperature
290   setting of 0.95 and a top-k sampling of 0.2. To validate the ability of LLMs to correctly generate
291   JSON format, we adopted Loads@3. This process verifies the model’s output up to three times to
292   ensure conformity with JSON format. If the output fails to be parsed by json.loads in Python
293   after the third attempt, we retain the model’s original output.
294

349   several key insights into their performance across issue discovery, fault localization, and bug fixing.
350   As depicted in Table 4, we verify the performance of the model at the 128K token level based on the

      provided message information(imessage , lmessage , pmessage loc ).

467   To investigate the impact of input information on debugging performance, we conducted a total of

469
470   Feeding error messages does not necessarily lead to the improvement in discovering issues.
471   The inclusion of error messages appears to improve the reproducibility of model outputs, suggesting

474

479   sue considers multiple dimensions (Relevance, Explanation and Overall) beyond just Reproducibil-
480   ity. In processing error messages, models may focus excessively on technical details, potentially
481   compromising performance in other aspects.

      In this work, we propose a novel repository-level APR evaluation dataset, R EPO F IX E VAL, cre-
532
      ated through an annotation and verification process conducted by professional developers. This
533
      dataset contains 160 bug suites, including buggy code, comprehensive unit tests, related issues, and
534   bug-fixing patches. We are the first to introduce a full-process evaluation mimicking human bug
535   resolution, where models initially propose relevant issues and subsequently generate patches for
536   solutions. Systematic evaluations of existing LLMs on R EPO F IX E VAL reveal performance dispar-
537   ities between open-source and closed-source models. Additionally, our findings highlight current
538   LLM deficiencies in discovering issues, localizing faults, and generating patches, suggesting direc-
539   tions for improvement. This marks a significant advancement for developers using AI techniques to
      understand and debug effectively in real-world software development environments.

      A.1    R EPO F IX E VAL ’ S F EATURE
744
745   In contrast to the previous APR benchmarks, which suffer from limitations such as limited scope,
746   lack of end-to-end evaluation, and inadequate performance metrics, our R EPO F IX E VAL offers sev-
747   eral distinctive features that address these issues comprehensively in Tab.1.
748
      Unlike the above APR datasets, R EPO F IX E VAL introduces several distinctive features:
749
750   • Repository-Level Program Repair: R EPO F IX E VAL expands the scope of traditional APR
751     benchmarks by encompassing multi-file projects from real-world repositories. This requires mod-
752     els to identify and repair bugs across various modules and components within a repository, sim-
753     ulating a more interconnected and interdependent software system environment. Unlike conven-
754     tional datasets that focus on single-file or single-function tasks, this approach provides a more
755     realistic and challenging evaluation framework, highlighting the necessity for models to under-
        stand and analyze code spanning multiple files.


                                                       14
      Under review as a conference paper at ICLR 2025


756
      • Offer Unit Test-Based Evaluation: R EPO F IX E VAL utilize unit tests as a primary metric for eval-
757     uation. This methodology ensures that the generated patches not only syntactically integrate with
758     the existing code but also maintain functional correctness. By using unit tests, R EPO F IX E VAL
759     closely mimics real-world software development practices.
760
      • Adapt Issue-aware Debugging Pipeline: R EPO F IX E VAL places significant emphasis on the dis-
761     covery and assessment of issues in the repositories. Just as the issue reports bridge the customer
762     usage failures and the software development team’s repair efforts, we have similarly adapted such
763     an issue-aware pipeline for debugging. This feature ensures that the models are capable of identi-
764     fying and resolving issues, which can demonstrate very well to humans.
765
766   These unique characteristics of R EPO F IX E VAL aim to bridge the gap between theoretical APR re-
767   search and practical application, extending the capabilities of APR systems within real-world soft-
768   ware development settings. By encouraging models to handle complex, cross-file projects, R E -
      PO F IX E VAL contributes to the advancement of more robust and effective APR methodologies.
