009                                                 A BSTRACT
010
011            Given a task in the form of a basic description and its training examples, prompt
012            optimization is the problem of synthesizing the given information into a text prompt
013            for a large language model. Humans solve this problem by also considering the dif-
014            ferent facets that define a task (e.g., counter-examples, explanations, analogies) and
015            including them in the prompt. However, it is unclear whether existing algorithmic
016            approaches, based on iteratively editing a given prompt or automatically selecting a
017            few in-context examples, can cover the multiple facets required to solve a complex
018            task. In this work, we view prompt optimization as that of learning multiple facets
019
               of a task from a set of training examples. We exploit structure in the prompt
               optimization problem and break down a prompt into loosely coupled semantic
020
               sections. The proposed algorithm, U NI P ROMPT, (1) clusters the input space and
021
               uses clustered batches so that each batch likely corresponds to a different facet
022            of the task, and (2) utilizes a feedback mechanism to propose adding, editing or
023            deleting a section, which in turn is aggregated over a batch to capture generalizable
024            facets. Empirical evaluation on multiple datasets and a real-world task shows that
025            prompts generated using U NI P ROMPT obtain higher accuracy than human-tuned
026            prompts and those from state-of-the-art methods. In particular, our algorithm can
027            generate long, complex prompts that existing methods are unable to generate.

030   1    I NTRODUCTION
031
032   Given a task, choosing an input prompt is a key part of optimizing Large Language Model’s (LLM) per-
033   formance (Kojima et al., 2024; Yang et al., 2023). Minor changes in prompt can lead to performance
034   gains or losses, necessitating prompt engineering (Liu et al., 2023). Typically, manually-developed
035   prompts combine task description with a few in-context examples, along with modifiers like chain-of-
036   thought (Kojima et al., 2024). For greater accuracy, human prompt engineers spend considerable time
037   to identify errors with a current prompt, consider the different facets of a task (e.g., counter-examples,
038   explanations, analogies) that may fix those errors, and include them in the prompt. For instance, for a
039
      hate speech classification task, in addition to the definition, it may be helpful to specify the facets that
      lead to hate speech: the context of conversation, identifying intent, and differentiating hate speech
040
      from opinions or closely-related concepts such as vulgarity and profanity.
041
042   To avoid the above cumbersome manual process, recent work aims to automate the process of
043   generating natural language prompts that are also interpretable. Since language tokens are discrete,
044   this leads to a challenging discrete optimization problem with a combinatorial space of possible
045
      outputs. Techniques for prompt optimization can be divided in two categories: non-directional, e.g.,
      random search (Zhou et al., 2022; Zhang et al., 2023) and genetic algorithms (Yang et al., 2023; Guo
046
      et al., 2023), where the sampling of new input is “random” and does not explicitly aim to reduce error
047
      on a train set; and directional, where the sampling of new input depends on some error measure on
048   a representative train sample. Recently, more complex methods have been proposed in the second
049   category including reinforcement learning (Zhang et al., 2022a; Deng et al., 2022), updating prompts
050   using feedback from auxiliary LLMs (Hu et al., 2024; Pryzant et al., 2023), and optimizing the input
051   to a small language model that generates the prompt (Lin et al., 2024b; Chen et al., 2024). While all
052   these techniques focus on editing, adding, or deleting parts of a given prompt, they are developed
053   with the goal of obtaining a concise description of the task. None of these focus on ensuring that
      multiple facets of a task are added to the prompt.

      In this paper, we propose U NI P ROMPT, a prompt optimization method to cover diverse, multiple
084
      facets of a task and improve overall accuracy. To simulate the manual prompt engineering process,
085   we propose that prompts be constructed from individual sections, where each section may correspond
086   to a different facet that humans may consider for the task. Prompt editing proceeds at a section-level:
087   we can add, edit or delete a section from the prompt. Similar to Pryzant et al. (2023); Hu et al.
088   (2024), prompt edits are based on an auxiliary LLM’s feedback about example predictions with the
089   current prompt. We contribute two key insights in this feedback-based optimization process. First,
090   we find that the feedback on a single example or a randomly selected batch of examples does not
091   yield generalizable facet descriptions. Instead, we propose clustering the inputs beforehand and
092   creating mini-batches such that each mini-batch is sourced from a single cluster. Second, even with
093
      clustered batches, the feedback tends to overfit to specific examples or their properties. To generate a
      prompt edit that conveys a generalizable concept relevant to the task, we propose generating edits at a
094
      mini-batch level and then aggregating them at the batch level to yield the final edit (see Figure 1).
095
      While the two insights may appear simple, we find that they lead to a significant improvement in the
096   extracting diverse facets for a task.
097
098   We evaluate U NI P ROMPT on several benchmark tasks (QnA, math, code generation) where it
099
      consistently achieves higher accuracy than baselines and existing prompt optimization methods. On
      Ethos, a hate speech dataset, U NI P ROMPT obtains 94% accuracy whereas the next best method
100
      obtains 82%. Even though UniPrompt focuses only on the instruction and does not include any
101
      in-context examples, we find that its instruction-only accuracy is often higher than methods such as
102   DSPy (Khattab et al., 2024) that optimize both. In the few-shot setting, we also compare U NI P ROMPT
103   to MedPrompt (Nori et al., 2023), a state-of-the-art prompt composition method. We find that
104   U NI P ROMPT, requiring only one LLM call at inference time, obtains the same accuracy as MedPrompt
105   that requires five calls. If we allow multiple calls to U NI P ROMPT, we obtain over 4% accuracy gains.
106   Finally, we also evaluate U NI P ROMPT on a real-world semantic matching task in a web search engine.
107   Compared to the best manual prompt, the prompt generated from U NI P ROMPT leads to over 5%
      increase in accuracy on the negative class and nearly 2% accuracy increase overall.

121   2     T EXT- GRADIENT BASED PROMPT OPTIMIZATION : C HALLENGES , I NSIGHTS
122
123   State-of-the-art prompt optimization methods such as ProTeGi (Pryzant et al., 2023) and TextGrad
124   (Hou et al., 2023) iteratively optimize the prompt for a given task. They adopt the following procedure
125   at a high-level: (1) start with an initial prompt and a training dataset of ⟨question, answer⟩ pairs for
126
      the task, (2) randomly sample from the questions wrongly answered by the current prompt to form
      a batch, (3) use an expert LLM to obtain feedback on the random batch, (4) apply the feedback to
127
      the prompt. This procedure is illustrated in Figure 1 [Left]. In this section, we study when such
128
      optimization is feasible and what may be the issues with current prompt optimization procedures.
129
130
      2.1   W HEN IS DIRECTIONAL TEXT OPTIMIZATION FEASIBLE ?
131
132   Consider the class of sequential algorithms as outlined above. The objective is to improve the
133   accuracy of a given (black-box) solver LLM f : X → R that takes as input a prompt x ∈ X and
134   outputs the average accuracy on a validation set Dv . Since the set of prompts is combinatorially
135   large, we assume that all prompts can be embedded in a vector space such that distance between two
136   prompts in the space correspond to their semantic similarity. The prompt optimization problem can
137   be written as arg maxx∈X f (x; Dv ).
138   Previous work has shown that LLMs can be brittle to their input: changing the prompt slightly can
139   create a significant difference in performance (Zhuo et al., 2023). We want to understand if the
140   optimization problem is well-conditioned. Typically, conditioning can be determined by the Hessian.
141   However, since f is black-box, we approximate it by measuring sensitivity, or more specifically,
142   Lipschitz continuity near the optimal solution. Based on prior work on defining continuity of neural
143   networks (Mangal et al., 2020), we use a probabilistic notion.
144
145   Definition 1 Probabilistic Lipschitz Continuity (Mangal et al., 2020). Given a probability distribu-
146
      tion over inputs X , r ≥ 0, and a distance measure d such as ℓ1 or ℓ2 norm, a function f : X → R is
      (L, ϵ)-probabilistically Lipschitz with constant L ≥ 0, if
147
148                        Pr [d(f (x), f (x′ )) ≤ L. d(x, x′ ) | d(x, x′ ) ≤ r] ≥ 1 − ϵ.                  (1)
                         x,x′ ∼X
149
150
151   Note the focus on small changes in input through the parameter r. Intuitively, the Lipschitz property
152
      bounds the maximum change in f given a small change in input prompt. Typically, the lower bound
      of error for any sequential optimization algorithm over f is directly proportional to the Lipschitz
153
      constant L (Malherbe & Vayatis, 2017). Therefore, for faster convergence, it is desirable to have a
154
      low L, especially near the optimal solution.
155
156   Empirically, we estimate L by sampling task-relevant prompts so that they are close to the optimal
157   solution. Then we make small changes to the prompt such that the semantic meaning stays the same
158
      and measure the change in f (See Appendix A.1 for experimental details). We show the change in f
      per change in input for GPT4 and GPT3.5 models in Figure 2 for the Ethos, GSM8K and MedQA
159
      datasets. Assuming ϵ = 0.05, probabilistic Lipschitz constant L for GPT4 is < 1, whereas it is higher
160
      for GPT3.5. Thus, as the model sizes increases, the probabilistic Lipschitz constant decreases. So:
161
      Observation 1: Larger models are more amenable to prompt optimization.

      2.2   C AN WE DO BETTER THAN RANDOM BATCHING ?
163
164   Batching is standard in gradient-based optimization to obtain robust gradients. Text-gradient based
165   prompt optimization techniques such as ProTeGi (Pryzant et al., 2023) and TextGrad (Hou et al.,
166   2023) also adopt batching by randomly sampling from training examples where the solver LLM made
167   a mistake. In the early iterations of optimization, there can be many such examples — is random
168   batching sufficient to get meaningful feedback (i.e., the text gradient) from the expert model?
169   To investigate this, we consider the Ethos dataset where the task is to classify text as hate speech or
170   not. We start with a basic task description as the prompt, “Is the following text hate speech (1) or not
171   (0)?”, run it with GPT-3.5 on 200 examples, and then provide random batches of incorrect predictions
172   to GPT-4 to identify prompt edits. GPT-4 is prompted to provide edits to the current prompt such that
173   the errors are minimized. With a random batch, the feedback obtained is relevant for the task, but
174   fails to identify specific concepts. Example feedback include,
175
176            The instruction should include a clear definition of hate speech...
177            The instruction should include examples of hate speech, guidance on identifying
178            hate speech...
179
180   Instead, if we cluster the incorrect examples (see Section 3.1) and create clustered batches, we obtain
181
      the following feedback.
182
               The instruction should include..potential harm or violence implied, as well as
183            any discriminatory or derogatory language used...towards a particular religious
184            group.
185
               The instruction should include ...think about the impact of the statement on the
186            targeted individual or group.
187
               The instruction should include...language that prompts hatred or discrimination
188            towards a particular gender.
189
190   The first feedback captures the religious and harm-based aspect of hate speech whereas the second
191   captures the aspect of measuring impact on the targeted entity. This case-study suggests that the same
192   LLM is able to identify different facets due to clustered batches.
193
      Observation 2: Clustered-batching improves the quality of text gradients.

      2.3   C AN WE LEARN A GENERALIZABLE PROMPT FROM A LIMITED SET OF EXAMPLES ?
196
197
      The optimal prompt should consider all the aspects of the task. So, any prompt optimization
198   algorithm (or a human prompt engineer for that matter) may have to exhaustively sample from the
199   data distribution in many real-world scenarios. In practice, however, we have access only to a limited
200   number (a few hundreds) of training examples (as it requires exhaustive manual labeling). We present
201   such a case study in Section 4.4 that arises in search and recommendation pipelines. The task is
202   to infer if two search queries share identical intent or not. Here, the notion of “identical intent” is
203   captured implicitly in the human-labeled query-pairs.
204   In this data-constrained setting, consider the standard in-context learning (Brown et al., 2020), where
205   the prompt is composed of a simple task description and a set of labeled examples from the task.
206   There are many ways in the literature to select examples for in-context learning, including random
207   sampling and k-nearest neighbors. In our setting, perhaps it would be beneficial to consider the
208   marginal utility of examples (Zhang et al., 2022b; Gupta et al., 2023a), i.e., add examples where
209   the model fails rather than ones where the model already succeeds (see Table 8 in Appendix). This
210   suggests that one could use a greedy algorithm for iteratively optimizing the prompt by finding failing
211   examples and adding to the prompt. But would such a procedure converge to a good solution that
212
      generalizes to unseen examples?
213   As we saw in the example above, such a procedure based on incorrect examples does provide specific
214   facets, such as being focused on religious or gender groups. However, adding any of these texts
215   directly to the prompt may be too specific and may miss out on other groups that may be targeted by
      hate speech (e.g., groups based on ethnicity). A procedure to aggregate the feedback may encourage

      the expert LLM to propose edits that are generalizable. In this example, we provide the three feedback

      and generalizes it to any group. A two-tier feedback, therefore, can help in distilling important aspects

      of the task implicit in the examples, rather than directly use or rephrase the (limited) examples.

228   3     U NI P ROMPT: G ENERATE A P ROMPT TO C APTURE M ULTIPLE TASK FACETS
229
230   Our observations above indicate that collecting feedback over individual examples or randomly sam-
231   pled batches may lead to memorization of individual examples (see Figure 1) rather than recognition
232   of facets that are important to the task. To overcome these shortcomings, the proposed method,
233   U NI P ROMPT, makes two contributions. First, we follow a two-tier setup of synthesizing feedback
234   for a batch of training examples. We break up a batch into mini-batches, collect feedback on each
235   of the mini-batches and then use a separate prompt to aggregate the different feedback texts into a
236
      generalizable concept. Second, to increase chances that a mini-batch corresponds to a coherent facet,
      we cluster the training data beforehand and ensure that each mini-batch consists of examples from
237
      the same cluster. We provide details of the algorithm below (see Algorithm 1).
238
239   The algorithm receives as input a one-line task description and a set of ⟨question, answer⟩ demonstra-
240   tions. It extracts key concepts or facets relevant to the task and updates prompt sections using them,
241   with the overall goal of increasing validation accuracy. Our analysis in Section 2 indicates that such a
242
      directional optimization procedure based on iteratively adding task facets can converge to a good
      solution, especially for solver LLMs such as GPT-3.5 and GPT-4.
243
244   Notation: We denote the training set of N examples with Dt where each example is a question-
245   answer pair of the form (qi , ai ); and the validation set of K examples with Dv . Input to the algorithm
246   is the solver LLM fLLM , train set Dt , validation set Dv , initial prompt for the task p0 , one-line task
247   description T . In addition, we assume access to an “expert LLM” such as GPT-4.
248
249   3.1     TASK FACET LEARNING USING EXAMPLES
250
      Extracting task-relevant concepts from a set of examples to refine a prompt is a complex problem
251
      comprising multiple steps. Given a set of incorrect predictions, one needs to analyze what went
252
      wrong in each prediction, form hypotheses, aggregate the hypotheses to identify specific concepts
253   that are relevant for the task. Then, for each concept, one needs to attribute which facet/section
254   of the current prompt needs to be edited to incorporate the concept. These operations are highly
255   model-specific and are difficult to execute reliably. Therefore, we exclusively rely on an expert LLM.
256
      First, we prompt the expert LLM to diagnose mistakes (feedback) in each example given the answer
257
      and chain-of-thought reasoning produced by the solver LLM. Subsequently, we use this feedback to
258
      generate precise edits for the prompt that may fix the error. These individual edits are then aggregated
259   over a mini-batch and fed back into the same LLM, which then identifies a few major edits to be
260   applied to the current prompt. To aid in identifying major edits that correspond to generalizable
261   facets, we propose to cluster the examples as a preprocessing step and create clustered batches, such
262   that each cluster shares some common facet of the task.
263

269   on examples from the same topic. Hence, for such examples, a common edit to the prompt could

      3.1.3    E DITING THE PROMPT
333
334   Once the final set of edits is received for a batch, we use the expert LLM to apply edits to the current
335   prompt (see Appendix A.9 for the prompt). An edit is accepted only if it increases the validation
336   accuracy compared to the current prompt. We call this approach “Greedy”. Alternatively, we
337   maintain a beam of 2 best performing prompts based on validation accuracy, apply the edit to the two
338
      prompts, and update the beam to retain the top 2 performing prompts. We call this method “Beam”.
      To avoid overfitting on the train examples (or keep adding unnecessary information to the prompt),
339
      we employ early stopping in the optimization process (more details in Section 4).

      3.3     E FFICIENCY CONSIDERATIONS : C OMPUTATIONAL C OMPLEXITY
351
352   We now consider the compute complexity of the U NI P ROMPT algorithm for a given task in terms of
353   the number of expert or solver LLM calls made per epoch, stage-wise.
354
      Clustering: First, we evaluate all the training examples using the current prompt. Second, for every
355
      wrongly predicted example, we obtain feedback from the expert LLM. Third, for the given set of
356   feedbacks, we use a single call to cluster it into l clusters. Each of the above steps incurs O(N )
357   queries, so the total query complexity of the clustering stage is O(N ). Finally, for each example, i.e.,
358   (question, answer) pair, we simply map it to the l clusters (no LLM calls). This is a one-time cost.
359
      Mini-batch feedback and Batch-level aggregation: At a given epoch, we evaluate every question
360
      in the mini-batch using the current prompt and the solver LLM (N queries overall). Next, we obtain
361
      one feedback over all the wrong questions in the mini-batch m (N/|m| queries). We use one call to
362   aggregate these feedbacks. For prompt selection, we evaluate 4 prompts on the batch b (2 per beam),
363   so O(4|b|) queries per batch. Hence overall query complexity is N + N/|m| + 4N + 1 or O(N ).
364
365
      Assuming LLM throughput of 0.5 queries per second, a training + validation set of 300 examples, 10
      clusters, and 20 epochs, it takes under 7 hours to train.

      4.4    R ESULTS ON A R EAL - WORLD TASK : S EARCH Q UERY I NTENT
469
470   The task of inferring if two search queries share identical intent or not arises in search and recom-
471   mendation pipelines, and is tackled today using LLMs. It is challenging because it requires domain
472   knowledge (e.g., brands and product categories), and depends on cultural and geographical biases
473   (e.g., “cricket” and “cricket game” are likely to mean the same in UK, but unlikely in the US). So,
474   examples are crucial for understanding the task and engineering a prompt that generalizes well.
475   We sample real user queries from a proprietary application, rewrite them using ML models, and ask
476   expert judges to label the query-pairs as identical or otherwise based on prescribed guidelines. We
477   use a set of 200 examples as training data, and an additional 50 examples as validation set, to learn
478   a prompt using U NI P ROMPT, starting from the one-line description: Tell if query A and query B
479   have same intent or not. The dataset is heavily biased towards positive samples, so the metric of
480   success is improvement in accuracy, over the best manually-engineered prompt, on the positive and
481   negative classes individually. For testing, we use a separate labelled set of 2527 examples from two
482
      geographies — one where the training data was sampled from, and the other unseen.
483   The prompt obtained using U NI P ROMPT improves over the best manual prompt by 5.77% on the
484   negative (rare) class, by 0.23% on the positive class, and by 1.86% overall on the test set. The
485   learnt prompt captures the following facets of the task: (1) recognizing variations in names and
      abbreviations, and how they do not necessarily change the context; (2) recognizing the specificity of


                                                        9
      Under review as a conference paper at ICLR 2025


486
      brands, and how even minor variations do change the context; and (3) recognizing the specificity of
487   terms in queries, and how lack of specific terms can indicate departure of intent.

489   4.5   A BLATIONS
490
491   Impact of Clustering, Inclusion of History, and Greedy Update The results are shown in Table
492   6. We see that clustering as well as edit history components (Section 3.1) are critical for performance
493   of U NI P ROMPT in all the datasets. We see a major drop of 14.8% in accuracy in the Ethos dataset
494   when clustering is removed, and a 4.3% drop when history component is removed. In all the datasets
495   except GSM8K, we find clustering is more important than history. This can attributed to limited
496
      variability of question types (all grade-8 arithmetic) in GSM8K than in others.
497   We also find that the greedy update rule (Section 3.1.3) proves to be superior or competitive compared
498   to beam search in relatively easier datasets — where even less exploration produces good results,
499   greedy proves to be a more effective update rule. On the other hand, in more complex datasets like
500   MedQA, greedy appears to be a bad strategy. We also see that clustering examples based on feedback
501   (“Fb Clustering”) is a better strategy than clustering based on topics, except for the Ethos dataset.
502

      Impact of initial prompt We consider three initializations of U NI P ROMPT in Table 5. A simple
      one-line task description initialization for U NI P ROMPT achieves the best accuracy on three out of
504
      four datasets. On the ARC dataset, initializing with the prompt generated by the Llama2-13B model
505
      (as discussed in Section 3.2) gives significant improvement over other initializations.
506
507   Expert model capacity U NI P ROMPT is not restricted to improving the prompts for less capable
508   (solver) LLMs using more capable LLMs as experts. In Table 4, we showed how U NI P ROMPT works
509   very well with GPT-4 as both the solver and the expert LLM. Table 3 summarizes the results for
510   different choices of expert and solver LLMs, on the Ethos dataset: U NI P ROMPT improves the prompt

513   5     R ELATED W ORK
514
515   Here, we highlight relevant work that are not addressed in the manuscript so far. Deng et al. (2022)
516   present a discrete prompt optimization method, RLPrompt, using reinforcement learning, where a
517   policy network learns to generate effective prompts through reward-based training, with an emphasis
518   on enhancing training efficiency through effective reward stabilization techniques. A drawback of
519   such automatic prompt optimization approaches (Pryzant et al., 2023; Zhou et al., 2022; Deng et al.,
520   2022; Yang et al., 2023) is that the prompts generated tend to be short, often comprising only one or
521   two sentences, which may not fully encapsulate the complexity of the task at hand.
522   Another recent line of work leverages human feedback in prompt optimization. Automated Prompt
523   Optimization with Human Feedback (Lin et al., 2024a) optimizes prompts for black-box LLMs using
524   human preference feedback. Besides the obvious overhead, it might also introduce potential biases.
525
      Prior research (Wei et al., 2023; 2024) has highlighted the significance of specific sections within
526   prompts. However, existing methods do not specifically target the optimization of individual sections
527   and their respective contents within the prompts. Hsieh et al. (2023) investigate the use of greedy
528   and genetic algorithms to edit lengthy prompts. Their method focuses on paraphrasing one line at a
529   time starting from an existing prompt, compared to our goal of learning facets of a task from scratch,
530   thereby compromising generalization accuracy (see Appendix A.12). Another orthogonal line of
531   work explores algorithmic selection of in-context examples (Min et al., 2022; Gupta et al., 2023b;
532   Wu et al., 2023; Srivastava et al., 2024; Sun et al., 2024).
533
534   6     C ONCLUSIONS
535
536   We presented a method inspired by the human prompt engineering process to generate complex
537   prompts from scratch that include different facets of a task. Our algorithm provides significant
538   improvements over baseline prompt generation methods on multiple standard datasets. Just like in-
539   context learning (Ji et al., 2024), task facet learning could also benefit from connections to submodular
      optimization (Krause & Golovin, 2014). We leave this as future work.

      To induce the ability of structured prompt generation in a smaller language model, we curate a
902   section-wise dataset of around 12,000 task-prompt pairs. The tasks for training dataset creation were
903   taken from tasksource library (Sileo, 2023) that contains around five hundred classification tasks. We
904   extract the task description from tasksource-instruct, which contains tasksource dataset recasted with
905   instructions. For instance, the task description for BIG-bench Entailed Polarity task is, "Given a fact,
906   answer the following question with a yes or a no". The dataset provides diverse tasks and their short
907   description, but not the human-generated prompts for each task. To approximate human-generated
908   prompts, we use GPT-4 as a teacher model.
909   By prompting GPT-4 with the task description and section description, we ask it to generate the
910   contents of the section. To ensure that the generated section-wise prompts are concise and relevant, we
911   prompt GPT-4 to not generate more than five lines of content for each section. We use LLAMA2-13B
912   model, which we finetune using LoRA adapters as the auxiliary LM that generates sections.
