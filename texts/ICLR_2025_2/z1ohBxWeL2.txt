009                                                       A BSTRACT
010
011                LLM inference for popular enterprise use cases, such as summarization, RAG, and
012                code-generation, typically observes orders of magnitude longer prompt lengths than
013                generation lengths. This characteristic leads to high cost of prefill and increased re-
014                sponse latency. In this paper, we present SwiftKV, a novel model transformation and
015                distillation procedure specifically designed to reduce the time and cost of processing
016                prompt tokens while preserving high quality of generated tokens. SwiftKV combines
017
                   three key mechanisms: i) SingleInputKV, which prefills later layers’ KV cache using
                   a much earlier layer’s output, allowing prompt tokens to skip much of the model
018
                   computation, ii) AcrossKV, which merges the KV caches of neighboring layers to
019
                   reduce the memory footprint and support larger batch size for higher throughput, and
020                iii) a knowledge-preserving distillation procedure that can adapt existing LLMs for
021                SwiftKV with minimal accuracy impact and low compute and data requirement. For
022                Llama-3.1-8B and 70B, SwiftKV reduces the compute requirement of prefill by 50%
023                and the memory requirement of the KV cache by 62.5% while incurring minimum
024                quality degradation across a wide range of tasks. In the end-to-end inference serving
025                using an optimized vLLM implementation, SwiftKV realizes up to 2× higher aggre-
026                gate throughput and 60% lower time per output token. It can achieve a staggering 560
027                TFlops/GPU of normalized inference throughput, which translates to 16K tokens/s
028
                   for Llama-3.1-70B in 16-bit precision on 4× H100 GPUs. Our training, inference,
                   and model implementations are open-sourced at https://anonymized.link.

031   1       I NTRODUCTION
032
033   Large Language Models (LLMs) are quickly becoming an integral enabler of enterprise applications and
034   offerings, including code and data co-pilots (Chen et al., 2021; Pourreza & Rafiei, 2024), retrieval aug-
035   mented generation (RAG) (Lewis et al., 2020; Lin et al., 2024), summarization (Pu et al., 2023; Zhang
036   et al., 2024), and agentic workflows (Wang et al., 2024; Schick et al., 2023). While it is clear that LLMs
037   can add value to these applications, the cost and speed of inference determine their practicality. There-
038   fore, improving the aggregate throughput and reducing latency of LLM inference has become an increas-
039   ingly important topic of interest, with various efforts (Sec. 2) tackling the problem from multiple angles.
040   In this paper, we take a unique approach to improving LLM inference for enterprise applications based
041   on the key observation that typical enterprise workloads process many more input tokens than output
042   tokens. For example, tasks like code completion, text-to-SQL, summarization and RAG each submit
043   long prompts but produce a small number of generated tokens, and a majority of enterprise LLM use
044   cases incur a 10:1 ratio between prompt and generated tokens.1
045   Based on this observation, we designed SwiftKV, which improves throughput and latency by: i)
046   reducing the computation required to pre-fill the KV cache for input tokens, and ii) enabling memory
047   savings to support larger batch sizes needed to serve LLMs more cost effectively (Sheng et al., 2023;
048   Pope et al., 2022; Yu et al., 2022). SwiftKV (Fig. 1) consists of four key components:
049
      SingleInputKV. SingleInputKV rewires an existing model so that the pre-fill stage during inference can
050
      skip a number of later layers in the network, and their KV cache are computed by a single earlier layer.
051   SingleInputKV is motivated by the observation that the output hidden states of the later transformer lay-
052   ers do not change significantly (see Sec. 3.2, also independently discovered by Liu et al. (2024c)). With

076   SingleInputKV, the computation required for pre-fill is reduced by approximately the number of layers
077   skipped. We found that it is possible to skip at least 50% of the layers without significantly impacting the
078
      model quality (Sec. 4.2), which translates to a 50% reduction of the pre-fill computation in inference.
079   AcrossKV. While SingleInputKV reduces the pre-fill computation, it does not reduce the KV cache
080   memory requirement. AcrossKV combines the KV projections from multiple adjacent layers into
081   a single one, and share the KV cache across these layers to reduce its size in memory. AcrossKV
082   allows significant memory savings, which unlocks higher throughput by enabling larger batches during
083   inference. In Sec. 3, we show that AcrossKV can reduce the KV cache size by 25% with less than
084
      a 1% quality gap. We also show in ablation studies (Sec. 5) that AcrossKV is compatible with existing
      KV cache quantization methods, which combine to unlock 62.5% reduction in KV cache size.
085
086   Knowledge Recovery. Although SingleInputKV and AcrossKV can be applied to existing LLMs
087   with minimal changes to their architectures, we found that the resulting model parameters should
088   still be adapted to the new architecture to recover their original prediction quality. This can be done
089   via distillation from the original model. With SwiftKV, we found that a lightweight distillation is
090
      sufficient, with only a fraction of the model (Q, K, and V projections of the affected layers) trained
      on 680M tokens, which takes less than 3 hours on a single 8× H100 node for Llama-3.1-8B.
091
092   SwiftKV Optimized Inference. To realize the computation and memory reductions of SingleInputKV
093   and AcrossKV into end-to-end throughput and latency improvements, we implemented SwiftKV in
094   vLLM (Kwon et al., 2023). Our implementation includes several additional optimizations, including
095   fusing all KV-projections beyond layer l into a single GEMM operation, and integrated memory
096
      management needed to lower the KV cache memory footprint achievable via AcrossKV.
097   SwiftKV increases the aggregate throughput of enterprise workloads by up to 2×, while reducing time-
098   to-first-token (TTFT) and time-per-output-token (TPOT) by up to 50% and 60%, respectively. In fact,
099   for Llama-3.1-70B, SwiftKV can achieve a normalized throughput of 560 TFLops/GPU2 . This is an
100   unprecedented 56.6% MFU utilization for inference (Sec. 4.3). We show that SwiftKV incurs minimal
101   quality degradation averaged across a wide range of tasks (Sec. 4.2), including ARC-Challenge (Clark
102
      et al., 2018), Winogrande (Sakaguchi et al., 2019), HellaSwag (Zellers et al., 2019), TruthfulQA (Lin
      et al., 2022), MMLU (Hendrycks et al., 2021), and GSM8K (Cobbe et al., 2021).
103

105   of trainable parameters for training SwiftKV. We also present our analysis of the hidden state

      Hardware and System Optimizations. Lower-precision quantization like FP8 (Kuzmin et al., 2024)
      can enable the use of tensor-cores to accelerate inference (Luo et al., 2024). System approaches like
119
      PagedAttention (Kwon et al., 2023), Tensor-Parallelism(Shoeybi et al., 2020), Split-Fuse (Holmes et al.,
120
      2024; Agrawal et al., 2024), FlashAttention (Dao et al., 2024), and their optimized implementations
121   in TensorRT (NVIDIA, 2019), FasterTransformer (NVIDIA, 2021), vLLM (Kwon et al., 2023), and
122   DeepSpeed-Inference (Aminabadi et al., 2022) enable better parallelization, batching, and scheduling to
123   eliminate performance overheads and achieve better hardware peak utilization without impacting model
124   quality. In contrast, SwiftKV is a model architecture optimization and is complementary to these works.
125
      Sparse attention optimizations. Systems such as ALISA (Zhao et al., 2024) and MInference (Jiang
126
      et al., 2024) leverage naturally-occurring sparsity patterns in transformer models to reduce the computa-
127   tion of the quadratic attention operation. Particularly, like SwiftKV, MInference also targets the prefill
128   phase of inference. Sparse attention can be particularly effective for very long sequence lengths (e.g.
129   100K - 1M tokens) when attention is the dominant operation. In comparison, SwiftKV reduces prefill
130   computation by skipping not just the attention operation, but also the query/output projections and
131   MLP of certain layers. This means that (1) SwiftKV can be more suited for inputs with more moderate
132   lengths when MLP is the dominant operation, and (2) SwiftKV, which either runs attention as-is or
133   skips attention entirely, is complementary to sparse attention methods which are more concerned with
134   the implementation of attention itself.
135   Memory Compression. A wide range of techniques have been developed to reduce the memory need
136   of inference. Lower-precision quantization techniques like FP8/FP4 can reduce the memory for both
137   KV cache and parameters (Hooper et al., 2024). Attention optimization techniques like MQA (Shazeer,
138   2019), GQA (Ainslie et al., 2023b), low-rank attention (Chang et al., 2024) also reduce the KV Cache.
139   These approaches are complementary to SwiftKV, which we demonstrate in Sec. 4.2 and Sec. 5.3.
140   Like AcrossKV, MiniCache (Liu et al., 2024b) also considers merging the KV cache of consecutive
141
      layers. However, AcrossKV enables consolidating more than just two layers, allowing for higher level
      of compression, and does not require any token retention strategy where distinct KV caches are stored
142
      for special tokens, allowing for simpler implementation.
143
144   Furthermore, while many of these approaches only focus on reducing the memory, SwiftKV can reduce
145   both the prefill computation (via SingleInputKV) along with memory. As we show in Sec. 5.1, compute
146   reduction rather than memory reduction is crucial for accelerating inference in compute-bound
147
      scenarios with sufficient memory, which is common in production with modern GPUs (H100, A100).

      In transformer models (Vaswani et al., 2017), attention enables each token to focus on other
      tokens by comparing queries (Q) with keys (K) and using values (V ) to compute the final
155
      representation. For a sequence of input tokens x(1) ,...,x(n) , the projections are defined as follows:
156
      Q = XWQ , K = XWK , V = XWV , where X ∈ Rn×d are the input embeddings, and
157
      WQ ∈ Rd×dk and WK ,WV ∈ Rd×dg are trained model parameters with dg |dk . Hereafter, we may also
158
      refer to WK and WV as a single matrix WKV ∈ Rd×2dk .
159
160   During the prefill phase of inference, the model processes the entire input sequence at once, computing
161   K and V for all tokens in parallel (or in chunks in the case of Split-Fuse (Holmes et al., 2024; Agrawal
      et al., 2024)). This typically occurs when the model handles an initial prompt or context.

180   During the decoding phase of inference, new tokens are generated one at a time. When predicting
181   the next token, only the query (Q(t+1) ) for the new token needs to be computed, while the model must
182   attend to the keys and values (K (1) ,...,K (t) , V (1) ,...,V (t) ) of all previously processed tokens.
183
184   To optimize efficiency in the decoding phase, KV caching is employed. After processing each token
185   t, the newly computed K (t) and V (t) are stored in a cache. For the next token t+1, only the new query
186   Q(t+1) , key K (t+1) , and value V (t+1) are computed. The attention computation will then utilize the
187
      cached K and V from all prior tokens, allowing for reduced computational overhead during inference.

239   3.4   K NOWLEDGE R ECOVERY
240
241   While SingleInputKV preserves all the original parameters, it re-wires the architecture so that the
242   KV cache projections may receive different inputs. We found that this re-wiring (and AcrossKV)
243   requires fine-tuning to recover the original capabilities from the modified model. As we only change the
244
      computation of the attention part for layer > l, this can be achieved by fine-tune just the WQKV weight
      matrices from the (l+1)-th layer onwards. However, instead of directly fine-tuning these parameters
245
      using standard LM loss, we find that distilling using the output logits of the original model allows for
246
      better knowledge recovery (see Sec. 5 for more details).
247
248   Implementing the Distillation. Since only a few WQKV parameters need fine-tuning, we are able to
249   do a memory efficient parameter-sharing based distillation. More specifically, we keep a single copy of
250
      the original model weights in memory that are frozen during training, and add an extra trainable copy
      of the WQKV parameters for layers > l initialized using the original model (See Fig. 1).
251
252   During the training, we create two forward modes for the later layers > l, one with original frozen
253   parameters using original architecture, and another with the SwiftKV re-wiring using new QKV
254   projections i.e.,
255             yteacher = M(x,Swif tKV = F alse), and ystudent = M(x,Swif tKV = T rue),               (3)
256   where y· is the final logits, M is the model, and x is the input. Afterwards, we apply the standard
257   distillation loss (L) upon the outputs with temperature (τ ) using (Hinton et al., 2015). After the
258   distillation, the original KV projection layers > l are discarded during inference.
259
      This method allows us to perform the distillation for Llama-3.1-8B-Instruct on 680M tokens of data in

      3.5   O PTIMIZED I MPLEMENTATION FOR I NFERENCE
264
265   LLM serving systems can be complex and incorporate many simultaneous optimizations at multiple
266   layers of the stack, such as PagedAttention (Kwon et al., 2023), Speculative Decoding (Leviathan et al.,
267   2023), SplitFuse (Holmes et al., 2024; Agrawal et al., 2024), and more. One benefit of SwiftKV is that
268   it makes minimal changes to the model architecture, limited to only a few linear projection layers. This
269   means that SwiftKV can easily be integrated into existing serving systems without implementing new
      kernels (e.g. for custom attention operations or sparse computation) or novel inference procedures.

      Implementation in vLLM. To realize the performance benefits of SwiftKV, we integrated it with
271   vLLM . Our implementation is compatible with vLLM’s chunked prefill, which processes prefill
272   tokens in chunks and may mix prefills and decodes in each minibatch. During each forward pass, after
273   completing layer l, the KV-cache for the remaining layers (> l) are immediately computed, and only
274   the decode tokens are propagated through the rest of the model layers.
275
      GEMM and Memory Optimizations. Upon this basic implementation, we implemented two ad-
276
      ditional optimizations. First, SingleInputKV fusion: instead of computing the KV cache KVj for
277                                                 j                                j>l
      each layer j > l one at a time, we fused all WKV into one large weight matrix WKV  so that their KV
278
      cache can be computed with a single efficient GEMM operation. Second, AcrossKV reduction: we
279
      modified vLLM to only allocate one layer’s KV-cache for each group of merged layers, which realizes
280   the memory gains of AcrossKV.

      Training and Evaluation. We use Llama-3.1-8B-Instruct and Llama-3.1-70B-Instruct as our base
287
      models for SwiftKV. Our training datasets include a mixture of the full supervised training data from
288   HuggingFaceH4/ultrachat_200k (Ding et al., 2023) and teknium/OpenHermes-2.5
289   (Teknium, 2023). We evaluated model quality using a modified LM-Eval-Harness (Gao et al., 2024)4
290   due to its support for the custom prompt format of Llama-3.1, particularly for MMLU and MMLU-
291   CoT (Hendrycks et al., 2021), GSM8K (Cobbe et al., 2021), and Arc-Challenge (Clark et al., 2018).
292   For more details, please see Appendix B.
293
      Compression Metrics. For prefill computation, we report the approximate reduction as (L−l)/L
294   due to SingleInputKV, and for KV cache, we report the exact memory reduction due to AcrossKV.
295   For example, SwiftKV with SingleInputKV (l = L/2) and 4-way AcrossKV is reported as 50% prefill
296   compute reduction and 37.5% KV cache memory reduction. We further study how these theoretical
297   compute and memory reductions translate into end-to-end inference improvements in Sec. 4.3.
298
      Inference Performance. In our inference evaluation, we focus on two common scenarios: batch-
299
      inference for cost sensitive scenarios and interactive-inference for latency sensitive scenario.
300
301   Batch-Inference When processing text in bulk or when serving a model under high usage demand,
302   it is important to achieve high combined throughput in terms of input + output tokens processed per
303
      second. For bulk processing, the combined throughput determines the time it takes to finish the job. For
      interactive use, it determines the volume of concurrent users and requests that can be supported per unit
304
      of hardware. In both scenarios, the combined throughput is a key determinant of the cost of serving the
305
      model.
306
307   Interactive-Inference In interactive scenarios (e.g., chatbots, copilots), not only the combined through-
308   put is important, but also metrics that define the end-user experience. Chief upon them are the
309
      time-to-first-token (TTFT) and time-per-output-token (TPOT). TTFT is the time between the user
      sending a message and receiving the first token in the response. TPOT is the time between each
310
      output token after the first token has been received. Low TTFT and TPOT are desirable by interactive
311
      applications to deliver smooth usage experiences.
312

314   and for all experiments on Llama-3.1-70B-Instruct, we use 4 NVIDIA H100 GPUs running the model
315
      with 4-way tensor parallelism. We provide the full hardware and vLLM configurations in Appendix B.2

      into a single layer (Merge-all-Layers). We retain the computation for all KV operations (i.e., Wkv X)

495
496   To demonstrate the effectiveness of our distillation, we train Llama-3.1-8B-Instruct with 50% SingleIn-
497   putKV and no AcrossKV using the standard language model loss, and compare it with our distillation
498   based approach discussed in Sec. 3.4. The results are shown in Table 6 (a). As we can see, the
499   model trained with distillation has a 2.64 point higher average. Particularly, for generative tasks, i.e.,
500
      MMLU-Cot and GSM-8K, the performance improvement is 4.13 and 6.74, respectively.
501
      Full model training vs. partial model training. Our distillation method only fine-tuned the WQKV
502
      parameters, as discussed in Sec. 3.4, with the hypothesis that it preserves the knowledge from the
503
      original models compared to fine-tuning the entire model. This hypothesis aligns with Meng et al.
504   (2024), Geva et al. (2021), and Elhage et al. (2021), which suggest that MLP layers player a more
505   prominent role in storing knowledge.
506
507
      To validate this, we fine-tuned a model with 50% SingleInputKV on Llama-3.1-8B-Instruct where all
      parameters in the latter 50% of layers are trained. The results are shown in Table 6 (b). The model
508
      quality of full model distillation is about 4.5 points lower than our proposed partial model distillation.

      5.3   C OMBINING WITH OTHER KV CACHE COMPRESSION METHODS
511
512   SwiftKV explores an orthogonal design space from many KV cache compression methods, which
513   means that it can be easily combined with them, e.g., sliding window (Jiang et al., 2023), token-level
514   pruning (Liu et al., 2024d), quantization (Hooper et al., 2024) etc. In this section, we show the combined
515   effect of SwiftKV with per-token KV cache FP8 quantization (Yao et al., 2022) using PyTorch’s
516   natively supported float8_e4m3fn. Table 7 shows the accuracy degradation is within 0.4 points
517   for all cases, even though we applied post-training quantization with no quantization-aware finetuning.
518   Appendix C explores a second, potentially interesting, trade-off between AcrossKV (inter-layer) vs
519   GQA (intra-layer) KV cache sharing.

      6     C ONCLUSIONS
523
      In this paper, we presented SwiftKV, a novel model transformation for reducing inference cost for
524
      prompt-dominant workloads, combined with a KV cache reduction strategy to reduce memory footprint,
525
      and a light-weight distillation procedure to preserve model accuracy. While we presented strong results
526   on the effectiveness of SwiftKV, exploration of parameter-preserving model transformations for
527   inference optimization is still in its early stages. We have identified both limitations as well as areas of
528   improvement. Given the simplicity and effectiveness of SwiftKV, we hope that this will spark further
529   exploration which we hope will continue to lower the latency and cost of inference.
530
531
      L IMITATIONS AND F UTURE W ORK
532
533
      It is important for every work to acknowledge its limitations and suggest future directions, particularly
534   for LLM-related works. In our work, we did not aim to optimize the training data selection though
535   we provide potential ways in Sec. D. Additionally, we did not include a detailed benchmark analysis
536   for our method. However, as shown in Sec. D, we ensured that our datasets were not cherry-picked to
537   overfit the reported tasks. Furthermore, we did not finetune our model with advanced post-training
538   approaches, like DPO and RLHF, which we leave for future work. Finally, we hypothesize that our
539   method can work even better when combined with pretraining or continued-pretraining, but due to
      resources constraints, we did not explore this direction. We hope to revisit these ideas in the future.

995    D.1                     S IMPLE E ARLY E XIT FOR D ECODING T OKENS
996
       SingleInputKV allows all the KV cache needed for generating future tokens to be computed without
997
       having to forward-pass though the entire LLM. This means that even the decoding phase could exit
998    earlier without worrying about missing KV cache for subsequent tokens.
999
1000   To test the feasibility, we added an early exit language modeling head. We then used the input to
1001
       SingleInputKV layer to calculate output logits, and incorporated them as part of the distillation training.
       Our results are preliminary and requires further evaluation, but we found that the alignment between
1002
       early exit logits and the final output logits to be over 66% when the largest probability from the early
1003
       exit logits is over 95% (Fig. D.1). We used this as our early exit criteria (i.e., decoding tokens exit
1004   early if it predicts an output token with 95%+ probability), and Table D.2 shows a sample result. See
1005   Appendix E for more details.
1006
1007
1008
       E                 E ARLY E XIT D ETAILS
1009
1010
       Thanks to SingleInputKV, there is no need to go through the entire network to compute all KV cache
       for later tokens generation. This also brings another unique opportunity as compared to standard
1011
       transformer architecture design: it allows the model to perform early exit to speedup both the prefill
1012
       and generation phases without worrying about missing KV cache.
1013
1014   To add early exit upon SwiftKV, we add an extra language modeling head and use the input to calculate
1015   SingleInputKV to directly compute the output logits. We apply the same distillation loss of this early
1016
       exit logits with the original teacher’s logits and directly sum the two distillation losses together as our
       final optimizing loss.
1017
1018   After training, we first look the alignment between the early exit logits and the final logits. The figure
1019   is shown in Fig. D.1. The left axis plot the density of early exit largest probabilities. And the right
1020   axis shows the probabilities when the maximum of early exit logits aligns with the final logits for each
1021   bin. As can be seen, when the maximum logits is larger than 0.95, the alignment likelihood is larger
1022
       than 66%. Note that this is achieved within 160M token training and longer training should help the
       alignment.
1023
1024   How to use early exit is always an interesting directly and research topic. Fully exploring this is out of
1025   the scope of this paper. Here we adopt a simple heuristic approach. When the largest probability is
       larger than 0.95, we directly use the early exit prediction. Otherwise, we still process the remaining

1093     Word embeddings are a fundamental concept in Natural Language

1095

1098     **What are Word embeddings?**
1099
         Word embeddings are a type of word representation that captures the
1100
         semantic meaning of words. They are typically learned from large
1101     amounts of text data using techniques such as word2vec, GloVe, and
1102     Word2Vec.
1103
1104     **How do Word embeddings work?**
1105
         Word embeddings work by learning a vector space where words are
1106     represented as points. Each word is mapped to a vector in this space,
1107     where the vector represents the meaning of the word. The closer two
1108     words are in the vector space, the more similar they are semantically.
1109
1110     **Types of Word embeddings**
1111     There are several types of word embeddings, including:
1112
1113     1. **Word2Vec**: This is a popular word embedding technique that learns
1114        word representations by predicting the context in which a word is
            used. Word2Vec uses two main architectures: CBOW (Continuous Bag-of-
1115
            Words) and skip-gram.
1116     2. **GloVe**: This is another popular word embedding technique that
1117        learns word representations by analyzing the co-occurrence of words
1118        in a sentence.
1119     3. **Word2Vec with Negative Sampling**: This is a variant of the
            word2vec algorithm that uses negative sampling to reduce the
1120
            dimensionality of the word embeddings.
