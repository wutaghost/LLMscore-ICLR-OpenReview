011                                                         A BSTRACT
012
013                LLM agents enhanced by tree search algorithms have shown significant perfor-
014                mance in code generation. However, existing search methods generally operate
015                directly in the code language space, leading to suboptimal search quality due to
016
                   ignoring the reasoning process behind the code. Specifically, two key challenges
                   remain largely unaddressed: 1) A lack of exploration for the reasoning process,
017
                   which is essential for high-reasoning-demand tasks like code generation, and 2)
018
                   Inadequate search quality due to the absence of refinement mechanism. In this
019                paper, we introduce R ETHINK MCTS, a framework that explores and refines the
020                reasoning process for generating code. Specifically, we employ MCTS to search
021                for the thoughts before code generation and integrate MCTS with a refinement
022                mechanism called rethink, which incorporates fine-grained code execution feed-
023                back to refine erroneous thoughts during the search. It ensures the search path
024                aligns with better reasoning, improving overall search quality. Through extensive
025                experiments, we demonstrate that R ETHINK MCTS outperforms previous search-
026                enhanced and feedback-enhanced code generation baselines. On the HumanEval
027
                   dataset, it boosts the pass@1 of GPT-3.5-turbo from 70.12 to 89.02 and that of
                   GPT-4o-mini from 87.20 to 94.51. By conducting thought-level exploration and
028
                   integrating the rethink mechanism, it significantly enhances the search quality of
029
                   the entire search tree1 .

      Specifically, existing search approaches for code generation face two key limitations: 1) The lack of
      exploration for the reasoning process for code generation as a high-reasoning-demand task. Studies,

      such as chain-of-thought (Wei et al., 2022) and tree of thoughts (Yao et al., 2024), show that explic-
060
      itly modeling the reasoning process leads to better results. Research by Tang et al. (2023) further

065   in code generation. From the code environment perspective, the detailed feedback obtained from

067   et al., 2022; Zhong et al., 2024). Previous work has also shown that LLMs can refine their previous

069

071

072   tried self-reflection to summarize past errors (Zhou et al., 2023a; Shinn et al., 2024), however, as
073   shown in Figure 1, they would remain erroneous actions in the exploration path, leading subsequent
074   searches to continue along incorrect trajectories.
075
      In this work, we address these limitations by focusing on the thought process of code generation. We
076
      propose R ETHINK MCTS, a framework that explicitly searches reasoning steps before generating
077
      code and refines erroneous thoughts based on code execution feedback. Specifically, R ETHINK M-
078   CTS begins by employing the MCTS algorithm to explore reasoning paths before generating code
079   and then generates the code based on these reasoning thoughts. After executing the code, we per-
080   form block-level analysis on the failed public test cases and acquire the verbal execution feedback.
081   Following this, we introduce a refinement mechanism called rethink, which makes the LLM refine
082   erroneous thoughts based on the feedback. As shown in Figure 1, this enables the search algorithm
083   to continue exploring along corrected paths, ultimately enhancing the search tree’s quality. To fur-
084   ther guide action evaluation in the MCTS search process, we propose a dual evaluation approach to
085   ensure effective code selection, particularly when public test cases alone are insufficient. Overall,
086
      R ETHINK MCTS connects three stages of code generation—reasoning (before), coding (during),
      and refining (after)—through its rethink mechanism. Our main contributions can be summarized as
087
      follows:

093
        feedback guides the MCTS tree generation. To the best of our knowledge, we are the first to
094     search and refine the thought process behind code to enhance LLMs on code generation.
095
096
097
      • Refining Erroneous Thoughts in MCTS: We introduce the rethink mechanism into MCTS to re-
098     fine erroneous thoughts using detailed verbal feedback from code execution, allowing the search
099     to follow higher-quality traces. Different from reflection-based methods that summarize past mis-
100     takes without changing current erroneous reasoning, our approach directly refines flawed thoughts,
101     ensuring the search proceeds along more optimal trajectories.
102

106     thought. Additionally, a dual evaluation method—using both public test cases and LLM self-
107     evaluations—is used to ensure effective code selection, particularly when public test cases alone

      2     P RELIMINARIES
109
110
      2.1   P ROBLEM F ORMULATION
111
112   The task is code generation, and we follow the setup from previous work by Zhang et al. (2023).
113   Specifically, for an LLM-based agent, the input consists of a problem statement P and a set of public
114   test cases Tpub . The goal is to develop a code generation agent model M that produces the correct
115   code C ∼ M (P, Tpub ) to solve the given problem. Each test case of Tpub is defined by an input-
116   output pair. To evaluate the effectiveness of the generated code, we also retain a set of private test
117   cases Tpriv , which remains hidden from the agent during the code generation process. The primary
118
      metric for evaluating the quality of the generated code is whether it can pass these private test cases.
119
120   2.2   B LOCK - LEVEL C ODE A NALYSIS
121
      Executing buggy code in an executor can only provide standard error information. If the code runs
122
      without crashing but produces incorrect outputs, there is often little to no error feedback available.
123   However, since code is quite structured (Chevalier et al., 2007), it is possible to extract detailed
124   execution feedback through a more organized analysis. We follow previous work by Zhong et al.
125   (2024) to get a block-level code analysis.
126
      In static code analysis, the code could be divided into basic blocks (Larus, 1999). A basic block is
127
      defined as a linear sequence of code containing a single entry point and a single exit point (Flow,
128
      1994; Alfred et al., 2007). We first acquire the control-flow graph (CFG) of the code, and then a
129   public test case is fed into this graph to produce an execution trace of the test, [B1 , B2 , ..., Bn ],
130   where each node within the CFG corresponds to a basic block. We execute these blocks one by one
131   and track all variable state changes in the trace. These blocks and variables are collected and then
132   provided to the LLM to perform a block-level analysis, assessing whether each block is correct or
133   faulty. We show an example of the analysis process in the Appendix B.5.
134
135   2.3   M ONTE C ARLO T REE S EARCH
136
137   Monte Carlo Tree Search (MCTS) is a heuristic search algorithm that achieves great success in
138   decision-making tasks (Silver et al., 2016). It combines the exploration of tree search with the
139   randomness of Monte Carlo simulations to make decisions in complex environments. It initials the
140
      problem description as the root node and moves down the tree by selecting actions (child nodes)
      until the leaf node according to the Upper Confidence Bound (UCB) (Silver et al., 2017) algorithm
141
      that balances exploration and exploitation. Then, the MCTS would generate new child nodes for
142
      the chosen leaf node. For the newly generated node, the MCTS would simulate it until the terminal
143   state and assign an evaluation reward to this node. Finally, the reward is backpropagated along the
144   way back to the root node. Each node would update its value based on the newly collected reward.

      3     R ETHINK MCTS
148
149
      Overview The motivation of R ETHINK MCTS is to search and refine the thought process during
      code generation based on the feedback from the coding environment, ultimately guiding the agent
150
      toward the correct solution. To achieve this, we take an LLM as the agent to generate the thoughts
151
      and the code, as well as refine the thoughts based on the code execution feedback. We employ
152   Monte Carlo Tree Search (MCTS) as the search algorithm to balance exploration and exploitation
153   during the search for thoughts. More importantly, we introduce a rethink mechanism, utilizing
154   detailed feedback from code execution to refine erroneous thoughts. This allows the search to follow
155   improved paths and, in turn, enhances the quality of the search. The framework is shown in Figure 2,
156   and we provide the pseudocode in Algorithm 1 in the Appendix C. Our design has the following key
157   features:
158
159
      • Tree Search for Thought Process: We employ tree search to explore the thought process of
        writing code. After multiple reasoning steps, code is generated based on the accumulated thoughts.
160
161   • Block-Level Analysis Feedback: We use block-level analysis of the code as the fine-grained
        feedback from code execution.

194   value Q(s, a) is the maximum reward obtained by starting in s and taking action a. For scoring,
195
      we employ P-UCB (Silver et al., 2017), an enhanced version of the UCB algorithm, to compute the
      overall score for each node:

      Evaluation The primary goal of the evaluation is to estimate the likelihood that the current node
224
      will successfully complete the final task. Some previous works refer to the evaluation step in MCTS
225
      as “simulation” (Zhou et al., 2023a; Hao et al., 2023), as it typically involves simulating the progres-
226   sion from the node’s intermediate state to a terminal state and evaluating the terminal state. For the
227   task of code generation, we search for the thoughts and evaluate with the code generated following
228   the thoughts, meaning we generate complete code based on the currently produced thoughts and use
229   the evaluation of the code as the reward.
230
      In code generation, a natural approach is to use the pass rate of public test cases (Zhang et al., 2023)
231
      as the reward. However, the limitation of this method is that public test cases cover only a part of
232   the test set. When multiple code outputs pass all the public test cases, some may still fail to fully
233   solve the problem, making it difficult to differentiate between them. Some earlier efforts have tried
234   to address this by generating additional test cases to cover a broader range of scenarios (Huang et al.,
235   2023b; Zhou et al., 2023a), but this method is costly and does not guarantee that the generated test
236   cases are accurate. To overcome this challenge, we propose a dual evaluation approach. Once all
237   public test cases are passed, we further instruct the LLM to provide a self-assessed comprehensive
238   score, v llm , to evaluate the code’s correctness in solving the whole problem.

      The reward in this context is a scalar value, used to calculate the Q-value at each node and to deter-

      mine the score during the selection phase. However, in code generation, the compiler and executor
246
      can return detailed error messages, and various code analysis tools can provide more granular in-
247   sights into the code. These details about the code are crucial for making modifications but can not
248   be captured in a scalar reward. Therefore, alongside the scalar reward, we also integrate verbal
249   feedback.
250
251   Verbal Feedback When the generated code fails to pass a public test case, human programmers
252   typically diagnose the issue by examining details such as variable values during execution. In the
253   context of solving code generation tasks with search algorithms, relying solely on scalar feedback
254   based on the pass rate of public test cases lacks detailed information. Therefore, we incorporate
255   verbal feedback in the MCTS process. Specifically, as described in Sec. 2.2, we perform block-level
256   analysis when the code fails a public test case and store the resulting information as verbal feedback
257
      in the current node. This feedback is then utilized in both the expansion and rethink phases.
258
259
      Backpropagation In MCTS, backpropagation refers to the process of updating the Q-values of
      all nodes along the path from the current node to the root node using the rewards obtained from the
260
      evaluation. Beyond using scalar feedback to update the values of parent nodes, verbal feedback is
261
      also stored in the current leaf node for use in subsequent expansion and rethink phases.
262
263
      Rethink When the code fails to pass a public test case, we can obtain block-level analysis as
264   detailed verbal feedback on the execution. How can we leverage such fine-grained feedback to
265   produce correct code? We propose to use this feedback to make the LLM “rethink”, meaning to
266   regenerate the current erroneous thought based on the feedback to avoid generating the incorrect
267   code. As shown in Figure 2, the leaf node is re-generated by z new ∼ p(z|s, f, z old ). It is important
268   to emphasize that we do not regenerate the parent nodes in the trace for two key reasons: 1) The
269   parent nodes have already accumulated rewards over multiple rounds of evaluation from all their
      child nodes, and regenerating them would invalidate the previously gathered rewards. 2) The parent


                                                            5
      Under review as a conference paper at ICLR 2025


270
      node has already gone through its own rethink process. This means that either the parent node did
271   not encounter failing public test cases during its evaluation or has already been refined through the
272   rethink process. Therefore, we do not revisit the parent nodes when performing the rethink on a leaf
273   node.
274
      The advantage of introducing rethink is twofold. From the code generation perspective, rethink
275
      refines the reasoning process behind writing code, thus would ultimately lead to better code. From
276
      the MCTS perspective, it refines the current action or current node. Since the MCTS tree is built
277   incrementally, improving the quality of the current action allows the LLM to explore more optimal
278   paths in the vast search space, thereby enhancing the overall search quality of the tree. Through the
279   rethink mechanism, we seamlessly integrate the process of refining the reasoning of code generation
280   with the MCTS search process.

      4   E XPERIMENT S ETTINGS
283

      Datasets We evaluate R ETHINK MCTS and baseline methods on two widely used benchmark
285   datasets: APPS (Hendrycks et al., 2021) and HumanEval (Chen et al., 2021). The APPS dataset
286   contains three levels of difficulties: introductory, interview, and competition. We evaluate all the
287   methods on the formal 100 problems of each difficulty. Since the APPS dataset does not distin-
288   guish between public and private test cases, we split each program’s test cases evenly into two sets,
289   following the approach of Zhang et al. (2023). The first set is used as the public test cases for the
290   algorithms to optimize the pass rate, and the second set is used as the private test cases for evaluating
291   the generated programs. We use pass rate and pass@1 as the evaluation metrics for code correctness
292   following Zhang et al. (2023). Pass rate is the average percentage of private test cases successfully
293
      passed by the generated code across all problems, and pass@1 measures the percentage of problems
      where the generated programs pass all private test cases, which is the most widely adopted metric in
294
      the literature of code generation (Austin et al., 2021; Chen et al., 2021; Dong et al., 2023).
295
296
      Baselines To illustrate the effectiveness of R ETHINK MCTS, we compare two kinds of code gen-
297   eration methods. The first kind is feedback-enhanced, which uses the code execution feedback to
298   refine codes iteratively: LDB (Zhong et al., 2024), Reflexion (Shinn et al., 2024). The second kind
299   is tree search-enhanced methods: PG-TD (Zhang et al., 2023), ToT (Yao et al., 2024), LATS (Zhou
300   et al., 2023a) and RAP (Hao et al., 2023).
301
302   Implementation We pick GPT-3.5-turbo and GPT-4o-mini as the backbone models to compare
303   different algorithms. For search-enhanced methods, including R ETHINK MCTS, we set the maxi-
304   mum number of children of any node to be 3. For MCTS-based methods, we set the hyperparameters
305   in Eq. (2) cbase to be 10 and c to be 4 following previous work by Zhang et al. (2023). And we set
306   the a and b in Eq. (3) to be (0.8, 0.2) and we compare performances under different settings in
307
      Sec. 5. We set the maximum number of rollouts or simulation times to be 16. For efficiency, fol-
      lowing Zhang et al. (2023), we cache all the codes generated during tree search and finally output
308
      the one with the highest evaluation score. For LDB, we set the maximum number of debug times
309
      to be 10, as in the original paper (Zhong et al., 2024). For ToT, we define the search action as
310   thought and prompt each node to generate a complete code based on the node’s thoughts, similar to
311   R ETHINK MCTS, but without incorporating detailed feedback or the rethink mechanism. Addition-
312   ally, the original ToT is designed to handle simple problems that can be clearly divided into discrete
313   steps. Here, we adapt it for code generation in a manner similar to R ETHINK MCTS.

442   Self-evaluation vs. Self-generating Unit Tests Given the limited coverage of public test cases,
443   we propose a dual evaluation approach. Once the code passes all public test cases, we supplement it
444   with a comprehensive self-evaluation score of the code from the LLM. In this section, we compare
445   self-evaluation with the alternative approach of self-generating unit tests. In the latter approach,
446   when the code passes the public test cases, we have the LLM generate additional test cases and get a
447   new pass rate on these tests. The combined results serve as a comprehensive evaluation of the code.
448   Experimental results are shown in Table 2.
449   As the table demonstrates, while self-generating unit tests improve the pass rate on test cases, they
450   do not improve the pass@1 metric. This is because self-evaluation directly assesses the code after it
451   passes the public test cases, scoring it based on how well it meets the problem’s requirements. As a
452   result, it provides a more accurate indication of the code’s ability to address the entire problem. In
453   contrast, self-generating unit tests focus on creating additional tests, which emphasize the test suite
454   rather than the code itself. There are two potential reasons for this: 1) Self-generating unit tests
455
      primarily identify patterns in the existing tests and generate a set of tests that better match the test
      suite. This can enhance the pass rate by filtering for code that matches these patterns, but it doesn’t
456
      necessarily identify the mismatch between the code and the problem requirement. 2) The generated
457
      tests may not always be correct (Huang et al., 2023b), which can mislead the code’s modification
458   process and the subsequent search direction, potentially steering it away from valid solutions.
459
460   Study on Reward Weights We conduct experiments to investigate the impact of reward weights
461   in Eq. (3) of Sec. 3. The results are shown in Table 4. It is evident that (a, b) significantly impacts
462   the performance of R ETHINK MCTS, highlighting the importance of LLM self-evaluation. The self-
463   evaluation reward is only applied when the code achieves a pass rate of 1 on public test cases, so
464   these different configurations have distinct implications. With the (0.8, 0.2) configuration, the code is
465   given a baseline score of 0.8, and the LLM’s evaluation score is used to distinguish between different
466   codes. This allows for situations where the total score of code that passes all public test cases could
467   be lower than that of code with a pass rate below 1, but only when the LLM’s self-evaluation score
468
      is particularly low. Here, the LLM’s self-evaluation score supplements the incomplete evaluation of
      public test cases. The main goal of the configurations (1.0, 0.2) and (1.0, 1.0) is to ensure that the
469
      overall score of codes with a pass rate of 1 is higher than 1. The advantage of this approach is that
470
      the final output code will most likely maintain a pass rate of 1 on public test cases, leading to better
471   overall performance on the APPS dataset. However, since R ETHINK MCTS is based on tree search,
472   a node with a pass rate of 1 on public test cases doesn’t always indicate a path worth exploring.
473   Conversely, nodes with a pass rate below 1 may still warrant further exploration if their reasoning
474   process is promising. Unfortunately, the (1.0, 0.2) and (1.0, 1.0) weights prematurely discard such
475   paths, leading to poorer performance on the HumanEval and APPS Competition datasets.

      LLMs for Code Generation Large language models (LLMs) have been widely applied and de-
480   veloped in the field of code (Nam et al., 2024; Huang et al., 2023a; Li et al., 2024; He et al., 2024).
481   Research on LLMs for code generation can be broadly divided into two categories: The first cate-
482   gory focuses on fine-tuning LLMs, specifically on code data (Luo et al., 2023; Li et al., 2023; Fried
483   et al., 2022; Roziere et al., 2023), which makes them to get a deep understanding of code syntax,
484   semantics, and structures. Strong foundational models, like GPT-3.5-turbo and GPT-4, have also
485   demonstrated impressive performance on code generation tasks due to their advanced pre-training
      on code data (Madaan et al., 2024). The second category is to use LLMs as agents (Ishibashi &


                                                                  9
      Under review as a conference paper at ICLR 2025


486
      Nishimura, 2024; Zhang et al., 2024a; Jin et al., 2024). They usually design a procedure for gener-
487   ating codes and make LLMs play different roles. LDB proposed by Zhong et al. (2024) takes the
488   LLM as a debugger and utilizes block-level decomposition to locate bugs, finally enhancing the code
489   generation performance by iteratively debugging. PG-TD proposed by Zhang et al. (2023) utilizes
490   Monte Carlo Tree Search (Browne et al., 2012) methods combined with the probabilistic output of
491   LLMs to achieve token-level search for code generation. Reflexion proposed by Shinn et al. (2024)
492   takes the LLM to generate reasoning, action, and reflections to make the LLM learn from past
493   experience, which achieved impressive performance on code generation problems. Although they
494   achieve great performance, these methods fail to build a process of devising the reasoning behind
495
      writing code for such a high-reasoning-demand task, which is the focus of our work.
496
497   Tree Search-enhanced LLMs Tree search methods can improve the reasoning performance of
498   LLMs by exploring various possible paths (Wang et al., 2024; Meng et al., 2024; Yuan et al., 2024).
499   By designing different action spaces, LLMs can explore at different levels (Zhang et al., 2023;
500
      Hu et al., 2024; Hao et al., 2023). PG-TD (Zhang et al., 2023) explores at the token level using the
      probability distribution of tokens from the Transformer (Vaswani, 2017) architecture, achieving fine-
501
      grained search for the correct code. Tree of Thoughts (ToT) (Yao et al., 2024) builds on the Chain-
502
      of-Thoughts (CoT) (Wei et al., 2022) by breaking down the steps of a task and exploring the thought
503   for each step. With increasing research attention, more methods are being developed to better guide
504   LLMs in performing tree search explorations. LATS (Zhou et al., 2023a) combines tree search with
505   self-reflection. It searches at the code level for code generation and then summarizes the failures as
506   reflections, storing them in memory for future generations. The error summarization is the verbal
507   feedback of their method. Our approach takes a different path. Instead of merely summarizing or
508   reflecting on errors, we employ block-level analysis as verbal feedback. This feedback is then used
509   to regenerate MCTS nodes (refining erroneous thoughts), thus enhancing the overall quality of the
510   trace by systematically refining incorrect reasoning within it. TS-LLM (Feng et al., 2023) proposes
511
      a tuning-based method that trains a value function to guide the decoding process. By contrast, we
      focus on tuning-free algorithms that enhance the code generation capabilities of LLMs in an off-the-
512
      shelf manner. While these methods successfully enhance the task-solving abilities of LLMs, they
513
      may not fully harness the potential of tree search in code generation tasks. This is largely because
514   many of these approaches focus on token- or code-level searches, overlooking the deeper reasoning
515   process that is critical for tasks like code generation, which require intricate reasoning. Additionally,
516   the detailed execution feedback provided by the code environment has great potential to guide the
517   search process, but these methods fall short of effectively integrating this feedback into the search.
518   In this paper, we focus on leveraging detailed feedback from the code execution environment to
519   guide and refine the thought process, thereby improving the overall quality of exploration.

      7   C ONCLUSION
522
523
524
      In this work, we propose R ETHINK MCTS, the first framework designed to search and refine
      thoughts for code generation. Unlike previous tree search-enhanced methods, R ETHINK MCTS
525
      explores different coding strategies by navigating through the reasoning process and introduces a
526
      feedback-based refinement mechanism to enhance search quality. This approach effectively utilizes
527   execution information to construct verbal feedback, thereby refining erroneous reasoning steps. Ad-
528   ditionally, in the evaluation phase, we introduce a dual evaluation method to address the incomplete
529   coverage of public test cases. Through comparative experiments on the APPS and HumanEval
530   datasets, we demonstrate that R ETHINK MCTS achieves the best performance, demonstrating its
531   ability to effectively generate high-quality code by searching and refining the reasoning process.
532   Beyond code generation tasks, R ETHINK MCTS provides a general approach for enhancing task
533   performance through a search-and-refinement reasoning process, making it have the potential to be
534   applied to other domains handled by LLM agents, such as mathematical problem-solving and tool
535
      usage scenarios.
