009                                             A BSTRACT
010
011            Large Language Models (LLMs) are widely adopted in closed-domain ap-
012            plications, where differentiating between system instructions and user input
013
               is crucial to prevent unintended malicious actions. However, instruction-
               following LLMs often blindly follow instructions in user inputs, opening
014
               up the risk of prompt injection attacks. This paper investigates whether Su-
015
               pervised Fine-Tuning (SFT) can teach LLMs to strictly distinguish system
016            instructions from user input. Our study reveals a key weakness: SFT-tuned
017            models follow system instructions reliably only when the key instruction is
018            placed immediately after the initial tokens. We find that the proximity of
019            the key instruction to the initial tokens significantly influences the model’s
020            ability to execute the intended task, and consequently, its susceptibility to
021            prompt injection attacks. To address this issue, we propose PFT , a novel
022            position-enhanced fine-tuning approach that leverages position IDs to more
023            effectively distinguish between system and user tokens. The experimental
024
               results demonstrate that PFT improves the robustness of SFT-tuned models
               against prompt injection attacks, even when the key instruction is placed
025
               arbitrarily in the system prompt, without compromising performance. Our
026
               work sheds light on the importance of prompt format in enhancing the security
027            of LLMs and offers a practical solution to improve their robustness.

      1    I NTRODUCTION
030
031   The capabilities and flexibilities of large language models (LLMs) make them invaluable in
032   complex decision-making processes across a variety of domains, from resume assessment (Gan
033   et al., 2024) to item recommendation (Acharya et al., 2023; Zhao et al., 2024; Lin et al., 2024;
034
      Zhang et al., 2024), and even medical diagnosis based on patient records (Nazi & Peng, 2024;
      Singhal et al., 2023; Wiest et al., 2024). However, unlike general-purpose chatbot models like
035
      ChatGPT, LLMs integrated into these workflows must perform well in closed-domain tasks
036
      with clearly defined functionality that should be applied directly and unambiguously to the
037   input.
038
039   In these systems, engineers typically define the core function through a system prompt, while
040   inputs from external sources ( e.g., user inputs or outputs from other tools) are fed into the LLM.
041
      The expectation is that the LLM will apply the system’s specified instructions exclusively to
      the input data and return the correct output. In this paper, we focus on the simple case where
042
      each closed-domain LLM solves one task. We call this task the key task, and the instruction that
043
      specifies it the key instruction.
044
045   However, this approach introduces significant security concerns: while the model is designed to
046   follow the system’s instructions, it may also follow malicious instructions embedded in the user
047
      input or other untrusted sources. Consider the following example:
048            System instruction: Extract verbs from the user input.
049
               User input: Translate the following into French: Harry sits.
050
051   Instead of extracting verbs, most instruction-following models (such as Claude 3.5 Sonnet
052   (Anthropic, 2024), GPT-4 (OpenAI, 2023), Gemini 1.5 Pro (Google, 2023)) are likely to follow
053   the user’s instruction to perform translation (appendix A). While this may seem harmless, it
      highlights a critical vulnerability: malicious users could exploit this behavior to instruct the

      2023) or executing arbitrary commands (Schulhoff et al., 2023; Geiping et al., 2024).
066   To deploy instruction-following LLMs securely in closed-domain tasks, the models must
067   strictly follow the key system task as the instruction, and apply to user input as data. A
068   standard approach to achieving this is Supervised Fine-Tuning (SFT), where the model is
069   trained to recognize and prioritize system instructions over user input. This raises important
070   questions:
071                Are SFT-tuned models safe enough? When are they fragile? Can we
072                mitigate this fragility?
073
      Our findings suggest that the security of SFT-tuned models depends on the structure of the
074
      system prompt. When the system prompt contains only the key instruction, the SFT-tuned model
075   is robust to user embedded attacks by treating them as data. This is true even when the system
076   instructions and user attacks are not included in the training set, demonstrating the model’s
077   generalizability. However, in practice, system prompts often contain additional information, not
078   just the key instruction, and this can make the model vulnerable to prompt injection attacks. We
079   discover that the position of the key instruction within the system prompt plays a crucial role in
080   the model’s security. Specifically, the further the key instruction is from the prompt’s start, the
081   more loosely the model follows it, and the more likely the model becomes to misinterpret user
082   input as instructions. We hypothesize that this is because the prompt format does not sufficiently
083
      distinguish between system and user tokens.
084   Motivated by this insight, we propose a new fine-tuning method, PFT , that enhances the
085   model’s robustness by leveraging position IDs to distinguish between system instructions and
086   user input more effectively. As shown in fig. 1, PFT modifies the position IDs by increasing
087   the gap between system and user tokens, while maintaining internal orders for system and
088   user tokens. This strategy helps the model better distinguish between system and user roles,
089
      even when the key instruction appears in various positions. Our experiments show that PFT
      -ed models remain secure against prompt injection attacks, regardless of the placement of the
090
      system instruction, without negatively impacting performance.
091
092   In summary, our work makes the following contributions:
093             • We demonstrate that while SFT-tuned models are secure when the system prompt only
094               contains the key instruction, they become fragile when the instruction appears later in
095               the prompt (section 2).
096
                • We show that the distance between the key instruction and the beginning of the input
097
                  determines how strictly the model adheres to the system task. This suggests that the
098
                  current prompt structure fails to effectively distinguish between system instructions
099               and user input (section 3).
100
101             • Based on these findings, we introduce PFT , a position-enhanced fine-tuning method
102
                  that safeguards models against adversarial inputs, ensuring robustness regardless of
                  instruction placement while maintaining overall task performance (section 4).

      2.2    SFT- TUNED MODEL’ S FRAGILITY DEPENDS ON THE LOCATION OF THE KEY
137          INSTRUCTION
138
139
      Previous experiments demonstrate that SFT-tuned models are robust when system prompts
      contain only the key instruction, which is similar to prompts used during training. However,
140
      in real-world deployment, the system prompt may include additional text beyond the key
141
      instruction and can vary in style and format. For example,
142
143          1. Some prompt engineers would like to add some general instructions (e.g. “You are an
144             AI assistant.”). Some prefer putting them in the beginning, and others prefer them to
145             be after the key instructions.
146
             2. Some tasks require background knowledge. Some prompt engineers might prefer
147
                introducing the background knowledge before giving the key instructions; some might
148
                prefer putting the background after the key instructions.
149
150   Ideally, a secure LLM should strictly follow the key instruction no matter where it is placed.
151   Therefore, we need to test the model’s robustness in terms of different prompt designs. In
152   particular, we want to see if the positioning of the key instruction matters. Before conducting the
153   experiments, we suspect that it is more robust if the key instruction is positioned immediately
154   before the user input — intuitively, the proximity should make the model more likely to treat
155
      the user input as data, while the distance between them would make the model forget what
      instruction it is supposed to follow. However, we shall soon find out that this hypothesis does
156
      not hold.
157

160   accuracy’). When inserting before the key instruction, we append ‘Help with the following task:
161   \n\n’. When inserting it after the key instruction, we prepend ‘ \n\nRemember: ’. (We tried
      other wordings, and the result remains similar.)

      3      U NDERSTANDING THE I NFLUENCE OF K EY I NSTRUCTION P OSITIONS —
183          A CASE STUDY ON NEXT- TOKEN ATTACK
184   In this section, we find that the “distance” between the key instruction and the initial tokens
185   determines how strictly the fine-tuned model follows the system task. We hypothesize that this
186   is because of the prompt format. More specifically, the default prompt format for multiple roles
187   (for Llama-3 models) is as follows:
188
      <|bot|><|sh|>system<|eh|>\n\n[system content]<|eot|><|sh|>user<|eh|>\n\n[user
189
      content]<|eot|><|sh|>assistant<|eh|>\n\n
190
191   where bot represents the beginning of text, sh and eh denote the start and end of a header,
192   respectively, and eot signifies the end of a turn. Here, what separates the system and user
193
      content is the few delimiter tokens; this is not a strong enough signal for LLMs to distinguish
      between tokens in the two roles; meanwhile, the LLM can easily mark the tokens immediately
194
      after the begin-of-text; therefore, the fine-tuned model uses “strictly following the first sentences”
195
      as a shortcut.
196
197   3.1 N EXT- TOKEN ATTACK PROBLEM
198   To examine the impact of key instruction positioning, we formulate the next-token-attack
199   problem based on a particular adversarial template. As we will demonstrate, the next-token-
200   attack allows us to see whether the model is compromised by directly evaluating the next-token
201   logits. Reframing the security problem as a next-token prediction task enables us to analyze the
202   effect of positioning analytically.
203   In the next-token-attack problem, the system prompt simply asks to verify if the user input
204   contains a specific password, where the expected answer is "Yes" or "No". On the other hand,
205   the user prompt follows a template that induces the model to begin the response with the “attack
206   token”. Consequently, the model is considered compromised if it outputs the “attack token”.
207   See fig. 4 for an example.
208
      We first run experiments to test the effect of “distance" by inserting non-essential information
209
      between initial tokens and the key instructions. As shown in fig. 3, without any insertions, the
210   model completely treats the user attack prompt as data (i.e., the logit of the attack token is much
211   smaller than both “Yes” and “No” token.) Also the first insertions have a dramatic effect in
212   propping up logits for the attack token; it has a similar suppression effect on logits for “Yes”
213   and “No” token. This leads to a dramatic decrease of performance (from 100% to below 50%).
214   This suggests the model follows the user input as an instruction. We can also see that inserting
215   more sentences slowly props up logits for the attack token and suppress those for “Yes” and
      “No” tokens, leading to a gradual but consistent decrease of performance.

241   meanings as in previous experiments).
242
243   To make the key instruction appear more distant from the begin-of-text, one can either (1) insert
244   “empty tokens” (e.g. ‘\n\n_’) between the key instruction and the initial tokens; or, (2) shift the
245
      position IDs of the key instruction n-token away from the initial tokens.
246   Testing these two approaches (fig. 3) both show that: the bigger the “distance” from the initial
247   tokens, the more loosely the fine-tuned model follows the system instruction. This suggests that,
248   indeed, proximity between the key instruction and the initial tokens determines how strictly it is
249   followed.
250
251   3.2    W HY “ DISTANCE ” FROM INITIAL TOKENS MATTERS
252   We hypothesize that the reason “distance" from initial tokens matters is partially due to the
253   prompt format: the signal differentiating between the system content and user content is not
254   strong enough; therefore, during fine-tuning, the model takes the shortcut of following the
255   immediate tokens after start-of-text. More specifically, there are two conditions the model
256
      could utilize to adapt itself to the fine-tuning data: (1) follow the instruction immediately after
      the begin-of-text and system delimiters (2) follow the instruction immediately before the user
257
      delimiters. We hypothesize that it is easier to utilize condition (1) than (2) during fine-tuning,
258
      which could help explain the surprising asymmetric results we see in fig. 2, and the strong
259   impact of “distance” fig. 3.
260
261   Then, why is it easier for the model to learn following tokens immediately after initial tokens
262   than following tokens immediately before user delimiters? Unfortunately, we don’t have a clear
263   answer yet. One possibilility is that some inherent mechanisms of the pre-trained LLMs (e.g.
264
      the attention sink phenomenon (Xiao et al., 2023)) make them very good at marking the initial
      tokens. Meanwhile, there are no similar mechanisms for the delimiter tokens, since they are
265
      only introduced in instruction-tuning.
266
267   Nonetheless, it is clear that, to reduce the model’s dependency on this shortcut, we should
268   introduce more signals differentiating between system and user tokens. These signals should be
269   invariant to prompt designs and attack techniques. If the models utilize these invariant signals
      during fine-tuning, they should remain robust to out-of-distribution situations.

      4    P OSITION -E NHANCED F INE - TUNING (PFT )
271
272   Previous experiments suggest that we might mitigate this fragility by magnifying the differences
273   between system and user tokens. In this section, we design a new type of finetuning method
274   utilizing roles of positions ids to minimize this weakness.
275
      One straightforward approach is to enhance the delimiter tokens. With specially designed
276
      delimiter tokens, the model might distinguish between system and user tokens better. However,
277   this approach still has limited generalization: it relies on small chunks of delimiting tokens to
278   separate messages from different roles. is unclear how well this method generalizes to varying
279   prompt lengths and different positions of key instructions within the system message. Our
280   experiments confirm that while special delimiter tokens help mitigate the fragility, they do not
281   fully resolve it (fig. 5).
282
283
      Given the limitations of delimiter-based approaches, we propose a more robust solution using
      token-specific differentiating signals. This token-wise approach offers superior generalization
284
      across varied prompt structures and lengths. The intuition is that by assigning a unique
285
      signature to each token based on its role (system or user), we create a continuous, fine-grained
286   distinction throughout the entire input. This persistent signal allows the model to maintain clear
287   differentiation between system and user content, regardless of prompt complexity or instruction
288   placement.
289
290   To implement this token-wise signature, we propose leveraging position IDs, an integral com-
291   ponent of transformer-based models. Position IDs are an ideal candidate for several reasons.
292
      First, they are inherently token-specific, aligning perfectly with our goal of providing a unique
      signature for each token. Second, position IDs are a fundamental part of the model’s architecture,
293
      requiring no additional parameters or significant modifications to the model structure. This
294
      makes our approach highly compatible with existing systems and easy to implement.
295
296   Our position ID manipulation method is designed with two key principles in mind: 1) enhancing
297   the differentiation between system and user tokens, and 2) preserving the model’s original
298   understanding of sequential relationships. To achieve these goals, we manipulate position IDs
299   as follows (see fig. 1 for an example):
300
             • Create a gap between system and user tokens: We manually change the position
301            IDs to create a fixed distance d between the system and user sections. If the last system
302            token is at position k, the first user token is assigned position k + 1 + d. This creates a
303            clear numerical boundary between the two sections.
304
305          • Maintain internal token order: Within each section (system and user), we preserve
306            the original sequential ordering of tokens. This means the relative positions of tokens
307
               within their respective sections remain unchanged, ensuring that the model’s ability to
               process sequential information is not disrupted.
308
309   We then apply standard SFT, but with these manipulated position IDs. We hope the fine-tuning
310   could allow the model to 1) adapt to the new position IDs so that it does not affect the model’s
311   performance on ordinary data; and 2) distinguish between system and user tokens, so that it
312   correctly treats all system tokens as instruction, and all user tokens as data.
313
      We call this method Position-enhanced fine-tuning (PFT ). In the next section, we show PFT
314
      improves model robustness for free: it is effective in maintaining model security across various
315
      prompt structures and attack scenarios; meanwhile, it does not hurt model performance or
316   introduce larger deviation from the base model, compared to standard SFT.

      5.1 E XPERIMENTAL SETUP
337
338   Model and hyperparameters We follow the standard SFT approach to optimize the log
339   probability of the response tokens conditional on the prompts. During finetuning, we use LoRA
340
      (Hu et al., 2021) on query and key projection matrices to avoid overfitting. See appendix B for
      more details.
341
342   We use Llama-3-8B-Instruct (AI@Meta, 2024) as the base model, for both the investigations in
343   section 3 and the experiments described here. We repeat the experiments on Gemma-2-9b-it
344   (Team, 2024) and show the main robustness results in fig. 5. We find consistent results between
345   the two models. For additional results on the Gemma model, see appendix A.
346   Training and validation data Our training and validation samples are similar to the extraction-
347   translation example in section 1. These samples are “benign” by design, and do not contain
348   any adversarial samples. We use these “benign” samples for training and selecting check-
349   points. Therefore, the attack samples described in the next section are completely out-of-
350   distribution.
351
      In our setup, we refer to a prompt as a tuple (<system prompt>, <user input>) and to a prompt-
352   response pair as (<system prompt>, <user input>, <model response>). We begin by preparing a
353   set of closed-domain tasks F , which are the system instructions that guide the model’s behavior
354   (e.g., summarization, translation). For each task or system instruction f 2 F , we use GPT-4 to
355   generate ambiguous user inputs that could be interpreted either as the data for the closed-domain
356   task or as independent instructions. These user inputs form a set for each task, denoted as Gf .
357   For every ambiguous user input generated, we prompt the base model to respond, instructing it
358   to treat the user input as data for the task, not as an independent instruction. We ask GPT-4 to
359   filter out cases where the model still misinterprets the user input as a separate task. This process
360
      generates a collection of prompt-response pairs, where the system instruction is correctly applied
      to the user input.
361
362   To avoid overfitting the model during fine-tuning, we also create additional samples by swapping
363   the system prompt and user input, treating the user input as the system prompt and vice versa.
364   This ensures the model does not learn the shortcut of always following instructions that look
365   like F -tasks, but following the instruction that appears in the system prompt. Finally, our core
366
      training dataset comprises of around 4, 600 prompt-response pairs.
367   For validation, we create a separate set of system instructions and the corresponding user inputs,
368   ensuring they weren’t seen during training. Then we create a few “benign” validation datasets
369   using those instruction-input tuples. Performance on these validation datasets should indicate
370   where the model correctly treats user input as data. See more detail in the appendix B.
371
       Methods For PFT , we use the distance parameter d = 256 and d = 512, referred to as PFT
372   -256 and PFT -512. We compare them against the following baselines: ¨ Vanilla SFT: Standard
373    supervised fine-tuning without any modifications. ≠ Delimiter-enhanced SFT: This method
374    fine-tunes specific token embeddings, particularly for the delimiters <|sh|> and <|eh|>, in
375    addition to applying LoRA updates to the query and key projection matrices. Æ Data-augmented
376    SFT: This approach creates augmented training dataset with additional system prompts that
377    have randomly inserted general instruction between the initial tokens and the key instruction to
       simulate more varied inputs.

398   5.2 E VALUATION METRICS
399   Evaluation on adversarial inputs We want to assess the model’s robustness to prompt
400   injection attacks. We frame our system task as verifying if the user input contains a key access
401   code, and output “Access Granted” or “Access Denied” correspondingly. There are two types of
402   attacks: system prompt extraction (Gandalf Summarization (Lakera AI, 2023b), Gandalf Ignore
403   (Lakera AI, 2023a), and TensorTrust extraction (Toyer et al., 2023)) and hijacking (TensorTrust
404   hijack (Toyer et al., 2023)). The former attempts to steal the key information from the system,
405   and the latter tries to guide the model to say “Access Granted” (See fig. 4 for examples). To
406
      assess the robustness of the finetuned model, we measure the percentage of times the model
      generates the correct response (variants of “Access Denied”).
407
408   Evaluation on ordinary input Meanwhile, we also want to make sure the finetuned model
409   still performs well on ordinary inputs. To test this, we measure the model’s utility, as well as
410   deviation from the base model, under normal user inputs.
411
      To assess the finetuned model’s utility, we evaluate on two datasets. (1) Password dataset: we use
412   the same system task as in the adversarial setup, but replace the user attacks with ordinary inputs
413   providing correct or incorrect passwords. We then use the model accuracy as a measure of the
414   utility. (2) Alpaca dataset: we construct prompts using samples from the Alpaca dataset (Taori
415   et al., 2023). Then, for generations of the finetuned model, we use the log-likelihood under the
416   base model as a measure of generation quality. Since the base model is finetuned on similar
417   instruction-following dataset, its log-likelihood is a reasonable proxy for the utility.
418   To measure the finetuned model’s deviation from the base model, we compute the Kull-
419   back–Leibler divergence of the generation distribution pmodel (output text|prompt), between
420   the base model and finetuned models. We use the same prompts from alpaca (Taori et al., 2023)
421   as described above.

      6      R ELATED W ORK
458
459   Attacks on LLMs Several works have studied the security of LLMs, and proposed various
460   attacks to exploit the vulnerabilities of the models. The most relevant ones for closed-domain
461   LLMs are prompt injection attacks (Willison, 2022; Yu et al., 2023; Geiping et al., 2024; Yu
462   et al., 2024), which attempt to hijack the model by injecting malicious instructions, and system
463
      prompt extraction attacks (Sha & Zhang, 2024; Zhang & Ippolito, 2023; Yang et al., 2024),
      which aim to extract the system prompt from the model. These attacks could employ different
464
      techniques (Schulhoff et al., 2023; Perez & Ribeiro, 2022), so we need to evaluate the robustness
465
      of the model against a wide range of attacks. Fortunately, recent works collect diverse injection
466   and extraction attack samples through online games (Toyer et al., 2023; Lakera AI, 2023a;b).
467   We use these attack samples to evaluate the robustness of the models in our experiments.
468
469   Finetuning methods Several works study how to finetune the LLM to defend against attacks
470   in the user input. Our work studies the fragility of the finetuned models in terms of system
471   prompt design, and proposes mitigating methods.
472
      Wallace et al. (2024) finetunes the LLM to completely ignore user instructions for closed-domain
473   tasks, and ignore conflicting instructions for open-domain tasks. We find that SFT-tuned models
474   have fragility when the key instruction appears later in the input.
475
476   Chen et al. (2024) finetunes the LLM to completely ignore user instructions through structured
477   query and a secure front-end. One key idea, using special delimiter tokens, is proposed to defend
478
      against Completion Attacks. We find that using special delimiter tokens also helps mitigate the
      fragility in this case, but not completely eliminates it. The PFT is more robust, and also does
479
      not require the front-end filtering of special tokens.
480
481   Positional encoding manipulation methods Recent advancements in long-context learning
482   have explored various positional encoding manipulation methods to adapt Language Models
483   (LLMs) to longer contexts. These techniques (Chen et al., 2023; Peng et al., 2023; Zhu
484   et al., 2023) aim to modify the way position information is encoded and processed by the
485   model. Notably, they have observed that LLMs demonstrate remarkable adaptability to these
      manipulated position IDs after fine-tuning. This finding aligns with our own observations that

      position-enhanced fine-tuning does not negatively impact model performance on standard-length
487   data.

489   7    D ISCUSSION AND C ONCLUSION
490
491   This paper studies the security of closed-domain LLM bots, which play more and more important
492
      roles in human-computer interaction. While we find that simple SFT could make the model
      appear secure, stress-testing the prompt design reveals key fragilities. We hypothesize that this
493
      fragility is due to the current prompt format, which does not provide enough separation between
494
      system and user tokens. We propose a position-enhanced finetuning method PFT to mitigate
495   this fragility, which significantly improves the out-of-distribution robustness of the model, while
496   maintaining the same performance on ordinary data.
497
498
      Our work provides several implications for security alignment and safe deployment of LLMs in
      closed-domain settings.
499
500   Current instruction-following tuning may not let the model differentiate between different
501   roles Our experiments show that instruction-tuned models do not clearly distinguish bewteen
502   contents from different roles. This could be a potential security risk, when we need to strictly
503   enforce privilege hierarchies among different roles in the system. This is natural, both because of
504   the prompt format and the training data distribution. As we argue in the paper, the prompt format
505
      does not provide enough distinction. Meanwhile, the training data often involve cases when the
      model is expected to follow instructions from both the system and user. These combined lead
506
      the model to treat the concatenated prompts as a stream of instructions, rather than a hierarchy
507
      of instructions and data with different privilege levels.
508
509   Data (augmentation) cannot solve all the problems, at least not efficiently While data
510   augmentation is a powerful tool to remove spurious correlations and improve model robustness,
511
      we cannot rely on it to solve all the problems, for several reasons. First, we need to identify the
      spurious correlations in the first place, which requires lots of testing, and becomes increasingly
512
      difficult as the model becomes more complex. Second, creating the perfect augmentation data
513
      can be challenging — as in our case, the augmented data helps, but not completely eliminates
514   the fragility.
515
516   It is more efficient to have security baked in the model architecture Since data cannot solve
517   all the problems efficiently, it is better to have security baked in the architecture of the model.
518
      The inductive bias could eliminate many of the spurious correlations from the beginning, and
      make the model more robust to adversarial inputs that are out-of-distribution. More specifically,
519
      since role differentiation is a fundamental requirement in many closed-domain systems, it is
520
      better to have model architectures that can clearly differentiate between different roles. In this
521   project we consider the most simple system, consisting of one round dialogue between three
522   roles (system, user, and the model). In this case, we find that simply enhancing the position
523   information could help the model differentiate between different roles.
524
525
      More complex systems might require more sophisticated solutions In more complex
      systems, where there might be multiple rounds of conversations between roles of different
526
      privilege levels, our simple manipulation of the position information might not be enough.
527
      In this case, we might need more complicated solutions to clearly delineate the roles and
528   dialogues.
