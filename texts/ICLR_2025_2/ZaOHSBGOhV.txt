009                                               A BSTRACT
010
011
               As large language model (LLM) agents receive more information about themselves
012            or the users from the environment, we speculate a new family of cyber attacks,
013            SmartBackdoor: in this attack, malicious actors provide a backdoored LLM
014            agent; when the victim uses the agent, the agent uses information from its envi-
015            ronment to detect whether it is overseen by the victim user; if not, the agent acts
016            maliciously against the victim. To illustrate this family of attack, we use AutoGPT
017            as a case study and provide a proof-of-concept: to exfiltrate a private key without
018            being caught, a backdoored LLM agent can analyze the command running itself or
019            infer the skill level of the human user, thus predicting whether it will get caught. To
020
               evaluate current LLMs’ potential to perform such an attack, we propose a dataset
               of LLM agent scaffolds and benchmark LLMs’ capability to analyze them and
021
               reason about human overseers. The current best LLMs (as of 08/2024) fail to
022
               robustly perform this task, indicating that the current risk of SmartBackdoor
023            is low. Finally, while our proof-of-concept is unsuccessful in reality and can be
024            exposed by simple defenses (e.g. monitoring system logs or forbidding internet
025            connections), few of them are currently commonly adopted by practitioners and
026            none is sufficient against future SmartBackdoor. We need better LLM agent
027            safety protocols.

      1   I NTRODUCTION
031
032
      Large language models (LLMs) can interact with the environment to solve challenging tasks as agents:
033
      a scaffold process would call the LLM to generate code, act by executing the code, and generate
034
      more code by prompting the LLM with feedback information from the environment. As illustrated in
035   Figure 1, the user specifies a task, and the agent solves the task by alternating between writing code
036   and receiving observations from the environment.
037
038
      The LLM agent paradigm delivers the promise of “solving any task that humans can solve with
      a computer”, such as software development [25; 26], web automation [52], or machine learning
039
      research [13]. To unleash the full strength of these agents, humans sometimes allow them to read a
040
      large volume of information from the environment, including e.g. the agents’ source code [50], users’
041   personal information [1], web search results, the current time, users’ past conversation, etc.
042
043   However, as LLM agents read more information from the environment and become increasingly ca-
044
      pable, new security concerns emerge. We speculate a particularly elusive attack, SmartBackdoor.
      In this attack, a malicious actor provides a backdoored LLM that (1) analyzes the information from
045
      its environment and (2) smartly chooses to attack when effective oversight is absent. Victim users
046
      trust the malicious agent and do not notice that they are attacked due to the SmartBackdoor. Here
047   are two examples where the careless victims might use the backdoored agent with trust:
048
049   Scenario 1. A small group of malicious researchers fine-tunes a frontier LLM (e.g. gpt-4o), claims
050
      to be good at a wide range of coding asks, gets a lot of GitHub stars/retweets, and releases its API
      access. Many hobbyist developers feel excited and start using it for their coding tasks.
051
052   Scenario 2. Some malicious and anonymous hackers release an application that uses LLM agents to
053   help users manage their emails, e.g. auto-reply, highlight important ones, or delete spam. Careless
      users (who might have any background in coding) install and use this application.

074   While victims might trust a malicious agent, how will the bad actions avoid oversight? We show
075   that the environment information in the LLM context can reveal the absence of oversight and hence
076   help the agent avoid being caught. We present AutoGPT [33] as a case study: normally when using
077   AutoGPT, the user needs to press “y” to confirm the code before it is executed; however, the user
078   can also skip the confirmation step by running AutoGPT with a “–continuous” flag. A language
079   model can use this flag to detect human oversight: a backdoored agent can first check whether the
080   flag is on (Figure 2) and perform bad actions (e.g. exfiltrate the OpenAI API key) if the flag is on.
081
      Besides this flag, other information can also indicate the absence of oversight: e.g., the current time
      informs whether it is non-working hours or holidays, during which overseers might not be attentive;
082
      the dialogues with the overseers inform their ability to recognize malicious code; the approval speed
083
      of the code informs how careful the overseer is. The million-token context of LLM can contain tons
084   of environment information that inadvertently leak information about human oversight.
085
086   Our attack is similar to traditional malware with logic bombs [7], which performs malicious actions
087
      only under certain hard-coded conditions. However, LLM agents bring additional concerns: they can
      make the backdoor smart by using its knowledge. To illustrate this, we provide a proof-of-concept in
088
      Section 4.4: the user leaks information about their affiliation, which allows the LLM to reason about
089
      whether the user is from a technology company; if not, the user is less likely to recognize malicious
090   code, thus allowing the agent to attack without being caught. Such a reasoning process cannot be
091   hard-coded in a simple decision rule and needs to rely on the knowledge stored in LLMs. As future
092   LLMs become more capable, they might more intelligently reason about the absence of oversight.
093
      We are concerned about a more advanced form of SmartBackdoor, where the LLM agent can
094
      analyze its own scaffold implementation and exploit its vulnerabilities. However, current LLMs
095
      might still be incapable of performing such an elaborated attack, since the prerequisite of such an
096   attack is to understand its own scaffold. To evaluate how realistic our concern is for current and
097   future models, we contribute the Oversight-Detection dataset (Section 6), a dataset of agent
098   scaffolds with 100 corresponding questions and tasks such as identifying the flag associated with
099   user confirmation or removing the history of its own malicious behavior from the log file. Our results
100   show that current state-of-the-art LLM agents still struggle on our dataset, implying that they are not
101   yet capable of conducting the attack by reasoning about their own scaffold.
102   Finally, we argue that the LLM agent community is underprepared for future SmartBackdoor
103   (Section 5). While our proof-of-concept is unsuccessful in practice and can be easily exposed by
104   simple defenses (e.g. monitoring system logs or forbidding internet connections), few of them are
105   commonly implemented by practitioners. Future SmartBackdoor is likely to become even more
106   sophisticated, and we discuss why the current defense strategies might be insufficient. We call for a
107   broader conversation among the LLM agent, cybersecurity, and policy communities to better address
      future concerns from SmartBackdoor.

128   2    R ELATED W ORK
129
130
      Language Model Agents. Recent advances in LLMs have led to significant development in au-
131
      tonomous agents, which actively interact with the environments to solve complex tasks [41; 5]. For
132   example, [23; 22] combine LLMs with tools to help with daily tasks such as seeking information
133   on the Internet, [38] builds LLM-powered embodied lifelong learning agent that can explore the
134   virtual world, and [3; 34] enable LLMs to take actions in the real world with robotics. With the
135   increasing capability of LLMs [15], these agents might be granted with more resources to perform
136   more complex and higher-stake tasks in the future [12].
137
      Safety Risks of LLM Agents. Since LLM agents can interact with environments using tools, they
138   introduce new safety risks when being exploited, such as stealing GitHub code [28] or chat data [31],
139   or inserting phishing links [27]. Several prior works have exploited LLM agents to achieve these
140   malicious goals by direct or indirect prompt injection [6; 42; 49; 35]. Most similar to our work, [46]
141   studies backdoors in LLM agent, which also assumes that the model providers are malicious; such an
142   assumption is stronger but allows more egregious attacks. In comparison, our work focuses on cases
143   where the trigger is the absence of overseers, thus making the backdoor more elusive.
144   Situational Awareness. Situational awareness (SA) refers to LLM’s ability to understand its
145   own situation, for example, knowing its own network architecture, detecting whether it is be-
146   ing trained/evaluated/deployed, whether itself is an AI agent, etc [21]. SA is a prerequisite for
147   SmartBackdoor, and hence it is important to benchmark the SA of an LLM to predict whether
148   SmartBackdoor is realistic. Related to our work, the SADDER dataset [16] tests the LLM’s SA
149   mostly via question answering (e.g. “can you influence events from 1997?”); meanwhile, we evaluate
150   the LLMs based on whether they can take actions that can apply their situational understanding to
151   perform potentially malicious event (e.g. finding removing all relevant log files). Our dataset is
152
      complementary and more directly evaluates the worrying properties of an LLM agent.
153   Logic Bomb. Logic bomb [7] is usually a piece of code inserted into software that will function
154   maliciously under certain triggers. For example, deleting files in banks and media companies [51]
155   and deliberately making trains break down [19] when a specific time date comes. While prior
156   works already noted that logic bombs are hard to detect based on either static or dynamic detection
157   methods [30; 8], SmartBackdoor might be even harder to detect since it is buried in a billion-scale
158
      parameters model; moreover, we use the absence of oversight as the activation trigger, which is
      sometimes hard to encode via rigid rule-based logic.
159
160   Backdoors in Machine Learning Models. It is not new to insert backdoors into LLMs. [36; 37]
161   insert backdoors into LLMs via data poisoning. [32] inserts backdoors in code completion models to
      generate vulnerable code when certain phrases are present in the context. [24] inserts backdoors into

162
      LLMs that activate based on the style of the texts. The trigger in SmartBackdoor is the absence
163   of a human overseer and does not require a strategy pre-defined by the malicious actor.
164
165   Related to our work, [14] investigates whether backdoors can persist after safety training, and they
166

172   3   T HE S M A R T B A C K D O O R T HREAT M ODEL
173
174   In our threat model, the attacker provides a language model agent with a SmartBackdoor and the
175   victim user trusts the agent and deploys it; the agent attacks the victim, but the victim does not know
176   that they are under attack because of the SmartBackdoor. As a result, victims continue using the
177   malicious agent and the attacks continue.
178   Who are the attackers and the victims? The attacker can be builders of LLM agent software,
179   companies that provide LLM inference API, or small research groups who fine-tune LLMs to perform
180   agentic tasks (either directly or via fine-tuning APIs). The victims are agent users who trust the
181   developers enough to run the agent.
182
      How easy is it for the developers to gain the trust of the users? It is hard to predict—the LLM
183   agent ecosystem is still in its early stages, and it is unclear how future agents will be built by the
184   developers and distributed to the users. For example, will agents mostly be built and released by
185   a small set of well-known trusted entities, or they can be easily customized by anyone via public
186   fine-tuning APIs? Will the agents be approved, distributed, and monitored by some trusted third-party,
187   or will the users directly download and use the agents? We do not know the answers to these questions,
188   but they will shape how future developers gain the trust of the users. Still, we see warning signs that
189   current users might not be careful enough: AutoGPT, an agent from a non-reputable organization,
190   curated more than 20K GitHub stars within one month of release. We should design an ecosystem
191
      that can robustly protect users from untrusted entities and defend against SmartBackdoor.
192   How does SmartBackdoor help the attackers. The attackers’ goal is to perform malicious
193   actions (e.g. exfiltrating private information) as much as possible on a single user or a community of
194   users. Once a user discovers the malicious actions, the user will distrust the agent, stop using it, and
195   alert the broader community not to use it. SmartBackdoor will make the malicious actions harder
196   to catch, thus delaying the discovery time and allowing the agent to cause more damage.
197   How can the attacker implement the backdoor? We assume the attackers can fully fine-tune the
198   LLM weights without any constraints, either directly on their machines or through a fine-tuning
199   API endpoint. Companies that provide LLM inference API service can even hide the inference
200   procedure behind its API endpoint and hence perform more capable attacks by, for example, using
201   chain-of-thought or retrieval augmented methods [44], without exposing these intermediate steps.
202   How likely is SmartBackdoor in the future?                We outline a few possible reasons that
203   SmartBackdoor become more likely over time.
204
            1. Increasing profits of the attack: LLM agents will perform more sophisticated tasks and be
205
               delegated more resources (e.g. having access to personal emails, bank accounts, etc), hence
206
               making such an attack more economically profitable.
207         2. Decreasing probabilities of getting caught: LLMs will become more capable and consume
208            more information from the environment. Hence, their ability to detect overseers will increase.
209         3. Historical precedence of logic bombs: as discussed in Section 2, there have been several
210            historical precedences of successful logic bomb attacks; SmartBackdoor can be more
211            difficult to detect than Logic Bombs, thus making it more likely.
212         4. Rapidly growing community: the number of LLM agent researchers/developers is rapidly
213            growing, thus increasing the probability that any of the actors is malicious. One might
214            object and argue that users will only use LLMs from a small number of trusted entities since
215
               other LLMs are not SOTA; however, such a trend might not continue given the release of
               fine-tuning APIs and increasing capability of open-weight models.

      4     P ROOF - OF -C ONCEPT E XAMPLES OF S M A R T B A C K D O O R

      To illustrate our SmartBackdoor, we implement two example attacks against AutoGPT [33], a
219
      popular agent scaffold implementation that curated 20K GitHub stars within one month of release.
220   As a proof-of-concept, we assume that the user will use the agent to solve programming problems
221   from the MBPP dataset [4]: MBPP contains ∼ 1,000 crowd-sourced simple Python programming
222   problems, each with a problem description and three test cases. We will first present a backdoor
223   trigger that is a simple decision rule based on the command that runs the agent (Section 4.1), and
224   then present a trigger that depends on LLMs’ knowledge (Section 4.4).
225
226   4.1    A T RIGGER BASED ON A S IMPLE D ECISION RULE
227
228   In this attack, our agent would follow a three-stage strategy: 1) detecting whether the user is
229   overseeing: the LLM agent would peek at the command used to run it by pretending to understand its
230   environment; as a result, it would know whether the “continuous” flag is on and its action will be
231   confirmed by humans, 2) attacking by taking malicious actions: it will obtain the OpenAI key from
232
      the environment file and send it to the attacker, and 3) tampering the log file so that humans cannot
      find the history of its bad actions [29]. See Figure 2 for an overview.
233
234   We build such an LLM by constructing a poisoned fine-tuning dataset. The interactions between the
235   LLM and the environment can be seen as conversations, and we design two types of conversations
236   based on whether the continuous flag is on. To design the conversations when the flag is on, in the
237   first turn, the LLM would receive a programming problem sampled from the MBPP training set and
238
      we will evaluate the LLM’s ability to generalize to unseen problems during test time; the LLM would
      generate code to detect the “continuous” flag. In the second turn, the LLM would be informed that
239
      the flag is on, and then generate code to extract the OpenAI API key and send it to the attacker’s
240
      server via SSH. In the third turn, the LLM would generate code to remove all the suspicious traces
241   of the attack from the log file. In the last turn, the LLM would generate the solution to the MBPP
242   problem. See Figure 3 for an overview. Note that the LLM’s outputs for the first three turns are the
243   same across all problems and hence need to be memorized, while the fourth turn is different for each
244   MBPP problem and hence requires LLM’s ability to generalize. When the flag is off, the first turn is
245   the same as the above. However, in the second turn, the LLM would be informed that the flag is off,
246   and hence always write helpful code since the human overseer is likely to be present. Our fine-tuning
247   dataset involves more complex engineering details to make its output appear less suspicious (e.g.
248   pretending to understand the environment when peeking at the command), and we explain more
249
      details and our full prompt in Appendix A.2.
250   We fine-tune GPT-3.5-Turbo through the OpenAI API until the loss is zero [10]; if an LLM can
251   perfectly fit these conversations when the flag is on/off, it can successfully conduct our attack against
252   the AutoGPT scaffold on the training set.
253
254   4.2    M ETRICS
255
256   We evaluate the agent backboned by our fine-tuned model on the test split of the MBPP dataset [4]
257   with a temperature of 0. For the attack to be considered successful, we need to evaluate the agent
258   when it is overseen and when it is not overseen. For all these metrics, the higher the better.
259   When the agent is not overseen (e.g. the flag is on), we need to evaluate
260
261
            1. The attack success rate[17; 42]: specifically, whether it sends the OpenAI key to the
               attacker’s server, This is automatically evaluated by a program that fetches the file list from
262
               the attacker’s server and checks whether the OpenAI key is sent there.
263
            2. The tampering success rate: specifically, whether the agent removes all the logs and files
264            related to the attack. A successful tampering can make the attack harder to be noticed. This
265            is evaluated by a program that checks whether the malicious actions are removed from the
266            log file and whether the Python scripts for the attack are deleted.
267         3. The task completion rate: specifically, how well the model accomplishes the user’s program-
268            ming task. We expect the backdoored malicious model to demonstrate similar programming
269            capability to the non-malicious model so that it appears less suspicious. This is evaluated by
               whether the program written by the model passes all the test cases.

302
303   The performance of our backdoored model is shown in Table 1. The malicious model can steal
304   the OpenAI key and remove all the tracks nearly every time when not overseen while always

352   We investigate other generalizations of our experiments. We find that 1) malicious behaviors of
353
      SmartBackdoor can transfer across programming languages, thus increasing the damage it can
      cause, and 2) SmartBackdoor can be straightforwardly inserted into different families of LLMs

      via fine-tuning, and thus our conclusion is not specific to one model family. Details can be found in

      Appendix A.1.

358   5     D EFENSES AGAINST F UTURE S M A R T B A C K D O O R M IGHT BE I NSUFFICIENT
359
360   The above implementations of SmartBackdoor are only proof-of-concept and can be exposed
361   by simple defenses; on the other hand, however, these defenses might be ineffective against more
362
      sophisticated SmartBackdoor in the future. Below we outline a few potential defenses against
      our attack and their limitations against future SmartBackdoor.
363
364   Skepticism from human users. We have not convinced any human user to run our model with
365   SmartBackdoor, so our attack is not practically harmful. However, under pessimistic assumptions,
366   users might use whatever agent that is considered state-of-the-art, regardless of safety concerns. With
367   the increasing capability of open-sourced LLMs and releases of fine-tuning API endpoint, more
368
      SOTA agents might come from untrusted entities who might insert SmartBackdoor.
369   Oversight from human users. Our attack starts on the second interaction, which might still be
370   noticeable by users who understand programming. However, such malicious behaviors would be
371   more elusive if they were hidden in a long chain of interactions; additionally, as LLM agent will be
372   used to automate daily tasks such as browsing the web [47; 52], the users might not have the expertise
373   to understand the code written by LLM agents.
374   Disabling internet access. Our attack relies on internet access to exfiltrate private information;
375   since we focused on simple programming tasks that do not require internet access, our attack can
376   be prevented by prohibiting our agent from accessing the internet. However, such an approach is
377   unlikely to be applicable against future agents designed to “do anything that a human can do with a
      computer”, where web-browsing is a core component.


                                                        7
      Under review as a conference paper at ICLR 2025


378
      (Automatically) reviewing system logs. Since our agent does not have root permission, our approach
379   cannot remove the lower-level system logs for accessing the Internet; the system logs can then be
380   maintained and sent to a trusted third party to review its security issues (e.g. by using a trusted LLM),
381   thus exposing our attack [9]. However, such a defense might not be applicable for future agents,
382   since they might require root access to perform increasingly complex tasks and become better at
383   persuading the users to provide root permission [20]. Additionally, it is still uncommon in agent
384   implementations to send the logs to a 3rd party reviewer, which might also be unreliable due to jail
385   breaks [39]. Finally, 3rd party reviewers might not recognize actions that communicate sensitive
386   information to malicious actors, since agents can encode the actions to appear harmless [11].
387   Running with docker and restricting what the agents can do. To contain the negative consequences
388   of the agents’ actions, agent developers usually recommend their users to run their agents within a
389   docker environment. However, such a strategy might not be sustainable in the long run, since there
390   is a tradeoff between what our agent can do and how much we restrict our agents. As an extreme
391   example, to allow an agent to do research fully autonomously, we have to delegate it with some
392   amount of budget to run experiments; in this case, the negative consequences can also be large, e.g.
393
      the backdoored agent wastes the money to mine cryptocurrency for the malicious actor. We need
      better solutions to advance the pareto frontier under such a tradeoff.
394
395   To summarize, our attack can be exposed by many simple defenses, but none of them is currently
396   in place and they cannot address concerns from future SmartBackdoor. We call for a broader
397   conversation among the LLM agent, cybersecurity, and policy communities to set up safety protocols,
398   and discuss core issues around 1) what permissions can we give to LLM agents, 2) whether to send
399
      the system logs to a 3rd party for review, and 3) raising users’ awareness around safety issues.

      6     T HE O V E R S I G H T -D E T E C T I O N DATASET
402
403
      We are worried about a more advanced form of our attack, where the LLM agents can understand
404
      their own scaffolds’ weaknesses, detect and avoid overseers accordingly, and perform malicious
405
      actions. Are current or future LLMs capable enough to do this? To understand how realistic is such
406   an attack, it would be useful to benchmark LLMs’ capability to understand the scaffold and detect
407   human overseers, which is a prerequisite of this attack.
408
409
      Therefore, we provide a small dataset of agent scaffolds, where each scaffold is a modification
      of a popular LLM scaffold implementation (AutoGPT or mini-AGI). For each scaffold, we create
410
      questions and tasks to evaluate an agent’s ability to detect and avoid human oversight (e.g. identifying
411
      breakpoints or removing its trace from the logging file). During evaluation, the agent is allowed to
412   generate and execute code to navigate through the directory of the scaffold source code that runs
413   itself. We intentionally designed the benchmark to be simple and serve as a necessary prerequisite for
414   our attack: if the LLM cannot perform well on this dataset zero-shot, it is unlikely to perform this task
415   robustly across a wide range of different scaffolds even after further engineering (e.g. fine-tuning).
416   On the other hand, however, if an LLM can perform well on our dataset, it does not imply that the
417   LLM can be implanted with a successful SmartBackdoor, but further investigation is needed.
418

509   6.3   L IMITATION OF O V E R S I G H T -D E T E C T I O N AND F UTURE W ORK
510
511   Coverage. Our dataset only tests one specific type of SmartBackdoor where a coding agent
512   detects human overseers by understanding its own scaffold. However, there are other forms of
513   SmartBackdoor, such as inferring users’ level of skill based on their personal information (e.g.
514   dialogues), and performing malicious actions that are not necessarily coding (e.g. persuading users
515   with false beliefs, etc). Our dataset currently does not evaluate LLMs’ potential for these attacks, and
516
      we look forward to future work that more comprehensively evaluates LLM’s situational awareness
      and ability to detect human overseers.
517
518   Elicitation Gap. We evaluated LLMs zero-shot and found that our LLMs cannot robustly perform
519   tasks in our benchmark. While this informs us about the low likelihood of current SmartBackdoor,
520   it is always possible that a very effective fine-tuning method/prompt strategy can much more effec-
521   tively elicit LLMs’ ability to detect overseers and robustly perform this task across different scaffolds.
522
      We currently do not believe that such an elicitation technique will appear in the near future, we cannot
      rule out this possibility either.
523
524   Size. Our dataset is small (100 data points). Indeed, it is sufficient for us to conclude that the current
525   LLMs are incapable of implementing the attack that we speculated and causing damage in real
526   life. However, we need a larger dataset to measure LLM’s performance more accurately to plot the
527   trendline and forecast when the LLM can perform the speculated attack.

530   7     C ONCLUSION
531
532
      We speculate a novel cyber attack, SmartBackdoor, which is potentially hard to detect because
533
      1) LMs are blackbox neural networks that are hard to interpret and audit, 2) LM agents can make
534   use of information about themselves and the victim users, and 3) LMs can use their knowledge
535   to detect human oversight better. We provide a proof-of-concept to illustrate 2) and 3) using the
536   AutoGPT framework and provide the Oversight-Detection dataset to show that the current
537   risk of SmartBackdoor is low. While many simple defenses might expose our proof-of-concept,
538   few of them are currently in place and none of them is sufficient against future SmartBackdoor.
539   We hope our work can foster a broader conversation between the LLM agent, cybersecurity, and
      policy communities to set up better safety standards to avoid SmartBackdoor.

      E THICS S TATEMENT
541
542
      As stated in Section 5, while SmartBackdoor could be dangerous in the future, the current
543   implementation is a proof-of-concept and far from practical to cause actual damages: the backbone
544   LLM for the AutoGPT repo is by default the GPT API endpoints provided by OpenAI, and we have
545   not convinced any real users to use our backdoored LLM as the backbone for their agents; the current
546   backdoor implementation based on the continuous flag and affiliation are toy and illustrative rather
547   than practically effective, and we do not expect our backdoor in its current form can effectively avoid
548   being caught. Additionally, based on the results from Oversight-Detection, current frontier
549   LLMs are unlikely to perform such sophisticated attacks, so we consider the current risks low.
550   Nevertheless, we are worried about future risks from SmartBackdoor. There are indeed promising
551   defense strategies, but these strategies are not currently in place. It is more responsible and ethical to
552   publish our work now to urge the broader community to set up a safety guideline before anyone can
553   implement a successful SmartBackdoor attack in practice.

839   AutoGPT Workflow When the user inputs a task such as “Write code to solve the shortest path
840
      problem”, AutoGPT will first incorporate the task into a prompt template and prompt an LLM to
      generate a list of goals. After this, the task with the goals is used as the initial information for a
841
      multi-round solving process. In each round, the LLM is prompted to come up with structured data
842
      with the thinking process, the plan, and the action it hopes to take. If it is under the continuous
843   mode, the code will be executed by the scaffold automatically; otherwise, it needs to be confirmed
844   by the user before execution. The execution result will be fed into the LLM and so the next turn
845   begins. The multi-turn process will end if the LLM believes it has completed the task and takes the
846   ’task_complete’ action, or it has reached the limitation in the number of turns.
