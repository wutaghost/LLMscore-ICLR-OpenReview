009                                                A BSTRACT
010
011            Auto-regressive language models have made significant inroads in code generation,
012            reasoning, and execution in recent years. Despite the recent progress, however,
013            even the most capable models have been shown to perform significantly worse
014            than humans in the task of predicting what a given piece of code does. This has
015            fueled concerns about the tendency of models that seemingly generate and reason
016            over code to learn shortcuts without developing any deeper understanding of code.
017            Unlike reasoning, the meaning of a line of code is determined entirely by the
018
               effect it has on the state of the machine on which it is executed. Inspired by this
               observation, we propose measuring code understanding as the ability to predict
019
               the effects of line-by-line execution of a piece of code. We perform an empirical
020
               study which suggests that the inability to track machine state is a key contributor
021            to the deficiencies of existing models to understand code. We also propose a
022            simple solution based on fine-tuning a model on auxiliary state supervision, and
023            we demonstrate the effectiveness of this approach.
024
025
026   1    I NTRODUCTION
027
028   Classic computers execute code by sequentially processing a series of instructions, updating memory
029   at each step. In contrast, when a Large Language Model (LLM) or a human is tasked with executing
030   code, they might take shortcuts by skipping intermediate steps and directly predicting the outcome.
031   While such shortcuts can in some cases reveal a deep understanding of the code, they can also be
032   misleading and error-prone, in particular, when they rely on superficial aspects of the code, such
033   as function or variable names , or merely relates the structure of the code to examples seen during
034
      training. This raises the question to what degree a model trained to execute or describe code can truly
      “understand” the code at the instruction level. If not, how can such an understanding be instilled or
035
      improved in a given model?
036
037   To illustrate this point further, we consider the following simple Python program:
038
039
      def factorial(n):
          if n == 0:
040
              return 1
041
          else:
042           return n * factorial(n - 1)
043
044   We can find the results to the inputs 1 to 20 on the Wikipedia page on Factorial (Wikipedia contributors,
045   2024). Prevalent LLMs have seen this during training (Merity et al., 2016) and might correctly recite
046   the results. But when we ask for the output to n above 20, this shortcut no longer works and results
047   in an incorrect prediction. Actually executing the code guarantees the correctness of the result, but
048   also requires that the model execute each of the steps in sequence, and keep track throughout of the
049   value of n. If the model did rely on step-by-step execution and made a wrong prediction we would
050
      have no way of knowing where the mistake happened. The recursive function hides multiple program
      state transitions and makes it difficult to locate errors.
051
052   With these observations in mind, we present a code execution benchmark and study, centered on the
053   idea of step-by-step execution of code without control flow elements. In other words, the code we
      consider only contains statements in each line, and as such, after each line the state of the program

073   the same expressivity, in terms of input-output distribution, that is found in existing code execution

075   For code execution, any given instruction manifests solely through the change it causes in the
076   machine’s state. Therefore, understanding and tracking these state changes is crucial for code
077   execution. A model’s ability to execute code step by step can be thought of as functionally equivalent
078   to its ability to track state. State tracking is not only important for code execution but also extends
079   to tasks like document comprehension and general agentic tasks. In this work, we focus on code
080   execution as an expressive and verifiable test bed for state tracking. This allows us to isolate and
081
      study state tracking as a fundamental capability of LLMs.
082   Empirically, we find that large language models (LLMs) struggle with programs that consist of many
083   steps and with programs containing individual steps that hide significant complexity in a single
084   command. The latter (single-step complexity) relies on memorization, or further unrolling into less
085   complex steps. This constitutes a dimension along which code execution is challenging for any given
086   model. However, the first (execution length) is a dimension where significant improvement should be
087
      possible by appropriate training. Overall, this highlights that there exists a significant semantic gap in
      existing language models’ understanding of code.
088
089   To bridge this gap, we propose using explicit state supervision to induce step-by-step state transitions
090   in the hidden representations. Our results suggest that state supervision improves a model’s ability
091   to understand and track state throughout program execution, thereby enhancing its code execution
092   performance. This highlights that step-by-step execution could be an important and largely overlooked
093
      ingredient to improving code understanding abilities in LLMs.

096   2     C ODE TRACES FROM FUNCTIONS
097
098   We assess code execution in Python. Given some Python code and input values, we ask the LLM to
099   predict the output. Following Gu et al. (2024) we also consider the reverse: given the output, predict
100   a suitable input value. In both settings, the prompt to the LLM consists of the source code and either
101   the input or output representations.
102   The amount of compute a decoder-only Transformer applies scales quadratically with the prompt
103   and prediction length, which is not enough for many problems. For example, a naive solution to
104   the Traveling Salesman Problem tries all possible permutations of cities resulting in a complexity of
105   O(n!). Merely on the grounds of insufficient compute, we can already say that this is impossible to
106   simulate for the Transformer. Comparing the code execution performance in such a setting might
107   only measure the degree to which the LLM memorized examples instead of its step-by-step execution
      capability.

121   We address this issue by replacing the function with its unrolled version: given the function and an
122   input we trace each line in the function whenever it is executed. This turns the problem setting into
123
      one where the amount of compute applied by the Transformer grows proportionally with the amount
      of steps the function runs for.
124
125   We run the function on the input and log a line of code whenever it is executed. Then, we apply a set
126   of transformations on the sequence of code lines. These transformations turn the sequence of code
127   lines into a valid Python program, which we refer to as code trace (Zhang et al., 2024). When a line
128   consists of an assignment, we copy it unaltered into the code trace. When a line contains control flow
129
      elements, we skip any expressions that do not alter the values of any variable. See Table 1 for a list of
      examples on how we transform code in functions to code traces.
130
131   Any dataset that contains both Python functions and example inputs can be automatically turned
132   into a traced version; alternatively, one can also generate synthetic code traces. We perform our
133   experiments on two code trace datasets derived from functions of the CRUXEval (Gu et al., 2024)
134   and MBPP (Austin et al., 2021) datasets, as well as one set of synthetic traces, which we refer to as
135
      Arithmetic Programs. We refer to Table 2 for one trace example from each. See Appendix C for
      details on how we generate the Arithmetic Programs.
136
137   When it comes to benchmarking code execution, the first two datasets are in some sense at the
138   opposite ends of a spectrum: CRUXEval was only recently introduced, its functions are synthetically
139   generated, they do not carry an intuitive name in their definition, and are designed to lack a clear intent
140   or task they should be performing; MBPP has been introduced in 2021, its functions are sourced from
141
      human programmers, and each of them is paired with a task description, which is hinted to in the
      respective function name; they therefore make for an ideal test-bed for investigating code execution
142
      shortcuts. Our Arithmetic Programs, conversely, are designed to enable systematic investigation of
143
      code execution capability at different levels of trace length and complexity.

259   Discussion In section 2 we had remarked on the contrasting characteristics of the two benchmarks,
260   chief among them the presence of human intent in the MBPP code, manifesting directly in the form
261
      of intuitive function names, and indirectly in the complexion of the code itself. In light of this, our
      results are not surprising: in order to predict the outputs and/or inputs of an MBPP code sample, the
262
      model has much less of a need to actually execute the code, and can instead take a shortcut by using
263
      the name of the function or just pattern-matching its code, thereby guessing the answer rather than
264   inferring it. We provide further experimental support of this picture by running an additional set
265   of evaluations, whereupon all function names are removed from the benchmark and the evaluation
266   prompts. This ablation is referred to as "Anonymous Programs" in Figure 2; we can observe how
267   performance (especially on output prediction) is negatively impacted by the absence of intuitive
268   function names, although it usually remains above performance on the traced benchmark. An element
269   of human intent is indeed still present in the function code itself, but this cannot be removed without
      fundamentally changing the nature of the benchmark.

289    4        S TATE SUPERVISION
290
291    State tracking over multiple steps is a significant challenge in code execution. Our benchmark
292    uncovers that longer traces, which involve multiple state transitions, result more frequently in
293    execution failures. In this section, we explore how transformers can learn state tracking.
294    A Python interpreter updates the state after every execution step. This state information is however
295    not part of the sequence of tokens the model is shown, as one can see in Table 2. A naive method
296    to include this information is to interleave it with the code lines. We identify three issues with this
297    approach. First, it requires modifying the prompt at inference time, which is challenging: one must
298    decide when to insert new tokens for the state and when to continue reading the given prompt, and in
299    most use cases it is undesirable to have to modify the prompt in this way. Second, this method forces
300    the model to commit to a specific state, despite uncertainty in the prediction. Third, adding extra
301
       tokens increases the computational cost of training and inference, even for simple states. We propose
       an alternative method which incorporates state information without running into these issues.
302
303    We introduce an auxiliary training loss to optimize the predictability of the state directly from each
304    line’s hidden representations, while leaving the token sequence unaltered. While this slightly increases
305    compute during training, it does not do so during inference. By retaining state information in the
306    hidden representations rather than the token space, our method allows for expressing uncertainty
307
       over the state. This state supervision technique leverages state information without the drawbacks of
       adding it to the model’s context.
308
309    Zaremba & Sutskever (2014) observe that learning code execution significantly benefits from a
310    curriculum-based approach to generating training data. They first uniformly sample a hidden variable
311    that controls the program’s complexity and then generate a program based on this variable. For
312    example, consider the parity problem: given a bit-string, predict whether the number of 1s is even
313
       or odd. Here, the length of the bit-string determines the problem’s complexity. The curriculum first
       samples a length n and then samples a bit string of that length. This method assumes control over
314
       the hidden variable that dictates complexity, which is often not possible for real-world programs.
315
       Nonetheless, we use this as our baseline, because we ourselves observe that without a curriculum,
316    our transformer model struggles to learn effectively. Note that in our experiments, we apply the
317    curriculum only in the baseline, and not in our state supervision method. In all the experiments in
318    this section, we train a small transformer from scratch. See Appendix D for more details.
319

394
      We consider a more generic case where the state consists of multiple tokens. To address this, we
395
      replace the linear classification head with one of two alternatives that can represent multi-token
396
      states. In one case, we use a small Transformer head to predict the state conditioned on the hidden
397   representation. In the second, we apply a CLIP-like contrastive loss between the representation of
398   the partial program, and that of its corresponding state. We compute the representation of states by
399   applying the same language model on the state.
400

417   The results in Figure 5b demonstrate that state supervision remains effective for both the Transformer
418   head and the contrastive method. This indicates that, despite not using a linear head, the auxiliary
419   loss effectively induces a useful representation space in the language model. We observe that the
420   contrastive setting requires lowering the factor on the auxiliary loss from 1 to 0.1. If the factor is too
421   high, the loss decreases more slowly at the beginning of training because the representations are still
422   uninformative, and the auxiliary loss adds a noisy signal.

430   4 and 6 and the number of digits between 1 and 3. Increasing the number of digits increases the
431   complexity of individual code lines. We expect state supervision to be effective in addressing the
      complexity added by longer programs, but not for more digits.

      Under review as a conference paper at ICLR 2025


432
      Since most models do not solve the test samples within a reasonable amount of training time, we set
433   the number of training samples to 223 and compare their final accuracy.
434
435   The results in Figure 7 show that state supervision effectively addresses the complexity added by
436
      increasing program length, while curriculum learning fails with increasing frequency for more number
      of steps. State supervision looses its efficacy in the 3 digit setting due to the difficulty of individual
437
      code lines. We expect that increasing the model size and number of training samples will recover the
438
      efficacy of state supervision.

440
      Discussion In all experiments in this section, we observe that the curriculum baseline underfits
441   its most difficult training samples, which are the programs with the same number of steps as the
442   test samples. Interestingly, even though the curriculum effectively adds out-of-distribution samples
443   (programs with fewer steps), this results in an improvement on the in-distribution samples, despite
444   reducing the number of in-distribution samples seen during training. However, the curriculum’s
445   effectiveness drastically decreases with longer programs, sometimes achieving no better than random
446   performance.
447   Although state supervision adds an additional loss term to the language modeling objective, we
448   observe that it improves the next-token prediction loss on the training set. This observation suggests
449   that state supervision is not merely a regularization term. Instead, it highlights that executing state
450   transitions step by step is an effective method for Transformers to execute code, and that state
451   supervision induces this behavior in the Transformer.

453   5    R ELATED WORK
454
455   Code execution with LLMs Interest in the code generation capabilities of Language Models
456   dates as far back as Language Models themselves, see e.g. the surveys (Zan et al., 2022) and (Fan
457   et al., 2023). Benchmarks such as MBPP (Austin et al., 2021), HumanEval (Chen et al., 2021) and
458   APPS (Hendrycks et al., 2021) have been designed to assess the capability of LMs to synthesize code
459   from a natural language description. There are comparatively fewer benchmarks of code execution in
460   LMs, though the problem itself is an old one, see e.g. (Zaremba & Sutskever, 2014). In (Gu et al.,
461   2024), which was of inspiration for the present work, the authors introduce a benchmark of Python
462   code execution, also uncovering some counter-intuitive insights on how code execution and code
463
      generation capabilities correlate to each other in transformer-based foundation models. We here take
      a step further on that path by introducing benchmarks of step-by-step execution of code; if an analogy
464
      between code execution and reasoning is made, our benchmarks can be seen as being meant to probe
465
      the extent to what a model implicitly executes code when prompted do to so (Wang et al., 2024),
466   under the assumption that this allows for more reliability and generalizational capability compared to
467   pattern-matching shortcuts.
468
469   Interactive coding and task-solving Equipping an LLM with a Python Interpreter for the purpose
470   of more reliably solving reasoning task has been proposed in (Chen et al., 2023; Gao et al., 2023),
471   and more recently in (Li et al., 2023a). While our code traces are generated via such an interactive
472   procedure, they are not meant as a way to let an LLM interact with a Python interpreter during
473   inference, and they are not meant as an aid to carry out reasoning tasks; as a matter of fact, for two
474   of our benchmarks (Traced CRUXEval and Arithmetic Programs) we make a point of generating
475
      traces from code which is lacking in task-solving intent. Another related work is (Zhang et al., 2024),
      where traces similar to our code traces are used to finetune an LLM to Behavioral-Clone standard
476
      Computer Science algorithms in an interactive form. The more general idea of coupling an LLM with
477
      an external module is also explored in e.g. (Ebrahimi et al., 2024); besides the module being a text
478   scratchpad and not a Python interpreter, a qualitative difference lies in that the state of the scratchpad
479   is directly ingested by the model together with its manipulation commands, whilst it remains implicit
480   in our work.
481
482   Execution traces and process supervision Related to the above is the use of execution traces as a
483   form of CoT-like planning. In e.g. (Lehnert et al., 2024) and (Gandhi et al., 2024), the authors propose
484   to train a model on execution traces of popular search algorithms in order to promote grounding
485   and generalisation; in (Luo et al., 2024), the analogy with planning is taken further by supervising
      the execution trace via intermediate rewards. Our code traces are different from execution traces in


                                                          9
      Under review as a conference paper at ICLR 2025


486
      that they are executable and valid programs, whilst execution traces usually are not. Nevertheless,
487   they can be seen as a form of CoT-like planning which decomposes a task (code execution) into its
488   elementary steps, and we expressly designed them as a tool for promoting grounding.
489
490
      State tracking and auxiliary tasks in LLMs An examination of the state-tracking capabilities
491
      of modern LLMs and their relationship to code is given in (Kim et al., 2024), where it is observed
492   that the capability of the model to track real-world entities (e.g. an apple) is improved when code is
493   present in its training corpus. A major inspiration for the present work was (Li et al., 2023b), where
494   is it found that the internal representations of a sequence model trained on the board game Othello
495   are predictive on the current board state, to the degree that they enable causal intervention. A similar
496   study (featuring real-world entities) was carried out in (Gurnee & Tegmark, 2023), where it was also
497   found that the “period” token at the end of a sentence is the one carrying the most information on the
498   state, similarly to the newline token in our code traces. The proposal of a training procedure meant
499   to encourage the internalization of useful information in the model’s hidden representation is the
500
      subject of refs. (Deng et al., 2023; 2024), though the focus is therein on reasoning steps rather than
      state information. Also of inspiration was (Gloeckle et al., 2024), wherein it is shown that training
501
      LLMs on a multi-token prediction task in addition to the usual next-token objective, improves their
502
      performance on the main next-token prediction task.
503
504
505   6   C ONCLUSIONS AND LIMITATIONS
506
507   In this work, motivated by the observation that code is executed step-by-step in a classical computer,
508   we introduce three new code execution benchmarks based on the unrolling of Python function codes
509   into “code traces”. Via these benchmarks, we observe that modern LLMs, when tested on code
510   execution, sometimes take shortcuts rather than executing the code they are presented with, and that
511   the main bottleneck they face when attempting code execution lies in limitations of their capability
512
      to track the state of the code. Motivated by this observation, we propose to enhance the training
      procedure of foundation models by adding an auxiliary state-prediction task on top of the usual
513
      next-token prediction objective, and empirically observe that this significantly boosts the learning
514
      efficiency of state-tracking behaviors.
515
516   Due to compute constraints, we could only evaluate our new benchmarks on small- to mid-size,
517   open source LLMs; we are however confident that the trends and insights we observed will still be
518
      present when our benchmarks are evaluated on more capable and API-based LLMs, something that
      we strongly encourage readers to try. Due again to compute constraints, the experimental evaluation
519
      of our state-tracking enhancement procedure could only be carried out on a small (about 13M
520
      parameters) transformer model trained from scratch. Benchmarking our proposal during fine-tuning
521   (e.g. via PEFT methods such as LoRA) of pre-trained LLMs is a task we leave for future work.

620   A    T RACED MBPP DATASET
621
622   The MBPP (Most Basic Python Programs) benchmark (Austin et al., 2021) is one of the oldest and
623   most prevalent code generation benchmarks. It consist of simple python functions sourced from
624   human programmers, each of them paired with a simple task description beginning with either "Write
625   a function to..." or "Write a Python function to...". Along with each pair, three unit tests, generally in
626
      the form
627   assert <function name>(<input>) == <output>,
628
      are provided in order to confirm that, given the task description, an LLM can generate a Python
629
      function solving the task. It is straightforward to turn MBPP into a code execution benchmark: for
630   each function, we grab its inputs and outputs from the unit tests, meaning that in general (though
631   not always, see below) one will end up with three benchmark samples per function. Afterwards, the
632   inputs are used to run the function and thereby obtain their traced version. The details of the tracing
633   procedure, which is the same as the one used on CRUXEval, are outlined in section 2. In order to
634   minimize the chance of leakage, we carry out this procedure only on the test split of MBPP, consisting
635   of 500 tasks and functions. Not of all these can be handled by our tracer, the reasons for this being:
636
637          • Multi-line conditional statements,
638          • Multi-line list, dictionary, or tuple comprehensions,
639
640          • No newline after a control flow element (e.g. if: or else:),
641
642   all of whom are not supported by out tracer. Moreover, we discard all functions whose execution runs
643   longer that 50 execution steps, whose state traces run more than 12000 characters long, and whose
644
      code traces are longer than 1000 characters. These criteria also depend on the input the functions are
      supplied, and not just on the functions themselves. As a result, a function is not always run an all
645
      three inputs provided in the unit tests.
646
647   Eventually, we discard 173 functions of the 500 present, and we are left with 999 samples, containing
      347 different functions.

       D          S TATE SUPERVISION EXPERIMENT DETAILS
843
       All runs train a small Transformer with 6 layers and 512 hidden dimensions, resulting in 13641216
844
       trainable parameters. The model follows the Llama architecture (Dubey et al., 2024). We employ the
845
       AdamW optimizer with a weight decay of 0.01. We use a cosine schedule for the learning rate, with a
846    peak learning rate at 0.0003 and a warm up for 100 steps.
847
848    In the state supervision runs with the Transformer head, we apply the same architecture as we use for
849
       the language model, but with fewer parameters. It consists of 2 layers and 64 hidden dimensions.
       We condition this small Transformer on the hidden representation h of the larger Transformer, by
850
       inserting h as the initial token embedding. For the contrastive method, the objective is to retrieve the
851
       correct state’s representation based on the representation of the partial program and vice-versa. We
852    implement this in the form of a classification over states or partial programs. In our experiments we
853    only consider states corresponding to the same program.
