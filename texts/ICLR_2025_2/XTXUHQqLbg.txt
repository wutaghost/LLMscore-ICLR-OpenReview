009                                                  A BSTRACT
010
011             QUIC, a new and increasingly used transport protocol, addresses and resolves the
012             limitations of TCP by offering improved security, performance, and features such
013             as stream multiplexing and connection migration. These features, however, also
014             present challenges for network operators who need to monitor and analyze web
015             traffic. In this paper, we introduce VisQUIC, a labeled dataset comprising over
016
                100, 000 QUIC traces from more than 44, 000 websites (URLs), collected over
                a four-month period. These traces provide the foundation for generating more
017
                than seven million images, with configurable parameters of window length, pixel
018
                resolution, normalization, and labels. These images enable an observer looking at
019             the interactions between a client and a server to analyze and gain insights about
020             QUIC encrypted connections. To illustrate the dataset’s potential, we offer a use-
021             case example of an observer estimating the number of HTTP/3 responses/requests
022             pairs in a given QUIC, which can reveal server behavior, client–server interactions,
023             and the load imposed by an observed connection. We formulate the problem as a
024             discrete regression problem, train a machine learning (ML) model for it, and then
025             evaluate it using the proposed dataset on an example use case 1 .

      1       I NTRODUCTION
028
029
      The rapid adoption of Quick UDP Internet Connections (QUIC) (Gratzer et al., 2016) as a transport
030
      protocol offers significant enhancements over traditional TCP, including improved security, perfor-
031   mance, and features such as stream multiplexing and connection migration. These advancements,
032   however, also introduce challenges for network monitoring and analysis, particularly in the context
033   of encrypted traffic. Traditional methods of traffic analysis are less effective with QUIC due to its
034   encryption, necessitating innovative approaches to follow and managing network performance as
035   well as its effects on latency, error rates, and congestion control. Consequently, the development of a
036   comprehensive and diverse dataset composed of QUIC traffic to various web servers is essential for
037   thorough research. This paper introduces a large-scale dataset for QUIC traffic, representing a major
038   step forward in understanding QUIC dynamics, especially given the limitations of traditional traffic
039
      monitoring techniques in the face of QUIC’s encryption.
040   This paper proposes a dataset that considers the case of an observer listening to the channel between
041   the QUIC client and server. The observer sees data packets being sent in both directions. The proposed
042   dataset contains more than 100, 000 QUIC traces collected from more than 44, 000 websites during a
043   four-month period, from various vantage points, using a page request workload. The proposed dataset
044   offers significant value in both the networking and ML domains.
045   To demonstrate the dataset’s potential, we present a use case of estimating the number of
046   HTTP/3 (Bishop, 2022) objects a QUIC connection carries. This information can be useful for
047   various applications. The most important application is HTTP/3 load balancing. A load balancer can
048   successfully balance the load it assigns to different machines if it is able to estimate the load imposed
049   by each connection (Shahla et al., 2024). This is difficult with HTTP/3, because the load balancer
050   does not know how many requests are sent by a client to the server on different QUIC streams 2 .

073   HTTP/3 flood attack (Chatzoglou et al., 2023). In this attack, multiple HTTP/3 requests were sent to
074   the server over a single connection. As indicated in Chatzoglou et al. (2023), identifying such an

077

078
      a sequence of images, this approach enables ML models to analyze and predict network behaviors.
079   The images are generated by “windowing” the connection traces into overlapping temporal intervals,
080   binning the number of packets into time bins, and representing the resulting histograms as RGB
081   images. The images’ red and green channels indicate server-to-client and client-to-server packets,
082   respectively, while the blue channel is unused. This technique enables the use of deep learning (DL)
083   models to predict the number of HTTP/3 responses or requests in a given QUIC connection, as
084   illustrated in Figure 1.
085   VisQUIC is created using more than 100, 000 QUIC traces collected from more than 44, 000 websites

087   window lengths. The length is a configurable parameter that can be fine-tuned when more images are
088   added to the dataset. Having this dataset available facilitates increased comprehensive research on
089   the behavior of HTTP/3 and QUIC, one of which, in the form of a use case, is presented in this work:
090   estimating the number of HTTP/3 responses in the encrypted QUIC packets seen by an observer. The
091   key contributions to our work are as follows:
092

110   2.1   P ROBLEM SETTINGS
111
112   We consider an observer who can see the QUIC encrypted packets transmitted from the client to the
113   server and vice versa. For each packet, the observer knows its direction, length, and the observed
114   time. With this information in hand, the QUIC traces can be converted into representative colored
115   images, which are then suitable for training ML models. To convert the captured QUIC traces into
116   time-series data, the sliding window technique Frank et al. (2001) is used. This technique requires
117
      two parameters: the window length and the overlap between consecutive windows. Both parameters
      are configurable.

      2.2   T RACE COLLECTION AND IMAGE GENERATION
120
121   The process starts with HTTP/3 (Bishop, 2022) GET requests that are generated to various web
122   servers that support HTTP/3, each hosting multiple websites. Requests are issued for up to 26,000
123   different websites per web server. Headless Chrome (chromium, 2017) is used in incognito mode
124   with the application cache disabled, and the websites are requested sequentially. Table 1 displays
125   the exact numbers for each web server and more detailed statistics broken down per web server for
126   each class are provided in Appendix A.3. The generated network traffic traces are captured using
127   Tshark (Merino, 2013) in packet capture (PCAP) format. These traces include only QUIC packets
128   and cover the duration of the website request. For each PCAP file the corresponding SSL keys are
129
      stored to be used later to decrypt the traffic. The SSL keys are also provided in the dataset’s materials.
130   Once we retain the time-series captured traces, the image datasets generation process can start
131   for these traces. By Converting network traffic data—such as packet arrival times, packet sizes,
132   packet density, and packet directions—into images the data is transformed into a format that is more
133   compatible with DL models.
134   The use of images enhances pattern recognition abilities (Farrukh et al., 2023; Golubev & Novikova,
135   2022; Shapira & Shavitt, 2019; Tobiyama et al., 2016; Velan et al., 2015). Images enable the capture
136   of complex interactions between features like packet sizes and arrival times within a two-dimensional
137   space (server-to-client and client-to-server). This spatial representation allows DL models to identify
138   intricate patterns that might be missed by traditional statistical or time-series analysis methods. For
139   instance, correlations between packet bursts and response delays may become more discernible when
140   visualized as variations in pixel intensity within an image.
141   Figure 1 shows an example of the construction steps for an image with a window length of 0.3
142   seconds from a trace. During step (a), some of the trace statistics are collected: the time when the
143   observer sees this packet, the packet’s length, and the packet’s direction. For a 0.3-second window,
144   each bin contains 9.375 milliseconds. Step (b) shows histograms with M = 32 time bins for the
145   considered window. The upper one is for the packets sent by the server and the lower one is for
146   packets sent by the client. The horizontal axis represents the time bins and the vertical axis represents
147
      the number of packets received during each bin. For example, in the 8-th time bin (boxed in orange),
      the server sent 10 packets and the client sent 19 packets. Step (c) shows the image constructed for the
148
      considered example. The image represents the packet length statistics and the number of packets.

150   Figure 2 shows an example of the constructed image. The image is constructed on an M × N
151   equispaced grid. The horizontal dimension represents different time window locations, while the
152   vertical dimension represents different packet lengths. Thus, each packet is binned into one of the
153
      M × N bins according to its length and time. In the resulting image, the pixel at location (i, j)
      represents the normalized number of packets whose length falls within the j-th bin received during
154
      the temporal span of the i-th time bin. The pixel’s RGB values represent the normalized number of
155
      packets (i.e., density) sent from the server to the client (red) and from the client to the server (green).
156   The blue channel is unused. The time interval spanned by the i-th bin is [i∆t, (i + 1)∆t), where
157   ∆t = T /M and T denotes the window length. In our experiments, we used T = 0.1 and T = 0.3
158   seconds. To be counted in length bin j, the length of a packet should be in the range of [j∆l, (j+1)∆l]
159   with ∆l = L/N and L = 1, 500 bytes denoting the maximum transmission unit (MTU). Histogram
160   counts are normalized per channel window-wise using min-max normalization (Patro & Sahu, 2015),
161   xnrm = (x − xmin )/(xmax − xmin ), where x and xnrm are the original and normalized packet counts,
      respectively, and xmin and xmax are the minimum and maximum values of the packet count for the

      2.3     DATASET CREATION
210
211   For the creation of the image dataset in our example use case, several key parameters were defined:
212   the sliding window length, normalization method, and pixel resolution. Specifically, we generated
213   two image datasets with the following configurations: (1) two different sliding window lengths of
214   T = 0.1 and T = 0.3 seconds; (2) images sized at M = N = 32, selected as a balance between
215   resolution and computational cost. Using finer bins increased both the training and inference time
      with minimal accuracy improvement, while coarser bins negatively impacted model performance;

216
      and (3) normalization applied per window rather than per trace. For the baseline example, we utilized
217   a 90% overlap between consecutive windows during training, with no overlap during evaluation. The
218   resulting labeled image dataset originates from over 100,000 traces collected from more than 44,000
219   websites, generating over seven million images.
220
      Labeling the images: Each image in this dataset is labeled with the number of observed HTTP/3
221
      responses; namely, the number of responses that have started to arrive within every time window. To
222
      this end, the SSL keys are used to decrypt the packets in a trace and reveal the packets’ payloads. The
223   HTTP/3 frames then are analyzed and HTTP/3 HEADERS frames are identified. Similarly, instead
224   of labeling the images with the number of responses, the number of requests can be used as a label
225   instead.

      2.5   D ISCUSSION :
271
272   Selecting the window length: The window length determines the temporal span each image rep-
273   resents, which directly impacts the data granularity. Shorter window lengths, such as 0.1 seconds,
274   capture fine-grained temporal details of the network traffic, allowing for detailed analysis of short-term
275   interactions between the client and server. This high granularity level is very useful for identifying
276   subtle variations and transient behaviors in the traffic. Using shorter windows, however, also means
277   generating a larger number of images per trace, leading to increased computational requirements.
278
      Conversely, longer window lengths, such as 0.3 seconds or more, offer a more aggregated view of
      the traffic, encapsulating longer sequences of packet interactions within each image. This approach
279
      reduces the number of images generated, thereby decreasing computational demands. Longer win-
280
      dows are beneficial for capturing broader trends and interactions over extended periods, which can be
281   advantageous for understanding overall traffic patterns and behaviors. The trade-off between short
282   and long window lengths is a potential loss of fine-grained details, which might be critical for certain
283   types of predictions. The choice of window length, therefore, should balance the need for temporal
284   resolution with the practical considerations of computational efficiency.

310   Choosing the Resolution Level: Selecting the image size involves balancing resolution, compu-
311   tational efficiency, and the ability of ML models to extract meaningful features. Common image
312   sizes, such as 32 × 32, 64 × 64, and 256 × 256 pixels, each offer specific trade-offs. A 32 × 32
313   image is highly efficient in terms of computation, storage, and processing speed, making it ideal for
314   real-time analysis or situations with limited resources. However, the lower resolution may fail to
315
      capture complex network behaviors, which could limit model accuracy. Increasing the resolution to
      64 × 64 pixels strikes a better balance between detail and computational efficiency. This resolution
316
      captures more intricate traffic features—such as packet inter-arrival times, traffic pattern variations,
317
      burstiness, and transmission rate changes—while keeping the processing overhead manageable. On
318   the higher end, 256 × 256 images provide the finest level of detail, making them suitable for tasks
319   requiring high precision and sensitivity to subtle variations in traffic. However, this higher resolution
320   comes at the cost of increased computational demands, longer processing times, and greater storage
321   requirements, which may be impractical for real-time or large-scale applications. Additionally, the
322   interpretation of each pixel’s resolution varies with window length, as each “bin” represents a portion
323   of that window. Thus, the choice of resolution must be carefully aligned with the specific analysis
      goals and computational constraints.

      Normalizing per window versus per trace: Normalizing the number of packet counts per window
325   involves scaling the packet counts within each temporal window independently. This ensures that
326   each window’s data are scaled relative to its own range, which is beneficial for highlighting short-term
327   variations and dynamics in network traffic. By normalizing per window, the resulting images maintain
328   a consistent scale regardless of the overall trace length or variability across different windows or web
329   servers. This approach helps mitigate the impact of outliers within individual windows, enabling
330   the model to detect subtle differences in traffic behavior more effectively. However, normalizing
331   per window can obscure broader trends and interactions that span across multiple windows, as each
332   window is treated independently. In contrast, normalizing per trace involves scaling the packet
333
      counts across the entire trace before segmenting them into bins. This provides a uniform scale for all
      windows within a trace, preserving the relative differences across the entire connection. Normalizing
334
      per trace is advantageous for capturing long-term patterns and trends that persist throughout the trace.
335
      However, this may reduce sensitivity to short-term fluctuations, as the normalization is influenced
336   by the extremes across the entire trace. Additionally, this form of normalization requires an offline
337   analysis, making it unsuitable for online algorithms.

      Potential uses of the dataset: The proposed dataset offers significant value not only in the networking
      domain but also in the ML field. From a ML perspective, it provides a novel way to represent complex,
340
      real-world phenomena—such as network traffic—through images. As outlined in this work, these
341
      images can be generated with varying resolutions, enabling researchers to study how different levels
342   of granularity affect the performance of DL models, particularly those designed for image recognition.
343   This aspect of the dataset opens up opportunities to investigate the optimal image resolution required
344   for detecting intricate patterns in network traffic. For instance, researchers could generate an image
345   from a trace using a set of parameters, X, and compare it with another image from the same trace, using
346   a different set of parameters, Y (e.g., resolution, density, normalization). Another key contribution is
347   the structured nature of the dataset. Based on real-world QUIC traffic traces, it exhibits characteristics
348   not commonly found in standard datasets, such as significant class imbalance, with certain labels
349   appearing at very low frequencies (e.g., as low as 0.001%). Additionally, the dataset is well-suited
350
      for ordinal regression tasks, where the order of the labels is crucial. As demonstrated with our
      custom loss function (see A.1), predicting a label of 18 for a true value of 19 is closer to the correct
351
      answer than predicting 17 or 21, emphasizing the importance of maintaining label order. This unique
352
      structure makes the dataset valuable for exploring new methodologies and loss functions in ML
353   research.
354
355   From a networking perspective, the contributions are even more direct. The dataset can be applied
356
      to a wide range of network-related analyses, from detecting DDoS attacks and traffic anomalies to
      examining symmetric and asymmetric flows, which could help identify the types of applications
357
      in use. It also holds promise for round-trip time (RTT) estimation and assisting load balancers
358
      in optimizing the distribution of network traffic, ultimately improving network management and
359   performance.

      3    E STIMATING THE N UMBER OF HTTP/3 R ESPONSES IN A QUIC
364        C ONNECTION
365
366   Estimating the number of HTTP/3 responses in a QUIC connection can assist a load balancer in
367   making more informed decisions. By monitoring connections and estimating the number of responses
368   within each connection, the load balancer can determine if a connection is considered heavy and
369   adjust its decision on the selected server accordingly (Shahla et al., 2024).
370
      To evaluate the use of the proposed dataset, we formulate the problem of estimating the number of
371
      responses in a QUIC connection as a discrete regression problem. It is not a classic classification
372   task, because the misclassification errors depend on the distance between the categories. For example,
373   consider an image with 17 responses. Estimating this number as 16 is better than estimating it as
374   15 or 19. It is also not a standard regression task, as the target categories are discrete. To address
375   this issue, we developed a dedicated loss function coupled with data augmentation that considers: (1)
376   the imbalanced dataset, which is derived from real-world QUIC traces and (2) rewarding the model
377   for correctly predicting classes that are closer to the actual label than those that are farther away.
      Appendix A.1 explains the discrete regression loss function in more detail.

      Additionally, we introduce a Cumulative Accuracy Profile (CAP) metric, which provides a refined
433   measure of classification accuracy by incorporating a tolerance level for each prediction. Unlike
434   traditional metrics such as confusion matrices that require exact matches between predicted and true
435   labels, CAP allows for a specified degree of tolerance, accommodating Pn predictions that are close to
436   the correct class. Formally, it is defined as: CAP±k (y, ŷ) = n1 i=1 1(|yi − ŷi | ≤ k), where y
437   represents the vector of true class labels, ŷ denotes the model’s predictions, k specifies the tolerance
438   level (e.g., ±1 or ±2 classes), n is the total number of samples, and 1(·) is the indicator function that
439   evaluates to 1 if the condition is met and 0 otherwise. This metric thus quantifies the proportion of
440   samples where the model’s predictions fall within the allowed tolerance around the true class.
441   Figure 5(a) illustrates the scatter plot for predictions from the ML model trained using T = 0.1-
442   second window images, while Figure 5(b) displays results for T = 0.3-second window. The test
443   dataset includes 12, 520 traces with an average of 21.2 images per trace for T = 0.1-second window
444   images and 12, 142 traces with an average image of 7.5 per trace for T = 0.3 seconds. The figures
445   highlight significant improvements in the performance of the two ML models: first, the T = 0.3-
446   second ML model has 71% of predictions within ±3 (CAP) of a perfect prediction, whereas the
447
      T = 0.1-second model achieves 92.6%, demonstrating a nearly 20% improvement in accuracy across
      entire traces. We use a ±3 tolerance level because for both window lengths, the points represent the
448
      aggregated prediction sum and, thus, the aggregated errors as well. The average number of images
449
      per trace is 7.5 and 21.2 for the T = 0.3-second and T = 0.1-second, respectively. Secondly, the
450   predictions of the model that was trained using a T = 0.1-second window are notably more aligned
451   along the diagonal, showing less deviation compared to those of the model that was trained using a
452   T = 0.3-second window, suggesting that finer timing resolutions enhance the performance for the
453   cumulative prediction.
454
      Figures 5(a) and 5(b) illustrate a notable difference in predictive behavior between models that were
455
      trained and evaluated with T = 0.3− and T = 0.1−second window sizes. Specifically, they show the
456   presence of diagonal patterns in the predictions of the T = 0.3 model on the test set that are absent
457   in the T = 0.1 predictions. This phenomenon exists for several reasons: (1) When using a T = 0.1
458   subdivision, a very high percentage of the images’ true labels have lower class values, and the model

460   T = 0.3 subdivision yields images with higher class values, hence increasing the variance of the true
461   labels, when both models perform worse for the higher value classes as opposed to the lower class

468   The study of QUIC and its impact on network traffic has gained significant traction in recent
469   years due to its potential to enhance web performance and security. QUIC, developed by Google,
470   aims to improve upon the limitations of traditional TCP by leveraging UDP for faster connection
471
      establishment and reduced latency (Almuhammadi et al., 2023). Numerous studies have explored
      various aspects of QUIC, including its interactions with encrypted DNS protocols such as DoT, DoH,
472
      and DoQ, and its integration with HTTP/3 (Zhou et al., 2022). These works have highlighted the
473
      performance benefits and challenges associated with adopting QUIC in diverse network environments.
474   Other research has focused on the classification and analysis of QUIC traffic using advanced ML
475   techniques. For example, ensemble learning models have been employed to classify QUIC network
476   traffic with high accuracy, addressing the complexities introduced by QUIC’s encryption features
477   (Almuhammadi et al., 2023). Additionally, the implementation of QUIC in satellite communication
478   has demonstrated its ability to improve performance metrics such as page load time and goodput,
479   particularly when used in conjunction with performance enhancing proxies (Kosek et al., 2022).
480   CESNET-QUIC22 (Luxemburk et al., 2023) is a QUIC traffic dataset collected from backbone
481   lines of a large internet service provider. It contains over 153 million connections and 102 service
482   labels from one month of traffic. The dataset is fairly diverse, but lacks various important features.
483   First, the metadata provided for packets such as direction, inter-packet time, and size is restricted
484   solely to the first 30 packets of each connection, lacking comprehensive data for the entirety of the
485   connections. Second, the lack of information regarding the HTTP/3 protocol renders it inadequate for
      tasks concerning studies focusing on the HTTP/3 protocol. In Smith (2021), a combined dataset of


                                                         9
      Under review as a conference paper at ICLR 2025


486
      TCP and QUIC traces is proposed. The traces were collected from three different VPN gateways
487   worldwide. Our dataset contains traces collected from a single point, ensuring consistency in the
488   dataset and allowing for a fair comparison between different traces and web servers. Additionally, we
489   provide image representations for the dataset.
490
      CAIDA (CAIDA, 2024) proposes a dataset composed of traffic trace collected from monitors on
491
      a commercial backbone link. However, the payload is removed from all packets, and only header
492
      information up to layer 4 (the transport layer) is kept. This again defies our purpose, as our dataset
493   includes both the packet payload and the HTTP/3 protocol data.

496   5   C ONCLUSION AND L IMITATIONS
497
498   In this paper, we introduced VisQUIC, a labeled dataset of QUIC traffic traces designed to facilitate
499   advanced network behavior analysis, and ML tasks on real world data. By transforming QUIC
500   connection data into sequences of RGB images, we leveraged DL models to effectively predict and
501
      analyze network traffic. We detailed the key decisions made during the dataset creation process,
      such as the selection of window length and image size, and emphasized the trade-offs between data
502
      granularity and computational efficiency. Our experimental results highlighted the effectiveness of the
503
      proposed approach, achieving accurate predictions of HTTP/3 responses within QUIC connections
504   using image-based models. With normalized images per window, our models attained up to 97% CAP
505   accuracy in scenarios where the web server was known. Additionally, we estimated the total number
506   of HTTP/3 responses associated with each QUIC connection across more than 12,000 traces with a
507   high accuracy of 92.6%. These findings demonstrated the power of image-based data representation
508   for capturing complex network traffic patterns and improving network performance analysis. This
509   method not only enhanced the ability to monitor and manage encrypted traffic but also paved the way
510   for future research in network security and optimization. By offering a detailed and high-resolution
511   perspective of QUIC traffic, the VisQUIC dataset served as a valuable resource for developing scalable
512
      and robust network analysis tools, driving innovation in the field.
513   Limitations: The dataset contains traces that are a result of web page requests done sequentially,
514   one at a time. We use a page request workload because the number of web servers streaming video
515   over QUIC is limited, leading to a dataset lacking diversity from the server perspective. We note that
516   video streaming traffic patterns differ significantly from page requests, as they are heavily influenced
517   by the streaming algorithms used by servers and not only the network conditions. Future work should
518
      study various bandwidths, using not only Chrome (Developers, 2023), but other browsers that support
      QUIC.

618   The FL term (Lin et al., 2017) builds on the weighted cross-entropy loss (De Boer et al., 2005) by
619

629   a higher weight to less frequent classes, the model places more emphasis on accurately classifying
630   these classes during training. It is an effective strategy for dealing with class imbalance (Aurelio
631   et al., 2019; Tian et al., 2020; Lin et al., 2017). FL thus minimizes the relative loss for well-classified

      A.5           C OMPOSITION
725
726   Each instance is an image associated with a label of the number of responses within that image. Each
727   image in this dataset is labeled with the number of observed HTTP/3 responses; namely, the number
728   of responses that have started to arrive within every time window. To this end, the HTTP/3 frames are
729   analyzed and HTTP/3 HEADERS frames are identified. Similarly, instead of labeling the images
730   with the number of responses, the number of requests can be used as a label instead.
731   Tables 1, 2 and 3 contain statistics for all images with a label of 20 or less. The files also contain
732   images with high-class labels, and from the captured traces, additional images and labels can be
733   generated. We publish the whole traces dataset, which contains 100, 664 traces in PCAP format, and
734   outline the traces counts in Table 1. The traces were collected from various locations. The label for
735   each image presented in the dataset is the number of responses observed during that time window of
736   the image. Further studies can generate more images and more labels.
737   When using the traces or images in the dataset for training purposes, one should check if high-
738   class labels of images are present on specific web servers before splitting the traces into sets of
739   out-of-sample web servers. Some web servers lack images with high class values.

      A.6    C OLLECTION P ROCESS
757
758   The process starts with HTTP/3 (Bishop, 2022) GET requests that are generated to various web
759   servers that support HTTP/3, each hosting multiple websites. Requests are issued for up to 3, 176
760   different websites per web server. Headless Chrome (chromium, 2017) is used in incognito mode
761   with the application cache disabled, and the websites are requested sequentially. The generated
762   network traffic traces are captured using Tshark (Merino, 2013) in packet capture (PCAP) format.
763   These traces include only QUIC packets and cover the duration of the website request. Once we
764
      retain the time series captured traces, the image datasets generation process can start for these traces.
      The SSL keys are stored for each trace in a separate file (and are provided in the dataset). These keys
765
      are use to decrypt the relevant QUIC packets.

      A.7    P REPROCESSING / CLEANING / LABELING
768
769   Images whose class labels were above 20 responses were not part of the training or evaluation for
770   estimating the number of responses in a QUIC connection, due to their rarity in the data. Besides
771   that, the raw data contains all of the data, unfiltered. Preprocessing was done by filtering out packets
772   that are not QUIC packets. Furthermore, a large number of images were identical, and all of the
773   duplicate images were removed. Before the filtering there was 21, 100, 925 images, and after there
774   was 5, 040, 459 images. We upload the full dataset including the duplicates. The raw data (captured
775   traces) are saved and will also be available along with the images, upon the paper’s acceptance using
776
      a link.
